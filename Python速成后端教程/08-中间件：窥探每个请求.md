## 三、分步操作

### 步骤 1：中间件是什么？——快递站安检机的故事

想象你开了一家快递站。每天有成千上万个包裹（请求）进来，你的员工（接口函数）负责处理每个包裹——该签收的签收、该退回的退回。

但有一天，老板提了两个要求：

1. **每个包裹进站时，在系统里记一笔**：什么时间、谁寄的、多大、多久处理完。
2. **同一家快递公司 10 秒内最多收 5 个包裹**，超过就拒收——防止有人恶意塞包裹把站点堵死。

你的第一反应可能是：在每个员工的工位上贴一张纸条，提醒他们"处理包裹前先记日志"、"收包裹前先检查是不是同一家公司的"。

**但如果有 20 个员工、50 个工位呢？** 你贴得过来吗？万一有人没看到纸条呢？万一哪天规则变了，你要把所有纸条全部重贴一遍？

**更好的办法是在快递站门口装一台安检机。** 所有包裹进来，不管最终由哪个员工处理，都必须先过安检机。安检机负责：
- 扫描包裹，记录日志（谁寄的、什么时候进来的）
- 检查规则，决定放行还是拒收

**这个安检机，就是中间件（Middleware）。**

```
所有请求 → [安检机①] → [安检机②] → [安检机③] → 你的接口函数
                                                      │
所有请求 ← [安检机①] ← [安检机②] ← [安检机③] ← 你的接口函数返回
```

每一条请求进来，**依次穿过所有中间件**，然后到达你的接口函数。函数的返回值再**原路返回，依次穿过所有中间件**，最后发回给客户端。

**中间件的核心特点**：
- 对所有请求生效（不需要在每个接口里单独写逻辑）
- 按注册顺序依次执行（先注册的先跑）
- 可以做请求前处理（记日志、检查权限、限流），也可以做响应后处理（压缩数据、加安全头）

### 步骤 2：写第一个中间件——请求日志中间件

现在我们来写一个最简单的中间件：**记录每个请求的方法、路径、状态码和耗时，打印到终端。**

#### 2.1 新建 `middleware.py`

在 `blog_backend/` 目录下新建一个文件，命名为 `middleware.py`：

```python
import time
from fastapi import Request


async def log_requests(request: Request, call_next):
    """
    请求日志中间件：
    记录每个请求的方法、路径、状态码和耗时。
    """
    # 1. 记录请求开始时间
    start_time = time.time()

    # 2. 调用下一个中间件（或最终的接口函数）
    response = await call_next(request)

    # 3. 计算耗时
    duration = time.time() - start_time
    duration_ms = round(duration * 1000, 2)  # 转换成毫秒，保留两位小数

    # 4. 打印日志
    print(f"{request.method} {request.url.path} {response.status_code} {duration_ms}ms")

    # 5. 返回响应（必须！）
    return response
```

**逐行解释**：

- `import time`：Python 内置的时间模块，我们用 `time.time()` 来记录时间戳（精确到秒的小数点后）。
- `from fastapi import Request`：FastAPI 的 `Request` 对象，它包含了当前请求的所有信息——方法、路径、请求头、客户端 IP 等等。
- `async def log_requests(request: Request, call_next):`：中间件函数的签名。注意两个参数：
  - `request: Request`：当前请求对象，FastAPI 自动传进来。
  - `call_next`：一个**回调函数**。调用它，就是"把请求传给下一个中间件（或最终的接口函数）"。这就像快递站的传送带——你检查完包裹，按一下按钮，包裹就传送到下一站。
- `start_time = time.time()`：记下"现在几点"，精确到毫秒。
- `response = await call_next(request)`：**这是中间件最关键的一行。** 把请求传给下游，等下游处理完，拿到响应。`await` 是因为中间件是异步的——传送带上的包裹在下一站处理时，你在原地等着。
- `duration = time.time() - start_time`：用当前时间减去开始时间，算出耗时。
- `print(...)`：把日志打印到终端。
- `return response`：**必须返回 response！** 如果不返回，客户端就收不到任何数据了——想象快递站的安检机把包裹吞了，不还给传送带。

> **为什么是 `async`？** FastAPI 的中间件是异步的，这意味着你的日志记录不会阻塞其他请求的处理。你可以把它理解成"一边等包裹过下一站，一边可以看看下一个包裹"——但为了简单，本教程的中间件都是顺序等待的，不影响你理解核心概念。

#### 2.2 在 `main.py` 中注册中间件

打开 `blog_backend/main.py`，在 `app = FastAPI()` 后面加上一行注册代码：

```python
from fastapi import FastAPI
from fastapi.responses import JSONResponse
from blog_backend.middleware import log_requests  # 新增：导入中间件

app = FastAPI()

# 新增：注册中间件
app.middleware("http")(log_requests)


@app.get("/")
def read_root():
    return {"message": "Hello World"}


# ... 其余代码保持不变
```

> **注意**：`app.middleware("http")(log_requests)` 这一行必须放在 `app = FastAPI()` 之后，所有路由（`@app.get`、`@app.post` 等）之前。放在路由之后也能工作，但放在前面更符合"安检机在大门口"的直觉。

保存文件。因为开启了 `--reload`，服务器会自动重启。

#### 2.3 测试：看终端里的日志

现在访问 `http://127.0.0.1:8000`，然后切回终端。你会看到类似这样的输出：

```

```

再访问 `http://127.0.0.1:8000/docs`，终端会多出好几行（因为 Swagger 页面会发多个请求来加载资源）：


用 curl 也试试：

```bash

```

终端输出：


**你什么都没在接口函数里做，日志就自动打印了。** 这就是中间件的威力——它像一个透明的玻璃罩，罩在所有接口外面，你看不到它，但它确实在工作。

> **日志格式的含义**：`GET / 200 3.21ms` 拆开看——
> - `GET`：请求方法
> - `/`：请求路径
> - `200`：响应状态码
> - `3.21ms`：处理耗时（毫秒）

### 步骤 3：逐行拆解 `@app.middleware("http")` 装饰器

在第 02 章我们见过 `@app.get("/")`，你当时只需要知道"它是一个路牌"。现在我们来认真看看 `@app.middleware("http")` 这个装饰器变体。

**它的写法不是 Python 标准的 `@` 语法，而是函数调用形式**：

```python
app.middleware("http")(log_requests)
```

拆开看：

- `app.middleware("http")`：调用 `app` 的 `middleware` 方法，参数 `"http"` 表示"我要注册一个 HTTP 中间件"。FastAPI 还支持 WebSocket 中间件（参数是 `"websocket"`），但本教程只讲 HTTP。
- `(...)(log_requests)`：`app.middleware("http")` 返回了一个**装饰器函数**，然后把这个装饰器函数应用在 `log_requests` 上。等价于：

```python
# 等价写法（不推荐，但帮你理解）
@app.middleware("http")
async def log_requests(request, call_next):
    ...
```

两种写法完全等价，但把注册和函数定义分开更清晰——特别是当中间件定义在别的文件里（比如 `middleware.py`）时。

> **为什么不是 `@app.middleware("http")` 直接写在 `log_requests` 上面？** 因为 `log_requests` 定义在 `middleware.py` 里，它没有 `app` 这个变量。我们在 `main.py` 里有 `app`，所以注册逻辑放在 `main.py` 里更合理。

### 步骤 4：认识 `Request` 对象——中间件能看到什么？

中间件函数收到的 `request: Request` 对象里，装着当前请求的**所有信息**。以下是最常用的几个属性：

| 属性 | 类型 | 含义 | 示例值 |
|------|------|------|--------|
| `request.method` | `str` | HTTP 方法 | `"GET"`, `"POST"`, `"DELETE"` |
| `request.url.path` | `str` | URL 路径 | `"/"`, `"/blogs"`, `"/blogs/5"` |
| `request.url.query` | `str` | 查询字符串 | `"page=2&size=5"`（如果是 `/blogs?page=2&size=5`） |
| `request.client.host` | `str` | 客户端 IP 地址 | `"127.0.0.1"` |
| `request.headers` | `dict` | 请求头 | `{"user-agent": "curl/8.x", ...}` |
| `request.state` | `object` | 请求级别的临时存储（后面会用到） | — |

我们在步骤 2 的日志中间件里已经用了 `request.method` 和 `request.url.path`。接下来在步骤 5 的限流器里，我们会用到 `request.client.host` 来获取客户端 IP。

### 步骤 5：写一个简易限流器——10 秒内同一 IP 最多 5 次请求

现在来写一个更实用的中间件：**限流器（Rate Limiter）**。它的作用是防止某个客户端（同一 IP）在短时间内发太多请求，把服务器打垮。

> ⚠️ **重要声明**：本节实现的限流器用 Python 字典存储 IP 的访问计数，数据存在内存里。**重启服务器就会清零，多进程不共享，而且没有过期自动清理机制。** 这是教学演示用的简化版，帮助你理解限流的核心原理。生产环境请使用 Redis + 专业限流算法（如令牌桶、滑动窗口），本教程不展开。

#### 5.1 在 `middleware.py` 中追加限流器

打开 `blog_backend/middleware.py`，在 `log_requests` 函数后面追加以下代码：

```python
import time
from fastapi import Request
from fastapi.responses import JSONResponse

# 限流器存储：以 IP 为 key，访问记录为 value
# 结构：{"127.0.0.1": [时间戳1, 时间戳2, 时间戳3, ...]}
_rate_limit_store: dict[str, list[float]] = {}

# 限流配置
RATE_LIMIT_WINDOW = 10      # 时间窗口：10 秒
RATE_LIMIT_MAX_REQUESTS = 5  # 最大请求数：5 次


async def rate_limiter(request: Request, call_next):
    """
    简易限流中间件：
    同一 IP 在 10 秒内最多 5 次请求。
    超过限制返回 429 Too Many Requests。
    """
    # 1. 获取客户端 IP
    client_ip = request.client.host if request.client else "unknown"

    # 2. 获取当前时间
    now = time.time()

    # 3. 获取该 IP 的历史访问记录，没有就初始化为空列表
    if client_ip not in _rate_limit_store:
        _rate_limit_store[client_ip] = []

    # 4. 清理时间窗口之外的旧记录（只保留最近 10 秒内的）
    _rate_limit_store[client_ip] = [
        t for t in _rate_limit_store[client_ip]
        if now - t < RATE_LIMIT_WINDOW
    ]

    # 5. 检查是否超过限制
    if len(_rate_limit_store[client_ip]) >= RATE_LIMIT_MAX_REQUESTS:
        # 返回 429 错误，不放行
        return JSONResponse(
            status_code=429,
            content={
                "error": "Too Many Requests",
                "message": f"同一 IP 在 {RATE_LIMIT_WINDOW} 秒内最多 {RATE_LIMIT_MAX_REQUESTS} 次请求",
                "retry_after_seconds": RATE_LIMIT_WINDOW,
            },
        )

    # 6. 没超限，记录本次访问时间
    _rate_limit_store[client_ip].append(now)

    # 7. 放行，调用下一个中间件/接口
    response = await call_next(request)
    return response
```

**逐行解释**：

- `_rate_limit_store: dict[str, list[float]] = {}`：一个 Python 字典，key 是 IP 地址，value 是一个时间戳列表（记录该 IP 每次访问的时间）。变量名前面加 `_` 是 Python 约定——表示"这是模块内部用的，外部不要直接碰"。
- `RATE_LIMIT_WINDOW = 10` 和 `RATE_LIMIT_MAX_REQUESTS = 5`：限流参数，集中定义方便修改。
- `client_ip = request.client.host`：从请求对象中获取客户端 IP。`if request.client else "unknown"` 是防御性写法——万一某些极端情况下 `request.client` 为 `None`（比如通过代理），不会报错。
- 步骤 4 的清理逻辑：用一个列表推导式 `[t for t in ... if now - t < 10]`，把所有超过 10 秒的旧记录删掉。这样字典不会无限膨胀。
- 步骤 5：如果该 IP 在 10 秒内已经有 5 条记录，直接返回 `429 Too Many Requests`，**不调用 `call_next`**——包裹被安检机截住了，根本没送到员工手上。
- 步骤 6：没超限，把当前时间戳追加到记录列表里。
- 步骤 7：调用 `call_next`，放行。

> **`429 Too Many Requests` 是什么？** 429 是 HTTP 标准状态码，专门用于限流场景。意思是"你请求太多了，等一会儿再试"。客户端收到 429 后，通常会等一段时间再重试。

### 步骤 6：中间件执行顺序——先注册的先跑

现在你的 `main.py` 里有两个中间件：`log_requests` 和 `rate_limiter`。它们按**注册顺序**执行：

```
请求进来
  │
  ▼
log_requests  → 打印"开始处理"
  │
  ▼
rate_limiter  → 检查是否超限
  │
  ├── 超限 → 直接返回 429，不继续往下
  │
  └── 没超限 → 继续
       │
       ▼
    你的接口函数 → 处理请求，返回数据
       │
       ▼
rate_limiter  ← 拿到响应（但不做任何处理）
       │
       ▼
log_requests  ← 计算耗时，打印日志，返回响应
       │
       ▼
客户端收到响应
```

**关键观察**：当限流器拦截了请求（返回 429），日志中间件依然会记录它——因为日志中间件在限流器外面，请求先穿过日志中间件，再穿过限流器。限流器返回的 429 响应会原路返回，穿过日志中间件时被记录。

你可以在终端里验证：疯狂刷新时，你会看到类似这样的日志：

```
GET / 429 0.15ms
GET / 429 0.12ms
GET / 429 0.11ms
```

限流器拦截的请求，日志中间件也忠实地记录了下来——状态码是 429，耗时极短（因为限流器直接返回了，没走到接口函数）。

#### ❌ 常见错误：中间件顺序搞反了

❌ 错误示例：

```python
# 先注册限流器，后注册日志中间件
app.middleware("http")(rate_limiter)
app.middleware("http")(log_requests)
```

**后果**：限流器在日志中间件外面，所以限流器拦截的 429 请求**不会被日志中间件记录**——因为请求在到达日志中间件之前就被截住了。这可能不是你想要的行为。

✅ 正确示例（如果你想记录所有请求，包括被拦截的）：

```python
# 日志中间件在最外层，记录所有请求
app.middleware("http")(log_requests)
app.middleware("http")(rate_limiter)
```

**规则**：多个中间件的执行顺序取决于注册顺序，先注册的在外面，后注册的在里面。想象洋葱——一层一层剥开，再一层一层合上。

### 🤔 想多一点：中间件 vs 依赖注入——什么时候用哪个？

现在你手里有两把"瑞士军刀"：中间件（Middleware）和依赖注入（Dependency Injection，后面章节会详细讲）。它们都能在请求到达接口函数之前做一些事情，但它们的设计哲学完全不同。

#### 比喻：公寓楼的门禁 vs 房间里的插座

**中间件**就像公寓楼大门口的门禁系统。每个进楼的人，不管你去几楼、去哪个房间，都必须先过门禁。门禁不管你最终要去哪，它只做一件事——检查你有没有门禁卡（权限）、记录你几点进楼（日志）、限制同一张卡 10 秒内只能刷一次（限流）。

**依赖注入**就像房间里的插座。每个房间（接口）**需要什么就插什么**——这个房间需要插台灯（数据库连接），那个房间需要插充电器（当前用户信息）。不是所有房间都需要同一个电器，每个房间按需使用。

#### 核心区别

| 维度 | 中间件（Middleware） | 依赖注入（Dependency Injection） |
|------|---------------------|----------------------------------|
| **作用范围** | 对所有请求生效（全局） | 只对声明了依赖的接口生效（按需） |
| **何时执行** | 在路由匹配之前就执行 | 在路由匹配之后，进入函数之前执行 |
| **能决定"放不放行"吗** | 能——可以不调用 `call_next`，直接返回错误 | 通常不能直接拦截请求，但可以抛异常 |
| **能访问请求数据吗** | 能——有完整的 `Request` 对象 | 能——FastAPI 的依赖函数也能访问 `Request` |
| **适合做什么** | 日志、限流、CORS 头、全局异常捕获 | 获取当前用户、数据库会话、权限校验 |
| **比喻** | 公寓楼大门口的门禁 | 房间里按需插入的插座 |

#### 什么时候该用中间件？

- **需要拦截所有请求**：比如限流、IP 黑名单、全局 CORS 处理。
- **需要在请求处理前后都做事情**：比如日志记录（记开始时间 + 结束时间）、性能监控。
- **跟业务逻辑无关的基础设施**：压缩响应内容、添加安全头（`X-Content-Type-Options` 等）。

#### 什么时候该用依赖注入？

- **只有部分接口需要**：比如只有管理后台的接口需要管理员权限，普通接口不需要。
- **需要获取数据库连接或当前用户这种"有状态"的东西**。
- **需要灵活组合**：一个接口可以注入多个依赖，不同接口注入不同的依赖。

#### 一句话总结

**中间件是"无差别安检"，依赖注入是"按需取工具"。** 如果你想让所有请求都过一遍（而且不关心它最终去哪个接口），用中间件。如果只有部分接口需要某个功能，用依赖注入。

## 四、完整代码清单

### `blog_backend/middleware.py`（新建文件）

```python
import time
from fastapi import Request
from fastapi.responses import JSONResponse


async def log_requests(request: Request, call_next):
    """
    请求日志中间件：
    记录每个请求的方法、路径、状态码和耗时。
    """
    start_time = time.time()
    response = await call_next(request)
    duration = time.time() - start_time
    duration_ms = round(duration * 1000, 2)
    print(f"{request.method} {request.url.path} {response.status_code} {duration_ms}ms")
    return response


# 限流器存储：以 IP 为 key，访问记录为 value
_rate_limit_store: dict[str, list[float]] = {}

# 限流配置
RATE_LIMIT_WINDOW = 10      # 时间窗口：10 秒
RATE_LIMIT_MAX_REQUESTS = 5  # 最大请求数：5 次


async def rate_limiter(request: Request, call_next):
    """
    简易限流中间件：
    同一 IP 在 10 秒内最多 5 次请求。
    超过限制返回 429 Too Many Requests。
    """
    client_ip = request.client.host if request.client else "unknown"
    now = time.time()

    if client_ip not in _rate_limit_store:
        _rate_limit_store[client_ip] = []

    # 清理时间窗口之外的旧记录
    _rate_limit_store[client_ip] = [
        t for t in _rate_limit_store[client_ip]
        if now - t < RATE_LIMIT_WINDOW
    ]

    # 检查是否超过限制
    if len(_rate_limit_store[client_ip]) >= RATE_LIMIT_MAX_REQUESTS:
        return JSONResponse(
            status_code=429,
            content={
                "error": "Too Many Requests",
                "message": f"同一 IP 在 {RATE_LIMIT_WINDOW} 秒内最多 {RATE_LIMIT_MAX_REQUESTS} 次请求",
                "retry_after_seconds": RATE_LIMIT_WINDOW,
            },
        )

    # 记录本次访问
    _rate_limit_store[client_ip].append(now)

    response = await call_next(request)
    return response
```

## 七、术语附录

| 术语 | 英文 | 通俗解释 | 本章出现位置 |
|------|------|----------|-------------|
| 中间件 | Middleware | 在请求到达接口函数之前和之后执行的"钩子"函数。就像快递站的安检机——每个包裹进来都要先过它。可以修改请求、拦截请求、记录日志、添加响应头等。**字面陷阱**：它不是"中间的那个部件"——它是"夹在请求和接口之间的那一层"。 | 步骤 1 |
| 限流 | Rate Limiting | 限制某个客户端在单位时间内的请求次数。比如"同一 IP 10 秒内最多 5 次请求"。目的是防止恶意攻击或某个客户端无意中打垮服务器。**字面陷阱**：不是"限制流量"（那是带宽控制），而是"限制请求频率"。 | 步骤 5 |
| 依赖注入 | Dependency Injection（DI） | 一种设计模式：接口函数声明自己需要什么（比如"我需要数据库连接"），框架自动把对应的东西传进来。和中间件的区别：中间件对所有请求生效，依赖注入是"按需声明、按需注入"。 | 步骤 7（🤔 想多一点） |
| 429 状态码 | 429 Too Many Requests | HTTP 状态码，表示"你请求太多了，歇会儿"。客户端收到 429 后，通常会等待一段时间再重试。响应中通常包含 `Retry-After` 头，告诉客户端等多久。 | 步骤 5 |
| 洋葱模型 | Onion Model | 描述中间件执行顺序的比喻：请求从最外层中间件进入，一层层剥到核心（接口函数），响应再从核心一层层返回。就像一个洋葱，从外到内，再从内到外。 | 步骤 6 |

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| 忘记调用 `call_next` | 请求一直转圈，最终超时 | 中间件没有把请求传给下游，请求卡在中间件里了 | 确保函数里有 `response = await call_next(request)` |
| 忘记 `return response` | 客户端收到空响应或报错 | 中间件拿到了响应但没有返回给调用者 | 确保函数最后有 `return response` |
| 中间件注册顺序搞反 | 限流器拦截的请求没被日志记录 | 限流器在日志外面，请求在到达日志前就被截住了 | 日志中间件注册在最外层 |
| 限流器用字典存储，重启清零 | 重启服务器后限流计数归零 | 数据在内存里，不是持久化存储 | 这是教学版的限制，生产环境用 Redis |
| 限流器没有自动清理过期数据 | 字典越来越大，内存占用上升 | 没有后台任务清理过期 IP | 代码里用列表推导式在每次请求时清理，但长期不活跃的 IP 会残留。教学版够用，生产环境用 Redis 的 TTL。 |
| 用 `print` 做日志 | 终端输出杂乱，无法按级别过滤 | `print` 没有日志级别，没有时间戳格式化 | 本教程为简化使用 `print`，生产环境改用 `logging` 模块 |
| `--reload` 模式下中间件不生效 | 改了中间件代码，服务器没重启 | 热重载不是 100% 可靠，特别是改了被导入的模块 | 手动 `Ctrl+C` 停止，再重新 `uvicorn blog_backend.main:app --reload` |
