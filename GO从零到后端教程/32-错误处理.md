# 第32章 · 错误处理

> "如果你写Java，可能习惯了try-catch——出了问题就'抛'出去，让上层去'接'。Go不这样。Go的做法是：错误就是普通的返回值，你每次函数调用后老老实实检查一下。这很啰嗦，但也很坦率——错误就在你面前，你无法假装看不见。"

---

## 32.1 error接口：Go错误机制的基石

Go的error就是一个接口，极其简单：

```go
type error interface {
    Error() string
}
```

就一个方法 `Error() string`。任何有这个方法的东西，都是一个error。

你甚至可以直接拿string来当error用：

```go
type myError string

func (e myError) Error() string {
    return string(e)
}
```

当然，标准库已经做好了，你不需要自己写。

## 32.2 创建错误

### errors.New：最基础的错误

```go
import "errors"

err := errors.New("文件不存在")
fmt.Println(err)
```

输出：`文件不存在`

### fmt.Errorf：带格式化的错误

```go
err := fmt.Errorf("用户 %s 不存在", userName)
fmt.Println(err)
```

输出：`用户 张三 不存在`

`fmt.Errorf` 就像 `fmt.Sprintf` + `errors.New`。

### 何时创建

错误不是用来"标记"正常流程的——错误就是错误：

```go
var ErrNotFound = errors.New("记录不存在")
var ErrPermissionDenied = errors.New("权限不足")
var ErrInvalidInput = errors.New("输入不合法")
```

这些预定义的错误叫 **sentinel error**（哨兵错误），用于调用方判断错误的类型。

## 32.3 错误处理模式：if err != nil

这就是Go代码中最常见的三行：

```go
result, err := doSomething()
if err != nil {
    return err
}
```

之后继续用 `result`。

### 为什么这么啰嗦？

很多从其他语言转来的人觉得这很烦："到处都是 `if err != nil`，代码有一半是错误处理。"

Go的设计者认为：**错误不是特殊情况，它是正常流程的一部分。** 文件不存在、网络超时、用户输入不合法——这些都是预期内可能发生的事，你每次函数调用都应该考虑"如果失败了怎么办"。

`try-catch` 的问题在于：你可以在一个大 `try` 块里写几十行代码，然后在最后的 `catch` 里统一处理。这看起来很"省代码"，但它让错误被"藏"起来了——你看不到哪一行可能出错，也搞不清出了错应该怎么恢复。

Go的显式错误检查虽然啰嗦，但让代码的执行路径完全透明。

### 不要忽略错误

```go
result, _ := doSomething()
```

**永远不要用 `_` 忽略错误！** 除非你非常确定这个函数不可能失败（几乎没有什么函数是绝对不可能失败的）。

即使是 `fmt.Println` 理论上也可能出错（比如磁盘满了写不了标准输出）。

唯一可以"稍微"接受的忽略场景是在 `defer` 中：

```go
defer file.Close()
```

`Close()` 返回error，但defer里确实很难处理。更好的做法是明确处理：

```go
defer func() {
    if err := file.Close(); err != nil {
        log.Printf("关闭文件失败: %v", err)
    }
}()
```

---

> ❌ **常见错误 1：忽略了 error 返回值**
> 
> ```go
> result := doSomething()   // ❌ doSomething 返回 (Result, error)，但没接 error
> // 无法知道操作是否成功，result 可能是零值/无效数据
> ```
> 
> Go 中几乎所有 I/O、网络、解析操作都返回 error。不接收 error，你就是在假设"天下太平、万事顺利"——而线上环境永远不会这样。
> 
> ✅ **正确做法**：
> ```go
> result, err := doSomething()   // ✅ 同时接收 result 和 err
> if err != nil {
>     return err
> }
> ```

> ❌ **常见错误 2：用 `_` 忽略 error**
> 
> ```go
> result, _ := doSomething()   // ❌ 把错误丢掉了
> fmt.Println(result)
> ```
> 
> `_` 是 Go 的"黑洞"——被它吞掉的值再也找不回来。如果 `doSomething` 失败了，`result` 可能是零值或半初始化状态，后续代码会在错误的基础上继续执行，产生更难排查的 Bug。
> 
> ✅ **正确做法**：
> ```go
> result, err := doSomething()
> if err != nil {
>     log.Printf("doSomething 失败: %v", err)
>     return   // 或做降级处理
> }
> fmt.Println(result)   // ✅ 确认成功后才使用 result
> ```

> ❌ **常见错误 3：`err != nil` 检查后没有 return，继续使用可能无效的返回值**
> 
> ```go
> data, err := os.ReadFile("config.json")
> if err != nil {
>     log.Println("读取失败:", err)
>     // ❌ 没有 return！继续往下执行
> }
> // 下面的代码使用了 data，但 data 可能是 nil 或空切片
> config := parseConfig(data)   // ❌ data 可能无效，导致后续逻辑全乱
> ```
> 
> 检查到 error 后如果忘了 `return`（或 `continue`/`break`），代码会继续使用一个可能无效的返回值。这比不检查 error 更隐蔽——你看起来"处理了"错误（打了日志），但实际上程序在错误的状态下继续运行。
> 
> ✅ **正确做法**：
> ```go
> data, err := os.ReadFile("config.json")
> if err != nil {
>     log.Println("读取失败:", err)
>     return   // ✅ 或者 return err / return defaultValue
> }
> config := parseConfig(data)   // ✅ 走到这里 data 一定有效
> ```

## 32.4 错误包装（Go 1.13+）

Go 1.13引入了错误包装机制，让错误可以像"俄罗斯套娃"一样一层层包裹。

### %w：包装错误

```go
func readConfig() error {
    data, err := os.ReadFile("config.json")  // os.ReadFile 文件操作详见第38章
    if err != nil {
        return fmt.Errorf("读取配置文件失败: %w", err)
    }
    return nil
}
```

`%w`（w = wrap）把原始错误 `err` 包裹在新的错误里。

调用方看到的错误信息是：
```
读取配置文件失败: open config.json: no such file or directory
```

它同时包含了"读取配置文件失败"这个上下文，和原始错误的具体原因。

### 比喻：快递包裹

原始错误是商品，`fmt.Errorf(..., %w, err)` 是在外面包了一层快递盒，快递盒上写着"读取配置文件失败"。你可以一层层拆开快递盒，找到最里面的商品（原始错误）。

### errors.Unwrap：解开一层

```go
func errors.Unwrap(err error) error
```

### errors.Is：判断错误链中是否包含某个错误

不用包装时，判断错误类型：

```go
if err == ErrNotFound {
    ...
}
```

用了包装后，`==` 就不灵了——外层错误不等于内层错误。用 `errors.Is`：

```go
if errors.Is(err, ErrNotFound) {
    fmt.Println("记录不存在")
}
```

`errors.Is` 会沿着错误链（一层层 `Unwrap`）查找，看是否包含目标错误。

```go
var ErrNotFound = errors.New("记录不存在")

func getUser() error {
    return fmt.Errorf("getUser: %w", ErrNotFound)
}

err := getUser()
fmt.Println(errors.Is(err, ErrNotFound))
```

输出：`true`

### errors.As：判断错误链中是否有某种类型的错误

```go
type ValidationError struct {
    Field string
    Msg   string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("字段 %s: %s", e.Field, e.Msg)
}

func validate() error {
    return &ValidationError{Field: "email", Msg: "格式不正确"}
}

func process() error {
    err := validate()
    if err != nil {
        return fmt.Errorf("验证失败: %w", err)
    }
    return nil
}

func main() {
    err := process()
    var valErr *ValidationError
    if errors.As(err, &valErr) {
        fmt.Println("验证错误，字段:", valErr.Field)
    }
}
```

`errors.As` 沿着错误链查找，看是否有某个类型的错误。

## 32.5 自定义错误类型

有时候你需要错误携带更多信息（比如错误码、哪个字段出错了）：

```go
type AppError struct {
    Code    int
    Message string
    Cause   error
}

func (e *AppError) Error() string {
    if e.Cause != nil {
        return fmt.Sprintf("[%d] %s: %v", e.Code, e.Message, e.Cause)
    }
    return fmt.Sprintf("[%d] %s", e.Code, e.Message)
}

func (e *AppError) Unwrap() error {
    return e.Cause
}
```

实现了 `Unwrap()` 方法后，`errors.Is` 和 `errors.As` 就能穿透这个自定义错误。

在Web开发中，你通常会有这样一个错误类型，包含HTTP状态码：

```go
type APIError struct {
    Status  int    `json:"-"`
    Code    string `json:"code"`
    Message string `json:"message"`
}

func (e *APIError) Error() string {
    return e.Message
}

func NotFound(msg string) *APIError {
    return &APIError{Status: 404, Code: "NOT_FOUND", Message: msg}
}

func BadRequest(msg string) *APIError {
    return &APIError{Status: 400, Code: "BAD_REQUEST", Message: msg}
}
```

## 32.6 错误处理最佳实践

### 1. 错误只处理一次

```go
func process() error {
    err := step1()
    if err != nil {
        log.Printf("step1失败: %v", err)
        return err
    }
    return nil
}
```

这里错误被处理了**两次**——一次打日志，一次返回。上层调用者拿到错误后可能会再打一次日志。最终一个错误在日志里出现三四次。

**原则：要么处理它（解决掉），要么传播它（返回给上层）。不要两样都做。**

正确做法：

```go
func process() error {
    err := step1()
    if err != nil {
        return fmt.Errorf("process: %w", err)
    }
    return nil
}
```

只传播，不打日志。让最上层（main函数或请求处理器）统一记录日志。

### 2. 传播错误时添加上下文

不要直接 `return err`，加上"在哪里出错的"：

```go
return fmt.Errorf("加载用户 %d: %w", userID, err)
```

而不是：
```go
return err
```

前者让你看到错误链时能快速定位问题在哪一层。

### 3. Sentinel Error

把常见的错误情况定义为包级别的变量：

```go
var ErrNotFound = errors.New("not found")
var ErrTimeout = errors.New("timeout")
var ErrInvalidArgument = errors.New("invalid argument")
```

调用方可以用 `errors.Is(err, ErrNotFound)` 精确判断。

### 4. 不要检查error的字符串

```go
if err.Error() == "record not found" {
    ...
}
```

**不要这么做！** 错误信息可能随时变化，而且不同的库返回的错误信息格式不同。用 `errors.Is` 或 `errors.As` 来判断错误类型。

### 5. 用defer处理可能失败但不应中断的操作

```go
func copyFile(src, dst string) (err error) {
    r, err := os.Open(src)
    if err != nil {
        return err
    }
    defer r.Close()

    w, err := os.Create(dst)
    if err != nil {
        return err
    }
    defer func() {
        if closeErr := w.Close(); closeErr != nil && err == nil {
            err = closeErr
        }
    }()

    _, err = io.Copy(w, r)  // io.Copy 文件操作详见第38章
    return err
}
```

`Close()` 的错误也应该被处理——如果写入过程中一切都好，但关闭时磁盘满了导致数据没写入，应该把这个错误报告出来。

## 🤔 想多一点：错误处理真的这么烦吗？

刚开始写Go，你会觉得 `if err != nil` 到处都是，很烦。但写了两周之后，你会发现……它其实是在"省时间"。

因为 `try-catch` 虽然代码少，但排查线上Bug的时候是大问题：抛出异常的那个点，和捕获异常的那个点，中间隔了十几层调用栈。你要在日志里翻来翻去，猜到底是哪个环节出了问题。

Go的错误处理让你"当场"就得面对错误——这一行执行完之后，要么成功继续，要么失败处理。没有惊喜，没有隐藏的控制流。

这不叫丑，这叫"直面现实"。
