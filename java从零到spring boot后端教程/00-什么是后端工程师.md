# 第00章 · 什么是后端工程师

> "你在浏览器里点了'登录'按钮，1秒后页面跳转到了你的个人主页——这1秒里，发生了上百次你看不见的对话。而后端工程师，就是编排这场'看不见的对话'的人。"

## 00.1 用餐厅讲清楚前后端

想象你走进一家高级餐厅：

- **你（顾客）** = 浏览器/手机App（前端）
- **服务员** = HTTP协议（传递请求和响应）
- **厨房** = 后端服务器（处理业务逻辑）
- **食材仓库** = 数据库（持久化存储）
- **菜单** = API文档（告诉前端能点什么菜）

你坐下后，拿起菜单（打开App），对服务员说"我要一份牛排"（点击按钮发送HTTP请求）。服务员把订单送到厨房（请求到达后端），厨房从仓库取食材（查询数据库），厨师做好牛排（业务逻辑处理），服务员端回来放在你面前（HTTP响应返回给前端）。

| 餐厅角色 | 对应技术 | 做什么 |
|---------|---------|--------|
| 顾客 | 前端（浏览器/App） | 展示界面，收集用户操作 |
| 服务员 | HTTP协议 | 在前端和后端之间传递数据 |
| 厨房 | 后端（Spring Boot服务） | 处理业务逻辑，校验数据 |
| 食材仓库 | 数据库（MySQL） | 持久化存储数据 |
| 菜单 | API文档 | 定义前后端通信契约 |

> 🤔 想多一点：服务员不会让你直接进厨房翻冰箱，同样，前端也不应该直接操作数据库。后端存在的核心意义是**校验权限、保护数据、封装复杂逻辑**——不然谁都能直接改数据库里的余额了。

## 00.2 一个HTTP请求的完整旅程

现在用技术语言把这个过程走一遍。假设你在浏览器输入 `https://example.com/api/users`，按下回车：

```
你（浏览器）
  │
  │  ① 浏览器输入URL，按下回车
  │
  ▼
DNS解析 ─── ② 把 example.com 翻译成 IP 地址（如 12.34.56.78）
  │               > DNS就像电话簿：你说"找张三"，它告诉你"打138xxxx"
  │
  ▼
TCP连接 ─── ③ 浏览器与服务器建立TCP连接（三次握手）
  │               > 就像打电话：先确认对方在不在，再开始说话
  │
  ▼
TLS握手 ─── ④ 如果是HTTPS，先加密握手交换密钥
  │               > 相当于你和服务员约定了一套只有你们俩懂的暗号
  │
  ▼
HTTP请求 ── ⑤ 浏览器发送：GET /api/users HTTP/1.1
  │               Host: example.com
  │               Accept: application/json
  │
  ▼
Nginx ───── ⑥ 反向代理接收请求，转发给后面的Spring Boot
  │               > Nginx像餐厅门口的分流员，把客人引到正确的餐位
  │
  ▼
Spring Boot ── ⑦ Controller 接收请求：
  │                @GetMapping("/api/users")
  │                public List<User> getUsers() {
  │                    return userService.findAll();
  │                }
  │          ⑧ Service 层处理业务逻辑（权限校验、数据过滤）
  │          ⑨ Repository 层查询数据库：SELECT * FROM users
  │
  ▼
MySQL ────── ⑩ 数据库返回查询结果（若干行数据）
  │
  ▼
Spring Boot ── ⑪ 将Java对象序列化为JSON：
  │                [{"id":1, "name":"张三"}, {"id":2, "name":"李四"}]
  │
  ▼
HTTP响应 ── ⑫ 返回：HTTP/1.1 200 OK
  │               Content-Type: application/json
  │               body: [{"id":1, "name":"张三"}, ...]
  │
  ▼
浏览器 ──── ⑬ 接收JSON，JavaScript渲染成你看到的用户列表
```

> 📊 可视化演示见 [java_00_http_journey_visual.html](java_00_http_journey_visual.html)

这13个步骤，在正常网络条件下，发生在 **几十到几百毫秒** 之间。而后端工程师的工作，就是确保第⑥到第⑫步稳定、安全、高效地运行。

## 00.3 后端工程师的日常真实工作

别被"工程师"三个字吓到。后端工程师的日常大概是这样：

**写代码（40%）**
- 写接口：`@PostMapping("/api/orders")` 处理下单请求
- 写业务逻辑：计算优惠、校验库存、生成订单号
- 写数据库操作：设计表结构、写SQL或使用JPA

**改Bug（25%）**
- "为什么用户下单后库存没减？" → 发现没加事务注解 `@Transactional`
- "为什么查询很慢？" → 发现没建索引，加上后从3秒变0.01秒

**联调（15%）**
- 前端说："你接口返回的字段名是 `user_name`，我这边用的是 `userName`，对不上啊！"
- 于是你需要跟前端约定好JSON字段命名规范

**看日志/监控（10%）**
- "凌晨3点CPU飙到95%是什么情况？" → 排查发现是定时任务没做分页，一次查了100万条数据

**开会/写文档（10%）**
- 评审新需求的技术方案
- 写接口文档给前端同事看

**新手区（第00-43章）**：即使你从来没写过Java，也能从头跟上。我们假定你只会开关电脑，其他全部从零教起。

**进阶区（第44-93章）**：开始真正动手写后端服务。你会经历第一次"接口调通"的狂喜，也会经历"为什么报500"的折磨——这都是正常的过程。

**高手区（第94-124章）**：从"能写代码"到"能写生产级代码"。你会理解为什么有些系统能扛住双十一的流量而有些不能。

> 💡 **一个建议**：不要跳章。地基篇的每一章都跟后面的Java内容有直接关联。比如你现在觉得"学IP地址干嘛"，等学到第114章部署服务器时你会感谢现在打好基础的自己。
