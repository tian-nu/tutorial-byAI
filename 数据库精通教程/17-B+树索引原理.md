## 一、磁盘 IO 为什么是瓶颈？

### 一个比喻：去图书馆找书

想象你去一个巨大的图书馆，要找一本特定的书：

- **书架** = 磁盘（硬盘/SSD）：数据存储的地方，容量大但速度慢
- **你手上的书** = 内存（RAM）：正在处理的数据，速度快但容量小
- **图书管理员** = MySQL：帮你找书的人

你要找《MySQL 从入门到精通》：
- **没有目录（索引）**：管理员从第一个书架第一本开始，一本一本翻 → 找 100 万本书可能要一整天
- **有目录（索引）**：管理员查目录卡 → "T 类，3 排，5 层" → 直接走过去 → 30 秒搞定

数据库也一样：**磁盘 IO 是最大的瓶颈**。

### 数据对比（刻在脑子里！）

| 操作 | 耗时 | 比喻 |
|------|------|------|
| CPU 执行一条指令 | ~0.3ns | 你眨一下眼 |
| 内存访问（RAM） | ~100ns | 你伸手拿桌上的杯子 |
| **磁盘寻道（HDD）** | **~10ms** | **你开车去隔壁城市** |
| SSD 随机读 | ~0.1ms | 你走到隔壁房间 |

**关键结论**：一次磁盘寻道 ≈ 10 万次内存访问。如果你的查询需要 100 次磁盘 IO，而别人只需要 3 次，差距就是 30 倍以上。

### 没有索引时：全表扫描

全表扫描（Full Table Scan）*此术语见附录E* 就是"一本一本翻"：

```sql
-- 假设 users 表有 1000 万行，name 列上没有索引
SELECT * FROM users WHERE name = '张三';
```

MySQL 会这样做：
1. 从磁盘读第一行 → 检查 name 是不是 '张三'
2. 从磁盘读第二行 → 检查
3. ...重复 1000 万次...

即使每行只有 200 字节，总共也要读 2GB 数据。按磁盘 100MB/s 的速度，**约 20 秒**。如果数据不在缓存中，每次都要物理读盘，那就是灾难。

所以问题来了：**怎么让 MySQL 少读数据？**

## 二、为什么不用二叉树？

你可能会想："用二叉树啊！左边小右边大，查找 O(log n)，多快！"

### 二叉树的问题：树太高了

假设有 1 亿行数据，用二叉搜索树存储：

```
            root
           /    \
          /      \
         /        \
       ...        ...
      /              \
   叶子              叶子
```

- 树的高度 ≈ log₂(1 亿) ≈ **27 层**
- 每一层访问需要一次磁盘 IO（因为节点分散在磁盘不同位置）
- 查找一次 = **27 次磁盘 IO** ≈ 27 × 10ms = **270ms**

270ms 看起来还行？但这是**一个查询**的时间。如果每秒 1000 个查询，数据库直接崩溃。

### 核心问题：每个节点只存一个键

二叉树每个节点只存一个值，所以树的高度等于数据量的对数。如果我们能让**每个节点存更多的键**，树的高度就会大幅降低。

这就是 B 树（B-Tree）*此术语见附录E* 的核心思想：**让每个节点存多个键，降低树的高度。**

## 三、B+树结构详解

### 3.1 B+树长什么样？

B+树（B+ Tree）*此术语见附录E* 是 MySQL InnoDB 存储引擎使用的索引结构。它有三层：

```
                    [30 | 60]                    ← 根节点（内部节点）
                   /    |    \
                  /     |     \
            [10|20]  [40|50]  [62|68|72|78]     ← 内部节点
           /   |   \   /  |  \
          /    |    \  /   |   \
      [3|7] [12|15|18] [22|25|28] ...           ← 叶子节点
        ↕       ↕         ↕
      有序链表 ←→ 有序链表 ←→ 有序链表
```

**B+树的规则**：
1. **内部节点**（非叶子）：只存键值（key），不存数据。键值用于"指路"。
2. **叶子节点**：存键值 + 数据（或数据指针）。所有叶子在同一层。
3. **叶子节点之间用指针连接**，形成有序链表。

### 3.2 查找过程

查找 key=45 的过程（以图示为例）：

```
第 1 步：根节点 [30, 60]
  30 ≤ 45 ≤ 60 → 走中间子节点

第 2 步：内部节点 [40, 50]
  40 ≤ 45 ≤ 50 → 走中间子节点

第 3 步：叶子节点 [42, 45, 48]
  找到 45 → 取出对应数据 ✓
```

**仅 3 次磁盘 IO！** 每次读取一个节点（一个页），3 次就够了。

### 3.3 B+树 vs B 树

B 树和 B+树经常被混为一谈，但它们有关键区别：

| 对比维度 | B 树 | B+树 |
|---------|------|------|
| 数据存储 | 所有节点都存数据 | 只有叶子节点存数据 |
| 内部节点 | 存键 + 数据 | 只存键（更小，能存更多） |
| 叶子链表 | 无 | 有（有序链表，支持范围查询） |
| 范围查询 | 需要中序遍历，IO 多 | 定位起点后顺序扫描，IO 少 |
| 同样高度 | 存的键更少 | 存的键更多（树更矮） |

**为什么 MySQL 选 B+树？**
- 内部节点只存键 → 一个节点能存更多键 → 树更矮 → IO 更少
- 叶子节点有链表 → 范围查询（`BETWEEN`、`ORDER BY`）极快
- 所有数据都在叶子 → 查询性能稳定（每次都是树的高度次 IO）

> **想多一点**：B 树的名字其实有争议。有人说 B 是 "Balanced"（平衡），有人说是 "Bayer"（发明者之一），还有人说是 "Boeing"（波音公司资助的研究）。我们不需要纠结名字，记住 B+树是 B 树的改进版即可。

## 四、页（Page）的概念

页（Page）*此术语见附录E* 是 InnoDB 存储引擎管理磁盘空间的最小单位，默认 **16KB**（16384 字节）。

### 一个页能存多少索引项？

假设主键是 BIGINT（8 字节），指向子节点的指针也是 8 字节（实际 6 字节，我们算宽松一点），那么一个内部节点中：

```
每个索引项 = 8 字节（键）+ 8 字节（指针）= 16 字节
一个页（16KB）= 16384 字节
一个页能存的索引项 ≈ 16384 / 16 ≈ 1024 个
```

也就是说，一个内部节点可以有约 1000 个子节点。

### 3 层 B+树能存多少数据？

- 根节点：1 个页，约 1000 个索引项 → 指向 1000 个内部节点
- 第二层：1000 个页，每个约 1000 个索引项 → 指向 1000 × 1000 = **100 万**个叶子节点
- 第三层（叶子）：100 万个页，每个页存约 10-20 行数据 → **1000 万 ~ 2000 万行**

**3 层 B+树能存千万级数据，查找只需要 3 次 IO！**

这就是为什么 `SELECT * FROM users WHERE id = 9999999` 可以瞬间返回——主键索引的 B+树只有 3 层，3 次磁盘 IO 就能定位到数据。

## 五、聚集索引 vs 二级索引

### 5.1 聚集索引（Clustered Index）

聚集索引（Clustered Index）*此术语见附录E* 是 InnoDB 的**主键索引**。它的特点：

- **叶子节点存的是完整行数据**（所有列）
- 每张表**有且只有一个**聚集索引
- 如果没有显式定义主键，InnoDB 会选第一个 UNIQUE 索引；如果都没有，创建隐藏的 `row_id`

```
聚集索引的叶子节点：
┌──────────────────────────────────────────────┐
│ key=3  │ id=3, name='Alice', age=25, city='北京'  │  ← 完整行数据
│ key=7  │ id=7, name='Bob',   age=30, city='上海'  │
│ key=12 │ id=12,name='Charlie',age=22, city='广州' │
└──────────────────────────────────────────────┘
```

### 5.2 二级索引（Secondary Index）

二级索引（Secondary Index）*此术语见附录E* 是你手动创建的非主键索引。它的特点：

- **叶子节点存的是主键值**（不是完整行数据！）
- 一张表可以有多个二级索引
- 查找流程：二级索引 → 找到主键值 → 回表到聚集索引 → 取完整行

```
二级索引（在 age 列上）的叶子节点：
┌─────────────────────────────────┐
│ key=21 │ 主键值=62              │
│ key=22 │ 主键值=12              │
│ key=23 │ 主键值=52              │
│ ...    │ ...                    │
└─────────────────────────────────┘

然后拿着主键值去聚集索引中找完整行 → 这就是"回表"！
```

### 5.3 回表（Table Lookup）

回表（Table Lookup）*此术语见附录E* 是指：通过二级索引找到主键值后，再回到聚集索引中查找完整行数据的过程。

```sql
-- 假设在 age 列上建了二级索引
SELECT * FROM users WHERE age = 25;
```

执行过程：
1. 在 age 索引的 B+树中查找 age=25 → 叶子节点存的是主键 id 值
2. 拿着 id 值去聚集索引的 B+树中查找 → 取出完整行 `(id, name, age, city, ...)`

**回表的问题**：多一次 B+树查找 = 多 3~4 次 IO。如果查询返回 1000 行，就要回表 1000 次！

**怎么避免回表？** → 覆盖索引（第 19 章详解）。如果查询只需要 `id` 和 `age`，二级索引中已经有了，就不需要回表。

## 六、为什么主键建议用自增 ID 而非 UUID？

这是一个经典面试题，现在你应该能自己回答了。

### 自增 ID（AUTO_INCREMENT）

新数据的主键值越来越大，总是追加到 B+树的最右侧叶子节点：

```
插入 id=101：
  → 定位到最右侧叶子 [..., 99, 100]
  → 直接追加： [..., 99, 100, 101]
  → 如果叶子满了，分裂，但只影响最右侧
```

**优点**：插入效率高，页分裂少，磁盘利用率高。

### UUID

UUID 是随机字符串，如 `'a3f2b1c9-...'`。新数据的主键值可能落在树的任何位置：

```
插入 id='a3f2b1c9-...'（UUID）：
  → 可能插在树中间某个叶子节点
  → 如果叶子满了，需要页分裂
  → 分裂后，数据被"挤"到不同位置
  → 原本连续的页变得不连续 → 磁盘碎片化
```

**问题**：
1. **页分裂频繁**：随机插入导致大量页分裂，性能下降
2. **磁盘碎片化**：数据分散在磁盘各处，范围查询变慢
3. **索引体积大**：UUID 是 CHAR(36) 或 BINARY(16)，比 BIGINT(8) 大得多

### 页分裂详解（选读）

页分裂（Page Split）*此术语见附录E* 是指：当一个页（16KB）满了，需要插入新数据时，InnoDB 会分配一个新页，把原页的一部分数据移到新页，再插入新数据。

```
插入前：[1, 3, 5, 7, 9]  ← 页已满（假设最多存 5 个）
插入 6：
  分裂 → 页A: [1, 3, 5]  页B: [6, 7, 9]
  同时更新父节点的指针指向两个页
```

对于自增主键，分裂只发生在最右侧，插入新页即可。对于随机 UUID，分裂随机发生，导致 B+树频繁调整。

**结论**：主键用自增 BIGINT，除非你有特殊理由（如分布式系统需要全局唯一 ID）。

## 七、查看索引

```sql
-- 查看某张表的所有索引
SHOW INDEX FROM users;

-- 更多字段
SHOW INDEX FROM users\G
```

输出解读：

| 字段 | 含义 |
|------|------|
| Key_name | 索引名称（PRIMARY 表示主键索引） |
| Column_name | 索引列 |
| Index_type | 索引类型（BTREE 表示 B+树） |
| Cardinality | 索引中不同值的数量（估算值，用于优化器选择索引） |

## 我做得对不对？

### 验证方法

用你自己的话回答以下问题（不需要 SQL）：

1. **为什么全表扫描慢？** → 因为要逐行从磁盘读数据，数据量大时 IO 次数多。
2. **为什么二叉树不适合做数据库索引？** → 树太高，1 亿行数据约 27 层，每次查询 27 次 IO。
3. **B+树怎么解决这个问题？** → 每个节点存多个键，3 层就能存千万级数据，IO 次数固定为 3 次。
4. **为什么主键建议用自增 ID 而非 UUID？** → UUID 随机插入导致频繁页分裂和磁盘碎片化，自增 ID 追加到最右侧，效率高。
5. **什么是回表？** → 二级索引叶子只存主键值，需要再查聚集索引取完整行数据。

## 不对怎么办？

### 常见错误 1：认为索引就是"排序好的数据"

索引不是把数据排序后存一份。索引是**独立的数据结构**（B+树），它指向原始数据，而不是复制一份数据。

- ❌ 错误理解：索引 = 排序后的数据副本
- ✅ 正确理解：索引 = 独立 B+树结构，叶子存数据（聚集索引）或主键（二级索引）

### 常见错误 2：不知道为什么 UUID 主键性能差

如果你只记住了"UUID 性能差"但不知道为什么，面试官一问就露馅。

**核心原因就一条**：UUID 随机插入导致 B+树频繁页分裂，而自增 ID 插入总是在最右侧，几乎不分裂。

### 常见错误 3：混淆聚集索引和二级索引

- 聚集索引：**每表只有一个**，叶子存完整行。主键就是聚集索引。
- 二级索引：**可以有多个**，叶子存主键值。需要回表。

试试问自己：`SELECT * FROM users WHERE name = '张三'` 在 name 列有二级索引的情况下，一共发生了多少次 B+树查找？

答案：**两次**——先在 name 索引中找到主键值，再回表查聚集索引。
