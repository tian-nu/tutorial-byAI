# 附录I · 技术术语速查

> "指针是什么？编译到底干了什么？汇编又是什么？全书中提到的各种底层术语、编程概念、架构名词——如果遇到看不懂的词语，回这里查。每个术语用最直白的大白话解释，附带书中首次出现的位置。"

## I.1 计算机底层

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **CPU** | 计算机的"大脑"，只会做算术和逻辑判断，但做得极快（每秒几十亿次） | 第1章 |
| **寄存器** | CPU内部的超快存储单元，比内存快几百倍，但数量极少（几十个），相当于你办公桌上的便签纸 | 第1章 |
| **二进制** | 只有0和1的数字系统。计算机底层一切——文字、图片、视频——本质上都是一串0和1 | 第1章 |
| **汇编语言** | 比机器码（纯0101）可读一点的语言，每条指令直接对应CPU的一个操作，是"最低级"的编程语言 | 第1章 |
| **机器码** | CPU能直接"吃"的指令，纯二进制。你写的任何高级语言代码，最终都要变成机器码才能运行 | 第1章 |
| **编译** | 把人类能看懂的代码翻译成机器能看懂的机器码，这个过程叫编译。就像把中文菜谱翻译成机器指令 | 第1章 |
| **编译器** | 执行"编译"这个翻译工作的工具。Go的编译器叫`gc`，C的编译器叫`gcc` | 第1章 |
| **内存（RAM）** | 计算机的"临时工作台"，运行中的程序和数据都放在这里。断电就清空，速度快 | 第2章 |
| **内存地址** | RAM里每个存储位置的编号，从0开始。就像小区里每套房子的门牌号 | 第2章 |
| **指针** | 存"门牌号"的变量。这个变量里装的不是一个值，而是这个值在内存中的地址（门牌号） | 第2章 |
| **堆（Heap）** | 内存中一块"公共区域"，谁想用就申请一块，用完了还回去。Go里用`new()`或`make()`分配的东西一般在堆上 | 第2章 |
| **栈（Stack）** | 内存中一块"临时便签区"，函数调用时自动分配，函数返回自动释放。每个Goroutine有自己的栈 | 第2章 |
| **硬盘（HDD/SSD）** | 计算机的"仓库"，存文件、照片、数据库。断电数据还在，但比内存慢几百到几千倍 | 第3章 |
| **操作系统（OS）** | 硬件和软件之间的"翻译官+管家"。管理CPU、内存、硬盘，提供程序运行的环境。你用的Windows、macOS、Linux都是操作系统 | 第4章 |
| **进程** | 一个正在运行的程序实例。每个进程有自己独立的内存空间，互不干扰 | 第4章 |
| **线程** | 进程内部的一条"执行流"。一个进程可以有多个线程，共享同一块内存。相当于一个厨房里的多个厨师 | 第4章 |
| **上下文切换** | CPU从一个任务切换到另一个任务时，要保存当前状态、加载新状态，这个开销叫上下文切换。换太多会变慢 | 第4章 |
| **内核** | 操作系统的核心。负责跟硬件直接打交道的最底层部分。你写的代码不能直接访问硬件，必须通过内核 | 第4章 |
| **系统调用** | 你的程序请求内核办事的方式。比如"我要读这个文件"——程序发起系统调用，内核去操作硬盘 | 第99章 |

## I.2 编程语言概念

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **C语言** | 1972年诞生，几乎所有现代操作系统和数据库都用C写的。它是Go语言的老祖宗，接近底层但还算人类可读 | 第1章 |
| **高级语言** | 离人类语言近、离机器语言远的编程语言。Go、Python、Java都是高级语言 | 第1章 |
| **低级语言** | 离机器语言近的编程语言。"低级"不是贬义——不是说它差，是它更接近硬件。汇编和C都属于低级语言 | 第1章 |
| **编译型语言** | 写完代码后需要先"编译"成可执行文件才能运行的语言。Go、C、Rust是编译型 | 第15章 |
| **解释型语言** | 不需要提前编译，运行时一行一行翻译执行。Python、JavaScript是解释型 | 第15章 |
| **静态类型** | 变量在代码写好时就确定了类型，编译时检查。比如`var x int`——x这辈子只能是int。Go是静态类型 | 第17章 |
| **动态类型** | 变量类型在运行时才确定，同一个变量可以先存数字再存字符串。Python和JavaScript是动态类型 | 第17章 |
| **强类型** | 不同类型不能混用，必须显式转换。`"hello" + 3`会报错。Go和Python都是强类型 | 第17章 |
| **弱类型** | 不同类型可以自动混用。`"hello" + 3`可能变成`"hello3"`。JavaScript是弱类型 | 第17章 |
| **垃圾回收（GC）** | 自动清理不再使用的内存。你只需要分配内存，不用手动释放——像有清洁工定期扫垃圾 | 第15章 |
| **闭包** | 一个函数"记住"了它出生时的外部变量。即使外部函数已经返回，闭包还能访问那些变量 | 第27章 |
| **反射** | 程序在运行时检查自己的结构——"我这个变量到底是什么类型？有几个字段？"。强大但慢 | 第31章 |
| **序列化** | 把程序里的数据结构（结构体、Map等）转成能在网络上传输或存文件的格式（JSON、XML等） | 第39章 |
| **反序列化** | 序列化的逆过程——把JSON字符串还原成程序里的结构体 | 第39章 |

## I.3 网络与协议

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **协议** | 通信双方约定好的"规矩"。就像写信要写邮政编码、收件人——HTTP协议规定了请求必须包含方法（GET/POST）、路径、头部等 | 第6章 |
| **IP地址** | 互联网上每台设备的"门牌号"，格式如`192.168.1.1`。IPv4大约43亿个地址，已经不够用了 | 第6章 |
| **DNS** | 互联网的电话号码簿——把你输入的`www.baidu.com`翻译成IP地址`110.242.68.66` | 第7章 |
| **TCP** | 可靠的"快递员"协议——保证数据不丢、不乱、不重复，按顺序送达。三次握手建立连接，四次挥手断开 | 第9章 |
| **UDP** | 不可靠的"明信片"协议——发出去不管，丢了就丢了。快但没有保证。适合视频通话、游戏等 | 第10章 |
| **HTTP** | 浏览器和服务器之间"对话"的语言。你每次打开网页，都在发HTTP请求 | 第11章 |
| **HTTPS** | HTTP的"加密版"。数据在传输过程中被加密，中间人看不到内容。网址前的🔒小锁就是HTTPS | 第13章 |
| **TLS/SSL** | HTTPS用到的加密协议。在HTTP和TCP之间加了一层加密，就像给普通信封加上了密码锁 | 第13章 |
| **WebSocket** | HTTP的"升级版"——建立一次连接后，双方可以随时互发消息，不需要客户端反复问"有消息吗" | 第14章 |
| **API** | 程序之间"对话"的接口。后端写API，前端调用API——就像餐厅的菜单：告诉你有什么菜、怎么点 | 第0章 |
| **RESTful** | 一种设计API的风格——用URL表示"资源"，用HTTP方法（GET/POST/PUT/DELETE）表示"操作" | 第65章 |
| **JSON** | 一种轻量级数据格式，人和机器都能轻松读。后端API最常用的数据交换格式 | 第39章 |

## I.4 数据库与缓存

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **数据库** | 有组织地存储数据的地方。不是简单的文件——能高效查询、事务保证安全、多人同时访问不冲突 | 第50章 |
| **SQL** | 和数据库"对话"的语言。`SELECT * FROM users WHERE age > 18`——英文一样的语法 | 第51章 |
| **索引** | 数据库里的"目录"。给常用查询的列建索引，就像给书加目录——不用从头翻到尾 | 第55章 |
| **事务** | 一组操作打包执行——要么全成功，要么全失败。就像转账：A扣钱和B加钱必须同时成功 | 第56章 |
| **ACID** | 事务的四个特性：原子性（全或无）、一致性（规则不破）、隔离性（互不干扰）、持久性（写入不丢） | 第56章 |
| **ORM** | 把数据库表映射成程序里的结构体。你不用写SQL，直接操作结构体就行。GORM是Go最流行的ORM | 第59章 |
| **连接池** | 预先建好一批数据库连接放着。请求来了直接拿一个用，用完还回去——避免反复"拨号"的开销 | 第59章 |
| **缓存** | 把经常要用的数据放在"伸手就能够到"的位置。Redis就是最流行的缓存工具 | 第60章 |
| **Redis** | 数据存在内存里的"超快存储"。不像MySQL存硬盘那么慢，但内存容量有限，一般当缓存用 | 第60章 |
| **缓存穿透** | 查一个数据库里根本不存在的key，每次都穿透缓存打到数据库。恶意攻击常用这招 | 第62章 |
| **缓存击穿** | 一个热点key在过期的瞬间，大量请求同时打到数据库。就像热门商品秒杀瞬间的流量洪峰 | 第62章 |
| **缓存雪崩** | 大量缓存在同一时间过期，数据库瞬间承受所有请求——像雪崩一样压垮系统 | 第62章 |

## I.5 架构与运维

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **中间件** | 在请求处理链中的"插件"。日志记录、权限检查、跨域处理——每个中间件做一件事 | 第69章 |
| **微服务** | 把一个大应用拆成多个独立小服务。每个服务有自己的数据库，通过API通信。就像超市拆成专卖店 | 第93章 |
| **单点登录（SSO）** | 登录一次，所有系统都认。就像公司门禁卡——刷刷进大门，所有办公室都不用再刷 | 第81章 |
| **消息队列（MQ）** | 异步通信的"中间人"。生产者扔消息进去，消费者慢慢取出来处理——各干各的，互不打扰 | 第85章 |
| **负载均衡** | 把请求分给多台服务器，压力分散。Nginx是最常用的反向代理+负载均衡工具 | 第96章 |
| **反向代理** | 站在服务器前面"接客"。客户端以为在跟服务器说话，其实是在跟代理说话——代理再把请求转发给后端的服务器 | 第96章 |
| **容器（Docker）** | 把应用和环境打包在一起，到哪都能跑。就像一个标准集装箱——不管装什么货，运输体系都一样 | 第102章 |
| **CI/CD** | 持续集成+持续部署。代码提交后自动测试→自动打包→自动部署，一条流水线全自动 | 第103章 |
| **熔断** | 当一个服务频繁出错时，暂时"切断"对它的调用，直接返回兜底结果。就像电路过载跳闸保护 | 第97章 |
| **降级** | 系统压力大时，主动关掉非核心功能，保证核心功能正常。就像春运时火车站暂停售票厅的大屏幕以保障售票系统 | 第97章 |
| **限流** | 控制请求速率，"一秒最多1000个请求"。超过的直接拒绝——保护系统不被冲垮 | 第97章 |
| **服务发现** | 微服务架构中，服务实例的地址是动态变化的。服务发现自动跟踪"现在有哪些实例、IP和端口是什么" | 第93章 |

## I.6 测试概念

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **单元测试** | 测试单个函数的正确性。给定输入，检查输出是否和预期一致 | 第106章 |
| **Mock** | 测试用的"替身"。测试发邮件功能时，不用真的发邮件——用Mock假装发了 | 第107章 |
| **集成测试** | 多个模块一起测。确保数据库、API、中间件能配合工作，而不是各自孤立地正确 | 第108章 |
| **性能测试** | 测系统的速度和负载能力。2000人同时刷首页，响应时间还在200毫秒内吗？ | 第109章 |
| **基准测试（Benchmark）** | 测量一段代码的执行速度。Go的`testing.B`提供了精确的纳秒级计时 | 第109章 |
