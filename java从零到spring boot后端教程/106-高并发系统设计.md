# 106-高并发系统设计

> 💡 一个游乐场每天能接待 5000 人。突然有一天，抖音把它炒火了，来了 50000 人——大门挤破了、排队 3 小时、厕所排到马路上。高并发系统设计就是给游乐场做"扩容改造"——多开几个入口、安排调度车辆、设置预约入场。本章梳理从"一天 1000 人"到"一天 1 亿人"的系统进化路线。

## 106.1 高并发设计的"四层宝塔"

```
┌─────────────────────────────────┐
│  第 4 层：CDN + 静态化            │  ← 离用户最近
├─────────────────────────────────┤
│  第 3 层：负载均衡 + 反向代理       │
├─────────────────────────────────┤
│  第 2 层：缓存（Redis + 本地缓存）   │
├─────────────────────────────────┤
│  第 1 层：数据库（读写分离 + 分库分表）│  ← 离用户最远
└─────────────────────────────────┘

优化方向：让请求在越靠外的层就被解决，尽量不打到数据库。
```

## 106.2 QPS 估算

### 核心公式

```
QPS = (PV × 0.8) ÷ (86400 × 0.2)

其中：
- PV：日页面浏览量
- 0.8：假设 80% 的请求集中在一天中的高峰期
- 86400：一天的秒数
- 0.2：假设高峰期持续 20% 的时间（约 5 小时）
```

### 举例

```
日 PV = 1000 万
日 QPS = 1000 万 ÷ 86400 ≈ 115

高峰期 QPS = (1000 万 × 0.8) ÷ (86400 × 0.2) ≈ 463

峰值 QPS = 高峰期 QPS × 3 = 1389  ← 按 3 倍冗余设计
```

### 各级处理能力参考

| 组件 | 单机 QPS |
|------|----------|
| Nginx | 5 万 - 10 万 |
| Redis | 5 万 - 10 万 |
| Spring Boot (简单接口) | 1000 - 5000 |
| MySQL (单表) | 1000 - 2000 |

## 106.3 缓存层——离数据库越远越好

### 多级缓存架构

```
请求 → CDN（静态资源）
   │ 命中？返回
   │ 未命中 ↓
请求 → Nginx 本地缓存
   │ 命中？返回
   │ 未命中 ↓
请求 → Redis 集群
   │ 命中？返回
   │ 未命中 ↓
请求 → 数据库
   │ 查出来，写入 Redis
   └ 返回
```

### 缓存三大问题

| 问题 | 现象 | 解法 |
|------|------|------|
| **缓存穿透** | 查一个不存在的 key，每次都穿透到 DB | 布隆过滤器 / 缓存空值 |
| **缓存击穿** | 热点 key 过期瞬间，大量请求打到 DB | 互斥锁 / 永不过期 + 异步刷新 |
| **缓存雪崩** | 大量 key 同时过期，DB 瞬间被打死 | 过期时间加随机值 / 集群高可用 |

```java
// 缓存穿透：缓存空值
public User getUser(Long id) {
    String key = "user:" + id;
    User user = redis.get(key);
    if (user != null) {
        return "NULL_USER".equals(user.getUsername()) ? null : user;
    }
    user = db.findById(id);
    if (user == null) {
        redis.set(key, new User("NULL_USER"), Duration.ofMinutes(5));
    } else {
        redis.set(key, user, Duration.ofMinutes(30));
    }
    return user;
}

// 缓存雪崩：过期时间加随机
redis.set(key, value, Duration.ofMinutes(30 + new Random().nextInt(5)));
```

## 106.4 数据库层——读写分离 + 分库分表

### 读写分离

```
写请求（INSERT/UPDATE/DELETE）
        │
        ▼
    ┌───────┐
    │ Master │ ─── 同步 ───→ ┌───────┐
    └───────┘               │ Slave1 │
                             ├───────┤
                             │ Slave2 │
                             └───────┘
                                  ▲
                                  │
读请求（SELECT）──────────────────┘
```

```yaml
spring:
  datasource:
    master:
      url: jdbc:mysql://master:3306/mydb
    slave:
      url: jdbc:mysql://slave1:3306/mydb,slave2:3306/mydb
```

### 分库分表

```
分库：按业务垂直拆分
  ├── user_db（用户表）
  ├── order_db（订单表）
  └── product_db（商品表）

分表：按数据水平拆分
  order_db:
    ├── order_0（userId % 4 = 0）
    ├── order_1（userId % 4 = 1）
    ├── order_2（userId % 4 = 2）
    └── order_3（userId % 4 = 3）
```

> 🤔 想多一点：水平分表的代价是 JOIN、GROUP BY、跨分片事务都变得极其困难。设计分表方案时，选分片键（Sharding Key）是核心决策——最好让 90% 的查询都落在单个分片上。

## 106.5 异步与削峰

```
同步（低并发）：请求 → 处理 → 返回

异步（高并发）：
  请求 → 写入 MQ → 立刻返回 "受理中"
                   │
                   └→ 消费者慢慢处理 → 处理完通知用户
```

```java
@PostMapping("/orders")
public Result<String> createOrder(@RequestBody OrderRequest request) {
    // 1. 基本校验
    validate(request);

    // 2. 写入消息队列，立刻返回（不等处理完）
    rabbitTemplate.convertAndSend("order.exchange", "order.create", request);
    return Result.success("订单已受理，请稍后查看结果");
}

@RabbitListener(queues = "order.queue")
public void processOrder(OrderRequest request) {
    // 实际处理：扣库存、生成订单、扣款...
}
```

## 106.6 限流算法

### 令牌桶（Token Bucket）——最常用

```
令牌桶容量 = 10
令牌生成速率 = 5 个/秒

请求到达 → 桶里有令牌？→ 有：拿走令牌，放行 → 无：拒绝（429）
```

### 漏桶（Leaky Bucket）

```
请求 → 进漏桶 → 以恒定速率流出
不管请求多快，处理速度恒定（强行削峰）
```

### 滑动窗口

```
1 分钟窗口，限制 100 次请求
窗口随时间滑动，超过就拒绝
```

## 106.7 性能优化的黄金法则

```
1. 先测量，再优化（没有监控数据 = 瞎优化）
2. 80% 的瓶颈在 20% 的代码（二八定律）
3. 缓存 > 异步 > 并行 > 换语言
4. 数据库是最难扩展的，尽量减少打到 DB 的请求
5. 不要过早优化——但要知道"什么时候该优化"
```

## 106.8 小结

| 知识点 | 核心内容 |
|--------|----------|
| 四层架构 | CDN → 负载均衡 → 缓存 → 数据库 |
| QPS 估算 | PV × 0.8 ÷ (86400 × 0.2) |
| 缓存三问题 | 穿透（空值缓存）、击穿（互斥锁）、雪崩（随机过期） |
| 数据库扩展 | 读写分离 → 垂直分库 → 水平分表 |
| 限流 | 令牌桶 / 漏桶 / 滑动窗口 |
| 异步 | MQ 削峰填谷 |

## 106.9 自测题

**1. 日 PV 500 万，估算高峰期 QPS 和峰值 QPS。单台 Spring Boot 能否承受？**

**2. 热点数据（如双 11 商品详情）的 Redis key 突然过期了，大量请求打到 MySQL。这属于缓存三问题中的哪一个？如何解决？**

**3. 秒杀系统为什么不能直接用"查库存 → 扣库存 → 创建订单"的同步流程？**

---

**答案提示**：1→高峰期 QPS ≈ (500 万 × 0.8) ÷ (86400 × 0.2) ≈ 231；峰值 QPS ≈ 693。单台 Spring Boot 正常情况下可以承受（1000-5000 QPS）。2→缓存击穿。解决方案：① 热点 key 设置永不过期，后台异步更新；② 用分布式锁，同一时刻只允许一个请求回源 DB 重建缓存。3→因为瞬时流量是平时的 1000 倍以上，同步流程下所有请求排队查数据库，数据库连接池瞬间耗尽。解决：前端 → CDN 静态化 → 验证码削峰 → MQ 排队 → Redis 预减库存 → 异步处理订单。下一章——系统设计面试经典题。
