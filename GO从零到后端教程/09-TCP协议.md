# 第9章 · TCP协议

> "你寄出一本书，快递公司保证：不丢件、不损坏、不乱序、不重复。这就是TCP——互联网的'靠谱快递员'。"

---

## 9.1 TCP是什么

TCP = **Transmission Control Protocol** = 传输控制协议

把它想象成一个**极其负责的快递员**。你告诉它："把这箱东西送到隔壁城市的小王手里。"它保证：

| 承诺 | 含义 |
|------|------|
| **不丢** | 每一个包裹都能送到，不会中途消失 |
| **不乱** | 先寄的先到，后寄的后到，顺序和发出时一模一样 |
| **不重复** | 同一个包裹不会莫名其妙送两次 |
| **不出错** | 包裹里的东西和发出的完全一样，没有破损 |

> 🤔 想多一点：那为什么不所有网络通信都用TCP？因为保证可靠性是有代价的——慢。TCP要做握手、要确认、要重传，这都是额外开销。所以有些场景（比如视频通话）宁可用不那么可靠但更快的UDP。我们下章会详细讲。

### TCP在协议栈中的位置

在网络分层模型中，TCP 位于第四层——**传输层**：

```
应用层：    HTTP, FTP, SMTP, DNS...
              ↓ "把数据交给传输层"
传输层：    TCP, UDP
              ↓ "把数据段交给网络层"
网络层：    IP
              ↓ "把数据包交给链路层"
链路层：    网卡、WiFi、光纤
```

TCP的工作是：从上层（应用层）接收任意长度的数据流，切成合适大小的**数据段**（Segment），确保它们按序、完整地到达目的地。上层不需要关心丢包、乱序——TCP全帮你处理好了。

## 9.2 三次握手（Three-Way Handshake）

这是TCP中最核心的概念，也是**面试必问**。

### 打电话的比喻

你给朋友打电话的过程：

```
你：   "喂？听得到吗？"           （第1步）
朋友： "听得到！你听得到我吗？"    （第2步）
你：   "听得到！那咱们开始聊吧。"  （第3步）
```

为什么是三次？你有没有想过，如果只做两次——

```
你：   "喂？听得到吗？"           （第1步）
朋友： "听得到！"                 （第2步）
```

这里有个问题：**你不知道朋友那边能不能听到你说的话**。朋友知道你听得到他吗？不知道。万一朋友的回话你根本听不到呢？你们就开始"喂？喂？喂？"地死循环了。

三次握手的本质是：**确认双方都有发送能力和接收能力。**

### 三次握手的具体过程

```
客户端                             服务端
  │                                  │
  │  ──── SYN (seq=x) ──────────→   │  第1次：客户端发SYN
  │                                  │  "我想连接你，我的序号是x"
  │                                  │
  │  ←── SYN+ACK (seq=y, ack=x+1) ─ │  第2次：服务端回应
  │                                  │  "好的，我的序号是y，我收到了你的x"
  │                                  │
  │  ──── ACK (seq=x+1, ack=y+1) ─→ │  第3次：客户端确认
  │                                  │  "收到你的y，咱们可以开始了"
  │                                  │
  │═══════ TCP连接已建立 ═══════│
```

三个关键标志位：

| 标志位 | 含义 | 通俗理解 |
|--------|------|----------|
| **SYN** | Synchronize | "我想建立连接" |
| **ACK** | Acknowledge | "我收到了" |
| **FIN** | Finish | "我没话说了，准备结束" |

### 为什么必须是三次？

面试中最高频的问题之一。核心答案就一句话：**TCP是全双工的，双方都要确认自己有发送和接收的能力。**

我们仔细分析：

```
只做一次握手（客户端发SYN）：
  → 服务端不知道客户端能否收到自己的数据 ❌

只做两次握手（客户端发SYN，服务端回SYN+ACK）：
  → 服务端不知道客户端是否收到了自己的SYN+ACK ❌
  → 历史连接问题：如果客户端第一个SYN在网络中滞留了
    很久才到达服务端，服务端会以为是一个新连接请求，
    回SYN+ACK后就傻傻地等着，浪费资源。

做三次握手：
  → 第一步：客户端说"我能发"（服务端收到了SYN，确认客户端能发）
  → 第二步：服务端说"我能收也能发"（客户端收到了SYN+ACK，确认服务端能收能发）
  → 第三步：客户端说"我收到了你的确认"（服务端确认客户端也能收）
  → 双方都确认对方既能发也能收！✅
```

> 🤔 想多一点：为什么不是四次握手？三次就够了。服务端的SYN和ACK可以合并在一个包里（SYN+ACK），没必要拆成两个包。拆成两个包反而浪费一趟往返时间。

### 用Go代码观察三次握手

虽然你通常不需要手动实现TCP握手（操作系统内核帮你做了），但可以用Go的TCP服务器来感受这个过程：

```go
package main

import (
    "fmt"
    "net"
)

func main() {
    listener, err := net.Listen("tcp", ":8888")
    if err != nil {
        fmt.Println("监听失败:", err)
        return
    }
    defer listener.Close()
    fmt.Println("TCP服务器已启动，等待连接...")

    for {
        conn, err := listener.Accept()
        if err != nil {
            fmt.Println("接受连接失败:", err)
            continue
        }
        fmt.Println("一个新客户端连接了！三次握手已完成。")
        fmt.Printf("客户端地址: %s\n", conn.RemoteAddr())

        buf := make([]byte, 1024)
        n, _ := conn.Read(buf)
        fmt.Printf("收到数据: %s\n", string(buf[:n]))
        conn.Write([]byte("收到！\n"))
        conn.Close()
    }
}
```

当你运行这个程序并用 `telnet localhost 8888` 或 `nc localhost 8888` 连接时，操作系统在后台已经悄悄完成了三次握手。`Accept()` 返回时，连接已经建立好了。

## 9.3 四次挥手（Four-Way Wavehand）

连接建立后总要断开。TCP断开连接需要**四次挥手**。

### 挂电话的比喻

再想一下挂电话的情景——但这次是两个人都不想失礼：

```
你：   "我说完了，你还有要说的吗？"              （第1次）
朋友： "好的，我知道了。我还有点要说的..."        （第2次）
       ...（朋友继续说完剩下的话）...
朋友： "好，我也说完了。"                        （第3次）
你：   "好的，我知道了。拜拜。"                   （第4次）
```

为什么是四次？因为**TCP是全双工的**——双方都可以同时发送数据。你这一方说"我不发了"，不代表对方也发完了。所以需要**双方各自独立地关闭自己这一侧**。

### 四次挥手的具体过程

```
客户端                         服务端
  │                              │
  │  ── FIN (seq=u) ────────→   │  第1次：客户端说"我不发了"
  │                              │  客户端进入 FIN-WAIT-1
  │                              │
  │  ←── ACK (ack=u+1) ──────   │  第2次：服务端说"收到"
  │                              │  服务端进入 CLOSE-WAIT
  │                              │  客户端进入 FIN-WAIT-2
  │                              │
  │         ... 服务端可能还在发数据 ...
  │                              │
  │  ←── FIN (seq=v) ────────   │  第3次：服务端也说"我也不发了"
  │                              │  服务端进入 LAST-ACK
  │                              │
  │  ── ACK (ack=v+1) ──────→   │  第4次：客户端说"收到"
  │                              │  客户端进入 TIME-WAIT
  │                              │  服务端收到后进入 CLOSED
  │                              │
  │    ... 客户端等2MSL后关闭 ...
```

### TIME-WAIT状态

注意到吗？客户端发出最后一个ACK后，不是立刻关闭，而是进入了一个叫**TIME-WAIT**的状态，要等大约**2MSL**（Maximum Segment Lifetime，最大报文生存时间，通常是2分钟，但实际上很多系统设为30秒~60秒）才真正关闭。

为什么？想象这个场景：

```
客户端发出最后一个ACK → 但它在网络中丢失了！
服务端没收到ACK → 服务端会重新发一次FIN
客户端的连接已经关了 → FIN到了没人接收
服务端永远收不到ACK → 服务端永远无法正常关闭 😱
```

所以客户端在2MSL内保持"半死不活"状态：如果收到服务端重传的FIN，就再发一次ACK。这保证了**最后一个ACK即使丢了，也能被重传修复**。

> 🤔 想多一点：这也是为什么你重启一个服务器程序时，偶尔会碰到 `bind: address already in use` 的错误。上一个进程可能还处于TIME-WAIT状态，端口还没释放。解决方法：设置 `SO_REUSEADDR` socket选项，允许重用TIME-WAIT状态的端口。

## 9.4 TCP可靠性机制

TCP到底是怎么保证"不丢、不乱、不重复、不出错"的？它有一整套机制：

### 9.4.1 序号和确认号

每个TCP数据包都有一个**序号**（Sequence Number）和一个**确认号**（Acknowledgment Number）。

```
发送方：我要发3个包，序号分别是 1000, 2000, 3000
接收方：收到1000 → 回复 ack=1001（"下一个我要1001"）
        收到2000 → 回复 ack=2001（"下一个我要2001"）
        收到3000 → 回复 ack=3001（"下一个我要3001"）
```

序号让接收方知道**包的顺序**，确认号让发送方知道**对方收到了哪些**。

### 9.4.2 超时重传

发送方每发一个包，就启动一个计时器。如果在规定时间内没收到确认，就**重发**。

这就像你寄快递，如果对方3天没回应"收到了"，你就再寄一次。

### 9.4.3 流量控制（Flow Control）

接收方处理不过来怎么办？TCP用**滑动窗口**机制解决这个问题。

```
接收方： "我的缓冲区还剩65535字节空间，别发太快！"
发送方： "好的，我只发65535字节以内，等你处理完了再发"
```

每次回复ACK时，接收方都会在TCP头里带上自己的**窗口大小**（Window Size）——即"我还能吃下多少"。发送方据此调整发送速度。这就是**流量控制**。

类比：一个吃播主播吃汉堡。观众（发送方）问他："你还能吃几个？"主播（接收方）说："嘴里还有3个，再给1个就行。"观众就只给1个，等主播说"嘴里空了，再来10个吧"再加速。

### 9.4.4 拥塞控制（Congestion Control）

流量控制是"接收方吃不消"，拥塞控制是"网络吃不消"。

当网络拥堵时，路由器会丢弃数据包。TCP检测到丢包后，不会立刻猛发，而是启动**拥塞控制**：

1. **慢启动**（Slow Start）：从1个包开始，每收到确认就翻倍（1→2→4→8→16...），探测网络容量。
2. **拥塞避免**（Congestion Avoidance）：达到阈值后，线性增长（每次加1），不再翻倍。
3. **快速重传**（Fast Retransmit）：收到3个重复ACK（说明后面的包到了但中间有个包没到），不等到超时就立刻重传。
4. **快速恢复**（Fast Recovery）：快速重传后，不重头开始慢启动，而是把窗口减半后继续拥塞避免。

> 🤔 想多一点：1986年互联网经历了一次著名的"拥塞崩溃"——吞吐量从32Kbps暴跌到40bps。正是这次事件催生了TCP拥塞控制算法的诞生。现在有几十种拥塞控制算法变种，如CUBIC（Linux默认）、BBR（Google开发），它们各有优劣，适用于不同场景。

类比：一条高速公路（网络）堵车了，更多的车（包）冲上高速只会更堵。所以交警（拥塞控制）开始限流，等路况好转再放行。

### 9.4.5 校验和（Checksum）

每个TCP包都带一个**校验和**——把包里的所有数据做一个数学运算，得到一个数值。接收方收到包后，用同样的算法再算一遍，和发送方附带的校验和对比。不一致？说明数据在传输中损坏了，丢弃这个包，等发送方重传。

这就像买水果时，摊主先称一下说"5斤整"，你回家自己称发现是4.8斤——数据被篡改了，回去找摊主（重传）。

## 本章小结

| 知识点 | 一句话概括 |
|--------|------------|
| TCP是什么 | 传输层的可靠传输协议，保证不丢、不乱、不重复、不出错 |
| 三次握手 | SYN → SYN+ACK → ACK，确认双方发送和接收能力都正常 |
| 四次挥手 | FIN → ACK → FIN → ACK，双方各自独立关闭自己一侧 |
| TIME-WAIT | 客户端发完最后ACK后等2MSL，防止最后的ACK丢失 |
| 序号/确认号 | 靠序号保证顺序，靠确认号保证送达 |
| 超时重传 | 发出去没收到确认？重发！ |
| 流量控制 | 接收方告诉发送方自己还能吃多少 |
| 拥塞控制 | 网络堵了自动减速，避免把路压垮 |
| 校验和 | 数据坏了能发现，丢弃等重传 |
