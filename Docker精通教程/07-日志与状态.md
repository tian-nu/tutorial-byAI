## 我在做什么？

容器跑起来了，但你看不到它里面发生了什么。它有没有报错？内存占用多少？CPU 有没有飙到 100%？哪个进程在吃资源？这些信息不会自动出现在你面前——你需要用 Docker 提供的"诊断工具"去查看。

这一章教你四个核心命令：`docker logs`（看日志）、`docker inspect`（看配置）、`docker stats`（看资源）、`docker top`（看进程）。学完之后，你就能像医生一样"诊断"容器的健康状况。

学完这一章，你能：
- 用 `docker logs` 查看容器日志，用 `-f` 实时跟踪，用 `--tail` 控制输出量
- 用 `docker inspect` 查看容器的完整配置信息，用 `--format` 提取特定字段
- 用 `docker stats` 实时监控容器的 CPU、内存、网络、磁盘使用
- 用 `docker top` 查看容器内运行的进程
- 理解日志轮转的重要性，避免日志撑爆磁盘

## 一、`docker logs`：看容器在说什么

### 基本用法

```bash
docker logs my-nginx
```

输出的是容器**主进程**的标准输出（stdout）和标准错误（stderr）。对于 Nginx，你会看到类似这样的日志：

```
/docker-entrypoint.sh: /docker-entrypoint.d/ is not empty...
2026/01/01 12:00:00 [notice] 1#1: using the "epoll" event method
2026/01/01 12:00:00 [notice] 1#1: nginx/1.27.4
2026/01/01 12:00:00 [notice] 1#1: start worker processes
172.17.0.1 - - [01/Jan/2026:12:00:05 +0000] "GET / HTTP/1.1" 200 615
```

> 注意：`docker logs` 只能看到**主进程的输出**。如果容器内的程序把日志写到文件里（比如 `/var/log/nginx/access.log`），`docker logs` 是看不到的。你需要进入容器或用卷挂载才能访问那些文件。

### 三个最常用的选项

**`-f`（`--follow`）：实时跟踪**

```bash
docker logs -f my-nginx
```

效果：和 `tail -f` 一样，终端会持续显示新的日志，直到你按 `Ctrl+C` 退出。适合在另一个终端窗口里"盯着"日志看。

```
# 你在终端 1 里跑
docker logs -f my-nginx

# 在终端 2 里访问
curl http://localhost:8080

# 终端 1 会立刻显示对应的访问日志
```

**`--tail`：只看最后 N 行**

```bash
# 只看最后 100 行
docker logs --tail 100 my-nginx

# 只看最后 10 行
docker logs --tail 10 my-nginx
```

适合日志量很大的容器——你不想等它把所有日志都打印出来。

**`--since` / `--until`：按时间范围过滤**

```bash
# 最近 1 小时的日志
docker logs --since 1h my-nginx

# 最近 30 分钟的日志
docker logs --since 30m my-nginx

# 从某个时间点开始
docker logs --since 2026-01-01T10:00:00 my-nginx

# 到某个时间点为止
docker logs --until 2026-01-01T12:00:00 my-nginx

# 时间范围组合
docker logs --since 1h --until 30m my-nginx
```

### 组合使用

```bash
# 实时跟踪最近 50 条，带时间戳
docker logs -f --tail 50 --timestamps my-nginx
```

`--timestamps`（`-t`）会在每条日志前加上时间戳：

```
2026-01-01T12:00:00.123456789Z /docker-entrypoint.sh: ...
2026-01-01T12:00:00.234567890Z 172.17.0.1 - - [01/Jan/2026:12:00:05] "GET / HTTP/1.1" 200
```

### 日志驱动的坑：日志会无限增长

`docker logs` 默认使用 `json-file` 日志驱动，日志会一直积累，**不会自动删除**。如果你有一个容器跑了三个月，每天产生几 GB 日志，`docker logs` 会输出所有日志——可能把终端卡死，也可能撑爆磁盘。

```bash
# 查看日志文件的大小
docker inspect --format='{{.LogPath}}' my-nginx
# 输出类似：/var/lib/docker/containers/abc123/abc123-json.log
ls -lh /var/lib/docker/containers/abc123/abc123-json.log
# 可能看到：-rw-r----- 1 root root 2.3G Jan 1 12:00 abc123-json.log
```

**解决方法：在启动容器时配置日志轮转**

```bash
docker run -d \
  --name my-nginx \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  nginx:1.27.4
```

含义：
- `max-size=10m`：单个日志文件最大 10MB，超过后轮转
- `max-file=3`：最多保留 3 个轮转文件

这样，日志最多占用 10MB × 3 = 30MB 的磁盘空间。

## 二、`docker inspect`：容器的"全身体检报告"

### 基本用法

```bash
docker inspect my-nginx
```

输出是一个巨大的 JSON 对象，包含容器的**所有配置信息**：

- 容器 ID、创建时间、状态
- 镜像信息
- 网络配置（IP 地址、端口映射）
- 挂载的卷
- 环境变量
- 资源限制
- 重启策略
- ……几乎你能想到的关于这个容器的所有信息

> 输出很长，通常几百行。建议用 `| less` 或 `| head` 来控制输出量。

### 用 `--format` 提取特定字段

`--format` *此术语见附录C* 使用 Go 模板语法，从 JSON 中提取你需要的字段。

```bash
# 查看容器的 IP 地址
docker inspect --format='{{.NetworkSettings.IPAddress}}' my-nginx
# 输出：172.17.0.2

# 查看容器的状态
docker inspect --format='{{.State.Status}}' my-nginx
# 输出：running

# 查看容器的 PID（在宿主机上的进程 ID）
docker inspect --format='{{.State.Pid}}' my-nginx
# 输出：12345

# 查看容器是否在运行
docker inspect --format='{{.State.Running}}' my-nginx
# 输出：true

# 查看容器的退出码（如果已停止）
docker inspect --format='{{.State.ExitCode}}' my-nginx
# 输出：0（正常退出）

# 查看容器的启动命令
docker inspect --format='{{.Config.Cmd}}' my-nginx
# 输出：[nginx -g daemon off;]

# 查看容器的环境变量
docker inspect --format='{{range .Config.Env}}{{println .}}{{end}}' my-nginx
```

### 常用的 `--format` 查询

| 你要查什么 | 命令 |
|-----------|------|
| 容器 IP | `docker inspect --format='{{.NetworkSettings.IPAddress}}' 容器名` |
| 容器状态 | `docker inspect --format='{{.State.Status}}' 容器名` |
| 容器是否在运行 | `docker inspect --format='{{.State.Running}}' 容器名` |
| 宿主机 PID | `docker inspect --format='{{.State.Pid}}' 容器名` |
| 镜像名 | `docker inspect --format='{{.Config.Image}}' 容器名` |
| 日志文件路径 | `docker inspect --format='{{.LogPath}}' 容器名` |
| 端口映射 | `docker inspect --format='{{range $p, $conf := .NetworkSettings.Ports}}{{$p}} -> {{(index $conf 0).HostPort}}{{"\n"}}{{end}}' 容器名` |
| 挂载卷 | `docker inspect --format='{{range .Mounts}}{{.Source}} -> {{.Destination}}{{"\n"}}{{end}}' 容器名` |

> 不需要背这些命令。需要时回来查，或者用 `docker inspect 容器名 | grep 关键词` 快速定位。

## 三、`docker stats`：实时资源监控

### 基本用法

```bash
docker stats
```

输出一个实时刷新的表格，显示所有运行中容器的资源使用情况：

```
CONTAINER ID   NAME        CPU %     MEM USAGE / LIMIT     MEM %     NET I/O          BLOCK I/O
abc123def456   my-nginx    0.01%     8.5MiB / 15.62GiB     0.05%     1.2kB / 856B     0B / 0B
def456abc789   my-mysql     2.35%     256MiB / 15.62GiB     1.60%     45kB / 23kB      12MB / 0B
```

| 列名 | 含义 | 注意 |
|------|------|------|
| **CONTAINER ID** | 容器 ID | 前 12 位 |
| **NAME** | 容器名 | 你指定的或随机生成的 |
| **CPU %** | CPU 使用百分比 | 这是**实时瞬时值**，不是平均值 |
| **MEM USAGE / LIMIT** | 内存使用 / 限制 | LIMIT 是宿主机总内存（如果没有设置容器内存限制） |
| **MEM %** | 内存使用百分比 | 基于 LIMIT 计算 |
| **NET I/O** | 网络收发量 | 累计值，不是实时速率 |
| **BLOCK I/O** | 磁盘读写量 | 累计值 |

### 只看特定容器

```bash
docker stats my-nginx

# 只看多个指定容器
docker stats my-nginx my-mysql
```

### 只看一次（不持续刷新）

```bash
docker stats --no-stream
```

适合脚本中获取数据。

### 重要：`docker stats` 的数据不是累计值

一个常见的误解：以为 `docker stats` 显示的 CPU % 是"从容器启动到现在的平均值"。**不是的。** Docker 实时计算的是**当前时刻和上一时刻之间的 CPU 使用率**。

如果你看到 CPU 0.01%，不代表容器一直很闲——可能 5 分钟前它飙到了 90%，只是现在刚好空闲。

### 自定义输出格式

```bash
# 只显示容器名、CPU、内存
docker stats --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}" --no-stream

# 输出：
# NAME                CPU %               MEM USAGE / LIMIT
# my-nginx            0.01%               8.5MiB / 15.62GiB
```

## 四、`docker top`：看容器里跑了什么进程

### 基本用法

```bash
docker top my-nginx
```

输出：

```
UID                 PID                 PPID                C                   STIME               TTY                 TIME                CMD
root                12345               12320               0                   12:00               ?                   00:00:00            nginx: master process nginx -g daemon off;
systemd+            12390               12345               0                   12:00               ?                   00:00:00            nginx: worker process
systemd+            12391               12345               0                   12:00               ?                   00:00:00            nginx: worker process
```

| 列名 | 含义 |
|------|------|
| **UID** | 进程的用户 |
| **PID** | 进程在**容器内**的 PID |
| **PPID** | 父进程 PID |
| **C** | CPU 使用率 |
| **STIME** | 进程开始时间 |
| **TTY** | 关联的终端 |
| **TIME** | 累计 CPU 时间 |
| **CMD** | 进程的命令行 |

> 注意：这里显示的 PID 是**容器内部的 PID 命名空间**中的 PID，和宿主机上的 PID 不同。你可以用 `docker inspect --format='{{.State.Pid}}' my-nginx` 查看宿主机上的 PID。

### 什么时候用 `docker top`？

- 怀疑容器内有多余的进程在跑 → 用 `docker top` 确认
- 容器 CPU 高但不知道哪个进程在吃资源 → 用 `docker top` 看进程列表
- 想确认容器的主进程是否是你期望的那个

## 五、`docker port`：查看端口映射

### 基本用法

```bash
docker port my-nginx
```

输出：

```
80/tcp -> 0.0.0.0:8080
80/tcp -> [::]:8080
```

含义：容器内的 80 端口映射到了宿主机的 8080 端口（IPv4 和 IPv6）。

### 查看特定端口

```bash
docker port my-nginx 80
# 输出：
# 0.0.0.0:8080
# [::]:8080
```

## 六、实战：诊断一个容器

启动一个 Nginx 容器，然后全面诊断它：

```bash
# 1. 启动容器
docker run -d --name diagnose-nginx -p 8080:80 nginx:1.27.4

# 2. 查看日志，确认启动成功
docker logs --tail 20 diagnose-nginx

# 3. 查看容器详细信息
docker inspect --format='IP: {{.NetworkSettings.IPAddress}} | Status: {{.State.Status}} | Image: {{.Config.Image}}' diagnose-nginx

# 4. 查看资源使用
docker stats --no-stream diagnose-nginx

# 5. 查看容器内进程
docker top diagnose-nginx

# 6. 查看端口映射
docker port diagnose-nginx

# 7. 访问一下，产生日志
curl http://localhost:8080

# 8. 实时跟踪日志（在另一个终端窗口跑）
docker logs -f --tail 10 diagnose-nginx

# 9. 清理
docker stop diagnose-nginx
docker rm diagnose-nginx
```

## 我做得对不对？

### 验证方法

请按顺序执行以下操作，**全部成功**才算通过：

```bash
# 1. 启动一个 Nginx 容器
docker run -d --name log-test -p 8080:80 nginx:1.27.4

# 2. 查看最后 10 行日志，确认 Nginx 启动成功
docker logs --tail 10 log-test
# 应该看到 "start worker process" 之类的信息

# 3. 用 curl 访问多次，产生日志
curl http://localhost:8080
curl http://localhost:8080
curl http://localhost:8080

# 4. 查看实时资源使用
docker stats --no-stream log-test
# 应该看到 CPU 和内存数据

# 5. 查看容器内进程
docker top log-test
# 应该看到 nginx master 和 worker 进程

# 6. 用 inspect 查看容器 IP
docker inspect --format='{{.NetworkSettings.IPAddress}}' log-test
# 应该输出类似 172.17.0.x 的 IP

# 7. 查看端口映射
docker port log-test
# 应该输出 80/tcp -> 0.0.0.0:8080

# 8. 清理
docker stop log-test && docker rm log-test
```

**预期结果**：
- 步骤 2：日志显示 Nginx 正常启动
- 步骤 4：`docker stats` 显示正常的 CPU 和内存使用
- 步骤 5：`docker top` 显示 nginx 的 master 和 worker 进程
- 步骤 6：`docker inspect` 输出有效的 IP 地址

## 不对怎么办？

### 常见错误 1：`docker logs` 输出过多，终端卡死

❌ 错误做法：

```bash
# 对运行了三个月的容器直接跑 docker logs
docker logs my-production-nginx
# 终端被几 GB 的日志淹没，卡死了
```

✅ 正确做法：

```bash
# 先限制输出量
docker logs --tail 100 my-production-nginx

# 或按时间范围
docker logs --since 1h my-production-nginx
```

> 养成习惯：查看日志时先加 `--tail`，确认输出量可控后再去掉限制。

### 常见错误 2：认为 `docker stats` 的 CPU 是累计值

❌ 错误认知：`docker stats` 显示的 CPU 0.01% 意味着容器一直很闲。

✅ 正确认知：`docker stats` 显示的是**瞬时值**，只代表当前时刻和上一采样时刻之间的 CPU 使用率。容器可能 5 分钟前飙到了 90%，只是你现在看的时候刚好空闲。

**想准确监控，用专门的监控工具**（Prometheus + Grafana 等），`docker stats` 适合快速看一眼。

### 常见错误 3：日志不轮转，磁盘被占满

**现象**：某天突然发现磁盘满了，排查发现 `/var/lib/docker/containers/` 下有几十 GB 的日志文件。

**原因**：启动容器时没有配置日志轮转，`json-file` 驱动默认不限制日志大小。

**解决**：

```bash
# 以后启动容器时配置日志轮转
docker run -d \
  --name my-nginx \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  nginx:1.27.4

# 对已经在运行的容器，可以修改 Docker 守护进程的全局配置
# 编辑 /etc/docker/daemon.json（Linux）
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
# 重启 Docker：sudo systemctl restart docker
```

### 常见错误 4：`docker inspect` 输出太长，找不到想要的字段

❌ 错误做法：`docker inspect my-nginx`，然后肉眼在几百行 JSON 里找。

✅ 正确做法：

```bash
# 方法 1：用 --format 提取
docker inspect --format='{{.NetworkSettings.IPAddress}}' my-nginx

# 方法 2：用 grep 过滤
docker inspect my-nginx | grep -A 5 "IPAddress"

# 方法 3：用 jq 处理（需要安装 jq）
docker inspect my-nginx | jq '.[0].NetworkSettings.IPAddress'
```

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 查看容器日志 | `docker logs` 容器名 | 只能看到主进程的 stdout/stderr；日志文件内的日志看不到 |
| 实时跟踪日志 | `docker logs -f` 容器名 | 按 Ctrl+C 退出，不影响容器 |
| 限制日志输出量 | `--tail 100`、`--since 1h` | 日志量大时先加限制，避免终端卡死 |
| 日志轮转 | `--log-opt max-size=10m --log-opt max-file=3` | 不配置的话日志会无限增长，可能撑爆磁盘 |
| 查看容器详细信息 | `docker inspect` 容器名 | 输出几百行 JSON，用 `--format` 或 `grep` 提取特定字段 |
| 提取特定字段 | `docker inspect --format='{{.State.Status}}'` | Go 模板语法，常用字段见表 |
| 实时资源监控 | `docker stats` | CPU 和网络 I/O 是瞬时值/累计值，注意区分 |
| 查看容器内进程 | `docker top` 容器名 | 显示容器内 PID 命名空间中的进程 |
| 查看端口映射 | `docker port` 容器名 | 确认端口映射是否正确 |

## 本篇最可能出错的地方及原因

### 1. 日志不轮转，磁盘悄然被占满

**原因**：`json-file` 日志驱动默认不限制日志大小。很多人不知道需要配置 `--log-opt`，容器跑了几个月后磁盘满了才发现。

**排查**：`docker inspect --format='{{.LogPath}}' 容器名 | xargs ls -lh` 查看日志文件大小。

**解决**：所有生产容器启动时都加上 `--log-opt max-size=10m --log-opt max-file=3`。

### 2. `docker stats` 的 CPU 数据被误解

**原因**：`docker stats` 的 CPU % 是瞬时值，不是平均值。看到 0.01% 不代表容器不耗资源。

**建议**：`docker stats` 适合快速检查，长期监控用 Prometheus + Grafana 等专业工具。

### 3. `docker logs` 只能看到 stdout/stderr

**原因**：很多应用把日志写到文件（如 `/var/log/nginx/access.log`），`docker logs` 看不到这些。新手以为 `docker logs` 能看所有日志。

**解决**：要么让应用把日志输出到 stdout/stderr（推荐），要么用卷挂载日志目录到宿主机。
