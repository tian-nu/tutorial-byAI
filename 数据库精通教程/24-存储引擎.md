## 我在做什么？

前面三章我们讲了事务、MVCC、锁，每次说到"InnoDB 怎么怎么样"。你有没有想过：**MySQL 为什么是 InnoDB？有没有别的选择？**

实际上，MySQL 是一个"可插拔存储引擎"架构——你可以给不同的表选择不同的"存储方式"。最常见的引擎有 InnoDB、MyISAM、Memory，它们各有各的脾气。

这一章我们分三条线讲：
1. **引擎对比线**：InnoDB vs MyISAM vs Memory，各有什么优劣
2. **redo log / binlog 分叉线**：这两个日志经常被搞混，我们用"崩溃恢复"和"主从复制"两个视角分开讲
3. **InnoDB 内存结构线**：Buffer Pool、Change Buffer、Log Buffer 各司其职

学完这一章，你能解释"为什么 InnoDB 支持崩溃恢复而 MyISAM 不支持"，能说出 redo log 和 binlog 的 5 个区别，以及理解两阶段提交在 MySQL 中的含义。

## 一、存储引擎是什么？

**存储引擎（Storage Engine）** *此术语见附录E* 是 MySQL 存储数据的"底层实现"。你可以理解为：MySQL 把"怎么存数据"这件事外包给了不同的引擎，每个引擎有不同的特点。

```sql
-- 查看 MySQL 支持哪些引擎
SHOW ENGINES;
```

输出中最常见的三个：

| Engine | Support | 一句话 |
|--------|---------|--------|
| InnoDB | DEFAULT | 默认引擎，支持事务、行锁、外键、崩溃恢复 |
| MyISAM | YES | 老牌引擎，不支持事务，读速度快 |
| MEMORY | YES | 数据全在内存，重启就没了 |

**关键概念**：引擎是**表级别**的，不是库级别的。你可以 `users` 表用 InnoDB，`logs` 表用 MyISAM：

```sql
CREATE TABLE users (
  id INT PRIMARY KEY
) ENGINE=InnoDB;

CREATE TABLE access_logs (
  id INT, log TEXT
) ENGINE=MyISAM;
```

## 二、三大引擎对比

### 2.1 核心特性对比表

| 特性 | InnoDB | MyISAM | Memory |
|------|--------|--------|--------|
| 事务支持 | ✅ 支持 | ❌ 不支持 | ❌ 不支持 |
| 行级锁 | ✅ 支持 | ❌ 只有表锁 | ❌ 只有表锁 |
| 外键 | ✅ 支持 | ❌ 不支持 | ❌ 不支持 |
| 崩溃恢复 | ✅ 支持（redo log） | ❌ 不支持 | ❌ 不支持（内存数据） |
| MVCC | ✅ 支持 | ❌ 不支持 | ❌ 不支持 |
| 全文索引 | MySQL 5.6+ 支持 | ✅ 支持 | ❌ 不支持 |
| 数据存储 | 磁盘（共享表空间） | 磁盘（.MYD + .MYI） | 内存 |
| 读性能 | 好 | 很好（简单场景） | 极快 |
| 写性能 | 好 | 一般（表锁） | 极快 |
| 压缩 | 支持 | 支持（读多写少场景） | 不支持 |
| 默认 | ✅ MySQL 5.5+ 默认 | MySQL 5.5 之前默认 | 否 |

### 2.2 InnoDB：现在的默认引擎，全能选手

InnoDB 是 MySQL 5.5 之后的默认引擎，也是现在绝大多数场景的首选。它的核心优势：
- **事务（ACID）**：支持完整的 ACID 事务
- **行级锁**：高并发写入性能好
- **外键**：数据库层保证引用完整性
- **崩溃恢复**：redo log 保证数据不丢

**什么时候用 InnoDB？** 几乎所有场景。除非你有非常明确的理由，否则就选 InnoDB。

### 2.3 MyISAM：老牌引擎，读多写少场景

MyISAM 是 MySQL 早期默认引擎，它的设计哲学是"简单、快"。

**优点**：
- 读速度快（没有事务、锁开销小）
- 表容易压缩，节省磁盘
- 全文索引比 InnoDB 更成熟（MySQL 5.6 之前）

**致命缺点**：
- **不支持事务**：数据一致性没保障
- **只有表锁**：写入时整张表锁住，高并发写性能极差
- **不支持崩溃恢复**：数据库崩溃后，表可能损坏，需要用 `REPAIR TABLE` 修复

**什么时候用 MyISAM？** 很少了。只有在"读多写少 + 不在乎事务 + 不在乎崩溃"的场景下考虑，比如日志归档表。

### 2.4 Memory：数据全在内存，重启就没了

Memory 引擎把数据全存在内存中，速度极快，但数据库重启后数据全部丢失。

**什么时候用 Memory？** 临时表、会话数据、缓存。但注意：Redis 通常比 Memory 引擎更适合做缓存。

## 三、redo log：崩溃恢复的保障

前两章我们提到 redo log 多次，现在正式讲它。

### 3.1 问题：内存改了但磁盘没写，怎么办？

回想一下数据的写入过程：
1. 你要修改一行数据 → MySQL 先从磁盘读到内存（Buffer Pool）
2. 在内存中修改 → 此时内存和磁盘不一致了（这叫"脏页"）
3. 某个时刻，把脏页刷回磁盘

如果步骤 2 完成后、步骤 3 之前，数据库崩溃了——**内存中的修改丢失，磁盘上还是旧数据。**

这就是 redo log 要解决的问题。

### 3.2 WAL 机制：先写日志，再写磁盘

**WAL（Write-Ahead Logging）** *此术语见附录E*：**在修改数据之前，先把"我要怎么改"写进日志。**

```
正常流程：
  1. 修改 Buffer Pool 中的数据页
  2. 把修改记录写入 redo log（先写日志！）
  3. 返回"写入成功"给客户端
  4. 后台慢慢把脏页刷回磁盘

崩溃恢复：
  1. 数据库重启，发现 redo log 中有记录
  2. 重放 redo log 中的修改 → 把 Buffer Pool 恢复到崩溃前的状态
  3. 数据不丢！
```

### 3.3 redo log 的关键特性

| 特性 | 说明 |
|------|------|
| 层级 | InnoDB 引擎层，不是 Server 层 |
| 类型 | 物理日志（记录"在哪个页的哪个位置改了哪个值"） |
| 写入方式 | 循环写（ib_logfile0, ib_logfile1） |
| 大小 | 固定大小，配满后从头覆盖 |
| 何时写 | 事务执行过程中持续写 |
| 何时刷盘 | 事务提交时（`innodb_flush_log_at_trx_commit=1`） |

**比喻**：redo log 就像一个"操作备忘录"。你修改数据前，先在备忘录上写"我要改第 3 页超市的账本，把张三的余额从 1000 改成 900"。万一你突然被打断（崩溃），回来一看备忘录，就知道刚才做了什么，继续做完。

### 3.4 关键参数

```sql
-- 查看 redo log 相关参数
SHOW VARIABLES LIKE 'innodb_log%';

-- innodb_flush_log_at_trx_commit 的值非常重要：
-- 0：每秒刷一次，性能最好，但可能丢失 1 秒数据
-- 1：每次提交都刷盘，最安全，性能最差（默认）
-- 2：每次提交写 OS 缓存，每秒刷盘，折中
```

## 四、binlog：主从复制的基石

binlog 和 redo log 经常被混淆，但它们是**完全不同的东西**。

### 4.1 binlog 是什么？

**binlog（Binary Log）** *此术语见附录E* 是 MySQL Server 层的日志，记录所有"修改数据或结构"的 SQL 语句（或行变更）。

**主要用途**：
- **主从复制**：从库通过读取主库的 binlog 来同步数据
- **数据恢复**：备份 + binlog 可以实现基于时间点的恢复

### 4.2 binlog 的关键特性

| 特性 | 说明 |
|------|------|
| 层级 | MySQL Server 层，所有引擎共用 |
| 类型 | 逻辑日志（记录"执行了什么 SQL"或"哪行变成什么样"） |
| 写入方式 | 追加写，文件写满后新建 |
| 大小 | 不固定，通过 `max_binlog_size` 控制，过期的可配置清理 |
| 何时写 | 事务提交时写入 |
| 格式 | STATEMENT（记 SQL）/ ROW（记行变更）/ MIXED |

### 4.3 binlog 三种格式

```sql
-- 查看 binlog 格式
SHOW VARIABLES LIKE 'binlog_format';
```

| 格式 | 记录内容 | 优点 | 缺点 |
|------|---------|------|------|
| STATEMENT | 记录 SQL 语句 | 日志小 | 某些函数（UUID、NOW）在主从结果不一致 |
| ROW | 记录每行变更 | 最精确，推荐 | 日志大（UPDATE 全表会记录每行） |
| MIXED | 混合，默认 STATEMENT，必要时 ROW | 折中 | 有些场景可能选错 |

### 4.4 redo log vs binlog：一张表分清

这是面试高频问题，必须能张口就来：

| 维度 | redo log | binlog |
|------|----------|--------|
| 所属层级 | **InnoDB 引擎层** | **MySQL Server 层** |
| 日志类型 | 物理日志（页的修改） | 逻辑日志（SQL 或行的变更） |
| 写入方式 | 循环写（固定大小） | 追加写（不断新建文件） |
| 主要用途 | **崩溃恢复（Crash Recovery）** | **主从复制、数据恢复** |
| 何时产生 | 事务执行过程中持续写 | 事务提交时写入 |
| 能不能关？ | 不能（InnoDB 必须） | 可以关（`--skip-log-bin`） |
| 谁用？ | InnoDB 自己用 | 从库、备份工具用 |

**比喻助记**：
- **redo log** = 你的草稿纸，随时记录修改计划，写完就丢掉，作用是防止忘记
- **binlog** = 你的工作日志本，每完成一件事就记一笔，作用是给同事看（从库）/ 以后回溯（数据恢复）

## 五、两阶段提交：让 redo log 和 binlog 保持一致

这里有一个很微妙的问题：**事务提交时，既要写 redo log，又要写 binlog。如果两个日志的写入顺序不一致，或者写了一个另一个没写，会怎样？**

### 5.1 问题场景

假设先写 redo log，再写 binlog：
- redo log 写完了，但 binlog 没写完，系统崩溃
- 重启后，redo log 恢复了这个事务
- 但这在 binlog 中不存在 → 从库收不到这个事务 → 主从数据不一致！

假设先写 binlog，再写 redo log：
- binlog 写完了，但 redo log 没写完，系统崩溃
- 重启后，主库没有这个事务
- 但从库已经通过 binlog 同步了这个事务 → 主从数据不一致！

### 5.2 两阶段提交解决方案

**两阶段提交（Two-Phase Commit）** *此术语见附录E* 在 MySQL 语境下，是指协调 redo log 和 binlog 的一致性。注意：**这不是分布式事务的两阶段提交，是 MySQL 内部写日志的机制。**

```
两阶段提交流程：

1. PREPARE 阶段（准备阶段）：
   - 事务执行完毕，先写 redo log，标记为 "prepare" 状态
   
2. COMMIT 阶段（提交阶段）：
   - 写 binlog
   - 把 redo log 标记为 "commit" 状态
   - 事务提交完成
```

**崩溃恢复时的判断逻辑：**

| 重启时 redo log 状态 | 重启时 binlog 状态 | 处理 |
|---------------------|-------------------|------|
| prepare | 有完整记录 | 提交事务（redo log → commit） |
| prepare | 无记录 | 回滚事务 |
| commit | — | 已完成，无需处理 |

这样就能保证：**redo log 和 binlog 要么都成功，要么都失败，不会出现不一致。**

## 六、undo log：补充物理存储

第 22 章我们讲了 undo log 在 MVCC 中的角色——它是版本链的"原材料"。这里补充一下它的物理存储。

**undo log** 存储在 InnoDB 的**共享表空间（ibdata1）** 或独立的 undo 表空间中。它记录的是"修改前的数据"，用于：
- 事务回滚（ROLLBACK 时，用 undo log 恢复旧值）
- MVCC（构建版本链，让快照读看到历史版本）

```sql
-- 查看 undo 表空间设置
SHOW VARIABLES LIKE 'innodb_undo%';
```

## 七、InnoDB 内存结构

InnoDB 在内存中维护了几个重要的缓冲区，这是理解 InnoDB 性能的关键。

### 7.1 Buffer Pool（缓冲池）

**Buffer Pool** *此术语见附录E* 是 InnoDB 中最大的内存区域，缓存数据页和索引页。

**工作原理**：
1. 读取数据时，先查 Buffer Pool → 命中就返回，不命中就从磁盘读
2. 修改数据时，在 Buffer Pool 中修改，标记为"脏页"
3. 后台线程定期把脏页刷回磁盘

**Buffer Pool 就像图书馆的书桌**：你从书架（磁盘）拿书放到书桌上（Buffer Pool），在书桌上阅读和修改。书桌越大，能同时放的书就越多，来回跑书架的次数就越少。

```sql
-- 查看 Buffer Pool 大小（默认 128MB）
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';

-- 查看 Buffer Pool 命中率（越高越好，理想值 > 99%）
SHOW STATUS LIKE 'Innodb_buffer_pool_read%';
-- Innodb_buffer_pool_read_requests：总请求次数
-- Innodb_buffer_pool_reads：需要读磁盘的次数
-- 命中率 = (requests - reads) / requests * 100%
```

### 7.2 Change Buffer（变更缓冲）

**Change Buffer** *此术语见附录E* 是 Buffer Pool 的一部分，专门缓存**对非唯一二级索引的修改**。

**为什么需要它？** 当你插入一行数据，除了更新主键索引，还要更新所有二级索引。如果这些二级索引页不在 Buffer Pool 中，就需要读磁盘。Change Buffer 先把这些修改缓存在内存中，等对应的索引页被加载到 Buffer Pool 时再合并（merge），从而减少磁盘 IO。

### 7.3 Log Buffer（日志缓冲）

**Log Buffer** 是 redo log 的内存缓冲区。事务执行时，先把 redo log 写入 Log Buffer，达到一定条件再刷到磁盘的 redo log 文件中。

```sql
-- 查看 Log Buffer 大小（默认 16MB）
SHOW VARIABLES LIKE 'innodb_log_buffer_size';
```

### 7.4 一条 UPDATE 的完整旅程

把三个缓冲区串起来，看一条 `UPDATE` 在 InnoDB 中是怎么走的：

```
1. 从磁盘读取目标数据页 → 放入 Buffer Pool
2. 在 Buffer Pool 中修改数据 → 变成脏页
3. 把修改记录写入 Log Buffer（redo log 内存版）
4. 事务提交时：
   a. Log Buffer 刷到磁盘 redo log（PREPARE 阶段）
   b. 写 binlog
   c. redo log 标记为 COMMIT（两阶段提交完成）
   d. 返回客户端"提交成功"
5. 后台线程慢慢把脏页刷回磁盘
```

## 八、查看和切换引擎

```sql
-- 查看所有引擎
SHOW ENGINES;

-- 查看某张表用的什么引擎
SHOW TABLE STATUS LIKE 'accounts';
-- 或者
SELECT ENGINE FROM information_schema.TABLES 
WHERE TABLE_SCHEMA = 'my_shop' AND TABLE_NAME = 'accounts';

-- 查看所有表的引擎
SELECT TABLE_NAME, ENGINE FROM information_schema.TABLES 
WHERE TABLE_SCHEMA = 'my_shop';

-- 切换引擎（注意：会锁表，重建数据）
ALTER TABLE accounts ENGINE = MyISAM;

-- 新建表时指定引擎
CREATE TABLE test (id INT) ENGINE = Memory;
```

## 九、验证方法：自己动手测

### 实验 1：证明 MyISAM 不支持事务

```sql
-- 创建一张 MyISAM 表
CREATE TABLE test_myisam (id INT, val INT) ENGINE = MyISAM;
INSERT INTO test_myisam VALUES (1, 100);

-- 开启事务，修改数据
BEGIN;
UPDATE test_myisam SET val = 200 WHERE id = 1;
SELECT * FROM test_myisam;  -- 看到 200

-- 回滚
ROLLBACK;
SELECT * FROM test_myisam;  -- 还是 200！MyISAM 不支持回滚！

-- 清理
DROP TABLE test_myisam;
```

### 实验 2：查看 Buffer Pool 命中率

```sql
-- 先记录当前状态
SHOW STATUS LIKE 'Innodb_buffer_pool_read%';

-- 执行一些查询
SELECT * FROM accounts;
SELECT * FROM accounts WHERE id = 1;

-- 再次查看
SHOW STATUS LIKE 'Innodb_buffer_pool_read%';
-- 计算命中率
```

### 实验 3：理解 Memory 引擎的"易失性"

```sql
-- 创建 Memory 表
CREATE TABLE test_memory (id INT, val VARCHAR(100)) ENGINE = Memory;
INSERT INTO test_memory VALUES (1, 'hello');

-- 查询，有数据
SELECT * FROM test_memory;

-- 重启 MySQL 后（或断开连接，如果是临时表）
-- 再查询：数据全部丢失！
-- SELECT * FROM test_memory;  -- 空表

-- 清理
DROP TABLE test_memory;
```

## 十、常见错误与排错

### 错误 1：不了解引擎特性就建表

**现象**：你建了一张表，后来发现它不支持事务、不支持外键、崩溃后数据损坏。

**原因**：没有指定引擎，用了默认引擎（通常是 InnoDB），或者无意中指定了 MyISAM。

**解决**：建表时明确指定 `ENGINE = InnoDB`，养成好习惯。检查已有表：
```sql
SELECT TABLE_NAME, ENGINE FROM information_schema.TABLES 
WHERE TABLE_SCHEMA = 'my_shop';
```

### 错误 2：MyISAM 场景用了 InnoDB（或反过来）

**现象 1**：你的日志表，只有追加写入，没有事务需求，用了 InnoDB → 写入性能不如 MyISAM。

**现象 2**：你的业务表，需要事务和外键，用了 MyISAM → 数据一致性没保障。

**选型建议**：
- 需要事务 → 必须 InnoDB
- 纯读多写少、不在乎崩溃 → 可考虑 MyISAM
- 临时数据、缓存 → 可考虑 Memory（但 Redis 更合适）
- **不确定就用 InnoDB**，99% 的情况不会错

### 错误 3：混淆 redo log 和 binlog

**现象**：面试时被问"redo log 和 binlog 有什么区别"，答不上来或答混了。

**记住一张表**：redo log 是 InnoDB 层的物理日志（崩溃恢复），binlog 是 Server 层的逻辑日志（主从复制）。一个管"别丢数据"，一个管"同步给别人"。

### 错误 4：Buffer Pool 太小导致性能差

**现象**：数据库查询很慢，`SHOW STATUS` 看到 Buffer Pool 命中率很低（< 95%）。

**原因**：Buffer Pool 太小，数据页频繁从磁盘加载。

**解决**：增大 Buffer Pool：
```sql
-- 查看当前大小
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';
-- 建议设为物理内存的 50%-70%（专用数据库服务器）
-- 在 my.cnf 中配置：
-- innodb_buffer_pool_size = 4G
```

| MyISAM | 老引擎，不支持事务，读速度快 | 不支持崩溃恢复，慎用 |

| Buffer Pool | 缓存数据页和索引页 | 越大越好，建议物理内存的 50-70% |
| Change Buffer | 缓存二级索引的修改 | 减少磁盘 IO |
| Log Buffer | redo log 的内存缓冲 | 默认 16MB |
| redo log | InnoDB 层物理日志，崩溃恢复 | 循环写，固定大小 |
| binlog | Server 层逻辑日志，主从复制 | 追加写，三种格式 |
| WAL | 先写日志，再写磁盘 | 核心机制 |
| 两阶段提交 | 协调 redo log 和 binlog | MySQL 语境，协调 redo log 和 binlog 的一致性 |
| undo log | 存储共享表空间，记录旧值 | 支持回滚和 MVCC 版本链 |
| 切换引擎 | `ALTER TABLE t ENGINE = InnoDB;` | 会锁表，重建数据 |
| Buffer Pool 命中率 | `SHOW STATUS LIKE 'Innodb_buffer_pool_read%';` | 理想值 > 99% |
