## 什么是上下文窗口：AI的"短期记忆"

先讲一个比喻。

你参加期末考试，老师允许你带一张**草稿纸**进考场。这张草稿纸能写多少字是有限的——比如一张A4纸，正反两面，大概能写2000字。

你在考试时，把公式、定理、例题都记在草稿纸上。当你做题时，你低头看一眼草稿纸，回忆一下，然后写答案。

**AI的"上下文窗口"（📖 术语：Context Window）就是这张草稿纸。**

- 你发的Prompt + AI的回复 + 你之前和AI说的所有话 = 都写在这张草稿纸上
- 草稿纸写满了 → AI就"忘记"最早的内容
- 草稿纸大小取决于AI模型 —— 有的只有4K大小，有的能到200K

> 📖 **术语：上下文（Context）** = AI在处理你的当前请求时，能够"看到"的所有信息。包括你当前的Prompt、之前的对话历史、系统提示词、引用的文件内容等。AI的所有回答，都基于这个上下文窗口内的信息。

## 单位：tokens（词元）

上下文窗口的大小用**tokens**（📖 术语：词元）来衡量，不是"字数"。

> **Token（词元）** = AI阅读文本的最小单位。它不是字，也不是单词，而是把文字切成的小积木块。

**一个token大概是多少？**

| 语言 | 1 Token ≈ |
|------|----------|
| 英文 | 约 0.75 个单词（"hello" = 1 token, "hippopotamus" ≈ 3 tokens）|
| 中文 | 约 1.5-2 个汉字（"你好" = 2 tokens, "人工智能" = 3 tokens）|
| 代码 | 变化很大，Python的 `def` 是1 token，`requests` 是2 tokens |

**一个更直观的估算：**

| 上下文窗口大小 | 大概能放多少中文字 |
|--------------|------------------|
| 4K tokens | 约 3,000 字（一篇短文章） |
| 8K tokens | 约 6,000 字（一篇长文章） |
| 32K tokens | 约 24,000 字（中篇小说一章） |
| 128K tokens | 约 96,000 字（一本中篇小说） |
| 200K tokens | 约 150,000 字（一本长篇小说） |

**看起来很大对吧？但注意：**

- 你的Prompt + AI的回复 + 对话历史 + 引用的文件 + 系统提示词，**全部**都在这个窗口里
- 你一次对话来回10轮，可能已经用掉了好几万tokens
- 如果你@了一个大文件，这个文件的内容会直接吃掉大量tokens

## 上下文不是越多越好：信息过载的危害

你可能会想："那我把整个项目代码都扔给AI，信息越多越好嘛！"

**错。** 这就像考试时你带了整本教科书进考场，翻到眼花——还不如带一页重点笔记。

**为什么信息过载会降低AI输出质量？**

1. **注意力稀释**：AI要处理的信息越多，越容易忽略你真正关心的部分。你给它整个项目10个文件，它可能注意到一个无关紧要的注释，而忽略了你真正想改的那个函数。

2. **噪声干扰**：不相关的代码会干扰AI的判断。比如你只是想改登录逻辑，但你把整个项目的路由配置、前端模板、数据库迁移脚本都扔进去了——AI可能被这些无关内容带偏。

3. **速度变慢**：上下文越大，AI处理越慢，响应时间越长。

4. **成本增加**：很多AI工具按token计费，你扔进去的每一条信息都在花钱。

**类比：**
> 你给AI的上下文，就像你给厨师的食材清单。你给他"面粉、鸡蛋、糖、黄油"——他做蛋糕。你给他"整个超市的库存清单"——他反而不知道你要做什么了。

## 如何精简上下文：只给相关的

**核心原则：按需提供，精准投放。**

### ❌ 坏做法：把整个项目扔给AI

```
（@了项目全部20个文件，总共5000行代码）
帮我给登录函数加个错误处理。
```

> 问题：AI看到5000行代码，可能被其他模块的代码干扰，输出的错误处理风格和你的项目不一致，甚至改到不该改的地方。

### ✅ 好做法：只给AI它需要看的部分

```
我在写一个Flask博客系统，下面是用户登录的现有代码。
帮我加上错误处理：
1. 邮箱格式验证
2. 密码非空检查
3. 数据库查询失败时返回友好错误

现有代码：
```python
@app.route('/login', methods=['POST'])
def login():
    data = request.get_json()
    email = data['email']
    password = data['password']
    user = db.execute("SELECT * FROM users WHERE email = ?", (email,)).fetchone()
    if user and bcrypt.checkpw(password.encode(), user['password']):
        token = jwt.encode(...)
        return jsonify({"token": token})
    return jsonify({"error": "登录失败"})
```

> 你只给了AI 20行相关代码，它就能精准地帮你改这个函数，不会被其他模块干扰。

---

### 精简上下文的几个技巧

| 技巧 | 说明 | 对比 |
|------|------|------|
| **只贴相关函数** | 不要贴整个文件，贴你要改的那个函数 | ❌ `@models.py` → ✅ 贴`def login()`代码 |
| **不贴依赖** | 不需要贴`import`语句，除非你改的是依赖 | ❌ 贴30行import → ✅ 只贴核心逻辑 |
| **不贴无关模块** | 改登录逻辑，就别贴注册逻辑 | ❌ 贴整个auth模块 → ✅ 只贴login函数 |
| **用文字描述结构** | 用"我有一个User表，包含id/email/password字段"代替贴整个SQL | ❌ 贴整个schema.sql → ✅ 一句话描述表结构 |

## @符号的用法：引用外部信息

在Cursor和Trae这类AI编程工具中，你可以用 `@` 符号来**引用外部信息**，让AI能访问到不在你当前对话中的内容。

**常见@引用方式：**

| 引用方式 | 含义 | 示例 |
|---------|------|------|
| `@file` | 引用单个文件 | `@file app.py` — AI会读取这个文件 |
| `@folder` | 引用整个文件夹 | `@folder src/` — AI会读取文件夹内所有文件 |
| `@web` | 搜索网络 | `@web Flask JWT 最新文档` — AI联网搜索 |
| `@docs` | 引用官方文档 | `@docs Flask` — 在Cursor中引用Flask官方文档 |

**@的威力：**

```
帮我分析 @file models.py 中的User模型，看看有没有SQL注入风险。
```

> AI会直接读取你的models.py，然后针对性地分析。你不用复制粘贴代码，AI自己会去读。

**@的坑：**

- `@folder` 可能引用几百个文件，直接撑爆上下文窗口
- 用 `@file` 时，确保你引用的文件确实相关，不要"顺手"引用

## 什么时候清上下文：AI开始"犯糊涂"了

你有没有遇到过这种情况：

- 对话进行了20轮，AI突然开始答非所问
- AI给了你一段代码，但明显是之前某个需求的代码，不是当前需求的
- AI开始"忘记"你10轮之前说过的话
- AI的输出质量明显下降，代码越来越啰嗦

**这些都是上下文满了的信号。**

### 上下文满了的典型表现

| 症状 | 原因 |
|------|------|
| AI开始"忘事" | 早期对话内容被挤出上下文窗口 |
| 输出质量下降 | 上下文太拥挤，AI注意力分散 |
| 回答和你的问题不相关 | AI"混淆"了不同时期的对话 |
| 代码越来越长但质量越来越差 | AI在"凑字数"而非"解决问题" |

### 怎么办：清上下文，重新开始

**方法一：开启新对话**

最简单粗暴的方式。结束当前对话，开启一个新对话，把当前要做的事情重新描述一遍。

**方法二：总结切换**

在清上下文之前，先让AI帮你总结一下当前进度：

```
请帮我总结一下到目前为止我们做了什么，各个文件的状态是什么。
```

然后把AI的总结复制下来，开启新对话，把总结作为上下文贴进去。

**方法三：有选择地清**

在Cursor中，你可以删除对话历史中的某些轮次，保留最近的几轮，让AI"忘记"不重要的早期对话。

## 自测

1. 上下文窗口是什么？用"考试草稿纸"的比喻解释。
2. "帮我写一个函数"这个Prompt，如果把整个项目（5000行代码）都@给AI，会有什么问题？
3. AI开始答非所问、忘记早期对话内容，最可能的原因是什么？你应该怎么办？

> **答案**：
> 1. 上下文窗口就是AI处理当前请求时能看到的全部信息，像草稿纸一样有容量限制。对话历史、Prompt、文件引用都在里面，写满了就"忘记"旧内容。
> 2. 信息过载 — AI注意力被稀释，可能被无关代码干扰，输出质量下降，响应变慢，还可能浪费token配额。应该只贴相关函数和必要描述。
> 3. 最可能的原因是上下文满了。应该开启新对话，或者先让AI总结当前进度，再带着总结开新对话继续。
