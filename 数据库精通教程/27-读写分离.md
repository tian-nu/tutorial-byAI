## 我在做什么？

第 26 章你搭好了主从复制，主库有一个从库跟着同步。但仔细想想，**从库有什么用？**

如果只让主库干活，从库在旁边干瞪眼追着同步，那从库就是个"吃白饭的备用机"。太浪费了。

**读写分离（Read-Write Splitting）** *此术语见附录E* 的核心思想：
- **写操作（INSERT/UPDATE/DELETE）→ 走主库**
- **读操作（SELECT）→ 走从库**

这样：主库压力减半，从库不再干瞪眼，整体吞吐量翻倍。就像餐厅里，大厨（主库）只管炒菜，帮厨（从库）只管上菜，各司其职。

学完这一章，你能：
- 理解读写分离的架构和工具全景
- 用 ProxySQL *此术语见附录E* 配置读写分离
- 处理主从延迟带来的"读不到刚写入的数据"问题
- 了解 MySQL Router *此术语见附录E* 和负载均衡策略

## 二、读写分离架构

### 2.1 两种实现方式

**方式一：应用层实现（代码里判断）**

```python
# ❌ 简单粗暴，但耦合严重，每个项目都要写一遍
def execute_sql(sql):
    if sql.strip().upper().startswith('SELECT'):
        conn = slave_connection   # 读走从库
    else:
        conn = master_connection  # 写走主库
    return conn.execute(sql)
```

- ✅ 不依赖额外中间件
- ❌ 代码耦合，每个项目都要写
- ❌ 主从切换时改代码重启
- ❌ 多语言项目要写多份

**方式二：中间件实现（ProxySQL / MySQL Router）**

```
应用 → ProxySQL（拦截 SQL 自动路由）→ 主库（写）/ 从库（读）
```

- ✅ 应用无感知，代码不用改
- ✅ 主从切换时改中间件配置即可
- ✅ 支持连接池、查询缓存、SQL 防火墙

**推荐：生产环境用中间件。**

### 2.2 架构图

```
                         ┌─────────────┐
                         │   应用服务    │
                         └──────┬──────┘
                                │
                                │ 连接 ProxySQL（端口 6033）
                                ▼
                         ┌─────────────┐
                         │  ProxySQL    │
                         │              │
                         │  SQL 分析    │
                         │  SELECT? → 从库 │
                         │  INSERT/UPDATE/DELETE? → 主库 │
                         │              │
                         └──┬──────┬───┘
                            │      │
              写请求 ──────┘      └────── 读请求
               ▼                          ▼
        ┌─────────────┐          ┌─────────────┐
        │   主库       │  复制    │   从库       │
        │  (Master)    │ ──────► │  (Slave)     │
        │  端口 3306   │  binlog │  端口 3306   │
        └─────────────┘          └─────────────┘
```

ProxySQL 就是一个"数据库代理"，它拦在应用和数据库之间，分析每条 SQL，自动决定发给主库还是从库。

## 三、ProxySQL 入门：安装 + 配置

### 3.1 安装 ProxySQL

**Docker 安装（推荐，最快）：**

```bash
docker run -d \
  --name proxysql \
  --network mysql-repl \
  -p 6033:6033 \
  -p 6032:6032 \
  proxysql/proxysql:2.5
```

两个端口：
- `6033`：应用连接端口（你的程序连这个）
- `6032`：管理端口（配置 ProxySQL 用）

**直接安装：**

```bash
# Debian/Ubuntu
wget https://github.com/sysown/proxysql/releases/download/v2.5.5/proxysql_2.5.5-ubuntu22_amd64.deb
dpkg -i proxysql_2.5.5-ubuntu22_amd64.deb
systemctl start proxysql
```

### 3.2 配置后端 MySQL 服务器

连上 ProxySQL 管理端口：

```bash
# 管理端口 6032
mysql -u admin -padmin -h 127.0.0.1 -P 6032
```

> 默认用户名密码都是 `admin`，生产环境第一时间改密码！

```sql
-- 配置主库（写组 hostgroup_id=0）
INSERT INTO mysql_servers (hostgroup_id, hostname, port, weight) VALUES
(0, 'mysql-master', 3306, 1);

-- 配置从库（读组 hostgroup_id=1）
INSERT INTO mysql_servers (hostgroup_id, hostname, port, weight) VALUES
(1, 'mysql-slave', 3306, 1);

-- 查看配置
SELECT * FROM mysql_servers;
```

**hostgroup_id 约定**：
- `0` = 写组（所有非 SELECT 走这里）
- `1` = 读组（SELECT 走这里）

### 3.3 配置 MySQL 用户和监控

```sql
-- 配置后端 MySQL 的登录用户（和主库的复制用户不同，这是应用连接用的）
INSERT INTO mysql_users (username, password, default_hostgroup) VALUES
('root', 'root123', 0);
-- default_hostgroup=0 表示默认走写组

-- 配置监控用户（ProxySQL 用它来检查后端 MySQL 是否健康）
SET mysql-monitor_username='root';
SET mysql-monitor_password='root123';

-- 加载配置到运行时
LOAD MYSQL SERVERS TO RUNTIME;
LOAD MYSQL USERS TO RUNTIME;
LOAD MYSQL VARIABLES TO RUNTIME;

-- 保存到磁盘（否则重启 ProxySQL 配置丢失）
SAVE MYSQL SERVERS TO DISK;
SAVE MYSQL USERS TO DISK;
SAVE MYSQL VARIABLES TO DISK;
```

### 3.4 配置读写分离规则

这是最关键的一步——告诉 ProxySQL "什么 SQL 走哪条路"。

```sql
-- 把 SELECT 请求路由到读组（hostgroup_id=1）
INSERT INTO mysql_query_rules (rule_id, active, match_pattern, destination_hostgroup, apply) VALUES
(1, 1, '^SELECT.*', 1, 1);

-- 把 SELECT ... FOR UPDATE 强制路由到写组（因为这是带锁的查询，必须走主库）
INSERT INTO mysql_query_rules (rule_id, active, match_pattern, destination_hostgroup, apply) VALUES
(2, 1, '^SELECT.*FOR UPDATE', 0, 1);

-- 加载规则
LOAD MYSQL QUERY RULES TO RUNTIME;
SAVE MYSQL QUERY RULES TO DISK;
```

**规则匹配逻辑**：ProxySQL 按 `rule_id` 从小到大匹配，匹配到第一个就停止。所以 `SELECT ... FOR UPDATE` 的规则要放在普通 `SELECT` 前面！

### 3.5 验证读写分离

现在通过 ProxySQL 的 6033 端口连接数据库：

```bash
# 通过 ProxySQL 连接（端口 6033）
mysql -u root -proot123 -h 127.0.0.1 -P 6033
```

```sql
-- 执行一个写操作
INSERT INTO my_shop.test_rw (name) VALUES ('via proxysql');

-- 执行一个读操作
SELECT * FROM my_shop.test_rw;
```

查看 ProxySQL 的统计信息，确认路由正确：

```bash
mysql -u admin -padmin -h 127.0.0.1 -P 6032
```

```sql
-- 查看每个 hostgroup 处理了多少查询
SELECT hostgroup, srv_host, srv_port, Queries
FROM stats_mysql_connection_pool;

-- 应该看到：
-- hostgroup=0（写组）收到了 INSERT
-- hostgroup=1（读组）收到了 SELECT
```

## 四、主从延迟问题：读不到刚写入的数据

这是读写分离最大的坑，必须认真对待。

### 4.1 场景复现

```
1. 用户注册 → INSERT INTO users ... → 主库执行成功
2. 跳转到个人主页 → SELECT * FROM users WHERE id = 123 → 路由到从库
3. 从库还没同步这条数据 → 查不到！→ 用户看到"用户不存在" 🤯
```

这就是 **主从延迟** *此术语见附录E*：主库写完了，从库还没同步过来。

### 4.2 解决方案

**方案一：关键读操作强制走主库**

```sql
-- 在 SQL 前加注释，ProxySQL 识别后路由到主库
/* master */ SELECT * FROM users WHERE id = 123;
```

在 ProxySQL 中配置规则识别这个注释：

```sql
INSERT INTO mysql_query_rules (rule_id, active, match_pattern, destination_hostgroup, apply) VALUES
(3, 1, '.*\/\* master \*\/.*', 0, 1);
```

**方案二：写后立即读，强制走主库（应用层）**

```python
# 伪代码：注册后立即查用户信息，走主库
user_id = db.execute_write("INSERT INTO users ...")
user = db.execute_read_on_master("SELECT * FROM users WHERE id = ?", user_id)
```

**方案三：等待从库同步（业务侧）**

```python
# 注册后等 100ms 再查，给从库同步时间
user_id = db.execute("INSERT INTO users ...")
time.sleep(0.1)
user = db.execute("SELECT * FROM users WHERE id = ?", user_id)
```

- ✅ 简单
- ❌ 不精确，100ms 够不够不确定
- ❌ 用户体验差（多等 100ms）

**方案四：缓存刚写入的数据**

```python
# 写操作后把数据也放到 Redis，读的时候先查 Redis
user_id = db.execute("INSERT INTO users ...")
cache.set(f"user:{user_id}", user_data, expire=5)  # 缓存 5 秒
# 读的时候先查缓存，没有再从库查
```

**推荐组合**：方案一 + 方案二。大部分场景用方案二（写后立即读走主库），特殊场景用方案一（注释标记）。

## 五、MySQL Router 对比 ProxySQL

### 5.1 MySQL Router 是什么？

MySQL Router 是 MySQL 官方出品的轻量级路由中间件，配合 InnoDB Cluster 使用效果最好。

**对比 ProxySQL：**

| 对比项 | ProxySQL | MySQL Router |
|--------|----------|-------------|
| 开发者 | 第三方（ProxySQL） | MySQL 官方 |
| 功能 | 丰富（查询缓存、SQL 防火墙、读写分离） | 轻量（路由 + 故障切换） |
| 配置复杂度 | 较高 | 较低 |
| 查询缓存 | ✅ 支持 | ❌ 不支持 |
| SQL 改写 | ✅ 支持 | ❌ 不支持 |
| 社区活跃度 | 很活跃 | 官方维护 |
| 推荐场景 | 需要高级功能 | 配合 InnoDB Cluster |

**一句话**：简单用 MySQL Router，功能多要灵活用 ProxySQL。

### 5.2 负载均衡策略

多个从库时，ProxySQL 怎么分配读请求？

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| 轮询（Round Robin） | 依次分配 | 从库性能一致 |
| 权重（Weight） | 按权重比例分配 | 从库性能不一致 |
| 最少连接（Least Connections） | 发到连接最少的从库 | 长连接场景 |
| 最快响应 | 发到响应最快的从库 | 追求低延迟 |

ProxySQL 默认按权重轮询，你可以通过 `weight` 参数调整：

```sql
-- 从库 A 性能好，权重 10
INSERT INTO mysql_servers (hostgroup_id, hostname, port, weight) VALUES
(1, 'slave-a', 3306, 10);

-- 从库 B 性能差，权重 3
INSERT INTO mysql_servers (hostgroup_id, hostname, port, weight) VALUES
(1, 'slave-b', 3306, 3);
-- 10 个请求里，约 7 个发到 A，3 个发到 B
```

## 六、健康检查与故障切换

### 6.1 ProxySQL 自动健康检查

ProxySQL 会定期 ping 后端 MySQL，如果发现挂了，自动从路由表里移除：

```sql
-- 查看监控配置
SELECT * FROM global_variables WHERE variable_name LIKE 'mysql-monitor%';
```

关键参数：

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `mysql-monitor_ping_interval` | 60000ms | 每 60 秒 ping 一次 |
| `mysql-monitor_ping_timeout` | 1000ms | ping 超时时间 |
| `mysql-monitor_connect_interval` | 60000ms | 连接检查间隔 |

### 6.2 手动故障切换

如果主库挂了，需要手动把从库提升为主库，然后更新 ProxySQL 配置：

```sql
-- ProxySQL 管理端口
-- 把旧主库标记为 OFFLINE
UPDATE mysql_servers SET status='OFFLINE_HARD' WHERE hostname='mysql-master';

-- 把新主库（原从库）移到写组
UPDATE mysql_servers SET hostgroup_id=0 WHERE hostname='mysql-slave';

-- 加载生效
LOAD MYSQL SERVERS TO RUNTIME;
SAVE MYSQL SERVERS TO DISK;
```

## 我做得对不对？

### 验证方法

```bash
# 1. 确认 ProxySQL 运行中
mysql -u admin -padmin -h 127.0.0.1 -P 6032 -e "SELECT * FROM runtime_mysql_servers;"
# 应该看到主库（hostgroup=0）和从库（hostgroup=1）

# 2. 通过 ProxySQL 执行写操作
mysql -u root -proot123 -h 127.0.0.1 -P 6033 my_shop -e "
  CREATE TABLE IF NOT EXISTS rw_test (id INT AUTO_INCREMENT PRIMARY KEY, msg VARCHAR(50));
  INSERT INTO rw_test (msg) VALUES ('hello from proxy');
"

# 3. 直接连主库和从库，确认数据是否一致
docker exec mysql-master mysql -u root -proot123 my_shop -e "SELECT * FROM rw_test;"
docker exec mysql-slave mysql -u root -proot123 my_shop -e "SELECT * FROM rw_test;"
# 两个结果一致 → ✅

# 4. 检查 ProxySQL 统计，确认 INSERT 走了写组
mysql -u admin -padmin -h 127.0.0.1 -P 6032 -e "
  SELECT hostgroup, srv_host, Queries FROM stats_mysql_connection_pool;
"
# hostgroup=0 的 Queries 增加了 → INSERT 走了写组 ✅
```

## 不对怎么办？

### 常见错误 1：所有请求都走主库（读写分离没生效）

现象：检查 ProxySQL 统计，所有 Queries 都在 hostgroup=0（写组）。

排查：

```sql
-- 检查规则是否加载
SELECT * FROM runtime_mysql_query_rules;

-- 检查规则是否匹配
SELECT * FROM stats_mysql_query_rules;
-- 看 Hits 字段，如果都是 0 说明规则没匹配到
```

原因：`match_pattern` 写错了，或者规则没加载到 RUNTIME。

✅ 解决：重新设置规则，确保 `LOAD MYSQL QUERY RULES TO RUNTIME;`。

### 常见错误 2：主从延迟导致数据不一致

现象：刚插入的数据，马上查不到。

解决：见本章第四节，四种方案选一个。**推荐"写后立即读走主库"**。

### 常见错误 3：ProxySQL 配置忘记保存，重启后丢失

现象：ProxySQL 重启后，所有配置都没了。

✅ 解决：每次改完配置，都要执行 `SAVE ... TO DISK;`。

```sql
LOAD MYSQL SERVERS TO RUNTIME;
SAVE MYSQL SERVERS TO DISK;   -- 别忘了这行！
```

### 常见错误 4：`SELECT ... FOR UPDATE` 走了从库

现象：带 `FOR UPDATE` 的查询在从库执行，锁不住数据。

✅ 解决：在读写分离规则中，把 `SELECT.*FOR UPDATE` 的规则排在普通 `SELECT` 前面，rule_id 更小。

## 最可能出错的地方及原因

1. **读写分离规则写了但没加载**：只执行了 INSERT 没执行 `LOAD ... TO RUNTIME`，原因是不了解 ProxySQL 的多层配置模型（内存 → RUNTIME → DISK）。
2. **主从延迟导致"查不到刚写入的数据"**：注册后立即跳转个人页，从库还没同步，原因是没考虑复制延迟。
3. **`SELECT FOR UPDATE` 走了从库**：规则匹配顺序错误，原因是没理解 ProxySQL 按 rule_id 从小到大匹配。
4. **ProxySQL 重启后配置丢失**：只加载到 RUNTIME 没 SAVE 到 DISK，原因是不了解 ProxySQL 的持久化机制。
5. **所有请求都路由到主库**：规则不匹配或没配置，原因是正则表达式写错或没配置读组。
