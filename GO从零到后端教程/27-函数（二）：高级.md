# 第27章 · 函数（二）：高级

> "普通函数像自动售货机——投币、选货、出货，每次都一样。但闭包像一个有记忆的售货机——它会记住上次你买了什么、还剩多少库存。defer则像一个负责任的管家——不管你今天过得怎么样，睡前一定会锁门。"

## 27.1 可变参数：来多少接多少

有时候你不知道调用者会传几个参数。比如写一个求和函数，你希望 `sum(1, 2)` 和 `sum(1, 2, 3, 4, 5)` 都能用。

Go用 `...` 来表示可变参数：

```go
func sum(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

fmt.Println(sum(1, 2))
fmt.Println(sum(1, 2, 3, 4, 5))
fmt.Println(sum())
```

输出：
```
3
15
0
```

`nums ...int` 的意思是：`nums` 是一个 `[]int` 类型的切片，调用者可以传任意个 `int` 类型的参数（包括0个）。在函数内部，`nums` 就是一个普通的切片，你可以用 `len()`、`range`、`append` 等所有切片操作。

### 比喻：餐厅的自助餐

普通参数就像套餐——三道菜，多了不要少了不行。可变参数就像自助餐——你拿5盘也行，拿10盘也行，一盘不拿也行（空切片）。

### 把切片传给可变参数

如果你已经有一个切片，想把它展开传给可变参数函数：

```go
scores := []int{85, 92, 78, 95}
total := sum(scores...)
```

`scores...` 把切片"展开"成一个个独立的参数。这就像把打包好的饭盒一个个拿出来摆在桌上。

### 可变参数必须是最后一个

```go
func greet(prefix string, names ...string) {
    for _, name := range names {
        fmt.Println(prefix, name)
    }
}

greet("你好，", "张三", "李四", "王五")
```

输出：
```
你好， 张三
你好， 李四
你好， 王五
```

`...string` 必须放在最后——Go编译器需要知道前面的参数到哪里结束，可变参数从哪里开始。把可变参数放在最后一个位置，编译器就很清楚：前面的是普通参数，后面全部是可变参数。

### fmt.Println的秘密

现在你应该能理解 `fmt.Println` 的签名了：

```go
func Println(a ...interface{}) (n int, err error)
```

它接收任意类型、任意数量的参数——你用 `Println("hello", 42, true, 3.14)` 它全都能接住。这就是可变参数 + 空接口的组合威力。

## 27.2 匿名函数：没名字的函数

通常函数都有名字——`sum`、`greet`、`main`。但有时候你只需要"临时用一下"，没必要给它起个名字。

匿名函数就是"即时贴"——写完就用，不存档：

```go
func main() {
    greeting := func(name string) string {
        return "你好，" + name
    }

    fmt.Println(greeting("张三"))
}
```

`func(name string) string { ... }` 就是一个匿名函数。它没有函数名，直接赋值给变量 `greeting`。然后你可以通过 `greeting("张三")` 来调用它。

### 定义即调用

更常见的写法是——定义的同时立刻调用：

```go
func main() {
    message := func() string {
        return "这个函数只活一次"
    }()

    fmt.Println(message)
}
```

注意最后的 `()`——这就是调用。函数定义完立刻执行，返回值赋给 `message`。

**比喻**：普通函数是你常备的厨具——每次炒菜都用，值得给它一个固定位置（名字）。匿名函数是一次性筷子——用完就扔，不需要给它个名字。

### 匿名函数做参数

匿名函数最常见的用途是作为另一个函数的参数：

```go
func operate(a, b int, op func(int, int) int) int {
    return op(a, b)
}

func main() {
    add := operate(3, 4, func(x, y int) int {
        return x + y
    })
    fmt.Println("3 + 4 =", add)

    multiply := operate(3, 4, func(x, y int) int {
        return x * y
    })
    fmt.Println("3 × 4 =", multiply)
}
```

`operate` 接收一个"操作函数"作为参数。你可以把加法逻辑传进去，也可以把乘法逻辑传进去。这就像去干洗店——你把衣服给店员（参数a和b），但洗涤方式（操作函数op）可以每次不同：干洗、水洗、熨烫。

## 27.3 闭包（Closure）：函数 + 记忆

这是本章的**重中之重**。闭包是Go面试必考题，也是理解很多高级模式的基础。

### 什么是闭包

**闭包（函数能"记住"并访问其外部变量，即使外部函数已返回，详见附录I） = 函数 + 它引用的外部变量**。

听起来很抽象？我们用一个具体例子。

### Counter：一个会自己计数的函数

```go
func newCounter() func() int {
    count := 0
    return func() int {
        count++
        return count
    }
}

func main() {
    counter := newCounter()

    fmt.Println(counter())
    fmt.Println(counter())
    fmt.Println(counter())
}
```

输出：
```
1
2
3
```

`newCounter` 返回了一个匿名函数。这个匿名函数引用了 `newCounter` 内部的局部变量 `count`。

正常情况下，函数执行完，局部变量就消失了。但闭包是个例外：匿名函数"抓着" `count` 不放，`count` 的生命周期被延长了。每次调用 `counter()`，它操作的是**同一个** `count`，所以数字在增长。

### 比喻：自动编号机

你去银行取号，有一个机器不停地吐出递增的号码：1、2、3... 这个机器就是一个闭包。它的"记忆"就是上次吐到第几号了。

`newCounter` 就像工厂——生产一个自动编号机。你调用一次 `newCounter()`，得到一个编号机。每个编号机都有自己的独立计数，互不干扰：

```go
counter1 := newCounter()
counter2 := newCounter()

fmt.Println(counter1())
fmt.Println(counter1())
fmt.Println(counter2())
fmt.Println(counter1())
```

输出：
```
1
2
1
3
```

`counter1` 的计数到了3，`counter2` 的计数还在1——它们是两台独立的机器，各有各的记忆。

### 闭包的实际用途

**1. 生成不同行为的函数**

```go
func makeMultiplier(factor int) func(int) int {
    return func(n int) int {
        return n * factor
    }
}

double := makeMultiplier(2)
triple := makeMultiplier(3)

fmt.Println(double(5))
fmt.Println(triple(5))
```

`double` 和 `triple` 是两个不同的函数，一个记住了 `factor=2`，一个记住了 `factor=3`。

**2. 实现中间件模式**

你在后续的Gin框架中会看到大量这样的代码：

```go
func loggingMiddleware(handler func()) func() {
    return func() {
        fmt.Println("开始执行")
        handler()
        fmt.Println("执行完毕")
    }
}
```

### 闭包的陷阱：循环里的闭包

这是Go面试中最高频的陷阱题：

```go
func main() {
    var funcs []func()

    for i := 0; i < 3; i++ {
        funcs = append(funcs, func() {
            fmt.Println(i)
        })
    }

    for _, f := range funcs {
        f()
    }
}
```

输出是什么？你可能会猜：
```
0
1
2
```

实际输出是：
```
3
3
3
```

**为什么？** 三个闭包都引用了同一个变量 `i`。当循环结束时，`i` 变成了3。然后调用三个闭包，它们打印的都是这个已经变成3的 `i`。

**本质**：闭包记住的是**变量本身**，不是变量**当时的值**。就像你电话号码簿里存的不是"去年他的电话号码"，而是指向他当前号码的引用——他换号了，你查到的是新号。

### 修复方法

**方法一：在循环里创建局部副本**

```go
for i := 0; i < 3; i++ {
    i := i
    funcs = append(funcs, func() {
        fmt.Println(i)
    })
}
```

`i := i` 是一个经典Go写法：在循环体内部用 `:=` 声明一个新的局部变量 `i`，把外层 `i` 的值拷贝过来。每个闭包"抓住"的是自己那个独立的局部 `i`，互不干扰。

**方法二：通过参数传入**

```go
for i := 0; i < 3; i++ {
    val := i
    funcs = append(funcs, func() {
        fmt.Println(val)
    })
}
```

原理相同：让每个闭包引用不同的变量。

## 27.4 defer：延迟执行

`defer`（延迟执行，把指定操作推迟到函数返回前执行，详见附录I）是Go里非常有特色的关键字。它的意思是：**"把这件事推迟到函数返回之前再做"**。

### 基本用法

```go
func readFile() {
    fmt.Println("打开文件")
    defer fmt.Println("关闭文件")
    fmt.Println("读取文件内容")
    fmt.Println("处理数据")
}
```

输出：
```
打开文件
读取文件内容
处理数据
关闭文件
```

`defer fmt.Println("关闭文件")` 被推迟到了函数末尾执行。

### 比喻：离场关灯

你进一个房间开会：开门 → 拉窗帘 → 讨论 → 写白板 → 散会。但你一进门就贴了一张纸条："离开时关灯"。

defer就是这张纸条——**你提前写好"离开时要做什么"，然后放心做其他事，最后离开时自动执行**。

### defer的三个经典用途

**1. 关闭文件**

```go
func readConfig(path string) (string, error) {
    file, err := os.Open(path)
    if err != nil {
        return "", err
    }
    defer file.Close()

    data, err := io.ReadAll(file)  // io.ReadAll 文件操作详见第38章
    if err != nil {
        return "", err
    }
    return string(data), nil
}
```

打开文件后立刻 `defer file.Close()`——不管后面是正常读取还是中途出错返回，文件都会被关闭。

**2. 释放锁**

```go
func updateBalance(accountID int, amount float64) {
    mu.Lock()
    defer mu.Unlock()  // mu 是 sync.Mutex 类型，同步原语详见第36章

    balance := loadBalance(accountID)
    balance += amount
    saveBalance(accountID, balance)
}
```

锁住之后立刻 `defer mu.Unlock()`——即使 `loadBalance` 或 `saveBalance` panic了，锁也会被释放。

**3. 记录函数执行时间**

```go
func slowOperation() {
    start := time.Now()
    defer func() {
        fmt.Println("耗时:", time.Since(start))
    }()

    time.Sleep(2 * time.Second)
}
```

### defer的执行顺序：LIFO（后进先出）

如果有多个defer，它们按照"后进先出"的顺序执行——像叠盘子：最后放上去的，最先拿下来。

```go
func demo() {
    defer fmt.Println("第1个defer")
    defer fmt.Println("第2个defer")
    defer fmt.Println("第3个defer")
    fmt.Println("函数体")
}
```

输出：
```
函数体
第3个defer
第2个defer
第1个defer
```

为什么是LIFO？因为defer本质上是一个**栈**。每遇到一个defer语句，就把要执行的函数压入栈中。函数返回时，从栈顶逐个弹出执行。

### defer的关键细节：参数在声明时就确定了

```go
func demo() {
    i := 1
    defer fmt.Println("defer里i的值:", i)
    i = 100
    fmt.Println("函数体里i的值:", i)
}
```

输出：
```
函数体里i的值: 100
defer里i的值: 1
```

`defer fmt.Println("defer里i的值:", i)` 这句话**执行defer声明时**，`i` 是1，所以defer"记住"了参数值是1。虽然后面 `i` 变成了100，但defer不管——它在声明那一刻就把参数的值拍下来了。

**但如果你传的是指针或闭包呢？**

```go
func demo() {
    i := 1
    defer func() {
        fmt.Println("defer里i的值:", i)
    }()
    i = 100
}
```

输出：
```
defer里i的值: 100
```

因为匿名函数通过闭包捕获的是**变量本身**，不是变量的值的快照。defer执行时 `i` 已经是100了。

## 27.5 panic和recover

### panic：程序崩溃

`panic`（程序主动触发崩溃，打印调用栈后停止运行，详见附录I）让程序立即停止运行，并打印调用栈：

```go
func main() {
    fmt.Println("开始")
    panic("出大事了！")
    fmt.Println("结束")
}
```

输出：
```
开始
panic: 出大事了！

goroutine 1 [running]:
main.main()
        /path/to/main.go:7 +0x...
```

`panic` 之后的代码都不会执行。这就像开车时突然爆胎——车立刻失控，后面的路程（代码）全废了。

### recover：救回来

`recover`（在defer中捕获panic，让程序从崩溃中恢复，详见附录I）只能在 `defer` 函数中使用，用来"接住"panic：

```go
func safeDivide(a, b int) {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("被recover救回来了:", r)
        }
    }()

    if b == 0 {
        panic("除数不能为0")
    }
    fmt.Println("结果:", a/b)
}

func main() {
    safeDivide(10, 0)
    fmt.Println("程序继续运行")
}
```

输出：
```
被recover救回来了: 除数不能为0
程序继续运行
```

程序没有崩溃，`main` 最后一行正常执行了。

### 比喻：安全气囊

`panic` 是撞车，`recover` 是安全气囊。撞车（panic）本身是坏事，但安全气囊（recover）能让你活下来。但你不应该因为有安全气囊就故意撞车——同理，`panic/recover` 不应该被用作常规的错误处理。

### 重要：panic不是异常处理！

很多从Java/Python转来的程序员习惯用异常来处理错误。Go里**不应该**这么用。

Go的正确做法是：**用error返回值处理可预期的错误，用panic处理不可恢复的严重错误**。

| 场景 | 处理方式 |
|------|---------|
| 文件不存在 | 返回error |
| 网络超时 | 返回error |
| 用户输入不合法 | 返回error |
| 数组越界 | 这是bug，应该panic |
| 不能accept端口 | 程序无法继续，panic |
| 数据库配置缺失 | 启动时panic |

### panic/recover的正确使用场景

**1. 程序启动时的致命错误**

```go
func main() {
    config, err := loadConfig()
    if err != nil {
        panic("无法加载配置: " + err.Error())
    }
}
```

没有配置程序完全没法跑，不如直接panic。

**2. 库内部处理不可恢复错误，但对外暴露为error**

少数场景下，库的内部实现可能用 `recover` 把panic转成error返回，避免库的使用者因为一个bug而整个程序崩溃。但这属于防御性编程，不是推荐日常使用。

**3. 保证程序不崩溃**

Web框架（比如Gin）内置了 `recover` 中间件——即使用户的某个请求处理panic了，服务器也不会整个挂掉，而是返回500并继续服务其他请求。
