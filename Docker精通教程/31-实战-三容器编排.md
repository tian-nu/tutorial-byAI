## 我在做什么？

前面两章你容器化了单个应用：Node.js 和 Python 各一个。但真实的项目很少只有一个服务——通常是**前端（Nginx） + 后端 API（Node.js） + 数据库（MySQL）**的组合。

这一章，我们搭一个完整的三层架构。你会学到 Docker Compose 真正强大的地方：**一次命令，三个服务全部启动，网络自动配置，健康检查保证启动顺序，数据卷持久化数据库**。你还会在 Nginx 里配置反向代理*此术语见附录C*——让前端请求先经过 Nginx，Nginx 把 API 请求转发给后端，把静态文件请求自己处理。

学完这一章，你能：
- 写一个三服务的 compose.yml（Nginx + Node.js + MySQL）
- 配置 Nginx 反向代理，区分静态文件和 API 请求
- 用 healthcheck 确保 MySQL 就绪后才启动后端
- 理解"容器名代替 localhost"的核心原则
- 在浏览器里走通"前端 → API → 数据库"全链路

## 一、整体架构

```
浏览器
  │
  ▼
┌──────────────────────────────────────────────────┐
│  Docker Compose 网络 (app-network)                │
│                                                    │
│  ┌──────────┐    proxy_pass     ┌──────────┐     │
│  │  Nginx   │ ───────────────→  │  Node.js │     │
│  │  :80     │                   │  :3000   │     │
│  │ 静态文件  │                   │  API     │     │
│  └──────────┘                   └────┬─────┘     │
│       │                              │            │
│       │  /api/* 转发给后端             │ 连接数据库  │
│       │                              │            │
│       │                         ┌────▼─────┐     │
│       │                         │  MySQL   │     │
│       │                         │  :3306   │     │
│       │                         └────┬─────┘     │
│       │                              │            │
│       │                         ┌────▼─────┐     │
│       │                         │  mysql-  │     │
│       │                         │  data    │     │
│       │                         │ (volume) │     │
│       │                         └──────────┘     │
└──────────────────────────────────────────────────┘
```

三条关键路径：
1. **静态文件**：浏览器 → Nginx:80 → 直接返回 HTML/CSS/JS
2. **API 请求**：浏览器 → Nginx:80 → 反向代理到 Node.js:3000 → 处理业务逻辑
3. **数据库**：Node.js:3000 → MySQL:3306 → 读写数据，数据持久化到 volume

## 三、后端代码（Node.js + Express）

### backend/package.json

```json
{
  "name": "three-tier-backend",
  "version": "1.0.0",
  "main": "src/index.js",
  "scripts": {
    "start": "node src/index.js"
  },
  "dependencies": {
    "express": "^4.18.2",
    "mysql2": "^3.6.5"
  }
}
```

### backend/src/index.js

```javascript
const express = require('express');
const mysql = require('mysql2/promise');
const app = express();
const PORT = process.env.PORT || 3000;

// 数据库连接池
const pool = mysql.createPool({
  host: process.env.DB_HOST || 'localhost',
  port: process.env.DB_PORT || 3306,
  user: process.env.DB_USER || 'appuser',
  password: process.env.DB_PASSWORD || 'your_db_password_here',
  database: process.env.DB_NAME || 'appdb',
  waitForConnections: true,
  connectionLimit: 10,
});

// 健康检查
app.get('/health', async (req, res) => {
  try {
    await pool.query('SELECT 1');
    res.json({ status: 'OK', database: 'connected' });
  } catch (err) {
    res.status(500).json({ status: 'ERROR', database: 'disconnected', error: err.message });
  }
});

// 获取用户列表
app.get('/api/users', async (req, res) => {
  try {
    const [rows] = await pool.query('SELECT id, name, email FROM users');
    res.json({ users: rows });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

// 根路径
app.get('/api', (req, res) => {
  res.json({ message: 'Backend API is running', env: process.env.NODE_ENV || 'development' });
});

app.listen(PORT, () => {
  console.log(`Backend server running on port ${PORT}`);
});
```

**关键点**：数据库连接的主机名是 `process.env.DB_HOST`，默认值是 `'localhost'`。在容器里，这个值会通过 Compose 环境变量设为 `db`（MySQL 服务名）。这是**容器名代替 localhost** 的核心体现。

### backend/Dockerfile

```dockerfile
FROM node:18-alpine

WORKDIR /app

COPY package.json .

RUN npm install --production

COPY src/ ./src/

EXPOSE 3000

CMD ["node", "src/index.js"]
```

```html
        el.textContent = JSON.stringify(data, null, 2);

      const el = document.getElementById('users-result');
      el.textContent = '请求中...';

        const data = await res.json();

```

这个页面做了两件事：调用 `/api/health` 和 `/api/users`。注意：前端的 fetch URL 是 `/api/health`，不是 `http://localhost:3000/health`。这是 Nginx 反向代理发挥作用的地方——浏览器请求的是 Nginx，Nginx 把 `/api/*` 转发给后端。

### frontend/nginx.conf — 反向代理的核心

```nginx
server {
    listen 80;
    server_name localhost;

    # 静态文件：Nginx 自己处理
    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    # API 请求：转发给后端 Node.js 容器
    location /api/ {
        proxy_pass http://backend:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**这就是反向代理的魔法**：

```
浏览器请求: GET /api/users
    │
    ▼
Nginx (端口 80)
    │ 匹配 location /api/
    │ 规则：proxy_pass http://backend:3000
    ▼
Node.js (端口 3000)
    │ 处理 /api/users
    │ 查询 MySQL
    ▼
返回 JSON → Nginx → 浏览器
```

### ❌ 常见错误：`proxy_pass` 地址写错

```nginx
# ❌ 错误 1：写成 localhost
location /api/ {
    proxy_pass http://localhost:3000;   # localhost 是 Nginx 容器自己，不是 Node.js
}

# ❌ 错误 2：写成 127.0.0.1
location /api/ {
    proxy_pass http://127.0.0.1:3000;   # 同上
}

# ❌ 错误 3：写成宿主机地址
location /api/ {
    proxy_pass http://192.168.1.100:3000;  # 你本机 IP 在容器里不一定能访问
}

# ✅ 正确：用 Compose 服务名
location /api/ {
    proxy_pass http://backend:3000;   # backend 是 Compose 里 Node.js 的服务名
}
```

**核心原则**：在 Compose 网络里，**服务名就是主机名**。Nginx 要访问 Node.js，就写 `http://backend:3000`；Node.js 要访问 MySQL，就写 `host: 'db'`。

### frontend/Dockerfile

```dockerfile
FROM nginx:alpine

# 删除默认配置
RUN rm /etc/nginx/conf.d/default.conf

# 用我们的配置
COPY nginx.conf /etc/nginx/conf.d/default.conf

# 拷贝静态文件
COPY html/ /usr/share/nginx/html/

```

## 六、compose.yml — 三服务编排

```yaml
# compose.yml — 三层架构：Nginx + Node.js + MySQL
services:
  # ===== 1. 前端：Nginx 静态文件 + 反向代理 =====
  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    ports:
      - "8080:80"
    depends_on:
      - backend
    restart: unless-stopped
    networks:
      - app-network

  # ===== 2. 后端：Node.js API =====
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    environment:
      NODE_ENV: production
      DB_HOST: db
      DB_PORT: "3306"
      DB_USER: appuser
      DB_PASSWORD: your_db_password_here
      DB_NAME: appdb
    depends_on:
      db:
        condition: service_healthy
    restart: unless-stopped
    networks:
      - app-network

  # ===== 3. 数据库：MySQL =====
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: your_root_password_here
      MYSQL_DATABASE: appdb
      MYSQL_USER: appuser
      MYSQL_PASSWORD: your_db_password_here
    volumes:
      - mysql-data:/var/lib/mysql
      - ./db/init.sql:/docker-entrypoint-initdb.d/init.sql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$$MYSQL_ROOT_PASSWORD"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    restart: unless-stopped
    networks:
      - app-network

# ===== 数据卷 =====
volumes:
  mysql-data:

# ===== 自定义网络 =====
networks:
  app-network:
    driver: bridge
```

### 逐段解析

**frontend（Nginx）**：
- `build: ./frontend` — 用 frontend 目录下的 Dockerfile 构建
- `ports: "8080:80"` — 宿主机 8080 映射到容器 80
- `depends_on: backend` — 先等 backend 启动（但不等它 ready，因为没有 `condition: service_healthy`）
- 对于 Nginx 不需要等后端 ready——Nginx 自带重试，后端没起来时返回 502，起来后自动恢复

**backend（Node.js）**：
- `DB_HOST: db` — **这是关键**：用服务名 `db` 而不是 `localhost` 连接 MySQL
- `depends_on: db: condition: service_healthy` — **必须等 MySQL 健康检查通过**才启动
- MySQL 需要 20-30 秒初始化，如果没有 `condition: service_healthy`，后端会在 MySQL 还没准备好时尝试连接，导致 `ECONNREFUSED`

**db（MySQL）**：
- `healthcheck` — 用 `mysqladmin ping` 检测 MySQL 是否真正可用
- `$$MYSQL_ROOT_PASSWORD` — 两个 `$` 是 YAML 转义，最终传给容器的是 `$MYSQL_ROOT_PASSWORD`
- `start_period: 30s` — 给 MySQL 首次启动 30 秒初始化时间，期间不检查健康状态
- `init.sql` 挂载到 `/docker-entrypoint-initdb.d/` — MySQL 首次启动自动执行

### 为什么 healthcheck 里是 `$$`？

在 compose.yml 中，`$` 是 YAML 的变量字符。`$MYSQL_ROOT_PASSWORD` 会被 YAML 解析成环境变量插值。但 healthcheck 的 test 命令里，我们需要的是**字面量** `$MYSQL_ROOT_PASSWORD`（让 MySQL 容器里的 shell 去解析），不是 Compose 去解析。两个 `$` 就是告诉 YAML："别解析，给我一个 `$`"。

```yaml
# ❌ 一个 $ → YAML 尝试解析为变量，可能为空
test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$MYSQL_ROOT_PASSWORD"]

# ✅ 两个 $$ → YAML 输出一个 $，容器内 shell 正确解析
test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$$MYSQL_ROOT_PASSWORD"]
```

## 九、不对怎么办？

### 常见错误 1：后端用 `localhost` 连接 MySQL

```javascript
// ❌ 错误
const pool = mysql.createPool({
  host: 'localhost',   // 或 '127.0.0.1'
  // ...
});
```

**这是多容器编排中最常见的错误。** 在 Compose 网络中，每个容器有自己的 `localhost`。Node.js 容器里的 `localhost` 指向 Node.js 容器自己——它上面没有 MySQL。

```
Node.js 容器                           MySQL 容器
┌─────────────────────┐               ┌─────────────────────┐
│ localhost → Node.js │               │ localhost → MySQL   │
│ db → MySQL 容器      │               │ db → MySQL 容器     │
└─────────────────────┘               └─────────────────────┘
```

```javascript
// ✅ 正确：用服务名
const pool = mysql.createPool({
  host: process.env.DB_HOST || 'db',   // 通过环境变量传入，默认 'db'
  // ...
});
```

**排查**：如果怀疑网络不通，进入后端容器 ping 一下：

```bash
docker compose exec backend ping db
# 预期：能 ping 通，说明 Compose DNS 正常工作
```

### 常见错误 2：`proxy_pass` 地址写错

```nginx
# ❌ proxy_pass 写了 localhost 或 127.0.0.1
location /api/ {
    proxy_pass http://localhost:3000;
}
```

`localhost` 在 Nginx 容器里是 Nginx 自己，不是 Node.js 后端。

```nginx
# ✅ 用 Compose 服务名
location /api/ {
    proxy_pass http://backend:3000;
}
```

**排查**：

```bash
# 进入 Nginx 容器，测试能否访问后端
docker compose exec frontend sh -c "wget -qO- http://backend:3000/api/health"
# 预期：{"status":"OK","database":"connected"}
```

### 常见错误 3：MySQL 还没初始化完，后端就尝试连接

```bash
docker compose logs backend
# Error: connect ECONNREFUSED 172.20.0.3:3306
```

❌ 原因：`depends_on` 没有加 `condition: service_healthy`，或者 `condition` 加了但 healthcheck 没配。

```yaml
# ❌ 只等容器启动，不等 MySQL 准备好
backend:
  depends_on:
    - db
```

```yaml
# ✅ 等 MySQL 真正可用
backend:
  depends_on:
    db:
      condition: service_healthy
```

而且 `db` 服务必须配 healthcheck：

```yaml
db:
  healthcheck:
    test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$$MYSQL_ROOT_PASSWORD"]
    interval: 10s
    timeout: 5s
    retries: 5
    start_period: 30s
```

**如果已经配了 healthcheck 还是失败**：检查 `start_period` 是否够长。MySQL 8.0 首次启动需要 20-30 秒，如果 `start_period` 太短，healthcheck 可能在 MySQL 初始化期间就开始检测，多次失败后容器被标记为 unhealthy。

### 常见错误 4：healthcheck 的 `$$` 写成了 `$`

```yaml
# ❌ 一个 $ — YAML 尝试解析 $MYSQL_ROOT_PASSWORD 为空
test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$MYSQL_ROOT_PASSWORD"]
```

```yaml
# ✅ 两个 $$ — YAML 输出一个字面量 $
test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$$MYSQL_ROOT_PASSWORD"]
```

### 常见错误 5：端口冲突

```bash
# Error: port is already allocated
```

如果 8080 或 3000 端口被占用，换一个：

```yaml
# compose.yml
```

## 本篇最可能出错的地方及原因

### 1. 后端用 `localhost` 连接 MySQL（最高频）

**这是多容器编排的"头号杀手"。** 在传统开发中，后端和数据库装在同一台机器上，`localhost` 能连上。但在 Docker Compose 里，每个服务是独立的容器，`localhost` 指向容器自己。

**从"传统思维"到"容器思维"的转变**：在 Compose 网络中，Docker 内置了 DNS 服务器。每个服务名（如 `db`、`backend`）自动注册为 DNS 记录。Node.js 容器里 `ping db` 能解析到 MySQL 容器的 IP。所以数据库连接字符串的主机名应该是 Compose 服务名，而不是 `localhost`。

**排查**：`docker compose exec backend sh -c "nc -zv db 3306"`——如果 MySQL 端口可达，说明网络没问题，是代码里 host 写错了。

### 2. Nginx `proxy_pass` 地址写错

和上面同一个根因——在 Nginx 容器里，`localhost` 是 Nginx 自己。`proxy_pass http://localhost:3000` 会让 Nginx 把请求转发给自己，而不是后端。

**正确写法**：`proxy_pass http://backend:3000`。`backend` 是 compose.yml 里后端服务的名字。

### 3. MySQL 健康检查的 `$$` 写成了 `$`

YAML 里 `$` 是特殊字符，会被解析为变量插值。`-p$MYSQL_ROOT_PASSWORD` 会被 YAML 尝试解析环境变量 `MYSQL_ROOT_PASSWORD`——但此时 healthcheck 命令还没进入容器，YAML 解析出来的是空字符串。结果：`mysqladmin ping -p`（空密码）→ MySQL 拒绝连接 → healthcheck 失败 → 后端永远不启动。

**两个 `$`** 是 YAML 的转义语法：`$$` → 输出一个字面量 `$`。容器内的 shell 拿到 `$MYSQL_ROOT_PASSWORD` 后，在容器环境里正确解析。

### 4. MySQL 初始化太慢，healthcheck 在 `start_period` 结束前就失败了

MySQL 8.0 首次启动需要初始化 InnoDB 系统表空间、创建用户和数据库、执行 `init.sql`。这个过程需要 20-30 秒。如果 `start_period` 设为 10s，healthcheck 在 MySQL 初始化期间就开始检测，连续失败后容器被标记为 unhealthy。

**解决**：`start_period: 30s` 或更长。如果 `init.sql` 很大，可能需要更长时间。

### 5. `docker compose down` 后数据丢了

如果没用 volume 持久化，`docker compose down` 删除容器后数据就没了。必须配 `volumes: mysql-data:/var/lib/mysql`。如果已经配了但还是丢了数据，检查是否用了 `docker compose down -v`（`-v` 会删除数据卷）。
