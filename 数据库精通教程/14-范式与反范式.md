## 我在做什么？

第 04 章建表的时候，你凭直觉给 `users` 和 `orders` 分了表——用户信息放 `users`，订单信息放 `orders`，用 `user_id` 关联。这个直觉是对的，但你想过没有：**为什么不能把所有信息塞进一张表里？**

举个例子：如果一张表既存用户名、又存订单号、又存商品名，会有什么问题？答案是：**同一份数据可能被重复存多次，修改一处忘了改另一处，数据就不一致了。**

范式（Normalization）就是回答这个问题的。它是一套"表设计质量标准"，帮你判断一张表设计得是否合理。反范式化则是"故意违反范式以换取性能"——因为有时候，太规范的设计反而让查询慢得像蜗牛。

学完这一章，你能：
- 判断一张表是否符合 1NF、2NF、3NF
- 把一个不合理的表结构逐步规范化到 3NF
- 知道什么时候该反范式化，以及反范式化的代价

## 一、为什么需要范式？—— 一个让你后怕的演示

先看一张"把所有信息塞一起"的表：

```sql
-- ❌ 反例：一张"万能表"，什么都往里塞
CREATE TABLE bad_design (
    order_id    INT,
    user_name   VARCHAR(50),
    user_phone  VARCHAR(20),
    product     VARCHAR(100),
    category    VARCHAR(50),
    price       DECIMAL(10,2),
    created_at  DATETIME
);
```

插入一些数据看看：

```sql
INSERT INTO bad_design VALUES
(1, '张三', '13800001111', 'iPhone 15', '手机', 6999.00, '2024-01-15'),
(2, '张三', '13800001111', 'AirPods',   '耳机', 1299.00, '2024-01-16'),
(3, '李四', '13900002222', 'iPhone 15', '手机', 6999.00, '2024-01-17'),
(4, '李四', '13800003333', 'MacBook',   '电脑', 9999.00, '2024-01-18');
--                                      ↑ 注意：李四改手机号了！
```

发现问题了吗？

1. **数据冗余**：张三的手机号 `13800001111` 存了两次，"iPhone 15" 的价格 `6999.00` 存了两次。磁盘空间浪费还在其次，更可怕的是——
2. **更新异常**：李四换了手机号，你在第 3 行更新了，但第 4 行忘了改。现在数据库里李四有两个手机号，哪个是真的？
3. **插入异常**：如果来了一个新用户"王五"，还没下过单，你往哪存他？这表里没地方存"没有订单的用户"。
4. **删除异常**：如果删除订单 3，李四的全部信息就没了——连他这个人是谁都不知道了。

> **想多一点**：这四种异常——冗余、更新异常、插入异常、删除异常——是数据库理论里的经典问题。它们不是 MySQL 的 bug，而是**表结构设计错误**导致的。范式就是用来解决这些异常的一把"手术刀"。

## 二、第一范式（1NF）：每个字段不可再分

**1NF** *此术语见附录E* 是最基本的要求，只有一条规则：

> **每个字段必须是原子值（不可再分）。**

### 违反 1NF 的例子

```sql
-- ❌ 违反 1NF：address 列存了"省+市+区"，不是原子值
CREATE TABLE users_v1 (
    id      INT,
    name    VARCHAR(50),
    address VARCHAR(200)   -- 存的是"北京市朝阳区望京街道"
);
```

"北京市朝阳区望京街道"包含了省、市、区、街道四个信息，不是一个原子值。如果你要查询"所有住在朝阳区的用户"，只能用 `LIKE '%朝阳区%'` 模糊匹配——慢、不准确，而且索引失效。

### 修正为 1NF

```sql
-- ✅ 符合 1NF：把地址拆成独立的列
CREATE TABLE users_v1_fixed (
    id       INT,
    name     VARCHAR(50),
    province VARCHAR(20),   -- 省
    city     VARCHAR(20),   -- 市
    district VARCHAR(30),   -- 区
    street   VARCHAR(100)   -- 街道
);
```

现在 `WHERE city = '朝阳区'` 可以用索引，精确又高效。

### 另一个常见反例：一列存多个值

```sql
-- ❌ 违反 1NF：phone_numbers 列存了"13800001111,13900002222"
INSERT INTO users_v1 (id, name, phone_numbers) VALUES
(1, '张三', '13800001111,13900002222');
```

用逗号分隔多个值看起来方便，但当你需要查询"谁有手机号 13900002222"时，只能用 `LIKE '%13900002222%'`——又回到了模糊匹配的问题。

```sql
-- ✅ 修正：拆成独立行，用一张"用户电话"表
CREATE TABLE user_phones (
    id       INT AUTO_INCREMENT PRIMARY KEY,
    user_id  INT NOT NULL,
    phone    VARCHAR(20) NOT NULL
);
INSERT INTO user_phones (user_id, phone) VALUES
(1, '13800001111'),
(1, '13900002222');
```

> **想多一点**：1NF 看似简单，但"原子性"是相对的。比如"姓名"通常作为一个字段存，但如果你需要按姓或名分别查询，你可能需要拆成 `first_name` 和 `last_name`。**原子性的判断标准不是"能不能拆"，而是"在这个业务场景下，拆开有没有查询需求"。**

## 三、第二范式（2NF）：非主键列完全依赖于主键

**2NF** *此术语见附录E* 的核心规则：

> **非主键列必须完全依赖于主键，不能只依赖主键的一部分。**

这句话什么意思？先看一个违反 2NF 的例子：

### 违反 2NF 的例子

```sql
-- ❌ 违反 2NF：学生选课表
CREATE TABLE student_course (
    student_id  INT,          -- 学号
    course_id   INT,          -- 课程编号
    student_name VARCHAR(50), -- 学生姓名
    course_name VARCHAR(100), -- 课程名称
    score       INT,          -- 成绩
    PRIMARY KEY (student_id, course_id)  -- 联合主键
);
```

主键是 `(student_id, course_id)` 的联合主键。现在检查每一列：

- `score`（成绩）：同时依赖于 `student_id`（哪个学生）和 `course_id`（哪门课）—— ✅ 完全依赖主键
- `student_name`（学生姓名）：只依赖于 `student_id`，跟 `course_id` 无关—— ❌ 部分依赖主键
- `course_name`（课程名称）：只依赖于 `course_id`，跟 `student_id` 无关—— ❌ 部分依赖主键

**问题**：
- "张三"这个名字，选了 3 门课就存 3 次。如果张三改名了，你得改 3 个地方。
- 如果一门课还没有学生选，这门课就存不进来（插入异常）。

### 修正为 2NF

把表拆开，让每个非主键列只依赖自己的主键：

```sql
-- ✅ 符合 2NF：拆分后

-- 学生表（student_name 只依赖 student_id）
CREATE TABLE students (
    student_id   INT PRIMARY KEY,
    student_name VARCHAR(50) NOT NULL
);

-- 课程表（course_name 只依赖 course_id）
CREATE TABLE courses (
    course_id   INT PRIMARY KEY,
    course_name VARCHAR(100) NOT NULL
);

-- 选课表（score 依赖完整的联合主键）
CREATE TABLE enrollments (
    student_id INT,
    course_id  INT,
    score      INT,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (course_id)  REFERENCES courses(course_id)
);
```

现在：
- 张三的名字只存一处，改名只需改一次。
- 新课程即使没有学生选，也能存入 `courses` 表。

### 还有一个更实用的反例

```sql
-- ❌ 违反 2NF：订单表里存了用户名
CREATE TABLE orders_v2 (
    order_id   INT PRIMARY KEY,
    user_id    INT,
    user_name  VARCHAR(50),   -- 用户名只依赖 user_id，不依赖 order_id
    product    VARCHAR(100),
    amount     DECIMAL(10,2)
);
```

`user_name` 只依赖于 `user_id`，而主键是 `order_id`，所以 `user_name` 不完全依赖于主键——违反 2NF。

```sql
-- ✅ 修正：用户名只存 users 表
-- orders 表只存 user_id，通过 JOIN 获取用户名
CREATE TABLE orders_v2_fixed (
    order_id   INT PRIMARY KEY,
    user_id    INT,
    product    VARCHAR(100),
    amount     DECIMAL(10,2),
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

> **想多一点**：2NF 的适用场景是有**联合主键**的表。如果表只有一个主键列（比如自增 ID），那 2NF 自动满足——因为不存在"只依赖主键的一部分"这个情况。所以实际工作中，2NF 主要出现在**多对多中间表**（如选课表、订单明细表）的设计中。

## 四、第三范式（3NF）：非主键列不传递依赖于主键

**3NF** *此术语见附录E* 的核心规则：

> **非主键列不能传递依赖于主键。即：非主键列只能直接依赖于主键，不能通过另一个非主键列间接依赖。**

### 违反 3NF 的例子

```sql
-- ❌ 违反 3NF：订单表里存了用户手机号
CREATE TABLE orders_v3 (
    order_id   INT PRIMARY KEY,
    user_id    INT,
    user_phone VARCHAR(20),   -- 违反 3NF！
    product    VARCHAR(100),
    amount     DECIMAL(10,2)
);
```

分析依赖关系：
- `order_id` → `user_id`（主键决定用户ID）
- `user_id` → `user_phone`（用户ID决定手机号）
- 所以 `order_id` → `user_id` → `user_phone`，这就是**传递依赖** *此术语见附录E*

**问题**：
- 如果用户改了手机号，所有该用户的订单记录都要更新。
- 如果某条订单数据被误删，用户手机号也跟着丢了。
- 如果用户还没下过单，他的手机号存不进来。

### 修正为 3NF

```sql
-- ✅ 符合 3NF：手机号回归 users 表
-- orders 表只保留直接依赖于 order_id 的列
CREATE TABLE orders_v3_fixed (
    order_id   INT PRIMARY KEY,
    user_id    INT,
    product    VARCHAR(100),
    amount     DECIMAL(10,2),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 手机号在 users 表中，通过 user_id 关联
-- SELECT o.*, u.phone FROM orders o JOIN users u ON o.user_id = u.id;
```

### 一个更贴近生活的例子

```sql
-- ❌ 违反 3NF：员工表里存了部门信息
CREATE TABLE employees (
    emp_id     INT PRIMARY KEY,
    emp_name   VARCHAR(50),
    dept_id    INT,
    dept_name  VARCHAR(50),   -- 部门名称
    dept_floor VARCHAR(20)    -- 楼层
);
```

依赖链：`emp_id` → `dept_id` → `dept_name` / `dept_floor`。`dept_name` 和 `dept_floor` 都通过 `dept_id` 传递依赖于 `emp_id`。

```sql
-- ✅ 修正：部门信息独立成表
CREATE TABLE departments (
    dept_id    INT PRIMARY KEY,
    dept_name  VARCHAR(50) NOT NULL,
    dept_floor VARCHAR(20)
);

CREATE TABLE employees_fixed (
    emp_id   INT PRIMARY KEY,
    emp_name VARCHAR(50),
    dept_id  INT,
    FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
);
```

## 五、BCNF：比 3NF 更严格（了解即可）

> **本节标注"了解即可"**。BCNF 在日常开发中很少成为瓶颈，你只需要知道它存在。

**BCNF** *此术语见附录E*（Boyce-Codd Normal Form，波伊斯-科德范式）是 3NF 的加强版。3NF 只要求"非主键列不传递依赖于主键"，而 BCNF 要求**所有决定因素都必须是候选键**。

简单说：如果一张表里存在 `A → B` 的依赖关系，那么在 BCNF 下，A 必须是候选键。3NF 允许 A 不是候选键（只要 B 是主键列即可），但 BCNF 不允许。

**绝大多数情况下，满足 3NF 的表自动满足 BCNF。** 只有极少数"多个候选键重叠"的复杂场景才会出现违反 BCNF 的情况。你不需要记住 BCNF 的细节，知道"3NF 上面还有更严格的 BCNF"就够了。

## 六、综合实战：从不规范到 3NF 的完整改造

以 `my_shop` 数据库为背景，假设你一开始建了一张"万能表"：

```sql
-- 初始状态：一张大表，违反 1NF + 2NF + 3NF
CREATE TABLE bad_orders (
    order_id     INT,
    customer     VARCHAR(50),           -- 存了"张三|13800001111"
    products     VARCHAR(200),          -- 存了"iPhone 15,2个"
    product_cat  VARCHAR(50),           -- 商品分类
    order_date   VARCHAR(20),           -- 日期存成了字符串
    total_amount DECIMAL(10,2)
);
```

### 第一步：满足 1NF（原子化）

```sql
-- 把 customer 拆成 name + phone
-- 把 products 拆成商品名和数量两个列
-- 把 order_date 改为 DATE 类型
CREATE TABLE orders_1nf (
    order_id      INT PRIMARY KEY,
    customer_name VARCHAR(50),
    customer_phone VARCHAR(20),
    product_name  VARCHAR(100),
    quantity      INT,
    product_cat   VARCHAR(50),
    order_date    DATE,
    total_amount  DECIMAL(10,2)
);
```

### 第二步：满足 2NF（消除部分依赖）

假定主键是 `(order_id, product_name)`（一个订单可以有多个商品）。检查：

- `customer_name`：只依赖 `order_id`，不依赖 `product_name` → 部分依赖 ❌
- `customer_phone`：同上 → 部分依赖 ❌
- `product_cat`：只依赖 `product_name`，不依赖 `order_id` → 部分依赖 ❌
- `quantity`：同时依赖 `order_id` 和 `product_name` → 完全依赖 ✅
- `order_date`：只依赖 `order_id` → 部分依赖 ❌
- `total_amount`：只依赖 `order_id` → 部分依赖 ❌

拆分为：

```sql
-- 订单主表（order_id 是主键）
CREATE TABLE orders_2nf (
    order_id      INT PRIMARY KEY,
    customer_name VARCHAR(50),
    customer_phone VARCHAR(20),
    order_date    DATE,
    total_amount  DECIMAL(10,2)
);

-- 订单明细表（order_id + product_name 联合主键）
CREATE TABLE order_items_2nf (
    order_id     INT,
    product_name VARCHAR(100),
    product_cat  VARCHAR(50),
    quantity     INT,
    PRIMARY KEY (order_id, product_name),
    FOREIGN KEY (order_id) REFERENCES orders_2nf(order_id)
);
```

### 第三步：满足 3NF（消除传递依赖）

检查 `orders_2nf`：
- `order_id` → `customer_name` → `customer_phone` → 传递依赖 ❌

检查 `order_items_2nf`：
- `order_id` + `product_name` → `product_cat`，但 `product_name` 单独就能决定 `product_cat` → 传递依赖 ❌

```sql
-- ✅ 最终：完整的 3NF 设计

-- 用户表
CREATE TABLE users_3nf (
    user_id       INT PRIMARY KEY AUTO_INCREMENT,
    user_name     VARCHAR(50) NOT NULL,
    user_phone    VARCHAR(20) NOT NULL
);

-- 商品表
CREATE TABLE products_3nf (
    product_id   INT PRIMARY KEY AUTO_INCREMENT,
    product_name VARCHAR(100) NOT NULL,
    category     VARCHAR(50),
    price        DECIMAL(10,2) NOT NULL
);

-- 订单主表
CREATE TABLE orders_3nf (
    order_id     INT PRIMARY KEY AUTO_INCREMENT,
    user_id      INT NOT NULL,
    order_date   DATE NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users_3nf(user_id)
);

-- 订单明细表
CREATE TABLE order_items_3nf (
    order_id    INT,
    product_id  INT,
    quantity    INT NOT NULL,
    unit_price  DECIMAL(10,2) NOT NULL,
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (order_id)   REFERENCES orders_3nf(order_id),
    FOREIGN KEY (product_id) REFERENCES products_3nf(product_id)
);
```

**改造前后对比**：

|  | 改造前（反范式） | 改造后（3NF） |
|------|------|------|
| 用户名修改 | 需要改 N 条订单记录 | 只需改 users 表 1 条 |
| 商品分类修改 | 需要改所有包含该商品的订单 | 只需改 products 表 |
| 新增用户 | 必须有订单才能存 | 用户独立存在 |
| 删除订单 | 可能丢失用户信息 | 用户数据不受影响 |
| 查询复杂度 | 单表查询 | 需要 JOIN 3 张表 |

## 七、反范式化：故意违反范式

### 7.1 什么时候需要反范式？

范式化减少冗余，但带来一个代价：**查询需要 JOIN 更多表**。当你需要高频查询"订单 + 用户名 + 手机号"时，每次都要 JOIN `users` 表，在数据量很大时，JOIN 可能成为性能瓶颈。

**反范式化** *此术语见附录E* 就是**故意保留一些冗余，用数据一致性风险换取查询性能**。

典型场景：
- **报表查询**：每天跑一次的销售报表，数据量上千万，JOIN 太慢。把用户名、商品名直接冗余到订单表里，报表秒出。
- **高频读取、低频写入**：用户昵称很少改，但每次展示订单列表都要显示。把昵称冗余到订单表，省掉一次 JOIN。
- **数据仓库 / 分析库**：专门用于分析的数据库，数据是定期导入的，不需要实时一致。

### 7.2 反范式化的代价：数据不一致

```sql
-- 反范式化：orders 表冗余了 user_name 和 user_phone
CREATE TABLE orders_denormalized (
    order_id     INT PRIMARY KEY AUTO_INCREMENT,
    user_id      INT,
    user_name    VARCHAR(50),    -- 冗余
    user_phone   VARCHAR(20),    -- 冗余
    product      VARCHAR(100),
    amount       DECIMAL(10,2),
    order_date   DATE
);
```

**问题演示**：

```sql
-- 用户下单时，把用户名和手机号写入 orders
INSERT INTO orders_denormalized (user_id, user_name, user_phone, product, amount, order_date)
VALUES (1, '张三', '13800001111', 'iPhone 15', 6999.00, '2024-01-15');

-- 后来张三改了手机号，只更新了 users 表...
UPDATE users SET phone = '13900009999' WHERE id = 1;

-- 但 orders 表里的 user_phone 还是旧的！
SELECT user_id, user_name, user_phone FROM orders_denormalized WHERE user_id = 1;
-- 输出：1 | 张三 | 13800001111  ← 和 users 表不一致！
```

**这就是冗余的代价**：你必须自己保证数据一致性。如果业务代码忘了同步更新，就会出现两个不同的值。

### 7.3 反范式化的安全措施

如果决定反范式化，建议做以下保护：

1. **用触发器自动同步**（但触发器有性能开销，高并发慎用）
2. **应用层事务保证**：更新用户手机号时，在同一事务中更新所有冗余字段
3. **定期数据校验脚本**：每晚跑一次，检查 `orders.user_phone` 和 `users.phone` 是否一致，不一致则告警
4. **明确标注冗余列**：在表注释中写清楚"此列为冗余字段，源数据来自 `users.phone`"

```sql
-- 在 COMMENT 中标注冗余
CREATE TABLE orders_denormalized (
    -- ...
    user_phone VARCHAR(20) COMMENT '冗余字段，源数据来自 users.phone。修改用户手机号时需同步更新此列'
);
```

### 7.4 过度范式化的反面：JOIN 地狱

```sql
-- ❌ 过度范式化：为了"完美 3NF"，连城市都拆成单独的表
CREATE TABLE provinces (
    province_id INT PRIMARY KEY,
    province_name VARCHAR(20)
);
CREATE TABLE cities (
    city_id INT PRIMARY KEY,
    city_name VARCHAR(20),
    province_id INT
);
CREATE TABLE districts (
    district_id INT PRIMARY KEY,
    district_name VARCHAR(30),
    city_id INT
);
-- 查询一个订单的完整地址，需要 JOIN 4 张表！
SELECT o.id, p.province_name, c.city_name, d.district_name, a.street
FROM orders o
JOIN addresses a ON o.address_id = a.id
JOIN districts d ON a.district_id = d.district_id
JOIN cities c ON d.city_id = c.city_id
JOIN provinces p ON c.province_id = p.province_id;
```

如果你的地址数据基本不变、查询频率很高，把省市区直接存成字符串反而更合理。**范式是工具，不是信仰。** 关键是理解取舍，而不是死记规则。

## 我做得对不对？

### 验证方法

```sql
-- 1. 验证 1NF：检查是否有列存了多个值
SELECT * FROM bad_orders WHERE products LIKE '%,%';
-- 如果 products 列能用逗号分割出多个值，则违反 1NF

-- 2. 验证 2NF：检查联合主键表
-- 以 order_items 为例，查看是否有列只依赖联合主键的一部分
-- 思考：product_name 是否只依赖 product_id（联合主键的一部分）？
-- 如果是，则违反 2NF

-- 3. 验证 3NF：检查是否存在传递依赖
-- 自问：orders 表里的 user_phone，是直接依赖 order_id 还是通过 user_id 间接依赖？
-- 如果是间接依赖（order_id → user_id → user_phone），则违反 3NF

-- 4. 实操：在 my_shop 中检查现有表结构
USE my_shop;
DESC users;
DESC orders;
-- 判断：users 和 orders 符合第几范式？
```

### 判断练习

给定以下表结构，判断违反了几 NF，并给出修改方案：

```sql
-- 练习题：请判断此表违反了几 NF
CREATE TABLE school_records (
    student_id   INT,
    student_name VARCHAR(50),
    class_id     INT,
    class_name   VARCHAR(50),
    teacher_id   INT,
    teacher_name VARCHAR(50),
    subject      VARCHAR(50),
    score        INT,
    PRIMARY KEY (student_id, subject)
);
```

> **答案**（翻到下一页前先自己思考）：
> - `student_name` 只依赖 `student_id` → 违反 2NF（部分依赖）
> - `class_name` 依赖 `student_id` → `class_id` → `class_name` → 同时违反 2NF 和 3NF
> - `teacher_name` 依赖 `teacher_id`，而 `teacher_id` 依赖 `(student_id, subject)` → 违反 3NF（传递依赖）
> - 修正方案：拆为 students、classes、teachers、records 四张表

## 不对怎么办？

### 常见错误 1：把"拆分"等同于范式化

```sql
-- ❌ 为了"范式化"而盲目拆分——把 users 表的 email 单独拆成 emails 表
CREATE TABLE users_wrong (
    id       INT PRIMARY KEY,
    username VARCHAR(50)
);
CREATE TABLE emails (
    id      INT PRIMARY KEY,
    user_id INT,
    email   VARCHAR(100)
);
-- email 和 user_id 是一一对应的，拆出来没有任何好处，只是增加了 JOIN 成本
```

**判断标准**：拆表的前提是**消除冗余**。如果两个表是一一对应关系，拆出来只是自找麻烦。

### 常见错误 2：过度范式化导致 JOIN 地狱

已经在上文 7.4 节演示过了。记住：**3NF 是经验上的"甜点区"——既避免了大部分异常，又不会让查询过于复杂。**

### 常见错误 3：认为"一张表查询快，所以永远不范式化"

```sql
-- ❌ 把一切塞进一张表
SELECT * FROM big_table WHERE user_id = 1;
-- 单表查询确实快，但：
-- 1. 用户改名你得改 1000 条记录
-- 2. 表越来越宽，索引效率下降
-- 3. 磁盘空间浪费严重
```

**权衡原则**：
- 读写比 > 100:1（读远多于写）→ 可以考虑反范式
- 读写比 < 10:1（写也比较频繁）→ 保持 3NF
- 不确定时 → 先从 3NF 开始，发现性能瓶颈后再针对性反范式

### 常见错误 4：1NF 中的"原子性"判断过于死板

```sql
-- 这个问题不是非黑即白：
-- address 列存"北京市朝阳区"到底算不算违反 1NF？

-- 如果业务上从不需要按"区"查询，那存一起就合理
-- 如果需要按"区"筛选，就必须拆开

-- 关键不是"能不能拆"，而是"业务上需不需要拆"
```

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 范式的目的 | 消除冗余、更新异常、插入异常、删除异常 | 范式是工具，不是信仰 |
| 1NF | 字段不可再分（原子性） | 原子性是相对的，取决于业务查询需求 |
| 2NF | 非主键列完全依赖于主键（无部分依赖） | 主要适用于有联合主键的表；单主键表自动满足 2NF |
| 3NF | 非主键列不传递依赖于主键（无传递依赖） | 3NF 是日常开发推荐的"甜点区" |
| BCNF | 比 3NF 更严格，了解即可 | 绝大多数满足 3NF 的表已满足 BCNF |
| 反范式化 | 故意保留冗余，用一致性风险换查询性能 | 适用场景：报表、高频读低频写、数据仓库 |
| 冗余的代价 | 数据不一致风险 | 安全措施：触发器、应用层事务、定期校验、标注冗余列 |
| 过度范式化 | 拆表过多导致 JOIN 地狱 | 一一对应关系不要拆表 |
| 范式化决策 | 读写比 < 10:1 用 3NF，> 100:1 考虑反范式 | 不确定时从 3NF 开始，遇瓶颈再反范式 |
| 判断方法 | 自问：列依赖什么？有没有间接依赖？ | 画出依赖链：主键 → 列A → 列B 就是传递依赖 |
