# 第91章 · 设计模式（Go实现）

> "设计模式不是背下来的23个名字——是前辈们踩过无数坑后总结的'套路'。Go没有Java那种满屏的抽象类和继承，但设计模式的思想是通用的。本章精选三类九种最常用的模式，每种都配场景、完整Go代码和优缺点分析。不要求你背下来，但要求你下次写代码时能想起来——'这里好像可以用责任链'。"

---

## 91.1 创建型模式

创建型模式解决的问题：**如何优雅地创建对象**。不是在每个地方都 `NewXXX()`，而是把创建逻辑封装起来。

### 单例模式（Singleton）

**场景**：整个程序只需要一个Redis连接池、一个日志实例、一个配置对象。如果到处创建新实例，资源浪费还容易出Bug。

**Go的实现**：`sync.Once`——保证了初始化代码在多goroutine并发的环境下只执行一次。

```go
package config

import (
	"encoding/json"
	"os"
	"sync"
)

type Config struct {
	DBHost string `json:"db_host"`
	DBPort int    `json:"db_port"`
	APIKey string `json:"api_key"`
}

var (
	instance *Config
	once     sync.Once
)

func GetConfig() *Config {
	once.Do(func() {
		instance = &Config{}
		data, _ := os.ReadFile("config.json")
		json.Unmarshal(data, instance)
	})
	return instance
}
```

使用：

```go
cfg1 := config.GetConfig()
cfg2 := config.GetConfig()
fmt.Println(cfg1 == cfg2)
```

无论多少个goroutine同时调用 `GetConfig()`，`once.Do` 里的函数只执行一次。第二次调用直接返回已有的实例。

**优缺点**：

| 优点 | 缺点 |
|------|------|
| 全局唯一，避免重复初始化 | 隐藏了依赖关系——谁在用这个单例看不出来 |
| 线程安全（sync.Once保证） | 难以测试——单例状态在测试间"泄漏" |
| 实现简单 | 可能变成全局状态的"垃圾桶" |

### 工厂模式（Factory，根据参数动态创建不同对象的模式，详见附录I）

**场景**：根据不同的输入参数创建不同类型的对象。比如支付系统：用户选微信支付→创建WechatPay，选支付宝→创建Alipay。

**Go的实现**：Go没有构造函数重载，工厂模式用函数返回接口来模拟。

```go
type PaymentMethod interface {
	Pay(amount float64) error
	Refund(orderID string) error
}

type WechatPay struct {
	AppID string
}

func (w *WechatPay) Pay(amount float64) error {
	fmt.Printf("微信支付: %.2f元 (AppID=%s)\n", amount, w.AppID)
	return nil
}

func (w *WechatPay) Refund(orderID string) error {
	fmt.Printf("微信退款: 订单=%s\n", orderID)
	return nil
}

type Alipay struct {
	PartnerID string
}

func (a *Alipay) Pay(amount float64) error {
	fmt.Printf("支付宝支付: %.2f元 (PartnerID=%s)\n", amount, a.PartnerID)
	return nil
}

func (a *Alipay) Refund(orderID string) error {
	fmt.Printf("支付宝退款: 订单=%s\n", orderID)
	return nil
}

func NewPayment(method string) (PaymentMethod, error) {
	switch method {
	case "wechat":
		return &WechatPay{AppID: "wx123456"}, nil
	case "alipay":
		return &Alipay{PartnerID: "2088xxxx"}, nil
	default:
		return nil, fmt.Errorf("不支持的支付方式: %s", method)
	}
}
```

使用：

```go
pay, err := NewPayment("wechat")
if err != nil {
    log.Fatal(err)
}
pay.Pay(99.00)
```

**优点**：调用方不需要知道 `WechatPay` 或 `Alipay` 的具体类型，只需要知道 `PaymentMethod` 接口。新增支付方式（如Apple Pay）只需在 `NewPayment` 里加一个case，调用方代码不动。

### 建造者模式（Builder）

**场景**：创建一个有很多可选参数的对象。Go里最常见的是创建HTTP请求、配置数据库连接池，或者构建复杂的结构体。

Go没有函数参数的默认值，一个结构体有10个字段时，传统的做法很痛苦。建造者模式通过"链式调用"解决。

```go
type ServerConfig struct {
	Host        string
	Port        int
	MaxConn     int
	TLSEnabled  bool
	CertFile    string
	KeyFile     string
	ReadTimeout time.Duration
}

type ServerConfigBuilder struct {
	config ServerConfig
}

func NewServerConfigBuilder() *ServerConfigBuilder {
	return &ServerConfigBuilder{
		config: ServerConfig{
			Host:        "0.0.0.0",
			Port:        8080,
			MaxConn:     100,
			ReadTimeout: 30 * time.Second,
		},
	}
}

func (b *ServerConfigBuilder) Host(host string) *ServerConfigBuilder {
	b.config.Host = host
	return b
}

func (b *ServerConfigBuilder) Port(port int) *ServerConfigBuilder {
	b.config.Port = port
	return b
}

func (b *ServerConfigBuilder) MaxConn(max int) *ServerConfigBuilder {
	b.config.MaxConn = max
	return b
}

func (b *ServerConfigBuilder) WithTLS(certFile, keyFile string) *ServerConfigBuilder {
	b.config.TLSEnabled = true
	b.config.CertFile = certFile
	b.config.KeyFile = keyFile
	return b
}

func (b *ServerConfigBuilder) Build() ServerConfig {
	return b.config
}
```

使用：

```go
config := NewServerConfigBuilder().
	Host("192.168.1.100").
	Port(443).
	MaxConn(1000).
	WithTLS("/etc/certs/server.crt", "/etc/certs/server.key").
	Build()
```

比起传统方式：

```go
config := ServerConfig{
    Host: "192.168.1.100",
    Port: 443,
    MaxConn: 1000,
    TLSEnabled: true,
    CertFile: "/etc/certs/server.crt",
    KeyFile: "/etc/certs/server.key",
    ReadTimeout: 30 * time.Second,
}
```

建造者的优势在于：
1. 可以给字段设置合理的默认值（比零值好）
2. 可以在 `Build()` 中做参数校验（比如"如果TLSEnabled=true但CertFile为空就报错"）
3. 链式调用语义清晰

## 91.2 结构型模式

结构型模式解决的问题：**如何把不同的类和对象组合成更大的结构**。

### 适配器模式（Adapter）

**场景**：你有一个旧系统的接口，现在要对接一个新系统，但接口不兼容。不想改旧代码也不想改新代码→适配器来翻译。

**比喻**：去香港旅游，你的插头是三脚的，酒店插座是英标三孔——你需要一个转换插头（适配器）。

**Go实现**：

```go
type OldLogger interface {
	LogInfo(msg string)
	LogError(msg string, err error)
}

type FileLogger struct{}

func (l *FileLogger) LogInfo(msg string) {
	fmt.Printf("[INFO] 写入文件: %s\n", msg)
}

func (l *FileLogger) LogError(msg string, err error) {
	fmt.Printf("[ERROR] 写入文件: %s | %v\n", msg, err)
}

type NewLogSystem struct{}

func (s *NewLogSystem) WriteLog(level string, message string, metadata map[string]string) {
	fmt.Printf("[%s] %s | meta=%v\n", level, message, metadata)
}

type LogAdapter struct {
	newSystem *NewLogSystem
}

func (a *LogAdapter) LogInfo(msg string) {
	a.newSystem.WriteLog("INFO", msg, map[string]string{"source": "app"})
}

func (a *LogAdapter) LogError(msg string, err error) {
	a.newSystem.WriteLog("ERROR", msg, map[string]string{
		"source": "app",
		"error":  err.Error(),
	})
}
```

旧代码继续用 `OldLogger` 接口，适配器在内部翻译成新系统的调用。

### 装饰器模式（Decorator）

**场景**：在不修改原有代码的情况下，给一个对象动态添加功能。Go里最常见的装饰器就是HTTP中间件。

**Go实现**——为函数添加日志和计时：

```go
type Handler func(ctx context.Context, req interface{}) (interface{}, error)

func LoggingDecorator(handler Handler) Handler {
	return func(ctx context.Context, req interface{}) (interface{}, error) {
		log.Printf("请求开始: %v", req)
		result, err := handler(ctx, req)
		if err != nil {
			log.Printf("请求失败: %v", err)
		} else {
			log.Printf("请求成功: %v", result)
		}
		return result, err
	}
}

func RetryDecorator(handler Handler, maxRetries int) Handler {
	return func(ctx context.Context, req interface{}) (interface{}, error) {
		var lastErr error
		for i := 0; i <= maxRetries; i++ {
			result, err := handler(ctx, req)
			if err == nil {
				return result, nil
			}
			lastErr = err
			time.Sleep(time.Duration(i*100) * time.Millisecond)
		}
		return nil, lastErr
	}
}
```

使用：像层层包装礼物一样：

```go
var coreHandler Handler = func(ctx context.Context, req interface{}) (interface{}, error) {
    return "处理完成", nil
}

handler := LoggingDecorator(RetryDecorator(coreHandler, 3))
result, _ := handler(context.Background(), "某个请求")
```

### 代理模式（Proxy）

**场景**：控制对某个对象的访问——比如"访问前要检查权限"、"结果要缓存"、"要限制访问频率"。

**Go实现**——访问数据库的缓存代理：

```go
type UserRepository interface {
	GetUser(id string) (*User, error)
}

type DBUserRepository struct {
	db *sql.DB
}

func (r *DBUserRepository) GetUser(id string) (*User, error) {
	row := r.db.QueryRow("SELECT id, name FROM users WHERE id = ?", id)
	var user User
	err := row.Scan(&user.ID, &user.Name)
	return &user, err
}

type CachedUserRepository struct {
	real  UserRepository
	cache map[string]*User
	mu    sync.RWMutex
}

func NewCachedRepo(real UserRepository) *CachedUserRepository {
	return &CachedUserRepository{
		real:  real,
		cache: make(map[string]*User),
	}
}

func (r *CachedUserRepository) GetUser(id string) (*User, error) {
	r.mu.RLock()
	user, ok := r.cache[id]
	r.mu.RUnlock()
	if ok {
		return user, nil
	}

	user, err := r.real.GetUser(id)
	if err != nil {
		return nil, err
	}

	r.mu.Lock()
	r.cache[id] = user
	r.mu.Unlock()
	return user, nil
}
```

上层代码只知道 `UserRepository` 接口，不关心背后是直查数据库还是有缓存。

## 91.3 行为型模式

行为型模式解决的问题：**对象之间如何通信和协作**。

### 策略模式（Strategy）

**场景**：同一个问题有多种算法/方案，运行时动态选择。比如：商品排序（按价格/按销量/按评分）、文件压缩（Gzip/Zstd/Snappy）、消息序列化（JSON/Protobuf/MessagePack）。

**Go实现**：

```go
type SortStrategy interface {
	Sort(products []Product) []Product
}

type SortByPrice struct{}

func (s *SortByPrice) Sort(products []Product) []Product {
	sorted := make([]Product, len(products))
	copy(sorted, products)
	sort.Slice(sorted, func(i, j int) bool {
		return sorted[i].Price < sorted[j].Price
	})
	return sorted
}

type SortBySales struct{}

func (s *SortBySales) Sort(products []Product) []Product {
	sorted := make([]Product, len(products))
	copy(sorted, products)
	sort.Slice(sorted, func(i, j int) bool {
		return sorted[i].Sales > sorted[j].Sales
	})
	return sorted
}

type ProductSorter struct {
	strategy SortStrategy
}

func (ps *ProductSorter) SetStrategy(s SortStrategy) {
	ps.strategy = s
}

func (ps *ProductSorter) Sort(products []Product) []Product {
	return ps.strategy.Sort(products)
}
```

使用：

```go
sorter := &ProductSorter{}
sorter.SetStrategy(&SortByPrice{})
cheapest := sorter.Sort(products)

sorter.SetStrategy(&SortBySales{})
hottest := sorter.Sort(products)
```

**关键优势**：新增一种排序方式（如"按评分"），只需新增一个实现 `SortStrategy` 接口的类型，不动任何已有代码。

### 观察者模式（Observer）

**场景**：一个对象状态变化时，自动通知所有"关注者"。发布订阅、事件监听、UI框架的数据绑定都是这个思想。

**Go实现**——订单状态变更通知：

```go
type Observer interface {
	OnOrderStatusChanged(orderID string, oldStatus, newStatus string)
}

type OrderNotifier struct {
	observers []Observer
}

func (n *OrderNotifier) Subscribe(o Observer) {
	n.observers = append(n.observers, o)
}

func (n *OrderNotifier) Notify(orderID, oldStatus, newStatus string) {
	for _, o := range n.observers {
		o.OnOrderStatusChanged(orderID, oldStatus, newStatus)
	}
}

type SMSObserver struct {
	phone string
}

func (s *SMSObserver) OnOrderStatusChanged(orderID, oldStatus, newStatus string) {
	fmt.Printf("[短信通知] %s: 订单%s状态变更 %s→%s\n", s.phone, orderID, oldStatus, newStatus)
}

type LogObserver struct{}

func (l *LogObserver) OnOrderStatusChanged(orderID, oldStatus, newStatus string) {
	fmt.Printf("[日志] 订单%s: %s → %s\n", orderID, oldStatus, newStatus)
}
```

使用：

```go
notifier := &OrderNotifier{}
notifier.Subscribe(&SMSObserver{phone: "13800138000"})
notifier.Subscribe(&LogObserver{})

notifier.Notify("ORD-001", "pending", "paid")
```

### 责任链模式（Chain of Responsibility）

**场景**：一个请求要经过层层审批，每一层可以"处理"或"传递给下一层"。请假审批（组长→经理→总监）、Web中间件链、订单风控检查链。

**Go实现**——订单风控检查链：

```go
type RiskCheckHandler interface {
	Handle(order Order) error
	SetNext(RiskCheckHandler) RiskCheckHandler
}

type BaseRiskHandler struct {
	next RiskCheckHandler
}

func (b *BaseRiskHandler) SetNext(handler RiskCheckHandler) RiskCheckHandler {
	b.next = handler
	return handler
}

func (b *BaseRiskHandler) HandleNext(order Order) error {
	if b.next != nil {
		return b.next.Handle(order)
	}
	return nil
}

type AmountCheckHandler struct {
	BaseRiskHandler
	maxAmount float64
}

func (h *AmountCheckHandler) Handle(order Order) error {
	if order.Amount > h.maxAmount {
		return fmt.Errorf("订单金额%.2f超过上限%.2f，需人工审核", order.Amount, h.maxAmount)
	}
	fmt.Printf("[风控] 金额检查通过: %.2f\n", order.Amount)
	return h.HandleNext(order)
}

type FrequencyCheckHandler struct {
	BaseRiskHandler
	maxPerHour int
}

func (h *FrequencyCheckHandler) Handle(order Order) error {
	count := getOrderCountInLastHour(order.UserID)
	if count > h.maxPerHour {
		return fmt.Errorf("用户%s一小时内下单%d次，疑似刷单", order.UserID, count)
	}
	fmt.Printf("[风控] 频率检查通过: %d次/小时\n", count)
	return h.HandleNext(order)
}

type BlacklistCheckHandler struct {
	BaseRiskHandler
}

func (h *BlacklistCheckHandler) Handle(order Order) error {
	if isBlacklisted(order.UserID) {
		return fmt.Errorf("用户%s在黑名单中", order.UserID)
	}
	fmt.Printf("[风控] 黑名单检查通过\n")
	return h.HandleNext(order)
}
```

使用：

```go
amountCheck := &AmountCheckHandler{maxAmount: 10000}
freqCheck := &FrequencyCheckHandler{maxPerHour: 5}
blacklistCheck := &BlacklistCheckHandler{}

amountCheck.SetNext(freqCheck).SetNext(blacklistCheck)

err := amountCheck.Handle(order)
if err != nil {
    fmt.Printf("风控拦截: %v\n", err)
    return
}
```

**优势**：
- 每个检查独立，可以任意组合顺序
- 新增检查项不影响已有代码
- 调用方只需要知道链的第一个节点

### 模板方法模式（Template Method）

**场景**：一个算法的骨架固定，但某些步骤留给子类自定义。Go没有继承，用**函数参数**实现。

```go
type DataExporter struct {
	FetchData  func() ([]map[string]interface{}, error)
	FormatRow  func(row map[string]interface{}) string
}

func (e *DataExporter) Export(filename string) error {
	data, err := e.FetchData()
	if err != nil {
		return err
	}

	file, err := os.Create(filename)
	if err != nil {
		return err
	}
	defer file.Close()

	for _, row := range data {
		line := e.FormatRow(row)
		file.WriteString(line + "\n")
	}

	return nil
}
```

使用：

```go
csvExporter := &DataExporter{
	FetchData: func() ([]map[string]interface{}, error) {
		return queryAllUsers()
	},
	FormatRow: func(row map[string]interface{}) string {
		return fmt.Sprintf("%v,%v,%v", row["id"], row["name"], row["email"])
	},
}
csvExporter.Export("users.csv")

jsonExporter := &DataExporter{
	FetchData: func() ([]map[string]interface{}, error) {
		return queryAllUsers()
	},
	FormatRow: func(row map[string]interface{}) string {
		data, _ := json.Marshal(row)
		return string(data)
	},
}
jsonExporter.Export("users.jsonl")
```

骨架相同（取数据→遍历→写文件），细节不同（CSV一行 vs JSON一行）。

🤔 **想多一点**：Go需要设计模式吗？

有人会说："设计模式是Java/C++的遗产——因为那些语言表达能力不足，才需要这些套路。Go语言不需要。"

这句话只对了一半。Go确实不需要"工厂工厂模式"或者"抽象单例建造者装饰器"这种叠床架屋的东西。但模式背后的**思想**是通用的：解耦、封装变化、面向接口编程。

本章的Go实现都很简洁——不是因为"简化了设计模式"，而是因为Go的接口和函数一等公民让这些模式表达起来更自然。比如"模板方法模式"在Java里需要抽象类和继承，在Go里只需要一个函数参数。

**记住：模式是手段，不是目的。** 如果一段代码用了三个设计模式反而更难懂了——那就是在炫技，不是在解决问题。

## 本章小结

| 模式 | 分类 | Go实现方式 | 典型场景 |
|------|------|-----------|---------|
| 单例 | 创建型 | sync.Once | 全局配置、连接池、日志实例 |
| 工厂 | 创建型 | 函数返回接口 | 支付方式、数据库驱动 |
| 建造者 | 创建型 | 链式调用 | 复杂配置对象、HTTP请求构建 |
| 适配器 | 结构型 | 接口包装 | 新老系统对接、第三方API统一 |
| 装饰器 | 结构型 | 函数包装函数 | HTTP中间件、日志/重试/计时 |
| 代理 | 结构型 | 接口+组合 | 缓存、权限检查、限流 |
| 策略 | 行为型 | 接口+组合 | 排序算法、支付策略、序列化格式 |
| 观察者 | 行为型 | 接口+注册列表 | 事件通知、发布订阅 |
| 责任链 | 行为型 | 接口+next指针 | 审批流、风控检查、中间件链 |
| 模板方法 | 行为型 | 函数参数+结构体 | 数据导出、ETL流程 |

> 🚀 下一章：设计模式讲的是"怎么写好一个类"，SOLID原则讲的是"怎么设计好一组类"。五个原则，五个反例，五个正例——让你的代码从"能跑"升级到"好维护"。
