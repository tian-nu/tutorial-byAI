# 102-微服务基础

> 💡 单体架构是一座"超大型商场"——服装、餐饮、电影院全在一个楼里。今天是情人节，电影院排队 500 人，整栋楼的电梯、空调、停车场全瘫痪了。微服务是把一栋楼拆成"步行街"——电影院一栋楼、餐馆一栋楼、服装店一栋楼。电影院爆满只影响自己那栋楼，其他店铺照常营业。

## 本章目标
- 理解单体架构的痛点与微服务的优势
- 掌握微服务的核心概念：服务拆分、服务注册发现、API 网关
- 知道微服务引入的新问题（复杂度并不消失，只是转移）
- 理解什么时候该用微服务，什么时候不该用

## 102.1 单体架构的四种痛

```
┌──────────────────────────────────────────────┐
│             单体应用（WAR / JAR）              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐     │
│  │ 用户模块  │ │ 订单模块  │ │ 商品模块  │     │
│  └──────────┘ └──────────┘ └──────────┘     │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐     │
│  │ 支付模块  │ │ 物流模块  │ │ 营销模块  │     │
│  └──────────┘ └──────────┘ └──────────┘     │
│                一套代码库                     │
└──────────────────────────────────────────────┘
```

| 痛点 | 表现 |
|------|------|
| **难扩展** | 订单模块压力大，你只能把整个应用多部署几份 |
| **难维护** | 100 万行代码在一个仓库，改动牵一发而动全身 |
| **难部署** | 改一行代码 → 编译 20 分钟 → 部署整个应用 |
| **技术锁定** | 只能用一种语言、一种数据库 |

## 102.2 微服务怎么解决

```
                    ┌──────────────┐
                    │  API 网关     │
                    └──┬──┬──┬──┬──┘
                       │  │  │  │
        ┌──────────────┼──┼──┼──┼──────────────┐
        ▼              ▼  │  │  ▼              ▼
┌──────────┐  ┌──────────┐ │ │ ┌──────────┐ ┌──────────┐
│ 用户服务  │  │ 订单服务  │ │ │ │ 商品服务  │ │ 支付服务  │
│ (Java)   │  │ (Java)   │ │ │ │ (Java)   │ │ (Go)     │
│ MySQL    │  │ MySQL    │ │ │ │ ES       │ │ MySQL    │
└──────────┘  └──────────┘ │ │ └──────────┘ └──────────┘
                           │ │
                    ┌──────┘ └──────┐
                    ▼              ▼
            ┌──────────┐  ┌──────────┐
            │ 物流服务  │  │ 营销服务  │
            │ (Python) │  │ (Java)   │
            │ PG       │  │ Redis    │
            └──────────┘  └──────────┘
```

| 能力 | 说明 |
|------|------|
| **独立部署** | 改订单服务，只部署订单服务 |
| **独立扩展** | 订单忙就多部署几个订单服务实例 |
| **技术异构** | 用户服务用 Java + MySQL，营销服务用 Python + Redis |
| **故障隔离** | 物流服务挂了，不影响用户下单 |
| **团队自治** | 每个小团队负责一个服务，独立开发、测试、上线 |

## 102.4 服务注册与发现

问题：订单服务调用用户服务，但用户服务部署了 3 台机器，IP 是动态的。

```
注册：用户服务启动 → 向注册中心报告 "我在 192.168.1.10:8081"
发现：订单服务问注册中心 "用户服务在哪？" → 返回 [192.168.1.10:8081, 192.168.1.11:8081, 192.168.1.12:8081]
```

### Spring Cloud 中的实现

```yaml
# 用户服务
spring:
  application:
    name: user-service
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848
```

```java
// 订单服务调用用户服务
@FeignClient(name = "user-service")  // 只需服务名，不需要 IP
public interface UserClient {
    @GetMapping("/api/users/{id}")
    UserDTO getUserById(@PathVariable Long id);
}
```

## 102.5 API 网关

API 网关是所有请求的"大门"：

```
客户端
  │ GET /api/users/1001
  ▼
API 网关 (Gateway)
  │ 1. 认证：token 是否有效？
  │ 2. 限流：这个 IP 请求是否超限？
  │ 3. 路由：/api/users/** → user-service
  │ 4. 负载均衡：user-service 有 3 个实例，选哪个？
  ▼
user-service (具体实例)
```

Spring Cloud Gateway 配置：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service    # lb = load balance
          predicates:
            - Path=/api/users/**
          filters:
            - StripPrefix=0
```

## 102.6 微服务引入的新问题

微服务不消除复杂度，只是把它从"代码内部"转移到了"服务之间"：

| 新问题 | 解决方案 |
|--------|----------|
| 服务发现 | Nacos / Consul / Eureka |
| 配置管理 | Nacos Config / Apollo |
| 服务间调用 | Feign (HTTP) / gRPC / Dubbo |
| 分布式事务 | Seata / 消息队列最终一致性 |
| 链路追踪 | SkyWalking / Zipkin / Jaeger |
| 日志聚合 | ELK (Elasticsearch + Logstash + Kibana) |
| 容错与降级 | Sentinel / Resilience4j |
| 部署 | Docker + Kubernetes |

## 102.7 什么时候不该用微服务？

| 条件 | 建议 |
|------|------|
| 团队 < 5 人 | 单体，否则运维成本压死开发 |
| 业务不复杂 | 单体 + 模块化开发（按包分层） |
| 没有 DevOps 能力 | 微服务需要自动部署、监控、容器化 |
| 用户量 < 10 万 | 单体 + 单机 + 读写分离足够 |

> 🤔 想多一点：Martin Fowler 说过："在你确定需要微服务之前，先以单体架构构建系统。" 很多公司从微服务退回到"模块化单体"（Modular Monolith），因为运维微服务的成本远超性能收益。

## 102.8 小结

| 知识点 | 核心内容 |
|--------|----------|
| 单体痛点 | 难扩展、难维护、难部署、技术锁定 |
| 微服务优势 | 独立部署、独立扩展、技术异构、故障隔离 |
| 核心组件 | 注册中心 + 配置中心 + API 网关 + 服务调用 |
| 服务发现 | 服务启动注册 → 调用方动态发现 → 负载均衡 |
| 先决条件 | 团队、DevOps、监控 → 缺少任何一环都不要上微服务 |

**2. 什么是服务发现？为什么微服务需要它？**

**3. 一个 5 人创业团队，用户量 5000 人/天。项目经理说要"微服务化"。你的建议是什么？为什么？**

---

**答案提示**：1→难扩展（单体只能整体扩，微服务可单服务扩）；难维护（100 万行代码耦合，微服务分了代码仓库和数据库）。2→微服务实例的 IP 和端口是动态变化的（扩缩容、故障迁移），调用方不能写死 IP。服务发现通过注册中心动态获取当前可用的实例列表。3→建议暂不微服务化。5 人团队没有足够的 DevOps 能力，5000 DAU 单体完全可以支撑。先用模块化单体（按 package 分层），业务验证后再逐步拆分。下一章——微服务通信。
