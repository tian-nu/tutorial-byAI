## 47.1 什么是重构

> **📖 术语：重构（Refactoring）**——在不改变代码外部行为的前提下，改善代码的内部结构。就像重新布置房间：家具还是那些家具，但摆放更合理、动线更流畅、找东西更方便。

**一个比喻**：

你搬进新家时，随手把衣服塞进了衣柜。过了一个月，衣柜乱成一团——找一件T恤要翻5分钟。

**重构就是**：把衣柜清空，衣服分类叠好，按季节排列——**衣服还是那些衣服，但找起来快多了。**

在代码中：
- 变量名从 `x` 改成 `user_count` → 重构
- 把一段重复的代码抽成一个函数 → 重构
- 把一堆 `if/elif` 简化为查字典 → 重构
- 把"注册用户"改成"注册管理员" → **不是重构，是改功能**

**重构的黄金法则**：外面看起来一模一样，里面变得更好。

## 47.2 AI擅长重构的四大场景

### 场景1：重命名（Rename）

这是最常见的重构。变量名从 `a`、`b`、`tmp` 改成有意义的名称。

#### ❌ 重构前
```python
def calc(a, b):
    r = 0
    for i in a:
        if i > b:
            r += i
    return r
```

#### ✅ 重构后（让AI重命名）
```python
def sum_greater_than(numbers, threshold):
    total = 0
    for number in numbers:
        if number > threshold:
            total += number
    return total
```

**Prompt**：
```
帮我把这段代码中的变量名改成有意义的名称，保持功能不变。
```

### 场景2：提取函数（Extract Function）

一段代码太长，逻辑揉在一起，难以理解。拆成多个小函数。

#### ❌ 重构前
```python
def process_order(order):
    # 验证订单
    if not order.get('user_id'):
        return "错误：缺少用户ID"
    if not order.get('items'):
        return "错误：订单为空"
    
    # 计算总价
    total = 0
    for item in order['items']:
        price = item.get('price', 0)
        quantity = item.get('quantity', 1)
        total += price * quantity
    
    # 应用折扣
    if total > 1000:
        total *= 0.9  # 满1000打9折
    elif total > 500:
        total *= 0.95  # 满500打95折
    
    # 生成订单号
    import random
    import string
    order_id = 'ORD-' + ''.join(random.choices(string.digits, k=8))
    
    return {
        'order_id': order_id,
        'user_id': order['user_id'],
        'total': round(total, 2),
        'status': '已确认'
    }
```

这个函数做了4件事：验证、计算、打折、生成订单号。**太长了，读起来累。**

#### ✅ 重构后（让AI提取函数）

```python
def validate_order(order):
    """验证订单数据"""
    if not order.get('user_id'):
        return "错误：缺少用户ID"
    if not order.get('items'):
        return "错误：订单为空"
    return None  # 验证通过


def calculate_total(items):
    """计算订单总价"""
    total = 0
    for item in items:
        price = item.get('price', 0)
        quantity = item.get('quantity', 1)
        total += price * quantity
    return total


def apply_discount(total):
    """根据金额应用折扣"""
    if total > 1000:
        return total * 0.9
    elif total > 500:
        return total * 0.95
    return total


def generate_order_id():
    """生成订单号"""
    import random
    import string
    return 'ORD-' + ''.join(random.choices(string.digits, k=8))


def process_order(order):
    """处理订单（主函数）"""
    error = validate_order(order)
    if error:
        return error
    
    total = calculate_total(order['items'])
    total = apply_discount(total)
    
    return {
        'order_id': generate_order_id(),
        'user_id': order['user_id'],
        'total': round(total, 2),
        'status': '已确认'
    }
```

**效果**：主函数 `process_order` 从30行变成了10行，每个子函数只做一件事，清晰明了。

**Prompt**：
```
帮我把这段代码重构，把不同职责的代码提取成独立的函数。保持功能完全不变。
```

### 场景3：消除重复代码（DRY — Don't Repeat Yourself）

如果同一段代码出现了两次以上，就应该提取出来。

#### ❌ 重构前（重复代码）
```python
def calculate_student_score(homework, exam, bonus):
    total = homework * 0.3 + exam * 0.7
    if total > 90:
        level = 'A'
    elif total > 80:
        level = 'B'
    elif total > 70:
        level = 'C'
    else:
        level = 'D'
    return total + bonus, level


def calculate_teacher_score(teaching, research, bonus):
    total = teaching * 0.5 + research * 0.5
    if total > 90:
        level = 'A'
    elif total > 80:
        level = 'B'
    elif total > 70:
        level = 'C'
    else:
        level = 'D'
    return total + bonus, level
```

**重复的等级判断逻辑**出现了两次。

#### ✅ 重构后（消除重复）

```python
def get_level(score):
    """根据分数返回等级"""
    if score > 90:
        return 'A'
    elif score > 80:
        return 'B'
    elif score > 70:
        return 'C'
    else:
        return 'D'


def calculate_student_score(homework, exam, bonus):
    total = homework * 0.3 + exam * 0.7
    return total + bonus, get_level(total)


def calculate_teacher_score(teaching, research, bonus):
    total = teaching * 0.5 + research * 0.5
    return total + bonus, get_level(total)
```

**Prompt**：
```
这两段代码有重复的等级判断逻辑，帮我把重复部分提取出来，保持功能不变。
```

### 场景4：简化条件表达式

有时候 `if/elif/else` 写得太复杂，可以用更简洁的方式替代。

#### ❌ 重构前（复杂条件）
```python
def get_status_text(code):
    if code == 0:
        return "待处理"
    elif code == 1:
        return "处理中"
    elif code == 2:
        return "已完成"
    elif code == 3:
        return "已取消"
    else:
        return "未知状态"
```

#### ✅ 重构后（用字典映射）
```python
def get_status_text(code):
    status_map = {
        0: "待处理",
        1: "处理中",
        2: "已完成",
        3: "已取消",
    }
    return status_map.get(code, "未知状态")
```

**Prompt**：
```
帮我把这段if/elif代码简化，用更优雅的方式实现。
```

## 47.3 重构Prompt模板

把要重构的代码发给AI，使用这个模板：

```
帮我重构这段代码，要求：
1. 目标：[重命名变量 / 提取函数 / 消除重复 / 简化条件 / 你自己描述]
2. 保持功能完全不变（外部行为一模一样）
3. 保留原有注释
4. 重构后请解释你做了哪些改动，为什么
```

## 47.4 重构后验证：一定要跑测试！

**重构完必须验证！** 重构的黄金法则是"功能不变"，但你怎么知道功能真的没变？

### 如果你有测试（第44章）

```bash
python -m unittest test_your_module.py
```

所有测试应该仍然通过。如果有测试失败，说明重构**改了功能**——那就不叫重构了，叫"改出Bug了"。

### 如果你没有测试

至少手工跑一下核心功能，确认和之前一样。

### 让AI帮你验证

```
我重构了这段代码，请帮我检查重构后功能是否和原来一致。如果有不一致的地方，请指出。
[贴重构前代码]
[贴重构后代码]
```

## 47.5 什么时候应该重构，什么时候不应该

| 应该重构 | 不应该重构 |
|----------|-----------|
| 代码重复了两次以上 | 代码只写了一次，也没打算改 |
| 函数超过50行，难以理解 | 这段代码下周就要删掉 |
| 变量名让你猜不透含义 | 项目马上要交付，重构风险大 |
| 准备加新功能，但现有代码太乱 | 你对代码还不够熟悉，不确定行为 |

**一句话**：重构是"磨刀不误砍柴工"，但也要在合适的时候磨。不要为了重构而重构。

## 47.6 重构的"安全网"：Git

重构前，**务必提交一次代码**（或者至少保存当前状态）。

```bash
git add .
git commit -m "重构前的状态"
```

如果重构后发现问题，可以一键回退：

```bash
git checkout -- .  # 撤销所有修改
```

或者更安全的方式——开一个新分支：

```bash
git checkout -b refactor/cleanup
# 在新分支上重构，不影响主分支
```

**Git是重构的安全网。** 有了它，你可以大胆重构，不怕改坏。

1. 重构和"改功能"有什么区别？举一个"重构"的例子和一个"改功能"的例子。
2. AI帮你在重构中提取了一个函数，重构后你应该优先做什么来验证？
3. 重构前为什么建议提交一次Git？如果重构改坏了怎么办？

> **答案**：
> 1. 重构不改功能，只改结构。重构例子：把变量名 `x` 改成 `user_count`、把重复代码提取成函数。改功能例子：把"注册用户"改成"注册管理员"、把折扣从8折改成7折。前者外面看不出来，后者外面能看出来。
> 2. 跑测试（或手工验证核心功能），确认重构后功能和原来一致。如果测试失败，说明重构引入了Bug，需要修正。
> 3. Git是"安全网"——重构前提交代码，如果重构改坏了，可以 `git checkout -- .` 一键回退到重构前的状态。更好的做法是开新分支，在新分支上重构。
