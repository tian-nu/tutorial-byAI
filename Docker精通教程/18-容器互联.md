## 我在做什么？

上一章说了，默认 bridge 网络里容器之间**只能用 IP 通信**。但 IP 是会变的——容器重启就可能换 IP。你总不能每次重启都去更新配置文件里的 IP 地址吧？

这一章教你**自定义网络** *此术语见附录C*。创建之后，Docker 会为网络里的每个容器提供**自动 DNS 解析** *此术语见附录C*——容器名就是域名，直接用容器名就能相互访问。从此告别硬编码 IP。

学完这一章，你能：
- 创建自定义 bridge 网络
- 把多个容器加入同一网络，用容器名互相访问
- 搭建一个 Web + 数据库的互联架构
- 理解 `--link` 为什么被废弃以及为什么不该用它

## 一、自定义网络：给容器装上"电话簿"

### 比喻

默认 bridge 网络像一个没有电话簿的公寓楼——你知道邻居住在哪个房间号（IP），但房间号可能随时换。

自定义网络则给每个住户配了一个永久门牌——**容器名**。不管房间号怎么变，你只要喊名字就能找到它。

### 创建自定义网络

```bash
docker network create mynet
# 输出：mynet 的网络 ID

docker network ls
# NETWORK ID     NAME      DRIVER    SCOPE
# abc123def456   bridge    bridge    local
# def456abc789   host      host      local
# ghi789jkl012   none      null      local
# xyz999...      mynet     bridge    local    ← 你创建的网络
```

注意 DRIVER 是 `bridge`——自定义网络本质上也是 bridge 网络，但比默认的多了**内置 DNS 服务器**。

### 想多一点：自定义网络比默认 bridge 多什么？

| 特性 | 默认 bridge | 自定义 bridge |
|------|-----------|-------------|
| 容器间通信 | 仅 IP | IP + 容器名 |
| DNS 解析 | 无 | 自动，容器名 → IP |
| IP 分配 | 动态，重启可变化 | 动态，但 DNS 自动更新 |
| 网络隔离 | 所有容器在同一个网段 | 不同网络之间完全隔离 |
| 可配置子网 | 固定 172.17.0.0/16 | 创建时可自定义 |

## 二、实战：Web + 数据库通过容器名通信

### 步骤 1：创建自定义网络

```bash
docker network create app-net
```

### 步骤 2：启动数据库容器

```bash
docker run -d \
  --name db \
  --network app-net \
  -e MYSQL_ROOT_PASSWORD=your_root_password_here \
  -e MYSQL_DATABASE=appdb \
  mysql:8.0
```

注意：
- `--name db` —— 容器名叫 `db`
- `--network app-net` —— 加入自定义网络 `app-net`

### 步骤 3：启动一个测试容器，验证能用容器名 ping 通

```bash
# 启动一个 Alpine 容器，也加入 app-net
docker run --rm \
  --name tester \
  --network app-net \
  alpine:latest ping -c 3 db
```

输出：

```
PING db (172.18.0.2): 56 data bytes
64 bytes from 172.18.0.2: seq=0 ttl=64 time=0.123 ms
64 bytes from 172.18.0.2: seq=1 ttl=64 time=0.098 ms
64 bytes from 172.18.0.2: seq=2 ttl=64 time=0.105 ms

--- db ping statistics ---
3 packets transmitted, 3 packets received, 0% packet loss
```

关键：`ping db` 而不是 `ping 172.18.0.2`。容器名 `db` 被 DNS 解析成了 IP `172.18.0.2`。

### 步骤 4：启动 Web 应用，连接数据库

现在启动一个应用容器，通过容器名连接数据库：

```bash
docker run -d \
  --name webapp \
  --network app-net \
  -p 8080:8080 \
  -e DB_HOST=db \
  -e DB_PORT=3306 \
  -e DB_USER=root \
  -e DB_PASSWORD=your_root_password_here \
  -e DB_NAME=appdb \
  your-webapp-image
```

注意 `DB_HOST=db`——不是 `127.0.0.1`，不是 `localhost`，就是容器名 `db`。Docker 的内置 DNS 会自动把 `db` 解析为数据库容器的 IP。

### 验证连通性

```bash
# 从 webapp 容器测试数据库连接
docker exec webapp sh -c "nc -zv db 3306"
# db (172.18.0.2:3306) open
```

## 三、深入：Docker 的内置 DNS 是怎么工作的？

### 架构

当你创建自定义网络时，Docker 会在该网络里启动一个**嵌入式 DNS 服务器**（127.0.0.11）。每个加入该网络的容器，其 `/etc/resolv.conf` 都会被自动配置指向这个 DNS 服务器。

```bash
docker exec db cat /etc/resolv.conf
# nameserver 127.0.0.11
# options ndots:0
```

当容器 A 尝试连接 `db` 时：
1. 容器 A 的 DNS 请求发到 127.0.0.11
2. Docker 内置 DNS 查找 `db` → 找到 IP `172.18.0.2`
3. 返回 IP 给容器 A
4. 容器 A 用 IP 连接容器 B

> 注意：Docker 内置 DNS 只解析容器名，不解析容器名加网络别名之外的任何东西。对于 `example.com` 这种外部域名，Docker 会转发到宿主机的 DNS 服务器。

### 网络别名

一个容器可以有多个名字：

```bash
docker run -d \
  --name myapp \
  --network app-net \
  --network-alias api \
  --network-alias backend \
  nginx:alpine
```

现在同一个容器既可以用 `myapp`、`api`、`backend` 任意一个名字访问。这在做蓝绿部署或 A/B 测试时非常有用。

## 四、`docker network connect`：动态加入网络

容器不一定在启动时就要加入网络。你可以随时把运行中的容器添加到某个网络：

```bash
# 启动一个"裸"容器
docker run -d --name late-joiner nginx:alpine

# 稍后把它加入 app-net
docker network connect app-net late-joiner

# 验证
docker exec late-joiner ping -c 1 db
# 通了！
```

同样，也可以断开连接：

```bash
docker network disconnect app-net late-joiner
```

### 一个容器可以同时属于多个网络

```bash
docker network create net-a
docker network create net-b

docker run -d --name multi-net \
  --network net-a \
  nginx:alpine

docker network connect net-b multi-net

# 现在 multi-net 同时在 net-a 和 net-b 里
docker inspect multi-net -f '{{ json .NetworkSettings.Networks }}' | python3 -m json.tool
```

这让你可以构建复杂的网络拓扑：比如前端容器在 `frontend` 网络，后端容器在 `backend` 网络，API 网关同时连接两个网络。

## 五、`--link`：已经废弃，但你应该认识它

在 Docker 1.9 之前，没有自定义网络，容器间通信只能用 `--link` *此术语见附录C*：

```bash
# ❌ 已废弃的写法
docker run -d --name db mysql:8.0
docker run --name app --link db:database myapp
```

`--link` 会在 `/etc/hosts` 里添加一条记录，把 `database` 指向 `db` 的 IP。

### 为什么废弃？

1. **单向**：`--link` 是单向的——app 可以访问 db，但 db 不能访问 app
2. **静态**：如果 db 容器重启换了 IP，`--link` 不会更新 `/etc/hosts`，通信就断了
3. **依赖链复杂**：多个容器互相链接时，启动顺序必须严格按依赖链
4. **功能已被自定义网络覆盖**：自定义网络提供双向 DNS、自动更新、任意顺序启动

如果你看到旧项目里有 `--link`，把它迁移到自定义网络——新项目直接创建自定义网络。

## 六、常见错误

### 错误 1：容器在默认 bridge 网络，却用容器名访问

```bash
# ❌ 两个容器都在默认 bridge 网络
docker run -d --name mydb mysql:8.0
docker run --name myapp --link mydb:db myimage
# 或者
docker run --name myapp -e DB_HOST=mydb myimage
# myapp 找不到 mydb！
```

❌ 原因：默认 bridge 网络没有 DNS 解析。容器名对默认 bridge 无效。

✅ 正确做法：

```bash
# 创建自定义网络
docker network create mynet

# 两个容器都加入
docker run -d --name mydb --network mynet mysql:8.0
docker run --name myapp --network mynet -e DB_HOST=mydb myimage
# myapp 可以通过 mydb 访问数据库
```

### 错误 2：网络名拼写错误

```bash
docker run --network app-net myimage
# 容器启动成功，但它在默认 bridge 网络里！
```

❌ 原因：`app-net` 不存在，Docker 的默认行为是**回退到 bridge 网络**，而且不报错。

✅ 预防：

```bash
# 先确认网络存在
docker network ls | grep app-net

# 或用 docker compose（它会在网络不存在时报错）
```

### 错误 3：不同网络之间无法通信

```bash
docker network create net-a
docker network create net-b

docker run -d --name ctr-a --network net-a nginx:alpine
docker run -d --name ctr-b --network net-b nginx:alpine

docker exec ctr-a ping ctr-b
# 不通！
```

❌ 原因：不同自定义网络之间是**完全隔离**的。这是安全设计，不是 bug。

✅ 如果需要互通：

```bash
# 方案 1：把 ctr-b 也加入 net-a
docker network connect net-a ctr-b

# 方案 2：把 ctr-a 也加入 net-b
docker network connect net-b ctr-a

# 现在它们都在 net-a 和 net-b 里了
```

### 错误 4：用 `--link` 替代自定义网络

```bash
# ❌ 2024 年了还在用 --link
docker run --link mysql:db myapp
```

`--link` 在 Docker 文档中标注为 legacy，可能在未来版本中彻底移除。而且它不支持动态 IP 更新——这是最致命的缺陷。

## 七、我做得对不对？

### 验证方法

```bash
# 1. 确认自定义网络存在
docker network ls | grep app-net

# 2. 确认容器都在网络中
docker network inspect app-net
# 检查 Containers 字段，确认 db 和 webapp 都在

# 3. 从 webapp ping db（容器名）
docker exec webapp ping -c 2 db
# 预期：能 ping 通，延迟 < 1ms

# 4. 从 db ping webapp（容器名——证明双向 DNS）
docker exec db ping -c 2 webapp
# 预期：也能 ping 通

# 5. 验证 DNS 解析
docker exec webapp nslookup db
# 预期：返回 db 的 IP 地址

# 6. 验证网络隔离
docker run --rm alpine:latest ping db
# 预期：不通（因为不在 app-net 里）
```

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 自定义网络 | `docker network create 网络名` | 内置 DNS，容器名自动解析，推荐所有多容器场景 |
| DNS 解析 | 容器名→IP，127.0.0.11 | 仅限自定义网络，默认 bridge 没有 |
| 网络别名 | `--network-alias 别名` | 一个容器可以有多个名字 |
| 动态加入网络 | `docker network connect/disconnect` | 容器运行时随时加入/退出网络 |
| 多网络 | 一个容器可同时属于多个网络 | 构建复杂网络拓扑（前端/后端隔离） |
| `--link` | 已废弃 | 单向、静态、不更新 IP，用自定义网络替代 |
| 网络隔离 | 不同自定义网络之间不通 | 安全设计，需要互通时用 `docker network connect` |

## 本篇最可能出错的地方及原因

### 1. 忘了加 `--network`，容器掉进默认 bridge

```bash
docker run -d --name db mysql:8.0           # 忘记 --network
docker run -d --name webapp --network app-net myimage
```

`db` 在默认 bridge 网络，`webapp` 在 `app-net`。`webapp` 尝试 `ping db` 永远不通。你排查半天，发现是少写了 `--network`。

**口诀：启动容器时把 `--network` 写在 `--name` 旁边，养成习惯。**

### 2. 搜索"容器互联"找到了 `--link` 的旧教程

你搜"Docker 容器通信"，前几条结果全是 2015 年的博客，教你用 `--link`。你照着做了，虽然能跑，但这是 10 年前的方案。**判断方法：如果教程里没有 `docker network create`，说明它过时了。**

### 3. 容器名和主机名混淆

```bash
# 容器名是 db
docker run --name db --network mynet mysql:8.0

# 在另一个容器里
docker exec app ping db  # ✅ 能通——容器名
docker exec app ping db.mynet  # ❌ 不通——这不是 Docker 的 DNS 格式
```

Docker 的 DNS 只解析容器名（和网络别名），不解析 `容器名.网络名` 这种格式。

### 4. 自定义网络里容器重启后 ping 不通——其实是 DNS 缓存

```bash
docker restart db
# db 的 IP 可能变了

docker exec webapp ping db
# 第一次可能报错，第二次就好了
# 原因：DNS 缓存过期后重新解析，新 IP 生效
```

Docker 内置 DNS 会检测容器 IP 变化并更新。但容器内的应用可能有 DNS 缓存（如 JVM 默认缓存 DNS 30 秒），导致短暂不可用。生产环境应配置连接重试机制。

### 5. 不同 Docker Compose 项目的网络默认不互通

```bash
# 项目 A
docker compose -p proj-a up

# 项目 B
docker compose -p proj-b up
```

Docker Compose 会为每个项目创建独立的默认网络（如 `proj-a_default` 和 `proj-b_default`）。这两个网络之间默认不通。如果需要跨项目通信，需要显式配置外部网络。
