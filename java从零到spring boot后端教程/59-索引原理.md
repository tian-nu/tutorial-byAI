# 第59章 · 索引原理

> "你去图书馆找一本《三体》，最蠢的办法是从第一个书架第一本书开始，一本本翻到最后一架——这叫'全表扫描'。聪明人先去查索引柜里的小卡片，按字母顺序找到'三体'在哪架哪层——这叫'索引查找'。数据库索引和图书馆的卡片柜完全一样，只是卡片柜里的卡片换成了B+树的节点。"

---

## 59.1 没有索引的世界

假设 `users` 表有100万条数据，我们查一条：

```sql
SELECT * FROM users WHERE username = 'zhangsan';
```

**没有索引时**：MySQL从磁盘上读取第一行，比一下username是不是'zhangsan'；不是，读下一行……一直读到第70万行找到了。每次读取都涉及磁盘IO（最慢的操作），100万行可能读几十万次磁盘，耗时数秒。

**有索引时**：MySQL先查索引结构，像翻字典一样定位到'zhangsan'所在的那一页，然后直接跳到数据页读取，只需要几次磁盘IO，耗时毫秒级。

> 📊 没有索引是**O(n)**，有索引是**O(log n)**。当n=100万时，log₂(1,000,000)≈20，而线性查找平均要查50万次。这就是**2万倍**的性能差距。

## 59.2 B+树：索引的底层结构

MySQL InnoDB引擎使用**B+树**作为索引的数据结构。请一步步跟着下面的图示理解。

### 59.2.1 从二分查找说起

如果数据是排好序的，查找'zhangsan'可以这样做：翻到中间，比一下，比'zhangsan'小就往后翻，大了就往前翻，每次排除一半。

但数据存在磁盘上，是分散的数据页，没法直接"翻到中间"。你需要一个结构来快速定位"中间在哪"——B+树就是这个结构。

### 59.2.2 B+树的逐步构造

**第1步：最简单的结构——一个根节点指向叶子节点**

```
         ┌──────────────┐
         │  [100, 200]  │  ← 根节点（非叶子节点，只存索引键值+指针）
         └──┬────┬──────┘
       ┌────┘    └──────┐
       ▼                ▼
┌──────────────┐ ┌──────────────┐
│ 1..100的     │ │ 101..200的   │  ← 叶子节点（存完整数据或主键值+数据指针）
│ 全部数据      │ │ 全部数据      │
└──────────────┘ └──────────────┘
```

非叶子节点存的是**键值+子节点指针**，作用是导航。
叶子节点存的是**真正的数据**。

**第2步：数据多了，加一层**

```
              ┌─────────────────┐
              │    [500]        │  ← 根节点
              └───┬──────┬──────┘
        ┌─────────┘      └──────────┐
        ▼                           ▼
   ┌───────────┐              ┌───────────┐
   │ [100,300] │              │ [700,900] │  ← 内部节点
   └─┬───┬───┬─┘              └─┬───┬───┬─┘
  ┌──┘   │   └──┐          ┌───┘   │   └───┐
  ▼      ▼      ▼          ▼       ▼       ▼
┌────┐┌────┐┌────┐    ┌────┐ ┌────┐ ┌────┐
│1.. ││101.││301.│    │501.│ │701.│ │901.│  ← 叶子节点
│100 ││.300││.500│    │.700│ │.900│ │.1000│
└────┘└────┘└────┘    └────┘ └────┘ └────┘
     ↕ 双向链表连接所有叶子节点 ↕
```

### 59.2.3 B+树的五个关键特性

| 特性 | 含义 | 好处 |
|------|------|------|
| **N叉树** | 每个节点可以有多个子节点（通常几百个） | 树的高度极低，3~4层就能存千万级数据 |
| **数据只在叶子节点** | 非叶子节点只存键值和指针 | 非叶子节点能存更多键值，树更矮 |
| **叶子节点有序且双向链表连接** | 所有叶子节点按索引键值从小到大排序，用双向链表连起来 | 范围查询极快——找到起点，沿着链表往后读就行 |
| **节点对应磁盘页** | 一个B+树节点大小=InnoDB一个数据页（默认16KB） | 一次磁盘IO读取一个完整节点，内存换IO |
| **自平衡** | 插入和删除时自动分裂/合并节点 | 树高度始终稳定，插入删除后查询性能不会退化 |

### 59.2.4 一次B+树查找的过程

查找 `username = 'zhangsan'`：

```
1. 从根节点开始（常驻内存）
                              ┌─────────────────┐
                              │   根节点(内存)     │
                              │ ['lisi','wangwu'] │
                              └───┬─────┬─────┬───┘
                     'zhangsan'在这之间
                                  │
2. 加载中间节点（1次磁盘IO）        ▼
                          ┌─────────────────────┐
                          │   内部节点            │
                          │ ['sunqi','zhangsan'] │
                          └──┬──────────────────┘
                   'zhangsan'在这里
                             │
3. 加载叶子节点（1次磁盘IO）    ▼
                    ┌──────────────────┐
                    │   叶子节点         │
                    │ ['zhangsan', ...] │ ← 找到了！总共2次磁盘IO
                    └──────────────────┘
```

如果是全表扫描，平均需要扫描50万行数据页，假设每页存100行，就是5000次磁盘IO。B+树只需2~3次。这就是为什么索引快。

## 59.3 聚簇索引 vs 非聚簇索引

这是MySQL面试的重灾区，必须搞清楚。

### 59.3.1 聚簇索引（Clustered Index）

**聚簇索引 = 主键索引。叶子节点直接保存整行数据。**

```
B+树结构:
┌─────────────────┐
│   非叶子节点       │  ← 存主键值 + 子节点指针
│   (主键值范围)     │
└────────┬────────┘
         ▼
┌─────────────────┐
│   叶子节点         │  ← 每个叶子页里存的是完整的行数据
│ id=1 | 张三 | 25  │
│ id=2 | 李四 | 30  │
│ id=3 | 王五 | 28  │
└─────────────────┘
```

关键点：
- InnoDB中，**每张表有且仅有一个聚簇索引**，通常是主键
- 如果没有显式定义主键，InnoDB会用第一个UNIQUE NOT NULL索引，再找不到就建一个隐藏的row_id做聚簇索引
- 数据行物理上按主键顺序存储——这就是为什么用 `AUTO_INCREMENT` 主键性能好：新数据永远追加到末尾，不会导致页分裂

### 59.3.2 非聚簇索引（Secondary Index，二级索引）

**你手动创建的普通索引 = 非聚簇索引。叶子节点不存完整数据，只存索引列值 + 主键值。**

```
非聚簇索引（username索引）:
┌─────────────────┐
│   非叶子节点       │  ← 存 username 值 + 指针
└────────┬────────┘
         ▼
┌─────────────────────┐
│   叶子节点            │
│ 'lisi'    → id=2    │  ← 只存 (username, 主键)
│ 'wangwu'  → id=3    │
│ 'zhangsan'→ id=1    │   没有其他列的数据！
└─────────────────────┘
```

### 59.3.3 回表

查 `SELECT * FROM users WHERE username = 'zhangsan'` 时：

1. **索引查找**：在 username 索引的B+树中定位到 `'zhangsan'`，拿到 `id=1`
2. **回表**：拿着 `id=1`，再去主键索引（聚簇索引）的B+树中查找一次，取出完整行数据

```
username索引B+树                主键索引B+树
┌──────────────┐              ┌──────────────────┐
│ 'zhangsan'   │              │ id=1              │
│     ↓        │   回表       │ 张三, 25,          │
│   id=1 ──────┼──────────────→ zhangsan@xx.com   │
└──────────────┘              └──────────────────┘
```

**回表是性能开销**。如果查询只需要索引列+主键，就不需要回表——这叫**覆盖索引**，第60章细讲。

> 📌 一张图总结：
> ```
> 聚簇索引（主键）：叶子 = 整行数据
> 非聚簇索引（其他）：叶子 = 索引列 + 主键值 → 回表取完整数据
> ```

## 59.4 联合索引与最左前缀原则

### 59.4.1 什么是联合索引

在多个列上建一个索引：

```sql
CREATE INDEX idx_a_b_c ON table_name (a, b, c);
```

这个索引的B+树按 `(a, b, c)` 的组合值排序。先按a排序，a相同的按b排序，b相同的按c排序。

```
联合索引 (a, b, c) 的叶子节点顺序示意：
┌───────────────────┐
│ (1, 10, 'x')      │
│ (1, 20, 'y')      │  ← a=1 时按b排序
│ (1, 20, 'z')      │  ← b相同时按c排序
│ (2, 5,  'a')      │
│ (2, 15, 'b')      │  ← a=2 时按b排序
│ (3, 1,  'c')      │
└───────────────────┘
```

### 59.4.2 最左前缀原则

联合索引 `(a, b, c)` 相当于创建了三个索引：
- `(a)` ——索引可用 ✅
- `(a, b)` ——索引可用 ✅
- `(a, b, c)` ——索引可用 ✅

但**单独查 b 或 c 或 (b, c) 时索引不生效**。

```sql
-- 索引 (a, b, c)

WHERE a = 1                    -- ✅ 使用 (a) 部分
WHERE a = 1 AND b = 10         -- ✅ 使用 (a, b) 部分
WHERE a = 1 AND b = 10 AND c=5 -- ✅ 使用完整索引
WHERE a = 1 AND c = 5          -- ⚠️ 只用 (a)，b被跳过，c用不了
WHERE b = 10                   -- ❌ 跳过a，索引完全失效
WHERE b = 10 AND c = 5         -- ❌ 跳过a，索引完全失效
```

**生活比喻**：联合索引像一本按"省→市→区"排序的电话簿。你只能从省开始查，跳过省直接查市是查不到的——因为书是按省排列的，不同省的"海淀区"散布在书的不同位置。

### 59.4.3 范围查询断开最左前缀

```sql
-- 索引 (a, b, c)

WHERE a = 1 AND b > 10 AND c = 5
-- ✅ 用了 (a, b)，但 b 用了范围查询后，c 无法使用索引

WHERE a = 1 AND b = 10 AND c > 5
-- ✅ 用了完整的 (a, b, c)
```

> 💡 等值查询可以"跨列匹配"，范围查询只能"匹配到本列为止"。所以联合索引中，**把范围查询的列放最后**。

## 59.5 索引的代价

索引不是免费的：

| 代价 | 说明 |
|------|------|
| **空间** | 每个索引都是一棵独立的B+树，占用磁盘空间 |
| **写入性能** | INSERT/UPDATE/DELETE 时需要同时维护所有索引的B+树（插入、分裂、合并） |
| **查询优化器负担** | 索引太多，MySQL选索引反而可能选错 |

**经验法则**：一张表的索引数量控制在**5个以内**。每加一个索引都要问自己：这个查询真的需要吗？

## 本章小结

| 概念 | 要点 |
|------|------|
| B+树 | InnoDB的索引结构，N叉自平衡树，数据只在叶子节点 |
| 聚簇索引 | 主键索引，叶子节点存完整行数据，表有且仅有一个 |
| 非聚簇索引 | 二级索引，叶子节点存索引列+主键值，其他列需要回表 |
| 回表 | 非聚簇索引找到主键后，再去聚簇索引取完整数据 |
| 联合索引 | 多列索引，按定义顺序排序 |
| 最左前缀 | 联合索引从最左列开始匹配，跳过左列则索引失效 |
| 索引代价 | 占用空间，拖慢写入 |

## 自测题

1. **为什么InnoDB推荐使用自增整数作为主键？**

2. **联合索引 `(name, age, city)` 中，以下SQL能用到索引吗？**
   ```sql
   -- A
   SELECT * FROM users WHERE name = '张三' AND city = '北京';
   -- B
   SELECT * FROM users WHERE age > 25 AND name = '张三';
   -- C
   SELECT * FROM users WHERE name LIKE '张%' AND age = 25;
   ```

3. **一张表有500万行数据，没有索引时查一条数据要多少时间？建了索引之后呢？描述整个过程。**

<details>
<summary>参考答案（做完再看）</summary>

1. 自增整数做聚簇索引主键有两个好处：第一，新数据永远追加到B+树的末尾，不会在中间插入导致页分裂和页空间浪费；第二，主键越小，二级索引叶子节点中存的主键值越小（因为二级索引都要存主键做回表），整个二级索引的B+树也更小，查询更快。UUID做为主键正好相反——随机插入导致频繁页分裂，且128位的UUID让二级索引膨胀严重。

2. A：能用，但只用到了(name)部分，city被跳过的age挡住无法使用索引。B：能用，MySQL优化器会自动调整条件顺序，等价于 `WHERE name='张三' AND age>25`，用了(name, age)部分。C：能用(name, age)部分，LIKE '张%'是范围查询，所以age可以使用（和B的情况类似——等值在前，范围在后）。

3. 没有索引时，MySQL全表扫描，500万行涉及大量磁盘IO，可能耗时数秒到几十秒。建了索引后，B+树高度约3-4层，需3-4次磁盘IO定位叶子节点，然后读取数据页，整体耗时在1-5毫秒。性能提升约1000-10000倍。过程：MySQL在索引B+树中从根节点开始，每次根据键值比较选择子节点指针，到达叶子节点后如果是覆盖索引直接返回，否则用主键值回表取完整行数据。
