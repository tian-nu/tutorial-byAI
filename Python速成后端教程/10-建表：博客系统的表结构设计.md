## 三、分步操作

### 步骤 1：博客系统需要哪些表？——从需求反推

在上一章，我们在 `blog.db` 里创建了一张 `test` 表练手。现在我们要来真的了——**为博客系统设计真正的数据表**。

在开始设计之前，先问自己一个问题：**一个博客系统里，有哪些"东西"需要存到数据库？**

闭上眼睛想一想你平时看的技术博客，页面上有什么：
- 有**文章**：标题、正文、发布时间
- 有**作者**：每篇文章都是一个具体的人写的
- 有**评论**：文章下面有人留言

这三种"东西"就是我们需要存储的数据类型。每种类型对应数据库里的一张表。

#### 为什么是"三张表"而不是"一张大表"？

❌ **错误思路——一张大表包罗万象**：

```
┌────┬──────────┬──────────┬────────┬─────────────────────┬──────────────────┐
│ id │  title   │  author  │ content│     created_at      │    comment_text   │
├────┼──────────┼──────────┼────────┼─────────────────────┼──────────────────┤
│ 1  │ Python.. │ 小明     │ 今天.. │ 2026-06-01 10:00:00 │ 写得太好了！     │
│ 1  │ Python.. │ 小明     │ 今天.. │ 2026-06-01 10:00:00 │ 我也是这么想的   │
│ 2  │ FastAPI  │ 小红     │ 后端.. │ 2026-06-02 14:30:00 │ 学到了！         │
└────┴──────────┴──────────┴────────┴─────────────────────┴──────────────────┘
```

这么做的问题很明显：
1. **数据重复**：文章 ID=1 的标题、作者、内容，因为下面有两条评论，就重复存了两遍。硬盘空间浪费不说，万一哪天要改标题，你得把所有包含这条文章的行全部改一遍。
2. **信息混乱**：作者"小明"的名字存在文章表里，但"小明"还有自己的密码、昵称、注册时间，这些信息该往哪塞？
3. **不可扩展**：如果以后要加"点赞"功能呢？再加一列？那表就越来越宽，越来越乱。

✅ **正确思路——每种"实体"一张表**：

- **用户表（users）**：专门存"人"的信息
- **文章表（posts）**：专门存"文章"的信息
- **评论表（comments）**：专门存"评论"的信息

每张表只管自己的事，表之间通过**外键**来建立联系（后面会详细讲）。

### 步骤 2：用户表（users）——先有"人"

用户表是整个系统的"地基"。没有用户，就没有文章；没有文章，就没有评论。所以**用户表必须最先建**。

#### 字段设计

| 字段名 | 类型 | 含义 | 约束 |
|--------|------|------|------|
| `id` | INTEGER | 用户唯一编号 | PRIMARY KEY（主键，自动递增） |
| `username` | TEXT | 登录用户名 | NOT NULL, UNIQUE（不能空，不能重复） |
| `password_hash` | TEXT | 密码哈希值 | NOT NULL（不能空） |
| `nickname` | TEXT | 显示昵称 | NOT NULL（不能空） |
| `created_at` | TEXT | 注册时间 | NOT NULL, DEFAULT CURRENT_TIMESTAMP |

**逐字段解释**：

- **`id`**：主键（PRIMARY KEY）。每一行数据的"身份证号"，全局唯一。你以后说"把 3 号用户的昵称改成'小刚'"，数据库就靠 `id=3` 精准定位到这一行。`INTEGER PRIMARY KEY` 在 SQLite 中还有一个特殊效果：如果插入时不指定 id，SQLite 会自动给它分配一个递增的数字（1, 2, 3, 4...）。

- **`username`**：登录时用的用户名。`UNIQUE` 约束保证"不能有两个人用同一个用户名注册"——就像你注册微信时，如果提示"该用户名已被占用"，就是数据库的 `UNIQUE` 约束在起作用。

- **`password_hash`**：重要！这里存的**不是**密码原文，而是密码的"哈希值"。什么是哈希？暂时理解为"密码经过一套数学运算后的乱码"。比如 `123456` 经过哈希后变成 `$2b$12$LJ3m4...`（一串看不懂的东西）。**存哈希值的原因是：即使数据库被黑客偷走，他也拿不到用户的真实密码。** 此术语需进附录。

- **`nickname`**：显示用的昵称，可以和用户名不同。比如用户名是 `zhangsan`，昵称可以是"张三"。

- **`created_at`**：注册时间。`DEFAULT CURRENT_TIMESTAMP` 的意思是：如果插入时不指定这个字段，自动填入当前时间。

#### 建表 SQL

```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    nickname TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

> **`AUTOINCREMENT` 是什么？** 它是 `INTEGER PRIMARY KEY` 的加强版——保证 ID 只增不减。即使你删除了 ID=3 的用户，下一个新用户也不会再用 3，而是从 4 开始。这在大多数场景下是更安全的选择。

### 步骤 3：文章表（posts）——文章属于谁？

文章是博客系统的核心。每篇文章由某个用户撰写，所以文章表需要"记住"作者是谁。

#### 字段设计

| 字段名 | 类型 | 含义 | 约束 |
|--------|------|------|------|
| `id` | INTEGER | 文章唯一编号 | PRIMARY KEY AUTOINCREMENT |
| `title` | TEXT | 文章标题 | NOT NULL |
| `content` | TEXT | 文章正文 | NOT NULL |
| `author_id` | INTEGER | 作者的用户 ID | NOT NULL, FOREIGN KEY → users(id) |
| `created_at` | TEXT | 发布时间 | NOT NULL, DEFAULT CURRENT_TIMESTAMP |
| `updated_at` | TEXT | 最后修改时间 | NOT NULL, DEFAULT CURRENT_TIMESTAMP |

**逐字段解释**：

- **`title`** 和 **`content`**：跟第 07 章内存版 CRUD 里的字段一样，只是这次它们要存到硬盘上了。

- **`author_id`**：这是本章最关键的字段。它存的是一个数字——用户在 `users` 表中的 `id`。比如 `author_id = 3` 的意思是"这篇文章是 3 号用户写的"。**关键：`author_id` 本身只是一个数字，它和 `users` 表之间的关系是通过外键约束来保证的。** 此术语需进附录。

- **`created_at`** 和 **`updated_at`**：两个时间戳。`created_at` 记录文章什么时候发布，`updated_at` 记录文章最后一次修改是什么时候。这两个字段分开存，是因为"发布时间"和"修改时间"是两个不同的概念——你可以今天发布一篇文章，明天修改它，发布时间不变，但修改时间变成了明天。

#### 建表 SQL

```sql
CREATE TABLE posts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    author_id INTEGER NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (author_id) REFERENCES users (id)
);
```

**最后一行 `FOREIGN KEY (author_id) REFERENCES users (id)`** 就是外键约束。它的意思是："`author_id` 这个字段的值，必须在 `users` 表的 `id` 列里存在"。翻译成人话：**"你不能说一篇文章是'999 号用户'写的，除非 999 号用户真的存在。"** 此术语需进附录。

### 步骤 4：评论表（comments）——评论属于谁、属于哪篇文章？

评论最复杂，因为它同时和**两**张表有关系：一条评论是"某个人"写的，并且发布在"某篇文章"下面。

#### 字段设计

| 字段名 | 类型 | 含义 | 约束 |
|--------|------|------|------|
| `id` | INTEGER | 评论唯一编号 | PRIMARY KEY AUTOINCREMENT |
| `content` | TEXT | 评论内容 | NOT NULL |
| `post_id` | INTEGER | 所属文章 ID | NOT NULL, FOREIGN KEY → posts(id) |
| `author_id` | INTEGER | 评论者用户 ID | NOT NULL, FOREIGN KEY → users(id) |
| `created_at` | TEXT | 评论时间 | NOT NULL, DEFAULT CURRENT_TIMESTAMP |

**逐字段解释**：

- **`content`**：评论正文。通常比文章内容短得多——几百字以内。

- **`post_id`**：这条评论挂在哪篇文章下面。比如 `post_id = 5` 表示"这是第 5 号文章的一条评论"。

- **`author_id`**：谁写的这条评论。跟文章表的 `author_id` 一样，指向 `users` 表的 `id`。

- **`created_at`**：评论时间。评论通常不需要 `updated_at`——大部分博客系统不允许用户修改已发出的评论（或者修改后会有"已编辑"标记）。

#### 建表 SQL

```sql
CREATE TABLE comments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content TEXT NOT NULL,
    post_id INTEGER NOT NULL,
    author_id INTEGER NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (post_id) REFERENCES posts (id),
    FOREIGN KEY (author_id) REFERENCES users (id)
);
```

注意：评论表有**两个**外键约束——一个指向 `posts`，一个指向 `users`。这是完全合法的，也是一个很常见的模式。你可以把它想象成：一条评论"左手拉着文章，右手拉着用户"。

### 步骤 5：外键是什么？——用身份证号来理解

在步骤 3 和 4 中，你看到了 `FOREIGN KEY (xxx) REFERENCES yyy (id)` 这种写法。现在我们来深入理解外键。

#### 比喻：身份证号

假设你去银行办卡。柜员让你填表，其中有一栏是"紧急联系人身份证号"。你随手填了一个 `123456789012345678`。柜员在系统里一查，说："这个身份证号不存在，你不能填一个假号码。"

**外键就是数据库里的"身份证号校验器"。**

- `posts` 表里的 `author_id` 就像"作者身份证号"。你填 `author_id = 5`，数据库就会去 `users` 表里查："有没有 ID = 5 的用户？"如果有，通过；如果没有，拒绝。
- `comments` 表里的 `post_id` 就像"文章身份证号"。你填 `post_id = 99`，数据库就去 `posts` 表里查。"99 号文章存在吗？如果不存在，拒绝。"

**外键约束的核心价值**：保证**数据完整性（Data Integrity）**。此术语需进附录。它确保你的数据不会出现"幽灵引用"——比如一篇文章的作者 ID 指向一个不存在的用户，或者一条评论挂在不存在的一篇文章下面。

没有外键约束的世界：
```
posts 表：
┌────┬──────────┬───────────┐
│ id │  title   │ author_id │
├────┼──────────┼───────────┤
│ 1  │ Python.. │ 999       │  ← 999 号用户？根本没这个人！
└────┴──────────┴───────────┘
```

有了外键约束，插入 `author_id = 999` 时数据库直接报错，拒绝插入。**数据从一开始就是干净的。**

#### 一张图看懂三张表的关系

```

┌────┬──────────┐          ┌────┬───────┬───────────┐  ┌────┬─────────┬─────────┬───────────┐
│ id │ username │          │ id │ title │ author_id │  │ id │ content │ post_id │ author_id │
├────┼──────────┤          ├────┼───────┼───────────┤  ├────┼─────────┼─────────┼───────────┤
│ 1  │ alice    │◄─────────│ 1  │ Py..  │     1     │◄─│ 1  │ 很棒！  │    1    │     2     │──┐
│ 2  │ bob      │◄──┐      │ 2  │ Fast..│     2     │  │ 2  │ 赞同！  │    1    │     1     │  │
```

### 🤔 想多一点：为什么一定要用外键？——"数据完整性"的真实代价

你可能会想："我在代码里注意一下不就行了？每次插入文章前，先检查一下作者 ID 是否存在。为什么还要在数据库层面加外键约束？"

**因为代码会出错，但数据库不会撒谎。**

假设你的博客系统有 100 个接口，其中 3 个涉及插入文章。你在 2 个接口里加了作者 ID 的校验，第 3 个接口忘了加。某天，一个 bug 导致第 3 个接口插入了一条 `author_id = 999` 的文章。此后，你所有"根据作者 ID 查用户"的代码都会在这条脏数据上崩溃。

**外键约束就是把"校验"这件事从代码层下沉到数据库层。** 数据库是最后一道防线——不管你的代码写得有多烂，不管有多少个接口操作数据，数据库都会在数据写入硬盘之前做最后一次检查。就像机场安检：你可以在登机口排队（代码层校验），但过安检门（数据库外键约束）时还是要再过一遍——两道防线比一道更安全。

**代价是什么？** 外键约束会让插入和删除操作稍微变慢——因为数据库需要额外查一次被引用的表。但这个代价在个人博客这种规模下完全可以忽略不计。为了那一点点性能提升而放弃数据完整性，相当于为了省 5 毛钱的车费走路 10 公里——不值得。

**结论**：在本教程中，所有表关系都加外键约束。等你以后做高并发秒杀系统时，再考虑"要不要去掉外键用代码保证"也不迟。

### 步骤 6：在 SQLite 中执行建表语句

现在把前面设计的三张表真正建出来。先确保你位于项目根目录（能看到 `blog.db` 文件的那一层）。

#### 重要：建表顺序不能乱！

**必须先建被引用的表，再建引用别人的表。** 因为创建 `posts` 表时，SQLite 需要确认 `users` 表已经存在——否则 `FOREIGN KEY (author_id) REFERENCES users (id)` 找不到 `users`，会报错。

✅ **正确顺序**：`users` → `posts` → `comments`

❌ **错误顺序**：先建 `posts` → 报错 `no such table: users`

#### 方式一：SQLite 命令行（推荐——直观）

打开终端，进入 SQLite：

```bash
sqlite3 blog.db
```

然后依次执行三条建表语句。**注意：如果你在教程 09 中创建了 `test` 表，可以先删掉它，保持数据库干净：**

```sql
DROP TABLE IF EXISTS test;
```

然后建表：

```sql
-- 1. 用户表（必须先建！）
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    nickname TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 2. 文章表（引用 users）
CREATE TABLE posts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    author_id INTEGER NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (author_id) REFERENCES users (id)
);

-- 3. 评论表（引用 posts 和 users）
CREATE TABLE comments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content TEXT NOT NULL,
    post_id INTEGER NOT NULL,
    author_id INTEGER NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (post_id) REFERENCES posts (id),
    FOREIGN KEY (author_id) REFERENCES users (id)
);
```

**每执行一条，如果什么都没发生，`sqlite>` 提示符又回来了——成功了。** 如果报错，仔细看错误信息，通常是指出哪个表不存在（说明建表顺序错了）或者 SQL 语法错误（少了逗号、括号不匹配等）。

#### 方式二：Python 交互环境

```python
import sqlite3
conn = sqlite3.connect("blog.db")
cursor = conn.cursor()

# 删除上一章的测试表（如果存在）
cursor.execute("DROP TABLE IF EXISTS test;")

# 依次创建三张表
cursor.execute("""
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    nickname TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);
""")

cursor.execute("""
CREATE TABLE posts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    author_id INTEGER NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (author_id) REFERENCES users (id)
);
""")

cursor.execute("""
CREATE TABLE comments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content TEXT NOT NULL,
    post_id INTEGER NOT NULL,
    author_id INTEGER NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (post_id) REFERENCES posts (id),
    FOREIGN KEY (author_id) REFERENCES users (id)
);
""")

conn.commit()
print("三张表创建成功！")
```

### ⚠️ 关键坑点：SQLite 默认不开启外键约束！

这是一个**极高概率踩坑的点**。SQLite 出于历史兼容性原因，默认**不执行**外键约束检查。也就是说，即使你写了 `FOREIGN KEY (author_id) REFERENCES users (id)`，如果你不手动开启外键，SQLite 照样允许你插入一个不存在的作者 ID。

**验证这个坑**：

```sql
-- 不开启外键的情况下
INSERT INTO posts (title, content, author_id) VALUES ('测试', '内容', 999);
-- 成功了！999 号用户根本不存在，但 SQLite 没有拦截！
```

**为什么 SQLite 这么做？** 这是历史遗留问题。SQLite 最早的设计目标之一就是"尽量兼容旧版本"，而外键约束是后来才加的功能。为了不让老用户的代码突然崩掉，SQLite 把它设成了默认关闭。

**解决方案**：每次连接数据库时，执行一条命令开启外键：

```sql
PRAGMA foreign_keys = ON;
```

**在 SQLite 命令行中**：进入 `sqlite>` 后，第一步就执行这条命令。

**在 Python 中**：连接数据库后，第一步就执行：

```python
conn = sqlite3.connect("blog.db")
conn.execute("PRAGMA foreign_keys = ON;")
```

> 🔥 **这是最容易忘记的一步。** 如果你发现插入了一个不存在的作者 ID 却没有报错，99% 是因为忘了 `PRAGMA foreign_keys = ON;`。我们会在下一章 `database.py` 中把这个配置固化到代码里，每次连接自动开启。

**现在，我们先在 SQLite 命令行中开启外键**：

```sql
PRAGMA foreign_keys = ON;
```

验证是否开启成功：

```sql
PRAGMA foreign_keys;
```

如果返回 `1`，说明外键已开启。返回 `0` 说明没开启。

现在来试试外键约束到底管不管用。试着插入一篇"作者 ID 不存在"的文章：

```sql
INSERT INTO posts (title, content, author_id) VALUES ('测试', '内容', 999);
```

❌ 报错！你会看到类似这样的输出：

```
Error: FOREIGN KEY constraint failed
```

**外键约束起作用了！** SQLite 检查了 `author_id = 999`，发现 `users` 表里没有 ID 为 999 的用户，拒绝插入。

再试试插入一条"文章 ID 不存在"的评论：

```sql
INSERT INTO comments (content, post_id, author_id) VALUES ('测试评论', 999, 1);
```

❌ 同样报错：`FOREIGN KEY constraint failed`。因为 `posts` 表里没有 ID 为 999 的文章。

---

### ❌ 常见错误 → ✅ 解决方案

| 错误信息 / 现象 | 原因 | 解决 |
|----------|------|------|
| 插入不存在的作者 ID 成功，没报错 | 忘了执行 `PRAGMA foreign_keys = ON;` | 先执行 `PRAGMA foreign_keys = ON;`，再重新插入 |
| `no such table: users`（建 posts 表时） | 建表顺序错了，先建了 posts | 先建 users，再建 posts，最后建 comments |
| `FOREIGN KEY constraint failed` | 插入的外键值在被引用表中不存在 | 先确认被引用的数据确实存在。比如要先插入用户，再插入文章 |
| `UNIQUE constraint failed: users.username` | 插入的用户名重复了 | 换一个不重复的用户名，或者先 `DELETE FROM users WHERE username = 'xxx'` |
| SQLite 命令行中 `.schema` 看不到表 | 当前数据库文件不是你想的那个 | 确认你 `sqlite3 blog.db` 的目录正确，`dir blog.db` 确认文件存在 |
| `PRAGMA foreign_keys` 返回 0 | 外键确实没开启 | 执行 `PRAGMA foreign_keys = ON;` |

```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    password_hash TEXT NOT NULL,
    nickname TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP

CREATE TABLE posts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
```

## 七、术语附录

| 术语 | 英文 | 通俗解释 | 本章出现位置 |
|------|------|----------|-------------|
| 外键 | Foreign Key | 表中的一个字段，它的值必须存在于另一张表的某个字段中。就像"身份证号"——你填的号码必须在公安局的数据库里真实存在。**字面陷阱**："外"不是说它不重要，而是说它引用了"外部"（另一张表）的数据。 | 步骤 3、4、5 |
| 数据完整性 | Data Integrity | 数据在存储后仍然准确、一致、没有矛盾。比如"一篇文章的作者 ID 指向一个不存在的用户"就是数据完整性被破坏。外键约束是保证数据完整性的手段之一。**字面陷阱**："完整性"不是"完整无缺"的意思，而是"数据之间没有逻辑矛盾"。 | 步骤 5、🤔 想多一点 |
| 建表语句 | CREATE TABLE | SQL 中用来创建新表的命令。格式：`CREATE TABLE 表名 (列名 类型 约束, ...);`。建表语句定义了表的结构——有哪些列、每列什么类型、有什么限制。 | 步骤 2、3、4、6 |
| 表关系 | Table Relationship | 表和表之间通过外键建立的联系。常见的有：一对多（一个用户→多篇文章）、多对一（多条评论→一篇文章）。本教程中三张表的关系是：用户 1:N 文章，用户 1:N 评论，文章 1:N 评论。 | 步骤 5 |
| 密码哈希 | Password Hash | 把密码原文通过数学算法变成一串"乱码"。特点：相同输入总是得到相同输出，但无法从输出反推输入。数据库存哈希值而不存明文，即使数据库泄露，攻击者也拿不到用户真实密码。 | 步骤 2 |
| 主键 | Primary Key | 表中每一行数据的唯一标识，就像身份证号。SQLite 中 `INTEGER PRIMARY KEY AUTOINCREMENT` 会自动生成递增的 ID。 | 步骤 2 |

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| SQLite 外键默认不开启 | 插入不存在的作者 ID 没有报错 | SQLite 历史兼容性，外键约束默认关闭 | 每次连接数据库后执行 `PRAGMA foreign_keys = ON;` |
| 建表顺序错误 | `no such table: users` 报错 | 在创建 posts 表时引用了还不存在的 users 表 | 严格按照 users → posts → comments 的顺序建表 |
| 外键字段类型不一致 | 外键约束不生效 | `author_id` 定义为 `TEXT`，但 `users.id` 是 `INTEGER`，类型不匹配 | 外键字段和被引用字段的类型必须一致（都用 `INTEGER`） |
| 忘记 `conn.commit()` | 数据在 Python 中能看到，但关掉后丢失 | SQLite 在 Python 中默认不自动提交 | 建表/插入/更新/删除操作后执行 `conn.commit()` |
| 在已存在的表上重复 `CREATE TABLE` | `table already exists` 报错 | 表已经建过了，不能重复建 | 要么先 `DROP TABLE IF EXISTS 表名;`，要么检查表是否存在再决定建不建 |
| 在 SQLite 命令行中忘记分号 | 输入换行后没有执行 | SQLite 以分号 `;` 作为语句结束标志 | 每条 SQL 语句末尾加 `;` 再回车 |
