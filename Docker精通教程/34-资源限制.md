## 我在做什么？

你在一台 4 核 8GB 的服务器上跑了 3 个容器。某天，一个 Java 应用突然开始疯狂 GC，内存飙到 7GB，另外两个容器直接被 OOM Killer *此术语见附录C* 杀死。你看着监控，发现罪魁祸首是一个你没设资源限制的容器——它想吃多少就吃多少，就像餐馆里没人管的"自助餐终结者"。

Docker 默认**不对容器做任何资源限制**。一个容器可以用光宿主机的所有 CPU 和内存，把其他容器"饿死"。这一章教你给每个容器戴上"紧箍咒"——设定内存上限、CPU 配额、Swap 限制，让每个容器只能吃自己碗里的，不能抢别人的。

学完这一章，你能：
- 用 `--memory` 限制容器内存，理解 OOM 的触发条件
- 用 `--cpus` 限制容器 CPU 使用量
- 区分 `--cpus`（绝对限制）和 `--cpu-shares`（相对权重）
- 理解 `--memory-swap` 的作用
- 在 Compose 中用 `deploy.resources` 配置资源限制
- 用 `docker stats` 实时监控容器资源使用

## 一、问题：没有资源限制的容器有多危险？

### 一个实验：让容器吃掉所有内存

```bash
# 启动一个"内存炸弹"容器
docker run -d --name memory-hog alpine:latest \
  sh -c 'dd if=/dev/zero of=/dev/null bs=1G count=100'
# 这个容器会尝试分配大量内存

# 观察宿主机状态
free -h
#              total   used    free
# Mem:         7.7Gi   7.5Gi   0.2Gi   ← 几乎被吃光
# Swap:        2.0Gi   2.0Gi   0B      ← swap 也满了
```

这时候你的 SSH 连接可能已经卡住了，其他容器也在崩溃边缘。**没有限制的容器就像没有阀门的水管——一旦失控，整个系统遭殃。**

### 为什么 Docker 默认不限制？

Docker 的设计哲学是"开箱即用"——它假设你只想快速跑一个容器，不想被各种配置绊住。这个假定在开发环境没问题，但在生产环境是灾难性的。

> **想多一点**：把 Docker 主机想象成一栋公寓楼，每个容器是一个租户。默认情况下，所有租户共用水电燃气，没有独立的电表和水表。一个租户用空调开到 16 度，其他租户的电压就不够了。资源限制就是在每个租户门口装独立电表——你用多少是你的事，但不能超过合同规定的上限。超过上限怎么办？电表跳闸（OOM Kill），而不是整栋楼跳闸（宿主机崩溃）。

## 二、内存限制：`--memory`

### 基本用法

```bash
docker run -d --name my-app --memory 256m my-app:latest
```

`--memory`（或 `-m`）限制容器能使用的最大物理内存。如果容器尝试分配超过这个限制的内存，Docker 会触发 OOM（Out Of Memory），杀死容器内的进程。

### 内存单位

```bash
--memory 256m      # 256 MB
--memory 1g        # 1 GB
--memory 512000k   # 512000 KB
--memory 2.5g      # 2.5 GB（支持小数）
```

### 实战：观察 OOM

```bash
# 限制容器只能使用 50MB 内存
docker run -d --name oom-test --memory 50m alpine:latest \
  sh -c 'tail /dev/zero'

# 等几秒
docker ps -a --filter name=oom-test
# STATUS: Exited (137)  ← 137 是 OOM 被杀的标准退出码

# 查看 OOM 事件
docker inspect oom-test --format='{{.State.OOMKilled}}'
# true
```

退出码 137 = 128 + 9（SIGKILL）。OOM Killer 发的就是 SIGKILL 信号。

### 实战：限制一个 Java 容器

Java 应用特别容易 OOM，因为 JVM 默认会拿宿主机的内存来算堆大小：

```bash
# ❌ 错误：容器限制 256MB，但 JVM 默认堆可能设得更大
docker run -d --name java-app --memory 256m openjdk:21 \
  java -jar app.jar
# 很快 OOM，因为 JVM 不知道自己在容器里，按宿主机 8GB 设堆

# ✅ 正确：配合 JVM 参数
docker run -d --name java-app --memory 512m openjdk:21 \
  java -XX:MaxRAMPercentage=75.0 -jar app.jar
# JVM 会使用容器内存的 75%（约 384MB）作为最大堆
```

> **关键原则**：`--memory` 限制的是容器内存，但 JVM 不知道这个限制。你需要用 `-XX:MaxRAMPercentage` 或 `-Xmx` 告诉 JVM 它能用多少。**两者缺一不可**——只设 `--memory` 不设 `-Xmx`，JVM 可能 OOM；只设 `-Xmx` 不设 `--memory`，容器可能吃掉宿主机内存。

## 三、Swap 限制：`--memory-swap`

### 什么是 `--memory-swap`？

`--memory-swap` 控制的是**内存 + swap 的总量**，不是单独的 swap 大小。

```bash
docker run -d --memory 256m --memory-swap 512m my-app
```

- `--memory 256m`：物理内存上限 256MB
- `--memory-swap 512m`：内存 + swap 上限 512MB
- 可用 swap = 512MB - 256MB = 256MB

### 三种常见的 swap 配置

| 配置 | 含义 |
|------|------|
| `--memory 256m --memory-swap 256m` | 禁用 swap（swap = 0） |
| `--memory 256m --memory-swap 512m` | 256MB 物理内存 + 256MB swap |
| `--memory 256m --memory-swap -1` | 256MB 物理内存 + 无限制 swap（不推荐） |
| 只设 `--memory 256m`（不设 `--memory-swap`） | 默认 swap = 内存 × 2 = 512MB |

### 什么时候需要 swap？

```bash
# ❌ 完全禁用 swap 在某些场景下可能有问题
docker run -d --memory 256m --memory-swap 256m my-app
# 应用内存尖峰时直接 OOM，没有缓冲

# ✅ 给一点 swap 作为缓冲
docker run -d --memory 256m --memory-swap 384m my-app
# 128MB 的 swap 缓冲区，应对短期内存尖峰
```

> **想多一点**：swap 就像信用卡额度——它不是让你长期依赖的，而是应对临时资金周转的。应用正常的运行内存应该在物理内存内，swap 只在短暂的尖峰时发挥作用。如果应用长期使用 swap，性能会严重下降（磁盘 IO 比内存慢几个数量级），这时候应该增加 `--memory` 而不是 `--memory-swap`。

## 四、CPU 限制

### `--cpus`：绝对限制

```bash
# 限制容器最多使用 1.5 个 CPU 核心
docker run -d --cpus 1.5 my-app
```

`--cpus` 是硬限制。无论宿主机有多少空闲 CPU，这个容器最多只能用 1.5 个核心的计算能力。

```bash
# 实战：体验 CPU 限制
# 启动一个 CPU 密集型容器（不限制）
docker run -d --name cpu-unlimited alpine:latest \
  sh -c 'while true; do :; done'

# 启动另一个 CPU 密集型容器（限制 0.5 核）
docker run -d --name cpu-limited --cpus 0.5 alpine:latest \
  sh -c 'while true; do :; done'

# 监控
docker stats cpu-unlimited cpu-limited
# cpu-unlimited: CPU ~100%   ← 吃掉一整个核心
# cpu-limited:   CPU ~50%    ← 只用半个核心
```

### `--cpu-shares`：相对权重

```bash
docker run -d --cpu-shares 512 my-app
```

`--cpu-shares` **不是绝对限制**，而是**相对权重**。默认值是 1024。

- 只有容器 A（shares=1024）时，它可以用 100% CPU
- 容器 A（shares=1024）和容器 B（shares=512）同时跑时，A 拿 2/3，B 拿 1/3
- 如果 A 空闲，B 可以拿到更多 CPU

```bash
# 实战：验证 cpu-shares
# 启动两个容器，权重 2:1
docker run -d --name app1 --cpu-shares 1024 alpine:latest \
  sh -c 'while true; do :; done'
docker run -d --name app2 --cpu-shares 512 alpine:latest \
  sh -c 'while true; do :; done'

docker stats app1 app2
# app1: CPU ~66%
# app2: CPU ~33%
# 比例 2:1 ✓
```

### `--cpus` vs `--cpu-shares`：怎么选？

| 场景 | 用哪个 | 原因 |
|------|--------|------|
| 需要严格限制，防止一个容器影响其他 | `--cpus` | 硬限制，绝对不超过 |
| 希望空闲时让某些容器用更多 CPU | `--cpu-shares` | 软限制，空闲资源可共享 |
| 生产环境 | `--cpus` | 可预测性 > 灵活性 |
| 批处理任务 | `--cpu-shares` | 让批处理在空闲时跑快一点 |

```bash
# ❌ 混淆：以为 cpu-shares 是绝对限制
docker run -d --cpu-shares 512 my-app
# 如果宿主机只有这一个容器在跑，它还是能用 100% CPU

# ✅ 要绝对限制，用 --cpus
docker run -d --cpus 0.5 my-app
# 无论如何，最多用 0.5 核
```

### 其他 CPU 相关参数

```bash
# --cpuset-cpus：绑定到特定 CPU 核心
docker run -d --cpuset-cpus 0,2 my-app
# 只能用 CPU 0 和 CPU 2（适合 NUMA 架构优化）

# --cpu-period 和 --cpu-quota：更细粒度的 CPU 控制
# 一般用 --cpus 就够了，这两个是底层实现参数
```

## 五、Docker Compose 中的资源限制

### 基础配置

```yaml
# compose.yml
services:
  web:
    image: nginx:alpine
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 256M
        reservations:
          cpus: '0.25'
          memory: 128M
```

### limits vs reservations

| 配置 | 含义 | 比喻 |
|------|------|------|
| `limits` | 硬上限，容器不能超过 | 房租合同：最多住 2 人 |
| `reservations` | 软预留，调度时保证 | 酒店预订：保证给你留 1 间房 |

```yaml
services:
  api:
    image: my-api:latest
    deploy:
      resources:
        limits:
          cpus: '2.0'        # 最多用 2 核
          memory: 512M       # 最多用 512MB
        reservations:
          cpus: '0.5'        # 保证至少 0.5 核
          memory: 256M       # 保证至少 256MB
```

> **注意**：`deploy.resources` 在 `docker compose up` 中**默认被忽略**。它只在 Swarm 模式下生效（`docker stack deploy`）。如果你用 `docker compose up`，需要用 `docker compose --compatibility up` 来启用资源限制，或者在 Compose v2 中用旧的 `mem_limit` / `cpus` 语法。

### Compose 兼容写法（不依赖 Swarm）

```yaml
# compose.yml — 兼容传统写法
services:
  web:
    image: nginx:alpine
    mem_limit: 256m          # 内存限制
    cpus: 0.5                # CPU 限制
    memswap_limit: 384m      # Swap 限制
```

## 七、实战：完整演示

### 场景：一台服务器跑三个容器

```bash
# 1. 数据库：需要较多内存，稳定运行
docker run -d --name db \
  --memory 512m --memory-swap 512m \
  --cpus 1.0 \
  postgres:16-alpine

# 2. 后端 API：中等资源
docker run -d --name api \
  --memory 256m --memory-swap 384m \
  --cpus 0.5 \
  my-api:latest

# 3. 后台任务（批处理）：低优先级，但允许空闲时多跑
docker run -d --name worker \
  --memory 128m \
  --cpu-shares 256 \
  my-worker:latest

# 监控所有容器
docker stats --no-stream
```

### 验证：内存限制生效

```bash
# 查看 api 容器的内存限制
docker inspect api --format='{{.HostConfig.Memory}}'
# 268435456（即 256MB，256 × 1024 × 1024）

# 在容器内查看可见内存
docker exec api free -h
#              total   used   free
# Mem:         256Mi   50Mi   206Mi
# 容器看到的内存上限就是 256MB
```

## 九、不对怎么办？

### 常见错误 1：Java 容器不配 `-Xmx`，配合 `--memory` 导致 OOM

```dockerfile
# ❌ 错误：只设了容器内存限制，没设 JVM 堆大小
FROM openjdk:21
COPY app.jar .
CMD ["java", "-jar", "app.jar"]
```

```bash
docker run -d --memory 256m java-app
# 容器看到 256MB 内存，但 JVM 按宿主机内存算堆（比如 2GB）
# 结果：JVM 尝试分配 2GB 堆 → 超过容器 256MB 限制 → OOM Kill
```

✅ 修正：

```dockerfile
FROM openjdk:21
COPY app.jar .
# 75% 是保守值，留 25% 给非堆内存（Metaspace、线程栈、Native 内存）
CMD ["java", "-XX:MaxRAMPercentage=75.0", "-jar", "app.jar"]
```

### 常见错误 2：`--cpus` 和 `--cpu-shares` 混淆

```bash
# ❌ 错误：以为 --cpu-shares 512 是限制 0.5 核
docker run -d --cpu-shares 512 my-app
# 如果宿主机空闲，这个容器还是能用 100% CPU

# ✅ 正确：要限制 0.5 核，用 --cpus
docker run -d --cpus 0.5 my-app
```

记住：**`--cpus` 是天花板（绝对上限），`--cpu-shares` 是谈判桌（相对权重）。**

### 常见错误 3：`--memory-swap` 理解错误

```bash
# ❌ 错误：以为 memory-swap 是 swap 大小
docker run -d --memory 256m --memory-swap 256m
# 以为：256MB 内存 + 256MB swap = 512MB 总量
# 实际：256MB 内存 + 0 swap = 256MB 总量（--memory-swap = --memory 表示禁用 swap）

# ✅ 正确：memory-swap 是 内存+swap 的总量
docker run -d --memory 256m --memory-swap 512m
# 256MB 内存 + 256MB swap = 512MB 总量
```

### 常见错误 4：`docker stats` 中 MEM USAGE 显示不准

```bash
docker stats my-app
# MEM USAGE 可能显示得比实际小
```

**原因**：`docker stats` 显示的是 RSS（Resident Set Size），不包括 page cache 等。实际内存使用可能更高。此外，如果容器内进程使用了 mmap 文件映射，这些也不计入 RSS。

**排查**：进入容器用 `free -h` 或 `cat /sys/fs/cgroup/memory/memory.usage_in_bytes` 查看 cgroup 级别的精确内存使用。

### 常见错误 5：Compose 中 `deploy.resources` 不生效

```yaml
# compose.yml
services:
  web:
    image: nginx:alpine
    deploy:
      resources:
        limits:
          memory: 256M
```

```bash
docker compose up -d
# 资源限制没生效！容器还是能随便用内存
```

✅ 修正方式一：用兼容模式

```bash
docker compose --compatibility up -d
```

✅ 修正方式二：用传统语法

```yaml
services:
  web:
    image: nginx:alpine
    mem_limit: 256M
    cpus: 0.5
```

## 十、术语解释

| 术语 | 解释 |
|------|------|
| **OOM（Out Of Memory）** *此术语见附录C* | Linux 内核在内存不足时强制杀死进程的机制。Docker 中用 `--memory` 限制容器内存，超限时容器内进程被 OOM Kill，退出码 137 |
| **`--memory`** *此术语见附录C* | 容器能使用的物理内存上限。超限触发 OOM。单位：b/k/m/g |
| **`--cpus`** *此术语见附录C* | 容器能使用的 CPU 核心数上限。`--cpus 1.5` 表示最多用 1.5 个核心。是硬限制 |
| **`--cpu-shares`** *此术语见附录C* | CPU 相对权重，默认 1024。只在 CPU 争抢时生效，空闲时容器可以超过权重使用。是软限制 |
| **`--memory-swap`** *此术语见附录C* | 内存 + swap 的总量上限。设为和 `--memory` 相同表示禁用 swap。不设时默认是 `--memory` 的 2 倍 |
| **`docker stats`** *此术语见附录C* | 实时显示容器的 CPU、内存、网络、磁盘 IO 使用量。是 `docker` 内置的监控命令 |

## 本篇最可能出错的地方及原因

### 1. Java 容器只设 `--memory` 不设 `-Xmx`（最高频）

**这是 Java + Docker 场景下最经典的错误。** JVM 默认根据宿主机内存计算堆大小，而不是根据容器的 `--memory` 限制。容器设了 256MB 内存，JVM 以为有 8GB，把自己的堆设到 2GB——结果直接被 OOM Kill。

**解决方案**：`-XX:MaxRAMPercentage=75.0`（Java 8 用 `-XX:+UseCGroupMemoryLimitForHeap`）。这个参数让 JVM 读取 cgroup 的内存限制，按比例分配堆。

### 2. `--memory-swap` 的语义陷阱

很多文档把 `--memory-swap` 称为"swap 限制"，但它的实际含义是"内存+swap 的总量上限"。`--memory 256m --memory-swap 256m` 不是"256MB 内存 + 256MB swap"，而是"256MB 总量，swap 为 0"。

**记忆方法**：`--memory` 是"物理内存"，`--memory-swap` 是"总计"，`--memory-swap` 减去 `--memory` 才是"可用的 swap"。

### 3. `docker compose up` 中 `deploy.resources` 静默失效

`deploy` 段是 Docker Swarm 的配置，`docker compose up` 默认会忽略它。写了配置但没生效，没有报错，没有警告——静默失效是这类错误最危险的地方。

**排查**：`docker inspect` 查看容器的 `HostConfig.Memory` 是否为 0。如果是 0，说明限制没生效。

### 4. `docker stats` 的内存数据不等于实际使用量

`docker stats` 的 MEM USAGE 是 RSS，不包括 page cache。如果你的应用大量使用文件缓存，实际内存使用可能远高于 `docker stats` 显示的值。OOM 可能发生在 `docker stats` 显示只有 50% 内存使用率的时候。

**精确排查**：进入容器，查看 `/sys/fs/cgroup/memory/memory.usage_in_bytes` 和 `memory.stat`。

### 5. `--cpu-shares` 误用为绝对限制

这是"语义混淆"型错误。`--cpu-shares` 名字里有"cpu"，容易让人以为它是"CPU 限制"。实际上它是"CPU 权重"，只在多个容器竞争 CPU 时才生效。如果只有一个容器在跑，`--cpu-shares 1` 也能用满所有 CPU。
