## 三、分步操作

### 步骤 1：回顾——07 内存版的数据存在哪？

在教程 07 中，我们用了一行代码来"模拟数据库"：

```python
# blog_backend/main.py - 内存版
posts_db: list[dict] = []  # 这就是我们的"数据库"
```

**这个"数据库"有什么问题？**

- 数据存在 Python 进程的**内存**里
- 程序一退出（服务器一停止），内存被操作系统回收，数据就没了
- 下次启动，`posts_db = []` 又被重新赋值为空列表——一切清零
- 就像你在沙滩上写字，潮水一来，什么都没了

**比喻回顾**：
- 内存版 = 写在便签纸上，揉了扔垃圾桶就没了
- 数据库版 = 刻在石头上，刻上去就一直在，风吹雨打也不会消失

**我们今天要做的事**：把五个接口中所有操作 `posts_db` 列表的代码，换成操作数据库的代码。**五个接口的 URL、参数、响应格式都不变**——变的只是数据从哪来、到哪去。

> 🤔 想多一点：为什么分两步走？为什么不一开始就用数据库？

**因为"分而治之"是学习编程最好的方法。**

如果一开始就让你同时学：
1. 接口怎么设计（路由、参数、错误处理）
2. 数据库怎么连接、怎么写 ORM 查询、怎么提交事务

你报错的时候根本不知道是接口写错了还是数据库写错了——两个坑叠加，调试起来非常痛苦。

我们先走第一步：不管数据库，先把五个接口的逻辑和形状写对，用最简单的内存列表跑通。现在接口逻辑已经烂熟于心了，再来换数据库——你只需要专注于"数据库操作"这一件新东西，任何报错都能直接定位到数据库相关的代码。

**这就像学画画：先画轮廓，再填颜色。轮廓对了，填颜色才不会乱。**

### 步骤 3：改造第一个接口 —— `GET /posts` 获取全部文章

先来改最简单的：获取全部文章列表。

#### 内存版代码（原来的）：

```python
@app.get("/posts")
def get_all_posts():
    return posts_db
```

#### 数据库版代码（改成这样）：

```python
from fastapi import FastAPI, Depends
from fastapi.responses import JSONResponse
from sqlalchemy.orm import Session  # 新增：导入 Session
from blog_backend.schemas import BlogCreate
from blog_backend.models import Post  # 新增：导入 Post ORM 模型
from blog_backend.database import get_db  # 新增：导入 get_db 依赖

app = FastAPI()

# 删掉这行：posts_db: list[dict] = []  # 内存版"数据库"，不再需要了

# === 获取全部文章 ===
@app.get("/posts")
def get_all_posts(db: Session = Depends(get_db)):  # 新增：注入数据库 Session
    posts = db.query(Post).all()  # 新增：从数据库查所有文章。此术语需进附录：query
    return posts
```

**逐行解释**：

- `db: Session = Depends(get_db)`：这是 FastAPI 的**依赖注入**。每次请求进来，FastAPI 自动给你创建一个数据库会话（`Session`），请求处理完自动关闭。你不用管连接怎么开怎么关，Framework 帮你搞定。
- `db.query(Post)`：从数据库查询 `Post` 表。此术语需进附录：query 就是"查询"的意思，你告诉数据库："我要查 Post 这张表"。
- `.all()`：把查询结果中的**所有行**都拿出来，返回一个列表。如果表是空的，返回空列表 `[]`——和内存版的 `posts_db` 行为一模一样！
- `return posts`：直接返回列表。SQLAlchemy 的 ORM 对象，FastAPI 自动转成 JSON 返回——和内存版返回字典列表的行为完全一样。

**❌ 常见错误 → ✅ 正确示例**：

❌ 错误示例：把 `Post` 当成字典用

```python
# 错误：Post 是 ORM 模型类，不是 dict
posts = db.query(dict).all()  # 错！
posts = db.query("Post").all()  # 错！
```

✅ 正确示例：`db.query()` 的参数是**ORM 模型类**

```python
# 正确：直接传你导入的 Post 类
posts = db.query(Post).all()
```

**验证方法**：

保存文件，服务器自动重启（因为 `--reload` 开着）。用 curl 测试：

```bash
curl http://127.0.0.1:8000/posts
```

预期输出（现在数据库还是空的）：

```json
[]
```

和内存版空列表的输出一样——改造成功！

### 步骤 4：改造第二个接口 —— `GET /posts/{id}` 获取单篇文章

接下来改造"按 ID 获取单篇文章"。

#### 内存版代码（原来的）：

```python
@app.get("/posts/{post_id}")
def get_post(post_id: int):
    for post in posts_db:
        if post["id"] == post_id:
            return post
    return JSONResponse(status_code=404, content={"error": f"文章 #{post_id} 不存在"})
```

#### 数据库版代码（改成这样）：

```python
# === 获取单篇文章 ===
@app.get("/posts/{post_id}")
def get_post(post_id: int, db: Session = Depends(get_db)):
    post = db.query(Post).filter(Post.id == post_id).first()  # 此术语需进附录：filter
    if post is None:  # 没找到
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在"}
        )
    return post
```

**逐行解释**：

- `db.query(Post)`：还是从 `Post` 表查，和上一步一样。
- `.filter(Post.id == post_id)`：加上过滤条件——只保留 `id` 等于 `post_id` 的行。此术语需进附录：filter 就是"过滤"，把不满足条件的筛掉，只留下满足条件的。
- `.first()`：拿满足条件的**第一行**。因为 `id` 是主键，肯定唯一，所以最多只有一行。如果找不到，返回 `None`。
- `if post is None`：如果返回 `None` 说明没找到，返回 404——和内存版逻辑一样。
- 如果找到了，直接返回 `post` 对象——FastAPI 自动转 JSON。

**对比内存版**：
- 内存版需要你自己写 `for` 循环遍历列表找
- 数据库版只需要 `filter(Post.id == post_id).first()`——SQLAlchemy 帮你生成 SQL 去查，比手写循环简洁太多

**验证方法**：

先用 POST 创建一篇文章（我们马上改 POST 接口，先测 GET），然后：

```bash
curl http://127.0.0.1:8000/posts/1
```

如果文章存在，返回文章 JSON；如果不存在，返回 404——和内存版行为一致。

### 步骤 5：改造第三个接口 —— `POST /posts` 创建文章

这一步稍微复杂一点，因为涉及到**写入数据库**和**提交事务**。

#### 内存版代码（原来的）：

```python
@app.post("/posts")
def create_post(blog: BlogCreate):
    new_id = len(posts_db) + 1
    new_post = {
        "id": new_id,
        "title": blog.title,
        "content": blog.content,
        "published": blog.published,
    }
    posts_db.append(new_post)
    return new_post
```

#### 数据库版代码（改成这样）：

```python
# === 创建文章 ===
@app.post("/posts")
def create_post(blog: BlogCreate, db: Session = Depends(get_db)):
    # 创建 ORM 对象
    new_post = Post(  # Post 是我们在 models.py 中定义的 ORM 模型
        title=blog.title,
        content=blog.content,
        published=blog.published,
    )
    # 添加到会话
    db.add(new_post)  # 此术语需进附录：add
    # 提交事务
    db.commit()  # 此术语需进附录：commit
    # 刷新对象，获取数据库生成的字段（id、created_at）
    db.refresh(new_post)  # 此术语需进附录：refresh
    # 返回创建好的文章
    return new_post
```

**逐行解释，这一步很重要，别跳！**：

1. `new_post = Post(...)`：创建一个 `Post` 对象。这只是在 Python 内存里创建了一个对象，还没写到数据库。参数就是我们模型定义的字段：`title`、`content`、`published`。**`id` 和 `created_at` 不用你给**——数据库会自动生成。

2. `db.add(new_post)`：把这个新对象**添加到当前数据库会话**。这一步还没真正写入数据库，只是告诉 SQLAlchemy："这个对象是新的，下次提交的时候把它插进去"。此术语需进附录。

3. `db.commit()`：**提交事务**——把这一次会话中所有的修改（插入、更新、删除）真正写到数据库文件里。这一步做完，数据就真的落盘了。此术语需进附录。如果不 commit，数据只在会话里，程序退出就没了。

4. `db.refresh(new_post)`：**刷新对象**——从数据库重新读取一遍这个对象的数据。为什么需要这一步？因为 `id` 和 `created_at` 是数据库自动生成的，你刚 `add()` 完，Python 这边的 `new_post` 对象还没有这两个字段的值。`refresh()` 之后，数据库生成的值就会被填到 `new_post` 对象里，你返回给前端的时候就能看到了。此术语需进附录。

> 🤔 想多一点：`db.refresh()` 到底什么时候需要？

- 如果你的主键是**自增 ID**，而且需要把生成的 ID 返回给客户端——必须 `refresh()`，否则客户端拿不到新生成的 ID。
- 如果你的主键是客户端生成的（比如 UUID），而且所有字段都是客户端传的——不需要 `refresh()`，因为你已经有所有值了。
- 在我们这个案例中，`id` 是自增的，我们需要返回给客户端，所以必须 `refresh()`。

**❌ 常见错误 → ✅ 正确示例**：

❌ 错误 1：只 `add()` 忘记 `commit()`

```python
db.add(new_post)
# 忘记写 db.commit()！
return new_post
```

**后果**：程序不报错，返回也正常，但数据根本没写到数据库。重启服务器后，这条记录就消失了——因为它只在会话里，没落盘。

✅ 正确：`add()` 之后必须 `commit()`

```python
db.add(new_post)
db.commit()
db.refresh(new_post)
return new_post
```

❌ 错误 2：`commit()` 之后忘记 `refresh()`

```python
db.add(new_post)
db.commit()
# 忘记 db.refresh()
return new_post
```

**后果**：返回给客户端的 JSON 中，`id` 和 `created_at` 是 `null`（因为 Python 对象里还没有这两个值），虽然数据库里已经有了。

✅ 正确：要返回自增 ID 就必须 `refresh()`

**验证方法**：

保存文件，用 curl 创建一篇文章：

```bash
curl -X POST http://127.0.0.1:8000/posts ^
  -H "Content-Type: application/json" ^
  -d "{\"title\": \"数据库第一篇文章\", \"content\": \"数据真的存在硬盘了！\", \"published\": true}"
```

**预期输出**：

```json
{
  "id": 1,
  "title": "数据库第一篇文章",
  "content": "数据真的存在硬盘了！",
  "published": true,
  "created_at": "2026-06-05T12:34:56",
  "updated_at": "2026-06-05T12:34:56"
}
```

看到了吗？`id` 是 `1`（数据库自动生成），`created_at` 和 `updated_at` 也有值了。这就是 `refresh()` 的作用——把数据库生成的字段读回来给你。

再验证列表：

```bash
curl http://127.0.0.1:8000/posts
```

应该能看到刚才创建的文章。

### 步骤 6：改造第四个接口 —— `PUT /posts/{id}` 更新文章

接下来改造更新接口。更新的逻辑是：**先查出来，改字段，再提交**。

#### 内存版代码（原来的）：

```python
@app.put("/posts/{post_id}")
def update_post(post_id: int, blog: BlogCreate):
    for i, post in enumerate(posts_db):
        if post["id"] == post_id:
            posts_db[i] = {
                "id": post_id,
                "title": blog.title,
                "content": blog.content,
                "published": blog.published,
            }
            return posts_db[i]
    return JSONResponse(status_code=404, content={"error": ...})
```

#### 数据库版代码（改成这样）：

```python
# === 更新文章 ===
@app.put("/posts/{post_id}")
def update_post(post_id: int, blog: BlogCreate, db: Session = Depends(get_db)):
    # 先查找这篇文章
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法更新"}
        )
    # 找到了，修改字段
    post.title = blog.title
    post.content = blog.content
    post.published = blog.published
    # 提交事务
    db.commit()  # 此术语需进附录：commit
    # 返回更新后的文章
    return post
```

**逐行解释**：

- `post = db.query(Post).filter(Post.id == post_id).first()`：和 GET 接口一样，先按 ID 查找。
- `if post is None`：没找到返回 404——和之前逻辑一样。
- `post.title = blog.title`：找到之后，直接修改 ORM 对象的属性！SQLAlchemy 会自动追踪你对对象属性的修改，你不用写 `UPDATE` 语句，改完属性直接 `commit()` 就行——非常方便。
- `db.commit()`：提交修改——修改真正写到数据库。
- 返回 `post`——已经修改好了，直接返回。

**对比内存版**：比内存版还简洁！内存版需要你找索引、替换整个字典，数据库版找到对象直接改属性就行。

**验证方法**：

我们刚才创建了 ID 为 1 的文章，现在更新它：

```bash
curl -X PUT http://127.0.0.1:8000/posts/1 ^
  -H "Content-Type: application/json" ^
  -d "{\"title\": \"数据库第一篇文章（已更新）\", \"content\": \"更新后的内容，数据真的存好了！\", \"published\": true}"
```

预期输出：

```json
{
  "id": 1,
  "title": "数据库第一篇文章（已更新）",
  "content": "更新后的内容，数据真的存好了！",
  "published": true,
  "created_at": "2026-06-05T12:34:56",
  "updated_at": "2026-06-05T12:40:00"
}
```

注意 `updated_at` 已经变成更新时间了——这是我们在 `models.py` 中配置的 `onupdate=func.now()`，SQLAlchemy 自动帮你更新，不用你写代码！

再 GET 验证一下：

```bash
curl http://127.0.0.1:8000/posts/1
```

标题和内容确实变了——更新成功！

### 步骤 7：改造第五个接口 —— `DELETE /posts/{id}` 删除文章

最后一个接口：删除文章。逻辑：**先查，找到就删，然后提交**。

#### 内存版代码（原来的）：

```python
@app.delete("/posts/{post_id}")
def delete_post(post_id: int):
    for i, post in enumerate(posts_db):
        if post["id"] == post_id:
            deleted = posts_db.pop(i)
            return {"message": f"文章 #{post_id}「{deleted['title']}」已删除"}
    return JSONResponse(status_code=404, content={"error": ...})
```

#### 数据库版代码（改成这样）：

```python
# === 删除文章 ===
@app.delete("/posts/{post_id}")
def delete_post(post_id: int, db: Session = Depends(get_db)):
    # 先查找这篇文章
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法删除"}
        )
    # 找到了，删除
    db.delete(post)  # 此术语需进附录：delete
    # 提交事务
    db.commit()  # 此术语需进附录：commit
    # 返回成功消息
    return {"message": f"文章 #{post_id}「{post.title}」已删除"}
```

**逐行解释**：

- 查找逻辑和之前一样：`db.query(Post).filter(Post.id == post_id).first()`
- `db.delete(post)`：告诉 SQLAlchemy，这个对象要删除。和 `add()` 一样，这一步只是标记删除，还没真正从数据库删掉。此术语需进附录。
- `db.commit()`：提交事务——删除真正生效。
- 返回成功消息——和内存版格式一样。

**验证方法**：

删除 ID 为 1 的文章：

```bash
curl -X DELETE http://127.0.0.1:8000/posts/1
```

预期输出：

```json
{"message":"文章 #1「数据库第一篇文章（已更新）」已删除"}
```

再验证删除：

```bash
curl http://127.0.0.1:8000/posts/1
```

应该返回 404：

```json
{"error":"文章 #1 不存在"}
```

删除成功！

### 步骤 8：终极验证——重启服务，数据还在吗？

现在来做最关键的实验——这也是我们今天做这一切的目的：**重启服务器，数据还在吗？**

**步骤 1**：先创建两篇文章，确认它们在数据库里：

```bash
# 创建文章 1
curl -X POST http://127.0.0.1:8000/posts -H "Content-Type: application/json" -d '{"title": "重启测试1", "content": "重启后我还在吗？", "published": true}'

# 创建文章 2
curl -X POST http://127.0.0.1:8000/posts -H "Content-Type: application/json" -d '{"title": "重启测试2", "content": "我应该还在。", "published": false}'

# 查看列表
curl http://127.0.0.1:8000/posts
```

你应该看到两篇文章，ID 分别是 2 和 3。

**步骤 2**：重启服务器。在终端按 `Ctrl+C` 停止 uvicorn，然后重新启动：

```bash
uvicorn blog_backend.main:app --reload
```

**步骤 3**：服务器启动完成后，再次获取文章列表：

```bash
curl http://127.0.0.1:8000/posts
```

**预期输出**：

```json
[
  {"id": 2, "title": "重启测试1", ...},
  {"id": 3, "title": "重启测试2", ...}

```

**两篇文章都还在！** 🎉

数据真的写到硬盘上的 `blog.db` 文件里了——服务器重启，数据不丢。这就是**持久化**。此术语需进附录：持久化 = 把数据从内存保存到硬盘，断电/重启后数据仍然存在。

### 步骤 9：代码对比——内存版 vs 数据库版，差异有多大？

我们来做一个并排对比，直观感受一下改造前后的差异：

| 接口 | 内存版核心代码 | 数据库版核心代码 |
|------|---------------|-----------------|
| `GET /posts` | `return posts_db` | `return db.query(Post).all()` |
| `GET /posts/{id}` | `for post in posts_db: if post["id"] == post_id: return post` | `post = db.query(Post).filter(Post.id == post_id).first()` |
| `POST /posts` | `new_id = len(...)+1; posts_db.append(...)` | `Post(...); db.add(...); db.commit(); db.refresh()` |
| `PUT /posts/{id}` | `找到后 posts_db[i] = 新字典` | `找到后 post.title = new_title; db.commit()` |
| `DELETE /posts/{id}` | `找到后 posts_db.pop(i)` | `找到后 db.delete(post); db.commit()` |

**你发现了吗？**

- **接口逻辑完全没变**：还是"查所有、查单个、创建、更新、删除"
- **URL 路径完全没变**：还是 `/posts`、`/posts/{id}`
- **请求参数完全没变**：还是那些参数
- **响应格式完全没变**：还是那个 JSON 结构，调用方根本不知道你底层换了存储
- **唯一变的**：就是"数据从哪来、到哪去"这几行代码

**这就是**"分层设计"**的好处**：接口层和数据层分离，你换底层存储，接口层不用大改——对调用方完全透明。

### `blog_backend/main.py`（本教程最终数据库版）

```python
from fastapi import FastAPI, Depends
from fastapi.responses import JSONResponse
from sqlalchemy.orm import Session
from blog_backend.schemas import BlogCreate
from blog_backend.models import Post
from blog_backend.database import get_db

app = FastAPI()

# （内存版的 posts_db 已经删掉了）


@app.get("/")
def read_root():
    return {"message": "Hello World"}


# === 获取全部文章 ===
@app.get("/posts")
def get_all_posts(db: Session = Depends(get_db)):
    posts = db.query(Post).all()
    return posts


# === 创建文章 ===
@app.post("/posts")
def create_post(blog: BlogCreate, db: Session = Depends(get_db)):
    new_post = Post(
        title=blog.title,
        content=blog.content,
        published=blog.published,
        summary=blog.summary,
    )
    db.add(new_post)
    db.commit()
    db.refresh(new_post)
    return new_post


# === 获取单篇文章 ===
@app.get("/posts/{post_id}")
def get_post(post_id: int, db: Session = Depends(get_db)):
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在"}
        )
    return post


# === 更新文章 ===
@app.put("/posts/{post_id}")
def update_post(post_id: int, blog: BlogCreate, db: Session = Depends(get_db)):
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法更新"}
        )
    post.title = blog.title
    post.content = blog.content
    post.published = blog.published
    post.summary = blog.summary
    db.commit()
    return post


# === 删除文章 ===
@app.delete("/posts/{post_id}")
def delete_post(post_id: int, db: Session = Depends(get_db)):
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法删除"}
        )
    db.delete(post)
    db.commit()
    return {"message": f"文章 #{post_id}「{post.title}」已删除"}
```

## 六、术语附录

| 术语 | 英文 | 通俗解释 | 本章出现位置 |
|------|------|----------|-------------|
| query | — | SQLAlchemy ORM 的查询入口，`db.query(Model)` 表示"要查询某个模型对应的表"。本章就是 `db.query(Post)`。 | 步骤 3 |
| filter | — | 在查询中添加过滤条件，只保留满足条件的行。本章用 `filter(Post.id == post_id)` 按 ID 过滤。 | 步骤 4 |
| add | — | 把一个新建的 ORM 对象添加到数据库会话，准备插入。调用 `add()` 后还没真正写入，需要 `commit()` 才落盘。 | 步骤 5 |
| commit | — | 提交当前数据库事务，把会话中所有的修改（插入/更新/删除）真正写入数据库。不 commit，修改只在内存中，重启就丢。 | 步骤 5 |
| refresh | — | 从数据库重新读取对象的数据，用于获取数据库自动生成的字段（如自增 ID、自动时间戳）。`add()` + `commit()` 后需要 refresh 才能拿到自动生成的值。 | 步骤 5 |
| delete | — | 标记一个 ORM 对象为删除，需要调用 `commit()` 才真正从数据库删掉。和 `add()` 对应，一个标记插入一个标记删除。 | 步骤 7 |
| 持久化 | Persistence | 把数据从易失性的内存保存到永久性存储（硬盘、SSD），断电、重启、程序退出后数据仍然存在。本章把数据从内存（重启丢）持久化到 SQLite 数据库文件（重启还在）。 | 步骤 8 |

## 七、小结

| 你学到了什么 | 一句话总结 |
|--------------|-----------|
| `GET /posts` 数据库版 | `db.query(Post).all()` 查询所有文章 |
| `GET /posts/{id}` 数据库版 | `db.query(Post).filter(Post.id == id).first()` 按 ID 查找 |
| `POST /posts` 数据库版 | `Post()` → `db.add()` → `db.commit()` → `db.refresh()` |
| `PUT /posts/{id}` 数据库版 | 先查 → 改对象属性 → `db.commit()` |
| `DELETE /posts/{id}` 数据库版 | 先查 → `db.delete()` → `db.commit()` |
| `db.refresh()` 的作用 | 拿到数据库自动生成的字段（自增 ID、created_at） |
| 什么时候需要 `commit()` | 任何修改数据库的操作（增/删/改）都需要 commit，查询不需要 |
| 持久化 | 数据写到硬盘，重启不丢——这就是数据库比内存列表真正多出来的东西 |
| 分层设计好处 | 换底层存储不影响上层接口，调用方完全感知不到 |

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| `add()` 之后忘记 `commit()` | 创建/更新/删除后，重启数据就没了 | 修改只在会话里，没落盘 | 增删改操作必须记得 `db.commit()` |
| `commit()` 之后忘记 `refresh()` | 返回的 JSON 中自增 ID 是 null | Python 对象还没拿到数据库生成的值 | 创建完自增 ID 的记录，必须 `refresh()` |
| `db.query(dict)` 或 `db.query("Post")` | 查询报错 | `query()` 的参数必须是 ORM 模型类，不是字符串也不是 dict | 导入 `Post` 类，直接传 `Post` |
| 改造完 curl 命令不能用了 | 报错 404 或参数错误 | 你改了 URL 或参数 | 原来内存版的 curl 命令完全能用，URL 和参数都没变，不用改 |
| 修改了 ORM 对象属性忘记 `commit()` | 重启后修改又回去了 | 修改只在内存，没写到数据库 | 改完属性必须 `db.commit()` |
