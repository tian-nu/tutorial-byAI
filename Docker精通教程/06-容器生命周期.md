## 我在做什么？

你已经会用 `docker run` 启动容器了。但容器的"一生"不止是"启动 → 运行 → 停止"这么简单。容器有完整的状态机：创建、运行、暂停、停止、删除。每个状态之间如何转换？`docker stop` 和 `docker kill` 有什么本质区别？已停止的容器为什么还占着磁盘空间？

这一章，我们把容器的"生老病死"搞清楚。学完之后，你看到 `docker ps -a` 里的任何状态都知道该怎么处理。

学完这一章，你能：
- 画出容器的完整状态图，理解每个状态的含义
- 区分 `docker stop`（优雅停止）和 `docker kill`（暴力杀死）
- 用 `docker pause/unpause` 冻结/恢复容器
- 用 `docker container prune` 清理所有已停止容器
- 理解为什么已停止的容器会占用磁盘空间

## 一、容器的一生：状态流转图

### 先把全貌看清楚

一个容器从创建到删除，会经历以下状态：

```
                    docker create
                         │
                         ▼
              ┌──────────────────┐
              │    Created        │  ← 容器已创建，但从未启动
              │  (已创建，未启动)   │
              └────────┬─────────┘
                       │ docker start
                       ▼
              ┌──────────────────┐
     ┌───────│    Running        │───────┐
     │       │  (正在运行)        │       │
     │       └────────┬─────────┘       │
     │                │                  │
     │ docker pause   │ docker stop      │ docker kill
     │                │ (SIGTERM)        │ (SIGKILL)
     ▼                ▼                  ▼
┌──────────┐   ┌──────────────┐   ┌──────────────┐
│  Paused   │   │   Stopped    │   │   Stopped    │
│ (已暂停)   │   │  (已停止)     │   │  (已停止)     │
└────┬─────┘   └──────┬───────┘   └──────┬───────┘
     │                │                  │
     │ docker         │ docker start     │ docker start
     │ unpause        │                  │
     ▼                ▼                  ▼
  Running          Running            Running
                     │
                     │ docker rm
                     ▼
              ┌──────────────────┐
              │    Deleted        │  ← 容器从系统中彻底删除
              │  (已删除)          │
              └──────────────────┘
```

> 容器生命周期 *此术语见附录C* 的核心规则：**容器的生命周期 = 它的主进程（PID 1）的生命周期。** 主进程活着 → 容器 Running；主进程退出 → 容器 Stopped。

### 每个状态的含义

| 状态 | 含义 | 什么时候出现 | 能做什么 |
|------|------|------------|---------|
| **Created** | 容器已创建，但主进程从未启动 | `docker create` 后，还没 `docker start` | 可以修改配置，然后 `docker start` |
| **Running** | 容器正在运行，主进程活着 | `docker start` 后，主进程正常运行 | 正常操作 |
| **Paused** | 容器被冻结，进程暂停，内存保留 | `docker pause` 后 | `docker unpause` 恢复 |
| **Stopped** | 容器已停止，主进程退出 | `docker stop`、`docker kill`、主进程自行退出 | 可以 `docker start` 重新启动，或 `docker rm` 删除 |
| **Deleted** | 容器已从系统中删除 | `docker rm` 后 | 不存在了，无法恢复 |

## 二、优雅停止 vs 暴力杀死：`docker stop` vs `docker kill`

### 这是本章最重要的概念

停止容器有两种方式，区别在于**给不给容器清理现场的机会**。

### `docker stop`：优雅停止

```bash
docker stop my-nginx
```

`docker stop` 会向容器主进程发送 **SIGTERM** *此术语见附录C* 信号（信号编号 15）。SIGTERM 的意思是"请你优雅地退出"——应用收到这个信号后，可以：

- 关闭数据库连接
- 写入未保存的数据
- 完成正在处理的请求
- 释放资源

然后 Docker 等待 10 秒（默认）。如果 10 秒后容器还没退出，Docker 发送 **SIGKILL** *此术语见附录C* 信号（信号编号 9），强制杀死进程。

```
docker stop 的过程：
──────────────────────────────────────────────────
发送 SIGTERM → 等待 10 秒 → 还没退出 → 发送 SIGKILL
──────────────────────────────────────────────────
```

### `docker kill`：暴力杀死

```bash
docker kill my-nginx
```

`docker kill` 直接发送 **SIGKILL** 信号。SIGKILL 不会被进程捕获，操作系统直接终止进程，不给任何清理机会。

### 对比

| | `docker stop` | `docker kill` |
|------|-------------|-------------|
| 发送的信号 | 先 SIGTERM，10 秒后 SIGKILL | 直接 SIGKILL |
| 进程能清理现场吗 | 能（如果程序处理了 SIGTERM） | 不能 |
| 速度 | 最多等 10 秒 | 瞬间 |
| 安全吗 | 安全，数据不会丢失 | 可能丢失未保存的数据 |
| 适合场景 | 日常停止 | 容器无响应、紧急情况 |

### 修改等待时间

```bash
# 只等 5 秒就发 SIGKILL
docker stop -t 5 my-nginx

# 等 30 秒（给足清理时间）
docker stop -t 30 my-nginx
```

> 打个比方：`docker stop` 就像你对接线员说"请帮我转接"——对方等你准备好；`docker kill` 就像你直接拔电话线——通话立刻中断，不管对方在说什么。

## 三、重新启动容器：`docker start` 和 `docker restart`

### `docker start`：启动已存在的容器

```bash
# 先停止
docker stop my-nginx

# 再启动（容器还在，只是重新跑起来）
docker start my-nginx
```

`docker start` 只对**已经存在的容器**有效。它不会创建新容器，只是重新启动一个已停止的容器。

### `docker start` vs `docker run`

| | `docker start` | `docker run` |
|------|--------------|------------|
| 操作对象 | 已存在的容器 | 镜像 |
| 效果 | 重新启动已停止的容器 | 创建新容器 + 启动 |
| 会创建新容器吗 | 不会 | 会 |
| 常用场景 | 恢复已停止的服务 | 首次部署 |

### `docker restart`：重启

```bash
docker restart my-nginx
```

`docker restart` = `docker stop` + `docker start`。它先停止容器，再启动。

```bash
# 等价于
docker stop my-nginx && docker start my-nginx
```

区别：`restart` 是原子操作，`stop && start` 不是。但大多数情况下没有区别，用哪个都行。

## 四、暂停与恢复：`docker pause` / `docker unpause`

### 和 `stop` 完全不同

`docker pause` 不会停止容器，而是**冻结**它。容器内的进程被暂停，但内存保持不变。

```bash
# 暂停容器
docker pause my-nginx

# 容器状态变为 Paused
docker ps
# STATUS 列显示：Up 10 minutes (Paused)

# 恢复容器
docker unpause my-nginx

# 容器恢复正常运行
docker ps
# STATUS 列显示：Up 10 minutes
```

### `pause` vs `stop` 的本质区别

| | `docker pause` | `docker stop` |
|------|-------------|-------------|
| 容器状态 | Paused | Stopped |
| 进程呢 | 被冻结（cgroup freezer） | 被终止 |
| 内存呢 | 保持 | 释放 |
| 恢复后 | 从暂停点继续 | 重新启动，状态丢失 |
| 恢复速度 | 瞬间 | 需要重新初始化 |
| 网络连接 | 可能会中断 | 中断 |

> 打个比方：`pause` 就像按了电视遥控器的"暂停"键——画面定格，但电视还在通电；`stop` 就像关了电视——重新开机后需要重新加载。

### `pause` 的适用场景

- 调试：暂停容器，检查宿主机状态，然后恢复
- 资源紧张时临时冻结非关键容器
- 对容器做快照前暂停（与 `docker commit` 配合）

> **想多一点**：`pause` 用的是 Linux 内核的 **cgroup freezer** 功能。它不发送任何信号给进程，而是直接在内核层面冻结进程调度。进程本身不知道被暂停了，恢复后它从被暂停的指令继续执行，仿佛什么都没发生。这比 `stop` 快得多，但网络连接可能会因为超时而断开。

## 五、删除容器：`docker rm` 和 `docker container prune`

### `docker rm`：删除指定容器

```bash
# 删除已停止的容器
docker rm my-nginx

# 强制删除运行中的容器（不推荐日常使用）
docker rm -f my-nginx

# 删除多个容器
docker rm my-nginx my-ubuntu my-redis
```

### 删除运行中的容器会怎样？

```bash
docker rm my-nginx
# Error: You cannot remove a running container abc123.
#        Stop the container before attempting removal or force remove
```

❌ 错误做法：直接 `docker rm` 运行中的容器。

✅ 正确做法：

```bash
# 方案 1：先停再删
docker stop my-nginx
docker rm my-nginx

# 方案 2：强制删除（风险：数据可能丢失）
docker rm -f my-nginx
```

### 删除所有已停止的容器（一行命令）

```bash
# 方法 1：用 prune（推荐）
docker container prune

# 输出：
# WARNING! This will remove all stopped containers.
# Are you sure you want to continue? [y/N] y
# Deleted Containers:
# abc123def456...

# 方法 2：用管道（不推荐，容易误删）
docker rm $(docker ps -aq)
```

> `docker container prune` 会列出将要删除的容器并让你确认，比直接用管道安全得多。

### 已停止的容器为什么占空间？

容器停止后，它的**可写层**（容器层）还在磁盘上。这个可写层里包含了你在容器运行期间做的所有修改：安装的软件、创建的文件、修改的配置等。

```bash
# 查看 Docker 的磁盘占用
docker system df

# 输出示例：
# TYPE            TOTAL     ACTIVE    SIZE      RECLAIMABLE
# Containers      15        2         2.5GB     2.3GB (92%)
# ↑ 15 个容器，只有 2 个在运行，其余 13 个已停止但占着 2.3GB 空间
```

频繁 `docker run` 又不删除，已停止的容器会越积越多。建议每周清理一次：

```bash
docker container prune
```

## 六、实战：完整走一遍容器生命周期

```bash
# 1. 创建容器（但不启动）
docker create --name lifecycle-test nginx:1.27.4
# 输出：容器 ID

# 2. 查看容器状态（Created）
docker ps -a --filter "name=lifecycle-test"
# STATUS: Created

# 3. 启动容器
docker start lifecycle-test

# 4. 查看状态（Running）
docker ps --filter "name=lifecycle-test"
# STATUS: Up X seconds

# 5. 暂停容器
docker pause lifecycle-test

# 6. 查看状态（Paused）
docker ps --filter "name=lifecycle-test"
# STATUS: Up X seconds (Paused)

# 7. 恢复容器
docker unpause lifecycle-test

# 8. 查看状态（Running）
docker ps --filter "name=lifecycle-test"
# STATUS: Up X seconds

# 9. 优雅停止
docker stop lifecycle-test

# 10. 查看状态（Stopped）
docker ps -a --filter "name=lifecycle-test"
# STATUS: Exited (0) X seconds ago

# 11. 重新启动
docker start lifecycle-test

# 12. 确认运行中
docker ps --filter "name=lifecycle-test"
# STATUS: Up X seconds

# 13. 强制杀死
docker kill lifecycle-test

# 14. 删除容器
docker rm lifecycle-test

# 15. 确认已删除
docker ps -a --filter "name=lifecycle-test"
# 没有输出
```

## 我做得对不对？

### 验证方法

请按顺序执行以下操作，**全部成功**才算通过：

```bash
# 1. 创建一个容器，启动它，暂停它，恢复它，停止它，删除它
docker create --name test-lifecycle nginx:1.27.4
docker start test-lifecycle
docker ps --filter "name=test-lifecycle"        # 应该看到 Running
docker pause test-lifecycle
docker ps --filter "name=test-lifecycle"        # 应该看到 Paused
docker unpause test-lifecycle
docker ps --filter "name=test-lifecycle"        # 应该看到 Running
docker stop test-lifecycle
docker ps -a --filter "name=test-lifecycle"     # 应该看到 Exited
docker rm test-lifecycle
docker ps -a --filter "name=test-lifecycle"     # 应该没有输出

# 2. 故意创建几个已停止的容器，然后用 prune 清理
docker run --rm alpine echo "this one auto-deletes"
docker run alpine echo "this one stays"
docker run alpine echo "this one also stays"
docker ps -a                                      # 看到两个 Exited 容器
docker container prune                            # 清理它们
docker ps -a                                      # 应该只剩还在运行的容器
```

**预期结果**：
- 每一步 `docker ps` 的 STATUS 列都符合预期状态
- `docker container prune` 清除了所有已停止容器
- 最终 `docker ps -a` 中看不到 `test-lifecycle` 和两个 alpine 容器

## 不对怎么办？

### 常见错误 1：用 `docker rm` 删除运行中的容器

```bash
docker rm my-nginx
# Error: You cannot remove a running container
```

**原因**：`docker rm` 只能删除已停止的容器。运行中的容器需要先停再删。

**解决**：

```bash
docker stop my-nginx && docker rm my-nginx
# 或
docker rm -f my-nginx   # 不推荐，除非你确定不需要优雅清理
```

### 常见错误 2：`docker kill` 后用 `docker start` 容器起不来

**现象**：`docker kill` 后容器状态是 `Exited (137)`，`docker start` 后容器还是起不来。

**原因**：退出码 137 = 128 + 9（SIGKILL 信号编号）。如果容器内的程序在收到 SIGKILL 后留下了损坏的状态文件（比如数据库的锁文件），下次启动时就可能失败。

**排查**：

```bash
# 查看容器日志
docker logs my-nginx

# 如果日志显示类似 "lock file exists" 之类的错误
# 进入容器手动清理（如果容器能启动的话）
docker start my-nginx
docker exec -it my-nginx bash
# 清理损坏的文件
```

### 常见错误 3：`docker container prune` 误删了想保留的容器

**现象**：执行 `docker container prune` 后，发现某个重要的已停止容器也被删了。

**原因**：`prune` 会删除**所有**已停止的容器，没有例外。

**预防**：

```bash
# 在 prune 之前，先看看将要删除哪些容器
docker ps -a --filter "status=exited"

# 如果确认没有问题，再执行
docker container prune

# 或者用标签/名字来管理容器，只删除特定容器
docker rm $(docker ps -a --filter "status=exited" --filter "name=test-*" -q)
```

### 常见错误 4：`docker stop` 要等 10 秒，觉得太慢

**现象**：`docker stop my-nginx` 等了 10 秒才返回。

**原因**：默认等待时间是 10 秒。容器内的程序没有正确处理 SIGTERM 信号，Docker 等了 10 秒后发 SIGKILL 才停止。

**解决**：

```bash
# 缩短等待时间
docker stop -t 3 my-nginx

# 或者直接用 kill（如果你确定不需要优雅停止）
docker kill my-nginx
```

### 常见错误 5：`docker pause` 后网络连接中断

**现象**：暂停容器后，外部请求超时。

**原因**：`pause` 冻结了容器内的所有进程，包括网络处理进程。外部客户端不知道容器被暂停了，继续发送请求，等待超时。

**这不是 bug，是设计如此。** `pause` 不适合需要维持网络连接的服务。如果你需要临时"静音"一个服务，考虑用 `docker stop` 或在应用层面做限流。

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 容器状态图 | Created → Running ⇄ Paused → Stopped → Deleted | 容器的生命周期 = 主进程的生命周期 |
| 优雅停止 | `docker stop`（SIGTERM → 等 10 秒 → SIGKILL） | 给程序清理现场的机会，推荐日常使用 |
| 暴力杀死 | `docker kill`（直接 SIGKILL） | 不给清理机会，适合容器无响应时使用 |
| 启动已存在的容器 | `docker start` 容器名 | 和 `docker run` 不同，不创建新容器 |
| 重启 | `docker restart` 容器名 | = stop + start |
| 暂停/恢复 | `docker pause` / `docker unpause` | 冻结进程，保留内存；网络连接可能中断 |
| 删除容器 | `docker rm` 容器名 | 不能删除运行中的容器（除非 `-f`） |
| 清理所有已停止容器 | `docker container prune` | 会提示确认，比直接管道安全 |
| 退出码 137 | 128 + 9（SIGKILL） | 表示容器被 `docker kill` 或 OOM Killer 杀死 |
| 暂停 vs 停止 | pause 保留内存，stop 释放内存 | pause 恢复快但网络可能断，stop 恢复慢但干净 |

## 本篇最可能出错的地方及原因

### 1. `docker stop` 和 `docker kill` 搞混

**原因**：两者都能停止容器，新手以为"差不多"。实际上 `stop` 是优雅退出（SIGTERM → SIGKILL 兜底），`kill` 是直接杀死。对于数据库这类需要清理现场的服务，用 `kill` 可能导致数据损坏。

**建议**：日常用 `stop`，只有在容器卡死、`stop` 超时时才用 `kill`。

### 2. 已停止容器占用磁盘空间

**原因**：新手以为容器停止后就不占空间了。实际上已停止容器的可写层还在磁盘上。频繁 `docker run` 测试又不清理，几周后磁盘可能被占满。

**排查**：`docker system df` 查看容器占用的磁盘空间。

**解决**：定期 `docker container prune`，或在测试容器时加 `--rm`。

### 3. `docker pause` 后服务不可用

**原因**：`pause` 冻结了所有进程，包括网络服务。外部请求会超时。这不是 bug，但很多人不知道这一点。

**建议**：`pause` 不适合生产环境中的 Web 服务。它更适合调试场景或临时冻结非关键后台任务。
