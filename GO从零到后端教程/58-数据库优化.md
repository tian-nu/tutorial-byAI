# 第58章 · 数据库优化

> "凌晨三点，运维群炸了——用户下单页面加载要30秒，监控显示MySQL的CPU使用率99%。你被叫起来排查。这时候你不能说'我也不知道，要不重启一下？'。数据库优化是后端工程师的基本功——它遵循一套清晰的排查链条：找到慢SQL→看执行计划→加索引或改写→验证效果。如果还不行，就要动表结构、上缓存、甚至分库分表。"

## 58.1 慢查询日志

慢查询日志是数据库优化的"起跑线"——先知道哪条SQL慢，才能对症下药。

### 开启慢查询日志

```sql
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
```

- `slow_query_log`：开启慢查询日志。
- `long_query_time`：超过多少秒的查询被记录（设为1秒是合理起点，上线后可调到0.5秒）。
- `slow_query_log_file`：日志文件路径。

永久生效需写入配置文件 `my.cnf`：

```ini
[mysqld]
slow_query_log = 1
long_query_time = 1
slow_query_log_file = /var/log/mysql/slow.log
log_queries_not_using_indexes = 1
```

`log_queries_not_using_indexes`：即使查询时间没超过阈值，只要没走索引就记录——对审查"是不是漏建索引了"很有用。

### 慢查询日志内容

一条典型的慢查询日志：

```
# Time: 2024-01-15T03:27:18.123456Z
# User@Host: root[root] @ localhost []
# Query_time: 3.241526  Lock_time: 0.000123  Rows_sent: 10  Rows_examined: 500000
SET timestamp=1705291638;
SELECT * FROM users WHERE email LIKE '%test%' ORDER BY created_at DESC;
```

关键信息：
- `Query_time: 3.24秒`——执行时间。
- `Lock_time: 0.000123秒`——等待锁的时间（如果这个值大，说明锁冲突严重）。
- `Rows_examined: 500000`——扫描了50万行（但只返回了10行，严重浪费）。
- `Rows_sent: 10`——返回了10行。

**优化目标**：缩小 `Rows_examined` 和 `Rows_sent` 之间的差距。扫描50万行只返回10行——典型的缺索引。

### 分析工具

原始慢查询日志文件很难读。用 `mysqldumpslow` 工具汇总：

```bash
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log
```

- `-s t`：按总执行时间排序。
- `-t 10`：只显示Top 10。

或者用 **pt-query-digest**（Percona Toolkit 中的王牌工具），生成详细的慢查询分析报告。

## 58.2 SQL优化套路

### 套路一：不用 SELECT *

```sql
SELECT * FROM users WHERE id = 1;
```

改为：

```sql
SELECT id, username, email FROM users WHERE id = 1;
```

为什么：
- 减少数据传输量（如果表有50个字段，用 `*` 全部拉出来浪费带宽和内存）。
- 让覆盖索引有发挥空间（SELECT的字段刚好在索引中 → Using index → 不回表）。
- 代码可维护性——你不知道未来会不会有人给users表加个 `BLOB biography` 字段。

### 套路二：避免在索引字段上使用函数

```sql
SELECT * FROM orders WHERE DATE(created_at) = '2024-01-01';
```

`DATE()` 包裹索引字段 → 索引失效。改为范围查询：

```sql
SELECT * FROM orders
WHERE created_at >= '2024-01-01' AND created_at < '2024-01-02';
```

### 套路三：小表驱动大表

如果必须JOIN，让小表做驱动表（外层循环）：

```sql
SELECT *
FROM small_table s
JOIN big_table b ON s.id = b.small_id;
```

而不是反过来。MySQL优化器通常能自动识别并调换顺序，但有些复杂查询它判断不准——用EXPLAIN验证。

### 套路四：深分页优化

```sql
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 1000000;
```

OFFSET太大会扫描并丢弃前100万行。改为**游标分页**：

```sql
SELECT * FROM users WHERE id > 1000000 ORDER BY id LIMIT 10;
```

前提是id有索引且递增。如果排序字段不是id（比如按created_at），可以建联合索引 `(created_at, id)` 并用同样的游标思路。

### 套路五：用UNION ALL替代OR

```sql
SELECT * FROM users WHERE email LIKE '%zhang%' OR phone LIKE '%138%';
```

如果email和phone分别有索引，OR导致只能用一个索引（或全表扫描）。拆成UNION：

```sql
SELECT * FROM users WHERE email LIKE '%zhang%'
UNION ALL
SELECT * FROM users WHERE phone LIKE '%138%';
```

各自走各自的索引。

### 套路六：避免隐式类型转换

```sql
SELECT * FROM users WHERE phone = 13800001111;
```

phone是VARCHAR，传入整数 → 隐式转换 → 索引失效。改为：

```sql
SELECT * FROM users WHERE phone = '13800001111';
```

### 套路七：用IN代替多个OR

```sql
SELECT * FROM users WHERE id = 1 OR id = 3 OR id = 5 OR id = 7;
```

改为：

```sql
SELECT * FROM users WHERE id IN (1, 3, 5, 7);
```

MySQL优化器对IN的处理比OR好得多——IN会被转化为多个等值查询。

## 58.3 表结构优化

### 字段选择原则

| 原则 | 正例 | 反例 | 原因 |
|------|------|------|------|
| 最小够用 | `age TINYINT UNSIGNED` | `age INT` | 1字节 vs 4字节，100万行差3MB |
| 非空优先 | `NOT NULL DEFAULT ''` | 默认允许NULL | NULL需要额外1字节标记，且索引复杂 |
| 定长优先 | `CHAR(11)` 存手机号 | `VARCHAR(11)` | 手机号固定长度，CHAR更高效 |
| 精确金额 | `DECIMAL(12,2)` | `FLOAT` | 浮点数有精度损失 |

### 垂直拆分

一张表有80个字段，但大多数查询只需要其中10个——这叫"宽表"。垂直拆分把这些"冷数据"（不常查的大字段）移到另一张表：

```
users:
id | username | email | phone | age | ...

user_profiles:
user_id | bio | avatar_url | address | social_links | ...
```

主表保留高频查询的字段，profile表用 `user_id` 关联，仅在需要时JOIN。好处：
- 主表行更紧凑，一页能放更多行，减少IO。
- 冷数据不污染Buffer Pool（内存缓冲池）。

### 水平拆分

当一张表达到千万级甚至亿级，即使有索引也开始吃力。水平拆分把数据按行切开——不同的行放在不同的表（或不同的数据库）里。

## 58.4 分库分表概念

当单表数据量超过**2000万行**（经验值，实际取决于硬件和业务），通常考虑分表。

### 分表方式

**取模分表（Hash）**：

```sql
users_0, users_1, users_2, users_3
```

根据 `user_id % 4` 决定数据落在哪张表。均匀分布，但扩容麻烦（取模数一变全部数据要重分配）。

**范围分表（Range）**：

```sql
users_2023, users_2024, users_2025
```

按年份范围。扩容方便（加新表就行），但数据分布不均匀（老表大新表小）。

**哈希槽分片**：

一致性哈希的变种——先预分16384个槽，然后灵活分配槽到物理节点。Redis Cluster就是这种做法。

### 分库

当单台数据库服务器扛不住了——磁盘满了、CPU爆了、网络IO到上限——把表分散到多台数据库服务器上。分库通常和分表一起做。

### 分库分表带来的新问题

| 问题 | 说明 |
|------|------|
| 跨库JOIN | 用户在DB0，订单在DB1——没法SQL JOIN，只能应用层做 |
| 分布式事务 | 扣款在DB0，加款在DB1——保证一致性极其复杂 |
| 全局唯一ID | 不能用AUTO_INCREMENT了（各库自增会冲突），需要雪花算法等方案 |
| 分片键选择 | 选user_id分片，按order_id查询就要扫所有分片 |
| 扩容迁移 | 从4库扩到8库，数据要重新分布 |

**一句话总结**：分库分表不是银弹，能用缓存、索引、读写分离解决的问题，不要动分库分表。动了以后，架构复杂度翻倍。

## 58.5 读写分离

大部分应用**读多写少**——100次浏览才1次下单。读写分离利用这个特点：写操作走主库，读操作走从库。

```
            ┌──────────┐
  写请求 →  │  主库     │ (Master)
            │  INSERT  │
            │  UPDATE  │
            │  DELETE  │
            └────┬─────┘
                 │ 主从复制（binlog）
          ┌──────┼──────┐
          ↓      ↓      ↓
      ┌──────┐┌──────┐┌──────┐
读请求→│ 从库1││ 从库2││ 从库3│ (Slaves)
      │SELECT││SELECT││SELECT│
      └──────┘└──────┘└──────┘
```

### 实现方式

**应用层路由**：代码里判断是读还是写，路由到不同数据源。

**中间件代理**：使用 ProxySQL、MaxScale 或 MySQL Router——对应用透明，中间件负责路由和负载均衡。

### 主从延迟问题

主库写入后，要经过binlog传输→从库回放，这个过程有**延迟**（通常是毫秒级，高峰可能秒级）。

典型bug：用户注册后立刻跳转到个人主页——个人主页查的是从库，但从库还没同步到这个新用户的数据 → 报"用户不存在"。

**解决方案**：
- **强制走主库**：这种"写后立刻读"的场景指定读主库。
- **延迟容忍**：显示"注册成功，数据正在同步中……"这种提示。
- 半同步复制：主库提交前确保至少一个从库收到了binlog。

### MHA / Orchestrator

读写分离架构需要高可用保障——主库挂了怎么办？

- **MHA**（Master High Availability）：主库故障自动切换到从库，小于30秒完成。
- **Orchestrator**：更现代的方案，可视化拓扑管理，故障恢复更智能。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 慢查询日志 | 开启slow_query_log，long_query_time设1秒，mysqldumpslow分析 |
