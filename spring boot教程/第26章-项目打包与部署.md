## 26.1 Maven打包

### 26.1.1 从源码到可执行jar

在第15章我们用`mvn spring-boot:run`启动过项目，但那需要Maven环境和源码。生产环境不可能装Maven再跑源码——我们需要一个**自包含的可执行文件**。

Spring Boot的`spring-boot-maven-plugin`会将项目编译后的class文件和所有依赖的jar包打成一个**Fat Jar**（也叫Uber Jar），一个文件包含一切：

```bash
# 进入项目根目录，执行打包
mvn clean package -DskipTests
```

命令拆解：

| 部分 | 含义 |
|------|------|
| `clean` | 先删除`target/`目录，确保从头构建 |
| `package` | 编译+打包，生成jar文件 |
| `-DskipTests` | 跳过测试**执行**，但测试代码仍然会被编译 |

打包成功后，在`target/`目录下会看到：

```
target/
├── classes/                    # 编译后的class文件
├── generated-sources/
├── ems-0.0.1-SNAPSHOT.jar     # ← 这就是Fat Jar！
├── ems-0.0.1-SNAPSHOT.jar.original  # 原始jar（不含依赖，很小）
└── test-classes/
```

> 🚨 **坑点：两个jar文件的区别**
> - `xxx.jar.original`是Maven默认打的原始jar，只包含你自己的class，**没有依赖**，直接`java -jar`会报`NoClassDefFoundError`
> - `xxx.jar`是Spring Boot插件打的Fat Jar，包含所有依赖，**这才是我们要部署的文件**
> - 如果你发现jar包只有几十KB，说明Spring Boot插件没生效——检查`pom.xml`中是否配置了`spring-boot-maven-plugin`

### 26.1.2 Fat Jar的内部结构

用`jar -tf ems-0.0.1-SNAPSHOT.jar | head -30`可以查看Fat Jar的内容：

```
META-INF/
META-INF/MANIFEST.MF                    # 启动入口：Main-Class=org.springframework.boot.loader.JarLauncher
org/
org/springframework/boot/loader/        # Spring Boot启动器
BOOT-INF/
BOOT-INF/classes/                       # 你自己的class文件
BOOT-INF/lib/                           # 所有第三方依赖jar
  ├── spring-boot-3.2.0.jar
  ├── mybatis-spring-boot-starter-3.0.3.jar
  ├── mysql-connector-j-8.0.33.jar
  └── ...（几十上百个jar）
```

运行`java -jar ems-0.0.1-SNAPSHOT.jar`时，Spring Boot的JarLauncher会：
1. 创建一个自定义类加载器
2. 加载`BOOT-INF/lib/`下所有依赖
3. 加载`BOOT-INF/classes/`下你的代码
4. 调用你的`main`方法启动应用

### 26.1.3 jar vs war

| 维度 | jar（推荐） | war |
|------|------------|-----|
| **打包方式** | `spring-boot-maven-plugin`打Fat Jar | `maven-war-plugin`打war包 |
| **运行方式** | `java -jar app.jar` 直接运行 | 需要部署到Tomcat等Servlet容器 |
| **内嵌服务器** | ✅ 内嵌Tomcat/Undertow | ❌ 不含，由外部容器提供 |
| **适用场景** | 微服务、容器化部署 | 需要与传统Tomcat共存的老项目 |
| **Spring Boot推荐** | ✅ 默认方式 | 仅特殊场景 |

**99%的Spring Boot项目都应该打jar包。** 只有当你必须将应用部署到已有的Tomcat集群中时，才需要打war包。

### 26.1.4 打包常见问题

**问题一：mapper.xml没打进jar包**

如果你把MyBatis的XML映射文件放在`src/main/java`目录下（和Mapper接口同目录），Maven默认不会将`java`目录下的XML文件复制到jar中——Maven只处理`src/main/resources`下的资源。

**解决方案**：在`pom.xml`中添加resources配置：

```xml
<build>
    <resources>
        <resource>
            <directory>src/main/resources</directory>
        </resource>
        <resource>
            <directory>src/main/java</directory>
            <includes>
                <include>**/*.xml</include>
            </includes>
        </resource>
    </resources>
</build>
```

> 🚨 **坑点：加了resources配置后resources目录反而不复制了？**
> - 一旦你显式配置了`<resources>`标签，Maven就**不再使用默认的资源处理规则**
> - 所以必须同时保留`src/main/resources`的resource声明，否则`application.yml`等配置文件不会被打包
> - 更好的做法：把mapper.xml统一放在`src/main/resources/mapper/`下，从根源避免这个问题

**问题二：Fat Jar体积太大**

一个简单的Spring Boot项目，Fat Jar轻松50-100MB+。这是因为所有依赖都打包在一起。如果只改了一行业务代码，重新打包后整个jar都要重新上传部署。

优化方案：
- **layertools分层jar**（第27章Docker中详细讲解）：将依赖层和应用层分离，只重建变化的层
- **瘦jar + 外部lib目录**：用`spring-boot-thin-launcher`或手动解压lib到外部目录

**问题三：跳过测试的两种方式**

| 参数 | 效果 |
|------|------|
| `-DskipTests` | 跳过测试**执行**，但测试代码仍然会被编译 |
| `-Dmaven.test.skip=true` | 跳过测试**编译和执行**，速度更快 |

生产打包推荐`-DskipTests`（至少保证测试代码能编译通过），CI/CD流水线中可以用`-Dmaven.test.skip=true`加速（前提是测试已在前面阶段跑过）。

### 26.1.5 验证打包结果

```bash
# 1. 打包
mvn clean package -DskipTests

# 2. 本地验证能否启动
java -jar target/ems-0.0.1-SNAPSHOT.jar --spring.profiles.active=dev

# 3. 看到Spring Boot Banner和启动成功日志后，Ctrl+C停止
```

> **经验法则**：部署到服务器之前，先在本地用`java -jar`方式启动一次，确认jar包没有问题。不要直接上传到服务器才发现打包有误——来回传文件浪费时间。

## 26.2 Linux基础速览

### 26.2.1 为什么必须学Linux

你的Spring Boot应用最终运行在Linux服务器上。无论是阿里云ECS、腾讯云CVM还是AWS EC2，服务器操作系统几乎都是Linux。不掌握基本的Linux操作，你连日志都看不了。

### 26.2.2 Linux目录结构速查

Linux的目录结构遵循FHS（Filesystem Hierarchy Standard），每个目录有明确用途：

| 目录 | 用途 | 示例 |
|------|------|------|
| `/etc` | 系统配置文件 | `/etc/nginx/nginx.conf`、`/etc/my.cnf` |
| `/var` | 可变数据（日志、缓存） | `/var/log/`、`/var/lib/mysql/` |
| `/opt` | 第三方应用安装目录 | `/opt/ems/`、`/opt/jdk-17/` |
| `/home` | 普通用户主目录 | `/home/ems/` |
| `/tmp` | 临时文件（重启可能清空） | 临时上传文件等 |
| `/usr` | 系统级程序和库 | `/usr/bin/java`、`/usr/local/` |
| `/root` | root用户主目录 | — |
| `/proc` | 虚拟文件系统（进程信息） | `/proc/{pid}/status` |
| `/dev` | 设备文件 | `/dev/null`、`/dev/sda1` |

**部署约定**：我们的应用放在`/opt/ems/`，日志放在`/var/log/ems/`，配置文件放在`/opt/ems/config/`。

### 26.2.3 文件权限

Linux中每个文件和目录都有三组权限：

```
-rwxr-xr-x  1 ems ems  4096 May 27 10:00 app.jar
│├──┤├──┤├──┤
│ │   │   └── 其他用户：r-x（读+执行）
│ │   └────── 同组用户：r-x（读+执行）
│ └────────── 所有者：rwx（读+写+执行）
└──────────── 文件类型：-普通文件 d目录 l链接
```

**权限与数字的对应关系**：

| 权限 | 二进制 | 数字 |
|------|--------|------|
| `r--` | 100 | 4 |
| `-w-` | 010 | 2 |
| `--x` | 001 | 1 |
| `rwx` | 111 | 7 |
| `r-x` | 101 | 5 |
| `rw-` | 110 | 6 |
| `r--` | 100 | 4 |

常用权限组合：

| 数字 | 含义 | 使用场景 |
|------|------|----------|
| `755` | 所有者rwx，其他人r-x | 可执行文件、目录 |
| `644` | 所有者rw-，其他人r-- | 普通文件、配置文件 |
| `600` | 所有者rw-，其他人无权限 | 密钥文件、敏感配置 |

```bash
chmod 755 deploy.sh       # 给部署脚本加执行权限
chmod 600 /opt/ems/config/application-prod.yml  # 保护生产配置
chown ems:ems /opt/ems/   # 将目录所有者改为ems用户
```

### 26.2.4 常用命令速查

**文件与目录操作**：

```bash
ls -la /opt/ems/          # 列出所有文件（含隐藏文件）及详细信息
cd /opt/ems/              # 切换目录
pwd                       # 显示当前目录
cp -r /opt/ems/ /backup/  # 递归复制目录
mv app.jar app.jar.bak    # 移动/重命名
rm -rf /tmp/old/          # 递归强制删除目录
mkdir -p /opt/ems/config  # 递归创建目录（父目录不存在也创建）
touch deploy.sh           # 创建空文件
```

> 🚨 **坑点：`rm -rf /` 是毁灭性操作！**
> - 这条命令会删除整个文件系统，服务器直接报废
> - 更常见的事故：`rm -rf / opt/ems/`（多了一个空格，变成删除`/`和`opt/ems/`）
> - **安全习惯**：
>   - 永远先`ls`确认路径，再`rm`
>   - 用`rm -ri`交互式删除（每个文件都确认）
>   - 重要操作前先备份

**压缩与解压**：

```bash
tar -czvf ems-backup.tar.gz /opt/ems/   # 压缩
tar -xzvf ems-backup.tar.gz -C /opt/    # 解压到/opt/
```

参数记忆：**c**reate / e**x**tract / **z**ip(gzip) / **v**erbose / **f**ile

**进程与系统监控**：

```bash
ps -ef | grep java              # 查找Java进程
top                             # 实时查看CPU/内存占用（按q退出）
free -h                         # 查看内存使用情况
df -h                           # 查看磁盘使用情况
```

**日志查看**：

```bash
tail -f /var/log/ems/app.log    # 实时查看日志末尾（Ctrl+C停止）
tail -100 /var/log/ems/app.log  # 查看最后100行
grep "ERROR" /var/log/ems/app.log           # 搜索包含ERROR的行
grep -A 5 "Exception" /var/log/ems/app.log  # 搜索并显示后5行上下文
find /opt/ems/ -name "*.log" -mtime +7      # 查找7天前的日志文件
```

**服务管理**：

```bash
systemctl start nginx       # 启动服务
systemctl stop nginx        # 停止服务
systemctl restart nginx     # 重启服务
systemctl status nginx      # 查看服务状态
systemctl enable nginx      # 设置开机自启
systemctl disable nginx     # 取消开机自启
journalctl -u ems -f        # 实时查看ems服务的系统日志
journalctl -u ems --since "1 hour ago"  # 查看最近1小时的日志
```

### 26.2.5 SSH远程连接与文件传输

**SSH连接**：

```bash
ssh ems@192.168.1.100 -p 22       # 默认端口22可省略
ssh ems@192.168.1.100 -p 2222     # 指定端口
```

**SCP文件传输**：

```bash
# 上传jar包到服务器
scp target/ems-0.0.1-SNAPSHOT.jar ems@192.168.1.100:/opt/ems/app.jar

# 从服务器下载日志
scp ems@192.168.1.100:/var/log/ems/app.log ./app.log

# 传输整个目录
scp -r ./config/ ems@192.168.1.100:/opt/ems/config/
```

**rsync增量同步**（比scp快得多，只传变化的部分）：

```bash
rsync -avz --progress target/ems-0.0.1-SNAPSHOT.jar ems@192.168.1.100:/opt/ems/app.jar
```

> **小技巧**：配置SSH免密登录，避免每次输入密码：
> ```bash
> # 本地生成密钥对
> ssh-keygen -t ed25519
> # 将公钥复制到服务器
> ssh-copy-id ems@192.168.1.100
> ```

## 26.3 服务器环境搭建

### 26.3.1 JDK 17安装

**方式一：包管理器安装（推荐）**

Ubuntu/Debian：

```bash
sudo apt update
sudo apt install openjdk-17-jdk -y
```

CentOS/RHEL：

```bash
sudo yum install java-17-openjdk java-17-openjdk-devel -y
```

**方式二：手动安装（适用于无外网或需特定版本）**

```bash
# 1. 下载JDK（从Adoptium获取）
wget https://github.com/adoptium/temurin17-binaries/releases/download/jdk-17.0.9%2B9/OpenJDK17U-jdk_x64_linux_hotspot_17.0.9_9.tar.gz

# 2. 解压到/opt
sudo tar -xzvf OpenJDK17U-jdk_x64_linux_hotspot_17.0.9_9.tar.gz -C /opt/

# 3. 配置环境变量
sudo vim /etc/profile.d/jdk.sh
```

写入以下内容：

```bash
export JAVA_HOME=/opt/jdk-17.0.9+9
export PATH=$JAVA_HOME/bin:$PATH
```

```bash
# 4. 使配置生效
source /etc/profile.d/jdk.sh

# 5. 验证
java -version
# openjdk version "17.0.9" 2023-10-17
```

> 🚨 **坑点：确认JAVA_HOME指向正确版本**
> - 服务器上可能存在多个JDK版本（系统自带OpenJDK 8 + 你安装的17）
> - `which java`查看实际使用的是哪个
> - 如果指向系统自带的旧版本，检查`/etc/alternatives/java`的链接
> - Spring Boot 3.x **必须** JDK 17+，用JDK 8启动直接报错

### 26.3.2 MySQL 8安装

Ubuntu/Debian：

```bash
sudo apt update
sudo apt install mysql-server -y
sudo systemctl start mysql
sudo systemctl enable mysql
```

CentOS/RHEL：

```bash
sudo yum install mysql-server -y
sudo systemctl start mysqld
sudo systemctl enable mysqld
```

**安装后安全配置**：

```bash
sudo mysql_secure_installation
# 按提示设置root密码、移除匿名用户、禁止远程root登录等
```

> 🚨 **坑点：MySQL 8安装后root默认只能Unix Socket登录**
> - MySQL 8的root用户默认使用`auth_socket`（Ubuntu）或`caching_sha2_password`插件认证
> - 这意味着只有系统root用户或mysql用户才能登录MySQL
> - 应用程序用JDBC连接时，需要密码认证的用户
>
> **解决方案**：创建专用用户或修改root认证方式：
>
> ```bash
> # 用sudo登录MySQL
> sudo mysql
> ```
>
> ```sql
> -- 方案一：创建专用用户（推荐）
> CREATE USER 'ems'@'%' IDENTIFIED BY 'YourStrongPassword123!';
> CREATE DATABASE ems CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
> GRANT ALL PRIVILEGES ON ems.* TO 'ems'@'%';
> FLUSH PRIVILEGES;
>
> -- 方案二：修改root允许远程登录（不推荐用于生产）
> ALTER USER 'root'@'localhost' IDENTIFIED WITH caching_sha2_password BY 'YourPassword123!';
> CREATE USER 'root'@'%' IDENTIFIED WITH caching_sha2_password BY 'YourPassword123!';
> GRANT ALL PRIVILEGES ON *.* TO 'root'@'%';
> FLUSH PRIVILEGES;
> ```

**导入EMS数据库**：

```bash
mysql -u ems -p ems < ems-schema.sql
```

### 26.3.3 Redis 7安装

Ubuntu/Debian：

```bash
sudo apt install redis-server -y
sudo systemctl start redis-server
sudo systemctl enable redis-server
```

CentOS/RHEL（需EPEL源）：

```bash
sudo yum install epel-release -y
sudo yum install redis -y
sudo systemctl start redis
sudo systemctl enable redis
```

> 🚨 **坑点：Redis默认bind 127.0.0.1只能本地连接**
> - Redis配置文件`/etc/redis/redis.conf`中默认`bind 127.0.0.1`
> - 如果Spring Boot应用和Redis在同一台服务器，这没问题
> - 如果应用在另一台服务器，需要修改bind或注释掉
> - **同时必须设置密码**（Redis默认无密码，公网暴露=裸奔）
>
> ```bash
> sudo vim /etc/redis/redis.conf
> ```
>
> 修改以下配置：
>
> ```ini
> # bind 127.0.0.1              # 注释掉或改为具体IP
> bind 0.0.0.0                  # 允许所有IP连接（配合密码使用）
> requirepass YourRedisPassword123!   # 设置密码
> ```
>
> ```bash
> sudo systemctl restart redis-server
> ```

验证Redis连接：

```bash
redis-cli -a YourRedisPassword123!
# 127.0.0.1:6379> ping
# PONG
```

### 26.4.1 上传jar包

```bash
# 在本地执行
scp target/ems-0.0.1-SNAPSHOT.jar ems@192.168.1.100:/opt/ems/app.jar
```

在服务器上创建必要的目录结构：

```bash
sudo mkdir -p /opt/ems/config
sudo mkdir -p /var/log/ems
sudo chown -R ems:ems /opt/ems/
sudo chown -R ems:ems /var/log/ems/
```

### 26.4.2 nohup方式启动

最简单的启动方式：

```bash
nohup java -jar /opt/ems/app.jar > /var/log/ems/app.log 2>&1 &
```

命令拆解：

| 部分 | 含义 |
|------|------|
| `nohup` | 忽略SIGHUP信号，关闭终端后进程不退出 |
| `java -jar /opt/ems/app.jar` | 启动应用 |
| `>` | 重定向标准输出 |
| `/var/log/ems/app.log` | 日志文件 |
| `2>&1` | 将标准错误也重定向到标准输出（合并到一个文件） |
| `&` | 后台运行 |

> 🚨 **坑点：nohup默认输出到nohup.out → 磁盘写满！**
> - 如果不指定重定向，nohup会将所有输出写入当前目录的`nohup.out`文件
> - 这个文件**永远不会轮转**，应用跑几个月后可能长到几十GB
> - 日志框架（Logback）的输出不受影响（它自己写文件），但System.out和未捕获异常的堆栈会写入nohup.out
> - **解决方案**：
>   - 方案一：重定向到`/dev/null`丢弃（`> /dev/null 2>&1`），完全依赖日志框架
>   - 方案二：重定向到指定文件，配合logrotate做日志轮转
>   - 方案三：**用systemd管理（推荐）**，让journald接管日志

**查看和停止进程**：

```bash
# 查看Java进程
ps -ef | grep java
# ems  12345  1  3 10:00 ?  00:00:15 java -jar /opt/ems/app.jar

# 查看Java进程（JDK自带工具，更直观）
jps
# 12345 app.jar

# 停止进程
kill 12345          # 发送SIGTERM(15)信号，优雅关闭
```

**kill vs kill -9**：

| 命令 | 信号 | 行为 | 适用场景 |
|------|------|------|----------|
| `kill PID` | SIGTERM(15) | 请求进程优雅退出，Spring Boot会执行shutdown hook | **推荐**，正常停止 |
| `kill -9 PID` | SIGKILL(9) | 操作系统立即强制杀掉进程，无法拦截 | 最后手段，进程卡死时 |

> 🚨 **坑点：kill -9可能导致数据丢失**
> - Spring Boot收到SIGTERM时，会执行`@PreDestroy`、shutdown hook等清理逻辑
>   - 关闭数据库连接池
>   - 完成正在处理的请求
>   - 保存缓存数据
> - `kill -9`直接杀进程，以上清理全部跳过
>   - 数据库连接未正常关闭 → 连接泄露
>   - 正在写入的文件可能损坏
>   - 正在处理的事务可能丢失

### 26.4.3 Systemd服务管理（推荐！）

nohup方式简单粗暴，但缺点明显：服务器重启后应用不会自动启动，进程崩溃也不会自动恢复。**systemd是Linux系统的服务管理器**，将应用注册为systemd服务后，可以实现开机自启、自动重启、统一日志管理。

**第一步：创建专用用户**（不要用root运行应用）

```bash
sudo useradd -r -s /sbin/nologin ems    # 创建系统用户，不允许登录
sudo chown -R ems:ems /opt/ems/
sudo chown -R ems:ems /var/log/ems/
```

**第二步：编写服务文件**

创建`/etc/systemd/system/ems.service`：

```ini
[Unit]
Description=EMS Application
After=network.target mysql.service redis.service
Wants=mysql.service redis.service

[Service]
Type=simple
User=ems
Group=ems
WorkingDirectory=/opt/ems
ExecStart=/usr/bin/java -jar /opt/ems/app.jar \
    --spring.profiles.active=prod \
    -Xms256m -Xmx512m \
    -Duser.timezone=Asia/Shanghai
ExecStop=/bin/kill -SIGTERM $MAINPID
Restart=on-failure
RestartSec=10
StandardOutput=journal
StandardError=journal

# 安全加固
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/opt/ems /var/log/ems

[Install]
WantedBy=multi-user.target
```

配置项详解：

| 配置 | 含义 |
|------|------|
| `After=network.target` | 网络就绪后才启动 |
| `Type=simple` | 主进程就是服务进程 |
| `User=ems` | 以ems用户身份运行（安全） |
| `Restart=on-failure` | 非正常退出时自动重启 |
| `RestartSec=10` | 重启前等待10秒 |
| `StandardOutput=journal` | 标准输出交给systemd日志 |
| `WantedBy=multi-user.target` | 多用户模式下启动 |

**第三步：启用并启动服务**

```bash
# 重新加载systemd配置（修改.service文件后必须执行）
sudo systemctl daemon-reload

# 设置开机自启
sudo systemctl enable ems

# 启动服务
sudo systemctl start ems

# 查看状态
sudo systemctl status ems
```

**第四步：查看日志**

```bash
# 实时查看日志
journalctl -u ems -f

# 查看最近100行
journalctl -u ems -n 100

# 查看今天的日志
journalctl -u ems --since today

# 查看指定时间段的日志
journalctl -u ems --since "2026-05-27 10:00:00" --until "2026-05-27 12:00:00"
```

> **对比nohup和systemd**：
>
> | 维度 | nohup | systemd |
> |------|-------|---------|
> | 开机自启 | ❌ | ✅ |
> | 崩溃自动重启 | ❌ | ✅ |
> | 日志管理 | 手动重定向 | journalctl统一管理 |
> | 启停命令 | kill PID | systemctl start/stop |
> | 运行用户 | 当前用户 | 可指定专用用户 |
> | 生产环境 | ❌ 不推荐 | ✅ 推荐 |

### 26.4.4 外部配置文件

生产环境的配置和开发环境不同（数据库地址、密码、日志级别等）。Spring Boot支持在jar包外部放置配置文件，外部配置优先级高于jar包内部：

```
/opt/ems/
├── app.jar                              # jar包
├── config/
│   └── application-prod.yml             # 外部配置（优先级最高）
└── application-prod.yml                 # 外部配置（次高优先级）
```

Spring Boot配置文件加载优先级（从高到低）：

1. `jar包同目录/config/`下的配置
2. `jar包同目录/`下的配置
3. `jar包内classpath:/config/`下的配置
4. `jar包内classpath:/`下的配置

这样你只需要修改外部配置文件，无需重新打包：

```yaml
# /opt/ems/config/application-prod.yml
server:
  port: 8080

spring:
  datasource:
    url: jdbc:mysql://localhost:3306/ems?useSSL=false&serverTimezone=Asia/Shanghai&characterEncoding=utf8mb4
    username: ems
    password: ${DB_PASSWORD}    # 从环境变量读取，不写明文
  data:
    redis:
      host: localhost
      port: 6379
      password: ${REDIS_PASSWORD}

logging:
  file:
    name: /var/log/ems/app.log
  level:
    root: INFO
    com.example.ems: DEBUG
```

## 26.5 Nginx反向代理

### 26.5.1 为什么需要Nginx

你的Spring Boot应用运行在8080端口，但用户访问的是80或443端口。Nginx作为反向代理，架在用户和应用之间：

```
用户浏览器 → Nginx(80/443) → Spring Boot(8080)
                  │
                  └── 还能做：负载均衡、静态资源服务、SSL终止、限流
```

Nginx的核心价值：
- **安全**：应用不直接暴露，Nginx挡在前面
- **性能**：Nginx处理静态资源的性能远超Tomcat
- **灵活**：一个Nginx可以代理多个后端应用
- **SSL终止**：HTTPS加解密在Nginx完成，后端只需HTTP

### 26.5.2 Nginx配置结构

Nginx配置采用嵌套结构，核心层级为`http → server → location`：

```
nginx.conf
├── events { ... }              # 事件模型配置
└── http {                      # HTTP服务配置
    ├── upstream backend { ... } # 负载均衡组
    ├── server { ... }          # 虚拟主机1（HTTP）
    └── server { ... }          # 虚拟主机2（HTTPS）
        ├── location / { ... }  # 路由规则1
        └── location /api/ { ... }  # 路由规则2
    }
```

### 26.5.3 反向代理完整配置

创建`/etc/nginx/conf.d/ems.conf`：

```nginx
# HTTP → HTTPS 重定向
server {
    listen 80;
    server_name api.example.com;
    return 301 https://$host$request_uri;
}

# HTTPS 主配置
server {
    listen 443 ssl http2;
    server_name api.example.com;

    # SSL证书配置
    ssl_certificate /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;

    # SSL优化配置
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    # API请求代理到Spring Boot
    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 超时设置
        proxy_connect_timeout 30s;
        proxy_read_timeout 60s;
        proxy_send_timeout 30s;

        # 缓冲设置
        proxy_buffering on;
        proxy_buffer_size 4k;
        proxy_buffers 8 4k;
    }

    # 静态资源直接由Nginx处理（性能更好）
    location /static/ {
        alias /opt/ems/static/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # 健康检查端点
    location /health {
        proxy_pass http://127.0.0.1:8080/actuator/health;
    }
}
```

> 🚨 **坑点：proxy_pass路径末尾的`/`**
>
> 这是Nginx最容易踩的坑之一：
>
> ```nginx
> # 情况一：proxy_pass末尾有 /
> location /api/ {
>     proxy_pass http://127.0.0.1:8080/;
> }
> # 请求 /api/employees → 转发到 http://127.0.0.1:8080/employees
> # /api/ 被替换掉了
>
> # 情况二：proxy_pass末尾无 /
> location /api/ {
>     proxy_pass http://127.0.0.1:8080;
> }
> # 请求 /api/employees → 转发到 http://127.0.0.1:8080/api/employees
> # /api/ 被保留了
> ```
>
> **规则**：`proxy_pass`末尾有URI（哪怕只是一个`/`），Nginx会将location匹配的部分替换掉；末尾没有URI，则原样拼接。
>
> 对于Spring Boot项目，如果Controller的路径是`/api/employees`，应该用**情况二**（末尾无`/`）。

### 26.5.4 负载均衡入门

当单台服务器扛不住时，可以部署多个Spring Boot实例，Nginx做负载均衡：

```nginx
upstream ems_backend {
    # 默认轮询策略
    server 192.168.1.101:8080 weight=3;   # 权重3，接收3/4的请求
    server 192.168.1.102:8080 weight=1;   # 权重1，接收1/4的请求

    # 备用服务器（主服务都挂了才启用）
    server 192.168.1.103:8080 backup;

    # 保持会话一致性（同一用户始终路由到同一台）
    ip_hash;
}

server {
    listen 80;
    server_name api.example.com;

    location /api/ {
        proxy_pass http://ems_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

Nginx负载均衡策略：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 轮询（默认） | 依次分配 | 服务器性能相近 |
| weight | 按权重分配 | 服务器性能不同 |
| ip_hash | 按IP哈希固定分配 | 需要会话保持 |
| least_conn | 分配给连接数最少的 | 请求处理时间差异大 |

### 26.5.5 验证Nginx配置

```bash
# 测试配置文件语法
sudo nginx -t
# nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
# nginx: configuration file /etc/nginx/nginx.conf test is successful

# 重新加载配置（不中断服务）
sudo nginx -s reload

# 或者重启Nginx
sudo systemctl restart nginx
```

## 26.6 HTTPS配置

### 26.6.1 为什么必须HTTPS

- **安全**：HTTP是明文传输，密码、Token等敏感信息可被中间人截获
- **合规**：微信小程序、App Store等平台强制要求HTTPS
- **SEO**：Google等搜索引擎优先展示HTTPS网站
- **HTTP/2**：主流浏览器只在HTTPS下支持HTTP/2

### 26.6.2 Let's Encrypt免费证书

Let's Encrypt提供免费的SSL证书，配合certbot工具实现自动申请和续期：

```bash
# 安装certbot
sudo apt install certbot python3-certbot-nginx -y    # Ubuntu
sudo yum install certbot python3-certbot-nginx -y     # CentOS

# 自动申请证书并配置Nginx（一键搞定）
sudo certbot --nginx -d api.example.com

# 按提示操作：
# 1. 输入邮箱（用于证书过期提醒）
# 2. 同意服务条款
# 3. 选择是否将HTTP重定向到HTTPS（选是）
```

certbot会自动修改Nginx配置，添加SSL相关指令。

### 26.6.3 手动配置SSL证书

如果你使用云服务商的证书（如阿里云免费证书），需要手动配置：

```bash
# 创建证书目录
sudo mkdir -p /etc/nginx/ssl

# 上传证书文件
scp cert.pem key.pem ems@192.168.1.100:/tmp/
sudo mv /tmp/cert.pem /etc/nginx/ssl/
sudo mv /tmp/key.pem /etc/nginx/ssl/
sudo chmod 600 /etc/nginx/ssl/key.pem    # 私钥权限必须严格
```

Nginx SSL配置已在26.5.3节给出，这里补充SSL优化要点：

```nginx
# SSL协议版本：只允许TLS 1.2和1.3（禁用不安全的SSL 3.0、TLS 1.0/1.1）
ssl_protocols TLSv1.2 TLSv1.3;

# 加密套件：只允许高强度加密
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;

# HSTS：告诉浏览器未来一年内只使用HTTPS访问
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

### 26.6.4 证书自动续期

> 🚨 **坑点：证书过期忘记续期 → 用户无法访问**
> - Let's Encrypt证书有效期只有90天
> - 忘记续期会导致浏览器显示"不安全"警告，用户无法正常访问
> - **certbot自动续期**：

```bash
# 测试续期命令（不会真正续期，只是验证流程）
sudo certbot renew --dry-run

# 如果测试通过，certbot会自动添加cron定时任务
# 查看定时任务
sudo systemctl list-timers | grep certbot

# 手动续期
sudo certbot renew
sudo nginx -s reload    # 续期后重新加载Nginx
```

certbot安装时通常会自动创建systemd timer，每天检查两次是否需要续期。如果没有，手动添加：

```bash
# 添加cron任务：每天凌晨2点检查续期
sudo crontab -e
# 添加以下行：
0 2 * * * certbot renew --quiet --post-hook "nginx -s reload"
```

## 26.7 常见部署问题排查

### 26.7.1 端口冲突

**现象**：启动时报`Port 8080 already in use`

```bash
# 查看端口占用
sudo netstat -tlnp | grep 8080
# tcp6  0  0  :::8080  :::*  LISTEN  12345/java

# 或者用ss命令（更快）
sudo ss -tlnp | grep 8080

# 解决方案一：杀掉占用端口的进程
kill 12345

# 解决方案二：修改应用端口
# application-prod.yml
server:
  port: 8081
```

### 26.7.2 内存不足

**现象**：`java.lang.OutOfMemoryError: Java heap space`或启动直接被系统杀掉

```bash
# 查看内存使用
free -h
#               total    used    free    shared  buff/cache   available
# Mem:          1.9Gi   1.5Gi   100Mi   12Mi       300Mi        200Mi
# Swap:         0B      0B      0B

# 查看Java进程内存占用
ps -o pid,rss,command -p $(pgrep -f app.jar)
# PID   RSS    COMMAND
# 12345 524288 java -jar /opt/ems/app.jar   # RSS单位KB，即512MB
```

**解决方案**：限制JVM堆内存

```bash
java -Xms128m -Xmx512m -jar /opt/ems/app.jar
```

| 参数 | 含义 | 建议 |
|------|------|------|
| `-Xms128m` | JVM初始堆内存 | 设为最大堆的50%-75% |
| `-Xmx512m` | JVM最大堆内存 | 1GB内存的服务器设512m，留一半给系统和其他进程 |
| `-Xss256k` | 每个线程的栈大小 | 默认1MB，微服务可适当减小 |

### 26.7.3 时区问题

**现象**：数据库中存储的时间比实际时间少8小时

```bash
# 查看服务器时区
timedatectl
# Time zone: UTC (UTC, +0000)   ← 问题所在！

# 修改时区为中国标准时间
sudo timedatectl set-timezone Asia/Shanghai
```

JVM启动参数也必须指定时区：

```bash
java -Duser.timezone=Asia/Shanghai -jar /opt/ems/app.jar
```

> 🚨 **坑点：时区问题可能在三个层面出现**
> - **操作系统时区**：`timedatectl`查看和修改
> - **JVM时区**：`-Duser.timezone=Asia/Shanghai`启动参数
> - **数据库时区**：MySQL连接URL中的`serverTimezone=Asia/Shanghai`
> - 三个层面必须统一，否则时间对不上

### 26.7.4 后台进程被杀（OOM Killer）

**现象**：应用运行一段时间后突然消失，systemd显示`code=killed status=9`

```bash
# 查看系统日志，搜索OOM Killer记录
sudo dmesg | grep -i "out of memory"
# Out of memory: Kill process 12345 (java) score 800 or sacrifice child

# 或者
sudo journalctl -k | grep -i "oom"
```

**原因**：Linux内核在内存不足时，会自动杀掉占用内存最多的进程（OOM Killer）。Java应用通常是头号目标。

**解决方案**：

```bash
# 方案一：增加swap空间（临时缓解）
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
# 永久生效
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# 方案二：调整OOM策略（让内核尽量不杀Java进程）
sudo vim /etc/systemd/system/ems.service
# 在[Service]中添加：
OOMScoreAdjust=-500    # 降低被杀优先级（-1000到1000，越低越不容易被杀）

sudo systemctl daemon-reload
sudo systemctl restart ems

# 方案三：升级服务器内存（根本解决）
```

## EMS v9（上）：传统部署版

现在让我们将EMS v8（Redis缓存版）部署到Linux服务器上，完成传统方式的部署。

### 部署目录结构

```
/opt/ems/
├── app.jar                              # Fat Jar
├── config/
│   └── application-prod.yml             # 生产配置
└── static/                              # 静态资源（如有前端）
```

### 生产配置文件

`/opt/ems/config/application-prod.yml`：

```yaml
server:
  port: 8080
  shutdown: graceful                     # 优雅关闭

spring:
  datasource:
    url: jdbc:mysql://localhost:3306/ems?useSSL=false&serverTimezone=Asia/Shanghai&characterEncoding=utf8mb4
    username: ems
    password: ${DB_PASSWORD}
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
  data:
    redis:
      host: localhost
      port: 6379
      password: ${REDIS_PASSWORD}
  lifecycle:
    timeout-per-shutdown-phase: 30s      # 优雅关闭超时时间

mybatis:
  mapper-locations: classpath:mapper/**/*.xml
  configuration:
    map-underscore-to-camel-case: true

logging:
  file:
    name: /var/log/ems/app.log
  level:
    root: WARN
    com.example.ems: INFO

springdoc:
  swagger-ui:
    enabled: false                       # 生产环境关闭Swagger
```

### Systemd服务文件

`/etc/systemd/system/ems.service`：

```ini
[Unit]
Description=EMS Application
After=network.target mysql.service redis.service
Wants=mysql.service redis.service

[Service]
Type=simple
User=ems
```

### 一键部署脚本

`/opt/ems/deploy.sh`：

```bash
#!/bin/bash
set -e

APP_NAME="ems"
JAR_NAME="app.jar"
DEPLOY_DIR="/opt/ems"
BACKUP_DIR="/opt/ems/backup"
LOG_DIR="/var/log/ems"

echo "===== 开始部署 ${APP_NAME} ====="

# 1. 备份当前版本
if [ -f "${DEPLOY_DIR}/${JAR_NAME}" ]; then
    BACKUP_FILE="${BACKUP_DIR}/${JAR_NAME}.$(date +%Y%m%d%H%M%S)"
    mkdir -p ${BACKUP_DIR}
    cp ${DEPLOY_DIR}/${JAR_NAME} ${BACKUP_FILE}
    echo "已备份到 ${BACKUP_FILE}"
fi

# 2. 停止服务
echo "停止服务..."
sudo systemctl stop ${APP_NAME} || true
sleep 5

# 3. 替换jar包（从上传目录复制）
if [ -f "/tmp/${JAR_NAME}" ]; then
    cp /tmp/${JAR_NAME} ${DEPLOY_DIR}/${JAR_NAME}
    echo "已替换jar包"
else
    echo "错误：未找到 /tmp/${JAR_NAME}"
    exit 1
fi

# 4. 启动服务
echo "启动服务..."
sudo systemctl start ${APP_NAME}

# 5. 等待启动并检查状态
echo "等待应用启动..."
sleep 15

if sudo systemctl is-active --quiet ${APP_NAME}; then
    echo "===== 部署成功！====="
    sudo systemctl status ${APP_NAME} --no-pager
else
    echo "===== 部署失败！====="
    sudo journalctl -u ${APP_NAME} -n 50 --no-pager
    exit 1
fi
```

```bash
chmod +x /opt/ems/deploy.sh

# 使用方式：
# 1. 先scp上传新jar到/tmp/app.jar
# 2. 执行部署脚本
./deploy.sh
```
