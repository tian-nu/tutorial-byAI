## 一、目标与完成效果

**一句话目标**：用 10 行代码写出人生中第一个后端程序，让浏览器访问 `http://127.0.0.1:8000` 时显示 `{"message": "Hello World"}`。

**完成后的可观测效果**：
- 你在浏览器地址栏输入一个地址，回车，浏览器返回一段 JSON 数据（白色背景的纯文本）。
- 你再输入另一个地址（末尾加 `/docs`），浏览器显示一个精美的 API 文档页面，你甚至可以在上面直接点按钮测试接口。
- 你改一行代码，保存，页面立刻刷新——不用手动重启服务器。

## 三、分步操作

### 步骤 1：新建 `main.py`，写入代码

打开你的编辑器（推荐 VS Code），在 `blog_backend/` 目录下新建一个文件，命名为 `main.py`。

把下面这 10 行代码一个字一个字敲进去（**不要复制粘贴**，敲一遍能帮你建立肌肉记忆）：

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/")
def read_root():
    return {"message": "Hello World"}
```

保存文件。

> **为什么让你手敲？** 就像学骑自行车，看别人骑一百遍不如自己上去摔一次。手敲代码能让你注意到每个字符：括号是圆还是方？冒号在哪儿？缩进对不对？这些细节复制粘贴时你根本不会在意，但报错的时候它们就是元凶。

---

### 步骤 2：逐行解释——这 10 行代码到底在说什么

我们用"开一家小餐厅"的比喻来理解每一行。

```python
from fastapi import FastAPI
```

**翻译**：从 `fastapi` 这个工具箱里，拿出一个叫 `FastAPI` 的工具。

**比喻**：你要开餐厅，先去工商局领一张营业执照模板。`FastAPI` 就是那张模板——你拿到它才能开始"注册"你的餐厅（创建你的网站应用）。

---

```python
app = FastAPI()
```

**翻译**：用刚拿到的模板，创建一份属于你的"营业执照"实体，命名为 `app`。

**比喻**：你在模板上填了餐厅名字，拿到了一张真实的营业执照。从此以后，这张执照（`app`）就代表你的整个餐厅（你的整个网站后端）。

---

```python
@app.get("/")
def read_root():
    return {"message": "Hello World"}
```

这三行要放在一起理解。

**`@app.get("/")`**——这就是传说中的**装饰器**（decorator）。先别被这个词吓到。你现在只需要知道它的意思：**"当有人访问网站首页（路径是 `/`）时，执行下面这个函数"**。

> ⚠️ **装饰器原理先别管**。你现在把它当成一个"标签"或"路牌"就好——你往函数上贴这个标签，FastAPI 就知道"哦，有人来首页的时候要叫这个函数去接待"。装饰器背后的 `@` 语法、闭包、高阶函数这些东西，我们会在后续章节专门讲。**你现在要学的不是造发动机，是开车。**

**比喻**：你在餐厅门口挂了一个牌子，上面写着：

> "当有客人走进大门（访问 `/`）时，叫服务员 `read_root` 去接待。"

`return {"message": "Hello World"}` 就是服务员端上来的菜——一份 JSON 格式的数据。

---

**`def read_root():`**——这就是**路径操作函数**（path operation function）。它就是一个普通的 Python 函数，唯一的特别之处是：它被 `@app.get("/")` 这个路牌标记了，所以 FastAPI 会在特定时刻调用它。

**比喻**：`read_root` 是你雇的餐厅服务员。客人来了，他就端上一盘菜（返回数据）。至于他怎么知道客人来了、怎么把菜送过去——那是 FastAPI 这个"餐厅经理"的事，你不用操心。

---

**术语快速索引**：
| 术语 | 通俗解释 | 详细见附录 |
|------|----------|-----------|
| 装饰器（decorator） | 给函数贴的"路牌"，告诉 FastAPI 什么路径触发什么函数 | 附录 A |
| 路径操作函数（path operation function） | 被装饰器标记的普通 Python 函数，负责处理某个路径的请求 | 附录 A |
| JSON | 一种数据格式，看起来像 Python 字典，是前后端通信的"通用语言" | — |

### 步骤 3：启动服务器

打开终端（PowerShell 或命令提示符），确保你在项目根目录（能看到 `blog_backend/` 文件夹的那一层），并且虚拟环境已激活（终端前面有 `(venv)` 字样）。

然后执行：

```bash
uvicorn blog_backend.main:app --reload
```

**这是什么意思？拆开看：**

- `uvicorn`：一个**服务器程序**。你的代码（FastAPI 应用）只是"菜谱"，需要一个厨师（服务器）来执行它。
- `blog_backend.main:app`：告诉 uvicorn **去哪儿找你的应用**。
  - `blog_backend.main` = 文件夹 `blog_backend` 里的文件 `main.py`（去掉 `.py` 后缀）
  - `:app` = 文件里叫 `app` 的那个变量（就是你在步骤 1 里创建的 `app = FastAPI()`）
- `--reload`：**热重载**开关。打开之后，你改代码保存，服务器自动重启，不用手动关掉再开。

> 🔥 **这是最容易出错的点**：如果你把文件直接放在了项目根目录（没有 `blog_backend/` 文件夹），命令就是 `uvicorn main:app --reload`。`main:app` 里的 `main` **永远等于你的 Python 文件名（去掉 `.py`）**，`app` **永远等于你的 FastAPI() 实例变量名**。

**运行成功后你会看到类似这样的输出：**

```
INFO:     Will watch for changes in these directories: ['D:\\Project\\...']
INFO:     Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)
INFO:     Started reloader process [12345] using WatchFiles
INFO:     Started server process [67890]
INFO:     Waiting for application startup.
INFO:     Application startup complete.
```

看到 `running on http://127.0.0.1:8000` 就说明成功了！

---

#### ❌ 常见错误 → ✅ 解决方案

| 错误信息 | 原因 | 解决 |
|----------|------|------|
| `'uvicorn' 不是内部或外部命令` | 虚拟环境没激活 | 先执行 `venv\Scripts\activate` |
| `Could not import module 'blog_backend.main'` | 你执行命令的目录不对 | `cd` 到能看到 `blog_backend/` 文件夹的目录 |
| `Address already in use` | 端口 8000 被占用了 | 换个端口：`uvicorn blog_backend.main:app --reload --port 8001` |
| 改了代码没反应 | `--reload` 忘加了 | 按 `Ctrl+C` 停止，重新加上 `--reload` 执行 |

### 步骤 4：浏览器访问 `http://127.0.0.1:8000`

打开你最喜欢的浏览器（Chrome、Edge、Firefox 都行），在地址栏输入：

```
http://127.0.0.1:8000
```

回车。

你应该看到的是一页白底黑字（或浏览器默认的 JSON 显示风格）：

```json
{"message": "Hello World"}
```

**🎉 恭喜！你的第一个后端程序跑起来了！**

**到底发生了什么？** 用餐厅比喻复盘一下：

1. 你（浏览器）走进餐厅大门（访问 `http://127.0.0.1:8000/`）
2. 门口的引导员（FastAPI 路由系统）看了一眼门牌号 `/`，查表发现："这个客人应该由服务员 `read_root` 接待"
3. 服务员 `read_root` 跑过来，端上一盘菜：`{"message": "Hello World"}`
4. 你（浏览器）吃到了这盘菜（显示了 JSON 数据）

> **`127.0.0.1` 是什么？** 这是计算机的"自言自语地址"。它代表"我自己"。`127.0.0.1:8000` 的意思是："在我自己的电脑上，通过 8000 号门（端口）找我自己的程序"。你的浏览器和你的后端程序在**同一台电脑**上，所以用 `127.0.0.1`。

### 步骤 5：打开 Swagger 自动文档——你什么都没做，文档就有了

在浏览器地址栏输入：

```
http://127.0.0.1:8000/docs
```

回车。

你会看到一个精美的网页——这就是 **Swagger**（也叫 Swagger UI）。

页面上显示了你刚才写的那个接口：

- `GET / Read Root`——就是你写的 `@app.get("/")` 对应的接口
- 点一下展开，能看到详细的请求和响应格式
- 甚至有个 **"Try it out"** 按钮——你点一下，再点 **"Execute"**，它就直接帮你调用接口，返回 `{"message": "Hello World"}` 给你看！

**是不是很神奇？** 你只写了一个函数，FastAPI 就自动帮你生成了：
- 接口文档（这个接口叫什么、路径是什么）
- 交互式测试工具（Try it out）
- 请求和响应的数据格式定义

**为什么需要 Swagger？** 想象你是厨师，前厅服务员和后厨之间需要一个"点菜传话系统"。后端写的接口就像后厨的菜，前端同学（或者调用你接口的其他程序员）需要知道：有什么菜、怎么点、点完会得到什么。Swagger 就是这个"菜单+点菜系统"——自动生成、自动维护、永远跟代码同步。

### 步骤 6：修改返回内容，体验热重载

现在我们来做一个最简单的修改，验证 `--reload` 热重载是否在工作。

**操作步骤：**

1. 回到编辑器，打开 `blog_backend/main.py`
2. 把 `"Hello World"` 改成 `"你好，世界！"`（或者任何你想说的话）
3. **按 `Ctrl+S` 保存**

修改后的代码：

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/")
def read_root():
    return {"message": "你好，世界！"}  # 改了这一行
```

4. 切回终端观察——你应该看到类似这样的输出：

```
WARNING:  WatchFiles detected changes in 'main.py'. Reloading...
INFO:     Shutting down
INFO:     Waiting for application shutdown.
INFO:     Application shutdown complete.
INFO:     Finished server process [67890]
INFO:     Started server process [12345]
INFO:     Waiting for application startup.
INFO:     Application startup complete.
```

整个过程不到一秒。uvicorn 检测到文件变化，自动关掉旧服务器，启动新的，你的修改立即生效。

5. 切回浏览器，刷新 `http://127.0.0.1:8000`——你应该看到：

```json
{"message": "你好，世界！"}
```

### 🤔 想多一点：为什么需要 `--reload`？

**先想象一个没有 `--reload` 的世界：**

你改了代码 → 保存 → 切到终端 → `Ctrl+C` 停掉服务器 → 重新敲 `uvicorn blog_backend.main:app` → 启动完成 → 切回浏览器 → 刷新。每次改一行代码都要重复这一整套动作。

**一天改 50 次代码，这套动作就要做 50 次。** 这就像暑假作业——你做一道题，老师要求你合上作业本再打开才能继续做下一道。荒谬吧？

`--reload` 就是帮你省掉这套机械劳动的工具。它背后是一个叫 `watchdog`（看门狗）的机制——uvicorn 会盯着你的代码文件，一旦发现变化，就自动重启服务器。

**⚠️ 重要提醒**：`--reload` 只在**开发阶段**使用。想象一下，如果你在线上运行的服务器也开了 `--reload`，那任何风吹草动都会导致服务器重启，用户请求全部中断。生产环境用的是 `uvicorn blog_backend.main:app`（没有 `--reload`），配合专业的进程管理工具。

**一句话总结**：`--reload` 是开发时的电梯，生产环境请走楼梯（更稳当）。

## 八、已知坑点与禁止事项

1. **`uvicorn main:app` 报 "could not import"**：检查你当前所在的目录。必须在能看到 `blog_backend/` 文件夹的目录下执行 `uvicorn blog_backend.main:app --reload`，不是在 `blog_backend/` 里面执行。

2. **端口 8000 被占用**：常见于上次没正常关闭 uvicorn（比如直接关了终端窗口）。解决方案：
   - 换个端口：`uvicorn blog_backend.main:app --reload --port 8001`
   - 或者找到占用的进程杀掉（Windows：`netstat -ano | findstr 8000` 找到 PID，然后 `taskkill /PID <PID> /F`）

3. **改了代码浏览器没变化**：先确认 `--reload` 参数加上了。加上后还是没反应？试试硬刷新浏览器（`Ctrl+Shift+R` 或 `Ctrl+F5`）。

4. **`--reload` 别带去生产环境**：生产环境部署时，`uvicorn` 前面不加 `--reload`。热重载是给开发用的"自动挡"，生产环境需要"手动挡"来保证稳定。
