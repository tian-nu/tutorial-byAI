## 一、整数：从 TINYINT 到 BIGINT

整数是最简单的数据类型，但选错大小会浪费空间或溢出。

### 1.1 五种整数类型

| 类型 | 字节数 | 有符号范围 | 无符号范围 |
|------|--------|-----------|-----------|
| TINYINT | 1 | -128 ~ 127 | 0 ~ 255 |
| SMALLINT | 2 | -32,768 ~ 32,767 | 0 ~ 65,535 |
| MEDIUMINT | 3 | -838 万 ~ 838 万 | 0 ~ 1677 万 |
| INT | 4 | -21 亿 ~ 21 亿 | 0 ~ 42 亿 |
| BIGINT | 8 | -9 百京 ~ 9 百京 | 0 ~ 18 百京 |

### 1.2 有符号 vs 无符号

```sql
-- 有符号（默认）：可以存负数
CREATE TABLE test_int (
    age TINYINT,           -- 范围 -128~127
    score INT              -- 范围 -21亿~21亿
);

-- 无符号（UNSIGNED）：只能存非负数，范围翻倍
CREATE TABLE test_uint (
    age TINYINT UNSIGNED,  -- 范围 0~255
    user_id INT UNSIGNED   -- 范围 0~42亿，适合自增主键
);
```

**选型直觉**：
- 年龄 → `TINYINT UNSIGNED`（0~255，够用）
- 自增主键 → `INT UNSIGNED`（42 亿行之前够用）或 `BIGINT UNSIGNED`
- 状态码 → `TINYINT`（0/1/2 等小数字）
- 数量、金额 → `INT` 或 `BIGINT`（按分存储）

### 1.3 整数溢出演示

```sql
-- TINYINT 最大值 127
CREATE TABLE test_overflow (val TINYINT);
INSERT INTO test_overflow VALUES (127);  -- 成功
INSERT INTO test_overflow VALUES (128);  -- ERROR 1264: Out of range value

-- 严格模式下报错，非严格模式下截断为 127
SELECT @@sql_mode;  -- 查看是否含 STRICT_TRANS_TABLES
```

## 二、浮点数精度：0.1 + 0.2 ≠ 0.3

这是一个经典的计算机科学问题，不是 MySQL 特有的。

### 2.1 演示：浮点数不精确

```sql
-- 创建测试表
CREATE TABLE test_float (
    val_f FLOAT,
    val_d DOUBLE
);

INSERT INTO test_float VALUES (0.1, 0.1);
INSERT INTO test_float VALUES (0.2, 0.2);

-- 看看 0.1 + 0.2 的结果
SELECT
    val_f,
    val_f + 0.2 AS sum_result,
    (val_f + 0.2) = 0.3 AS is_equal
FROM test_float
WHERE val_f = 0.1;
```

结果可能是：
```
+-------+------------+----------+
| val_f | sum_result | is_equal |
+-------+------------+----------+
|   0.1 | 0.30000004 |        0 |
+-------+------------+----------+
```

**`0.1 + 0.2 = 0.30000004`，不是 `0.3`！**

### 2.2 为什么会这样？

想象你只有 10 个手指，要表示 1/3。你只能写成 `0.3333333333`（10 位），但 1/3 真正的值是 `0.33333...`（无限循环）。不管你用多少位，总是有误差。

计算机用二进制存储小数，许多十进制小数在二进制中是无限循环的。比如：
- 十进制的 `0.1` → 二进制是 `0.0001100110011...`（无限循环）
- 十进制的 `0.2` → 二进制也是无限循环

FLOAT（4 字节）和 DOUBLE（8 字节）只能存有限位，所以必然有舍入误差。

### 2.3 FLOAT vs DOUBLE

| 类型 | 字节 | 精度（有效位数） | 适用场景 |
|------|------|----------------|---------|
| FLOAT | 4 | ~7 位 | 科学计算、游戏坐标 |
| DOUBLE | 8 | ~15 位 | 高精度科学计算 |
| DECIMAL | 可变 | 完全精确 | **金额、财务数据** |

## 三、DECIMAL(M, D)：精确小数，金额专用

**DECIMAL** *此术语见附录E* 是定点数类型，它**不会丢失精度**。因为它在内部用字符串方式存储（确切说是二进制编码的十进制），不经过二进制浮点数转换。

### 3.1 语法

```sql
DECIMAL(M, D)
-- M：总位数（精度），1~65
-- D：小数位数，0~30，D ≤ M
```

举例：
- `DECIMAL(10, 2)`：最多 10 位，其中 2 位小数 → 整数部分最多 8 位 → 范围 `-99999999.99 ~ 99999999.99`
- `DECIMAL(5, 0)`：5 位整数，无小数 → 等价于整数

### 3.2 精度验证

```sql
CREATE TABLE test_decimal (
    price_f FLOAT,
    price_d DOUBLE,
    price_dec DECIMAL(10, 2)
);

INSERT INTO test_decimal VALUES (0.1, 0.1, 0.1);
INSERT INTO test_decimal VALUES (0.2, 0.2, 0.2);

SELECT
    price_f + 0.2 AS float_sum,
    price_d + 0.2 AS double_sum,
    price_dec + 0.2 AS decimal_sum
FROM test_decimal
WHERE price_dec = 0.1;
```

```
+-----------+------------+-------------+
| float_sum | double_sum | decimal_sum |
+-----------+------------+-------------+
| 0.30000004| 0.30000000000000004 | 0.30 |
+-----------+------------+-------------+
```

**只有 DECIMAL 返回了精确的 0.30！**

> **想多一点**：很多人觉得"金额也不大，FLOAT 误差才 0.00000004，无所谓吧？"——错。第一，误差会累积，100 万次计算后误差可能达到几块钱。第二，金融系统有审计要求，你账上差一分钱都过不了审计。**金额类数据，永远用 DECIMAL**。

## 四、字符串：CHAR、VARCHAR、TEXT

### 4.1 CHAR vs VARCHAR 的本质区别

**CHAR** *此术语见附录E* 是**定长**字符串，**VARCHAR** *此术语见附录E* 是**变长**字符串。

```
CHAR(10)：  不管存什么，始终占 10 个字符的空间
VARCHAR(10)：存多少占多少（+1~2 字节记录长度）
```

```sql
CREATE TABLE test_strings (
    code CHAR(5),       -- 固定 5 字符
    name VARCHAR(50)    -- 最长 50 字符，实际按内容长度存储
);

INSERT INTO test_strings VALUES ('AB', 'AB');

-- 验证存储长度
SELECT
    code,
    CHAR_LENGTH(code) AS code_len,
    LENGTH(code) AS code_bytes,
    name,
    CHAR_LENGTH(name) AS name_len,
    LENGTH(name) AS name_bytes
FROM test_strings;
```

结果：
```
+------+----------+------------+------+----------+------------+
| code | code_len | code_bytes | name | name_len | name_bytes |
+------+----------+------------+------+----------+------------+
| AB   |        2 |          5 | AB   |        2 |          2 |
+------+----------+------------+------+----------+------------+
```

注意 `code_bytes = 5`：虽然只存了"AB"（2 个字符），但 CHAR(5) 用空格填充到 5 个字符。而 VARCHAR 的 `name_bytes = 2`，只存了实际内容。

### 4.2 选型标准：CHAR vs VARCHAR

| 选择 CHAR | 选择 VARCHAR |
|-----------|-------------|
| 定长数据（手机号、身份证、MD5） | 变长数据（姓名、邮箱、地址） |
| 长度很短（≤10）且基本固定 | 长度差异大 |
| 频繁更新（CHAR 不会产生碎片） | 存储空间优先 |
| 状态码、枚举值、性别 | 备注、描述、URL |

### 4.3 VARCHAR 的长度限制

- `VARCHAR(n)` 中 `n` 是**字符数**，不是字节数
- 最大 65,535 **字节**（所有列共享，不是每列）
- 一个中文字符占 3 字节（utf8mb4），所以 `VARCHAR(21845)` 约等于 65,535 字节
- 实际可用长度受行大小限制（InnoDB 行最大约 8KB，溢出列存到单独页）

```sql
-- ❌ 常见错误：VARCHAR 不给长度
CREATE TABLE bad (name VARCHAR);  -- 语法错误！

-- ✅ 正确：给长度
CREATE TABLE good (name VARCHAR(100));
```

### 4.4 TEXT 系列

当字符串超过 VARCHAR 上限（约 65KB）时，用 TEXT：

| 类型 | 最大长度 | 存储方式 |
|------|---------|---------|
| TINYTEXT | 255 字节 | 行内 |
| TEXT | 64 KB | 行外 |
| MEDIUMTEXT | 16 MB | 行外 |
| LONGTEXT | 4 GB | 行外 |

**TEXT** *此术语见附录E* 系列不能有默认值，不能用于索引（除非指定前缀长度）。

## 五、日期时间：DATE、TIME、DATETIME、TIMESTAMP

### 5.1 先铺垫：服务器时区

在讨论日期时间类型之前，必须先搞清楚一件事：**你的 MySQL 服务器用的是什么时区？**

```sql
-- 查看时区设置
SELECT @@global.time_zone, @@session.time_zone;
-- 如果返回 SYSTEM，表示跟随操作系统时区

-- 查看操作系统时区对应的具体值
SELECT NOW();  -- 当前时间，用了哪个时区就是哪个时间
```

**时区为什么重要？** 因为 `TIMESTAMP` 类型存的是 UTC 时间戳，读取时自动转换到当前会话时区。如果服务器在北京（UTC+8），你从纽约（UTC-5）连过来，看到的时间可能差 13 个小时。

### 5.2 四种时间类型的对比

| 类型 | 范围 | 时区转换 | 存储字节 | 适用场景 |
|------|------|---------|---------|---------|
| DATE | 1000-01-01 ~ 9999-12-31 | 无 | 3 | 生日、节假日 |
| TIME | -838:59:59 ~ 838:59:59 | 无 | 3 | 时长、营业时间 |
| DATETIME | 1000-01-01 00:00:00 ~ 9999-12-31 23:59:59 | 无 | 5（旧版8） | 用户可见时间、历史记录 |
| TIMESTAMP | 1970-01-01 00:00:01 ~ 2038-01-19 03:14:07 | **有** | 4 | 记录创建/更新时间、日志 |

### 5.3 DATETIME vs TIMESTAMP 的深入对比

**DATETIME** *此术语见附录E*：存的就是你写进去的值，不转换，不考虑时区。

**TIMESTAMP** *此术语见附录E*：内部存的是**UTC 时间戳**（从 1970-01-01 00:00:00 UTC 开始的秒数）。写入时从当前时区转 UTC，读取时从 UTC 转回当前时区。

```sql
-- 演示时区转换
CREATE TABLE test_time (
    dt DATETIME,
    ts TIMESTAMP
);

-- 当前时区（假设是 UTC+8，北京时间）
INSERT INTO test_time VALUES (NOW(), NOW());

SELECT * FROM test_time;
-- 假设当前是 2024-01-15 12:00:00
-- dt: 2024-01-15 12:00:00
-- ts: 2024-01-15 12:00:00  ← 看起来一样

-- 切换时区到 UTC+0
SET time_zone = '+00:00';

SELECT * FROM test_time;
-- dt: 2024-01-15 12:00:00  ← 不变！DATETIME 不转换
-- ts: 2024-01-15 04:00:00  ← 变了！TIMESTAMP 转换为 UTC+0

-- 恢复时区
SET time_zone = '+08:00';
```

### 5.4 TIMESTAMP 的 2038 问题

TIMESTAMP 用 4 字节存**有符号 32 位整数**表示从 1970-01-01 开始的秒数：
- 最大值：2,147,483,647 秒 → 对应 `2038-01-19 03:14:07 UTC`
- 超过这个时间，TIMESTAMP 就溢出了！

```sql
-- 试试插入 2039 年
INSERT INTO test_time (ts) VALUES ('2039-01-01 00:00:00');
-- 可能报错或截断，取决于 sql_mode
```

**对策**：需要存 2038 年之后的时间，用 `DATETIME`。

### 5.5 自动初始化与自动更新

```sql
CREATE TABLE posts (
    id INT PRIMARY KEY AUTO_INCREMENT,
    content TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,           -- 插入时自动填
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP  -- 更新时自动刷新
);
```

`ON UPDATE CURRENT_TIMESTAMP`：每次 UPDATE 这一行，`updated_at` 自动更新为当前时间。

## 六、ENUM 和 SET：了解即可

### 6.1 ENUM：单选枚举

**ENUM** *此术语见附录E* 允许你定义一个列只能取指定的几个值：

```sql
CREATE TABLE test_enum (
    status ENUM('pending', 'approved', 'rejected')
);

INSERT INTO test_enum VALUES ('pending');   -- 成功
INSERT INTO test_enum VALUES ('approved');  -- 成功
INSERT INTO test_enum VALUES ('cancelled'); -- 失败或插入空字符串（取决于 sql_mode）
```

### 6.2 ENUM 的两个大坑

**坑 1：扩展困难**。想加一个 `'cancelled'` 状态？需要 `ALTER TABLE ... MODIFY`，这是一条 DDL，会锁表。

**坑 2：排序行为奇特**。ENUM 按**内部数字索引**排序，不是按字符串字母顺序：

```sql
INSERT INTO test_enum VALUES ('approved'), ('pending'), ('rejected');
SELECT status FROM test_enum ORDER BY status;
-- 结果：approved, pending, rejected
-- 而不是：approved, pending, rejected（按字母）
-- 因为内部索引：approved=2, pending=1, rejected=3
-- ORDER BY 按索引排序：1→2→3 = pending→approved→rejected
```

### 6.3 SET：多选

SET 允许一个列存多个值（用逗号分隔），类似"多选"：

```sql
CREATE TABLE test_set (
    tags SET('mysql', 'redis', 'docker', 'k8s')
);

INSERT INTO test_set VALUES ('mysql,redis');  -- 成功
INSERT INTO test_set VALUES ('mysql,docker'); -- 成功
```

SET 的问题和 ENUM 类似：扩展需要 ALTER TABLE，且查询效率低。

**我的建议**：ENUM 和 SET 了解即可，生产环境用 `VARCHAR` + 应用层约束替代。或者用独立的"字典表"：

```sql
-- 替代 ENUM 的更好方案：字典表
CREATE TABLE order_status (
    code VARCHAR(20) PRIMARY KEY,
    label VARCHAR(50) NOT NULL
);
INSERT INTO order_status VALUES ('pending', '待审核'), ('approved', '已通过'), ('rejected', '已拒绝');

-- orders 表引用
CREATE TABLE orders (
    id INT PRIMARY KEY,
    status VARCHAR(20),
    FOREIGN KEY (status) REFERENCES order_status(code)
);
```

## 七、JSON：MySQL 5.7+ 的原生 JSON

**JSON** *此术语见附录E* 类型让你把 JSON 文档直接存到 MySQL 中，并支持 JSON 函数进行查询和修改。

```sql
-- 创建含 JSON 列的表
CREATE TABLE user_profiles (
    user_id INT PRIMARY KEY,
    profile JSON
);

-- 插入 JSON 数据
INSERT INTO user_profiles VALUES
(1, '{"name": "alice", "age": 25, "tags": ["mysql", "redis"]}'),
(2, '{"name": "bob", "age": 30, "tags": ["python", "docker"]}');

-- JSON 查询
SELECT
    user_id,
    profile->>'$.name' AS name,           -- 提取 name 字段
    profile->>'$.age' AS age,             -- 提取 age 字段
    JSON_EXTRACT(profile, '$.tags[0]') AS first_tag  -- 提取 tags 数组第一个元素
FROM user_profiles
WHERE JSON_CONTAINS(profile->'$.tags', '"mysql"');
```

### JSON 的适用场景

- ✅ 结构不固定的数据（如用户自定义属性）
- ✅ 日志、API 响应缓存
- ✅ 配置数据（JSON 格式更自然）
- ❌ 需要频繁查询/索引的字段（先用虚拟列生成索引）
- ❌ 需要 JOIN 的数据（JSON 不支持 JOIN）

## 八、字符集 utf8 vs utf8mb4

第 04 章建表时写了 `CHARACTER SET utf8mb4`，当时只说"能存 emoji"。现在展开解释。

### 8.1 MySQL 的 utf8 是个"假 utf8"

MySQL 的 `utf8` 字符集最多只用 3 字节编码，只能覆盖 Unicode 的**基本多语言平面**（BMP）。但 emoji 和许多罕见汉字在**补充平面**，需要 4 字节。

`utf8mb4`（UTF-8 Multi-Byte 4）才是真正的 UTF-8，支持 4 字节字符。

### 8.2 演示：utf8 存 emoji 会怎样

```sql
-- 用 utf8 建表
CREATE TABLE test_utf8 (
    content VARCHAR(100)
) CHARACTER SET utf8;

INSERT INTO test_utf8 VALUES ('😀');  -- emoji
-- ERROR 1366 (HY000): Incorrect string value: '\xF0\x9F\x98\x80'

-- 用 utf8mb4 建表
CREATE TABLE test_utf8mb4 (
    content VARCHAR(100)
) CHARACTER SET utf8mb4;

INSERT INTO test_utf8mb4 VALUES ('😀');  -- 成功！
```

### 8.3 结论

**永远用 `utf8mb4`**。`utf8` 是 MySQL 的历史遗留问题，现在没有任何理由用它。

```sql
-- 建库时
CREATE DATABASE my_shop CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

-- 建表时
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50)
) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

## 九、选型决策速查表

| 场景 | 推荐类型 | 理由 |
|------|---------|------|
| 自增主键 | INT UNSIGNED / BIGINT UNSIGNED | 大范围，非负 |
| 金额 | DECIMAL(10,2) | 精确，不丢精度 |
| 姓名 | VARCHAR(50) | 变长，省空间 |
| 手机号 | CHAR(11) | 定长，查询快 |
| 身份证号 | CHAR(18) | 定长 |
| 邮箱 | VARCHAR(100) | 变长 |
| 性别 | TINYINT(1) | 0/1 表示，比 ENUM 灵活 |
| 状态码 | VARCHAR(20) + 字典表 | 比 ENUM 灵活 |
| 创建时间 | TIMESTAMP | 自动时区转换，自动更新 |
| 历史时间（如生日） | DATETIME | 不受时区影响，范围大 |
| 长文本（文章） | TEXT / MEDIUMTEXT | 超过 64KB |
| URL | VARCHAR(2048) | URL 标准最大长度 |
| IP 地址 | INT UNSIGNED（INET_ATON） | 省空间，查询快 |
| 布尔值 | TINYINT(1) | 0/1，兼容性好 |
| JSON 数据 | JSON | 原生支持，可查询 |
| 哈希值（MD5/SHA） | CHAR(32) / CHAR(64) | 定长，适合索引 |

## 我做得对不对？

### 验证方法

```sql
-- 1. 创建一张包含所有数据类型的测试表
CREATE TABLE type_test (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    -- 整数
    age TINYINT UNSIGNED,
    score BIGINT,
    -- 小数
    price_f FLOAT,
    price_d DOUBLE,
    price_dec DECIMAL(10,2),
    -- 字符串
    code CHAR(5),
    name VARCHAR(50),
    note TEXT,
    -- 日期时间
    dt DATETIME,
    ts TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    -- JSON
    extra JSON,
    -- 布尔
    is_active TINYINT(1) DEFAULT 1
) CHARACTER SET utf8mb4;

-- 2. 插入边界值测试
INSERT INTO type_test (age, price_f, price_d, price_dec, code, name, note, dt, extra)
VALUES
(255, 0.1, 0.1, 99999999.99, 'AB', N'测试中文名', '这是一段长文本', '2038-01-19 03:14:07', '{"key": "value"}');

-- 3. 验证存储大小
SELECT
    id,
    CHAR_LENGTH(name) AS name_chars,  -- 字符数
    LENGTH(name) AS name_bytes,       -- 字节数
    CHAR_LENGTH(code) AS code_chars,
    LENGTH(code) AS code_bytes        -- CHAR(5) 总是 5 字节（英文）
FROM type_test;

-- 4. 验证浮点精度
SELECT
    price_f + 0.2 AS float_sum,
    price_d + 0.2 AS double_sum,
    price_dec + 0.2 AS decimal_sum,
    (price_f + 0.2) = 0.3 AS float_eq_03,
    (price_dec + 0.2) = 0.3 AS decimal_eq_03
FROM type_test
WHERE price_dec = 0.1;

-- 5. 验证 TIMESTAMP 时区
SELECT @@global.time_zone, @@session.time_zone;
SELECT dt, ts FROM type_test;
SET time_zone = '+00:00';
SELECT dt, ts FROM type_test;  -- DATETIME 不变，TIMESTAMP 变了
SET time_zone = '+08:00';

-- 6. 清理
DROP TABLE type_test;
```

## 不对怎么办？

### 常见错误 1：金额用 FLOAT 或 DOUBLE

```sql
-- ❌ 错误：金额用 FLOAT
CREATE TABLE orders (
    id INT PRIMARY KEY,
    amount FLOAT  -- 金额会算错！
);

-- ❌ 错误：金额用 DOUBLE 也不行（精度更高但仍有误差）
CREATE TABLE orders (
    amount DOUBLE
);

-- ✅ 正确：金额用 DECIMAL
CREATE TABLE orders (
    amount DECIMAL(10, 2)  -- 精确到分
);
```

### 常见错误 2：VARCHAR 不给长度

```sql
-- ❌ 错误
CREATE TABLE users (name VARCHAR);  -- 语法错误！

-- ✅ 正确
CREATE TABLE users (name VARCHAR(50));
```

### 常见错误 3：过度使用 VARCHAR(255)

```sql
-- ❌ 过度："反正 255 够用"
CREATE TABLE users (
    phone VARCHAR(255),  -- 手机号是定长的，用 CHAR(11)
    md5 VARCHAR(255)     -- MD5 固定 32 字符，用 CHAR(32)
);

-- ✅ 正确：根据实际需求选
CREATE TABLE users (
    phone CHAR(11),      -- 定长，查询更快
    md5 CHAR(32)         -- 定长，适合索引
);
```

### 常见错误 4：TIMESTAMP 2038 问题

```sql
-- ❌ 可能出问题：存 2038 年后的日期
CREATE TABLE events (
    event_date TIMESTAMP  -- 2038-01-19 之后溢出
);

-- ✅ 正确：需要存远期日期用 DATETIME
CREATE TABLE events (
    event_date DATETIME
);
```

### 常见错误 5：用 utf8 存中文或 emoji

```sql
-- ❌ 错误：utf8 只能存 3 字节字符
CREATE TABLE articles (content TEXT) CHARACTER SET utf8;

-- ✅ 正确：永远用 utf8mb4
CREATE TABLE articles (content TEXT) CHARACTER SET utf8mb4;
```

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 整数类型 | TINYINT→BIGINT，有符号/无符号 | 选合适的范围，不要全部用 INT；自增主键用 INT UNSIGNED |
| 浮点精度问题 | FLOAT/DOUBLE 有舍入误差 | 用 `0.1+0.2` 演示；金额永远用 DECIMAL |
| 定点数 DECIMAL | DECIMAL(M,D)，精确存储 | 金额专用；M 是总位数，D 是小数位数 |
| CHAR vs VARCHAR | CHAR 定长，VARCHAR 变长 | 定长数据（手机号）用 CHAR；变长数据（姓名）用 VARCHAR |
| TEXT 系列 | TINYTEXT / TEXT / MEDIUMTEXT / LONGTEXT | 不能有默认值；不能全列索引 |
| 日期时间 | DATE/TIME/DATETIME/TIMESTAMP | TIMESTAMP 有时区转换和 2038 问题；DATETIME 无时区 |
| ENUM / SET | 了解即可，生产慎用 | 扩展困难（需 ALTER TABLE）；排序行为奇特 |
| JSON | 原生 JSON 类型，JSON_EXTRACT 查询 | 适合结构不固定的数据；需要频繁查询的字段先用虚拟列索引 |
| utf8 vs utf8mb4 | MySQL 的 utf8 是"假 utf8" | 永远用 utf8mb4，能存 emoji 和所有 Unicode 字符 |
| 选型速查表 | 见第九节 | 建表时直接参考，避免选错 |
