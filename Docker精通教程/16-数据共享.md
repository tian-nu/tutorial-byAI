## 我在做什么？

上一章你学会了给一个容器配上"保险箱"（Volume），删了容器数据不丢。但现实场景中，经常需要**多个容器共享同一份数据**：

- 两个 Web 服务器共用同一套静态文件
- 一个日志收集器和一个应用容器共用同一个日志目录
- 一个备份容器定时读取数据库容器的数据目录

怎么让多个容器看到同一个 Volume？这一章教你两种方法，以及一个逐渐退出历史舞台但你可能在旧项目中遇到的模式。

学完这一章，你能：
- 让多个容器同时挂载同一个 named volume
- 用 `--volumes-from`*此术语见附录C* 让一个容器直接"继承"另一个容器的挂载
- 理解数据卷容器*此术语见附录C*模式及其历史背景
- 在两个 Nginx 容器之间共享静态文件，修改一处两边都生效

## 一、最简单的方法：多容器挂同一个 Volume

其实没有魔法。你只需要在启动多个容器时，指定**同一个 volume 名字**：

```bash
# 创建一个共享卷
docker volume create shared-web

# 容器 A：挂载 shared-web 到 /usr/share/nginx/html
docker run -d --name web-a \
  --mount type=volume,source=shared-web,target=/usr/share/nginx/html \
  nginx:alpine

# 容器 B：也挂载 shared-web 到同一个路径
docker run -d --name web-b \
  --mount type=volume,source=shared-web,target=/usr/share/nginx/html \
  nginx:alpine
```

现在 `web-a` 和 `web-b` 共用同一份数据。你在宿主机上通过任意一个容器修改卷里的文件，另一个容器立刻能看到变化。

### 想多一点：这有什么风险？

多个容器同时写同一个文件，就是**并发写入**问题。Docker 不提供文件锁——这取决于你容器内的应用程序怎么处理。如果两个 Nginx 同时试图写同一个日志文件，数据可能损坏。所以共享 Volume 最常见的是**读多写少**的场景（静态文件、配置、只读数据）。

## 二、`--volumes-from`：共享另一个容器的挂载配置

还有一种方式：你不想重新声明一遍 `--mount` 参数，而是"照抄"另一个容器的所有挂载。这时用 `--volumes-from`：

```bash
# 语法
docker run --volumes-from 源容器名 新容器
```

### 实战：两个 Nginx 共享静态文件

#### 步骤 1：创建共享卷，写入测试文件

```bash
# 创建卷
docker volume create nginx-shared

# 启动一个临时容器，写入测试文件
docker run --rm \
  --mount type=volume,source=nginx-shared,target=/usr/share/nginx/html \
  alpine:latest sh -c "echo '<h1>共享的数据</h1>' > /usr/share/nginx/html/index.html"
```

#### 步骤 2：启动第一个 Nginx（数据源容器）

```bash
docker run -d --name nginx-source \
  --mount type=volume,source=nginx-shared,target=/usr/share/nginx/html \
  -p 8081:80 \
  nginx:alpine
```

验证：

```bash
curl http://localhost:8081
# 输出：<h1>共享的数据</h1>
```

#### 步骤 3：启动第二个 Nginx，用 `--volumes-from` 继承挂载

```bash
docker run -d --name nginx-follower \
  --volumes-from nginx-source \
  -p 8082:80 \
  nginx:alpine
```

注意：`nginx-follower` 的启动命令里**没有** `--mount` 或 `-v`。它通过 `--volumes-from nginx-source` 继承了 `nginx-source` 的所有挂载配置。

验证：

```bash
curl http://localhost:8082
# 输出：<h1>共享的数据</h1>
```

#### 步骤 4：修改数据，两边都看到变化

```bash
# 通过任意一个容器修改卷中的文件
docker exec nginx-source sh -c \
  "echo '<h1>数据已更新！</h1>' > /usr/share/nginx/html/index.html"

# 两个容器都返回新内容
curl http://localhost:8081
# 输出：<h1>数据已更新！</h1>

curl http://localhost:8082
# 输出：<h1>数据已更新！</h1>
```

这就是共享 Volume 的核心价值：**一处修改，多处生效。**

## 三、数据卷容器模式（纯数据容器）

这是 `--volumes-from` 的经典用法：创建一个**什么都不做、只持有数据卷**的容器。

```bash
# 1. 创建一个"数据卷容器"——它只负责持有卷，不运行任何服务
docker create \
  --name data-store \
  --mount type=volume,source=nginx-shared,target=/usr/share/nginx/html \
  alpine:latest /bin/true
#      ─────┬─────
#     /bin/true 是一个立即退出的命令，
#     容器创建后处于 "已退出" 状态，但它定义的卷挂载仍然有效

# 2. 其他容器通过 --volumes-from 引用它
docker run -d --name app1 --volumes-from data-store -p 8081:80 nginx:alpine
docker run -d --name app2 --volumes-from data-store -p 8082:80 nginx:alpine
```

### 想多一点：这个模式还有必要吗？

在 Docker 早期（1.9 之前），没有 `docker volume create` 命令，named volume 的概念也不成熟。数据卷容器是当时**唯一**的方法来持久化和共享数据。

现在你有 `docker volume create` 和 `--mount`，数据卷容器模式已经不那么必要了。但你可能在旧项目、旧教程中频繁看到它，所以你应该认识它。

### 数据卷容器 vs named volume 直接挂载

| 维度 | 数据卷容器 | named volume 直接挂载 |
|------|-----------|---------------------|
| 可读性 | 差——不看 `data-store` 不知道挂了什么 | 好——每个容器的 `--mount` 清晰可见 |
| 维护性 | 差——多了一层间接引用 | 好——挂载声明就在容器启动命令里 |
| 历史原因 | Docker 1.9 之前的唯一方案 | 现代推荐方式 |
| 何时还用它 | 需要把一系列挂载"打包"成一个整体供多个容器继承 | 绝大多数场景 |

## 四、常见错误

### 错误 1：`--volumes-from` 引用的容器不存在

```bash
# ❌ 引用的容器还没创建
docker run --volumes-from data-store nginx:alpine
# Error: No such container: data-store
```

✅ 正确顺序：

```bash
# 1. 先创建数据源容器
docker create --name data-store \
  --mount type=volume,source=myvol,target=/data \
  alpine:latest /bin/true

# 2. 再启动依赖它的容器
docker run --volumes-from data-store nginx:alpine
```

### 错误 2：以为 `--volumes-from` 是"实时同步"两个容器的挂载

```bash
# 你启动了 data-store
docker create --name data-store -v myvol:/data alpine:latest /bin/true

# 然后你修改了 data-store 的挂载（停掉重建）
docker rm data-store
docker create --name data-store -v myvol:/data -v myvol2:/extra alpine:latest /bin/true

# 但 app1 的挂载配置不会自动更新！
# app1 启动时复制了 data-store 当时的挂载列表，
# 之后 data-store 怎么变，app1 都不受影响
```

❌ 原因：`--volumes-from` 在**容器启动时**复制源容器的挂载列表。它不是"订阅"关系，启动后两者独立。

### 错误 3：共享 Volume 时，一个容器写入的数据另一个容器看不到

```bash
# ❌ 容器 A 和容器 B 挂载了"同名但不同"的卷
docker run --name A -v vol-a:/data alpine:latest
docker run --name B -v vol-b:/data alpine:latest
# vol-a 和 vol-b 是两个不同的 volume！
```

✅ 用 `docker volume ls` 和 `docker inspect` 确认挂载的是同一个卷。

### 错误 4：数据卷容器退出后，以为数据没了

```bash
docker create --name data-store -v myvol:/data alpine:latest /bin/true
# data-store 状态是 Exited

docker volume ls
# myvol 仍然在

# 即使 docker rm data-store
docker rm data-store
docker volume ls
# myvol 还在！
```

数据卷容器的生存状态和卷的生存状态**完全独立**。容器退出了，卷还在；容器删除了，卷也还在（除非你用 `docker rm -v`）。

## 五、我做得对不对？

### 验证方法

```bash
# 1. 确认共享卷存在
docker volume ls | grep nginx-shared

# 2. 确认两个 Nginx 容器都在运行
docker ps --filter "name=nginx" --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"

# 3. 访问两个容器，确认返回相同内容
curl http://localhost:8081
curl http://localhost:8082

# 4. 修改卷中文件，确认两边都更新
docker exec nginx-source sh -c "echo '<h1>验证通过</h1>' > /usr/share/nginx/html/index.html"
curl http://localhost:8081
curl http://localhost:8082
# 预期：两个都返回 <h1>验证通过</h1>

# 5. 查看两个容器的挂载信息
docker inspect nginx-source -f '{{ json .Mounts }}' | python3 -m json.tool
docker inspect nginx-follower -f '{{ json .Mounts }}' | python3 -m json.tool
# 预期：两个容器的 Mounts 列表指向同一个卷名
```

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 多容器挂载同一 volume | 多个 `docker run` 指定相同卷名 | 最简单直接的方式 |

| 卷生命周期 | 卷独立于容器，容器删除卷不删 | `docker volume prune` 清理孤儿卷 |

## 本篇最可能出错的地方及原因

### 1. `--volumes-from` 引用的容器还没创建

这是最常见的顺序错误。`--volumes-from` 不会自动创建容器，它只是引用一个已存在的容器。如果你在 Docker Compose 里用 `volumes_from`，Compose 会处理依赖顺序；但纯 `docker run` 时你必须自己保证顺序。

### 2. 误以为 `--volumes-from` 是"动态订阅"

很多人以为 `--volumes-from` 是"我订阅了源容器的挂载，源容器挂载变了我也跟着变"。实际上它是在启动时**一次性复制**挂载配置。Docker 里没有"热更新挂载"的概念。

### 3. 共享 Volume 时，两个容器同时写同一文件

```bash
# 容器 A 和 B 同时写入同一个日志文件
docker exec container-a sh -c "echo 'log from A' >> /shared/app.log"
docker exec container-b sh -c "echo 'log from B' >> /shared/app.log"
# 可能正常，也可能日志行交错混乱
```

Docker 不提供文件级锁。如果有多容器同时写的需求，应该用数据库、消息队列或其他支持并发的存储方案，而不是共享文件系统。

### 4. 数据卷容器的 `docker create` vs `docker run`

```bash
# ❌ 如果用 docker run
docker run --name data-store -v myvol:/data alpine:latest
# 容器启动后立即退出（alpine 没有默认进程），但没问题

# ✅ 更推荐 docker create
docker create --name data-store -v myvol:/data alpine:latest /bin/true
# create 只创建容器不启动，/bin/true 作为占位命令
```

`docker create` 和 `docker run` 都能创建数据卷容器，但 `create` 语义更明确："我只创建，不运行。"

### 5. 清理时忘了 `--volumes-from` 创建的匿名卷

如果你用 `--volumes-from` 引用了一个使用了匿名卷的容器，那些匿名卷也会被继承。清理时可能留下大量匿名卷。定期 `docker volume prune` 是好习惯。
