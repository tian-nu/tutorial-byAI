# 第43章 · 链表

> "切片就像一列紧密排列的火车车厢——你可以在O(1)时间跳到第5节车厢，但如果你想在第2节和第3节之间插入一节新车厢，那就得把第3节之后所有的车厢都往后挪。链表呢？它像一串用绳子连起来的珠子——每个珠子只知道自己后面是谁。你要想找第5颗珠子，必须从第1颗开始一颗一颗数过去。但如果你想在两颗珠子之间插一颗新的——只需解开绳子、串进去、再系上，O(1)搞定。"

## 43.1 单向链表

单向链表是最简单的链表——每个节点有两个字段：**值**和**指向下一个节点的指针**。

```go
type ListNode struct {
    Val  int
    Next *ListNode
}
```

当这些节点一个一个串起来，就形成了一条链：

```
头节点               节点2               节点3
┌────┬─────┐      ┌────┬─────┐      ┌────┬─────┐
│ 1  │  ●──┼─────→│ 2  │  ●──┼─────→│ 3  │ nil │
└────┴─────┘      └────┴─────┘      └────┴─────┘
```

最后一个节点的Next是`nil`——表示链到此为止。

### 创建链表

```go
func createList(values []int) *ListNode {
    if len(values) == 0 {
        return nil
    }
    head := &ListNode{Val: values[0]}
    current := head
    for i := 1; i < len(values); i++ {
        current.Next = &ListNode{Val: values[i]}
        current = current.Next
    }
    return head
}

head := createList([]int{1, 2, 3, 4, 5})
```

**遍历链表（打印所有值）**：

```go
func printList(head *ListNode) {
    for head != nil {
        fmt.Print(head.Val, " → ")
        head = head.Next
    }
    fmt.Println("nil")
}
```

输出：`1 → 2 → 3 → 4 → 5 → nil`

### 查找：O(n)

链表无法像切片那样跳过中间元素，必须从头一个个往下找：

```go
func find(head *ListNode, target int) *ListNode {
    for head != nil {
        if head.Val == target {
            return head
        }
        head = head.Next
    }
    return nil
}
```

**比喻**：链表是一串钥匙串在绳子上。你要找写着"仓库"的那把钥匙，必须从第一把开始，一个一个翻过来看——不能跳过。所以链表查找是O(n)。

### 插入头部：O(1)

在头部插入一个新节点是最简单的——把新节点的Next指向原来的头，新节点就变成头了。

```go
func insertHead(head *ListNode, val int) *ListNode {
    newNode := &ListNode{Val: val, Next: head}
    return newNode
}
```

```
插入前: head → [2] → [3] → nil
insertHead(head, 1)
插入后: head → [1] → [2] → [3] → nil
```

一步搞定，不用移动任何元素。O(1)。

### 插入尾部：O(n)

尾部插入需要先遍历到最后一个节点（找到Next是nil的那个），然后接上去。

```go
func insertTail(head *ListNode, val int) *ListNode {
    newNode := &ListNode{Val: val}
    if head == nil {
        return newNode
    }
    current := head
    for current.Next != nil {
        current = current.Next
    }
    current.Next = newNode
    return head
}
```

需要遍历整条链——O(n)。

### 插入中间（在某个节点后）：O(1)

先找到目标节点，然后把新节点插在它后面。

```go
func insertAfter(node *ListNode, val int) {
    newNode := &ListNode{Val: val, Next: node.Next}
    node.Next = newNode
}
```

```
插入前: [1] → [3] → [4] → nil
insertAfter(节点1, 2)
插入后: [1] → [2] → [3] → [4] → nil
```

注意：找到"节点1"这个位置需要O(n)遍历。但一旦找到了，插入操作本身只需改两个指针——O(1)。

**这就是链表相比切片的最大优势。** 切片在中间插入一个元素，需要把后面所有元素往后挪：

```go
s := []int{1, 3, 4}
s = append(s[:1], append([]int{2}, s[1:]...)...)
```

这行代码背后：创建新切片→复制前1个元素→复制新元素→复制后2个元素——O(n)。而链表只需要改两个指针。

### 删除节点：O(1)（给定前驱节点）

删除节点——把当前节点的Next指针"跳过"要删的那个节点，直接指向下下个。

```go
func deleteAfter(prev *ListNode) {
    if prev == nil || prev.Next == nil {
        return
    }
    prev.Next = prev.Next.Next
}
```

```
删除前: [1] → [2] → [3] → nil
deleteAfter(节点1)
删除后: [1] → [3] → nil    （节点2被GC回收）
```

同样，找到prev需要O(n)，但删除操作本身是O(1)。

## 43.2 双向链表

单向链表只能往前走，不能回头——这在某些场景下很不方便。双向链表每个节点既指向下一个，也指向上一个。

Go标准库中的 `container/list` 就是双向链表的实现：

```go
import "container/list"

l := list.New()
l.PushBack(1)
l.PushBack(2)
l.PushBack(3)

for e := l.Front(); e != nil; e = e.Next() {
    fmt.Println(e.Value)
}
```

`container/list` 的节点结构（简化版）：

```go
type Element struct {
    prev, next *Element
    Value      interface{}
}
```

- `prev`：指向前一个节点
- `next`：指向后一个节点
- `Value`：存储的值（`interface{}` 类型，可以存任何数据）

双向链表支持：
- `PushFront(v)`：头部插入，O(1)
- `PushBack(v)`：尾部插入，O(1)（因为维护了尾指针）
- `InsertBefore(v, mark)`：在某个节点前插入，O(1)
- `InsertAfter(v, mark)`：在某个节点后插入，O(1)
- `MoveToFront(e)`：把节点移到头部，O(1)
- `MoveToBack(e)`：把节点移到尾部，O(1)
- `Remove(e)`：删除节点，O(1)

**双向链表的妙处**：因为每个节点知道前驱和后继，给定任意一个节点，你可以在O(1)时间完成插入或删除——不需要像单向链表那样先找到前驱节点。

### 自己实现一个简单双向链表

```go
type DoublyNode struct {
    Val  int
    Prev *DoublyNode
    Next *DoublyNode
}

type DoublyList struct {
    head *DoublyNode
    tail *DoublyNode
}

func (dl *DoublyList) PushBack(val int) {
    node := &DoublyNode{Val: val}
    if dl.tail == nil {
        dl.head = node
        dl.tail = node
        return
    }
    dl.tail.Next = node
    node.Prev = dl.tail
    dl.tail = node
}

func (dl *DoublyList) Remove(node *DoublyNode) {
    if node.Prev != nil {
        node.Prev.Next = node.Next
    } else {
        dl.head = node.Next
    }
    if node.Next != nil {
        node.Next.Prev = node.Prev
    } else {
        dl.tail = node.Prev
    }
}
```

## 43.3 链表 vs 切片：一张对比表

| 操作 | 切片 | 单向链表 | 双向链表 |
|------|------|---------|---------|
| 按下标访问 | O(1) | O(n) | O(n) |
| 头部插入 | O(n) | O(1) | O(1) |
| 尾部插入 | 均摊O(1)* | O(n) | O(1) |
| 中间插入（已定位） | O(n) | O(1) | O(1) |
| 删除（已定位） | O(n) | O(1)** | O(1) |
| 遍历 | O(n) | O(n) | O(n) |
| 内存连续性 | 连续（缓存友好） | 离散（缓存不友好） | 离散（缓存不友好） |
| 额外内存开销 | 无（纯数据） | 每个节点多一个指针 | 每个节点多两个指针 |

\* 切片尾部插入的"均摊O(1)"是指：大部分时候不扩容直接写（O(1)），偶尔扩容（O(n)），平均下来是O(1)。

\** 单向链表的O(1)删除需要前驱指针；如果只有当前节点，需要O(n)找到前驱。

### 什么时候用链表？

链表并不是"更高级的切片"。它的适用场景比较特定：

1. **频繁在头部或中间插入/删除**：实现LRU缓存、消息队列中的延迟队列
2. **元素数量频繁变化且无法预分配**：不适合用切片（会频繁扩容）
3. **需要从中间删除大量元素**：如游戏开发中的子弹管理——子弹飞出屏幕就删除
4. **不需要随机访问**：如果你主要做遍历，不需要按下标跳转

**大多数时候，切片就够了。** 链表在Go中实际使用并不多——因为切片已经足够好用，而且CPU缓存对连续内存极其友好（切片的遍历可能比链表快5~10倍）。

## 43.4 链表经典算法

链表是面试的"必考题"。下面5个经典算法，每一个都可能出现在白板上。

### 算法一：反转链表（迭代法）

**问题**：输入 `1→2→3→4→5→nil`，输出 `5→4→3→2→1→nil`

**思路**：遍历链表，每次把当前节点的Next指向前一个节点。

```
操作过程（动画分解）：

初始:    prev=nil    cur=1→2→3→4→5→nil
步骤1:   prev=1→nil  cur=2→3→4→5→nil   （1.Next指向nil）
步骤2:   prev=2→1    cur=3→4→5→nil     （2.Next指向1）
步骤3:   prev=3→2→1  cur=4→5→nil       （3.Next指向2）
步骤4:   prev=4→3→2→1 cur=5→nil        （4.Next指向3）
步骤5:   prev=5→4→3→2→1 cur=nil        （5.Next指向4）
结束:    返回prev
```

**代码实现**：

```go
func reverseList(head *ListNode) *ListNode {
    var prev *ListNode
    cur := head
    for cur != nil {
        next := cur.Next
        cur.Next = prev
        prev = cur
        cur = next
    }
    return prev
}
```

三个指针就够了：
- `prev`：已经反转好的部分的头
- `cur`：当前正在处理的节点
- `next`：暂存cur的下一个（因为cur.Next会被改写）

时间复杂度O(n)，空间复杂度O(1)。

### 算法二：反转链表（递归法）

```go
func reverseListRecursive(head *ListNode) *ListNode {
    if head == nil || head.Next == nil {
        return head
    }
    newHead := reverseListRecursive(head.Next)
    head.Next.Next = head
    head.Next = nil
    return newHead
}
```

递归的思考方式：先反转后面的所有节点（递归调用），然后把当前节点接在反转后的链表尾部。

递归法更简洁但空间复杂度是O(n)（调用栈深度），不如迭代法的O(1)。

### 算法三：快慢指针找中点

**问题**：找到链表的中间节点。如果是偶数长度，返回中间两个的第二个。

**思路**：快指针每次走两步，慢指针每次走一步。当快指针到末尾时，慢指针刚好在中间。

```go
func findMiddle(head *ListNode) *ListNode {
    slow, fast := head, head
    for fast != nil && fast.Next != nil {
        slow = slow.Next
        fast = fast.Next.Next
    }
    return slow
}
```

```
演示: 1→2→3→4→5→nil
slow:  1  2  3
fast:  1  3  5
返回 3 ✓

演示: 1→2→3→4→nil
slow:  1  2  3
fast:  1  3  nil
返回 3 ✓
```

时间复杂度O(n)，空间复杂度O(1)。快慢指针是链表问题中最常用的技巧之一。

### 算法四：判断链表是否有环

**问题**：判断链表中是否存在环（某个节点的Next指向了前面的节点，形成循环）。

**思路**：同样是快慢指针。如果有环，快指针最终会追上慢指针（像操场跑圈，快的总会套圈慢的）。如果没有环，快指针会遇到nil。

```go
func hasCycle(head *ListNode) bool {
    slow, fast := head, head
    for fast != nil && fast.Next != nil {
        slow = slow.Next
        fast = fast.Next.Next
        if slow == fast {
            return true
        }
    }
    return false
}
```

**追上的原理**：假设环的长度是L。慢指针进入环时，快指针已经在环里。快指针每次比慢指针多走一步——相当于快指针在以每次1步的速度追赶慢指针。如果相距k步，k步后必然追上。

**进阶问题——找到环的入口**：

```go
func detectCycle(head *ListNode) *ListNode {
    slow, fast := head, head
    for fast != nil && fast.Next != nil {
        slow = slow.Next
        fast = fast.Next.Next
        if slow == fast {
            slow = head
            for slow != fast {
                slow = slow.Next
                fast = fast.Next
            }
            return slow
        }
    }
    return nil
}
```

相遇后，把一个指针放回头部，两个指针同时每次走一步——再次相遇的位置就是环的入口。这背后的数学原理比较巧妙：设链表头到环入口距离为a，环入口到相遇点距离为b，相遇点到环入口（沿环继续走）距离为c。快指针走过的距离是慢指针的两倍：2(a+b) = a+b+c+b，解得a=c。所以从头和从相遇点同时出发，会在环入口相遇。

### 算法五：合并两个有序链表

**问题**：两个升序链表，合并成一个升序链表。

**思路**：类似归并排序的merge步骤。两个指针分别指向两个链表的头部，每次取较小的那个节点接到结果链表后面。

```go
func mergeTwoLists(l1 *ListNode, l2 *ListNode) *ListNode {
    dummy := &ListNode{}
    cur := dummy
    for l1 != nil && l2 != nil {
        if l1.Val < l2.Val {
            cur.Next = l1
            l1 = l1.Next
        } else {
            cur.Next = l2
            l2 = l2.Next
        }
        cur = cur.Next
    }
    if l1 != nil {
        cur.Next = l1
    }
    if l2 != nil {
        cur.Next = l2
    }
    return dummy.Next
}
```

`dummy`（哨兵节点）是一个常用技巧：它简化了边界条件处理（不需要特殊判断结果链表是否为空）。最后返回`dummy.Next`才是真正的头。

```
l1: 1→3→5→nil
l2: 2→4→6→nil

过程：
dummy→1→2→3→4→5→6→nil
返回 dummy.Next = 1→2→3→4→5→6→nil
```

时间复杂度O(n+m)，空间复杂度O(1)。

## 🤔 想多一点：为什么Go标准库有list但大家不怎么用？

`container/list` 是Go标准库的一等公民，但你去看大多数Go项目的代码，几乎看不到它的身影。原因：

### 1. 切片太好用了

Go的切片设计得极其顺手——append、copy、切片子串，所有常用操作都非常简洁。而`container/list`的操作需要处理`*Element`指针，代码更啰嗦。

### 2. CPU缓存是个大问题

现代CPU的缓存机制决定了：连续内存访问极快（切片），跳跃式内存访问很慢（链表）。

```
切片遍历：[1][2][3][4][5]  ← 一个缓存行(64字节)就能装下好几个int
链表遍历：[1] → [2] → [3] → [4] → [5]  ← 每个节点可能在内存的任意位置
```

CPU每次从内存读一个缓存行（64字节）。切片可能一次读4~8个元素；链表每次都只读一个节点，剩下的63字节里可能有用的只有一个指针。

**在Go里，遍历一个100万元素的切片可能只要几毫秒，但遍历同样大小的链表可能要几十毫秒——差距可达10倍以上。**

### 3. GC压力

链表的每个节点是独立分配在堆上的小对象。100万个节点就是100万个堆对象——GC扫描它们会非常痛苦。切片只分配一块连续内存，GC扫描时一块就完事。

### 结论：链表是"懂原理但慎用"

链表的学习价值在于理解指针操作、理解数据结构之间的权衡。但在Go的实际开发中，**80%的场景切片就够了，19%的场景用map，只有1%是链表。**

什么情况下那1%真的值得用？当你需要O(1)的中间插入/删除，且元素数量很大（几十万+），且随机访问的需求很少时。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 单向链表 | `Node{Val, Next}`，每个节点只知道下一个 |
| 双向链表 | `Node{Val, Prev, Next}`，知道前后，Go标准库`container/list` |
| 链表查找 | O(n)，必须从头遍历 |
| 链表插入/删除 | O(1)指针操作，但需要O(n)定位 |
| 切片vs链表 | 切片O(1)随机访问+C缓存友好，链表O(1)插入删除+C缓存不友好 |
| 反转链表 | 迭代法：三个指针prev/cur/next，O(n)+O(1)；递归法简洁但O(n)栈空间 |
| 快慢指针 | 找中点/判环，经典双指针技巧 |
| 判环 | 快慢指针相遇则有环；找环入口需数学推导 |
| 合并有序链表 | 哨兵节点+dummy技巧，逐个比较，O(n+m) |
| 实战建议 | 链表重在理解原理，Go生产中大多用切片 |
