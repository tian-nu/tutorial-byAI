# 79-AOP切面编程

> 💡 古代皇帝的"太监"制度——皇帝批奏折时，太监递笔研墨，但从不参与核心决策。笔墨纸砚的准备、奏折的归档，这些"横切关注点"由太监完成，皇帝只管决策。AOP 就是你的太监总管——在你的业务方法执行前后自动完成日志、事务、权限校验等杂活，而你的业务代码干干净净，一行杂活代码都不留。

## 79.1 AOP 依赖与启用

Spring Boot 中 AOP 已集成在 `spring-boot-starter-aop` 中，如果没有单独引入则添加：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>
```

在启动类或任一配置类上加 `@EnableAspectJAutoProxy`（Spring Boot 自动配置已默认启用，一般无需手动加）：

```java
@SpringBootApplication
@EnableAspectJAutoProxy   // 通常可省略，Spring Boot 自动启用
public class DemoApplication { }
```

## 79.2 五种 Advice（通知）类型

| Advice 类型 | 注解 | 执行时机 | 典型用途 |
|-------------|------|----------|----------|
| 前置通知 | `@Before` | 目标方法执行前 | 权限校验、参数日志 |
| 后置通知 | `@After` | 目标方法执行后（无论成功或异常） | 资源释放 |
| 返回通知 | `@AfterReturning` | 目标方法正常返回后 | 缓存结果、返回值日志 |
| 异常通知 | `@AfterThrowing` | 目标方法抛出异常后 | 异常告警、错误日志 |
| 环绕通知 | `@Around` | 包围目标方法，最强大 | 性能计时、事务、重试 |

执行顺序：

```
@Around（前半段）
  → @Before
    → 目标方法执行
  → @AfterReturning（正常） 或 @AfterThrowing（异常）
  → @After（无论如何）
@Around（后半段）
```

## 79.3 切点表达式——execution 语法

AOP 的核心是如何精确定位"要在哪些方法上切入"。最常用的是 `execution` 表达式：

```
execution(修饰符? 返回类型 包名.类名.方法名(参数列表) throws?)
```

`?` 表示可省略。最常用的形式：

```java
// 切入 com.example.demo.service 包下所有类的所有方法
@Pointcut("execution(* com.example.demo.service.*.*(..))")
public void serviceLayer() {}

// 切入所有 Service 结尾的类的所有方法
@Pointcut("execution(* com.example.demo.service.*Service.*(..))")
public void allServices() {}

// 切入所有 find 开头的方法
@Pointcut("execution(* com.example.demo..*.find*(..))")
public void findMethods() {}

// 切入所有 public 方法
@Pointcut("execution(public * *(..))")
public void publicMethods() {}

// 组合多个切点
@Pointcut("serviceLayer() && !findMethods()")
public void serviceButNotFind() {}
```

### execution 表达式拆解

```java
execution(* com.example.demo.service.UserService.getUser(Long))
         │  └──────────────────────┘ └──────┘ └───────┘
         │           包.类           方法名    参数类型
         │
        返回类型（* 表示任意类型）
```

`..` 的含义：
- 在包路径中：`com.example..service.*` 表示"com.example 及其所有子包下的 service 包"
- 在参数列表中：`(..)` 表示"任意参数"

## 79.4 实战：日志切面——记录方法执行时间

### 自定义注解（方便标记需要记录的方法）

```java
package com.example.demo.annotation;

import java.lang.annotation.*;

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface LogExecutionTime {
}
```

### 切面类

```java
package com.example.demo.aspect;

import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.Around;
import org.aspectj.lang.annotation.Aspect;
import org.aspectj.lang.annotation.Pointcut;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.stereotype.Component;
import org.springframework.util.StopWatch;

@Aspect
@Component
public class LoggingAspect {

    private static final Logger log = LoggerFactory.getLogger(LoggingAspect.class);

    // 切点 1：所有带 @LogExecutionTime 注解的方法
    @Pointcut("@annotation(com.example.demo.annotation.LogExecutionTime)")
    public void logExecutionTimeAnnotation() {}

    // 切点 2：所有 Service 层的方法
    @Pointcut("execution(* com.example.demo.service.*.*(..))")
    public void serviceLayer() {}

    // 组合切点：满足任一条件
    @Pointcut("logExecutionTimeAnnotation() || serviceLayer()")
    public void methodsToLog() {}

    @Around("methodsToLog()")
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        String className = joinPoint.getSignature().getDeclaringTypeName();
        String methodName = joinPoint.getSignature().getName();
        Object[] args = joinPoint.getArgs();

        log.info("→ 调用 {}.{}() 参数: {}", className, methodName, args);

        StopWatch stopWatch = new StopWatch();
        stopWatch.start();

        Object result = joinPoint.proceed();  // 执行目标方法

        stopWatch.stop();
        log.info("← 返回 {}.{}() 耗时: {}ms 结果: {}",
                className, methodName, stopWatch.getTotalTimeMillis(), result);

        return result;
    }
}
```

### 使用

```java
@Service
public class UserService {

    @LogExecutionTime  // 加上这个注解，该方法自动被日志切面拦截
    public User create(User user) {
        // 模拟耗时操作
        try { Thread.sleep(100); } catch (InterruptedException ignored) {}
        return user;
    }
}
```

控制台输出：

```
→ 调用 com.example.demo.service.UserService.create() 参数: [User(id=null, username=zhangsan)]
← 返回 com.example.demo.service.UserService.create() 耗时: 105ms 结果: User(id=1, username=zhangsan)
```

## 79.5 @AfterThrowing——异常告警切面

```java
@Aspect
@Component
public class ExceptionAlertAspect {

    @AfterThrowing(
            pointcut = "execution(* com.example.demo.service.*.*(..))",
            throwing = "ex"
    )
    public void alertOnException(JoinPoint joinPoint, Exception ex) {
        String method = joinPoint.getSignature().toShortString();
        // 实际项目中这里可以发送钉钉/飞书/邮件告警
        System.err.println("⚠ 方法 " + method + " 抛出异常: " + ex.getMessage());
    }
}
```

## 79.6 Filter / Interceptor / AOP —— 终极对比

这是本章的核心表格。三者都能拦截请求，但拦截的层面和粒度完全不同：

| 维度 | Filter | Interceptor | AOP |
|------|--------|-------------|-----|
| **所处层级** | Servlet 容器层 | Spring MVC 层 | Spring 业务方法层 |
| **拦截对象** | HTTP 请求 | Controller 方法 | 任意 Spring Bean 的方法 |
| **能否拿到 HTTP 请求/响应** | ✅ 原生 | ✅ 原生 | ❌（需要手动从 RequestContextHolder 获取） |
| **能否拿到方法参数** | ❌ | ✅ 能拿到 | ✅ 能拿到（JoinPoint.getArgs()） |
| **能否修改方法参数/返回值** | ❌ | ❌（postHandle 只能看不能改） | ✅（@Around 可以改参数和返回值） |
| **能否注入 Spring Bean** | 需要 DelegatingFilterProxy | ✅ 直接注入 | ✅ 直接注入 |
| **异常不吞没** | 手动 try-catch | afterCompletion 保证执行 | @AfterThrowing 或 @Around try-catch |
| **精细控制粒度** | 粗（URL 级别） | 中（Controller 方法级别） | 细（任意方法级别，支持参数类型过滤） |
| **典型用途** | 编码设置、CORS、XSS 防御 | 认证鉴权、操作日志 | 缓存、事务、性能监控、重试 |

> 🤔 想多一点：Filter 运作在"酒店大堂"——你还没进门，只看你的身份证（request）。Interceptor 运作在"客房走廊"——你知道他要去 508 房间（Controller），但不能进房间改里面的东西。AOP 运作在"房间里"——你可以站在床边递毛巾、倒水（改参数/返回值），也可以命令人换房间（改方法执行路径）。

## 79.7 AOP 的底层原理——JDK 动态代理 vs CGLIB

AOP 之所以能"在不改源码的情况下增强方法"，靠的是**代理模式**。

```
客户端 → 代理对象（加了切面逻辑）→ 目标对象（原始业务代码）
```

Spring AOP 默认使用两种代理方式：

| | JDK 动态代理 | CGLIB 代理 |
|------|------|------|
| 条件 | 目标类必须实现接口 | 目标类可以没有接口 |
| 代理的是 | 接口 | 子类 |
| 限制 | 只能调用接口中定义的方法 | final 类和方法不能被代理 |
| Spring Boot 默认 | 如果类有接口就用它 | Spring Boot 2.x+ 默认使用 CGLIB |

> 你几乎不需要关心这个选择。Spring Boot 默认 `spring.aop.proxy-target-class=true`（即 CGLIB 代理），覆盖了绝大多数场景。

## 79.8 AOP 的陷阱——同類內部方法調用

**这是 AOP 最经典的坑**：

```java
@Service
public class UserService {

    @LogExecutionTime
    public void methodA() {
        System.out.println("A 执行");
        this.methodB();  // ← 直接调用，不经过代理！
    }

    @LogExecutionTime
    public void methodB() {
        System.out.println("B 执行");
    }
}
```

当你调用 `methodA()` 时，你会看到：
```
→ 调用 methodA()
A 执行
B 执行          ← methodB 的 @LogExecutionTime 没有生效！
← 返回 methodA()
```

**原因**：`this.methodB()` 是 Java 的普通方法调用，它直接调用当前对象本身的方法，而不经过 Spring 的代理对象。代理增强只对"从外部通过代理对象调用"的方法生效。

**解决方案一**：注入自己

```java
@Service
public class UserService {

    @Autowired
    private UserService self;  // 注入自己的代理

    public void methodA() {
        self.methodB();  // 通过代理调用
    }
}
```

**解决方案二**：拆分到不同的 Bean

```java
@Service
public class UserServiceA {
    @Autowired
    private UserServiceB userServiceB;

    public void methodA() {
        userServiceB.methodB();  // 跨 Bean 调用，走代理
    }
}

@Service
public class UserServiceB {
    @LogExecutionTime
    public void methodB() { }
}
```

**解决方案三**：使用 `AopContext.currentProxy()`

```java
@Service
public class UserService {

    public void methodA() {
        ((UserService) AopContext.currentProxy()).methodB();
    }
}
```

需要在配置中开启 `@EnableAspectJAutoProxy(exposeProxy = true)`。

## 80.1 没有全局异常处理的世界

Controller 中的错误处理散落各处：

```java
@GetMapping("/{id}")
public ResponseEntity<?> getById(@PathVariable Long id) {
    try {
        User user = userService.findById(id);
        return ResponseEntity.ok(Result.success(user));
    } catch (RuntimeException e) {
        return ResponseEntity.status(404).body(Result.error(404, e.getMessage()));
    }
}

@PostMapping
public ResponseEntity<?> create(@Valid @RequestBody User user) {
    try {
        User created = userService.create(user);
        return ResponseEntity.ok(Result.success(created));
    } catch (RuntimeException e) {
        return ResponseEntity.status(400).body(Result.error(400, e.getMessage()));
    }
}
```

问题：
- 每个方法都要 try-catch，代码膨胀
- 异常处理格式不统一（有的返回 400，有的返回 500）
- 新增一个异常类型需要改所有 Controller

## 80.2 @ControllerAdvice——一个类统一所有异常

```java
package com.example.demo.exception;

import com.example.demo.common.Result;
import lombok.extern.slf4j.Slf4j;
import org.springframework.http.HttpStatus;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.ResponseStatus;
import org.springframework.web.bind.annotation.RestControllerAdvice;

@Slf4j
@RestControllerAdvice   // = @ControllerAdvice + @ResponseBody
public class GlobalExceptionHandler {

    // 1. 处理自定义业务异常
    @ExceptionHandler(BusinessException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public Result<Void> handleBusinessException(BusinessException e) {
        log.warn("业务异常: {}", e.getMessage());
        return Result.error(e.getCode(), e.getMessage());
    }

    // 2. 处理参数校验异常
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public Result<Void> handleValidationException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors().stream()
                .map(f -> f.getField() + ": " + f.getDefaultMessage())
                .reduce((a, b) -> a + "; " + b)
                .orElse("参数校验失败");
        log.warn("参数校验失败: {}", message);
        return Result.error(400, message);
    }

    // 3. 处理其他所有未捕获的异常（兜底）
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public Result<Void> handleException(Exception e) {
        log.error("系统异常", e);
        // 生产环境不要返回 e.getMessage()，应返回统一提示避免信息泄露
        return Result.error(500, "服务器内部错误，请稍后重试");
    }
}
```

Controller 变干净了：

```java
@GetMapping("/{id}")
public Result<User> getById(@PathVariable Long id) {
    return Result.success(userService.findById(id));
}

@PostMapping
public Result<User> create(@Valid @RequestBody User user) {
    return Result.success(userService.create(user));
}
```

没有任何 try-catch。Service 层抛出的任何异常都会被 `GlobalExceptionHandler` 捕获并转为统一格式。

> 🤔 想多一点：`@RestControllerAdvice` 是全局的——它拦截**所有 Controller** 抛出的异常。如果你需要只拦截特定包下的 Controller，使用 `@ControllerAdvice("com.example.demo.controller")`。

## 80.3 自定义 BusinessException

区分"代码出错"和"业务规则不满足"：

```java
package com.example.demo.exception;

import lombok.Getter;

@Getter
public class BusinessException extends RuntimeException {

    private final int code;

    public BusinessException(int code, String message) {
        super(message);
        this.code = code;
    }

    public BusinessException(String message) {
        this(400, message);
    }
}
```

Service 层中使用：

```java
@Service
public class UserService {

    private final Map<Long, User> userStore = new ConcurrentHashMap<>();

    public User findById(Long id) {
        User user = userStore.get(id);
        if (user == null) {
            throw new BusinessException(404, "用户不存在: " + id);
        }
        return user;
    }

    public User create(User user) {
        if (userStore.values().stream().anyMatch(u -> u.getUsername().equals(user.getUsername()))) {
            throw new BusinessException("用户名已存在: " + user.getUsername());
        }
        user.setId((long) (userStore.size() + 1));
        userStore.put(user.getId(), user);
        return user;
    }
}
```

### 异常码约定

建议建立错误码枚举：

```java
package com.example.demo.exception;

import lombok.Getter;

@Getter
public enum ErrorCode {

    SUCCESS(200, "操作成功"),
    BAD_REQUEST(400, "请求参数错误"),
    UNAUTHORIZED(401, "未登录或 token 过期"),
    FORBIDDEN(403, "无权限"),
    NOT_FOUND(404, "资源不存在"),
    CONFLICT(409, "资源冲突"),
    INTERNAL_ERROR(500, "服务器内部错误");

    private final int code;
    private final String message;

    ErrorCode(int code, String message) {
        this.code = code;
        this.message = message;
    }
}
```

```java
// 使用
throw new BusinessException(ErrorCode.NOT_FOUND.getCode(), "用户不存在: " + id);
```

## 80.4 异常处理的优先级

当有多个 `@ExceptionHandler` 时，Spring 按**精确匹配优先**的原则选择：

```
try {
    userService.findById(id);  // 抛了 BusinessException
} catch (BusinessException e)  ← 精确匹配，优先处理
  catch (RuntimeException e)   ← 次精确
  catch (Exception e)          ← 最宽泛
```

注意：Spring 不会同时执行多个 handler。一旦匹配到一个，后面的不再执行。

## 80.5 处理常见特殊异常

### 404 异常（NoHandlerFoundException）

Spring Boot 默认不抛出 404，需要额外配置：

```yaml
spring:
  mvc:
    throw-exception-if-no-handler-found: true
  web:
    resources:
      add-mappings: false    # 禁用默认静态资源映射
```

```java
@ExceptionHandler(NoHandlerFoundException.class)
@ResponseStatus(HttpStatus.NOT_FOUND)
public Result<Void> handle404(NoHandlerFoundException e) {
    return Result.error(404, "接口不存在: " + e.getRequestURL());
}
```

### HTTP 消息不可读（请求体格式错误）

```java
@ExceptionHandler(HttpMessageNotReadableException.class)
@ResponseStatus(HttpStatus.BAD_REQUEST)
public Result<Void> handleNotReadable(HttpMessageNotReadableException e) {
    return Result.error(400, "请求体格式错误，请检查 JSON 格式");
}
```

### 请求方法不支持

```java
@ExceptionHandler(HttpRequestMethodNotSupportedException.class)
@ResponseStatus(HttpStatus.METHOD_NOT_ALLOWED)
public Result<Void> handleMethodNotSupported(HttpRequestMethodNotSupportedException e) {
    return Result.error(405, "不支持的请求方法: " + e.getMethod());
}
```

## 80.6 最佳实践总结

```java
// ❌ 不好的写法：Controller 中到处 try-catch
// ❌ 不好的写法：直接向客户端抛出堆栈信息
// ❌ 不好的写法：所有异常都用 500 返回

// ✅ 好的写法：
//   1. Service 层抛出 BusinessException（业务规则不满足）
//   2. Service 层抛出标准异常（IllegalArgumentException 等）表示代码逻辑缺陷
//   3. GlobalExceptionHandler 统一拦截并转为 Result 格式
//   4. 参数校验异常单独处理，合并为一条友好的错误信息
//   5. 生产环境不向客户端暴露堆栈信息
```

| @RestControllerAdvice | 全局异常处理入口 |
| @ExceptionHandler | 按异常类型精确匹配处理方法 |
| BusinessException | 自定义业务异常，携带错误码和消息 |
| MethodArgumentNotValidException | Jakarta Validation 校验失败时抛出 |
| 异常优先级 | 精确匹配 > 父类匹配 |
| 兜底处理 | @ExceptionHandler(Exception.class) 捕获所有未处理异常 |

## 80.8 自测题

**1. 以下关于 `@ExceptionHandler` 的描述，哪一项是正确的？**

A. 一个 `@ExceptionHandler` 只能处理一种异常  
B. 多个 `@ExceptionHandler` 匹配时，Spring 会执行所有匹配的方法  
C. `@ExceptionHandler` 必须写在 Controller 类内部  
D. `@ExceptionHandler(Exception.class)` 可以捕获所有未被其他 handler 处理的异常  

**2. Service 层中的 `IllegalArgumentException` 和自定义 `BusinessException`，在全局异常处理器中应该如何区分处理？**

**3. 你有一个全局异常处理器处理 `BusinessException`，但某个 Controller 中也写了一个 `@ExceptionHandler(BusinessException.class)`。当这个 Controller 中抛出 `BusinessException` 时，哪个处理器会生效？**

---

**答案提示**：1→D。2→`BusinessException` 表示可预期的业务规则不满足（返回 400）；`IllegalArgumentException` 表示代码参数传递有误（返回 500 + 记录日志排查）。3→Controller 中的 handler 会生效——Spring 优先选择"就近"的 handler，全局 handler 作为兜底。下一章——Spring Data JPA 深入。
