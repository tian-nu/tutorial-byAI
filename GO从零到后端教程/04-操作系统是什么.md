# 第4章 · 操作系统是什么

> "你写了Go代码，编译成了可执行文件，双击运行——在这背后，操作系统干了什么？它怎么管理CPU、内存和硬盘？进程、线程、并发、并行到底有什么区别？"

---

## 4.1 操作系统的作用

### 翻译官 + 管家

操作系统（OS，Operating System）扮演两个角色：

**角色一：翻译官** —— 把程序的请求翻译成硬件能懂的命令。

你的Go程序说："我要从硬盘读一个文件。"但硬盘只懂电信号。操作系统充当翻译官，把"读文件"翻译成"给硬盘控制器发0x2F指令，参数是扇区号12345"。

**角色二：管家** —— 管理所有硬件资源，不让程序互相打架。

你在听歌、写文档、同时又开着浏览器。三个程序都想用CPU、内存和声卡。操作系统确保每个程序都能分到资源，不会互相抢。

```
┌─────────────────────────────────────────┐
│             应用程序层                    │
│    Go程序    Chrome     VS Code    ...   │
├─────────────────────────────────────────┤
│             操作系统层                    │
│    进程管理 | 内存管理 | 文件系统 | 网络栈  │
├─────────────────────────────────────────┤
│             硬件层                        │
│    CPU    内存    硬盘    网卡    显卡     │
└─────────────────────────────────────────┘
```

### 常见操作系统

| 操作系统 | 用在 | 特点 |
|----------|------|------|
| Windows | 个人电脑（90%+） | 图形界面好，生态丰富 |
| Linux | 服务器（90%+）、Android | 开源免费，稳定，命令行强大 |
| macOS | 苹果电脑 | Unix底层，设计好 |
| FreeBSD | 高端服务器 | 网络栈优秀 |

**你作为后端工程师，你的代码最终运行在Linux服务器上**（极大概率）。所以后面第99章会专门讲Linux。现在先理解通用概念。

## 4.2 进程：一个正在运行的程序

### 什么是进程

你硬盘上有个 `hello.exe` 文件，这是**程序**——一段静态的机器指令。

你双击运行它，操作系统把它加载到内存里，分配CPU时间去执行它——这个**正在运行中的程序**就是**进程**。

```
程序（硬盘上的文件）  →  加载到内存 + CPU分配时间  →  进程（运行中的实例）
```

同一个程序可以启动多次，每次启动都是一个独立的进程。比如你开了三个终端窗口，就有三个终端进程。

### 进程包含什么

一个进程拥有的资源包括：

- **内存空间**：存代码和数据的地方
- **文件描述符**：打开了哪些文件、网络连接
- **CPU时间**：操作系统分配给它的执行时间
- **PID**（进程ID）：唯一的数字标识
- **环境变量**：PATH、HOME等

### 进程隔离

进程之间是**隔离**的。一个进程崩溃不会影响其他进程。

- Chrome崩了，你的VS Code不会受影响
- 一个Go服务挂了，同一个服务器上的MySQL还在正常运行

这种隔离是操作系统通过**虚拟内存**实现的——每个进程看到的是自己独立的"虚拟地址空间"，看不到其他进程的内存。

### 查看进程

```bash
# Windows
tasklist

# Linux / Mac
ps aux        # 列出所有进程
ps aux | grep go   # 找出包含"go"的进程
```

## 4.3 线程：进程里的"工人"

### 什么是线程

一个进程内部可以有多个**线程**（thread）。线程是CPU调度的最小单位。

**比喻：**
- **进程** = 一个公司
- **线程** = 公司里的员工

一个公司至少有一个员工（主线程），可以有很多员工（多线程）。所有员工共享公司的资源（内存空间、文件、连接），但每个员工有自己的工位（栈、寄存器）。

### 为什么需要多线程

想象一个Web服务器：它需要同时处理100个用户的请求。

如果只有一个线程：
- 处理请求1 → 处理请求2 → 处理请求3 → ...
- 请求1在等数据库返回的时候，CPU闲着没事干，但请求2只能在后面排队

如果有多个线程：
- 线程A处理请求1，在等数据库时，CPU切换到线程B处理请求2
- 充分利用CPU，吞吐量大大提升

### 线程的代价

操作系统线程不便宜：
- **创建**：需要分配栈空间（通常1MB），设置寄存器
- **切换**：保存当前线程的寄存器 → 恢复目标线程的寄存器（也叫上下文切换）→ 约1~10微秒
- **内存**：每个线程约1MB栈空间，1000个线程就是1GB

这就是为什么Go发明了Goroutine——一种更轻量的"线程"（初始栈只有2KB，可以动态增长）。（Goroutine是Go语言的轻量级线程，第34章会详细讲）

### Go的Goroutine比线程好在哪

| 特性 | 操作系统线程 | Goroutine |
|------|------------|-----------|
| 初始栈大小 | ~1MB | ~2KB |
| 创建开销 | 大 | 极小 |
| 切换开销 | 需要内核（OS核心，详见附录I）参与 | 在用户态切换 |
| 数量 | 几百到几千 | 可以轻松十万+ |
| 调度 | 操作系统调度 | Go运行时调度 |

## 4.4 并发 vs 并行

### 一个烧水泡面的例子

说明两者区别的经典例子——烧水泡面：

**并发（Concurrency）：**
你一个人在家。你先把水烧上（水壶在烧），烧水的时候你去泡面（打开面饼、放调料），水开了你就去关火。

你**一个人**在做三件事（烧水、泡面、等水开），交替切换，看起来像同时在做好几件事。这就是**并发**——在同一时间段内处理多件事情。

**并行（Parallelism）：**
你和室友在家。你烧水，室友泡面。两个人**真正同时**在做不同的事情。这就是**并行**——在同一时刻处理多件事情。

### 技术定义

- **并发** = 逻辑上同时（时间片轮转，交替执行）
- **并行** = 物理上同时（不同CPU核心同时执行不同任务）

### 在Go代码里

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
// 并发：启动多个Goroutine
// 它们可能会在同一个CPU核心上交替执行（并发）
// 也可能在不同核心上同时执行（并行）
// 这取决于Go运行时如何调度

for i := 0; i < 5; i++ {
    go func(n int) {
        fmt.Printf("Goroutine %d 在运行\n", n)
    }(i)
}
```

- 如果电脑只有1个CPU核心：所有Goroutine是**并发**执行（交替）
- 如果电脑有4个CPU核心：最多4个Goroutine**并行**执行，其余的等待

### 为什么这两者经常被搞混

因为用户看到的都是"同时在做很多事情"。但从实现上看是完全不同的：
- 并发靠的是**时间片轮转**（一个人来回切换）
- 并行靠的是**多核心**（多个人一起上）

Go的口号是"不要通过共享内存来通信，而要通过通信来共享内存"——这句话解决的就是**并发编程的复杂性**（多个人同时改一个变量怎么办？）。

## 本章小结

| 概念 | 一句话 |
|------|--------|
| 操作系统 | 翻译官+管家，管理硬件、服务程序 |
| 进程 | 一个正在运行的程序实例，拥有独立内存空间 |
| 线程 | 进程里的执行单元，共享进程内存 |
| Goroutine | Go的轻量级"线程"，比OS线程便宜得多 |
| 并发 | 一个人交替做多件事（一核多用） |
| 并行 | 多个人同时做事（多核同时用） |
