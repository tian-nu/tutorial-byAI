## 我在做什么？

第 38 章你学会了用 `#define` 做文本替换。但预处理器还有一个更强大的能力——**条件编译**：根据不同的条件，让编译器看到不同的代码。这就像你写了一份"万能演讲稿"，根据听众是谁（Windows 用户还是 Linux 用户？调试模式还是发布模式？），自动选择不同的段落。这一章你将学会用 `#ifdef`、`#ifndef`、`#if` 等指令实现条件编译，理解头文件保护机制，以及用条件编译管理调试代码。

读完本章你能：用 `#ifdef`/`#ifndef` 判断宏是否定义，用 `#if`/`#elif`/`#else` 做数值条件判断，写出头文件保护防止重复包含，用 `#ifdef DEBUG` 管理调试代码，理解条件编译在跨平台代码中的核心作用。

## 一、条件编译是什么？

### 1.1 预处理器也能做"if-else"

C 语言有两个层面的条件判断：
- **运行时条件**：`if-else` 语句，程序运行时判断，所有分支代码都会编译进可执行文件。
- **编译时条件**：`#if`、`#ifdef` 等指令，预处理阶段判断，不满足条件的代码根本不会进入编译。

```text
源文件
    |
    |  #ifdef WINDOWS
    |      代码 A（Windows 专用）
    |  #else
    |      代码 B（Linux 专用）
    |  #endif
    |
    v
预处理器（根据 WINDOWS 是否定义，保留 A 或 B）
    |
    v
编译器（只看到被选中的代码）
```

### 1.2 为什么需要条件编译

- **跨平台**：Windows 和 Linux 的系统调用不同，需要不同代码。
- **调试/发布**：调试时需要打印日志，发布时需要去掉。
- **功能开关**：同一套代码编译出不同版本（免费版/专业版）。
- **头文件保护**：防止同一个头文件被重复包含导致编译错误。

## 二、#ifdef 与 #ifndef：判断宏是否定义

### 2.1 #ifdef：如果定义了

```c
#ifdef 宏名
    // 如果宏名被 #define 过，这段代码保留
#endif
```

```c
#include <stdio.h>

#define DEBUG  // 定义 DEBUG 宏（值不重要，定义了就行）

int main() {
#ifdef DEBUG
    printf("调试模式：程序启动\n");
#endif

    printf("Hello, World!\n");

#ifdef DEBUG
    printf("调试模式：程序结束\n");
#endif

    return 0;
}
```

编译运行：

```
调试模式：程序启动
Hello, World!
调试模式：程序结束
```

如果注释掉 `#define DEBUG`，编译后的程序就只输出 `Hello, World!`——两行调试信息完全不存在于可执行文件中。

### 2.2 #ifndef：如果没定义

```c
#ifndef 宏名
    // 如果宏名没有被 #define 过，这段代码保留
#endif
```

```c
#include <stdio.h>

#ifndef PI
#define PI 3.14159
#endif

int main() {
    printf("PI = %f\n", PI);
    return 0;
}
```

这段代码的意思是：如果 `PI` 还没有被定义，就定义它为 `3.14159`。如果之前已经定义了（比如通过编译命令行 `gcc -DPI=3.14`），就使用已有的定义。

### 2.3 #else 分支

```c
#ifdef 宏名
    // 定义了
#else
    // 没定义
#endif
```

```c
#include <stdio.h>

// #define ENABLE_LOGGING

int main() {
#ifdef ENABLE_LOGGING
    printf("日志功能已启用\n");
#else
    printf("日志功能已禁用\n");
#endif

    return 0;
}
```

### 2.4 通过命令行定义宏

不需要修改源代码，在编译时通过 `-D` 选项定义宏：

```bash
# 定义 DEBUG 宏
gcc -DDEBUG main.c -o main

# 定义宏并指定值
gcc -DBUFFER_SIZE=4096 main.c -o main

# 定义多个宏
gcc -DDEBUG -DVERSION=2 main.c -o main
```

这是条件编译常用的技巧——编译调试版时加 `-DDEBUG`，编译发布版时去掉。

## 三、头文件保护：防止重复包含

### 3.1 问题：重复包含

假设你有一个头文件 `math_utils.h`：

```c
// math_utils.h
#define PI 3.14159

double circle_area(double r);
```

另一个头文件 `geometry.h` 包含了它：

```c
// geometry.h
#include "math_utils.h"

double sphere_volume(double r);
```

你的 `main.c` 同时包含了两个头文件：

```c
// main.c
#include "math_utils.h"
#include "geometry.h"

int main() {
    return 0;
}
```

预处理器展开后，`math_utils.h` 的内容出现了两次：

```c
// 第一次出现（来自 #include "math_utils.h"）
#define PI 3.14159
double circle_area(double r);

// 第二次出现（来自 #include "geometry.h" → #include "math_utils.h"）
#define PI 3.14159          // 重复定义！编译器可能警告或报错
double circle_area(double r);  // 重复声明！编译器报错
```

### 3.2 解决方案：头文件保护

```c
// math_utils.h（正确写法）
#ifndef MATH_UTILS_H       // 如果 MATH_UTILS_H 还没定义
#define MATH_UTILS_H       // 定义它

#define PI 3.14159

double circle_area(double r);

#endif                     // 结束条件编译
```

执行流程：

```text
第一次 #include "math_utils.h":
  → MATH_UTILS_H 没定义 → 定义它，保留所有代码

第二次 #include "math_utils.h"（通过 geometry.h 间接包含）:
  → MATH_UTILS_H 已经定义了 → 跳过所有代码，直到 #endif
```

结果：头文件内容只被处理一次，重复定义/声明的问题消失。

### 3.3 命名惯例

保护宏的命名通常基于头文件名，转为全大写，用下划线分隔：

| 头文件 | 保护宏 |
|--------|--------|
| `math_utils.h` | `MATH_UTILS_H` |
| `student.h` | `STUDENT_H` |
| `my_project/config.h` | `MY_PROJECT_CONFIG_H` |

有些项目会在保护宏前后加双下划线（如 `__MATH_UTILS_H__`），但以双下划线开头的标识符是 C 标准保留的，建议避免。用单下划线前缀或后缀也行（`_MATH_UTILS_H_`），但同样存在一定风险。最安全的方式就是不加特殊前缀：`MATH_UTILS_H`。

### 3.4 完整示例

```c
// math_utils.h
#ifndef MATH_UTILS_H
#define MATH_UTILS_H

#define PI 3.14159

// 计算圆的面积
double circle_area(double r) {
    return PI * r * r;
}

#endif  // MATH_UTILS_H
```

```c
// geometry.h
#ifndef GEOMETRY_H
#define GEOMETRY_H

#include "math_utils.h"

double sphere_volume(double r) {
    return 4.0 / 3.0 * PI * r * r * r;
}

#endif  // GEOMETRY_H
```

```c
// main.c
#include <stdio.h>
#include "math_utils.h"
#include "geometry.h"

int main() {
    printf("圆面积: %f\n", circle_area(5.0));
    printf("球体积: %f\n", sphere_volume(5.0));
    return 0;
}
```

## 四、#if、#elif、#else：数值条件编译

### 4.1 #if 判断数值

```c
#if 常量表达式
    // 如果常量表达式为真（非零），保留
#elif 另一个常量表达式
    // 否则如果这个为真
#else
    // 否则
#endif
```

```c
#include <stdio.h>

#define VERSION 3

int main() {
#if VERSION == 1
    printf("这是版本 1：基础功能\n");
#elif VERSION == 2
    printf("这是版本 2：增加了高级功能\n");
#elif VERSION == 3
    printf("这是版本 3：增加了专业功能\n");
#else
    printf("未知版本\n");
#endif

    return 0;
}
```

输出：

```
这是版本 3：增加了专业功能
```

### 4.2 defined() 操作符

`defined()` 可以在 `#if` 中检查宏是否定义，比 `#ifdef` 更灵活：

```c
#if defined(DEBUG) && (VERSION >= 2)
    // 只在 DEBUG 已定义且 VERSION >= 2 时保留
#endif
```

`defined(DEBUG)` 等价于 `#ifdef DEBUG`，但 `defined()` 可以用在复杂表达式中：

```c
#if defined(FEATURE_A) || defined(FEATURE_B)
    // 如果 FEATURE_A 或 FEATURE_B 任一被定义
#endif

#if !defined(NDEBUG)
    // 如果 NDEBUG 没有被定义（即调试模式）
#endif
```

### 4.3 在 #if 中判断宏未定义

```c
// 写法一：用 #ifndef（推荐，简单直观）
#ifndef FOO
    // FOO 未定义
#endif

// 写法二：用 #if !defined()（适合复杂条件）
#if !defined(FOO)
    // FOO 未定义
#endif

// 写法三：复杂条件
#if defined(FOO) && !defined(BAR)
    // FOO 定义了但 BAR 没定义
#endif
```

### 4.4 跨平台代码示例

```c
#include <stdio.h>

int main() {
#if defined(_WIN32) || defined(_WIN64)
    printf("运行在 Windows 上\n");
    printf("使用 Windows API 获取系统信息...\n");
    // Windows 特定的代码
#elif defined(__linux__)
    printf("运行在 Linux 上\n");
    printf("使用 Linux 系统调用获取系统信息...\n");
    // Linux 特定的代码
#elif defined(__APPLE__)
    printf("运行在 macOS 上\n");
    // macOS 特定的代码
#else
    printf("运行在未知操作系统上\n");
#endif

    return 0;
}
```

不同平台编译时，预处理器会自动选择对应分支。`_WIN32`、`__linux__`、`__APPLE__` 是各编译器自动定义的平台宏。

## 五、调试代码管理：#ifdef DEBUG

### 5.1 基本模式

```c
#include <stdio.h>

#define DEBUG  // 发布时注释掉这一行

int main() {
    int sum = 0;

#ifdef DEBUG
    printf("[DEBUG] 开始计算...\n");
#endif

    for (int i = 1; i <= 10; i++) {
        sum += i;
#ifdef DEBUG
        printf("[DEBUG] i=%d, sum=%d\n", i, sum);
#endif
    }

    printf("1 到 10 的和是: %d\n", sum);

#ifdef DEBUG
    printf("[DEBUG] 计算完成\n");
#endif

    return 0;
}
```

编译调试版：

```bash
gcc -DDEBUG main.c -o main_debug
```

编译发布版（不定义 DEBUG）：

```bash
gcc main.c -o main_release
```

发布版的可执行文件中，所有 `#ifdef DEBUG` 块内的代码都消失了，不占用任何空间，也没有任何性能开销。

### 5.2 封装调试宏

每次都写 `#ifdef DEBUG` 有点啰嗦，可以封装成一个宏：

```c
#include <stdio.h>

#ifdef DEBUG
#define DEBUG_PRINT(fmt, ...) \
    printf("[%s:%d] " fmt, __FILE__, __LINE__, ##__VA_ARGS__)
#else
#define DEBUG_PRINT(fmt, ...)  // 空宏，展开后什么都不留
#endif

int main() {
    DEBUG_PRINT("程序启动\n");

    int sum = 0;
    for (int i = 1; i <= 10; i++) {
        sum += i;
        DEBUG_PRINT("i=%d, sum=%d\n", i, sum);
    }

    printf("结果: %d\n", sum);
    DEBUG_PRINT("程序结束\n");

    return 0;
}
```

`##__VA_ARGS__` 是 GNU 扩展，当可变参数为空时，自动删除前面的逗号。标准 C 中可以用 `__VA_OPT__`（C23），但 `##__VA_ARGS__` 在 gcc 中更通用。

### 5.3 多级日志

```c
#include <stdio.h>

// 定义日志级别：0=无日志，1=错误，2=警告，3=信息
#define LOG_LEVEL 2

#if LOG_LEVEL >= 1
#define LOG_ERROR(fmt, ...)   fprintf(stderr, "[ERROR] " fmt, ##__VA_ARGS__)
#else
#define LOG_ERROR(fmt, ...)
#endif

#if LOG_LEVEL >= 2
#define LOG_WARN(fmt, ...)    fprintf(stderr, "[WARN]  " fmt, ##__VA_ARGS__)
#else
#define LOG_WARN(fmt, ...)
#endif

#if LOG_LEVEL >= 3
#define LOG_INFO(fmt, ...)    printf("[INFO]  " fmt, ##__VA_ARGS__)
#else
#define LOG_INFO(fmt, ...)
#endif

int main() {
    LOG_ERROR("文件打开失败: %s\n", "data.txt");
    LOG_WARN("内存使用率超过 80%%\n");
    LOG_INFO("程序初始化完成\n");

    return 0;
}
```

编译时，`LOG_LEVEL` 以下的日志代码会被彻底删除。你可以通过不同的编译命令生成不同日志级别的版本：

```bash
# 只输出错误的版本
gcc -DLOG_LEVEL=1 main.c -o main

# 输出所有日志的版本
gcc -DLOG_LEVEL=3 main.c -o main
```

## 六、#error 与 #warning：编译时提示

### 6.1 #error：强制编译失败

```c
#if !defined(__STDC_VERSION__) || __STDC_VERSION__ < 201112L
#error "此程序需要 C11 或更高版本！"
#endif
```

如果编译器不满足条件，编译过程会中止并显示错误信息：

```
error: #error "此程序需要 C11 或更高版本！"
```

### 6.2 #warning：编译时警告

```c
#if defined(_WIN32)
#warning "Windows 版本的此功能尚未完全测试，请谨慎使用"
#endif
```

编译时显示警告但不中止：

```
warning: #warning "Windows 版本的此功能尚未完全测试，请谨慎使用"
```

### 6.3 实用场景：确保依赖满足

```c
// 确保必要的宏已定义
#ifndef BUFFER_SIZE
#error "BUFFER_SIZE 未定义！请在编译时指定 -DBUFFER_SIZE=值"
#endif

// 确保平台支持
#if !defined(__linux__) && !defined(__APPLE__)
#error "此程序仅支持 Linux 和 macOS"
#endif
```

## 七、#pragma once：头文件保护的替代方案

### 7.1 语法

```c
// math_utils.h
#pragma once

#define PI 3.14159
double circle_area(double r);
```

`#pragma once` 告诉编译器：这个文件在整个编译过程中只包含一次。效果和 `#ifndef` 保护一样，但只需要一行。

### 7.2 #pragma once vs #ifndef

| 特性 | `#pragma once` | `#ifndef` 保护 |
|------|----------------|----------------|
| 简洁性 | 一行 | 三行 |
| 可移植性 | 几乎所有编译器都支持，但不是 C 标准 | C 标准的一部分 |
| 唯一性 | 编译器根据文件路径判断 | 依赖宏名的唯一性 |
| 软链接/硬链接 | 可能出现问题（同一文件不同路径） | 由宏名控制，不依赖路径 |
| 推荐度 | 新项目常用 | 保守项目、跨平台库必备 |

实际项目中，如果你确定目标编译器支持，用 `#pragma once` 可以省不少事。但如果你在写一个需要广泛分发的库，用 `#ifndef` 保护更保险。

### 7.3 双保险写法

有些项目同时使用两种保护：

```c
#ifndef MATH_UTILS_H
#define MATH_UTILS_H
#pragma once

// 头文件内容...

#endif
```

## 八、常见错误

### 8.1 忘记 #endif

```c
// ❌ 编译错误
#ifdef DEBUG
    printf("debug\n");
// 缺少 #endif

// ✅
#ifdef DEBUG
    printf("debug\n");
#endif
```

### 8.2 头文件保护宏名冲突

```c
// math_utils.h
#ifndef MATH_H    // ❌ 太通用，容易和其他文件冲突
#define MATH_H

// utils.h
#ifndef MATH_H    // 同一个宏名！第二个文件的内容会被跳过
#define MATH_H

// ✅ 用包含路径的宏名
// math_utils.h  →  MATH_UTILS_H
// string_utils.h → STRING_UTILS_H
```

### 8.3 #if 中使用了变量

```c
int x = 5;
#if x > 3        // ❌ 错误！x 是变量，预处理器不知道它的值
    printf("x > 3\n");
#endif

// ✅ 只能用宏
#define X 5
#if X > 3
    printf("X > 3\n");
#endif
```

`#if` 只能使用**编译时已知**的常量表达式。变量值要到运行时才知道，预处理器在编译前就运行了。

### 8.4 #ifdef 与 #if defined() 的混淆

```c
// 这两行等价
#ifdef FOO
#if defined(FOO)

// 但 #ifdef 不能做复杂判断，只能用 defined()
#if defined(FOO) && defined(BAR)   // ✓
#ifdef FOO && BAR                    // ❌ 语法错误
```

## 九、我做得对不对？

### 9.1 验证条件编译

```c
#include <stdio.h>

#define FEATURE_X
#define VERSION 2

int main() {
    printf("特性检查:\n");

#ifdef FEATURE_X
    printf("  FEATURE_X: 已启用\n");
#else
    printf("  FEATURE_X: 已禁用\n");
#endif

#ifdef FEATURE_Y
    printf("  FEATURE_Y: 已启用\n");
#else
    printf("  FEATURE_Y: 已禁用\n");
#endif

#if VERSION == 1
    printf("  版本: 基础版\n");
#elif VERSION == 2
    printf("  版本: 专业版\n");
#endif

    return 0;
}
```

预期输出：

```
特性检查:
  FEATURE_X: 已启用
  FEATURE_Y: 已禁用
  版本: 专业版
```

### 9.2 验证头文件保护

创建三个文件：

`math_utils.h`：

```c
#ifndef MATH_UTILS_H
#define MATH_UTILS_H

#define PI 3.14159

#endif
```

`geometry.h`：

```c
#ifndef GEOMETRY_H
#define GEOMETRY_H

#include "math_utils.h"

#define E 2.71828

#endif
```

`main.c`：

```c
#include <stdio.h>
#include "math_utils.h"
#include "geometry.h"

int main() {
    printf("PI = %f, E = %f\n", PI, E);
    return 0;
}
```

编译：

```bash
gcc main.c -o main
```

应该正常编译通过。如果去掉 `math_utils.h` 中的 `#ifndef`/`#define`/`#endif` 保护，由于 `PI` 被重复定义，编译器会报警告或错误。

### 9.3 验证调试宏

```c
#include <stdio.h>

// 注释掉下一行，比较输出差异
#define DEBUG

#ifdef DEBUG
#define DBG(fmt, ...) printf("[DEBUG] " fmt, ##__VA_ARGS__)
#else
#define DBG(fmt, ...)
#endif

int factorial(int n) {
    DBG("factorial(%d) 被调用\n", n);
    if (n <= 1) {
        DBG("factorial(%d) 返回 1\n", n);
        return 1;
    }
    int result = n * factorial(n - 1);
    DBG("factorial(%d) 返回 %d\n", n, result);
    return result;
}

int main() {
    printf("5! = %d\n", factorial(5));
    return 0;
}
```

定义 `DEBUG` 时，输出包含调试信息；不定义时，输出只有 `5! = 120`。

## 十、不对怎么办？

| 现象 | 可能原因 | 解决 |
|------|----------|------|
| 编译错误：`expected '...' before '...'` 在头文件附近 | 头文件缺少保护，被重复包含 | 给头文件加上 `#ifndef`/`#define`/`#endif` |
| 条件编译的代码始终不执行 | 宏名拼写错误、大小写不对 | 检查宏名拼写，注意 `#ifdef DEBUG` 和 `#ifdef debug` 不同 |
| `#ifdef` 后面的代码被 IDE 灰掉 | IDE 认为宏未定义，但编译时通过 `-D` 定义了 | 在代码中显式 `#define` 或 IDE 设置中配置宏 |
| `#if` 条件判断错误 | `#if` 中使用了变量或运行时值 | `#if` 只能用 `#define` 的宏和常量表达式 |
| 头文件保护宏名冲突 | 两个头文件用了相同的保护宏名 | 用包含路径的独特宏名 |
| `#error` 导致编译失败 | 你的编译环境不满足条件 | 根据 `#error` 提示调整编译选项或环境 |
| `#pragma once` 不生效 | 编译器不支持或被软链接迷惑 | 改用 `#ifndef` 保护 |

---

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 条件编译 | 预处理器根据条件选择保留哪些代码，不满足条件的代码不编译 |
| #ifdef / #ifndef | 判断宏是否已定义 |
| #if / #elif / #else | 判断常量表达式 |
| #endif | 结束条件编译块 |
| defined() | 在 `#if` 中检查宏是否定义，可组合复杂条件 |
| 头文件保护 | `#ifndef HEADER_H` / `#define HEADER_H` / `#endif` 防止重复包含 |
| #pragma once | 头文件保护的简化写法（非标准但广泛支持） |
| -D 编译选项 | 通过命令行定义宏：`gcc -DDEBUG -DVERSION=2` |
| #error | 强制编译失败，用于检查依赖 |
| #warning | 编译时显示警告信息 |
| 调试代码 | 用 `#ifdef DEBUG` 包裹调试代码，发布时自动去除 |
