# 第56章 · 事务

> "银行转账：A账户扣100元，B账户加100元。如果扣完A的100元后，MySQL服务器突然断电了——A的钱没了，B也没收到。A第二天打电话过来，你说'可能被黑洞吞了'。这就是没有事务的世界。事务是数据库的原子弹保险按钮：要么两件事都做完，要么一件事都不做。转账这个问题，60年前就开始困扰计算机科学家了。"

---

## 56.1 什么是事务

**事务**（一组不可分割的SQL操作，要么全做要么全不做，详见附录I）是一组不可分割的SQL操作。就像化学中的原子——要么整体发生，要么整体不发生，没有"做了一半"的状态。

经典转账场景：

```sql
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
```

如果没有事务保护：

| 时间点 | 事件 | A余额 | B余额 |
|--------|------|-------|-------|
| T1 | 开始 | 1000 | 500 |
| T2 | A扣100 | 900 | 500 |
| T3 | **系统崩溃** | 900 | 500 |
| ... | 100元凭空消失 | | |

有了事务保护：

| 时间点 | 事件 | 效果 |
|--------|------|------|
| T1 | BEGIN（开始事务） | |
| T2 | A扣100成功 | |
| T3 | B加100失败（系统崩溃） | |
| T4 | MySQL恢复后发现事务未完成 | 自动ROLLBACK（回滚） |
| T5 | A的余额恢复为1000 | 就像什么都没发生过 |

这就是事务的核心价值：**保证数据一致性**。

## 56.2 ACID四大特性

事务有四个不可妥协的特性——简称ACID（原子性、一致性、隔离性、持久性四特性，详见附录I）。

### A：原子性（Atomicity）

**要么全做，要么全不做**。

就像在ATM机上取款：必须在钞票吐出来之前先扣款。扣款成功但钱卡住了——这不行。原子性保证"扣款+吐钞"要么一起成功，要么一起失败。

InnoDB通过**undo log**（回滚日志）实现原子性：执行UPDATE时，先把旧值写入undo log。如果事务失败，根据undo log把数据恢复到事务开始前的状态。

### C：一致性（Consistency）

**事务前后，数据库必须从一个一致状态到另一个一致状态**。

一致性更像是一种"约束"——由原子性、隔离性、持久性共同保证。比如转账前后，总金额必须不变（A+B=1500）。如果A扣了100但B没加，总金额变成了1400——一致性被破坏。

还有数据库的约束：主键不重复、外键必须存在、NOT NULL字段不能为空……事务结束后这些约束依然满足。

### I：隔离性（Isolation）

**多个事务同时操作数据时，互不干扰**。

A正在转账（A扣100→B加100），同时C在查询余额。C看不到A扣了但B还没加的中间状态——C要么看到完整的转账前，要么看到完整的转账后。

隔离性通过**锁机制**和**MVCC**（多版本并发控制）实现——第57章详细讲。

### D：持久性（Durability）

**一旦事务提交成功，结果就是永久的——即使数据库立刻崩溃，重启后数据还在**。

InnoDB通过**redo log**（重做日志）实现持久性：执行事务时，先把改动写入redo log（顺序写磁盘——很快），再修改内存中的数据页，最后异步刷到数据文件。如果事务提交后崩溃了，恢复时"重放"redo log，保证数据不丢。

🤔 **想多一点**：如果你在MySQL里执行了COMMIT，然后立刻拔电源——数据还在吗？在。因为COMMIT的时候，redo log已经写到了磁盘（`innodb_flush_log_at_trx_commit = 1` 保证每次提交都刷盘）。即使数据文件还没更新，redo log可以"重做"每一笔已提交的事务。

## 56.3 隔离级别

隔离性是平衡**性能**和**数据安全**的精妙设计。隔离越强，数据越安全，但性能越差——所以要分级。

MySQL支持四种隔离级别，从弱到强：

### READ UNCOMMITTED（读未提交）——最低级

事务A可以读到事务B**未提交**的数据。

```
事务A: BEGIN
事务A: UPDATE users SET age=30 WHERE id=1     (age: 25 → 30, 未提交)
事务B: SELECT age FROM users WHERE id=1      → 读到 30!
事务A: ROLLBACK                               (age变回25)
事务B：读到的30是"脏数据"——永远不会真正存在的数据
```

**问题**：脏读（Dirty Read）——读到了别人未提交的数据，而这些数据可能被回滚。

**生产中几乎没人用这个级别**——读到的数据随时可能被回滚，等于数据完全不可信。

### READ COMMITTED（读已提交）——Oracle默认

事务A只能读到事务B**已经提交**的数据。

```
事务A: BEGIN
事务A: UPDATE users SET age=30 WHERE id=1    (age:25→30, 未提交)
事务B: SELECT age FROM users WHERE id=1      → 读到 25（事务A未提交，看不到变化）
事务A: COMMIT                                 (age永久=30)
事务B: SELECT age FROM users WHERE id=1      → 读到 30（事务A已提交，看得到）
```

**问题**：不可重复读（Non-Repeatable Read）——同一个事务内两次读同一个数据，结果不一样。第一次读到25，第二次读到30。

**适用场景**：对精度要求不太高，但需要一定并发性能的应用。PostgreSQL的默认级别也是这个。

### REPEATABLE READ（可重复读）——MySQL InnoDB默认

同一个事务内，**每次读到的数据都是一样的**——即使其他事务已经提交了修改。

```
事务A: BEGIN
事务A: SELECT age FROM users WHERE id=1      → 读到 25
事务B: BEGIN
事务B: UPDATE users SET age=30 WHERE id=1
事务B: COMMIT                                 (age永久=30)
事务A: SELECT age FROM users WHERE id=1      → 仍然读到 25！
事务A: COMMIT
```

事务A在整个生命周期中，看到的users表都是事务开始时刻的"快照"——这是一种时间旅行般的体验。

**问题**：MVCC解决了大部分幻读，但**当前读**（SELECT ... FOR UPDATE）下仍可能存在幻读。

### SERIALIZABLE（串行化）——最高级

所有事务**一个接一个地串行执行**，完全隔离。

```
事务A: BEGIN
事务A: SELECT ...  (锁定)
事务B: BEGIN
事务B: INSERT ...  → 等待（因为事务A锁住了相关范围）
事务A: COMMIT
事务B: INSERT ...  → 执行
```

**问题**：并发性能极差——事务B要等事务A完成才能动。高并发场景下不可用。

## 56.4 并发问题对照表

| 并发问题 | 说明 | READ UNCOMMITTED | READ COMMITTED | REPEATABLE READ | SERIALIZABLE |
|----------|------|:---:|:---:|:---:|:---:|
| 脏读 | 读到未提交的修改 | ✗ | ✓ | ✓ | ✓ |
| 不可重复读 | 同事务内两次读同一行，值不同 | ✗ | ✗ | ✓ | ✓ |
| 幻读 | 同事务内两次范围查询，行数不同 | ✗ | ✗ | 部分✓* | ✓ |

（✓ = 该隔离级别可解决此问题，✗ = 该隔离级别无法解决此问题）

*MySQL InnoDB在REPEATABLE READ下通过**间隙锁**（Gap Lock）解决了大部分幻读（对于当前读）。快照读天然无幻读。

### 脏读详解

读到了别人未提交的数据。好比你在餐厅点菜，服务员还没确认下单，你就跟朋友说"菜已经点了"——结果服务员告诉你他们忘了，菜根本没下。

### 不可重复读详解

同一个事务内，两次读同一行结果不一样。好比你看天气预报说今天是晴天，结果出门下雨了——信息在你手中"变质了"。

### 幻读详解

同一个事务内，两次范围查询返回的行数不同——第一次查出来10行，第二次查出来12行，多的那2行像"凭空出现"一样。

```
事务A: SELECT * FROM users WHERE age > 20  → 10行
事务B: INSERT INTO users (age) VALUES (25)
事务B: COMMIT
事务A: SELECT * FROM users WHERE age > 20  → 11行（多了一行！）
```

**不可重复读 vs 幻读**：不可重复读是针对**同一行**值变了（UPDATE改了一行的值），幻读是针对**行数**变了（INSERT/DELETE了行）。

## 56.5 MySQL事务操作

### 基础操作

```sql
START TRANSACTION;

UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

COMMIT;
```

如果中途发现问题：

```sql
START TRANSACTION;

UPDATE accounts SET balance = balance - 100 WHERE id = 1;

ROLLBACK;
```

ROLLBACK让所有改动撤销，数据库回到事务开始前的状态。

### BEGIN vs START TRANSACTION

```sql
BEGIN;
START TRANSACTION;
```

两者都能开启事务，但有微小区别：`START TRANSACTION` 后面可以加修饰符，如 `START TRANSACTION READ ONLY`（只读事务）。

### SAVEPOINT：事务中的存档点

```sql
START TRANSACTION;

UPDATE accounts SET balance = balance - 100 WHERE id = 1;
SAVEPOINT after_deduct;

UPDATE accounts SET balance = balance + 100 WHERE id = 2;

ROLLBACK TO SAVEPOINT after_deduct;

COMMIT;
```

ROLLBACK TO SAVEPOINT只撤销到存档点，之前的操作保留。这就好比游戏存档——打BOSS前存个档，挂了从这里重来，不用从头开始。

### 设置隔离级别

```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

查看当前隔离级别：

```sql
SELECT @@transaction_isolation;
```

### 自动提交

MySQL默认开启了**自动提交**——每条SQL语句自动包装在一个事务中：

```sql
SHOW VARIABLES LIKE 'autocommit';
```

关闭自动提交后，每条语句不再自动提交，需要手动COMMIT：

```sql
SET autocommit = 0;
```

但在应用开发中，尽量不要依赖关闭自动提交——用显式的 `START TRANSACTION ... COMMIT` 代码更清晰。

### 只读事务

```sql
START TRANSACTION READ ONLY;

SELECT * FROM users WHERE id = 1;

COMMIT;
```

只读事务不允许写操作——加了双重保险，防止不小心改数据。对纯查询的批量操作很有用。
