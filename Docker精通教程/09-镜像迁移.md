## 我在做什么？

到目前为止，你用的镜像都是从 Docker Hub 拉取的。但如果你的服务器没有外网呢？如果你想把一台机器上的镜像搬到另一台机器上呢？如果你修改了容器，想把修改保存成新镜像呢？

这一章教你镜像的"搬运"技术：从 `docker commit`（保存容器修改为镜像）、`docker tag`（打标签）、`docker save/load`（离线导出导入）、到 `docker push`（推送到云端仓库）。学完之后，你的镜像不再被困在一台机器上，可以在任何地方分发。

学完这一章，你能：
- 用 `docker commit` 把容器修改保存为新镜像（以及理解为什么不推荐）
- 用 `docker tag` 给镜像打标签
- 用 `docker save / docker load` 在离线环境传输镜像
- 用 `docker push` 推送镜像到 Docker Hub
- 完成离线环境镜像传输的完整流程

## 一、`docker commit`：把容器变成镜像

`docker commit` *此术语见附录C* 把一个**容器的当前状态**保存为一个新镜像。就像你在一道菜上加了调料后，给这道新菜拍张照，以后可以照着这张照片重新做。

### 基本用法

```bash
docker commit 容器名 新镜像名:标签
```

**场景**：你进入了一个 Nginx 容器，修改了默认页面，安装了一些工具，现在想把这个"修改后的状态"保存成一个新镜像，以后可以直接用。

```bash
# 1. 启动一个 Nginx 容器
docker run -d --name commit-test -p 8080:80 nginx:1.27.4

# 2. 进入容器，做一些修改
docker exec -it commit-test bash
echo "<h1>Custom Page</h1>" > /usr/share/nginx/html/index.html
apt update && apt install -y curl   # 装了一个工具
exit

# 3. 验证修改
curl http://localhost:8080
# 看到 "Custom Page"

# 4. 把容器保存为新镜像
docker commit commit-test my-custom-nginx:v1

# 5. 查看新镜像
docker images my-custom-nginx
# REPOSITORY          TAG       IMAGE ID       SIZE
# my-custom-nginx     v1        xyz789abc012   250MB
# ↑ 比原始 nginx 镜像大了不少，因为装了 curl 等工具

# 6. 用新镜像启动容器
docker run -d --name custom-nginx -p 8081:80 my-custom-nginx:v1
curl http://localhost:8081
# 看到 "Custom Page"——修改被保留了
```

### 为什么不推荐 `docker commit`？

`docker commit` 看起来很方便，但有几个严重问题：

| 问题 | 说明 |
|------|------|
| **不可追溯** | 镜像里到底改了什么？只有你自己知道。别人（包括三个月后的你）无法知道这个镜像是怎么来的。 |
| **镜像臃肿** | 你在容器里装的每个包、创建的每个文件，都会留在镜像里。没办法清理。 |
| **不可重复** | 你能手动再做一次吗？如果容器不小心删了，你能重新做出一模一样的镜像吗？ |
| **安全风险** | 容器里可能残留了临时文件、密码、密钥等敏感信息。 |

> **正确做法**：用 **Dockerfile** 来定义镜像的构建过程。Dockerfile 是一个文本文件，记录每一步操作，可追溯、可重复、可审查。这将在后面的章节详细展开。

### `docker commit` 的合法用途

虽然不推荐日常使用，但 `docker commit` 在以下场景是合理的：

- **调试**：容器里出了问题，你想保存现场，发给同事分析
- **快速原型**：临时测试一个想法，不想写 Dockerfile
- **创建基础镜像**：从一个干净的 Linux 镜像开始，手动装好基础工具，commit 作为内部基础镜像

### 提交时添加元信息

```bash
docker commit \
  --author "Your Name <your_email@example.com>" \
  --message "Added curl and custom index.html" \
  commit-test my-custom-nginx:v1
```

这样以后用 `docker inspect my-custom-nginx:v1` 可以看到作者和提交信息。

## 二、`docker tag`：给镜像贴标签

`docker tag` *此术语见附录C* 给镜像**多加一个标签**，不复制镜像数据。就像给同一本书贴两个不同的标签——书还是那本书，只是多了一个称呼。

### 基本用法

```bash
docker tag 源镜像[:标签] 目标镜像[:标签]
```

`docker tag` 不给镜像复制一份——它只是给同一个镜像**多加一个标签**。

```bash
# 给 nginx:1.27.4 多加一个标签
docker tag nginx:1.27.4 my-nginx:stable

# 查看——两个标签指向同一个 IMAGE ID
docker images nginx
# REPOSITORY   TAG       IMAGE ID
# nginx        1.27.4    abc123def456
# my-nginx     stable    abc123def456   ← 同一个 ID
```

### `docker tag` 的常见用途

**用途 1：给镜像起一个便于记忆的名字**

```bash
docker tag nginx:1.27.4 my-project/nginx:production
```

**用途 2：准备推送到 Docker Hub**

```bash
# Docker Hub 的格式：用户名/仓库名[:标签]
docker tag my-custom-nginx:v1 your_username/my-custom-nginx:v1

# 然后才能 push
docker push your_username/my-custom-nginx:v1
```

**用途 3：给版本打语义化标签**

```bash
docker tag my-app:abc123def456 my-app:latest
docker tag my-app:abc123def456 my-app:v2.1.0
docker tag my-app:abc123def456 my-app:stable
```

## 三、`docker save` / `docker load`：离线传输镜像

### 什么时候需要？

- 服务器没有外网，无法从 Docker Hub 拉取镜像
- 镜像太大，网络传输慢，用 U 盘/移动硬盘更快
- 想把镜像归档保存

### `docker save`：导出镜像为 tar 文件

`docker save` *此术语见附录C* 把镜像及其所有层打包成一个 tar 文件，方便离线传输或归档。就像把一整本食谱复印装订成册，可以邮寄给任何人。

```bash
# 基本用法
docker save -o 输出文件名.tar 镜像名:标签

# 导出 nginx 镜像
docker save -o nginx-1.27.4.tar nginx:1.27.4

# 导出多个镜像到一个文件
docker save -o my-images.tar nginx:1.27.4 alpine:latest

# 用管道压缩（减小文件体积）
docker save nginx:1.27.4 | gzip > nginx-1.27.4.tar.gz
```

### `docker load`：从 tar 文件导入镜像

`docker load` *此术语见附录C* 从 tar 文件中恢复镜像到本地，是 `docker save` 的逆操作。就像收到邮寄的食谱后，把它放回你的书架。

```bash
# 基本用法
docker load -i 输入文件名.tar

# 导入 nginx 镜像
docker load -i nginx-1.27.4.tar

# 输出：
# Loaded image: nginx:1.27.4

# 从压缩文件导入
docker load < nginx-1.27.4.tar.gz
```

### 完整离线传输流程

假设你在有网的机器 A 上下载了镜像，现在要传到没网的机器 B 上：

**机器 A（有网）**：

```bash
# 1. 拉取镜像
docker pull nginx:1.27.4
docker pull alpine:latest

# 2. 导出镜像
docker save -o images.tar nginx:1.27.4 alpine:latest

# 3. 把 images.tar 复制到机器 B
# 用 U 盘、scp、内网共享等任何方式
scp images.tar user@machine-b:/home/user/
```

**机器 B（无网）**：

```bash
# 1. 导入镜像
docker load -i images.tar

# 输出：
# Loaded image: nginx:1.27.4
# Loaded image: alpine:latest

# 2. 验证
docker images
# 应该能看到 nginx:1.27.4 和 alpine:latest

# 3. 正常使用
docker run -d --name my-nginx -p 8080:80 nginx:1.27.4
```

### 验证离线传输是否成功

最严格的验证方法：完全删除本地镜像，然后从 tar 恢复，确认能正常使用。

```bash
# 1. 导出镜像
docker save -o nginx.tar nginx:1.27.4

# 2. 删除本地镜像
docker rmi nginx:1.27.4

# 3. 确认已删除
docker images nginx
# 应该看不到 nginx:1.27.4

# 4. 从 tar 恢复
docker load -i nginx.tar

# 5. 确认恢复成功
docker images nginx
# 应该看到 nginx:1.27.4

# 6. 用恢复的镜像启动容器
docker run --rm nginx:1.27.4 nginx -v
# 确认 Nginx 能正常运行
```

## 四、`docker push`：推送镜像到 Docker Hub

`docker push` *此术语见附录C* 把本地镜像上传到远程仓库（如 Docker Hub），让其他人或服务器可以拉取。就像把你写的食谱上传到菜谱网站，全世界都能看到。

### 前置条件

1. 注册 Docker Hub 账号（hub.docker.com）
2. 在本地登录 Docker Hub

```bash
docker login
# 输入你的 Docker Hub 用户名和密码
# Login Succeeded
```

> 密码不会显示在终端上，这是正常的。密码会被存储在 `~/.docker/config.json` 中，注意不要泄露这个文件。

### 推送镜像

```bash
# 1. 给镜像打上 Docker Hub 格式的标签
docker tag nginx:1.27.4 your_username/my-nginx:1.27.4

# 2. 推送
docker push your_username/my-nginx:1.27.4

# 输出：
# The push refers to repository [docker.io/your_username/my-nginx]
# abc123: Pushed
# def456: Pushed
# ...
# 1.27.4: digest: sha256:xxxxxxxx size: 1234
```

### 推送成功后

- 任何人都可以在 Docker Hub 上看到你的镜像（如果你设为公开仓库）
- 任何人都可以 `docker pull your_username/my-nginx:1.27.4`
- 你也可以在 Docker Hub 网站上管理镜像

### 推送到私有仓库

```bash
# 1. 给镜像打上私有仓库的标签
docker tag my-app:v1 your-registry.com:5000/my-app:v1

# 2. 推送
docker push your-registry.com:5000/my-app:v1
```

私有仓库的搭建不在本章范围内，但原理是一样的：打标签 → 推送。

## 五、镜像迁移方式对比

| 方式 | 命令 | 适用场景 | 优点 | 缺点 |
|------|------|---------|------|------|
| Docker Hub | `docker push/pull` | 有外网的机器 | 方便、自动分层、全球可访问 | 需要外网、公开仓库免费 |
| 离线导出 | `docker save/load` | 无外网的服务器 | 不需要网络、可靠 | 文件大、手动传输 |
| 私有仓库 | `docker push/pull` | 企业内部 | 安全、集中管理 | 需要搭建维护 |
| `docker commit` | `docker commit` | 快速保存容器状态 | 快速 | 不可追溯、臃肿、不推荐 |

## 六、实战：完整走一遍离线传输流程

```bash
# ===== 模拟离线传输（在同一台机器上验证） =====

# 1. 拉取一个镜像
docker pull alpine:latest

# 2. 查看镜像信息
docker images alpine
# 记录 IMAGE ID 和 SIZE

# 3. 导出镜像
docker save -o alpine.tar alpine:latest

# 4. 查看 tar 文件大小
ls -lh alpine.tar

# 5. 删除本地镜像
docker rmi alpine:latest

# 6. 确认已删除
docker images alpine
# 应该没有输出

# 7. 从 tar 文件导入
docker load -i alpine.tar

# 8. 确认导入成功
docker images alpine
# 应该看到 alpine:latest，IMAGE ID 和 SIZE 和之前一样

# 9. 用导入的镜像运行容器
docker run --rm alpine echo "Alpine is back!"
# 输出：Alpine is back!

# 10. 清理 tar 文件
rm alpine.tar
```

## 我做得对不对？

### 验证方法

请按顺序执行以下操作，**全部成功**才算通过：

```bash
# 1. 拉取一个 alpine 镜像
docker pull alpine:latest

# 2. 导出 alpine 镜像为 tar 文件
docker save -o alpine.tar alpine:latest

# 3. 删除本地 alpine 镜像
docker rmi alpine:latest

# 4. 确认 alpines 镜像已删除

# 应该没有输出

# 5. 从 tar 文件加载 alpine 镜像
docker load -i alpine.tar

# 6. 确认 alpine 镜像已恢复

# 应该看到 alpine:latest

# 7. 用恢复的镜像启动容器，验证能正常工作

# 预期输出：Transfer successful!

rm alpine.tar
```

**预期结果**：

- 步骤 5：看到 `Loaded image: alpine:latest`
- 步骤 6：`docker images alpine` 显示 alpine 镜像
- 步骤 7：输出 `Transfer successful!`

## 不对怎么办？

### 常见错误 1：`docker commit` 导致镜像巨大

**现象**：`docker commit` 后镜像大小从 187MB 变成了 1.5GB。

**原因**：你在容器里执行了 `apt update` 和 `apt install`，这些操作在容器可写层添加了大量文件。`docker commit` 会把可写层的所有内容打包进新镜像，包括 apt 缓存、临时文件等。

**建议**：不要用 `docker commit` 做生产镜像。用 Dockerfile 构建，可以在构建过程中清理缓存。

### 常见错误 2：`docker save` 导出的 tar 文件很大

**现象**：`docker save nginx:latest` 导出的文件有好几百 MB。

**原因**：`docker save` 保存的是镜像的**所有层**，包括基础操作系统层。如果基础层是 Debian（~120MB），再加上 Nginx 的层，总大小自然不小。

**减小体积的方法**：

```bash
# 推荐：用 gzip 压缩
docker save nginx:1.27.4 | gzip > nginx.tar.gz

# 或者用更激进的压缩
docker save nginx:1.27.4 | xz > nginx.tar.xz
```

### 常见错误 3：`docker push` 前忘记打标签

```bash
docker push my-nginx:1.27.4
# Error: denied: requested access to the resource is denied
```

**原因**：Docker Hub 的镜像名格式必须是 `用户名/仓库名:标签`。`my-nginx` 这种短名字默认指向 `docker.io/library/my-nginx`，这是一个官方仓库，你没有权限推送。

**解决**：

```bash
docker tag my-nginx:1.27.4 your_username/my-nginx:1.27.4
docker push your_username/my-nginx:1.27.4
```

### 常见错误 4：`docker load` 后镜像名是 `<none>:<none>`

**现象**：

```bash
docker load -i some-image.tar
docker images
# REPOSITORY   TAG       IMAGE ID
# <none>       <none>    abc123def456
```

**原因**：这个 tar 文件是在 `docker save` 时只用了镜像 ID 而没有用标签名，或者是从 `docker commit` 创建的镜像导出的。

**解决**：

```bash
# 给这个镜像打标签
docker tag abc123def456 my-image:latest
```

### 常见错误 5：`docker login` 后密码泄露

**现象**：密码明文存储在 `~/.docker/config.json` 中。

**解决**：Docker 默认用 base64 编码存储密码（不是加密！）。如果你担心安全，使用**访问令牌（Access Token）**代替密码：

1. 登录 Docker Hub 网页 → Account Settings → Security → New Access Token
2. 用 token 登录：`docker login -u your_username`，然后输入 token 而不是密码

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 容器保存为镜像 | `docker commit 容器名 新镜像:标签` | 不推荐日常使用，不可追溯、不可重复、镜像臃肿 |
| 给镜像打标签 | `docker tag 源镜像 目标镜像` | 不复制镜像，只是多加一个标签 |
| 导出镜像 | `docker save -o 文件.tar 镜像:标签` | 保存所有层，文件可能很大；用 gzip 压缩减小体积 |
| 导入镜像 | `docker load -i 文件.tar` | 离线环境的必备操作 |
| 推送镜像到 Docker Hub | `docker push 用户名/仓库:标签` | 需要先 `docker login`，标签格式必须是 `用户名/仓库` |
| 离线传输流程 | save → 拷贝 → load | 验证方法：删除本地镜像 → load → 确认能正常使用 |
| 镜像迁移方式对比 | Docker Hub / 离线 tar / 私有仓库 / commit | 各有适用场景，优先用 Dockerfile + Docker Hub |

## 本篇最可能出错的地方及原因

### 1. `docker commit` 用于生产环境

**原因**：新手觉得 `docker commit` 很方便，改完容器直接保存成镜像。但这种方式完全不可追溯——三个月后你完全不知道这个镜像里改了什么。而且 `docker commit` 会把所有临时文件、apt 缓存都打进镜像，导致镜像臃肿。

**建议**：`docker commit` 只用于调试和快速原型。生产环境必须用 Dockerfile。

### 2. `docker save` 导出的文件太大

**原因**：`docker save` 保存的是完整镜像的所有层，包括基础操作系统层。新手以为 `alpine` 只有 7MB，导出也应该是 7MB——实际上导出的 tar 包含了所有层的数据，可能比 `docker images` 显示的 SIZE 更大。

**建议**：用 `gzip` 压缩可以显著减小体积。离线传输时优先考虑压缩。

### 3. `docker push` 标签格式错误

**原因**：Docker Hub 的标签格式必须是 `用户名/仓库名:标签`。新手直接用 `my-image:latest` 推送，会报权限错误，因为 `my-image` 默认指向官方仓库。

**建议**：推送前先 `docker tag` 加上用户名前缀，养成习惯。
