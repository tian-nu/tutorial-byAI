## 我在做什么？

你点开这本书，说明你决定学 C 语言。但你可能心里打鼓："2026 年了，Python、Go、Rust 满天飞，为什么还要学一个 50 多年前的老语言？"

这一章就用大白话回答这个问题。读完你能清楚地告诉别人：C 语言凭什么活着，它藏在哪些你每天都在用的东西里，以及这本书打算怎么带你"啃"下它。

## 一、C 语言的出身：一个传奇的开端

### 1.1 贝尔实验室的地下室

1972 年，美国新泽西州，*贝尔实验室* *此术语见附录F* 的一间办公室里，Dennis Ritchie *此术语见附录F* 做了一件改变世界的事——他发明了 C 语言。

这件事的背景很有趣。当时贝尔实验室的工程师 Ken Thompson 正在写一个叫 *UNIX* *此术语见附录F* 的操作系统。最早的 UNIX 是用汇编语言写的——那是一种和硬件直接对话的"原始语言"，写起来极其痛苦：

```
汇编代码写 "把两个数加起来" 的样子（示意）：
MOV AX, [0x1000]
ADD AX, [0x1004]
MOV [0x1008], AX
```

你看，只是做个加法，就要写三行，而且必须精确指定数据放在哪个地址。换一台不同型号的机器，这些代码就得重写。

Ritchie 想："能不能有一种语言，写起来像人话，但又足够底层、足够快，能用来写操作系统？"

于是 C 语言诞生了。

### 1.2 一个"刚好够"的设计哲学

C 语言的设计理念可以概括为一句朴素的话：

> 给程序员足够的控制权，但不要用繁琐的语法妨碍他。

C 没有后来的 Java 或 Python 那么多"保护措施"。它信任程序员——你让程序往哪个内存地址写数据，它就写，哪怕那个地址是错的。这种"信任"是双刃剑：高手能写出极致性能，新手可能写出灾难性 bug。

打个比方：

| 语言 | 比喻 |
|------|------|
| Python | 自动驾驶汽车——你告诉它"带我去公司"，它帮你处理一切 |
| Java | 带安全气囊的轿车——很多保护机制，但不够灵活 |
| C | 手动挡赛车——没有辅助，但你能掌控引擎的每一个细节 |

**学 C 语言，本质上是在学"计算机到底怎么工作的"。**

## 二、C 语言还活着吗？比你想象的更健康

你可能觉得 C 语言是"爷爷辈"的技术了。来看看它实际藏在哪里：

### 2.1 操作系统内核

你正在用的 Windows、macOS、Linux——它们的 *内核* *此术语见附录F* 几乎全是 C 语言写的。

```
你的眼睛 -> 屏幕上的窗口 -> 操作系统内核(C语言) -> 硬件
```

内核是操作系统的核心，负责管理 CPU、内存、硬盘。它必须极快、极可靠，而且直接和硬件打交道。能胜任这件事的语言，到今天也只有 C（和少量汇编）。

### 2.2 数据库

MySQL、PostgreSQL、SQLite——这些管理着全世界数据的软件，底层都是 C。

当你"双十一"下单时，淘宝的数据库每秒要处理几十万次请求。每一毫秒的优化都有意义。C 语言的极致性能在这里不可替代。

### 2.3 编程语言的"语言"

Python 的解释器（CPython）是 C 写的。Ruby 的解释器是 C 写的。PHP 的解释器是 C 写的。Node.js 的 V8 引擎是 C++ 写的，而 C++ 正是从 C 演化而来。

这幅图描述了这种关系：

```
你的代码层
+-----------------------------+
|  Python 代码                |
|  def hello():               |
|      print("Hello")         |
+-----------------------------+
         |
         v
解释器层 (C 语言写)
+-----------------------------+
|  CPython 解释器             |
|  把 Python 代码翻译成       |
|  机器能懂的指令             |
+-----------------------------+
         |
         v
操作系统层 (C 语言写)
+-----------------------------+
|  Linux/Windows/macOS 内核   |
+-----------------------------+
         |
         v
硬件层
+-----------------------------+
|  CPU / 内存 / 硬盘          |
+-----------------------------+
```

一句话：**你写 Python 的时候，C 语言正在你的代码下面默默干活。**

### 2.4 嵌入式系统

你的微波炉、汽车刹车控制系统、心脏起搏器、路由器、智能手表——这些设备里的芯片运行的程序，绝大多数是 C 语言写的。

这类设备叫 *嵌入式系统* *此术语见附录F*。它们的特点是：内存小（可能只有几 KB）、处理器慢、必须省电。C 语言编译出来的程序极小、极快，是嵌入式的唯一选择。

### 2.5 游戏引擎

Unreal Engine、Unity 的底层渲染引擎，以及那些你玩 3A 大作时"每秒 60 帧"的流畅画面——它们的核心渲染代码是 C/C++。每帧只有 16 毫秒的时间来绘制整个画面，C 语言才能挤出每一微秒的性能。

## 三、C 语言和其他语言的关系

### 3.1 家族谱系

C 语言不是孤立的，它是整个现代编程世界的"根节点"：

```
1972  C 语言诞生
      |
+-----+-----+-----------+
|     |     |           |
v     v     v           v
C++  Objective-C  Java  C#
(1983) (1984)  (1995) (2000)
      |                  |
      v                  v
    Swift              ......
  (2014)
```

- **C++**：C 语言加上了面向对象。保留了 C 的全部性能，增加了更强大的抽象能力。
- **Java**：借鉴了 C/C++ 的语法，但加上了 *虚拟机* *此术语见附录F* 来保证"一次编写，到处运行"。
- **C#**：微软对 Java 的回应，语法和 C 有血缘关系。
- **Python/Ruby/JavaScript 等脚本语言**：语法不像 C，但它们的 *解释器* *此术语见附录F* 大多是 C 写的。

### 3.2 学了 C，学其他语言会更快

C 语言的语法是"现代编程语言的普通话"。你学了 C：

- C++ 语法几乎一样，只需加学面向对象。
- Java/C# 的循环、判断、变量声明，和 C 一模一样。
- 理解 Python 的列表为什么是动态的（因为 C 帮你管理了内存）。

### 4.2 为什么第零篇不写代码？

很多 C 语言教材第一章就写代码：

```c
#include <stdio.h>
int main() {
    printf("Hello, World!\n");
    return 0;
}
```

然后读者满脸问号：

- `#include` 是什么？
- `printf` 为什么叫这个名字？
- 这个程序怎么"跑"起来的？谁调用了 `main`？
- 输出的字到底怎么显示到屏幕上的？

本书的第零篇（第 0-5 章）先把计算机的基本工作原理讲清楚。当你理解了 CPU、内存、操作系统之后，再回来看 `Hello World`，你会有一种"原来如此"的通透感。

### 4.3 本书学习方法建议

1. **不要跳章**：第零篇的每一章都在为下一篇铺路。跳过去后面会看不懂。
2. **手打代码，不要复制粘贴**：学编程就像学弹琴——光看乐谱是学不会的。
3. **完成每章末尾的练习题**：它们不是"加分题"，是"必修题"。
4. **遇到不懂的，先往下读两页**：有时候答案就在下一页。
5. **如果还是不懂，回头重读**：C 语言的很多概念需要反复咀嚼。

## 五、对编程零基础读者的特别说明

如果你以前从没写过代码，这一节专门为你写。

### 5.1 编程是什么？

编程就是**写一份"说明书"，告诉计算机按什么步骤做什么事**。

计算机很"笨"——它不会猜你的意思，只会严格按你写的代码执行。代码就像菜谱：

```
菜谱（人类语言）              代码（C 语言）
------------------            -----------------
把鸡蛋打散                     int eggs = 3;
倒油进锅                      heatPan();
油热后倒入蛋液                  while (!cooked) {
翻炒至金黄色                        stir();
盐少许                             }
盛盘                          serve();
```

区别是：人看菜谱会自动补全（"少许"大概是多少年经验来判断），计算机不会。你必须告诉它每一个细节。

### 5.2 这本书用了很多比喻

你会看到很多"门牌号"（内存地址）、"仓库 vs 桌面"（硬盘 vs 内存）、"翻译官"（编译器）这样的比喻。这不是为了凑字数—— C 语言的很多概念高度抽象，不借助比喻，几乎不可能在脑海中"看到"它们。

每个比喻都对应一个精确的技术概念，文中会标注出来。

## 不对怎么办？

**常见困惑 1**："我还是不明白为什么 C 能写操作系统而 Python 不能。"

解答：操作系统需要直接操控硬件——比如"把第 0x1000 号内存地址改成 0xFF"。C 语言允许你这么做，Python 不允许（为了保护你不犯错）。你可以把操作系统想成房子的地基，必须用最牢靠的钢筋混凝土（C），不能用泡沫板（Python）。

**常见困惑 2**："这么多术语我记不住。"

解答：不要死记硬背。术语会在后续章节反复出现，用着用着就记住了。你也可以在附录 F 随时查阅。现在只需要有个印象即可。

**常见困惑 3**："第零篇还没写代码，我是不是在浪费时间？"

解答：想象你要学开车。有些驾校第一天就让你上路开，结果你连刹车和油门都分不清。有些驾校先让你在教室里学交通规则、认识仪表盘——然后再上路。本书选第二种。第零篇的每一章，后面写代码时你都会感激它。
