## 一、秒杀场景分析

### 1.1 一个典型的秒杀长什么样？

某电商平台宣布：今晚 8 点，iPhone 15 限量 100 台，五折秒杀。

- 8:00:00.000 —— 秒杀开始
- 8:00:00.001 —— 瞬间涌入 50 万并发请求
- 8:00:00.050 —— 库存从 100 变成 0
- 8:00:00.100 —— 还有 49.9 万请求在排队，但它们已经抢不到了

### 1.2 秒杀和普通下单的区别

| 维度 | 普通下单 | 秒杀 |
|------|---------|------|
| 并发量 | 平稳，每秒几百到几千 | 瞬时爆发，每秒几十万 |
| 库存 | 充足，很少售罄 | 极度稀缺，100 件对 50 万人 |
| 业务逻辑 | 复杂（购物车→下单→支付→物流） | 极简（抢到→下单→支付） |
| 数据库压力 | 可控 | 瞬间打满 |
| 一致性要求 | 强一致 | 最终一致（库存扣减必须强一致） |

### 1.3 秒杀的三大核心挑战

| 挑战 | 描述 | 如果没做好，会发生什么 |
|------|------|----------------------|
| **超卖** | 100 件货卖出 101 件 | 用户投诉、平台赔钱、上新闻 |
| **高并发** | 50 万 QPS 同时涌入 | 数据库连接池耗尽、服务雪崩 |
| **数据库压力** | 瞬间几十万次写操作 | 主库 CPU 100%、从库延迟飙升 |

## 二、方案演进：从错误到正确到高性能

这是本章最核心的部分。我们用一个真实的库存扣减场景，一步步展示三个方案的设计思路和问题。

### 2.1 建表

```sql
CREATE DATABASE IF NOT EXISTS seckill
    DEFAULT CHARACTER SET utf8mb4
    DEFAULT COLLATE utf8mb4_unicode_ci;

USE seckill;

-- 秒杀商品表（简化版，只关注核心字段）
CREATE TABLE seckill_products (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    original_price DECIMAL(12,2) NOT NULL,
    seckill_price DECIMAL(12,2) NOT NULL,
    stock INT UNSIGNED NOT NULL COMMENT '剩余库存',
    total_stock INT UNSIGNED NOT NULL COMMENT '总库存',
    start_time DATETIME NOT NULL,
    end_time DATETIME NOT NULL,
    status TINYINT NOT NULL DEFAULT 1 COMMENT '1:未开始 2:进行中 3:已结束',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_status_time (status, start_time, end_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 秒杀订单表
CREATE TABLE seckill_orders (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    order_no VARCHAR(32) NOT NULL UNIQUE,
    user_id BIGINT UNSIGNED NOT NULL,
    product_id BIGINT UNSIGNED NOT NULL,
    seckill_price DECIMAL(12,2) NOT NULL COMMENT '秒杀价快照',

    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_product (user_id, product_id) COMMENT '防止重复秒杀',
    INDEX idx_product_id (product_id),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.2 方案一：直接 UPDATE（❌ 会超卖）

**最直观的想法：** 用户请求来了，直接扣库存。

```sql
-- 方案一：没有 WHERE 条件检查库存
UPDATE seckill_products SET stock = stock - 1 WHERE id = 1;
```

**问题出在哪？**

假设库存还剩 1 件。两个请求同时执行上述 SQL：

```
时间线：
请求 A: 读到 stock = 1
请求 B: 读到 stock = 1
请求 A: UPDATE stock = 1 - 1 = 0  ← 成功
请求 B: UPDATE stock = 0 - 1 = -1 ← 也成功了！超卖！
```

虽然 InnoDB 的行锁保证了两个 UPDATE 是串行执行的，但**没有 WHERE 条件来阻止库存为负的情况**。即使加了 `WHERE stock > 0`：

```sql
-- 方案一改进：加 WHERE 条件
UPDATE seckill_products SET stock = stock - 1 WHERE id = 1 AND stock > 0;
```

这个 SQL 在 MySQL 层面是安全的——InnoDB 行锁 + WHERE 条件保证了不会减到负数。但问题是：**在高并发下，大量请求会排队等待行锁**，导致响应时间急剧增加（见方案二的性能分析）。

**结论：** 方案一在低并发下可以工作，但性能极差，不适合秒杀。

---

### 2.3 方案二：事务 + 行锁 + WHERE 条件（✅ 防超卖，但性能差）

**思路：** 用事务包装，`SELECT ... FOR UPDATE` 显式加行锁，`WHERE stock > 0` 防超卖。

```sql
START TRANSACTION;

-- 步骤 1：锁定库存行并检查
SELECT stock FROM seckill_products WHERE id = 1 FOR UPDATE;
-- 如果 stock > 0，继续

-- 步骤 2：扣库存
UPDATE seckill_products SET stock = stock - 1 WHERE id = 1 AND stock > 0;

-- 步骤 3：检查 ROW_COUNT()
-- SELECT ROW_COUNT(); → 1 表示成功，0 表示库存不足

-- 步骤 4：创建订单
INSERT INTO seckill_orders (order_no, user_id, product_id, seckill_price)
VALUES (CONCAT('SK', UNIX_TIMESTAMP(), LPAD(1, 8, '0')), 100, 1, 3499.00);

COMMIT;
```

**这个方案能防超卖吗？** 能。

**为什么？** `SELECT ... FOR UPDATE` 对 `id=1` 这一行加了排他锁（X Lock）*此术语需进附录E*。其他事务在读取同一行时会被阻塞，直到当前事务提交。所以任何时候只有一个事务能操作这行数据，加上 `WHERE stock > 0` 的检查，绝对不会超卖。

**那问题在哪？性能。**

MySQL 的行锁是在 InnoDB 层实现的，每次加锁/解锁都有开销。50 万个请求排队抢同一行的锁，就像 50 万人排一个窗口——不管窗口处理多快，排队本身就耗尽了资源。

**实测数据（模拟）：**
- 100 并发请求，方案二耗时约 2-3 秒
- 1000 并发请求，方案二耗时约 20-30 秒
- 10000 并发请求，方案二直接超时（`Lock wait timeout`）

**结论：** 方案二正确但不高效。适合低并发（< 100 QPS），不适合秒杀。

### 2.4 方案三：Redis 预减库存 + 异步下单（✅ 防超卖 + 高性能）

**核心思路：** 把库存扣减从 MySQL 搬到 Redis。

Redis 是单线程处理命令的，这意味着 `DECR` 操作天然原子——不需要加锁，没有锁竞争。Redis 的吞吐量是每秒 10 万+，而 MySQL 行锁的吞吐量只有几千。

**架构图：**

```
用户请求
  │
  ▼
┌─────────────────────────────────┐
│  1. Redis DECR stock           │  ← 原子操作，无需加锁
│     if stock < 0 → 拒绝        │
│     if stock >= 0 → 继续        │
└────────────┬────────────────────┘
             │
             ▼
┌─────────────────────────────────┐
│  2. 发送消息到消息队列          │  ← 削峰：把瞬间的 50 万请求
│     （Kafka/RabbitMQ）          │     变成稳定的消费速度
└────────────┬────────────────────┘
             │
             ▼
┌─────────────────────────────────┐
│  3. 消费者：MySQL 事务下单      │  ← 以可控速度（如每秒 1000）
│     START TRANSACTION          │     消费消息，创建订单
│     INSERT INTO orders         │
│     COMMIT                     │
└─────────────────────────────────┘
```

### 2.5 Redis 预减库存实现

```bash
# ============================================================
# 秒杀开始前，将库存同步到 Redis
# ============================================================
redis-cli> SET seckill:stock:1 100
OK

# ============================================================
# 秒杀进行中，每次请求执行 DECR
# ============================================================
redis-cli> DECR seckill:stock:1
(integer) 99     # 库存从 100 变成 99，返回值 >= 0，允许下单

redis-cli> DECR seckill:stock:1
(integer) -1     # 库存已耗尽，返回值 < 0，拒绝请求

# ============================================================
# 用 Lua 脚本保证原子性（DECR + 判断 + 记录用户）
# ============================================================
```

**Lua 脚本：原子化的秒杀逻辑**

为什么需要 Lua 脚本？因为 Redis 的单个命令是原子的，但**多个命令组合不是原子的**。秒杀需要同时做三件事：
1. 检查库存（DECR）
2. 记录已抢到的用户（防重复秒杀）
3. 返回结果

这三个操作必须原子执行，否则可能出现"库存扣了但没记录用户"的情况。

```lua
-- seckill.lua
-- KEYS[1]: 库存 key (seckill:stock:1)
-- KEYS[2]: 已抢用户集合 key (seckill:users:1)
-- ARGV[1]: 用户 ID

local stock = redis.call('DECR', KEYS[1])

if stock < 0 then
    -- 库存不足，恢复库存（因为 DECR 已经减了）
    redis.call('INCR', KEYS[1])
    return -1  -- 表示库存不足
end

-- 检查用户是否已抢过（防重复秒杀）
local exists = redis.call('SISMEMBER', KEYS[2], ARGV[1])
if exists == 1 then
    -- 用户已抢过，恢复库存
    redis.call('INCR', KEYS[1])
    return -2  -- 表示重复秒杀
end

-- 记录用户已抢
redis.call('SADD', KEYS[2], ARGV[1])
return stock  -- 返回剩余库存
```

执行 Lua 脚本：

```bash
redis-cli --eval seckill.lua seckill:stock:1 seckill:users:1 , 100
# 返回: 99（剩余库存）或 -1（库存不足）或 -2（重复秒杀）
```

### 2.6 MySQL 异步下单

Redis 扣库存成功后，发送消息到消息队列，由消费者异步写入 MySQL：

```sql
-- 消费者执行的 SQL（在事务中）
START TRANSACTION;

-- 插入订单
INSERT INTO seckill_orders (order_no, user_id, product_id, seckill_price)
VALUES ('SK2024060515300000001', 100, 1, 3499.00);

-- 同步更新 MySQL 库存（用于对账，不是用于扣库存）
UPDATE seckill_products SET stock = stock - 1 WHERE id = 1 AND stock > 0;

COMMIT;
```

**注意：** 这里 UPDATE 的目的不是扣库存（库存已经在 Redis 扣过了），而是**保持 MySQL 和 Redis 的最终一致性**。如果 Redis 挂了，可以从 MySQL 恢复库存数据。

### 2.7 三种方案对比

| 维度 | 方案一（直接 UPDATE） | 方案二（事务+行锁） | 方案三（Redis+异步） |
|------|---------------------|-------------------|---------------------|
| **防超卖** | ❌ 可能超卖 | ✅ 不超卖 | ✅ 不超卖 |
| **并发性能** | 极差（所有请求排队等行锁） | 差（行锁串行化） | 好（Redis 单线程原子操作） |
| **QPS 上限** | ~500 | ~2000 | ~100000+ |
| **实现复杂度** | 低 | 低 | 中高 |
| **数据一致性** | 弱 | 强 | 最终一致 |
| **适用场景** | 不能用于生产 | 低并发秒杀 | 高并发秒杀（生产环境） |

## 三、常见错误分析

### 3.1 错误一：先查库存再扣库存（非原子操作）

这是最常见的错误，99% 的初学者都会犯：

```sql
-- ❌ 错误示范：先查再扣
-- 步骤 1：查库存
SELECT stock FROM seckill_products WHERE id = 1;
-- 假设返回 stock = 1

-- 步骤 2：在应用层判断
-- if stock > 0 → 扣库存
-- 问题：在步骤 1 和步骤 2 之间，另一个请求也读了 stock = 1

-- 步骤 3：扣库存
UPDATE seckill_products SET stock = stock - 1 WHERE id = 1;
```

**为什么是错的？** 因为 SELECT 和 UPDATE 之间有缝隙。就像两个人同时看到"还剩 1 间房"，然后同时下单——两个人都以为自己抢到了。

**正确做法：** 把判断逻辑放在 UPDATE 的 WHERE 条件中：

```sql
-- ✅ 正确：判断和扣减在一条 SQL 中完成
UPDATE seckill_products SET stock = stock - 1 WHERE id = 1 AND stock > 0;
-- 然后检查 ROW_COUNT()，如果为 0 说明库存不足
```

### 3.2 错误二：在 Redis 里扣了但 MySQL 没扣

**场景：** Redis DECR 成功了，但消息队列发送失败，或者消费者处理失败，导致 MySQL 里的订单没创建。

**后果：** 用户以为自己抢到了，但实际上没有订单记录。

**解决：**
1. 消息队列开启持久化和确认机制（ACK）
2. 消费者处理失败时重试（至少 3 次）
3. 定时对账：对比 Redis 已扣库存和 MySQL 已创建订单数量，不一致时告警

### 3.3 错误三：认为 Redis 是万能的

**错误想法：** "Redis 这么快，所有的库存都放 Redis 不就行了？"

**问题：** Redis 的数据在内存中，如果宕机了，内存数据可能丢失（取决于持久化策略）。RDB 快照可能丢失最近几分钟的数据，AOF 也可能丢失最后一秒的数据。

**正确做法：** Redis 负责"快速过滤"（99% 的请求在 Redis 层就被拒绝了），MySQL 负责"最终记录"（只有抢到的请求才写 MySQL）。两者各司其职。

## 四、分库分表方案

### 4.1 为什么秒杀需要分库分表？

秒杀系统的订单表增长极快。一场秒杀可能产生几十万条订单，日积月累，订单表会变得非常大。单表超过 500 万行后，查询性能会明显下降。

### 4.2 分片策略：按用户 ID 分片

```sql
-- 分片键：user_id
-- 分片算法：user_id % 4 → 路由到 4 个库之一

-- 库 0（user_id % 4 = 0）: seckill_db_0
-- 库 1（user_id % 4 = 1）: seckill_db_1
-- 库 2（user_id % 4 = 2）: seckill_db_2
-- 库 3（user_id % 4 = 3）: seckill_db_3
```

**为什么按用户 ID 分片而不是按订单 ID？**

因为秒杀系统最常用的查询是"我的订单"——按 `user_id` 查询。如果按 `user_id` 分片，这个查询只需要访问一个库。如果按 `order_id` 分片，查"我的订单"需要遍历所有库。

### 4.3 分片后的查询

```sql
-- 查询用户 100 的订单（user_id=100，100%4=0，路由到库 0）
-- 直接去 seckill_db_0 查询
SELECT * FROM seckill_orders WHERE user_id = 100;

-- 查询某场秒杀的所有订单（需要跨库查询）
-- 分别查 seckill_db_0、seckill_db_1、seckill_db_2、seckill_db_3
-- 然后在应用层合并结果
```

### 4.4 使用 ShardingSphere 简化分片

```yaml
# ShardingSphere 配置示例（简化版）
rules:
  - !SHARDING
    tables:
      seckill_orders:
        actualDataNodes: seckill_db_${0..3}.seckill_orders
        databaseStrategy:
          standard:
            shardingColumn: user_id
            shardingAlgorithmName: database_inline
        shardingAlgorithms:
          database_inline:
            type: INLINE
            props:
              algorithm-expression: seckill_db_${user_id % 4}
```

```sql
CREATE DATABASE IF NOT EXISTS seckill
    DEFAULT CHARACTER SET utf8mb4
    DEFAULT COLLATE utf8mb4_unicode_ci;

USE seckill;

CREATE TABLE seckill_products (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    original_price DECIMAL(12,2) NOT NULL,
    seckill_price DECIMAL(12,2) NOT NULL,
    stock INT UNSIGNED NOT NULL,
    total_stock INT UNSIGNED NOT NULL,
    start_time DATETIME NOT NULL,
    end_time DATETIME NOT NULL,
    status TINYINT NOT NULL DEFAULT 1,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_status_time (status, start_time, end_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE seckill_orders (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    order_no VARCHAR(32) NOT NULL UNIQUE,
    user_id BIGINT UNSIGNED NOT NULL,
    product_id BIGINT UNSIGNED NOT NULL,
    seckill_price DECIMAL(12,2) NOT NULL,
    status TINYINT NOT NULL DEFAULT 1,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_product (user_id, product_id),
    INDEX idx_product_id (product_id),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

```lua
-- seckill.lua
local stock = redis.call('DECR', KEYS[1])

if stock < 0 then
    redis.call('INCR', KEYS[1])
    return -1  -- 库存不足
end

local exists = redis.call('SISMEMBER', KEYS[2], ARGV[1])
if exists == 1 then
    redis.call('INCR', KEYS[1])
    return -2  -- 重复秒杀
end

redis.call('SADD', KEYS[2], ARGV[1])
return stock  -- 剩余库存
```
