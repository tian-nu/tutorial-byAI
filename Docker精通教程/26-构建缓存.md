## 我在做什么？

你改了一行代码，运行 `docker build`，结果等了 2 分钟——Docker 把 npm install 重新跑了一遍。明明你只改了 `app.js`，为什么 Docker 要重新安装所有依赖？难道它不知道依赖没变吗？

Docker 其实知道。它有一个**层缓存** *此术语见附录C* 机制，但能不能用上，取决于你 Dockerfile 里**指令的排列顺序**。这一章教你让 Docker 的缓存为你工作，把"改一行代码等 2 分钟"变成"改一行代码等 3 秒"。

学完这一章，你能：
- 理解层缓存的工作原理（什么时候命中，什么时候失效）
- 把依赖安装和代码复制分开，让缓存覆盖 90% 的构建步骤
- 用 `.dockerignore` 排除无用文件，加速 COPY 层
- 在需要时用 `--no-cache` 强制完整构建

## 一、层缓存原理回顾

### 每条指令一层

你在第 10 章学过：Dockerfile 里每条指令（`FROM`、`RUN`、`COPY`）都会生成一个镜像层。当时你可能觉得这只是"概念上的分层"，但层还有一个核心特性：**缓存**。

```dockerfile
FROM node:20-alpine
RUN npm install -g npm@10
COPY package.json .
RUN npm install
COPY . .
CMD ["node", "server.js"]
```

Docker 构建时，会为每一条指令计算一个**哈希值**（基于指令内容和 COPY 的文件内容）。下一次构建时，如果某条指令的哈希值没变，Docker 就直接复用缓存中的那一层，跳过这条指令。

### 缓存失效的连锁反应

关键规则：**一旦某一层的缓存失效，它之后的所有层都失效。**

```
FROM node:20-alpine           ← 层 1：缓存命中 ✅
RUN npm install -g npm@10     ← 层 2：缓存命中 ✅
COPY package.json .           ← 层 3：缓存命中 ✅
RUN npm install               ← 层 4：缓存命中 ✅
COPY . .                      ← 层 5：失效！❌（源码变了）
CMD ["node", "server.js"]     ← 层 6：也失效！❌（连锁）
```

`COPY . .` 复制了所有源码，改一行代码就导致这一层失效。它失效后，后面的 `CMD` 也失效——虽然 CMD 内容没变。

这就是"改一行代码，npm install 重新跑"的根源，对吗？**不对。** 在这个例子里，npm install 在 COPY 源码之前，所以 npm install 不会重新跑。那问题出在哪？

### 真正的问题：错误的指令顺序

很多人这样写 Dockerfile：

```dockerfile
# ❌ 常见错误：COPY . . 在前，npm install 在后
FROM node:20-alpine
WORKDIR /app
COPY . .                    # 复制所有文件（包括源码）
RUN npm install             # 安装依赖
CMD ["node", "server.js"]
```

```
COPY . .                    ← 失效！❌（源码变了）
RUN npm install             ← 也失效！❌（连锁）
CMD ["node", "server.js"]   ← 也失效！❌（连锁）
```

`COPY . .` 复制了所有文件——`package.json`、`server.js`、`src/` 下所有文件。改一行 `server.js`，`COPY . .` 就失效，然后 `RUN npm install` 也没了缓存，重新安装 500 个依赖包。

> **想多一点**：Docker 的缓存机制很像一个**流水线工厂**。每条指令是一个工位，每个工位产出一个半成品（层）。如果第 3 个工位的输入没变（哈希没变），它可以直接用上次的半成品，跳过工作。但如果第 3 个工位的输入变了，不仅第 3 个工位要重新做，第 4、5、6...所有后面的工位都要重新做——因为它们的输入变了。这就是"连锁失效"。

## 二、优化策略：依赖先装，代码后拷

### 黄金法则

> **把不常变的放在前面，经常变的放在后面。**

- 依赖文件（`package.json`、`go.mod`、`pom.xml`）：很少变
- 源码文件（`.js`、`.go`、`.java`）：经常变

### Node.js 优化前后对比

**❌ 优化前（错误顺序）：**

```dockerfile
FROM node:20-alpine
WORKDIR /app
COPY . .                    # 复制所有 → 源码一变就失效
RUN npm install             # 连锁失效 → 每次都重装
CMD ["node", "server.js"]
```

```
构建时间线：
  COPY . .          → 每次都重新复制（源码变了）
  RUN npm install   → 每次都重新安装 500 个包 → 2 分钟
  CMD               → 每次都重新创建
```

**✅ 优化后（正确顺序）：**

```dockerfile
FROM node:20-alpine
WORKDIR /app
# 第一步：先只复制依赖描述文件
COPY package.json package-lock.json ./
# 第二步：安装依赖（只在 package.json 变时才重新跑）
RUN npm ci
# 第三步：最后复制源码
COPY . .
CMD ["node", "server.js"]
```

```
构建时间线：
  COPY package*.json ./   → 缓存命中 ✅（package.json 没变）
  RUN npm ci              → 缓存命中 ✅（依赖没变）
  COPY . .                → 失效 ❌（源码变了）
  CMD                     → 失效 ❌（连锁，但几乎不耗时）
```

效果：改一行 `server.js` 后重新构建，`npm ci` 那层显示 `Using cache`，构建时间从 2 分钟降到 3 秒。

### 为什么用 `npm ci` 而不是 `npm install`？

- `npm install` 会检查 `package.json` 和 `package-lock.json` 是否一致，不一致就更新 `package-lock.json`。在 Docker 构建里，这可能导致不确定的行为。
- `npm ci`（clean install）严格按照 `package-lock.json` 安装，不修改任何文件，速度更快，行为确定。**在 CI/CD 和 Docker 构建中，永远用 `npm ci`。**

## 三、实战：Node.js 应用缓存优化完整演示

### 准备项目

```bash
mkdir cache-demo && cd cache-demo
```

```json
// package.json
{
  "name": "cache-demo",
  "version": "1.0.0",
  "dependencies": {
    "express": "^4.18.2"
  }
}
```

```javascript
// server.js
const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.send('Hello from cache-optimized Docker!');
});

app.listen(3000, () => {
  console.log('Server running on http://localhost:3000');
});
```

生成 `package-lock.json`：

```bash
npm install
```

### 步骤 1：错误的 Dockerfile（对比用）

```dockerfile
# Dockerfile.bad
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm install
CMD ["node", "server.js"]
```

```bash
docker build -f Dockerfile.bad -t cache-demo-bad .
```

```bash
# 第一次构建：正常
# 第二次构建（不改任何文件）：
docker build -f Dockerfile.bad -t cache-demo-bad .
# 应该看到 "Using cache" 在每一层

# 第三次构建（改一行 server.js）：
echo "// test comment" >> server.js
docker build -f Dockerfile.bad -t cache-demo-bad .
# 输出：
# => CACHED [1/4] FROM node:20-alpine
# => CACHED [2/4] WORKDIR /app
# => [3/4] COPY . .              ← 失效！
# => [4/4] RUN npm install       ← 也失效！重新安装...
```

npm install 那层失效了，重新安装所有依赖。

### 步骤 2：优化后的 Dockerfile

```dockerfile
# Dockerfile
FROM node:20-alpine
WORKDIR /app
# 先复制依赖文件
COPY package.json package-lock.json ./
# 安装依赖
RUN npm ci
# 最后复制源码
COPY . .
CMD ["node", "server.js"]
```

```bash
docker build -t cache-demo .
```

### 步骤 3：验证缓存效果

```bash
# 第一次构建（完整构建）
docker build -t cache-demo .

# 改一行源码
echo "// another comment" >> server.js

# 第二次构建
docker build -t cache-demo .
```

输出中你会看到：

```
[+] Building 1.2s (9/9) FINISHED
 => [1/5] FROM docker.io/library/node:20-alpine         0.0s
 => CACHED [2/5] WORKDIR /app                           0.0s
 => CACHED [3/5] COPY package*.json ./                  0.0s
 => CACHED [4/5] RUN npm ci                             0.0s   ← 关键！用了缓存
 => [5/5] COPY . .                                      0.1s
 => exporting to image                                  0.5s
```

`RUN npm ci` 显示 `CACHED`——依赖没重新安装！构建只用了 1.2 秒。

### 步骤 4：什么时候 npm ci 会重新跑？

只有当你改了 `package.json`（添加/删除依赖）时：

```bash
# 添加一个新依赖
npm install lodash
# package.json 和 package-lock.json 都变了

docker build -t cache-demo .
# 输出：
# => [3/5] COPY package*.json ./              ← 失效！文件变了
# => [4/5] RUN npm ci                         ← 也失效！重新安装
# => [5/5] COPY . .                           ← 失效
```

这才是正确的行为：依赖变了，应该重新安装。但改源码时不应该重新安装依赖。

## 四、`.dockerignore`：不要让缓存做无用功

### 问题：COPY 复制了不该复制的文件

```dockerfile
COPY . .
```

这条命令把当前目录下**所有文件**都复制进了镜像。包括：
- `node_modules/`（几百 MB，本地调试用的，构建时会重新装）
- `.git/`（整个 Git 历史，镜像里完全不需要）
- `*.log`、`dist/`、`.env` 等

这些文件每次构建都可能不同（尤其是 `node_modules`），导致 `COPY . .` 层几乎总是失效——即使你只改了源码。

### 解决方案：`.dockerignore`

创建 `.dockerignore` 文件（和 `.gitignore` 语法相同）：

```bash
# .dockerignore
node_modules
.git
.gitignore
*.log
dist
.env
Dockerfile
.dockerignore
```

现在 `COPY . .` 只会复制必要的文件，排除了那些会"污染缓存"的大型目录。

### 效果验证

```bash
# 没有 .dockerignore 时
docker build -t test .
# Sending build context to Docker daemon  350MB
# （因为 node_modules 被包含在构建上下文中）

# 有 .dockerignore 后
docker build -t test .
# Sending build context to Docker daemon  2.5MB
# （排除了 node_modules 和 .git）
```

> **想多一点**：`.dockerignore` 有两个层面的好处。第一，它减少了"构建上下文"的大小，让 `docker build` 的第一步（打包上下文）更快。第二，它让 `COPY . .` 层更稳定——`node_modules` 不在上下文里，就不会因为本地 `node_modules` 的变化导致 `COPY . .` 缓存失效。你在第 10 章学过构建上下文，现在你看到了它的实际运用：控制哪些文件进入上下文，就是在控制哪些文件影响缓存。

## 五、其他语言的缓存优化模式

### Go 项目

Go 的依赖由 `go.mod` 和 `go.sum` 定义。思路一样：先复制依赖文件，下载依赖，再复制源码。

```dockerfile
FROM golang:1.22-alpine
WORKDIR /app
# 先复制依赖文件
COPY go.mod go.sum ./
# 下载依赖（缓存层）
RUN go mod download
# 最后复制源码
COPY . .
# 编译
RUN go build -o server .
CMD ["./server"]
```

效果：改 `.go` 源码不会触发 `go mod download` 重新下载依赖。

### Python 项目

```dockerfile
FROM python:3.12-slim
WORKDIR /app
# 先复制依赖文件
COPY requirements.txt .
# 安装依赖（缓存层）
RUN pip install --no-cache-dir -r requirements.txt
# 最后复制源码
COPY . .
CMD ["python", "app.py"]
```

### Java (Maven) 项目

```dockerfile
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
# 先复制 pom.xml（依赖描述）
COPY pom.xml .
# 下载所有依赖（缓存层）
RUN mvn dependency:go-offline
# 最后复制源码
COPY src ./src
# 编译
RUN mvn package -DskipTests
```

效果：改 `.java` 源码不会触发 Maven 重新下载几百 MB 的依赖 jar 包。

## 六、`--no-cache`：强制从头构建

有时候你需要**不使用缓存**，从零开始构建。比如：
- 基础镜像有安全更新，你想拉最新的
- 你怀疑缓存出了问题（构建结果不一致）
- `RUN apt-get update` 这类命令需要每次都拉最新包列表

```bash
docker build --no-cache -t myapp .
```

输出中所有层都标记为 `[+]` 而不是 `CACHED`，每一步都重新执行。

### 什么时候用 `--no-cache`？

| 场景 | 是否用 `--no-cache` |
|------|-------------------|
| 日常开发，改了一行代码 | ❌ 不用。缓存让你 3 秒构建完 |
| 改的是 `package.json` | ❌ 不用。依赖层会自动失效 |
| 改的是基础镜像（`FROM` 行） | 一般来说不用，Docker 会检测到 FROM 变了 |
| 怀疑构建结果不对 | ✅ 用。排除缓存问题 |
| 发布前最终构建 | ✅ 建议用。确保用的是最新的基础镜像和安全补丁 |
| `RUN apt-get update` 后装软件 | ✅ 建议 `--no-cache` 或把 `update` 和 `install` 放同一行 |

## 八、不对怎么办？

### 常见错误 1：依赖安装放在源码复制之后

```dockerfile
# ❌ 最典型的错误
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm install
CMD ["node", "server.js"]
```

**症状**：改任何源码，npm install 都重新跑。

✅ 修正：把依赖文件的 COPY 和 npm install 移到源码 COPY 之前。

### 常见错误 2：`.dockerignore` 里忘了排除 `node_modules`

```dockerfile
COPY . .
```

```bash
docker build .
# Sending build context to Docker daemon  500MB
# 构建卡在 "Sending build context" 很久
```

**原因**：本地 `node_modules` 目录很大，被 COPY 进了构建上下文。

✅ 修正：创建 `.dockerignore` 并加入 `node_modules`。

### 常见错误 3：`npm ci` 报错，但本地 `npm install` 没问题

```bash
# npm ci 报错
# npm ERR! cipm can only install packages with an existing package-lock.json
```

**原因**：`npm ci` 要求必须有 `package-lock.json`。如果你本地只用了 `npm install` 但没生成 lock 文件，或者 lock 文件和 `package.json` 不一致，`npm ci` 就会报错。

✅ 修正：

```bash
# 确保 package-lock.json 存在且和 package.json 一致
rm -rf node_modules package-lock.json
npm install
# 然后确保 Dockerfile 里 COPY 了 package-lock.json
```

### 常见错误 4：`RUN apt-get update` 和 `apt-get install` 分开写

```dockerfile
# ❌ 错误
RUN apt-get update
RUN apt-get install -y nginx
```

**问题**：`apt-get update` 那层有缓存，可能用的是几周前的包列表。如果软件源更新了，`install` 那层可能因为找不到旧版本的包而失败。

✅ 修正：合并到一行：

```dockerfile
RUN apt-get update && apt-get install -y nginx && rm -rf /var/lib/apt/lists/*
```

一条 RUN 指令 = 一层。合并后，`update` 和 `install` 在同一层，要么都用缓存，要么都重新执行，不会出现"update 是旧的，install 是新的"这种不一致。

### 常见错误 5：把经常变的文件放在 `.dockerignore` 里，但 Dockerfile 需要它

```bash
# .dockerignore
package.json        # ❌ 错误！Dockerfile 里需要 COPY package.json
```

`.dockerignore` 排除了 `package.json`，导致 `COPY package.json .` 找不到文件。

✅ 修正：`.dockerignore` 只排除那些**不需要**进镜像的文件。构建需要的文件不要排除。

## 九、术语解释

| 术语 | 解释 |
|------|------|
| **层缓存** *此术语见附录C* | Docker 为每条指令生成的层会被缓存。下次构建时，如果指令内容和输入文件没变，Docker 直接复用缓存层，跳过执行 |
| **`--no-cache`** *此术语见附录C* | `docker build` 的参数，强制从头构建，不使用任何缓存 |
| **构建优化** *此术语见附录C* | 通过调整 Dockerfile 指令顺序，让缓存命中率最大化，从而加速构建 |
| **`npm ci`** | `npm clean-install`，严格按照 `package-lock.json` 安装，不修改 lock 文件，比 `npm install` 更快更确定 |
| **`.dockerignore`** | 和 `.gitignore` 语法相同，指定哪些文件不进入构建上下文，也不被 COPY 指令复制 |

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 层缓存原理 | 每条指令一层，哈希没变就复用缓存 | 一层失效，后面所有层都失效（连锁反应） |

| `.dockerignore` | 排除 `node_modules`、`.git`、`*.log` 等 | 减少上下文大小，稳定 COPY 层的缓存 |
| `--no-cache` | `docker build --no-cache -t myapp .` | 发布前最终构建建议使用 |
| 合并 RUN | `RUN apt-get update && apt-get install -y ...` | 避免 update 有缓存但 install 没缓存的矛盾 |

## 本篇最可能出错的地方及原因

### 1. 依赖安装和源码复制顺序写反

**这是最高频的错误。** 很多开发者直觉上把 `COPY . .` 放在最前面（"先把所有文件复制进去，然后装依赖"），结果改了源码就要重新装依赖。这就像把蔬菜先放进冰箱，再去找菜谱——每次换菜谱，冰箱里的蔬菜也要重新洗一遍。

**记忆口诀**：package.json 是"购物清单"，npm install 是"采购"，COPY . . 是"做饭"。先按清单采购，再做饭。清单没变就不需要重新采购。

### 2. 本地 `node_modules` 污染构建上下文

**原因**：没有 `.dockerignore`，或者 `.dockerignore` 里没写 `node_modules`。本地 `node_modules` 可能有几百 MB，不仅拖慢 `docker build` 的第一步，还导致 `COPY . .` 层经常失效。

**排查**：看 `docker build` 输出的第一行 `Sending build context to Docker daemon XXXMB`。如果 XXX 很大（几百 MB），说明缺少 `.dockerignore`。

### 3. `npm ci` 要求 `package-lock.json`，但没 COPY 进去

**原因**：Dockerfile 里只 COPY 了 `package.json`，没 COPY `package-lock.json`。`npm ci` 找不到 lock 文件就报错。

```dockerfile
# ❌ 只复制了 package.json
COPY package.json ./
RUN npm ci          # 报错：缺少 package-lock.json
```

✅ 修正：

```dockerfile
COPY package.json package-lock.json ./
RUN npm ci
```

### 4. 缓存"太有效"导致安全补丁没更新

**原因**：`FROM node:20-alpine` 被缓存了，几周前拉取的镜像没有最新的安全补丁。即使 `node:20-alpine` 官方镜像更新了，你的本地缓存还是旧的。

**解决**：定期用 `docker build --no-cache --pull` 构建。`--pull` 会强制拉取最新的基础镜像，即使本地已有缓存。

```bash
docker build --no-cache --pull -t myapp .
```

### 5. 多阶段构建中，第一阶段也要考虑缓存

**原因**：以为多阶段构建只有第二阶段重要，第一阶段随便写。但第一阶段也参与构建，第一阶段慢了，整个构建就慢了。

✅ 在多阶段构建的第一阶段也应用缓存优化：

```dockerfile
# ✅ 第一阶段也遵循"依赖先装，代码后拷"
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download       # 缓存依赖下载
COPY . .
RUN go build -o server .
```
