# 第10章 · UDP协议

> "你在群里发了一个红包，有人抢到了，有人没抢到——你不在乎谁没抢到，你只在乎发出去的那一瞬间快不快。这就是UDP。"

---

## 10.1 UDP是什么

UDP = **User Datagram Protocol** = 用户数据报协议

如果说TCP是一个极度负责的快递员，那UDP就是一个 **"发了就忘"的快递员**。你交给它一个包裹，它扔出去就跑，不回头看一眼——丢没丢？不管。到没到？不管。顺序对不对？不管。

### UDP的"四不管"

| 不管什么 | 后果 |
|----------|------|
| **不管丢包** | 数据可能消失在网络里，发送方不知道，接收方也不通知 |
| **不管顺序** | 先发的可能后到，后发的可能先到，甚至可能不到 |
| **不管重复** | 同一个数据包可能被路由器转发多次 |
| **不管拥堵** | 不管网络通不通畅，照发不误，发不出去就丢掉 |

听上去很糟糕对吧？但UDP有一个巨大的优势：**快，极快。**

没有三次握手，没有四次挥手，没有确认机制，没有重传逻辑，没有滑动窗口，没有拥塞控制——少了这么多东西，UDP的头部只有**8个字节**（TCP头部至少20个字节），处理起来极轻。

类比：TCP是**挂号信**（签字确认、追踪查件、丢件赔偿），UDP是**明信片**（写完了扔邮筒，爱到不到）。

## 10.2 TCP vs UDP 全方位对比

| 维度 | TCP | UDP |
|------|-----|-----|
| **连接方式** | 面向连接（先握手再传数据） | 无连接（直接发，不需要建立连接） |
| **可靠性** | 可靠，保证送达 | 不可靠，不保证送达 |
| **顺序** | 保证有序 | 不保证有序 |
| **速度** | 慢（握手、确认、重传有开销） | 快（没有这些开销） |
| **头部大小** | 20~60字节 | 8字节 |
| **流量控制** | 有（滑动窗口） | 没有 |
| **拥塞控制** | 有（慢启动、拥塞避免等） | 没有 |
| **适用场景** | 文件传输、网页、邮件、数据库 | 视频直播、语音通话、游戏、DNS查询 |
| **比喻** | 打电话（确认接通后才说话） | 发短信（发了就发了） |
| **英文助记** | **T**rustable **C**ourier **P**rotocol | **U**nreliable **D**ata **P**rotocol |

> 🤔 想多一点：那UDP是不是就完全没用？恰恰相反。现实世界中UDP无处不在。你看B站视频、打王者荣耀、用微信语音——底层的音视频数据全是UDP。丢一个视频帧？画面卡了一瞬间，马上又恢复了。但如果你用TCP传视频，每丢一个包都要停下来重传——那视频就卡死了。这就是**"宁丢勿等"**的哲学。

## 10.3 UDP的使用场景

### 10.3.1 视频直播与语音通话

你看直播时，主播的画面是一帧一帧传过来的。如果某一帧在传输中丢了——

- **TCP的做法**：暂停播放，等重传，等到了再继续。画面卡住，体验极差。
- **UDP的做法**：丢就丢了，跳过这一帧，继续播下一帧。你最多看到画面花了一下，但不会卡住。

对于实时性要求极高的音视频通话，**晚到的数据 = 没用的数据**。你宁愿收到99%的数据流畅播放，也不要收到100%的数据但卡成PPT。

### 10.3.2 在线游戏

你正在打王者荣耀。你的英雄向前走了一步，这个"走了一步"的指令用UDP发给了服务器。

如果用TCP：丢包了→重传→延迟从30ms飙到200ms→你的英雄卡在原地→你被对面拿了人头😫

用UDP：那个"走了一步"的指令丢了就丢了，下一个指令"往前走"已经在路上了。服务器看到的只是少了一个中间步骤——它用插值/预测算法补上就行。

游戏领域有一个说法：**"UDP is for fast-paced action, TCP is for reliability."**

### 10.3.3 DNS查询

还记得第7章的DNS吗？DNS查询默认用的就是UDP！

为什么？一个DNS查询包通常只有几十个字节。用TCP要三次握手再发请求，加起来3个包。用UDP一个包就搞定了——发出去，等回复，完事。如果回复没收到（丢了），客户端过几秒再发一次就行，代价几乎为零。

### 10.3.4 DHCP（自动获取IP地址）

当你连上WiFi时，你的设备需要获取一个IP地址。这个过程中用的是DHCP协议，而DHCP跑在UDP上。

为什么？因为此时你的设备**还没有IP地址**，没办法建立TCP连接！所以只能用UDP广播"有人在吗？请给我一个IP地址！"

### 10.3.5 物联网（IoT）传感器

一个温度传感器每10秒上报一次温度数据（比如"28.5°C"）。丢了一次数据？没关系，10秒后还有下一次。用TCP的话，每次上报都要建立连接，对传感器这种低功耗设备来说太奢侈了。

## 10.4 用Go体验UDP

```go
package main

import (
    "fmt"
    "net"
)

func main() {
    addr, _ := net.ResolveUDPAddr("udp", ":9999")
    conn, _ := net.ListenUDP("udp", addr)
    defer conn.Close()
    fmt.Println("UDP服务器在9999端口等待数据...")

    buf := make([]byte, 1024)
    for {
        n, clientAddr, _ := conn.ReadFromUDP(buf)
        fmt.Printf("收到来自 %s 的数据: %s\n", clientAddr, string(buf[:n]))

        conn.WriteToUDP([]byte("收到啦！\n"), clientAddr)
    }
}
```

注意和TCP代码的区别：
- 没有 `Listen` + `Accept`（没有连接概念，直接 `ListenUDP`）
- 读取数据时要获取发送方的地址 `ReadFromUDP`
- 回复时也要指定接收方的地址 `WriteToUDP`

这就是UDP的"无连接"——服务器不用欢迎谁，谁发数据过来它就处理谁。

## 10.5 QUIC：UDP的进化

你可能想说："UDP好是好，但如果我也想要可靠性和速度兼得呢？"

这就是 **QUIC**（Quick UDP Internet Connections）协议的使命——它**在UDP之上自己实现了一套类似TCP的可靠性机制**。HTTP/3就基于QUIC。

```
传统方案：        HTTP → TLS → TCP → IP
HTTP/3方案：      HTTP → QUIC(含TLS) → UDP → IP
```

QUIC把TLS加密内建在协议里，并且用UDP作为底层传输，绕过了TCP的队头阻塞问题（一个包丢了不会阻塞同一连接上的其他数据流）。这是后话，知道有这么个东西就行。

> 🤔 想多一点：TCP已经用了40多年（1981年的RFC 793），它设计时互联网还很单纯——谁也没想到40年后会有4K视频、实时游戏和5G网络。但TCP的问题是它被**焊死在内核里**了——想升级TCP？你得改操作系统内核，这几乎不可能大规模推广。QUIC的聪明之处在于：它跑在**用户空间**，在UDP之上实现。你可以像升级Chrome浏览器一样升级QUIC协议，而不需要等微软/苹果/Red Hat修改他们的操作系统。
