## 我在做什么？

容器在后台跑着，你想进去看看——检查配置文件、查看日志文件、或者临时装个工具调试。但容器是隔离的，你不能像操作本地文件一样直接访问容器内部。

这一章教你三种"进入容器"的方式：`docker exec`（最常用）、`docker attach`（慎用）、`docker cp`（拷贝文件）。学完之后，你就能在容器里像在本地一样自由操作。

学完这一章，你能：
- 用 `docker exec -it` 进入容器内部执行命令
- 用 `docker exec` 执行单条命令，不用进入交互模式
- 理解 `docker exec` 和 `docker attach` 的本质区别，避免误用
- 用 `docker cp` 在宿主机和容器之间复制文件
- 实战：进入 Nginx 容器修改默认页面

## 一、`docker exec`：进入容器的"正门"

`docker exec` *此术语见附录C* 是在正在运行的容器中**启动一个新进程**的命令。它不会影响容器的主进程，就像你在一间正在营业的餐厅里"临时借用一个灶台"——用完就走，餐厅照常营业。

### 基本用法

```bash
docker exec -it my-nginx bash
```

这行命令让你进入 `my-nginx` 容器内部，打开一个 bash shell。你会看到终端提示符变了：

```
root@abc123def456:/#
```

现在你就在容器内部了。你可以像操作一台 Linux 机器一样操作它：

```bash
# 查看文件
ls /etc/nginx/

# 编辑文件（如果容器里有编辑器）
vi /etc/nginx/nginx.conf

# 查看进程
ps aux

# 查看网络
cat /etc/hosts
```

> 注意：容器内的环境通常非常精简，很多工具（如 `vi`、`nano`、`curl`）可能没有安装。你只能使用容器镜像里自带的工具。

### 退出容器

```bash
exit
# 或按 Ctrl+D
```

退出后，容器**继续运行**（除非你退出的进程是容器的主进程）。`docker exec` 启动的是一个**额外的进程**，不是主进程，所以退出不影响容器。

### `-i` 和 `-t` 的含义（复习）

| 选项 | 全称 | 作用 | 不写会怎样 |
|------|------|------|-----------|
| `-i` | `--interactive` | 保持标准输入（STDIN）打开 | 你无法在容器里输入命令 |
| `-t` | `--tty` | 分配伪终端（pseudo-TTY） | 输出没有格式、没有颜色、没有提示符 |

和 `docker run -it` 一样，`-i` 和 `-t` 必须一起用。

### 执行单条命令（不进入交互模式）

```bash
# 查看 Nginx 配置文件
docker exec my-nginx cat /etc/nginx/nginx.conf

# 查看 Nginx 版本
docker exec my-nginx nginx -v

# 列出目录
docker exec my-nginx ls -la /usr/share/nginx/html/

# 检查网络连通性（如果容器里有 curl 或 wget）
docker exec my-nginx curl http://localhost
```

这种方式不需要 `-it`，因为你不打算和容器交互——只是执行一条命令，拿到输出就结束。

## 二、容器没有 bash 怎么办？

### 不是所有镜像都有 bash

Alpine Linux 镜像（如 `alpine:latest`、`nginx:alpine`）为了极致精简，只装了 `sh`（ash），没有 `bash`。

```bash
# ❌ 错误：Alpine 容器没有 bash
docker run -d --name alpine-test alpine:latest sleep 3600
docker exec -it alpine-test bash
# Error: executable file "bash" not found in $PATH

# ✅ 正确：用 sh
docker exec -it alpine-test sh
```

### 如何知道容器里有什么 shell？

```bash
# 方法 1：试试 bash，不行就试 sh
docker exec -it 容器名 bash || docker exec -it 容器名 sh

# 方法 2：查看镜像的 Dockerfile 或文档
# 一般规则：
# - Ubuntu/Debian 镜像 → bash
# - Alpine 镜像 → sh
# - CentOS/Fedora 镜像 → bash
# - 精简版镜像（slim/alpine） → 可能只有 sh
```

> 口诀：**先试 bash，失败换 sh。都不行，镜像可能连 shell 都没有（比如 `scratch` 镜像）。**

## 三、`docker exec` vs `docker attach`：一个关键区别

### `docker attach`：连到主进程

`docker attach` *此术语见附录C* 和 `docker exec` 完全不同——它不启动新进程，而是把你的终端**直接连接到容器的主进程（PID 1）**。

```bash
docker attach my-nginx
```

`docker attach` 把你的终端**连接到容器的 PID 1（主进程）**的标准输入/输出。

| | `docker exec` | `docker attach` |
|------|-------------|-------------|
| 连接对象 | 容器内**新启动的**进程 | 容器的**主进程（PID 1）** |
| 退出后容器怎样 | 容器继续运行 | ⚠️ 容器可能停止！ |
| 能开多个吗 | 能，可以同时开多个 | 只能连到主进程 |
| 适合场景 | 检查、调试、执行命令 | 查看主进程的实时输出 |

### `docker attach` 的危险之处

**场景：你 attach 到一个 Nginx 容器，然后按了 Ctrl+C。**

```bash
docker attach my-nginx
# 看到 Nginx 的日志输出
# 你想退出，按 Ctrl+C
# 容器停了！
```

**为什么容器停了？** 因为 `Ctrl+C` 发送 SIGINT 信号给主进程（Nginx），Nginx 收到信号后退出。主进程退出 → 容器停止。

**{{：如果你用 `docker exec -it my-nginx bash` 进入，然后按 Ctrl+C 或 `exit`，只退出你启动的 bash 进程，主进程不受影响。**}}

### `docker attach` 的正确用法：安全退出

如果你不幸 attach 了，想退出但**不想停止容器**，按 `Ctrl+P` 然后 `Ctrl+Q`（先按 P 再按 Q，不要松开 Ctrl）。

```bash
# 安全退出 attach（不停止容器）
Ctrl+P, Ctrl+Q
```

> 记住这个组合键，它可能救你一次。但我建议你**尽量不用 `docker attach`**，用 `docker exec` 和 `docker logs` 替代。

### 什么时候真的需要用 `docker attach`？

- 容器的主进程是你自己写的交互式程序，你想和它交互
- 你在调试容器启动时的错误，想看到主进程的实时输出

其余情况，**用 `docker exec` 和 `docker logs` 就够了**。

## 四、`docker cp`：在宿主机和容器之间搬运文件

`docker cp` *此术语见附录C* 让你在宿主机和容器之间复制文件，就像在两个文件夹之间拖拽文件一样。它不需要容器里有任何特殊服务——只要容器在运行（或已停止但未被删除），就能复制。

### 基本语法

```bash
# 从宿主机复制到容器
docker cp 宿主机路径 容器名:容器内路径

# 从容器复制到宿主机
docker cp 容器名:容器内路径 宿主机路径
```

### 从宿主机复制到容器

```bash
# 把本地文件复制到容器内
echo "<h1>Hello from Docker!</h1>" > index.html
docker cp ./index.html my-nginx:/usr/share/nginx/html/index.html

# 验证
docker exec my-nginx cat /usr/share/nginx/html/index.html
# 输出：<h1>Hello from Docker!</h1>
```

### 从容器复制到宿主机

```bash
# 把容器内的 Nginx 配置文件复制出来
docker cp my-nginx:/etc/nginx/nginx.conf ./nginx.conf.bak

# 验证
cat ./nginx.conf.bak
```

### 复制整个目录

```bash
# 复制容器内的整个目录到宿主机
docker cp my-nginx:/etc/nginx ./nginx-config-backup

# 复制宿主机的目录到容器
docker cp ./my-html-files my-nginx:/usr/share/nginx/html/
```

### `docker cp` 的注意事项

1. **路径必须是绝对路径或相对于当前目录的路径**，不能用 `~` 这种 shell 简写。
2. **复制文件时，目标路径的父目录必须存在**。如果目标路径以 `/` 结尾，Docker 会把它当作目录。
3. **权限问题**：复制到容器的文件，权限由容器内的用户决定（通常是 root）。

## 五、实战：修改 Nginx 的默认页面

现在，你进入一个 Nginx 容器，修改它的默认欢迎页面：

### 步骤 1：启动 Nginx 容器

```bash
docker run -d --name edit-nginx -p 8080:80 nginx:1.27.4
```

### 步骤 2：确认默认页面

```bash
curl http://localhost:8080
# 看到 Welcome to nginx! 的 HTML
```

### 步骤 3：进入容器，查看默认页面文件

```bash
docker exec -it edit-nginx bash
# 在容器内执行：
cat /usr/share/nginx/html/index.html
```

你会看到默认的 Nginx 欢迎页 HTML。

### 步骤 4：修改默认页面

在容器内，覆盖 `index.html`：

```bash
# 在容器内执行（用 echo 写一个新页面）
echo "<html><body><h1>Hello from Docker! This page was modified inside the container.</h1></body></html>" > /usr/share/nginx/html/index.html
exit
```

### 步骤 5：验证修改

```bash
curl http://localhost:8080
# 看到你修改后的页面内容
```

### 步骤 6：用 `docker cp` 方式也能实现

```bash
# 在宿主机上创建文件
echo "<html><body><h1>Modified via docker cp</h1></body></html>" > custom-index.html

# 复制到容器
docker cp ./custom-index.html edit-nginx:/usr/share/nginx/html/index.html

# 验证
curl http://localhost:8080
```

### 步骤 7：清理

```bash
docker stop edit-nginx
docker rm edit-nginx
```

## 我做得对不对？

### 验证方法

请按顺序执行以下操作，**全部成功**才算通过：

```bash
# 1. 启动 Nginx 容器
docker run -d --name exec-test -p 8080:80 nginx:1.27.4

# 2. 用 docker exec 执行单条命令，查看 Nginx 版本
docker exec exec-test nginx -v
# 预期输出：nginx version: nginx/1.27.4

# 3. 进入容器内部
docker exec -it exec-test bash

# 4. 在容器内执行（逐行输入）：
ls /etc/nginx/
cat /etc/nginx/nginx.conf | head -5
exit

# 5. 用 docker cp 把容器的配置文件复制出来
docker cp exec-test:/etc/nginx/nginx.conf ./nginx.conf.copy

# 6. 确认文件已复制到本地
ls -la ./nginx.conf.copy

# 7. 清理
docker stop exec-test && docker rm exec-test
rm ./nginx.conf.copy
```

**预期结果**：
- 步骤 2：输出 Nginx 版本号
- 步骤 4：能正常进入容器，`ls` 能看到配置文件列表
- 步骤 5-6：`nginx.conf.copy` 文件在本地创建成功

## 不对怎么办？

### 常见错误 1：容器没有 bash

```bash
docker exec -it my-alpine bash
# Error: executable file "bash" not found in $PATH
```

**原因**：Alpine 等精简镜像没有 bash，只有 sh。

**解决**：

```bash
docker exec -it my-alpine sh
```

### 常见错误 2：`docker attach` 退出后容器也停了

```bash
docker attach my-nginx
# 看到日志输出
# 按 Ctrl+C
# 容器停止了！
```

**原因**：`Ctrl+C` 发送 SIGINT 给主进程，主进程退出 → 容器停止。

**解决**：

- **安全退出**：按 `Ctrl+P` 然后 `Ctrl+Q`（不要松开 Ctrl）
- **更好的做法**：不要用 `docker attach`，用 `docker exec` 和 `docker logs` 替代

### 常见错误 3：`docker cp` 路径不对

❌ 错误做法：

```bash
# 目标路径不以 / 结尾，docker 不知道你要复制到目录还是文件
docker cp ./index.html my-nginx:/usr/share/nginx/html
# 如果 /usr/share/nginx/html 不存在，会报错
```

✅ 正确做法：

```bash
# 明确指定完整路径（包括文件名）
docker cp ./index.html my-nginx:/usr/share/nginx/html/index.html

# 或者先确认目录存在
docker exec my-nginx ls /usr/share/nginx/html/
docker cp ./index.html my-nginx:/usr/share/nginx/html/
```

### 常见错误 4：进入了容器，但什么工具都没有

**现象**：进入容器后想用 `ps`、`netstat`、`curl` 等工具，提示 `command not found`。

**原因**：容器镜像为了精简，去掉了大部分非必要的工具。

**解决**：

```bash
# 如果是 Ubuntu/Debian 镜像，可以临时安装
apt update && apt install -y curl procps net-tools

# 但更推荐的做法是：在宿主机上完成诊断
# 宿主机有全部工具，docker exec 只是执行命令
docker exec my-nginx cat /proc/1/status
docker top my-nginx
docker inspect my-nginx
```

> 容器不是虚拟机，不要试图让它"完整"。用宿主机的工具来诊断容器，而不是在容器里装工具。

### 常见错误 5：`docker exec` 不需要 `-it` 时也加上了

```bash
# 执行单条命令时，不需要 -it
docker exec my-nginx nginx -v
# 可以正常运行

# 加了 -it 也能跑，但没必要
docker exec -it my-nginx nginx -v
```

`-it` 只在需要**交互**时使用（比如进入 bash shell）。执行单条命令时，加不加 `-it` 不影响结果。

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 进入容器内部 | `docker exec -it 容器名 bash` | `-i` 保持输入 + `-t` 分配终端，必须一起用 |
| 执行单条命令 | `docker exec 容器名 命令` | 不需要 `-it`，直接拿到输出 |
| 容器没有 bash | 用 `sh` 替代 `bash` | Alpine 等精简镜像只有 sh |
| `docker exec` vs `docker attach` | exec 启动新进程，attach 连主进程 | attach 退出会停止容器（按 Ctrl+P+Q 安全退出） |
| 宿主机与容器间复制文件 | `docker cp 源路径 目标路径` | 宿主机 → 容器：`docker cp ./file 容器:/path/`；容器 → 宿主机反过来 |
| 复制目录 | `docker cp` 支持目录 | 目标路径的父目录必须存在 |
| 容器内工具有限 | 精简镜像没有 ps/curl/vi 等 | 用宿主机的工具诊断容器，不要在容器里装工具 |

## 本篇最可能出错的地方及原因

### 1. `docker attach` 退出后容器也停了（最危险的坑）

**原因**：`docker attach` 连接的是容器的主进程（PID 1）。按 `Ctrl+C` 发送 SIGINT 给主进程，主进程退出 → 容器停止。

**预防**：尽量不用 `docker attach`。如果用了，退出时按 `Ctrl+P, Ctrl+Q` 而不是 `Ctrl+C`。

### 2. 进入 Alpine 容器时报 bash 不存在

**原因**：Alpine 是极致精简的 Linux 发行版，镜像只有 7MB，没有 bash。很多新手用 `alpine` 标签的镜像（如 `nginx:alpine`）时不知道这一点。

**解决**：`docker exec -it 容器名 sh`。记住：先试 bash，不行换 sh。

### 3. 在容器里装了一堆调试工具

**原因**：新手把容器当虚拟机用，进去后 `apt install` 一堆工具。但容器是**不可变的**——这些安装的修改在容器删除后就没了，且增加了镜像运行时的负担。

**最佳实践**：调试工具在宿主机上跑，用 `docker exec` 在容器内执行单条命令，用 `docker cp` 搬文件出来分析。
