## 107.1 短链接系统（TinyURL）

### 需求分析

| 维度 | 要求 |
|------|------|
| 功能 | 长 URL → 短 URL；访问短 URL → 302 重定向到长 URL |
| 流量 | 日生成 100 万条，日访问 1 亿次 |
| 存储 | 读写比 100:1 |
| 短链长度 | 7 位字符（`[0-9a-zA-Z]` 共 62 字符，62⁷ ≈ 3.5 万亿） |

### 设计步骤

```
1. 长 URL → hash(长URL) → 取前 7 位 → Base62 编码 → 短码
2. 存储：短码 → 长 URL 映射
3. 访问：短码 → 查 DB → 302 重定向
```

### 架构图

```
                      ┌─────────────┐
                      │  Nginx LB   │
                      └──────┬──────┘
                ┌────────────┼────────────┐
                ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ API 服务  │ │ API 服务  │ │ API 服务  │  ← 缩短 / 重定向
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │            │            │
             ▼            ▼            ▼
        ┌──────────────────────────────────────┐
        │         Redis 缓存（热点短链）         │  ← 读多写少，缓存提速
        └──────────────────┬───────────────────┘
                           │ 未命中
                           ▼
        ┌──────────────────────────────────────┐
        │   MySQL (short_code → long_url)       │
        │   short_code 是唯一索引               │
        │   按月分表                             │
        └──────────────────────────────────────┘
```

### 核心 SQL

```sql
CREATE TABLE `short_url` (
    `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
    `short_code` VARCHAR(7) NOT NULL COMMENT '短码，如 Ab3xK9m',
    `long_url` VARCHAR(2048) NOT NULL COMMENT '原始长 URL',
    `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP,
    `expire_at` DATETIME COMMENT '过期时间，NULL=永不过期',
    UNIQUE KEY `uk_short_code` (`short_code`)
) ENGINE=InnoDB;

INSERT INTO short_url (short_code, long_url)
VALUES ('Ab3xK9m', 'https://www.example.com/very/long/url?...');
```

### 短码生成策略

```java
// 方案：发号器（雪花算法 ID）→ Base62 编码
public class ShortUrlGenerator {
    private static final String BASE62 = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz";

    public static String encode(long id) {
        StringBuilder sb = new StringBuilder();
        while (id > 0) {
            sb.append(BASE62.charAt((int) (id % 62)));
            id /= 62;
        }
        return sb.reverse().toString();
    }
    // 1000000 → "4c92"
}
```

### 重定向服务

```java
@GetMapping("/{shortCode}")
public ResponseEntity<Void> redirect(@PathVariable String shortCode) {
    String longUrl = shortUrlService.getLongUrl(shortCode);
    return ResponseEntity.status(HttpStatus.FOUND)
            .header("Location", longUrl)
            .build();
}
```

### 避免短码重复：预生成

```
发号器 → 自增 ID → Base62 编码 → 检查是否被占 → 存入 DB
使用唯一索引保证不重复：
  INSERT ... ON DUPLICATE KEY → 失败则换下一个 ID
```

## 107.2 秒杀系统

### 需求分析

| 维度 | 要求 |
|------|------|
| 场景 | 100 万人抢 1 万件商品 |
| 核心约束 | 不超卖、不宕机、用户体验不差 |
| 时间窗口 | 10:00:00 ~ 10:01:00（1分钟） |

### 架构图

```
用户端
  │
  ▼
┌────────────────────────────────┐
│  CDN + 静态化（商品详情页）      │  ← 第 1 层：不走后端
└──────────────┬─────────────────┘
               │ 仅秒杀按钮请求
               ▼
┌────────────────────────────────┐
│  Nginx 限流（limit_req）        │  ← 第 2 层：拦截大部分无效请求
└──────────────┬─────────────────┘
               │
               ▼
┌────────────────────────────────┐
│  验证码 / 答题                  │  ← 第 3 层：延迟请求，过滤脚本
└──────────────┬─────────────────┘
               │
               ▼
┌────────────────────────────────┐
│  秒杀接口                        │
│  ① Redis 预减库存（Lua脚本）     │  ← 第 4 层：原子操作
│     if stock > 0: stock--       │
│     else: 返回"已售罄"           │
│  ② 返回"排队中"                  │
└──────────────┬─────────────────┘
               │ 抢到资格的用户
               ▼
┌────────────────────────────────┐
│  MQ（秒杀成功队列）               │  ← 第 5 层：削峰
└──────────────┬─────────────────┘
               │
               ▼
┌────────────────────────────────┐
│  订单服务（消费者）               │  ← 第 6 层：慢处理
│  MySQL 扣库存 → 生成订单         │
└────────────────────────────────┘
```

### 核心 SQL & Redis

```sql
-- 商品表
CREATE TABLE `product` (
    `id` BIGINT PRIMARY KEY,
    `name` VARCHAR(200),
    `total_stock` INT COMMENT '总库存',
    `seckill_stock` INT COMMENT '秒杀库存',
    `seckill_price` DECIMAL(10,2),
    `start_time` DATETIME,
    `end_time` DATETIME
);

-- 秒杀订单表（按 userId 分表）
CREATE TABLE `seckill_order_0` (
    `id` BIGINT PRIMARY KEY,
    `user_id` BIGINT,
    `product_id` BIGINT,
    `status` TINYINT DEFAULT 0 COMMENT '0=待支付 1=已支付 2=已取消',
    `created_at` DATETIME(3),
    UNIQUE KEY `uk_user_product` (`user_id`, `product_id`)
);
```

```java
// Redis Lua 脚本：原子减库存
String luaScript = """
    local key = KEYS[1]
    local stock = tonumber(redis.call('get', key))
    if stock and stock > 0 then
        redis.call('decr', key)
        return 1  -- 抢到
    else
        return 0  -- 售罄
    end
    """;

Long result = redisTemplate.execute(
    new DefaultRedisScript<>(luaScript, Long.class),
    List.of("seckill:stock:" + productId)
);
```

### 防超卖的核心

```
MySQL 层面：UPDATE product SET seckill_stock = seckill_stock - 1
            WHERE id = ? AND seckill_stock > 0
            ↓
            行锁保证不超卖，但性能差

Redis 层面：预减库存（Lua 原子操作）→ 扛住海量并发
            ↓
            MQ 异步写到 MySQL，最终一致
```

## 107.3 消息推送系统

### 需求分析

| 维度 | 要求 |
|------|------|
| 功能 | 向指定用户推送消息（App Push / 短信 / 邮件） |
| 规模 | 1 亿用户，日推送 10 亿条 |
| 要求 | 不丢消息、不重复推送、延迟 < 5 秒 |

### 架构图

```
┌──────────────┐
│  运营后台     │  ← 创建推送任务："给北京地区用户推送活动通知"
└──────┬───────┘
       │
       ▼
┌──────────────────────┐
│  推送任务服务          │
│  ① 根据条件圈定用户    │  ← "北京 + 活跃 + 安卓"
│  ② 分批写入任务表      │
│  ③ 发送 MQ            │
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐
│  Kafka                │  ← 任务分发
│  Topic: push-tasks    │
│  分区：按 userId hash  │
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐
│  推送工人（多实例）     │
│  ① 消费任务            │
│  ② 查询用户设备 token   │
│  ③ 调用 APNs/FCM/个推  │  ← 实际推送
│  ④ 上报结果            │
└──────────────────────┘
```

### 核心 SQL

```sql
CREATE TABLE `push_task` (
    `id` BIGINT PRIMARY KEY,
    `batch_id` VARCHAR(32) COMMENT '批次号',
    `user_id` BIGINT,
    `title` VARCHAR(100),
    `body` VARCHAR(500),
    `device_token` VARCHAR(255),
    `platform` TINYINT COMMENT '1=iOS 2=Android',
    `status` TINYINT DEFAULT 0 COMMENT '0=待推送 1=成功 2=失败',
    `retry_count` INT DEFAULT 0,
    `created_at` DATETIME,
    INDEX `idx_batch_status` (`batch_id`, `status`)
);

CREATE TABLE `user_device` (
    `user_id` BIGINT,
    `device_token` VARCHAR(255),
    `platform` TINYINT,
    `last_active` DATETIME,
    PRIMARY KEY (`user_id`, `device_token`)
);
```

### 关键设计点

1. **批量写入**：一次推送可能覆盖千万用户，逐条 INSERT 太慢 → 批量 INSERT + 异步 MQ
2. **失败重试**：推送工人失败 → `retry_count+1` → 重试 3 次 → 还失败进死信队列
3. **频率控制**：同一用户 1 分钟内最多收 3 条推送（令牌桶限流）
4. **去重**：同一批次 + 同一用户 → `batch_id + user_id` 唯一索引

## 107.4 Feed 流系统（朋友圈/微博）

### 需求分析

| 维度 | 要求 |
|------|------|
| 功能 | 用户发布动态 → 粉丝看到；用户刷自己的 Feed |
| 规模 | 1 亿用户，大 V 有 1000 万粉丝 |
| 延迟 | 发布后 5 秒内粉丝可见 |

### 两种模式

```
Push 模式（写扩散）：
  用户发动态 → 写入所有粉丝的收件箱
  优点：读快（O(1) 查自己收件箱）
  缺点：大 V 写 1000 万条 → 写扩散爆炸

Pull 模式（读扩散）：
  用户发动态 → 只写自己的发件箱
  刷 Feed → 拉取所有关注者的发件箱 → 合并排序
  优点：写简单
  缺点：读慢（关注 1000 人时拉 1000 次）

生产方案：Push + Pull 混合
  普通用户 → Push（粉丝少，写扩散代价小）
  大 V → Pull（粉丝多，不写扩散，粉丝读时再拉）
```

### 架构图

```
用户发布动态
  │
  ▼
┌────────────────┐
│  动态服务       │
│  ① 写入 MySQL   │
│  ② 判断粉丝数    │
│     < 5000 → Push │
│     >= 5000 → 标记大V │
└───┬────────────┘
    │
    ├── Push ──→ ┌──────────────┐
    │            │  粉丝收件箱   │
    │            │  (Redis List)│  ← 每个用户一个 List，存最新 200 条
    │            │  user:{id}:feed│
    │            └──────────────┘
    │
    └── 大V标记 ──→ ┌──────────────┐
                    │  大V发件箱   │
                    │  (Redis ZSet)│  ← 粉丝刷 Feed 时拉取
                    └──────────────┘

用户刷 Feed
  │
  ▼
① 从自己收件箱读 Push 的动态（Redis List，分页）
② 查出关注的大 V 列表
③ 拉大 V 的发件箱（ZSet，按时间排序）
④ 合并 ① + ③，按时间排序分页返回
```

### 核心 SQL & Redis 数据结构

```sql
CREATE TABLE `feed` (
    `id` BIGINT PRIMARY KEY,
    `user_id` BIGINT COMMENT '发布者',
    `content` TEXT,
    `type` TINYINT COMMENT '1=图文 2=视频 3=转发',
    `created_at` DATETIME(3),
    INDEX `idx_user_time` (`user_id`, `created_at`)
);
```

```java
// Redis 数据结构设计

// 普通用户收件箱：List（最新 200 条）
// Key: feed:inbox:{userId}
// Value: List of feedId
// 发布时：LPUSH feed:inbox:{followerId} feedId  (推送给每个粉丝)
//         LTRIM feed:inbox:{followerId} 0 199   (只保留最新 200 条)
// 读取时：LRANGE feed:inbox:{userId} {offset} {offset+19}  (分页)

// 大 V 发件箱：ZSet（score = 时间戳）
// Key: feed:outbox:{bigVUserId}
// Value: ZSet {feedId: timestamp}
// 发布时：ZADD feed:outbox:{bigVUserId} timestamp feedId
// 读取时：ZREVRANGE feed:outbox:{bigVUserId} {offset} {offset+19}

// 刷 Feed 时：
// 1. LRANGE feed:inbox:{myUserId} ...
// 2. 遍历关注的大V，ZREVRANGE feed:outbox:{bigVId} ...
// 3. 合并排序，取前 20 条
```

### 冷热分离

```
活跃用户（最近 7 天登录）：Feed 数据全在 Redis
非活跃用户：最近 200 条在 Redis，更早的在 MySQL
```

## 107.5 四题总结

| 系统 | 核心难点 | 核心技术 |
|------|----------|----------|
| 短链接 | 短码生成不重复、高并发重定向 | Base62 编码 + 发号器 + 302 重定向 |
| 秒杀 | 不超卖、扛并发 | Redis Lua 预减库存 + MQ 削峰 + 限流 |
| 消息推送 | 海量用户、失败重试、去重 | Kafka 分区 + 批量推送 + 重试机制 |
| Feed 流 | 大 V 写扩散、实时性 | Push+Pull 混合 + Redis List/ZSet |

## 107.7 自测题

**1. 设计短链接系统时，为什么不用 URL 的 MD5 值直接当短码？**

**2. 秒杀系统中 Redis 预减库存已经扣成功了，但最后写 MySQL 时失败了（比如网络抖动）。用户以为自己抢到了但实际没生成订单。怎么处理？**

**3. Feed 流系统中，一个用户同时是普通用户（被 Push）和大 V（被 Pull）。他的 Feed 如何合并排序？**

---

**答案提示**：1→MD5 输出 16 进制（128 bit），转为 7 位短码信息大量丢失，碰撞概率太高。应用发号器自增 ID + Base62 编码。2→在 MQ 消费者中重试 + 幂等处理。如果最终失败，发一条"秒杀失败"的异步通知给用户，同时 Redis 补偿库存（incr）。不能简单回滚——因为 Redis 和 MySQL 是两套存储，需要补偿机制。3→从自己的 Redis 收件箱 List 拉取普通用户的动态（按时间有序），再拉取关注的大 V 发件箱 ZSet（按时序拉），两路数据在应用层做 K 路归并排序（类似归并算法），按时间戳合并后分页返回。恭喜完成本教程全部 107 章！
