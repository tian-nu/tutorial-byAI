## 43.1 遇到Bug，你的第一反应是什么？

我猜你的第一反应是这些中的某一个：

1. 盯着屏幕，自己一行一行看代码
2. 把代码改了又改，越改越乱
3. 打开百度/Google，搜索错误信息
4. 找朋友帮忙看

**在AI编程时代，这些做法都太慢了。** 正确的第一反应是：

> **把完整错误信息复制粘贴给AI。** 就这么简单。

---

## 43.2 什么是Bug和调试

> **📖 术语：Bug（缺陷）**——代码中的错误，导致程序行为不符合预期。这个词的来源很有趣：1947年，一只飞蛾（bug）飞进了计算机的继电器，导致故障。从此程序员把"程序错误"叫做Bug。

> **📖 术语：调试（Debugging）**——找到Bug、修复Bug的过程。字面意思就是"去虫"（de-bug）。

在传统编程中，调试是最痛苦的环节——你可能花一下午找一个Bug，最后发现是少写了一个等号。AI编程时代，调试变成了**"把错误信息给AI，让它帮你分析"**。

---

## 43.3 Bug处理口诀：不要瞎猜，复制粘贴

**四步口诀**：

```
① 把完整错误信息复制给AI
② 告诉AI你做了什么、想要什么结果
③ 让AI分析原因、给出修复方案
④ 按方案改 → 验证
```

下面我们一步步拆解。

---

### 步骤①：把完整错误信息复制给AI

**这是最关键的一步。** 很多人只给AI一句"代码报错了"，AI完全不知道怎么回事。

#### ❌ 错误做法：只给AI一句话

```
我的代码报错了，帮我修一下。
```

AI只能盲猜，大概率猜不对。

#### ✅ 正确做法：把完整错误信息全部贴过去

```
我运行 python app.py 时报了以下错误，帮我分析原因：

'''
Traceback (most recent call last):
  File "app.py", line 15, in <module>
    result = calculate_average(scores)
  File "app.py", line 8, in calculate_average
    total += score
TypeError: unsupported operand type(s) for +=: 'int' and 'str'
'''
```

**为什么要贴完整信息？** 因为错误信息包含了三样关键东西：

1. **错误类型**（`TypeError`）：告诉AI"这是什么类型的错误"
2. **错误位置**（`app.py, line 8`）：告诉AI"在哪一行出错"
3. **错误描述**（`unsupported operand type(s) for +=: 'int' and 'str'`）：告诉AI"具体错在哪"

这三样东西凑齐，AI就能精准定位问题。

---

### 步骤②：告诉AI你做了什么、想要什么结果

光有错误信息还不够，AI还需要知道**你的意图**。

```
这个函数的作用是：接收一个分数列表，计算平均分。
我传入的数据是：scores = [85, 90, '92', 88]（注意第三个是字符串）
我想要的结果是：忽略非数字的项，返回数字的平均值
```

**为什么需要这一步？** 因为有时候"错误"不是语法错误，而是"代码逻辑对，但不是你想要的效果"。比如上面的例子——AI可能觉得"应该把所有字符串转成数字再算"，但你的需求是"忽略非数字项"。两种方案技术上都没错，但只有一种是你想要的。

---

### 步骤③：让AI分析原因、给出修复方案

把信息给AI后，这样问：

```
请帮我：
1. 分析这个错误的原因
2. 给出修复方案（不要只给代码，先解释思路）
3. 写出修复后的完整代码
```

**为什么要AI先解释思路？** 因为这样你能判断AI的方案是否合理。如果AI说"把字符串转成数字"，但你想"跳过非数字"——那你就能在AI写代码之前纠正它，不用返工。

---

### 步骤④：按方案改 → 验证

AI给了修复方案后，应用修改，然后**重新运行**，确认Bug修好了。

**但注意**：修好一个Bug之后，**重新跑一遍所有相关功能**。修A时可能引入Bug B——这叫"回归Bug"，AI编程中特别常见。

## 43.4 完整案例：一次真实的调试过程

### 场景

你写了一个计算平均分的函数，但运行时报错了。

### 你的原始代码

```python
def calculate_average(scores):
    total = 0
    for score in scores:
        total += score
    return total / len(scores)

scores = [85, 90, "92", 88]
result = calculate_average(scores)
print(f"平均分：{result}")
```

### 运行报错

```
Traceback (most recent call last):
  File "app.py", line 8, in <module>
    result = calculate_average(scores)
  File "app.py", line 4, in calculate_average
    total += score
TypeError: unsupported operand type(s) for +=: 'int' and 'str'
```

### 你的完整Prompt（四步合一）

```
我运行这段代码时报错了，错误信息如下：

'''
Traceback (most recent call last):
  File "app.py", line 8, in <module>
    result = calculate_average(scores)
  File "app.py", line 4, in calculate_average
    total += score
TypeError: unsupported operand type(s) for +=: 'int' and 'str'
'''

我的代码是：
```python
def calculate_average(scores):
    total = 0
    for score in scores:
        total += score
    return total / len(scores)

scores = [85, 90, "92", 88]
result = calculate_average(scores)
print(f"平均分：{result}")
```

这个函数的作用是：计算一个分数列表的平均值。
我想要的行为是：如果列表里有非数字的项（比如字符串），跳过它，只计算数字的平均值。

请帮我：
1. 分析错误原因
2. 给出修复思路
3. 写出修复后的完整代码
```

### AI的回复（预期）

> **错误原因**：`scores` 列表中混入了字符串 `"92"`。当循环到它时，`total += "92"` 试图把 `int` 和 `str` 相加，Python 不允许这种操作，所以抛出 `TypeError`。
>
> **修复思路**：在循环中加一个类型检查——如果 `score` 是数字（`int` 或 `float`），就累加；如果是字符串，跳过。同时注意，跳过的项不应该计入总数，所以分母也要用有效数字的个数。
>
> **修复后代码**：

```python
def calculate_average(scores):
    valid_scores = []
    for score in scores:
        if isinstance(score, (int, float)):
            valid_scores.append(score)
    
    if len(valid_scores) == 0:
        return 0  # 没有有效分数时返回0
    
    total = sum(valid_scores)
    return total / len(valid_scores)

scores = [85, 90, "92", 88]
result = calculate_average(scores)
print(f"平均分：{result}")
```

> 运行结果：`平均分：87.66666666666667`（(85+90+88) / 3 = 87.67）

**一次搞定，没有返工。** 这就是四步口诀的力量。

## 43.5 常见报错信息速查

你会遇到成千上万种报错，但这四种是最常见的。**记住它们的中文意思，不用背，遇到了回来查就行。**

| 错误类型 | 中文意思 | 什么时候会出现 | 典型原因 |
|----------|----------|---------------|---------|
| `SyntaxError` | 语法错误 | 代码根本跑不起来 | 少写了括号、冒号、引号 |
| `IndentationError` | 缩进错误 | Python特有 | 空格和Tab混用、缩进层级不对 |
| `NameError` | 变量名没定义 | 使用了不存在的变量 | 拼写错误、变量还没定义就用 |
| `TypeError` | 类型错误 | 操作了不兼容的类型 | 数字+字符串、列表当函数调用 |

### SyntaxError 示例

```python
# ❌ 少写了右括号
print("Hello World"

# 错误信息：SyntaxError: unexpected EOF while parsing
```

```python
# ✅ 修正
print("Hello World")
```

### IndentationError 示例

```python
# ❌ 缩进混乱
def greet(name):
print("Hello")  # 这行没有缩进！
    return name

# 错误信息：IndentationError: expected an indented block
```

```python
# ✅ 修正
def greet(name):
    print("Hello")
    return name
```

### NameError 示例

```python
# ❌ 变量名拼写错误
user_name = "Alice"
print(usr_name)  # 少写了 e

# 错误信息：NameError: name 'usr_name' is not defined
```

```python
# ✅ 修正
user_name = "Alice"
print(user_name)
```

### TypeError 示例

```python
# ❌ 数字和字符串做加法
result = 100 + "元"

# 错误信息：TypeError: unsupported operand type(s) for +: 'int' and 'str'
```

```python
# ✅ 修正
result = str(100) + "元"  # 先转成字符串再拼接
# 或
result = f"{100}元"
```

## 43.6 如果AI也修不好，怎么办？

有时候AI给了修复方案，你应用了，还是报错。再来一次，还是不对。**第三次了，还是不对。**

**这时候，不要再继续让AI修了。** 记住一个原则：

> **超过3次修正还不对 → 不是AI的问题，是你的问题描述有问题。**

具体做法：

### 1. 简化问题，做最小复现

把整个项目暂时放一边，单独写一个**只有5行代码的小脚本**，只复现那个Bug。

```python
# 简化的测试脚本 — 只测试"字符串和数字混合"的问题
test_list = [1, "2", 3]
for item in test_list:
    print(item * 2)  # 数字正常，字符串会重复，预期行为吗？
```

### 2. 把这个最小复现+完整错误信息，开一个新会话给AI

清空上下文，开一个新的Chat，把最小复现的代码和错误信息贴进去。**不要带之前的对话历史**——之前的对话可能已经让AI陷入了错误的思路。

### 3. 换个AI模型试试

如果Cursor的模型不行，试试Trae，或者用ChatGPT网页版。不同模型对不同问题的理解能力不同。

### 4. 换个角度描述问题

不要用同样的措辞。比如：
- 之前说"修这个Bug"
- 换成"这段代码有一个TypeError，帮我解释为什么Python不允许int和str相加，以及怎么处理"

## 43.7 调试的核心心态

**不要怕Bug。Bug是编程的一部分，就像学走路一定会摔跤。** AI编程时代，Bug不再是"你水平不行"的标志——它只是"AI猜错了，需要你纠正它"。

每一次调试，你都在教AI更准确地理解你的需求。**调试不是浪费时间，而是在训练你的AI搭档。**

## 自测

1. 遇到Bug时，为什么不能只对AI说"帮我修一下"？正确的做法是什么？
2. `TypeError` 和 `NameError` 有什么区别？各举一个例子。
3. 如果同一个Bug让AI修了4次还是不对，你应该怎么办？（至少说两种做法）

> **答案**：
> 1. 因为AI不知道"哪里有问题"，只能盲猜。正确做法是把完整错误信息（包括错误类型、位置、描述）复制粘贴给AI，同时告诉AI你做了什么、想要什么结果。
> 2. `TypeError` 是类型错误，比如 `"hello" + 5`（字符串和数字相加）；`NameError` 是变量名没定义，比如拼写错误的变量名。两者都是常见错误，但根因不同。
> 3. ①简化问题，做一个最小复现的脚本，单独给AI看；②开一个新会话，清空混乱的上下文；③换个AI模型试试；④换个角度重新描述问题。不要死磕同一个对话。
