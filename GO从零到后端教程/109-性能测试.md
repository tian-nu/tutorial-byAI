## 109.1 Go Benchmark

### 什么是Benchmark

Benchmark（基准测试，测量代码执行速度的Go内置工具，详见附录I）是Go内置的**性能测试**。和单元测试类似，但目的不是验证"对不对"，而是测量"有多快"。

### 基本规则

1. Benchmark函数以 `Benchmark` 开头，接受 `*testing.B` 参数
2. 放在 `_test.go` 文件中
3. 用 `go test -bench=.` 运行

### 第一个Benchmark

```go
package calc

import "testing"

func Add(a, b int) int {
    return a + b
}

func BenchmarkAdd(b *testing.B) {
    for i := 0; i < b.N; i++ {
        Add(1, 2)
    }
}
```

运行：
```bash
go test -bench=.
go test -bench=Add           # 只跑名字含Add的Benchmark
go test -bench=. -benchmem   # 同时显示内存分配
go test -bench=. -count=5    # 重复跑5次，取稳定的结果
```

输出：
```
BenchmarkAdd-8    1000000000    0.25 ns/op    0 B/op    0 allocs/op
```

### b.N是什么

`b.N` 不是你来设定的——Go的Benchmark框架会自动调整它。框架先从一个很小的值开始（比如1），如果跑得太快（不到1秒），就把 `b.N` 翻倍再来一次，直到测试运行了足够长的时间（默认1秒）能产生统计上可信的结果。

这就是为什么你在 `BenchmarkAdd` 里写了一个循环 `for i := 0; i < b.N; i++`——`b.N` 可能最后是1000000000，意味着你的函数被执行了10亿次来测算平均耗时。

## 109.2 解读Benchmark结果

### 输出格式

```
BenchmarkStringConcat-8    5000000    250 ns/op    48 B/op    5 allocs/op
```

| 部分 | 含义 | 解读 |
|------|------|------|
| BenchmarkStringConcat-8 | 函数名-使用的CPU核数 | 用了8个逻辑CPU |
| 5000000 | 测试循环执行了多少次 | 跑了500万次 |
| 250 ns/op | 每次操作平均耗时（纳秒） | 1次调用约250纳秒=0.25微秒 |
| 48 B/op | 每次操作分配的内存（字节） | 每次调用分配48字节 |
| 5 allocs/op | 每次操作的内存分配次数 | 每次调用5次内存分配 |

### 三大指标详解

**ns/op（每次操作耗时纳秒）：** 越小越好。这是速度的直接衡量。

**B/op（每次操作分配字节数）：** 越小越好。内存分配是GC（垃圾回收）的主要触发因素。分配越少，GC压力越小，程序越稳定。

**allocs/op（每次操作分配次数）：** 越小越好。即使每次只分配1字节，分配次数多也意味着频繁的GC扫描。

### 实战示例对比

**字符串拼接：`+` vs `fmt.Sprintf` vs `strings.Builder`**

```go
var strHello = "hello"
var strSpace = " "
var strWorld = "world"

func BenchmarkPlusConcat(b *testing.B) {
    for i := 0; i < b.N; i++ {
        _ = strHello + strSpace + strWorld
    }
}

func BenchmarkSprintf(b *testing.B) {
    for i := 0; i < b.N; i++ {
        _ = fmt.Sprintf("%s %s", strHello, strWorld)
    }
}

func BenchmarkBuilder(b *testing.B) {
    for i := 0; i < b.N; i++ {
        var sb strings.Builder
        sb.WriteString(strHello)
        sb.WriteString(strSpace)
        sb.WriteString(strWorld)
        _ = sb.String()
    }
}
```

典型输出：
```
BenchmarkPlusConcat-8    300000000     4.5 ns/op     0 B/op    0 allocs/op
BenchmarkSprintf-8        20000000    75.0 ns/op    16 B/op    2 allocs/op
BenchmarkBuilder-8        50000000    24.0 ns/op    24 B/op    2 allocs/op
```

**解读：**
- `+` 拼接约4.5ns，字符串拼接本身开销很低，但注意这需要创建新字符串
- `fmt.Sprintf` 最慢（75ns），但可读性最好
- `strings.Builder` 约24ns，虽然单次稍慢但循环中内存分配更可控

**结论：** 少量字符串拼接用 `+`，循环中大量拼接用 `strings.Builder`（预分配更佳），非性能关键路径可以用 `fmt.Sprintf`（可读性好）。

### Benchmark中避免编译器优化

```go
var result string

func BenchmarkSomething(b *testing.B) {
    var s string
    for i := 0; i < b.N; i++ {
        s = expensiveOperation()
    }
    result = s
}
```

如果 `s` 没有被使用，编译器可能把整个 `expensiveOperation()` 优化掉——你的Benchmark结果就会变成"0 ns/op"。把结果赋值给包级变量 `result` 可以防止这种优化。

## 109.3 pprof性能分析

### pprof是什么

pprof（Go内置性能分析工具，可定位CPU和内存瓶颈，详见附录I）是Go内置的性能分析工具。如果说Benchmark告诉你"这个函数慢"，pprof告诉你**"为什么慢"和"哪里最慢"**。

### 三种Profile类型

| Profile | 测量什么 | 回答的问题 |
|---------|----------|-----------|
| **CPU Profile** | 每个函数花了多少CPU时间 | "为什么CPU 100%？" |
| **Memory Profile** | 每个函数分配了多少内存，哪些内存还活着 | "为什么内存占用这么高？" |
| **Goroutine Profile** | 当前有多少个Goroutine，分别在干什么 | "为什么Goroutine泄漏了？" |

### 在代码中集成pprof

```go
package main

import (
    "net/http"
    _ "net/http/pprof"
)

func main() {
    go func() {
        http.ListenAndServe("localhost:6060", nil)
    }()

    // 你的主逻辑...
}
```

然后访问 `http://localhost:6060/debug/pprof/` 就能看到各种Profile的入口。

### go tool pprof交互式分析

**采样CPU Profile（30秒）：**

```bash
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30
```

进入交互模式后，常用命令：

```
(pprof) top 10           # 显示CPU耗时前10的函数
(pprof) top 10 -cum      # 按累计耗时排序（含子调用）
(pprof) list 函数名       # 显示函数源码+每行耗时
(pprof) web              # 生成调用图（需要graphviz）
(pprof) pdf              # 生成PDF报告
```

**top输出解读：**

```
Showing nodes accounting for 2.5s, 89.29% of 2.8s total

      flat  flat%   sum%        cum   cum%
     1.10s 39.29% 39.29%      1.20s 42.86%  runtime.mallocgc
     0.50s 17.86% 57.14%      0.50s 17.86%  runtime.memmove
     0.30s 10.71% 67.86%      0.30s 10.71%  crypto/sha256.block
```

| 列 | 含义 |
|----|------|
| flat | 该函数**自身**消耗的时间 |
| cum | 该函数**及其调用的所有子函数**消耗的总时间 |

`flat` 大 = 这个函数本身写得慢（需要优化它）
`cum` 大但 `flat` 小 = 这个函数调了慢的子函数（需要优化它的子函数或减少不必要的调用）

### 内存分析

```bash
go tool pprof http://localhost:6060/debug/pprof/heap
```

```
(pprof) top 10
(pprof) list 函数名
```

重点关注 `inuse_space`（当前还占着的内存）和 `alloc_space`（累计分配过的内存）。

### Goroutine分析

```bash
go tool pprof http://localhost:6060/debug/pprof/goroutine
```

```
(pprof) top 10
(pprof) traces          # 显示Goroutine的完整调用栈
```

如果Goroutine数量异常高（比如几千个），说明可能有Goroutine泄漏。

## 109.4 火焰图

### 什么是火焰图

火焰图（Flame Graph）是CPU调用的可视化。x轴宽度代表CPU时间占比（越宽越耗时），y轴是调用栈深度（越高调用链越长）。

```
┌──────────────────────────────────────────────────────┐
│                    main                              │ 顶层
│   ┌──────────────┐  ┌─────────────────────────────┐  │
│   │  http.Handler │  │     processRequest           │  │
│   │  ┌──────────┐ │  │  ┌──────────────────┐      │  │
│   │  │json.Decode│ │  │  │  queryDatabase    │      │  │
│   │  │ (10%)    │ │  │  │  ┌─────────────┐  │      │  │
│   │  └──────────┘ │  │  │  │ mysql.Query  │  │      │  │
│   │               │  │  │  │ (60%宽度!)   │  │      │  │  ← 瓶颈在这！
│   │               │  │  │  └─────────────┘  │      │  │
│   └───────────────┘  │  └──────────────────┘      │  │
│                      └─────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

一眼就能看出 `mysql.Query` 占了60%的CPU时间——这就是性能瓶颈。

### 生成火焰图

```bash
go tool pprof -http=:8080 http://localhost:6060/debug/pprof/profile?seconds=30
```

浏览器自动打开 `http://localhost:8080`，你会看到：

- **VIEW菜单 → Flame Graph**：火焰图
- **VIEW菜单 → Top**：函数耗时排名
- **VIEW菜单 → Graph**：调用关系图

### 实战排查流程

假设你的API响应变慢了：

```bash
go tool pprof -http=:8080 http://localhost:6060/debug/pprof/profile?seconds=30
```

1. 打开火焰图，发现80%的宽度都在 `crypto/bcrypt.CompareHashAndPassword` 上
2. 你意识到：每次请求都在重新验证密码哈希——应该用缓存！
3. 优化：登录成功后缓存Token，后续请求验证Token而不是验证密码哈希
4. P99延迟从800ms降到了15ms

这就是pprof的价值——**数据驱动的性能优化，不是凭感觉瞎猜。**

## 109.5 压测工具：wrk

### wrk是什么

wrk是一个高性能的HTTP压测工具。它从外部模拟大量并发请求，测试你的服务在压力下的表现。

### 安装

```bash
sudo apt-get install wrk         # Ubuntu
brew install wrk                  # macOS
```

### 基本使用

```bash
wrk -t4 -c100 -d30s http://localhost:8080/api/users
```

| 参数 | 含义 |
|------|------|
| `-t4` | 使用4个线程 |
| `-c100` | 保持100个并发连接 |
| `-d30s` | 持续压测30秒 |
| `-s script.lua` | 加载Lua脚本（高级用法） |

### 解读输出

```
Running 30s test @ http://localhost:8080/api/users
  4 threads and 100 connections
  Thread Stats   Avg      Stdev     Max   +/- Stdev
    Latency     5.23ms    2.15ms  45.67ms   88.50%
    Req/Sec     4.85k   430.21     5.32k    72.00%
  578234 requests in 30.10s, 72.45MB read
Requests/sec:  19210.43
Transfer/sec:      2.41MB
```

**关键指标：**

| 指标 | 含义 | 好/坏 |
|------|------|-------|
| Latency Avg | 平均延迟 | 越低越好 |
| Latency Max | 最大延迟 | 关注异常值 |
| Req/Sec | 每秒请求数（QPS） | 越高越好 |
| Requests/sec | 总QPS | 系统的吞吐能力 |
| +/- Stdev | 延迟的稳定度 | 越高越稳定 |

### 进阶：Lua脚本自定义请求

创建 `post.lua`：

```lua
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.body = '{"email":"alice@example.com","password":"secret123"}'
```

```bash
wrk -t4 -c100 -d30s -s post.lua http://localhost:8080/api/users
```

### 压测最佳实践

1. **不要在公网压测生产环境**——你可能把服务器打死。在测试环境或本地进行。
2. **逐步加压**：从 `-c10` 开始，逐步增加到 `-c50`、`-c100`、`-c500`，观察系统在哪个点开始退化。
3. **压测时开启pprof**：一边加压一边采集CPU/内存Profile，找到真正的瓶颈。
4. **关注长尾延迟**：Avg可能很好看，但P99可能很糟糕。少数用户的糟糕体验就能毁掉口碑。
5. **压测不仅是测Go应用**：也包括Nginx、数据库连接池、操作系统文件描述符限制等。

## 🤔 想多一点：什么时候做性能优化？

**常见误区：** 一开始就疯狂优化，把所有变量都预分配、用池化代替`make`、手写汇编级别的优化——然后需求变了，你的优化全白费了。

**正确流程：**
1. 先写**正确**的代码（简单、可读、可维护）
2. 写Benchmark，记录基线数据
3. 用pprof找到真正的瓶颈
4. 只优化瓶颈（不要优化非热点路径——你节省0.1ns在一个只调用10次的函数上毫无意义）
5. 用Benchmark验证优化效果

**Donald Knuth的名言值得铭记：**
> "过早的优化是万恶之源。"
> "Premature optimization is the root of all evil."

在97%的情况下，可读性比微小的性能提升更重要。只有当你的API真的扛不住流量时，才该拿起pprof和Benchmark。
