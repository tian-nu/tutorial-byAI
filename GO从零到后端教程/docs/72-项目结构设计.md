# 第72章 · 项目结构设计

> "写10行代码可以全塞在 `main.go` 里。写1000行也还行。写10000行？你会发现自己不断往上翻往下翻，改一个函数要跨越500行。项目结构就是把混乱的代码按职责分类、分层、分文件——就像图书馆把书按学科分区一样，让你和你的同事能快速找到任何一段代码。"

---

## 72.1 MVC模式

MVC是最经典的Web开发架构模式，诞生于上世纪70年代的Smalltalk语言，至今仍是大多数Web框架的默认组织方式。

### 三层分家

| 层 | 全称 | 职责 | 餐厅比喻 |
|----|------|------|----------|
| **M**odel | 模型 | 数据结构、数据库操作 | 食材的标准规格（一块鸡胸肉200g，切成2cm丁） |
| **V**iew | 视图 | 用户看到的界面 | 摆盘——菜装在什么盘子里，怎么装饰 |
| **C**ontroller | 控制器 | 接收请求、调用Model、选择View | 厨师——收到订单，取食材，做好装盘 |

### 一个请求在MVC中的旅程

```
用户访问 GET /articles/5
  ↓
Controller: ArticleController.Show(5)
  ├─ 调用 Model: Article.FindByID(5)  → 查数据库
  ├─ 拿到文章数据
  └─ 选择 View: article_detail.html
       └─ 把数据填进模板，生成HTML返回
```

在后端API项目中，**View层通常不存在**（因为返回的是JSON，不是HTML）。这时MVC退化成**MC**模型——Model + Controller。

## 72.2 分层架构

在Go后端项目中，标准的做法不是MVC，而是**分层架构**：

```
Handler（Controller）→ Service → Repository（DAO）→ Model
```

### 各层职责

| 层 | 职责 | 不能做的事 |
|----|------|-----------|
| **Handler** | 处理HTTP请求/响应，参数校验，调用Service | 不能直接操作数据库 |
| **Service** | 业务逻辑，事务管理，调用多个Repository | 不能处理HTTP请求参数 |
| **Repository** | 数据库CRUD，缓存操作 | 不能写业务逻辑 |
| **Model** | 数据结构定义（与数据库表对应） | 不能有业务逻辑方法 |

**比喻**：
- Handler是服务员——接菜单（请求参数），叫厨师（Service），端菜出去（响应）
- Service是主厨——决定这道菜怎么做（业务逻辑），吩咐帮厨（Repository）去取食材
- Repository是帮厨——冰箱里取食材（数据库查询），但不管做什么菜
- Model是菜谱上的标准规格——一道菜有哪些成分（字段）

### 代码示例

`model/user.go`：

```go
package model

import "time"

type User struct {
    ID        uint      `json:"id"`
    Name      string    `json:"name"`
    Email     string    `json:"email"`
    CreatedAt time.Time `json:"created_at"`
    UpdatedAt time.Time `json:"updated_at"`
}
```

`repository/user_repo.go`：

```go
package repository

import (
    "database/sql"
    "myweb/model"
)

type UserRepository struct {
    db *sql.DB
}

func NewUserRepository(db *sql.DB) *UserRepository {
    return &UserRepository{db: db}
}

func (r *UserRepository) FindByID(id uint) (*model.User, error) {
    user := &model.User{}
    row := r.db.QueryRow("SELECT id, name, email, created_at, updated_at FROM users WHERE id = ?", id)
    err := row.Scan(&user.ID, &user.Name, &user.Email, &user.CreatedAt, &user.UpdatedAt)
    if err != nil {
        return nil, err
    }
    return user, nil
}

func (r *UserRepository) FindAll(page, size int) ([]model.User, int64, error) {
    var users []model.User
    var total int64

    r.db.QueryRow("SELECT COUNT(*) FROM users").Scan(&total)

    offset := (page - 1) * size
    rows, err := r.db.Query(
        "SELECT id, name, email, created_at, updated_at FROM users LIMIT ? OFFSET ?",
        size, offset,
    )
    if err != nil {
        return nil, 0, err
    }
    defer rows.Close()

    for rows.Next() {
        var u model.User
        rows.Scan(&u.ID, &u.Name, &u.Email, &u.CreatedAt, &u.UpdatedAt)
        users = append(users, u)
    }
    return users, total, nil
}
```

`service/user_service.go`：

```go
package service

import (
    "errors"
    "myweb/model"
    "myweb/repository"
)

type UserService struct {
    repo *repository.UserRepository
}

func NewUserService(repo *repository.UserRepository) *UserService {
    return &UserService{repo: repo}
}

func (s *UserService) GetUser(id uint) (*model.User, error) {
    if id == 0 {
        return nil, errors.New("用户ID不能为0")
    }
    return s.repo.FindByID(id)
}

func (s *UserService) GetUserList(page, size int) ([]model.User, int64, error) {
    if page < 1 {
        page = 1
    }
    if size < 1 || size > 100 {
        size = 20
    }
    return s.repo.FindAll(page, size)
}
```

`handler/user_handler.go`：

```go
package handler

import (
    "net/http"
    "strconv"

    "github.com/gin-gonic/gin"
    "myweb/pkg/response"
    "myweb/service"
)

type UserHandler struct {
    svc *service.UserService
}

func NewUserHandler(svc *service.UserService) *UserHandler {
    return &UserHandler{svc: svc}
}

func (h *UserHandler) GetUser(c *gin.Context) {
    idStr := c.Param("id")
    id, err := strconv.Atoi(idStr)
    if err != nil {
        response.Error(c, 10001, "ID格式错误")
        return
    }

    user, err := h.svc.GetUser(uint(id))
    if err != nil {
        response.Error(c, 10002, "用户不存在")
        return
    }

    response.Success(c, user)
}
```

## 72.3 标准目录结构

一个中等规模的Go Web项目的推荐目录结构：

```
myweb/
├── cmd/
│   └── server/
│       └── main.go              # 程序入口
├── internal/
│   ├── handler/                 # HTTP处理层（Controller）
│   │   ├── user_handler.go
│   │   └── article_handler.go
│   ├── service/                 # 业务逻辑层
│   │   ├── user_service.go
│   │   └── article_service.go
│   ├── repository/              # 数据访问层
│   │   ├── user_repo.go
│   │   └── article_repo.go
│   └── model/                   # 数据结构层
│       ├── user.go
│       └── article.go
├── pkg/
│   ├── response/                # 统一响应工具
│   │   └── response.go
│   └── middleware/               # 中间件
│       ├── auth.go
│       ├── cors.go
│       └── logger.go
├── config/
│   ├── config.go                # 配置结构体
│   └── config.yaml              # 配置文件
├── routes/
│   └── router.go                # 路由注册
├── docs/                        # Swagger文档
├── migrations/                  # 数据库迁移脚本
├── go.mod
├── go.sum
└── Makefile
```

### 各目录说明

| 目录 | 用途 | 可见性 |
|------|------|--------|
| `cmd/` | 程序入口，每个子目录是一个可执行程序 | — |
| `internal/` | 私有代码，**外部项目不能导入** | Go编译器强制私有 |
| `pkg/` | 公共库，可被外部项目导入 | 公开 |
| `config/` | 配置文件与加载逻辑 | — |
| `routes/` | 路由注册，连接handler和路由 | — |
| `migrations/` | 数据库表结构变更脚本 | — |

**`internal` vs `pkg` 的选择**：如果你的代码只在本项目中使用，放 `internal/`。如果可能被其他项目导入复用（比如你写了一个通用的验证码生成库），放 `pkg/`。

## 72.4 配置管理：Viper

硬编码的配置（数据库地址、端口号、密钥）是灾难。你需要一个配置管理系统。Viper是Go生态中最流行的配置库。

### 安装

```bash
go get github.com/spf13/viper
```

### 读取YAML配置

`config/config.yaml`：

```yaml
server:
  port: 8080
  mode: debug

database:
  host: localhost
  port: 3306
  user: root
  password: mypassword
  dbname: myweb

jwt:
  secret: my-secret-key
  expire: 7200
```

`config/config.go`：

```go
package config

import (
    "fmt"

    "github.com/spf13/viper"
)

type Config struct {
    Server   ServerConfig   `mapstructure:"server"`
    Database DatabaseConfig `mapstructure:"database"`
    JWT      JWTConfig      `mapstructure:"jwt"`
}

type ServerConfig struct {
    Port int    `mapstructure:"port"`
    Mode string `mapstructure:"mode"`
}

type DatabaseConfig struct {
    Host     string `mapstructure:"host"`
    Port     int    `mapstructure:"port"`
    User     string `mapstructure:"user"`
    Password string `mapstructure:"password"`
    DBName   string `mapstructure:"dbname"`
}

type JWTConfig struct {
    Secret string `mapstructure:"secret"`
    Expire int    `mapstructure:"expire"`
}

var AppConfig *Config

func LoadConfig() error {
    viper.SetConfigName("config")
    viper.SetConfigType("yaml")
    viper.AddConfigPath("./config")

    if err := viper.ReadInConfig(); err != nil {
        return fmt.Errorf("读取配置文件失败: %w", err)
    }

    AppConfig = &Config{}
    if err := viper.Unmarshal(AppConfig); err != nil {
        return fmt.Errorf("解析配置失败: %w", err)
    }

    return nil
}
```

### 在main.go中使用

```go
func main() {
    if err := config.LoadConfig(); err != nil {
        panic(err)
    }

    gin.SetMode(config.AppConfig.Server.Mode)

    r := gin.Default()
    r.Run(fmt.Sprintf(":%d", config.AppConfig.Server.Port))
}
```

### 环境变量覆盖

Viper支持用环境变量覆盖配置文件中的值：

```go
viper.AutomaticEnv()
viper.SetEnvPrefix("MYWEB")
viper.BindEnv("database.password", "MYWEB_DB_PASSWORD")
```

这样你可以把敏感信息（数据库密码、API密钥）通过环境变量注入，不写到配置文件里。

## 72.5 依赖注入：Wire入门

当你有了Handler→Service→Repository三层结构，创建对象变得繁琐：

（Wire是进阶主题，初学者可先手动注入依赖，等项目变大后再回来学习）

```go
db, _ := sql.Open("mysql", dsn)

userRepo := repository.NewUserRepository(db)
userSvc := service.NewUserService(userRepo)
userHandler := handler.NewUserHandler(userSvc)

articleRepo := repository.NewArticleRepository(db)
articleSvc := service.NewArticleService(articleRepo)
articleHandler := handler.NewArticleHandler(articleSvc)
```

当项目有20个Service和20个Handler时，这堆初始化代码就变成了一团乱麻。**依赖注入**（Dependency Injection，DI）解决的就是这个问题——你只声明"我需要什么"，由框架自动帮你组装。

### Google Wire

Wire是Google推出的Go依赖注入代码生成器。它**不是在运行时反射注入**，而是在编译期生成初始化代码。

安装：

```bash
go install github.com/google/wire/cmd/wire@latest
```

### Provider定义

`internal/wire/provider.go`：

```go
package wire

import (
    "database/sql"

    "github.com/google/wire"
)

var DBProvider = wire.NewSet(NewDB)

func NewDB() *sql.DB {
    db, err := sql.Open("mysql", "...")
    if err != nil {
        panic(err)
    }
    return db
}
```

### Injector定义

`internal/wire/wire.go`：

```go
//go:build wireinject
// +build wireinject

package wire

import (
    "database/sql"

    "github.com/google/wire"
    "myweb/internal/handler"
    "myweb/internal/repository"
    "myweb/internal/service"
)

func InitializeUserHandler(db *sql.DB) *handler.UserHandler {
    wire.Build(
        repository.NewUserRepository,
        service.NewUserService,
        handler.NewUserHandler,
    )
    return nil
}
```

然后在项目根目录运行 `wire`，它会自动生成 `wire_gen.go` 文件，包含完整的初始化代码。你只需要在 `main.go` 中调用：

```go
userHandler := wire.InitializeUserHandler(db)
```

Wire的价值在大型项目中才体现——当依赖图复杂到几十个节点时，手工维护初始化顺序会非常痛苦，Wire帮你自动化这个过程。

## 🤔 想多一点：过度设计的陷阱

### 1. 不要为一周写不完500行的项目搞四层架构

如果你的项目只是一个简单的CRUD API，总共不到2000行代码，直接在handler里操作数据库也是可以的。分层架构是为了管理复杂度，不是为了看起来"专业"。**代码量<2000行，一个 `main.go` + 几个文件就够了。**

### 2. Handler→Service→Repository不是铁律

如果某个Handler的逻辑足够简单（比如直接返回一串常量数据），不需要Service层：

```go
r.GET("/health", func(c *gin.Context) {
    c.JSON(200, gin.H{"status": "ok"})
})
```

强行给每个Handler配一个Service是教条主义。

### 3. 不要过早引入Wire

Wire的学习曲线不低。当你的依赖关系少于10个对象时，手工初始化比Wire更清晰。只有当你有15个以上的Service并且相互依赖复杂时，才考虑Wire。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| MVC | Model（数据）+ View（界面）+ Controller（控制流） |
| 分层架构 | Handler → Service → Repository → Model |
| Handler | 处理HTTP请求/响应，参数校验，调Service |
| Service | 业务逻辑，调Repository，管理事务 |
| Repository | 数据库CRUD，不写业务逻辑 |
| Model | 数据结构定义（struct + tag） |
| 目录结构 | `cmd/`入口 + `internal/`私有 + `pkg/`公共 + `config/`配置 + `routes/`路由 |
| Viper | 读取YAML/JSON/ENV配置，`mapstructure` tag映射 |
| Viper优先级 | 代码设置 > 环境变量 > 配置文件 > 默认值 |
| Wire | 编译期依赖注入，`wire.Build` + `wire.NewSet` |
| 避免过度设计 | <2000行代码不需要四层架构 |
