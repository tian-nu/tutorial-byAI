# 118-性能测试

> 💡 你的接口在本地测试，10 个用户时响应 50ms。上线后 1000 个用户同时访问——响应变成了 5 秒。性能问题往往不是代码写得不对，而是代码在"高并发"下撑不住。本章教你用 JMeter 和 Spring Boot Actuator 找到瓶颈，并给出常见的优化手段。

---

## 本章目标
- 使用 Spring Boot Actuator 监控应用运行状态
- 使用 JMeter 进行压力测试
- 解读性能指标（QPS、响应时间、错误率）
- 掌握常见性能优化手段

## 118.1 Spring Boot Actuator——应用体检报告

### 添加依赖

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

### 配置

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus
  endpoint:
    health:
      show-details: always
```

### 常用端点

| 端点 | 路径 | 作用 |
|------|------|------|
| health | `/actuator/health` | 健康检查 |
| info | `/actuator/info` | 应用信息 |
| metrics | `/actuator/metrics` | 所有指标列表 |
| metrics/http.server.requests | `/actuator/metrics/http.server.requests` | HTTP 请求统计 |
| metrics/jvm.memory.used | `/actuator/metrics/jvm.memory.used` | JVM 内存使用 |
| threaddump | `/actuator/threaddump` | 线程快照（找死锁） |
| heapdump | `/actuator/heapdump` | 堆转储（分析内存泄漏） |

### 验证

```bash
curl http://localhost:8080/actuator/health
# {"status":"UP"}
```

## 118.2 JMeter——性能压测

### 安装

从 [jmeter.apache.org](https://jmeter.apache.org/download_jmeter.cgi) 下载，解压即可。进入 `bin/` 目录，运行：
- Windows：`jmeter.bat`
- Mac/Linux：`./jmeter`

### 创建第一个测试计划

```
Test Plan
└── Thread Group（线程组）
    ├── 线程数：100（模拟100个并发用户）
    ├── Ramp-up：10（10秒内逐步增加到100）
    ├── 循环次数：10（每个用户执行10次）
    └── HTTP Request（HTTP请求）
        ├── 协议：http
        ├── 服务器：localhost
        ├── 端口：8080
        └── 路径：/api/users
```

### 添加监听器

- **察看结果树**：看每个请求的响应内容
- **聚合报告**：看吞吐量、平均响应时间
- **图形结果**：看响应时间趋势

### 命令行运行（适合 CI/CD）

```bash
jmeter -n -t test-plan.jmx -l result.jtl -e -o report/
```

- `-n`：非 GUI 模式
- `-t`：测试计划文件
- `-l`：结果输出文件
- `-e -o`：生成 HTML 报告

## 118.3 解读性能指标

| 指标 | 含义 | 健康值 |
|------|------|--------|
| **QPS / TPS** | 每秒处理请求数 | 越高越好，取决于业务 |
| **Average RT** | 平均响应时间 | < 200ms（内部接口） |
| **P99 RT** | 99% 的请求响应时间 | < 1s |
| **Error Rate** | 错误率 | < 0.1% |
| **并发数** | 同时处理的请求数 | 不超线程池上限 |

### JMeter 聚合报告解读

```
Label           #Samples  Average  Min  Max   Error%  Throughput
/api/users      1000      45       10   500   0.00%   200.5/sec
/api/users/1    1000      80       20   2000  2.50%   100.2/sec
```

- `/api/users/1` 的最大响应时间 2000ms 和 2.5% 错误率——有问题！
- 可能是数据库查询慢或锁竞争。

## 118.4 常见性能瓶颈与优化

### 1. 数据库连接池耗尽

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000
```

- 打印当前连接数：`/actuator/metrics/hikari.connections.active`
- 活跃连接持续等于 `maximum-pool-size`：池太小，调大或排查慢 SQL

### 2. 慢 SQL

在 `application.yml` 中设置慢 SQL 阈值：

```yaml
spring:
  jpa:
    properties:
      hibernate:
        session:
          events:
            log:
              LOG_QUERIES_SLOWER_THAN_MS: 100
```

超过 100ms 的 SQL 会打印日志，然后：
- 检查是否缺少索引（`EXPLAIN`）
- 检查是否有 N+1 问题（用 `@EntityGraph` 或 `JOIN FETCH`）

### 3. Tomcat 线程池

```yaml
server:
  tomcat:
    threads:
      max: 200
      min-spare: 10
    accept-count: 100
    max-connections: 8192
```

### 4. 缓存

```java
@Cacheable(value = "users", key = "#id")
public User findById(Long id) {
    return userRepository.findById(id).orElseThrow();
}
```

### 5. 异步处理

```java
@Async
public CompletableFuture<Void> sendNotification(Long userId) {
    // 耗时操作，不阻塞主线程
}
```

需要 `@EnableAsync`。

## 118.5 压力测试最佳实践

1. **递增加压**：10 → 50 → 100 → 200，找到拐点（QPS 不再增长、错误率陡增）
2. **预热**：先跑少量请求让 JIT 编译生效
3. **独立环境**：不在生产环境压测，测试环境配置与生产一致
4. **监控全链路**：压测时同时看 CPU、内存、数据库连接数、GC 频率
5. **不要只看平均值**：P99 甚至 P999 更能反映用户体感

## 自测题

1. 压力测试时，平均响应时间 50ms 但 P99 是 3000ms，这说明什么问题？
2. Actuator 的 `/actuator/metrics/jvm.memory.used` 和 `/actuator/metrics/hikari.connections.active` 分别告诉你什么？
3. 你发现 QPS 在并发数到 50 后不再增长，可能的原因有哪些？

<details>
<summary>点击查看答案</summary>

1. 少数请求的响应时间极长（长尾延迟），可能是 GC 暂停、数据库锁等待、或慢 SQL 导致。大部分请求正常，但少量用户体感很差。
2. `jvm.memory.used` 是 JVM 当前使用的堆内存量，接近最大值说明可能内存不够或内存泄漏；`hikari.connections.active` 是当前正在使用的数据库连接数，接近 `maximum-pool-size` 说明连接池可能不够。
3. 可能原因：①数据库连接池耗尽（连接等待）；②Tomcat 线程池耗尽；③CPU 打满；④数据库出现锁竞争；⑤网络带宽瓶颈。
</details>
