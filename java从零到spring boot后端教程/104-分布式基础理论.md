## 104.1 CAP 定理

CAP 定理说：一个分布式系统**最多只能同时满足三项中的两项**。

```
          C（Consistency 一致性）
          /          \
         /            \
        /    CAP       \
       /                \
P（分区容错）─────────── A（可用性）
```

### C — Consistency（一致性）

所有节点在同一时刻看到的数据完全相同。

```
写操作 → 节点 A (x=1)
          │ 同步
          ▼
          节点 B (x=1)
          │ 同步
          ▼
          节点 C (x=1)

读操作 ← 无论读哪个节点，x 都是 1
```

### A — Availability（可用性）

每个请求都能收到一个非错误的响应（但不保证是最新数据）。

### P — Partition Tolerance（分区容错）

网络分区发生时（节点间通信中断），系统仍能继续运作。

### 为什么不能三个全要？

```
网络分区发生：
┌──────────┐     ✗ ✗ ✗     ┌──────────┐
│ 节点 A   │ ←──通信中断──→ │ 节点 B   │
│ 收到写请求│               │          │
└──────────┘               └──────────┘

你现在只能二选一：
- 选 C+A：拒绝写请求，直到网络恢复（但这个网络分区可能永远不恢复→系统不可用）
  但是网络分区客观存在，你无法避免——所以 P 是"无法选择不要"的
- 实际上你能选的：CP 还是 AP？
```

### 实际选择

| 选择 | 含义 | 典型系统 |
|------|------|----------|
| **CP** | 分区时牺牲可用性，保证一致性 | ZooKeeper、Etcd、HBase |
| **AP** | 分区时牺牲一致性，保证可用性 | Eureka、Cassandra |

> 🤔 想多一点：CAP 中的 C 是强一致性（所有节点数据完全一致）。现实中的系统更多追求**最终一致性**——允许短暂的不一致，但最终会一致。这就是 BASE 理论。

## 104.2 BASE 理论

BASE = BA + S + E：

| 缩写 | 含义 | 解释 |
|------|------|------|
| **BA** | Basically Available | 基本可用（允许响应变慢或降级） |
| **S** | Soft State | 软状态（允许系统存在中间状态） |
| **E** | Eventually Consistent | 最终一致性（不要求实时一致，但最终会一致） |

### ACID vs BASE

| | ACID（传统数据库） | BASE（分布式系统） |
|------|------|------|
| 核心追求 | 强一致性 | 高可用 |
| 一致性 | 强一致 | 最终一致 |
| 典型场景 | 银行转账 | 社交网络点赞 |
| 代价 | 性能差、难以横向扩展 | 开发复杂、需处理数据不一致 |

## 104.3 一致性 Hash

### 传统 Hash 的问题

```
3 台缓存服务器：
key "user:1001" → hash("user:1001") % 3 = 1 → 服务器 1
key "user:1002" → hash("user:1002") % 3 = 2 → 服务器 2

加一台服务器（3→4）：
key "user:1001" → hash("user:1001") % 4 = 2 → 服务器 2  ← 位置变了！
key "user:1002" → hash("user:1002") % 4 = 1 → 服务器 1  ← 也变了！
```

结果：几乎所有缓存都失效，瞬间全部打到数据库——**缓存雪崩**。

### 一致性 Hash 的解决

```
把整个 hash 空间看作一个环（0 ~ 2³²-1）：

        0
    ┌───────┐
    │       │  服务器 A: hash(A) = 1000
    │   A   │  服务器 B: hash(B) = 5000
    │       │  服务器 C: hash(C) = 9000
    │   B   │
    │       │  key 落在哪个区间，就归哪台服务器：
    │   C   │  1001 → A  (1000 < 1001 ≤ 5000)
    │       │  5001 → B  (5000 < 5001 ≤ 9000)
    └───────┘  9001 → C  (9000 < 9001 ≤ 1000，绕回)
```

**加一台服务器 D**：只影响相邻的一小段范围，其他 key 的映射不变。

## 104.4 分布式事务

微服务下，一个操作可能涉及多个数据库（订单库 + 库存库 + 账户库）。如何保证"下单扣库存扣钱"三个操作全成功或全失败？

### 方案一：两阶段提交（2PC）

```
准备阶段：
  Coordinator：你们能提交吗？
  Order DB：能（锁定数据）
  Inventory DB：能
  Account DB：能

提交阶段：
  Coordinator：全部提交！
  Order DB：提交 ✓
  Inventory DB：提交 ✓
  Account DB：提交 ✗ ← 这时已经晚了！
```

缺点：单点故障 + 阻塞 + 数据不一致。

### 方案二：TCC（Try-Confirm-Cancel）

```
Try（预留资源）：
  冻结库存 1 件
  冻结余额 100 元

Confirm（确认执行）：
  扣除库存 1 件
  扣除余额 100 元

Cancel（回滚）：
  解冻库存 1 件
  解冻余额 100 元
```

### 方案三：最终一致性（消息队列）

```
1. 订单服务：创建订单（状态=待支付）
2. 发消息到 MQ："请扣库存"
3. 库存服务：消费消息，扣库存。成功后发消息"库存已扣"
4. 支付服务：消费消息，扣款
5. 任何一步失败 → 发补偿消息回滚
```

这就是**可靠消息最终一致性**。虽然中途有短暂的不一致（库存扣了但钱没扣），但最终会通过补偿达到一致。

## 104.5 小结

| 知识点 | 核心内容 |
|--------|----------|
| CAP | 一致性 / 可用性 / 分区容错 — 最多同时满足两个 |
| BASE | 基本可用 + 软状态 + 最终一致性 |
| 一致性 Hash | 解决分布式缓存扩缩容时的雪崩问题 |
| 分布式事务 | 2PC / TCC / 最终一致性消息 |

## 104.6 自测题

**1. 你在设计一个分布式配置中心（如 ZooKeeper）。当网络分区发生时，你选 CP 还是 AP？为什么？**

**2. 缓存集群从 3 台扩到 4 台，为什么会导致缓存雪崩？一致性 Hash 怎么解决？**

**3. 电商下单流程涉及订单服务、库存服务、支付服务。为什么不用传统的 ACID 事务，而用最终一致性？**

---

**答案提示**：1→选 CP。配置中心的数据一致性比可用性更重要——错误配置导致的问题远比短暂不可用严重。ZooKeeper 本身就是 CP 系统。2→因为 `hash(key) % N` 在 N 变化时几乎所有 key 的映射都会变，缓存全部失效。一致性 Hash 用环结构，加节点只影响相邻范围。3→三个服务有各自的数据库，没法用一个数据库事务。且电商允许短暂不一致（如库存扣了但支付未完成，通过超时取消补偿），不需要强一致性锁死资源。下一章——负载均衡与反向代理。
