## 三、分步操作

### 步骤 1：需求场景——前端搜索框

你的博客前端有一个搜索框，用户输入 "python"，点击搜索。前端发一个请求：

```
GET /posts?keyword=python
```

后端拿到 `keyword=python`，从数据库中找出所有标题或内容包含 "python" 的文章，返回给前端。

**比喻**：搜索 = 图书馆的检索系统。你说"我想找跟 Python 相关的书"，图书管理员（后端）不会把所有书搬出来让你自己翻，而是去检索系统里查目录，只把匹配的书递给你。

---

### 步骤 2：最简单方案——`LIKE` 模糊匹配

在 SQL 中，"包含某个词"对应 `LIKE` 关键字：

```sql
SELECT * FROM posts
WHERE title LIKE '%python%';
```

**逐行解释**：
- `LIKE`：SQL 的模糊匹配关键字。`=` 是精确匹配，`LIKE` 是模糊匹配。
- `%`：通配符，代表"任意字符，任意长度"。`%python%` 的意思是"前面可以有任何字符，中间有 python，后面可以有任何字符"。
- `'%python%'`：匹配 `"python入门"`、`"学python"`、`"python"`、`"PYTHON教程"`（取决于数据库的大小写敏感性）。

**`LIKE` 的三种常见模式**：

| 模式 | 含义 | 示例匹配 |
|------|------|----------|
| `'%keyword%'` | 包含 keyword | `"python入门"`、`"学python"`、`"python"` |
| `'keyword%'` | 以 keyword 开头 | `"python入门"`、`"python"`，不匹配 `"学python"` |
| `'%keyword'` | 以 keyword 结尾 | `"学python"`、`"python"`，不匹配 `"python入门"` |

**本教程用 `'%keyword%'`**——只要标题或内容里出现过这个词，就算匹配。

**比喻**：`%` 就像扑克牌里的"癞子"——可以代替任何牌。`%python%` = "随便什么牌 + python + 随便什么牌"。

---

### 步骤 3：SQLAlchemy 层面——`contains()` 方法

SQLAlchemy 把 `LIKE '%keyword%'` 封装成了更 Pythonic 的方法：`.contains()`。

```python
# 搜索标题中包含 "python" 的文章
posts = db.query(Post).filter(Post.title.contains("python")).all()
```

**对应 SQL**：

```sql
SELECT * FROM posts
WHERE posts.title LIKE '%python%';
```

**`contains` vs `like`**：

| SQLAlchemy 方法 | 对应 SQL | 用法 |
|-----------------|----------|------|
| `.contains("x")` | `LIKE '%x%'` | 包含 x |
| `.startswith("x")` | `LIKE 'x%'` | 以 x 开头 |
| `.endswith("x")` | `LIKE '%x'` | 以 x 结尾 |
| `.like("x")` | `LIKE 'x'` | 原始模式，通配符自己写 |

**建议用 `contains`**：不用手动写 `%`，代码更清晰，不容易出错。

---

### 步骤 4：同时搜索标题和内容——`or_` 组合条件

用户的搜索需求通常是：**标题或内容中只要有一个匹配就行**。比如搜索 "python"，文章标题是 "Python 教程" 匹配，或者文章内容里写了 "python" 也匹配。

这需要 SQL 的 `OR` 逻辑：

```sql
SELECT * FROM posts
WHERE title LIKE '%python%'
   OR content LIKE '%python%';
```

SQLAlchemy 中用 `or_()` 来实现：

```python
from sqlalchemy import or_

posts = db.query(Post).filter(
    or_(
        Post.title.contains("python"),
        Post.content.contains("python"),
    )
).all()
```

**逐行解释**：
- `from sqlalchemy import or_`：导入 `or_` 函数。注意后面有下划线——因为 `or` 是 Python 关键字，不能直接当函数名。
- `or_(条件1, 条件2, ...)`：多个条件中任意一个满足就匹配。相当于 SQL 的 `WHERE 条件1 OR 条件2 OR ...`。
- `Post.title.contains("python")`：标题包含 "python"。
- `Post.content.contains("python")`：内容包含 "python"。

**比喻**：`or_` 像一个"或门"——只要有一个条件为真，总结果就是真。搜索 "python" 时，标题匹配了？（开门）内容匹配了？（开门）只要有一个开门，就算通过。

### 步骤 5：搜索 + 分页组合——`keyword` 和 `page` 一起用

用户搜索 "python" 并想看第 1 页，每页 5 条：

```
GET /posts?keyword=python&page=1&size=5
```

**后端逻辑**：

1. 如果 `keyword` 有值 → 加上 `filter` 条件，只查匹配的文章。
2. 如果 `keyword` 为空 → 不加 `filter`，查所有文章。
3. 分页逻辑不变——查 `total`、算 `skip`、`.offset().limit()`。

打开 `blog_backend/main.py`，改造 `get_all_posts`：

```python
# blog_backend/main.py —— 在文件顶部追加导入
from sqlalchemy import or_


@app.get("/posts", response_model=PaginatedResponse[BlogResponse])
def get_all_posts(
    page: int = 1,
    size: int = 10,
    keyword: str = "",  # 新增：搜索关键词
    db: Session = Depends(get_db),
):
    # 边界保护
    if page < 1:
        page = 1
    if size < 1:
        size = 1
    if size > 100:
        size = 100

    # 构建查询
    query = db.query(Post)

    # 如果有搜索关键词，加上过滤条件
    if keyword:
        query = query.filter(
            or_(
                Post.title.contains(keyword),
                Post.content.contains(keyword),
            )
        )

    # 查询总条数（在应用过滤条件之后）
    total = query.count()

    # 分页
    skip = (page - 1) * size
    posts = query.offset(skip).limit(size).all()
    pages = math.ceil(total / size) if total > 0 else 0

    return {
        "items": posts,
        "total": total,
        "page": page,
        "size": size,
        "pages": pages,
    }
```

**逐行解释**：

- `keyword: str = ""`：新增查询参数，默认空字符串。不传 `keyword` 时行为不变。
- `query = db.query(Post)`：先创建一个查询对象，不立即执行。
- `if keyword:`：Python 的空字符串 `""` 是 falsy 的，所以只有用户真正传了 `keyword` 才会加过滤条件。
- `query = query.filter(...)`：在查询对象上叠加过滤条件。注意我们重新赋值了 `query`——这是构建动态查询的关键技巧：**先创建基础查询，根据条件逐步叠加**。
- `total = query.count()`：`count()` 在过滤条件之后执行，所以统计的是**匹配搜索条件**的文章总数，而不是所有文章的总数。
- 分页逻辑完全不变——`skip`、`offset`、`limit` 都和教程 20 一模一样。

**关键设计**：`keyword` 为空时，`query` 没有被 `.filter()` 修改，行为 = 查询所有文章（和教程 20 完全兼容）。这就是"可选参数"的优雅之处——不加参数，功能不变。

**比喻**：`query` 就像一个"购物清单"，你可以一步步往上面加条件：
1. `query = db.query(Post)`：清单 = "我要文章"
2. `query = query.filter(...)`：清单 = "我要文章，且标题或内容包含 python"
3. `query = query.offset(skip).limit(size)`：清单 = "我要文章，且标题或内容包含 python，且只要第 2 页的 10 条"
4. `.all()`：拿着清单去仓库取货。

### 步骤 7：性能提示——`LIKE '%keyword%'` 为什么慢？

你可能会想：搜索搞定了，就这么简单？是的，功能上就这么简单。但有一个重要的性能问题你需要知道。

**`LIKE '%keyword%'` 无法使用数据库索引。** 此术语需进附录：索引。

**为什么？** 数据库索引就像书的目录——目录按"第一个字"排序，所以你可以快速找到"以 P 开头"的章节。但 `LIKE '%keyword%'` 问的是"哪些章节里出现过 keyword？"，这要求数据库扫描每一页、每一行，看看里面有没有这个关键词。

**比喻**：
- `LIKE 'python%'`（以 python 开头）= 翻到目录的 P 部分，找到 "Python" → 快（能用索引）。
- `LIKE '%python%'`（包含 python）= 把整本书从头到尾读一遍，标记所有出现 "python" 的地方 → 慢（全表扫描）。

**数据量对比**：

| 数据量 | `LIKE '%keyword%'` 耗时 | 用户体验 |
|--------|------------------------|----------|
| 100 条 | ~1ms | 无感 ✅ |
| 1 万条 | ~50ms | 稍有延迟，可接受 |
| 10 万条 | ~500ms | 明显卡顿 ⚠️ |
| 100 万条 | ~5 秒 | 无法接受 ❌ |

**对于我们的博客系统**，几百到几千篇文章时，`LIKE '%keyword%'` 完全够用。但如果文章量增长到万级以上，你需要升级搜索方案。

### 步骤 8：🤔 想多一点——进阶搜索方案预告

当你有了 10 万+ 篇文章，搜索变慢时，有两条升级路线：

| 方案 | 原理 | 适合场景 | 学习成本 |
|------|------|----------|----------|
| SQLite FTS5 | SQLite 自带的全文检索引擎，对文本建立倒排索引，搜索速度极快 | SQLite 项目，数据量在百万级以内 | 低（几行 SQL） |
| Elasticsearch | 专业的搜索引擎，独立部署，支持中文分词、高亮、权重排序等高级功能 | 大型项目，需要复杂搜索功能 | 高（需要学习新工具） |

**倒排索引**（FTS5 和 Elasticsearch 的核心）：传统索引是"文章 → 关键词"，倒排索引是"关键词 → 哪些文章"。比如：
- 传统：文章 1 → 包含 [python, 入门, 教程]
- 倒排：python → [文章 1, 文章 3, 文章 5]；入门 → [文章 1, 文章 7]

搜索 "python" 时，直接查倒排索引 → 找到 [1, 3, 5] → 快速返回。不需要扫描文章内容。

**这些留给进阶教程**。现在你只需要知道：`LIKE '%keyword%'` 是搜索的"入门版"，够用但不够快。以后项目变大了，记得回来升级。

## 六、术语附录

| 术语 | 英文 | 通俗解释 | 本章出现位置 |
|------|------|----------|-------------|
| LIKE | — | SQL 的模糊匹配关键字。`LIKE '%x%'` = 包含 x，`LIKE 'x%'` = 以 x 开头。和 `=` 的区别：`=` 是精确匹配，`LIKE` 是模糊匹配。 | 步骤 2 |
| `%` 通配符 | Wildcard | SQL `LIKE` 中的特殊字符，代表"任意字符，任意长度"。`%python%` = 前面随便 + python + 后面随便。 | 步骤 2 |
| `contains()` | — | SQLAlchemy 的方法，对应 SQL 的 `LIKE '%x%'`。不用手动写 `%` 通配符。 | 步骤 3 |
| `or_()` | — | SQLAlchemy 的 OR 组合函数。`or_(条件1, 条件2)` = 条件1 或 条件2 满足即可。注意后面有下划线（`or` 是 Python 关键字）。 | 步骤 4 |
| 全表扫描 | Full Table Scan | 数据库逐行检查表中所有数据，没有索引可用。`LIKE '%keyword%'` 会导致全表扫描。 | 步骤 7 |
| 索引 | Index | 数据库的"目录"，用于加速查询。`LIKE '%keyword%'` 无法使用索引，因为通配符在开头。 | 步骤 7 |
| 全文索引 | Full-Text Index | 专门为文本搜索优化的索引，用倒排索引实现。SQLite 的叫 FTS5，MySQL 的叫 FULLTEXT。 | 步骤 8 |
| FTS5 | — | SQLite 自带的全文检索引擎，支持倒排索引、布尔搜索、前缀查询等。几行 SQL 就能用。 | 步骤 8 |
| Elasticsearch | — | 专业的搜索引擎，独立部署，支持中文分词、高亮、权重排序等高级功能。适合大型项目。 | 步骤 8 |
| 倒排索引 | Inverted Index | "关键词 → 哪些文章"的索引结构。搜索时直接查关键词对应的文章 ID 列表，不需要扫描正文。 | 步骤 8 |

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| SQLite 大小写敏感 | 搜索 "python" 找不到 "Python" | SQLite 的 `LIKE` 默认对 ASCII 大小写不敏感，但对 Unicode 可能敏感 | 可以用 `func.lower()` 统一转小写：`func.lower(Post.title).contains(keyword.lower())` |
| 搜索空字符串 | 返回所有文章（因为 `LIKE '%%'` 匹配一切） | `keyword=""` 时 `contains("")` 等价于 `LIKE '%%'` | 用 `if keyword:` 判断，空字符串时跳过 `filter` |
| `or_` 写成 `or` | 报错 `SyntaxError` | `or` 是 Python 关键字，不能当函数名 | 导入 `from sqlalchemy import or_`，注意下划线 |
| 忘记 `count()` 在过滤之后 | `total` 显示的是所有文章的总数，而不是搜索结果的总数 | 先 `count()` 再加 `filter` | 确保 `total = query.count()` 在 `if keyword: query = query.filter(...)` 之后执行 |
| 搜索特殊字符 | 用户输入 `%` 或 `_` 导致意外行为 | `%` 和 `_` 是 SQL LIKE 的通配符，用户输入 `%` 会匹配所有文章 | 生产环境考虑用全文搜索（FTS5）代替 `LIKE`，或手动转义通配符 |
