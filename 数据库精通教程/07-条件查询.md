## 我在做什么？

上一章你学会了 SELECT——从表中取数据。但现实需求很少是"把所有数据取出来"，更多时候是"取**满足某些条件**的数据"。

比如："邮箱是 Gmail 的用户"、"订单金额大于 5000 的"、"用户名以 'a' 开头的"。这些"条件"就是 `WHERE` *此术语见附录E* 要做的事。

这一章，我们学 WHERE 的各种过滤方式——比较、逻辑、范围、模糊匹配。尤其是**NULL 的陷阱**，这是 SQL 初学者绕不开的坑。

学完这一章，你可以像 Google 搜索那样精确地筛选数据库中的数据。

## 一、比较运算符：最简单的条件

WHERE 子句的基本结构：

```sql
SELECT 列名 FROM 表名 WHERE 条件;
```

### 1.1 等于和不等于

```sql
-- 查找 id 为 1 的用户
SELECT * FROM users WHERE id = 1;

-- 查找所有不是 Gmail 的用户
SELECT * FROM users WHERE email NOT LIKE '%@gmail.com';
```

### 1.2 大小比较

```sql
-- 订单金额大于 5000 的
SELECT * FROM orders WHERE amount > 5000;

-- 订单金额小于等于 3000 的
SELECT * FROM orders WHERE amount <= 3000;
```

### 1.3 不等于

```sql
-- 查找 id 不为 1 的用户
SELECT * FROM users WHERE id != 1;
-- 或
SELECT * FROM users WHERE id <> 1;  -- 两种写法等价，<> 是 SQL 标准
```

### 1.4 全部比较运算符一览

| 运算符 | 含义 | 示例 |
|--------|------|------|
| `=` | 等于 | `WHERE id = 1` |
| `!=` 或 `<>` | 不等于 | `WHERE id != 1` |
| `>` | 大于 | `WHERE amount > 5000` |
| `<` | 小于 | `WHERE amount < 3000` |
| `>=` | 大于等于 | `WHERE amount >= 5000` |
| `<=` | 小于等于 | `WHERE amount <= 3000` |

## 二、逻辑运算符：组合多个条件

### 2.1 AND：同时满足

```sql
-- 金额大于 5000 且用户 ID 为 1 的订单
SELECT * FROM orders
WHERE amount > 5000 AND user_id = 1;
```

### 2.2 OR：满足其一

```sql
-- 用户 ID 为 1 或 2 的订单
SELECT * FROM orders
WHERE user_id = 1 OR user_id = 2;
```

### 2.3 NOT：取反

```sql
-- 不是 Gmail 的用户
SELECT * FROM users
WHERE NOT email LIKE '%@gmail.com';
```

### 2.4 优先级陷阱：AND 优先于 OR！

这是 SQL 中最容易出错的陷阱之一。AND 的优先级高于 OR——就像数学中乘法优先于加法。

❌ 错误做法——没有括号：

```sql
-- 这段代码的本意："订单金额大于 5000，且用户是 1 或 2"
SELECT * FROM orders
WHERE amount > 5000 AND user_id = 1 OR user_id = 2;
```

实际执行逻辑：`(amount > 5000 AND user_id = 1) OR user_id = 2`

翻译成人话：**"金额大于 5000 且用户是 1" 的订单，或者 "用户是 2" 的所有订单**。

结果：用户 2 的**所有订单**（包括金额小于 5000 的）都被查出来了！这不是你想要的。

✅ 正确做法——用括号明确优先级：

```sql
-- 明确意图：金额大于 5000，且用户是 1 或 2
SELECT * FROM orders
WHERE amount > 5000 AND (user_id = 1 OR user_id = 2);
```

> **防呆技巧**：当你写 `WHERE ... AND ... OR ...` 时，**永远加括号**。不要依赖优先级规则，不要考验自己或同事的记性。

## 三、BETWEEN：范围查询

`BETWEEN` *此术语见附录E* 用来查询某个范围内的值。**它是包含边界的**——`BETWEEN a AND b` 等价于 `>= a AND <= b`。

```sql
-- 订单金额在 3000 到 10000 之间（含 3000 和 10000）
SELECT * FROM orders
WHERE amount BETWEEN 3000 AND 10000;
```

等价于：

```sql
SELECT * FROM orders
WHERE amount >= 3000 AND amount <= 10000;
```

### 日期范围

BETWEEN 对日期同样有效：

```sql
-- 注册时间在 2025 年 1 月 1 日到 2025 年 6 月 30 日之间的用户
SELECT * FROM users
WHERE created_at BETWEEN '2025-01-01' AND '2025-06-30';
```

> **注意**：BETWEEN 后面的两个值必须是"小值 AND 大值"——`BETWEEN 1 AND 10` 正确，`BETWEEN 10 AND 1` 不会报错但永远返回空结果（因为没有任何值同时满足 >= 10 且 <= 1）。

## 四、IN：多值匹配

`IN` *此术语见附录E* 用来检查一个值是否在给定列表中：

```sql
-- 查询城市为北京、上海、广州的用户
SELECT * FROM users
WHERE city IN ('北京', '上海', '广州');
```

等价于：

```sql
SELECT * FROM users
WHERE city = '北京' OR city = '上海' OR city = '广州';
```

但 IN 更简洁，尤其是在列表很长的时候。

### NOT IN

```sql
-- 排除某些城市
SELECT * FROM users
WHERE city NOT IN ('北京', '上海');
```

> **警告**：`NOT IN` 有一个陷阱——如果子查询的返回结果中包含 NULL，整个 `NOT IN` 条件会返回空！这个问题等到第 10 章讲子查询时再展开。

## 五、LIKE：模糊匹配

`LIKE` *此术语见附录E* 用来做模糊匹配——你不需要精确知道要找什么，只需要知道大概的样子。

### 5.1 通配符

| 通配符 | 含义 | 示例 |
|--------|------|------|
| `%` | 匹配**任意多个**字符（包括 0 个） | `'a%'` 匹配以 a 开头的所有字符串 |
| `_` | 匹配**恰好一个**字符 | `'a_'` 匹配 "ab"、"ac"，不匹配 "a" 或 "abc" |

### 5.2 常见用法

```sql
-- 邮箱以 @gmail.com 结尾的用户
SELECT * FROM users WHERE email LIKE '%@gmail.com';

-- 用户名以 'a' 开头
SELECT * FROM users WHERE username LIKE 'a%';

-- 用户名中包含 'li' 的
SELECT * FROM users WHERE username LIKE '%li%';

-- 用户名恰好 5 个字符
SELECT * FROM users WHERE username LIKE '_____';  -- 5 个下划线
```

### 5.3 LIKE 的性能问题

```sql
-- 这种写法很慢！
SELECT * FROM users WHERE username LIKE '%keyword%';
```

为什么慢？因为 `LIKE '%keyword%'` 以 `%` 开头，意味着数据库**无法使用索引**——它必须扫描每一行，检查用户名是否包含 "keyword"。

如果表有 1000 万行，这个查询可能需要几秒甚至几十秒。

> **这不是说不要用 LIKE**——而是要知道它的代价。如果你经常需要"搜索包含某个关键词"，应该考虑用**全文索引**（Full-Text Index），第 12 章和第 36 章会讲到。

## 六、NULL 的陷阱（重点）

这是本章最重要的部分。NULL 是 SQL 中最容易出错的概念，没有之一。

### 6.1 NULL 是什么？

`NULL` *此术语见附录E* 在数据库中表示"**未知**"或"**不存在**"。它不是 0，不是空字符串 `''`，不是 false——它是"我不知道"。

用生活比喻来理解：

| 值 | 含义 | 比喻 |
|----|------|------|
| 0 | 我知道，结果是 0 | 考试得了 0 分 |
| `''`（空字符串）| 我知道，结果是空的 | 交了一张白卷 |
| NULL | 我不知道 | 根本没来考试 |

### 6.2 `= NULL` 永远为假！

❌ 错误做法：

```sql
-- 想查 phone 为空的用户
SELECT * FROM users WHERE phone = NULL;
-- 结果：永远返回 0 行！即使表中有 phone 为 NULL 的行！
```

为什么？因为 NULL 表示"未知"。两个"未知"值能比较相等吗？不能——你不知道它们是不是同一个东西。

✅ 正确做法：

```sql
-- 查 phone 为空的用户
SELECT * FROM users WHERE phone IS NULL;

-- 查 phone 不为空的用户
SELECT * FROM users WHERE phone IS NOT NULL;
```

### 6.3 亲自验证：NULL 不等于任何值，包括它自己

```sql
-- 执行这个查询，看看结果
SELECT NULL = NULL;
```

输出：

```
+-------------+
| NULL = NULL |
+-------------+
|        NULL |
+-------------+
```

返回的不是 `1`（TRUE），也不是 `0`（FALSE），而是 `NULL`！

在 SQL 的三值逻辑中：
- `TRUE AND NULL` = `NULL`
- `FALSE AND NULL` = `FALSE`
- `TRUE OR NULL` = `TRUE`
- `FALSE OR NULL` = `NULL`

**任何与 NULL 的比较运算，结果都是 NULL（未知）。** 而 WHERE 子句只保留条件为 TRUE 的行，NULL 和 FALSE 的行都会被过滤掉。

### 6.4 NULL 的常见坑

**坑 1：NOT IN 遇到 NULL**

```sql
-- 假设有一行的 phone 是 NULL
SELECT * FROM users WHERE phone NOT IN ('1380001', '1380002');
-- 如果 phone 是 NULL 的行，不会出现在结果中！
-- 因为 NULL NOT IN (...) 的结果是 NULL，不是 TRUE
```

**坑 2：COUNT 忽略 NULL**

```sql
-- 如果 phone 列有 NULL 值
SELECT COUNT(phone) FROM users;
-- 返回的是非 NULL 的行数，不是总行数！
```

这和 `COUNT(*)` 不同——`COUNT(*)` 统计所有行（含 NULL），`COUNT(col)` 只统计该列非 NULL 的行。第 09 章会详细讲。

**坑 3：字符串拼接遇到 NULL**

```sql
SELECT CONCAT('Hello ', NULL, '!');
-- 返回 NULL！整个拼接结果都是 NULL
```

> **想多一点**：为什么数据库要搞出 NULL 这么麻烦的东西？因为"未知"是真实世界的一部分。一个人的手机号可能确实不知道（还没填），一个订单的取消时间可能确实不存在（还没取消）。如果用 0 或空字符串表示"没有"，那你怎么区分"手机号是空字符串"和"手机号未知"？NULL 解决的就是这个"语义区分"的问题。

## 我做得对不对？

### 验证方法

```sql
-- 1. 测试比较运算符：查找 id=1 的用户
SELECT * FROM users WHERE id = 1;
-- 应该只返回 alice 一行

-- 2. 测试 AND/OR 优先级：
-- 不计括号，执行以下查询
SELECT * FROM orders WHERE amount > 5000 AND user_id = 1 OR user_id = 2;
-- 然后加上括号再执行
SELECT * FROM orders WHERE amount > 5000 AND (user_id = 1 OR user_id = 2);
-- 对比两次结果，观察差异

-- 3. 测试 BETWEEN：查找金额 3000-10000 的订单
SELECT * FROM orders WHERE amount BETWEEN 3000 AND 10000;
-- 手动检查：iPhone 15 (6999) 和 MacBook Pro (12999) 中，只有 6999 在范围内

-- 4. 测试 NULL = NULL
SELECT NULL = NULL;
-- 返回 NULL，不是 1 也不是 0

-- 5. 测试 LIKE：查找 Gmail 用户
SELECT * FROM users WHERE email LIKE '%@gmail.com';
-- 应该只返回 bob
```

## 不对怎么办？

### 常见错误 1：`= NULL` 而不是 `IS NULL`

❌ 错误做法：

```sql
SELECT * FROM users WHERE phone = NULL;
-- 永远返回 0 行！
```

✅ 正确做法：

```sql
SELECT * FROM users WHERE phone IS NULL;
SELECT * FROM users WHERE phone IS NOT NULL;
```

**记住：NULL 不能和任何东西比较，包括它自己。一律用 IS NULL 和 IS NOT NULL。**

### 常见错误 2：AND 和 OR 优先级混淆

❌ 错误做法：

```sql
-- 意图：金额大于 5000 且（用户是 1 或 2）
SELECT * FROM orders
WHERE amount > 5000 AND user_id = 1 OR user_id = 2;
-- 实际逻辑：(amount > 5000 AND user_id = 1) OR user_id = 2
```

✅ 正确做法：

```sql
SELECT * FROM orders
WHERE amount > 5000 AND (user_id = 1 OR user_id = 2);
```

**混合使用 AND 和 OR 时，永远加括号。不要省那两个字符。**

### 常见错误 3：LIKE 通配符不分

❌ 错误做法：

```sql
-- 想找用户名恰好 3 个字符的
SELECT * FROM users WHERE username LIKE '___';  -- 对了，3 个下划线
-- 想找以 'a' 开头的
SELECT * FROM users WHERE username LIKE '_a%';  -- 错了！第二个字符是 a，不是第一个
```

✅ 正确做法：

- `%` = 任意多个字符（包括 0 个）
- `_` = 恰好一个字符
- `LIKE 'a%'` = 以 a 开头
- `LIKE '%a'` = 以 a 结尾
- `LIKE '%a%'` = 包含 a
- `LIKE 'a__'` = a 开头且总共 3 个字符

### 常见错误 4：字符串比较没加引号

❌ 错误做法：

```sql
SELECT * FROM users WHERE username = alice;  -- alice 没加引号
```

MySQL 报错：`ERROR 1054 (42S22): Unknown column 'alice' in 'where clause'`

✅ 正确做法：

```sql
SELECT * FROM users WHERE username = 'alice';
```

MySQL 把没加引号的东西当作列名。所以 `alice` 被当作"找 alice 这一列"，而不是"找值为 'alice' 的行"。

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 比较运算符 | `=`, `!=`/`<>`, `>`, `<`, `>=`, `<=` | 字符串值必须用单引号包裹 |
| 逻辑运算符 | `AND`, `OR`, `NOT` | AND 优先级高于 OR，混合使用时**必须加括号** |

| NULL 的三值逻辑 | `NULL = NULL` 返回 NULL | WHERE 只保留条件为 TRUE 的行，NULL 和 FALSE 都会被过滤 |
