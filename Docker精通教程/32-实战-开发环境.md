## 我在做什么？

你有没有经历过这种场景：新同事入职，第一天花半天装环境——Node.js、PostgreSQL、Redis、各种系统级依赖。然后发现版本不对，又花半天调试。最后终于跑起来了，但"在我机器上能跑啊"的噩梦才刚刚开始。

这一章教你用 Docker 解决这个问题的终极方案：**Dev Containers** *此术语见附录C*。你不再需要在新电脑上装任何运行时环境——只要装 VS Code 和 Docker，然后一个命令，**完整的开发环境就在容器里跑起来了**。改代码立刻生效，断点调试照常工作，团队里所有人的环境完全一致。

学完这一章，你能：
- 搭建一个包含 Node.js + PostgreSQL + Redis 的完整开发环境
- 用 VS Code Dev Containers 连接到容器里开发
- 理解 bind mount 在开发模式下的正确用法
- 给数据库预填充种子数据（seed data）
- 让团队新成员在 5 分钟内进入开发状态

## 一、为什么开发环境也要容器化？

### 传统方式的问题

```
新人入职第一天：
  09:00  安装 Node.js 18
  09:15  等等，项目用的是 Node 16，重装
  09:30  安装 PostgreSQL 14
  10:00  配置 pg_hba.conf，改密码
  10:30  安装 Redis
  10:45  装各种系统级依赖（libpq-dev、build-essential...）
  11:30  npm install 报错，缺 native 模块
  12:00  终于跑起来了，但测试挂了，因为数据库编码不对
  14:00  配环境变量
  16:00  开始写代码（第一天快结束了）
```

### 容器化之后

```
新人入职：
  09:00  安装 Docker Desktop
  09:15  git clone https://github.com/your-team/project.git
  09:16  cd project && docker compose up -d
  09:18  VS Code 弹出 "Reopen in Container" → 点一下
  09:19  开始写代码
```

**从半天到 5 分钟。** 这就是开发环境容器化的价值。

### 核心思想

```
传统方式                          容器化方式
┌──────────────────┐             ┌──────────────────┐
│ 每个开发者自己装环境  │             │ 环境定义在文件里    │
│ 版本不统一          │             │ 版本精确锁定        │
│ 配置靠口口相传      │             │ 配置在 compose.yml  │
│ "我机器上能跑"      │             │ 所有人环境一样      │
└──────────────────┘             └──────────────────┘
```

## 二、VS Code Dev Containers 简介

VS Code 有一个官方扩展叫 **Dev Containers**（以前叫 Remote - Containers）。它的工作原理：

```
你的 VS Code 窗口
┌──────────────────────────────────────────────┐
│  UI 界面（在宿主机上）                          │
│  ┌────────────────────────────────────────┐  │
│  │ 编辑器、终端、调试器、扩展...             │  │
│  │ 这些都在宿主机上运行                      │  │
│  └────────────────────────────────────────┘  │
│                    │                          │
│                    │ VS Code Server 协议       │
│                    ▼                          │
│  ┌────────────────────────────────────────┐  │
│  │ VS Code Server（在容器里）               │  │
│  │ 文件系统、终端进程、代码执行              │  │
│  │ 这些都在容器里运行                        │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

简单说：**你的编辑器界面在宿主机上，但所有代码执行、文件读写、终端命令都在容器里**。你感觉就像在本地开发一样，但实际上你的代码运行在一个完全隔离的 Docker 容器里。

#### backend/src/index.js

```javascript
const express = require('express');
const { Pool } = require('pg');
const app = express();
const PORT = process.env.PORT || 3000;

// PostgreSQL 连接池
const pool = new Pool({
  host: process.env.DB_HOST || 'localhost',
  port: process.env.DB_PORT || 5432,
  user: process.env.DB_USER || 'devuser',
  password: process.env.DB_PASSWORD || 'your_dev_password_here',
  database: process.env.DB_NAME || 'devdb',
});

// 健康检查
app.get('/health', async (req, res) => {
  try {
    await pool.query('SELECT 1');
    res.json({ status: 'OK', database: 'connected' });
  } catch (err) {
    res.status(500).json({ status: 'ERROR', database: 'disconnected', error: err.message });
  }
});

// 获取用户列表
app.get('/api/users', async (req, res) => {
  try {
    const { rows } = await pool.query('SELECT id, name, email, created_at FROM users ORDER BY id');
    res.json({ users: rows });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

app.listen(PORT, () => {
  console.log(`Dev server running on port ${PORT}`);
});
```

#### backend/Dockerfile

```dockerfile
FROM node:18-alpine

# 安装 nodemon 全局（开发环境需要热重载）
RUN npm install -g nodemon

WORKDIR /app

# 先拷 package.json
COPY package.json .

# 装全部依赖（包括 devDependencies）
RUN npm install

# 再拷源代码
COPY src/ ./src/

EXPOSE 3000

# 默认开发模式：nodemon 热重载
CMD ["nodemon", "src/index.js"]
```

注意：这个 Dockerfile 和之前的生产版不同——**nodemon 是全局安装的，CMD 默认用 nodemon**。这是开发环境 Dockerfile 的典型特征：把"开发体验"放在第一位。

#### compose.yml

```yaml
# compose.yml — 开发环境基础配置
services:
  # ===== Node.js 后端 =====
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    ports:
      - "3000:3000"
    environment:
      NODE_ENV: development
      DB_HOST: db
      DB_PORT: "5432"
      DB_USER: devuser
      DB_PASSWORD: your_dev_password_here
      DB_NAME: devdb
    volumes:
      # 关键：bind mount 源代码实现热重载
      - ./backend/src:/app/src
      # 排除 node_modules，不让宿主机的覆盖容器里的
      - /app/node_modules
    depends_on:
      db:
        condition: service_healthy
    networks:
      - dev-network

  # ===== PostgreSQL 数据库 =====
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: devuser
      POSTGRES_PASSWORD: your_dev_password_here
      POSTGRES_DB: devdb
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./db/seed.sql:/docker-entrypoint-initdb.d/seed.sql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U devuser -d devdb"]
      interval: 5s
      timeout: 3s
      retries: 5
      start_period: 10s
    networks:
      - dev-network

volumes:
  pgdata:

networks:
  dev-network:
    driver: bridge
```

#### docker-compose.override.yml（开发模式专属覆盖）

```yaml
# docker-compose.override.yml
# 这个文件会在 docker compose up 时自动被合并到 compose.yml
# 只在开发环境使用，CI/CD 和部署时忽略

services:
  backend:
    # 开发模式命令：nodemon 带 --legacy-watch（Docker 文件系统需要）
    command: nodemon --legacy-watch src/index.js
    environment:
      NODE_ENV: development
      DEBUG: "express:*"
```

> **注意**：`--legacy-watch` 是 nodemon 的一个参数。在 Docker 容器里，默认的文件监听机制（`inotify`）可能不触发于 bind mount 的文件变化。`--legacy-watch` 用轮询方式检测文件变化，虽然会多消耗一点 CPU，但在 Docker 容器里更可靠。


逐字段解释：

```
"name": "Node.js + PostgreSQL Dev"
    ↓ VS Code 左下角显示的环境名称

"dockerComposeFile": ["../compose.yml", "../docker-compose.override.yml"]
    ↓ 指定用哪个 compose 文件启动开发容器
    ↓ 两个文件都会被加载，override 里的设置会覆盖基础配置

"service": "backend"
    ↓ "我要连接到 compose 里的 backend 服务"
    ↓ VS Code 的终端和文件系统都在 backend 容器里

"workspaceFolder": "/app"
    ↓ 容器里打开的工作目录

"customizations": ...
    ↓ 自动安装的 VS Code 扩展和设置
    ↓ 团队每个人打开这个容器后，扩展配置完全一致

"forwardPorts": [3000, 5432]
    ↓ 自动转发这些端口到宿主机

"postCreateCommand": "npm install"
    ↓ 容器创建后自动执行的命令
    ↓ 确保 node_modules 装好了

"remoteUser": "node"
    ↓ 容器里以哪个用户身份运行 VS Code Server
    ↓ node:18-alpine 镜像自带 node 用户，避免 root 权限问题
```

```bash
# 查看日志
```

2. 打开 `dev-env-project/` 目录

6. 打开终端（`` Ctrl+` ``），你会发现已经在容器里了：`node@container-id:/app$`

## 六、不对怎么办？

### 常见错误 1：bind mount 覆盖了容器里的 `node_modules`

```yaml
# ❌ 挂载整个 backend 目录
volumes:
  - ./backend:/app
```

宿主机的 `backend/` 里没有 `node_modules`（或者有但版本不对），挂到容器里后，容器里 `npm install` 装好的 `node_modules` 被覆盖了。

```yaml
# ✅ 挂载 src/ 源码，排除 node_modules
volumes:
  - ./backend/src:/app/src
  - /app/node_modules    # 这个匿名卷保护容器里的 node_modules
```

`/app/node_modules` 这个匿名卷的作用：告诉 Docker "这个目录不要被 bind mount 覆盖，保留容器里的内容"。它就像一个"保护罩"，让 bind mount 绕开 `node_modules`。

### 常见错误 2：nodemon 在 Docker 里不检测文件变化

```bash
# 改了代码，nodemon 没反应
```

❌ 原因：Docker 容器里的 bind mount 文件系统，不一定触发 Linux 的 `inotify` 事件。nodemon 默认用 `inotify` 监听文件变化，没收到事件就不重启。

✅ 解决：加 `--legacy-watch` 参数，让 nodemon 用轮询方式检测：

```yaml
# docker-compose.override.yml
services:
  backend:
    command: nodemon --legacy-watch src/index.js
```

或者在 Dockerfile 里：

```dockerfile
CMD ["nodemon", "--legacy-watch", "src/index.js"]
```

### 常见错误 3：Windows 路径问题

```yaml
# ❌ Windows 绝对路径写法不正确
volumes:
  - C:\Users\YourName\project\backend\src:/app/src
```

在 compose.yml 里，Windows 路径需要特别注意：

```yaml
# ✅ 用相对路径（推荐）
volumes:
  - ./backend/src:/app/src

# ✅ 如果必须用绝对路径，转义反斜杠
volumes:
  - C:\\Users\\YourName\\project\\backend\\src:/app/src
```

**推荐始终用相对路径**——相对路径没有跨平台兼容性问题。

### 常见错误 4：Dev Containers 找不到 compose 文件

```json
// ❌ 路径错误
{
  "dockerComposeFile": "compose.yml"   // 写成了相对 .devcontainer/ 的路径
}
```

`devcontainer.json` 里的 `dockerComposeFile` 的路径是**相对于 .devcontainer 目录**的。如果 compose.yml 在项目根目录，需要写 `../compose.yml`：

```json
// ✅
{
  "dockerComposeFile": [
    "../compose.yml",
    "../docker-compose.override.yml"
  ]
}
```

### 常见错误 5：PostgreSQL 健康检查命令写错

```yaml
# ❌ pg_isready 参数不对
healthcheck:
  test: ["CMD", "pg_isready"]
```

`pg_isready` 不带参数时检查默认数据库和用户，可能不匹配。

```yaml
# ✅ 明确指定用户和数据库
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U devuser -d devdb"]
```

注意：PostgreSQL 的 healthcheck 用 `CMD-SHELL`（而不是 `CMD`），因为 `pg_isready` 需要 shell 来解析参数。

### 常见错误 6：`docker compose up` 后端口被占用

```bash
# Error: port is already allocated
```

如果 5432 端口被本地 PostgreSQL 占用：

```yaml
# 换一个宿主机端口
services:
  db:
    ports:
      - "5433:5432"    # 宿主机 5433 → 容器 5432
```

但要相应更新 `DB_PORT` 环境变量。或者，如果你只在容器间通信（不需要宿主机直连数据库），可以去掉 `ports` 映射——容器之间的网络不受 `ports` 影响。

## 七、深入思考：团队开发环境的最佳实践

### 1. 为什么用 compose.override.yml？

`docker-compose.override.yml` 是 Docker Compose 的约定文件名。当你运行 `docker compose up` 时，Compose 会自动合并 `compose.yml` 和 `docker-compose.override.yml`（如果存在）。

这个机制的好处：

```
compose.yml                    compose.override.yml
（基础配置，所有人共享）          （本地覆盖，每个人自定义）
├── 服务定义                     ├── 开发用端口
├── 数据库配置                   ├── 调试标志
├── 网络设置                     ├── 本地环境变量
└── 提交到 Git ✅                └── 加入 .gitignore ✅
```

把通用的、团队共享的配置放 `compose.yml`，把个人的、本地的覆盖放 `compose.override.yml`（加入 `.gitignore`）。这样既有统一标准，又有个人灵活性。

### 2. 数据库种子数据的管理

开发环境需要种子数据来测试功能。`docker-entrypoint-initdb.d/` 里的 SQL 文件只在**数据库首次创建时**执行。如果数据库卷已经存在，不会重新执行。

如果你需要重置种子数据：

```bash
# 删除数据卷，重新初始化
docker compose down -v
docker compose up -d
```

### 3. Git 忽略什么？

```gitignore
# .gitignore
.env
docker-compose.override.yml
node_modules/
```

`compose.yml` 和 `Dockerfile` 要提交——它们是"环境的定义"，是团队的共识。`compose.override.yml` 不提交——它是"个人的定制"，每个人可能不同。

### backend/src/index.js

```javascript
const express = require('express');
const { Pool } = require('pg');
const app = express();
const PORT = process.env.PORT || 3000;

const pool = new Pool({
  host: process.env.DB_HOST || 'localhost',
  port: process.env.DB_PORT || 5432,
  user: process.env.DB_USER || 'devuser',
  password: process.env.DB_PASSWORD || 'your_dev_password_here',
  database: process.env.DB_NAME || 'devdb',
});

app.get('/health', async (req, res) => {
  try {
    await pool.query('SELECT 1');
    res.json({ status: 'OK', database: 'connected' });
  } catch (err) {
    res.status(500).json({ status: 'ERROR', database: 'disconnected', error: err.message });
  }
});

app.get('/api/users', async (req, res) => {
  try {
    const { rows } = await pool.query('SELECT id, name, email, created_at FROM users ORDER BY id');
    res.json({ users: rows });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

app.listen(PORT, () => {
  console.log(`Dev server running on port ${PORT}`);
});
```

## 本篇最可能出错的地方及原因

### 1. Bind mount 覆盖了容器里的 `node_modules`（最高频）

**这是开发环境容器化最经典的错误。** 用 `- ./backend:/app` 把整个项目目录挂进去，宿主机上没装 `node_modules`，容器里的也没了。

**解决方案的核心是"匿名卷保护"**：

```yaml
volumes:
  - ./backend/src:/app/src       # 只挂源码目录
  - /app/node_modules             # 匿名卷：保护容器里的 node_modules
```

这个匿名卷是 Docker 的一个精巧设计：它告诉 Docker "这个路径的内容不要被 bind mount 覆盖，保留镜像里的原始内容"。相当于给 `node_modules` 加了一个"安全气囊"。

### 2. nodemon 在 Docker 里不检测文件变化

**根因**：nodemon 默认使用 `inotify`（Linux 内核的文件事件监控）来检测文件变化。但在 Docker 的 bind mount 文件系统上，`inotify` 事件传递可能不可靠——尤其是 macOS 和 Windows 上的 Docker Desktop，因为文件系统经过了虚拟化层。

**解决方案**：`--legacy-watch` 让 nodemon 不去依赖 `inotify`，而是每隔一段时间主动扫描文件。虽然会多消耗一点 CPU，但在 Docker 开发中是最可靠的方案。

```yaml
command: nodemon --legacy-watch src/index.js
```

### 3. `devcontainer.json` 的路径问题

`.devcontainer/` 目录的位置和 `dockerComposeFile` 的路径是新手最容易搞混的：

- `.devcontainer/` 必须在项目根目录下
- `dockerComposeFile` 的路径相对于 `.devcontainer/` 目录
- 如果 compose.yml 在项目根目录，路径是 `../compose.yml`

### 4. PostgreSQL `pg_isready` 健康检查命令不生效

PostgreSQL 的 `pg_isready` 和 MySQL 的 `mysqladmin ping` 不同——它需要明确指定用户和数据库。不带参数时可能返回错误的状态码。

```yaml
# ❌ 可能不工作
healthcheck:
  test: ["CMD", "pg_isready"]

# ✅ 明确指定
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U devuser -d devdb"]
```

另外注意：PostgreSQL 的 healthcheck 用 `CMD-SHELL`，因为参数需要 shell 解析。而 MySQL 的 `mysqladmin ping` 用 `CMD` 数组格式就够了。

### 5. 种子数据只在首次创建时执行

`docker-entrypoint-initdb.d/` 里的 SQL 文件只在数据库**数据目录为空**时执行。如果你改了 `seed.sql`，重新 `docker compose up` 不会重新执行——因为数据卷里已经有数据了。

**要重新执行种子数据**：`docker compose down -v && docker compose up -d`（`-v` 删除数据卷）。
