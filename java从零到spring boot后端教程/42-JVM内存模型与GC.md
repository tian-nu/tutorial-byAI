# 第42章 JVM 内存模型与 GC

> ⚠️ **本章为选读章节**。JVM 内存模型和垃圾回收是面试重灾区，也是深入理解 Java 的必经之路。但对于刚开始学 Java 后端开发的你来说，可以先跳到第 43 章学注解反射，以及后续的 Spring Boot 实战。本章适合在需要排查内存问题、准备面试、或项目中遇到 OOM/频繁 Full GC 时回头精读。

> 你写的每一行 Java 代码，都在 JVM 这片土地上运行。JVM 帮你管理内存、回收垃圾——就像有一个隐形的清洁工默默打扫房间。但有时候他扫得太勤影响你工作（频繁 GC 导致 STW），有时候他又忘了扫（内存泄漏）。本章教你读懂这个清洁工的工作方式。

## 42.1 JVM 运行时数据区

> 🏷️ 此术语需进附录：**JVM 内存模型**——JVM 在运行时将内存划分为多个区域：堆、方法区、虚拟机栈、本地方法栈、程序计数器。其中堆和方法区是线程共享的，其他是线程私有的。

```
JVM 运行时数据区
├── 线程共享区域
│   ├── 堆（Heap）          ← 存放对象实例（GC 主战场）
│   └── 方法区（Method Area） ← 类信息、常量、静态变量（Java 8 后改为元空间）
│
└── 线程私有区域（每线程一份）
    ├── 虚拟机栈（VM Stack）  ← 方法调用的栈帧（局部变量、操作数栈等）
    ├── 本地方法栈（Native Stack） ← native 方法调用
    └── 程序计数器（PC Register）  ← 当前线程执行的字节码行号
```

## 42.2 堆（Heap）—— GC 的主战场

> 🏷️ 此术语需进附录：**堆**（Heap）——JVM 中最大的一块内存区域，所有对象实例和数组都在此分配。堆是 GC 管理的核心区域，分为新生代和老年代。

### 42.2.1 堆的分代结构

```
┌─────────────────────────────────────────────┐
│                    Heap                     │
│  ┌───────────────────────┐  ┌────────────┐ │
│  │      新生代 (Young)    │  │ 老年代(Old)│ │
│  │  ┌─────┬─────┬─────┐  │  │            │ │
│  │  │Eden │ S0  │ S1  │  │  │            │ │
│  │  │ 8/10│ 1/10│ 1/10│  │  │  存放长期  │ │
│  │  └─────┴─────┴─────┘  │  │  存活对象  │ │
│  │    新对象诞生地        │  │            │ │
│  └───────────────────────┘  └────────────┘ │
└─────────────────────────────────────────────┘
```

| 区域 | 说明 | 默认比例 |
|------|------|---------|
| **Eden**（伊甸园） | 绝大多数新对象在此分配 | 新生代的 80% |
| **S0 (Survivor 0)** | 存活区 0（From 区） | 新生代的 10% |
| **S1 (Survivor 1)** | 存活区 1（To 区） | 新生代的 10% |
| **老年代 (Old/Tenured)** | 长期存活的对象 | 堆的剩余部分 |

> 🏷️ 此术语需进附录：**GC 分代假说**——"弱分代假说"认为大多数对象朝生夕死（IBM 研究：98% 的对象在第一次 Minor GC 时死亡）。分代 GC 正是基于这个假说设计的：新生代频繁回收短期对象，老年代偶尔回收长期对象。

### 42.2.2 对象分配与晋升过程

```
1. 新对象 → 优先在 Eden 分配
2. Eden 满 → Minor GC
3. 存活对象 → 复制到 Survivor 区（S0 ↔ S1 轮流做 From/To）
4. 每熬过一次 Minor GC，对象年龄 +1
5. 年龄 ≥ 15（默认）→ 晋升到老年代（Promotion）
6. 老年代满 → Full GC（或 Major GC）
```

## 42.3 GC 算法与收集器

### 42.3.1 三种基础 GC 算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **标记-清除** | 标记存活对象 → 清除未标记的 | 简单 | 产生内存碎片 |
| **标记-复制** | 将存活对象复制到新区域 → 清空旧区域 | 无碎片，适合新生代 | 浪费一半空间 |
| **标记-整理** | 标记存活对象 → 移动到一端 → 清除边界外 | 无碎片 | 移动成本高，STW 时间长 |

```
标记-复制（新生代 Minor GC 使用）：
GC前: Eden [A B C D E F]  S0(From) [G H]  S1(To) [空]
              ↓ Minor GC：标记存活对象如 A、C、G
GC后: Eden [空]           S0(To) [空]      S1(From) [A C G]
      然后是下一轮：S0 和 S1 角色互换
```

### 42.3.2 常见 GC 名词

> 🏷️ 此术语需进附录（以下术语均需进附录）：

| 术语 | 全称 | 含义 |
|------|------|------|
| **Minor GC** | — | 新生代 GC，只回收 Eden + Survivor，频繁但快速 |
| **Major GC** | — | 老年代 GC（通常等价于 Full GC） |
| **Full GC** | — | 整个堆 + 方法区的 GC，STW 最长，最影响性能 |
| **STW** | Stop The World | GC 时暂停所有用户线程 |
| **G1 GC** | Garbage First | 区域化 GC，可预测停顿，Java 9+ 默认 |
| **ZGC** | Z Garbage Collector | 超低延迟 GC，目标停顿 < 1ms，支持 TB 级堆 |

### 42.3.3 G1 GC（Java 9+ 默认）

G1 不再按"新生代/老年代"物理划分，而是把堆分成多个大小相等的 **Region**：

```
G1 的 Region 布局：
┌───┬───┬───┬───┬───┬───┬───┬───┐
│ E │ S │ O │ E │ O │ E │ H │ O │   E=Eden, S=Survivor, O=Old, H=Humongous
└───┴───┴───┴───┴───┴───┴───┴───┘
 每个 Region 大小相同（1MB-32MB），角色可以动态切换
```

G1 的核心优势：**可预测的停顿时间**。通过 `-XX:MaxGCPauseMillis=200` 设定目标停顿时间，G1 优先回收垃圾最多的 Region（这就是 Garbage First 名字的由来）。

### 42.3.4 ZGC（Java 21 的生产力利器）

ZGC 是 JDK 11 引入的实验性 GC，JDK 15 正式发布，JDK 21 已非常成熟：

- 目标：停顿时间 < 1ms，堆大小 8MB ~ 16TB
- 核心技术：**染色指针（Colored Pointers）** + **读屏障（Load Barrier）**
- 几乎所有 GC 工作与用户线程**并发**执行
- 适用场景：大内存、低延迟应用（交易系统、实时服务）

```
各 GC 停顿时间对比（参考）：
Serial GC:    数百毫秒 ~ 数秒
Parallel GC:  数十毫秒 ~ 数百毫秒
CMS GC:       数十毫秒（但可能失败）
G1 GC:        数毫秒 ~ 数十毫秒
ZGC:          < 1ms（大部分情况 < 0.1ms）
```

## 42.4 jstat：查看 GC 情况

`jstat` 是 JDK 自带的 JVM 监控工具，不需要任何额外安装。

```bash
# 查看 Java 进程的 PID
jps -l

# 每 1000ms 输出一次 GC 统计（假设 PID=12345）
jstat -gc 12345 1000

# 输出示例：
#  S0C    S1C    S0U    S1U      EC       EU        OC         OU       MC     MU
# 10752.0 10752.0  0.0   0.0   65536.0  12345.6   174784.0   45678.2   44800.0 43562.3
#  |      |      |      |      |        |         |          |         |       |
#  S0容量  S1容量 S0使用 S1使用 Eden容量 Eden使用  老年代容量  老年代使用 元空间容量 元空间使用
```

常用参数：

| 命令 | 作用 |
|------|------|
| `jstat -gc <pid>` | GC 堆容量和使用量 |
| `jstat -gcutil <pid>` | GC 百分比统计 |
| `jstat -gcutil <pid> 1000 10` | 每秒输出一次，共 10 次 |
| `jstat -gccapacity <pid>` | 堆各区域容量 |

## 42.5 jmap：查看堆内存

```bash
# 查看堆中各类对象的数量和大小（直方图）
jmap -histo <pid>

# 只看存活对象
jmap -histo:live <pid>

# 导出堆转储文件（用于分析内存泄漏）
jmap -dump:format=b,file=heap.hprof <pid>

# 查看堆配置
jmap -heap <pid>
```

> ⚠️ `jmap -dump` 会触发 Full GC（如果加了 `:live`），在生产环境谨慎使用。建议用 `-XX:+HeapDumpOnOutOfMemoryError` 参数让 JVM 在 OOM 时自动导出。

## 42.6 常见 JVM 参数

```bash
# 堆内存设置
-Xms2g          # 初始堆大小
-Xmx4g          # 最大堆大小
-Xmn1g          # 新生代大小
-XX:NewRatio=2  # 老年代/新生代比例（2 表示老年代是新生代的 2 倍）
-XX:SurvivorRatio=8  # Eden/Survivor 比例（8 表示 Eden:S0:S1=8:1:1）

# GC 日志（Java 9+ 统一日志）
-Xlog:gc*:file=gc.log:time,level,tags

# GC 选择
-XX:+UseG1GC            # 使用 G1（Java 9+ 默认）
-XX:+UseZGC             # 使用 ZGC
-XX:MaxGCPauseMillis=200  # G1 的目标停顿时间

# OOM 排查
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dump.hprof
```

## 42.7 ❌ 常见错误

### 错误1：不设置 JVM 最大堆内存

```bash
# ❌ 使用默认值（通常是物理内存的 1/4，容器环境可能读宿主内存）
java -jar app.jar

# ✅ 明确指定堆大小
java -Xms512m -Xmx512m -jar app.jar
```

### 错误2：把 -Xms 和 -Xmx 设得不一样

```bash
# ⚠️ -Xms256m -Xmx4g —— 堆会在 256M~4G 之间动态伸缩
# 每次扩容都会触发 GC，抖动频繁
# ✅ 生产环境建议 -Xms 和 -Xmx 设相同值，避免堆大小波动
java -Xms4g -Xmx4g -jar app.jar
```

### 错误3：忽视 Full GC

频繁 Full GC 是性能问题的头号警报。排查步骤：

1. 用 `jstat -gcutil <pid> 1000` 观察 FGC（Full GC 次数）是否持续增长
2. 用 `jmap -histo <pid>` 看哪些对象占内存最多
3. 用 MAT 或 JProfiler 分析堆转储文件
4. 常见原因：内存泄漏（静态集合持有对象）、大对象直接分配在老年代

### 错误4：在低延迟场景使用 Serial/Parallel GC

```bash
# ❌ 低延迟场景仍然使用 Parallel GC（吞吐优先但停顿较长）
# ✅ 低延迟场景使用 G1 或 ZGC
-XX:+UseG1GC   # 毫秒级停顿
-XX:+UseZGC    # 亚毫秒级停顿
```

## 42.8 小结

| 知识点 | 一句话总结 |
|--------|-----------|
| 堆结构 | 新生代（Eden+S0+S1）+ 老年代，对象在新生代分配，长寿则晋升 |
| Minor GC | 回收新生代，频繁但快速，使用标记-复制算法 |
| Full GC | 回收整个堆 + 方法区，STW 时间长，应尽量避免 |
| G1 GC | 区域化 GC，Java 9+ 默认，可预测停顿 |
| ZGC | 超低延迟 GC，停顿 < 1ms，支持 TB 级堆 |
| jstat | 实时查看 GC 统计的命令行工具 |
| jmap | 查看堆内存、导出堆转储 |
| JVM 参数 | -Xms/-Xmx 设置堆大小，生产环境建议设为相同值 |

## 42.9 自测题

**1.** 简述一个 Java 对象从创建到被回收的完整生命周期（经历哪些区域、哪些 GC）。

**2.** Minor GC 和 Full GC 的区别是什么？为什么 Full GC 比 Minor GC 影响大得多？

**3.** 如果你的 Spring Boot 应用内存使用持续增长，最终 OOM，你会用哪些工具和步骤排查？

> 答案提示：1. Eden 分配 → Minor GC 后进 Survivor → 反复 Minor GC 年龄 +1 → 年龄够进老年代 → 老年代满触发 Full GC 回收；2. Minor GC 只回收新生代（小区域，标记-复制算法快），Full GC 回收整个堆（大区域，通常伴随 STW 时间长）；3. jstat 看 GC 趋势 → jmap -histo 看大对象 → jmap -dump 导出堆转储 → MAT 分析 → 定位内存泄漏代码。
