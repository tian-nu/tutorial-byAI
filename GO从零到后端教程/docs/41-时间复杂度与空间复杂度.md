# 第41章 · 时间复杂度与空间复杂度

> "你说你的算法很快——但'很快'是多快？在我的电脑上跑5秒，在你的电脑上可能跑0.5秒，在树莓派上可能跑50秒。这就像用'我的车很快'来描述车速——没有基准，没有单位，毫无意义。算法分析需要一套脱离硬件、脱离具体机器的通用语言——这就是大O表示法。"

---

## 41.1 为什么需要复杂度分析

先看一个实验。下面两段代码，哪个更快？

**代码A：**

```go
func sumA(n int) int {
    result := 0
    for i := 1; i <= n; i++ {
        result += i
    }
    return result
}
```

**代码B：**

```go
func sumB(n int) int {
    return n * (n + 1) / 2
}
```

直觉上你可能觉得B更快——因为B只有一行。但如果n=5呢？A循环5次，B只算一次。差距不大。

如果n=100000000呢？A要循环1亿次，B还是只算一次。差距巨大。

问题是：你能准确说出"A比B慢多少倍"吗？你说不出。因为：
- 在我的MacBook Pro上，A可能跑0.3秒
- 在你的Windows笔记本上，A可能跑0.8秒
- 在阿里云ECS上，A可能跑0.1秒

**用秒表衡量算法是不可靠的。** 不同硬件、不同语言、不同编译器优化，都会产生不同的"速度"。

我们需要一套**脱离具体机器**的分析方法——只关注"当数据量n变化时，算法执行步骤的**增长趋势**"。这就是时间复杂度（衡量算法执行步骤随数据量增长的趋势，详见附录I）。

> **比喻**：评估两个快递员谁效率高。你不能看"送第一单谁快"（太小了看不出），也不能看"在杭州送快还是在铁岭送快"（城市不同没法比）。你应该看的是——"当快递数量翻10倍时，谁的送件时间增长更少"。那个增长更少的，算法效率更高。

## 41.2 大O表示法

大O表示法（Big O Notation，用数学语言描述算法增长率，详见附录I）描述的是：**当输入规模n趋近无穷大时，算法执行时间的增长率。**

它的核心思想只有两条：

**规则一：忽略常数**
- O(2n) → O(n)
- O(100n + 500) → O(n)
- O(3) → O(1)

为什么可以忽略？因为当n很大时（比如n=10000000），前面的常数2、100、500根本不重要。

打个比方：张三的年薪是2n万（n是工作年数），李四的年薪是n万。前几年李四可能领先（因为有500万的签约费），但工作10年后张三拿200万，李四只拿100万——常数差距早就被线性增长淹没了。

**规则二：只保留最高阶**
- O(n² + n) → O(n²)
- O(n³ + n² + 100n) → O(n³)
- O(2ⁿ + n¹⁰⁰) → O(2ⁿ)

为什么可以忽略低阶？因为当n很大时（比如n=10000），n²是1亿，n是1万——n只占n²的万分之一，完全可以忽略。

> **核心思想**：大O表示法不关心你到底跑了多少步，它只关心——当数据量翻倍时，你的执行时间会怎么变。

## 41.3 常见复杂度等级

从快到慢，依次是：

```
O(1)     <   O(log n)   <   O(n)   <   O(n log n)   <   O(n²)   <   O(2ⁿ)
常量时间     对数时间        线性时间     线性对数时间       平方时间      指数时间
```

### O(1) — 常量时间

不管输入多大，执行时间恒定不变。

```go
func getFirst(arr []int) int {
    return arr[0]
}
```

arr有10个元素还是100万个元素，取第一个元素都是一步搞定。

再比如：

```go
func getUserAge(userMap map[string]int, name string) int {
    return userMap[name]
}
```

哈希表的查找是O(1)——不管这个map里有多少人，找张三的年龄都是一步。

**O(1)是算法的终极追求**——但大多数问题做不到。

> **O(log n) — 对数时间**

每执行一步，问题规模缩小一半（或固定比例）。

最典型的是二分查找——之前第40章已经看过。100万个有序元素，只需要约20次比较（因为2²⁰ ≈ 100万）。

```go
func binarySearch(arr []int, target int) int {
    left, right := 0, len(arr)-1
    for left <= right {
        mid := left + (right-left)/2
        if arr[mid] == target {
            return mid
        } else if arr[mid] < target {
            left = mid + 1
        } else {
            right = mid - 1
        }
    }
    return -1
}
```

log n的增长极其缓慢——n从100万变成10亿，log n只从20变成30。这是仅次于O(1)的效率。

**比喻**：一本1000页的字典，你找"hello"这个单词。你不会从第1页开始一页页翻（O(n)），而是翻到中间，看是到了h之前还是之后——每次翻页，找的范围减半。1000页只需要翻大约10次。

> **O(n) — 线性时间**

执行时间与输入规模成正比——数据翻倍，时间翻倍。

```go
func findMax(arr []int) int {
    maxVal := arr[0]
    for _, v := range arr {
        if v > maxVal {
            maxVal = v
        }
    }
    return maxVal
}
```

遍历一次数组找最大值，必须把每个元素都看一遍——这是O(n)的底线。**任何需要"看一遍所有数据"的操作，不可能比O(n)更快。**

> **O(n log n) — 线性对数时间**

比O(n)稍慢，但比O(n²)快得多。这是大多数高效排序算法的时间复杂度——快速排序、归并排序、堆排序都在这个级别。

n log n是怎么来的？想象一下：你把数据分成log n层（像二分查找那样层层二分），每一层你都要处理所有n个元素。所以总共做了n × log n次操作。

```go
func sortAndPrint(arr []int) {
    sort.Ints(arr)
    for _, v := range arr {
        fmt.Println(v)
    }
}
```

Go的`sort.Ints`用的是pdqsort，时间复杂度O(n log n)。然后遍历打印是O(n)。整体是O(n log n) + O(n) = O(n log n)（低阶被忽略）。

> **O(n²) — 平方时间**

两层嵌套循环，每一层都遍历n次。n=1000时，要执行100万次。

```go
func printPairs(arr []int) {
    for i := 0; i < len(arr); i++ {
        for j := 0; j < len(arr); j++ {
            fmt.Println(arr[i], arr[j])
        }
    }
}
```

冒泡排序、选择排序、插入排序都是O(n²)。小数据量没事，一旦n超过10000，性能就会严重下降。

**感受一下差距**：如果n=100000，O(n²)需要100亿次操作。如果每次操作1纳秒（极高性能假设），要跑10秒。而O(n log n)只需要约170万次——不到2毫秒。

**比喻**：O(n²)就像你要跟全班50个同学每个人说一遍"你好，我叫XXX"。你一个人去跟49个人说（49次），然后每个同学也来跟你说（又是49次），最后整个教室炸了——所有人都在互相打招呼，总共2500次交互。

> **O(2ⁿ) — 指数时间**

每增加一个元素，执行时间翻倍。这是最可怕的增长速度——n=30时就需要10亿次操作，n=100时宇宙可能都等不到结果。

最经典的例子：斐波那契数列的朴素递归。

```go
func fib(n int) int {
    if n <= 1 {
        return n
    }
    return fib(n-1) + fib(n-2)
}
```

为什么是O(2ⁿ)？因为fib(n)会调用fib(n-1)和fib(n-2)，而fib(n-1)又会调用fib(n-2)和fib(n-3)……每个调用分叉成两个，形成了一个巨大的递归树。

n=5时已经调用了15次；n=50时，大概要调用400亿次——你的电脑会直接卡死。

**如果你在面试中写出了O(2ⁿ)的解，面试官一定会追问"能优化吗？"** 答案是能——用记忆化（缓存中间结果）降到O(n)，或者用动态规划。

> **复杂度增长可视化**

假设每次操作耗时1毫秒，不同复杂度处理不同数据量的耗时：

| n | O(1) | O(log n) | O(n) | O(n log n) | O(n²) | O(2ⁿ) |
|---|------|----------|------|------------|-------|-------|
| 10 | 1ms | 3ms | 10ms | 33ms | 100ms | 1秒 |
| 100 | 1ms | 7ms | 100ms | 0.7秒 | 10秒 | 4×10¹⁶年 |
| 1000 | 1ms | 10ms | 1秒 | 10秒 | 17分钟 | ∞ |
| 100万 | 1ms | 20ms | 17分钟 | 5.6小时 | 31年 | ∞ |

看到了吗？O(2ⁿ)的n=100就连宇宙年龄（138亿年）都不够用。而O(log n)的n=100万只需要20毫秒。

## 41.4 空间复杂度

时间复杂度衡量执行步骤，空间复杂度（衡量算法额外内存消耗的增长趋势，详见附录I）衡量**额外内存消耗**。

同样的原则——忽略常数、保留最高阶。

### O(1) 空间

算法只需要固定大小的额外空间（不随n增长）。

```go
func sum(arr []int) int {
    total := 0
    for _, v := range arr {
        total += v
    }
    return total
}
```

不管arr有多大，只需要一个`total`变量——O(1)空间。

### O(n) 空间

额外空间和输入规模成正比。

```go
func copySlice(arr []int) []int {
    result := make([]int, len(arr))
    for i, v := range arr {
        result[i] = v
    }
    return result
}
```

创建了一个和输入一样大的切片——O(n)空间。

### 空间换时间

很多时候，时间复杂度与空间复杂度是一对矛盾——**用空间可以换时间**（不是绝对的但很常见）。

```go
func findDuplicates(arr []int) []int {
    seen := make(map[int]bool)
    result := []int{}
    for _, v := range arr {
        if seen[v] {
            result = append(result, v)
        } else {
            seen[v] = true
        }
    }
    return result
}
```

用了一个map来记录已经见过的数。时间复杂度从O(n²)（两层循环对比）降到了O(n)，但空间从O(1)升到了O(n)。

**这是编程中的经典权衡**：
- 要快就要多用内存（缓存、预计算、索引）
- 要省内存就可能要多算几次（不存中间结果，用到时再算）

> 比喻：你家楼下有个便利店，你每次做饭缺什么就下楼买（省空间，费时间）。或者你一次性把一周的菜都买回来塞冰箱（费空间，省时间）。两种策略各有适用场景——你不能为了"省空间"每次炒菜前都下楼买酱油，也不能为了"省时间"把整个菜市场搬回家。

## 41.5 实战分析多段代码

面试中最常见的题目就是："分析这段代码的时间和空间复杂度。"

### 示例1：单层循环

```go
func example1(arr []int) {
    for i := 0; i < len(arr); i++ {
        fmt.Println(arr[i])
    }
}
```

时间复杂度：O(n)。循环执行n次。空间复杂度：O(1)。没有任何额外空间分配。

### 示例2：非对称两层循环

```go
func example2(arr []int) {
    n := len(arr)
    for i := 0; i < n; i++ {
        for j := i; j < n; j++ {
            fmt.Println(arr[i], arr[j])
        }
    }
}
```

不要不假思索地说"两层循环就是O(n²)"。仔细看：内层循环从i开始，不是从0开始。
- i=0时，j循环n次
- i=1时，j循环n-1次
- i=2时，j循环n-2次
- ...
- 总共：n + (n-1) + (n-2) + ... + 1 = n(n+1)/2

n(n+1)/2 = n²/2 + n/2。忽略常数1/2和低阶n/2，复杂度是O(n²)。**所以结果还是O(n²)**——虽然比完整的两层循环少了一半，但增长率是一样的。

### 示例3：每层规模裂变

```go
func example3(arr []int) {
    n := len(arr)
    for i := 0; i < n; i += 2 {
        fmt.Println(arr[i])
    }
}
```

时间复杂度：O(n)。虽然i每次加2只执行了n/2次，但n/2的增长率仍然是O(n)——常数因子被忽略。

### 示例4：多个操作的组合

```go
func example4(arr1 []int, arr2 []int) {
    for _, v := range arr1 {
        fmt.Println(v)
    }
    for _, v := range arr2 {
        fmt.Println(v)
    }
}
```

两个独立的循环，分别遍历arr1和arr2。时间复杂度是O(n + m)，其中n=len(arr1)，m=len(arr2)。如果n≈m，可以简写为O(n)。

**注意**：两个并列循环是O(n+n)=O(n)，不是O(n²)。只有嵌套才是相乘。

### 示例5：递归分析

```go
func factorial(n int) int {
    if n <= 1 {
        return 1
    }
    return n * factorial(n-1)
}
```

递归了n次，每次递归做O(1)的工作。时间复杂度O(n)。空间复杂度也是O(n)——因为递归调用栈深度为n。

### 示例6：二分递归（陷阱！）

```go
func mergeSort(arr []int) []int {
    if len(arr) <= 1 {
        return arr
    }
    mid := len(arr) / 2
    left := mergeSort(arr[:mid])
    right := mergeSort(arr[mid:])
    return merge(left, right)
}
```

每层把问题分成两半（log n层），每层合并所有元素（O(n)工作）。总共O(n log n)。空间：每次合并创建新切片，O(n)。

merge 函数的简单实现：
```go
func merge(left, right []int) []int {
    result := make([]int, 0, len(left)+len(right))
    i, j := 0, 0
    for i < len(left) && j < len(right) {
        if left[i] <= right[j] {
            result = append(result, left[i])
            i++
        } else {
            result = append(result, right[j])
            j++
        }
    }
    result = append(result, left[i:]...)
    result = append(result, right[j:]...)
    return result
}
```

### 复杂度速查表

| 代码模式 | 时间复杂度 | 例子 |
|---------|-----------|------|
| 简单操作（赋值、加减、取下标、map读） | O(1) | `x = arr[0]` |
| 单层循环 | O(n) | `for i:=0; i<n; i++` |
| 循环内每次规模减半 | O(log n) | 二分查找 |
| 双层嵌套循环（等长） | O(n²) | 冒泡排序 |
| 三层嵌套循环 | O(n³) | Floyd最短路 |
| 循环+内部分治 | O(n log n) | 归并排序 |
| 递归二分（每次调两次） | O(2ⁿ) | 斐波那契朴素递归 |
| 全排列 | O(n!) | 旅行商问题暴力解 |

## 🤔 想多一点：大O的局限性

大O表示法看着很美，但实际工程中有几件事要注意：

### 1. 常数因子在小数据时很重要

两个算法都是O(n)，但一个做100n次操作，一个做n次操作——n=10时，差距是1000次vs10次，100倍。大O会告诉你"一样快"，实际差了100倍。

**结论**：小数据量时，不要只看大O，要看实际profile。

### 2. O(n)不一定比O(n log n)快

如果O(n)算法的常数是10000，O(n log n)算法的常数是1，那在n < 2¹⁰⁰⁰⁰（天文数字）之前，O(n log n)都比O(n)快。当然这是极端情况，实际上标准库里的O(n log n)排序确实比你自己写的O(n)循环快。

### 3. 空间换时间不总是值得的

用O(n)空间把时间从O(n²)降到O(n)很好。但用O(n²)空间把O(n²)降到O(n log n)——可能不值得。内存也是有成本的（分配时间、缓存未命中、GC压力）。

### 4. 不同的n，瓶颈不同

如果在Web服务器上，n通常是请求并发数，最多几千。O(n²)对n=1000是100万次操作，也就几毫秒。但如果在数据分析场景，n可能是几亿——O(n²)直接不可用。

**所以复杂度分析是指导原则，不是教条。** 理解它，但不要盲目崇拜它。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 复杂度分析目的 | 脱离硬件衡量算法效率，只看增长趋势 |
| 大O规则 | 忽略常数、只保留最高阶 |
| O(1) | 常量时间，不管输入多大都一样快，如map查找、数组下标 |
| O(log n) | 每次操作规模减半，如二分查找，增长极慢 |
| O(n) | 线性时间，数据翻倍时间翻倍，如单层循环遍历 |
| O(n log n) | 高效排序级别，如快排、归并、堆排 |
| O(n²) | 平方时间，两层嵌套循环，小数据可用大数据挂 |
| O(2ⁿ) | 指数时间，每加一个元素时间翻倍，纯递归陷阱 |
| 空间复杂度 | 额外内存消耗，同样用大O，常见O(1)/O(n) |
| 空间换时间 | 经典权衡，以内存换取更快的执行速度 |
| 实战技巧 | 并列循环→相加O(n)，嵌套循环→相乘O(n²)，分治递归→分析递归树 |
