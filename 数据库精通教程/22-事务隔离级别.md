## 我在做什么？

上一章你学会了用事务保证"一组操作要么全成功要么全失败"。但还有个问题没解决：**多个事务同时执行时，互相之间怎么看到对方的修改？**

回想一下，事务隔离性要求"事务之间互不干扰"。但完全隔离意味着所有事务必须排队执行，并发性能会很差。所以实际数据库提供了**四种隔离级别**，让你在"隔离程度"和"并发性能"之间做权衡。

这一章我们分三步走：
1. 先理解三个并发问题（脏读、不可重复读、幻读）——知道"要隔离什么"
2. 四种隔离级别——知道"怎么隔离"
3. MVCC 原理——知道"怎么做到的"

学完这一章，你能用两个窗口复现三种并发问题，理解 MVCC 的 undo log 版本链和 ReadView 快照机制，并说出"快照读"和"当前读"的区别。

## 一、三个并发问题：同时操作会出什么乱子？

我们先搞清楚"如果不隔离，会发生什么"。下面是并发场景中最常见的三个问题，每个我们都用两个窗口来演示。

### 1.1 脏读（Dirty Read）：读到别人"正在写"的数据

**脏读** *此术语见附录E* 是说：事务 A 修改了数据但还没提交，事务 B 就读到了这个"未提交"的修改。如果事务 A 后来回滚了，事务 B 读到的就是根本不存在的数据——就像你看了别人草稿纸上的答案，结果人家擦掉了，你的答案就错了。

**两个窗口演示脏读：**

```
-- 先查一下当前各个隔离级别
-- 我们要在 READ UNCOMMITTED 级别下才能看到脏读
```

| 操作顺序 | 窗口 1（事务 A） | 窗口 2（事务 B） |
|---------|-----------------|-----------------|
| 1 | `SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;` | `SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;` |
| 2 | `BEGIN;` | `BEGIN;` |
| 3 | `UPDATE accounts SET balance = 999 WHERE name = 'A';` | — |
| 4 | — | `SELECT balance FROM accounts WHERE name = 'A';` |
| 5 | 结果：A 自己看到 999（正常） | **结果：看到 999！这就是脏读！** |
| 6 | `ROLLBACK;`（反悔了，不转了） | — |
| 7 | — | `SELECT balance FROM accounts WHERE name = 'A';` |
| 8 | 结果：A=900 | **结果：又变回 900 了** |

你看，窗口 2 先看到 999，后来又变成 900——它读到的 999 是"脏数据"，因为事务 A 根本没提交，而且后来回滚了。

**脏读的危害**：如果你的应用根据"脏数据"做了决策（比如"余额够了，可以下单"），等人家回滚之后，你的决策依据就错了。

### 1.2 不可重复读（Non-Repeatable Read）：同一事务两次读，结果不同

**不可重复读** *此术语见附录E* 是说：事务 A 内，两次读取同一行数据，中间有人（事务 B）修改了这行并提交了，导致两次读到的值不一样。

**比喻**：你去超市看一瓶可乐 3 块钱，走到收银台准备付钱，收银员扫出来是 4 块——因为在你走的过程中，有人改了价格并通知了所有人。

**两个窗口演示不可重复读：**

```
-- 注意：READ COMMITTED 级别下会出现不可重复读
```

| 操作顺序 | 窗口 1（事务 A） | 窗口 2（事务 B） |
|---------|-----------------|-----------------|
| 1 | `SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;` | `SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;` |
| 2 | `BEGIN;` | — |
| 3 | `SELECT balance FROM accounts WHERE name = 'A';` → 看到 900 | — |
| 4 | — | `BEGIN;` → `UPDATE accounts SET balance = 500 WHERE name = 'A';` → `COMMIT;` |
| 5 | `SELECT balance FROM accounts WHERE name = 'A';` → **看到 500！** | — |
| 6 | `COMMIT;` | — |

同一个事务 A 里，第一次读到 900，第二次读到 500——这就是"不可重复读"。

**不可重复读 vs 脏读的区别**：
- 脏读：读到的是**未提交**的数据
- 不可重复读：读到的是**已提交**的数据，但两次读到的值不一样

### 1.3 幻读（Phantom Read）：同一事务两次读，行数不同

**幻读** *此术语见附录E* 是说：事务 A 内，两次执行相同的范围查询，中间有人（事务 B）插入了新行并提交了，导致第二次查询出现"凭空多出来的行"。

**比喻**：你去图书馆查"所有 2024 年出版的书"，第一次查有 10 本。你正要根据这 10 本做统计，管理员又上架了 3 本 2024 年的新书，你再查就是 13 本——这 3 本就"凭空冒出来了"。

**两个窗口演示幻读：**

```
-- 注意：幻读在 READ COMMITTED 和普通 REPEATABLE READ 下可能出现
-- 但 InnoDB 的 REPEATABLE READ 用临键锁解决了幻读问题
-- 我们先在 READ COMMITTED 下演示幻读的现象
```

| 操作顺序 | 窗口 1（事务 A） | 窗口 2（事务 B） |
|---------|-----------------|-----------------|
| 1 | `SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;` | `SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;` |
| 2 | `BEGIN;` | — |
| 3 | `SELECT * FROM accounts WHERE balance > 500;` → 返回 2 行 | — |
| 4 | — | `BEGIN;` → `INSERT INTO accounts (name, balance) VALUES ('C', 800);` → `COMMIT;` |
| 5 | `SELECT * FROM accounts WHERE balance > 500;` → **返回 3 行！多了一行 C！** | — |
| 6 | `COMMIT;` | — |

**幻读 vs 不可重复读的区别**：
- 不可重复读：**同一行**的值变了（别人 UPDATE 了）
- 幻读：**行数**变了（别人 INSERT 或 DELETE 了）

## 二、四种隔离级别：逐级加强

知道问题是什么了，接下来看"怎么防"。SQL 标准定义了四种隔离级别，从低到高：

### 2.1 隔离级别总览

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 并发性能 |
|---------|------|----------|------|---------|
| **READ UNCOMMITTED**（读未提交） | ❌ 有 | ❌ 有 | ❌ 有 | 最高 |
| **READ COMMITTED**（读已提交） | ✅ 无 | ❌ 有 | ❌ 有 | 较高 |
| **REPEATABLE READ**（可重复读） | ✅ 无 | ✅ 无 | ⚠️ 部分解决 | 中等 |
| **SERIALIZABLE**（串行化） | ✅ 无 | ✅ 无 | ✅ 无 | 最低 |

隔离级别越高，数据越安全，但并发性能越差——就像你为了安全在门上装了 5 把锁，每次开门就会慢一些。

### 2.2 逐级详解

**READ UNCOMMITTED（读未提交）**：最低级别，完全不隔离。一个事务可以读到另一个事务未提交的修改。性能最高，但一般不会有人用，因为脏读太危险了。

**READ COMMITTED（读已提交）**：解决了脏读——只能读到别人已经提交的数据。但不可重复读和幻读还在。这是 Oracle、PostgreSQL 的默认级别。

**REPEATABLE READ（可重复读）**：解决了脏读和不可重复读——同一个事务内，多次读同一行，结果始终一致。幻读理论上还存在，但 **InnoDB 用临键锁（Next-Key Lock）解决了幻读！** 这是 MySQL 的默认级别。

**SERIALIZABLE（串行化）**：最高级别，事务完全串行执行。所有事务排队执行，就像根本没有并发一样。绝对安全，但性能最差。

### 2.3 一个重要澄清：MySQL 的 REPEATABLE READ 怎么解决幻读的？

前面说了，理论上 REPEATABLE READ 不解决幻读。但 MySQL 的 InnoDB 引擎说："我偏不，我要解决！"

InnoDB 用了一种叫**临键锁（Next-Key Lock）** *此术语见附录E* 的技术——不仅是锁住查询到的行，还锁住这些行之间的间隙，防止别人插入新数据。

具体锁机制我们下一章（第 23 章）展开讲，这里你只需要知道结论：**用 InnoDB 的话，REPEATABLE READ 级别下，幻读基本上被解决了**（少数特殊情况除外，比如 `SELECT ... FOR UPDATE` 和快照读的混合使用）。

### 2.4 查看和设置隔离级别

```sql
-- 查看当前会话的隔离级别
SELECT @@transaction_isolation;
-- 或者（MySQL 8.0 之前）
SELECT @@tx_isolation;

-- 查看全局隔离级别
SELECT @@global.transaction_isolation;

-- 设置当前会话的隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置全局隔离级别（影响所有新连接，但不影响当前已有连接）
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

## 三、MVCC 原理：InnoDB 怎么做到"不同人看到不同版本"？

前面讲了隔离级别，你可能会想："InnoDB 怎么做到 REPEATABLE READ 下，同一个事务内两次读结果一样？它是不是把数据锁住了？"

答案是：**不是锁，是 MVCC（Multi-Version Concurrency Control，多版本并发控制）** *此术语见附录E*。

### 3.1 一句话理解 MVCC

**MVCC 的核心思想：不阻塞读，不阻塞写，每个事务都看到自己那个"时刻"的数据快照。**

就像拍照片：
- 你 9:00 开始一个事务 → 数据库给你拍了一张"数据快照"
- 9:01 有人改了数据 → 跟你没关系，你看的还是 9:00 的快照
- 9:02 你再次查询 → 还是 9:00 的快照，数据不变

这就是"快照读"——你读到的不是最新数据，而是事务开始时的版本。

### 3.2 核心机制一：undo log 版本链

**undo log 就像修改前的照片备份，版本链就是照片的时间线。**

当你修改一行数据，InnoDB 会做两件事：
1. 把**修改前的旧值**记录到 undo log 里（"拍一张修改前的照片"）
2. 把这行当前的数据存下来（"新照片"）

如果你再改一次，又会记录一次。这样多次修改的 undo log 串起来，就是一条**版本链**——你可以沿着这条链，看到这行数据的历史版本。

**假设 A 的余额变化过程：**

```
初始余额 = 1000
  ↓
事务1：改成 900 → undo log 记录"旧值=1000"
  ↓
事务2：改成 800 → undo log 记录"旧值=900"
  ↓
事务3：改成 700 → undo log 记录"旧值=800"
```

版本链就是：
```
当前值=700 → undo(旧值=800) → undo(旧值=900) → undo(旧值=1000)
```

> ⚠️ 注意：undo log 的完整物理存储结构和作用，我们在第 24 章存储引擎中详细展开。本章只关注 MVCC 视角下"版本链"这个概念——你理解成"数据的历史版本可以串成一条链"就够了。

### 3.3 核心机制二：ReadView（快照）

**ReadView 就像事务开始时拍的一张快照。你只能看到快照时刻之前的数据。**

当你执行一个普通 `SELECT`（快照读），InnoDB 会创建一个 ReadView，它记录了：
- 当前系统中**活跃事务**（已开始但未提交的事务）的 ID 列表
- 最早活跃事务 ID（`m_low_limit_id`）
- 下一个将要分配的事务 ID（`m_up_limit_id`）

ReadView 用这三个信息来判断：版本链中的某个版本，我能不能看到？

**可见性判断规则**（不需要背，理解就行）：

| 情况的版本事务 ID | 能看见吗？ | 原因 |
|------------------|-----------|------|
| 等于我自己的事务 ID | ✅ 能 | 我自己改的，当然能看见 |
| 小于所有活跃事务的 ID | ✅ 能 | 这个版本在我开始前就已经提交了 |
| 在活跃事务列表中 | ❌ 不能 | 这个版本是别人正在改的，还没提交 |
| 大于等于下一个事务 ID | ❌ 不能 | 这个版本在我开始之后才开始，还没提交 |

### 3.4 一个完整例子：串起来

假设有三个事务同时进行：

```
时间线：
t1: 事务 A 开始（id=1），读 A 的余额 → 1000
t2: 事务 B 开始（id=2），改 A 的余额为 900 → undo log 记录旧值 1000
t3: 事务 B 提交
t4: 事务 C 开始（id=3），改 A 的余额为 800 → undo log 记录旧值 900
t5: 事务 C 提交
t6: 事务 A 再读 A 的余额
```

在 REPEATABLE READ 下：
- 事务 A 在 t1 创建了 ReadView，那时候活跃事务只有它自己
- t6 再读时，版本链是：`当前值=800 → undo(旧值=900) → undo(旧值=1000)`
- `800` 的版本是事务 C（id=3）改的，事务 C 在事务 A 之后开始 → **不可见！**
- `900` 的版本是事务 B（id=2）改的，事务 B 在事务 A 之后开始 → **不可见！**
- `1000` 的版本是最早的，在事务 A 开始前就已存在 → ✅ **可见！**
- 所以事务 A 第二次读到 1000，和第一次一样——**可重复读！**

在 READ COMMITTED 下：
- 每次查询**都创建一个新的 ReadView**
- t6 读时，新 ReadView 中事务 B 和 C 都已提交 → 不在活跃事务列表中
- `800` 的版本可见 → 读到 800
- 和第一次读到的 1000 不一样 → **不可重复读！**

这就是 MVCC 的精髓：**同一个数据，不同的事务看到不同的版本，全依赖 ReadView 的可见性判断。**

### 3.5 快照读 vs 当前读

MVCC 里有一个非常容易混淆的概念：**`SELECT` 和 `SELECT ... FOR UPDATE` 读到的数据可能不一样！**

| 类型 | SQL 语句 | 读的是什么？ | 会加锁吗？ |
|------|---------|------------|-----------|
| 快照读（Snapshot Read） | `SELECT ...` | 基于 MVCC 的历史版本 | 不加锁 |
| 当前读（Current Read） | `SELECT ... FOR UPDATE` | 最新的已提交数据 | 加排他锁 |
| 当前读 | `SELECT ... LOCK IN SHARE MODE` | 最新的已提交数据 | 加共享锁 |
| 当前读 | `UPDATE ...` / `DELETE ...` / `INSERT ...` | 最新的已提交数据 | 加排他锁 |

**快照读** *此术语见附录E*：就是你普通的 `SELECT`，它基于 MVCC 的 ReadView，读的是你事务开始时的快照。**不加锁，不阻塞别人。**

**当前读** *此术语见附录E*：就是 `SELECT ... FOR UPDATE` 或 `UPDATE`/`DELETE`，它读的是磁盘上最新的已提交数据，且会加锁防止别人修改。**会阻塞别人。**

**一个容易踩坑的场景：**

```sql
-- 事务 A
BEGIN;
SELECT balance FROM accounts WHERE name = 'A';  -- 快照读，看到 900
-- 此时事务 B 修改了 A 的余额为 500 并提交
SELECT balance FROM accounts WHERE name = 'A';  -- 快照读，还是 900（REPEATABLE READ）
-- 现在你想根据这个 900 做 UPDATE
UPDATE accounts SET balance = balance - 100 WHERE name = 'A';
-- 这条 UPDATE 是当前读！它看到的是 500，然后减 100 变成 400！
-- 你以为余额是 900，实际已经变成 400 了！
```

这个陷阱在面试中经常出现。记住：**普通 SELECT 是快照读，UPDATE 是当前读，两者看到的数据可以不同。**

## 四、验证方法：双窗口完整实验

你需要打开两个 MySQL 窗口，把下面三组实验都做一遍。

### 实验 1：脏读验证

```sql
-- 两个窗口都执行
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- 窗口 1
BEGIN;
UPDATE accounts SET balance = 999 WHERE name = 'A';
-- 等窗口 2 查完
ROLLBACK;

-- 窗口 2
BEGIN;
-- 在窗口 1 执行 UPDATE 之后、ROLLBACK 之前执行
SELECT * FROM accounts WHERE name = 'A';  -- 应该看到 999（脏读）
-- 在窗口 1 ROLLBACK 之后
SELECT * FROM accounts WHERE name = 'A';  -- 变成原来的值
COMMIT;
```

### 实验 2：不可重复读验证

```sql
-- 两个窗口都执行
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 窗口 1
BEGIN;
SELECT * FROM accounts WHERE name = 'A';  -- 记下第一次看到的余额
-- 等窗口 2 提交完
SELECT * FROM accounts WHERE name = 'A';  -- 应该看到不同的值！
COMMIT;

-- 窗口 2
-- 在窗口 1 第一次查询之后执行
UPDATE accounts SET balance = 500 WHERE name = 'A';
COMMIT;
```

### 实验 3：REPEATABLE READ 下的可重复读

```sql
-- 两个窗口都执行
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 窗口 1
BEGIN;
SELECT * FROM accounts WHERE name = 'A';  -- 记下第一次看到的余额
-- 等窗口 2 提交完
SELECT * FROM accounts WHERE name = 'A';  -- 应该和第一次一样！
COMMIT;

-- 窗口 2
UPDATE accounts SET balance = 300 WHERE name = 'A';
COMMIT;
```

## 五、常见错误与排错

### 错误 1：认为 REPEATABLE READ 不能解决幻读

**错误认识**：很多人死记硬背"REPEATABLE READ 不能解决幻读"，但 MySQL/InnoDB 用临键锁已经解决了。

**正确理解**：SQL 标准说 REPEATABLE READ 不解决幻读，但 **InnoDB 的实现超出了标准**。在 InnoDB 的 REPEATABLE READ 下，普通 `SELECT` 用快照读不会出现幻读，`SELECT ... FOR UPDATE` 用临键锁也不会出现幻读。

### 错误 2：混淆快照读和当前读

**现象**：你写了一个 `SELECT` 查到余额 1000，然后又 `UPDATE` 余额，结果发现扣款后余额不对——原来在 `SELECT` 和 `UPDATE` 之间，别人已经改了数据。

**原因**：`SELECT` 是快照读，看到的是事务开始时的值；`UPDATE` 是当前读，看到的是最新值。

**解决方案**：
- 如果需要读最新并锁定，使用 `SELECT ... FOR UPDATE`（当前读 + 加锁）
- 或者用乐观锁（版本号），下一章讲

### 错误 3：忘记设置隔离级别，在不同级别下测试混乱

**现象**：你按照教程测试，但结果和教程不一样——因为你没有先设置隔离级别。

**解决**：每次测试前，明确设置隔离级别，确认后再测试：
```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
SELECT @@transaction_isolation;  -- 确认设置成功
```

### 错误 4：在 READ COMMITTED 下，认为事务内所有读都一样

**现象**：你以为一个事务内读到的数据都一致，但实际在 READ COMMITTED 下每次读都可能不同。

**原因**：READ COMMITTED 每次查询都创建新的 ReadView，所以能看到别人已提交的修改。

**解决**：如果你需要事务内读一致，用 REPEATABLE READ（MySQL 默认）。

| 不可重复读 | 同一行两次读值不同 | READ COMMITTED 下出现 |
| 幻读 | 同一查询两次行数不同 | 理论存在，InnoDB 用临键锁解决 |
| READ UNCOMMITTED | 最低隔离级别，三种问题都有 | 几乎不用 |
| READ COMMITTED | 解决脏读，Oracle/PG 默认 | 有不可重复读和幻读 |
| REPEATABLE READ | 解决脏读+不可重复读 | MySQL 默认，InnoDB 额外解决幻读 |
| SERIALIZABLE | 最高隔离级别，串行执行 | 性能最差 |
| 查看隔离级别 | `SELECT @@transaction_isolation;` | 确认级别后再测试 |
| 设置隔离级别 | `SET SESSION TRANSACTION ISOLATION LEVEL ...;` | 只影响当前会话 |
| MVCC | 多版本并发控制 | 不加锁，通过版本链实现隔离 |
| undo log 版本链 | 每条修改记录旧值，串成链 | 可以回溯到历史版本 |
| ReadView | 事务开始时的数据快照 | 决定哪些版本可见 |
| 快照读 | `SELECT ...` | 基于 MVCC，不加锁 |
| 当前读 | `SELECT ... FOR UPDATE` | 读最新数据，加锁 |
| 快照读 vs 当前读 | 同一事务内可能看到不同数据 | UPDATE 是当前读，不是快照读 |
