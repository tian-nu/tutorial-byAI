## 我在做什么？

前面的章节解决了一个问题：**单机数据库怎么跑得更稳更快**。但有一个天花板你绕不过去——**单机能撑的数据量是有上限的**。

- 一台机器最多存 2TB 数据
- 单表超过 2000 万行，索引就吃不消
- 单机 TPS 上限几千到几万，再多就崩了

**分库分表（Sharding）** *此术语见附录E* 就是用来突破这个天花板的。核心思想很简单：**把一份数据拆成多份，分散到多台机器上**。

就像开超市——一家店只能服务周边 5 万人，你要服务全国，就得在每个城市开分店。分库分表就是把你的"数据库超市"开成连锁店。

学完这一章，你能：
- 区分垂直拆分和水平拆分，知道什么时候用哪种
- 选择一个合适的分片键
- 理解跨分片查询的挑战和解决方案
- 了解 ShardingSphere 的基本用法
- 理解雪花算法生成全局唯一 ID

> 📊 可视化演示见 [28-分库分表_visual.html](28-分库分表_visual.html) — 通过交互式空间关系图，展示垂直拆分 → 水平拆分 → 数据路由的完整流程。

## 一、两种拆分策略：垂直 vs 水平

### 1.1 垂直拆分（Vertical Sharding）*此术语见附录E*

**按业务模块拆分，把不同的表放到不同的库。**

```
拆分前：一个库
┌──────────────────────────────────────┐
│              my_shop                  │
│  ┌──────┐ ┌──────┐ ┌──────────┐     │
│  │ users│ │orders│ │products  │     │
│  └──────┘ └──────┘ └──────────┘     │
│  ┌──────┐ ┌──────────┐              │
│  │logs  │ │analytics │              │
│  └──────┘ └──────────┘              │
└──────────────────────────────────────┘

拆分后：四个库
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ user_db  │  │ order_db │  │product_db│  │  log_db  │
│ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │
│ │users │ │  │ │orders│ │  │ │prod  │ │  │ │logs  │ │
│ └──────┘ │  │ └──────┘ │  │ └──────┘ │  │ └──────┘ │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
```

**特点**：
- ✅ 结构清晰，按业务模块隔离
- ✅ 不同库可以独立扩容、独立备份
- ❌ 不能解决单表数据量过大的问题（users 表还是在一个库里）
- ❌ 跨库 JOIN 变成分布式查询，需要应用层处理

**适用场景**：微服务架构，每个服务有自己的数据库。用户服务连 user_db，订单服务连 order_db。

### 1.2 水平拆分（Horizontal Sharding）*此术语见附录E*

**按数据行拆分，把同一张表的数据分散到多个库。**

```
拆分前：单表 1000 万行
┌──────────────────────────────┐
│         orders 表            │
│  id 1 ~ 10,000,000          │
└──────────────────────────────┘

拆分后：四张表，每张 250 万行
┌──────────────┐ ┌──────────────┐
│  orders_0    │ │  orders_1    │
│  id % 4 = 0  │ │  id % 4 = 1  │
└──────────────┘ └──────────────┘
┌──────────────┐ ┌──────────────┐
│  orders_2    │ │  orders_3    │
│  id % 4 = 2  │ │  id % 4 = 3  │
└──────────────┘ └──────────────┘
```

**特点**：
- ✅ 解决单表数据量过大的问题，每个分片数据量可控
- ✅ 读写压力分散到多台机器
- ❌ 跨分片查询复杂（JOIN、排序、聚合）
- ❌ 分片键选择不好会导致数据倾斜

**适用场景**：单表数据量超过千万、写入 TPS 超过单机上限。

### 1.3 一句话总结

| 维度 | 垂直拆分 | 水平拆分 |
|------|---------|---------|
| 拆什么 | 拆表（不同表放不同库） | 拆行（同一张表的数据分散） |
| 解决问题 | 业务耦合、单库并发瓶颈 | 单表数据量过大、单机 TPS 上限 |
| 复杂度 | 较低 | 较高 |
| 跨分片查询 | 跨库 JOIN（应用层） | 跨分片 JOIN/排序/聚合 |
| 典型场景 | 微服务 | 大表优化 |

**实际生产环境，通常是垂直拆分 + 水平拆分组合使用。**

## 二、分片键选择：分库分表最重要的一步

**分片键（Sharding Key）** *此术语见附录E* 决定了数据如何分布。选错了，后面全是坑。

### 2.1 好分片键的标准

1. **数据均匀分布**：每个分片的数据量差不多，不能有的分片 1 亿条，有的 100 条
2. **查询能带分片键**：大部分查询都知道分片键的值，能精确路由到一个分片
3. **避免跨分片查询**：JOIN 的两张表最好用相同的分片键

### 2.2 常见分片键选择

**用户 ID 取模（最常用）**：

```sql
-- 4 个分片，按 user_id % 4 路由
-- user_id=1 → 分片 1
-- user_id=2 → 分片 2
-- user_id=5 → 分片 1
```

- ✅ 分布均匀
- ✅ 查询用户数据时能精确路由
- ❌ 扩容时（从 4 个变 8 个），`user_id % 4` 和 `user_id % 8` 结果不同，数据需要迁移

**时间范围**：

```sql
-- 按月份或年份拆分
-- orders_202401 → 1 月数据
-- orders_202402 → 2 月数据
```

- ✅ 扩容简单（加新表就行）
- ✅ 历史数据归档方便
- ❌ 热点数据（最近的表压力大，旧表没压力）→ **数据倾斜**

**地理位置**：

```sql
-- 按省份或城市
-- users_beijing → 北京用户
-- users_shanghai → 上海用户
```

- ✅ 业务直观
- ❌ 分布不均匀（北京可能 1000 万，青海可能 10 万）

### 2.3 反例：最差的分片键选择

❌ 用性别（gender）做分片键——只有男/女/其他，最多 3 个分片，分布极不均匀

❌ 用状态（status）做分片键——pending/processing/done，也是几个值，热点严重

❌ 用自增 ID 做分片键——ID 逐步递增，最新数据全在一个分片

## 三、ShardingSphere 入门

> ⚠️ **语言依赖说明**：ShardingSphere-JDBC 是 Java 生态的组件，需要在 Java 项目中使用。如果你用的是 Python/Go/Node.js 等语言，可以使用 **ShardingSphere-Proxy**（独立部署的代理，任何语言都能连）或者参考本章的设计思路，寻找对应语言的分库分表方案。

### 3.1 ShardingSphere 是什么？

Apache ShardingSphere *此术语见附录E* 是一个分布式数据库中间件生态，提供三种接入方式：

| 方式 | 说明 | 适用 |
|------|------|------|
| **ShardingSphere-JDBC** | Java 的 JDBC 驱动增强，嵌入应用 | Java 项目 |
| **ShardingSphere-Proxy** | 独立部署的代理，兼容 MySQL 协议 | 任何语言 |
| **ShardingSphere-Sidecar** | 云原生方案（Kubernetes） | 云原生架构 |

### 3.2 ShardingSphere-Proxy 安装（Docker，推荐）

```bash
# 下载 Proxy
docker run -d \
  --name shardingsphere-proxy \
  --network mysql-repl \
  -p 3309:3307 \
  -v /path/to/config:/opt/shardingsphere-proxy/conf \
  apache/shardingsphere-proxy:5.4.0
```

### 3.3 配置分片规则

配置文件 `server.yaml`：

```yaml
# server.yaml
authority:
  users:
    - user: root
      password: root123
  privilege:
    type: ALL_PERMITTED

props:
  proxy-frontend-database-protocol-type: MySQL
```

配置文件 `config-sharding.yaml`：

```yaml
# config-sharding.yaml
databaseName: sharding_db

dataSources:
  ds_0:
    url: jdbc:mysql://mysql-master:3306/db_0
    username: root
    password: root123
  ds_1:
    url: jdbc:mysql://mysql-slave:3306/db_1
    username: root
    password: root123

rules:
  - !SHARDING
    tables:
      t_order:
        actualDataNodes: ds_${0..1}.t_order_${0..1}
        # ds_0.t_order_0, ds_0.t_order_1, ds_1.t_order_0, ds_1.t_order_1
        tableStrategy:
          standard:
            shardingColumn: order_id
            shardingAlgorithmName: t_order_inline
        keyGenerateStrategy:
          column: order_id
          keyGeneratorName: snowflake

    shardingAlgorithms:
      t_order_inline:
        type: INLINE
        props:
          algorithm-expression: t_order_${order_id % 4}
          # 4 个分片，路由到 t_order_0 ~ t_order_3

    keyGenerators:
      snowflake:
        type: SNOWFLAKE
```

### 3.4 验证分片路由

连接 ShardingSphere-Proxy：

```bash
mysql -u root -proot123 -h 127.0.0.1 -P 3309
```

```sql
USE sharding_db;

-- 创建逻辑表（Proxy 会自动创建实际分片表）
CREATE TABLE t_order (
    order_id BIGINT PRIMARY KEY,
    user_id INT,
    amount DECIMAL(10,2)
);

-- 插入几条数据，观察分片路由
INSERT INTO t_order (user_id, amount) VALUES (1, 99.00);
INSERT INTO t_order (user_id, amount) VALUES (2, 150.00);
INSERT INTO t_order (user_id, amount) VALUES (3, 200.00);

-- 查询所有数据（Proxy 会汇总所有分片的结果）
SELECT * FROM t_order;
-- 看到 3 条数据，但你不知道它们分别在哪个分片

-- 用分片键精确查询（Proxy 只路由到一个分片，效率最高）
SELECT * FROM t_order WHERE order_id = 1234567890;
```

**验证数据分布**：直接连各个分片库看看：

```bash
docker exec mysql-master mysql -u root -proot123 db_0 -e "SELECT * FROM t_order_0;"
docker exec mysql-master mysql -u root -proot123 db_0 -e "SELECT * FROM t_order_1;"
docker exec mysql-slave mysql -u root -proot123 db_1 -e "SELECT * FROM t_order_0;"
docker exec mysql-slave mysql -u root -proot123 db_1 -e "SELECT * FROM t_order_1;"
# 每条数据只在一个分片里，不会重复
```

## 四、跨分片问题与解决方案

分片后，SQL 不再简单。以下问题你一定会遇到：

### 4.1 跨分片 JOIN

❌ 问题：

```sql
-- 用户表和订单表按不同分片键拆分，JOIN 无法执行
SELECT u.username, o.amount
FROM users u JOIN orders o ON u.id = o.user_id;
-- users 按 user_id 分片，orders 按 order_id 分片
-- 一个用户的数据可能在多个分片
```

✅ 解决方案：

**方案一：绑定表（Binding Table）**——让 JOIN 的两张表用相同的分片键

```yaml
# 让 order 和 order_item 按相同的 order_id 分片
bindingTables:
  - t_order, t_order_item
# 这样 JOIN 在一个分片内就能完成
```

**方案二：应用层 JOIN**——分别查两张表，在应用层合并结果

```python
# 查用户信息
user = db.query("SELECT * FROM users WHERE id = ?", user_id)
# 查订单列表
orders = db.query("SELECT * FROM orders WHERE user_id = ?", user_id)
# 在应用层组装
result = {"user": user, "orders": orders}
```

**方案三：冗余字段**——在订单表里冗余存用户名，避免 JOIN

```sql
-- 订单表加冗余字段
CREATE TABLE t_order (
    order_id BIGINT PRIMARY KEY,
    user_id INT,
    username VARCHAR(50),  -- 冗余！避免查用户表
    amount DECIMAL(10,2)
);
```

### 4.2 跨分片排序与分页

❌ 问题：

```sql
-- 全局排序，需要从每个分片取数据，然后在 Proxy 层合并排序
SELECT * FROM t_order ORDER BY amount DESC LIMIT 10;
-- 每个分片取前 10 → 4 × 10 = 40 条 → 排序取前 10
```

✅ 处理方式：ShardingSphere 会自动处理，但要注意性能。如果数据量大，考虑用搜索引擎（Elasticsearch）做列表查询。

### 4.3 跨分片事务（分布式事务）

❌ 问题：一个事务跨多个分片，不能简单用 MySQL 的 `BEGIN/COMMIT`。

✅ 解决方案：

**Seata**（阿里开源，AT 模式）：对业务代码侵入小，推荐入门使用。

```yaml
# ShardingSphere 集成 Seata
rules:
  - !SHARDING
    # ... 分片规则 ...
  - !TRANSACTION
    defaultType: BASE
    providerType: Seata
```

**XA 协议**：两阶段提交（2PC），性能差，一般不用。

**最终一致性**：消息队列 + 本地事务表，异步补偿。

> 分布式事务是分布式系统的一大难题，本书不深入展开。建议：**尽量减少跨分片事务，让每个事务尽量落在一个分片内。**

## 五、全局唯一 ID：雪花算法

分片后，自增 ID 失效了——每个分片的自增 ID 都从 1 开始，会重复。需要全局唯一的 ID 生成方案。

### 5.1 雪花算法（Snowflake）*此术语见附录E*

Twitter 开源的ID生成算法，原理极其精妙：

```
雪花 ID 结构（64 位）：
┌─────────────────────────────────────────────────────────────────┐
│ 1bit │    41 bits    │  10 bits   │         12 bits             │
│ 未用  │   时间戳(ms)  │  机器ID     │    序列号（同一毫秒内递增）  │
│ 0     │ 从自定义纪元  │ 1024 台机器│    4096 个/ms               │
└─────────────────────────────────────────────────────────────────┘

例如：1234567890123456789
```

**特点**：
- ✅ 全局唯一，不依赖数据库
- ✅ 趋势递增（按时间），对索引友好
- ✅ 高性能，单机每秒可生成几十万个
- ❌ 依赖机器时钟，时钟回拨会出问题

**想多一点：为什么雪花 ID 对索引友好？**

第 17 章学过，B+树索引喜欢有序插入（避免页分裂）。雪花 ID 天然按时间递增，大部分场景下插入顺序和时间顺序一致，索引性能好。UUID 随机，插入时页分裂严重。

### 5.2 ShardingSphere 内置雪花算法

```yaml
keyGenerators:
  snowflake:
    type: SNOWFLAKE
    props:
      worker-id: 1  # 机器 ID，不同机器不能重复
```

## 六、分片后扩容：数据迁移

### 6.1 扩容的痛苦

假设你从 4 个分片扩到 8 个：
- 原来的数据按 `user_id % 4` 分布
- 现在要按 `user_id % 8` 分布
- 大部分数据的路由都变了，需要迁移

### 6.2 扩容方案

**方案一：翻倍扩容（最简单）**

从 4 个分片扩到 8 个，每个旧分片拆成 2 个：
- 分片 0 → 分片 0（`user_id % 8 = 0`）+ 分片 4（`user_id % 8 = 4`）
- 分片 1 → 分片 1（`user_id % 8 = 1`）+ 分片 5（`user_id % 8 = 5`）
- ...

数据迁移量：每个旧分片只迁移一半数据，可控。

**方案二：一致性哈希（Consistent Hashing）** *此术语见附录E*

不展开原理，一句话：一致性哈希让扩容时只有部分数据需要迁移，比取模算法好很多。

**方案三：预分片**

一开始就建 32 个分片（逻辑分片），但物理上只部署 4 台机器，每个机器跑 8 个分片。扩容时，把部分逻辑分片迁移到新机器即可，不需要重新哈希。

## 我做得对不对？

### 验证方法

```bash
# 1. 启动 ShardingSphere-Proxy 后，连接测试
mysql -u root -proot123 -h 127.0.0.1 -P 3309 -e "SHOW DATABASES;"
# 应该看到 sharding_db

# 2. 创建分片表，插入数据
mysql -u root -proot123 -h 127.0.0.1 -P 3309 sharding_db -e "
  CREATE TABLE IF NOT EXISTS t_order (order_id BIGINT PRIMARY KEY, user_id INT, amount DECIMAL(10,2));
  INSERT INTO t_order (user_id, amount) VALUES (1, 99.00), (2, 150.00), (3, 200.00), (4, 250.00);
"

# 3. 直接连各分片库，确认数据按规则分布
docker exec mysql-master mysql -u root -proot123 db_0 -e "SELECT COUNT(*) FROM t_order_0;"
docker exec mysql-master mysql -u root -proot123 db_0 -e "SELECT COUNT(*) FROM t_order_1;"
# 每个分片的数据量应该大致均匀

# 4. 通过 Proxy 查询，确认所有数据都能查到
mysql -u root -proot123 -h 127.0.0.1 -P 3309 sharding_db -e "SELECT COUNT(*) FROM t_order;"
# 应该返回 4
```

## 不对怎么办？

### 常见错误 1：分片键选择不当导致数据倾斜

现象：查看各分片，有的分片 500 万行，有的分片 10 万行。

❌ 原因：用了低基数列（如 status、gender）做分片键。

✅ 解决：换成高基数列（如 user_id、order_id），确保数据均匀分布。

### 常见错误 2：跨分片 JOIN 不懂怎么处理

现象：分片后原来的 JOIN 查询报错或极慢。

✅ 解决：
1. 先看能不能用**绑定表**（相同分片键）——ShardingSphere 配置 `bindingTables`
2. 不行就**冗余字段**——在订单表里存用户名
3. 再不行就**应用层 JOIN**——分别查两张表，代码里合并

### 常见错误 3：分片后忘了全局唯一 ID

现象：多分片插入后，ID 重复了。

原因：每个分片都有自己的自增 ID，从 1 开始。

✅ 解决：使用雪花算法或 ShardingSphere 内置的 `SNOWFLAKE` key generator。

## 最可能出错的地方及原因

1. **分片键选择不当**：用了低基数列（gender/status），导致数据严重倾斜，原因是不知道分片键需要高基数。
2. **跨分片 JOIN 直接写 SQL 不处理**：原来的 JOIN 查询在分片后报错，原因是没意识到分片后表分散在不同库。
3. **分片后自增 ID 重复**：每个分片自增 ID 都从 1 开始，插入同 ID 数据冲突，原因是忘了全局唯一 ID。
4. **扩容时数据迁移成本巨大**：取模分片算法下扩容需要全量数据迁移，原因是没预分片或没考虑一致性哈希。
5. **所有查询都走全分片扫描**：不带分片键的查询，Proxy 需要查询所有分片再合并，性能极差，原因是没在查询中带分片键。
