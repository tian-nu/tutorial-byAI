## 18.1 配置文件：YAML vs Properties

### 18.1.1 Spring Boot的配置文件

在第15章创建第一个Spring Boot项目时，你一定注意到了`src/main/resources/`下有个`application.properties`文件。它是Spring Boot项目的配置中枢——数据源、服务器端口、日志级别、MyBatis映射路径……几乎一切都可以通过它来配置。

Spring Boot支持两种配置文件格式：

| 维度 | `.properties` | `.yml` / `.yaml` |
|------|--------------|-----------------|
| **语法** | `key=value` 键值对 | 缩进层级结构 |
| **可读性** | 扁平，配置多了冗长 | 层级清晰，一目了然 |
| **重复前缀** | 需要反复写相同前缀 | 通过缩进消除重复 |
| **列表/map** | 支持较弱 | 原生支持 |
| **Spring Boot默认** | ✅ 传统默认 | ✅ 现代项目主流 |
| **同时存在时** | 会被yml**覆盖**（yml优先级更高） | — |

> 🚨 **坑点：不要同时使用两种格式**
> - Spring Boot会**同时加载** `application.properties` 和 `application.yml`
> - 如果有同名配置，`application.yml` **优先级更高**（因为YAML先加载，Properties后加载——后加载的覆盖先加载的）
> - 你心想"大部分配置在YAML里，数据库密码放Properties单独管理" → **两文件有相同key时，Properties的值被忽略**
> - **最佳实践**：整个项目**只用一种格式**，推荐 `application.yml`

### 18.1.2 YAML语法基础

YAML（YAML Ain't Markup Language）是一种专为配置文件设计的数据序列化语言。在深入Spring Boot配置之前，先掌握YAML的基础语法。

**规则一：缩进表示层级关系**

```yaml
server:                    # 顶层
  port: 8080               # server的子属性
  servlet:
    context-path: /api     # 三级嵌套
```

等价于Properties的：

```properties
server.port=8080
server.servlet.context-path=/api
```

> 🚨 **坑点：YAML缩进不能用Tab！必须用空格！**
> - 如果你习惯用Tab缩进（很多程序员都习惯），YAML会直接报错
> - 错误信息通常不友好：`mapping values are not allowed here` 或 `found character '\t' that cannot start any token`
> - **根治方法**：在IDEA中设置 `File → Settings → Editor → Code Style → YAML → Use tab character: 取消勾选`，缩进量设为2或4
> - IDEA默认对`.yml`文件使用空格缩进，但如果你改了全局设置可能受影响

**规则二：列表用 `-` 前缀**

```yaml
# Properties写法：
# servers[0]=192.168.1.1
# servers[1]=192.168.1.2

servers:
  - 192.168.1.1
  - 192.168.1.2
```

也可以写成内联形式：

```yaml
servers: [192.168.1.1, 192.168.1.2]
```

**规则三：映射（Map）**

```yaml
# Properties写法：
# app.cache.users.ttl=300
# app.cache.users.max-size=1000
# app.cache.orders.ttl=60
# app.cache.orders.max-size=500

app:
  cache:
    users:
      ttl: 300
      max-size: 1000
    orders:
      ttl: 60
      max-size: 500
```

**规则四：引号的使用**

```yaml
# 普通字符串不需要引号
name: 张三

# 包含特殊字符（:、#、{、}、[、]、,、&、*、?、|、-、<、>、=、!、%、@、`）需要加引号
greeting: "Hello: World"       # 冒号会被误解析
path: "C:\\Users\\admin"       # 反斜杠需要双引号（单引号会原样输出）

# 单引号 vs 双引号
single: 'Hello\nWorld'         # 原样输出：Hello\nWorld（不转义）
double: "Hello\nWorld"         # 输出：Hello（换行）World（转义生效）
```

> 🚨 **坑点：特殊字符串不加引号导致解析失败**
> - `password: abc#123` → YAML把`#`之后当作注释，password实际值是`abc`
> - `value: 123,456` → 可能被解析为字符串而非数字
> - `time: 12:00:00` → 冒号被解析为层级分隔符
> - **原则**：不确定时就加引号，加双引号最安全

**规则五：多行文本**

```yaml
# | 保留换行符
description: |
  这是一段
  多行文本
  换行符会被保留

# > 折叠换行（将换行替换为空格）
summary: >
  这是一段
  多行文本
  最终会合并为一行
```

### 18.1.3 同一配置两种写法的对比实战

以第17章的数据源配置为例，展示Properties和YAML两种写法：

**application.properties 写法**：

```properties
# ==================== 数据源配置 ====================
spring.datasource.url=jdbc:mysql://localhost:3306/ems?useSSL=false&serverTimezone=Asia/Shanghai&characterEncoding=utf8mb4
spring.datasource.username=root
spring.datasource.password=123456
spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver

# HikariCP连接池
spring.datasource.hikari.maximum-pool-size=20
spring.datasource.hikari.minimum-idle=5
spring.datasource.hikari.connection-timeout=3000
spring.datasource.hikari.idle-timeout=600000

# MyBatis
mybatis.mapper-locations=classpath:mapper/*.xml
mybatis.type-aliases-package=com.example.ems.entity
mybatis.configuration.map-underscore-to-camel-case=true
mybatis.configuration.log-impl=org.apache.ibatis.logging.stdout.StdOutImpl

# 服务器
server.port=8080
server.servlet.context-path=/api
```

**application.yml 写法**：

```yaml
# ==================== 数据源配置 ====================
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/ems?useSSL=false&serverTimezone=Asia/Shanghai&characterEncoding=utf8mb4
    username: root
    password: 123456
    driver-class-name: com.mysql.cj.jdbc.Driver
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 3000
      idle-timeout: 600000

# MyBatis
mybatis:
  mapper-locations: classpath:mapper/*.xml
  type-aliases-package: com.example.ems.entity
  configuration:
    map-underscore-to-camel-case: true
    log-impl: org.apache.ibatis.logging.stdout.StdOutImpl

# 服务器
server:
  port: 8080
  servlet:
    context-path: /api
```

对比明显：YAML通过缩进消除了大量重复前缀，配置结构一目了然。当配置项超过20个时，YAML的优势更加突出。

## 18.2 多环境配置

### 18.2.1 为什么需要多环境？

任何一个经历过生产事故的开发者都明白：**用同一套配置连接开发数据库和生产数据库是灾难性的。**

真实的软件开发至少有三个环境：

| 环境 | 用途 | 数据库 | 日志级别 | 特点 |
|------|------|--------|---------|------|
| **dev（开发）** | 程序员本地开发调试 | 本地MySQL | DEBUG | 频繁修改，允许报错 |
| **test（测试）** | QA人员功能测试 | 测试服务器MySQL | INFO | 模拟生产，数据可重置 |
| **prod（生产）** | 真实用户访问 | 生产服务器MySQL | WARN | 稳定第一，任何改动需审批 |

如果只有一套配置，每次部署都要手动改数据库地址、日志级别、端口……不仅繁琐，更危险的是：**一旦忘记修改就部署到生产，轻则功能异常，重则数据损坏。**

> 🚨 **真实事故案例**：某公司开发人员周五下班前用dev配置打包，周一运维直接部署到生产。结果生产系统连的是开发数据库，用户看到的是脏数据，所有业务数据混乱。花了整整两天时间恢复数据和排查问题。

### 18.2.2 多环境配置文件命名规则

Spring Boot的多环境配置基于 **`application-{profile}.yml`** 命名约定：

```
src/main/resources/
├── application.yml              # 主配置（公共配置 + 默认激活哪个环境）
├── application-dev.yml          # 开发环境
├── application-test.yml         # 测试环境
└── application-prod.yml         # 生产环境
```

加载规则：
1. `application.yml` **始终加载**，存放各环境公共的配置
2. `application-{profile}.yml` **按需加载**，存放环境特定的配置
3. 如果`application.yml`和`application-dev.yml`有**同名配置项**，环境配置**覆盖**主配置

### 18.2.3 三环境完整配置示例

**application.yml（主配置——公共部分）**：

```yaml
# ==================== 所有环境公共配置 ====================
server:
  port: 8080
  servlet:
    context-path: /api

spring:
  # 默认激活dev环境
  profiles:
    active: dev

  # 公共数据源配置（各环境通过application-{profile}.yml覆盖差异部分）
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 3000

# MyBatis公共配置
mybatis:
  mapper-locations: classpath:mapper/*.xml
  type-aliases-package: com.example.ems.entity
  configuration:
    map-underscore-to-camel-case: true

# 自定义应用配置（见18.3节）
app:
  name: EMS-员工管理系统
  version: 1.0.0
```

**application-dev.yml（开发环境）**：

```yaml
# ==================== 开发环境 ====================
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/ems_dev?useSSL=false&serverTimezone=Asia/Shanghai&characterEncoding=utf8mb4
    username: root
    password: dev123456

  # 开发环境显示SQL
  jpa:
    show-sql: true

# 开发环境日志级别：DEBUG（看详细信息）
logging:
  level:
    root: INFO
    com.example.ems: DEBUG           # 你项目的包 → DEBUG
    com.example.ems.mapper: DEBUG    # Mapper层 → DEBUG（看SQL参数）
    org.springframework: INFO

# 开发环境开启调试功能
debug: true

# 自定义配置
app:
  upload-path: D:/uploads/           # 开发环境文件上传本地路径
  cors-allowed-origins: http://localhost:5173  # Vue开发服务器
```

**application-test.yml（测试环境）**：

```yaml
# ==================== 测试环境 ====================
spring:
  datasource:
    url: jdbc:mysql://192.168.10.50:3306/ems_test?useSSL=true&serverTimezone=Asia/Shanghai&characterEncoding=utf8mb4
    username: ems_test_user
    password: Test@2024!Secure

# 测试环境日志：INFO（不需要DEBUG细节，但需要关键信息排查Bug）
logging:
  level:
    root: WARN
    com.example.ems: INFO
    org.springframework: WARN

# 自定义配置
app:
  upload-path: /data/ems/uploads/
  cors-allowed-origins: http://test.ems.com

# 测试环境关闭调试
debug: false
```

**application-prod.yml（生产环境）**：

```yaml
# ==================== 生产环境 ====================
spring:
  datasource:
    url: jdbc:mysql://10.0.1.100:3306/ems_prod?useSSL=true&serverTimezone=Asia/Shanghai&characterEncoding=utf8mb4&connectTimeout=5000&socketTimeout=15000
    username: ems_prod_user
    # 密码使用Jasypt加密！绝不明文存储（见18.5节）
    password: ENC(BL3fR8xK9mQ2vN7pW5tY1aD6jH4cS0)

    hikari:
      maximum-pool-size: 50          # 生产环境并发高，连接池大一些
      minimum-idle: 10
      connection-timeout: 5000
      idle-timeout: 300000

# 生产环境日志：WARN（只记录警告和错误，减少IO开销）
logging:
  level:
    root: WARN
    com.example.ems: WARN
  file:
    path: /var/log/ems/              # 写入文件，不是控制台
    name: ${logging.file.path}/ems-prod.log

# 生产环境关闭调试和信息端点
debug: false

# Actuator监控端点（见第19章）
management:
  endpoints:
    web:
      exposure:
        include: health,info         # 生产环境只暴露健康检查和基本信息
  endpoint:
    health:
      show-details: when-authorized  # 详细信息需授权

# 自定义配置
app:
  upload-path: /data/ems/uploads/
  cors-allowed-origins: https://ems.yourcompany.com
```

### 18.2.4 三种激活方式

**方式一：application.yml中指定（推荐开发阶段）**

```yaml
spring:
  profiles:
    active: dev
```

简单直接，但每次切换环境都要修改文件、重新编译打包。适合**个人开发**。

**方式二：环境变量（推荐生产部署）**

```bash
# Linux / Mac
export SPRING_PROFILES_ACTIVE=prod
java -jar ems.jar

# Windows PowerShell
$env:SPRING_PROFILES_ACTIVE="prod"
java -jar ems.jar

# 或者在 systemd 服务文件中
[Service]
Environment="SPRING_PROFILES_ACTIVE=prod"
```

Spring Boot支持将环境变量**自动映射**为配置属性：
- `SPRING_PROFILES_ACTIVE` → `spring.profiles.active`
- 规则：全大写、用下划线替代点和短横线、加`SPRING_`前缀去掉后就是配置key

**方式三：命令行参数（优先级最高，运维紧急调整）**

```bash
java -jar ems.jar --spring.profiles.active=prod

# 甚至可以同时指定多个profile
java -jar ems.jar --spring.profiles.active=prod,monitoring
```

命令行参数的格式是 `--配置项=值`，Spring Boot会自动解析并覆盖配置文件中的同名配置。

### 18.2.5 IDEA中切换环境的操作演示

在IntelliJ IDEA中，你不需要每次手动修改`application.yml`：

1. 打开 `Run → Edit Configurations`
2. 找到你的Spring Boot启动配置
3. 在 **Active profiles** 输入框中填入 `dev` 或 `test`
4. 或者在 **Environment variables** 中填入 `SPRING_PROFILES_ACTIVE=dev`

IDEA会在启动时自动传入对应的profile，覆盖`application.yml`中的`spring.profiles.active`配置。

启动日志中你会看到：

```
The following 1 profile is active: "dev"
```

### 18.2.6 Maven Profile与Spring Profile配合

如果你使用Maven管理项目，还可以在打包阶段就决定使用哪个环境的配置：

**pom.xml**：

```xml
<profiles>
    <profile>
        <id>dev</id>
        <properties>
            <spring.profiles.active>dev</spring.profiles.active>
        </properties>
        <activation>
            <activeByDefault>true</activeByDefault>
        </activation>
    </profile>
    <profile>
        <id>test</id>
        <properties>
            <spring.profiles.active>test</spring.profiles.active>
        </properties>
    </profile>
    <profile>
        <id>prod</id>
        <properties>
            <spring.profiles.active>prod</spring.profiles.active>
        </properties>
    </profile>
</profiles>
```

然后在`application.yml`中：

```yaml
spring:
  profiles:
    active: @spring.profiles.active@   # Maven打包时替换为实际值
```

打包命令：

```bash
# 开发环境打包
mvn clean package -Pdev

# 生产环境打包
mvn clean package -Pprod
```

Maven会在资源过滤阶段将`@spring.profiles.active@`替换为对应profile的值。这样jar包内就固定了环境配置，运维无需额外指定。

> 🚨 **坑点：Maven的@变量@替换和Spring Boot的`@ConfigurationProperties`可能冲突**
> - Maven资源过滤默认会替换所有`@xxx@`占位符
> - 如果你的`@ConfigurationProperties`类中有类似`@NotNull`等注解，不会被影响（那是Java注解，不是文件中的文本）
> - 但如果你的配置文件中真的有`@`字符需要使用，需要在pom.xml中使用`<delimiters>`或配置`<resources>`的`filtering`范围
> - **建议**：只对`application.yml`开启Maven过滤，其他资源文件关闭

### 18.2.7 使用 `@Profile` 注解按环境注册Bean

除了配置文件，你也可以用`@Profile`注解控制**哪些Bean在哪个环境生效**：

```java
@Configuration
public class DataInitializerConfig {

    @Bean
    @Profile("dev")  // 只在dev环境创建此Bean
    public CommandLineRunner devDataInitializer(EmployeeRepository repo) {
        return args -> {
            // 开发环境启动时插入测试数据
            repo.save(new Employee("张三", 25, "技术部", 15000.0));
            repo.save(new Employee("李四", 30, "产品部", 18000.0));
            System.out.println("✅ 开发环境测试数据初始化完成");
        };
    }

    @Bean
    @Profile("!dev")  // 非dev环境（test和prod）
    public CommandLineRunner nonDevWarning() {
        return args -> {
            System.out.println("⚠️ 当前不是开发环境，不初始化测试数据");
        };
    }
}
```

> 🚨 **坑点：`@Profile("!dev")`中`!dev`表示"非dev"，即包括test和prod。注意别写成`"!dev"`（是字符串）或`!("dev")`（语法错误）。也支持多环境：`@Profile({"dev", "test"})`表示dev或test都生效。

## 18.3 @ConfigurationProperties — 类型安全的配置绑定

### 18.3.1 为什么需要类型安全的配置？

截至目前，你获取配置的方式可能是：

```java
@Value("${app.name}")
private String appName;

@Value("${app.upload-path}")
private String uploadPath;

@Value("${app.cors-allowed-origins}")
private String corsOrigins;
```

这种方式可行，但有几个痛点：
1. **散装注入**：每个配置一个`@Value`，几十个配置就得几十个字段
2. **无类型安全**：`@Value("${app.max-retry}")`注入到`String`字段也能编译通过
3. **无校验**：配置文件写了`app.max-retry: -1`，你只能在运行时发现
4. **无IDE提示**：不知道有哪些配置项可用，只能去`application.yml`里查
5. **松散绑定不支持**：`@Value("${app.max-retry}")`只能用短横线，驼峰不行

**`@ConfigurationProperties`一次解决上述所有问题。**

### 18.3.2 使用方式一：@Component + @ConfigurationProperties

**步骤1：添加依赖（IDE自动提示必需）**

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-configuration-processor</artifactId>
    <optional>true</optional>
</dependency>
```

> 🚨 **坑点：不加`spring-boot-configuration-processor`依赖，IDE不提示自定义配置**
> - 配置`app.name`时，IDEA编辑`application.yml`不会自动补全
> - 编写配置类时，属性名写错也不会被警告
> - 这个依赖只在编译时生效（`optional=true`），不会打进jar包
> - **加上后需要重新构建项目**：`mvn compile` 或在IDEA中 `Build → Rebuild Project`
> - 完成后在`application.yml`中输入`app.`会弹出你定义的属性提示

**步骤2：编写配置类**

```java
package com.example.ems.config;

import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.stereotype.Component;

@Component
@ConfigurationProperties(prefix = "app")
public class AppConfig {

    /** 应用名称 */
    private String name;

    /** 应用版本 */
    private String version;

    /** 文件上传路径 */
    private String uploadPath;

    /** 允许的跨域来源 */
    private String corsAllowedOrigins;

    // ========== 必须提供getter和setter ==========
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    public String getVersion() { return version; }
    public void setVersion(String version) { this.version = version; }

    public String getUploadPath() { return uploadPath; }
    public void setUploadPath(String uploadPath) { this.uploadPath = uploadPath; }

    public String getCorsAllowedOrigins() { return corsAllowedOrigins; }
    public void setCorsAllowedOrigins(String corsAllowedOrigins) {
        this.corsAllowedOrigins = corsAllowedOrigins;
    }
}
```

> 🚨 **坑点：@ConfigurationProperties类必须有getter和setter**
> - Spring通过**反射调用setter**来绑定配置值
> - 如果没有setter，配置值无法注入 → 字段为null
> - 如果你使用Java 16+的**record**类型，可以用构造器绑定替代setter：
>   ```java
>   @ConfigurationProperties(prefix = "app")
>   public record AppConfig(String name, String version,
>                           String uploadPath, String corsAllowedOrigins) {}
>   ```
> - record方式更简洁，但不可变（配置在运行时不能修改），适用于大多数场景

**步骤3：使用配置**

```java
@Service
public class FileService {

    private final AppConfig appConfig;

    // 构造器注入（推荐）
    public FileService(AppConfig appConfig) {
        this.appConfig = appConfig;
    }

    public void saveFile(MultipartFile file) {
        String uploadDir = appConfig.getUploadPath();
        Path targetPath = Path.of(uploadDir, file.getOriginalFilename());
        // ...
        log.info("文件已保存到：{}", targetPath);
    }
}
```

### 18.3.3 使用方式二：@EnableConfigurationProperties

如果不想在配置类上加`@Component`（比如配置类在第三方jar包中，或者想更明确地声明哪些配置类生效），使用方式二：

```java
package com.example.ems.config;

import org.springframework.boot.context.properties.ConfigurationProperties;

// 注意：这里不需要 @Component
@ConfigurationProperties(prefix = "app")
public class AppConfig {
    // ... 同上的属性和getter/setter
}
```

然后在任意`@Configuration`类上启用：

```java
@Configuration
@EnableConfigurationProperties(AppConfig.class)
public class AppConfigAutoConfiguration {
    // 可以在这里对AppConfig做额外的初始化
}
```

两种方式的选择：
- **方式一**：简单直接，适合你自己的项目
- **方式二**：适合编写starter/公共库，或者需要按条件启用配置类时

### 18.3.4 嵌套配置（复杂对象）

配置类支持嵌套结构：

**application.yml**：

```yaml
app:
  name: EMS-员工管理系统
  thread-pool:
    core-size: 10
    max-size: 50
    queue-capacity: 200
    keep-alive-seconds: 60
```

**配置类**：

```java
@ConfigurationProperties(prefix = "app")
@Component
public class AppConfig {

    private String name;
    private ThreadPoolConfig threadPool = new ThreadPoolConfig();  // 必须初始化！

    // 内部静态类
    public static class ThreadPoolConfig {
        private int coreSize;
        private int maxSize;
        private int queueCapacity;
        private int keepAliveSeconds;
        // getter/setter...
    }

    // getter/setter...
}
```

> 🚨 **坑点：嵌套配置对象必须初始化**
> - 如果`private ThreadPoolConfig threadPool;`是null，Spring无法绑定其子属性
> - 必须写成 `private ThreadPoolConfig threadPool = new ThreadPoolConfig();`
> - 或者通过构造器初始化

### 18.3.5 JSR-303校验配置值

`@ConfigurationProperties`支持JSR-303（Bean Validation）校验注解，在应用启动时就能发现配置错误：

```java
@Component
@ConfigurationProperties(prefix = "app")
@Validated  // ← 开启校验！必不可少！
public class AppConfig {

    @NotBlank(message = "应用名称不能为空")
    private String name;

    @NotNull
    @Min(value = 1, message = "核心线程数至少为1")
    @Max(value = 100, message = "核心线程数不能超过100")
    private Integer maxRetry;

    @Positive(message = "线程存活时间必须为正数")
    private Integer keepAliveSeconds;

    // 校验嵌套对象
    @Valid  // ← 嵌套校验需要 @Valid
    @NotNull
    private ThreadPoolConfig threadPool = new ThreadPoolConfig();

    // getter/setter...
}
```

如果配置不满足校验规则，应用启动时会抛出`BindValidationException`并列出所有校验失败信息，**立即失败而非运行时才发现问题**。

### 18.3.6 松散绑定规则

Spring Boot支持**极为灵活的**属性名匹配，称为"松散绑定（Relaxed Binding）"：

```yaml
# 以下四种写法都能绑定到 maxRetry 属性：
app:
  max-retry: 3        # 短横线（kebab-case）— YAML/Properties 最标准
  maxRetry: 3         # 驼峰（camelCase）— Java属性名
  max_retry: 3         # 下划线（snake_case）— 某些环境变量风格
  MAX_RETRY: 3         # 全大写下划线 — 环境变量风格
```

推荐的写法规范：

| 文件类型 | 推荐命名风格 | 示例 |
|---------|-------------|------|
| `application.yml` | 短横线（kebab-case） | `max-retry: 3` |
| `application.properties` | 短横线或驼峰 | `app.max-retry=3` |
| Java字段名 | 驼峰（camelCase） | `private int maxRetry;` |
| 环境变量 | 全大写下划线 | `APP_MAX_RETRY=3` |

Spring Boot会自动完成所有这些格式之间的转换，你不需要手动写任何映射代码。

### 18.3.7 @Value vs @ConfigurationProperties 完整对比

| 对比维度 | `@Value` | `@ConfigurationProperties` |
|---------|----------|---------------------------|
| **绑定方式** | 逐个字段绑定 | 批量绑定（整个类） |
| **松散绑定** | ❌ 不支持（仅短横线） | ✅ 全支持 |
| **SpEL表达式** | ✅ 支持（如`#{systemProperties['java.home']}`） | ❌ 不支持 |
| **JSR-303校验** | ❌ 需要配合`@Validated`且有限制 | ✅ 原生支持 |
| **复杂类型（List/Map）** | ❌ 配置繁琐，需写完整路径 | ✅ 自动映射 |
| **IDE自动提示** | ❌ 无提示 | ✅ 需要`configuration-processor`依赖 |
| **元数据生成** | ❌ 不生成 | ✅ 自动生成`spring-configuration-metadata.json` |
| **适用场景** | 单个值、SpEL计算、少量配置 | 一组相关配置（>3个属性） |
| **代码量** | 每个属性一行 | 一个类封装全部 |

**使用建议**：

- 如果你的`application.yml`中有`app.xxx`这样的一组自定义配置 → 用`@ConfigurationProperties`
- 如果只需要注入一两个标准配置（如`server.port`） → 用`@Value`
- 如果需要SpEL表达式动态计算 → 用`@Value`
- **核心原则**：能归类的配置不要散装

## 18.4 配置优先级

### 18.4.1 完整优先级表

Spring Boot的配置来源多达14种，按优先级从高到低排列：

| 优先级 | 配置来源 | 说明 |
|--------|---------|------|
| 1（最高） | 命令行参数 `--server.port=9090` | 运维紧急调整 |
| 2 | `SPRING_APPLICATION_JSON` 环境变量 | JSON格式批量传配置 |
| 3 | Servlet初始化参数 | `web.xml`中的`<init-param>` |
| 4 | JNDI属性 | `java:comp/env` |
| 5 | Java系统属性 `-Dserver.port=9090` | `System.getProperties()` |
| 6 | 操作系统环境变量 | 如`SERVER_PORT=9090` |
| 7 | `RandomValuePropertySource` | `${random.int}` 随机值 |
| 8 | **jar包外** `application-{profile}.yml` | 部署时覆盖jar内配置（环境专属） |
| 9 | **jar包内** `application-{profile}.yml` | 环境专属默认配置 |
| 10 | **jar包外** `application.yml` | 部署时覆盖jar内配置（主配置） |
| 11 | **jar包内** `application.yml` | 主配置默认值 |
| 12 | `@PropertySource` 注解引入 | `@PropertySource("classpath:custom.properties")` |
| 13 | `SpringApplication.setDefaultProperties()` | 代码中设的默认值 |
| 14（最低） | `@ConfigurationProperties`类的默认值 | Java字段初始值 |

### 18.4.2 核心记忆口诀

记住以下三句话就能应对90%的场景：

1. **命令行最大**：`--server.port=9090` 覆盖一切
2. **外部盖内部**：jar包外的配置文件覆盖jar包内的
3. **环境盖主配**：`application-prod.yml` 覆盖 `application.yml`

### 18.4.3 实战验证

验证配置优先级的最佳方式是自己动手实验。创建一个简单的Spring Boot应用：

```java
@SpringBootApplication
public class ConfigPriorityDemo implements CommandLineRunner {

    @Value("${test.key:未配置}")
    private String testKey;

    public static void main(String[] args) {
        SpringApplication.run(ConfigPriorityDemo.class, args);
    }

    @Override
    public void run(String... args) {
        System.out.println("test.key = " + testKey);
    }
}
```

**application.yml（jar包内）**：

```yaml
test:
  key: 来自jar内application.yml
```

**实验一：命令行覆盖**

```bash
java -jar demo.jar --test.key=来自命令行
# 输出：test.key = 来自命令行
```

**实验二：jar外配置文件覆盖**

在jar包同级目录创建`application.yml`：
```yaml
test:
  key: 来自jar外application.yml
```
```bash
java -jar demo.jar
# 输出：test.key = 来自jar外application.yml（覆盖了jar内的）
```

**实验三：环境profile覆盖主配置**

```bash
# application-prod.yml 中有 test.key: 来自prod环境
java -jar demo.jar --spring.profiles.active=prod
# 输出：test.key = 来自prod环境
```

### 18.4.4 配置文件外部化

这个特性是运维人员的救星：你可以在**不重新打包**的情况下修改配置。

Spring Boot按以下顺序查找`application.yml`：

```
1. 当前目录的 /config 子目录（如 ./config/application.yml）
2. 当前目录（如 ./application.yml）
3. classpath 中的 /config 包（如 resources/config/application.yml）
4. classpath 根目录（如 resources/application.yml）← 默认位置
```

部署时的最佳实践：

```
/opt/ems/
├── ems.jar                    # Spring Boot可执行jar
├── config/                    # 外部配置目录（优先级最高）
│   └── application-prod.yml   # 生产环境配置
└── logs/                      # 日志目录
```

> 🚨 **坑点：同一配置多处设置 → 排查困难**
> - 当你发现`server.port`不是你预期的值时，可能要从14个来源中排查
> - 调试利器：在`application.yml`中加 `debug: true`，启动时会打印所有配置来源（但不推荐生产环境使用）
> - 更优雅的方式：访问Actuator的`/actuator/env`端点查看实际生效的配置（见第19章）
> - **预防原则**：每个配置项只在一个地方设置，避免在不同来源中重复定义

## 18.5 敏感信息加密 — Jasypt

### 18.5.1 问题场景

```yaml
# ❌ 这是灾难！
spring:
  datasource:
    password: MyProdPassword123!
```

这个配置文件会被提交到Git仓库。任何一个有代码仓库访问权限的人（包括离职员工）都能看到生产数据库的密码。即使你用私有仓库，密码明文存储在配置文件中本身就是安全漏洞。

### 18.5.2 Jasypt解决方案

**Jasypt（Java Simplified Encryption）** 是一个加密库，Spring Boot有官方starter集成。它的工作方式是：

```
配置文件写 ENC(密文) → 启动时用密钥解密 → 内存中是明文 → 安全使用
```

### 18.5.3 集成步骤

**步骤1：添加依赖**

```xml
<dependency>
    <groupId>com.github.ulisesbocchio</groupId>
    <artifactId>jasypt-spring-boot-starter</artifactId>
    <version>3.0.5</version>
</dependency>
```

**步骤2：生成密文**

Jasypt提供了命令行工具来加密。从Maven仓库下载`jasypt-1.9.3.jar`，然后在命令行执行：

```bash
# 语法：
# java -cp jasypt-1.9.3.jar org.jasypt.intf.cli.JasyptPBEStringEncryptionCLI \
#   input="你的真实密码" password="你的加密密钥" algorithm=PBEWithMD5AndDES

# 示例：
java -cp jasypt-1.9.3.jar org.jasypt.intf.cli.JasyptPBEStringEncryptionCLI \
  input="MyProdPassword123!" \
  password="my-secret-key-2024" \
  algorithm=PBEWithMD5AndDES

# 输出：
# ----ENVIRONMENT-----------------
# Runtime: Oracle Corporation OpenJDK 64-Bit Server VM 17.0.2
#
# ----ARGUMENTS-------------------
# input: MyProdPassword123!
# password: my-secret-key-2024
#
# ----OUTPUT----------------------
# BL3fR8xK9mQ2vN7pW5tY1aD6jH4cS0  ← 这就是密文
```

**步骤3：配置文件中使用密文**

```yaml
spring:
  datasource:
    password: ENC(BL3fR8xK9mQ2vN7pW5tY1aD6jH4cS0)
    # 任何需要加密的值都可用ENC()包裹
    username: root  # 用户名一般不需要加密
```

Jasypt自动识别`ENC()`格式并解密。

**步骤4：启动时传入密钥**

```bash
# 密钥绝不能写在配置文件中！！
java -jar ems.jar --jasypt.encryptor.password=my-secret-key-2024

# 或者通过环境变量
export JASYPT_ENCRYPTOR_PASSWORD=my-secret-key-2024
java -jar ems.jar

# systemd服务文件中
[Service]
Environment="JASYPT_ENCRYPTOR_PASSWORD=my-secret-key-2024"
```

### 18.5.4 进阶：自定义加密算法

默认的`PBEWithMD5AndDES`算法已经不够安全。推荐升级为`PBEWITHHMACSHA512ANDAES_256`：

```yaml
jasypt:
  encryptor:
    algorithm: PBEWITHHMACSHA512ANDAES_256
    # 密钥的外部化同上
```

> 🚨 **坑点：加密密钥绝不能写在配置文件中！**
> - 这就像把保险柜密码写在保险柜门上
> - 密钥必须通过**环境变量**或**启动参数**传入
> - 如果配置文件中同时有`jasypt.encryptor.password=xxx`和`ENC(xxx)`，任何人拿到配置文件就能解密

> 🚨 **坑点：密钥泄露 → 所有ENC密文可被解密**
> - 密钥是解密的唯一凭据，一旦泄露，所有`ENC()`密文形同虚设
> - 密钥需要像保护生产数据库密码一样保护
> - **最佳实践**：不同环境使用不同密钥（dev密钥、prod密钥分开）
> - 如果怀疑密钥泄露 → 立即更换密钥 → 重新加密所有密文 → 更新配置文件

### 18.5.5 测试中注入密钥

单元测试可能需要连接数据库，但测试环境不能每次手动传密钥。在测试代码中：

```java
@SpringBootTest
@ExtendWith(SpringExtension.class)
@TestPropertySource(properties = {
    "jasypt.encryptor.password=test-secret-key"
})
class EmployeeServiceTest {
    // 测试方法...
}
```

> 提醒：测试密钥不要与生产密钥相同，且测试密钥仅用于本地和CI环境。

### 18.5.6 代码中编程式加密解密

有时候你可能需要在代码中对部分数据进行加解密（如存储用户敏感字段到数据库），Jasypt也提供了编程式API：

```java
import org.jasypt.encryption.StringEncryptor;
import org.springframework.stereotype.Component;

@Component
public class SensitiveDataHandler {

    private final StringEncryptor stringEncryptor;

    public SensitiveDataHandler(StringEncryptor stringEncryptor) {
        this.stringEncryptor = stringEncryptor;
    }

    public String encrypt(String plainText) {
        return stringEncryptor.encrypt(plainText);
    }

    public String decrypt(String encryptedText) {
        return stringEncryptor.decrypt(encryptedText);
    }
}
```

注意：这和配置文件加密共用同一套密钥。

**第1题**：
最终运行在**9090**端口（properties的值生效）。

原因：Spring Boot先加载`application.yml`，后加载`application.properties`。后加载的Properties会覆盖先加载的YAML中同名配置。所以yml中的8080被properties中的9090覆盖。

但是，**强烈不建议**同时使用两种格式！维护两套配置文件是给自己找麻烦。

**第2题**：

```java
@Component
@ConfigurationProperties(prefix = "mail")
@Validated
public class MailConfig {

    @NotBlank(message = "邮件服务器地址不能为空")
    private String host;

    @Min(value = 1, message = "端口必须为正数")
    @Max(value = 65535, message = "端口不能超过65535")

    public static class SmtpProperties {
        private boolean smtpAuth;
        private boolean smtpStarttlsEnable;

    }

    // getter/setter...
}
```

**第3题**：

- `@Value("${app.max-retry}")`：**仅支持**短横线格式`max-retry`。因为`@Value`使用的是精确匹配，不走Spring Boot的松散绑定。
- `@ConfigurationProperties`的`maxRetry`字段：支持全部四种——`max-retry`、`maxRetry`、`max_retry`、`MAX_RETRY`。

所以`@ConfigurationProperties`支持更多。这也是推荐使用它的原因之一。

**第4题**：

方案一（配置文件外部化）：运维在jar包同级目录创建`config/application-prod.yml`，覆盖jar包内的数据库密码配置。由于jar包外配置优先级更高（优先级8 vs 9/10 vs 11），不需要重新打包。

方案二（环境变量 + Jasypt）：运维将加密密钥通过环境变量`JASYPT_ENCRYPTOR_PASSWORD`传入，jar包内配置文件使用`ENC(密文)`。密码变更时只需更新jar包内的密文（或也放到外部config目录），密钥不变。

两种方案可以结合使用：密钥用环境变量，其他配置用外部config目录。

**第5题**：

`debug: true`会让Spring Boot在启动时将所有自动配置的匹配情况输出到日志（几百到上千行），这在开发环境很有用（排查"为什么这个Bean没创建"），但在生产环境：
1. **日志量巨大**：每次启动产生MB级别的垃圾日志
2. **增加启动时间**：计算和打印报告需要额外时间
3. **暴露内部信息**：包括Bean名称、配置类路径等，有助于攻击者了解系统架构
4. 生产环境应该用更精细的日志控制（如第19章的logback-spring.xml）

**第6题**：

如果加密密钥泄露：
1. **立即更换密钥**：生成新密钥
2. **重新加密所有密文**：用新密钥重新执行jasypt CLI生成新密文
3. **更新所有配置文件**：替换旧密文为新密文
4. **更新密钥存储位置**：确保新密钥的安全存储（KMS、Vault、或至少环境变量）
5. **审计**：排查泄露来源，是配置文件误提交、聊天记录、还是服务器被入侵
6. **考虑使用更安全的方案**：如HashiCorp Vault、AWS Secrets Manager、阿里云KMS等专业的密钥管理服务（Jasypt适合小团队，专业方案适合大企业）

</details>
