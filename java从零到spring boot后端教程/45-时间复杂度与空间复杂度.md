# 45. 时间复杂度与空间复杂度

> 🏃 你去图书馆找一本《三体》。如果书架上的书按首字母排列，你直奔T区，10秒找到。如果所有书随机堆在地上，你可能要找一下午。——同样的数据量，不同的查找策略，耗时天壤之别。**复杂度分析就是给算法"估时"的工具：不跑代码，也能判断它快不快。**

---

## 45.1 为什么需要复杂度分析？

一个很朴素的想法：把两段代码都跑一遍，掐表看谁快，不就行了？

❌ **问题**：
1. **硬件不一致**：你的电脑跑0.1秒，老旧服务器可能跑2秒
2. **数据量不同**：100条数据差0.001秒，100万条可能差几百倍
3. **无法在写代码阶段判断**：总不能每次都部署到生产环境测一下

✅ **我们需要一把"尺子"**——不依赖硬件、不依赖具体数据量，只衡量**算法随数据量增长的变化趋势**。这就是**大O表示法（Big O Notation）**。

---

## 45.2 大O表示法

大O描述的是：**当输入数据量n趋近于无穷大时，算法执行次数的增长趋势。**

它只关心**增长最快的项**，忽略常数和低阶项：

| 实际执行次数 | 大O表示 | 为什么 |
|---|---|---|
| `3` | O(1) | 常数，不随n增长 |
| `2n + 3` | O(n) | n的主系数2和常数3都被忽略 |
| `3n² + 5n + 100` | O(n²) | 只保留最高阶n² |
| `5n³ + 2n²` | O(n³) | 只保留最高阶n³ |

**大O的直觉**：
- O(1)：不管数据多少，时间恒定（哈希查找）
- O(log n)：数据翻倍，时间只增加一点点（二分查找）
- O(n)：数据翻倍，时间也翻倍（遍历数组）
- O(n log n)：数据翻倍，时间增长略多于翻倍（快速排序）
- O(n²)：数据翻倍，时间变4倍（冒泡排序）
- O(2ⁿ)：数据加1，时间翻倍（穷举所有子集，噩梦级别）

## 45.3 常见复杂度详解（附Java代码分析）

### O(1) — 常数时间

```java
public int getFirst(int[] arr) {
    return arr[0];           // 无论arr多长，只执行1步
}
```

无论数组有100个元素还是1亿个，取第一个元素的时间完全一样。HashMap的get/put也是O(1)。

### O(log n) — 对数时间

```java
public int binarySearch(int[] arr, int target) {
    int left = 0, right = arr.length - 1;
    while (left <= right) {
        int mid = left + (right - left) / 2;
        if (arr[mid] == target) return mid;
        else if (arr[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return -1;
}
```

每次比较后，搜索范围减半。**10亿条数据只需约30次比较。** 这是数据库索引查找的底层原理。

### O(n) — 线性时间

```java
public int findMax(int[] arr) {
    int max = arr[0];
    for (int i = 1; i < arr.length; i++) {   // 遍历n-1个元素
        if (arr[i] > max) max = arr[i];
    }
    return max;
}
```

数据翻倍，耗时翻倍。遍历数组、遍历链表、`String.contains()` 都是O(n)。

### O(n log n) — 线性对数时间

```java
// 归并排序是经典的 O(n log n)
// 原理：每次将数组分成两半(log n层)，每层合并需要O(n)
public void mergeSort(int[] arr, int left, int right) {
    if (left >= right) return;
    int mid = left + (right - left) / 2;
    mergeSort(arr, left, mid);        // 递归左半
    mergeSort(arr, mid + 1, right);   // 递归右半
    merge(arr, left, mid, right);     // 合并O(n)
}
```

最好的比较排序算法都在O(n log n)。`Arrays.sort()` 对于对象数组用的TimSort就是O(n log n)。

### O(n²) — 平方时间

```java
public boolean hasDuplicate(int[] arr) {
    for (int i = 0; i < arr.length; i++) {
        for (int j = i + 1; j < arr.length; j++) {  // 嵌套循环
            if (arr[i] == arr[j]) return true;
        }
    }
    return false;
}
```

嵌套两层循环，外层跑n次，内层平均跑n/2次，总计约n²/2次 → O(n²)。

**n=10000时，执行约5000万次比较，肉眼可见卡顿。**

> 🤔 想多一点：上述去重如果用HashSet，只需O(n)。把所有元素放入HashSet（每个O(1)），总开销O(n)。选对数据结构，O(n²)直接变O(n)。

### O(2ⁿ) — 指数时间（红色警戒）

```java
public int fibonacci(int n) {
    if (n <= 1) return n;
    return fibonacci(n - 1) + fibonacci(n - 2);
}
```

计算fib(50)时，递归树有约2^50个节点，你等到宇宙热寂也算不完。需要用动态规划优化到O(n)。

## 45.4 空间复杂度

空间复杂度衡量的是**算法运行过程中额外占用的内存**，同样用大O表示。

```java
// 空间复杂度 O(1)：只用了几个变量，不随n增长
public int sum(int[] arr) {
    int total = 0;
    for (int num : arr) total += num;
    return total;
}

// 空间复杂度 O(n)：创建了一个和输入等长的新数组
public int[] copyArray(int[] arr) {
    int[] newArr = new int[arr.length];  // 分配了n个int的空间
    for (int i = 0; i < arr.length; i++) newArr[i] = arr[i];
    return newArr;
}

// 空间复杂度 O(n)：递归深度为n（调用栈占用）
public int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}
```

**后端工程师的空间敏感场景**：从数据库查出100万行数据，如果全部加载到内存再处理（O(n)空间），可能直接把JVM撑爆（OOM）。应该用游标或分页，逐批处理。

## 45.5 分析Java代码复杂度的实战步骤

面对一段Java代码，按以下顺序分析：

**第一步：看循环嵌套层数**
- 一层for → 基本是O(n)
- 两层for嵌套 → 基本是O(n²)
- for里套binarySearch → O(n log n)

**第二步：看循环变量的变化方式**
- `i++` → 线性
- `i *= 2` → 对数O(log n)
- `while (left <= right) { mid = ... }`  → 大概率O(log n)

**第三步：看集合操作**
- `list.get(i)` → ArrayList O(1)，LinkedList O(n)
- `map.get(key)` → HashMap O(1)
- `set.contains(x)` → HashSet O(1)，TreeSet O(log n)
- `list.contains(x)` → O(n)

**第四步：看递归**
- 每次递归问题规模减半 → O(log n)
- 每次递归问题规模减1 → O(n)
- 每次递归分成两个子问题 → 可能是O(2ⁿ)（如斐波那契）

## 45.6 实战练习：分析以下代码

```java
// 代码A
public void printPairs(int[] arr) {
    for (int i = 0; i < arr.length; i++) {         // n次
        for (int j = 0; j < arr.length; j++) {     // n次
            System.out.println(arr[i] + "," + arr[j]);
        }
    }
}
// 时间复杂度：O(n²)，两层完整嵌套循环

// 代码B
public void printPairsHalf(int[] arr) {
    for (int i = 0; i < arr.length; i++) {         // n次
        for (int j = i + 1; j < arr.length; j++) { // 平均n/2次
            System.out.println(arr[i] + "," + arr[j]);
        }
    }
}
// 时间复杂度：O(n²)，执行次数 ≈ n×(n-1)/2，最高阶仍是n²
// 虽然比代码A快约一半，但大O不区分——都是O(n²)

// 代码C
public int findInSorted(int[] sortedArr, int target) {
    int left = 0, right = sortedArr.length - 1;
    while (left <= right) {                        // 每次减半
        int mid = left + (right - left) / 2;
        if (sortedArr[mid] == target) return mid;
        else if (sortedArr[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return -1;
}
// 时间复杂度：O(log n)，空间复杂度：O(1)
```

## 45.7 ❌ 常见错误

| 错误 | 为什么错 | 正确做法 |
|---|---|---|
| "两个for嵌套一定是O(n²)" | 如果内层循环次数是常数（如`j<5`），是O(n) | 具体分析内层执行次数 |
| "O(n)一定比O(n²)快" | 当n很小时，O(n²)可能更快（常数因子不同） | 大O只关心增长趋势，不关心小n |
| "用了HashMap就一定O(1)" | 如果hashCode()设计不好，冲突严重，退化到O(n) | 理解哈希冲突（第49章） |
| 忽略`list.contains()`的开销 | ArrayList.contains()是O(n) | 频繁查找用HashSet |
| 递归不分析空间复杂度 | 递归深度就是调用栈深度，占空间 | 同时考虑时间和空间 |

## 小结

| 概念 | 一句话解释 | 判断方法 |
|---|---|---|
| 大O表示法 | 描述算法随数据量增长的趋势 | 忽略常数和低阶项 |
| O(1) | 时间不随数据量变化 | 无循环、无递归 |
| O(log n) | 数据翻倍，时间只加一点 | 每次减半的while |
| O(n) | 数据翻倍，时间翻倍 | 一层循环 |
| O(n log n) | 略多于翻倍 | 分治+合并（快排、归并） |
| O(n²) | 数据翻10倍，时间翻100倍 | 两层嵌套循环 |
| 空间复杂度 | 额外占用的内存 | new了多少东西、递归深度 |
