# 第39章 多线程（二）：同步

> 想象两个人在同一个银行账户上操作：一个人存 100 元，另一个人取 100 元。如果同时操作，读取到的余额都可能是旧值——这就是**竞态条件**（Race Condition）。本章学的同步机制，就是给共享资源加一把"锁"，一次只允许一个人操作。

---

## 39.1 线程安全问题演示

### 39.1.1 一个经典 Bug

```java
public class Counter {
    private int count = 0;

    public void increment() {
        count++;  // 看起来一行，实际是三步：读 → 加1 → 写
    }

    public int get() {
        return count;
    }
}

// 两个线程各加 10000 次
public static void main(String[] args) throws InterruptedException {
    Counter counter = new Counter();

    Thread t1 = new Thread(() -> {
        for (int i = 0; i < 10000; i++) {
            counter.increment();
        }
    });
    Thread t2 = new Thread(() -> {
        for (int i = 0; i < 10000; i++) {
            counter.increment();
        }
    });

    t1.start();
    t2.start();
    t1.join();
    t2.join();

    System.out.println(counter.get()); // 期望 20000，实际可能 18732
}
```

**为什么会出错？** `count++` 不是原子操作。两个线程可能同时读到 `count=42`，各加 1 后都写成 43——丢失了一次递增。

## 39.2 synchronized

> 🏷️ 此术语需进附录：**synchronized**——Java 内置的互斥锁机制。被 synchronized 修饰的代码块或方法，同一时刻只能有一个线程执行。synchronized 锁的是**对象**，而不是代码。

### 39.2.1 三种用法

```java
// 1. 同步代码块（锁指定对象）—— 推荐，粒度可控
public void method() {
    synchronized (this) {  // 锁当前实例
        // 临界区
    }
}

// 2. 同步实例方法（锁 this）
public synchronized void method() {
    // 等价于 synchronized(this) { 整个方法体 }
}

// 3. 同步静态方法（锁 Class 对象）
public static synchronized void method() {
    // 锁的是 Counter.class，不是实例
}
```

### 39.2.2 修复计数器

```java
public class SafeCounter {
    private int count = 0;
    private final Object lock = new Object();  // 专用锁对象

    public void increment() {
        synchronized (lock) {
            count++;
        }
    }

    public int get() {
        synchronized (lock) {
            return count;
        }
    }
}
// 现在输出一定是 20000
```

### 39.2.3 对象锁 vs 类锁

```java
public class Demo {
    // 实例方法 —— 锁对象实例（对象锁）
    public synchronized void instanceMethod() {
        // 同一实例上的所有 synchronized 实例方法互斥
    }

    // 静态方法 —— 锁 Class 对象（类锁）
    public static synchronized void staticMethod() {
        // 所有线程调用此静态方法都需要竞争同一把锁
    }
}

// 对象锁 和 类锁 是两把不同的锁，不互斥！
// 线程 A 调用 instanceMethod() 和 线程 B 调用 staticMethod() 可以同时执行
```

### 39.2.4 synchronized 的可重入性

同一个线程可以**多次获取**自己已经持有的锁：

```java
public class Reentrant {
    public synchronized void methodA() {
        System.out.println("进入 A");
        methodB();  // A 已经持有锁，进入 B 时自动放行
    }

    public synchronized void methodB() {
        System.out.println("进入 B");  // 不会死锁！
    }
}
```

这就是**可重入锁**——如果不可重入，`methodA` 调用 `methodB` 会死锁（自己等待自己释放锁）。

## 39.3 volatile

> 🏷️ 此术语需进附录：**volatile**——Java 的轻量级同步机制。保证变量的**可见性**（一个线程修改后，其他线程立即看到）和**禁止指令重排序**。但**不保证原子性**。

### 39.3.1 可见性问题

```java
// ❌ 没有 volatile，线程可能永远停不下来
public class FlagDemo {
    private static boolean running = true;  // 普通变量

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            while (running) {
                // 做一些工作
            }
            System.out.println("线程结束");
        });
        worker.start();

        Thread.sleep(1000);
        running = false;  // worker 线程可能永远看不到这个修改！
        System.out.println("已设为 false");
    }
}
```

原因：每个线程有自己的工作内存（CPU 缓存），`running` 的修改可能在主线程的缓存中，worker 线程读到的是自己缓存中的旧值。

```java
// ✅ 加上 volatile
private static volatile boolean running = true;
// 任何线程对 running 的修改，其他线程立即可见
```

### 39.3.2 volatile 不保证原子性

```java
private static volatile int count = 0;

// ⚠️ 即使有 volatile，count++ 仍然不是线程安全的！
count++;  // 读、改、写三步，volatile 只保证每一步看到的值是最新的
           // 但三步之间其他线程可能介入
```

**volatile 适用场景**：
- 状态标志位（如上面的 `running`）
- 双重检查锁定（DCL）单例模式中的 `instance` 变量
- 独立观察（一个线程写，多个线程读）

```java
// volatile 的正确用法：双重检查锁定单例
public class Singleton {
    private static volatile Singleton instance;  // 必须 volatile！

    public static Singleton getInstance() {
        if (instance == null) {  // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {  // 第二次检查
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

> 🤔 **想多一点**：DCL 单例中为什么必须 `volatile`？`instance = new Singleton()` 不是原子操作，它分为：①分配内存 ②初始化对象 ③赋值给 instance。CPU 可能重排序为 ①→③→②，此时另一个线程看到 instance 非 null 但对象还没初始化完。volatile 禁止了这种重排序。

## 39.4 wait / notify / notifyAll

这三个方法是 `Object` 类的方法，必须在 `synchronized` 块中调用。

### 39.4.1 生产者-消费者模型

```java
import java.util.LinkedList;
import java.util.Queue;

public class ProducerConsumer {
    private final Queue<Integer> queue = new LinkedList<>();
    private final int CAPACITY = 5;

    // 生产者
    public void produce() throws InterruptedException {
        int value = 0;
        while (true) {
            synchronized (queue) {
                while (queue.size() == CAPACITY) {
                    System.out.println("队列满了，生产者等待...");
                    queue.wait();  // 释放锁，等待通知
                }
                queue.add(value);
                System.out.println("生产：" + value);
                value++;
                queue.notifyAll();  // 唤醒所有等待的线程
            }
            Thread.sleep(500);
        }
    }

    // 消费者
    public void consume() throws InterruptedException {
        while (true) {
            synchronized (queue) {
                while (queue.isEmpty()) {
                    System.out.println("队列空了，消费者等待...");
                    queue.wait();
                }
                int value = queue.poll();
                System.out.println("消费：" + value);
                queue.notifyAll();
            }
            Thread.sleep(1000);
        }
    }
}
```

关键规则：
- `wait()` 释放锁，线程进入 WAITING 状态
- `notify()` 随机唤醒一个等待线程
- `notifyAll()` 唤醒所有等待线程
- 必须在 `synchronized` 块中调用，否则抛 `IllegalMonitorStateException`
- `wait()` 应该放在 `while` 循环中检查条件（防止虚假唤醒）

## 39.5 死锁演示

死锁是多线程最经典的 bug——两个线程互相等待对方释放锁，永远卡住。

```java
public class DeadlockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (lockA) {
                System.out.println("t1 持有 lockA");
                try { Thread.sleep(100); } catch (InterruptedException e) {}

                synchronized (lockB) {  // 等 lockB，但被 t2 持有
                    System.out.println("t1 持有 lockA 和 lockB");
                }
            }
        });

        Thread t2 = new Thread(() -> {
            synchronized (lockB) {
                System.out.println("t2 持有 lockB");
                try { Thread.sleep(100); } catch (InterruptedException e) {}

                synchronized (lockA) {  // 等 lockA，但被 t1 持有
                    System.out.println("t2 持有 lockB 和 lockA");
                }
            }
        });

        t1.start();
        t2.start();
        // 两个线程互相等待，永远卡住
    }
}
```

**死锁的四个必要条件**（缺一不可）：
1. **互斥**：资源只能被一个线程占用
2. **持有并等待**：线程持有资源的同时等待其他资源
3. **不可剥夺**：资源不能被强制释放
4. **循环等待**：形成环形等待链

**预防死锁**：
- 按固定顺序获取锁（例如总是先 lockA 再 lockB）
- 使用 `tryLock()` 带超时（`java.util.concurrent.locks.Lock`）
- 减少锁的持有时间

## 39.6 ❌ 常见错误

### 错误1：用 String 当锁对象

```java
// ❌ 危险！字符串常量池会导致意想不到的共享
synchronized ("lock") {
    // "lock" 是字符串常量池中的同一个对象
    // 不同模块可能用了相同字符串，造成意外的锁竞争！
}

// ✅ 用专用 Object
private final Object lock = new Object();
synchronized (lock) { ... }
```

### 错误2：synchronized 加在错误的粒度上

```java
// ❌ 锁对象可能被外部修改！
public synchronized void method() {
    // 锁的是 this，外部代码可以 synchronized(obj) 抢占锁
}

// ✅ 用私有锁对象
private final Object lock = new Object();
public void method() {
    synchronized (lock) { ... }
}
```

### 错误3：wait() 用 if 而非 while

```java
synchronized (queue) {
    // ❌ 虚假唤醒后会继续执行，条件可能已经不满足了
    if (queue.isEmpty()) {
        queue.wait();
    }
    queue.poll();

    // ✅ 用 while，被唤醒后重新检查条件
    while (queue.isEmpty()) {
        queue.wait();
    }
    queue.poll();
}
```

### 错误4：不在 synchronized 中调用 wait/notify

```java
// ❌ IllegalMonitorStateException
queue.wait();

// ✅
synchronized (queue) {
    queue.wait();
}
```

## 39.7 小结

| 知识点 | 一句话总结 |
|--------|-----------|
| synchronized | 互斥锁，保证原子性和可见性，可重入 |
| 锁的对象 | 实例方法锁 this，静态方法锁 Class，代码块锁指定对象 |
| volatile | 只保证可见性，不保证原子性，适合状态标志位 |
| wait/notify | 线程间通信，wait 释放锁，必须在 synchronized 中 |
| 死锁 | 两线程互相等待对方释放锁，按固定顺序获取锁可避免 |
| 虚假唤醒 | wait() 必须放在 while 循环中检查条件 |

## 39.8 自测题

**1.** 以下代码线程安全吗？如果不安全，如何修改？

```java
public class Counter {
    private volatile int count = 0;
    public void increment() { count++; }
}
```

**2.** 为什么 `wait()` 要放在 `while` 循环中而不是 `if` 中？

**3.** 写出以下场景可能的死锁情况，并说明如何避免：

```java
// 线程 A 调用 transfer(from=A, to=B)
// 线程 B 调用 transfer(from=B, to=A)
synchronized (from) {
    synchronized (to) {
        // 转账
    }
}
```

> 答案提示：1. 不安全——volatile 不保证 `count++` 的原子性，需加 synchronized；2. 防止虚假唤醒——被唤醒时条件可能已变化；3. 两个线程按不同顺序获取锁导致死锁，应统一获取锁的顺序（如按账户 ID 排序后始终先锁小的）。
