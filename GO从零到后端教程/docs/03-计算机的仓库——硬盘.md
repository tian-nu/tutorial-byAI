# 第3章 · 计算机的仓库——硬盘

> "内存快但断电就忘，硬盘慢但能记住。但到底有多慢？为什么数据库服务器都要用SSD？1纳秒和10毫秒之间差了多少个数量级？"

---

## 3.1 硬盘和内存的区别

### 一个比喻

| 角色 | 比喻 | 特点 |
|------|------|------|
| **内存** | 餐桌 | 菜放在餐桌上随时吃（快），但吃完就没了（断电消失） |
| **硬盘** | 冰箱 | 食材长期存放（持久），但每次拿要开门（慢） |

你正在写的文件在**内存**里（你随时能编辑）。你点了"保存"后，它被写到了**硬盘**上（关电脑也不丢）。如果突然断电，内存里的内容全没了，但硬盘上的还在。

### 为什么一定要分层

你不能把硬盘当内存用，因为硬盘比内存慢**10万倍到100万倍**。

你也不能把内存当硬盘用，因为内存断电数据就没了，而且太贵——1TB内存的价格大约是1TB SSD的10~20倍。

计算机就是靠"分层存储"来平衡速度、容量和价格。

## 3.2 HDD 和 SSD

### HDD：机械硬盘（老式）

HDD（Hard Disk Drive）就像一个**唱片机**：

```
        ┌─────────────┐
        │  磁头臂 →    │ ← 这个臂可以前后移动
        │    ↓         │
        │  ╔═══════╗   │
        │  ║ 磁盘片 ║  │ ← 这个盘高速旋转（5400/7200 RPM）
        │  ╚═══════╝   │
        └─────────────┘
```

- **盘片**：金属或玻璃的圆盘，上面有磁性涂层（存数据的）
- **磁头**：悬浮在盘片上方的读写装置（不是接触的！）
- **磁头臂**：移动磁头到正确的位置

读数据的过程：
1. 磁头臂移动到正确的"磁道"（半径位置）→ 这叫**寻道**，约5~10毫秒
2. 等盘片转到正确的"扇区" → 这叫**旋转延迟**，约2~4毫秒
3. 开始读写数据

这就是机械运动——寻道和旋转是**物理动作**，所以慢。

**优点**：便宜（1TB约200~300元）、技术成熟、数据恢复相对容易
**缺点**：慢、怕震动（摔一下磁头可能撞到盘片）、有噪音、耗电

### SSD：固态硬盘（新式）

SSD（Solid State Drive）是纯**电子芯片**——没有活动部件：

```
┌────────────────────┐
│  控制器芯片         │
│  ┌──────┐┌──────┐ │
│  │NAND闪存││NAND闪存│ │
│  │ 芯片1 ││ 芯片2 │ │
│  └──────┘└──────┘ │
│  ┌──────┐┌──────┐ │
│  │NAND闪存││NAND闪存│ │
│  │ 芯片3 ││ 芯片4 │ │
│  └──────┘└──────┘ │
└────────────────────┘
```

NAND闪存通过**电子**而不是磁来存储数据，读取数据只需要电信号——不需要任何机械运动。

**优点**：极快（比HDD快几十倍到几百倍）、安静、不怕震动、省电
**缺点**：贵（但越来越便宜）、写入寿命有限（但正常使用几年没问题）、数据损坏后难恢复

### 对比

| 特性 | HDD（机械硬盘） | SSD（固态硬盘） |
|------|----------------|----------------|
| 随机读取速度 | ~0.5~2 MB/s | ~500~7000 MB/s |
| 随机读取延迟 | ~10毫秒 | ~0.1毫秒 |
| 噪音 | 有（转盘声） | 无 |
| 抗震 | 差（怕摔） | 好 |
| 价格/GB | ~0.3元 | ~0.5~1元 |
| 写入寿命 | 几乎无限 | 有限（但够用） |
| 适合场景 | 大容量存储、备份 | 操作系统、数据库 |

### 为什么数据库服务器要用SSD

数据库需要**频繁的随机读写**（查不同的行、更新不同的记录）。HDD每次寻道都要5~10毫秒，对于数据库来说太慢了。

- HDD每秒约100~200次随机IO
- SSD每秒可达10万~100万次随机IO

差距是1000倍以上。所以现在所有正经的数据库服务器（MySQL、PostgreSQL、MongoDB等）都跑在SSD上。

## 3.3 文件是什么

### 文件 = 硬盘上的一段数据

从物理角度看，一个文件就是硬盘上一段连续的（或者不连续的）磁化区域（HDD）或电荷状态（SSD）。

从操作系统的角度看，文件是一个抽象——你不用关心它存在硬盘的哪个位置，你只需要**路径**和**文件名**就能找到它。

### 文件系统：管理文件的那套规则

文件系统就像**图书馆的管理系统**：哪本书在哪个书架、怎么找、怎么借、怎么还。

常见文件系统：

| 文件系统 | 哪个系统用 | 特点 |
|----------|-----------|------|
| NTFS | Windows | 日志、权限、压缩 |
| ext4 | Linux | 稳定、通用 |
| XFS | Linux（CentOS默认） | 适合大文件 |
| APFS | macOS | 快照、加密、空间共享 |

### 路径：从根目录出发找文件

**绝对路径**：从根目录 `/`（Linux/Mac）或盘符 `C:\`（Windows）开始

```
Linux:   /home/zhangsan/projects/hello.go
Windows: C:\Users\zhangsan\projects\hello.go
```

**相对路径**：从当前所在目录出发

```
当前在 /home/zhangsan/
./projects/hello.go   ← 等同于上面的绝对路径
../lisi/file.txt       ← 上一级的lisi目录
```

- `.` = 当前目录
- `..` = 上一级目录

### 在Go里操作文件路径

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
import "path/filepath"

// 拼接路径（自动处理平台的斜杠差异）
path := filepath.Join("home", "zhangsan", "hello.go")
// Linux: home/zhangsan/hello.go
// Windows: home\zhangsan\hello.go

// 获取文件名
filepath.Base("/home/zhangsan/hello.go")  // "hello.go"

// 获取目录部分
filepath.Dir("/home/zhangsan/hello.go")   // "/home/zhangsan"

// 获取扩展名
filepath.Ext("/home/zhangsan/hello.go")   // ".go"
```

## 3.4 磁盘IO为什么慢

### 数量级对比（非常重要！）

这是每一个后端工程师都应该刻在脑子里的数据：

```
操作                         约花费的时间      相对速度
─────────────────────────────────────────────────────────
CPU执行一条指令（L1缓存命中）     ~1 纳秒          ┃（光速）
L2缓存命中                      ~5 纳秒          ┃
L3缓存命中                      ~15 纳秒         ┃
从主内存读取                    ~100 纳秒         ┃
从SSD随机读取                    ~100 微秒         ┃┃┃
从HDD随机读取                    ~10 毫秒          ┃┃┃┃┃┃┃┃┃┃┃┃┃
从另一个数据中心（网络）           ~100 毫秒         ┃┃┃┃┃┃┃┃┃┃┃┃┃┃┃┃┃┃┃┃┃
```

换算一下：

- 1纳秒 = 10⁻⁹秒（十亿分之一秒）
- 1微秒 = 10⁻⁶秒（百万分之一秒）
- 1毫秒 = 10⁻³秒（千分之一秒）

如果CPU执行一条指令是**1秒**：
- 读内存 = **100秒**（约2分钟）
- 读SSD = **100,000秒**（约28小时）
- 读HDD = **10,000,000秒**（约116天）
- 跨数据中心 = **100,000,000秒**（约3年）

**现在你明白为什么"尽量减少IO"是后端性能优化的第一条了吧？**

### 这对你写代码有什么影响

**1. 不要循环里查数据库**

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
// 坏的写法：每条记录去查一次——N次网络IO + 磁盘IO
for _, userID := range userIDs {
    user := db.FindByID(userID)  // 每次都去数据库！
}

// 好的写法：一次查出所有——1次IO
users := db.FindByIDs(userIDs)  // SELECT * FROM users WHERE id IN (1,2,3,...)
```

**2. 用缓存（Redis）避免重复读数据库**

```go
// 每次请求都查数据库 → 慢
user := db.FindByID(userID)

// 先查缓存 → 缓存有就直接返回，没有才查数据库 → 快
user := cache.Get(userID)
if user == nil {
    user = db.FindByID(userID)
    cache.Set(userID, user, 10*time.Minute)
}
```

**3. 批量操作而不是逐条操作**

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
// 坏：逐条插入，100条数据 = 100次IO
for _, user := range users {
    db.Insert(user)
}

// 好：批量插入，100条数据 = 1次IO
db.BatchInsert(users)
```

## 本章小结

| 概念 | 一句话 |
|------|--------|
| 内存 vs 硬盘 | 内存是餐桌（快，临时），硬盘是冰箱（慢，永久） |
| HDD | 有转盘和磁头，物理运动所以慢 |
| SSD | 纯芯片，电子读写所以快 |
| 文件系统 | 管理硬盘上文件的那套规则（NTFS/ext4/APFS） |
| 磁盘IO | 读硬盘比读内存慢10万倍，能省则省 |

---

> 🤔 **记住这些数量级**：CPU 1纳秒 → 内存100纳秒 → SSD 100微秒 → HDD 10毫秒。这些数字会在你设计系统、优化性能时反复指导你——离CPU越近越贵越快，越远越便宜越慢。后端工程师的本质工作就是在这条"速度-成本"线上找最优解。
