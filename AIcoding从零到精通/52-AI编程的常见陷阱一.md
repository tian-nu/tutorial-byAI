你有驾照了，车也能开了，但你有没有发现——一个人开车的时候，最容易犯的错往往不是"技术不好"，而是**"以为自己技术很好"**？

AI编程也一样。当你从"AI帮我写第一行代码"走到"AI帮我写整个项目"，最大的敌人不是AI不够强，而是**你用AI的方式出了问题**。

## 陷阱一：过度依赖——"我的脑子去哪儿了？"

### 症状

你打开Cursor，想写一个函数。你没有思考这个函数应该怎么设计——你直接对AI说："帮我写一个处理用户登录的函数。"

AI写了。你复制粘贴。跑通了。开心。

下一个功能，你又对AI说："帮我写一个处理……"

**三个月后**，你发现自己：
- 离开AI，连一个 `for` 循环都写不利索
- AI改了你的代码，你不知道改了哪里、为什么这么改
- 面试的时候别人问你"这段代码为什么这么写"，你只能回答"AI写的"

这就是**过度依赖**——你把大脑外包给了AI，然后你发现，你的大脑变小了。

### 为什么会这样？

把AI想象成**计算器**。计算器是个好东西，但如果你从小学一年级就用计算器，从来不自己算加减乘除——那你永远不会真正理解"加法是什么"。

AI编程也一样。它可以帮你写代码，但如果你从不动脑思考"这段代码为什么这样写"，你就永远不会真正理解编程。

更可怕的是：**你越不会，AI就越不可靠**。因为当AI犯错的时候（它一定会，还记得第03章的"幻觉"吗？），你根本没有能力发现——你已经退化了。

### 怎么办？

**一句话：永远比AI多想一步。**

在你问AI之前，先花30秒自己在脑子里过一遍：

1. 我要做什么？（一句话说清楚）
2. 这个功能大概分几步？（123列出来）
3. 有哪些边界条件？（输入为空怎么办？网络断了怎么办？）

然后**你带着这三步去指挥AI**，而不是把空白的大脑扔给AI等答案。

```
❌ 坏的打开方式：
你："帮我写一个函数。"
AI：（写了）
你：（复制粘贴，不思考）
→ 你什么都没学到，AI错了你也看不出来。

✅ 好的打开方式：
你："我想写一个函数，输入文件路径，返回文件的行数。
    需要处理文件不存在的情况，返回-1。
    用Python写。"
AI：（写了）
你：（读一遍代码，理解逻辑，跑一下测试）
→ 你在"指导"AI，而不是被AI指导。
```

> **记住：AI是你的工具，不是你大脑的替代品。如果你停止思考，你就变成了AI的附属品。**

## 陷阱二：盲目信任——"AI写的，肯定没问题"

### 症状

最典型的场景：

1. AI帮你写了一个"用户登录"功能，你直接上线了
2. 三天后，你发现任何人都能用空密码登录
3. 或者更惨——用户数据泄露了，因为你没发现AI把密码明文存储了

你质问AI："你为什么这么写？"——AI不会回答你，因为它只是一台猜字机器。**责任在你，不在AI。**

### 为什么这么危险？

AI写的代码和人类写的代码有一个本质区别：**人类写代码的时候是"理解→编写"，AI写代码的时候是"猜字→输出"。**

AI不知道你的系统有什么安全要求，不知道你的数据库密码在哪里，不知道这段代码会跑在几百万用户的服务器上。AI只知道——"根据我见过的代码模式，你应该写成这样。"

换句话说：**AI对"正确"的定义是"看起来对"，不是"真的对"。**

### 真实案例

有程序员让AI写了一段处理用户上传文件的代码：

```python
# ❌ AI生成的代码 —— 直接上线了
from flask import Flask, request

app = Flask(__name__)

@app.route('/upload', methods=['POST'])
def upload_file():
    file = request.files['file']
    file.save(file.filename)  # ← 危险！
    return "上传成功"
```

看起来很合理，对吧？问题是：

- 用户上传了一个叫 `../../../etc/passwd` 的文件，你的服务器系统文件被覆盖了
- 用户上传了一个10GB的文件，你的服务器硬盘爆了
- 用户上传了一个 `.exe` 病毒文件，你的服务器变成了病毒分发站

**AI不会告诉你这些风险。它只是把"看起来合理的代码"写出来了。** 安全性、边界检查——它一概不负责。

### 怎么办？

**一句话：永远不要信任AI写的代码，永远跑一遍再上线。**

```
AI写完代码 → 你读一遍（理解逻辑）
           → 你跑一遍（验证功能）
           → 你检查安全（有密钥吗？有注入风险吗？有边界问题吗？）
           → 测试通过 → 上线
```

把这四步刻在脑子里。跳过任何一步，都可能在某个深夜把你从床上叫起来。

## 陷阱三：幻觉中毒——"我调了3小时，才发现是AI瞎编的"

### 症状

你让AI帮你写一个"读取Excel文件"的功能。AI信心满满地输出：

```python
# ❌ AI的幻觉输出
import pandas as pd
from pandas.io.excel import ExcelMagicParser  # ← 不存在！

data = ExcelMagicParser("data.xlsx").parse_all()
```

你复制粘贴。运行。报错：`ModuleNotFoundError: No module named 'pandas.io.excel'`

你觉得是环境问题。你重装了pandas。还是报错。你觉得是版本问题。你降级了pandas。还是报错。你去Google搜 `ExcelMagicParser`——搜不到。

**三个小时过去了。** 你突然意识到：**这个类根本就不存在，是AI编的。**

这就是"幻觉中毒"——你花了大量时间调试一个根本不存在的"错误"，最终发现不是你的问题，是AI编造了一个虚假的现实。

### 为什么会中招？

因为AI有两个特点让你防不胜防：

1. **它说得太自信了**：AI不会说"我猜可能是这样"。它会说"你应该用 `ExcelMagicParser`"，语气坚定得像在背诵官方文档。这本就是幻觉的特征——AI自己都不知道自己在胡说。
2. **你懒得查文档**：因为AI太快了，你习惯了"问AI → 复制 → 粘贴"，你已经忘了"查官方文档"这个动作。当AI编造了一个不存在的API，你不会先去查文档验证——你会先怀疑自己的环境配置。

### 怎么办？

**记住一个铁律：AI说的任何你"没听过"的函数/类/参数名，先验证再用。**

```
验证三步走：
1. 这个名字你以前听过吗？
   → 没听过 → 警惕！
2. 把函数名复制到Google / 官方文档搜索
   → 搜不到 → 幻觉！
3. 搜到了 → 看文档确认参数是否正确
   → 参数不对 → 还是幻觉（部分幻觉）
```

还有一个更简单的办法：

```bash
# 在Python中直接验证
>>> import pandas
>>> hasattr(pandas.io, 'excel')
True
>>> dir(pandas.io.excel)  # 看看里面有什么
# 如果看不到 ExcelMagicParser → 幻觉
```

**养成习惯：AI生成的代码跑之前，先看一眼用了哪些库、调了哪些方法。不认识的就查一下。** 这会帮你省下无数个"调试3小时"的下午。

## 陷阱四：修复死循环——"改了5次还没对，我要疯了"

### 症状

你遇到一个bug，让AI帮你修。

AI给了方案A → 运行，还是报错 → 你把错误信息给AI → AI给了方案B → 运行，报另一个错 → 你又把错误信息给AI → AI给了方案C → 又报错……

**循环了5轮**。每次AI都会说："抱歉，我重新分析一下这个问题……"，然后给出一个新的方案。你越来越焦虑，AI越来越迷茫，bug还是那个bug。

最后你崩溃了："AI根本不行！"

### 为什么会陷入死循环？

这不是AI的问题——**这是你的问题。**

当你反复让AI修同一个bug超过3次，说明以下至少有一项已经发生：

1. **AI完全理解错了你的问题**：它在你给的错误方向上越走越远
2. **你的上下文已经被污染了**：来回对话太多次，AI已经不知道前面说过什么了
3. **问题本身超出了AI的能力范围**：比如涉及复杂的线程竞态、特定框架的隐藏行为
4. **你给的错误信息不完整**：AI在看一个模糊的错误描述盲猜

**继续让AI改下去，就像用头撞同一堵墙，期望墙会先碎。** 不会的。

### 怎么办？

**铁律：同一个问题，AI改3次还没对，立刻停下来。**

停下来之后做什么：

```
步骤一：换一个思路
  → 把刚才的对话清空，开一个全新的AI对话
  → 换个说法描述你的问题
  → 换个角度：能不能换个实现方式？而不是死磕当前方案？

步骤二：拆小问题
  → 你的bug可能不是"一个bug"，而是"三个bug叠在一起"
  → 把大问题拆成3个小问题，一个一个让AI修
  → 每个小问题修好后，验证通过再进行下一个

步骤三：求助人类
  → 去Stack Overflow搜一下报错信息
  → 去GitHub Issues看看有没有人遇到过同样的问题
  → 问一个有经验的同事/朋友
  → 有时候AI真的不如一个搜对关键词的人类

步骤四：自己理解代码
  → 停下来，自己一行一行读AI写的代码
  → 很多时候你读一遍就发现：哦，这里逻辑就不对
  → AI可能在修A的时候把B改坏了，你没发现
```

> **记住：AI改3次还不对，不是AI不行，是你的使用方式出问题了。停下来，换思路，拆问题——比死磕管用100倍。**

## 小结

| 陷阱 | 一句话 | 应对策略 |
|------|--------|----------|
| 过度依赖 | 把大脑外包给AI，能力退化 | 动脑30秒再问AI，你指导AI不是AI指导你 |
| 盲目信任 | AI写的代码直接上线不测试 | 读一遍、跑一遍、检查安全、测试通过再上线 |
| 幻觉中毒 | 调试3小时发现是AI编造的东西 | AI说的没听过的名字先验证再用 |
| 修复死循环 | 来回改3次+还不对还在改 | 停下来！开新对话、拆问题、求助人类、自己读代码 |

2. AI给你修了4次bug还没修好，你接下来应该怎么做？列出至少三个具体动作。

3. "过度依赖"和"不用AI"是两个极端。你觉得什么才是"正确地使用AI"？用你自己的话描述一下。

> **答案**：
> 1. 首先怀疑这是幻觉——Python的 `dict` 确实没有 `length` 属性（正确的是 `len(dict)`）。把这个方法名复制到Python文档或Google搜索确认。如果搜不到官方用法，就是幻觉。把错误信息和你的分析一起发给AI让它修正。
> 2. ①停掉当前对话，开一个新对话，换个说法描述问题；②把大问题拆成2-3个小问题，逐个解决；③去Stack Overflow搜报错信息，或问有经验的同事；④自己逐行读代码，理解逻辑后再让AI改。
> 3. 参考答案：正确使用AI = 把它当副驾驶而不是自动驾驶。你负责方向（架构设计、需求分析）、AI负责执行（写代码、修bug）。你永远比AI多想一步，AI的输出你永远验证一遍。你不是不用AI，也不是依赖AI——你是"驾驭"AI。
