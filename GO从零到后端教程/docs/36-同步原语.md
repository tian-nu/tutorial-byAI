# 第36章 · 同步原语

> "Channel像流水线——有序、清晰，但有时候你只需要一把'锁'来保护一个共享变量，不想搞那么复杂的流水线。Go理解这一点，所以在Channel之外还提供了传统的同步工具：互斥锁、读写锁、WaitGroup……"

---

## 36.1 sync.Mutex：互斥锁

### 什么是竞态条件

```go
var counter int

func increment() {
    counter++
}

func main() {
    for i := 0; i < 1000; i++ {
        go increment()
    }
    time.Sleep(1 * time.Second)
    fmt.Println(counter)
}
```

你期望输出 `1000`，实际可能输出 `987` 或 `993` 或别的什么。

为什么？因为 `counter++` 不是原子操作——它分三步：
1. 读取当前值
2. 加1
3. 写回

两个Goroutine同时读到 `counter=42`，各自加1，各自写回 `43`——明明是两次加1，结果只加了1。

这叫**竞态条件（Race Condition）**。

### Mutex登场

**比喻：厕所门锁**

家里只有一个厕所。你进去，锁上门——别人进不来。你用完，解锁——别人才能进去。

`sync.Mutex` 就是这把锁：

```go
var (
    counter int
    mu      sync.Mutex
)

func increment() {
    mu.Lock()
    counter++
    mu.Unlock()
}

func main() {
    for i := 0; i < 1000; i++ {
        go increment()
    }
    time.Sleep(1 * time.Second)
    fmt.Println(counter)
}
```

输出：`1000`

`mu.Lock()`——锁门。其他Goroutine运行到 `mu.Lock()` 时会被阻塞，直到有人 `Unlock()`。

### defer Unlock：最佳实践

```go
func updateBalance(accountID int, amount float64) {
    mu.Lock()
    defer mu.Unlock()

    balance := loadBalance(accountID)
    balance += amount
    saveBalance(accountID, balance)
}
```

用 `defer mu.Unlock()` 保证无论函数怎么退出（正常返回、提前return、甚至panic），锁都会被释放。不这么做的话，一旦中间某步出错了return了但忘了Unlock，程序就死锁了。

## 36.2 sync.RWMutex：读写锁

Mutex的问题是：不管是读还是写，都得排队。但"读取"操作本身不会修改数据，完全可以**多个人同时读**——只有"写"需要独占。

**比喻：图书馆**

一本书，多个人可以同时借阅（读）——互不干扰。但如果有人要修订这本书（写），就必须等所有人还书，然后独占。

```go
var (
    data map[string]string
    rwMu sync.RWMutex
)

func readData(key string) string {
    rwMu.RLock()
    defer rwMu.RUnlock()
    return data[key]
}

func writeData(key, value string) {
    rwMu.Lock()
    defer rwMu.Unlock()
    data[key] = value
}
```

| 操作 | 锁方法 | 解锁方法 |
|------|--------|---------|
| 读 | `RLock()` | `RUnlock()` |
| 写 | `Lock()` | `Unlock()` |

规则：
- 多个读可以同时持有锁（`RLock` 之间不互斥）
- 写锁是独占的（`Lock` 和任何 `Lock` / `RLock` 都互斥）
- 写锁优先级高于读锁（有Goroutine在等写锁时，新的读者会被阻塞——防止写者饥饿）

### 何时用RWMutex？

当你的数据**读多写少**时，RWMutex能显著提升并发性能。比如一个配置缓存——每分钟可能被读几千次，但每小时才更新一次——用Mutex所有读全排队，用RWMutex读可以并发。

## 36.3 sync.WaitGroup：等待一组Goroutine

还记得上一章我们用 `time.Sleep` 等Goroutine跑完吗？丑陋且不可靠。`WaitGroup` 是优雅的解决方案。

### 比喻：旅游团

导游带团出去逛，团员各自去拍照。导游不需要一个个去找——他在集合点等着，到了约定时间，清点人数，人到齐了就走。

```go
func main() {
    var wg sync.WaitGroup

    for i := 1; i <= 5; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            fmt.Printf("Goroutine %d 开始工作\n", id)
            time.Sleep(time.Duration(id) * 100 * time.Millisecond)
            fmt.Printf("Goroutine %d 完成工作\n", id)
        }(i)
    }

    wg.Wait()
    fmt.Println("所有Goroutine都完成了")
}
```

### 三个关键方法

| 方法 | 作用 | 比喻 |
|------|------|------|
| `Add(n)` | 计数器+n，表示有n个任务 | 导游说"等n个人" |
| `Done()` | 计数器-1，表示一个任务完成 | 一个团员归队 |
| `Wait()` | 阻塞直到计数器归零 | 导游在集合点等 |

### 注意事项

**1. Add必须在启动Goroutine之前调用**

```go
var wg sync.WaitGroup
wg.Add(1)
go func() {
    defer wg.Done()
}()
wg.Wait()
```

如果把 `wg.Add(1)` 放在Goroutine里面，`wg.Wait()` 可能在 `Add` 还没执行时就返回了（因为计数器是0）。

**2. 传指针，不传值**

```go
func worker(wg *sync.WaitGroup, id int) {
    defer wg.Done()
}

func main() {
    var wg sync.WaitGroup
    wg.Add(1)
    go worker(&wg, 1)
    wg.Wait()
}
```

`sync.WaitGroup` 必须通过指针传递——值传递会复制，一个WaitGroup的副本的 `Done()` 不会影响原版的计数器。

## 36.4 sync.Once：只执行一次

有些初始化操作只需要做一次——不管有多少Goroutine同时调用：

```go
var (
    config *Config
    once   sync.Once
)

func loadConfig() *Config {
    once.Do(func() {
        fmt.Println("加载配置...")
        config = &Config{Host: "localhost", Port: 8080}
    })
    return config
}

func main() {
    for i := 0; i < 10; i++ {
        go loadConfig()
    }
    time.Sleep(1 * time.Second)
}
```

不管 `loadConfig()` 被多少个Goroutine同时调用，`"加载配置..."` 只会打印**一次**。

### 比喻：班级点名

老师问"班长是谁？"——全班的回答应该是同一个人。不能因为问得多就变出多个班长。

`sync.Once` 确保：**不管调用多少次，函数只执行一次。**

### 实际用途

- 单例模式（只创建一个数据库连接池）
- 初始化全局配置
- 注册驱动

## 36.5 sync/atomic：原子操作

对于简单的计数器，Mutex有点"杀鸡用牛刀"——原子操作更轻量：

```go
import "sync/atomic"

var counter int64

func increment() {
    atomic.AddInt64(&counter, 1)
}

func main() {
    var wg sync.WaitGroup
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            atomic.AddInt64(&counter, 1)
        }()
    }
    wg.Wait()
    fmt.Println(counter)
}
```

输出：`1000`

`atomic.AddInt64` 是**原子操作**——读取、加、写回，三步合成一步，不可分割。

### 常用原子操作

```go
var n int64

atomic.AddInt64(&n, 1)
atomic.StoreInt64(&n, 42)
val := atomic.LoadInt64(&n)
swapped := atomic.CompareAndSwapInt64(&n, 42, 43)
```

`CompareAndSwap`（CAS）是"比较并交换"：如果当前值等于预期值，就换成新值。这是很多无锁数据结构的基础。

### atomic vs Mutex

| 特性 | atomic | Mutex |
|------|--------|-------|
| 适用场景 | 简单的计数器、标志位 | 复杂的多步操作 |
| 性能 | 更快（无锁） | 较慢（有锁争用） |
| 使用难度 | 简单但局限 | 灵活但容易出错 |
| 保护范围 | 单个变量 | 一段代码块 |

**选择原则**：
- 只是计数？用 `atomic`
- 需要保护一段代码（比如读→判断→写）？用 `Mutex`

## 🤔 想多一点：Channel vs 同步原语

很多Go初学者会有这样的困惑：既然有Channel，为什么还需要Mutex？

答案是：**不同的场景，不同的工具**。

Channel擅长的是：**Goroutine之间的通信和协调**。比如"生产者-消费者"模式、事件通知、任务分发。

Mutex擅长的是：**保护共享状态**。比如一个全局计数器、一个缓存Map。

一个经验法则：

> 如果你在问"数据从谁流向谁"，用Channel。
> 如果你在问"谁正在碰这个变量"，用Mutex。

两者不是非此即彼的关系——大型项目中，Channel和同步原语往往一起出现。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| Mutex | `Lock()`/`Unlock()`，保护临界区；用 `defer Unlock()` |
| RWMutex | `RLock()`/`RUnlock()` 读不互斥，`Lock()`/`Unlock()` 写独占 |
| WaitGroup | `Add()`/`Done()`/`Wait()`，等待一组Goroutine；传指针 |
| Once | `Do(func())`，只执行一次（单例初始化） |
| atomic | 原子操作，比Mutex快但只能保护单个变量 |
