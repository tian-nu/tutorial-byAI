## 三、分步操作

### 步骤 1：Loading 状态 — 别让用户对着白屏发呆

**我在做什么？**
用户点进页面，浏览器去后端取数据，这中间可能有 200ms 到好几秒的等待。如果页面上什么都不显示，用户会以为"这页面坏了吧？"然后关掉走人。这就好比你去餐厅点菜，服务员说"稍等"然后消失了 5 分钟——你会不会走？

**核心模式**：给每个数据请求配三个状态变量：

```js
const data = ref(null)        // 数据本身
const isLoading = ref(false)  // 是否正在加载
const error = ref(null)       // 错误信息（如果有）
```

**改造首页文章列表**。假设 `src/views/Home.vue` 当前是这样：

```html
<template>
  <div class="home">
    <article v-for="post in posts" :key="post.id">
      <h2>{{ post.title }}</h2>
      <p>{{ post.excerpt }}</p>
    </article>
  </div>
</template>

<script setup>
import { ref, onMounted } from 'vue'

const posts = ref([])

onMounted(async () => {
  const res = await fetch('/api/posts')
  posts.value = await res.json()
})
</script>
```

**加上三个状态后**：

```html
<template>
  <div class="home">
    <!-- 🌀 加载中 -->
    <div v-if="isLoading" class="loading-container">
      <div class="spinner"></div>
      <p>正在加载文章...</p>
    </div>

    <!-- ❌ 加载失败 -->
    <div v-else-if="error" class="error-container">
      <p>😵 {{ error }}</p>
      <button @click="loadPosts">重试</button>
    </div>

    <!-- 📭 列表为空 -->
    <div v-else-if="posts.length === 0" class="empty-container">
      <p>📝 还没有文章，来写第一篇吧~</p>
      <router-link to="/create">去写文章</router-link>
    </div>

    <!-- ✅ 正常列表 -->
    <template v-else>
      <article v-for="post in posts" :key="post.id">
        <h2>{{ post.title }}</h2>
        <p>{{ post.excerpt }}</p>
      </article>
    </template>
  </div>
</template>

<script setup>
import { ref, onMounted } from 'vue'

const posts = ref([])
const isLoading = ref(false)
const error = ref(null)

async function loadPosts() {
  isLoading.value = true
  error.value = null       // 🎯 每次新请求先清空上次的错误
  try {
    const res = await fetch('/api/posts')
    if (!res.ok) throw new Error('服务器开小差了，请稍后重试')
    posts.value = await res.json()
  } catch (e) {
    error.value = e.message || '网络错误，请检查连接'
  } finally {
    isLoading.value = false
  }
}

onMounted(() => {
  loadPosts()
})
</script>
```

**加一个简单的旋转动画 CSS**（追加到 `<style scoped>` 中）：

```css
.spinner {
  width: 40px;
  height: 40px;
  border: 4px solid #e0e0e0;
  border-top: 4px solid #3498db;
  border-radius: 50%;
  animation: spin 0.8s linear infinite;
  margin: 40px auto;
}
@keyframes spin {
  to { transform: rotate(360deg); }
}
.loading-container, .error-container, .empty-container {
  text-align: center;
  padding: 60px 20px;
  color: #666;
}
```

🤔 **想多一点**：为什么用 `v-if` / `v-else-if` / `v-else` 链条而不用多个独立的 `v-if`？因为这四种状态是互斥的——一个页面在同一时刻只能是加载中、出错、空数据、有数据这四种状态之一。用 `v-else-if` 链保证只有一个分支被渲染，逻辑清晰且不会有"加载中和错误提示同时显示"的诡异画面。

❌ **错误示例**：把 `catch` 块里的错误只 `console.log` 而不存到 `error` 变量里，用户永远不知道出了什么事。
✅ **正确示例**：`catch` 中设置 `error.value`，模板中用 `v-else-if="error"` 展示。

**我做得对不对？**
在浏览器 DevTools Network 面板将网速设为 "Slow 3G"，刷新首页，应该看到 loading 旋转动画几秒后变成文章列表。

**不对怎么办？**
- loading 动画一闪而过或根本看不到 → 数据太快返回了，正常现象。可以在 `loadPosts` 开头加 `await new Promise(r => setTimeout(r, 1000))` 模拟慢速测试。
- 动画一直在转停不下来 → 检查 `finally` 块是否存在，以及请求是否卡住了。

### 步骤 2：空状态 — "暂无数据"也是正常状态

**我在做什么？**
列表为空不等于出错——一个刚建好的博客本来就是空的。但如果不加提示，用户看到白茫茫一片会以为页面坏了。空状态设计就像商场里新开的店铺："正在装修，敬请期待"——告诉用户这里功能正常，只是暂时没内容。

模板中已经写了 `v-else-if="posts.length === 0"` 的空状态段落（步骤 1 已包含）。这里重点强调空状态的设计原则：

- **不要只说"暂无数据"**：加上引导行动（如"去写第一篇文章"）
- **不要用红色**：空状态不是错误，用中性色调
- **给一个出路**：放一个跳转按钮或链接

🤔 **想多一点**：一个优秀的空状态至少包含三要素——**图标**（降低认知负担）、**文案**（告知当前状态）、**行动入口**（告诉用户下一步做什么）。三步少一步都会让用户困惑。

### 步骤 3：try/catch — 捕获取数据时的所有异常

**我在做什么？**
`fetch` 不会因为 HTTP 状态码是 4xx/5xx 就抛出异常——`fetch` 只会在"网络层面失败"（DNS 解析失败、连接被拒等）时 reject。所以你需要**手动检查** `res.ok`。

**规范模板**：

```js
async function fetchData(url) {
  try {
    const res = await fetch(url)
    // 🎯 fetch 不自动抛异常，需要手动检查
    if (!res.ok) {
      // 根据状态码给不同提示
      if (res.status === 401) throw new Error('请先登录')
      if (res.status === 403) throw new Error('没有权限执行此操作')
      if (res.status === 404) throw new Error('请求的资源不存在')
      if (res.status >= 500) throw new Error('服务器内部错误，请稍后重试')
      throw new Error(`请求失败 (${res.status})`)
    }
    return await res.json()
  } catch (e) {
    // 🎯 网络断开会走到这里（fetch reject）
    // 🎯 手动 throw 的也会走到这里
    throw e  // 让调用方处理
  }
}
```

❌ **错误示例**：认为 `fetch` 返回 404 时会进 `catch`。
✅ **正确示例**：始终检查 `res.ok`，不 `ok` 时手动 `throw`。

### 步骤 4：网络断开提示

**我在做什么？**
用户可能在地铁上信号断断续续，也可能 Wi-Fi 突然断了。浏览器提供了 `navigator.onLine` 来检测网络状态，还有 `online` / `offline` 事件监听变化。

**创建一个全局网络状态提示组件** `src/components/NetworkStatus.vue`：

```html
<template>
  <Teleport to="body">
    <div v-if="!isOnline" class="offline-bar">
      ⚠️ 网络已断开，部分功能不可用
    </div>
  </Teleport>
</template>

<script setup>
import { ref, onMounted, onUnmounted } from 'vue'

const isOnline = ref(navigator.onLine)

function handleOnline()  { isOnline.value = true }
function handleOffline() { isOnline.value = false }

onMounted(() => {
  window.addEventListener('online', handleOnline)
  window.addEventListener('offline', handleOffline)
})

onUnmounted(() => {
  window.removeEventListener('online', handleOnline)
  window.removeEventListener('offline', handleOffline)
})
</script>

<style scoped>
.offline-bar {
  position: fixed;
  top: 0;
  left: 0;
  right: 0;
  background: #e74c3c;
  color: #fff;
  text-align: center;
  padding: 8px;
  font-size: 14px;
  z-index: 10000;
}
</style>
```

**在 App.vue 中引入**（全局生效）：

```html
<template>
  <NetworkStatus />
  <BlogHeader />
  <router-view />
</template>

<script setup>
import NetworkStatus from '@/components/NetworkStatus.vue'
import BlogHeader from '@/components/BlogHeader.vue'
</script>
```

🤔 **想多一点**：`navigator.onLine` 的判断有一定局限性——它只告诉你"设备连上了某个网络"，不代表"能访问互联网"。比如连上了 Wi-Fi 但 Wi-Fi 本身没网，`onLine` 仍然是 `true`。所以它只是第一道防线，真正的请求仍然需要 `try/catch`。二者是互补关系，不是替代关系。

**我做得对不对？**
- 在浏览器 DevTools Network 面板中切换到 "Offline"，页面顶部应出现红色提示条。
- 切换回 "Online"，红色条消失。

### 步骤 5：全局错误提示组件（优雅降级）

**我在做什么？**
有些错误发生在组件内部，靠单个页面的 `error` 变量覆盖不到全部场景。比如图片加载失败、某个计算属性抛异常。我们需要一个全局的错误边界。

**Vue 3 全局错误处理**，在 `src/main.js` 中注册：

```js
import { createApp } from 'vue'

const app = createApp(App)

// 全局错误处理
app.config.errorHandler = (err, instance, info) => {
  console.error('全局捕获到错误:', err)
  console.error('出错组件:', instance)
  console.error('错误信息:', info)

  // 可以对接错误上报服务（如 Sentry），这里先用 toast 提示用户
  // 避免在生产环境下显出 Vue 内部报错信息
}
```

**创建一个全局 ErrorBoundary 组件** `src/components/ErrorBoundary.vue`：

```html
<template>
  <slot v-if="!hasError" />
  <div v-else class="error-boundary">
    <h2>😵 页面出现了意外错误</h2>
    <p>{{ error?.message || '未知错误' }}</p>
    <button @click="reset">重新加载此区域</button>
  </div>
</template>

<script setup>
import { ref, onErrorCaptured } from 'vue'

const hasError = ref(false)
const error = ref(null)

onErrorCaptured((err) => {
  hasError.value = true
  error.value = err
  return false  // 阻止错误继续向上传播
})

function reset() {
  hasError.value = false
  error.value = null
}
</script>

<style scoped>
.error-boundary {
  text-align: center;
  padding: 60px 20px;
}
</style>
```

**使用方式**——在 `App.vue` 中用 ErrorBoundary 包裹 `<router-view>`：

```html
<template>
  <ErrorBoundary>
    <router-view />
  </ErrorBoundary>
</template>
```

## 六、小结

| 状态 | 何时出现 | 用户感受 | 关键代码 |
|------|----------|----------|----------|
| **Loading** | 数据请求进行中 | "系统在处理，稍等" | `v-if="isLoading"` + spinner CSS |
| **空状态** | 数据返回了但数组为空 | "功能正常，只是没内容" | `v-else-if="list.length === 0"` |
| **错误状态** | 请求失败或异常 | "出错了，但我知道该怎么办" | `v-else-if="error"` + 重试按钮 |
| **正常状态** | 数据返回成功 | "一切正常" | `v-else` |
| **断网提示** | 浏览器检测到网络断开 | "不是网站的锅" | `navigator.onLine` + 事件监听 |

## 七、术语附录

| 术语 | 解释 |
|------|------|
| **Loading** | 加载状态，数据尚未返回时展示的过渡界面。目的是告诉用户"没有卡死，正在处理"。 |
| **骨架屏 (Skeleton Screen)** | Loading 的高级形式——用灰色方块占位，模拟真实内容的布局轮廓，让用户感知到即将出现的内容结构。比 spinner 体验更好但实现更复杂。 |
| **try/catch** | JavaScript 异常捕获语法。`try` 块中抛出异常时自动跳到 `catch`。**注意**：`fetch` 不因 HTTP 4xx/5xx 进入 `catch`，需要手动检查 `res.ok`。 |
| **优雅降级 (Graceful Degradation)** | 当某些功能不可用时，系统仍能以降低体验的方式运行，而非完全崩溃。例如后端挂了，前端展示"请稍后重试"而非白屏。 |

---

## 八、已知坑点与禁止事项

- **禁止忘记 `finally`**：`isLoading = false` 必须在 `finally` 中，否则异常时 loading 动画永不停。
- **禁止在 catch 中只 `console.log`**：用户看不到控制台，必须把错误信息展示在页面上。
- **`navigator.onLine` 不可靠**：它只检测"设备是否连上网络"，不代表"能访问互联网"。不能替代 `try/catch`。
- **空状态不是错误状态**：不要把"列表为空"用红色错误样式展示，会让用户误以为出 bug 了。
