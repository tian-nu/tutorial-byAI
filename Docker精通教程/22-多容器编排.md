## 我在做什么？

前两章你学了 Compose 是什么、compose.yml 怎么写。现在是时候来一次真正的实战了。

我们选一个经典的场景：**WordPress + MySQL**。WordPress 是世界上最流行的 CMS（内容管理系统），它需要 MySQL 来存储数据。用 Compose 启动 WordPress 是多容器编排的"Hello World"——它只涉及两个服务，但涵盖了 `depends_on`、`volumes`、`environment`、`healthcheck` 等几乎所有核心概念。

学完这一章，你能：
- 写一个完整的 WordPress + MySQL 的 compose.yml
- 用 `docker compose up -d` 一键启动
- 解决 MySQL 还没准备好 WordPress 就启动的经典问题
- 用 `docker compose ps` 查看服务状态
- 在浏览器中访问运行中的 WordPress

## 一、场景分析：WordPress 需要什么？

WordPress 是一个 PHP 写的 Web 应用，它需要：

```
浏览器 → WordPress 容器（Apache/PHP，端口 80）
            │
            └──→ MySQL 容器（数据库，端口 3306）
                   │
                   └──→ mysql-data 卷（数据持久化）
```

WordPress 需要 MySQL 以下信息：
- 数据库主机名（在 Compose 网络里就是服务名 `db`）
- 数据库名、用户名、密码

这些信息通过环境变量传给 WordPress 容器。

## 二、完整 compose.yml

```yaml
# compose.yml — WordPress + MySQL
services:
  # ===== WordPress =====
  wordpress:
    image: wordpress:latest
    ports:
      - "8080:80"
    environment:
      WORDPRESS_DB_HOST: db
      WORDPRESS_DB_USER: wordpress
      WORDPRESS_DB_PASSWORD: your_wp_password_here
      WORDPRESS_DB_NAME: wordpress
    volumes:
      - wp-content:/var/www/html/wp-content
    depends_on:
      db:
        condition: service_healthy
    restart: unless-stopped

  # ===== MySQL =====
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: your_root_password_here
      MYSQL_DATABASE: wordpress
      MYSQL_USER: wordpress
      MYSQL_PASSWORD: your_wp_password_here
    volumes:
      - mysql-data:/var/lib/mysql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$$MYSQL_ROOT_PASSWORD"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    restart: unless-stopped

volumes:
  wp-content:
  mysql-data:
```

### 逐段讲解

**WordPress 服务**：
- `image: wordpress:latest`：官方 WordPress 镜像，内置 Apache + PHP
- `ports: "8080:80"`：WordPress 监听 80 端口，映射到宿主机 8080
- `WORDPRESS_DB_HOST: db`：注意！这里写的是 `db`，不是 `localhost`，不是 `127.0.0.1`。在 Compose 的默认网络里，**服务名就是主机名**。`db` 会被 DNS 解析为 MySQL 容器的 IP 地址
- `wp-content:/var/www/html/wp-content`：只持久化上传的文件和插件，WordPress 核心代码不持久化（每次升级重新拉镜像即可）

**MySQL 服务**：
- `healthcheck`：用 `mysqladmin ping` 检测 MySQL 是否真正可用
- `start_period: 30s`：MySQL 8.0 首次启动需要初始化数据目录，可能需要 20-30 秒。这 30 秒内不进行健康检查，给 MySQL 初始化时间
- 注意 `$$MYSQL_ROOT_PASSWORD`：YAML 里 `$$` 是 `$` 的字面量，确保 shell 能正确展开环境变量

**depends_on 的关键作用**：

```
没有 condition: service_healthy 时：
  T0: db 容器启动
  T1: wordpress 容器启动（不等 MySQL 就绪）
  T2: WordPress 尝试连接 MySQL → 失败！
  T3: MySQL 初始化完成（但 WordPress 已经启动失败了）

有 condition: service_healthy 时：
  T0: db 容器启动
  T1-T30: MySQL 在初始化（healthcheck 还没通过）
  T30: mysqladmin ping 成功 → db 状态变为 healthy
  T31: wordpress 容器启动（此时 MySQL 已经可用）
  T32: WordPress 连接 MySQL → 成功！
```

## 三、启动

### 步骤 1：创建项目目录

```bash
mkdir wordpress-demo
cd wordpress-demo
```

### 步骤 2：创建 compose.yml

把上面的 compose.yml 内容复制进去。**记得改密码**（把 `your_wp_password_here` 和 `your_root_password_here` 换成你自己的密码）。

### 步骤 3：启动

```bash
docker compose up -d
```

输出示例：

```
[+] Running 4/4
 ✔ Network wordpress-demo_default    Created
 ✔ Volume "wordpress-demo_mysql-data" Created
 ✔ Volume "wordpress-demo_wp-content" Created
 ✔ Container wordpress-demo-db-1     Started
```

注意：此时可能只看到 `db-1` 启动了，`wordpress-1` 还在等。因为 `depends_on` 的 `condition: service_healthy` 在等 MySQL 就绪。等待 30 秒左右，你会看到：

```
 ✔ Container wordpress-demo-wordpress-1  Started
```

### 步骤 4：查看状态

```bash
docker compose ps
```

输出示例：

```
NAME                          IMAGE              COMMAND                  SERVICE      CREATED          STATUS                    PORTS
wordpress-demo-db-1           mysql:8.0          "docker-entrypoint.s…"   db           2 minutes ago    Up 2 minutes (healthy)    3306/tcp, 33060/tcp
wordpress-demo-wordpress-1    wordpress:latest   "docker-entrypoint.s…"   wordpress    30 seconds ago   Up 30 seconds             0.0.0.0:8080->80/tcp
```

关键信息：
- `db` 服务的 STATUS 是 `Up 2 minutes (healthy)`——`(healthy)` 表示 healthcheck 通过了
- `wordpress` 服务的 STATUS 是 `Up 30 seconds`——它比 db 晚启动约 1.5 分钟，证明 `depends_on` 生效了

## 五、服务依赖问题的深度解析

### 为什么 `depends_on` 不够？

`depends_on` 只控制 Docker 容器的启动顺序，不关心容器内的应用是否就绪。这是设计选择，不是 bug——Docker 不知道你容器里跑的是什么，它只知道"进程启动了"。

MySQL 容器启动 ≠ MySQL 数据库可用。MySQL 的启动过程：

```
1. 容器启动（PID 1: mysqld）
2. 检查数据目录 /var/lib/mysql
3. 如果是首次启动 → 初始化数据目录（创建系统表空间、redo log、undo log）
4. 创建 root 用户和指定数据库
5. 加载 InnoDB 缓冲池
6. 监听 3306 端口
7. 此时 mysqladmin ping 才能成功
```

步骤 3-6 在 MySQL 8.0 首次启动时可能需要 20-30 秒。如果 WordPress 在这之前就尝试连接，会得到 `Connection refused` 或 `Unknown database 'wordpress'`。

### healthcheck 的工作原理

```yaml
healthcheck:
  test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$$MYSQL_ROOT_PASSWORD"]
  interval: 10s     # 每 10 秒检查一次
  timeout: 5s       # 单次检查超时时间
  retries: 5        # 连续失败 5 次才标记为 unhealthy
  start_period: 30s # 启动后前 30 秒不检查（给初始化时间）
```

Docker 每 10 秒执行一次 `mysqladmin ping`。如果成功，状态变为 `healthy`。如果连续失败 5 次，状态变为 `unhealthy`。`start_period` 期间失败不计入 `retries`。

### 没有 healthcheck 时的应对方案

如果你的 Compose 版本不支持 `condition: service_healthy`（旧版 `docker-compose`），可以用以下变通方案：

**方案 A：在应用启动脚本里加等待逻辑**

```yaml
services:
  wordpress:
    image: wordpress:latest
    depends_on:
      - db
    entrypoint: ["/bin/sh", "-c", "sleep 30 && docker-entrypoint.sh apache2-foreground"]
```

这个方案很粗暴——等 30 秒。但 30 秒是拍脑袋的，MySQL 可能 20 秒就初始化好了（你白等了 10 秒），也可能 40 秒才初始化好（你还是失败了）。

**方案 B：用 wait-for-it 脚本**

```yaml
services:
  wordpress:
    image: wordpress:latest
    depends_on:
      - db
    entrypoint: ["/bin/sh", "-c", "./wait-for-it.sh db:3306 -t 60 -- docker-entrypoint.sh apache2-foreground"]
```

`wait-for-it.sh` 是一个经典的等待脚本，它会循环检测 TCP 端口是否可连接。这比方案 A 好——它真正在等 MySQL 端口就绪，而不是盲目 sleep。但 TCP 端口通了 ≠ MySQL 完全初始化好了，仍有可能连上端口但查询失败。

> **最佳方案仍然是 `healthcheck` + `condition: service_healthy`。** 如果你的 Docker 版本够新，不要用变通方案。

## 七、不对怎么办？

### 常见错误 1：WordPress 启动时报 `Error establishing a database connection`

现象：

```
docker compose logs wordpress
# MySQL Connection Error: (2002) Connection refused
# 或
# Error establishing a database connection
```

❌ 原因：WordPress 在 MySQL 还没准备好之前就尝试连接了。

✅ 解决：

```yaml
# 1. 确认 db 服务有 healthcheck
# 2. 确认 wordpress 的 depends_on 用了 condition: service_healthy
# 3. 如果已经配了还是报错，检查 start_period 是否够长
```

如果已经配了 healthcheck 但 WordPress 仍然先启动，检查你的 Compose 版本：

```bash
docker compose version
# 需要 Docker Compose v2.1.0 以上才支持 condition: service_healthy
```

如果版本不够，升级 Docker Desktop 或 Docker Engine。

### 常见错误 2：`WORDPRESS_DB_HOST` 写成了 `localhost`

```yaml
# ❌ 错误
environment:
  WORDPRESS_DB_HOST: localhost
```

`localhost` 在 WordPress 容器里指的是 WordPress 容器自己，不是 MySQL 容器。WordPress 去自己的容器里找 MySQL，当然找不到。

```yaml
# ✅ 正确
environment:
  WORDPRESS_DB_HOST: db      # db 是 MySQL 服务名，Compose 网络里 DNS 会解析
```

### 常见错误 3：密码里有特殊字符，YAML 解析出错

```yaml
# ❌ 错误：密码里有特殊字符，YAML 可能误解析
environment:
  MYSQL_ROOT_PASSWORD: P@ssw0rd!#
```

如果密码包含 `#`、`:`、`{`、`}` 等 YAML 特殊字符，用引号包裹：

```yaml
# ✅ 正确
environment:
  MYSQL_ROOT_PASSWORD: "P@ssw0rd!#"
```

### 常见错误 4：`docker compose up` 后立即访问浏览器，看到空白页

❌ 原因：WordPress 容器启动后，Apache 和 PHP 还需要几秒初始化。

✅ 解决：等 5-10 秒再刷新浏览器。可以看 `docker compose logs wordpress` 确认 Apache 是否已经输出 "ready to handle connections"。

### 常见错误 5：数据卷权限问题

在某些 Linux 系统上，MySQL 容器可能因为数据卷权限问题无法写入：

```
docker compose logs db
# mysqld: Can't create/write to file '/var/lib/mysql/xxx'
```

✅ 解决：

```yaml
services:
  db:
    image: mysql:8.0
    user: "1000:1000"    # 指定运行用户
    volumes:
      - mysql-data:/var/lib/mysql
```

或者先手动创建数据卷并设置权限。Windows 和 macOS 的 Docker Desktop 通常不会遇到这个问题（因为文件系统是虚拟化的）。

## 本篇最可能出错的地方及原因

### 1. MySQL 初始化慢，WordPress 连接失败

**这是 WordPres + MySQL 场景最高频的故障。** 根因是 `depends_on` 只等容器启动，不等 MySQL 可用。MySQL 8.0 首次启动需要初始化 InnoDB 系统表空间、创建用户和数据库，这个过程需要 20-30 秒。WordPress 的 `depends_on` 在容器启动后立即放行，此时 MySQL 还没准备好。

**表现**：`Error establishing a database connection`。

**正确的解决**：healthcheck + `condition: service_healthy`。这是 Compose 处理数据库依赖的"标准答案"。如果你用了这个方案但还是失败，检查：
- Compose 版本是否 ≥ 2.1.0
- `start_period` 是否够长（MySQL 8.0 建议 30s 以上）
- healthcheck 命令里的 `$$` 是否写对了

### 2. `WORDPRESS_DB_HOST` 写成 `localhost`

这是"容器思维"和"传统服务器思维"的冲突。在传统服务器上，MySQL 和 Web 应用可能装在同一台机器上，用 `localhost` 连接。但在 Docker 里，每个服务是独立的容器，`localhost` 指向容器自己。

**排查**：`docker compose exec wordpress sh -c "ping db"`——如果 ping 不通，说明网络配置有问题；如果 ping 得通，说明 `db` 这个主机名是可以解析的，`WORDPRESS_DB_HOST` 应该填 `db`。

### 3. 忘了 `docker compose down -v`，数据卷残留

每次 `docker compose down` 不删数据卷，反复 `up`/`down` 后，`docker volume ls` 会看到一堆 `xxx_mysql-data` 卷。如果某个卷的数据已经损坏（比如 MySQL 初始化失败后残留），下次 `up` 时 MySQL 会复用这个损坏的数据目录，导致启动失败。

**解决**：如果你需要干净的重启，用 `docker compose down -v`。如果你需要保留数据，不要加 `-v`。
