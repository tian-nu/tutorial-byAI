# 86-认证基础

> 💡 你去酒店前台办入住，前台问："请问您是住客吗？" 你掏出身份证。前台核对系统："是的，张先生，您的房间是1208。——这就是认证（你是谁）。前台又看了一眼："您的房间是标准大床房，不能使用行政酒廊。" ——这就是授权（你能做什么）。本章讲的就是如何区分"你是谁"和"你能做什么"。

---

## 本章目标
- 理解认证（Authentication）与授权（Authorization）的区别
- 掌握常见认证方式：Session-Cookie、Token、JWT、OAuth2.0
- 知道什么时候该用哪种认证方式
- 理解"有状态"和"无状态"的含义

---

## 86.1 认证 vs 授权——必须分清

| 概念 | 英文 | 问的问题 | 生活中的类比 |
|------|------|----------|--------------|
| 认证 | Authentication | 你是谁？ | 掏身份证 / 输密码 / 刷脸 |
| 授权 | Authorization | 你能做什么？ | 工牌决定你能进哪些办公室 |

两者顺序固定：**先认证，再授权**。你不知道他是谁，就不可能决定他能做什么。

> 🤔 想多一点：你会发现很多系统把认证和授权混在一起讲。但面试时如果你能把这两个概念清楚分开，并说出它们的不同实现方式，面试官会觉得你基础扎实。

---

## 86.2 为什么需要认证？

HTTP 是无状态协议。每次请求都是独立的：

```
请求1：GET /api/users      → 服务器："谁在请求？不知道。"
请求2：POST /api/orders     → 服务器："还是不知道。"
```

服务器记不住"这个请求和上一个请求是同一个人发的"。认证就是为了让服务器记住你。

---

## 86.3 两种基本思路：有状态 vs 无状态

### 方式一：Session-Cookie（有状态）

```
第1次请求：
  客户端 ──POST /login (用户名+密码)──→ 服务器
  客户端 ←──返回 Set-Cookie: SESSIONID=abc123── 服务器（内存中存了 session 表）

第2次请求：
  客户端 ──GET /api/orders (自动带 Cookie: SESSIONID=abc123)──→ 服务器
  服务器查内存：abc123 → 用户张三
  客户端 ←──返回张三的订单数据── 服务器
```

**关键特征**：服务器要在内存/Redis 里存一份"session 表"，所以叫"有状态"。

| 优点 | 缺点 |
|------|------|
| 简单直观 | 服务器重启后 session 丢失（除非持久化到 Redis） |
| 可主动踢人（删 session 即可） | 分布式部署需要共享 session（Sticky Session 或 Redis） |
| Cookie 可设 HttpOnly 防 XSS | 移动 App 不擅长管理 Cookie |

### 方式二：Token（无状态）

```
第1次请求：
  客户端 ──POST /login (用户名+密码)──→ 服务器
  服务器：密码正确，生成 token = "eyJhbGciOi..."（含用户ID+过期时间+签名）
  客户端 ←──返回 {token: "eyJhbGciOi..."}── 服务器

第2次请求：
  客户端 ──GET /api/orders (Header: Authorization: Bearer eyJhbGci...)──→ 服务器
  服务器：验签通过，解析出用户ID=张三（不需要查数据库/Redis）
  客户端 ←──返回张三的订单数据── 服务器
```

**关键特征**：服务器不存任何东西，token 本身就包含所有信息。所以叫"无状态"。

| 优点 | 缺点 |
|------|------|
| 服务器无存储，天然支持分布式 | 无法主动踢人（发了 token 就收不回来） |
| 移动端友好 | token 比 session ID 大，每次请求多占带宽 |
| 跨域友好（不依赖 Cookie） | token 泄露后无法单独失效 |

## 86.4 常见认证方案一览

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| **Basic Auth** | 每次请求带 `Authorization: Basic base64(user:pass)` | 内部工具、极简 API |
| **Session-Cookie** | 登录后服务端存 session，客户端存 cookie | 传统 MVC Web 应用 |
| **JWT Token** | 登录后签发含用户信息的签名 token | REST API、微服务、移动端 |
| **OAuth2.0** | 委托第三方（微信/Google）认证 | 第三方登录、开放平台 |
| **API Key** | 预先分配固定密钥 | 机器对机器调用、SaaS API |
| **SSO（单点登录）** | 一处登录，多处自动认证 | 企业内部系统群 |

---

## 86.5 Basic Auth——最简单也最不安全的方案

```java
// 客户端请求
GET /api/admin HTTP/1.1
Authorization: Basic emhhbmdzYW46MTIzNDU2
```

`emhhbmdzYW46MTIzNDU2` 就是 `base64("zhangsan:123456")` 的结果。

**致命问题**：Base64 是编码，不是加密！任何人截获请求就能反解出密码。所以 Basic Auth **必须在 HTTPS 下使用**，否则等于明文传输密码。

## 86.6 你该用哪种？

```
┌──────────────────────────────────────┐
│ 你的项目类型？                        │
├──────────────────────────────────────┤
│ 前后端不分离（JSP/Thymeleaf）          │
│ → Session-Cookie                     │
├──────────────────────────────────────┤
│ 前后端分离（Vue/React + REST API）     │
│ → JWT Token                          │
├──────────────────────────────────────┤
│ 需要"用微信/Google 登录"              │
│ → OAuth2.0（其中 token 部分用 JWT）    │
├──────────────────────────────────────┤
│ 企业内部系统（OA/ERP）                │
│ → SSO + JWT 或 Session               │
├──────────────────────────────────────┤
│ 对外提供 API 给第三方调用              │
│ → API Key + HMAC 签名                 │
└──────────────────────────────────────┘
```

## 86.7 小结

| 知识点 | 核心内容 |
|--------|----------|
| 认证 vs 授权 | 认证 = 你是谁；授权 = 你能做什么 |
| 有状态 | Session-Cookie：服务器存 session |
| 无状态 | JWT Token：服务器不存，token 自包含 |
| Basic Auth | base64 编码用户名密码，必须 HTTPS |
| JWT | 第88章详解 |
| OAuth2.0 | 第89章详解 |
