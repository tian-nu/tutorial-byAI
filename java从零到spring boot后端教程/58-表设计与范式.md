# 第58章 · 表设计与范式

> "如果你把衣服、碗筷、书籍全堆在一个大箱子里，每次找东西都得翻个底朝天。数据库的表也一样——把什么数据都塞进一张表，修改一个用户的名字要同时改几十条订单记录。范式就是一套'收纳法则'，告诉你怎么把东西分门别类放好，既不会找不到，也不会改一处而动全身。"

---

## 58.1 什么是范式

**范式（Normal Form）** 是数据库表设计的规则体系，从1NF到5NF（还有BCNF），越来越严格。实际开发中，达到**第三范式（3NF）**就足够应对绝大多数场景。

范式的目标只有一个：**消除数据冗余，保证数据一致性**。

它不是法律，但违反它之前你必须清楚自己在付出什么代价。

---

## 58.2 第一范式（1NF）：原子性

> **每一列都是不可再分的原子值。**

### ❌ 违反1NF的设计

```sql
-- 错误：phone列存了多个电话号码
CREATE TABLE users (
    id   INT,
    name VARCHAR(20),
    phone VARCHAR(200)  -- 存 "13800138000,13900139000"
);

-- 错误：address列是一个复合值
CREATE TABLE users (
    id   INT,
    name VARCHAR(20),
    address VARCHAR(200)  -- 存 "北京市-朝阳区-望京SOHO"
);
```

### ✅ 符合1NF的设计

```sql
-- 方案一：拆列（如果电话数固定）
CREATE TABLE users (
    id      INT,
    name    VARCHAR(20),
    phone1  VARCHAR(20),
    phone2  VARCHAR(20)
);

-- 方案二：拆表（推荐，电话数量不固定）
CREATE TABLE users (
    id   INT,
    name VARCHAR(20)
);

CREATE TABLE user_phones (
    id      INT,
    user_id INT,
    phone   VARCHAR(20)
);
```

> 💡 方案二更好，因为用户可能有0~N个电话。拆列方案（phone1, phone2）无法应对"用户有3个电话"的情况。

---

## 58.3 第二范式（2NF）：完全依赖

> **非主键列必须完全依赖于主键（而不是部分依赖）。**

这条主要针对**复合主键**的表——如果主键只有一列，2NF自动满足。

### ❌ 违反2NF的设计

```sql
-- 复合主键：(order_id, product_id)
CREATE TABLE order_details (
    order_id     INT,
    product_id   INT,
    product_name VARCHAR(100),  -- 只依赖 product_id，不依赖 order_id
    quantity     INT,            -- 依赖 (order_id, product_id) 组合
    PRIMARY KEY (order_id, product_id)
);
```

分析：
- `quantity`（购买数量） → 依赖于 订单+商品 的组合 ✅
- `product_name`（商品名） → 只依赖于 product_id，和 order_id 无关 ❌

**问题**：同一个商品出现在100个订单中，product_name就要重复存100遍。如果商品改名，要改100处。

### ✅ 符合2NF的设计

```sql
CREATE TABLE products (
    id   INT PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE order_details (
    order_id   INT,
    product_id INT,
    quantity   INT,
    PRIMARY KEY (order_id, product_id)
);
```

商品名移到独立的 `products` 表，完全符合2NF。

---

## 58.4 第三范式（3NF）：无传递依赖

> **非主键列不能依赖于其他非主键列。**

### ❌ 违反3NF的设计

```sql
CREATE TABLE orders (
    id            INT PRIMARY KEY,
    customer_id   INT,
    customer_name VARCHAR(50),   -- 依赖于 customer_id，而非直接依赖 id
    customer_city VARCHAR(50),   -- 依赖于 customer_id，而非直接依赖 id
    amount        DECIMAL(10,2)  -- 直接依赖于 id ✅
);
```

依赖链：`id → customer_id → customer_name`（传递依赖）。

**问题**：如果客户改名字，需要更新所有该客户的订单记录。漏改一条，出现两个名字的数据不一致。

### ✅ 符合3NF的设计

```sql
CREATE TABLE customers (
    id   INT PRIMARY KEY,
    name VARCHAR(50),
    city VARCHAR(50)
);

CREATE TABLE orders (
    id          INT PRIMARY KEY,
    customer_id INT,
    amount      DECIMAL(10,2),
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);
```

客户信息独立成表，订单通过 `customer_id` 引用。

## 58.5 三范式的记忆口诀

| 范式 | 口诀 | 反例 |
|------|------|------|
| 1NF | **列不可拆** | 一列存多个电话号码 |
| 2NF | **完全依赖主键** | 复合主键时，部分列只依赖其中一部分 |
| 3NF | **不依赖别人的主键** | 订单表里存客户名字（名字依赖客户ID） |

> 📌 业内还有一种简单的判断法："一个字段改了，需要改多处吗？"如果需要，大概率违反了范式。

## 58.6 反范式化：故意违反范式

范式不是绝对真理。有时**故意违反范式**能换来巨大的性能提升。

### 经典案例：订单表存冗余的客户名

```sql
-- 严格3NF的订单表
CREATE TABLE orders (
    id          INT PRIMARY KEY,
    customer_id INT,
    amount      DECIMAL(10,2)
);

-- 反范式化的订单表
CREATE TABLE orders (
    id            INT PRIMARY KEY,
    customer_id   INT,
    customer_name VARCHAR(50),  -- 冗余！违反3NF
    amount        DECIMAL(10,2)
);
```

**为什么故意冗余？** 因为"查订单列表时显示客户名"是**每秒钟都在发生的查询**。如果用3NF设计，每次都要JOIN customers表：

```sql
-- 范式化：每次查订单列表都要JOIN
SELECT o.*, c.name
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.id;

-- 反范式化：直接查，快得多
SELECT * FROM orders;
```

**原则**：
- **规范化**：以写操作为主的表（改的比查的多）
- **反范式化**：以读操作为主的表（查的比改的多），或高频热点查询的关联字段

> 🤔 **想多一点**：这不矛盾吗？范式教我们这样设计，又要我们违反它？
>
> 不矛盾。范式是**理论最优**，反范式化是**工程最优**。就像一个物理公式假设"没有空气阻力"，但造飞机时必须考虑空气阻力。先用范式设计（保证数据正确性），然后在性能瓶颈处做反范式化（用冗余换速度），这是成熟工程师的标准做法。

## 58.7 ER图与实体关系

在设计表之前，先画ER图（Entity-Relationship Diagram，实体关系图）能避免后期大改。

### 三种实体关系

**1. 一对一（1:1）**

一个用户对应一个身份证号。

```
┌──────────┐     1    1    ┌──────────────┐
│  users   │───────────────│ user_profiles │
└──────────┘               └──────────────┘
```

实现：在任意一方的表里加外键。

```sql
CREATE TABLE user_profiles (
    id      INT PRIMARY KEY,
    user_id INT UNIQUE,   -- UNIQUE保证一对一
    avatar  VARCHAR(200),
    bio     TEXT,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

**2. 一对多（1:N）**

一个用户有多个订单。这是最常见的。

```
┌──────────┐     1    N    ┌──────────┐
│  users   │───────────────│  orders  │
└──────────┘               └──────────┘
```

实现：在"多"的那方加外键。

```sql
CREATE TABLE orders (
    id      INT PRIMARY KEY,
    user_id INT NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

**3. 多对多（M:N）**

一个学生可以选多门课，一门课可以有多名学生。

```
┌──────────┐     M    N    ┌──────────┐
│ students │───────────────│ courses  │
└──────────┘               └──────────┘
```

实现：必须建**中间表**。

```sql
CREATE TABLE student_courses (
    student_id INT,
    course_id  INT,
    score      INT,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students(id),
    FOREIGN KEY (course_id)  REFERENCES courses(id)
);
```

## 58.8 实际表设计练习：电商系统

设计一个简易电商系统的核心表结构（满足3NF）：

```sql
-- 用户表
CREATE TABLE users (
    id            INT AUTO_INCREMENT PRIMARY KEY,
    username      VARCHAR(50) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    email         VARCHAR(100) NOT NULL,
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 商品分类表
CREATE TABLE categories (
    id   INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50) NOT NULL UNIQUE,
    parent_id INT DEFAULT NULL,
    FOREIGN KEY (parent_id) REFERENCES categories(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 商品表
CREATE TABLE products (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    name        VARCHAR(200) NOT NULL,
    description TEXT,
    price       DECIMAL(10, 2) NOT NULL,
    stock       INT NOT NULL DEFAULT 0,
    category_id INT,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (category_id) REFERENCES categories(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 订单表
CREATE TABLE orders (
    id          INT AUTO_INCREMENT PRIMARY KEY,
    user_id     INT NOT NULL,
    total_amount DECIMAL(10, 2) NOT NULL,
    status      ENUM('待支付','已支付','已发货','已完成','已取消') NOT NULL DEFAULT '待支付',
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 订单明细表（中间表，处理订单和商品的多对多关系）
CREATE TABLE order_items (
    id         INT AUTO_INCREMENT PRIMARY KEY,
    order_id   INT NOT NULL,
    product_id INT NOT NULL,
    quantity   INT NOT NULL,
    unit_price DECIMAL(10, 2) NOT NULL COMMENT '下单时的单价，防止后续商品改价影响历史订单',
    FOREIGN KEY (order_id)   REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

> 💡 `order_items.unit_price` 为什么要冗余存单价？因为商品价格会变——今天降价你总不能把三个月前的订单金额也改了吧。这是**必要的反范式化**。

## 58.9 常见设计错误

| 错误 | 表现 | 后果 |
|------|------|------|
| 用逗号分隔存多值 | `tags VARCHAR(200)` 存 `"Java,MySQL,Spring"` | 查询"含Java标签的所有文章"需要用 `LIKE '%Java%'`，无法用索引，慢到怀疑人生 |
| 用JSON存所有字段 | 整张表只有id和data两列 | 丧失了SQL的查询、索引、约束能力，MySQL退化成了JSON文件存储 |
| 无限扩列 | `field1, field2, field3 ... field20` | 列名无意义，加一个属性就要ALTER TABLE，应该用行存储（另建表） |
| 把树形结构存成 `parent_id1, parent_id2` | 多级分类用多列存 | 深度不确定（也许有10级分类），应该用 `parent_id` + 自关联 |

## 本章小结

| 概念 | 要点 |
|------|------|
| 1NF | 列不可再分，每个字段是原子值 |
| 2NF | 非主键列完全依赖主键（复合主键时特别注意） |
| 3NF | 非主键列不依赖其他非主键列（无传递依赖） |
| 反范式化 | 故意冗余数据，用空间换时间，应对高频查询 |
| 1:1 | 任意一方加外键+UNIQUE |
| 1:N | "多"的一方加外键 |
| M:N | 建中间表，两列外键组成复合主键 |


1. 违反2NF和3NF。`employee_name`只依赖employee_id（部分依赖，违反2NF），`project_name`只依赖project_id。修复：拆成三张表——employees(employee_id, employee_name)、projects(project_id, project_name)、employee_projects(employee_id, project_id, hours)。

2. 关系：Category 1:N Article，Article M:N Tag。需要建 articles、categories、tags、article_tags（中间表）：
```sql
CREATE TABLE categories (id INT PRIMARY KEY, name VARCHAR(50));
CREATE TABLE articles (id INT PRIMARY KEY, title VARCHAR(200), category_id INT, FOREIGN KEY (category_id) REFERENCES categories(id));
CREATE TABLE tags (id INT PRIMARY KEY, name VARCHAR(50));
CREATE TABLE article_tags (article_id INT, tag_id INT, PRIMARY KEY (article_id, tag_id), FOREIGN KEY (article_id) REFERENCES articles(id), FOREIGN KEY (tag_id) REFERENCES tags(id));
```

3. 当查询频率远高于写入频率时。例如：订单列表页面每秒被访问数百次，每次都要JOIN客户表查客户名。在订单表中冗余存储customer_name，查询时不需要JOIN，IO减少、速度提升。代价是客户改名时需要批量更新所有相关订单——但这个操作频率极低，可以接受。
