# 第16章 · Go程序的基本结构

> "每一个Go程序都像一栋房子。package是房子的地址，import是建房子用的材料，func是房间。搞清楚了结构，你才知道每块砖该往哪放。"

---

## 16.1 package：Go程序的"地址"

### 什么是包

Go语言的代码是**按包（package）组织的**。每个 `.go` 文件的第一行必须声明自己属于哪个包。

你可以把包想象成一个**文件夹**——把功能相近的代码放在同一个包里。就像你不会把餐具放卧室、衣服放厨房一样，你也不会把数据库操作的代码和界面渲染的代码放在同一个包里。

### 两种包：可执行 vs 库

Go里只有两种包：

**1. main包——可执行程序**

```go
package main

func main() {
    fmt.Println("我是一个可以独立运行的程序")
}
```

声明为 `package main` 的包，编译后会生成一个**可以直接运行的可执行文件**（`.exe` 或二进制文件）。`main` 包必须包含一个 `main()` 函数作为程序入口。

**2. 库包——给别人用的代码**

```go
package mathutil

func Add(a, b int) int {
    return a + b
}
```

声明为其他名字的包（比如 `mathutil`、`database`、`handler`），编译后会生成一个**库文件**，不能被直接运行，只能被别的程序引用。就像你写的工具函数，不能自己独立运行，但可以被 `main` 包导入使用。

### 同目录必须同包

这是Go的第一个"奇怪"规则：**同一个目录下的所有 `.go` 文件，必须声明为同一个包名。**

```
myproject/
├── main.go              → package main
├── helper.go            → package main  （✅ 可以，同目录都叫main）
└── database/
    ├── mysql.go         → package database
    └── redis.go         → package database  （✅ 可以，同目录都叫database）
```

如果你尝试在同一个目录下创建不同包名的文件：

```
myproject/
├── main.go              → package main
└── helper.go            → package helpers  （❌ 编译错误！）
```

Go编译器会直接拒绝，告诉你"在同一个目录下找到了两个不同的包名"。

### 包的命名规范

- 包名应该**简短、小写、单数**。比如 `http`、`json`、`sql`，不能叫 `HTTP`、`JSONs`。
- 包名通常是它所放目录的名字。放在 `database/` 目录下的包就叫 `database`。
- 不要用 `util`、`common`、`misc` 这种"什么都能往里扔"的名字——这等于没有组织。

> 🤔 **想多一点**：Go强制同目录同包这个规则，初看起来有点死板。但它有一个巨大的好处：**消除选择困难**。你不需要纠结"这个文件应该放在哪个包"，目录结构直接决定了包结构，一目了然。这也是Go哲学"少即是多"的体现——不是不能支持更灵活的包组织方式，而是不想让程序员在这上面花时间。

## 16.2 import：Go程序的"采购清单"

`import` 就是你要告诉编译器："我要用到下面这些包里提供的东西，帮我找一下。"

### 单行导入

最简单的情况，只导一个包：

```go
import "fmt"
```

### 分组导入（最常用）

当你需要多个包时，用圆括号分组：

```go
import (
    "fmt"
    "os"
    "strings"
)
```

这不是语法糖，而是**Go推荐的标准写法**。即使是只导入一个包，很多团队也坚持用分组写法（为了统一风格）。

### 包的路径规则

标准库直接用包名：

```go
import "fmt"       // Go标准库里的
import "net/http"  // 标准库，net目录下的http包
```

第三方库用完整的模块路径：

```go
import "github.com/gin-gonic/gin"   // Gin框架
import "github.com/go-redis/redis"  // Redis客户端
```

你自己的项目里的包，用模块路径 + 相对目录：

```go
import "myproject/database"   // 你项目 database 目录下的包
import "myproject/handler"    // 你项目 handler 目录下的包
```

这里的 `myproject` 就是在 `go.mod` 里声明的模块名。

### 别名导入

有时候两个包的名字冲突了，或者包名太长/不直观，你可以给它起个**别名**：

```go
import (
    nf "net/http/pprof"
    "net/http"
)
```

比如 `net/http/pprof` 默认包名是 `pprof`，但你可以给它起个别名 `nf`，后面用 `nf.xxx` 来调用。

更常见的用法是给长名字起短别名：

```go
import (
    "github.com/gin-gonic/gin"
    gredis "github.com/go-redis/redis/v8"
)
```

### 点导入（不推荐）

```go
import . "fmt"
```

这个 `.` 的意思是"把这个包里所有公开的函数、变量直接导入到当前文件的名字空间"。之后你可以直接写 `Println("hello")` 而不需要 `fmt.Println("hello")`。

**绝大多数情况下不要这样做。** 因为它会让代码可读性崩塌——你看到 `Println`，完全不知道它来自哪个包。

### 匿名导入（最特别的写法）

```go
import _ "github.com/go-sql-driver/mysql"
```

前面加一个 `_`（下划线），表示"我不用这个包里的任何函数或变量，但我需要这个包的 `init()` 函数执行"。

什么时候用？最典型的场景是**数据库驱动注册**：

```go
import (
    "database/sql"
    _ "github.com/go-sql-driver/mysql"  // 注册MySQL驱动，但代码里不直接调用它
)
```

MySQL驱动的 `init()` 函数会在程序启动时自动执行，把自己注册到 `database/sql` 包中。你的代码只需要用 `database/sql` 的通用接口，不需要直接碰驱动包的代码。

> 🤔 **想多一点**：如果你导入了某个包但是在代码里完全没用到它，Go会直接报编译错误——"imported and not used"。这一点让很多从其他语言转过来的人很不适应，但它是个好设计——避免代码里积累大量无用的import。当然，对于匿名导入（`_`前缀），Go不会报错，因为你在显式告诉Go"我知道我没用这个包，但我是故意的"。

## 16.3 gofmt：让全世界的Go代码长得一模一样

如果你读过不同程序员写的Python代码，你会感受到各种风格——有人用4空格缩进，有人用2空格；有人喜欢在列表后面加逗号，有人不喜欢。这是Python的灵活性，也是它的代价——每个项目都可能有一套不同的风格。

Go用一种极端的方式解决了这个问题：**所有人都用同一种格式。**

### gofmt是什么

`gofmt`（Go Format）是Go官方的代码格式化工具。它**自动**调整你的代码：
- 缩进（统一用Tab，不是空格）
- 花括号位置
- 空格和换行
- import的顺序（标准库 → 第三方库 → 你的库）

### 怎么用

格式化单个文件：

```bash
gofmt -w main.go
```

`-w` 的意思是 write——直接把格式化结果写回文件。如果不加 `-w`，只会在屏幕上显示格式化后的结果，不修改文件。

格式化整个项目：

```bash
gofmt -w ./
```

现在更推荐用 `go fmt`（Go 1.0+自带）：

```bash
go fmt ./...
```

`./...` 表示当前目录及其所有子目录下的所有Go文件。

### 为什么强制统一格式

想象这个场景：你接手一个离职同事的项目，打开代码一看——缩进一会儿2空格一会儿4空格，花括号一会儿换行一会儿不换行，空行有的地方有有的地方没有。

你不得不花时间去"解码"代码风格，才能看懂逻辑。

Go的设计者说：**不要让程序员在选择代码风格上浪费哪怕一秒钟。** 所有人都用 `gofmt`，所有人的代码长得一模一样。你不需要看风格，直接看逻辑就行。

### gofmt不仅仅是格式化

`gofmt` 还有一个隐藏功能：它可以作为代码重构的基础。因为 `gofmt` 理解Go的语法结构，它可以把整个Go语法树导出给你看。

```bash
gofmt -w -r 'x.OldName -> x.NewName' ./
```

这条命令会把整个项目中所有的 `x.OldName` 改成 `x.NewName`——这是一种简单的自动重构。

### IDE集成

VS Code、GoLand等主流编辑器都支持保存时自动运行 `gofmt`。你只需要正常写代码，按 `Ctrl+S` 保存时格式就自动好了。**你不应该手动调整格式**——那是机器该做的事。

## 16.4 注释：写给未来的自己

注释是写给**人**看的，编译器会完全忽略它们。但好的注释能让你三个月后还能看懂自己写的代码。

### 单行注释

```go
// 这是单行注释
```

以 `//` 开头，后面直到行尾都是注释。这是Go里最常用的注释方式。

### 多行注释

```go
/*
这是一段多行注释。
你可以在这里写很多行文字，
解释一些比较复杂的事情。

比如：
- 为什么这里要用锁？
- 这段逻辑的边界条件是什么？
- 如果str是空字符串会怎样？
*/
func process(input string) error {
    return nil
}
```

以 `/*` 开头，`*/` 结束，中间可以跨多行。

### 文档注释（go doc）

这是Go最有特色的注释机制。如果你在一个**公开的**函数、类型、包的前面写注释，Go可以自动把这些注释提取成文档。

```go
// Add 计算两个整数的和。
// 如果计算结果超出int范围，行为是未定义的。
func Add(a, b int) int {
    return a + b
}
```

注意两点：
1. 注释必须以被注释的对象名开头（这里是 `// Add`）
2. 注释后面不要有空行，直接紧挨着函数声明

然后别人用 `go doc` 就能看到你的文档：

```bash
$ go doc Add
package mathutil

func Add(a, b int) int
    Add 计算两个整数的和。
    如果计算结果超出int范围，行为是未定义的。
```

对于包级别的注释，建议写在包声明前：

```go
// Package mathutil 提供常用的数学运算工具函数。
// 包含加、减、乘、除等基本运算。
package mathutil
```

### 注释应该写什么、不应该写什么

**应该写的内容：**
- 函数做什么（不是怎么写）
- 边界条件和特殊行为（"如果参数为nil会返回什么"）
- 为什么这样设计（"这里用map而不是切片是因为要O(1)查找"）
- 非显而易见的逻辑（"这个循环不用range是因为需要在遍历中删除元素"）

**不应该写的内容：**
- 代码已经说清楚的事（"// 把a加b赋值给c" —— 这个不用写，代码本来就是这个意思）
- 过时的注释（修改了代码一定要同步修改注释，否则注释比没注释更害人）
- 废话（"// 这是一个函数"）

### 好注释 vs 坏注释

```go
// ❌ 坏注释：重复代码
// 把a和b相加
result := a + b

// ✅ 好注释：解释为什么
// 用加法而不用位运算，因为这里需要考虑溢出检测
result := a + b
```

中间那个例子，好注释解释了"为什么这样做"而不是"做了什么"，这才是注释的价值。

## 本章小结

| 知识点 | 核心要点 |
|--------|----------|
| package | Go代码按包组织；main包生成可执行程序，其他包生成库；同目录必须同包 |
| main包 | 特殊包名，必须包含 `main()` 函数，编译后可直接运行 |
| import | 导入需要的包；分组写法是标准；别名/点导入/匿名导入各有用途 |
| 匿名导入 `_` | 只执行包的init()函数，不直接用包的代码（典型场景：数据库驱动注册） |
| gofmt | 官方格式化工具，强制统一风格；`go fmt ./...` 格式化整个项目 |
| 注释 | 单行 `//`、多行 `/* */`；文档注释可被 `go doc` 提取 |
| 注释原则 | 解释"为什么"而不是"做了什么"；过时注释比没注释更害人 |
