# 第74章 · 错误码与统一异常处理

> "不会有人把SQL报错原文 `ERROR 1062: Duplicate entry 'zhangsan' for key 'username'` 直接返回给客户端吧？但很多新手就是这么干的。本章教你设计一套专业级的错误码体系，把丑陋的系统错误翻译成调用方能理解的业务级错误信息，再用Recovery中间件兜住所有未捕获的panic。"

---

## 74.1 业务错误码设计

### 为什么需要错误码

前端收到错误后需要做两件事：
1. **展示给用户**：弹出友好的提示（"用户名已存在"而不是"duplicate entry"）
2. **程序判断**：根据错误码决定是否重试、是否跳转登录页、是否需要验证码

如果后端只返回字符串 "用户名已存在"，前端需要 `if (errMsg.includes("用户名已存在"))` 这种脆弱的方式来判断错误类型。一旦后端改了一个字的文案，前端的判断就失效了。

### 错误码设计原则

**模块号 + 错误序号**，通常5位数字：

```
1 0 0 0 1
│ └─┬─┘ └─┬─┘
│   │     └── 具体错误序号（两位）
│   └──────── 模块编号（两位）
└──────────── 服务编号（一位）
```

| 分段 | 含义 | 示例 |
|------|------|------|
| 第1位 | 服务编号（微服务架构下区分哪个服务） | 1=用户服务 |
| 第2-3位 | 模块编号 | 00=通用, 01=用户, 02=订单, 03=商品 |
| 第4-5位 | 具体错误 | 01, 02, 03... |

```go
package ecode

const (
    Success          = 0

    ErrInvalidParams = 10001
    ErrUnauthorized  = 10002
    ErrForbidden     = 10003
    ErrNotFound      = 10004
    ErrInternal      = 10005

    ErrUserNotFound      = 10101
    ErrUserAlreadyExists = 10102
    ErrPasswordWrong     = 10103
    ErrTokenExpired      = 10104

    ErrOrderNotFound     = 10201
    ErrOrderStatusInvalid = 10202
    ErrOrderPaid         = 10203

    ErrProductNotFound   = 10301
    ErrProductOutOfStock = 10302
)
```

### 错误码与错误信息映射

```go
var errMsgMap = map[int]string{
    ErrInvalidParams:     "参数错误",
    ErrUnauthorized:      "未登录或登录已过期",
    ErrForbidden:         "没有操作权限",
    ErrNotFound:          "资源不存在",
    ErrInternal:          "服务器内部错误",
    ErrUserNotFound:      "用户不存在",
    ErrUserAlreadyExists: "用户名已存在",
    ErrPasswordWrong:     "密码错误",
    ErrTokenExpired:      "登录已过期，请重新登录",
    ErrOrderNotFound:     "订单不存在",
    ErrProductOutOfStock: "商品库存不足",
}

func GetMsg(code int) string {
    if msg, ok := errMsgMap[code]; ok {
        return msg
    }
    return "未知错误"
}
```

## 74.2 自定义错误类型

单纯返回 `error` 无法区分"这是业务错误还是系统错误"、"错误码是多少"。我们需要自定义错误类型。

### 定义BusinessError

```go
type BusinessError struct {
    Code int
    Msg  string
    Err  error
}

func (e *BusinessError) Error() string {
    return fmt.Sprintf("业务错误 [%d]: %s", e.Code, e.Msg)
}

func NewBusinessError(code int, msg string) *BusinessError {
    return &BusinessError{Code: code, Msg: msg}
}

func NewBusinessErrorWithErr(code int, msg string, err error) *BusinessError {
    return &BusinessError{Code: code, Msg: msg, Err: err}
}
```

### Service层使用

```go
func (s *UserService) Register(req *RegisterRequest) error {
    exists, err := s.repo.ExistsByUsername(req.Username)
    if err != nil {
        return NewBusinessErrorWithErr(ecode.ErrInternal, "查询用户失败", err)
    }
    if exists {
        return NewBusinessError(ecode.ErrUserAlreadyExists, "用户名已存在")
    }

    err = s.repo.Create(&model.User{
        Username: req.Username,
        Password: hashPassword(req.Password),
    })
    if err != nil {
        return NewBusinessErrorWithErr(ecode.ErrInternal, "创建用户失败", err)
    }

    return nil
}
```

### Handler层统一处理

```go
func (h *UserHandler) Register(c *gin.Context) {
    var req RegisterRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        response.Error(c, ecode.ErrInvalidParams, "参数错误")
        return
    }

    if err := h.svc.Register(&req); err != nil {
        var bizErr *BusinessError
        if errors.As(err, &bizErr) {
            response.Error(c, bizErr.Code, bizErr.Msg)

            if bizErr.Err != nil {
                // logger为*zap.Logger，需 import "go.uber.org/zap"
                logger.Errorf("业务处理异常: %v", bizErr.Err)
            }
            return
        }

        logger.Errorf("未预期错误: %v", err)
        response.Error(c, ecode.ErrInternal, "服务器内部错误")
        return
    }

    response.Success(c, nil)
}
```

**核心逻辑**：用 `errors.As` 判断错误是不是 `*BusinessError` 类型。是的话把业务错误码和文案返回给客户端；不是的话记录详细日志，返回通用500错误。这样既保护了内部错误详情不泄露，又给客户端提供了准确的错误信息。

## 74.3 全局Recovery中间件

Gin默认的Recovery中间件返回的是 `500 Internal Server Error`，但没有我们的统一响应格式。而且它不会记录业务级日志。

### 自定义Recovery

```go
func CustomRecovery(logger *zap.Logger) gin.HandlerFunc {
    return func(c *gin.Context) {
        defer func() {
            if r := recover(); r != nil {
                traceID, _ := c.Get("traceID")

                logger.Error("服务panic",
                    zap.Any("traceID", traceID),
                    zap.Any("panic", r),
                    zap.String("path", c.Request.URL.Path),
                    zap.String("method", c.Request.Method),
                )

                debugStack := debug.Stack() // debug.Stack() 需要 import "runtime/debug"
                logger.Error("panic堆栈", zap.String("stack", string(debugStack)))

                c.JSON(http.StatusInternalServerError, Response{
                    Code: ecode.ErrInternal,
                    Msg:  "服务器内部错误，我们正在处理",
                    Data: nil,
                })
                c.Abort()
            }
        }()
        c.Next()
    }
}
```

`debug.Stack()` 会把完整的堆栈信息记录下来（Go标准库 `runtime/debug` 包）。客户端收到友好的500提示，你的日志中有完整的事故现场。

### 使用

```go
r := gin.New()
r.Use(TraceMiddleware)
r.Use(CustomRecovery(logger))
```

**不要用 `gin.Default()`**——它自带的Recovery用的是默认格式。用 `gin.New()` 手动添加中间件。

## 74.4 统一错误响应

把前面所有的错误处理逻辑整合到一个完整的响应包中。

`pkg/response/response.go`：

```go
package response

import (
    "net/http"

    "github.com/gin-gonic/gin"
    "myweb/pkg/ecode"
)

type Response struct {
    Code int         `json:"code"`
    Msg  string      `json:"msg"`
    Data interface{} `json:"data"`
}

func Success(c *gin.Context, data interface{}) {
    c.JSON(http.StatusOK, Response{
        Code: 0,
        Msg:  "success",
        Data: data,
    })
}

func Error(c *gin.Context, code int, msg string) {
    c.JSON(http.StatusOK, Response{
        Code: code,
        Msg:  msg,
        Data: nil,
    })
}

func HandleServiceError(c *gin.Context, err error, logger *zap.Logger) {
    var bizErr *BusinessError
    if errors.As(err, &bizErr) {
        if bizErr.Err != nil {
            logger.Error("业务异常",
                zap.Int("code", bizErr.Code),
                zap.Error(bizErr.Err),
            )
        }

        httpStatus := http.StatusOK
        switch bizErr.Code {
        case ecode.ErrUnauthorized:
            httpStatus = http.StatusUnauthorized
        case ecode.ErrForbidden:
            httpStatus = http.StatusForbidden
        case ecode.ErrNotFound:
            httpStatus = http.StatusNotFound
        case ecode.ErrInvalidParams:
            httpStatus = http.StatusBadRequest
        }

        c.JSON(httpStatus, Response{
            Code: bizErr.Code,
            Msg:  bizErr.Msg,
            Data: nil,
        })
        return
    }

    logger.Error("未预期错误", zap.Error(err))

    c.JSON(http.StatusInternalServerError, Response{
        Code: ecode.ErrInternal,
        Msg:  "服务器内部错误",
        Data: nil,
    })
}
```

### Handler的使用变得极其简洁

```go
func (h *UserHandler) Register(c *gin.Context) {
    var req RegisterRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        response.Error(c, ecode.ErrInvalidParams, "参数错误")
        return
    }

    result, err := h.svc.Register(&req)
    if err != nil {
        response.HandleServiceError(c, err, h.logger)
        return
    }

    response.Success(c, result)
}
```

所有Handler只需做三件事：
1. 参数校验失败 → `response.Error`
2. Service调用失败 → `response.HandleServiceError`（自动区分业务错误和系统错误）
3. 调用成功 → `response.Success`

## 🤔 想多一点：第三方服务的错误

当你调用微信支付、阿里云短信、SendGrid邮件等外部服务时，它们可能返回各种格式的错误。你需要在Service层把它们也转成你的BusinessError：

```go
func (s *OrderService) Pay(orderID uint) error {
    resp, err := s.wxpayClient.UnifiedOrder(params)
    if err != nil {
        logger.Error("微信支付请求失败", zap.Error(err))
        return NewBusinessError(ecode.ErrInternal, "支付服务暂时不可用")
    }

    if resp.ReturnCode != "SUCCESS" {
        logger.Error("微信支付返回错误",
            zap.String("returnMsg", resp.ReturnMsg),
            zap.String("errCode", resp.ErrCode),
        )
        return NewBusinessError(ecode.ErrPaymentFailed, "支付失败: "+resp.ReturnMsg)
    }

    return nil
}
```

**原则**：第三方服务的原始错误永远不要透传给客户端。你需要在边界层（Service）把它们转成你的业务错误码。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 错误码设计 | 服务号+模块号+错误序号，如 `10102` |
| 错误码映射 | `map[int]string`，code→中文文案 |
| BusinessError | 自定义错误类型，包含Code + Msg + 原始Err |
| errors.As | 判断 `err` 是不是 `*BusinessError` 类型 |
| 业务错误→客户端 | Code + Msg 直接返回 |
| 系统错误→客户端 | 500 + 通用文案，真实错误记录到日志 |
| 自定义Recovery | `defer recover()` + `debug.Stack()` 记录堆栈 |
| HandleServiceError | 一站式错误处理，自动区分业务/系统错误 |
| 第三方服务错误 | 边界层转成BusinessError，不透传原始错误 |
