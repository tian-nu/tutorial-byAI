## 我在做什么？

第 25 章你学会了备份，但那只是"事后补救"。你的数据库是单机运行的——**一台机器挂了，整个服务就挂了**。而且这台机器要同时承担读写请求，压力全扛在一个肩膀。

**主从复制（Replication）** *此术语见附录E* 就是解决这个问题的第一招：
- 主库（Master）负责写
- 从库（Slave）负责读（或者做备份）
- 主库的数据自动同步到从库

就像你有一本原稿（主库），复印了一份副本（从库）放在另一个地方。原稿上的修改，自动同步到副本上。原稿烧了，副本还在。

学完这一章，你能：
- 理解主从复制的完整原理（一张图讲清）
- 亲手搭建一个主从复制环境
- 用 GTID 管理复制
- 排查复制中断的常见问题
- 理解复制延迟的原因和解决方案

## 一、复制原理：一张图讲清

先别急着搭环境，搞清楚原理比操作重要一万倍。不理解原理，出了问题你只会重启。

### 1.1 核心流程（三个步骤）

```
┌─────────────────────────────────────────────────────────────────┐
│                         主库 (Master)                           │
│                                                                 │
│  客户端写请求 → MySQL Server → binlog（二进制日志）              │
│                                    │                            │
│                                    │ ① 主库写入 binlog           │
│                                    ▼                            │
│                           ┌────────────────┐                    │
│                           │  binlog 文件    │                    │
│                           │ mysql-bin.000001│                    │
│                           └────────┬───────┘                    │
└────────────────────────────────────┼────────────────────────────┘
                                     │
                                     │ ② 从库 IO 线程拉取 binlog
                                     │    （通过网络传输）
                                     ▼
┌─────────────────────────────────────────────────────────────────┐
│                        从库 (Slave)                              │
│                                                                 │
│  ┌──────────────────┐          ┌──────────────────┐             │
│  │ IO 线程 (I/O Thread) │  ───► │ relay log 中继日志 │            │
│  │ 拉取主库 binlog      │  写入   │ slave-relay-bin.xxx │         │
│  └──────────────────┘          └────────┬─────────┘             │
│                                         │                        │
│                                         │ ③ SQL 线程回放         │
│                                         ▼                        │
│                               ┌──────────────────┐              │
│                               │ SQL 线程           │             │
│                               │ 读取 relay log    │              │
│                               │ 执行 SQL 语句     │              │
│                               └──────────────────┘              │
└─────────────────────────────────────────────────────────────────┘
```

**想多一点：为什么从库不直接对主库执行 SQL？**

你可能会想：主库直接通过网络把 SQL 发给从库执行不就行了？为什么还要 binlog → IO 线程 → relay log → SQL 线程这么复杂？

答案：**异步解耦。**

- 如果主库直接发 SQL，主库执行完 SQL 还得等网络发送、从库执行，主库变慢了。
- 如果网络断了，主库还得重试，主库更慢了。
- binlog 是主库正常运转的产物（之前第 24 章学过，binlog 本来就是主库的 crash recovery 机制），主库写完 binlog 就完事了，不用管从库。
- 从库自己来拉取，从库想什么时候拉，主库不关心。这叫**异步复制** *此术语见附录E*。

### 1.2 三个关键角色

| 角色 | 位置 | 职责 |
|------|------|------|
| **binlog** | 主库磁盘 | 记录主库所有写操作，是复制的"数据源" |
| **IO 线程** *此术语见附录E* | 从库 | 连接主库，拉取 binlog，写入 relay log |
| **SQL 线程** *此术语见附录E* | 从库 | 读取 relay log，在从库上执行 SQL |

> 📊 可视化演示见 [26-主从复制_visual.html](26-主从复制_visual.html) — 分步展示这三个角色如何协作完成数据同步。

## 二、三种复制模式：性能 vs 安全的权衡

### 2.1 异步复制（默认）

**主库执行完事务，写入 binlog，就返回客户端"成功"。从库什么时候拉取，主库不管。**

```
主库：写数据 → 写 binlog → 返回客户端"成功" ✅
从库：过一会儿 → 拉取 binlog → 执行
```

- ✅ 性能最好，主库不受从库影响
- ❌ 主库挂了，从库可能还没同步到最新数据 → **数据丢失风险**

**适用场景**：大部分业务，能接受少量数据丢失（如日志、统计）。

### 2.2 半同步复制（Semi-Sync）*此术语见附录E*

**主库执行完事务，写入 binlog，至少等一个从库确认收到 binlog 后，才返回客户端"成功"。**

```
主库：写数据 → 写 binlog → 等待从库确认 → 返回客户端"成功" ✅
从库：收到 binlog → 写入 relay log → 回复主库"收到" ✅
```

- ✅ 强于异步，至少有一个从库有最新数据
- ❌ 性能略差（多等一个网络往返），如果从库全挂了会退化为异步（不阻塞主库）

**MySQL 配置**：`rpl_semi_sync_master_wait_for_slave_count = 1`（至少等 1 个从库确认）

**适用场景**：对数据一致性要求较高的业务（如订单、支付）。

### 2.3 全同步复制

**主库等所有从库都执行完事务，才返回客户端"成功"。**

```
主库：写数据 → 写 binlog → 等待所有从库执行完 → 返回客户端"成功" ✅
从库：收到 binlog → 写入 relay log → 执行完毕 → 回复主库"执行完" ✅
```

- ✅ 数据最安全，所有从库完全一致
- ❌ 性能最差，任何一个从库慢了，主库都跟着慢

**适用场景**：极少使用，因为 MySQL 本身的半同步已经够用。

### 一句话总结

| 模式 | 性能 | 数据安全 | 主库挂后数据丢失？ |
|------|------|---------|------------------|
| 异步 | ⭐⭐⭐ | ⭐ | 可能丢 |
| 半同步 | ⭐⭐ | ⭐⭐ | 大概率不丢 |
| 全同步 | ⭐ | ⭐⭐⭐ | 不丢 |

**生产环境 90% 用异步，金融/电商用半同步。**

## 三、搭建主从复制（极详细步骤）

> ⚠️ 环境要求：需要两台 MySQL 实例。推荐用 Docker 最方便，也可以用两台虚拟机。本节以 Docker 为例，单机模拟也能跑。

### 3.1 用 Docker 启动两台 MySQL

```bash
# 创建 Docker 网络，让两台容器能互相通信
docker network create mysql-repl

# 启动主库（端口 3307）
docker run -d \
  --name mysql-master \
  --network mysql-repl \
  -p 3307:3306 \
  -e MYSQL_ROOT_PASSWORD=root123 \
  -e MYSQL_DATABASE=my_shop \
  mysql:8.0 \
  --server-id=1 \
  --log-bin=/var/lib/mysql/mysql-bin.log \
  --binlog-format=ROW

# 启动从库（端口 3308）
docker run -d \
  --name mysql-slave \
  --network mysql-repl \
  -p 3308:3306 \
  -e MYSQL_ROOT_PASSWORD=root123 \
  -e MYSQL_DATABASE=my_shop \
  mysql:8.0 \
  --server-id=2 \
  --log-bin=/var/lib/mysql/mysql-bin.log \
  --binlog-format=ROW

# 等待 MySQL 启动完毕（约 10 秒）
sleep 15
```

**关键参数解释**：
- `--server-id=1`：每个 MySQL 实例的唯一 ID，主从不能重复！
- `--log-bin=...`：开启 binlog（主库必须，从库建议也开）
- `--binlog-format=ROW`：binlog 格式为 ROW（推荐，最安全）

### 3.2 主库配置：创建复制用户

连上主库（端口 3307）：

```bash
docker exec -it mysql-master mysql -u root -proot123
```

执行以下 SQL：

```sql
-- 创建专门用于复制的用户
CREATE USER 'repl'@'%' IDENTIFIED BY 'repl_pass_123';

-- 授予复制权限
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- 刷新权限
FLUSH PRIVILEGES;

-- 查看主库状态（记住 File 和 Position，后面要用！）
SHOW MASTER STATUS;
```

预期输出：

```
+------------------+----------+--------------+------------------+
| File             | Position | Binlog_Do_DB | Binlog_Ignore_DB |
+------------------+----------+--------------+------------------+
| mysql-bin.000003 |      157 |              |                  |
+------------------+----------+--------------+------------------+
```

**记下 `File` 和 `Position`！** 这两个值告诉从库"从主库 binlog 的哪个位置开始复制"。

### 3.3 主库初始数据导出（如果有数据）

如果主库已经有数据，需要先导出再导入从库：

```bash
# 导出主库数据（注意是端口 3307）
mysqldump -u root -proot123 -h 127.0.0.1 -P 3307 \
  --single-transaction --routines --triggers \
  --master-data=2 \
  my_shop > master_dump.sql

# 导入到从库（端口 3308）
mysql -u root -proot123 -h 127.0.0.1 -P 3308 my_shop < master_dump.sql
```

> `--master-data=2`：在导出的 SQL 里自动注释写入 `CHANGE MASTER TO` 需要的 binlog 位置信息，非常方便。

如果主库是空的，跳过这步。

### 3.4 从库配置：建立复制关系

连上从库（端口 3308）：

```bash
docker exec -it mysql-slave mysql -u root -proot123
```

执行以下 SQL：

```sql
-- 停止从库复制线程（如果之前有配置）
STOP SLAVE;

-- 建立复制关系
CHANGE MASTER TO
  MASTER_HOST='mysql-master',        -- Docker 下用容器名
  MASTER_USER='repl',                -- 复制用户
  MASTER_PASSWORD='repl_pass_123',   -- 复制用户密码
  MASTER_LOG_FILE='mysql-bin.000003', -- 记下的 File
  MASTER_LOG_POS=157;                -- 记下的 Position

-- 启动复制
START SLAVE;

-- 检查复制状态
SHOW SLAVE STATUS\G
```

`SHOW SLAVE STATUS\G` 输出很长，重点看这两行：

```
Slave_IO_Running: Yes          ← IO 线程运行正常 ✅
Slave_SQL_Running: Yes         ← SQL 线程运行正常 ✅
Seconds_Behind_Master: 0       ← 复制延迟 0 秒 ✅
```

**如果两个都是 `Yes`，恭喜！主从复制搭建成功！**

### 3.5 验证复制是否正常

在主库插入数据：

```bash
docker exec -it mysql-master mysql -u root -proot123
```

```sql
USE my_shop;
CREATE TABLE test_repl (id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(50));
INSERT INTO test_repl (name) VALUES ('hello replication!');
SELECT * FROM test_repl;
```

在从库查看同步结果：

```bash
docker exec -it mysql-slave mysql -u root -proot123
```

```sql
USE my_shop;
SELECT * FROM test_repl;
-- 应该看到同一条数据 ✅
```

如果从库立刻看到了数据，说明复制运行正常，延迟接近 0。

## 四、GTID 复制：告别手动记 Position

手动记 `MASTER_LOG_FILE` 和 `MASTER_LOG_POS` 太麻烦了，而且容易搞错。**GTID（Global Transaction Identifier）** *此术语见附录E* 就是来解决这个问题的。

### 4.1 GTID 是什么？

每个事务在集群中有一个全局唯一的 ID，格式：`server-uuid:transaction-id`

```
3E11FA47-71CA-11E1-9E33-C80AA9429562:1-23
│                                    │  │
│                                    │  └── 第 1 到 23 个事务
│                                    └── 在哪个 server 上执行的
└── 这个库的 UUID
```

从库不再需要知道"从 binlog 的哪个位置开始"，而是"从哪个 GTID 之后开始"。

### 4.2 启用 GTID 复制

在 Docker 启动时加参数：

```bash
# 主库
docker run -d \
  --name mysql-master-gtid \
  --network mysql-repl \
  -p 3317:3306 \
  -e MYSQL_ROOT_PASSWORD=root123 \
  mysql:8.0 \
  --server-id=11 \
  --log-bin=/var/lib/mysql/mysql-bin.log \
  --gtid-mode=ON \
  --enforce-gtid-consistency=ON

# 从库
docker run -d \
  --name mysql-slave-gtid \
  --network mysql-repl \
  -p 3318:3306 \
  -e MYSQL_ROOT_PASSWORD=root123 \
  mysql:8.0 \
  --server-id=12 \
  --log-bin=/var/lib/mysql/mysql-bin.log \
  --gtid-mode=ON \
  --enforce-gtid-consistency=ON
```

GTID 模式下，`CHANGE MASTER TO` 简化了：

```sql
-- 从库上执行
CHANGE MASTER TO
  MASTER_HOST='mysql-master-gtid',
  MASTER_USER='repl',
  MASTER_PASSWORD='repl_pass_123',
  MASTER_AUTO_POSITION=1;  -- 自动找位置，不用手动指定 File 和 Position！

START SLAVE;
```

**`MASTER_AUTO_POSITION=1`** 就是 GTID 的精髓：从库自动知道从哪里开始复制，你不需要手动记任何东西。

### 4.3 GTID 的额外好处：主从切换更简单

如果主库挂了，你要把一个从库提升为新主库：

```sql
-- 非 GTID：需要找到所有从库的 binlog 位置，手动对齐，复杂且容易出错
-- GTID：新主库执行，其他从库自动从正确位置继续
STOP SLAVE;
-- 其他从库只需指向新主库
CHANGE MASTER TO MASTER_HOST='new-master', MASTER_AUTO_POSITION=1;
START SLAVE;
```

**强烈建议生产环境开启 GTID。**

## 五、监控复制状态

### 5.1 关键监控指标

```sql
-- 从库执行
SHOW SLAVE STATUS\G
```

重点关注以下字段：

| 字段 | 含义 | 正常值 | 异常值 |
|------|------|--------|--------|
| `Slave_IO_Running` | IO 线程是否运行 | `Yes` | `No` → 连接主库失败 |
| `Slave_SQL_Running` | SQL 线程是否运行 | `Yes` | `No` → 执行 SQL 出错 |
| `Seconds_Behind_Master` | 从库落后主库多少秒 | 0-5 | 持续增长 → 延迟严重 |
| `Last_IO_Error` | 最近 IO 错误 | 空 | 连接失败原因 |
| `Last_SQL_Error` | 最近 SQL 错误 | 空 | SQL 执行失败原因 |

### 5.2 复制延迟的原因

`Seconds_Behind_Master` 飙升了，怎么排查？

**常见原因：**

| 原因 | 表现 | 解决方案 |
|------|------|---------|
| 主库有大事务 | 更新百万行，从库慢慢回放 | 拆大事务为小事务 |
| 从库性能差 | 从库 CPU/磁盘比主库弱 | 升级从库硬件 |
| 从库也在承载读请求 | 从库忙不过来 | 再加从库分摊读压力 |
| 网络带宽不足 | 主库 binlog 太大传不过来 | 升级网络 |
| 从库单线程回放（MySQL 5.6 及以下） | SQL 线程只有一个 | 升级到 MySQL 5.7+，开启并行复制 |

**MySQL 5.7+ 并行复制：**

```sql
-- 从库开启并行复制（基于逻辑时钟）
SET GLOBAL slave_parallel_type = 'LOGICAL_CLOCK';
SET GLOBAL slave_parallel_workers = 4;  -- 4 个并行工作线程
STOP SLAVE;
START SLAVE;
```

## 六、排错预案

### 6.1 server-id 重复

```sql
-- 从库 SHOW SLAVE STATUS\G 显示：
-- Last_IO_Error: Fatal error: The slave I/O thread stops because
-- master and slave have equal MySQL server ids
```

**原因**：主从库 server-id 一样（比如都用了默认的 1）。

**解决**：改了 server-id 需要重启 MySQL。

```sql
-- 检查 server-id
SELECT @@server_id;
-- 如果从库和主库一样，改 my.cnf 里的 server-id，重启。
```

### 6.2 IO 线程连接失败

```sql
-- 从库 SHOW SLAVE STATUS\G 显示：
-- Slave_IO_Running: Connecting
-- Last_IO_Error: error connecting to master 'repl@mysql-master:3306'
```

**排查步骤**：

```bash
# 1. 从库能 ping 通主库吗？
docker exec mysql-slave ping mysql-master

# 2. 主库端口 3306 开放了吗？
docker exec mysql-slave mysql -u repl -prepl_pass_123 -h mysql-master -e "SELECT 1;"

# 3. 复制用户的密码对吗？
# 回到主库，重新创建用户：
# ALTER USER 'repl'@'%' IDENTIFIED BY 'repl_pass_123';
# FLUSH PRIVILEGES;
```

### 6.3 SQL 线程报错（主键冲突）

```sql
-- SHOW SLAVE STATUS\G 显示：
-- Last_SQL_Error: Could not execute Write_rows event on table test.t;
-- Duplicate entry '1' for key 'PRIMARY'
```

**原因**：有人手贱在从库上直接写了数据，导致主键冲突。或者从库之前有残留数据。

**解决**：

```sql
-- 跳过这个错误（慎重！会丢数据）
SET GLOBAL SQL_SLAVE_SKIP_COUNTER = 1;
START SLAVE;

-- 或者直接跳过所有错误（更危险，仅调试用）
-- 不建议在生产环境使用
```

**根本解决方案**：**永远不要在从库上直接写数据！** 从库只读。

```sql
-- 设置从库只读
SET GLOBAL read_only = ON;
-- 但 super 用户仍然可以写，如果需要更严格的只读：
SET GLOBAL super_read_only = ON;  -- MySQL 5.7.8+
```

### 6.4 从库落后太多，追不上

```sql
-- Seconds_Behind_Master 持续增长，从库追不上
```

**排查**：先看大事务

```sql
-- 主库查看当前正在执行的大事务
SELECT * FROM information_schema.innodb_trx
WHERE trx_started < DATE_SUB(NOW(), INTERVAL 10 SECOND)
ORDER BY trx_started;
```

**解决**：
- 拆大事务为小事务
- 开启并行复制（`slave_parallel_workers = 4`）
- 如果实在追不上，重新做一次全量同步

## 我做得对不对？

### 验证方法

```bash
# 1. 主库插入数据
docker exec -it mysql-master mysql -u root -proot123 -e "
  USE my_shop;
  CREATE TABLE IF NOT EXISTS repl_test (id INT PRIMARY KEY AUTO_INCREMENT, ts DATETIME DEFAULT CURRENT_TIMESTAMP);
  INSERT INTO repl_test () VALUES ();
  SELECT * FROM repl_test;
"

# 2. 从库立即查询（应该在 1 秒内看到结果）
docker exec -it mysql-slave mysql -u root -proot123 -e "
  USE my_shop;
  SELECT * FROM repl_test;
"
# 如果看到相同的数据 → ✅ 复制正常

# 3. 从库检查复制状态
docker exec -it mysql-slave mysql -u root -proot123 -e "SHOW SLAVE STATUS\G" | grep -E "Running|Seconds_Behind"
# 预期：Slave_IO_Running: Yes, Slave_SQL_Running: Yes, Seconds_Behind_Master: 0
```

### 进阶验证：制造延迟再恢复

```sql
-- 主库：暂停 binlog 写入（模拟故障）
-- 主库：插入大量数据
-- 从库：观察 Seconds_Behind_Master 增长
-- 主库：恢复后，从库自动追赶上
```

## 不对怎么办？

### 常见错误 1：server-id 重复

```sql
-- ❌ 主库和从库 server-id 都是 1
-- 解决：从库改为 2 或更大的值，重启 MySQL
```

### 常见错误 2：从库直接写入数据

```sql
-- ❌ 有人在从库上 INSERT/UPDATE/DELETE
INSERT INTO users (username) VALUES ('hacker');
-- 主库也有这个 username 的插入 → 从库主键冲突
-- 或者主库没有 → 主从数据不一致，下次在这个表操作时冲突

-- ✅ 给从库加只读
SET GLOBAL super_read_only = ON;
-- 即使是 root 也不能在从库写数据了
```

```sql
-- ❌ 看到 SHOW SLAVE STATUS 报错，不管它

-- ✅ 收到告警立即处理

```

```sql

-- ✅ 用了 GTID 就用 MASTER_AUTO_POSITION=1，不要手动指定

  MASTER_USER='repl',
  MASTER_PASSWORD='repl_pass_123',
```

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 复制原理 | binlog → IO 线程 → relay log → SQL 线程 | 三个角色各司其职，异步解耦 |
| 异步复制 | 默认模式，性能最好 | 主库挂了可能丢数据 |
| 半同步复制 | `rpl_semi_sync_master_enabled=1` | 至少一个从库确认，性能略降 |
| 搭建主从 | server-id + binlog + 复制用户 + CHANGE MASTER TO + START SLAVE | server-id 不能重复！ |
| 验证复制 | `SHOW SLAVE STATUS\G` | IO + SQL 两个线程都是 Yes 才算正常 |
| GTID | `MASTER_AUTO_POSITION=1` | 自动找位置，简化主从切换 |
| 复制延迟 | `Seconds_Behind_Master` | 大事务、从库性能差、单线程回放 |
| 并行复制 | `slave_parallel_workers=4` | MySQL 5.7+ 支持 |
| 从库只读 | `SET GLOBAL super_read_only = ON` | 防止从库写入破坏一致性 |
| 排错 | IO 连接失败、SQL 主键冲突、server-id 重复 | 先看 `Last_IO_Error` 和 `Last_SQL_Error` |

## 最可能出错的地方及原因

1. **server-id 重复**：主从两台机器都用了默认的 server-id=1，原因是没意识到这个参数的重要性。
2. **从库直接写入**：有人在从库上 INSERT 数据，导致主键冲突，原因是不了解主从复制的单向性。
3. **复制中断不处理**：觉得 `SHOW SLAVE STATUS` 报错没事，几天后从库落后太多追不上，原因是没有监控告警。
4. **IO 线程连接失败**：防火墙没开 3306、复制用户密码错误、网络不通，原因是搭建时没逐步验证。
5. **大事务导致延迟**：一个事务更新百万行，从库回放慢，主从延迟飙升，原因是没拆分大事务。
