## 97.1 缓存策略

缓存是高并发系统的第一道防线。用得好，90%的请求根本到不了数据库。本节不是讲Redis怎么用（第58-62章已讲），而是讲**缓存架构的四种模式**。

### Cache Aside（旁路缓存）——最常用

**流程**：

```
读：
  1. 先查缓存 → 命中 → 直接返回
  2. 缓存未命中 → 查数据库 → 结果写入缓存 → 返回

写：
  1. 更新数据库
  2. 删除缓存（不是更新缓存！）
```

**为什么是"删除"而不是"更新"缓存？**

如果更新缓存→两个写操作几乎同时发生→缓存和数据库数据可能不一致：

```
时间线：
T1: 线程A更新DB: stock=10
T2: 线程B更新DB: stock=9
T3: 线程B更新缓存: stock=9
T4: 线程A更新缓存: stock=10（晚于B但覆盖了B的结果！）

结果：DB=9，缓存=10 → 不一致！
```

如果删除缓存→没有更新→下次读的时候自然会从DB加载最新数据。

**Go实现**：

```go
func GetProduct(productID string) (*Product, error) {
	data, err := rdb.Get(ctx, "product:"+productID).Result()
	if err == nil {
		var product Product
		json.Unmarshal([]byte(data), &product)
		return &product, nil
	}

	var product Product
	err = db.Where("id = ?", productID).First(&product).Error
	if err != nil {
		return nil, err
	}

	jsonData, _ := json.Marshal(product)
	rdb.Set(ctx, "product:"+productID, jsonData, 10*time.Minute)
	return &product, nil
}

func UpdateProduct(product Product) error {
	err := db.Save(&product).Error
	if err != nil {
		return err
	}
	rdb.Del(ctx, "product:"+product.ID)
	return nil
}
```

**Cache Aside的缺陷**：第一次请求（缓存为空时）会直接打到数据库。如果缓存恰好过期+瞬时大量请求涌入→全部打到数据库→数据库扛不住。这就是**缓存击穿**。

**缓存击穿的解决**——互斥锁（单机）或分布式锁（Redis）：

```go
func GetProductSafe(productID string) (*Product, error) {
	data, err := rdb.Get(ctx, "product:"+productID).Result()
	if err == nil {
		var product Product
		json.Unmarshal([]byte(data), &product)
		return &product, nil
	}

	lockKey := "lock:product:" + productID
	locked, _ := rdb.SetNX(ctx, lockKey, "1", 5*time.Second).Result()
	if !locked {
		time.Sleep(50 * time.Millisecond)
		return GetProductSafe(productID)
	}
	defer rdb.Del(ctx, lockKey)

	data2, err := rdb.Get(ctx, "product:"+productID).Result()
	if err == nil {
		var product Product
		json.Unmarshal([]byte(data2), &product)
		return &product, nil
	}

	var product Product
	db.Where("id = ?", productID).First(&product)
	jsonData, _ := json.Marshal(product)
	rdb.Set(ctx, "product:"+productID, jsonData, 10*time.Minute)
	return &product, nil
}
```

### Read/Write Through（读/写穿透）

缓存层完全代理数据库——应用只和缓存打交道，不直接访问数据库。

```
应用 → 缓存 → 数据库
```

- **Read Through**：读缓存miss→缓存自己去查数据库→回填缓存→返回
- **Write Through**：应用写缓存→缓存自己同步到数据库

Go里较少手动实现，一般用Redis + 定时任务，或者用ORM的缓存插件。

### Write Behind（写回）

应用只写缓存，**异步**批量写到数据库。这是四种策略中性能最高的，但也是最危险的——如果缓存挂了、还没写到数据库的数据就丢了。

```
应用 → 写缓存 → 立刻返回（不等待DB）
                   │
                   └→ 定时/批量刷到数据库
```

Kafka其实就是这种思想的体现。

### 总结

| 策略 | 性能 | 一致性 | 复杂度 | 推荐场景 |
|------|------|--------|--------|---------|
| Cache Aside | 高 | 最终一致 | 低 | 90%的场景 |
| Read/Write Through | 中 | 强一致 | 中 | 需要缓存层完全代理DB |
| Write Behind | 最高 | 弱 | 高 | 日志、统计、非关键数据 |

## 97.2 限流算法

限流是高并发系统的第二道防线——当流量确实超出了系统能力，不是硬扛，而是有策略地拒绝。

### 固定窗口

把时间分成固定大小的时间窗口（如1秒），统计每个窗口内的请求数。

```
窗口1 (0s-1s):  接收3个请求 → OK
窗口2 (1s-2s):  接收3个请求 → OK
窗口3 (2s-3s):  接收3个请求 → OK

如果1秒内来了30个 → 只接收前3个，后面27个拒绝
```

**致命问题**——边界突刺：

```
窗口1最后0.1秒内来了3个请求（全接收）
窗口2最初0.1秒内来了3个请求（全接收）

实际上0.2秒内涌入了6个请求——但限流器没拦住因为跨了两个窗口
```

### 滑动窗口

把"固定窗口"变成"滑动窗口"——不是看"当前这1秒"，而是看"过去1秒"。

```
时间轴: ...1.5s...1.8s...2.0s...2.3s...2.7s...
请求:     [请求A][请求B]       [请求C][请求D]
                                  ↑
                              现在时刻: 2.3s
                              检查过去1秒 (1.3s~2.3s): 只有C和D → 没超限
```

**Go实现**——用Redis的Sorted Set：

```go
func IsRateLimited(userID string, limit int, window time.Duration) bool {
	now := time.Now().UnixNano() / 1e6
	windowStart := now - window.Milliseconds()

	rdb.ZRemRangeByScore(ctx, "rate:"+userID, "0", strconv.FormatInt(windowStart, 10))

	count, _ := rdb.ZCard(ctx, "rate:"+userID).Result()
	if count >= int64(limit) {
		return true
	}

	rdb.ZAdd(ctx, "rate:"+userID, redis.Z{Score: float64(now), Member: now})
	rdb.Expire(ctx, "rate:"+userID, window)
	return false
}
```

### 漏桶（Leaky Bucket）

想象一个底部有小洞的桶——水（请求）从上面进来，从小洞匀速流出（处理请求）。如果进水的速度超过了出水的速度，桶满了→溢出的水被丢弃。

```
请求 ──→  ┌──桶──┐  ──→ 匀速流出（处理）
             满→溢出→丢弃
```

**特点**：输出速率绝对恒定——不管突发流量多大，处理速度永远是固定的。

```go
type LeakyBucket struct {
	capacity  int
	rate      time.Duration
	queue     chan struct{}
	stopCh    chan struct{}
}

func NewLeakyBucket(capacity int, rate time.Duration) *LeakyBucket {
	b := &LeakyBucket{
		capacity: capacity,
		rate:     rate,
		queue:    make(chan struct{}, capacity),
		stopCh:   make(chan struct{}),
	}
	go b.drain()
	return b
}

func (b *LeakyBucket) drain() {
	ticker := time.NewTicker(b.rate)
	defer ticker.Stop()
	for {
		select {
		case <-ticker.C:
			select {
			case <-b.queue:
			default:
			}
		case <-b.stopCh:
			return
		}
	}
}

func (b *LeakyBucket) Allow() bool {
	select {
	case b.queue <- struct{}{}:
		return true
	default:
		return false
	}
}
```

### 令牌桶（Token Bucket）

漏桶的反面——桶里存的是"令牌"，请求来了要从桶里取令牌，拿到令牌的才能被处理。

```
                       ┌──桶──┐
定时放入令牌 ──────────→│令牌  │←── 请求来取令牌
(匀速添加)              └──────┘
```

**漏桶 vs 令牌桶**：

| 维度 | 漏桶 | 令牌桶 |
|------|------|--------|
| 桶里存什么 | 请求 | 令牌 |
| 处理速率 | 绝对恒定 | 允许突发（桶里攒了令牌可以一次性发出去） |
| 适用场景 | 需要绝对平滑输出的场景 | 需要允许一定突发流量的场景 |

**Go实现**（标准库就有）：

```go
import "golang.org/x/time/rate"

limiter := rate.NewLimiter(rate.Limit(100), 200)

func handler(w http.ResponseWriter, r *http.Request) {
    if !limiter.Allow() {
        http.Error(w, "请求太频繁", 429)
        return
    }
    processRequest(r)
}
```

`rate.Limit(100)`：每秒产生100个令牌。`200`：桶容量（最多存200个令牌→允许200的突发）。

## 97.3 降级

降级（关闭非核心功能保障核心流程，详见附录I）是高并发系统的第三道防线——当压力大到连限流都不够用时，**关闭非核心功能保核心流程**。

### 降级层次

| 级别 | 措施 | 示例 |
|------|------|------|
| 轻度 | 关闭非核心功能 | 商品推荐从"AI推荐"降为"热门商品" |
| 中度 | 关闭辅助服务 | 积分明细不实时更新，改定时批处理 |
| 重度 | 仅保核心交易 | 关闭一切非支付相关功能 |

**Go实现**——配置开关：

```go
type DegradeConfig struct {
	RecommendationEnabled bool
	PointsServiceEnabled  bool
	SearchFromCacheOnly   bool
}

var degradeConfig DegradeConfig

func GetRecommendations(userID string) ([]Product, error) {
	if !degradeConfig.RecommendationEnabled {
		return getHotProducts(), nil
	}
	return aiRecommend(userID)
}
```

## 97.4 熔断

熔断是高并发系统的第四道防线——**当下游服务出问题时，快速失败而不是傻等。**

### 熔断器的三种状态

```
        ┌──────────┐
        │  关闭    │ ← 正常状态，请求直接放行
        │ (Closed) │
        └────┬─────┘
             │ 连续失败N次
             ▼
        ┌──────────┐
        │  打开    │ ← 熔断状态，请求直接失败
        │  (Open)  │
        └────┬─────┘
             │ 等待一段时间
             ▼
        ┌──────────┐
        │  半开    │ ← 试探性放行一个请求
        │(HalfOpen)│
        └────┬─────┘
        成功↙    ↘失败
    关闭状态    打开状态
```

**Go实现**——使用 `sony/gobreaker`：

```go
import "github.com/sony/gobreaker"

var cb *gobreaker.CircuitBreaker

func init() {
    settings := gobreaker.Settings{
        Name:        "user-service",
        MaxRequests: 3,
        Interval:    60 * time.Second,
        Timeout:     30 * time.Second,
        ReadyToTrip: func(counts gobreaker.Counts) bool {
            failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
            return counts.Requests >= 3 && failureRatio >= 0.6
        },
    }
    cb = gobreaker.NewCircuitBreaker(settings)
}

func CallUserService(userID string) (*User, error) {
    result, err := cb.Execute(func() (interface{}, error) {
        return userClient.GetUser(userID)
    })
    if err != nil {
        return nil, err
    }
    return result.(*User), nil
}
```

`ReadyToTrip` 定义了"什么时候熔断"：请求数≥3且失败率≥60%。

## 97.5 隔离

隔离是高并发系统的第五道防线——**把问题关在笼子里，不让它蔓延。**

### 三种隔离方式

**线程池隔离**：

不同业务用不同的goroutine池（虽然Go的goroutine天然轻量，但可以用 `ants` 等协程池库控制）：

```go
orderPool, _ := ants.NewPool(100)
notifyPool, _ := ants.NewPool(50)

orderPool.Submit(func() {
    processOrder(order)
})

notifyPool.Submit(func() {
    sendNotification(notif)
})
```

通知服务再有10000个任务堆积，也只会占用50个goroutine——不会影响订单处理的100个goroutine。

**信号量隔离**：

用channel做信号量，限制并发数：

```go
sem := make(chan struct{}, 10)

func handleWithSemaphore(req Request) {
    sem <- struct{}{}
    defer func() { <-sem }()
    processRequest(req)
}
```

最多10个并发处理，第11个请求会阻塞等待。

**集群隔离**：

`user-service` 部署在三台机器上。给每台机器打标签：
- 机器A：只处理VIP用户
- 机器B+C：处理普通用户

普通用户流量再大也影响不到VIP用户的体验。

## 97.6 超时和重试

超时和重试是高并发系统的**最后一道防线**——但它们也是最容易被"用错"的。

### 超时设置原则

**不是"越长越好"，也不是"越短越好"。**

- 太长→一个慢请求占用连接，堆积→连接池耗尽→整个系统雪崩
- 太短→正常慢的请求被无差别干掉→原本能成功的请求也失败了

```go
client := &http.Client{
    Timeout: 30 * time.Second,
}

ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
defer cancel()

req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
resp, err := client.Do(req)
```

建议：按接口类型设置不同超时：
- 查询接口（getUser）：3秒
- 创建接口（createOrder）：5秒
- 文件上传：60秒

### 重试的三个黄金法则

**法则一：重试必须幂等**

如果一个操作不幂等（如"扣款100元"），重试可能导致重复扣款。确保重试的接口是幂等的——或者不要让非幂等操作自动重试。

**法则二：指数退避 + 随机抖动**

```go
func retryWithBackoff(fn func() error, maxRetries int) error {
    for i := 0; i < maxRetries; i++ {
        err := fn()
        if err == nil {
            return nil
        }

        backoff := time.Duration(math.Pow(2, float64(i))) * time.Second
        jitter := time.Duration(rand.Int63n(int64(backoff / 2)))
        time.Sleep(backoff + jitter)
    }
    return fmt.Errorf("重试%d次后仍失败", maxRetries)
}
```

退避间隔：1秒 → 2秒 → 4秒 → 8秒……（每次翻倍）
随机抖动：1秒 ± 0.5秒（避免所有重试同时发生——**惊群效应**）

**法则三：设置最大重试次数和总超时时间**

不要无限重试。3次是比较合理的上限。

---

🤔 **想多一点**：为什么"超时不重试"有时更好？

很多开发者本能反应："超时了？重试！"但在微服务架构中，一个请求可能穿透5个服务：

```
用户 → API网关 → 订单服务 → 库存服务 → 支付服务 → 短信服务
```

如果每一层都"失败就重试3次"，一次用户请求最多可能产生 `3^5 = 243` 次内部调用——在系统已经压力巨大的时候，这些重试只会让它死得更快。

**建议**：
- 只在"最前端"重试（如API网关层）
- 内部服务调用不重试或只重试1次
- 或者用**请求ID去重**：后端收到相同请求ID的第二条请求时，直接返回"正在处理中，稍后查结果"

## 本章小结

| 知识点 | 要点 |
|--------|------|
| Cache Aside | 最常用：读cache→miss→查DB→回写；写DB→删cache |
| 缓存击穿 | 热点数据过期瞬间的高并发，用互斥锁/分布式锁解决 |
| 固定窗口 | 简单但有"边界突刺"问题 |
| 滑动窗口 | Redis Sorted Set实现，真正的"过去N秒"统计 |
| 漏桶 | 绝对恒定速率，不允许突发 |
| 令牌桶 | 允许突发，Go标准库 `golang.org/x/time/rate` |
| 降级 | 关闭非核心功能保核心流程 |
| 熔断 | 三状态：关闭→打开→半开→恢复或继续熔断 |
| 隔离 | 线程池/信号量/集群，把问题关在笼子里 |
| 超时 | 按接口设置，3~30秒不等 |
| 重试 | 幂等+指数退避+随机抖动+有限次数 |
