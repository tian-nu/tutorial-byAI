**一句话目标**：给博客系统加上"文章归属"——每篇文章记住是谁写的，更新和删除时检查"你是不是作者本人"，不是就拒绝。

## 三、分步操作

### 步骤 1：当前问题——登录用户可以改/删任何人的文章

在教程 16 中，我们加了一层保护：**没登录的人不能创建/修改/删除文章**。这解决了"游客乱改"的问题。

但还有一个漏洞：**登录用户可以改任何人的文章**。

**比喻**：你把公寓大门锁上了（教程 16：没登录不能进），但每个房间的门都没锁——任何住客（已登录用户）都能推开任意房间的门（改任何人的文章）。

**演示漏洞**：假设 alice 创建了一篇文章，bob 登录后去删它：

```bash
# 1. alice 登录，获取 token
$AliceToken = curl -X POST http://127.0.0.1:8000/login `
  -H "Content-Type: application/json" `
  -d '{"username":"alice","password":"alice123"}' `
  | python -c "import sys,json; print(json.load(sys.stdin)['access_token'])"

# 2. alice 创建一篇文章
curl -X POST http://127.0.0.1:8000/posts `
  -H "Content-Type: application/json" `
  -H "Authorization: Bearer $AliceToken" `
  -d '{"title":"alice 的私密日记","content":"只有我能看！","published":true}'
# → 返回 id=1 的文章

# 3. bob 登录，获取 token
$BobToken = curl -X POST http://127.0.0.1:8000/login `
  -H "Content-Type: application/json" `
  -d '{"username":"bob","password":"bob123"}' `
  | python -c "import sys,json; print(json.load(sys.stdin)['access_token'])"

# 4. bob 尝试删除 alice 的文章
curl -X DELETE http://127.0.0.1:8000/posts/1 `
  -H "Authorization: Bearer $BobToken"
```

**如果这一步成功了（返回删除成功的消息），说明漏洞存在——bob 删掉了 alice 的文章！** 这就是本章要修的问题。

> 🤔 想多一点：为什么教程 16 没顺便修这个问题？

**因为"能不能操作"和"操作谁的东西"是两个不同维度的权限。** 教程 16 解决的是第一个维度（身份认证），本章解决的是第二个维度（资源归属）。分开学，每个维度只关注一件事，不容易乱。

就像盖房子：先砌墙（教程 16：挡住没登录的人），再装门锁（本章：每个房间只能主人进）。

### 步骤 2：创建文章时自动绑定作者

修漏洞的第一步：**让每篇文章记住"是谁写的"**。这样以后检查才有依据。

打开 `blog_backend/main.py`，找到 `create_post` 接口，把 `author_id` 加进去：

```python
# === 创建文章（改造后） ===
@app.post("/posts")
def create_post(
    blog: BlogCreate,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)  # ← 教程 16 已加
):
    new_post = Post(
        title=blog.title,
        content=blog.content,
        published=blog.published,
        summary=blog.summary,
        author_id=current_user.id,  # ← 新增：绑定作者
    )
    db.add(new_post)
    db.commit()
    db.refresh(new_post)
    return new_post
```

**只有一行是新的**：`author_id=current_user.id`。意思是"这篇文章的作者是当前登录用户"。

**逐行解释**：
- `current_user: User = Depends(get_current_user)`：教程 16 已加，表示"当前请求必须带有效的 JWT token，否则直接返回 401"。
- `current_user.id`：当前登录用户的数据库 ID。`get_current_user` 从 token 中解析出用户信息，然后从数据库查出完整的 `User` 对象给你。
- `author_id=current_user.id`：把作者 ID 写入文章。从现在开始，每篇文章都知道"是谁写的"。

**不需要改 `BlogCreate` 模型**：`author_id` 是服务端自动填的，客户端不应该传（传了也不该信）。客户端只需要传 `title`、`content`、`published` 这些"文章内容"，作者由服务端根据 token 自动确定。

**比喻**：你去邮局寄信，你只需要写"信的内容"和"收件人地址"——寄件人（author_id）是邮局根据你的身份证自动填的，不需要你手写，也不能由你冒充别人。

**❌ 常见错误 → ✅ 正确示例**：

❌ 错误：让客户端传 `author_id`

```python
# 错误：信任客户端传的 author_id
new_post = Post(
    title=blog.title,
    content=blog.content,
    author_id=blog.author_id,  # ← 客户端传什么就信什么？！
)
```

**后果**：bob 可以伪造请求，把 `author_id` 设成 alice 的 ID，假装文章是 alice 写的——所有权限检查全部失效。

✅ 正确：author_id 由服务端根据 token 自动确定

```python
new_post = Post(
    title=blog.title,
    content=blog.content,
    author_id=current_user.id,  # ← 服务端从 token 中获取，信得过
)
```

**核心原则**：**永远不要信任客户端传来的身份信息。** 身份只能从 token 中解析，由服务端确定。

**验证方法**：

保存文件，服务器自动重启。用 alice 登录并创建一篇文章：

```bash
# 1. alice 登录
$AliceToken = curl -X POST http://127.0.0.1:8000/login `
  -H "Content-Type: application/json" `
  -d '{"username":"alice","password":"alice123"}' `
  | python -c "import sys,json; print(json.load(sys.stdin)['access_token'])"

# 2. 创建文章
curl -X POST http://127.0.0.1:8000/posts `
  -H "Content-Type: application/json" `
  -H "Authorization: Bearer $AliceToken" `
  -d '{"title":"alice 的文章","content":"这是 alice 写的。","published":true}'
```

预期输出中，`author_id` 应该等于 alice 的用户 ID（通常是 1）：

```json
{
  "id": 1,
  "title": "alice 的文章",
  "content": "这是 alice 写的。",
  "published": true,
  "author_id": 1,
  "created_at": "...",
  "updated_at": "..."
}
```

`author_id: 1` 说明文章绑定到了 alice（用户 ID=1）。✅

### 步骤 3：更新/删除前检查所有权

现在文章有主人了，下一步：**在更新和删除文章之前，检查"你是不是主人"。**

打开 `blog_backend/main.py`，修改 `update_post` 和 `delete_post` 两个接口。

#### 3.1 改造 `PUT /posts/{post_id}` —— 更新前检查

```python
# === 更新文章（改造后） ===
@app.put("/posts/{post_id}")
def update_post(
    post_id: int,
    blog: BlogCreate,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)
):
    # 先查找这篇文章
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法更新"}
        )

    # 新增：检查是不是作者本人
    if post.author_id != current_user.id:
        raise HTTPException(
            status_code=403,
            detail="你不能修改别人的文章"
        )

    # 是作者本人，允许修改
    post.title = blog.title
    post.content = blog.content
    post.published = blog.published
    post.summary = blog.summary
    db.commit()
    return post
```

**逐行解释新增部分**：

- `if post.author_id != current_user.id:`：比较"文章的作者 ID"和"当前登录用户的 ID"。如果不相等，说明当前用户不是文章作者。
- `raise HTTPException(status_code=403, detail="你不能修改别人的文章")`：**抛出 403 Forbidden 异常**。FastAPI 会自动把这个异常转成 HTTP 403 响应返回给客户端。注意这里用的是 `raise HTTPException(...)` 而不是 `return JSONResponse(...)`——两种写法都能返回错误响应，但 `HTTPException` 是 FastAPI 推荐的方式，更简洁，而且会自动生成标准的错误响应格式。
- 如果检查通过（`author_id == current_user.id`），继续执行原来的更新逻辑，完全不变。

**比喻**：你去酒店前台说"我要进 808 房间"。前台先查房卡系统——808 是你开的吗？不是 → 拒绝（403）。是 → 给你钥匙，请进。

#### 3.2 改造 `DELETE /posts/{post_id}` —— 删除前检查

```python
# === 删除文章（改造后） ===
@app.delete("/posts/{post_id}")
def delete_post(
    post_id: int,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)
):
    # 先查找这篇文章
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法删除"}
        )

    # 新增：检查是不是作者本人
    if post.author_id != current_user.id:
        raise HTTPException(
            status_code=403,
            detail="你不能删除别人的文章"
        )

    # 是作者本人，允许删除
    db.delete(post)
    db.commit()
    return {"message": f"文章 #{post_id}「{post.title}」已删除"}
```

逻辑和更新完全一样：**先查文章 → 检查主人 → 不匹配就 403 → 匹配就继续删。**

**注意检查顺序**：先判 404（文章不存在），再判 403（不是你的）。为什么？如果文章不存在，直接返回 404，不需要暴露"这篇文章存在但你不配"的信息。这是一个安全细节——不要让攻击者通过"403 vs 404"来探测哪些文章 ID 存在。

### 步骤 4：403 Forbidden vs 401 Unauthorized —— 两个"拒绝"的区别

你可能注意到了：教程 16 返回的是 **401**，本章返回的是 **403**。它们都是"拒绝访问"，但含义完全不同。

| | 401 Unauthorized | 403 Forbidden |
|------|------|------|
| **中文** | 未认证 | 禁止访问 |
| **含义** | "你是谁？我不知道你是谁。" | "我知道你是谁，但你不能做这个。" |
| **什么时候返回** | 没带 token、token 过期、token 无效 | 带了有效 token，但权限不够 |
| **客户端该怎么做** | 先去登录，拿到 token 再来 | 登录也没用，你没权限 |
| **本章场景** | 没登录就想改文章 | 登录了，但想改别人的文章 |

**比喻**：

- **401** = 你没带门禁卡，连公寓大门都进不去。保安说："你是谁？没卡不能进。"——你需要先去办卡（登录）。
- **403** = 你带了门禁卡，进了公寓大门，但想进 808 房间。门锁系统说："我知道你是 bob，但 808 是 alice 的房间，你不能进。"——你有卡也没用，这房间不是你的。

```
你（bob）→ 公寓大门（401 检查）→ 通过！→ 808 房间门（403 检查）→ 拒绝！
                 ↑                              ↑
           没带卡 = 401                    带了卡但不是你的房间 = 403
```

**一句话记忆**：
- **401**：问"你是谁？"
- **403**：说"我知道你是谁，但你不配。"

> 🤔 想多一点：为什么 HTTP 状态码叫 401 "Unauthorized" 而不是 "Unauthenticated"？

这是 HTTP 规范的一个历史遗留命名问题。401 的字面意思是"未授权"，但实际上它表示的是"未认证"（你没证明你是谁）。真正的"未授权"（你没权限）是 403。所以很多开发者会记混。

**记住这个口诀**：401 = 没带身份证，403 = 带了身份证但没被邀请。

### 步骤 6：抽取权限检查为公共函数

现在 `update_post` 和 `delete_post` 里各有一段一模一样的权限检查代码：

```python
if post.author_id != current_user.id:
    raise HTTPException(status_code=403, detail="...")
```

**重复代码 = 坏味道。** 如果以后加新功能（比如"只有作者能看草稿"），你又要复制粘贴一遍。万一哪天权限逻辑要改，你得满世界找所有复制粘贴的地方。

**解决方法**：把权限检查抽成一个公共函数，放在 `blog_backend/auth.py` 里。

打开 `blog_backend/auth.py`，在文件末尾追加：

```python
from fastapi import HTTPException
from blog_backend.models import User


def check_owner(obj, current_user: User):
    """
    检查 current_user 是否是 obj 的所有者。
    如果不是，抛出 403 Forbidden。

    要求 obj 必须有 author_id 属性。
    """
    if obj.author_id != current_user.id:
        raise HTTPException(
            status_code=403,
            detail="你不能操作别人的资源"
        )
```

**逐行解释**：

- `def check_owner(obj, current_user: User):`：接收两个参数——`obj` 是任意有 `author_id` 属性的对象（文章、评论、任何"有主人"的东西），`current_user` 是当前登录用户。
- `obj.author_id != current_user.id`：比较主人 ID。这个函数不关心 `obj` 是文章还是评论还是别的什么——只要它有 `author_id` 属性，就能用。
- `raise HTTPException(...)`：不匹配就抛 403。注意这里没有 `return`——因为异常直接中断了函数执行，调用方不需要 `if` 判断返回值。

**设计原则**：这个函数遵循"有罪推定"——默认不通过，抛异常。调用方只需写一行 `check_owner(post, current_user)`，如果没抛异常，就说明检查通过了，可以放心操作。如果抛了异常，FastAPI 自动处理，调用方什么都不用管。

**比喻**：`check_owner` 就像一个门卫函数。你把"房间"和"访客"交给它，它要么放行（没异常），要么拦下（抛 403）。你不需要关心它怎么检查的，只需要在关键入口调用它。

现在修改 `main.py` 中的 `update_post` 和 `delete_post`，用 `check_owner` 替换重复代码。

首先在 `main.py` 顶部导入 `check_owner`：

```python

```

然后修改两个接口：

**`update_post` 改造后**：

```python
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法更新"}
        )

    check_owner(post, current_user)  # ← 一行替代了原来的 if + raise

    post.title = blog.title
    post.content = blog.content
    post.published = blog.published
    post.summary = blog.summary
    db.commit()
    return post
```

### 步骤 7：🤔 想多一点——这个权限逻辑够用吗？

到这一步，你的博客系统有了一个基础的所有权检查：**只有作者本人能改/删自己的文章。**

**但真正的权限系统通常比这复杂得多。** 考虑以下几个场景：

| 场景 | 当前系统能处理吗？ | 为什么不够？ |
|------|:---:|------|
| 管理员可以删除任何人的文章（比如删违规内容） | ❌ | `check_owner` 只认 `author_id`，管理员也不是作者 |
| 编辑可以修改别人的文章（但不能删除） | ❌ | 权限粒度不够——只有"能"和"不能"两种，没有"能改不能删"这种细分 |
| VIP 用户的文章不能被任何人删除（包括管理员） | ❌ | 需要特殊标记和更复杂的优先级判断 |
| 某篇文章设置为"协作模式"，允许多人编辑 | ❌ | 所有权不是"一对一"的，需要"多对多" |

**这些场景需要一套更强大的权限系统：RBAC（Role-Based Access Control，基于角色的访问控制）。** 此术语需进附录。

**RBAC 的核心思想**：不直接给"人"分配权限，而是给"角色"分配权限，再把人分配到角色。

```
人（alice）→ 角色（管理员）→ 权限（删除任何文章）
人（bob）  → 角色（编辑）  → 权限（修改任何文章，不能删除）
人（charlie）→ 角色（作者）→ 权限（只能改/删自己的文章）
```

**对于我们的博客系统，当前的 `check_owner` 够用了。** 因为需求很简单：每个用户只能管自己的文章。RBAC 是进阶话题，留给后续教程（或者你未来的项目）。

**但你今天学到的 `check_owner` 模式是通用的**——无论以后你用 RBAC 还是更复杂的权限系统，核心思想不变：**在每个敏感操作之前，先检查"你有没有资格"**。`check_owner` 只是一个具体实现，你可以把它换成 `check_role(user, "admin")`、`check_permission(user, "delete_post")` 等等——模式是一样的。

```python
from fastapi import HTTPException
from blog_backend.models import User


def check_owner(obj, current_user: User):
    """
    检查 current_user 是否是 obj 的所有者。
    如果不是，抛出 403 Forbidden。

    要求 obj 必须有 author_id 属性。
```

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| 让客户端传 `author_id` | bob 可以伪造作者身份 | 客户端传来的数据不可信 | `author_id` 必须由服务端从 token 中获取：`author_id=current_user.id` |
| 忘记检查所有权 | 登录用户可以改/删任何人的文章 | 只加了 `get_current_user` 依赖，但没有比较 `author_id` | 在更新和删除接口中加 `check_owner(post, current_user)` |
| 403 检查放在 404 之前 | 攻击者可以通过"403 vs 404"探测哪些文章 ID 存在 | 检查顺序错误，先暴露了文章存在的信息 | 始终先检查资源是否存在（404），再检查权限（403） |
| 把 `check_owner` 的异常当成普通返回值 | 写了 `if not check_owner(...): return` 这样的代码 | `check_owner` 不返回布尔值，不匹配时直接抛异常，调用方不需要 `if` | 直接调用 `check_owner(post, current_user)`，不写 `if`，不检查返回值 |
| 401 和 403 搞混 | 用户没登录返回 403，或者越权返回 401 | 两个状态码含义不同，用错了客户端会困惑 | 401 = 没认证（没带 token），403 = 没授权（带了 token 但没权限） |
