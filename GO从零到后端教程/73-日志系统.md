# 第73章 · 日志系统

> "飞机上有黑匣子，记录飞行途中的一切数据——出了事故，调查员靠它还原真相。后端服务的黑匣子就是**日志系统**。没有日志，用户投诉'为什么我的订单不见了'的时候，你只能靠猜。有了日志，你只需搜索那个用户的ID，几分钟内就能还原整个请求链路。"

---

## 73.1 为什么要日志

### 日志的四大用途

| 用途 | 场景 |
|------|------|
| **排错** | 用户说"支付了但订单状态没变"——日志里能看到支付回调是否到达、处理过程有无报错 |
| **监控** | 凌晨三点数据库查询突然变慢——慢查询日志触发告警 |
| **审计** | 谁在什么时间修改了商品价格——操作日志留痕 |
| **数据分析** | 哪个API调用量最大、平均响应时间多少——日志聚合分析 |

### 没有日志的切肤之痛

```
用户：我的订单支付成功了但显示未支付！
你：让我查查……（打开数据库，数据是正常的，没有异常记录）
你：能告诉我几点支付的？
用户：大概是下午3点左右。
你：……（没有任何日志，完全无法排查）
```

### 有日志时

```
用户：我的订单支付成功了但显示未支付！
你：你的用户ID是什么？
用户：User_12345
你：（在日志系统搜索 User_12345）
[14:52:03] INFO 收到支付回调 orderID=ORD-789 payStatus=SUCCESS
[14:52:03] ERROR 更新订单状态失败 orderID=ORD-789 error="database connection timeout"
[14:52:03] WARN  支付回调重试第1次 orderID=ORD-789
[14:52:05] INFO 更新订单状态成功 orderID=ORD-789 status=PAID
你：找到了，下午2点52分支付回调成功了，但数据库当时有一个短暂超时，2秒后自动重试成功了。您现在刷新看看是不是已经显示已支付了？
```

**日志是你写的代码在生产环境中的"眼睛"。**

## 73.2 Go标准库log包

Go标准库 `log` 是最基础的日志工具。

### 基本用法

```go
import "log"

func main() {
    log.Println("服务启动")
    log.Printf("监听端口: %d", 8080)
    log.Fatalf("数据库连接失败: %v", err)
}
```

`log.Println` → 打印一行日志。
`log.Printf` → 格式化打印。
`log.Fatalf` → 打印后调用 `os.Exit(1)`，**程序直接退出**。

### 标准库log的不足

```go
2024/01/15 14:32:05 服务启动
2024/01/15 14:32:05 监听端口: 8080
```

- **没有日志级别**：你不能区分这行是Debug还是Error
- **没有结构化**：日志内容是纯文本字符串，无法按字段搜索
- **没有文件输出**：全打印到stderr
- **性能一般**：每次写日志都加锁

对于生产级项目，标准库 `log` 不够用。我们需要更强大的日志库。

## 73.3 结构化日志：zap

[zap](https://github.com/uber-go/zap) 是Uber开源的高性能结构化日志库。性能是标准库的10倍以上，且支持**结构化字段**。

### 安装

```bash
go get -u go.uber.org/zap
```

### 结构化日志是什么

**非结构化（传统）**：

```
[ERROR] 用户登录失败，用户名=张三，IP=192.168.1.1，原因=密码错误
```

要统计"因为密码错误失败的有多少个用户"，你得写正则表达式去解析日志文本。

**结构化（zap风格）**：

```json
{"level":"error","msg":"用户登录失败","username":"张三","ip":"192.168.1.1","reason":"密码错误","ts":"2024-01-15T14:32:05.000Z"}
```

每个字段都是独立的键值对，用Elasticsearch、Grafana Loki等日志平台可以按任意字段搜索和聚合。

### 两种Logger：SugaredLogger vs Logger

```go
package main

import (
    "go.uber.org/zap"
)

func main() {
    logger, _ := zap.NewProduction()
    defer logger.Sync()

    logger.Info("服务启动成功",
        zap.String("port", "8080"),
        zap.Int("pid", 12345),
    )

    sugar := logger.Sugar()
    sugar.Infow("用户登录",
        "username", "张三",
        "ip", "192.168.1.1",
    )
    sugar.Infof("请求耗时: %dms", 234)
}
```

| 特性 | Logger | SugaredLogger |
|------|--------|---------------|
| 性能 | 极高 | 高（比Logger慢约30%） |
| 用法 | `zap.String("key", val)` | `sugar.Infow("msg", "key", val)` |
| 格式化 | 不支持 `fmt.Sprintf` 风格 | 支持 `Infof("msg %s", val)` |
| 推荐场景 | 高频调用、性能关键路径 | 一般业务代码 |

大多数项目直接用 `SugaredLogger`——写法更简洁，性能损失在业务场景中几乎感觉不到。

### 两种预设：Production vs Development

```go
logger, _ := zap.NewProduction()
```

- JSON格式输出
- 只记录 `Info` 级别及以上
- 自动添加 `caller`（调用文件名+行号）
- 带有 `stacktrace`（Error及以上级别）

```go
logger, _ := zap.NewDevelopment()
```

- 控制台友好格式（带颜色）
- 记录 `Debug` 级别及以上
- 自动添加 `caller`
- 带有 `stacktrace`（Warn及以上级别）

**推荐**：开发时用 `NewDevelopment()`，上线前切换到 `NewProduction()`。

## 73.4 日志分级

不要把所有日志混在一起打。给日志分级别，让你在排查问题时直击要害。

### 五大级别

| 级别 | 含义 | 场景 |
|------|------|------|
| **Debug** | 调试信息 | "进入函数A，参数x=5" |
| **Info** | 常规操作信息 | "用户张三登录成功"、"订单ORD-789创建完成" |
| **Warn** | 警告，但还能继续 | "数据库连接池使用率90%"、"重试第3次" |
| **Error** | 出错了，需要关注 | "支付回调验签失败"、"文件上传保存失败" |
| **Fatal** | 致命错误，程序退出 | "无法连接数据库"、"配置文件缺失" |

### 级别的作用

- **开发环境**：打印Debug及以上所有日志
- **生产环境**：只打印Info及以上日志（Debug日志太多，影响性能且没有排查价值）

### 日志级别的使用原则

```
能够继续服务的异常 → Warn（告警但不中断）
需要人工介入的错误 → Error（记录并通知）
启动必须的资源缺失 → Fatal（终止程序）
```

不要在循环里打Info级别的日志——每秒钟几百条日志会把磁盘撑爆。高频操作用Debug级别。

## 73.5 日志切割：lumberjack

如果所有日志写进一个文件，一个月后这个文件可能会到几百GB。你需要**日志切割**——按大小或按时间把日志分成多个文件。

### 安装lumberjack

```bash
go get gopkg.in/natefinch/lumberjack.v2
```

### 集成到zap

```go
package main

import (
    "go.uber.org/zap"
    "go.uber.org/zap/zapcore"
    "gopkg.in/natefinch/lumberjack.v2"
)

func InitLogger() *zap.Logger {
    writer := &lumberjack.Logger{
        Filename:   "./logs/app.log",
        MaxSize:    100,
        MaxBackups: 10,
        MaxAge:     30,
        Compress:   true,
    }

    encoder := zapcore.NewJSONEncoder(zap.NewProductionEncoderConfig())
    core := zapcore.NewCore(encoder, zapcore.AddSync(writer), zapcore.InfoLevel)

    return zap.New(core)
}
```

### lumberjack配置说明

| 参数 | 含义 | 示例值 |
|------|------|--------|
| `Filename` | 日志文件路径 | `"./logs/app.log"` |
| `MaxSize` | 单个文件最大MB数，超了切割 | `100` |
| `MaxBackups` | 最多保留的旧文件数 | `10` |
| `MaxAge` | 旧文件保留天数 | `30` |
| `Compress` | 旧文件是否gzip压缩 | `true` |

切割后的文件命名：`app-2024-01-15T14-32-05.000.log`。

## 73.6 Gin集成zap中间件

替换Gin默认的Logger中间件，用zap记录请求日志。

```go
func GinZapLogger(logger *zap.Logger) gin.HandlerFunc {
    return func(c *gin.Context) {
        start := time.Now()
        path := c.Request.URL.Path
        query := c.Request.URL.RawQuery

        c.Next()

        cost := time.Since(start)

        logger.Info("HTTP请求",
            zap.Int("status", c.Writer.Status()),
            zap.String("method", c.Request.Method),
            zap.String("path", path),
            zap.String("query", query),
            zap.String("ip", c.ClientIP()),
            zap.Duration("cost", cost),
            zap.Int("body_size", c.Writer.Size()),
        )
    }
}
```

使用：

```go
logger := InitLogger()

r := gin.New()
r.Use(GinZapLogger(logger))
r.Use(gin.Recovery())
```

## 73.7 请求链路追踪：traceID

当一个用户请求进来，它可能经过多个微服务、多次数据库查询、多次缓存读写。如果每行日志都只是散落的独立信息，你很难把一个请求的所有行为串起来。**traceID** 解决了这个问题——给每个请求打上一个唯一ID，这个ID贯穿请求的整个生命周期。

### 中间件注入traceID

```go
import "github.com/google/uuid"

func TraceMiddleware(c *gin.Context) {
    traceID := c.GetHeader("X-Trace-ID")
    if traceID == "" {
        traceID = uuid.New().String()
    }

    c.Set("traceID", traceID)
    c.Header("X-Trace-ID", traceID)
    c.Next()
}
```

### 在handler中记录带traceID的日志

```go
r.GET("/users/:id", func(c *gin.Context) {
    traceID, _ := c.Get("traceID")
    logger.Info("查询用户",
        zap.String("traceID", traceID.(string)),
        zap.String("userID", c.Param("id")),
    )
})
```

现在当用户投诉"我的查询为什么这么慢"，你拿到traceID，在日志系统搜这个ID，就能看到这次请求的所有日志：

```
[traceID=abc123] 收到请求 GET /users/42
[traceID=abc123] 查询数据库 users WHERE id=42
[traceID=abc123] 数据库返回 1条记录，耗时 2.3s
[traceID=abc123] 响应 200 OK
```

一个ID把散落的所有日志串成完整的故事。

### 更好的做法：把traceID注入Custom Logger

```go
func LoggerWithTraceID(logger *zap.Logger) gin.HandlerFunc {
    return func(c *gin.Context) {
        traceID := c.GetHeader("X-Trace-ID")
        if traceID == "" {
            traceID = uuid.New().String()
        }
        c.Set("traceID", traceID)

        childLogger := logger.With(zap.String("traceID", traceID))
        c.Set("logger", childLogger)

        c.Next()
    }
}
```

Handler中使用：

```go
// 安全写法：MustGet在key不存在时会panic，改用Get做类型断言
loggerVal, exists := c.Get("logger")
if exists {
	if logger, ok := loggerVal.(*zap.Logger); ok {
		logger.Info("查询用户", zap.String("userID", id))
	}
}
```

每条日志自动带上traceID，不用每次手动写。

## 🤔 想多一点：日志的代价

### 1. 过多的日志比没日志更糟糕

生产环境每秒钟产生几十MB的Debug日志 = 磁盘很快耗尽、IO飙升拖慢整个服务。

**规则**：
- 生产环境只打Info及以上
- 关键路径打Info（登录成功、支付完成、创建订单）
- 异常路径打Error（支付失败、数据库超时、参数校验失败）
- Debug日志只在排查特定问题时临时开启

### 2. 不要在日志里记录敏感信息

密码、身份证号、银行卡号、Token——这些永远不要进日志。

```go
logger.Info("登录", zap.String("password", password))
```

这是安全灾难。日志文件可能会被第三方日志平台收集、被运维人员查看、被备份到不安全的存储。

### 3. 日志不是监控告警系统

日志记录"发生了什么"，但不主动通知你"出事了"。你需要搭配Prometheus（监控指标）+ Grafana（可视化）+ AlertManager（告警通知），形成完整的可观测性体系。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 日志用途 | 排错、监控、审计、数据分析 |
| 标准库log | `log.Println/Printf/Fatalf`，无分级，无结构化 |
| zap | 高性能结构化日志，支持字段搜索聚合 |
| SugaredLogger | `Infow/Infof` 风格，比Logger稍慢但更易用 |
| Production | JSON格式 + Info级别 + caller信息 |
| Development | 彩色输出 + Debug级别 + stacktrace |
| 日志分级 | Debug(调试) → Info(常规) → Warn(警告) → Error(错误) → Fatal(致命) |
| lumberjack | 按大小/时间切割日志文件，旧文件压缩 |
| Gin集成 | 替换默认Logger，用zap记录HTTP请求耗时/状态/IP |
| traceID | 给每个请求打唯一ID，贯穿全链路日志搜索 |
| 安全红线 | 密码/身份证/Token绝不进日志 |
