# 第1章 · 计算机的大脑——CPU

> "CPU是计算机的大脑——但大脑到底在干什么？为什么有些程序快有些程序慢？3.0GHz到底是什么意思？"

---

## 1.1 什么是CPU

### 用最简单的比喻

CPU = Central Processing Unit = 中央处理器。

把它想象成一个超级勤奋的**算术狂人**。这个人坐在一个小房间里，面前有一堆纸条。纸条上写着非常简单的指令：

```
把纸条A上的数字加1
把纸条B上的数字存到纸条C
如果纸条A大于10，跳到纸条D继续
```

这就是CPU干的全部事情——**执行指令**。它不聪明，它不会"思考"，它只是疯狂地、机械地、以极高的速度执行一条又一条的简单指令。

### CPU的本质

CPU是一块硅片，上面蚀刻着几十亿个微小的**晶体管**（transistor）。晶体管就像一个微小的电子开关——要么开（通电流），要么关（不通电流）。用这些开关的排列组合，就能实现加减乘除、比较大小、条件跳转等操作。

这张芯片大约只有你指甲盖那么大，但里面集成的晶体管数量比地球上的人口还多（一颗现代CPU有几十亿甚至上百亿个晶体管）。

### 指令是什么

CPU能理解的"语言"只有一种——**机器语言**（CPU原生指令，详见附录I）。机器语言就是纯数字（准确的说是二进制——全是0和1）。比如在x86架构的CPU上：

```
10110000 01100001
```

这条指令的意思是："把数字97放到一个叫AL的寄存器里"。

对人类来说这太反人类了，所以我们发明了汇编语言（下一节讲），后来又发明了高级语言（Go、Python等）。

CPU能执行的指令种类非常有限，通常包括：

| 指令类型 | 作用 | 比喻 | 对应的Go代码 |
|----------|------|------|-------------|
| 算术指令 | 加减乘除 | 算账 | `a + b`、`a * b` |
| 逻辑指令 | 与、或、非、异或 | 判断真伪 | `a && b`、`a || b` |
| 数据传输指令 | 把数据从一个地方搬到另一个地方 | 搬东西 | `x = 42`、`p = &x` |
| 控制转移指令 | 跳转到某条指令、条件跳转 | 改路线 | `if`、`for`、`goto` |
| 比较指令 | 比较两个数的大小 | 比大小 | `a > b`、`a == b` |

> 💡 **别被吓到**：你现在不需要记住这些指令类型。这张表只是想告诉你一件事——你以后写的每一行Go代码，最终都会变成上面这些最简单的操作。`fmt.Println("Hello")` 看起来是一行，实际上会被编译成几百条这样的基本指令。但作为程序员，你只需要写 `fmt.Println("Hello")`，编译器帮你翻译。

## 1.2 机器语言、汇编语言、高级语言

### 三者的关系

```
高级语言（Go）    →   汇编语言    →   机器语言（0和1）
你写的代码             人类可读的指令           CPU真正执行的
```

### 机器语言：CPU的母语

前面说过，CPU只认识0和1。机器语言就是这些0和1的组合。比如在x86架构上：

```
机器语言（十六进制表示）: B0 61
含义: 把数字97放到AL寄存器
```

### 汇编语言：给机器语言起名字

0和1太难记了，所以人类给每条机器指令起了个英文名：

```
机器语言: B0 61
   ↓
汇编语言: MOV AL, 61h    ← "MOV"是move的缩写，意思是"搬过去"
```

汇编语言和机器语言是**一一对应**的——一行汇编对应一条机器指令。只是把难记的数字换成了好记的英文缩写。

常见的汇编指令：

| 汇编指令 | 含义 | 通俗说法 |
|----------|------|----------|
| MOV | 搬数据 | 把这个数搬过去 |
| ADD | 加法 | 加起来 |
| SUB | 减法 | 减一下 |
| CMP | 比较 | 比大小 |
| JMP | 无条件跳转 | 跳到那行去 |
| JE | 相等就跳转 | 一样就跳 |
| JNE | 不等就跳转 | 不一样就跳 |
| CALL | 调用函数 | 去执行那个函数 |
| RET | 返回 | 回来 |

### 高级语言：给人看的，不是给CPU看的

汇编还是太难写了。于是有了高级语言——Go、Python、Java、C++等。

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
// Go代码（你写的）
func add(a, b int) int {
    return a + b
}
```

这段Go代码经过**编译**之后，会变成一大串汇编指令，再变成机器语言。编译的过程是自动的，你只需要运行 `go build`。

```
Go代码 → [go编译器] → 汇编语言 → [汇编器] → 机器语言 → [链接器] → 可执行文件(.exe)
```

### 编译 vs 解释

- **编译型语言**（Go、C、C++、Rust）：代码一次性全部翻译成机器语言，然后直接运行。**快，但改代码要重新编译。**
- **解释型语言**（Python、JavaScript、Ruby）：代码一行一行地翻译执行。**修改方便，但慢。**

Go是编译型语言。你写的Go程序会变成一个可以直接运行的 `.exe`（Windows）或可执行文件（Linux/Mac）。这个文件里全是CPU能直接读懂的机器指令，不需要安装Go就能运行。

## 1.3 时钟频率是什么

### CPU的心跳

CPU内部有一个**时钟信号**（clock signal），它像节拍器一样，每隔固定时间发出一个脉冲。CPU每收到一个脉冲，就执行一步操作。

这个节拍器的速度就是**时钟频率**，单位是**Hz（赫兹）**：

- 1 Hz = 每秒1次
- 1 KHz = 每秒1千次
- 1 MHz = 每秒100万次
- 1 GHz = 每秒10亿次

你买的电脑上写的"3.0 GHz"意味着：这个CPU的节拍器每秒敲30亿次。

### 但频率不是一切

你可能会想：频率越高越好吧？3.0 GHz 肯定比 2.0 GHz 快？

**不一定。**

因为不同的CPU在一个时钟周期内能做的事情不同。假设有两台CPU：

- CPU-A：频率2.0 GHz，每个周期能执行4条指令（设计先进）
- CPU-B：频率3.0 GHz，每个周期只能执行1条指令（老旧设计）

CPU-A 每秒执行：2.0 × 10⁹ × 4 = 80亿条指令
CPU-B 每秒执行：3.0 × 10⁹ × 1 = 30亿条指令

CPU-A 虽然频率低，但实际更快！

这就是为什么不能只看频率。CPU的性能取决于：
- 频率（心跳多快）
- IPC（Instructions Per Cycle，每个周期执行多少指令）
- 架构设计（流水线、分支预测、缓存大小等）

### 为什么不能无限制提高频率

主要是两个原因：
1. **功耗**：频率越高，功耗越大，发热越多。效率不划算。
2. **发热**：频率高 = 发热大 = 需要更好的散热，否则CPU会烧坏。

所以现在的CPU追求的不再是"更高的频率"，而是"每个周期做更多的活"。

## 1.4 核心数是什么

### 单核 vs 多核

早期CPU只有一个**核心**（core），也就是只有一个"算术狂人"在干活。

现在的CPU有多个核心——相当于**多个算术狂人**同时干活。

- 双核 = 2个算术狂人
- 四核 = 4个算术狂人
- 八核 = 8个算术狂人

**多个核心可以真正地"同时"执行不同的任务**（这叫做并行，和并发的区别在第4章会详细讲）。

### 用一个厨房比喻

- **单核CPU** = 一个厨师，一份一份地做菜
- **双核CPU** = 两个厨师，同时做两道菜
- **八核CPU** = 八个厨师，同时做八道菜

但多核不等于速度直接翻倍，因为：
- 有些任务没法拆分（你必须先切菜才能炒菜）
- 厨师之间需要协调（"这瓶酱油谁在用？"）
- 有些任务本来就很简单，不需要多个厨师

### Go和多核

Go的并发模型（Goroutine）天然适合多核CPU。你可以启动很多Goroutine，Go的运行时会把它们分配到不同的CPU核心上并行执行：

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
for i := 0; i < 8; i++ {
    go func(n int) {
        // 这个Goroutine可能运行在第n个CPU核心上
        fmt.Println("我在核心", n)
    }(i)
}
```

### 超线程（Hyper-Threading）

超线程是Intel的技术，让一个物理核心**假装成两个逻辑核心**。

比喻：一个厨师两只手——右手炒菜，左手切菜。虽然还是一个人，但能同时做两件事。效率提升大约20%~30%，不是翻倍。

在任务管理器里，你可能会看到"4核8线程"——意思是4个物理核心，每个核心开超线程，操作系统看到8个逻辑核心。

## 1.5 CPU缓存（L1 / L2 / L3）

### 为什么需要缓存

CPU执行指令的速度极快（纳秒级），但**从内存里读数据很慢**（比CPU慢100倍）。

这就像——你是CPU（思考极快），但每次需要信息都要跑到楼下的图书馆查（内存访问极慢）。你会疯掉的。

所以你就在桌上放了书（L1缓存）、身后放了书架（L2缓存）、房间角落放了书堆（L3缓存）。

### 三级缓存结构

```
┌─────────────────────────────────────────┐
│                  CPU                     │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐│
│  │ 核心1│  │ 核心2│  │ 核心3│  │ 核心4││
│  │ L1 L2│  │ L1 L2│  │ L1 L2│  │ L1 L2││
│  └──────┘  └──────┘  └──────┘  └──────┘│
│  ┌──────────────────────────────────┐   │
│  │           L3 缓存（共享）          │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
          │
          │  内存总线
          ↓
┌──────────────────┐
│    内存（RAM）    │
└──────────────────┘
```

| 缓存级别 | 大小 | 速度 | 位置 | 比喻 |
|----------|------|------|------|------|
| L1 | 几十KB | ~1纳秒 | 每个核心独享 | 你桌上的便签 |
| L2 | 几百KB | ~5纳秒 | 每个核心独享 | 你身后的抽屉 |
| L3 | 几MB~几十MB | ~15纳秒 | 所有核心共享 | 房间角落的书堆 |
| 内存 | 几GB~几十GB | ~100纳秒 | 插在主板上 | 楼下的图书馆 |

### 缓存的工作方式

CPU不会等你告诉它"去缓存里找这个数据"——它自己会判断。当一个程序访问某个内存地址时：

1. CPU先查L1缓存 → 有就直接用（**缓存命中**，最快）
2. L1没有查L2
3. L2没有查L3
4. L3没有才去内存读（**缓存未命中**，很慢）

如果CPU刚读了内存地址 `0x1000`，它会**预判**你接下来可能要读 `0x1001`、`0x1002`（这叫空间局部性），提前把它们也装进缓存。

### 对你的代码有什么影响

如果你写了一个循环遍历一个大数组：

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
// 好：连续访问内存，缓存友好
for i := 0; i < len(arr); i++ {
    sum += arr[i]  // i连续递增，CPU会预读后续元素到缓存
}

// 差：跳跃访问内存，缓存不友好（如果arr很大）
for i := 0; i < len(arr); i += 1000 {
    sum += arr[i]  // 跳跃访问，缓存频繁未命中
}
```

> 这就是为什么我们说"尽量顺序访问数组"——不仅代码好写，对CPU缓存也友好。这属于性能优化的知识，你现在知道这个概念就行，以后深入了会遇到。

## 1.6 32位和64位

### "位"是什么意思

"位"（bit）指的是CPU一次能处理的数据宽度。你可以理解为CPU内部"传送带"的宽度。

- **32位CPU**：传送带宽度是32位（32条线），一次最多搬32个0或1
- **64位CPU**：传送带宽度是64位（64条线），一次最多搬64个0或1

### 实际影响

**最大的影响是内存寻址能力：**

一个32位CPU能直接"看到"多少内存？

32位意味着地址总线有32根线，每根线可以是0或1。所以能表示的地址总数是：

> 2³² = 4,294,967,296 个地址 ≈ **4GB**

这就是为什么32位Windows最多只能认4GB内存。

而64位CPU：

> 2⁶⁴ ≈ 180亿GB（天文数字，远超物理可能）

实际上现在64位CPU只用了48根地址线，也能支持256TB的内存寻址。

**其他差异：**
- 64位CPU能直接处理更大的整数（最大能表示2⁶⁴大约是1844亿亿）
- 64位程序的指针占8字节，32位程序指针占4字节（但这是内存消耗增加的唯一代价）

### 对Go开发的影响

当你用 `go build` 编译Go程序时，默认编译出和你当前系统匹配的版本。但你可以**交叉编译**：

```bash
# 编译64位Linux版本
GOOS=linux GOARCH=amd64 go build

# 编译64位Windows版本
GOOS=windows GOARCH=amd64 go build

# 编译64位Mac版本（在Windows上也能编译！）
GOOS=darwin GOARCH=amd64 go build

# 编译32位版本（极少需要）
GOARCH=386 go build
```

- `amd64` = 64位x86架构（AMD发明的，Intel也用了）
- `386` = 32位x86架构
- `arm64` = 64位ARM架构（苹果M系列芯片、树莓派、手机）

Go的交叉编译极其简单——不需要安装目标平台的环境，Chrome就是这么回事，直接在Windows上就能编译出Linux服务器上能跑的二进制文件。这是Go的一大卖点。
