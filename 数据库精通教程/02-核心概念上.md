## 我在做什么？

上一章你知道了"数据库是什么"和"为什么需要它"。这一章，我们钻进**关系型数据库**（Relational Database）*此术语见附录E* 的肚子里，看看它到底长什么样。

你只需要理解 4 个概念：**表、行、列、主键**。不多，但每一个都是地基——后面所有 SQL 操作都建立在这 4 个概念之上。

## 一、表（Table）：像 Excel 的 Sheet

想象你打开一个 Excel 文件，里面有好几个 Sheet：

- Sheet1 叫"用户"
- Sheet2 叫"订单"
- Sheet3 叫"商品"

在数据库里，每一个 Sheet 就是一张**表（Table）** *此术语见附录E*。表是数据的基本容器，同一类数据放在同一张表里。

文字描述太抽象，我们用一张**用户表**来直观感受：

```
┌────┬──────────┬─────────────────────┬──────────┐
│ id │ username │       email         │  phone   │
├────┼──────────┼─────────────────────┼──────────┤
│ 1  │ alice    │ alice@example.com   │ 1380001  │
│ 2  │ bob      │ bob@gmail.com       │ 1380002  │
│ 3  │ charlie  │ charlie@outlook.com │ 1380003  │
└────┴──────────┴─────────────────────┴──────────┘
```

这就是一张"用户表"（通常命名为 `users`）。它看起来和 Excel 一模一样——二维表格，有行有列。

> **想多一点**：为什么叫"关系型"数据库？因为数据之间可以建立**关系**。比如"用户表"和"订单表"之间通过"用户的 ID"关联起来——这就是"关系"。下一章讲外键时你会看到这是怎么实现的。

---

## 二、行（Row）和列（Column）

一张表由两部分组成：

### 列（Column）*此术语见附录E* = 字段

列是表的"骨架"——定义了这张表存什么类型的信息。比如用户表有 4 列：`id`、`username`、`email`、`phone`。每一列有一个**名字**和一个**数据类型**（比如整数、字符串、日期）。

列也叫**字段（Field）**，这两个词在数据库领域可以互换使用。

### 行（Row）*此术语见附录E* = 记录

行是表的"血肉"——每一行就是一条具体的数据。比如上面那行 `| 1 | alice | alice@example.com | 1380001 |` 就是一条用户记录。

行也叫**记录（Record）**，这两个词同样可以互换。

用图书馆比喻来记住：

| 比喻 | 数据库概念 | 你能摸到的东西 |
|------|-----------|--------------|
| 书架上贴的标签（"姓名""电话"） | 列（Column） | 定义了数据格式 |
| 书架上的一本书 | 行（Row） | 一条具体的数据 |
| 整个书架 | 表（Table） | 同一类书的集合 |

## 三、主键（Primary Key）：每行的身份证号

仔细看上面那张用户表——第一列 `id` 有什么特别？

- `id = 1` 是 alice
- `id = 2` 是 bob
- `id = 3` 是 charlie

`id` 就是这张表的**主键（Primary Key）** *此术语见附录E*。

### 主键的两条铁律

1. **不可重复**：同一张表里，不能有两行拥有相同的主键值。就像两个人的身份证号不会一样。
2. **不可为空**：每一行必须有一个主键值。不能有一个人没有身份证号。

### 为什么需要主键？

假设没有主键，你想"把 alice 的邮箱改成 alice_new@example.com"：

```sql
-- 没有主键的话，你只能这样写：
UPDATE users SET email = 'alice_new@example.com' WHERE username = 'alice';
```

但如果系统里有两个叫 alice 的人呢？你会把两个人的邮箱都改了！

有了主键：

```sql
-- 有主键，精确到某一行的某一个人：
UPDATE users SET email = 'alice_new@example.com' WHERE id = 1;
```

`id = 1` 永远只对应**那一个人**，不会搞错。

### 主键用什么做？

最常见的做法是**自增整数**：每插入一行，`id` 自动加 1（1, 2, 3, 4, ...）。简单、高效、不会重复。

也有人用 UUID（一长串随机字符，如 `a1b2c3d4-...`），但这是进阶话题，后面章节再讨论。

## 四、两张表的关系：用户表 → 订单表

现在，我们不止一张表了。假设你还管理着一个电商系统的**订单表**：

```
用户表 users                       订单表 orders
┌────┬──────────┬───────┐         ┌────┬─────────┬──────────┬────────┐
│ id │ username │ email │         │ id │ user_id │  product │ amount │
├────┼──────────┼───────┤         ├────┼─────────┼──────────┼────────┤
│ 1  │ alice    │  ...  │         │ 1  │    1    │  iPhone  │  6999  │
│ 2  │ bob      │  ...  │         │ 2  │    1    │  iPad    │  3499  │
│ 3  │ charlie  │  ...  │         │ 3  │    2    │  MacBook │ 12999  │
└────┴──────────┴───────┘         └────┴─────────┴──────────┴────────┘
```

注意看订单表的 `user_id` 列：

- 订单 1 的 `user_id = 1`，意思是"这个订单属于用户 1（alice）"
- 订单 2 的 `user_id = 1`，意思是"这个订单也属于用户 1（alice）"
- 订单 3 的 `user_id = 2`，意思是"这个订单属于用户 2（bob）"

我们用 ASCII 关系图来展示：

```
     users                          orders
    ┌──────┐                      ┌──────────┐
    │  id  │◄─────────────────────│ user_id  │
    │ name │       通过 user_id   │ product  │
    │ email│       关联到 users   │ amount   │
    └──────┘                      └──────────┘
```

`orders.user_id` 存的是 `users.id` 的值。这样，你想知道"订单 1 是谁下的"——用 `user_id = 1` 去 users 表里找 `id = 1`，就是 alice。

## 五、为什么订单表里不直接存用户全部信息？

你可能会想："干嘛搞得这么复杂？直接把用户名、邮箱、电话全塞进订单表不行吗？"

❌ 错误做法——把用户信息全塞进订单表：

```
订单表（臃肿版）
┌────┬──────────┬─────────────────────┬──────────┬──────────┬────────┐
│ id │ username │       email         │  phone   │ product  │ amount │
├────┼──────────┼─────────────────────┼──────────┼──────────┼────────┤
│ 1  │ alice    │ alice@example.com   │ 1380001  │ iPhone   │  6999  │
│ 2  │ alice    │ alice@example.com   │ 1380001  │ iPad     │  3499  │
│ 3  │ bob      │ bob@gmail.com       │ 1380002  │ MacBook  │ 12999  │
└────┴──────────┴─────────────────────┴──────────┴──────────┴────────┘
```

问题来了：

1. **数据冗余**：alice 的信息存了两遍。如果她有 100 个订单，就存 100 遍。
2. **更新噩梦**：alice 换了手机号，你要更新 100 行！万一漏了一行，数据就不一致了。
3. **存储浪费**：同样的信息重复存储，浪费磁盘空间。

✅ 正确做法——只存一个"引用"（`user_id`）：

```
订单表（精简版）
┌────┬─────────┬──────────┬────────┐
│ id │ user_id │ product  │ amount │
├────┼─────────┼──────────┼────────┤
│ 1  │    1    │ iPhone   │  6999  │
│ 2  │    1    │ iPad     │  3499  │
│ 3  │    2    │ MacBook  │ 12999  │
└────┴─────────┴──────────┴────────┘
```

alice 换了手机号？只在 `users` 表里改一行，所有订单**自动**指向更新后的信息——因为订单只存了 `user_id`，真正的用户信息在 `users` 表里。

> **想多一点**：这就是"关系"的本质——**用引用代替复制**。这和你写代码时"用变量引用对象而不是复制整个对象"是同一个道理。这种思想在数据库设计中叫"规范化"（Normalization），后面第 14 章会详细讲。

## 不对怎么办？

### 常见错误 1：混淆主键和普通列

❌ 错误认知：`username` 也可以当主键，反正用户名不会重复。

✅ 正确认知：`username` 不是主键的好选择，因为：
- 用户名可能会变（alice 想改成 alice_2024），主键一变，所有引用它的地方都要改。
- 主键最好用**纯数字且无业务含义**的值（如自增 ID），这样它永远不需要变。

**主键是用来"标识"的，不是用来"描述"的。** 身份证号只管"你是你"，不管"你叫什么名字"。

### 常见错误 2：认为表名必须用英文复数

❌ 错误认知：表名必须叫 `users`、`orders`、`products`。

✅ 正确认知：表名用单数还是复数是**风格选择**，不是技术规则。重要的是**保持一致**——要么全用单数（`user`、`order`），要么全用复数（`users`、`orders`），不要混着来。本教程统一使用复数形式（`users`、`orders`），因为这是 MySQL 社区的常见习惯。

### 常见错误 3：认为一个表只能有一个列

❌ 错误认知：我只需要一个"用户表"，里面存 id 和 name 就够了。

✅ 正确认知：列的数量取决于你需要存什么信息。一个用户表可以有 2 列，也可以有 20 列。关键是**每一列存一个独立的信息**（不要一列存"姓名+电话"这种混合数据）。

## 本章小结

| 本章学了什么 | 对应概念 | 注意事项 |
|-------------|---------|---------|
| 表（Table） | 二维表格，同一类数据的容器 | 数据库中通常有多张表，每张表存一类数据 |
| 行（Row）/ 记录 | 一条具体数据 | 行也叫记录（Record），两个词可互换 |
| 列（Column）/ 字段 | 数据的属性定义 | 列也叫字段（Field），每列有名称和数据类型 |
| 主键（Primary Key） | 每行的唯一标识，不可重复、不可为空 | 推荐用自增整数，无业务含义 |
| 表之间的关联 | 用 `user_id` 存 `users.id` 的值 | 存引用而不是复制数据，避免冗余 |
| 表名约定 | 复数形式（users、orders） | 单复数不重要，但必须保持一致 |
