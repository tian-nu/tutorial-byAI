## 我在做什么？

前面四章你在单机数据库上加了备份、主从、读写分离、分库分表。但有一个问题始终没彻底解决：

**主库挂了怎么办？**

这不只是"数据丢了怎么办"（第 25 章备份解决了），而是"服务停了怎么办"——主库宕机，所有写操作都失败，用户看到的是"系统繁忙，请稍后重试"。

**高可用（High Availability，HA）** 就是解决这个问题的：**主库挂了，自动切换到备用库，业务不中断，用户无感知。**

这一章我不会让你搭一堆环境，而是带你**俯瞰 MySQL 高可用方案的完整版图**，让你知道：
- 有哪些方案可选
- 每个方案适合什么场景
- 选型的时候看什么指标

学完这一章，你能画出一张"高可用方案选型决策树"，在任何项目里给出靠谱的数据库高可用方案。

## 一、高可用的核心指标

选方案之前，先搞清楚评价标准。别被人忽悠"这个方案很牛"，你要问：

### 1.1 RPO 和 RTO

两个数字决定了你的方案有多"高可用"：

| 指标 | 全称 | 含义 | 比喻 |
|------|------|------|------|
| **RPO** | Recovery Point Objective | 故障后能容忍丢失多少数据 | 你最多能接受丢多少分钟的数据？ |
| **RTO** | Recovery Time Objective | 故障后多久能恢复服务 | 你最多能接受服务中断多少分钟？ |

```
RPO = 0：零数据丢失，故障后数据完全一致
RPO = 5min：最多丢 5 分钟的数据

RTO = 0：自动切换，用户无感知
RTO = 5min：服务中断 5 分钟，人工介入恢复
```

**选型的核心权衡**：RPO 越小，方案越复杂，成本越高。不是所有业务都需要 RPO=0。

### 1.2 三个关键问题

评估一个高可用方案，回答这三个问题：

1. **主库挂了，从库多久能接管？**（RTO）
2. **接管后，数据丢了多少？**（RPO）
3. **切换过程需要人工介入吗？**（自动化程度）

## 二、CAP 理论：数据库的天花板

在讲具体方案之前，先理解一个底层理论——**CAP 理论** *此术语见附录E*。

### 2.1 CAP 说的是什么？

一个分布式系统不可能同时满足：

- **C（Consistency）一致性**：所有节点看到的数据都一样
- **A（Availability）可用性**：每个请求都能收到响应（不报错）
- **P（Partition Tolerance）分区容错性**：网络分区（节点间通信断了）时系统仍能工作

**你只能同时满足两个**。就像"便宜、快、好"你只能选两个一样。

### 2.2 MySQL 的 CAP 选择

| 数据库 | 默认选择 | 含义 |
|--------|---------|------|
| **MySQL 主从** | CP（一致性 + 分区容错） | 主库和从库数据一致，但网络分区时可能拒绝写入 |
| **MySQL MGR** | CP 或 AP（可配置） | 可以调，但默认 CP |
| **MongoDB** | CP | 和 MySQL 类似 |
| **Cassandra** | AP | 可用性优先，数据可能不一致 |

**MySQL 默认是 CP**：宁可拒绝写入，也不能让数据不一致。这是金融、电商等场景最需要的特性。

### 2.3 不纠结 CAP

你不需要深入理解 CAP 的证明过程。只需要知道一个结论：

> **网络分区一定会发生（交换机故障、网线松动、云服务故障），所以 P 必须选。剩下的就是在 C 和 A 之间权衡——MySQL 选 C，保证数据一致性。**

## 三、MySQL 高可用方案全景

以下是 MySQL 生态中四种主流方案，按复杂度从低到高排列。

### 3.1 MHA（Master High Availability）*此术语见附录E*

**最成熟的主从切换方案，社区用了十多年。**

```
┌─────────────────────────────────────────────────────┐
│                     MHA Manager                     │
│  (监控主库，主库挂了自动把从库提升为新主库)            │
└────────┬──────────────────────────────┬─────────────┘
         │                              │
    ┌────▼────┐                    ┌────▼────┐
    │  主库    │  ── 复制 ────►   │  从库    │
    │ Master  │                    │ Slave   │
    └─────────┘                    └─────────┘
```

**工作流程**：
1. MHA Manager 持续监控主库
2. 主库宕机 → MHA 选一个数据最新的从库
3. 从所有从库上收集差异 binlog，应用到新主库
4. 新主库生效，其他从库指向新主库
5. 切换耗时：约 30 秒

**特点**：
- ✅ 成熟稳定，社区使用广泛
- ✅ 切换速度快（30 秒级别）
- ❌ 不是 MySQL 官方出品（第三方）
- ❌ 需要额外部署 MHA Manager
- ❌ 可能丢数据（异步复制）

**适用场景**：中小规模、异步复制、能接受少量数据丢失的业务。

### 3.2 MGR（MySQL Group Replication）*此术语见附录E*

**MySQL 官方多主复制方案，5.7.17+ 版本内置。**

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  Node 1  │◄───►│  Node 2  │◄───►│  Node 3  │
│ (Primary)│     │(Secondary)│    │(Secondary)│
└──────────┘     └──────────┘     └──────────┘
     ▲                                  ▲
     └──── 多主或单主模式 ───────────────┘
```

**两种模式**：

| 模式 | 说明 | 特点 |
|------|------|------|
| 单主模式 | 只有一个节点能写，其他只读 | 和主从复制类似，但自动切换 |
| 多主模式 | 所有节点都能写 | 冲突检测复杂，性能差，不推荐 |

**特点**：
- ✅ MySQL 官方内置，无需第三方
- ✅ 自动故障检测和切换
- ✅ 基于 Paxos 协议，强一致性
- ❌ 要求组内节点数 ≥ 3（必须奇数）
- ❌ 网络延迟敏感（所有节点必须快速响应）
- ❌ 性能比主从稍差

**适用场景**：要求强一致性、对数据零丢失有要求的业务。

### 3.3 InnoDB Cluster *此术语见附录E*

**MGR + MySQL Router + MySQL Shell 的组合方案，官方一站式高可用。**

```
┌──────────────────────────────────────────────────────────┐
│                    InnoDB Cluster                        │
│                                                          │
│  ┌────────────────┐    ┌────────────────┐               │
│  │  MySQL Router  │    │  MySQL Router  │  (负载均衡)    │
│  └───────┬────────┘    └───────┬────────┘               │
│          │                     │                         │
│          └──────────┬──────────┘                         │
│                     ▼                                    │
│            ┌────────────────┐                            │
│            │  MGR 集群       │                            │
│            │  Node1 Node2    │  (自动故障切换)            │
│            │  Node3          │                            │
│            └────────────────┘                            │
│                     ▲                                    │
│            ┌────────────────┐                            │
│            │  MySQL Shell   │  (管理工具)                 │
│            └────────────────┘                            │
└──────────────────────────────────────────────────────────┘
```

**三个组件分工**：
- **MGR**：数据复制 + 自动故障切换
- **MySQL Router**：读写分离 + 负载均衡 + 自动路由到新主库
- **MySQL Shell**：管理工具（部署、配置、监控）

**特点**：
- ✅ MySQL 官方完整方案，全套支持
- ✅ 部署简单（MySQL Shell 一键部署）
- ✅ 自动故障切换 + 读写分离一站式解决
- ❌ 只支持 MySQL 8.0+
- ❌ MGR 的限制它都有

**适用场景**：MySQL 8.0+ 的新项目，追求官方支持的团队。

### 3.4 Orchestrator *此术语见附录E*

**GitHub 开源的自动化故障检测与切换工具。**

```
┌─────────────────────────────────────────────────────┐
│                  Orchestrator                        │
│  ┌──────────────────────────────────────────────┐   │
│  │  拓扑发现 → 持续监控 → 故障检测 → 自动恢复    │   │
│  └──────────────────────────────────────────────┘   │
│                       │                              │
│        ┌──────────────┼──────────────┐              │
│        ▼              ▼              ▼              │
│   ┌────────┐    ┌────────┐    ┌────────┐           │
│   │ Master │    │ Slave1 │    │ Slave2 │           │
│   └────────┘    └────────┘    └────────┘           │
└─────────────────────────────────────────────────────┘
```

**特点**：
- ✅ 可视化 Web 界面，拓扑一目了然
- ✅ 支持复杂的拓扑结构（多级主从、环形复制）
- ✅ 自动恢复（不只是切换，还能修复复制中断）
- ✅ 有 API 和 Hook，可以自定义行为
- ❌ 本身也需要高可用部署（不能单点）
- ❌ 学习成本稍高

**适用场景**：有大量 MySQL 实例需要管理的大型团队、复杂拓扑结构。

## 五、选型决策树

根据你的业务规模和需求，照着这张决策树选：

```
开始
 │
 ├─ 你的数据库多大？
 │    │
 │    ├─ < 50GB，单机够用
 │    │    └─ 能不能接受 30 分钟停机？
 │    │         ├─ 能 → 主从 + 手动切换（最简单）
 │    │         └─ 不能 → MHA（成熟稳定）
 │    │
 │    ├─ 50GB ~ 500GB，需要复制
 │    │    └─ 能不能接受丢几秒数据？
 │    │         ├─ 能 → 主从 + Orchestrator（自动故障切换）
 │    │         └─ 不能 → InnoDB Cluster（官方强一致）
 │    │
 │    └─ > 500GB，分布式
 │         └─ 团队有没有 DBA？
 │              ├─ 有 → InnoDB Cluster + 分库分表
 │              └─ 没有 → 上云数据库（RDS/Aurora，别自己折腾）
 │
 └─ 提示：如果是创业公司，MySQL 主从 + 备份 + 定期演练就够了。
    别一上来就搞 InnoDB Cluster，过度设计是最大的坑！
```

## 六、过度设计：最大的坑

我见过太多项目：日活 100 人的小系统，上来就搞 MGR + InnoDB Cluster + 分库分表。结果：

- 运维成本翻了 5 倍
- 出问题没人敢动
- 性能还不如单机（MGR 的同步开销）
- 招人要求"精通 InnoDB Cluster"，月薪 5 万没人来

**建议路线**：

```
阶段 1：单机 MySQL（日活 < 1000）
    ↓ 数据量增长
阶段 2：主从 + 备份（日活 < 10000）
    ↓ 读写压力大
阶段 3：读写分离 + 多从库（日活 < 100000）
    ↓ 单表数据量过大
阶段 4：分库分表（日活 > 100000）
    ↓ 高可用要求
阶段 5：InnoDB Cluster / MGR
```

**不要跳级！** 每个阶段都有它的合理性和适用场景。架构是演进的，不是一步到位的。

## 不对怎么办？

### 常见错误 1：过度设计——小项目上 MGR

现象：日活 100 的博客系统，搞了 MGR 三节点集群。

❌ 问题：运维成本远超收益，MGR 的同步开销反而拖慢性能。

✅ 解决：单机 MySQL + 主从 + 每天备份，足够。日活几千之前不需要考虑高可用。

### 常见错误 2：没考虑网络分区的影响

现象：MGR 集群部署在同一个机房的同一台交换机下，交换机故障 → 整个集群不可用。

✅ 解决：MGR 节点尽量分布在不同机架、不同机房，甚至不同可用区。

### 常见错误 3：只做了切换，没做切回

现象：主库故障后自动切换到从库，但一个月后没人记得切回去，新主库跑了一个月也没备份。

✅ 解决：故障切换后，新主库应该立即配置备份策略。旧主库修复后，不是自动接回去，而是作为从库加入。

### 常见错误 4：以为高可用 = 不用备份

现象：有了 MGR 三节点，"数据丢不了"，就不做备份了。

❌ 错误！MGR 保护的是**可用性**（服务不中断），不是**误操作**。如果有人 `DROP TABLE`，MGR 会忠实地在所有节点上执行 DROP！

✅ 解决：**高可用和备份是两回事，缺一不可。** 备份保护的是误操作和逻辑错误，高可用保护的是机器故障。

## 最可能出错的地方及原因

1. **过度设计**：小项目上 MGR，运维成本爆炸，原因是不了解各方案的适用场景，以为"越高级越好"。
2. **高可用和备份混淆**：以为有了 MGR 就不用备份，结果误删表后数据全丢，原因是没理解高可用保护的是"机器故障"而非"操作失误"。
3. **没考虑网络分区**：MGR 节点全放同一台交换机下，交换机故障导致集群不可用，原因是没理解 CAP 理论中 P 的含义。
4. **选型只看功能不看团队**：选了 Orchestrator 但团队没人会维护，原因是没考虑运维成本。
5. **架构跳级**：日活 100 就上分库分表，业务复杂度爆炸，原因是没遵循"架构演进"原则。
