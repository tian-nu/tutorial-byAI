# 第62章 · 锁机制

> "15个窗口同时办业务，大家都去翻同一个档案柜。如果没有排队机制，张三正在改某份档案的半页，李四伸手过来把剩下半页改了——结果这份档案上半页是老数据，下半页是新数据，谁还敢用？锁就是排队机制——'我在用这个柜子，你等我一分钟'。"

---

## 62.1 为什么需要锁

事务的隔离性（ACID中的I）需要锁来实现。MVCC解决了**读-写**冲突（读不阻塞写），但**写-写**冲突必须靠锁——两个事务同时修改同一行数据，必须有人先来后到。

```
没有锁的世界：
事务A：UPDATE users SET age = 30 WHERE id = 1;  -- 改了一半
事务B：UPDATE users SET age = 25 WHERE id = 1;  -- 也改！交错操作
结果：数据损坏

有锁的世界：
事务A：获取id=1的锁 → UPDATE → 等待COMMIT后才释放锁
事务B：尝试获取id=1的锁 → 等待... 等待... → 拿到锁 → UPDATE → 完成
```

## 62.2 锁的分类

### 62.2.1 按粒度：表锁 vs 行锁

| 类型 | 加锁范围 | 并发度 | 开销 | 使用引擎 |
|------|---------|--------|------|---------|
| **表锁** | 整张表 | 低 | 低 | MyISAM、InnoDB（特殊场景） |
| **行锁** | 单行 | 高 | 高 | InnoDB |
| **页锁** | 一页（多行） | 中 | 中 | BDB引擎（极少用） |

InnoDB**默认使用行锁**——只锁住被操作的那几行，不影响其他行的并发操作。

```sql
-- 只锁id=1这一行，其他行不受影响
UPDATE users SET age = 30 WHERE id = 1;
```

> ⚠️ 行锁依赖索引。如果WHERE条件列没有索引，InnoDB无法精确定位到行，会**锁住所有扫描过的行**（甚至退化为表锁）。这就是为什么UPDATE和DELETE的WHERE条件列必须有索引。

### 62.2.2 按模式：共享锁 vs 排他锁

| 类型 | 英文 | 简称 | 含义 |
|------|------|------|------|
| **共享锁** | Shared Lock | S锁 / 读锁 | 我可以读，你也可以读，但谁都不能写 |
| **排他锁** | Exclusive Lock | X锁 / 写锁 | 我可以读写，谁都不能碰 |

**兼容矩阵**：

|  | 共享锁(S) | 排他锁(X) |
|--|----------|----------|
| **共享锁(S)** | ✅ 兼容 | ❌ 冲突 |
| **排他锁(X)** | ❌ 冲突 | ❌ 冲突 |

翻译成人话：S锁可以和S锁共存（多人同时读），X锁跟任何锁都不兼容（写的时候谁也别碰）。

### 手动加锁

```sql
-- 加共享锁（读锁）
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;
-- MySQL 8.0.22+ 推荐写法：
SELECT * FROM users WHERE id = 1 FOR SHARE;

-- 加排他锁（写锁）
SELECT * FROM users WHERE id = 1 FOR UPDATE;
```

> `SELECT ... FOR UPDATE` 是面试和工作中最常见的显式锁操作。它会在匹配的行上加排他锁，阻止其他事务修改这些行，直到当前事务提交。

## 62.3 InnoDB的三种行锁

InnoDB的行锁实际上有三种实现：

### 62.3.1 Record Lock（记录锁）

锁住**索引记录本身**。

```
索引B+树：
       [10] [20] [30] [40] [50]
                      ████
                   锁住 id=30
```

```sql
-- 对id=30这行加记录锁
SELECT * FROM users WHERE id = 30 FOR UPDATE;
```

### 62.3.2 Gap Lock（间隙锁）

锁住**索引记录之间的间隙**，防止其他事务在这个间隙中INSERT。

```
索引B+树：
       [10] → 间隙1 → [20] → 间隙2 → [30] → 间隙3 → [40]
                     ████████████████████
                     锁住间隙2（20~30之间）
```

```sql
-- 对 id>20 且 id<30 这个间隙加锁
SELECT * FROM users WHERE id BETWEEN 20 AND 30 FOR UPDATE;
```

这就是InnoDB**在REPEATABLE READ下解决幻读**的关键。间隙锁阻止了其他事务在查询范围内插入新行。

### 62.3.3 Next-Key Lock（临键锁）

**Record Lock + Gap Lock的组合**。锁住一条记录**和**它前面的间隙。

```
索引B+树：
       [10] → 间隙1 → [20] → 间隙2 → [30] → 间隙3 → [40]
       █████████████████████████████████████
       间隙1 + 记录20 + 间隙2 都被锁
```

这是InnoDB在REPEATABLE READ隔离级别下的**默认行锁算法**。

```sql
-- 以下查询在REPEATABLE READ下会对扫描到的行加Next-Key Lock
SELECT * FROM users WHERE age = 25 FOR UPDATE;
-- 如果age没有唯一索引，会锁住所有匹配行的记录+间隙
```

> 📌 Next-Key Lock是MySQL在REPEATABLE READ下同时解决不可重复读和幻读的武器。

## 62.4 死锁

### 62.4.1 什么是死锁

两个事务互相持有对方需要的锁，形成循环等待：

```
事务A：                           事务B：
LOCK id=1 (成功)                  LOCK id=2 (成功)
LOCK id=2 (等待B释放...)    →    LOCK id=1 (等待A释放...)

        A等B，B等A → 死锁！
```

```sql
-- 事务A
START TRANSACTION;
UPDATE users SET age = 30 WHERE id = 1;  -- 获取id=1的排他锁
-- 暂停，不提交

-- 事务B（同时执行）
START TRANSACTION;
UPDATE users SET age = 25 WHERE id = 2;  -- 获取id=2的排他锁
UPDATE users SET age = 25 WHERE id = 1;  -- 等待id=1的锁...

-- 回到事务A
UPDATE users SET age = 30 WHERE id = 2;  -- 等待id=2的锁...
-- ERROR 1213: Deadlock found when trying to get lock
```

MySQL检测到死锁后，会自动**回滚其中一个事务**（持有最少行锁的那个），另一个事务继续执行。

### 62.4.2 查看死锁日志

```sql
-- 查看最近一次死锁的详细信息
SHOW ENGINE INNODB STATUS\G
```

在输出中搜索 `LATEST DETECTED DEADLOCK` 部分。

### 62.4.3 避免死锁的实践

| 原则 | 做法 |
|------|------|
| **统一访问顺序** | 所有事务按相同顺序访问表和行（先锁id=1再锁id=2，不要反过来） |
| **缩短事务** | 事务越小越好，不要把耗时操作（发邮件、调接口）放在事务里 |
| **使用索引** | 没有索引的行锁退化为表锁，更容易死锁 |
| **降低隔离级别** | 如果可以接受，用READ COMMITTED代替REPEATABLE READ，后者有间隙锁更容易死锁 |
| **批量操作分批** | 一次锁太多行也容易死锁，分批操作 |

## 62.5 乐观锁与悲观锁

这是两种**并发控制思想**，而不只是数据库层面的锁。

### 62.5.1 悲观锁

**"肯定有人跟我抢，我先锁上再说。"**

```java
// 悲观锁：SELECT ... FOR UPDATE
@Transactional
public void deductStock(Long productId, int quantity) {
    // 查询商品库存并加排他锁
    Product product = mapper.selectForUpdate(productId);  // SELECT ... FOR UPDATE
    if (product.getStock() < quantity) {
        throw new BusinessException("库存不足");
    }
    mapper.deductStock(productId, quantity);
}
```

适用：写多读少、冲突概率高的场景（如秒杀扣库存）。

### 62.5.2 乐观锁

**"应该没人跟我抢，更新时检查一下版本就行。"**

用**版本号（version字段）**或**时间戳**实现：

```sql
-- 先查出版本号
SELECT id, stock, version FROM products WHERE id = 1;
-- 假设 stock=10, version=5

-- 更新时检查版本号是否未变
UPDATE products
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 5;
-- 如果affected rows = 0，说明版本号已经变了（被别人改过），重试
```

```java
// 乐观锁实现
public void deductStock(Long productId, int quantity) {
    while (true) {
        Product product = mapper.selectById(productId);
        int updated = mapper.deductStockWithVersion(
            productId, quantity, product.getVersion()
        );
        if (updated > 0) {
            break;  // 更新成功
        }
        // 更新失败（被其他事务改了），重试
    }
}
```

| 对比 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| 实现 | `SELECT ... FOR UPDATE` | version字段 + CAS更新 |
| 冲突处理 | 等待/阻塞 | 重试 |
| 适用场景 | 冲突多 | 冲突少 |
| 性能 | 等待时浪费连接资源 | 重试浪费CPU |
| Java生态 | JPA的 `@Lock(LockModeType.PESSIMISTIC_WRITE)` | JPA的 `@Version` |

> 💡 Spring Data JPA 内置了对乐观锁的支持，只需在实体类中加一个 `@Version` 标注的字段即可。

## 62.6 锁监控

```sql
-- 查看当前所有锁信息（MySQL 8.0）
SELECT * FROM performance_schema.data_locks;

-- 查看当前所有锁等待
SELECT * FROM performance_schema.data_lock_waits;

-- 查看当前所有事务
SELECT * FROM information_schema.INNODB_TRX;

-- 查看锁等待超时时间（默认50秒）
SHOW VARIABLES LIKE 'innodb_lock_wait_timeout';
```

如果某个事务长时间不提交（比如忘了COMMIT的查询窗口），可以通过以下方式强制终止：

```sql
-- 找到线程ID
SHOW PROCESSLIST;

-- 杀掉指定线程
KILL 123;  -- 123是线程ID
```

## 本章小结

| 概念 | 要点 |
|------|------|
| 表锁 vs 行锁 | InnoDB默认行锁，没索引时退化为表锁 |
| 共享锁(S) | 可多人同时读，不可写 |
| 排他锁(X) | 独享，其他人不能读也不能写 |
| Record Lock | 锁索引记录本身 |
| Gap Lock | 锁索引间隙，防止INSERT（解决幻读） |
| Next-Key Lock | Record+Gap，InnoDB默认行锁算法 |
| 死锁 | 两个事务循环等待对方的锁，MySQL自动检测并回滚一方 |
| 悲观锁 | 先锁再用，适合冲突多的场景 |
| 乐观锁 | 用版本号检查，冲突时重试，适合冲突少的场景 |

## 自测题

1. **什么是死锁？写出一个能产生死锁的SQL序列（两个事务）。**

2. **乐观锁和悲观锁的区别是什么？电商秒杀场景中扣减库存用哪种锁好？为什么？**

3. **InnoDB的行锁依赖索引。如果一条UPDATE语句WHERE条件列没有索引，会发生什么？**

<details>
<summary>参考答案（做完再看）</summary>

1. 死锁是两个或多个事务相互等待对方持有的锁，形成循环等待。例子：
   事务A：`UPDATE t SET v=1 WHERE id=1;`（获得id=1的锁）
   事务B：`UPDATE t SET v=2 WHERE id=2;`（获得id=2的锁）
   事务A：`UPDATE t SET v=1 WHERE id=2;`（等待id=2的锁）
   事务B：`UPDATE t SET v=2 WHERE id=1;`（等待id=1的锁）→ 死锁

2. 悲观锁是"先锁再用"（SELECT ... FOR UPDATE），乐观锁是"先查版本号，更新时验证版本号未变"。秒杀场景用乐观锁更好：因为锁冲突只在更新库存那一步，而且CAS重试比数据库行锁阻塞更轻量。如果用悲观锁，大量请求会在 `SELECT ... FOR UPDATE` 处排队等待，连接池瞬间耗尽。实际生产中，Redis+乐观锁+消息队列是经典秒杀方案。

3. 如果WHERE条件的列没有索引，InnoDB无法精确定位到具体行，会锁住**所有扫描到的行**。在REPEATABLE READ下甚至会产生间隙锁锁住大段范围，严重时相当于表锁——其他事务对该表的写操作全部被阻塞。这就是为什么DBA反复强调：WHERE和JOIN条件列必须建索引，尤其是UPDATE和DELETE语句。
</details>
