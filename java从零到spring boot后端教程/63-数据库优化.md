# 第63章 · 数据库优化

> "车能开了，但上高速抖得像拖拉机。数据库也一样——功能实现了不等于性能达标。当用户投诉'页面加载要5秒'时，你需要的不是加内存，而是一套系统的性能排查和优化的方法论。这章就是你的工具箱——从找到慢SQL到干掉它，每一步都有据可查。"

## 63.1 优化的第一步：找到慢SQL

优化最重要的原则：**不要猜测，用数据说话**。你直觉上觉得慢的那个查询，可能只占了1%的负载；真正吃资源的那个查询你可能根本没注意到。

### 63.1.1 开启慢查询日志

```sql
-- 查看慢查询日志是否开启
SHOW VARIABLES LIKE 'slow_query_log%';

-- 查看慢查询的时间阈值（默认10秒）
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询日志（开发环境）
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过1秒就记录
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
```

> ⚠️ 生产环境建议 `long_query_time = 0.5`（500毫秒）。10秒的默认值太宽松了，等你发现时用户已经跑光了。

### 63.1.2 分析慢查询日志

手动看日志文件太累了，用 **mysqldumpslow** 工具自动分析：

```bash
# 按查询时间排序，显示执行最慢的前10条
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# 按出现次数排序，显示最频繁的前10条
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

# 输出示例：
# Count: 3523  Time=2.15s (7578s)  Lock=0.00s  Rows=50000.0
# SELECT * FROM users WHERE username LIKE 'S'
```

### 63.1.3 performance_schema（推荐）

MySQL 5.7+自带的性能监控库，比慢查询日志更强大：

```sql
-- 查看哪些SQL语句执行次数最多
SELECT
    DIGEST_TEXT,
    COUNT_STAR AS exec_count,
    AVG_TIMER_WAIT / 1000000000 AS avg_ms,
    SUM_TIMER_WAIT / 1000000000 AS total_ms
FROM performance_schema.events_statements_summary_by_digest
ORDER BY total_ms DESC
LIMIT 10;
```

## 63.2 SQL优化十条军规

### 1. SELECT 只取需要的列

```sql
-- ❌ 差
SELECT * FROM users WHERE id = 1;

-- ✅ 好
SELECT id, username, email FROM users WHERE id = 1;
```

理由：减少网络传输，增加覆盖索引的机会。

### 2. 大批量INSERT使用批量语法

```sql
-- ❌ 差：1000条数据执行1000次
INSERT INTO logs (msg) VALUES ('msg1');
INSERT INTO logs (msg) VALUES ('msg2');
...

-- ✅ 好：一次插入
INSERT INTO logs (msg) VALUES
    ('msg1'), ('msg2'), ..., ('msg1000');
```

理由：减少网络往返和事务开销，性能差距可达10-50倍。

### 3. 用EXPLAIN检查，建立索引

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 1 ORDER BY created_at DESC LIMIT 10;

-- 如果 type=ALL → 建索引
CREATE INDEX idx_orders_user_created ON orders(user_id, created_at);
```

### 4. 用IN代替OR（当OR的列有索引时）

```sql
-- ❌ 差
SELECT * FROM users WHERE id = 1 OR id = 2 OR id = 3;

-- ✅ 好
SELECT * FROM users WHERE id IN (1, 2, 3);
```

### 5. 分页不要OFFSET太大

```sql
-- ❌ 慢：OFFSET 100000，MySQL要扫描并丢弃前10万行
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 100000;

-- ✅ 快：记住上一页最大ID
SELECT * FROM users WHERE id > 100000 ORDER BY id LIMIT 10;
```

### 6. WHERE条件中避免对列做函数/运算

```sql
-- ❌ 索引失效
SELECT * FROM users WHERE YEAR(created_at) = 2026;
SELECT * FROM products WHERE price + 10 > 100;

-- ✅
SELECT * FROM users WHERE created_at >= '2026-01-01' AND created_at < '2027-01-01';
SELECT * FROM products WHERE price > 90;
```

### 7. JOIN小表驱动大表，ON条件列有索引

```sql
-- 确保 orders.user_id 和 orders.product_id 有索引
SELECT u.name, p.name
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN products p ON o.product_id = p.id;
```

### 8. 能用UNION ALL就别用UNION

UNION会去重（隐含排序+DISTINCT），UNION ALL不去重，速度快很多。

### 9. 数值字段避免用字符串比较

```sql
-- ❌ 如果 status_code 是 INT 类型
SELECT * FROM orders WHERE status_code = '1';

-- ✅
SELECT * FROM orders WHERE status_code = 1;
```

### 10. COUNT(*) vs COUNT(col)

```sql
-- ✅ 统计总行数用 COUNT(*)，InnoDB已优化
SELECT COUNT(*) FROM users;

-- 统计某列非NULL行数用 COUNT(col)
SELECT COUNT(email) FROM users;
```

## 63.3 表结构优化

### 63.3.1 字段类型选择

| 场景 | 推荐类型 | 不要用 |
|------|---------|--------|
| 状态码 | TINYINT | VARCHAR |
| 年龄 | TINYINT UNSIGNED | INT |
| 金额 | DECIMAL(10,2) | FLOAT/DOUBLE |
| 手机号 | CHAR(11) | VARCHAR(20) |
| 定长唯一ID（如订单号） | CHAR(N) | VARCHAR(N) |
| IP地址 | INT UNSIGNED（4字节） | VARCHAR(15) |
| 真/假 | TINYINT(1) | VARCHAR(5) 存 "true"/"false" |

### 63.3.2 垂直拆分

把表中**不常用的大字段**拆分到另一张表：

```sql
-- 原表
CREATE TABLE articles (
    id      INT PRIMARY KEY,
    title   VARCHAR(200),
    content LONGTEXT,     -- 很少用，但占了大量空间
    created_at DATETIME
);

-- 拆分后
CREATE TABLE articles (
    id      INT PRIMARY KEY,
    title   VARCHAR(200),
    created_at DATETIME
);

CREATE TABLE article_contents (
    article_id INT PRIMARY KEY,
    content    LONGTEXT
);
```

好处：主表每行体积小，一个数据页能存更多行，缓存命中率更高。

### 63.3.3 避免NULL

NULL在索引、比较、聚合中都是麻烦制造者。能用NOT NULL DEFAULT的，就别留NULL。

```sql
-- ✅ 好
ALTER TABLE users MODIFY age TINYINT UNSIGNED NOT NULL DEFAULT 0;
```

## 63.4 架构层优化

### 63.4.1 读写分离

```
         ┌──────────┐
         │ 应用程序   │
         └─┬──────┬─┘
     写操作 │      │ 读操作
           ▼      ▼
     ┌─────────┐ ┌─────────┐ ┌─────────┐
     │ Master  │→│ Slave 1 │ │ Slave 2 │
     │ (主库)   │ │ (从库)   │ │ (从库)   │
     └─────────┘ └─────────┘ └─────────┘
       写入      复制  读取   读取
```

- 主库负责写操作，从库负责读操作
- 主从之间通过binlog异步复制
- 适合**读多写少**的场景

### 63.4.2 分库分表

当单表数据量超过**2000万行**（经验值，取决于行大小和硬件），考虑分表：

**水平分表**：按行拆分

```
users 表（5000万行）
      ↓ 按 user_id % 4 拆分
users_0 (id % 4 = 0)
users_1 (id % 4 = 1)
users_2 (id % 4 = 2)
users_3 (id % 4 = 3)
```

**垂直分库**：按业务拆分

```
单体数据库
      ↓
用户库（users）
订单库（orders）
商品库（products）
```

分库分表的复杂性在于：跨库JOIN不可用、分布式事务、全局ID生成等。**不到迫不得已不要分**。优先考虑索引优化、缓存、读写分离。

> 💡 Spring Boot中可用 **ShardingSphere**（原Sharding-JDBC）来实现分库分表，应用代码几乎不用改。

### 63.4.3 缓存

在应用层加Redis缓存，是性价比最高的优化手段。第68-70章会详细展开。

```
请求 → Redis（查缓存）→ 命中：返回（1ms）
                        → 未命中：查MySQL → 写回Redis → 返回（50ms）
```

## 63.5 参数调优速查

| 参数 | 建议值 | 说明 |
|------|--------|------|
| `innodb_buffer_pool_size` | 物理内存的60%~70% | InnoDB缓存池，最重要的性能参数 |
| `innodb_log_file_size` | 512M~2G | redo log文件大小，影响写入性能 |
| `innodb_flush_log_at_trx_commit` | 1（安全）/ 2（性能） | 1=每次提交刷盘，2=每秒刷盘 |
| `max_connections` | 500~1000 | 最大连接数，根据业务调整 |
| `query_cache_size` | **0**（MySQL 8.0 已移除） | 查询缓存已废弃，用Redis替代 |

```sql
-- 查看当前配置
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';

-- 动态调整（重启后失效，永久改需写入my.cnf）
SET GLOBAL innodb_buffer_pool_size = 4294967296;  -- 4GB
```

> 🚨 修改 `my.cnf` 中的参数后需要重启MySQL。`SET GLOBAL` 修改的参数重启后会丢失，所以调试时用 `SET GLOBAL` 测试，确认后在 `my.cnf` 中永久写入。

## 63.6 优化实战流程总结

```
发现问题（用户反馈慢/监控告警）
        │
        ▼
  找到慢SQL（慢查询日志 / performance_schema）
        │
        ▼
  EXPLAIN分析（看type/rows/Extra）
        │
   ┌────┴────┐
   ▼         ▼
 加索引    改写SQL
   │         │
   └────┬────┘
        ▼
  验证效果（EXPLAIN + 实际执行时间对比）
        │
        ▼
  还不够？→ 缓存（Redis）
        │
        ▼
  还不够？→ 读写分离 / 分库分表
```

## 本章小结

| 概念 | 要点 |
|------|------|
| 慢查询日志 | 记录执行超过阈值的SQL，`long_query_time` 建议设0.5秒 |
| mysqldumpslow | 慢查询日志分析工具 |
| SQL优化 | SELECT指定列、批量INSERT、EXPLAIN、避免OFFSET、避免函数索引失效 |
| 表结构 | 选对类型、垂直拆分大字段、用NOT NULL |
| 读写分离 | 主库写+从库读，基于binlog复制，适合读多写少 |
| 分库分表 | 水平按行拆分（取模），垂直按业务拆分，不到万不得已不分 |
| 缓存 | 应用层加Redis，性价比最高的优化 |
| innodb_buffer_pool_size | 最重要的MySQL配置参数，设为内存的60%-70% |

## 自测题

1. **慢查询日志发现一条SQL：`SELECT * FROM orders WHERE DATE(created_at) = '2026-05-29';` 平均耗时3秒。如何优化？**

2. **MySQL的 `innodb_buffer_pool_size` 参数是做什么的？为什么把它调大能显著提升性能？**

3. **什么是读写分离？它的工作原理是什么？适合什么场景？**

<details>
<summary>参考答案（做完再看）</summary>

1. 问题在于 `DATE(created_at)` 对索引列使用了函数，导致索引失效。优化为范围查询：`WHERE created_at >= '2026-05-29' AND created_at < '2026-05-30'`，并在 `created_at` 列上建索引。优化后 type 将从 ALL 变为 range。

2. `innodb_buffer_pool_size` 是InnoDB缓冲池大小，用于缓存数据和索引页。MySQL的数据是存在磁盘上的，但读写最快的介质是内存。缓冲池就是内存中的缓存区——数据被读到缓冲池后，后续读和写都先在内存中操作，减少磁盘IO。一般设为物理内存的60%~70%（给操作系统和其他进程留足够空间）。比如32GB内存的服务器，设为20GB左右。

3. 读写分离是将数据库的读和写操作分离到不同的数据库实例上：主库（Master）处理所有写操作，从库（Slave）处理所有读操作。原理：主库将写操作记录到binlog中，从库通过IO线程读取主库的binlog并回放（relay log），实现数据同步。适合**读多写少**的场景（如电商的商品浏览、新闻的列表页），因为读请求可以分摊到多个从库。不适合写密集型场景，且存在主从延迟的问题（刚写入主库的数据，立即去从库读可能读不到最新数据）。
</details>
