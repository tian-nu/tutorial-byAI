## 我在做什么？

你写了一个 Web 应用，用 Docker 跑起来，一切正常。但你有没有想过：这个容器里的进程，是以什么身份在运行？答案是 **root**。默认情况下，Docker 容器里的进程以 root 身份运行。如果攻击者通过你的 Web 应用拿到了一个 shell，他就在容器里拥有 root 权限。更糟的是，如果容器配置不当，他甚至可能逃逸到宿主机上。

"容器是隔离的，所以很安全"——这是 Docker 安全领域最大的误解。容器提供的隔离远不如虚拟机。这一章教你缩小容器的攻击面：用非 root 用户运行、扫描镜像漏洞、用最小基础镜像、保护密钥、设置只读文件系统。学完这些，你的容器从"筛子"变成"堡垒"。

学完这一章，你能：
- 理解容器默认 root 运行的风险
- 用 `USER` 指令创建非 root 用户运行应用
- 用 `docker scout` 扫描镜像 CVE 漏洞
- 选择最小基础镜像（alpine、distroless、scratch）
- 用 `--read-only` 保护文件系统
- 理解密钥管理的最佳实践（不用 ENV 传密码）

## 一、容器安全的第一课：你不是 root 吗？

### 默认 root 运行

```bash
# 启动一个最普通的容器
docker run -it alpine:latest sh

# 看看你是谁
whoami
# root
```

容器里的 root 和宿主机的 root 不是完全相同的——Docker 通过 Linux namespace 和 cgroup 做了隔离。但**这个隔离不是绝对的**。历史上出现过多次容器逃逸漏洞（如 CVE-2019-5736、CVE-2024-21626），让容器内的 root 可以操作宿主机文件系统。

### 风险有多大？

```bash
# 如果容器以 root 运行，且有不当的挂载
docker run -it --privileged -v /:/host alpine:latest sh
# 在容器里
whoami
# root
chroot /host
# 现在你在宿主机的根文件系统里了！
# 可以修改任何文件，包括 /etc/shadow
```

这是极端情况（`--privileged` + 挂载根目录），但说明了一个道理：**root 权限 + 宽松配置 = 宿主机沦陷的潜在风险。**

### 更常见的风险

```bash
# 一个 Web 应用容器以 root 运行
# 如果应用有漏洞（如命令注入），攻击者拿到 shell
# 攻击者可以：
# 1. 安装后门软件（apt-get install netcat）
# 2. 修改系统配置
# 3. 利用容器内的 root 权限尝试逃逸
# 4. 扫描内网其他服务
```

> **想多一点**：容器安全不是在"安全"和"不安全"之间二选一，而是在"攻击面"的连续谱上做减法。每一个安全措施——非 root 用户、最小镜像、只读文件系统、密钥管理——都是在缩小攻击面。一个攻击者在拿到 shell 后，发现自己是 `nobody` 用户、文件系统是只读的、没有 curl 没有 wget 没有 nc——他的攻击路径就被大幅压缩了。安全不是一堵墙，而是一层又一层的洋葱皮，每剥一层都让攻击者更痛苦。

## 二、`USER` 指令：不要用 root 跑应用

### 基本用法

```dockerfile
# ❌ 默认 root 运行
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm ci
EXPOSE 3000
CMD ["node", "server.js"]
# 这个 CMD 以 root 身份执行
```

```dockerfile
# ✅ 创建非 root 用户运行
FROM node:20-alpine

# 创建用户和用户组
RUN addgroup -g 1001 -S appgroup && \
    adduser -u 1001 -S appuser -G appgroup

WORKDIR /app
COPY --chown=appuser:appgroup . .
RUN npm ci

USER appuser          # 从此以后，所有命令以 appuser 身份运行
EXPOSE 3000
CMD ["node", "server.js"]
```

### 逐行解释

```dockerfile
RUN addgroup -g 1001 -S appgroup && \
    adduser -u 1001 -S appuser -G appgroup
```
- `-g 1001`：指定 GID 为 1001（固定值，避免不同系统冲突）
- `-S`：创建系统用户（无登录 shell、无 home 目录）
- `-u 1001`：指定 UID 为 1001
- `-G appgroup`：将用户加入 appgroup 组

```dockerfile
COPY --chown=appuser:appgroup . .
```
- `--chown`：复制文件时顺便改所有权，确保 appuser 有权限读

```dockerfile
USER appuser
```
- 从这一行开始，`RUN`、`CMD`、`ENTRYPOINT` 都以 `appuser` 身份执行

### 验证

```bash
docker build -t secure-app .
docker run --rm secure-app whoami
# appuser  ← 不是 root！

docker run --rm secure-app id
# uid=1001(appuser) gid=1001(appgroup)
```

### Debian/Ubuntu 镜像的写法

```dockerfile
FROM node:20
# debian 下的用户创建命令不同
RUN groupadd -r appgroup && \
    useradd -r -g appgroup -u 1001 appuser
USER appuser
```

### 端口问题：非 root 用户不能绑定 1024 以下的端口

```dockerfile
# ❌ 非 root 用户不能绑定 80 端口
USER appuser
EXPOSE 80
CMD ["node", "server.js"]
# 如果 server.js 监听 80 端口，会报 EACCES

# ✅ 方案一：监听高端口（>1024）
EXPOSE 3000
CMD ["node", "server.js"]

# ✅ 方案二：用 nginx 等反向代理在外部做端口转换
# compose.yml 里做端口映射：80:3000
```

## 三、不要暴露 Docker 守护进程端口

### 危险操作

```bash
# ❌ 危险：暴露 Docker Daemon 的 TCP 端口
dockerd -H tcp://0.0.0.0:2375 -H unix:///var/run/docker.sock
# 任何能连上 2375 端口的人，等于拥有了 root 权限
```

如果 Docker Daemon 的 TCP 端口（2375）暴露在公网上，攻击者可以：

```bash
# 攻击者从远程执行
docker -H tcp://your-server:2375 run -it --privileged -v /:/host alpine chroot /host
# 完全控制你的服务器
```

### 如果必须远程访问，用 TLS

```bash
# ✅ 启用 TLS 加密
dockerd --tlsverify \
  --tlscacert=/path/to/ca.pem \
  --tlscert=/path/to/server-cert.pem \
  --tlskey=/path/to/server-key.pem \
  -H tcp://0.0.0.0:2376
```

> **2375 是不加密端口，2376 是 TLS 加密端口。** 永远不要在没有 TLS 的情况下暴露 Docker Daemon。

## 四、镜像扫描：发现已知漏洞

### 用 `docker scout` 扫描

```bash
# 扫描本地镜像
docker scout quickview nginx:latest
# 会列出镜像的 CVE 漏洞数量和严重程度

# 详细扫描
docker scout cves nginx:latest
# 按严重程度列出每个 CVE，包括修复建议

# 对比两个镜像
docker scout compare nginx:1.25 nginx:1.27
```

### 用 Trivy 扫描（更详细）

```bash
# 安装 Trivy
# Linux: curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh
# macOS: brew install trivy
# Windows: choco install trivy

# 扫描镜像
trivy image nginx:latest
# 列出所有 CVE，按严重程度（CRITICAL、HIGH、MEDIUM、LOW）分类

# 只显示严重和高危漏洞
trivy image --severity CRITICAL,HIGH nginx:latest

# 扫描镜像的 Dockerfile 配置问题
trivy config ./Dockerfile
```

### 扫描结果示例

```
nginx:latest (alpine 3.19.1)
Total: 0 (CRITICAL: 0, HIGH: 0, MEDIUM: 0, LOW: 0)

→ 很好，没有已知漏洞！

node:20 (debian 12.5)
Total: 128 (CRITICAL: 2, HIGH: 15, MEDIUM: 45, LOW: 66)

→ 有 2 个严重漏洞！考虑升级到 node:20-alpine
```

### 扫描应该成为 CI/CD 的一部分

```yaml
# GitHub Actions 示例
- name: Scan Docker image
  run: |
    docker scout cves my-image:latest --format sarif --output scout-results.sarif
```

## 五、最小基础镜像：缩小攻击面

### 镜像大小 = 攻击面

```
镜像越大 → 包含的软件越多 → 潜在漏洞越多 → 攻击面越大
```

### 镜像选择梯度

| 镜像 | 大小 | 内容 | 适用场景 |
|------|------|------|---------|
| `ubuntu:latest` | ~77MB | 完整 Ubuntu 系统 | 几乎不用，太重 |
| `debian:bookworm-slim` | ~74MB | 精简版 Debian | 需要 apt 安装依赖 |
| `node:20` | ~1.1GB | Node.js + 完整 Debian | 开发环境 |
| `node:20-alpine` | ~130MB | Node.js + Alpine | 生产环境推荐 |
| `golang:1.22-alpine` | ~350MB | Go SDK + Alpine | 构建阶段 |
| `alpine:latest` | ~7MB | 最小 Linux | 运行静态二进制 |
| `distroless` | ~2MB | 只有你的程序 + 必要库 | 安全极致 |
| `scratch` | 0B | 什么都没有 | 纯静态二进制 |

### 实战：用 distroless

distroless 是 Google 维护的最小镜像，只包含你的程序和必要的运行时库，**没有 shell、没有包管理器、没有 curl**。

```dockerfile
# ===== 第一阶段：编译 =====
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o server .

# ===== 第二阶段：用 distroless =====
FROM gcr.io/distroless/static-debian12:nonroot
WORKDIR /app
COPY --from=builder /app/server .
EXPOSE 8080
CMD ["./server"]
```

什么是 `nonroot`？distroless 的 `nonroot` 变体默认以 `nonroot` 用户（UID 65532）运行，你不需要手动 `USER`。

### 实战：用 scratch（极致）

```dockerfile
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o server .

FROM scratch
COPY --from=builder /app/server /server
# 如果你需要 HTTPS 请求，还需要 CA 证书
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
EXPOSE 8080
CMD ["/server"]
```

`scratch` 完全为空——没有 shell，没有 `ls`，没有 `cat`，什么都没有。你甚至不能 `docker exec` 进去。如果你的程序能跑，这是最安全的镜像。

> **想多一点**：攻击者拿到 shell 后第一件事是什么？`ls`、`whoami`、`curl`、`wget`、`nc`。在 distroless 或 scratch 镜像里，这些命令全都不存在。攻击者面对的是一个"空房间"——连墙壁都没有。他能做的只有利用你程序本身的漏洞，而不是通过系统工具扩大战果。这就是"最小化攻击面"的核心思想：不是把门锁好，而是把不需要的房间全部拆掉。

## 六、只读文件系统：`--read-only`

### 用途

```bash
docker run -d --read-only --name secure-nginx nginx:alpine
```

容器启动后，整个文件系统是只读的。任何试图写文件的操作都会失败。这防止了攻击者在容器内写入恶意文件、修改配置、安装后门。

### 但有些目录需要写

nginx 需要写临时文件，MySQL 需要写数据目录。这些目录用 `tmpfs` 或 volume 挂载为可写：

```bash
docker run -d \
  --read-only \
  --tmpfs /var/cache/nginx \
  --tmpfs /var/run \
  nginx:alpine
```

`--tmpfs` 挂载的是内存文件系统，容器重启后数据丢失。对于缓存和临时文件来说，这正好。

### Compose 中的只读配置

```yaml
services:
  web:
    image: nginx:alpine
    read_only: true
    tmpfs:
      - /var/cache/nginx
      - /var/run
```

### 哪些目录需要可写？

| 应用 | 需要可写的目录 | 原因 |
|------|--------------|------|
| nginx | `/var/cache/nginx`, `/var/run` | 缓存和 PID 文件 |
| MySQL | `/var/lib/mysql` | 数据文件（用 volume 挂载） |
| Node.js | `/tmp` | 临时文件上传 |
| 通用 | `/tmp` | 很多应用默认写临时文件到 `/tmp` |

## 七、密钥管理：不要用 ENV 传密码

### 为什么 ENV 不安全？

```dockerfile
# ❌ 密码写在 Dockerfile 里
FROM my-app:latest
ENV DB_PASSWORD=your_super_secret_password_here
```

这个密码会被**永久嵌入镜像层**。任何人拿到镜像，都能通过 `docker history` 看到：

```bash
docker history my-app:latest
# IMAGE          CREATED BY
# abc123         ENV DB_PASSWORD=your_super_secret_password_here  ← 明文可见！
```

### 也不要在 docker run 里传

```bash
# ❌ 密码出现在命令行里，可能被 ps 看到
docker run -d -e DB_PASSWORD=your_secret_here my-app
```

```bash
# 另一个终端里
ps aux | grep docker
# 可以看到包含密码的完整命令
```

### 正确做法：用文件或外部密钥管理

#### 方式一：Docker Secrets（Swarm 模式）

```bash
# 创建 secret
echo "your_db_password_here" | docker secret create db_password -

# 在 service 中使用
docker service create \
  --name my-app \
  --secret db_password \
  my-app:latest
```

secret 被挂载到 `/run/secrets/db_password`，应用从文件读取：

```python
# Python 示例
with open('/run/secrets/db_password', 'r') as f:
    db_password = f.read().strip()
```

#### 方式二：环境变量文件（不提交到 Git）

```bash
# .env 文件（加入 .gitignore）
DB_PASSWORD=your_db_password_here

# compose 中引用
# compose.yml
services:
  app:
    environment:
      DB_PASSWORD: ${DB_PASSWORD}
```

```bash
docker compose --env-file .env up -d
```

#### 方式三：外部密钥管理

对于生产环境，用专业的密钥管理服务：

- **HashiCorp Vault**：开源密钥管理平台
- **AWS Secrets Manager**：AWS 上的托管密钥管理
- **Azure Key Vault**：Azure 上的托管密钥管理
- **Kubernetes Secrets**：如果你在 K8s 里

```
容器启动 → 向 Vault 认证 → 获取密钥 → 加载到内存 → 不落地到磁盘
```

## 八、其他安全实践

### 1. 不要用 `latest` 标签

```bash
# ❌ latest 标签不明确，不知道具体版本
docker pull nginx:latest
# 可能是 1.25，也可能是 1.27，完全取决于镜像仓库当前状态

# ✅ 锁定具体版本
docker pull nginx:1.27.0-alpine
```

### 2. 定期更新基础镜像

```bash
# 定期拉取最新版，重新构建
docker pull nginx:1.27.0-alpine
docker build --pull -t my-app:latest .
# --pull 确保用最新的基础镜像
```

### 3. 不要 `--privileged`

```bash
# ❌ 危险：--privileged 给了容器几乎所有的内核能力
docker run --privileged my-app

# ✅ 只给需要的权限
docker run --cap-add NET_BIND_SERVICE my-app
# 只加绑定 1024 以下端口的权限，而不是所有权限
```

### 4. 限制容器能力（Capabilities）

Linux capabilities 是"root 权限的碎片化"——把 root 的全能拆成 40+ 个独立的能力。

```bash
# 默认 Docker 容器只有 14 种能力
# 你可以进一步削减

# 只给最少的权限
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE my-app
```

### 5. 安全配置检查清单

```yaml
# compose.yml — 安全最佳实践
services:
  web:
    image: nginx:1.27.0-alpine      # ✅ 固定版本
    read_only: true                  # ✅ 只读文件系统
    user: "101:101"                  # ✅ 非 root 用户
    cap_drop:                        # ✅ 削减能力
      - ALL
    cap_add:
      - NET_BIND_SERVICE             # ✅ 只加需要的
    security_opt:
      - no-new-privileges:true       # ✅ 禁止进程获取新权限
    tmpfs:
      - /var/cache/nginx
      - /var/run
```

## 九、实战：从"裸奔"到"基本安全"

### 改造前：一个不安全的 Dockerfile

```dockerfile
FROM node:latest
WORKDIR /app
COPY . .
RUN npm install
ENV SECRET_KEY=your_actual_secret_key_here
EXPOSE 3000
CMD ["node", "server.js"]
```

问题：
- `FROM node:latest`：版本不明确
- 没有 `USER`：以 root 运行
- `ENV SECRET_KEY`：密钥嵌入镜像
- 没有 `--chown`：文件所有权是 root

### 改造后

```dockerfile
FROM node:20.15.0-alpine

# 创建非 root 用户
RUN addgroup -g 1001 -S appgroup && \
    adduser -u 1001 -S appuser -G appgroup

WORKDIR /app

# 复制依赖文件，改所有权
COPY --chown=appuser:appgroup package.json package-lock.json ./
RUN npm ci --production

# 复制源代码
COPY --chown=appuser:appgroup . .

USER appuser
EXPOSE 3000
CMD ["node", "server.js"]
```

运行：

```bash
docker build -t secure-app .
docker run -d \
  --read-only \
  --tmpfs /tmp \
  --name secure-app \
  -e SECRET_KEY="$(cat /path/to/secret_key)" \
  secure-app
```

## 十、我做得对不对？

### 验证清单

```bash
# 1. 确认容器以非 root 用户运行
docker run --rm secure-app whoami
# 预期：appuser（不是 root）

# 2. 确认容器没有多余能力
docker run --rm secure-app sh -c 'mount -o remount,rw /'
# 预期：Permission denied（只读文件系统）

# 3. 扫描镜像漏洞
docker scout quickview secure-app
# 预期：没有 CRITICAL 或 HIGH 漏洞

# 4. 确认镜像大小合理
docker images secure-app
# 预期：相对于完整版镜像，体积应该小很多

# 5. 确认密钥不在镜像里
docker history secure-app
# 检查：没有 ENV 行包含密钥

# 6. 确认容器文件系统是只读的
docker inspect secure-app --format='{{.HostConfig.ReadonlyRootfs}}'
# 预期：true
```

## 十一、不对怎么办？

### 常见错误 1：非 root 用户没有文件权限

```bash
# 症状：容器启动报 Permission denied
docker run secure-app
# Error: EACCES: permission denied, open '/app/config.json'
```

❌ 原因：用 `COPY . .` 复制文件时，文件所有者是 root。后面的 `USER appuser` 没有权限读这些文件。

✅ 修正：

```dockerfile
COPY --chown=appuser:appgroup . .
```

或者在 `USER appuser` 之前，先改权限：

```dockerfile
COPY . .
RUN chown -R appuser:appgroup /app    # 但这样会增加镜像层
```

### 常见错误 2：`--read-only` 后应用报错

```bash
# 症状：应用启动后报错，写文件失败
docker run --read-only my-app
# Error: EROFS: read-only file system, open '/app/logs/app.log'
```

✅ 修正：找出哪些目录需要写，用 `tmpfs` 或 volume 挂载：

```bash
docker run --read-only \
  --tmpfs /app/logs \
  --tmpfs /tmp \
  my-app
```

**排查方法**：先用 `strace` 或应用日志找出哪些路径需要写，然后逐个加 `--tmpfs`。

### 常见错误 3：Dockerfile 里写了密钥

```dockerfile
# ❌ 密钥嵌入镜像
ENV API_KEY=sk-1234567890abcdef
RUN echo $API_KEY > /app/config.json
```

这个密钥永久留在镜像层里。即使删了 `ENV` 行，`docker history` 也能看到。

✅ 修正：密钥在运行时传入，不在构建时写入：

```bash
docker run -e API_KEY="$(cat /path/to/api_key)" my-app
```

### 常见错误 4：`docker scout` 不扫描或报错

```bash
docker scout quickview my-app
# Error: image not found or not accessible
```

❌ 原因：镜像还没推送到 registry，docker scout 需要从 registry 获取漏洞数据。

✅ 修正：

```bash
# 先推送到 Docker Hub
docker tag my-app your-username/my-app:latest
docker push your-username/my-app:latest
docker scout quickview your-username/my-app:latest
```

或者用 Trivy（不需要推送）：

```bash
trivy image my-app:latest
```

### 常见错误 5：`USER` 写在 `COPY` 之前

```dockerfile
# ❌ 错误顺序
USER appuser
COPY . .         # 以 appuser 身份复制，但 appuser 在宿主机上可能没有权限
```

✅ 正确顺序：

```dockerfile
# 先以 root 复制文件（root 权限足够）
COPY --chown=appuser:appgroup . .

# 然后切换用户
USER appuser
```

## 十二、术语解释

| 术语 | 解释 |
|------|------|
| **USER** *此术语见附录C* | Dockerfile 指令，指定后续 `RUN`、`CMD`、`ENTRYPOINT` 以哪个用户身份运行。默认是 root。`USER appuser` 切换为非 root 用户 |
| **distroless** *此术语见附录C* | Google 维护的最小容器镜像，只包含程序运行时需要的最小文件集。没有 shell、没有包管理器、没有 curl。攻击面极小 |
| **scratch** *此术语见附录C* | Docker 的保留镜像名，代表一个完全空的镜像。适合运行纯静态编译的二进制程序。没有任何系统工具 |
| **CVE** *此术语见附录C* | Common Vulnerabilities and Exposures，公开的软件安全漏洞标识。如 CVE-2024-21626 是 runc 的一个容器逃逸漏洞 |
| **docker scout** *此术语见附录C* | Docker 官方提供的镜像漏洞扫描工具，可以分析镜像中的 CVE 漏洞并提供修复建议 |
| **`--read-only`** *此术语见附录C* | Docker 运行参数，将容器的文件系统挂载为只读。防止容器内的进程修改文件系统，缩小攻击面 |

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 默认 root 风险 | 容器默认以 root 运行 | 容器逃逸 + 权限滥用 = 宿主机风险 |
| 非 root 用户 | `USER appuser` + `adduser` + `--chown` | `COPY` 要在 `USER` 之前，且带 `--chown` |
| 端口限制 | 非 root 不能绑定 1024 以下端口 | 用高端口 + 反向代理或 Compose 端口映射 |
| Docker Daemon 安全 | 不暴露 2375 端口 | 2375 是不加密，2376 是 TLS 加密 |
| 镜像扫描 | `docker scout quickview` / `trivy image` | 定期扫描，CI/CD 中集成 |
| 最小基础镜像 | alpine → distroless → scratch | 越小 = 攻击面越小，但功能也越少 |
| 只读文件系统 | `--read-only` + `--tmpfs` | 需要给临时目录和缓存目录开可写 |
| 密钥管理 | Docker Secrets / 环境变量文件 / 外部密钥管理 | 永远不要把密钥嵌在镜像里或写在命令行 |
| 其他实践 | 固定版本、不 `--privileged`、削减 capabilities | 安全是洋葱，每一层都重要 |

## 本篇最可能出错的地方及原因

### 1. 密钥通过 ENV 写入镜像（最高频）

**这是安全审计中最常见的"高危"发现。** 开发者为了方便，把数据库密码、API Key 写在 Dockerfile 的 `ENV` 里。这个密钥会永久嵌入镜像层，任何人拿到镜像都能通过 `docker history` 看到明文。

**不可逆**：一旦密钥被写入镜像层并推送到了 registry，就无法从镜像历史中删除。即使你修改 Dockerfile 删掉 `ENV` 行，旧的镜像层已经存在于 registry 中。唯一的补救措施是**立即更换密钥**。

**预防**：在 CI/CD 中加一个检查步骤——扫描 Dockerfile 中是否包含 `ENV.*(KEY|SECRET|PASSWORD|TOKEN)` 模式。

### 2. `--read-only` 后应用崩溃，但不知道哪些目录需要写

**原因**：`--read-only` 是"全或无"的开关。很多应用默认写日志到 `/var/log`、写临时文件到 `/tmp`、写缓存到 `/var/cache`。设为只读后，这些写入全部失败，应用崩溃。

**排查方法**：先用 `--read-only` 启动，看应用日志里报什么错误。每个 "Permission denied" 或 "Read-only file system" 错误对应一个需要可写的目录。然后逐个加 `--tmpfs`。

### 3. 用 `USER` 但忘了 `--chown`，文件权限错误

```dockerfile
USER appuser
CMD ["node", "server.js"]
# 如果 server.js 的所有者是 root 且权限是 700，appuser 读不了
```

**症状**：容器启动报 `EACCES: permission denied`，但 `ls -la` 看文件确实存在。排查时容易忽略——文件存在，但属于 root，非 root 用户读不了。

**修正**：`COPY --chown=appuser:appgroup . .`，或者在 `USER` 之前 `RUN chown -R appuser:appgroup /app`。

### 4. `docker scout` 扫描不到本地镜像

**原因**：`docker scout` 需要从 Docker Hub 获取漏洞数据库。如果镜像没有推送到 registry，scout 可能无法获取信息。这是常见的"第一次使用"困惑。

**替代方案**：Trivy（`trivy image`）可以直接扫描本地镜像，不需要推送。

### 5. 用 `scratch` 镜像但程序需要 CA 证书

**原因**：程序调用 HTTPS API 时，需要 CA 证书来验证服务器证书。`scratch` 镜像里没有 `/etc/ssl/certs/` 目录。Go 程序默认会找这个路径，找不到就报 TLS 错误。

**修正**：在第一阶段复制 CA 证书：

```dockerfile
FROM golang:1.22-alpine AS builder
# ... 编译 ...

FROM scratch
COPY --from=builder /app/server /server
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
```

或者用 Go 的 `x509.SystemCertPool()` 在代码中嵌入证书。
