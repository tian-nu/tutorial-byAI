## 4.1 继承（`extends`）

### 4.1.1 为什么需要继承——代码复用与 is-a 关系

在开发中你会反复遇到这样的场景：多个类有共同的属性和方法。比如 `Dog`（狗）和 `Cat`（猫）都有名字、年龄，都会发出叫声——但它们的具体叫声不同。

**不用继承（重复代码）：**

```java
public class Dog {
    private String name;
    private int age;
    
    public Dog(String name, int age) { this.name = name; this.age = age; }
    
    public String getName() { return name; }
    public int getAge() { return age; }
    
    public void eat() { System.out.println(name + "在吃骨头"); }
    public void sleep() { System.out.println(name + "在睡觉"); }
    public void bark() { System.out.println(name + "汪汪叫"); }
}

public class Cat {
    private String name;
    private int age;
    
    public Cat(String name, int age) { this.name = name; this.age = age; }
    
    public String getName() { return name; }
    public int getAge() { return age; }
    
    public void eat() { System.out.println(name + "在吃鱼"); }
    public void sleep() { System.out.println(name + "在睡觉"); }
    public void meow() { System.out.println(name + "喵喵叫"); }
}
```

两个类中 `name`、`age`、`getName()`、`getAge()`、`sleep()` 是重复的。随着动物种类增多，重复会成倍增长。

**使用继承（消除重复）：**

```java
// 父类：提取公共代码
public class Animal {
    private String name;
    private int age;
    
    public Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }
    
    public String getName() { return name; }
    public int getAge() { return age; }
    
    public void eat() { System.out.println(name + "在吃东西"); }
    public void sleep() { System.out.println(name + "在睡觉"); }
}

// 子类：只需写自己特有的内容
public class Dog extends Animal {
    public Dog(String name, int age) {
        super(name, age);  // 调用父类构造方法
    }
    
    public void bark() { System.out.println(getName() + "汪汪叫"); }
}

public class Cat extends Animal {
    public Cat(String name, int age) {
        super(name, age);
    }
    
    public void meow() { System.out.println(getName() + "喵喵叫"); }
}

// 使用：
public class Test {
    public static void main(String[] args) {
        Dog dog = new Dog("旺财", 3);
        dog.eat();    // 继承自Animal：旺财在吃东西
        dog.sleep();  // 继承自Animal：旺财在睡觉
        dog.bark();   // Dog自己特有的：旺财汪汪叫
        
        Cat cat = new Cat("咪咪", 2);
        cat.eat();    // 继承自Animal：咪咪在吃东西
        cat.meow();   // Cat自己特有的：咪咪喵喵叫
    }
}
```

继承建立的不仅是代码复用关系，更重要的是**`is-a` 关系**：Dog **是一个** Animal（狗是动物），Cat **是一个** Animal（猫是动物）。这个 `is-a` 关系是多态的根基。

### 4.1.2 Java单继承规则

Java只允许**单继承**——一个类只能有一个直接父类：

```java
// ✅ 正确：单继承
public class Dog extends Animal { }

// ✅ 正确处理多重继承需求：通过接口实现（4.5节）
public class SmartPhone extends Device implements Phone, Camera, MusicPlayer { }

// ❌ 错误：Java不支持多继承
// public class X extends A, B { }  // 编译错误！
```

为什么Java禁止多继承？最主要的原因是**菱形继承问题（Diamond Problem）**：如果A类和B类都有同名方法 `doSomething()`，C同时继承A和B——编译器不知道C该调用哪一个。C++允许多继承但引入了虚继承这个复杂概念来解决。Java选择更简单的方式：用接口多实现来弥补单继承的限制。

### 4.1.3 子类能继承什么

父类的不同访问权限成员，在子类中有不同的访问能力：

```java
package com.example.parent;

public class Parent {
    public    int publicField    = 1;
    protected int protectedField = 2;
              int defaultField   = 3;  // 不写修饰符
    private   int privateField   = 4;
    
    public    void publicMethod()    {}
    protected void protectedMethod() {}
              void defaultMethod()   {}
    private   void privateMethod()   {}
}
```

```java
package com.example.child;
import com.example.parent.Parent;

public class Child extends Parent {
    public void test() {
        System.out.println(publicField);     // ✅ public：任何地方
        System.out.println(protectedField);   // ✅ protected：子类可见
        // System.out.println(defaultField);  // ❌ default：不同包不可见
        // System.out.println(privateField);  // ❌ private：只有本类
        
        publicMethod();       // ✅
        protectedMethod();    // ✅
        // defaultMethod();   // ❌ 不同包
        // privateMethod();   // ❌ 不可继承
    }
}
```

| 父类成员 | 同包子类 | 不同包子类 | 说明 |
|----------|---------|-----------|------|
| `public` | ✅ 可继承 | ✅ 可继承 | 完全开放 |
| `protected` | ✅ 可继承 | ✅ 可继承 | 专门为继承设计 |
| `default` | ✅ 可继承 | ❌ 不可继承 | 包级私有 |
| `private` | ❌ 不可继承 | ❌ 不可继承 | 仅本类 |

> ⚠️ `private` 成员虽然不能直接访问，但可以通过父类提供的public/protected方法间接访问。例如父类有 `private int age` 但提供了 `public int getAge()`，子类可以调用 `getAge()`。

### 4.1.4 `super` 关键字

`super` 是子类引用父类对象的"桥梁"。与 `this` 类似，`super` 也有三种用法：

**用法1：`super.方法名()` — 调用父类被重写的方法**

```java
public class Animal {
    public void speak() {
        System.out.println("动物在叫");
    }
}

public class Dog extends Animal {
    @Override
    public void speak() {
        super.speak();  // 先执行父类的speak()
        System.out.println("准确地说是汪汪叫");
    }
}

// 测试：
Dog dog = new Dog();
dog.speak();
// 输出：
// 动物在叫
// 准确地说是汪汪叫
```

**用法2：`super(参数)` — 调用父类构造方法**

```java
public class Animal {
    private String name;
    
    public Animal(String name) {
        this.name = name;
        System.out.println("Animal构造：" + name);
    }
}

public class Dog extends Animal {
    private String breed;
    
    public Dog(String name, String breed) {
        super(name);  // 必须显式调用父类有参构造
        this.breed = breed;
        System.out.println("Dog构造：" + name + ", 品种：" + breed);
    }
}

// 测试：
new Dog("旺财", "金毛");
// 输出：
// Animal构造：旺财
// Dog构造：旺财, 品种：金毛
```

> 🚨 **坑点：子类构造方法中如果不写 `super()`，编译器会自动插入 `super()`（无参）。但如果父类**没有**无参构造方法 → 编译错误！**
> 
> **原因**：编译器强制要求子类构造方法的第一行必须调用父类的某个构造方法（确保父类部分先初始化）。如果你没写，它自动补 `super()`。但如果父类不存在无参构造方法，`super()` 就找不到目标了。
> 
> **错误示例**：
> ```java
> public class Parent {
>     private String name;
>     // 只有有参构造，无参构造消失了！
>     public Parent(String name) { this.name = name; }
> }
> 
> public class Child extends Parent {
>     public Child() {
>         // 编译器自动插入 super() → 找不到Parent() → 编译错误！
>     }
> }
> ```
> 
> **解决方案**：
> ```java
> // 方案1：父类补上无参构造（推荐）
> public class Parent {
>     private String name;
>     public Parent() {}               // ← 加上
>     public Parent(String name) { this.name = name; }
> }
> 
> // 方案2：子类显式调用父类有参构造
> public class Child extends Parent {
>     public Child(String name) {
>         super(name);  // 显式调用有参构造
>     }
> }
> ```

### 4.1.5 方法重写（`@Override`）

方法重写是子类对父类方法进行重新实现，是多态的技术前提。

**重写的规则（五条必须同时满足）：**

| 规则 | 说明 |
|------|------|
| 1. 方法名相同 | 子类方法与父类方法名称完全一致 |
| 2. 参数列表相同 | 参数类型、数量、顺序完全一致 |
| 3. 返回类型兼容 | 基本类型必须一致；引用类型可以是父类返回类型的子类（协变返回类型） |
| 4. 访问权限 >= 父类 | 父类是protected，子类可以是protected或public；不能缩窄为private |
| 5. 异常范围 <= 父类 | 不能抛出比父类更宽泛的受检异常 |

```java
public class Animal {
    public Animal getParent() {
        return this;
    }
    
    protected void makeSound() throws Exception {
        System.out.println("...");
    }
}

public class Dog extends Animal {
    @Override
    public Dog getParent() {          // ✅ 协变返回类型：Dog是Animal的子类
        return this;
    }
    
    @Override
    public void makeSound() {          // ✅ 权限放大（protected→public）
                                       // ✅ 异常范围缩小（Exception→无）
        System.out.println("汪汪！");
    }
}
```

> 🚨 **坑点：参数列表不同 → 变成重载（Overload）而非重写（Override）**
> 
> **原因**：`@Override` 注解会让编译器帮你检查是否真的构成重写。参数列表不同意味着这是另一个方法，编译器会报错提醒你——前提是你**加了 `@Override` 注解**！
> 
> **错误示例**：
> ```java
> public class Parent {
>     public void doWork(int x) { }
> }
> 
> public class Child extends Parent {
>     @Override
>     public void doWork(long x) { }  // ❌ 参数类型不同，不是重写！编译错误
> }
> // 如果不加 @Override，就不会报错，但你就写了一个和父类无关的新方法
> ```

> 🚨 **坑点：返回类型如果是基本类型，必须完全一致**
> 
> ```java
> public class Parent {
>     public int getValue() { return 0; }
> }
> 
> public class Child extends Parent {
>     @Override
>     public long getValue() { return 0L; }  // ❌ 编译错误！int ≠ long
> }
> ```

## 4.2 多态——本章的核心

多态（Polymorphism）是整个面向对象编程中最具威力的特性。它表达了一种能力：**同一个行为，在不同的对象上展现出不同的结果**。如果你只能从本章记住一个知识点，那就是多态。

### 4.2.1 多态的定义——三步递进式理解

**步骤1：父类引用指向子类对象**

```java
Animal a = new Dog("旺财", 3);  // 父类类型 变量 = new 子类构造
```

这一步在多态的语境下叫做**向上转型（Upcasting）**——把子类对象当作父类来看待。这是自动的、安全的，因为 Dog "是一个" Animal。

**步骤2：编译时类型 vs 运行时类型**

```java
Animal a = new Dog("旺财", 3);
// 编译时类型：Animal（编译器看到的类型）
// 运行时类型：Dog（JVM实际执行的类型）
```

**步骤3：调用方法时，执行子类重写的方法**

```java
Animal a = new Dog("旺财", 3);
a.speak();  // 输出："旺财汪汪叫" —— 执行的是Dog的speak()，不是Animal的！
```

**多态的三个必要条件：**

1. **继承或实现**：存在父子类关系或接口实现关系
2. **方法重写**：子类重写了父类的方法
3. **父类引用指向子类对象**：`Parent p = new Child();`

### 4.2.2 核心口诀：「编译看左边，运行看右边」

这是理解多态的**终极口诀**。我们用一套完整的代码来验证它的每一种情况：

```java
// ===== 父类 =====
class Parent {
    int field = 1;                    // 成员变量
    static int staticField = 10;      // 静态变量
    
    public void method() {            // 普通方法
        System.out.println("Parent.method()");
    }
    
    public static void staticMethod() {  // 静态方法
        System.out.println("Parent.staticMethod()");
    }
}

// ===== 子类 =====
class Child extends Parent {
    int field = 2;                    // 同名成员变量
    static int staticField = 20;      // 同名静态变量
    
    @Override
    public void method() {            // 重写方法
        System.out.println("Child.method()");
    }
    
    public static void staticMethod() {  // 隐藏父类静态方法
        System.out.println("Child.staticMethod()");
    }
    
    public void onlyChildMethod() {   // 子类特有方法
        System.out.println("onlyChildMethod()");
    }
}
```

现在，我们创建一个关键的测试：

```java
public class PolymorphismTest {
    public static void main(String[] args) {
        Parent p = new Child();  // ★ 核心：父类引用指向子类对象
        
        // ============ 对比实验开始 ============
        
        // ── 实验1：成员方法 ──
        p.method();
        //   编译看左边(Parent)：Parent有method() → 编译通过 ✅
        //   运行看右边(Child) ：Child重写了method() → 执行Child.method()
        //   输出："Child.method()"  ← 多态生效！
        
        // ── 实验2：成员变量 ──
        System.out.println(p.field);
        //   编译看左边(Parent)：Parent有field → 编译通过 ✅
        //   运行看左边(Parent)：成员变量没有多态！→ 访问的是Parent.field
        //   输出：1 ← 不是2！！这就是最大的坑！
        
        // ── 实验3：静态方法 ──
        p.staticMethod();
        //   编译看左边(Parent)：Parent有staticMethod() → 编译通过 ✅
        //   运行看左边(Parent)：静态方法没有多态！→ 执行的是Parent.staticMethod()
        //   输出："Parent.staticMethod()" ← 不是Child的！
        
        // ── 实验4：静态变量 ──
        System.out.println(p.staticField);
        //   编译看左边(Parent)：Parent有staticField → 编译通过 ✅
        //   运行看左边(Parent)：静态变量没有多态 → 访问的是Parent.staticField
        //   输出：10 ← 不是20！
        
        // ── 实验5：子类特有方法 ──
        // p.onlyChildMethod();
        //   编译看左边(Parent)：Parent没有onlyChildMethod() → 编译错误！❌
        //   即使运行时的对象是Child，编译器只认Parent类型！
    }
}
```

**汇总对比 —— "编译看左边，运行看右边"完整版：**

| 访问目标 | 编译时 | 运行时 | 口诀 |
|----------|--------|--------|------|
| **成员方法** | 看左边（Parent）| **看右边（Child）** ✅ | **方法有多态** |
| **成员变量** | 看左边（Parent）| **看左边（Parent）** ❌ | **变量无多态** |
| **静态方法** | 看左边（Parent）| **看左边（Parent）** ❌ | **静态无多态** |
| **静态变量** | 看左边（Parent）| **看左边（Parent）** ❌ | **静态无多态** |

> 🚨 **坑点：成员变量没有多态性！（高频面试题）**
> 
> **这是Java面试中最常考的坑之一。** 当父类和子类有同名成员变量时，通过父类引用访问该变量，得到的是父类的变量值，而不是子类的。
> 
> **原因**：成员变量的访问在编译期就确定了（静态绑定），编译器根据引用的类型（编译看左边）来决定访问哪个变量。而成员方法的调用是动态绑定（运行看右边），JVM在运行时根据对象的实际类型来决定调用哪个方法。
> 
> ```java
> class Parent {
>     int x = 10;
>     public void show() { System.out.println("Parent x=" + x); }
> }
> class Child extends Parent {
>     int x = 20;
>     @Override
>     public void show() { System.out.println("Child x=" + x); }
> }
> 
> Parent p = new Child();
> System.out.println(p.x);    // 10 ← 父类的x，变量没有多态！
> p.show();                    // "Child x=20" ← 子类的方法，方法有多态！
> ```
> 
> **面试标准回答**：成员变量的绑定是静态的（编译期确定），引用是什么类型就访问哪个类型的变量；成员方法的绑定是动态的（运行时确定），调用的是对象实际类型中重写的方法。所以口诀是"编译看左边，运行看右边"——成员变量编译运行都看左边，成员方法编译看左边、运行看右边。

### 4.2.3 向上转型（Upcasting）与向下转型（Downcasting）

**向上转型**：子→父，自动完成，安全可靠：

```java
Dog dog = new Dog("旺财", 3);
Animal animal = dog;          // 自动向上转型，不需要强制转换
// 或直接写：
Animal animal2 = new Dog("旺财", 3);  // 最常见写法

animal.eat();                  // ✅ 可以调用Animal中定义的方法
// animal.bark();              // ❌ 不能调用Dog特有的方法（编译看左边）
```

**向下转型**：父→子，需要强制转换，有风险：

```java
Animal animal = new Dog("旺财", 3);
Dog dog = (Dog) animal;       // 强制向下转型
dog.bark();                    // ✅ 现在可以调用Dog特有的方法了

// ⚠️ 危险场景：
Animal animal2 = new Cat("咪咪", 2);
Dog dog2 = (Dog) animal2;     // 编译通过，但运行时抛出 ClassCastException！
// Cat不能转成Dog！
```

> 🚨 **坑点：直接强转可能 `ClassCastException`，必须先 `instanceof` 判断！**
> 
> **错误示例**：
> ```java
> public void handleAnimal(Animal animal) {
>     Dog dog = (Dog) animal;   // 如果animal实际是Cat，这里就崩了！
>     dog.bark();
> }
> ```
> 
> **正确做法**：
> ```java
> public void handleAnimal(Animal animal) {
>     if (animal instanceof Dog) {
>         Dog dog = (Dog) animal;
>         dog.bark();
>     } else if (animal instanceof Cat) {
>         Cat cat = (Cat) animal;
>         cat.meow();
>     }
> }
> ```

**`instanceof` 模式匹配（Java 16+）——更优雅的写法：**

```java
// 传统写法（Java 16之前）：
if (animal instanceof Dog) {
    Dog dog = (Dog) animal;    // 判断完还要强转一次
    dog.bark();
}

// Java 16+ 模式匹配（推荐！）：
if (animal instanceof Dog dog) {  // 一步完成：判断 + 声明 + 强转
    dog.bark();
}

if (animal instanceof Cat cat) {
    cat.meow();
}

// 结合逻辑运算：
if (animal instanceof Dog dog && dog.getAge() > 5) {
    System.out.println(dog.getName() + "是老狗了");
}
```

### 4.2.4 多态的应用场景——三个实战案例

**场景1：方法参数用父类类型——可传任意子类**

```java
public class Zoo {
    // 这个方法可以接受任何Animal的子类
    public void feed(Animal animal) {
        System.out.print("给" + animal.getName() + "喂食：");
        animal.eat();  // 多态！实际调用子类的eat()
    }
}

// 使用：
Zoo zoo = new Zoo();
zoo.feed(new Dog("旺财", 3));   // 给旺财喂食：旺财在吃东西
zoo.feed(new Cat("咪咪", 2));   // 给咪咪喂食：咪咪在吃鱼（假设Cat重写了eat）
zoo.feed(new Bird("小黄", 1));  // 将来增加Bird类，feed方法不需要改动！
```

这就是**对扩展开放、对修改关闭**（开闭原则）的体现——增加新动物时，`Zoo.feed()` 一行代码都不需要改。

**场景2：集合中存储多种子类**

```java
List<Animal> animals = new ArrayList<>();
animals.add(new Dog("旺财", 3));
animals.add(new Cat("咪咪", 2));
animals.add(new Dog("大黄", 5));
animals.add(new Cat("小花", 1));

// 统一处理：多态让每个动物用自己的方式"叫"
for (Animal a : animals) {
    a.speak();   // 多态！实际调用各自的speak()
}
// 输出：
// 旺财汪汪叫
// 咪咪喵喵叫
// 大黄汪汪叫
// 小花喵喵叫
```

## 4.3 `final` 关键字

`final` 的含义是"最终的、不可改变的"。它修饰不同的目标时有不同的效果：

### 4.3.1 `final` 类：不能被继承

```java
public final class String { }  // JDK中的String类就是final的

// public class MyString extends String { }  // ❌ 编译错误！String不能被继承
```

**为什么 `String` 要被 `final` 修饰？**
1. **安全性**：防止恶意子类篡改字符串行为（如重写 `equals()` 破坏HashMap）
2. **性能**：JVM知道String没有子类，可以进行内联优化
3. **不可变性保证**：`final` 配合 `private final char[]`，彻底保证String不可变

### 4.3.2 `final` 方法：不能被子类重写

```java
public class Parent {
    public final void criticalOperation() {
        // 这个方法实现的算法是核心机密，不允许子类改变
        System.out.println("核心操作");
    }
}

public class Child extends Parent {
    // @Override
    // public void criticalOperation() { }  // ❌ 编译错误！final方法不能重写
}
```

### 4.3.3 `final` 变量：赋值后不可变

**修饰基本类型——值不能改变：**

```java
final int MAX_VALUE = 100;
// MAX_VALUE = 200;  // ❌ 编译错误！
```

**修饰引用类型——引用不能改变，但对象的内容可以改变：**

> 🚨 **坑点：`final` 修饰的引用类型变量，变量指向的对象不能变，但对象内部的属性可以修改！**
> 
> ```java
> final List<String> names = new ArrayList<>();
> names.add("张三");      // ✅ 可以改变对象内容！
> names.add("李四");      // ✅ 可以
> names.remove(0);        // ✅ 可以
> // names = new ArrayList<>();  // ❌ 不能改变引用指向！
> 
> final StringBuilder sb = new StringBuilder("Hello");
> sb.append(" World");    // ✅ 可以修改字符串内容
> // sb = new StringBuilder();  // ❌ 不能改变引用
> ```
> 
> **面试记忆法**：`final` 锁住的是变量本身（栈中的值），对于引用类型，栈中存的是地址——所以锁住的是地址，地址指向的堆中对象该怎样还怎样。

## 4.4 `abstract` 抽象类

### 4.4.1 场景引入

回到动物体系的例子：每种动物都有叫声，但叫声各不相同。在 `Animal` 类中写 `speak()`，你该写什么实现？写 `"..."` ？不写实现让子类负责？——抽象方法正是为此而生。

```java
// 抽象类：不能new的类
public abstract class Animal {
    private String name;
    
    public Animal(String name) {
        this.name = name;
    }
    
    // 抽象方法：没有方法体，子类必须实现
    public abstract void speak();
    
    // 普通方法：有实现，子类可以直接继承使用
    public String getName() { return name; }
    public void sleep() { System.out.println(name + "在睡觉"); }
}

// 子类实现抽象方法
public class Dog extends Animal {
    public Dog(String name) { super(name); }
    
    @Override
    public void speak() {
        System.out.println(getName() + "汪汪叫");
    }
}

public class Cat extends Animal {
    public Cat(String name) { super(name); }
    
    @Override
    public void speak() {
        System.out.println(getName() + "喵喵叫");
    }
}

// 使用：
Animal a1 = new Dog("旺财");
a1.speak();  // 旺财汪汪叫（多态！）
// Animal a2 = new Animal("xxx");  // ❌ 抽象类不能实例化！
```

### 4.4.2 抽象类的特点

| 特点 | 说明 |
|------|------|
| 不能 `new` | `new Animal()` 编译错误，抽象类只能被继承 |
| 可以有构造方法 | 供子类通过 `super()` 调用 |
| 可以有普通方法和属性 | 抽象类不要求全是抽象方法 |
| 抽象方法必须在抽象类中 | 反过来，抽象类可以不包含抽象方法 |
| 子类必须实现所有抽象方法 | 除非子类也是抽象类 |

> 🚨 **坑点：`abstract` 不能与 `private`、`static`、`final` 共存**
> 
> | 组合 | 为什么不行 |
> |------|-----------|
> | `abstract` + `private` | 抽象方法需要被子类重写，`private` 让子类根本看不到这个方法，矛盾！ |
> | `abstract` + `static` | 静态方法属于类，可以通过类名直接调用。但抽象方法没有方法体，调用它要执行什么？ |
> | `abstract` + `final` | `final` 禁止重写，`abstract` 强制重写，水火不容！ |

## 4.5 `interface` 接口

### 4.5.1 接口的本质——定义规范 / 契约 / 能力

接口描述的是"**能做什么**"（can-do），而不是"**是什么**"（is-a）。最常见的类比是USB接口：

> USB接口定义了一套规范：插头形状、通信协议、供电能力。任何设备只要遵循USB规范，就可以插入任何USB插口。U盘、鼠标、键盘、打印机——这些设备本身完全不同，但它们都"能"通过USB通信。

```java
// 接口：定义"USB设备"这个能力
public interface USB {
    void connect();     // 连接
    void disconnect();  // 断开
    String getDeviceName();
}

// 实现类1：U盘 — 它能作为USB设备工作
public class USBFlashDrive implements USB {
    @Override
    public void connect() {
        System.out.println("U盘已连接，开始传输数据");
    }
    @Override
    public void disconnect() {
        System.out.println("U盘安全弹出");
    }
    @Override
    public String getDeviceName() {
        return "Kingston 64GB U盘";
    }
}

// 实现类2：鼠标 — 它也能作为USB设备工作
public class USBMouse implements USB {
    @Override
    public void connect() {
        System.out.println("鼠标已连接，光标准备就绪");
    }
    @Override
    public void disconnect() {
        System.out.println("鼠标已断开");
    }
    @Override
    public String getDeviceName() {
        return "Logitech 无线鼠标";
    }
}

// 使用：多态！接口类型引用指向实现类
public class Computer {
    public void plugUSB(USB device) {    // 参数类型是接口
        System.out.println("检测到设备：" + device.getDeviceName());
        device.connect();
        // ... 使用设备 ...
        device.disconnect();
    }
}

// 测试：
Computer pc = new Computer();
pc.plugUSB(new USBFlashDrive());  // 插U盘
pc.plugUSB(new USBMouse());       // 插鼠标
// 将来增加USB打印机、USB键盘……Computer.plugUSB() 一行不改
```

### 4.5.2 接口的演变历程

Java接口从诞生至今经历了三次重大升级，每一次都解决了一个实际问题：

```
时间线：Java接口的演变
═══════════════════════════════════════════════════════════════════
  
  Java 7 及以前                Java 8                        Java 9+
  ┌──────────────────┐    ┌──────────────────────┐    ┌──────────────────────┐
  │ 只能有：          │    │ 新增：               │    │ 新增：               │
  │ • 抽象方法        │    │ • default 方法  ←重磅│    │ • private 方法        │
  │   (public abstract)│   │   (有方法体！)       │    │   (default方法间复用) │
  │ • 常量             │    │ • static 方法        │    │                      │
  │   (public static   │    │                      │    │ 依然不能有：          │
  │    final)          │    │ 依然不能有：          │    │ • 实例变量            │
  │                    │    │ • 实例变量            │    │ • 构造方法            │
  └──────────────────┘    └──────────────────────┘    └──────────────────────┘
         ↓                        ↓                         ↓
    纯抽象契约              "接口也可以有实现了"         "实现代码也可以复用了"
```

**为什么 Java 8 要加入 `default` 方法？—— Collection 的故事**

Java 8 引入了 Stream API，Oracle 想在 `Collection` 接口中增加 `stream()` 方法。但问题来了：

- `Collection` 是 Java 最基础的接口之一
- 全世界有成千上万个自定义的 `Collection` 实现类
- 如果在接口中增加新的抽象方法，**所有这些实现类都必须立即实现 `stream()`，否则编译报错！**

这就是著名的"接口演进困境"。`default` 方法的解决方案优雅至极：

```java
public interface Collection<E> {
    // 原有的抽象方法（所有实现类已经实现了的）
    int size();
    boolean add(E e);
    // ... 几十个方法
    
    // Java 8 新增：default方法——有默认实现，实现类不需要强制实现！
    default Stream<E> stream() {
        // 提供默认实现
        return StreamSupport.stream(spliterator(), false);
    }
    
    default Stream<E> parallelStream() {
        return StreamSupport.stream(spliterator(), true);
    }
}
```

**Java 9 为何加入 `private` 方法？**

当接口中有多个 `default` 方法共享同一段逻辑时，`private` 方法可以提取公共代码：

```java
public interface ReportGenerator {
    default String generatePDF() {
        String data = fetchData();     // 公共逻辑
        return "PDF: " + data;
    }
    
    default String generateHTML() {
        String data = fetchData();     // 公共逻辑
        return "HTML: " + data;
    }
    
    // Java 9+：私有方法供default方法复用
    private String fetchData() {
        return "从数据库获取报表数据";
    }
}
```

### 4.5.3 函数式接口（`@FunctionalInterface`）

**函数式接口 = 有且仅有一个抽象方法的接口**。这是Lambda表达式的基础（第7章详讲）：

```java
@FunctionalInterface   // 该注解是可选的，但推荐加上——编译器会帮你检查
public interface Calculator {
    int calculate(int a, int b);  // 唯一的抽象方法
    
    // 可以有default方法和static方法，不影响"唯一抽象方法"的判定
    default void printVersion() {
        System.out.println("Calculator v1.0");
    }
}

// 传统使用方式：
Calculator add = new Calculator() {
    @Override
    public int calculate(int a, int b) {
        return a + b;
    }
};

// Lambda方式（第7章详讲）：
Calculator addLambda = (a, b) -> a + b;
System.out.println(addLambda.calculate(10, 20));  // 30
```

Java中常见的函数式接口：

| 接口 | 抽象方法 | 用途 |
|------|---------|------|
| `Runnable` | `void run()` | 线程执行 |
| `Comparator<T>` | `int compare(T o1, T o2)` | 比较排序 |
| `Consumer<T>` | `void accept(T t)` | 消费一个值 |
| `Supplier<T>` | `T get()` | 供给一个值 |
| `Function<T,R>` | `R apply(T t)` | 转换 |
| `Predicate<T>` | `boolean test(T t)` | 判断 |

### 4.5.4 接口的多实现——弥补单继承

```java
// 定义两个能力接口
public interface Flyable {
    void fly();
}

public interface Swimmable {
    void swim();
}

// 一个类可以实现多个接口
public class Duck extends Animal implements Flyable, Swimmable {
    public Duck(String name) { super(name); }
    
    @Override
    public void speak() { System.out.println(getName() + "嘎嘎叫"); }
    
    @Override
    public void fly() { System.out.println(getName() + "在飞"); }
    
    @Override
    public void swim() { System.out.println(getName() + "在游泳"); }
}
```

> 🚨 **坑点：两个接口有同名的 `default` 方法 → 必须重写来解决冲突！**
> 
> ```java
> public interface A {
>     default void hello() { System.out.println("A"); }
> }
> 
> public interface B {
>     default void hello() { System.out.println("B"); }
> }
> 
> // public class C implements A, B { }
> // ❌ 编译错误：C inherits unrelated defaults for hello() from types A and B
> 
> // 解决方案：显式重写，决定用哪个（或自己写新逻辑）
> public class C implements A, B {
>     @Override
>     public void hello() {
>         A.super.hello();  // 指定调用A的实现
>         // 或者 B.super.hello();
>         // 或者自己写全新逻辑
>     }
> }
> ```

### 4.5.5 接口 vs 抽象类 —— 完整对比

| 对比项 | 抽象类（abstract class） | 接口（interface） |
|--------|--------------------------|-------------------|
| **设计目的** | 代码复用、"is-a"关系 | 定义规范、"can-do"能力 |
| **关键字** | `abstract class` | `interface` |
| **构造方法** | ✅ 可以有 | ❌ 不能有 |
| **成员变量** | 任意类型 | 仅 `public static final` 常量 |
| **普通方法** | ✅ 可以 | Java 8+ 可以有 `default` / `static` 方法 |
| **抽象方法** | ✅ 可以 | ✅ 可以（Java 7及之前只有抽象方法） |
| **private方法** | ✅ 可以 | Java 9+ 可以 |
| **继承/实现** | 单继承（extends一个类） | 多实现（implements多个接口） |
| **实例化** | ❌ 不能new | ❌ 不能new |
| **何时使用** | 多个类共享大量代码、有明确的父子层级 | 定义跨层级的能力、实现类之间没有共同代码 |

**选型口诀**：**"继承用抽象，能力用接口。共享代码用抽象类，定义规范用接口。"**

## 4.6 内部类

### 4.6.1 成员内部类

定义在另一个类的内部，作为外部类的一个成员：

```java
public class Outer {
    private String name = "外部类";
    
    // 成员内部类
    public class Inner {
        public void print() {
            // 内部类可以直接访问外部类的所有成员（包括private）
            System.out.println("从内部类访问：" + name);
        }
    }
    
    public void createInner() {
        Inner inner = new Inner();  // 外部类内部可以直接new
        inner.print();
    }
}

// 外部使用（需要先创建外部类对象）：
Outer outer = new Outer();
Outer.Inner inner = outer.new Inner();  // 注意语法：outer.new
inner.print();  // 从内部类访问：外部类
```

### 4.6.2 静态内部类

用 `static` 修饰的内部类，**不依赖外部类的实例**：

```java
public class Outer {
    private static String staticName = "静态外部类";
    private String instanceName = "实例外部类";
    
    // 静态内部类
    public static class StaticInner {
        public void print() {
            System.out.println("访问静态成员：" + staticName);   // ✅
            // System.out.println("访问实例成员：" + instanceName); // ❌ 不能访问
        }
    }
}

// 外部使用（不需要创建外部类对象）：
Outer.StaticInner inner = new Outer.StaticInner();  // 直接new
inner.print();
```

> 📌 静态内部类是创建工具类或辅助类时的好选择。例如 `Map.Entry` 就是 `Map` 接口中的静态内部接口。

### 4.6.3 匿名内部类——重点，Lambda的前身

匿名内部类允许你在**不创建独立类文件**的情况下，直接实现一个接口或继承一个类：

```java
public class AnonymousDemo {
    public static void main(String[] args) {
        // 方式1：实现接口的匿名内部类
        Runnable task = new Runnable() {
            @Override
            public void run() {
                System.out.println("匿名内部类在执行任务");
            }
        };
        new Thread(task).start();
        
        // 方式2：继承抽象类的匿名内部类
        Animal anonymousCat = new Animal("小咪") {
            @Override
            public void speak() {
                System.out.println(getName() + "喵喵叫（匿名类）");
            }
        };
        anonymousCat.speak();
        
        // 方式3：最简洁的写法——直接在调用处写匿名内部类
        new Thread(new Runnable() {
            @Override
            public void run() {
                System.out.println("线程执行中...");
            }
        }).start();
    }
}
```

**为什么匿名内部类重要？** 在JDK 8引入Lambda之前，这是实现回调、事件监听、线程创建的主要方式。你会在旧代码中大量见到这种写法。第7章我们将学习Lambda如何将以上代码简化为：

```java
// Lambda 写法（预告）—— 比匿名内部类简洁数倍
new Thread(() -> System.out.println("线程执行中...")).start();
```

## 本章小结

本章是全书的第一个分水岭。学完本章，你应当能用Java写出结构清晰、可扩展的面向对象代码了。回顾核心知识点：

- **继承**：`class Dog extends Animal`。子类继承父类的public和protected成员，通过 `super` 调用父类构造和方法。`@Override` 确保方法正确重写。子类构造默认调用 `super()`，父类没有无参构造时必须显式 `super(参数)`。
- **多态（核心）**：父类引用指向子类对象 `Animal a = new Dog()`。**口诀："编译看左边，运行看右边"**—成员变量/静态成员看左边（无多态），成员方法看右边（有多态）。向上转型自动安全，向下转型必须先 `instanceof` 判断。
- **`final`**：类不可被继承、方法不可被重写、变量引用不可被改变（但对象内容可修改）。
- **`abstract`**：抽象类不能被实例化。抽象方法没有方法体，子类必须实现。`abstract` 不能与 `private`/`static`/`final` 共存。
- **接口**：定义"能做什么"的契约。Java 7→8（加入default/static方法）→9（加入private方法）→函数式接口为Lambda铺路。接口多实现弥补单继承限制。多接口default方法冲突时必须重写。
- **内部类**：成员内部类依赖外部类实例、静态内部类不依赖、匿名内部类是Lambda的前身。

> 📌 **学完面向对象的封装、继承、多态、抽象类和接口，接下来学习Java核心API（第5章）时，你会发现集合框架大量使用了接口和多态——`List<String> list = new ArrayList<>()` 正是多态的体现。继续前进！**

### 思考题答案

**答案1**：输出 `10` 和 `B.x=20`。
- `obj.x`：成员变量没有多态性，"编译看左边"——`obj` 的编译时类型是 `A`，所以访问的是 `A` 的 `x`，值为 `10`。
- `obj.print()`：成员方法有多态性，"运行看右边"——`obj` 的运行时类型是 `B`，所以调用的是 `B` 的 `print()`，输出 `B.x=20`。

**答案2**：两行都有问题。
- `private abstract void doPrivate()`：❌ 编译错误。`private` 方法子类不可见，而 `abstract` 要求子类重写，矛盾。
- `public static abstract void doStatic()`：❌ 编译错误。`static` 方法属于类，可以通过类名直接调用，但 `abstract` 方法没有方法体，无法执行。

**答案3**：第一行有问题。
- `private String apiKey = "sk-123456";`：❌ 接口中的变量默认是 `public static final` 的，不能声明为 `private`。正确：`String API_KEY = "sk-123456";`（隐式 `public static final`）或在 Java 9+ 中使用 `private` 方法（不是变量）。
- `static PaymentService getInstance()` 中使用匿名内部类：✅ 完全正确。可以在接口的 `static` 方法中返回接口的匿名实现。

**答案4**：
- 用法A：✅ `arr[0] = 100;` 正确。`final` 锁住的是 `arr` 引用（不能指向新数组），但数组内容可以修改。输出 `100`。
- 用法B：❌ `list = new ArrayList<>();` 错误。`final` 锁住 `list` 引用，不能让它指向新的 `ArrayList` 对象。但 `list.add("hello")` 是 ✅ 正确的。
- 用法C：❌ 不能。`PI` 被 `final` 修饰（值不可变），且是 `static`（类级别的常量），在类加载后就确定，永远不能修改。

**答案5**：

```java
// 接口
public interface Authenticatable {
    boolean login(String username, String password);
    void logout();
}

// 抽象类
public abstract class BaseUser implements Authenticatable {
    protected String name;
    protected String role;
    protected boolean loggedIn = false;
    
    public BaseUser(String name, String role) {
        this.name = name;
        this.role = role;
    }
    
    public abstract String[] getPermissions();
    
    @Override
    public boolean login(String username, String password) {
        if ("admin".equals(username) && "123456".equals(password)) {
            loggedIn = true;
            System.out.println(name + "(" + role + ")登录成功");
            return true;
        }
        System.out.println("登录失败");
        return false;
    }
    
    @Override
    public void logout() {
        loggedIn = false;
        System.out.println(name + "已登出");
    }
}

// 管理员
public class Admin extends BaseUser {
    public Admin(String name) { super(name, "管理员"); }
    
    @Override
    public String[] getPermissions() {
        return new String[]{"创建", "编辑", "删除", "管理用户"};
    }
}

// 编辑者
public class Editor extends BaseUser {
    public Editor(String name) { super(name, "编辑者"); }
    
    @Override
    public String[] getPermissions() {
        return new String[]{"编辑"};
    }
}

// 多态运用
public class AuthService {
    public void processLogin(Authenticatable user, String username, String password) {
        if (user.login(username, password)) {
            System.out.println("处理登录后逻辑...");
        }
    }
    
    public static void main(String[] args) {
        AuthService service = new AuthService();
        service.processLogin(new Admin("张三"), "admin", "123456");
        service.processLogin(new Editor("李四"), "admin", "wrong");
    }
}
```
