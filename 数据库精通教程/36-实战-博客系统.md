> 电商系统练的是"关系"——用户→订单→商品，一条线拉到底。博客系统不一样：它练的是"多对多"和"树形结构"。一篇文章可以有多个标签，一个标签下有 N 篇文章；一条评论可以有 N 条回复，回复还可以有回复——这就不是简单的 1:N 了。这一章，我们把这些"复杂关系"一个个拆开。

## 学完能做什么

- 设计多对多关系（文章↔标签）并正确使用中间表
- 使用 MySQL 全文搜索（ngram 解析器）实现中文搜索
- 用邻接表实现评论的树形结构（无限层级回复）
- 用 Redis 缓存文章阅读量（INCR）、热门文章排行（Sorted Set）、文章内容（String）
- 在 100 万文章 + 1000 万评论的规模下验证查询性能

## 一、需求分析：博客系统到底存什么？

### 1.1 核心场景

一个典型的博客系统，用户打开网站后：
1. 看到首页的热门文章列表（按阅读量排序）
2. 搜索文章（中文全文搜索）
3. 点进一篇文章，看到内容和评论
4. 评论可以互相回复（嵌套结构）
5. 给文章点赞
6. 每篇文章有多个标签（技术标签如"MySQL""Redis""Go"）

### 1.2 六大核心实体

| 实体 | 白话解释 | 存什么 |
|------|---------|--------|
| **用户（User）** | 写文章和评论的人 | 用户名、密码哈希、头像、简介 |
| **文章（Article）** | 博客的内容主体 | 标题、正文、摘要、阅读量、点赞数、状态 |
| **评论（Comment）** | 对文章的回复 | 内容、评论人、父评论 ID（实现嵌套回复） |
| **标签（Tag）** | 文章的分类标记 | 标签名（如"MySQL""Redis"） |
| **文章标签关联（ArticleTag）** | 多对多中间表 | 文章 ID + 标签 ID |
| **点赞（Like）** | 用户对文章的点赞 | 用户 ID + 文章 ID（联合唯一，防重复点赞） |

### 1.3 想多一点：为什么"阅读量"要放 Redis？

阅读量是一个**写操作极度频繁**的字段。一篇热门文章每秒可能有几百次阅读，如果每次都去 `UPDATE articles SET view_count = view_count + 1`，数据库的压力会非常大。

而且，阅读量"丢一点"是可以接受的——你少统计了 10 次阅读，用户根本看不出来。所以这是一个典型的"最终一致性"场景：用 Redis 的 `INCR` 原子递增，然后定期（比如每 5 分钟）批量同步回 MySQL。

### 1.4 想多一点：评论为什么用"邻接表"而不是"嵌套集"？

有两种常见的树形结构存储方案：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **邻接表** | 每条记录存 `parent_id` | 插入快、结构直观 | 查询多层级需要递归 |
| **嵌套集** | 每条记录存 `left`/`right` 值 | 查询子树快 | 插入/移动节点极慢 |

对于博客评论，**写多读也多**，而且评论树通常不会太深（最多 3-5 层）。邻接表是最合适的选择——插入一条评论只需要一个 `INSERT`，查子评论只需要 `SELECT * FROM comments WHERE parent_id = ?`。

## 二、ER 图设计

### 2.1 实体关系

```
用户 (users) ──1:N──> 文章 (articles)
用户 (users) ──1:N──> 评论 (comments)
文章 (articles) ──1:N──> 评论 (comments)
文章 (articles) ──M:N──> 标签 (tags)    ← 通过中间表 article_tags
评论 (comments) ──1:N──> 评论 (comments) ← 自引用，parent_id
用户 (users) ──M:N──> 文章 (articles)   ← 通过 likes 表（点赞）
```

### 2.2 字段清单

**users 表**：
- `id` BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
- `username` VARCHAR(50) NOT NULL UNIQUE
- `password_hash` CHAR(60) NOT NULL
- `avatar` VARCHAR(255) —— 头像 URL
- `bio` VARCHAR(500) —— 个人简介
- `status` TINYINT NOT NULL DEFAULT 1
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

**articles 表**：
- `id` BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
- `title` VARCHAR(200) NOT NULL
- `slug` VARCHAR(200) NOT NULL UNIQUE —— URL 友好的标题
- `summary` VARCHAR(500) —— 文章摘要
- `content` LONGTEXT NOT NULL —— 正文
- `view_count` INT UNSIGNED NOT NULL DEFAULT 0 —— 阅读量（从 Redis 定期同步）
- `like_count` INT UNSIGNED NOT NULL DEFAULT 0 —— 点赞数
- `comment_count` INT UNSIGNED NOT NULL DEFAULT 0 —— 评论数（冗余，避免 COUNT）
- `status` TINYINT NOT NULL DEFAULT 1 —— 1:已发布 0:草稿 2:已删除
- `author_id` BIGINT UNSIGNED NOT NULL
- `published_at` DATETIME
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
- `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

**tags 表**：
- `id` INT UNSIGNED AUTO_INCREMENT PRIMARY KEY
- `name` VARCHAR(50) NOT NULL UNIQUE
- `article_count` INT UNSIGNED NOT NULL DEFAULT 0 —— 该标签下的文章数（冗余）
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

**article_tags 表**（多对多中间表）：
- `article_id` BIGINT UNSIGNED NOT NULL
- `tag_id` INT UNSIGNED NOT NULL
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
- PRIMARY KEY (article_id, tag_id)

**comments 表**：
- `id` BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
- `article_id` BIGINT UNSIGNED NOT NULL
- `user_id` BIGINT UNSIGNED NOT NULL
- `parent_id` BIGINT UNSIGNED DEFAULT NULL —— 父评论 ID，NULL 表示顶级评论
- `content` TEXT NOT NULL
- `status` TINYINT NOT NULL DEFAULT 1 —— 1:正常 0:已删除
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

**likes 表**（点赞）：
- `user_id` BIGINT UNSIGNED NOT NULL
- `article_id` BIGINT UNSIGNED NOT NULL
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
- PRIMARY KEY (user_id, article_id)

## 三、建表 SQL

```sql
CREATE DATABASE IF NOT EXISTS blog
    DEFAULT CHARACTER SET utf8mb4
    DEFAULT COLLATE utf8mb4_unicode_ci;

USE blog;

-- ============================================================
-- 1. 用户表
-- ============================================================
CREATE TABLE users (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    password_hash CHAR(60) NOT NULL,
    avatar VARCHAR(255),
    bio VARCHAR(500),
    status TINYINT NOT NULL DEFAULT 1,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_username (username)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='用户表';

-- ============================================================
-- 2. 文章表
-- ============================================================
CREATE TABLE articles (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    slug VARCHAR(200) NOT NULL COMMENT 'URL友好标题，如 mysql-index-guide',
    summary VARCHAR(500),
    content LONGTEXT NOT NULL,
    view_count INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '从Redis定期同步',
    like_count INT UNSIGNED NOT NULL DEFAULT 0,
    comment_count INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '冗余字段，避免COUNT',
    status TINYINT NOT NULL DEFAULT 1 COMMENT '1:已发布 0:草稿 2:已删除',
    author_id BIGINT UNSIGNED NOT NULL,
    published_at DATETIME,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_slug (slug),
    INDEX idx_author_id (author_id),
    INDEX idx_status_published (status, published_at),
    INDEX idx_view_count (view_count),
    -- 全文索引：对标题和正文建全文索引
    FULLTEXT INDEX ft_title_content (title, content) WITH PARSER ngram,
    CONSTRAINT fk_article_author FOREIGN KEY (author_id)
        REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='文章表';

-- ============================================================
-- 3. 标签表
-- ============================================================
CREATE TABLE tags (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    article_count INT UNSIGNED NOT NULL DEFAULT 0,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_name (name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='标签表';

-- ============================================================
-- 4. 文章标签关联表（多对多中间表）
-- ============================================================
CREATE TABLE article_tags (
    article_id BIGINT UNSIGNED NOT NULL,
    tag_id INT UNSIGNED NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (article_id, tag_id),
    INDEX idx_tag_id (tag_id),
    CONSTRAINT fk_at_article FOREIGN KEY (article_id)
        REFERENCES articles(id) ON DELETE CASCADE,
    CONSTRAINT fk_at_tag FOREIGN KEY (tag_id)
        REFERENCES tags(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='文章标签关联表';

-- ============================================================
-- 5. 评论表（邻接表实现树形结构）
-- ============================================================
CREATE TABLE comments (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    article_id BIGINT UNSIGNED NOT NULL,
    user_id BIGINT UNSIGNED NOT NULL,
    parent_id BIGINT UNSIGNED DEFAULT NULL COMMENT '父评论ID，NULL=顶级评论',
    content TEXT NOT NULL,
    status TINYINT NOT NULL DEFAULT 1 COMMENT '1:正常 0:已删除',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_article_id (article_id),
    INDEX idx_parent_id (parent_id),
    INDEX idx_user_id (user_id),
    CONSTRAINT fk_comment_article FOREIGN KEY (article_id)
        REFERENCES articles(id) ON DELETE CASCADE,
    CONSTRAINT fk_comment_user FOREIGN KEY (user_id)
        REFERENCES users(id) ON DELETE CASCADE,
    CONSTRAINT fk_comment_parent FOREIGN KEY (parent_id)
        REFERENCES comments(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='评论表';

-- ============================================================
-- 6. 点赞表
-- ============================================================
CREATE TABLE likes (
    user_id BIGINT UNSIGNED NOT NULL,
    article_id BIGINT UNSIGNED NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (user_id, article_id),
    INDEX idx_article_id (article_id),
    CONSTRAINT fk_like_user FOREIGN KEY (user_id)
        REFERENCES users(id) ON DELETE CASCADE,
    CONSTRAINT fk_like_article FOREIGN KEY (article_id)
        REFERENCES articles(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='点赞表';
```

### 3.1 设计决策总结

| 决策 | 为什么 | 对应章节 |
|------|--------|---------|
| 全文索引用 `ngram` 解析器 | MySQL 默认解析器不支持中文分词，ngram 按字符 n-gram 分词 | 本章第四节 |
| `comment_count` 冗余 | 避免每次展示文章时 `COUNT(*)` 评论表 | 第 14 章（反范式） |
| 评论用邻接表（`parent_id`） | 插入快，适合博客的读写混合场景 | 本章第五节 |
| 点赞用联合主键 `(user_id, article_id)` | 天然防重复点赞 | 第 16 章 |
| `article_tags` 联合主键 | 同一文章不能重复关联同一标签 | 第 16 章 |

## 四、全文搜索：MySQL ngram 解析器

### 4.1 为什么需要全文搜索？

`LIKE '%关键词%'` 的问题是：全表扫描，无法使用普通索引（前置 `%` 导致索引失效）。100 万篇文章，`LIKE '%MySQL%'` 可能要扫十几秒。

全文索引（FULLTEXT Index）*此术语需进附录E* 是专门为文本搜索设计的索引结构。它会将文本拆成词条（token），建立倒排索引——就像书的目录，直接告诉你"MySQL 这个词出现在第 3、15、88 篇文章"。

### 4.2 ngram 解析器是什么？

MySQL 内置的全文解析器**不支持中文**——它按空格分词，而中文没有空格。所以我们需要 `ngram` 解析器。

ngram 的原理：把文本切成连续的 N 个字符。比如 "MySQL索引" 用 ngram（N=2）切分：
- "My"、"yS"、"SQ"、"QL"、"L索"、"索引"

虽然看起来有点"暴力"，但中文场景下效果不错——搜"索引"能匹配到，搜"MySQL"也能匹配到。

### 4.3 创建全文索引

```sql
-- 建表时已创建（见上文 DDL）：
-- FULLTEXT INDEX ft_title_content (title, content) WITH PARSER ngram;

-- 如果建表时没加，可以后续添加：
ALTER TABLE articles ADD FULLTEXT INDEX ft_title_content (title, content) WITH PARSER ngram;
```

### 4.4 全文搜索三种模式

```sql
-- 模式 1：自然语言模式（默认）
-- 搜索与"MySQL索引优化"最相关的文章，按相关性排序
SELECT id, title,
       MATCH(title, content) AGAINST('MySQL 索引 优化' IN NATURAL LANGUAGE MODE) AS relevance
FROM articles
WHERE MATCH(title, content) AGAINST('MySQL 索引 优化' IN NATURAL LANGUAGE MODE)
ORDER BY relevance DESC
LIMIT 10;

-- 模式 2：布尔模式（支持 + - ~ 等操作符）
-- + 表示必须包含，- 表示必须不包含，~ 表示降低权重
SELECT id, title
FROM articles
WHERE MATCH(title, content) AGAINST('+MySQL +索引 ~优化' IN BOOLEAN MODE)
LIMIT 10;

-- 模式 3：查询扩展模式
-- 先搜一次，取相关文章的高频词，再搜第二次
SELECT id, title
FROM articles
WHERE MATCH(title, content) AGAINST('MySQL' WITH QUERY EXPANSION)
LIMIT 10;
```

### 4.5 全文搜索的坑

```sql
-- ❌ 错误：搜索词太短
-- ngram 默认最小 token 长度为 2，搜一个字符会返回空
SELECT * FROM articles
WHERE MATCH(title, content) AGAINST('锁' IN NATURAL LANGUAGE MODE);
-- 结果: Empty set

-- ✅ 修复：修改 ngram_token_size
-- 在 my.cnf 中添加：
-- [mysqld]
-- ngram_token_size=1
-- 然后重建全文索引
```

```sql
-- ❌ 错误：停用词（stopword）被忽略
-- "the"、"is"、"的" 等词被 MySQL 默认忽略
SELECT * FROM articles
WHERE MATCH(title, content) AGAINST('的' IN NATURAL LANGUAGE MODE);
-- 结果: Empty set（"的"在停用词表中）

-- ✅ 查看停用词表
SELECT * FROM INFORMATION_SCHEMA.INNODB_FT_DEFAULT_STOPWORD;
```

### 4.6 全文搜索 vs LIKE 性能对比

```sql
-- 准备 100 万文章后，对比：

-- LIKE 方式（全表扫描）
SELECT * FROM articles WHERE content LIKE '%数据库%';
-- 耗时：~5-15 秒（取决于磁盘 IO）

-- 全文索引
SELECT * FROM articles
WHERE MATCH(title, content) AGAINST('数据库' IN NATURAL LANGUAGE MODE);
-- 耗时：~0.01-0.1 秒
```

## 五、评论树形结构：邻接表实战

### 5.1 邻接表的原理

每条评论有一个 `parent_id`，指向它的父评论。`parent_id = NULL` 表示这是一条顶级评论。

```
评论 1 (parent_id=NULL)
  ├── 评论 2 (parent_id=1)
  │     └── 评论 4 (parent_id=2)
  └── 评论 3 (parent_id=1)
```

### 5.2 查询某篇文章的评论树

```sql
-- 查询文章 ID=100 的顶级评论
SELECT c.id, c.content, c.parent_id, c.created_at,
       u.username, u.avatar
FROM comments c
JOIN users u ON c.user_id = u.id
WHERE c.article_id = 100
  AND c.parent_id IS NULL
  AND c.status = 1
ORDER BY c.created_at DESC
LIMIT 20;
```

```sql
-- 查询某条评论的所有子回复
SELECT c.id, c.content, c.parent_id, c.created_at,
       u.username
FROM comments c
JOIN users u ON c.user_id = u.id
WHERE c.parent_id = 1  -- 评论 ID=1 的所有子回复
  AND c.status = 1
ORDER BY c.created_at ASC;
```

### 5.3 用递归 CTE 查询完整评论树（MySQL 8.0+）

MySQL 8.0 支持递归 CTE（Common Table Expression）*此术语需进附录E*，可以一次性查出整棵树：

```sql
WITH RECURSIVE comment_tree AS (
    -- 基础查询：顶级评论
    SELECT c.id, c.article_id, c.user_id, c.parent_id, c.content,
           c.created_at, c.status, 0 AS depth,
           CAST(c.id AS CHAR(500)) AS path
    FROM comments c
    WHERE c.article_id = 100 AND c.parent_id IS NULL AND c.status = 1

    UNION ALL

    -- 递归查询：子评论
    SELECT c.id, c.article_id, c.user_id, c.parent_id, c.content,
           c.created_at, c.status, ct.depth + 1,
           CONCAT(ct.path, ',', c.id)
    FROM comments c
    INNER JOIN comment_tree ct ON c.parent_id = ct.id
    WHERE c.status = 1 AND ct.depth < 5  -- 限制最大深度，防止无限递归
)
SELECT ct.*, u.username, u.avatar
FROM comment_tree ct
JOIN users u ON ct.user_id = u.id
ORDER BY ct.path;
```

### 5.4 插入评论

```sql
-- 顶级评论（parent_id = NULL）
INSERT INTO comments (article_id, user_id, parent_id, content)
VALUES (100, 5, NULL, '这篇文章写得真好！');

-- 子回复（parent_id = 刚插入的评论 ID）
INSERT INTO comments (article_id, user_id, parent_id, content)
VALUES (100, 8, LAST_INSERT_ID(), '同意，特别是索引部分讲得很清楚');
```

## 六、Redis 缓存实战

### 6.1 阅读量缓存（INCR 原子递增）

阅读量是写操作最频繁的字段，用 Redis 的 `INCR` 来处理：

```bash
# 每次有人阅读文章，在 Redis 中递增
redis-cli> INCR article:view:100
(integer) 1573

# 获取当前阅读量
redis-cli> GET article:view:100
"1573"

# 批量获取多篇文章阅读量
redis-cli> MGET article:view:100 article:view:101 article:view:102
1) "1573"
2) "892"
3) "3456"
```

**定期同步回 MySQL**（可以用定时任务，比如每 5 分钟执行一次）：

```sql
-- 伪代码，实际在应用层遍历 Redis 中的 key
UPDATE articles SET view_count = view_count + 1573 WHERE id = 100;
```

### 6.2 热门文章排行（Sorted Set）

Redis 的 Sorted Set 天然适合排行榜——每个元素有一个分数，自动按分数排序。

```bash
# 文章被阅读时，增加分数
redis-cli> ZINCRBY hot:articles 1 article:100
"1"

# 获取 Top 10 热门文章
redis-cli> ZREVRANGE hot:articles 0 9 WITHSCORES
 1) "article:205"
 2) "9876"
 3) "article:100"
 4) "5678"
 5) "article:302"
 6) "4321"
 ...

# 获取某篇文章的排名
redis-cli> ZREVRANK hot:articles article:100
(integer) 1  # 排名第 2（从 0 开始）
```

```sql
-- 拿到热门文章 ID 列表后，去 MySQL 查详细信息
SELECT id, title, summary, view_count
FROM articles
WHERE id IN (205, 100, 302, ...)
  AND status = 1;
```

### 6.3 文章内容缓存（String）

文章内容很少改，但读得频繁。用 Redis String 缓存，带过期时间。

```bash
# 缓存文章内容，过期时间 1 小时
redis-cli> SETEX article:content:100 3600 "文章完整内容..."
OK

# 读取缓存
redis-cli> GET article:content:100
"文章完整内容..."

# 文章更新时，删除缓存（而不是更新缓存——这叫 Cache-Aside 模式）
redis-cli> DEL article:content:100
(integer) 1
```

### 6.4 缓存穿透/击穿/雪崩（回顾第 32 章）

| 问题 | 场景 | 本章的解决方案 |
|------|------|--------------|
| **缓存穿透** | 查询不存在的数据，缓存和 DB 都没有 | 布隆过滤器预判文章 ID 是否存在 |
| **缓存击穿** | 热点数据过期，大量请求打到 DB | 热门文章不设过期时间，手动更新 |
| **缓存雪崩** | 大量缓存同时过期 | 过期时间加随机值（如 3600 + rand(0,600)） |

WITH RECURSIVE comment_tree AS (
    SELECT c.id, c.article_id, c.parent_id, c.content, c.created_at,
           0 AS depth, CAST(c.id AS CHAR(500)) AS path
    FROM comments c
    WHERE c.article_id = 100 AND c.parent_id IS NULL AND c.status = 1
    UNION ALL
    SELECT c.id, c.article_id, c.parent_id, c.content, c.created_at,
           ct.depth + 1, CONCAT(ct.path, ',', c.id)
    FROM comments c
    INNER JOIN comment_tree ct ON c.parent_id = ct.id
    WHERE c.status = 1 AND ct.depth < 5
)
SELECT ct.*, u.username
FROM comment_tree ct
JOIN users u ON ct.user_id = u.id
ORDER BY ct.path;
