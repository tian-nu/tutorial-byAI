## 34.1 HashMap 扩容机制详解

### 34.1.1 核心参数

```java
// JDK HashMap 的关键常量
static final int DEFAULT_INITIAL_CAPACITY = 1 << 4; // 默认初始容量 = 16
static final float DEFAULT_LOAD_FACTOR = 0.75f;      // 默认负载因子
static final int TREEIFY_THRESHOLD = 8;               // 链表树化阈值
static final int UNTREEIFY_THRESHOLD = 6;             // 红黑树链化阈值
static final int MIN_TREEIFY_CAPACITY = 64;           // 最小树化容量
```

每个参数的意义：

| 参数 | 默认值 | 含义 |
|------|--------|------|
| 初始容量 | 16 | 桶数组的大小（始终是 2 的幂） |
| 负载因子 | 0.75 | 当 75% 的桶被占用时触发扩容 |
| 树化阈值 | 8 | 链表长度 ≥ 8 时转为红黑树 |
| 链化阈值 | 6 | 红黑树节点数 ≤ 6 时转回链表 |
| 最小树化容量 | 64 | 容量 < 64 时，优先扩容而非树化 |

### 34.1.2 扩容过程

**触发条件**：`size > capacity × loadFactor`（例如 16 × 0.75 = 12 时扩容）

```java
// 核心扩容代码（简化版）
final Node<K,V>[] resize() {
    Node<K,V>[] oldTab = table;
    int oldCap = (oldTab == null) ? 0 : oldTab.length;
    int newCap = oldCap << 1;  // 容量翻倍！

    Node<K,V>[] newTab = (Node<K,V>[])new Node[newCap];
    // 遍历旧桶，重新哈希到新桶
    for (int j = 0; j < oldCap; ++j) {
        Node<K,V> e = oldTab[j];
        if (e != null) {
            // 单节点：直接重新计算位置
            // 链表：拆分为两条链表（高位链 & 低位链）
            // 红黑树：拆分为两棵树
        }
    }
    return newTab;
}
```

**关键优化——高位链/低位链拆分**：

扩容时，不需要对每个节点重新计算 hashCode。因为容量是 2 的幂，新位置只有两种可能：
- **原位**：`hash & oldCap == 0` → 留在原来的索引位置
- **原位 + 旧容量**：`hash & oldCap != 0` → 放在 `原索引 + oldCap`

```java
// 具体拆分逻辑
Node<K,V> loHead = null, loTail = null; // 低位链（保持原位）
Node<K,V> hiHead = null, hiTail = null; // 高位链（移到新位置）

Node<K,V> next;
do {
    next = e.next;
    if ((e.hash & oldCap) == 0) {
        // 归入低位链
        if (loTail == null) loHead = e;
        else loTail.next = e;
        loTail = e;
    } else {
        // 归入高位链
        if (hiTail == null) hiHead = e;
        else hiTail.next = e;
        hiTail = e;
    }
} while ((e = next) != null);

// 低位链放回原位
if (loTail != null) {
    loTail.next = null;
    newTab[j] = loHead;
}
// 高位链放新位置
if (hiTail != null) {
    hiTail.next = null;
    newTab[j + oldCap] = hiHead;
}
```

> 🤔 **想多一点**：为什么要容量翻倍 + 高低位拆分而不是重新计算 hashCode？因为 `n = table.length` 是 2 的幂，`hash & (n-1)` 等价于 `hash % n`。扩容后 n 变 2n，新位置 = `hash & (2n-1)`。比 `hash & (n-1)` 多了一位，这一位正好是 `hash & n`。这就是"高低链"的数学本质。

### 34.1.3 链表 → 红黑树

当链表长度 ≥ 8 且容量 ≥ 64 时，链表转为红黑树。为什么是 8？

根据泊松分布，当负载因子为 0.75 时，链表长度达到 8 的概率约为 0.00000006（几乎不可能）。如果真的到了 8，说明哈希函数设计有问题，或者有人故意构造哈希碰撞攻击。

```
链表查找：O(n)，n=8 时最多查 8 次
红黑树查找：O(log n)，n=8 时最多查 3 次
```

### 34.1.4 容量为什么是 2 的幂

```java
// HashMap 保证容量为 2 的幂
static final int tableSizeFor(int cap) {
    int n = cap - 1;
    n |= n >>> 1;
    n |= n >>> 2;
    n |= n >>> 4;
    n |= n >>> 8;
    n |= n >>> 16;
    return (n < 0) ? 1 : (n >= MAXIMUM_CAPACITY) ? MAXIMUM_CAPACITY : n + 1;
}
// 作用：把任意整数变成 ≥ cap 的最小 2 的幂
// 例如 cap=10 → 16, cap=17 → 32
```

原因有三：
1. **索引计算高效**：`hash & (n-1)` 比 `hash % n` 快
2. **扩容时高低链拆分**：`hash & oldCap` 即可判断新位置
3. **减少哈希冲突**：2 的幂 -1 的二进制全是 1，哈希值分布更均匀

## 34.2 ConcurrentHashMap 演进

> 🏷️ 此术语需进附录：**ConcurrentHashMap**——线程安全的 HashMap。Java 7 使用分段锁（Segment），Java 8 改为 CAS + synchronized 锁单个桶，并发度大幅提升。

### 34.2.1 Java 7：分段锁

```
ConcurrentHashMap (Java 7)
├── Segment[0] (继承 ReentrantLock)
│   ├── HashEntry → HashEntry → ...
├── Segment[1]
│   ├── HashEntry → ...
├── ...
└── Segment[15]  (默认 16 个分段)
```

每个 Segment 独立加锁，不同 Segment 之间可以并发操作。默认并发级别 16，意味着最多 16 个线程同时写入不同段。

缺点：粒度太粗。即使两个 key 落在不同桶，如果落在同个 Segment 里也得抢锁。

### 34.2.2 Java 8：CAS + synchronized

JDK 8 彻底重写了 ConcurrentHashMap，放弃分段锁，改为**每个桶独立加锁**：

```java
// Java 8 写入简化逻辑
final V putVal(K key, V value, boolean onlyIfAbsent) {
    // 1. 如果桶为空，CAS 直接设置（无锁）
    if (tabAt(tab, i = (n - 1) & hash) == null) {
        if (casTabAt(tab, i, null, new Node<>(hash, key, value)))
            break;  // CAS 成功，不需要加锁
    }
    // 2. 如果桶不为空，synchronized 锁住桶的头节点
    synchronized (f) {
        // 遍历链表或红黑树，插入/覆盖
    }
}
```

核心改进：

| 对比项 | Java 7 | Java 8 |
|--------|--------|--------|
| 锁粒度 | Segment（16 个段） | 单个桶（bin） |
| 锁类型 | ReentrantLock | CAS + synchronized |
| 链表转树 | 不支持 | ≥8 转为红黑树 |
| 并发度 | 固定 16 | 等于桶数量 |

### 34.2.3 ConcurrentHashMap 用法

```java
import java.util.concurrent.ConcurrentHashMap;

ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

map.put("A", 1);                      // 线程安全
map.putIfAbsent("B", 2);              // 原子操作
map.computeIfAbsent("C", k -> 3);     // 不存在时原子计算

// 原子更新
map.compute("A", (k, v) -> v + 1);   // A = 2

// 批量操作
map.forEach(1, (k, v) -> System.out.println(k + "=" + v));
// 参数 1 是并行度阈值（元素数超过 1 时可能并行执行）

// 搜索
String result = map.search(1, (k, v) -> v > 1 ? k : null);
// 返回第一个 value > 1 的 key
```

> ⚠️ ConcurrentHashMap 的 key 和 value **都不能为 null**！这是和 HashMap 的一个重要区别。

## 34.3 Collections 工具类

`java.util.Collections` 提供了一系列操作集合的静态方法。

### 34.3.1 常用方法

```java
import java.util.*;

List<Integer> list = new ArrayList<>(List.of(3, 1, 4, 1, 5));

// 排序
Collections.sort(list);                       // [1, 1, 3, 4, 5]
Collections.reverse(list);                    // [5, 4, 3, 1, 1]
Collections.shuffle(list);                    // 随机打乱

// 查找
int idx = Collections.binarySearch(list, 3);  // 二分查找（需先排序）
int max = Collections.max(list);              // 最大值
int min = Collections.min(list);              // 最小值
int freq = Collections.frequency(list, 1);    // 出现频率

// 填充/替换
Collections.fill(list, 0);                    // 全部填充为0
Collections.replaceAll(list, 1, 99);          // 将所有1替换为99

// 不可变集合
List<Integer> unmodifiable = Collections.unmodifiableList(list);
// unmodifiable.add(6);  // ❌ UnsupportedOperationException

// 线程安全包装
List<Integer> syncList = Collections.synchronizedList(new ArrayList<>());
Map<String, Integer> syncMap = Collections.synchronizedMap(new HashMap<>());
```

### 34.3.2 线程安全包装的注意事项

```java
// ⚠️ synchronizedList 的迭代器不是线程安全的
List<String> syncList = Collections.synchronizedList(new ArrayList<>());
syncList.add("A");
syncList.add("B");

// ❌ 遍历时其他线程可能修改列表
for (String s : syncList) {  // 可能抛 ConcurrentModificationException
    System.out.println(s);
}

// ✅ 手动加锁
synchronized (syncList) {
    for (String s : syncList) {
        System.out.println(s);
    }
}
```

## 34.4 fail-fast 机制

> 🏷️ 此术语需进附录：**fail-fast**——迭代器遍历集合时，如果集合被其他方式修改（非迭代器自身的 remove），立即抛出 `ConcurrentModificationException`，防止出现不确定的行为。

```java
List<String> list = new ArrayList<>(List.of("a", "b", "c"));

// ❌ 增强 for 遍历中删除——抛 ConcurrentModificationException
for (String s : list) {
    if (s.equals("b")) {
        list.remove(s);  // 触发 fail-fast
    }
}
```

实现原理：集合内部有一个 `modCount` 计数器，每次结构性修改（增删）都会 +1。迭代器创建时记录 `expectedModCount = modCount`，每次 `next()` 检查两者是否相等。

```java
// ArrayList.Itr 简化源码
int expectedModCount = modCount;

public E next() {
    checkForComodification();
    // ...
}

final void checkForComodification() {
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
}
```

**安全删除方式**：
1. 迭代器自己的 `remove()`
2. Java 8 的 `removeIf()`
3. 先收集要删除的元素再批量删除

```java
// ✅ 方式1：迭代器 remove
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("b")) {
        it.remove();
    }
}

// ✅ 方式2：removeIf
list.removeIf(s -> s.equals("b"));
```

## 34.5 ❌ 常见错误

### 错误1：HashMap 初始化容量随意指定

```java
// ❌ new HashMap<>(100) —— 你以为容量是 100
// 实际：tableSizeFor(100) = 128（取 ≥100 的最小 2 的幂）

// ✅ 如果需要放 100 个元素，设置负载因子触发的阈值
// 容量 = 预期数量 / 负载因子 + 1
int capacity = (int) (100 / 0.75) + 1; // = 134
Map<String, String> map = new HashMap<>(capacity); // 内部会取到 256
```

### 错误2：对 ConcurrentHashMap 的复合操作不加锁

```java
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// ❌ 看似原子，实则不是！get 和 put 之间其他线程可能修改
if (map.get("key") == null) {
    map.put("key", 1);  // 两个线程可能同时到这里
}

// ✅ 用原子方法
map.putIfAbsent("key", 1);          // 原子
map.computeIfAbsent("key", k -> 1); // 原子
```

## 34.6 小结

| 知识点 | 一句话总结 |
|--------|-----------|
| HashMap 扩容 | 容量翻倍，高低链拆分，负载因子 0.75 |
| 树化 vs 链化 | 链表 ≥ 8 且容量 ≥ 64 转为红黑树；红黑树 ≤ 6 转回链表 |
| ConcurrentHashMap | Java 7 分段锁 → Java 8 CAS + synchronized 锁单桶 |
| Collections | 不可变集合、线程安全包装、排序/查找等工具方法 |
| fail-fast | 迭代时集合被修改则抛 ConcurrentModificationException |
| 容量计算 | HashMap 容量始终是 2 的幂，初始化时它自动向上取整 |

## 34.7 自测题

**1.** HashMap 的负载因子为什么默认是 0.75？如果设为 1.0 或 0.5 分别有什么影响？

**2.** 在 ConcurrentHashMap（Java 8）中，以下两个操作有什么区别？

```java
map.putIfAbsent("key", 1);
```
和
```java
if (map.get("key") == null) {
    map.put("key", 1);
}
```

**3.** 为什么 HashMap 扩容时，一个链表上的节点只会被拆分到两个桶中（原位置和原位置+旧容量），而不是被随机打散？

> 答案提示：1. 0.75 是时间和空间的折中——1.0 哈希冲突严重但省内存，0.5 查找快但浪费内存；2. putIfAbsent 是原子操作，get+put 不是；3. 因为容量是 2 的幂，新索引由 hash 多一位决定，只有 0 或 1 两种可能。
