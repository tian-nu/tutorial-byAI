# 第57章 · 锁机制

> "1000个人同时抢购最后一件库存，怎么办？答案是锁。就像公共厕所的门锁——一个人进去后锁上门，其他人只能在外面等。数据库的锁比厕所门锁复杂得多：有行级别的微观锁、表级别的宏观锁、读的时候可以共享的锁、写的时候必须独占的锁……还有一套名为MVCC的魔法，让大部分查询不需要加锁也能保证正确性。"

---

## 57.1 行锁 vs 表锁

### 表锁（Table Lock）

锁住整张表——在操作期间，整张表其他人不能读也不能写。

```sql
LOCK TABLES users READ;
```

**比喻**：你进入仓库后把大门锁了——任何人在你出来之前都进不去。

- **优点**：实现简单，开销小，不会死锁。
- **缺点**：并发极差。一个人锁了users表，其他499个请求全排队等着。
- **谁在用**：MyISAM引擎只用表锁（这也是它被InnoDB取代的原因之一）。

### 行锁（Row Lock）

只锁住被操作的那几行——同一张表其他行不受影响。

**比喻**：你只锁住你要修改的货箱，其他人可以同时操作其他货箱。

- **优点**：并发好，冲突少。
- **缺点**：开销大（锁更多需要管理），可能死锁。
- **谁在用**：InnoDB引擎默认使用行锁。

### InnoDB行锁的实现

InnoDB的行锁**不是锁在数据行上，而是锁在索引上**。这非常关键：

```sql
UPDATE users SET age = 30 WHERE name = 'zhangsan';
```

如果 `name` 字段有索引：InnoDB在name索引树上找到 'zhangsan'，在对应的索引记录上加锁。其他事务要修改同一行，必须等待。

如果 `name` 字段**没有索引**：InnoDB无法通过索引定位具体行，只能**全表扫描**，对扫描到的每一行都加行锁——效果等同于表锁！

**血的教训**：WHERE条件不走索引时，行锁退化为表锁。这也是为什么索引如此重要——不仅影响查询速度，还影响并发写入。

## 57.2 共享锁（S锁）与排他锁（X锁）

### 共享锁（Shared Lock / S锁 / 读锁）

加锁后，**其他人可以读但不能写**。

```sql
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;
```

（MySQL 8.0.22+ 推荐语法：`SELECT ... FOR SHARE`）

**比喻**：图书馆的一本书。你可以读，别人也可以同时读，但不能在上面写字。

多个事务可以同时持有同一行的S锁（读读共享）。但只要有人想写（申请X锁），必须等所有S锁释放。

### 排他锁（Exclusive Lock / X锁 / 写锁）

加锁后，**其他人既不能读也不能写**（除非你提交或回滚）。

以下操作自动加X锁：
- `UPDATE ...`
- `DELETE ...`
- `INSERT ...`（加的是插入意向锁，也是排他的）
- `SELECT ... FOR UPDATE`

```sql
SELECT * FROM users WHERE id = 1 FOR UPDATE;
```

**比喻**：你在修改一份重要文件，关上门不让任何人看——直到改完。

**S锁和X锁的相容矩阵**：

| | S锁 | X锁 |
|---|:---:|:---:|
| S锁 | ✓ 相容 | ✗ 冲突 |
| X锁 | ✗ 冲突 | ✗ 冲突 |

**关键理解**：读请求和写请求之间的锁竞争，是数据库并发性能的天花板。MVCC正是为了解决这个问题而存在的。

## 57.3 死锁

### 什么是死锁

两个事务互相等待对方释放锁——形成闭环，谁都动不了。

```
事务A:                                        事务B:
UPDATE users SET age=30 WHERE id=1;
  → 获得id=1的X锁
                                             UPDATE users SET age=25 WHERE id=2;
                                               → 获得id=2的X锁
UPDATE users SET age=31 WHERE id=2;
  → 等待B释放id=2的X锁...
                                             UPDATE users SET age=26 WHERE id=1;
                                               → 等待A释放id=1的X锁...
╔═══════╗                            ╔═══════╗
║   A   ║────→ 等待 id=2 ──→║   B   ║
║       ║←── 等待 id=1 ────║       ║
╚═══════╝                            ╚═══════╝
         ↑______ 闭环死锁 _______↑
```

**比喻**：十字路口四条车道的车同时往前开，谁也不让谁——全堵死了。

### MySQL如何处理死锁

InnoDB有**死锁检测机制**。检测到死锁后，选择一个事务（通常是持锁最少的那个）作为"牺牲品"——回滚它并抛出错误：

```
ERROR 1213 (40001): Deadlock found when trying to get lock; try restarting transaction
```

你的应用代码**必须处理这个错误**——不能当它不存在。标准做法是捕获1213错误后**重试**整个事务。

### 排查死锁

```sql
SHOW ENGINE INNODB STATUS;
```

输出中的 `LATEST DETECTED DEADLOCK` 段记录了最近一次死锁的详细信息：
- 哪两个事务互相卡住了
- 每个事务当时持有什么锁、在等待什么锁
- 最后一条执行的SQL是什么

有了这个输出，你就能分析出死锁原因，修改SQL顺序或加锁策略。

### 避免死锁的实践法则

1. **固定访问顺序**：多个事务访问多张表时，按相同的顺序操作。先A再B，不要有的先B再A。
2. **保持事务短小**：事务里不要掺杂耗时操作（网络请求、文件IO、用户交互）。
3. **尽可能用索引**：避免行锁退化为表锁（退化后更容易和别人冲突）。
4. **考虑使用低隔离级别**：READ COMMITTED在某些场景下死锁概率更低。
5. **一次锁定所有需要的数据**：`SELECT ... FOR UPDATE` 在事务开头就把需要的行锁住，而不是边操作边锁。

## 57.4 MVCC：多版本并发控制

MVCC是InnoDB实现高并发的核心魔法——它让**读不阻塞写，写不阻塞读**。

### MVCC的核心思想

不用锁来实现读一致性——而是给每行数据保留**多个版本**。每个事务看到的是"在它开始那个时刻"的数据快照。

**比喻**：你不是在仓库里直接翻货箱（那会挡别人的路）——而是走进仓库时有人递给你一张**仓库此刻的快照照片**。你拿着照片慢慢找，仓库里别人该怎么搬怎么搬，互不影响。

### 实现机制

每行数据在InnoDB中隐藏了三个字段：

| 隐藏字段 | 说明 |
|----------|------|
| DB_TRX_ID | 最近修改本行的事务ID（6字节） |
| DB_ROLL_PTR | 回滚指针，指向undo log中的旧版本（7字节） |
| DB_ROW_ID | 如果没有主键自动生成的rowid（6字节） |

undo log形成了一条**版本链**：

```
当前行 (TRX_ID=10, roll_ptr→):
    ↓
上一版本 (TRX_ID=8, roll_ptr→):
    ↓
上上一版本 (TRX_ID=5, roll_ptr→):
    ↓
最初版本 (TRX_ID=1)
```

### ReadView：决定能看到哪个版本

每个事务开始时生成一个ReadView，包含：

- `m_ids`：当前活跃的事务ID列表（还没提交的事务）。
- `min_trx_id`：活跃事务中最小的ID。
- `max_trx_id`：系统下一个要分配的事务ID。
- `creator_trx_id`：当前事务自己的ID。

查询一行数据时，遍历版本链，找到**第一个满足可见性条件**的版本：

```
版本可见 = (DB_TRX_ID < min_trx_id)                         已经提交的老版本
         OR (DB_TRX_ID == creator_trx_id)                   自己改的
         OR (DB_TRX_ID 不在 m_ids 中且 < max_trx_id)        别人已提交的
```

### 快照读 vs 当前读

**快照读（Snapshot Read）**：普通的SELECT走MVCC，不加锁，读的是历史版本。

```sql
SELECT * FROM users WHERE id = 1;
```

**当前读（Current Read）**：读取最新版本（不管ReadView），并加锁。

```sql
SELECT * FROM users WHERE id = 1 FOR UPDATE;
SELECT * FROM users WHERE id = 1 LOCK IN SHARE MODE;
UPDATE users SET age = 30 WHERE id = 1;
DELETE FROM users WHERE id = 1;
INSERT INTO users (...) VALUES (...);
```

当前读跳过MVCC，直接读最新版本——因为写操作必须基于最新数据，不能改一个已经过时的版本。

### REPEATABLE READ下的MVCC行为

在REPEATABLE READ级别下，事务只在**第一次快照读**时生成ReadView，之后一直用同一个ReadView——所以同一个事务内多次读结果一样。

在READ COMMITTED级别下，**每次快照读**都生成新的ReadView——所以能读到别人刚提交的修改（不可重复读的来源）。

🤔 **想多一点**：面试官问"MySQL的REPEATABLE READ是怎么解决不可重复读的？"答案就在MVCC的ReadView生成策略上——RR级别生成一个ReadView用到底，RC级别每次生成新的。这是一个面试高频问题，也是区分"会用MySQL"和"理解MySQL"的分水岭。

### MVCC的代价

- **undo log膨胀**：长事务会导致undo log版本链变长（因为旧版本不能被清理），占用空间、降低性能。
- **无法解决部分幻读**：当前读下MVCC无能为力，需要间隙锁（Gap Lock）配合。
- **不适合写多读少**：如果每条数据都被频繁修改，版本链很快变得很长，遍历成本上升。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 行锁 | InnoDB默认，锁在索引上，无索引时退化为表锁 |
| 表锁 | MyISAM使用，并发差但简单 |
| 共享锁(S) | 读锁，SELECT ... FOR SHARE，读读共享 |
| 排他锁(X) | 写锁，UPDATE/DELETE/INSERT自动加，读写都排他 |
| 死锁 | 两个事务互相等待，InnoDB自动检测+回滚，应用层要捕获重试 |
| 死锁避免 | 固定访问顺序、短事务、用索引、一次锁全、出错重试 |
| MVCC | 多版本并发控制，读不阻塞写，通过undo log版本链+ReadView实现 |
| ReadView | 决定版本可见性，RR下第一个ReadView用到底，RC下每次新生成 |
| 快照读 | 普通SELECT，通过MVCC读历史版本，不加锁 |
| 当前读 | SELECT FOR UPDATE / UPDATE / DELETE，读最新版，加锁 |
