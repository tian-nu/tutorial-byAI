## 我在做什么？

前面 20 章我们一直在聊"一个人操作数据库"：你写一条 SQL，数据库给你结果。但现实世界中，数据库往往是**多个人同时操作**——电商网站同时有几百个人下单，银行同时有几千个人转账。

更重要的是：有些操作必须"一起成功"，不能只成一半。比如银行转账：A 转 100 块给 B，必须满足"A 扣了 100，B 一定加了 100"。如果 A 扣了钱，数据库刚好崩溃，B 没收到——这钱就凭空消失了。

**事务（Transaction）** *此术语见附录E* 就是用来解决这个问题的。这一章我们学习事务的基本概念和控制语句，理解 ACID 四大特性，为下一章隔离级别打好基础。

学完这一章，你能：
- 说出 ACID 每个字母的含义（配生活例子）
- 正确使用 `BEGIN`/`COMMIT`/`ROLLBACK` 控制事务
- 理解自动提交和隐式提交的坑
- 用 `SAVEPOINT` 做部分回滚

## 一、事务是什么？一句话说清楚

**事务就是一组 SQL 操作，它们必须要么全成功，要么全失败。**

就像你去餐厅点菜：
- 你下单 → 厨房做菜 → 你付钱
- 如果厨房说"今天菜卖完了"，那整个订单取消——你不用付钱，厨房也不用做。不能说"做了一半，你付一半钱"。

最经典的例子就是**银行转账**：

```
事务开始
  1. 从账户 A 扣除 100 元
  2. 给账户 B 增加 100 元
事务结束
```

必须满足：
- 两步都成功 → 转账完成
- 任意一步失败 → 两步都撤销，回到转账前的状态

这就是事务的核心意义：**保证数据一致性，不让数据停留在"中间状态"。**

## 二、ACID 四大特性（每个字母配一个比喻）

ACID 是事务必须满足的四个特性，每个字母对应一个特性。很多面试都会问，你得能说出来，还要能举例子。

### 2.1 Atomicity（原子性）：要么全做，要么全不做

**比喻：签合同**——你要么签字盖章合同生效，要么不签合同作废，没有"签一半字合同算部分生效"这种说法。

原子性就是说：事务中的所有操作，是一个不可分割的原子单位。全部执行成功，才会提交；只要有一个失败，就必须回滚到事务开始前的状态。

转账例子：A 扣钱成功了，但 B 加钱失败（比如数据库崩溃），原子性要求必须把 A 扣的钱加回去，就像什么都没发生过一样。

### 2.2 Consistency（一致性）：从一个正确状态到另一个正确状态

**比喻：天平**——两边重量必须相等，不能一边重一边轻。

一致性是说：事务执行前，数据库满足所有约束（主键唯一、外键存在、余额不小于零等）；事务执行完，这些约束仍然满足。

转账例子：A 和 B 两个人的总金额，转账前后必须相等。转账前总金额是 1000，转账后总金额还是 1000——不能凭空多钱或少钱。

> 💡 思考题：原子性和一致性有什么区别？
> - 原子性：操作层面——做或不做
> - 一致性：结果层面——状态正确
> - 没有原子性就没有一致性，但光有原子性也不一定保证一致性（比如业务代码逻辑错了）

### 2.3 Isolation（隔离性）：事务之间互不干扰

**比喻：隔音房间**——每个事务都在自己的隔音房间里执行，听不到别的房间的声音。

隔离性是说：**多个事务同时执行时，一个事务的修改，对其他事务是不可见的**。你改你的，我改我的，互不干扰。

完全隔离意味着每个事务必须排队执行，那样并发性能就很差。所以实际数据库会提供不同的隔离级别，让你在"隔离程度"和"并发性能"之间做权衡——这是下一章的重点。

### 2.4 Durability（持久性）：一旦提交，永久保存

**比喻：刻在石头上**——字刻在石头上，雨淋水泡都不会消失。

持久性是说：**事务一旦提交成功，对数据的修改就是永久性的**。哪怕数据库宕机、重启，修改过的数据也不会丢失。

InnoDB 靠 redo log 保证持久性——具体我们第 24 章讲存储引擎时细说。

## 三、事务控制语句：从开始到结束

MySQL 给我们提供了几个简单命令来控制事务，我们用转账场景完整演示一遍。

### 3.1 准备工作：先建一个账户表

```sql
-- 创建账户表
CREATE TABLE accounts (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50) NOT NULL COMMENT '账户名',
  balance DECIMAL(10,2) NOT NULL DEFAULT 0.00 COMMENT '余额'
) ENGINE=InnoDB COMMENT '账户表';

-- 插入两个账户：A有1000元，B有1000元
INSERT INTO accounts (name, balance) VALUES ('A', 1000.00), ('B', 1000.00);

SELECT * FROM accounts;
```

预期结果：
```
+----+------+---------+
| id | name | balance |
+----+------+---------+
|  1 | A    | 1000.00 |
|  2 | B    | 1000.00 |
+----+------+---------+
```

### 3.2 完整流程：开始 → 执行 → 提交/回滚

```sql
-- 方式一：START TRANSACTION 开始事务
START TRANSACTION;

-- 方式二：BEGIN 是简写，效果一样
-- BEGIN;

-- A 转 100 元给 B
UPDATE accounts SET balance = balance - 100 WHERE name = 'A';
UPDATE accounts SET balance = balance + 100 WHERE name = 'B';

-- 检查结果（你自己在当前窗口看得到修改）
SELECT * FROM accounts;

-- 如果没问题，提交事务 —— 永久生效
COMMIT;

-- 如果有问题，回滚 —— 撤销所有修改，回到转账前
-- ROLLBACK;
```

### 验证方法

你需要打开**两个 MySQL 窗口**来验证：

| 窗口 1（事务发起者） | 窗口 2（旁观者） |
|---------------------|-----------------|
| `BEGIN;` → 执行 `UPDATE` | 执行 `SELECT * FROM accounts;` |
| **看到 A=900, B=1100** | **看到 A=1000, B=1000** ← 事务提交前看不到修改！ |
| 执行 `COMMIT;` | 重新执行 `SELECT * FROM accounts;` |
| 还是 A=900, B=1100 | **现在看到 A=900, B=1100** ← 提交后才可见 |

这就是隔离性的最基本体现。

### 3.3 回滚演示：出错了怎么办

```sql
BEGIN;

-- A 再转 100 给 B
UPDATE accounts SET balance = balance - 100 WHERE name = 'A';
SELECT * FROM accounts;  -- A=800, B=1100

-- 发现输错了，不想转了，全部撤销
ROLLBACK;

SELECT * FROM accounts;  -- 回到 A=900, B=1100
```

回滚之后，一切恢复原样，就像你根本没执行过那两条 `UPDATE`。

## 四、自动提交：你可能一直在用但不知道

MySQL 默认是**自动提交模式**——什么意思呢？

就是说：**你写的每一条 SQL，自动就是一个事务，执行完自动提交。**

看看你当前是不是自动提交：

```sql
SELECT @@autocommit;
```

如果返回 `1`，就是开启自动提交；返回 `0` 就是关闭。

### 4.1 自动提交模式的行为

```sql
-- 自动提交开启（默认）
-- 这条 UPDATE 自动就是一个事务，执行完自动提交
UPDATE accounts SET balance = 950 WHERE name = 'A';
-- 你没法 ROLLBACK 了，因为已经自动提交了
```

如果你要手动控制多个 SQL 在一个事务里，必须先关掉自动提交，或者显式写 `BEGIN`。

### 4.2 关闭自动提交

```sql
-- 关闭自动提交（当前会话有效）
SET autocommit = 0;

-- 现在你写的多条 SQL，都在同一个事务里
UPDATE accounts SET balance = balance - 50 WHERE name = 'A';
UPDATE accounts SET balance = balance + 50 WHERE name = 'B';

-- 必须手动 COMMIT 才会生效
COMMIT;
-- 或者 ROLLBACK 撤销
```

如果打开自动提交，你又写了 `BEGIN`，那么 `BEGIN` 之后到 `COMMIT`/`ROLLBACK` 之间的语句还是一个事务——`BEGIN` 会暂停自动提交，直到你提交或回滚。

## 五、隐式提交：那些会悄悄提交事务的语句

这是一个非常坑的点：**有些语句一执行，就会自动把你当前未提交的事务提交掉，根本不跟你打招呼。**

这个叫**隐式提交（Implicit Commit）** *此术语见附录E*。

### 常见的会触发隐式提交的语句：

| 类型 | 举例 |
|------|------|
| 所有 DDL 语句 | `CREATE TABLE` `ALTER TABLE` `DROP TABLE` `CREATE INDEX` 等 |
| 事务控制语句 | 在 `BEGIN` 里再写一个 `BEGIN` 或 `START TRANSACTION` |
| 锁语句 | `LOCK TABLES` `UNLOCK TABLES` |
| 其他 | `ALTER USER` `GRANT` `REVOKE` `RENAME TABLE` `TRUNCATE` 等 |

### 坑点演示：DDL 会隐式提交

```sql
BEGIN;
UPDATE accounts SET balance = 800 WHERE name = 'A';  -- 扣了 A 100

-- 这一句会触发隐式提交！上面的 UPDATE 已经提交了，回滚不了了！
CREATE TABLE test (id INT);  

ROLLBACK;  -- 没用！A 的 100 已经扣了，回滚不回来了
```

所以记住：**不要在事务中间执行 DDL！** 如果一定要加表结构，等事务提交完再做。

## 六、SAVEPOINT：只回滚一部分，不用全撤销

有时候，你的事务很长，执行了很多操作，走到某一步发现错了，但不想全部回滚——只想回到前面某个点重新来。

这时候可以用 **SAVEPOINT** *此术语见附录E*（保存点）。

### 用法演示

```sql
BEGIN;

-- 第一步：A 转 100 给 B，这步是对的
UPDATE accounts SET balance = balance - 100 WHERE name = 'A';
UPDATE accounts SET balance = balance + 100 WHERE name = 'B';
SAVEPOINT transfer_ab;  -- 在这里打个保存点

-- 第二步：B 转 50 给 C，输错了金额
UPDATE accounts SET balance = balance - 50 WHERE name = 'B';
UPDATE accounts SET balance = balance + 100 WHERE name = 'C';  -- 哦，输错了，应该是 +50

-- 不用回滚整个事务，只回到 transfer_ab 这个保存点
ROLLBACK TO transfer_ab;

-- 重新执行第二步，这次对了
UPDATE accounts SET balance = balance - 50 WHERE name = 'B';
UPDATE accounts SET balance = balance + 50 WHERE name = 'C';

-- 整个事务都对了，提交
COMMIT;
```

SAVEPOINT 就像你玩游戏存了个档，玩死了不用从头开始，读档重来就行。

### 注意事项

- `ROLLBACK TO sp1` 不会结束事务，只是回到保存点继续执行
- 不需要保存点了可以 `RELEASE SAVEPOINT sp1;` 删除它
- 事务提交后，所有保存点自动消失

## 七、验证方法：自己动手测一遍

按照这个顺序操作，验证你理解对了：

1. **打开两个 MySQL 窗口**，都连接到同一个数据库
2. 窗口 1：执行 `BEGIN;` → `UPDATE accounts SET balance = 1000 WHERE name = 'A';`
3. 窗口 2：执行 `SELECT balance FROM accounts WHERE name = 'A';` → 结果还是原来的 900，说明没提交看不到
4. 窗口 1：执行 `COMMIT;`
5. 窗口 2：再次查询 → 现在看到 1000，说明提交后才可见
6. 窗口 1：`BEGIN;` → `UPDATE accounts SET balance = 0 WHERE name = 'A';` → 不要提交
7. 测试隐式提交：窗口 1 执行 `CREATE TABLE temp (id int);` → 这会触发隐式提交
8. 窗口 1：`ROLLBACK;` → 查询 `A` 的余额还是 0，说明已经提交，回滚无效
9. 删除 `temp` 表：`DROP TABLE temp;`

如果你每一步都得到预期结果，说明你理解了。

## 八、常见错误与排错

### 错误 1：忘记 COMMIT，导致锁一直不释放

**现象**：你 `BEGIN` 之后改了数据，一直不 `COMMIT` 也不 `ROLLBACK`，然后就去吃饭了。别的事务要修改这行数据，就会一直卡住，直到锁等待超时。

**原因**：你改的数据加了行锁，不提交锁就不会释放。

**解决**：
- 事务写完尽快提交或回滚
- 如果用命令行，不小心退出了，MySQL 会自动回滚未提交的事务
- 如果是应用程序，一定要确保异常时回滚

**验证**：可以用下面这个命令看当前的锁：
```sql
-- MySQL 8.0+ 查看锁信息
SELECT * FROM performance_schema.data_locks;
```

### 错误 2：在自动提交模式下，以为多条语句在一个事务里

**现象**：你写了：
```sql
-- 自动提交开启（默认）
UPDATE accounts SET balance = balance - 100 WHERE name = 'A';
-- 这里网络断了，第二条没执行
UPDATE accounts SET balance = balance + 100 WHERE name = 'B';
```
结果 A 已经扣钱了，B 没加上——钱少了。

**原因**：自动提交模式下，每条语句都是独立事务，第一条执行完就自动提交了，第二条没执行也不会回滚第一条。

**解决**：多个相关操作要放在一个事务里，必须显式写 `BEGIN` 和 `COMMIT`：
```sql
BEGIN;
UPDATE ... A;
UPDATE ... B;
COMMIT;
```

### 错误 3：不知道 DDL 会隐式提交，导致回滚失败

**现象**：就是我们前面演示的，事务里执行 `CREATE TABLE`，然后发现错了 `ROLLBACK`，结果前面的修改已经提交了，回滚不了。

**解决**：记住——所有 DDL 都会隐式提交。不要把 DDL 和 DML 放在同一个事务里。

### 错误 4：用了 SAVEPOINT，最后忘记 COMMIT

**现象**：你 `ROLLBACK TO sp1` 改完了，然后就走了，事务一直挂着，锁不释放。

**解决**：记住：无论你用不用 SAVEPOINT，最后都要 `COMMIT` 或 `ROLLBACK` 结束事务。

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|--------------|---------|
| 事务定义 | 事务 = 一组要么全成功要么全失败的操作 | 原子性保证不出现中间状态 |
| Atomicity（原子性） | 要么全做，要么全不做 | 比喻：签合同 |

| 保存点 | `SAVEPOINT sp;` / `ROLLBACK TO sp;` | 只回滚部分，不用全撤销 |
