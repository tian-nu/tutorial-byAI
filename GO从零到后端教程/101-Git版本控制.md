# 第101章 · Git版本控制

> "你有没有过这种经历：代码改着改着发现改坏了，但已经Ctrl+S保存了——回不去了。或者你和同事同时改一个文件，合并的时候乱成一团。Git就是解决这些问题的。它让代码的每一次修改都有记录，可以随时回到任何一个历史版本。一个人用它像有个备份机，一群人用它像有张实时同步的地图。"

---

## 101.1 Git是什么：代码的时间机器

### 不用Git的世界有多恐怖

想象你和同事共同维护一份Word文档：
1. 你改了一段，发给同事
2. 同事也改了同一段，发给你
3. 你们互相覆盖了对方的修改
4. 最终版本变成了 `最终版_v3_修订_真正的最终版_2.docx`

Git彻底消灭了这种噩梦。

### Git的核心能力

Git是**分布式版本控制系统**。通俗讲：
- **记录每次修改**：谁、什么时候、改了什么东西
- **随时回退**：改坏了？一秒回到昨天的版本
- **多人协作**：每个人各自在独立分支上改，最后合并
- **分布式**：每个人的电脑上都有完整的代码历史（不依赖中央服务器）

### 类比：Git是代码界的"Ctrl+Z"

区别在于，Ctrl+Z只能回退最近的几步操作，关闭文件后就不能回退了。而Git能回退到代码诞生以来的任何一个历史时刻——即使是一年前的版本。

**暑假作业比喻**：
- 不用Git = 在唯一的作业本上直接写，写错了只能划掉
- 用Git = 每写一页就复印一张存档，写错了好几页可以回到之前某张复印稿重新写；同时还能让同学帮你改另一份复印稿

## 101.2 四区模型：工作区/暂存区/本地仓库/远程仓库

这是理解Git最关键的模型。不理解这四个区，你就永远无法真正理解Git。

```
工作区          暂存区          本地仓库         远程仓库
(Working)    (Staging)       (Local Repo)    (Remote Repo)
  │              │               │               │
  │   git add    │   git commit  │   git push    │
  ├─────────────>├──────────────>├──────────────>│
  │              │               │               │
  │              │               │   git fetch   │
  │              │               │<──────────────│
  │              │               │               │
  │              │   git reset   │               │
  │<─────────────│<──────────────│               │
  │              │               │               │
  │              │               │   git pull    │
  │              │               │<──────────────│
  │   git merge/rebase           │               │
  │<─────────────────────────────│               │
```

### 工作区（Working Directory）

你电脑上的那个文件夹——你实际写代码的地方。`git init` 后，这个文件夹就是工作区。你增删改文件，Git都知道，但还没记录。

### 暂存区（Staging Area / Index）

`git add` 把修改"放"进来。暂存区像一个**购物车**——你挑选了哪些修改要提交到下一版。不想提交的修改可以留在工作区。

### 本地仓库（Local Repository）

`git commit` 把暂存区的修改永久记录进来。本地仓库存在 `.git` 隐藏文件夹里，包含了所有历史版本。

### 远程仓库（Remote Repository）

托管在GitHub/GitLab/Gitee等平台上的仓库。`git push` 把本地仓库的提交推送到远程，`git pull` 把远程的新提交拉到本地。

### 🤔 想多一点：为什么要有个"暂存区"？

初学者常常不理解暂存区的存在必要性——为什么不能直接 commit 工作区的所有修改？

暂存区的设计给了你**精细控制提交粒度**的能力。假设你同时改了一个Bug和加了一个新功能：
1. `git add bug_fix.go` 只暂存Bug修复
2. `git commit -m "fix: 修复空指针异常"` 提交一个干净的Bug修复
3. `git add new_feature.go` 再暂存新功能
4. `git commit -m "feat: 添加用户导出功能"` 提交新功能

两次提交职责清晰，未来找问题或回退时不会互相干扰。这就是**原子性提交**。

## 101.3 基本操作

### git init：初始化仓库

```bash
git init                    # 在当前目录创建.git文件夹
git init myproject          # 创建myproject目录并初始化
```

执行后，目录里多了一个 `.git` 隐藏文件夹——这就是Git的"数据库"。所有版本历史都存这里。删除 `.git` 就等于删除了Git仓库的所有历史（但工作区的代码还在）。

### git clone：克隆远程仓库

```bash
git clone https://github.com/user/repo.git             # HTTPS方式
git clone git@github.com:user/repo.git                 # SSH方式（推荐，配好免密后很方便）
git clone -b dev https://github.com/user/repo.git      # 克隆指定分支
```

克隆做了三件事：
1. 从远程下载所有版本历史到本地仓库
2. 创建工作区拉出最新代码
3. 自动设置 `origin` 指向你克隆的远程地址

### git add：添加到暂存区

```bash
git add file.go              # 添加单个文件
git add *.go                 # 添加所有.go文件
git add .                    # 添加当前目录所有修改
git add -A                   # 添加所有修改（包括删除的文件）
git add -p                   # 交互式选择添加哪些修改块（精确控制！）
```

### git commit：提交到本地仓库

```bash
git commit -m "fix: 修复用户登录的空指针异常"
git commit -am "fix: 修复用户登录的空指针异常"    # -a=自动add所有已跟踪的文件修改
git commit --amend                                 # 修改最近一次提交
```

### git push：推送到远程

```bash
git push origin main        # 推送到origin的main分支
git push -u origin main     # -u=设置上游分支，之后只需git push
git push --force            # 强制推送（危险！覆盖远程，团队协作禁止使用）
```

### git pull：拉取远程更新

```bash
git pull                    # = git fetch + git merge
git pull --rebase           # = git fetch + git rebase（推荐）
```

### git fetch：只下载不合并

```bash
git fetch origin            # 下载远程所有分支的更新，但不合并到当前分支
```

`fetch` 是安全的——你只获取了"远程有什么新东西"，不影响你当前的代码。之后可以 `git merge` 手动合并，或者先用 `git log origin/main` 看看远程多了什么。

### git status：查看当前状态

```bash
git status                  # 显示哪些文件修改了、哪些已暂存、哪些未跟踪
git status -s               # 简短模式
```

输出解读：
```
On branch main
Changes to be committed:        # 已暂存，下次commit会包含
  modified:   src/user.go

Changes not staged for commit:  # 已修改但未暂存
  modified:   src/config.go

Untracked files:                # 新文件，Git还没跟踪
  src/new_feature.go
```

### git diff：查看具体修改

```bash
git diff                    # 工作区 vs 暂存区（还没add的修改）
git diff --staged           # 暂存区 vs 本地仓库（已add但还没commit的修改）
git diff HEAD               # 工作区+暂存区 vs 本地仓库（所有未提交的修改）
git diff main dev           # 比较两个分支
git diff HEAD~1 HEAD        # 比较最近一次提交和当前版本
```

### git log：查看提交历史

```bash
git log                     # 完整日志
git log --oneline           # 一行一条（最常用）
git log --oneline -10       # 最近10条
git log --oneline --graph   # 带分支图
git log --author="Alice"    # 只看Alice的提交
git log --since="2026-01-01"  # 只看某日期之后的
git log -p                  # 显示每次提交的具体修改内容
git log --oneline --graph --all   # 全部分支的可视化图形
```

## 101.4 分支管理

### 分支是什么

分支就是代码的**平行宇宙**。你从主线上分出一条岔路，在上面做实验、写新功能。做好了就合并回去，做废了就删掉——主线丝毫不受影响。

```
main:  A --- B --- C --- F --- G (合并了feature-x)
              \
feature-x:     D --- E
```

### 基本分支操作

```bash
git branch                  # 列出所有本地分支（当前分支前有*）
git branch -a               # 列出所有分支（含远程分支）
git branch feature-login    # 创建feature-login分支（不切换过去）
git checkout feature-login  # 切换到feature-login分支
git checkout -b feature-login  # 创建并切换（等价于上面两条）

git switch feature-login    # Git 2.23+的推荐写法（替代checkout切换分支）
git switch -c feature-login # 创建并切换
```

`checkout` 和 `switch` 的关系：`checkout` 一个命令做了太多事（切换分支+恢复文件），容易混淆。Git 2.23把"切换分支"的功能拆成了 `switch`，"恢复文件"的功能拆成了 `restore`。

### 合并分支：git merge

```bash
git checkout main           # 先切换到要合并入的分支
git merge feature-login     # 把feature-login合并到当前分支(main)
```

**Fast-forward合并（快进）：**
main分支在创建feature-login后没有新提交时发生。直接把main指针移到feature-login的最新位置，历史是线性的。

**三方合并：**
main和feature都有新提交时发生。Git自动创建一个"合并提交"（merge commit），把两边的工作合在一起。

### 冲突解决：完整实战

**场景模拟：** Alice和Bob同时改了 `config.go` 的同一个位置。

Alice的版本（在 `main` 分支）：
```go
var AppName = "MyApp"
```

Bob的版本（在 `feature` 分支）：
```go
var AppName = "SuperApp"
```

Bob做合并：
```bash
git checkout main
git merge feature
```

Git输出：
```
Auto-merging config.go
CONFLICT (content): Merge conflict in config.go
Automatic merge failed; fix conflicts and then commit the result.
```

打开 `config.go`，Git给你标出了冲突：
```go
<<<<<<< HEAD
var AppName = "MyApp"
=======
var AppName = "SuperApp"
>>>>>>> feature
```

- `<<<<<<< HEAD` 到 `=======` 是你当前分支（main）的内容
- `=======` 到 `>>>>>>> feature` 是你要合并进来的分支（feature）的内容
- 你需要**手动选择保留哪个**或**手动写一个新的**

解决冲突——假设最终决定用 `SuperApp`：
```go
var AppName = "SuperApp"
```

删除冲突标记，保存文件，然后：
```bash
git add config.go
git commit -m "merge: 合并feature分支，解决了config.go中的AppName冲突"
```

### 删除分支

```bash
git branch -d feature-login    # 删除已合并的分支（安全）
git branch -D feature-login    # 强制删除（即使没合并）
git push origin --delete feature-login  # 删除远程分支
```

### 解决冲突的最佳实践

1. **冲突前预防**：经常 `git pull`，保持本地和远程同步
2. **小步提交**：每个功能一个分支，做完了马上合并——搁置越久冲突越多
3. **冲突时沟通**：如果冲突涉及别人写的代码，叫上对方一起解决
4. **用可视化工具**：VS Code的合并编辑器、GitHub的Web冲突解决器都比纯手动靠谱
5. **解决后测试**：解决冲突后一定要跑一遍测试，确保没引入问题

## 101.5 高级操作

### rebase：变基

`merge` 产生"合并提交"，历史像一张网。`rebase` 让你的提交历史保持**一条直线**。

**原理：** rebase把你分支上的提交"搬到"目标分支的最新提交之后，仿佛你一开始就是从最新代码开始写的。

```
merge前:
main:   A --- B --- C
          \
feature:   D --- E

merge结果（三方合并）:
main:   A --- B --- C --- M
          \              /
feature:   D --- E -----/

rebase结果:
main:   A --- B --- C --- D' --- E'（D和E被"搬"到了C之后）
```

**操作：**
```bash
git checkout feature
git rebase main
```

如果rebase过程中有冲突，解决冲突后：
```bash
git add .
git rebase --continue     # 继续rebase
git rebase --abort        # 放弃rebase，回到rebase之前的状态
```

**rebase的黄金法则：不要对已推送的公共分支执行rebase！**

原因：rebase会改变提交的哈希值（D变成D'）。如果别人已经基于你的D写了很多代码，你一rebase，他的基础就没了——这会让整个团队抓狂。只在你自己本地的、还没推送的分支上做rebase。

### cherry-pick：挑选提交

把某个提交的修改"复制"到当前分支：

```bash
git cherry-pick abc123     # 把提交abc123的内容应用到当前分支
```

适用场景：你只想把某个Bug修复合并到另一个分支，而不想合并整个feature分支的所有内容。

### stash：暂存工作现场

你正在写代码，突然被叫去修一个紧急Bug——但当前的修改还没写到可以commit的程度。这时用 `stash`：

```bash
git stash                  # 把当前所有修改"暂存"起来，工作区变干净
git stash list             # 查看所有暂存
git stash pop              # 恢复最近的暂存并删除它
git stash apply            # 恢复最近的暂存但不删除
git stash drop             # 删除最近的暂存
git stash save "WIP: 正在写用户注册功能"   # 带备注的暂存
```

**完整工作流：**
```bash
git stash                  # 临时保存当前工作
git checkout main          # 切换到主分支修Bug
git checkout -b hotfix     # 创建修复分支
git commit -am "fix: 修复支付金额计算错误"
git checkout feature       # 切回原分支
git merge hotfix           # 合并修复
git stash pop              # 恢复之前的工作
```

### reset：重置（时光倒流）

```bash
git reset --soft HEAD~1    # 撤销commit，修改回到暂存区（最温和）
git reset --mixed HEAD~1   # 撤销commit和add，修改回到工作区（默认）
git reset --hard HEAD~1    # 撤销一切，包括工作区的修改（最危险！）
```

- `HEAD~1` = 上一个提交，`HEAD~3` = 上三个提交
- `--soft`：只移动HEAD指针，不改变暂存区和工作区
- `--mixed`：移动HEAD + 清空暂存区，工作区不变
- `--hard`：全部重置——提交、暂存区、工作区都回到指定版本

⚠️ `--hard` 不可恢复，使用前三思！

### revert：撤销（不改变历史）

```bash
git revert abc123          # 创建一个新提交，内容是"撤销abc123的修改"
```

`revert` 和 `reset` 的核心区别：
- `reset`：删除历史（假装那些提交从未存在过）
- `revert`：保留历史，追加一个"反操作"提交

对已推送的提交，只能用 `revert` 而不是 `reset`。因为如果别人已经拉取了你的提交，你 `reset` 后再 `push --force` 会把别人的历史弄乱。

## 101.6 .gitignore文件

有些文件不应该被Git跟踪：编译产物、依赖包、IDE配置、日志文件、含密码的环境变量文件等。

在项目根目录创建 `.gitignore`：

```gitignore
# 编译产物
*.exe
*.exe~
*.dll
*.so
*.dylib
myapp

# 测试产物
*.test
*.out

# Go相关
vendor/

# 依赖目录
node_modules/

# 环境变量（含密码，绝对不能提交）
.env
.env.local
*.env

# IDE配置
.idea/
.vscode/
*.swp
*.swo

# 操作系统
.DS_Store
Thumbs.db

# 日志文件
*.log

# 临时文件
tmp/
temp/
```

**注意事项：**
1. `.gitignore` 只对未跟踪的文件生效。如果一个文件已经被Git跟踪了，先要 `git rm --cached 文件名` 再从索引中移除。
2. 全局 `.gitignore`：`git config --global core.excludesfile ~/.gitignore_global`
3. 有一些文件应该提交但被忽略了？用 `git add -f 文件名` 强制添加。

## 101.7 Git工作流

### Git Flow（经典但复杂）

```
main:    ─────●────────────●─────── (发布版本)
               \          /
develop:  ──────●───●───●────── (开发主线)
                \     /
feature:         ●───●  (功能分支)

release:              ●──● (发布准备分支)
hotfix:                    ● (紧急修复分支)
```

**分支类型：**
- `main`：生产环境代码，只通过合并更新
- `develop`：开发主线，feature分支都合并到这
- `feature/*`：新功能开发
- `release/*`：发布前的测试和准备
- `hotfix/*`：生产环境的紧急修复

**优点：** 结构清晰，适合大型项目
**缺点：** 太重了，对小团队来说过度设计

### GitHub Flow（简单实用）

```
main:    ─────●───●─────●──────●────
               \   /     \    /
feature1:       ●─●        (已合并，删除)
feature2:                   ●──● (PR中)
```

只有一个核心分支 `main`：
1. 从 `main` 创建功能分支
2. 在功能分支上开发和提交
3. 发起Pull Request（PR）
4. 代码审查通过后合并到 `main`
5. 立即部署到生产环境

**适合：** 持续部署的项目，小型到中型团队。GitHub自身就是用这个工作流。

### Trunk Based（极简）

所有人都直接往 `main`（trunk）提交。功能通过**特性开关**（feature flag）控制。

**适合：** 成熟的DevOps团队，有完善的自动化测试和部署流水线。Google和Facebook大规模使用这种模式。

### 选择指南

| 团队规模 | 推荐工作流 |
|----------|-----------|
| 个人项目 | GitHub Flow |
| 2~5人小团队 | GitHub Flow |
| 5~20人 | Git Flow（简化版） |
| 20人以上 | Git Flow或Trunk Based |

## 101.8 Commit规范

### Conventional Commits（约定式提交）

格式：`<type>(<scope>): <description>`

```
feat(auth): 添加JWT刷新令牌功能
fix(order): 修复金额计算精度丢失问题
docs(api): 更新用户接口文档
refactor(user): 提取密码验证逻辑为独立函数
test(order): 添加订单取消的单元测试
chore(deps): 升级Go版本到1.22
```

**type类型：**

| Type | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug修复 |
| `docs` | 文档变更 |
| `style` | 代码格式（空格、缩进等，不影响功能） |
| `refactor` | 重构（既非新功能也非Bug修复） |
| `perf` | 性能优化 |
| `test` | 添加或修改测试 |
| `chore` | 构建过程或辅助工具的变动 |
| `ci` | CI/CD配置变更 |
| `build` | 影响构建系统或外部依赖的变更 |

**scope**是可选的，用来指明影响范围（如 `auth`、`order`、`db`）。

### 为什么规范Commit重要

1. **自动生成Changelog**：工具可以按type分类，自动生成发布说明
2. **语义化版本**：`fix` 触发补丁版本(1.0.0→1.0.1)，`feat` 触发次版本(1.0.1→1.1.0)，BREAKING CHANGE触发主版本(1.1.0→2.0.0)
3. **快速定位问题**：`git log --oneline --grep="fix"` 找到所有Bug修复
4. **代码审查友好**：审查者一看commit就知道这个提交在干什么

### 好的Commit信息 vs 坏的

```
❌ 改了一些东西
❌ fix bug
❌ update
❌ WIP
❌ 修复
❌ .

✅ fix(auth): 修复Token过期判断逻辑，增加10秒缓冲时间
✅ feat(api): 添加GET /api/users/:id接口
✅ refactor(db): 将SQL查询移至Repository层
```

好的Commit信息 = 一眼就能看懂改了什么，不需要去看具体代码。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| Git本质 | 分布式版本控制，代码的"时间机器" |
| 四区模型 | 工作区→git add→暂存区→git commit→本地仓库→git push→远程仓库 |
| 暂存区价值 | 精细控制提交粒度，实现原子性提交 |
| 基本操作 | clone/add/commit/push/pull/fetch/status/diff/log |
| 分支 | 平行宇宙，feature分支开发→merge回主线 |
| 冲突解决 | 手动选择保留的内容，沟通+测试验证 |
| merge vs rebase | merge保留真实历史，rebase线性化历史 |
| rebase黄金法则 | 不对已推送的公共分支做rebase |
| cherry-pick | 挑选特定提交到当前分支 |
| stash | 临时保存工作现场 |
| reset vs revert | reset删除历史，revert追加新提交 |
| .gitignore | 排除不应提交的文件 |
| Git Flow | main/develop/feature/release/hotfix |
| GitHub Flow | main + feature分支 + PR |
| Commit规范 | type(scope): description，feat/fix/docs/refactor等 |
| reflog | Git的终极救命稻草，找回"丢失"的提交 |
