## 一、需求分析：电商订单系统到底要存什么？

### 1.1 先想清楚：用户打开一个电商 App，经历了什么？

想象你打开淘宝，浏览商品，加入购物车，下单，填地址，付款。这个流程涉及的数据，就是我们数据库要存的东西。我们先不急着写 SQL，先用白话说清楚每一块存什么、为什么这么存。

### 1.2 六大核心实体

| 实体 | 白话解释 | 存什么 |
|------|---------|--------|
| **用户（User）** | 用 App 的人 | 用户名、密码哈希、手机号、邮箱、注册时间 |
| **商品（Product）** | 卖的东西 | 商品名、描述、价格、库存、分类、上下架状态 |
| **订单（Order）** | 一次购买的记录 | 订单号、用户 ID、总金额、订单状态、支付方式、收货地址快照 |
| **订单明细（OrderItem）** | 订单里每件商品 | 订单 ID、商品 ID、购买数量、购买时单价（快照） |
| **收货地址（Address）** | 用户的收货地址 | 用户 ID、收件人、电话、省市区、详细地址、是否默认 |
| **购物车（Cart）** | 用户的待购清单 | 用户 ID、商品 ID、数量、加入时间 |

### 1.3 想多一点：为什么订单要存"价格快照"？

这是个经典面试题。假设你在订单明细里只存了 `product_id`，每次查订单时去 `products` 表 JOIN 拿价格。三个月后，商品涨价了，你的历史订单显示的金额就变了——用户来找你："我明明付了 99，为什么订单显示 199？"

所以，**订单明细必须冗余存储下单时的价格**（`price_at_order`）。这是"反范式"（第 14 章）的典型应用——牺牲一点存储空间，换取历史数据的准确性。

### 1.4 想多一点：收货地址为什么也要"快照"？

同理。用户下单后可能修改或删除地址。如果订单只存 `address_id`，用户删了地址后，历史订单就找不到收货信息了。所以订单表里直接冗余存储收货地址的完整快照。

## 二、ER 图设计

### 2.1 实体关系

```
用户 (users) ──1:N──> 订单 (orders)
用户 (users) ──1:N──> 收货地址 (addresses)
用户 (users) ──1:N──> 购物车 (cart_items)
订单 (orders) ──1:N──> 订单明细 (order_items)
商品 (products) ──1:N──> 订单明细 (order_items)
商品 (products) ──1:N──> 购物车 (cart_items)
```

### 2.2 关系矩阵

| 实体 A | 实体 B | 关系 | 说明 |
|--------|--------|------|------|
| users | orders | 1:N | 一个用户可以有多个订单 |
| users | addresses | 1:N | 一个用户可以有多个收货地址 |
| users | cart_items | 1:N | 一个用户有一个购物车（多件商品） |
| orders | order_items | 1:N | 一个订单包含多个商品 |
| products | order_items | 1:N | 一个商品可以出现在多个订单中 |
| products | cart_items | 1:N | 一个商品可以被多个用户加入购物车 |

### 2.3 字段清单（只说关键字段，避免啰嗦）

**users 表**：
- `id` BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
- `username` VARCHAR(50) NOT NULL UNIQUE
- `password_hash` CHAR(60) NOT NULL —— bcrypt 固定 60 字符

- `email` VARCHAR(100) NOT NULL UNIQUE

**products 表**：
- `id` BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY

- `description` TEXT
- `price` DECIMAL(12,2) NOT NULL —— 价格用 DECIMAL，不用 FLOAT！
- `stock` INT UNSIGNED NOT NULL DEFAULT 0
- `category_id` INT UNSIGNED —— 商品分类
- `status` TINYINT NOT NULL DEFAULT 1 —— 1:上架 0:下架
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
- `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

**orders 表**：
- `id` BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY

- `status` TINYINT NOT NULL DEFAULT 1 —— 1:待付款 2:已付款 3:已发货 4:已完成 5:已取消
- `payment_method` TINYINT —— 1:支付宝 2:微信 3:银行卡
- `receiver_name` VARCHAR(50) NOT NULL —— 收货人快照
- `receiver_phone` VARCHAR(20) NOT NULL
- `receiver_province` VARCHAR(50)
- `receiver_city` VARCHAR(50)
- `receiver_district` VARCHAR(50)
- `receiver_detail` VARCHAR(255)
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
- `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

**order_items 表**：
- `id` BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
- `order_id` BIGINT UNSIGNED NOT NULL

- `quantity` INT UNSIGNED NOT NULL

**addresses 表**：
- `id` BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
- `user_id` BIGINT UNSIGNED NOT NULL

### 3.2 建表 SQL

```sql
-- ============================================================
-- 1. 用户表
-- ============================================================
CREATE TABLE users (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    password_hash CHAR(60) NOT NULL COMMENT 'bcrypt哈希',
    phone VARCHAR(20) NOT NULL,
    email VARCHAR(100) NOT NULL,
    status TINYINT NOT NULL DEFAULT 1 COMMENT '1:正常 0:禁用',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_username (username),
    UNIQUE KEY uk_phone (phone),
    UNIQUE KEY uk_email (email),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='用户表';

-- ============================================================
-- 2. 商品分类表（先建，因为商品表引用它）
-- ============================================================
CREATE TABLE categories (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    parent_id INT UNSIGNED DEFAULT 0 COMMENT '父分类ID，0表示顶级',
    sort_order INT DEFAULT 0,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_name (name),
    INDEX idx_parent_id (parent_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='商品分类表';

-- ============================================================
-- 3. 商品表
-- ============================================================
CREATE TABLE products (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    description TEXT,
    price DECIMAL(12,2) NOT NULL COMMENT '价格，DECIMAL避免浮点误差',
    stock INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '库存',
    category_id INT UNSIGNED,
    status TINYINT NOT NULL DEFAULT 1 COMMENT '1:上架 0:下架',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_category_id (category_id),
    INDEX idx_status_price (status, price),
    INDEX idx_name (name(50)),
    CONSTRAINT fk_product_category FOREIGN KEY (category_id)
        REFERENCES categories(id) ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='商品表';

-- ============================================================
-- 4. 收货地址表
-- ============================================================
CREATE TABLE addresses (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,
    receiver_name VARCHAR(50) NOT NULL,
    receiver_phone VARCHAR(20) NOT NULL,
    province VARCHAR(50),
    city VARCHAR(50),
    district VARCHAR(50),
    detail VARCHAR(255),
    is_default TINYINT NOT NULL DEFAULT 0 COMMENT '1:默认地址',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_id (user_id),
    CONSTRAINT fk_address_user FOREIGN KEY (user_id)
        REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='收货地址表';

-- ============================================================
-- 5. 购物车表
-- ============================================================
CREATE TABLE cart_items (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,
    product_id BIGINT UNSIGNED NOT NULL,
    quantity INT UNSIGNED NOT NULL DEFAULT 1,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_product (user_id, product_id),
    INDEX idx_user_id (user_id),
    CONSTRAINT fk_cart_user FOREIGN KEY (user_id)
        REFERENCES users(id) ON DELETE CASCADE,
    CONSTRAINT fk_cart_product FOREIGN KEY (product_id)
        REFERENCES products(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='购物车表';

-- ============================================================
-- 6. 订单表
-- ============================================================
CREATE TABLE orders (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    order_no VARCHAR(32) NOT NULL COMMENT '业务订单号',
    user_id BIGINT UNSIGNED NOT NULL,
    total_amount DECIMAL(12,2) NOT NULL COMMENT '订单总金额',
    status TINYINT NOT NULL DEFAULT 1 COMMENT '1:待付款 2:已付款 3:已发货 4:已完成 5:已取消',
    payment_method TINYINT COMMENT '1:支付宝 2:微信 3:银行卡',
    -- 收货地址快照（反范式，防止地址被修改后历史订单信息丢失）
    receiver_name VARCHAR(50) NOT NULL,
    receiver_phone VARCHAR(20) NOT NULL,
    receiver_province VARCHAR(50),
    receiver_city VARCHAR(50),
    receiver_district VARCHAR(50),
    receiver_detail VARCHAR(255),
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_order_no (order_no),
    INDEX idx_user_id (user_id),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at),
    CONSTRAINT fk_order_user FOREIGN KEY (user_id)
        REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='订单表';

-- ============================================================
-- 7. 订单明细表
-- ============================================================
CREATE TABLE order_items (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    order_id BIGINT UNSIGNED NOT NULL,
    product_id BIGINT UNSIGNED NOT NULL,
    product_name VARCHAR(200) NOT NULL COMMENT '商品名快照',
    quantity INT UNSIGNED NOT NULL,
    price_at_order DECIMAL(12,2) NOT NULL COMMENT '下单时价格快照',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_order_id (order_id),
    INDEX idx_product_id (product_id),
    CONSTRAINT fk_orderitem_order FOREIGN KEY (order_id)
        REFERENCES orders(id) ON DELETE CASCADE,
    CONSTRAINT fk_orderitem_product FOREIGN KEY (product_id)
        REFERENCES products(id) ON DELETE RESTRICT
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='订单明细表';
```

### 3.3 建表时我做了哪些设计决策？

| 决策 | 为什么 | 对应教程章节 |
|------|--------|-------------|
| 价格用 `DECIMAL(12,2)` | FLOAT/DOUBLE 有精度误差，算钱必须精确 | 第 12 章 |
| `order_items.price_at_order` 快照 | 防止商品涨价后历史订单金额变化 | 第 14 章（反范式） |
| `orders` 冗余地址字段 | 防止用户删除地址后历史订单信息丢失 | 第 14 章（反范式） |
| 外键 `ON DELETE CASCADE` | 删用户时自动清理关联数据 | 第 16 章 |
| 外键 `ON DELETE RESTRICT` | 有订单引用的商品不允许删除 | 第 16 章 |
| `cart_items` 联合唯一键 | 同一用户同一商品只能有一条购物车记录 | 第 16 章 |
| 所有表 `ENGINE=InnoDB` | 需要事务和外键支持 | 第 24 章 |

## 四、测试数据生成

### 4.1 为什么需要测试数据？

100 条数据看不出性能问题。我们需要模拟真实规模——10 万用户、10 万商品、10 万订单、50 万订单明细。这时候 EXPLAIN 的 rows 字段才有意义，慢查询才会暴露。

### 4.2 存储过程：批量插入

```sql
-- 设置允许函数创建（如果 MySQL 开启了二进制日志，需要此设置）
SET GLOBAL log_bin_trust_function_creators = 1;

DELIMITER $$

-- ============================================================
-- 存储过程：批量插入用户
-- ============================================================
CREATE PROCEDURE insert_users(IN num INT)
BEGIN
    DECLARE i INT DEFAULT 1;
    DECLARE batch_size INT DEFAULT 1000;
    
    WHILE i <= num DO
        INSERT INTO users (username, password_hash, phone, email, status)
        VALUES (
            CONCAT('user_', i),
            '$2a$10$placeholder_hash_60_characters_long_xxxxxxxxxxxx',
            CONCAT('138', LPAD(i, 8, '0')),
            CONCAT('user_', i, '@example.com'),
            1
        );
        SET i = i + 1;
        
        -- 每 1000 条提交一次，减少事务开销
        IF i % batch_size = 0 THEN
            COMMIT;
        END IF;
    END WHILE;
    COMMIT;
END$$

-- ============================================================
-- 存储过程：批量插入商品分类
-- ============================================================
CREATE PROCEDURE insert_categories()
BEGIN
    INSERT INTO categories (name, parent_id, sort_order) VALUES
    ('电子产品', 0, 1),
    ('服装', 0, 2),
    ('食品', 0, 3),
    ('图书', 0, 4),
    ('家居', 0, 5);
    
    INSERT INTO categories (name, parent_id, sort_order) VALUES
    ('手机', 1, 1),
    ('电脑', 1, 2),
    ('耳机', 1, 3),
    ('男装', 2, 1),
    ('女装', 2, 2),
    ('零食', 3, 1),
    ('饮料', 3, 2),
    ('小说', 4, 1),
    ('技术书', 4, 2),
    ('家具', 5, 1),
    ('厨具', 5, 2);
END$$

-- ============================================================
-- 存储过程：批量插入商品
-- ============================================================
CREATE PROCEDURE insert_products(IN num INT)
BEGIN
    DECLARE i INT DEFAULT 1;
    DECLARE cat_id INT;
    
    WHILE i <= num DO
        SET cat_id = 6 + (i % 11);  -- 随机分配到 11 个子分类
        
        INSERT INTO products (name, description, price, stock, category_id, status)
        VALUES (
            CONCAT('商品_', i),
            CONCAT('这是商品_', i, '的详细描述'),
            ROUND(RAND() * 9900 + 100, 2),  -- 价格 100~10000
            FLOOR(RAND() * 10000),           -- 库存 0~9999
            cat_id,
            1
        );
        SET i = i + 1;
        
        IF i % 1000 = 0 THEN
            COMMIT;
        END IF;
    END WHILE;
    COMMIT;
END$$

-- ============================================================
-- 存储过程：批量插入订单和订单明细
-- ============================================================
CREATE PROCEDURE insert_orders(IN num INT)
BEGIN
    DECLARE i INT DEFAULT 1;
    DECLARE v_user_id BIGINT;
    DECLARE v_order_no VARCHAR(32);
    DECLARE v_product_id BIGINT;
    DECLARE v_quantity INT;
    DECLARE v_price DECIMAL(12,2);
    DECLARE v_total DECIMAL(12,2);
    DECLARE v_item_count INT;
    DECLARE v_order_id BIGINT;
    DECLARE j INT;
    
    WHILE i <= num DO
        -- 随机用户
        SET v_user_id = FLOOR(1 + RAND() * 100000);
        -- 生成订单号
        SET v_order_no = CONCAT(
            DATE_FORMAT(NOW(), '%Y%m%d%H%i%s'),
            LPAD(i, 8, '0')
        );
        SET v_total = 0;
        
        -- 插入订单
        INSERT INTO orders (order_no, user_id, total_amount, status,
            receiver_name, receiver_phone, receiver_province, receiver_city,
            receiver_district, receiver_detail)
        VALUES (
            v_order_no,
            v_user_id,
            0,  -- 先填 0，后面更新
            FLOOR(1 + RAND() * 5),
            CONCAT('收货人_', v_user_id),
            CONCAT('138', LPAD(v_user_id, 8, '0')),
            '广东省',
            '深圳市',
            '南山区',
            CONCAT('科技园路', v_user_id, '号')
        );
        
        SET v_order_id = LAST_INSERT_ID();
        
        -- 每个订单随机 1~5 件商品
        SET v_item_count = FLOOR(1 + RAND() * 5);
        SET j = 1;
        WHILE j <= v_item_count DO
            SET v_product_id = FLOOR(1 + RAND() * 100000);
            SET v_quantity = FLOOR(1 + RAND() * 5);
            
            -- 获取商品价格（实际业务中应该用 SELECT INTO）
            SET v_price = (SELECT price FROM products WHERE id = v_product_id);
            IF v_price IS NULL THEN
                SET v_price = 199.00;
            END IF;
            
            INSERT INTO order_items (order_id, product_id, product_name, quantity, price_at_order)
            VALUES (v_order_id, v_product_id, CONCAT('商品_', v_product_id), v_quantity, v_price);
            
            SET v_total = v_total + (v_price * v_quantity);
            SET j = j + 1;
        END WHILE;
        
        -- 更新订单总金额
        UPDATE orders SET total_amount = v_total WHERE id = v_order_id;
        
        SET i = i + 1;
        IF i % 500 = 0 THEN
            COMMIT;
        END IF;
    END WHILE;
    COMMIT;
END$$

DELIMITER ;
```

## 五、核心业务查询

### 5.1 查询一：用户订单列表（最常用查询）

用户在"我的订单"页面看到自己的订单，按时间倒序，分页展示。

```sql
-- 查询用户 ID=100 的订单列表，第 1 页，每页 10 条
SELECT o.id, o.order_no, o.total_amount, o.status, o.created_at,
       COUNT(oi.id) AS item_count
FROM orders o
LEFT JOIN order_items oi ON o.id = oi.order_id
WHERE o.user_id = 100
GROUP BY o.id
ORDER BY o.created_at DESC
LIMIT 0, 10;
```

**验证：EXPLAIN 分析**

```sql
EXPLAIN SELECT o.id, o.order_no, o.total_amount, o.status, o.created_at,
               COUNT(oi.id) AS item_count
FROM orders o
LEFT JOIN order_items oi ON o.id = oi.order_id
WHERE o.user_id = 100
GROUP BY o.id
ORDER BY o.created_at DESC
LIMIT 0, 10;
```

你应该看到：
- `orders` 表走了 `idx_user_id` 索引（type: ref）
- `order_items` 表走了 `idx_order_id` 索引（type: ref）
- 扫描行数很少（rows < 100）

**如果查询慢（> 0.1 秒），检查：**
1. `orders.user_id` 是否有索引？→ 建表时已加 `INDEX idx_user_id (user_id)`
2. `order_items.order_id` 是否有索引？→ 建表时已加 `INDEX idx_order_id (order_id)`
3. 是否因为 `GROUP BY` 导致 `Using temporary`？→ 如果 `created_at` 不在索引中，可以考虑加联合索引 `INDEX idx_user_created (user_id, created_at)`

### 5.2 查询二：商品销量排行（Top N 查询）

运营后台需要看"哪些商品卖得最好"。

```sql
SELECT p.id, p.name, p.price,
       COALESCE(SUM(oi.quantity), 0) AS total_sold,
       COALESCE(SUM(oi.quantity * oi.price_at_order), 0) AS total_revenue
FROM products p
LEFT JOIN order_items oi ON p.id = oi.product_id
LEFT JOIN orders o ON oi.order_id = o.id AND o.status != 5  -- 排除已取消订单
WHERE p.status = 1  -- 只统计上架商品
GROUP BY p.id
ORDER BY total_sold DESC
LIMIT 20;
```

**验证：EXPLAIN 分析**

```sql
EXPLAIN SELECT p.id, p.name, p.price,
               COALESCE(SUM(oi.quantity), 0) AS total_sold
FROM products p
LEFT JOIN order_items oi ON p.id = oi.product_id
LEFT JOIN orders o ON oi.order_id = o.id AND o.status != 5
WHERE p.status = 1
GROUP BY p.id
ORDER BY total_sold DESC
LIMIT 20;
```

**优化要点（对应第 17-20 章知识）：**

如果这个查询慢，很可能是 `order_items` 表全表扫描了。优化思路：

```sql
-- 方案 1：在 order_items 上加覆盖索引（覆盖索引见第 19 章）
CREATE INDEX idx_product_quantity ON order_items(product_id, quantity);

-- 方案 2：如果数据量极大，考虑用汇总表（反范式思路）
-- 创建一个 daily_sales_summary 表，定时任务汇总
```

### 5.3 查询三：月度销售额统计

财务部门需要按月份统计销售额趋势。

```sql
SELECT DATE_FORMAT(o.created_at, '%Y-%m') AS month,
       COUNT(DISTINCT o.id) AS order_count,
       SUM(o.total_amount) AS total_revenue,
       AVG(o.total_amount) AS avg_order_value,
       COUNT(DISTINCT o.user_id) AS unique_customers
FROM orders o
WHERE o.status IN (2, 3, 4)  -- 已付款、已发货、已完成
  AND o.created_at >= '2024-01-01'
GROUP BY DATE_FORMAT(o.created_at, '%Y-%m')
ORDER BY month DESC;
```

**验证：EXPLAIN 分析**

```sql
EXPLAIN SELECT DATE_FORMAT(o.created_at, '%Y-%m') AS month,
               COUNT(DISTINCT o.id) AS order_count,
               SUM(o.total_amount) AS total_revenue
FROM orders o
WHERE o.status IN (2, 3, 4)
  AND o.created_at >= '2024-01-01'
GROUP BY DATE_FORMAT(o.created_at, '%Y-%m')
ORDER BY month DESC;
```

**优化要点：**

如果数据量很大（百万级订单），这个查询的本质是范围扫描 + 分组聚合。优化：

```sql
-- 创建联合索引覆盖 WHERE + GROUP BY
CREATE INDEX idx_status_created ON orders(status, created_at);

-- 如果还慢，考虑：
-- 1. 检查是否 Using filesort（Extra 字段），若是则调整索引列顺序
-- 2. 如果 status IN (...) 导致索引失效，考虑用 UNION ALL
-- 3. 极端场景下用汇总表预计算月度数据
```

### 6.2 模拟一条"慢查询"并优化

故意制造一条不走索引的查询：

```sql
-- ❌ 全表扫描：按商品名模糊搜索，LIKE 前置 % 导致索引失效
SELECT * FROM products WHERE name LIKE '%手机%';
```

```sql
EXPLAIN SELECT * FROM products WHERE name LIKE '%手机%';
-- type: ALL, rows: 100000, Extra: Using where
-- 这很慢！全表扫描 10 万行
```

**优化方案：**

```sql
-- ✅ 方案 1：全文索引（详见第 36 章博客系统）
ALTER TABLE products ADD FULLTEXT INDEX ft_name_desc (name, description);
SELECT * FROM products WHERE MATCH(name, description) AGAINST('手机' IN NATURAL LANGUAGE MODE);

-- ✅ 方案 2：如果搜索场景固定，用前缀 LIKE（可以走索引）
SELECT * FROM products WHERE name LIKE '手机%';  -- 前缀匹配可以走 idx_name 索引
```

## 七、事务演示：下单流程

### 7.1 下单的业务逻辑

一次下单需要做：
1. 检查库存（`SELECT stock FROM products WHERE id = ? AND stock >= ?`）
2. 扣库存（`UPDATE products SET stock = stock - ? WHERE id = ?`）
3. 创建订单（`INSERT INTO orders ...`）
4. 创建订单明细（`INSERT INTO order_items ...`）
5. 清空购物车（`DELETE FROM cart_items WHERE user_id = ? AND product_id = ?`）

这五步必须**要么全成功，要么全失败**。如果扣了库存但订单创建失败，用户的钱没扣但库存少了——这就造成了数据不一致。

### 7.2 用事务包装下单

```sql
-- 开启事务
START TRANSACTION;

-- 步骤 1：检查并扣减库存（行锁保证并发安全，对应第 23 章）
-- 使用 SELECT ... FOR UPDATE 锁定要操作的行
SELECT stock FROM products WHERE id = 1 FOR UPDATE;
-- 假设返回 stock = 50

-- 步骤 2：扣库存
UPDATE products SET stock = stock - 2 WHERE id = 1 AND stock >= 2;

-- 检查 affected rows，如果为 0 说明库存不足
-- SELECT ROW_COUNT(); → 1 表示成功，0 表示库存不足

-- 步骤 3：创建订单（地址快照）
INSERT INTO orders (order_no, user_id, total_amount, status,
    receiver_name, receiver_phone, receiver_province, receiver_city,
    receiver_district, receiver_detail)
VALUES (
    CONCAT('ORD', UNIX_TIMESTAMP(), '001'),
    100,  -- 用户 ID
    358.00,  -- 总金额
    1,  -- 待付款
    '张三', '13800138000', '广东省', '深圳市', '南山区', '科技园路 1 号'
);
SET @order_id = LAST_INSERT_ID();

-- 步骤 4：创建订单明细
INSERT INTO order_items (order_id, product_id, product_name, quantity, price_at_order)
VALUES (@order_id, 1, '商品_1', 2, 179.00);

-- 步骤 5：清空购物车中对应商品
DELETE FROM cart_items WHERE user_id = 100 AND product_id = 1;

-- 全部成功，提交事务
COMMIT;
```

### 7.3 事务回滚演示

```sql
-- 模拟：库存不足时回滚
START TRANSACTION;

-- 尝试扣库存
UPDATE products SET stock = stock - 99999 WHERE id = 1 AND stock >= 99999;
-- ROW_COUNT() = 0 → 库存不足！

-- 回滚事务
ROLLBACK;

-- 验证：库存没有变化
SELECT id, name, stock FROM products WHERE id = 1;
```

### 7.4 事务隔离级别验证（对应第 22 章）

打开两个 MySQL 连接（Session A 和 Session B），模拟并发下单：

**Session A：**
```sql
-- 设置隔离级别为 REPEATABLE READ（MySQL 默认）
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;

-- 读取库存
SELECT stock FROM products WHERE id = 1;  -- 假设返回 50

-- 扣库存
UPDATE products SET stock = stock - 5 WHERE id = 1;
-- 此时不提交，Session B 如果也来读这一行，会被阻塞（行锁）
```

**Session B：**
```sql
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;

-- 尝试读取同一行（会被阻塞，直到 Session A 提交或回滚）
SELECT stock FROM products WHERE id = 1 FOR UPDATE;
-- 如果 Session A 没提交，这里会等待 innodb_lock_wait_timeout 秒（默认 50 秒）
```

**Session A（继续）：**
```sql
COMMIT;
-- Session B 此时解除阻塞，读到的是 Session A 提交后的值
```
