## 我在做什么？

你有没有遇到过这种场景：服务器跑得好好的，突然所有服务都挂了。排查半天，发现是磁盘满了。再查，一个 50GB 的 `json.log` 文件躺在 `/var/lib/docker/containers/` 下面，默默地把你的磁盘吃光了。

Docker 默认会把每个容器的标准输出（stdout）和标准错误（stderr）写到宿主机的一个 JSON 文件里。这个文件**永远不会自动删除，永远不会自动轮转**。只要容器在跑，日志就在长。一个月跑下来，一个不怎么活跃的 Nginx 也能攒出几个 GB。

这一章教你搞定 Docker 的日志管理——从"放任不管，等着磁盘爆炸"到"主动管控，日志该滚滚、该扔扔、该集中收集中收"。学完这一章，你能：
- 理解 Docker 的日志驱动 *此术语见附录C* 机制
- 配置日志轮转，防止磁盘被日志撑爆
- 选择合适的日志驱动（json-file、journald、syslog、fluentd）
- 把日志送到集中式平台（ELK、Loki、CloudWatch）
- 看懂 `docker inspect` 里的日志配置

## 一、Docker 日志的默认行为

### 日志存在哪里？

当你启动一个容器：

```bash
docker run -d --name my-nginx nginx:alpine
```

Docker 会自动把容器的 stdout 和 stderr 收集起来，写入宿主机的一个文件：

```bash
# 默认路径
/var/lib/docker/containers/<container-id>/<container-id>-json.log
```

你可以这样找到它：

```bash
# 查看容器 ID
docker inspect --format='{{.Id}}' my-nginx

# 查看日志文件路径
docker inspect --format='{{.LogPath}}' my-nginx
# /var/lib/docker/containers/abc123.../abc123...-json.log

# 查看日志文件大小
ls -lh $(docker inspect --format='{{.LogPath}}' my-nginx)
# -rw-r----- 1 root root 2.3M Jun  6 10:00 abc123...-json.log
```

### 默认的日志驱动：json-file

Docker 默认使用 `json-file` 日志驱动。它的工作方式是：

```
容器 stdout/stderr
  │
  ▼
Docker Daemon 捕获
  │
  ▼
写入 /var/lib/docker/containers/<id>/<id>-json.log
  │
  │  （每行一条 JSON 记录）
  │  {"log":"127.0.0.1 - - [06/Jun/2026:10:00:00] ...\n","stream":"stdout","time":"2026-06-06T10:00:00.000Z"}
  │  {"log":"2026/06/06 10:00:01 [error] ...\n","stream":"stderr","time":"2026-06-06T10:00:01.000Z"}
  ▼
```

每行日志是一条 JSON，包含 `log`（日志内容）、`stream`（stdout 或 stderr）、`time`（时间戳）。

`docker logs` 命令就是从这个文件读取的：

```bash
docker logs my-nginx
# 等价于读取 /var/lib/docker/containers/<id>/<id>-json.log
```

### 问题：日志不轮转

默认的 `json-file` 驱动**不对日志做任何限制**。没有大小上限，没有轮转，没有自动清理。一个容器跑一年，日志文件长到 100GB 也不奇怪。

```bash
# 看看你的日志文件多大
du -sh /var/lib/docker/containers/*/
# 50G  /var/lib/docker/containers/abc123/
# 30G  /var/lib/docker/containers/def456/
# 15G  /var/lib/docker/containers/ghi789/
# 😱
```

> **想多一点**：Docker 为什么默认不做日志轮转？因为 Docker 的设计哲学是"最小惊讶原则"——它不知道你的日志重要不重要。有些人的日志是唯一的数据来源（比如审计日志），自动删了就是灾难。有些人的日志只是调试信息，丢了也无所谓。Docker 把这个决策权交给你，但代价是：如果你不做决策，默认就是"永远不删"。这就像一个房间，房东不给你配垃圾桶——你可以选择自己买一个，但如果你不买，垃圾就会堆满地。

## 二、配置日志轮转

### 容器级别配置（`docker run` 时指定）

```bash
docker run -d \
  --name my-nginx \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  nginx:alpine
```

| 参数 | 含义 | 说明 |
|------|------|------|
| `--log-opt max-size=10m` | 单个日志文件最大 10MB | 达到 10MB 后自动轮转 |
| `--log-opt max-file=3` | 最多保留 3 个日志文件 | 轮转时旧文件被删除 |

轮转后的文件命名：

```
/var/lib/docker/containers/<id>/
├── <id>-json.log          ← 当前日志文件（正在写入）
├── <id>-json.log.1        ← 上一个轮转文件
└── <id>-json.log.2        ← 最旧的轮转文件（再轮转就删了）
```

### 全局默认配置（Daemon 级别）

如果你想让**所有容器**默认启用日志轮转，编辑 `/etc/docker/daemon.json`：

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
```

然后重启 Docker：

```bash
sudo systemctl restart docker
```

> **注意**：`daemon.json` 的配置只对**新创建的容器**生效。已经在运行的容器不受影响。要让旧容器也生效，需要重建容器。

### Docker Compose 中配置

```yaml
# compose.yml
services:
  web:
    image: nginx:alpine
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
        labels: "app,env"      # 可选：只记录特定标签的日志
```

### 验证配置是否生效

```bash
# 查看容器的日志配置
docker inspect --format='{{json .HostConfig.LogConfig}}' my-nginx | python -m json.tool
```

输出：

```json
{
    "Type": "json-file",
    "Config": {
        "max-file": "3",
        "max-size": "10m"
    }
}
```

## 三、不只是 json-file：其他日志驱动

Docker 支持多种日志驱动，每种适用于不同的场景。

### 日志驱动一览

| 驱动 | 适用场景 | 特点 |
|------|---------|------|
| `json-file` | 默认，本地开发 | 简单，`docker logs` 可用 |
| `journald` | 使用 systemd 的 Linux 系统 | 日志进入 systemd journal，`journalctl` 查看 |
| `syslog` | 传统运维，有 syslog 服务器 | 日志发送到 syslog 服务 |
| `fluentd` | 需要集中式日志收集 | 日志发送到 Fluentd，再转发到 ES/Loki 等 |
| `gelf` | 使用 Graylog 的团队 | 日志发送到 Graylog |
| `awslogs` | AWS 上的容器 | 日志直接发送到 CloudWatch Logs |
| `gcplogs` | GCP 上的容器 | 日志发送到 Google Cloud Logging |
| `splunk` | 使用 Splunk 的企业 | 日志发送到 Splunk |
| `none` | 不需要日志 | 关闭日志收集 |

### 示例：用 journald

```bash
# 启动容器，使用 journald 日志驱动
docker run -d \
  --name my-nginx \
  --log-driver journald \
  --log-opt tag="nginx-{{.Name}}" \
  nginx:alpine

# 用 journalctl 查看
journalctl CONTAINER_NAME=my-nginx
```

### 示例：用 fluentd 发送到集中式平台

```bash
# 先启动一个 fluentd 容器
docker run -d -p 24224:24224 \
  -v /path/to/fluentd.conf:/fluentd/etc/fluent.conf \
  fluent/fluentd

# 启动应用容器，日志发送到 fluentd
docker run -d \
  --name my-app \
  --log-driver fluentd \
  --log-opt fluentd-address=localhost:24224 \
  --log-opt tag="docker.my-app" \
  my-app:latest
```

### 查看当前系统支持的日志驱动

```bash
docker info --format '{{.Plugins.Log}}'
# [awslogs fluentd gcplogs gelf journald json-file local splunk syslog]
```

## 四、`docker logs` 的局限

`docker logs` 很方便，但在生产环境有几个致命问题：

### 1. 不持久

```bash
# 容器删了，docker logs 就没了
docker rm my-nginx
docker logs my-nginx
# Error: No such container: my-nginx
```

日志文件在 `/var/lib/docker/containers/<id>/` 下，容器删除时这个目录也被删除。如果你用 `docker-compose down` 或 `docker rm`，日志就永远消失了。

### 2. 不方便搜索

```bash
# docker logs 只能按时间顺序看，不能搜索
docker logs my-nginx
# 输出几千行，你要找 "error" 相关的内容，只能人肉 grep

# 可以加管道，但效率低
docker logs my-nginx | grep "error"
# 每次都要把全部日志读出来再过滤
```

### 3. 多容器没法聚合

```bash
# 你有 10 个 nginx 实例，不能用一条命令看所有日志
docker logs nginx-1
docker logs nginx-2
docker logs nginx-3
# ... 需要 10 条命令，且没法按时间排序
```

### 4. 性能问题

```bash
# 日志文件很大时，docker logs 会很慢
docker logs nginx-with-10gb-logs
# 可能要等几十秒，甚至把 Docker Daemon 卡住
```

## 五、生产环境的日志方案

### 推荐架构

```
┌──────────────────────────────────────────────────┐
│                    容器集群                        │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐           │
│  │ 容器 A   │  │ 容器 B   │  │ 容器 C   │           │
│  │ stdout   │  │ stdout   │  │ stdout   │           │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘           │
│       │              │              │                │
│       ▼              ▼              ▼                │
│  ┌─────────────────────────────────────────────┐    │
│  │         日志驱动（fluentd / journald）         │    │
│  └─────────────────────┬───────────────────────┘    │
└────────────────────────┼────────────────────────────┘
                         │
                         ▼
              ┌─────────────────────┐
              │   日志收集器（Agent）  │
              │  Fluentd / Logstash  │
              │  Promtail / Vector   │
              └──────────┬──────────┘
                         │
                         ▼
              ┌─────────────────────┐
              │   日志存储与查询      │
              │  Elasticsearch      │
              │  Loki               │
              │  CloudWatch Logs    │
              └──────────┬──────────┘
                         │
                         ▼
              ┌─────────────────────┐
              │   可视化与告警        │
              │  Kibana / Grafana   │
              │  CloudWatch 告警    │
              └─────────────────────┘
```

### 轻量方案：Loki + Promtail + Grafana

如果你觉得 ELK 太重，Loki 是更轻量的选择：

```yaml
# compose.yml — Loki 日志收集方案
services:
  # 日志收集器
  promtail:
    image: grafana/promtail:latest
    volumes:
      - /var/lib/docker/containers:/var/lib/docker/containers:ro
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - ./promtail-config.yml:/etc/promtail/config.yml
    command: -config.file=/etc/promtail/config.yml

  # 日志存储
  loki:
    image: grafana/loki:latest
    ports:
      - "3100:3100"

  # 可视化
  grafana:
    image: grafana/grafana:latest
    ports:
      - "3000:3000"
    environment:
      GF_SECURITY_ADMIN_PASSWORD: your_admin_password_here
```

在 Grafana 里添加 Loki 数据源，就可以搜索、过滤、可视化所有容器的日志了。

### 云平台方案

如果你在 AWS 上，用 `awslogs` 驱动直接把日志送到 CloudWatch：

```bash
docker run -d \
  --name my-app \
  --log-driver awslogs \
  --log-opt awslogs-region=us-east-1 \
  --log-opt awslogs-group=/docker/my-app \
  --log-opt awslogs-create-group=true \
  my-app:latest
```

## 六、实战：从日志爆炸到日志可控

### 场景：模拟日志爆炸

```bash
# 创建一个疯狂输出日志的容器
docker run -d --name log-bomb alpine:latest \
  sh -c 'while true; do echo "Log line $(date)"; done'

# 等 30 秒
sleep 30

# 查看日志文件大小
ls -lh $(docker inspect --format='{{.LogPath}}' log-bomb)
# 可能已经几十 MB 了！
```

### 修复：配置日志轮转

```bash
# 停止并删除旧容器
docker stop log-bomb && docker rm log-bomb

# 重新启动，配置日志轮转
docker run -d \
  --name log-bomb \
  --log-opt max-size=5m \
  --log-opt max-file=2 \
  alpine:latest \
  sh -c 'while true; do echo "Log line $(date)"; done'

# 等一段时间，日志文件不会超过 5MB × 2 = 10MB
watch -n 5 'ls -lh $(docker inspect --format="{{.LogPath}}" log-bomb)'
```

### 验证清单


## 八、不对怎么办？

### 常见错误 1：只配了容器级别，没配全局默认

```bash
# ❌ 每个容器都要手动加 --log-opt
docker run -d --name app1 --log-opt max-size=10m --log-opt max-file=3 app1
docker run -d --name app2 --log-opt max-size=10m --log-opt max-file=3 app2
docker run -d --name app3 --log-opt max-size=10m --log-opt max-file=3 app3
# ... 漏了一个，那个就成了定时炸弹
```

✅ 修正：在 `daemon.json` 配全局默认，容器级别可覆盖：

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
```

### 常见错误 2：日志驱动选错了，没法切换

```bash
# ❌ 容器已经在运行，想换日志驱动
docker run -d --log-driver json-file --name my-app my-app:latest
# 跑了几天，想换成 fluentd...
docker run -d --log-driver fluentd --name my-app my-app:latest
# 不行，容器名冲突，而且旧日志在新驱动下看不到
```

✅ 日志驱动只能在**创建容器时**指定，运行时不能改。要换驱动，必须重建容器：

```bash
docker stop my-app
docker rm my-app
docker run -d --log-driver fluentd --name my-app my-app:latest
```

### 常见错误 3：`max-size` 设太小

```bash
# ❌ 日志轮转过于频繁
docker run -d --log-opt max-size=1m --log-opt max-file=1 my-app
# 应用日志量大的话，每分钟都在轮转，IO 开销大，且只保留 1MB 日志
```

✅ 根据应用的日志量设置合理的值：

```bash
# 一般应用
docker run -d --log-opt max-size=10m --log-opt max-file=3 my-app

# 日志量大的应用（如高流量 Nginx）
docker run -d --log-opt max-size=50m --log-opt max-file=5 my-app
```

### 常见错误 4：日志驱动和 `docker logs` 不兼容

```bash
# ❌ 用了 journald 驱动，还习惯用 docker logs
docker run -d --log-driver journald --name my-app my-app
docker logs my-app
# Error: "logs" command is supported only for "json-file" and "journald" logging drivers
# 等等，journald 也支持？是的，但输出格式不同
```

`docker logs` 只对 `json-file`、`journald` 和 `local` 驱动有效。用了 `fluentd`、`syslog` 等驱动后，`docker logs` 返回空。

### 常见错误 5：不清除旧日志，积累到磁盘满

```bash
# ❌ 配置了轮转，但旧日志文件从来不删
# 轮转只删 JSON 日志文件，但如果你换了驱动，旧文件还在
```

✅ 定期清理旧日志：

```bash
# 查看所有容器的日志文件大小
du -sh /var/lib/docker/containers/*/

# 安全清理：只清空不删除文件（避免容器还在写打开的文件句柄出错）
truncate -s 0 /var/lib/docker/containers/<id>/<id>-json.log

# 或者，如果容器已经不在了，直接删
find /var/lib/docker/containers/ -name "*-json.log" -mtime +30 -delete
```

## 九、术语解释

| 术语 | 解释 |
|------|------|
| **日志驱动（Logging Driver）** *此术语见附录C* | Docker 的插件机制，决定容器的 stdout/stderr 输出写到哪。可以是本地文件（json-file）、系统日志（journald、syslog）或远程服务（fluentd、awslogs） |
| **json-file** *此术语见附录C* | Docker 默认日志驱动，每行一条 JSON 记录写入宿主机文件。支持 `docker logs` 查看，但默认不轮转 |
| **journald** *此术语见附录C* | systemd 的日志系统，日志写入二进制 journal 文件，用 `journalctl` 查看。在大多数现代 Linux 发行版上默认运行 |
| **log-opt** *此术语见附录C* | Docker 日志驱动的配置选项，如 `max-size`、`max-file`。不同驱动支持的选项不同 |
| **fluentd** *此术语见附录C* | 开源日志收集器，支持从多种来源收集日志，统一处理后转发到多种后端。Docker 内置了 fluentd 日志驱动 |

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 默认日志行为 | `json-file` 驱动，日志在 `/var/lib/docker/containers/<id>/<id>-json.log` | 默认不轮转，不设上限 |
| 日志轮转 | `--log-opt max-size=10m --log-opt max-file=3` | 在 `docker run` 或 `daemon.json` 中配置 |
| 全局默认配置 | `/etc/docker/daemon.json` 中的 `log-driver` 和 `log-opts` | 只对新容器生效，旧容器需重建 |
| Compose 中配置日志 | `logging:` 下的 `driver` 和 `options` | 和 `docker run` 的 `--log-opt` 对应 |
| 其他日志驱动 | `journald` / `syslog` / `fluentd` / `awslogs` | 选驱动后 `docker logs` 可能失效 |
| `docker logs` 局限 | 不持久（容器删了日志就没了）、不方便搜索、多容器无法聚合 | 生产环境需要集中式日志平台 |
| 集中式日志方案 | Loki + Promtail + Grafana / ELK / CloudWatch | 日志从容器 → 收集器 → 存储 → 可视化 |
| 清理旧日志 | `truncate -s 0` 清空日志文件 | 不要直接 `rm` 正在写的日志文件 |
| 日志驱动切换 | 只能在创建容器时指定，不能运行时改 | 要换驱动必须重建容器 |

## 本篇最可能出错的地方及原因

### 1. 默认不轮转，日志悄悄撑爆磁盘（最高频）

**这是生产环境最常见的 Docker 故障之一。** 开发者用 `docker run` 或 `docker compose up` 启动服务后就不管了，几个月后服务器磁盘满了，所有服务停摆。

**根因**：Docker 的 `json-file` 驱动默认不设日志大小上限。每个容器一条日志写到天荒地老。

**预防**：**每台服务器必须在 `/etc/docker/daemon.json` 配置全局默认日志轮转。** 这是部署 Docker 服务器的标准操作，不是可选项。

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
```

### 2. `docker logs` 在生产环境查日志，把 Daemon 卡住

**原因**：`docker logs` 是同步操作，如果日志文件有几个 GB，它会阻塞 Docker Daemon 的 API 请求处理。同时其他 `docker ps`、`docker start` 等命令都会卡住。

**正确做法**：用 `tail` 直接读日志文件，或者用 `docker logs --tail 100` 只看最近 100 行。生产环境不要用 `docker logs` 查全量日志——那应该交给集中式日志平台。

### 3. 换了日志驱动后发现 `docker logs` 不输出了

**原因**：`docker logs` 只支持 `json-file`、`journald` 和 `local` 三种驱动。用了 `fluentd`、`syslog` 等驱动后，`docker logs` 返回空。

**预防**：在选择日志驱动前，确认团队是否依赖 `docker logs`。如果依赖，要么不用其他驱动，要么提前告知团队改用其他方式查日志。

### 4. `daemon.json` 配置不生效，因为没重启 Daemon

```bash
# ❌ 改了配置没重启
sudo vim /etc/docker/daemon.json
# 然后直接 docker run，发现没生效

# ✅ 必须重启 Docker Daemon
sudo systemctl restart docker
```

修改 `daemon.json` 后必须重启 Docker 服务。而且只对新创建的容器生效——已经在运行的容器不受影响。

### 5. 日志轮转配置了，但磁盘还是满了

**原因**：日志轮转只对 `json-file` 驱动生效。如果你用的是 `journald` 或 `fluentd`，日志轮转由对应的服务（systemd journal、fluentd）负责，Docker 的 `--log-opt max-size` 不起作用。此外，容器可能在其他地方写日志（比如应用自己写文件到 `/var/log/app.log`），那些不受 Docker 日志驱动管理。

**排查**：用 `du -sh /var/lib/docker/*` 和 `du -sh /var/log/*` 全面排查，确定是哪个目录在吃磁盘。
