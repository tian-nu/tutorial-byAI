## 我在做什么？

容器在 Docker 的虚拟网络里（172.17.x.x）运行得很开心，但外部的你、你的浏览器、你的客户端程序——它们不在 Docker 网络里，无法直接访问容器的 IP。

你需要一个**门**：外面的人敲宿主机的一个端口，Docker 把它转送到容器内部的端口。这个"门"就是端口映射 *此术语见附录C*。

学完这一章，你能：
- 用 `-p` 把宿主机端口映射到容器端口
- 用 `-P` 随机映射所有 EXPOSE 端口
- 理解端口映射的底层原理（docker-proxy *此术语见附录C*）
- 给 Nginx 和 MySQL 分别映射不同端口
- 诊断端口冲突和安全问题

## 一、`-p`：开一扇门

### 语法

```bash
docker run -p 宿主机端口:容器端口 镜像名
#           ─────┬───── ──┬──
#           外部访问   容器内监听
```

### 最简单的例子

```bash
docker run -d --name my-nginx -p 8080:80 nginx:alpine
```

拆解：
- `-p 8080:80`：宿主机 8080 端口 → 容器 80 端口
- 外部访问 `http://localhost:8080` → Docker 转发到容器内的 Nginx（监听 80）

验证：

```bash
curl http://localhost:8080
# 返回 Nginx 欢迎页 HTML
```

### 映射多个端口

```bash
docker run -d --name multi-port \
  -p 8080:80 \
  -p 8443:443 \
  -p 3306:3306 \
  nginx:alpine
```

每个 `-p` 开一扇门。你可以映射任意数量的端口。

### 指定监听地址

```bash
# 只监听本地回环（推荐：安全）
docker run -p 127.0.0.1:8080:80 nginx:alpine
# 只有本机可以访问 localhost:8080

# 绑定特定 IP
docker run -p 192.168.1.100:8080:80 nginx:alpine
# 只有通过 192.168.1.100 这个 IP 才能访问

# 不指定 IP（默认 0.0.0.0，所有网卡）
docker run -p 8080:80 nginx:alpine
# 任何能访问宿主机的设备都能连 8080
```

### 绑定 UDP 端口

```bash
# 默认是 TCP，如果需要 UDP
docker run -p 8080:80/udp nginx:alpine

# TCP 和 UDP 都需要
docker run -p 8080:80/tcp -p 8080:80/udp nginx:alpine
```

## 二、`-P`：一次性映射所有 EXPOSE 端口

如果你在 Dockerfile 里用 `EXPOSE` 声明了端口：

```dockerfile
FROM nginx:alpine
EXPOSE 80
EXPOSE 443
```

那么 `-P`（大写 P）会**随机映射**所有 EXPOSE 端口到宿主机的高端口（32768-60999）：

```bash
docker run -d --name auto-ports -P nginx:alpine

docker port auto-ports
# 80/tcp -> 0.0.0.0:32768
# 443/tcp -> 0.0.0.0:32769
```

### `-p` vs `-P` 对比

| 选项 | 行为 | 适用场景 |
|------|------|---------|
| `-p 8080:80` | 精确指定宿主端口 | 日常开发、生产部署 |
| `-p 127.0.0.1:8080:80` | 只绑本地，最安全 | 数据库、管理后台 |
| `-P` | 随机分配所有 EXPOSE | CI/CD 临时环境，避免端口冲突 |
| `-p 8080-8090:80` | 端口范围映射 | 多个同类容器 |

## 三、实战：同时映射 Nginx 和 MySQL

```bash
# 创建自定义网络
docker network create app-net

# MySQL：映射到 3307（避免和宿主机 MySQL 冲突）
docker run -d \
  --name db \
  --network app-net \
  -p 127.0.0.1:3307:3306 \
  -e MYSQL_ROOT_PASSWORD=your_root_password_here \
  mysql:8.0

# Nginx：映射到 8080
docker run -d \
  --name web \
  --network app-net \
  -p 8080:80 \
  nginx:alpine
```

### 验证

```bash
# 验证 Nginx
curl http://localhost:8080
# 返回 Nginx 欢迎页

# 验证 MySQL（需要 mysql 客户端）
mysql -h 127.0.0.1 -P 3307 -uroot -pyour_root_password_here -e "SELECT 1;"
# +---+
# | 1 |
# +---+
# | 1 |
# +---+
```

### 查看端口映射状态

```bash
docker port web
# 80/tcp -> 0.0.0.0:8080

docker port db
# 3306/tcp -> 127.0.0.1:3307
```

## 四、端口映射的底层原理：docker-proxy

当你执行 `docker run -p 8080:80` 时，发生了什么？

### 第一步：docker-proxy 启动

Docker 启动一个叫 `docker-proxy` *此术语见附录C* 的进程，它监听宿主机的 8080 端口。所有到 `0.0.0.0:8080` 的流量都会被这个进程接收。

### 第二步：流量转发

docker-proxy 把收到的数据包转发到容器的 80 端口。路径是：

```
外部请求 → 宿主机 eth0:8080 → docker-proxy → docker0 网桥 → 容器 eth0:80
```

### 第三步：iptables 规则

除了 docker-proxy，Docker 还会在 iptables 里添加 DNAT 规则，让内核直接转发流量（绕过 docker-proxy，性能更好）。docker-proxy 主要作为回退方案——当内核转发不可用时（如某些网络配置），docker-proxy 兜底。

### 想多一点：为什么需要两个转发路径？

Docker 最初只用 docker-proxy（用户态转发，性能差但可靠）。后来加了 iptables DNAT（内核态转发，性能好但依赖内核配置）。两者并存，互为备份。如果你在宿主机上执行 `ps aux | grep docker-proxy`，能看到每个端口映射都有一个 docker-proxy 进程。

## 五、常见错误

### 错误 1：宿主机端口被占用

```bash
# ❌ 8080 端口已经被宿主机的某个进程占用
docker run -p 8080:80 nginx:alpine
# Error: driver failed programming external connectivity on endpoint:
# Bind for 0.0.0.0:8080 failed: port is already allocated
```

✅ 排查：

```bash
# 查看谁占用了 8080
# Linux / macOS
sudo lsof -i :8080

# Windows
netstat -ano | findstr :8080

# 解决方案：
# 1. 换一个端口：docker run -p 8081:80 nginx:alpine
# 2. 停掉冲突进程
# 3. 如果冲突进程在另一个 Docker 容器里，停掉那个容器
```

### 错误 2：防火墙阻止端口

```bash
docker run -p 8080:80 nginx:alpine
curl http://localhost:8080  # ✅ 本机可以
# 但同事的电脑访问 http://你的IP:8080 → 超时
```

❌ 原因：防火墙（iptables / firewalld / ufw）阻止了 8080 端口的入站连接。

✅ 解决：

```bash
# Linux firewalld
sudo firewall-cmd --add-port=8080/tcp --permanent
sudo firewall-cmd --reload

# Linux ufw
sudo ufw allow 8080/tcp

# Linux iptables（直接添加规则）
sudo iptables -A INPUT -p tcp --dport 8080 -j ACCEPT
```

> ⚠️ Docker 通常会自动修改 iptables 规则让映射的端口可访问。但如果你手动管理防火墙（如 ufw），Docker 的规则可能被覆盖。这时需要手动放行。

### 错误 3：映射到 0.0.0.0 的安全风险

```bash
# ❌ 危险：数据库端口暴露给整个网络
docker run -p 3306:3306 mysql:8.0
# 任何能 ping 通你宿主机的人都能连你的 MySQL
```

✅ 安全做法：

```bash
# ✅ 只绑本地
docker run -p 127.0.0.1:3306:3306 mysql:8.0
# 只有本机能访问，外部完全看不到

# ✅ 或者不映射端口，只在自定义网络内通信
docker run --network app-net mysql:8.0
# 数据库不暴露任何端口给外部，只有同网络的容器能访问
```

### 错误 4：容器内端口和映射端口搞反

```bash
# ❌ docker run -p 80:8080 nginx:alpine
# 你以为：访问 localhost:80 → 容器内 8080
# 实际上：Nginx 监听的是 80，不是 8080！
# 结果：访问 localhost:80 没反应
```

✅ 口诀：`-p 外面:里面`。外面是宿主机的，里面是容器的。先想清楚容器内程序监听哪个端口，再决定外面用什么端口。

### 错误 5：`-P` 不知道映射到哪个端口

```bash
docker run -d --name test -P nginx:alpine
# 但访问 http://localhost 不通
```

`-P` 是随机端口，你需要查看映射结果：

```bash
docker port test
# 80/tcp -> 0.0.0.0:32768

# 正确访问方式
curl http://localhost:32768
```

## 六、我做得对不对？

### 验证方法

```bash
# 1. 确认端口映射
docker port web
# 预期：80/tcp -> 0.0.0.0:8080

docker port db
# 预期：3306/tcp -> 127.0.0.1:3307

# 2. 验证 Nginx 可访问
curl -s -o /dev/null -w "%{http_code}" http://localhost:8080
# 预期：200

# 3. 验证 MySQL 可连接
mysql -h 127.0.0.1 -P 3307 -uroot -pyour_root_password_here -e "SELECT 1;"
# 预期：返回 1

# 4. 确认 docker-proxy 在运行
docker exec web cat /proc/1/cmdline | tr '\0' ' '
# 查看容器内进程

# 5. 在宿主机上查看 docker-proxy 进程（Linux）
ps aux | grep docker-proxy
# 预期：每个端口映射对应一个 docker-proxy 进程
```

## 本篇最可能出错的地方及原因

### 1. 端口映射方向搞反——`-p 80:8080` 还是 `-p 8080:80`？

这是新手最常犯的错误。记住：`-p 外面:里面`。外面是你能访问到的端口，里面是容器内程序监听的端口。Nginx 默认监听 80，所以 `-p 8080:80`。如果你写成 `-p 80:8080`，访问 `localhost:80` 没反应，因为容器里没有程序监听 8080。

### 2. 数据库端口暴露到 0.0.0.0

```bash
docker run -p 3306:3306 mysql:8.0
```

你只是想本机开发测试，但 `-p 3306:3306` 等于对所有网络接口开放了 3306 端口。如果宿主机有公网 IP，你的 MySQL 直接暴露在公网上。**数据库端口永远用 `-p 127.0.0.1:3307:3306`。**

### 3. 端口被占用但不知道谁占的

`docker run` 报 `port is already allocated`，你 `docker ps` 看了一圈没有容器在用 8080。其实是宿主机某个非 Docker 进程占用了 8080。**用 `lsof -i :8080`（Linux/macOS）或 `netstat -ano | findstr :8080`（Windows）找出真凶。**

### 4. `-P` 映射了端口但不知道映射到哪里

```bash
docker run -d -P nginx:alpine
# 然后：curl http://localhost
# 不通！因为不是 80
```

`-P` 是随机端口，永远记得先用 `docker port 容器名` 查看映射结果。

### 5. 防火墙悄无声息地阻止了端口

你一切都做对了，`docker port` 也显示正确，但外部访问就是超时。排查链：
1. `curl http://localhost:端口` → 本机能不能通？
2. 本机能通 → 问题在防火墙
3. `sudo firewall-cmd --list-all` 或 `sudo ufw status` 查看防火墙状态
4. 放行对应端口或暂时关闭防火墙测试

### 6. docker-proxy 进程过多导致资源问题

如果你映射了 100 个端口，就会有 100 个 docker-proxy 进程。每个进程消耗少量内存和文件描述符。在极端场景下（如微服务网格），这可能导致资源问题。解决方案：使用 Docker Swarm 或 Kubernetes 的 Ingress 模式，它们用更高效的方式处理端口转发。
