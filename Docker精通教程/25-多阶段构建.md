## 我在做什么？

你写了一个 Go 程序，编译完只有 10MB。但你把它打进 Docker 镜像，镜像竟然有 600MB。六百兆，就为了跑一个十兆的程序。你感觉就像买了一个 10 块钱的东西，却付了 600 块的包装费。

这一章教你**多阶段构建** *此术语见附录C*——用两个 FROM 把"编译"和"运行"分开：第一阶段用大镜像编译，第二阶段只把编译好的程序复制到一个小镜像里。最终镜像只包含运行所需的最小文件，体积暴降 60 倍。

学完这一章，你能：
- 理解为什么单阶段构建会产生臃肿镜像
- 写出带 `FROM ... AS builder` 的多阶段 Dockerfile
- 用 `COPY --from=builder` 把第一阶段产物复制到第二阶段
- 把 Go、Java、前端项目的镜像体积缩小 10～60 倍

## 一、问题：镜像为什么这么大？

### 先看一个真实的例子

假设你有一个 Go 程序，源码只有一个文件：

```go
// main.go
package main

import "fmt"

func main() {
    fmt.Println("Hello from Docker!")
}
```

编译它：

```bash
go build -o hello main.go
ls -lh hello
# -rwxr-xr-x  1 user  staff   7.2M  hello
```

编译出来 7.2MB。一个简单的 Hello World，7MB 是 Go 运行时的大小，合理。

### 按你之前学的方式写 Dockerfile

```dockerfile
# ❌ 单阶段构建
FROM golang:1.22
WORKDIR /app
COPY main.go .
RUN go build -o hello main.go
CMD ["./hello"]
```

```bash
docker build -t hello-single .
docker images hello-single
# REPOSITORY      TAG       IMAGE ID       SIZE
# hello-single    latest    abc123def456   820MB
```

**820MB！** 怎么从 7MB 变成了 820MB？

### 镜像里到底装了什么？

```bash
docker history hello-single
```

你会看到每一层的大小：

```
IMAGE          CREATED BY                          SIZE
abc123def456   CMD ["./hello"]                     0B
def456abc123   RUN go build -o hello main.go       7.2MB
ghi789def456   COPY main.go .                      72B
jkl012ghi789   WORKDIR /app                        0B
mno345jkl012   FROM golang:1.22                    812MB
```

根源在 `FROM golang:1.22`——这个镜像 812MB，因为它包含了整个 Go SDK、编译器、标准库源码、Git、gcc 等等。你编译只需要它，但运行 `./hello` 完全不需要这些。就像你请了 100 个搬家工人来搬一个背包——搬完以后，100 个工人也跟着住进了你的新家。

> **想多一点**：镜像臃肿不只是"浪费点磁盘空间"的问题。它影响镜像推送和拉取的速度（CI/CD 流水线里每次都要等几分钟），影响容器启动速度，影响滚动更新的效率，而且更大的镜像意味着更大的攻击面——镜像里多出来的每一个库、每一个二进制都可能是一个未被发现的漏洞入口。在 Kubernetes 集群里，一个 800MB 的镜像要按节点数拉取，40 个节点就是 32GB 的带宽开销。

## 二、多阶段构建：分离编译和运行

### 核心思想

```
第一阶段（builder）：用大镜像 → 编译 → 产出二进制文件
第二阶段（runtime）：用小镜像 → 只复制二进制文件 → 最终镜像
```

### 多阶段 Dockerfile

```dockerfile
# ===== 第一阶段：编译 =====
FROM golang:1.22 AS builder
WORKDIR /app
COPY main.go .
RUN go build -o hello main.go

# ===== 第二阶段：运行 =====
FROM alpine:latest
WORKDIR /app
COPY --from=builder /app/hello .
CMD ["./hello"]
```

### 逐行解读

```dockerfile
FROM golang:1.22 AS builder
```
- `AS builder`：给这个阶段起个名字，叫 `builder`。后面用 `--from=builder` 引用它。

```dockerfile
COPY --from=builder /app/hello .
```
- `--from=builder`：这是多阶段构建的关键语法。正常情况下 `COPY` 从宿主机（构建上下文）复制文件；加了 `--from=builder` 后，Docker 会从**第一阶段产生的镜像**里面复制文件，而不是从宿主机。

### 构建并对比

```bash
# 多阶段构建
docker build -t hello-multi .
docker images hello-multi
# REPOSITORY      TAG       IMAGE ID       SIZE
# hello-multi     latest    xyz789abc123   13.5MB

# 对比单阶段
docker images | grep hello
# hello-single    latest    abc123def456   820MB
# hello-multi     latest    xyz789abc123   13.5MB
```

从 820MB 降到 13.5MB。减少了 **98%**。

### 13.5MB 里有什么？

```
alpine 基础镜像    ~7.5MB
hello 二进制文件    ~7.2MB
/alpine 的层开销    ~0.5MB
─────────────────────────
总计              ~13.5MB
```

没有 Go SDK、没有源码、没有编译器、没有 gcc。只有 alpine 和你的程序，干干净净。

> **想多一点**：你可能会想：alpine 还要 7.5MB，能不能更小？能。Go 编译时可以加 `CGO_ENABLED=0` 编译出纯静态二进制，然后用 `FROM scratch`——一个完全空的镜像（0 字节）。最终镜像只有你的程序，大约 7MB。但 `scratch` 没有 shell、没有 CA 证书、没有时区数据，你的程序需要自己处理这些。多阶段构建的厉害之处在于，"用小镜像"这个决策是独立的——你可以随时把 `FROM alpine` 换成 `FROM scratch` 或 `FROM debian:slim`，不影响编译阶段。

## 三、不只是 Go：多阶段构建的通用模式

多阶段构建不是 Go 的专利。任何"编译/构建"和"运行"需要不同依赖的场景，都可以用。

### 前端项目：npm build → nginx

```dockerfile
# ===== 第一阶段：构建前端 =====
FROM node:20-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build
# 产物在 /app/dist 目录

# ===== 第二阶段：用 nginx 托管 =====
FROM nginx:alpine
# 把构建产物复制到 nginx 的静态文件目录
COPY --from=builder /app/dist /usr/share/nginx/html
# 你的 nginx 默认配置就够了
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

效果对比：

| 阶段 | 镜像 | 大小 |
|------|------|------|
| 单阶段（node:20） | `node:20` + `node_modules` + 源码 | ~1.2GB |
| 多阶段（nginx:alpine） | `nginx:alpine` + 只有 dist 里的静态文件 | ~45MB |

**node_modules 没有进最终镜像。** 几百 MB 的 `node_modules` 只在编译时需要，运行 nginx 完全不需要它们。

### Java 项目：Maven 编译 → JRE 运行

```dockerfile
# ===== 第一阶段：编译 =====
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests
# 产物在 /app/target/myapp.jar

# ===== 第二阶段：运行 =====
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/myapp.jar .
EXPOSE 8080
CMD ["java", "-jar", "myapp.jar"]
```

效果：

| 阶段 | 基础镜像内容 | 大小 |
|------|-------------|------|
| 单阶段 | Maven + JDK + 源码 + 依赖缓存 | ~700MB |
| 多阶段 | 只有 JRE（无 JDK，无 Maven，无源码） | ~220MB |

> 注意区别：JDK（Java Development Kit）包含编译器 javac、调试工具等，JRE（Java Runtime Environment）只有运行环境。多阶段构建让你用 JDK 编译、用 JRE 运行，省掉 400MB+ 的编译工具。

## 四、多个 build stage

你不仅限于两个阶段。可以有三个、四个、任意多个。

### 场景：前端 + 后端一起构建

```dockerfile
# ===== 第一阶段：构建前端 =====
FROM node:20-alpine AS frontend-builder
WORKDIR /frontend
COPY frontend/package.json frontend/package-lock.json ./
RUN npm ci
COPY frontend/ .
RUN npm run build
# 产物：/frontend/dist

# ===== 第二阶段：构建后端 =====
FROM golang:1.22 AS backend-builder
WORKDIR /backend
COPY backend/ .
RUN go build -o server .

# ===== 第三阶段：运行 =====
FROM alpine:latest
WORKDIR /app
# 从后端 builder 复制二进制
COPY --from=backend-builder /backend/server .
# 从前端 builder 复制静态文件
COPY --from=frontend-builder /frontend/dist ./static
EXPOSE 8080
CMD ["./server"]
```

一个镜像里同时包含了编译好的 Go 后端和构建好的前端静态文件。三个不同的 base image（node、golang、alpine），各司其职，最终镜像只有 alpine + 你的 server + 静态文件。

### 无名阶段（不用 AS 命名）

阶段可以不命名。如果你只用 `COPY --from=0`（数字索引），第一个阶段是 0，第二个是 1：

```dockerfile
FROM golang:1.22
WORKDIR /app
COPY main.go .
RUN go build -o hello main.go

FROM alpine:latest
COPY --from=0 /app/hello .    # --from=0 引用第一个阶段（索引从 0 开始）
CMD ["./hello"]
```

但**强烈建议用 `AS` 命名**。`--from=0` 可读性差，而且如果你在后面加新阶段，数字索引会变——`--from=builder` 就不会。

## 五、实战：从 600MB 到 10MB 的完整演示

### 准备 Go 程序

```bash
mkdir go-multi-stage && cd go-multi-stage
```

```go
// main.go
package main

import (
    "fmt"
    "net/http"
)

func main() {
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintf(w, "Hello from a 10MB Docker image!")
    })
    fmt.Println("Server starting on :8080")
    http.ListenAndServe(":8080", nil)
}
```

### 单阶段 Dockerfile（对比用）

```dockerfile
# Dockerfile.single
FROM golang:1.22
WORKDIR /app
COPY main.go .
RUN go build -o server main.go
EXPOSE 8080
CMD ["./server"]
```

```bash
docker build -f Dockerfile.single -t goapp-single .
docker images goapp-single
# goapp-single   latest   ...   820MB
```

### 多阶段 Dockerfile

```dockerfile
# Dockerfile
FROM golang:1.22 AS builder
WORKDIR /app
COPY main.go .
# CGO_ENABLED=0 编译纯静态二进制，才能用 FROM scratch
RUN CGO_ENABLED=0 go build -o server main.go

FROM alpine:latest
# 或 FROM scratch（极致精简，但需要 CA 证书等额外处理）
WORKDIR /app
COPY --from=builder /app/server .
EXPOSE 8080
CMD ["./server"]
```

```bash
docker build -t goapp-multi .
docker images goapp-multi
# goapp-multi    latest   ...   13.5MB
```

### 验证两个镜像都能正常工作

```bash
# 测试单阶段镜像
docker run -d -p 8080:8080 --name goapp-single goapp-single
curl http://localhost:8080
# Hello from a 10MB Docker image!
docker stop goapp-single && docker rm goapp-single

# 测试多阶段镜像
docker run -d -p 8080:8080 --name goapp-multi goapp-multi
curl http://localhost:8080
# Hello from a 10MB Docker image!
docker stop goapp-multi && docker rm goapp-multi
```

两个镜像功能完全一样，但大小差了 60 倍。

## 七、不对怎么办？

### 常见错误 1：`COPY --from=builder` 但第一阶段没写 `AS builder`

```dockerfile
FROM golang:1.22
# ❌ 忘了写 AS builder
WORKDIR /app
COPY main.go .
RUN go build -o hello main.go

FROM alpine:latest
COPY --from=builder /app/hello .   # 报错：找不到名为 builder 的阶段
```

```bash
# 错误信息
# ERROR: failed to solve: failed to compute cache key:
# failed to calculate checksum of ref ...:
# "/app/hello": not found
```

✅ 修正：

```dockerfile
FROM golang:1.22 AS builder   # 加上 AS builder
```

### 常见错误 2：第二阶段 COPY 的路径写错

第一阶段的工作目录是 `/app`，编译产物在 `/app/hello`。如果你在第二阶段写：

```dockerfile
# ❌ 错误：路径和第一阶段不一致
COPY --from=builder /hello /hello
# 第一阶段编译产物在 /app/hello，不是 /hello
```

```bash
# 错误信息
# ERROR: failed to solve: failed to compute cache key:
# "/hello": not found
```

✅ 修正：

```dockerfile
COPY --from=builder /app/hello /hello
```

**排查方法**：在第一阶段加一个 `RUN ls -la` 确认文件位置：

```dockerfile
FROM golang:1.22 AS builder
WORKDIR /app
COPY main.go .
RUN go build -o hello main.go
RUN ls -la /app         # 构建时能看到文件列表，确认 hello 在 /app 下
```

### 常见错误 3：第一阶段的 base image 太大

```dockerfile
# ❌ 第一阶段用了不必要的大镜像
FROM ubuntu:latest AS builder
RUN apt-get update && apt-get install -y golang-go
# 第一阶段的镜像本身就很大，虽然最终镜像不受影响，但构建变慢
```

✅ 更优：

```dockerfile
FROM golang:1.22-alpine AS builder   # alpine 版本比完整版小很多
```

第一阶段的镜像**不影响最终镜像大小**（最终镜像只从第二阶段 `FROM` 开始），但影响构建速度——每次构建都要拉取第一阶段的基础镜像。用 `golang:alpine` 比 `golang:latest` 快很多。

### 常见错误 4：忘了加 `--from` 参数

```dockerfile
FROM alpine:latest
# ❌ 错误：没有 --from=builder
COPY /app/hello /hello
```

这会把**宿主机当前目录**下的 `/app/hello` 复制进镜像，而不是从第一阶段复制。如果宿主机上不存在这个文件，构建会报错 "file not found"。

✅ 修正：

```dockerfile
COPY --from=builder /app/hello /hello
```

### 常见错误 5：第二阶段基础镜像不兼容

```dockerfile
FROM golang:1.22 AS builder
RUN CGO_ENABLED=1 go build -o hello main.go   # CGO_ENABLED=1，依赖 glibc

FROM alpine:latest                             # alpine 用 musl libc，不是 glibc
COPY --from=builder /app/hello /hello
CMD ["/hello"]
```

```bash
docker run hello
# /hello: not found
# 或
# error while loading shared libraries: libc.so.6: cannot open shared object file
```

✅ 两种修正方式：

方式一（推荐）：编译纯静态二进制：

```dockerfile
FROM golang:1.22 AS builder
RUN CGO_ENABLED=0 go build -o hello main.go    # 编译纯静态，不依赖任何 libc
```

方式二：用 glibc 兼容的运行镜像：

```dockerfile
FROM debian:bookworm-slim                       # debian 使用 glibc
COPY --from=builder /app/hello /hello
CMD ["/hello"]
```

## 八、术语解释

| 术语 | 解释 |
|------|------|
| **多阶段构建** *此术语见附录C* | 一个 Dockerfile 里写多个 `FROM`，每个 FROM 开始一个新阶段。前一阶段的文件可以通过 `COPY --from=阶段名` 复制到后一阶段 |
| **build stage** *此术语见附录C* | 多阶段构建中的每一个 `FROM ... AS xxx` 块就是一个 build stage。每个 stage 可以使用完全不同的 base image |
| **`--from`** *此术语见附录C* | `COPY --from=阶段名` 的语法，从指定 build stage 复制文件，而不是从宿主机复制 |
| **`AS`** *此术语见附录C* | `FROM image AS 名称`，给 build stage 命名，方便后续引用 |
| **scratch** | Docker 的保留镜像名，代表一个完全空的镜像（0 字节）。适合运行纯静态二进制程序 |
| **CGO_ENABLED=0** | Go 编译选项，禁用 CGO，编译出纯静态二进制，不依赖系统的 libc |

## 本篇最可能出错的地方及原因

### 1. `COPY --from` 的源路径写错

**这是最高频的错误。** 第一阶段里文件在 `/app/hello`，但第二阶段写 `COPY --from=builder /hello /hello`——少写了 `/app`。Docker 的报错信息不会告诉你"第一阶段有这个文件，但路径不对"，它只会说 "not found"，让你以为文件根本没生成。

**排查**：在第一阶段末尾加 `RUN ls -la /app`（或者 `RUN ls -laR /app` 递归列出），构建时确认文件确实在指定路径。

### 2. 第一阶段 base image 太大，构建慢

**原因**：`FROM golang:latest`（完整版）而不是 `FROM golang:alpine`（精简版）。虽然第一阶段不进入最终镜像，但每次构建都要拉取它。在 CI/CD 里，这可能导致构建时间增加 2～3 分钟。

**预防**：优先用 alpine 版本的构建镜像，如 `golang:1.22-alpine`、`node:20-alpine`。

### 3. CGO 导致的 glibc/musl 不兼容

**原因**：Go 编译时用了 CGO（默认行为），编译出的二进制依赖系统的 glibc。但 alpine 用的是 musl libc，不是 glibc。运行时报错 "file not found" 或 "cannot open shared object file"。

**解决**：`CGO_ENABLED=0 go build`。如果你的程序确实需要 CGO（比如调用了 C 库），那就用 `FROM debian:slim` 而不是 `FROM alpine`。Debian 使用 glibc，和 Go 编译环境一致。

### 4. 忘了 `--from`，COPY 从宿主机复制

**原因**：写法 `COPY /app/hello /hello` 而不是 `COPY --from=builder /app/hello /hello`。这条命令会从宿主机当前目录找 `/app/hello`，找不到就报错。

**症状**：如果宿主机上恰好有 `/app/hello` 这个文件，构建不会报错，但镜像里是宿主机上的文件而不是第一阶段编译的文件——功能可能不对，但错误很难发现。

**预防**：写完 Dockerfile 后，用 `docker history` 检查每一层的内容，确认 `COPY` 的源来自正确的阶段。
