## 我在做什么？

第 15 章你画了 ER 图，写了建表 SQL。你注意到了没有——每条 `CREATE TABLE` 语句里除了列出列名和数据类型，还有一堆 `PRIMARY KEY`、`FOREIGN KEY`、`NOT NULL`、`UNIQUE`、`DEFAULT`。这些就是**约束（Constraint）** *此术语见附录E*。

约束是数据库的"免疫系统"——它不帮你查数据，但帮你**拦截坏数据**。就像你家的门禁系统：正常访客扫码通过，陌生人刷不开门。约束干的就是这个：合法数据放行，违规数据拒绝。

但约束也有代价。**外键约束** *此术语见附录E* 每次 INSERT/UPDATE 子表时都要去父表做检查，在高并发场景下，这个检查可能成为性能瓶颈。本章会量化这些代价，帮你做出正确的取舍。

学完这一章，你能：
- 理解并正确使用所有 6 种约束
- 设置外键的级联删除规则（CASCADE / SET NULL / RESTRICT）
- 测试约束违反场景，理解数据库的拒绝机制
- 在"数据库层约束"和"应用层约束"之间做正确选择

## 一、数据完整性：三个维度

在讲具体约束之前，先理解约束保护的到底是什么。**数据完整性** *此术语见附录E* 有三个维度：

| 维度 | 含义 | 比喻 | 对应约束 |
|------|------|------|---------|
| 实体完整性 | 每行数据有唯一标识，不能重复 | 每人有独一无二的身份证号 | PRIMARY KEY, UNIQUE |
| 参照完整性 | 表间引用关系必须有效 | 订单里的用户ID必须对应一个真实用户 | FOREIGN KEY |
| 域完整性 | 每列的值必须在规定范围内 | 年龄不能是负数，不能是"abc" | NOT NULL, CHECK, DEFAULT, 数据类型 |

如果你分别在 Excel 里管理用户表和订单表，用户删了但订单没删，就会出现"幽灵订单"（订单指向一个不存在的用户）。这就是参照完整性被破坏。约束就是防止这种问题的。

## 二、PRIMARY KEY（主键约束）

**PRIMARY KEY** *此术语见附录E* 是数据库中最基础的约束，你在第 04 章已经用过了。但这里要深入讲两个你之前没想过的选择。

### 2.1 PRIMARY KEY 的三个特性

```sql
CREATE TABLE demo_pk (
    id INT PRIMARY KEY
);
-- 等价于：
CREATE TABLE demo_pk (
    id INT NOT NULL UNIQUE
);
-- 但 PRIMARY KEY 还附带一个隐藏特性：自动成为聚集索引（第 17 章详解）
```

PRIMARY KEY 自动获得：
1. **NOT NULL**：主键不能为空
2. **UNIQUE**：主键值不能重复
3. **每表只能有一个 PRIMARY KEY**（但可以有多个 UNIQUE）

### 2.2 自增主键 vs UUID 主键

这是日常开发中第一个需要认真思考的选择：

```sql
-- 方案 A：自增整数主键
CREATE TABLE users_auto (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50)
);

-- 方案 B：UUID 主键
CREATE TABLE users_uuid (
    id CHAR(36) PRIMARY KEY DEFAULT (UUID()),
    username VARCHAR(50)
);
```

|  | 自增 INT | UUID |
|------|------|------|
| 存储大小 | 4 字节 | 36 字符（CHAR）或 16 字节（BINARY） |
| 顺序性 | 递增，新数据追加到末尾 | 随机，新数据插入到任意位置 |
| 索引效率 | 高（顺序插入，无页分裂） | 低（随机插入，频繁页分裂） |
| 分布式友好 | 差（多台机器可能产生相同 ID） | 好（不同机器生成不冲突） |
| 可猜测性 | 易（`/users/100` 可能猜到下一个是 101） | 难（UUID 不可猜测） |
| 业务含义 | 无（纯技术标识） | 无（纯技术标识） |

**选型建议**：
- 单机应用、内部系统 → 自增 INT（简单高效）
- 分布式系统、需要合并数据 → UUID（注意用 `BINARY(16)` 存储以节省空间）
- 需要排序、对外暴露的 ID → 雪花算法（Snowflake，趋势递增，兼顾分布式和性能，第 28 章详述）

```sql
-- 如果必须用 UUID，建议存为 BINARY(16) 而非 CHAR(36)
CREATE TABLE users_uuid_better (
    id BINARY(16) PRIMARY KEY DEFAULT (UUID_TO_BIN(UUID())),
    username VARCHAR(50)
);
-- 存储空间从 36 字节降到 16 字节，索引效率也更高
```

> **想多一点**：为什么 UUID 主键性能差？这涉及 B+树索引原理（第 17 章详述）。简单版解释：B+树的叶子节点是有序链表，自增主键的新数据直接追加到链表末尾，干净利落。UUID 的新数据插入到链表中间，可能触发**页分裂**——一个数据页满了，劈成两半，数据重新分配。分裂操作有磁盘 I/O 开销，频繁分裂会使索引碎片化，查询变慢。**所以你在第 04 章建表时用的 `INT AUTO_INCREMENT` 是正确选择，不要轻易换成 UUID。**

### 2.3 联合主键

```sql
-- 当单列无法唯一标识一行时，使用联合主键
CREATE TABLE order_items (
    order_id   INT,
    product_id INT,
    quantity   INT,
    PRIMARY KEY (order_id, product_id)  -- 联合主键
);
-- 同一订单中同一商品只能出现一次
```

## 三、FOREIGN KEY（外键约束）

**FOREIGN KEY** *此术语见附录E* 是维护表间引用关系的约束。它确保子表的某列值必须在父表的主键或唯一键中存在。

### 3.1 基本语法

```sql
CREATE TABLE orders (
    id      INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    amount  DECIMAL(10,2),
    -- 外键约束：user_id 必须在 users 表的 id 列中存在
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

### 3.2 级联操作：ON DELETE / ON UPDATE

当父表的数据被删除或更新时，子表怎么反应？有 5 种选项：

```sql
CREATE TABLE orders (
    id      INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    amount  DECIMAL(10,2),
    FOREIGN KEY (user_id) REFERENCES users(id)
        ON DELETE CASCADE    -- 父表行被删，子表相关行自动删除
        ON UPDATE CASCADE    -- 父表主键被更新，子表外键自动更新
);
```

| 选项 | 行为 | 使用场景 |
|------|------|---------|
| `CASCADE` *此术语见附录E* | 父表删/改，子表跟着删/改 | 订单明细 → 订单删了，明细没意义，级联删除 |
| `SET NULL` | 父表删/改，子表外键设为 NULL | 文章 → 作者删了，文章保留，作者设为 NULL |
| `RESTRICT` | 禁止父表删/改（如果子表有引用） | 用户 → 有订单的用户不能删，防止误操作 |
| `NO ACTION` | 同 RESTRICT，但检查时机不同（延迟检查） | 与 RESTRICT 几乎相同，MySQL 中等价 |
| `SET DEFAULT` | 设为默认值 | MySQL 不支持（InnoDB 忽略此选项） |

### 3.3 实战演示：CASCADE 的威力

```sql
-- 搭建测试环境
USE my_shop;

-- 创建测试用表
CREATE TABLE test_users (
    id   INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50)
);

CREATE TABLE test_orders (
    id      INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT,
    amount  DECIMAL(10,2),
    FOREIGN KEY (user_id) REFERENCES test_users(id)
        ON DELETE CASCADE
);

-- 插入测试数据
INSERT INTO test_users (name) VALUES ('张三'), ('李四');
INSERT INTO test_orders (user_id, amount) VALUES
(1, 100.00), (1, 200.00), (2, 300.00);

-- 查看数据
SELECT * FROM test_users;
-- +----+------+
-- | id | name |
-- +----+------+
-- |  1 | 张三 |
-- |  2 | 李四 |
-- +----+------+

SELECT * FROM test_orders;
-- +----+---------+--------+
-- | id | user_id | amount |
-- +----+---------+--------+
-- |  1 |       1 | 100.00 |
-- |  2 |       1 | 200.00 |
-- |  3 |       2 | 300.00 |
-- +----+---------+--------+

-- 删除张三
DELETE FROM test_users WHERE id = 1;

-- 再看订单表——张三的订单自动消失了！
SELECT * FROM test_orders;
-- +----+---------+--------+
-- | id | user_id | amount |
-- +----+---------+--------+
-- |  3 |       2 | 300.00 |
-- +----+---------+--------+
```

### 3.4 外键的性能代价（量化说明）

外键不是免费的。每次向子表 INSERT 或 UPDATE 外键列时，MySQL 会：

1. 在父表上执行一次隐式 `SELECT` 查找，确认引用的值存在
2. 如果父表没有对应的索引，这次查找可能变成全表扫描
3. 在父表引用行上加共享锁（SELECT 也受影响）

**量化数据**：
- 单条 INSERT 子表数据，外键检查增加约 **10-20%** 的写入耗时
- 批量 INSERT 时，影响更明显（每行都要检查）
- 如果父表的外键列没有索引，删除父表行时检查子表引用可能触发全表扫描

```sql
-- 验证：对比有无外键的写入性能
-- 创建两张结构相同但一张有外键、一张没有的表

CREATE TABLE parent (id INT PRIMARY KEY);
INSERT INTO parent VALUES (1), (2), (3);

CREATE TABLE child_no_fk (
    id       INT AUTO_INCREMENT PRIMARY KEY,
    parent_id INT
);

CREATE TABLE child_with_fk (
    id       INT AUTO_INCREMENT PRIMARY KEY,
    parent_id INT,
    FOREIGN KEY (parent_id) REFERENCES parent(id)
);

-- 向两张表插入 10000 条数据，对比时间
-- （在 MySQL 客户端中执行，观察耗时差异）
```

**建议**：
- 中低并发场景（< 1000 QPS 写入）→ 放心使用外键，数据安全优先
- 高并发场景（> 5000 QPS 写入）→ 考虑移除外键，在应用层用事务保证一致性
- 无论如何，**外键列必须建索引**（MySQL 会自动为外键列建索引，但如果你手动建表时跳过了，记得补上）

### 3.5 常见错误：忘了 ON DELETE 导致无法删除

```sql
-- ❌ 没有 ON DELETE 选项，默认是 RESTRICT
CREATE TABLE orders (
    id      INT PRIMARY KEY,
    user_id INT,
    FOREIGN KEY (user_id) REFERENCES users(id)
    -- 默认 ON DELETE RESTRICT，禁止删除有订单的用户
);

-- 删除用户时...
DELETE FROM users WHERE id = 1;
-- ERROR 1451: Cannot delete or update a parent row:
-- a foreign key constraint fails (`my_shop`.`orders`)

-- ✅ 解决方案取决于业务需求：
-- 方案 A：先手动删除关联订单
DELETE FROM orders WHERE user_id = 1;
DELETE FROM users WHERE id = 1;

-- 方案 B：建表时用 ON DELETE CASCADE
-- 方案 C：建表时用 ON DELETE SET NULL（如果业务允许）
```

## 四、UNIQUE（唯一约束）

**UNIQUE** *此术语见附录E* 保证某列（或某几列的组合）的值在表中不重复。

### 4.1 基本用法

```sql
CREATE TABLE users (
    id       INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL,
    email    VARCHAR(100) NOT NULL,
    phone    VARCHAR(20),
    UNIQUE KEY uk_username (username),   -- 列级唯一约束
    UNIQUE KEY uk_email (email),         -- 命名规范：uk_ 前缀
    UNIQUE KEY uk_phone (phone)          -- 手机号也唯一
);
```

### 4.2 UNIQUE 与 NULL 的关系

```sql
-- UNIQUE 列允许多个 NULL！
INSERT INTO users (username, email, phone) VALUES
('alice', 'alice@test.com', '13800001111'),
('bob',   'bob@test.com',   NULL),
('carol', 'carol@test.com', NULL);  -- 两个 NULL，不冲突！
```

**原因**：MySQL 中，NULL 表示"未知"，两个未知值无法比较，所以不视为重复。这是 SQL 标准的规定，不是 MySQL 的 bug。如果你需要"手机号要么为空要么唯一"，`UNIQUE` 天然满足这个需求。

### 4.3 联合唯一约束

```sql
-- 购物车：同一用户不能重复添加同一商品
CREATE TABLE cart_items (
    user_id    INT,
    product_id INT,
    quantity   INT,
    UNIQUE KEY uk_user_product (user_id, product_id)  -- 联合唯一
);
-- 插入重复组合会报错
INSERT INTO cart_items VALUES (1, 100, 1);
INSERT INTO cart_items VALUES (1, 100, 2);  -- ❌ Duplicate entry '1-100'
```

### 4.4 UNIQUE vs PRIMARY KEY

|  | PRIMARY KEY | UNIQUE |
|------|------|------|
| 每表数量 | 只能有 1 个 | 可以有多个 |
| NULL 值 | 不允许 | 允许（多个 NULL） |
| 自动建索引 | 是（聚集索引） | 是（二级索引） |
| 语义 | "这是行的唯一标识" | "这个值不能重复" |

## 五、NOT NULL（非空约束）

**NOT NULL** *此术语见附录E* 是最简单的约束：该列不能存 NULL。

```sql
CREATE TABLE users (
    id       INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL,   -- 用户名必填
    email    VARCHAR(100) NOT NULL,  -- 邮箱必填
    bio      TEXT,                   -- 个人简介选填（允许 NULL）
    age      INT DEFAULT 0           -- 年龄默认 0
);

-- 插入时省略 username → 报错
INSERT INTO users (email) VALUES ('test@test.com');
-- ERROR 1364: Field 'username' doesn't have a default value

-- 正确的做法：要么给值，要么设置 DEFAULT
INSERT INTO users (username, email) VALUES ('test', 'test@test.com');
```

**为什么 NULL 这么重要？** 回顾第 07 章的知识：`WHERE col = NULL` 永远为假，`COUNT(col)` 忽略 NULL。NULL 是 SQL 里最容易出 bug 的概念之一。**能用 NOT NULL 就用 NOT NULL，把 NULL 留给真正"可选"的字段。**

## 六、CHECK（检查约束）

**CHECK** *此术语见附录E* 约束让数据在插入/更新前通过自定义条件检查。

### 6.1 基本用法

```sql
-- MySQL 8.0.16+ 才真正支持 CHECK（之前版本语法不报错但实际不生效）
CREATE TABLE employees (
    id   INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    age  INT CHECK (age >= 18 AND age <= 65),
    salary DECIMAL(10,2) CHECK (salary > 0)
);

-- 插入非法数据 → 被拒绝
INSERT INTO employees (name, age, salary) VALUES ('张三', 16, 5000);
-- ERROR 3819: Check constraint 'employees_chk_1' is violated.
```

### 6.2 常见 CHECK 约束场景

```sql
-- 评分范围
CREATE TABLE reviews (
    id     INT PRIMARY KEY,
    rating TINYINT CHECK (rating >= 1 AND rating <= 5)
);

-- 枚举值模拟（MySQL 8.0.16+ 替代 ENUM 的一种方式）
CREATE TABLE orders (
    id     INT PRIMARY KEY,
    status VARCHAR(20) CHECK (status IN ('pending', 'paid', 'shipped', 'completed'))
);

-- 日期逻辑（开始日期不能晚于结束日期）
CREATE TABLE events (
    id         INT PRIMARY KEY,
    start_date DATE,
    end_date   DATE,
    CHECK (start_date <= end_date)
);

-- 多列约束
CREATE TABLE products (
    id          INT PRIMARY KEY,
    price       DECIMAL(10,2),
    sale_price  DECIMAL(10,2),
    CHECK (sale_price <= price)  -- 促销价不能高于原价
);
```

### 6.3 CHECK 的局限性

```sql
-- ❌ CHECK 不能用子查询（MySQL 限制）
CREATE TABLE orders (
    id      INT PRIMARY KEY,
    user_id INT,
    CHECK (user_id IN (SELECT id FROM users))  -- MySQL 不支持 CHECK 中的子查询！
);
-- 这种需求用 FOREIGN KEY，不要用 CHECK

-- ❌ CHECK 不能用自定义函数
-- ❌ CHECK 不能引用其他表的数据
```

## 七、DEFAULT（默认值约束）

**DEFAULT** *此术语见附录E* 在 INSERT 时不指定某列时，自动填入默认值。

```sql
CREATE TABLE users (
    id         INT PRIMARY KEY AUTO_INCREMENT,
    username   VARCHAR(50) NOT NULL,
    status     TINYINT DEFAULT 1 COMMENT '1=正常, 0=禁用',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 插入时只指定 username
INSERT INTO users (username) VALUES ('alice');
-- status 自动 = 1, created_at 自动 = 当前时间

-- ON UPDATE CURRENT_TIMESTAMP：更新行时自动更新此列
UPDATE users SET username = 'alice_new' WHERE id = 1;
-- updated_at 自动更新为当前时间
```

**DEFAULT 的实用场景**：
- `created_at` / `updated_at` 时间戳自动维护
- 状态字段（默认 `1` 表示正常，`0` 表示禁用）
- 计数字段（默认 `0`）
- 布尔字段（`TINYINT(1) DEFAULT 0`）

## 八、数据库层约束 vs 应用层约束

### 8.1 各自的优缺点

| 维度 | 数据库层约束 | 应用层约束 |
|------|------|------|
| 可靠性 | 最高——绕过应用直接操作数据库也无法破坏 | 低——Bug、绕过 API 直接操作、多应用共享数据库时都可能破坏 |
| 性能 | 有开销（外键 ~10-20% 写入性能下降） | 无额外数据库开销 |
| 灵活性 | 低——修改约束可能需要 ALTER TABLE（锁表） | 高——改代码即可 |
| 错误信息 | 通用（"Duplicate entry"） | 可定制（"该用户名已被注册，请换一个"） |
| 复杂逻辑 | 不支持（CHECK 不能做子查询） | 支持任意复杂逻辑 |

### 8.2 推荐策略

```
数据库层：保底防线
  ├── PRIMARY KEY       → 必须
  ├── NOT NULL          → 能加就加
  ├── UNIQUE            → 业务唯一字段（username, email）
  ├── DEFAULT           → 时间戳、状态默认值
  └── CHECK             → 简单的范围/枚举检查

应用层：用户友好 + 复杂逻辑
  ├── 数据格式验证（邮箱格式、手机号格式）
  ├── 业务规则校验（"余额不足"、"库存为0"）
  ├── 友好的中文错误提示
  └── 跨表复杂校验（"订单金额 = 所有订单项之和"）

FOREIGN KEY：视场景决定
  ├── 中低并发（< 1000 QPS 写入）→ 数据库层，数据安全优先
  └── 高并发（> 5000 QPS 写入）→ 应用层事务保证，性能优先
```

**一句话总结**：数据库约束是最后一道防线，即使应用层有校验，数据库约束也不应该省略。**你永远不知道未来谁会绕过应用直接操作数据库——可能是运维、可能是数据迁移脚本、可能是另一个微服务。**

## 我做得对不对？

### 验证方法

```sql
-- 在 my_shop 中创建测试表，验证每种约束
USE my_shop;

-- 1. PRIMARY KEY 验证
CREATE TABLE test_pk (id INT PRIMARY KEY);
INSERT INTO test_pk VALUES (1);
INSERT INTO test_pk VALUES (1);  -- ❌ Duplicate entry
INSERT INTO test_pk VALUES (NULL);  -- ❌ Column 'id' cannot be null

-- 2. FOREIGN KEY 验证
CREATE TABLE test_parent (id INT PRIMARY KEY);
INSERT INTO test_parent VALUES (1), (2);
CREATE TABLE test_child (
    id INT PRIMARY KEY,
    parent_id INT,
    FOREIGN KEY (parent_id) REFERENCES test_parent(id)
);
INSERT INTO test_child VALUES (1, 1);   -- ✅ 父表有 id=1
INSERT INTO test_child VALUES (2, 999); -- ❌ 父表没有 id=999

-- 3. UNIQUE 验证
CREATE TABLE test_unique (
    id INT PRIMARY KEY,
    code VARCHAR(10) UNIQUE
);
INSERT INTO test_unique VALUES (1, 'ABC');
INSERT INTO test_unique VALUES (2, 'ABC'); -- ❌ Duplicate entry
INSERT INTO test_unique VALUES (3, NULL);   -- ✅ NULL 可以重复
INSERT INTO test_unique VALUES (4, NULL);   -- ✅ 多个 NULL 不冲突

-- 4. NOT NULL 验证
CREATE TABLE test_notnull (
    id INT PRIMARY KEY,
```

## 不对怎么办？

### 常见错误 1：高并发场景使用外键导致性能瓶颈

```sql
-- ❌ 秒杀系统中，订单表有外键
CREATE TABLE seckill_orders (
    id      INT PRIMARY KEY,
    user_id INT,
    FOREIGN KEY (user_id) REFERENCES users(id)  -- 每次 INSERT 都检查 users 表
);
-- 10000 QPS 的插入场景下，外键检查成为瓶颈

-- ✅ 解决方案：移除外键，在应用层事务中保证一致性
CREATE TABLE seckill_orders (
    id      INT PRIMARY KEY,
    user_id INT  -- 无外键，写入性能更高
    -- 应用层：先查 users 确认 user_id 存在，再插入订单
    -- 在同一个事务中完成
);
```

### 常见错误 2：忘了 ON DELETE 导致无法删除父表数据

```sql
-- ❌ 建表时没写 ON DELETE，默认 RESTRICT
CREATE TABLE child (
    id INT PRIMARY KEY,
    parent_id INT,
    FOREIGN KEY (parent_id) REFERENCES parent(id)
    -- 等价于 ON DELETE RESTRICT
);
DELETE FROM parent WHERE id = 1;  -- ❌ 报错，因为子表有引用

-- ✅ 如果子表数据还有意义，用 ON DELETE SET NULL
CREATE TABLE child (
    id INT PRIMARY KEY,
    parent_id INT,
    FOREIGN KEY (parent_id) REFERENCES parent(id)
        ON DELETE SET NULL  -- 父表删除，子表外键变 NULL
);

-- ✅ 如果子表数据没有父表就没意义，用 ON DELETE CASCADE
CREATE TABLE child (
    id INT PRIMARY KEY,
    parent_id INT,
    FOREIGN KEY (parent_id) REFERENCES parent(id)
        ON DELETE CASCADE  -- 父表删除，子表一起删
);
```

### 常见错误 3：CHECK 约束在旧版 MySQL 上不生效

```sql
-- MySQL 8.0.15 及以下：CHECK 语法不报错，但实际不生效！
-- 检查你的 MySQL 版本
SELECT VERSION();

-- 如果版本 < 8.0.16，CHECK 约束只能用触发器或应用层替代
-- 或者在 INSERT/UPDATE 前用应用层代码校验
```

### 常见错误 4：UNIQUE 约束和 NULL 的"意外"

```sql
-- ❌ 以为 UNIQUE 列只能有一个 NULL
CREATE TABLE users (id INT, phone VARCHAR(20) UNIQUE);
INSERT INTO users VALUES (1, NULL);
INSERT INTO users VALUES (2, NULL);  -- ✅ 居然成功了！

-- 如果你需要"要么有值且唯一，要么为空且也不重复"
-- MySQL 中无法直接实现，需要应用层处理
-- 或者用触发器：BEFORE INSERT 检查 COUNT(phone IS NULL)
```

### 常见错误 5：应用层有校验就不加数据库约束

```sql
-- ❌ "反正代码里会检查邮箱格式，数据库不用加 UNIQUE"
CREATE TABLE users (
    id       INT PRIMARY KEY,
    email    VARCHAR(100)  -- 没有 UNIQUE
);
-- 结果：运维用脚本批量导入用户时，插入了重复邮箱

-- ✅ 数据库约束是最后一道防线
CREATE TABLE users (
    id       INT PRIMARY KEY,
    email    VARCHAR(100) NOT NULL,
    UNIQUE KEY uk_email (email)  -- 不管谁操作，数据库保证不重复
);
```

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 数据完整性 | 实体完整性、参照完整性、域完整性 | 约束是数据库的"免疫系统" |
| PRIMARY KEY | 自动 NOT NULL + UNIQUE，每表一个 | 自增 INT 适合单机，UUID 适合分布式但性能差 |
| 联合主键 | `PRIMARY KEY (col1, col2)` | 多列一起唯一标识一行 |
| FOREIGN KEY | 保证引用关系有效 | 高并发场景写入性能下降 10-20% |
| ON DELETE CASCADE | 父表删除，子表自动删除 | 订单明细有父表才存在，用 CASCADE |
| ON DELETE SET NULL | 父表删除，子表外键变 NULL | 文章作者被删，文章保留 |
| ON DELETE RESTRICT | 禁止删除有引用的父表行 | 默认选项，防止误删 |
| UNIQUE | 列值不重复 | 允许多个 NULL |
| NOT NULL | 列不能为空 | 能用 NOT NULL 就用，减少 NULL 隐患 |
| CHECK | 自定义条件检查 | MySQL 8.0.16+ 才真正支持，不支持子查询 |
| DEFAULT | 插入时自动填默认值 | 时间戳、状态码、布尔值常用 |
| 数据库 vs 应用层约束 | 数据库 = 保底防线，应用层 = 用户友好 | 数据库约束不可省略，以防绕过应用的操作 |
| 外键取舍 | < 1000 QPS 写入用数据库层，> 5000 用应用层 | 不确定时先用数据库层，遇瓶颈再优化 |
