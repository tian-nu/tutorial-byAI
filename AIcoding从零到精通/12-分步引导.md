## 为什么不能一次让AI写1000行代码

你试过一次性让AI写一个完整的项目吗？

```
帮我写一个完整的博客系统，包含用户注册、登录、发表文章、评论、分类、标签、搜索、后台管理、REST API、单元测试、Docker部署……
```

**如果你这么做过，结果大概率是：**

- AI写了1000行代码，但前言不搭后语，函数之间互相调用失败
- 前面写的函数和后面写的函数用的变量名不一致
- 代码跑到一半就报错，因为AI"忘记"了前面定义过的类
- 你花在"让AI修Bug"上的时间，比你自己写代码还多

**这不是AI不行，是你的工作方式不对。**

---

## 原因一：上下文有限

回顾第11章：AI的上下文窗口是有限的。你让它一口气写1000行代码，AI在写第800行的时候，可能已经"忘记"第200行写过什么了。结果就是前后不一致。

**类比：**
> 你让一个厨师一次性做100道菜，他做到第80道的时候，已经不记得第1道放了什么调料了。

---

## 原因二：AI会"忘事"

这是AI的一个特性，不是Bug。AI在生成长文本时，注意力会逐渐衰减：

- 前200行：质量很高，逻辑清晰
- 200-500行：开始出现小错误，变量名偶尔不一致
- 500-800行：明显有逻辑漏洞，函数签名对不上
- 800行以上：基本是"凑字数"，质量断崖式下降

**这不是AI的错，是你把任务描述得太"大"了。**

---

## 原因三：大任务容易写乱

一个1000行代码的项目，涉及到几十个函数、多个文件、数据库、API——这些之间的依赖关系很复杂。AI一次性处理这么多信息，很容易"乱掉"。

**正确做法：把大任务拆成小任务，一次只让AI做一件事。**

---

## Chain-of-Thought（思维链）：让AI先想，再写

> 📖 **术语：Chain-of-Thought（思维链，CoT）** = 让AI在给出最终答案之前，先展示它的思考过程。就像做数学题时"先列步骤，再计算结果"。

**普通Prompt vs 思维链Prompt：**

### ❌ 普通Prompt（不让AI思考）

```
帮我写一个Python函数，找出一个列表中所有重复的元素。
```

**AI直接输出（可能对，也可能有问题）：**

```python
def find_duplicates(lst):
    return [x for x in set(lst) if lst.count(x) > 1]
```
> 这个代码能跑，但性能很差（O(n²)），因为对每个元素都遍历了整个列表。

### ✅ 思维链Prompt（让AI先分析）

```
帮我写一个Python函数，找出一个列表中所有重复的元素。

请你先分析需求，列出需要考虑的情况，然后给出你的方案，最后再写代码。

需要考虑：
- 空列表怎么处理？
- 元素重复多次怎么处理？
- 时间复杂度要求？
- 是否保持原顺序？
```

**AI先分析，再写代码：**

```
分析：
1. 空列表 → 返回空列表
2. 重复多次 → 每个重复元素只出现一次
3. 时间复杂度 → O(n)用字典统计，不用count()（O(n²)）
4. 保持原顺序 → 遍历时按原顺序记录

方案：用字典统计每个元素的出现次数，然后筛选出count>1的。

代码：
```

```python
from typing import List, TypeVar

T = TypeVar('T')

def find_duplicates(lst: List[T]) -> List[T]:
    """找出列表中所有重复出现的元素，保持首次出现顺序。"""
    if not lst:
        return []
    
    count = {}
    for item in lst:
        count[item] = count.get(item, 0) + 1
    
    seen = set()
    result = []
    for item in lst:
        if count[item] > 1 and item not in seen:
            result.append(item)
            seen.add(item)
    
    return result
```

**看到区别了吗？** 思维链版的AI代码质量明显更高——它先分析了边界条件、性能要求，再写代码，而不是凭直觉一行搞定。

---

### 思维链在实际编程中的应用

**每次让AI写代码时，加上这句话：**

```
请先分析需求，列出需要考虑的边界情况，再给出你的方案，最后写代码。
```

这句话的威力在于：它强制AI在写代码之前进行"思考"，而不是直接跳到"写"。

## 任务拆分技巧：按功能 → 按文件 → 按函数

**三分法：**

| 拆分层级 | 每次任务大小 | 示例 |
|---------|------------|------|
| 第一层：按功能 | 一个完整功能模块 | "用户注册"、"文章发布" |
| 第二层：按文件 | 一个文件 | "models.py"、"routes.py" |
| 第三层：按函数 | 一个函数 | "register()函数"、"login()函数" |

**建议：新手从第二层开始（按文件拆分），熟练后可以按第一层（按功能拆分）。**

---

## 案例：拆分"用户注册系统"

假设你要做一个用户注册系统，包含：注册、登录、邮箱验证、密码重置。

### ❌ 坏做法：一次性全扔给AI

```
帮我写一个完整的用户注册系统，包含注册、登录、邮箱验证、密码重置。
```

> 结果：AI写了800行代码，函数之间互相调用失败，Bug一大堆。

### ✅ 好做法：拆成5个步骤

**步骤1：数据模型**

```
你现在是项目的初始化阶段。帮我创建用户系统的基础数据模型。

要求：
- 用Python + SQLAlchemy
- 创建User模型，包含字段：id, username, email, password_hash, is_verified, created_at, updated_at
- 密码不存明文，用bcrypt
- 只输出models.py的内容，不要写路由和其他代码
```

**步骤2：注册功能**

```
现在用户模型已经创建好（models.py中有User模型，字段包括id, username, email, password_hash, is_verified, created_at, updated_at）。
帮我写注册功能的路由。

要求：
- 用Flask
- POST /api/register
- 接收 username, email, password
- 验证：用户名3-20字符，邮箱格式正确，密码至少8位
- 密码用bcrypt加密后存到password_hash字段
- 返回JSON格式
- 只输出routes.py中的register函数，不要改其他代码
```

**步骤3：登录功能**

```
用户模型和注册功能已经完成。现在帮我写登录功能。

要求：
- POST /api/login
- 接收 email, password
- 用bcrypt验证密码
- 登录成功返回JWT token
- 返回JSON格式
- 只输出login函数
```

**步骤4：邮箱验证**

```
用户模型、注册、登录功能已完成。现在帮我写邮箱验证功能。

要求：
- 注册成功后自动发送验证邮件（用smtplib）
- GET /api/verify/<token> 验证邮箱
- 验证成功后更新is_verified为True
- 只输出verify_email相关函数
```

**步骤5：密码重置**

```
用户模型、注册、登录、邮箱验证已完成。现在帮我写密码重置功能。

要求：
- POST /api/reset-password/request 发送重置链接到邮箱
- POST /api/reset-password/confirm 重置密码
- 重置token有效期1小时
- 只输出reset_password相关函数
```

---

### 分步引导的好处总结

| 不分步 | 分步 |
|--------|------|
| 一次产出800行，Bug一堆 | 每次产出80-150行，几乎没Bug |
| 前后代码不一致 | 每步都基于前面已验证的代码 |
| 出问题你不知道从哪改起 | 每步独立，出问题定位快 |
| AI在后期"忘事" | 每步上下文都小，AI不会忘 |
| 改起来牵一发动全身 | 改哪步就改哪个函数 |

**经验法则：完成一步再走下一步，不要催。** 每步验证通过后，把这步的代码作为下一步的上下文贴进去。
