## 44.1 什么是单元测试

> **📖 术语：单元测试（Unit Test）**——给每个函数写一段专门的测试代码，用来验证这个函数在给定输入下，是否产生了正确的输出。"单元"指的是最小的可测试单位（在Python中一般就是一个函数）。

**一个比喻**：

你开了一家餐厅，雇了一个厨师。你怎么知道厨师做的菜对不对？

- **不测试**：客人吃了拉肚子，你才知道有问题——太晚了
- **品菜**：每道菜上桌前，你先尝一口，确认味道对——但你要尝每一道菜，太累了
- **单元测试**：你给厨师定好标准："番茄炒蛋 = 鸡蛋3个+番茄2个+盐5克+糖3克，炒3分钟，成品红色为主、黄色点缀"。然后你写一个"自动检测机器"——把菜放进去，机器自动告诉你"合格"还是"不合格"。

**单元测试就是那个"自动检测机器"**。你写好后，每次改代码，跑一遍测试，AI自动告诉你"改坏了没有"。

---

## 44.2 为什么AI特别擅长写测试

**因为测试代码是高度模式化的。** 大多数测试都是这样的套路：

```
1. 准备输入数据
2. 调用被测函数
3. 检查输出是否等于预期
4. 报告"通过"或"失败"
```

这种"套路化"的代码，恰好是AI的强项。AI不需要理解你的业务逻辑有多复杂——它只需要知道"输入是什么、输出应该是什么"，就能生成规范的测试代码。

**换句话说**：写业务逻辑，AI可能翻车；写测试，AI几乎不会翻车。

## 44.3 怎么让AI写测试

### 基本模板

把你要测试的函数发给AI，加上这句话：

```
帮我给这个函数写单元测试，要求：
1. 覆盖正常输入情况
2. 覆盖边界情况（空输入、极端值、错误输入）
3. 使用 Python 的 unittest 框架（或 pytest）
4. 每个测试用例要有清晰的注释，说明在测什么
```

### 完整案例

#### 你有一个函数

```python
def calculate_average(scores):
    """计算分数列表的平均值，跳过非数字项"""
    valid_scores = []
    for score in scores:
        if isinstance(score, (int, float)):
            valid_scores.append(score)
    
    if len(valid_scores) == 0:
        return 0
    
    return sum(valid_scores) / len(valid_scores)
```

#### 你发给AI的Prompt

```
帮我给下面这个函数写单元测试，使用 unittest 框架：

```python
def calculate_average(scores):
    """计算分数列表的平均值，跳过非数字项"""
    valid_scores = []
    for score in scores:
        if isinstance(score, (int, float)):
            valid_scores.append(score)
    
    if len(valid_scores) == 0:
        return 0
    
    return sum(valid_scores) / len(valid_scores)
```

要求：
1. 覆盖正常输入（全是数字）
2. 覆盖混合输入（数字+字符串混合）
3. 覆盖空列表
4. 覆盖全非数字的列表
5. 每个测试用例要有清晰的注释
```

#### AI生成的测试代码

```python
import unittest

class TestCalculateAverage(unittest.TestCase):
    
    def test_all_numbers(self):
        """测试：全是数字的列表"""
        result = calculate_average([80, 90, 100])
        self.assertEqual(result, 90.0)
    
    def test_mixed_types(self):
        """测试：数字和字符串混合"""
        result = calculate_average([85, 90, "92", 88])
        self.assertEqual(result, 87.66666666666667)
    
    def test_empty_list(self):
        """测试：空列表"""
        result = calculate_average([])
        self.assertEqual(result, 0)
    
    def test_all_strings(self):
        """测试：全是非数字项"""
        result = calculate_average(["A", "B", "C"])
        self.assertEqual(result, 0)
    
    def test_single_number(self):
        """测试：只有一个数字"""
        result = calculate_average([42])
        self.assertEqual(result, 42.0)
    
    def test_negative_numbers(self):
        """测试：包含负数"""
        result = calculate_average([-10, 10, 0])
        self.assertEqual(result, 0.0)

if __name__ == '__main__':
    unittest.main()
```

运行测试：

```bash
python -m unittest test_calculate_average.py
```

输出：

```
......
----------------------------------------------------------------------
Ran 6 tests in 0.001s

OK
```

**6个测试全部通过，你用了不到1分钟。** 如果让你手写这6个测试，至少需要10-15分钟。

## 44.4 验证AI写的测试：测试通过了就是对的吗？

**不一定！** 这是很多新手的误区。

### 测试通过 ≠ 测试写对了

AI可能写了一个"永远会通过的测试"——比如它忘了写断言（`assert`），或者断言的条件永远为真。

#### ❌ 例子：AI写了一个"假的"测试

```python
def test_average(self):
    result = calculate_average([1, 2, 3])
    # 忘了写 self.assertEqual(...)
    print("测试完成")  # 这个测试什么都测不了，但不会报错
```

这种测试运行时会"通过"（因为没有断言失败），但它什么都没验证。

#### ✅ 怎么验证AI写的测试是有效的？

1. **故意改错被测函数**，看测试会不会失败
2. **检查每个测试是否有断言**（`assertEqual`、`assertTrue`、`assertRaises` 等）
3. **检查断言的值是否合理**：`assertEqual(result, 0)` 比 `assertEqual(result, result)` 有效

**最简单的验证方法**：

```
把被测函数改错（比如 return 0），重跑测试。如果测试还通过，说明测试没写对。
```

## 44.5 什么是测试覆盖率

> **📖 术语：测试覆盖率（Test Coverage）**——你的测试代码"覆盖"了多少行被测代码。如果被测函数有20行，你的测试只执行了其中15行，覆盖率就是75%。

**一个比喻**：你写的测试就像一束光，照到的地方就是"被覆盖的"。函数里有些行可能从没被测试执行过（比如某个 `if` 分支），这些就是"黑暗角落"——Bug最喜欢藏在那里。

### 不要追求100%覆盖率

很多新手听说"测试覆盖率"后，就拼命追求100%。**没必要。**

- 100%覆盖率意味着你的测试执行了每一行代码
- 但执行了 ≠ 验证了正确性
- 追求100%覆盖率的代价很高（可能需要写很多意义不大的测试）
- **80%覆盖率是一个务实的目标**

### 怎么查看覆盖率

让AI帮你跑：

```bash
# 安装覆盖率工具
pip install coverage

# 运行测试并记录覆盖率
coverage run -m unittest test_calculate_average.py

# 查看覆盖率报告
coverage report -m
```

或者直接让AI帮你跑：

```
帮我运行这个测试文件，并检查测试覆盖率。如果覆盖率低于80%，告诉我是哪些行没覆盖到。
```

## 44.6 什么时候应该写测试，什么时候可以不写

| 应该写测试 | 可以不写测试 |
|-----------|-------------|
| 核心业务逻辑（算钱、算分、数据处理） | 一次性脚本（用完就扔） |
| 会被多次修改的代码 | 纯UI代码（HTML/CSS） |
| 多人协作的项目 | 快速原型验证 |
| 你过两周会忘记它怎么工作的代码 | 只调用了第三方库的简单包装 |

**一句话**：如果这个函数写错了会造成严重后果（算错钱、数据丢失），就写测试。如果错了只是"重新跑一下"，可以先不写。

2. AI生成的测试全部通过了，能说明测试写对了吗？怎么验证？

> **答案**：
> 1. 手工测试是每次改代码后，手动跑一遍功能看结果；单元测试是写好的"自动检测机器"，一行命令就能跑完所有测试。优势是省时间、不会漏测、改代码后能立刻发现Bug。
> 2. 不一定。AI可能写了"永远会通过"的测试（比如忘了写断言）。验证方法：故意改错被测函数，看测试会不会失败——如果还通过，说明测试没写对。
> 3. 100%覆盖率意味着测试执行了每一行代码，但"执行了"不等于"验证了正确性"。追求100%代价高、收益低，80%是一个务实的目标。
