## 我在做什么？

你写过代码，跑过项目，大概率也遇到过这样一个场景——

> 本地跑得好好的，部署到服务器就报错。同一个项目，同事的电脑上能跑，你的电脑上就是跑不起来。

你排查了半天，发现是 Python 版本不对、Node.js 版本不对、系统库少了某个 `.so` 文件、环境变量没设……最后你仰天长叹："有没有一种办法，能让我把本地能跑的环境**原封不动地打包**，扔到任何一台机器上都能跑？"

有的。这就是 Docker 要解决的问题。

学完这一章，你能用自己的话说出：Docker 是什么、容器和虚拟机有什么区别、以及 Docker 为什么成了现代软件开发的标准配置。

## 一、痛苦之源：环境不一致

### 你做过的那些"环境配置"噩梦

想想你搭建一个 Python 项目环境需要几步：

```bash
# 第1步：装 Python 3.11（还不能是 3.12，因为某个库不兼容）
# 第2步：装虚拟环境
python -m venv venv
# 第3步：激活虚拟环境
# 第4步：装依赖
pip install -r requirements.txt
# 第5步：装 MySQL 客户端库
# 第6步：配环境变量
# 第7步：发现还是报错，原来是系统缺了 libmysqlclient-dev
# 第8步：装完系统库，又发现 pip 版本太老……
```

这还只是**一个**项目。如果有两个项目，一个需要 Python 3.11，另一个需要 Python 3.9，你怎么办？如果项目 A 需要 MySQL 5.7，项目 B 需要 MySQL 8.0，你怎么办？

```
│    服务器：  Python 3.9,  Node 18, Ubuntu 22.04  │

└─────────────────────────────────────────────────┘
```

### 问题的本质

环境不一致的根源是：**一个软件能跑起来，依赖的不只是代码本身，还有操作系统、运行时版本、系统库、配置文件、环境变量……** 这些东西不在代码仓库里，却决定了代码能不能跑。

Docker 的思路很直接：**把这些东西全部打包进一个"箱子"里，连代码带环境一起分发。**

## 二、虚拟机 vs 容器：装房子 vs 带箱子

在 Docker 出现之前，解决环境不一致的常见方案是**虚拟机** *此术语见附录C*。要理解 Docker 为什么快，必须先理解虚拟机。

### 虚拟机：你在房子里又盖了一栋房子

虚拟机的工作方式：

```
┌──────────────────────────────────────┐
│            你的物理机（宿主机）        │
│                                      │
│  ┌────────────────────────────────┐  │
│  │        虚拟机（Guest）          │  │
│  │  ┌──────────────────────────┐  │  │
│  │  │  App A    App B          │  │  │
│  │  ├──────────────────────────┤  │  │
│  │  │  Binaries / Libraries    │  │  │
│  │  ├──────────────────────────┤  │  │
│  │  │  Guest OS（完整操作系统）  │  │  │
│  │  │  比如 Ubuntu 22.04       │  │  │
│  │  └──────────────────────────┘  │  │
│  │  Hypervisor（虚拟硬件层）       │  │
│  └────────────────────────────────┘  │
│                                      │
│  宿主机 OS（Windows / macOS / Linux） │
│  物理硬件（CPU、内存、硬盘）          │
└──────────────────────────────────────┘
```

虚拟机在宿主机上模拟了一套**完整的硬件**，然后在上面安装了一个**完整的操作系统**。每个虚拟机有自己的内核、自己的驱动、自己的系统进程。

**好处**：隔离得非常彻底，安全。

**代价**：
- 启动慢：要启动一个完整的操作系统，几十秒起步。
- 占资源：每个虚拟机至少要分几百 MB 内存，还要独占一部分 CPU 核心。
- 镜像大：一个虚拟机镜像动辄几个 GB。

> 打个比方：虚拟机就像你为了住进一个房间，先在房间里**重新盖了一栋楼**——连地基、墙壁、水电都要重新铺。

### 容器：你只带一个行李箱

Docker 容器 *此术语见附录C* 的工作方式完全不同：

```
┌──────────────────────────────────────┐
│            你的物理机（宿主机）        │
│                                      │
│  ┌──────────┐ ┌──────────┐          │
│  │ 容器 A    │ │ 容器 B    │          │
│  │  App A    │ │  App B    │          │
│  │ 依赖/Libs │ │ 依赖/Libs │          │
│  └──────────┘ └──────────┘          │
│                                      │
│         Docker Engine                │
│                                      │
│  宿主机 OS 内核（共享）               │
│  物理硬件（CPU、内存、硬盘）          │
└──────────────────────────────────────┘
```

容器**不装操作系统**，不虚拟硬件。它只打包应用及其依赖（库、配置文件、运行时），然后**共享宿主机的操作系统内核** *此术语见附录C*。

**好处**：
- 启动快：毫秒到秒级，因为不需要启动操作系统。
- 省资源：容器只是一个进程，内存占用小，CPU 不需要模拟硬件。
- 镜像小：通常几十 MB 到几百 MB。

**代价**：
- 隔离不如虚拟机彻底（共享内核，存在一定安全风险）。
- 容器必须和宿主机用**同一类操作系统内核**（Linux 容器只能在 Linux 宿主机上跑，Windows 容器只能在 Windows 宿主机上跑）。

> 打个比方：容器就像你住酒店——你不需要自己盖楼，你只需要一个**行李箱**（镜像），酒店（宿主机）提供基础设施（内核），你打开行李箱就能住。

### 一张表看懂核心差异

| 对比维度 | 虚拟机 | Docker 容器 |
|---------|--------|------------|
| 启动速度 | 30 秒 ~ 几分钟 | 毫秒 ~ 秒 |
| 内存占用 | 每个 GB 级 | 每个 MB 级 |
| 镜像大小 | 几个 GB | 几十 MB ~ 几百 MB |
| 包含内容 | 完整操作系统 + 应用 | 只含应用及其依赖 |
| 内核 | 各自独立内核 | 共享宿主机内核 |
| 隔离级别 | 硬件级隔离 | 进程级隔离 |
| 适合场景 | 跑不同操作系统、强安全隔离 | 微服务、CI/CD、快速扩缩 |

> **想多一点**：你可能会问："既然虚拟机更安全，为什么不都用虚拟机？"因为大部分场景下，你需要的不是"绝对安全"，而是"够用 + 快"。就像一个公司的不同部门，没必要每个部门都建一栋独立的办公楼——在同一栋楼里分楼层就够了。Docker 的隔离级别对绝大多数应用来说已经足够。

## 三、Docker 的三个核心概念

Docker 的世界里，有三个词你必须搞清楚。它们之间的关系，我用一个比喻帮你记住：

### 镜像（Image）*此术语见附录C* = 做菜的食谱

镜像是一个**只读模板**，里面定义了应用运行需要的一切：代码、运行时、库、环境变量、配置文件。

镜像本身是**静态的**，它不会运行。就像一本食谱——它告诉你放什么料、用什么火候，但食谱本身不会变成一道菜。

```bash
# 从 Docker Hub 拉取一个 Ubuntu 镜像
docker pull ubuntu:22.04

# 查看本地有哪些镜像
docker images
```

这行命令下载了一个 Ubuntu 22.04 的镜像到你的电脑上。此时它只是一个"食谱"，还没有变成"菜"。

### 容器（Container）*此术语见附录C* = 照着食谱做出来的菜

容器是镜像的**运行实例**。当你执行 `docker run` 时，Docker 照着镜像的内容，创建出一个独立运行的环境——这就是容器。

容器是**动态的**：它可以启动、停止、删除、重启。同一个镜像可以同时跑出多个容器，就像同一本食谱可以同时做出多道菜。

```bash
# 用 ubuntu:22.04 镜像启动一个容器
docker run -it ubuntu:22.04 bash
```

### 仓库（Registry）*此术语见附录C* = 菜谱网站

仓库是**存放镜像的地方**。最著名的公共仓库是 **Docker Hub**（hub.docker.com），上面有成千上万个官方和社区维护的镜像。

你可以从仓库**拉取（pull）**镜像到本地，也可以把自己制作的镜像**推送（push）**到仓库。

```bash
# 从 Docker Hub 拉取镜像
docker pull nginx:latest

# 把自己的镜像推送到 Docker Hub
docker push your_username/your_image:tag
```

### 三者的关系

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│   仓库 (Registry)  ──pull──>  镜像 (Image)            │
│   (Docker Hub)                 │                     │
│       ▲                        │ docker run          │
│       │ push                   ▼                     │
│       │               容器 (Container)                │
│       │               (镜像的运行实例)                 │
│                                                      │
│   一个镜像可以跑出多个容器                              │
│   一个容器只能来自一个镜像                              │
│                                                      │
└──────────────────────────────────────────────────────┘
```

## 四、Docker 能解决什么问题？

当你理解了上面三个概念，再回头看 Docker 能做什么，就一目了然了：

### 1. 环境一致性："在我机器上能跑"成为历史

你把应用和所有依赖打包成一个镜像，然后在任何装有 Docker 的机器上 `docker run`，环境一模一样。开发环境、测试环境、生产环境，用的是同一个镜像。

### 2. 快速部署：从"部署半天"到"一条命令"

传统部署：装系统 → 装依赖 → 配环境 → 传代码 → 启动 → 报错 → 排查 → 重来。

Docker 部署：`docker run your_image`。几秒钟，服务起来了。

### 3. 资源隔离：一个项目一个容器，互不干扰

项目 A 需要 Python 3.9，项目 B 需要 Python 3.11？没问题，各跑各的容器，互不影响。项目 A 的 MySQL 5.7 和项目 B 的 MySQL 8.0 可以同时跑在同一台机器上。

### 4. 弹性伸缩：流量来了，秒级扩容

双十一来了，流量暴增？多启动几个容器实例就行。`docker run` 一条命令，一个新的服务实例就上线了。用完就停，不浪费资源。

---

## 五、一句话总结

> **Docker 把应用和它的所有依赖打包成一个可移植的镜像，在哪里跑都一样。**

如果你只能记住一句话，记住这句就够了。

## 我做得对不对？

### 验证方法

合上这一章，用自己的话回答以下问题：

1. 为什么"Docker = 虚拟机"是错误的？
2. 镜像、容器、仓库三者之间的关系是什么？
3. 一个 Ubuntu 的 Docker 容器能在 Windows 上跑吗？为什么？

如果你能回答，说明你理解了 Docker 的本质。如果答不上来，回去重读第二节和第三节。

### 参考答案（读完再对）

1. 虚拟机包含完整操作系统，各自有独立内核，启动慢、占资源多；容器只打包应用和依赖，共享宿主机内核，启动快、省资源。
2. 镜像是静态模板（食谱），容器是镜像的运行实例（菜），仓库是存放镜像的地方（菜谱网站）。从仓库 pull 镜像，用镜像 run 出容器。
3. 能。Docker Desktop for Windows 会自动创建一个 Linux 虚拟机（WSL 2 或 Hyper-V），在虚拟机里跑 Linux 容器。对用户来说是透明的，但本质上容器还是跑在 Linux 内核上。

## 不对怎么办？

### 常见错误 1：认为 Docker = 虚拟机

❌ 错误认知：Docker 就是轻量级虚拟机，两种东西差不多。

✅ 正确认知：虚拟机是**硬件虚拟化**，每个虚拟机有自己的操作系统内核；容器是**操作系统级虚拟化**，所有容器共享同一个宿主机内核。区别不是"轻量"和"重量"，而是**架构完全不同**。

> 为什么这个错误很普遍？因为 Docker 和虚拟机都能"隔离环境"，表面的使用体验相似。但当你需要在一台机器上跑 100 个服务时，虚拟机方案需要 100 个操作系统，内存直接爆炸；Docker 方案只需要 100 个进程，内存开销小得多。

### 常见错误 2：认为 Docker 一定比虚拟机好

❌ 错误认知：Docker 全方位碾压虚拟机，虚拟机过时了。

✅ 正确认知：两者各有适用场景。Docker 擅长**轻量、快速、密度高**的场景（微服务、CI/CD）；虚拟机擅长**强隔离、跨平台、安全合规**的场景（多租户云服务、跑不同操作系统）。大公司通常是**两者混用**——在虚拟机上跑 Docker 容器。

### 常见错误 3：混淆"镜像"和"容器"

❌ 错误认知：镜像就是容器，容器就是镜像。

✅ 正确认知：镜像 = 类（Class），容器 = 实例（Instance）。就像"汽车图纸"和"你开的那辆汽车"——图纸能造出无数辆车，但你开的是其中一辆具体的车。镜像可以创建多个容器，每个容器是独立的运行实例。

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 环境不一致问题 | 代码能跑 ≠ 到处能跑 | Docker 解决的是"环境打包"问题，不是"代码打包"问题 |
| 虚拟机 vs 容器 | 虚拟机：独立 OS + Hypervisor；容器：共享内核 + 只打包应用 | 本质区别在架构，不是"轻量 vs 重量" |
| 镜像（Image） | 只读模板，静态 | 像食谱，本身不会运行 |
| 容器（Container） | 镜像的运行实例，动态 | 一个镜像可以跑多个容器，修改容器不影响镜像 |
| 仓库（Registry） | 存放镜像的服务器 | Docker Hub 是最大的公共仓库 |
| 内核共享 | 容器共享宿主机内核 | Linux 容器需要 Linux 内核，Windows/Mac 通过虚拟机间接支持 |
| Docker 四大能力 | 环境一致性、快速部署、资源隔离、弹性伸缩 | 不是 Docker 独有，但 Docker 让这些变得极其简单 |
