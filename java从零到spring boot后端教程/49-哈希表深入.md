# 49. 哈希表深入

> 📖 你在图书馆检索系统输入"三体"，系统瞬间告诉你：3楼A区第5排第3本。它没有遍历所有书架，而是通过书名计算出一个位置，直奔目的地。——这就是哈希表：**输入key→计算hash→直接定位，O(1)插入、O(1)查找、O(1)删除**。

---

## 49.1 从数组到哈希表：为什么需要哈希函数？

数组的查找最快是O(1)——前提是**你知道索引**。但现实中你很少知道索引，你只知道key（比如用户名"zhangsan"），需要找到它对应的value。

哈希表的思路：**用哈希函数把key转换成数组索引**。

```
key "zhangsan"  ──hashCode()──→  整数 123456789
                                      │
                              index = 123456789 % 16  →  13
                                      │
                              array[13] → value
```

**三步走**：
1. `key.hashCode()` → 得到一个int
2. `hash = hashCode ^ (hashCode >>> 16)` → 扰动，让高位也参与运算
3. `index = hash & (table.length - 1)` → 映射到数组索引

---

## 49.2 哈希冲突：不可避免的问题

哈希函数的输出是int（约42亿种可能），但数组长度通常是16、32这样的小数。不同key可能映射到同一个索引——这叫**哈希冲突**。

```
"张三".hashCode() % 16 = 3
"王五".hashCode() % 16 = 3    ← 冲突了！
```

### 解决方法1：链地址法（HashMap用的就是这个）

每个数组槽位不直接存value，而是存一个链表（或红黑树）的头节点。

```
数组:
[0] → null
[1] → null
[2] → null
[3] → "张三"=value1 → "王五"=value2 → null
[4] → null
...
```

查找时：先定位槽位3，然后在链表里遍历找匹配的key。

**JDK 1.8优化**：当链表长度≥8且数组长度≥64时，链表**转为红黑树**（查找从O(n)降到O(log n)），这是为了避免恶意攻击（构造大量冲突key让HashMap退化到O(n)）。

### 解决方法2：开放地址法

冲突了就找下一个空位：`index, index+1, index+2, ...`，直到找到空位。`ThreadLocal`内部用的就是这种方法。

## 49.3 HashMap源码核心机制

### 数据结构

```java
// HashMap核心字段（JDK源码简化）
public class HashMap<K,V> {
    // Node数组，每个Node是一个链表节点
    transient Node<K,V>[] table;

    // 实际存储的键值对数量
    transient int size;

    // 扩容阈值 = capacity * loadFactor
    int threshold;

    // 负载因子，默认0.75
    final float loadFactor = 0.75f;

    static class Node<K,V> {
        final int hash;    // key的哈希值（已扰动）
        final K key;
        V value;
        Node<K,V> next;    // 链表的下一个节点
    }
}
```

### put(K key, V value) 完整流程

```java
public V put(K key, V value) {
    // 1. 计算hash
    int hash = hash(key);

    // 2. 确定数组索引
    int i = (table.length - 1) & hash;

    // 3. 如果该位置为空，直接放入
    if (table[i] == null) {
        table[i] = newNode(hash, key, value, null);
    } else {
        // 4. 该位置已有节点，遍历链表/树
        Node<K,V> p = table[i];
        for (int binCount = 0; ; ++binCount) {
            if (p.hash == hash && key.equals(p.key)) {
                // key已存在，更新value
                return p.setValue(value);
            }
            if (p.next == null) {
                // 链表末尾，插入新节点
                p.next = newNode(hash, key, value, null);
                // 链表长度>=8时转为红黑树
                if (binCount >= 8) treeifyBin(tab, hash);
                break;
            }
            p = p.next;
        }
    }

    // 5. size+1，如果超过阈值则扩容
    if (++size > threshold) resize();
    return null;
}
```

### resize() 扩容

当`size > capacity * loadFactor`时触发扩容（默认16×0.75=12，即第13个元素触发）：

```
扩容前：capacity=16, each bucket可能有链表
扩容后：capacity=32, 重新计算每个节点的索引

关键优化（JDK 1.8）：不需要重新计算hash
旧索引 = hash & (16-1) = hash & 1111    (低4位)
新索引 = hash & (32-1) = hash & 11111   (低5位)
如果hash的第5位=0 → 索引不变
如果hash的第5位=1 → 索引 = 旧索引 + 16
```

## 49.4 equals和hashCode的契约

**这是Java中最容易踩的坑之一，面试必问。**

### 铁律（必须同时遵守）

1. **如果`a.equals(b) == true`，则`a.hashCode() == b.hashCode()` 必须是true**
2. **如果`a.hashCode() == b.hashCode()`，`a.equals(b)`不一定是true**（哈希冲突）
3. **重写equals时必须重写hashCode，反之亦然**

### ❌ 错误示例：只重写equals，不重写hashCode

```java
class User {
    String name;
    int age;

    User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof User)) return false;
        User user = (User) o;
        return age == user.age && Objects.equals(name, user.name);
    }
    // 没重写hashCode！！！继承了Object的hashCode（基于内存地址）
}

// 测试——出Bug了！
User u1 = new User("张三", 25);
User u2 = new User("张三", 25);
Map<User, String> map = new HashMap<>();
map.put(u1, "张三的资料");
System.out.println(map.get(u2));  // null！！！应该是"张三的资料"
// 因为u1和u2的hashCode不同（Object基于地址），HashMap找了不同的桶
```

### ✅ 正确示例

```java
class User {
    String name;
    int age;

    User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof User)) return false;
        User user = (User) o;
        return age == user.age && Objects.equals(name, user.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(name, age);  // equals用的字段，hashCode也必须用
    }
}
```

## 49.5 手写简易HashMap

```java
public class SimpleHashMap<K, V> {
    private static class Entry<K, V> {
        final K key;
        V value;
        Entry<K, V> next;

        Entry(K key, V value) {
            this.key = key;
            this.value = value;
        }
    }

    private Entry<K, V>[] table;
    private int size;
    private static final int DEFAULT_CAPACITY = 16;
    private static final float LOAD_FACTOR = 0.75f;

    @SuppressWarnings("unchecked")
    public SimpleHashMap() {
        table = new Entry[DEFAULT_CAPACITY];
        size = 0;
    }

    public void put(K key, V value) {
        int index = indexFor(key);
        Entry<K, V> head = table[index];

        // 检查是否已有相同key，有则更新
        for (Entry<K, V> e = head; e != null; e = e.next) {
            if (Objects.equals(e.key, key)) {
                e.value = value;
                return;
            }
        }

        // 头插法放入新节点
        Entry<K, V> newEntry = new Entry<>(key, value);
        newEntry.next = head;
        table[index] = newEntry;
        size++;

        if (size > table.length * LOAD_FACTOR) {
            resize();
        }
    }

    public V get(K key) {
        int index = indexFor(key);
        for (Entry<K, V> e = table[index]; e != null; e = e.next) {
            if (Objects.equals(e.key, key)) {
                return e.value;
            }
        }
        return null;
    }

    public V remove(K key) {
        int index = indexFor(key);
        Entry<K, V> prev = null;
        for (Entry<K, V> e = table[index]; e != null; e = e.next) {
            if (Objects.equals(e.key, key)) {
                if (prev == null) {
                    table[index] = e.next;
                } else {
                    prev.next = e.next;
                }
                size--;
                return e.value;
            }
            prev = e;
        }
        return null;
    }

    public int size() { return size; }

    private int indexFor(K key) {
        if (key == null) return 0;  // HashMap允许null key，放在索引0
        int h = key.hashCode();
        h = h ^ (h >>> 16);         // 扰动函数
        return h & (table.length - 1);
    }

    @SuppressWarnings("unchecked")
    private void resize() {
        Entry<K, V>[] oldTable = table;
        table = new Entry[oldTable.length * 2];
        size = 0;
        for (Entry<K, V> head : oldTable) {
            for (Entry<K, V> e = head; e != null; e = e.next) {
                put(e.key, e.value);  // rehash
            }
        }
    }
}
```

## 49.6 哈希表在后端的核心应用

### Spring IoC容器（再次强调）

```java
// Spring内部有多个Map来管理Bean
// DefaultListableBeanFactory 核心存储
private final Map<String, BeanDefinition> beanDefinitionMap = new ConcurrentHashMap<>(256);
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);
```

### 分布式一致性哈希

当有多个缓存服务器时，如何决定一个key放在哪台服务器？

```
普通哈希: server = hash(key) % 服务器数量
问题：加一台服务器，几乎所有key的映射都变了 → 缓存雪崩

一致性哈希：把服务器和key都映射到同一个哈希环上
key顺时针找到最近的服务器
加服务器：只影响环上一小段 → 大部分缓存仍然有效
```

### 去重、计数、缓存

// 去重：用户ID列表去重

## 49.7 ❌ 常见错误

| 错误 | 后果 | 正确做法 |
|---|---|---|
| 重写equals不重写hashCode | HashMap/HashSet行为异常 | 同时重写，用`Objects.hash()` |
| 用可变对象做HashMap的key | 改了对象，hash变了，找不到了 | key尽量用不可变对象（String、Integer） |
| HashMap多线程并发put | 可能导致死循环（JDK7）、数据丢失 | 用ConcurrentHashMap |
| `hashCode`每次都返回不同值 | 同一个对象每次放进HashMap都找不到 | 保证同对象的hashCode一致 |
| HashMap初始容量太小 | 频繁扩容影响性能 | 预估容量：`new HashMap<>(expectedSize / 0.75f + 1)` |

## 小结

| 概念 | 解释 |
|---|---|
| 哈希函数 | key→hash值→数组索引，核心是均匀分布 |
| 哈希冲突 | 不同key映射到同一索引 |
| 链地址法 | 每个桶存链表，冲突时追加到链表（≥8转红黑树） |
| 负载因子 | 默认0.75，size > capacity×0.75触发扩容 |
| equals/hashCode契约 | equals相等则hashCode必等，反之不成立 |
| HashMap时间复杂度 | 理想O(1)，冲突严重退化到O(n)或O(log n) |
