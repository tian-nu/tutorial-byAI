## 我在做什么？

上一章你学了 MVCC——它让不同事务看到不同版本的数据，实现了"读不阻塞写"。但 MVCC 只解决了**读**的问题，**写**的问题怎么办？

当两个事务同时修改同一行数据，或者修改的数据有重叠，就必须靠**锁（Lock）** *此术语见附录E* 来协调。锁就像红绿灯：绿灯通行的车（获得锁的事务）可以走，红灯的车（等待锁的事务）必须等。

这一章我们从上到下捋一遍 MySQL 的锁体系：全局锁 → 表锁 → 行锁 → 间隙锁 → 临键锁，然后学死锁排查和乐观锁/悲观锁策略。

学完这一章，你能：
- 说出 InnoDB 三种行锁的区别
- 理解间隙锁为什么能解决幻读
- 制造并排查一个死锁
- 在业务中选择乐观锁还是悲观锁

## 一、锁分类总览：从上到下，越来越细

MySQL 的锁从大到小分为三层：

| 层级 | 锁类型 | 粒度 | 影响范围 | 并发性能 |
|------|--------|------|---------|---------|
| 全局 | 全局读锁（FTWRL） | 整个数据库 | 所有表只能读，不能写 | 最差 |
| 表级 | MDL、表锁 | 整张表 | 影响整张表的读写 | 较差 |
| 行级 | 记录锁、间隙锁、临键锁 | 单行或间隙 | 只影响特定行 | 最好 |

**记住一句话：锁的粒度越小，并发性能越好，但实现越复杂。** InnoDB 支持行级锁，所以并发性能远好于 MyISAM（MyISAM 只有表锁）。

### 1.1 全局锁（FTWRL）

全称：Flash Tables With Read Lock。加了这个锁，**整个数据库实例变成只读**，所有表的写操作都会被阻塞。

```sql
-- 加全局读锁（危险！）
FLUSH TABLES WITH READ LOCK;

-- 释放全局锁
UNLOCK TABLES;
```

**什么时候用？** 做全库逻辑备份时，为了防止备份过程中数据变化。但 InnoDB 可以用 `mysqldump --single-transaction` 替代，不用锁全库。

**一句话总结**：你知道有这个东西就行，生产环境不要随便用。

### 1.2 表级锁

表级锁有两种：**MDL（元数据锁）** 和**表锁（Table Lock）**。

**MDL（Metadata Lock）** *此术语见附录E*：MySQL 自动加的，不需要你手动操作。当你执行 `SELECT` 时，MySQL 自动加 MDL 读锁；当你执行 `ALTER TABLE` 时，自动加 MDL 写锁。

MDL 的坑：如果你在 `ALTER TABLE` 时，有未提交的长事务持有 MDL 读锁，`ALTER TABLE` 会一直等待 MDL 写锁，后面的所有查询也会被阻塞——整个表"卡死"了。

**表锁（Table Lock）**：手动加的，用 `LOCK TABLES` 和 `UNLOCK TABLES`。

```sql
-- 显式加表锁（不推荐，InnoDB 有行锁就够了）
LOCK TABLES accounts READ;   -- 别人只能读，不能写
LOCK TABLES accounts WRITE;  -- 只有我能读写，别人全阻塞
UNLOCK TABLES;               -- 释放所有表锁
```

> ⚠️ 注意：`LOCK TABLES` 会触发隐式提交（第 21 章讲过），所以不要把它放在事务中间。

## 二、记录锁（Record Lock）：锁住单行

记录锁是最简单的行锁，直接锁住索引中的**一条记录**。

```sql
-- 对 id=1 的行加排他锁（X锁）
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
```

这就锁住了 `id=1` 这一行，其他事务不能修改也不能删除它，也不能对它加锁。

**关键是**：记录锁锁的是**索引记录**，不是数据行。如果 `WHERE` 条件没有命中索引，InnoDB 会退化为表锁（锁住所有行）！这也是为什么我们反复强调索引的重要性——没有索引，行锁升级为表锁。

---

## 三、间隙锁（Gap Lock）：锁住"不存在的数据"

间隙锁是 InnoDB 特有的锁，用来**防止别人在"间隙"中插入数据**。这正是 InnoDB 解决幻读的关键武器。

### 3.1 具体场景演示

假设 `accounts` 表中只有 `id=1, 5, 10` 三条记录：

```sql
-- 当前数据
SELECT id FROM accounts ORDER BY id;
-- +----+
-- | id |
-- +----+
-- |  1 |
-- |  5 |
-- | 10 |
-- +----+
```

InnoDB 把数据分成这样的间隙：

```
(-∞, 1] → (1, 5] → (5, 10] → (10, +∞)
```

现在执行：

```sql
SELECT * FROM accounts WHERE id BETWEEN 3 AND 7 FOR UPDATE;
```

这会锁住什么？**不是只锁 id=5 这一行！** 而是锁住：
- 记录锁：`id=5`（因为 5 在 3~7 之间）
- 间隙锁：`(1, 5]` 和 `(5, 10]` 两个间隙

**效果**：别人不能插入 id 在 3~10 之间的任何数据（包括 id=3,4,6,7,8,9），也不能修改 id=5。

### 3.2 间隙锁的特点

1. **间隙锁之间不冲突**：两个事务可以同时锁同一个间隙，因为它们锁的是"插入"这个动作，不是数据本身
2. **间隙锁只在 REPEATABLE READ 及以上级别生效**：READ COMMITTED 下没有间隙锁
3. **间隙锁的目的是防止幻读**：锁住间隙，别人就不能插入新行，你的范围查询结果就不会变

---

## 四、临键锁（Next-Key Lock）：记录锁 + 间隙锁

临键锁是 InnoDB 的**默认行锁**，它 = 记录锁 + 间隙锁。也就是说，锁住一行，同时锁住它前面的间隙。

还是用刚才的数据（id=1, 5, 10）：

```sql
SELECT * FROM accounts WHERE id = 5 FOR UPDATE;
```

在 REPEATABLE READ 下，实际锁住的是：
- 记录锁：`id=5`
- 间隙锁：`(1, 5]`（id=5 前面的间隙也被锁住了）

这意味着：**你不能在 id=5 前面插入新数据（比如 id=3），也不能修改 id=5。**

**为什么 InnoDB 默认用临键锁？** 因为它的设计目标就是"在 REPEATABLE READ 下解决幻读"。如果只锁记录不锁间隙，别人插入新数据就会产生幻读。

---

## 五、共享锁 vs 排他锁

这两种锁是锁的"性格"分类，无论记录锁还是临键锁，都有共享和排他之分。

| 锁类型 | 英文 | 符号 | 含义 | 冲突规则 |
|--------|------|------|------|---------|
| 共享锁 | Shared Lock | S 锁 | 我能读，你也能读，但你不能写 | S 和 S 兼容，S 和 X 冲突 |
| 排他锁 | Exclusive Lock | X 锁 | 只能我读写，你什么都不能做 | X 和一切冲突 |

### 5.1 共享锁（S 锁）

```sql
-- 加共享锁，别人可以读，但不能写
SELECT * FROM accounts WHERE id = 1 LOCK IN SHARE MODE;
-- MySQL 8.0.22+ 可以用新语法
SELECT * FROM accounts WHERE id = 1 FOR SHARE;
```

**使用场景**：你要读一行数据，并且不希望别人在你看的过程中修改它。比如查库存，你希望库存数量在你计算的过程中不变。

### 5.2 排他锁（X 锁）

```sql
-- 加排他锁，别人不能读（不能加锁读），也不能写
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
```

**使用场景**：你要修改一行数据，先锁住它，防止别人也改。比如扣库存，先锁住库存行，再扣减。

### 5.3 兼容矩阵

|  | 已有 S 锁 | 已有 X 锁 |
|--|----------|----------|
| 请求 S 锁 | ✅ 兼容 | ❌ 冲突 |
| 请求 X 锁 | ❌ 冲突 | ❌ 冲突 |

**一句话总结**：S 锁只防写，X 锁防一切。

## 六、死锁：两个事务互相等，谁也动不了

死锁是并发编程中最蛋疼的问题之一。在数据库中，死锁就是：**事务 A 等事务 B 释放锁，事务 B 又等事务 A 释放锁，两个人都卡住了。**

### 6.1 死锁产生的四个条件

1. **互斥**：资源不能共享，一次只能一个事务用
2. **持有并等待**：事务持有一个锁，又在等另一个锁
3. **不可剥夺**：锁不能被强行拿走，只能等事务自己释放
4. **循环等待**：A 等 B，B 等 A，形成环形等待

### 6.2 制造一个死锁（两个窗口）

```sql
-- 窗口 1：事务 A
BEGIN;
-- 先锁 id=1
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
-- 等 5 秒，让窗口 2 也锁住一行
-- 再锁 id=2
SELECT * FROM accounts WHERE id = 2 FOR UPDATE;  -- 这里会阻塞，等待窗口 2 释放
```

```sql
-- 窗口 2：事务 B
BEGIN;
-- 先锁 id=2（和窗口 1 顺序相反！）
SELECT * FROM accounts WHERE id = 2 FOR UPDATE;
-- 等 5 秒
-- 再锁 id=1
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;  -- 这里也会阻塞，形成死锁！
```

**结果**：其中一个窗口会报错：
```
ERROR 1213 (40001): Deadlock found when trying to get lock; try restarting transaction
```

MySQL 检测到死锁后，会自动回滚其中一个事务（通常选择回滚代价较小的那个），让另一个事务继续。

### 6.3 排查死锁

```sql
-- 查看最新的死锁信息
SHOW ENGINE INNODB STATUS\G;
```

在输出的信息中，找到 `LATEST DETECTED DEADLOCK` 这一段，里面会详细记录：
- 哪个事务被回滚了
- 两个事务分别持有哪些锁
- 两个事务分别在等待哪些锁
- 涉及的 SQL 语句

**读懂死锁日志的关键**：

```
*** (1) TRANSACTION:  -- 事务 1
-- 持有锁：RECORD LOCKS ... id=1
-- 等待锁：RECORD LOCKS ... id=2

*** (2) TRANSACTION:  -- 事务 2
-- 持有锁：RECORD LOCKS ... id=2
-- 等待锁：RECORD LOCKS ... id=1

*** WE ROLL BACK TRANSACTION (2)  -- 回滚了事务 2
```

### 6.4 预防死锁

**最有效的预防方法：按相同顺序加锁。**

```sql
-- ❌ 事务 A 先锁 id=1 再锁 id=2，事务 B 先锁 id=2 再锁 id=1 → 死锁！
-- ✅ 所有事务都先锁 id=1 再锁 id=2，或者先锁 id=2 再锁 id=1 → 不死锁
```

**其他预防措施**：
- 事务尽量小，尽快提交（锁持有时间短，冲突概率低）
- 避免在事务中执行耗时操作（比如调用外部 API）
- 如果业务允许，用 `SELECT ... FOR UPDATE NOWAIT`（MySQL 8.0+）——如果没有锁，立即报错而不是等待

## 七、乐观锁 vs 悲观锁：两种并发控制策略

锁可以分为"肢体锁"（悲观锁）和"思想锁"（乐观锁）——不对，正经说：

### 7.1 悲观锁（Pessimistic Lock）

**假设"一定会有人跟我抢"，所以先锁住，再操作。**

```sql
-- 悲观锁：先锁住，再修改
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;  -- 加排他锁
-- 检查余额够不够...
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

**优点**：安全，绝对不会冲突。
**缺点**：锁住期间别人全得等，并发性能差。

**适用场景**：冲突概率高，比如秒杀扣库存。

### 7.2 乐观锁（Optimistic Lock）

**假设"大概率没人跟我抢"，先操作，提交时检查有没有冲突。**

```sql
-- 乐观锁：用版本号 version
-- 1. 先读出版本号
SELECT balance, version FROM accounts WHERE id = 1;
-- 假设读到 balance=1000, version=5

-- 2. 修改时带版本号条件
UPDATE accounts 
SET balance = balance - 100, version = version + 1 
WHERE id = 1 AND version = 5;  -- 关键！只有版本号没变才更新

-- 3. 检查 affected rows
-- 如果 affected rows = 1 → 更新成功，提交
-- 如果 affected rows = 0 → 版本号变了，说明别人改过了 → 重试
```

**优点**：不锁数据，并发性能高。
**缺点**：如果冲突频繁，会大量重试，反而降低性能。

**适用场景**：冲突概率低，比如用户修改自己的个人信息。

### 7.3 对比总结

| 维度 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| 核心操作 | `SELECT ... FOR UPDATE` | 版本号 `WHERE version = ?` |
| 是否阻塞 | 是，会阻塞其他事务 | 否，失败了重试 |
| 冲突多时 | 等待时间长 | 重试次数多 |
| 冲突少时 | 不必要的锁开销 | 无锁，性能好 |
| 实现复杂度 | 简单 | 需要在表中加 version 字段 |
| 典型场景 | 库存扣减、余额转账 | 用户资料修改、文章编辑 |

## 八、锁等待超时

如果事务 A 锁住了数据不释放，事务 B 一直等，等多久？

```sql
-- 查看锁等待超时时间（默认 50 秒）
SHOW VARIABLES LIKE 'innodb_lock_wait_timeout';

-- 设置锁等待超时（当前会话）
SET innodb_lock_wait_timeout = 5;  -- 5 秒后报错
```

如果超过这个时间，事务 B 会报错：
```
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction
```

## 十、常见错误与排错

### 错误 1：没加索引导致行锁升级为表锁

**现象**：你只 `WHERE name = 'A'` 用 `FOR UPDATE`，结果整张表都被锁了，其他所有操作都卡住。

**原因**：`name` 列没有索引，InnoDB 需要扫描全表才能找到 `name='A'` 的行，这意味着它要锁住扫描到的每一行——也就是所有行。虽然叫"行锁"，但实际效果是表锁。

**解决**：
```sql
-- 检查 name 列有没有索引
SHOW INDEX FROM accounts;
-- 如果没有，加索引
CREATE INDEX idx_name ON accounts(name);
```

### 错误 2：SELECT FOR UPDATE 忘记 COMMIT 导致锁不释放

**现象**：你执行了 `SELECT ... FOR UPDATE`，然后去上厕所了。回来发现所有涉及这几行的操作都卡住了。

**原因**：`FOR UPDATE` 加的排他锁，必须等事务提交或回滚才释放。如果你忘了 `COMMIT`，锁就一直不释放。

**解决**：事务写完尽快提交。如果发现锁没释放，找到那个事务：
```sql
-- 查看当前正在运行的事务
SELECT * FROM information_schema.INNODB_TRX;
-- 找到对应的事务 ID，强制 kill（谨慎！）
-- KILL <thread_id>;
```

### 错误 3：在 READ COMMITTED 下测试间隙锁，发现没有

**现象**：你按照教程测试间隙锁，结果发现别人还是能插入数据。

**原因**：间隙锁只在 REPEATABLE READ 及以上级别生效。READ COMMITTED 下没有间隙锁。

**解决**：确认你的隔离级别：
```sql
SELECT @@transaction_isolation;
-- 如果不是 REPEATABLE READ
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

### 错误 4：乐观锁忘记检查 affected rows

**现象**：你用了乐观锁的 `WHERE version = ?` 条件，但没检查 `affected rows`。结果并发场景下，你以为更新成功了，其实没更新。

**解决**：乐观锁的关键步骤：
```sql
UPDATE accounts SET balance = balance - 100, version = version + 1 
WHERE id = 1 AND version = 5;

-- 一定要检查更新行数！
-- 如果 affected rows = 0，说明版本号已经变了，需要重试
```
