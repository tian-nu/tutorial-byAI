# 第50章 · 数据库基础与MySQL

> "从第一章到现在，我们写的所有程序都有一个共同的问题——只要程序关了，数据就没了。用户注册了账号、发表的文章、上传的文件，统统消失在内存里。就像你在沙滩上写字，潮水一来全都抹平。数据库，就是那堵水泥墙，刻上去的东西，十年后还在。"

---

## 50.1 什么是数据库

想象你开了一家快递公司。刚开始只有三个客户，你用一张纸就能记下所有信息：

```
张三 - 13800001111 - 北京市朝阳区
李四 - 13800002222 - 上海市浦东新区
王五 - 13800003333 - 广州市天河区
```

后来客户发展到三千个，你翻纸找"李四"找到眼冒金星。你又用Excel建了个表，可以筛选、排序、搜索，爽了几天。

再后来客户到了三十万。Excel打开都要卡三分钟，而且你和同事同时编辑时就乱了——他改他的，你改你的，保存时互相覆盖。

**数据库**就是解决这些问题的终极方案。你可以把它想象成一个**全自动的电子化仓库**：

| 仓库概念 | 数据库概念 | 说明 |
|----------|-----------|------|
| 仓库本身 | Database（数据库） | 存放所有数据的大房子 |
| 货架 | Table（表） | 同一类东西放在同一个货架上 |
| 周转箱 | Row（行） | 一条完整的记录 |
| 箱子里的每个物品 | Column（列） | 记录的某个属性 |

Excel是一张纸，一个人趴在上面用笔改。数据库是一个全自动仓库，有传送带、机械臂、监视器，可以**同时服务几千个人**而不混乱。

> **数据库的核心能力**：存（增）、查、改、删——简称**CRUD**（Create、Read、Update、Delete），同时保证多人操作不出错。

🤔 **想多一点**：你可能会问——"都2026年了，为什么不直接用Excel存数据？"
1. **并发冲突**：Excel被小王打开后，小李只能以"只读"方式打开，否则互相覆盖。
2. **数据量限制**：Excel超过10万行就开始卡，数据库轻松处理亿级数据。
3. **关联复杂**：你的用户表和订单表需要关联查询，Excel只能用VLOOKUP死撑。
4. **安全控制**：数据库可以精确到"小李只能查用户表，不能改"、"小王只能查自己负责的订单"。
5. **崩溃恢复**：数据库写一半断电了，重启后能自动恢复到一致状态，Excel直接文件损坏。

## 50.2 关系型数据库 vs 非关系型数据库

数据库世界分为两大门派：

### 关系型数据库（SQL数据库）

**比喻**：一个**格式极其严格的仓库**。每个货架（表）在放东西之前，必须先定义好每层放什么类型的东西。第一个格子只能放整数，第二个格子只能放不超过50个字的文本……所有货架之间还可以用"编号"互相引用。

- 代表选手：**MySQL**、PostgreSQL、Oracle、SQL Server
- 数据以**表**的形式组织，表和表之间通过**外键**建立关系
- 使用**SQL**（结构化查询语言，详见附录I）操作，所有关系型数据库（以表的形式组织数据，表之间通过外键关联，详见附录I）的SQL大同小异
- 适合：用户系统、电商订单、银行账目——总之，**结构固定、数据之间有关联**的场景

### 非关系型数据库（NoSQL数据库）

**比喻**：不设固定格式，想放什么就放什么。有三种典型代表：

**Redis = 大脑中的便签纸**
写一个字只要0.00001秒，但容量受限于大脑（内存），关机就忘（但可以"拍照"存到硬盘）。适合：临时缓存、实时排行榜、消息队列。

**MongoDB = 文件柜**
不需要预先设计表格结构，直接把一沓文档（JSON格式）塞进柜子。适合：日志记录、内容管理系统、数据结构频繁变动的场景。

**Elasticsearch = 超级搜索引擎**
扔进去一堆文本，它能帮你瞬间找到包含某个关键词的所有内容。适合：全文搜索、日志分析。

| 对比维度 | 关系型（如MySQL） | 非关系型（如Redis/MongoDB） |
|----------|------------------|---------------------------|
| 数据结构 | 严格，先定义表结构 | 灵活，随存随改 |
| 关联能力 | 强（JOIN、外键） | 弱或无 |
| 事务支持 | 完善（ACID） | 有限或不支持 |
| 扩展方式 | 垂直扩展为主 | 水平扩展方便 |
| 典型场景 | 金融、电商、ERP | 缓存、搜索、日志 |

**这本书的策略**：MySQL作为主力学透，MongoDB了解即可，Redis作为"性能加速器"重点学。

## 50.3 MySQL是什么

MySQL是目前世界上**最流行的开源关系型数据库**。它由瑞典公司MySQL AB开发，后被Sun收购，Sun又被Oracle收购——绕了一大圈，现在归Oracle管。

**为什么后端教程选MySQL？**

1. **免费开源**：不用付费，社区版完全够用。
2. **市场份额大**：全球超过80%的互联网公司在用（或曾经用过）MySQL。
3. **生态成熟**：文档多、社区活跃、各种工具和库支持。
4. **够用且好用**：对绝大多数后端项目来说，MySQL的性能绰绰有余。

**MySQL版本选择**：
- **5.7**：经典长期支持版，很多老项目还在用，稳定可靠。
- **8.0**：新长期支持版（当前推荐），窗口函数、CTE、JSON增强、角色管理、降序索引等大量新特性。**本书以8.0为准**。
- 9.0：2024年刚发布，太新，不推荐用于生产环境。

如果你去面试，面试官说"我们用的MySQL 5.7"，你别慌。5.7和8.0的核心SQL语法95%一样，学完8.0看5.7基本无缝。

## 50.4 MySQL安装（Windows）

### 步骤一：下载

打开 https://dev.mysql.com/downloads/installer/ ，下载 `mysql-installer-community-8.0.xx.msi`。

> 选 **Custom** 安装模式，勾选 MySQL Server 8.0 和 MySQL Workbench（图形化管理工具）。

### 步骤二：安装过程

安装器会引导你完成：

1. **Choosing a Setup Type**：选择 `Developer Default`，会安装MySQL Server、Workbench、Shell、连接器等全套工具。
2. **Check Requirements**：如果提示缺少 Visual C++ Redistributable，点Execute自动安装。
3. **Type and Networking**：端口保持默认的 `3306`（这是MySQL的"门牌号"）。
4. **Authentication Method**：选 `Use Strong Password Encryption`（8.0推荐方式）。
5. **Accounts and Roles**：设置root用户密码。**记住这个密码！忘了就得重装！**
6. **Windows Service**：保持默认（作为Windows服务自动启动）。

### 步骤三：验证安装

打开命令提示符：

```bash
mysql -u root -p
```

输入你刚才设置的密码。如果出现：

```
Welcome to the MySQL monitor...
mysql>
```

恭喜你，安装成功！这个 `mysql>` 就是MySQL的命令行交互界面——你在里面输入SQL语句，它执行并返回结果。

### 图形化工具

虽然命令行很酷，但日常开发中，看表结构和数据用图形化工具更直观：

- **MySQL Workbench**：MySQL官方出品，免费，功能全面。
- **DBeaver**：开源免费，支持多种数据库，界面漂亮。
- **Navicat**：付费，UI体验最好，很多公司用。
- **DataGrip**：JetBrains出品，付费，IDE级别体验。

本书主要使用命令行的方式演示SQL——因为考你面试的也是命令行。

## 50.5 MySQL安装（Linux / Ubuntu）

如果你的开发环境是Linux（推荐），安装更简单：

```bash
sudo apt update
sudo apt install mysql-server -y
```

安装完成后启动服务：

```bash
sudo systemctl start mysql
sudo systemctl enable mysql
```

首次安装后root用户没有密码，通过socket认证登录：

```bash
sudo mysql
```

进入后设置root密码：

```sql
ALTER USER 'root'@'localhost' IDENTIFIED WITH mysql_native_password BY '你的密码';
FLUSH PRIVILEGES;
EXIT;
```

之后就可以用密码登录了：

```bash
mysql -u root -p
```

如果需要远程访问（比如你本机Windows连Linux服务器上的MySQL），还需要：

```bash
sudo vim /etc/mysql/mysql.conf.d/mysqld.cnf
```

把 `bind-address = 127.0.0.1` 改为 `bind-address = 0.0.0.0`，然后：

```sql
CREATE USER '你的用户名'@'%' IDENTIFIED BY '你的密码';
GRANT ALL PRIVILEGES ON *.* TO '你的用户名'@'%';
FLUSH PRIVILEGES;
```

🤔 **想多一点**：`'用户名'@'%'` 中的 `%` 表示"任何IP地址都可以连接"。生产环境不建议这样设——应该限定为具体IP，比如 `'用户名'@'192.168.1.100'`。黑客最爱的密码是 `root@'%'` + 弱密码。

## 50.6 MySQL基础架构

打开MySQL这个"仓库"的大门，你会发现它内部不是一块铁板，而是分层的精密结构：

```
┌──────────────────────────────────┐
│         连接层（连接/认证）        │  ← 你敲 mysql -u root -p 到达这里
├──────────────────────────────────┤
│         服务层（核心大脑）         │
│  ┌──────┐ ┌──────┐ ┌────────┐   │
│  │SQL解析│→│优化器│→│ 执行器  │   │
│  └──────┘ └──────┘ └────────┘   │
├──────────────────────────────────┤
│         引擎层（存储引擎）         │  ← InnoDB / MyISAM / Memory
├──────────────────────────────────┤
│         文件层（硬盘文件）         │  ← 最终数据存在这里
└──────────────────────────────────┘
```

**连接层**：管理客户端连接、用户认证、权限校验。你登录MySQL时就是和这一层在握手。

**服务层**：MySQL的核心大脑。它负责：

1. **SQL解析**：把你写的SQL语句拆解成语义单元。比如 `SELECT name FROM users WHERE id = 1` → 分解为"选择name字段、来自users表、条件是id等于1"。
2. **查询优化器**：决定怎么执行最快。比如 `WHERE a=1 AND b=2`，优化器会判断先按a过滤还是先按b过滤——它看索引统计信息做决策。
3. **执行器**：拿着优化器的方案，调用存储引擎的接口去真正干活。

**引擎层**：这是MySQL和其他数据库最大的不同——**可插拔的存储引擎**。你可以给不同的表选不同的"发动机"：

- **InnoDB**（默认，5.5版本以后）：支持事务、行级锁、外键、崩溃恢复。**生产环境只用这个**。
- **MyISAM**（老古董）：不支持事务，表级锁，速度快但并发差。适合只读数据仓库。
- **Memory**：数据全放在内存里，关机就丢。适合临时表。

**文件层**：最终把数据以文件形式存到硬盘上。InnoDB的数据存在 `.ibd` 文件里，表结构存在 `.frm` 文件里（8.0开始frm也被整合了）。

🤔 **想多一点**：面试官爱问"一条SQL从输入到返回结果经历了什么？"答案就是上面的流程：连接认证→解析SQL→优化执行计划→调用存储引擎API→从磁盘/内存读取数据→返回结果。这个流程你必须背下来。

## 50.7 数据库基本操作

登录MySQL后，先学几个最基本的"仓库管理命令"：

### 查看所有数据库

```sql
SHOW DATABASES;
```

你会看到：

```
+--------------------+
| Database           |
+--------------------+
| information_schema |
| mysql              |
| performance_schema |
| sys                |
+--------------------+
```

这四个是MySQL的**系统数据库**，存储着用户信息、权限、性能数据等。**千万别删它们！**

### 创建数据库

```sql
CREATE DATABASE my_shop;
```

这等于在仓库区划出一块地，门牌号叫 `my_shop`。现在列表里会多出一条：

```sql
SHOW DATABASES;
```

### 使用（进入）数据库

```sql
USE my_shop;
```

你会看到：`Database changed`。现在所有的SQL操作都默认在这个数据库里进行——就像你走进了 `my_shop` 这个仓库。

### 查看当前在哪个数据库

```sql
SELECT DATABASE();
```

### 删除数据库

```sql
DROP DATABASE my_shop;
```

**红色警报**：这个操作不可逆！一秒钟，仓库里所有货架、所有数据，全部消失。生产环境执行这个命令之前请默念三遍"我真的要删吗"。手势一滑就会背上"删库跑路"的罪名。

## 50.8 字符集

这是一个坑了无数新手的黑暗领域。

### 什么是字符集

计算机只认识0和1。怎么把"你好"变成0和1？需要一个翻译表——A方案：`你=1010 你=0001 好=0101`，B方案：`你=0011 你=0110 好=1001`。

这个翻译表就是**字符集**（Character Set）。同样的0和1，用A方案翻译出来是"你好"，用B方案翻译出来可能变成"???"——这就是著名的**乱码**。

### MySQL中的字符集陷阱

MySQL默认的 `utf8` 字符集其实**不是真正的UTF-8**！它最多只支持3个字节的字符，而emoji（比如😀）需要4个字节。

如果你建表时用了 `utf8`，用户存了一个emoji昵称进去，MySQL会报错，或者把emoji截断成乱码。

**解决方案**：永远用 `utf8mb4`（UTF-8 Most Bytes 4，即完整的四字节UTF-8）。

### 创建数据库时指定字符集

```sql
CREATE DATABASE my_shop
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;
```

`COLLATE` 是排序规则。`utf8mb4_unicode_ci` 中的 `ci` 是 Case Insensitive（不区分大小写），即 `abc` 和 `ABC` 在比较时视为相同。

> **最佳实践**：创建任何数据库都加上 `CHARACTER SET utf8mb4`。养成肌肉记忆。
