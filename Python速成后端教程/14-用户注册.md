## 一、目标与完成效果

**一句话目标**：实现用户注册接口——用户提交用户名和密码，服务器把密码变成"土豆泥"（哈希）再存进数据库，即使数据库被偷，黑客也拿不到用户的真实密码。

**完成后的可观测效果**：
- 你用 curl 向 `POST /users/register` 发了一个注册请求，服务器返回了注册成功的响应，包含用户 ID 和昵称。
- 你打开 `blog.db` 数据库，查看 `users` 表，发现刚才注册的用户名确实存在，但密码字段不是 `123456`，而是一串像 `$2b$12$LJ3m4...` 这样的乱码——这就是哈希值。
- 你尝试用同一个用户名再次注册，服务器返回 400 错误，提示"用户名已被占用"——唯一性约束生效了。
- 你尝试用 2 个字符的用户名注册，服务器返回 422 验证错误，提示"用户名长度至少 3 个字符"——Pydantic 校验生效了。
- 你能用"土豆泥"这个比喻，给你不懂编程的朋友讲清楚哈希是什么。

### 步骤 1：注册逻辑全景——从"用户输入"到"数据入库"发生了什么？

在开始写代码之前，我们先花 3 分钟搞清楚整个注册流程。

闭上眼睛想一想你在任何网站注册账号的体验：
1. 你填了用户名和密码
2. 你点了"注册"按钮
3. 系统告诉你"注册成功"（或者"用户名已被占用"）

**这 3 秒内，服务器端发生了什么？**

```
用户提交 {username: "alice", password: "123456"}
          │
          ▼
    ┌─────────────┐
    │ 1. Pydantic │  ← 校验：用户名长度够不够？密码长度够不够？
    │    校验     │     不通过 → 直接返回 422，后面的步骤全跳过
    └──────┬──────┘
           │ 通过
           ▼
    ┌─────────────┐
    │ 2. 唯一性   │  ← 检查：数据库里有没有人已经用这个用户名注册了？
    │    检查     │     有 → 返回 400 "用户名已被占用"
    └──────┬──────┘
           │ 无重复
           ▼
    ┌─────────────┐
    │ 3. 密码哈希 │  ← 把 "123456" 变成 "$2b$12$LJ3m4..." 这种乱码
    │             │     原密码从此消失，只存哈希值
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 4. 存入数据库│  ← INSERT INTO users (username, password_hash, ...)
    │             │     密码字段存的是哈希值，不是 "123456"
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 5. 返回响应 │  ← {id: 1, username: "alice", nickname: "alice"}
    └─────────────┘
```

这五步中，第 3 步（密码哈希）是本章的核心新知识。其余四步你已经在前面的教程中学过了——Pydantic 校验（教程 06）、数据库查询（教程 11）、存数据（教程 13）、返回响应（教程 04）。

**所以本章本质上只教一件新东西：密码怎么安全地存储。** 其他的都是"把学过的技能串起来用"。

### 步骤 3：理解哈希——"把土豆做成土豆泥"

这是本章最核心的概念。请花 5 分钟认真理解，不要跳。

#### 3.1 哈希是什么？

**哈希（Hash）是一种"单向"的数学运算**：把任意长度的输入，变成固定长度的输出。这个输出就是"哈希值"。

**关键属性**：

| 属性 | 含义 | 比喻 |
|------|------|------|
| 确定性 | 同样的输入，永远得到同样的输出 | 同一颗土豆，用同样的机器压出来的土豆泥，形状永远一样 |
| 单向性 | 从输出无法反推输入 | 给你一碗土豆泥，你不可能还原出那颗土豆原来的样子 |
| 雪崩效应 | 输入改一点点，输出面目全非 | 土豆上划一道小口子，压出来的土豆泥纹路完全不同 |
| 定长输出 | 不管输入多长，输出长度固定 | 不管土豆多大，绞肉机出来的土豆泥都是统一的细度 |

**土豆泥比喻**：

- 土豆 = 用户的原始密码（比如 `123456`）
- 绞肉机 = 哈希算法（比如 bcrypt）
- 土豆泥 = 哈希值（比如 `$2b$12$LJ3m4...`）

**这个比喻的精髓**：
- 你可以把土豆做成土豆泥（正向计算），但**你永远不可能把土豆泥还原成土豆**（逆向不可能）。
- 但如果你有两碗土豆泥，你可以对比它们是不是**同一颗土豆**做的（验证密码时就是这样——把用户输入的密码哈希一下，和数据库里的哈希值对比，看是否一致）。
- 同一颗土豆，用同一台机器压，出来的土豆泥永远一模一样（确定性）。

**为什么不能存明文密码？** 此术语需进附录。

假设你的数据库被黑客偷走了。如果密码是明文存的：

```
users 表：
┌────┬──────────┬──────────┐
│ id │ username │ password │
├────┼──────────┼──────────┤
│ 1  │ alice    │ 123456   │  ← 黑客直接看到了密码！
│ 2  │ bob      │ password │  ← 黑客直接看到了密码！
└────┴──────────┴──────────┘
```

黑客拿到了所有人的真实密码，可以去尝试登录这些用户的邮箱、银行、其他网站（因为很多人多个网站用同一个密码）。

如果密码是哈希存的：

```
users 表：
┌────┬──────────┬──────────────────────────────────┐
│ id │ username │ password_hash                    │
├────┼──────────┼──────────────────────────────────┤
│ 1  │ alice    │ $2b$12$LJ3m4xK8...              │  ← 黑客看到的是乱码
│ 2  │ bob      │ $2b$12$Y7n9pQ2r...              │  ← 黑客看到的是乱码
└────┴──────────┴──────────────────────────────────┘
```

黑客只看到一堆乱码，无法反推出真实密码。即使黑客拿到的哈希值去尝试登录，服务器端验证时也会再次哈希后对比，不会直接比对明文。

> 🤔 想多一点：哈希和加密有什么区别？

**这是初学者最容易混淆的概念对。**

| | 哈希（Hash） | 加密（Encryption） |
|---|---|---|
| 方向 | 单向（不可逆） | 双向（可逆） |
| 密钥 | 不需要密钥 | 需要密钥 |
| 目的 | 验证数据是否一致 | 保护数据不被偷看 |
| 比喻 | 土豆 → 土豆泥（不可逆） | 信件 → 密码箱（有钥匙就能打开） |
| 典型算法 | bcrypt, SHA-256 | AES, RSA |

**加密**是"锁上"——有钥匙就能打开，拿回原文。**哈希**是"搅碎"——碎了就是碎了，永远拿不回原文。

**密码存储用哈希，不用加密。** 因为你不希望任何人（包括你自己）能"还原"出用户的密码。你只需要在用户登录时，验证他输入的密码和注册时的密码是否一致——而哈希正好能做这件事（同样输入 → 同样输出）。

### 步骤 4：创建 `auth.py` —— 密码哈希工具箱

现在我们要创建一个新文件，专门负责密码的哈希和验证。在 `blog_backend/` 目录下新建 `auth.py`。

**我在做什么？** 创建一个"密码哈希工具箱"，对外提供两个功能：① 把明文密码变成哈希值（注册时用）；② 验证明文密码和哈希值是否匹配（登录时用）。

**代码**（`blog_backend/auth.py` 完整版）：

```python
from passlib.context import CryptContext

# === 创建密码哈希上下文 ===
# schemes=["bcrypt"]: 使用 bcrypt 算法
# deprecated="auto": 自动处理过时的哈希算法（以后升级算法时有用）
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")


def hash_password(password: str) -> str:
    """把明文密码变成哈希值（注册时用）。

    比喻：把土豆放进绞肉机，出来土豆泥。
    """
    return pwd_context.hash(password)


def verify_password(plain_password: str, hashed_password: str) -> bool:
    """验证明文密码和哈希值是否匹配（登录时用）。

    比喻：把两颗土豆都做成土豆泥，对比两碗土豆泥是不是同一颗土豆做的。

    Args:
        plain_password: 用户输入的明文密码
        hashed_password: 数据库里存的哈希值

    Returns:
        True 表示密码正确，False 表示密码错误
    """
    return pwd_context.verify(plain_password, hashed_password)
```

**逐行解释**：

- **`CryptContext`**：passlib 的核心类。它管理你使用的哈希算法和配置。此术语需进附录。
- **`schemes=["bcrypt"]`**：指定使用 bcrypt 算法。你也可以配置多个算法作为备选——比如 `schemes=["bcrypt", "sha256_crypt"]`——passlib 会用第一个算法（bcrypt）来哈希新密码，验证时自动识别已有的哈希值用的是哪个算法。
- **`deprecated="auto"`**：如果以后你把 bcrypt 标记为"过时"并升级到更安全的算法，passlib 会自动在用户下次登录时用新算法重新哈希密码。这是一个"先埋好管道"的配置，本章暂时用不到，但留着它不碍事。
- **`pwd_context.hash(password)`**：输入明文密码，返回哈希值。输出格式像 `$2b$12$LJ3m4xK8abcdef...`。每次调用都会生成不同的哈希值（因为 bcrypt 每次生成一个随机"盐"——后面会讲），但 `verify()` 总能正确识别。
- **`pwd_context.verify(plain, hashed)`**：输入明文密码和哈希值，返回 `True` 或 `False`。它内部是这样工作的：从哈希值中提取"盐"和"轮数"等参数，用同样的参数对明文密码做哈希，然后对比两个哈希值是否一致。

> 🤔 想多一点："盐"（Salt）是什么？

哈希有个问题：如果两个用户的密码都是 `123456`，哈希后的结果是一样的。黑客可以预先计算常见密码的哈希值（这叫"彩虹表攻击"），然后跟数据库里的哈希值对比，就能反推出密码。

**盐（Salt）就是给每个密码加的"随机调料"。** 即使两个用户的密码完全一样，因为她们的盐不同，哈希值也完全不同。盐是随机生成的，存在哈希值里面（bcrypt 的哈希值格式已经包含了盐），验证时自动提取。

```
用户 A 密码: 123456 + 盐_randomA → 哈希: $2b$12$randomA$$$$$$$$$$$$$$$$$$...
用户 B 密码: 123456 + 盐_randomB → 哈希: $2b$12$randomB$$$$$$$$$$$$$$$$$$...
```

两个哈希值完全不一样！黑客无法通过"预计算"来破解。

### 步骤 5：扩展 User 模型 —— 加 `password_hash` 和 `nickname`

在教程 12 中，我们创建了 `User` 模型，但当时只有 `id`、`username`、`email`、`created_at` 四个字段。现在我们要扩展它，加上密码哈希和昵称。

**打开 `blog_backend/models.py`**，找到 `User` 类，把它改成下面这样。

**修改前（旧版）**：

```python
class User(Base):
    __tablename__ = "users"

    id = Column(Integer, primary_key=True, index=True)
    username = Column(String(50), unique=True, nullable=False)
    email = Column(String(100), unique=True, nullable=False)
    created_at = Column(DateTime, default=datetime.utcnow)

    posts = relationship("Post", back_populates="author")
```

**修改后（新版）**：

```python
class User(Base):
    __tablename__ = "users"

    id = Column(Integer, primary_key=True, index=True)
    username = Column(String(50), unique=True, nullable=False, index=True)
    password_hash = Column(String(128), nullable=False)  # 新增：密码哈希值，不是明文！
    nickname = Column(String(50), nullable=True)          # 新增：显示昵称
    email = Column(String(100), unique=True, nullable=True)  # 改为可选
    created_at = Column(DateTime, default=datetime.utcnow)

    posts = relationship("Post", back_populates="author")
```

**逐行解释**：

- **`password_hash`**：`String(128)` 够存 bcrypt 的哈希值了。`nullable=False` 表示每个用户必须有密码。**注意：这个字段存的是哈希值，不是明文密码。** 接口层永远不应该把明文密码传给这个字段——只传哈希值。
- **`nickname`**：`String(50)`，用于显示。`nullable=True` 表示昵称可选——如果用户不填，后端可以自动用用户名作为默认昵称。
- **`email`**：从 `nullable=False` 改为 `nullable=True`。因为本章的注册接口暂时不需要邮箱，但保留这个字段方便以后扩展。
- **`username`**：加了 `index=True`，因为注册时要频繁查询"用户名是否已存在"，建索引能加速查询。

**重要：修改了模型后，需要重建数据库表！**

因为 `users` 表的结构变了，旧表和新模型不匹配。最简单的方式是删除旧数据库文件，让 SQLAlchemy 重新建表：

```powershell
# 停止服务器（Ctrl+C）
# 删除旧数据库文件
del blog.db
# 重新启动服务器
uvicorn blog_backend.main:app --reload
```

服务器启动时，`Base.metadata.create_all(bind=engine)` 会自动按新模型建表。

> ⚠️ **警告**：`del blog.db` 会删除所有数据！如果你之前创建了测试文章，它们会一起消失。在实际项目中，你应该用数据库迁移工具（如 Alembic）来修改表结构，而不是删库重建。本章为了教学方便，直接删库重建。

### 步骤 6：扩展 Pydantic Schemas —— 注册请求和响应

在教程 06 中，我们创建了 `schemas.py` 来定义 `BlogCreate`。现在我们需要加两个新的 Pydantic 模型：用户注册的请求体和响应体。

**打开 `blog_backend/schemas.py`**，追加以下内容：

```python
from pydantic import BaseModel, Field


class BlogCreate(BaseModel):
    title: str
    content: str
    published: bool = False
    summary: str | None = None


# === 新增：用户注册相关 ===

class UserCreate(BaseModel):
    """用户注册请求体"""
    username: str = Field(
        min_length=3,
        max_length=20,
        description="用户名，3-20 个字符",
    )
    password: str = Field(
        min_length=6,
        max_length=100,
        description="密码，至少 6 个字符",
    )
    nickname: str | None = Field(
        default=None,
        max_length=50,
        description="昵称，不填则使用用户名",
    )


class UserResponse(BaseModel):
    """用户注册/登录成功后的响应体（不包含密码哈希！）"""
    id: int
    username: str
    nickname: str | None
    created_at: str

    class Config:
        from_attributes = True  # 允许从 ORM 对象直接转换
```

**逐行解释**：

- **`UserCreate`**：这是用户注册时前端发来的 JSON 格式。此术语需进附录：Pydantic model 就是"数据格式定义"——告诉 FastAPI "这个接口接收的数据长什么样"。
  - `username`：`min_length=3, max_length=20`——用户名必须在 3 到 20 个字符之间。
  - `password`：`min_length=6`——密码至少 6 个字符。**注意：`max_length=100` 不是限制用户密码长度，而是限制 HTTP 请求中这个字段的字符数上限。** 因为哈希后只有 60 个字符，原始密码多长都可以（但太长了也没必要，设个合理上限）。
  - `nickname`：可选字段，不填则为 `None`。
  - `Field` 是 Pydantic 提供的字段描述工具，`description` 参数会出现在 Swagger 文档中，方便前端开发者阅读。
- **`UserResponse`**：这是注册成功后返回给前端的 JSON 格式。**注意：它没有 `password` 字段，也没有 `password_hash` 字段！** 即使数据库里存了哈希值，也不应该返回给客户端——安全原则：永远不要返回任何与密码相关的数据。
  - `Config.from_attributes = True`：允许 FastAPI 从 SQLAlchemy 的 ORM 对象直接转换为 `UserResponse`。这是 Pydantic v2 的写法（旧版是 `orm_mode = True`）。

**❌ 常见错误 → ✅ 正确示例**：

❌ 错误：把 `password_hash` 放在 `UserResponse` 里

```python
class UserResponse(BaseModel):
    id: int
    username: str
    password_hash: str  # ❌ 错误！把哈希值返回给前端了
```

**后果**：虽然哈希值不能反推出密码，但把哈希值暴露给前端增加了安全风险——攻击者可以离线暴力破解哈希值。

✅ 正确：`UserResponse` 只包含不敏感的信息

```python
class UserResponse(BaseModel):
    id: int
    username: str
    nickname: str | None
    created_at: str
```

**新增注册接口**（放在文件末尾，已有接口的下面）：

```python
# === 用户注册 ===
@app.post("/users/register", response_model=UserResponse, status_code=201)
def register(user: UserCreate, db: Session = Depends(get_db)):
    # 1. 检查用户名是否已存在（唯一性约束）
    existing_user = db.query(models.User).filter(
        models.User.username == user.username
    ).first()
    if existing_user is not None:
        raise HTTPException(
            status_code=400,
            detail=f"用户名 '{user.username}' 已被占用",
        )

    # 2. 把明文密码变成哈希值（土豆 → 土豆泥）
    hashed_pw = hash_password(user.password)

    # 3. 创建用户 ORM 对象
    new_user = models.User(
        username=user.username,
        password_hash=hashed_pw,  # ← 存的是哈希值，不是明文！
        nickname=user.nickname if user.nickname else user.username,  # 没填昵称就用用户名
    )

    # 4. 写入数据库
    db.add(new_user)
    db.commit()
    db.refresh(new_user)

    # 5. 返回用户信息（不包含密码！）
    return new_user
```

**逐行解释，这一步很重要，别跳！**：

1. **`response_model=UserResponse`**：告诉 FastAPI 返回的 JSON 格式应该按 `UserResponse` 来。即使 `new_user` 对象里有 `password_hash` 字段，FastAPI 也不会把它放进返回的 JSON 里——因为 `UserResponse` 没有定义这个字段。**这是 FastAPI 自动帮你做的"安全过滤"——比手动删字段更可靠。**

2. **`db.query(models.User).filter(models.User.username == user.username).first()`**：查询数据库，看有没有人已经用这个用户名注册了。如果有，`existing_user` 不是 `None`，就返回 400 错误。**这就是唯一性约束在代码层的实现。** 此术语需进附录：唯一性约束 = 数据库中某个字段的值不能重复，本章通过"先查后插"来实现。

3. **`hash_password(user.password)`**：调用我们刚才在 `auth.py` 里写的函数，把明文密码 `"123456"` 变成哈希值 `"$2b$12$LJ3m4..."`。**从这一步起，明文密码就消失了**——后续所有代码只操作哈希值，永远不碰明文密码。

4. **`models.User(password_hash=hashed_pw)`**：创建 ORM 对象时，`password_hash` 字段存的是哈希值，不是明文。**这行代码是整个安全体系的关键——如果你不小心传了 `password_hash=user.password`（明文），那就前功尽弃了。**

5. **`db.add(new_user)` → `db.commit()` → `db.refresh(new_user)`**：这个三步走你已经很熟了——加入待办 → 写入数据库 → 读取数据库生成的字段（`id`、`created_at`）。

6. **`return new_user`**：直接返回 ORM 对象。FastAPI 看到 `response_model=UserResponse`，会自动过滤掉 `password_hash` 字段，只返回 `id`、`username`、`nickname`、`created_at`。

**为什么还要数据库层面的唯一性约束？**

你可能注意到了，我们在代码里手动检查了用户名是否重复，但 `models.py` 里 `username` 字段也设了 `unique=True`。**这不是重复，是双重保险。**

- **代码层检查**：能给出友好的错误提示（"用户名 'alice' 已被占用"），用户体验好。
- **数据库层检查**：是最后一道防线。如果代码有 bug 导致跳过了检查，数据库的 `UNIQUE` 约束会拒绝插入，不会产生脏数据。

**两道防线比一道更安全。** 就像你家门有锁（代码层），小区还有门禁（数据库层）——小偷过了第一道，还有第二道。

### 步骤 8：Pydantic 校验详情 —— 什么能过，什么不能过？

现在我们来梳理一下，Pydantic 到底帮我们校验了什么。

**场景 1：用户名太短**

```bash
curl -X POST http://127.0.0.1:8000/users/register ^
  -H "Content-Type: application/json" ^
  -d "{\"username\": \"ab\", \"password\": \"123456\"}"
```

❌ 返回 422：

```json
{
  "detail": [
    {
      "type": "string_too_short",
      "loc": ["body", "username"],
      "msg": "String should have at least 3 characters",
      "input": "ab"
    }
  ]
}
```

**Pydantic 在请求体到达接口函数之前就拦截了它。** 你的 `register` 函数根本不会执行——422 是 FastAPI 自动返回的。

**场景 2：密码太短**

```bash
curl -X POST http://127.0.0.1:8000/users/register ^
  -H "Content-Type: application/json" ^
  -d "{\"username\": \"alice\", \"password\": \"12345\"}"
```

❌ 返回 422：`"String should have at least 6 characters"`

**场景 3：缺少必填字段**

```bash
curl -X POST http://127.0.0.1:8000/users/register ^
  -H "Content-Type: application/json" ^
  -d "{\"username\": \"alice\"}"
```

❌ 返回 422：`"Field required"` 指向 `password` 字段。

**场景 4：用户名太长**

```bash
curl -X POST http://127.0.0.1:8000/users/register ^
  -H "Content-Type: application/json" ^
  -d "{\"username\": \"this_username_is_way_too_long\", \"password\": \"123456\"}"
```

❌ 返回 422：`"String should have at most 20 characters"`

**场景 5：一切正常**

```bash
curl -X POST http://127.0.0.1:8000/users/register ^
  -H "Content-Type: application/json" ^
  -d "{\"username\": \"alice\", \"password\": \"123456\"}"
```

✅ 返回 201：

```json
{
  "id": 1,
  "username": "alice",
  "nickname": "alice",
  "created_at": "2026-06-05T12:00:00"
}
```

**Pydantic 校验汇总**：

| 校验规则 | 定义位置 | 不通过时 |
|----------|----------|----------|
| 用户名长度 3-20 | `schemas.py` 中 `Field(min_length=3, max_length=20)` | 422 |
| 密码长度 ≥ 6 | `schemas.py` 中 `Field(min_length=6)` | 422 |
| 昵称长度 ≤ 50 | `schemas.py` 中 `Field(max_length=50)` | 422 |
| 用户名不能重复 | `main.py` 中手动检查 + 数据库 UNIQUE 约束 | 400 |

### 步骤 9：❌ 错误示例 vs ✅ 正确示例 —— 存明文密码的灾难

这是本章最重要的 ❌→✅ 对比。请认真看。

#### ❌ 错误示例：直接存明文密码

```python
# ❌ 危险！永远不要这样做！
@app.post("/users/register")
def register_bad(user: UserCreate, db: Session = Depends(get_db)):
    new_user = models.User(
        username=user.username,
        password_hash=user.password,  # ❌ 直接把明文密码存进去了！！！
        nickname=user.nickname,
    )
    db.add(new_user)
    db.commit()
    return new_user
```

**这个代码可以正常运行，接口也能成功返回——但它埋下了一颗定时炸弹。**

**后果链**：

1. 数据库里的 `password_hash` 列存的是 `"123456"`（明文）
2. 某天数据库文件被泄露（比如服务器被黑、备份文件被盗、Git 仓库误提交）
3. 攻击者打开 `blog.db`，看到所有人的密码
4. 攻击者用这些密码去尝试登录用户的邮箱、社交媒体、银行账户
5. **你的用户因为你的疏忽，在别的地方也遭受了损失**

**这就是为什么明文存储密码是"所有后端开发者的职业道德红线"。** 不是"建议不要"，是"永远不要"。

#### ✅ 正确示例：存哈希值

```python
# ✅ 正确！密码经过哈希后存储
@app.post("/users/register")
def register_good(user: UserCreate, db: Session = Depends(get_db)):
    # 检查用户名唯一性
    existing = db.query(models.User).filter(
        models.User.username == user.username
    ).first()
    if existing:
        raise HTTPException(status_code=400, detail="用户名已被占用")

    # 密码哈希（土豆 → 土豆泥）
    hashed_pw = hash_password(user.password)

    new_user = models.User(
        username=user.username,
        password_hash=hashed_pw,  # ✅ 存的是哈希值，不是明文
        nickname=user.nickname or user.username,
    )
    db.add(new_user)
    db.commit()
    db.refresh(new_user)
    return new_user
```

**对比效果**：

| | ❌ 错误做法 | ✅ 正确做法 |
|---|---|---|
| 数据库里存的 | `123456` | `$2b$12$LJ3m4xK8abcdef...` |
| 数据库被偷后 | 攻击者看到所有人的密码 | 攻击者只看到乱码，无法反推密码 |
| 身为开发者 | 你能看到用户的密码 | 你也不知道用户的密码是什么 |
| 职业道德 | 不及格 | 合格 |

> 🔥 **记住：你应该永远不知道用户的密码是什么。** 如果你能通过数据库查到用户的密码，你的系统就是有问题的。

from passlib.context import CryptContext

# === 创建密码哈希上下文 ===
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")


def hash_password(password: str) -> str:
    """把明文密码变成哈希值（注册时用）。"""
    return pwd_context.hash(password)


def verify_password(plain_password: str, hashed_password: str) -> bool:
    """验证明文密码和哈希值是否匹配（登录时用）。"""
    return pwd_context.verify(plain_password, hashed_password)

**第 6 步是最终判决**：如果 `password_hash` 列的值是以 `$2b$12$` 开头的乱码，本章通过。如果看到的是明文密码（`mysecret123`、`bobpass123`），你的代码有 bug——回头检查步骤 7。

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| 把明文密码存进数据库 | 接口正常，但数据库里密码是明文 | 没调用 `hash_password()`，直接把 `user.password` 赋给了 `password_hash` | 永远用 `hash_password(user.password)` 之后再存 |
| 修改 models.py 后没重建数据库 | 启动报错 `no such column: users.password_hash` | 旧表没有新加的列，SQLAlchemy 的 `create_all` 不会自动修改已有表 | 删掉 `blog.db` 重新建表（开发阶段），或使用 Alembic 迁移（生产环境） |
| `response_model` 里包含了 `password_hash` | 注册成功返回的 JSON 里包含了哈希值 | 在 `UserResponse` 里定义了 `password_hash` 字段 | 从 `UserResponse` 中删掉 `password_hash` 字段 |
| 忘记 `pip install "passlib[bcrypt]"` 的方括号 | 安装的 passlib 没有 bcrypt 支持 | `passlib` 和 `passlib[bcrypt]` 是不同的安装包，后者包含 bcrypt 算法库 | 用 `pip install "passlib[bcrypt]"`（注意方括号和引号） |
| 用户名唯一性只靠代码检查，没加数据库 UNIQUE 约束 | 并发注册时可能出现两个相同用户名 | 代码检查不是原子操作，两个请求同时查"用户名不存在"都能通过 | 在 `models.py` 中 `username` 字段加 `unique=True`，数据库层是最后的防线 |
| bcrypt 哈希值每次不同，误以为代码有 bug | 同一个密码两次 `hash()` 结果不一样 | bcrypt 每次生成随机盐，这是正常行为 | 用 `verify_password()` 来验证密码是否正确，不要直接对比哈希值 |
| 密码长度限制设得太小 | 用户用了密码管理器生成的 64 位密码，注册失败 | Pydantic 的 `max_length` 设得太小 | 密码字段的 `max_length` 设大一些（如 100），因为哈希后长度固定，原始密码再长也不影响存储 |
