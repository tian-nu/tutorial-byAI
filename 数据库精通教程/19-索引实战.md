## 一、创建索引

### 1.1 基本语法

```sql
-- 方法 1：CREATE INDEX
CREATE INDEX idx_name ON table_name(column_name);

-- 方法 2：ALTER TABLE ... ADD INDEX
ALTER TABLE table_name ADD INDEX idx_name(column_name);

-- 建表时直接指定
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50),
    email VARCHAR(100),
    INDEX idx_email(email)       -- 建表时创建索引
);
```

### 1.2 实战：给 users 表加索引

```sql
-- 先看看表里有什么数据
SELECT COUNT(*) FROM users;

-- 看看现在有哪些索引
SHOW INDEX FROM users;

-- 在 email 列上创建索引
CREATE INDEX idx_email ON users(email);

-- 再次查看
SHOW INDEX FROM users;
-- 现在应该能看到 idx_email 了

-- 用 EXPLAIN 验证
EXPLAIN SELECT * FROM users WHERE email = 'alice@example.com';
-- type: ref, key: idx_email, rows: 1 ✅
```

### 1.3 删除索引

```sql
-- 方法 1
DROP INDEX idx_name ON table_name;

-- 方法 2
ALTER TABLE table_name DROP INDEX idx_name;
```

## 二、联合索引与最左前缀原则

联合索引（Composite Index）*此术语见附录E* 是在**多个列**上创建的索引。

### 2.1 创建联合索引

```sql
-- 在 name 和 age 上创建联合索引
CREATE INDEX idx_name_age ON users(name, age);
```

**注意**：索引列的顺序很重要！`idx_name_age(name, age)` 和 `idx_age_name(age, name)` 是两个不同的索引。

### 2.2 最左前缀原则

最左前缀原则（Leftmost Prefix）*此术语见附录E* 是联合索引最重要的规则：

> **联合索引 `(a, b, c)` 相当于创建了 `(a)`、`(a, b)`、`(a, b, c)` 三个索引，但不等同于 `(b)` 或 `(c)` 或 `(b, c)`。**

用 EXPLAIN 验证：

```sql
-- 假设有联合索引 idx_name_age(name, age)

-- ✅ 走索引：使用了最左前缀 name
EXPLAIN SELECT * FROM users WHERE name = '张三';
-- key: idx_name_age, type: ref

-- ✅ 走索引：使用了 name 和 age（最左前缀 + 后续列）
EXPLAIN SELECT * FROM users WHERE name = '张三' AND age = 25;
-- key: idx_name_age, type: ref

-- ❌ 不走索引：跳过了最左列 name
EXPLAIN SELECT * FROM users WHERE age = 25;
-- key: NULL, type: ALL

-- ✅ 走索引：虽然 age 在前，但 name 也写了，MySQL 优化器会调整顺序
EXPLAIN SELECT * FROM users WHERE age = 25 AND name = '张三';
-- key: idx_name_age, type: ref
-- 注意：MySQL 优化器会自动调整 WHERE 条件的顺序来匹配最左前缀
```

### 2.3 最左前缀与范围查询

```sql
-- 联合索引 idx_name_age(name, age)

-- ✅ 走索引：name 等值，age 范围
EXPLAIN SELECT * FROM users WHERE name = '张三' AND age > 25;
-- key: idx_name_age, type: range

-- ⚠️ 部分走索引：age 后面的列不走索引
-- 假设有 idx_name_age_city(name, age, city)
EXPLAIN SELECT * FROM users WHERE name = '张三' AND age > 25 AND city = '北京';
-- 只有 name 和 age 走索引，city 部分不走（因为 age 用了范围查询）
```

**规则**：在联合索引中，一旦某个列使用了范围查询（`>`、`<`、`BETWEEN`、`LIKE`），它后面的列就不再走索引。

### 2.4 联合索引列顺序的选择

创建联合索引时，列的顺序决定了它的用途：

```sql
-- 哪个更好？
-- A: idx_name_age(name, age)
-- B: idx_age_name(age, name)

-- 答案取决于你的查询模式：
-- 如果经常查 WHERE name = ? AND age = ? → A 和 B 都可以
-- 如果经常查 WHERE name = ? → 只有 A 可以（因为 name 是最左前缀）
-- 如果经常查 WHERE age = ? → 只有 B 可以（因为 age 是最左前缀）
```

**选序原则**：
1. 选择性高的列放前面（能过滤更多行）
2. 经常单独查询的列放前面
3. 等值查询的列放前面，范围查询的列放后面

## 三、唯一索引

唯一索引（Unique Index）*此术语见附录E*：索引列的值必须唯一，但允许 NULL（MySQL 中允许多个 NULL）。

```sql
-- 创建唯一索引
CREATE UNIQUE INDEX idx_unique_email ON users(email);

-- 现在插入重复 email 会报错
INSERT INTO users (email) VALUES ('alice@example.com');
-- ERROR 1062: Duplicate entry 'alice@example.com' for key 'idx_unique_email'
```

**唯一索引 vs 普通索引**：
- 唯一索引多了"唯一性约束"的功能
- 查找性能：唯一索引在找到第一个匹配行后就可以停止，普通索引需要继续扫描找到所有匹配行
- 但日常使用中，性能差异可以忽略，主要用唯一索引来保证数据质量

---

## 四、前缀索引

前缀索引（Prefix Index）*此术语见附录E*：对于很长的字符串列（如 TEXT、长 VARCHAR），只对前 N 个字符建索引。

```sql
-- 对 email 的前 20 个字符建索引
CREATE INDEX idx_email_prefix ON users(email(20));

-- 对于很长的文本列
CREATE INDEX idx_content_prefix ON articles(content(50));
```

### 前缀长度怎么选？

```sql
-- 计算不同前缀长度下的选择性
SELECT
    COUNT(DISTINCT LEFT(email, 5)) / COUNT(*) AS sel_5,
    COUNT(DISTINCT LEFT(email, 10)) / COUNT(*) AS sel_10,
    COUNT(DISTINCT LEFT(email, 15)) / COUNT(*) AS sel_15,
    COUNT(DISTINCT LEFT(email, 20)) / COUNT(*) AS sel_20,
    COUNT(DISTINCT email) / COUNT(*) AS sel_full
FROM users;
```

选择性（Selectivity）*此术语见附录E* 越接近 1，说明前缀区分度越高。一般选择性能接近完整列且长度最短的前缀。

**前缀索引的局限**：
- 不能用于 ORDER BY
- 不能用于覆盖索引
- 如果前缀长度不够，区分度下降，可能导致更多回表

---

## 五、覆盖索引

覆盖索引（Covering Index）*此术语见附录E* 是指：**查询需要的所有列都在索引中，不需要回表**。

```sql
-- 假设有联合索引 idx_name_age(name, age)

-- ✅ 覆盖索引：id 是主键（自动在二级索引中），name 和 age 在索引中
EXPLAIN SELECT id, name, age FROM users WHERE name = '张三';
-- Extra: Using index  ← 覆盖索引！

-- ❌ 需要回表：city 不在索引中
EXPLAIN SELECT id, name, age, city FROM users WHERE name = '张三';
-- Extra: NULL  ← 需要回表取 city 列
```

### 覆盖索引的价值

回表一次 = 多一次 B+树查找 = 多 3~4 次 IO。如果查询返回 1000 行，回表 1000 次 = 3000 次额外 IO。覆盖索引避免了这一切。

**但注意**：不要为了覆盖索引把所有列都加到索引里。索引本身有维护成本（INSERT/UPDATE/DELETE 时都要更新索引），索引太多反而适得其反。

## 六、索引下推（ICP）

索引下推（Index Condition Pushdown，ICP）*此术语见附录E* 是 MySQL 5.6 引入的优化。在 EXPLAIN 中显示为 `Using index condition`。

### 6.1 没有 ICP 时

```sql
-- 假设有联合索引 idx_name_age(name, age)
SELECT * FROM users WHERE name LIKE '张%' AND age = 25;
```

没有 ICP 时：
1. 在索引中查找 `name LIKE '张%'` 的所有行（比如 100 行）
2. 对这 100 行，取出主键，**回表**取完整行
3. 在完整行上检查 `age = 25`
4. 如果 100 行中只有 5 行的 age = 25，那么 95 次回表是**浪费**的

### 6.2 有 ICP 时

有 ICP 时：
1. 在索引中查找 `name LIKE '张%'` 的所有行
2. **在索引中直接检查 `age = 25`**（因为索引中就有 age 列！）
3. 只对满足 age = 25 的行（5 行）回表
4. 节省了 95 次回表！

```sql
EXPLAIN SELECT * FROM users WHERE name LIKE '张%' AND age = 25;
-- Extra: Using index condition  ← ICP 生效！
```

**ICP 的触发条件**：
- 使用联合索引
- WHERE 条件中，一部分列走索引，另一部分列也在索引中但不走索引（如 LIKE 范围查询后的列）
- 查询需要回表（`SELECT *` 或选了不在索引中的列）

## 七、索引失效场景

这是最实用的一节。以下场景会导致索引失效，**每个场景都配 EXPLAIN 验证**。

### 7.1 LIKE 前导模糊

```sql
-- ✅ 走索引：后匹配
EXPLAIN SELECT * FROM users WHERE name LIKE '张%';
-- key: idx_name, type: range

-- ❌ 不走索引：前导模糊
EXPLAIN SELECT * FROM users WHERE name LIKE '%张三';
-- key: NULL, type: ALL

-- ❌ 不走索引：两端模糊
EXPLAIN SELECT * FROM users WHERE name LIKE '%张三%';
-- key: NULL, type: ALL
```

**原因**：B+树按照列值的顺序排列，前缀匹配可以定位范围，但前导模糊无法确定从哪开始扫描。

**解决方案**：如果确实需要模糊搜索，考虑用全文索引（FULLTEXT），或使用 Elasticsearch 等搜索引擎。

### 7.2 OR 条件含非索引列

```sql
-- 假设 name 有索引，city 没有索引

-- ❌ 不走索引：city 没有索引
EXPLAIN SELECT * FROM users WHERE name = '张三' OR city = '北京';
-- key: NULL, type: ALL

-- ✅ 走索引：两列都有索引
CREATE INDEX idx_city ON users(city);
EXPLAIN SELECT * FROM users WHERE name = '张三' OR city = '北京';
-- key: idx_name,idx_city（可能走 index merge）

-- ✅ 替代方案：用 UNION
SELECT * FROM users WHERE name = '张三'
UNION
SELECT * FROM users WHERE city = '北京';
```

### 7.3 函数/运算包裹索引列

```sql
-- 假设 created_at 有索引

-- ❌ 不走索引：YEAR() 函数包裹了索引列
EXPLAIN SELECT * FROM users WHERE YEAR(created_at) = 2024;
-- key: NULL, type: ALL

-- ✅ 走索引：不用函数
EXPLAIN SELECT * FROM users WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
-- key: idx_created_at, type: range
```

**原因**：索引存的是 `created_at` 的值，而不是 `YEAR(created_at)` 的值。函数改变了查找条件，MySQL 无法使用索引。

### 7.4 隐式类型转换

```sql
-- 假设 phone 是 VARCHAR 类型，有索引

-- ❌ 不走索引：phone 是 VARCHAR，但传入了数字
EXPLAIN SELECT * FROM users WHERE phone = 13800138000;
-- key: NULL, type: ALL
-- 实际执行：WHERE CAST(phone AS SIGNED) = 13800138000

-- ✅ 走索引：传入字符串
EXPLAIN SELECT * FROM users WHERE phone = '13800138000';
-- key: idx_phone, type: ref
```

**原因**：MySQL 在比较字符串和数字时，会把字符串转为数字。这相当于在索引列上加了函数 `CAST(phone AS SIGNED)`，导致索引失效。

### 7.5 != / <> 可能不走索引

```sql
-- != 不一定走索引，取决于优化器判断
EXPLAIN SELECT * FROM users WHERE status != 'deleted';
-- 可能走索引，也可能不走。取决于选择性。
```

**原因**：`!=` 意味着"除了这个值以外的所有值"，如果表中大部分数据都不是 'deleted'，优化器可能认为全表扫描更快。

**这不是绝对的**——如果表中 99% 的数据都是 'deleted'，`!= 'deleted'` 只返回 1% 的数据，MySQL 可能会走索引。一切以 EXPLAIN 为准。

## 八、索引选择策略

### 8.1 衡量选择性

```sql
-- 计算选择性：不同值数量 / 总行数
-- 选择性越接近 1，说明该列区分度越高
SELECT
    COUNT(DISTINCT email) / COUNT(*) AS email_selectivity,
    COUNT(DISTINCT gender) / COUNT(*) AS gender_selectivity,
    COUNT(DISTINCT status) / COUNT(*) AS status_selectivity
FROM users;
```

输出示例：
```
email_selectivity: 0.99  ← 高选择性，适合建索引
gender_selectivity: 0.02 ← 低选择性（只有男/女/其他），不适合建索引
status_selectivity: 0.05 ← 低选择性，不适合单独建索引
```

### 8.2 建索引的基本原则

1. **高选择性列优先**：选择性 > 0.1 的列才值得建索引
2. **WHERE 条件中的列**：经常出现在 WHERE 中的列
3. **JOIN 关联列**：外键列
4. **ORDER BY 列**：排序列
5. **避免过多索引**：每个索引都增加写入开销，一般不超过 5-6 个

### 8.3 索引不是越多越好

```sql
-- 查看表的所有索引
SHOW INDEX FROM users;
```

每个索引都会：
- 占用磁盘空间（B+树结构）
- 拖慢 INSERT/UPDATE/DELETE（每次写入都要更新所有索引）
- 占用 Buffer Pool 内存

**索引数建议**：普通业务表 3-5 个索引，核心表不超过 8-10 个。

## 我做得对不对？

### 验证方法

按顺序执行以下操作，每步都能解释结果：

```sql
-- 1. 创建测试环境
CREATE TABLE test_index (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50),
    age INT,
    city VARCHAR(50),
    email VARCHAR(100)

-- 插入一些测试数据
INSERT INTO test_index (name, age, city, email) VALUES
('张三', 25, '北京', 'zhangsan@example.com'),
('李四', 30, '上海', 'lisi@example.com'),
('王五', 25, '广州', 'wangwu@example.com'),
('赵六', 35, '北京', 'zhaoliu@example.com'),
('孙七', 28, '深圳', 'sunqi@example.com');

-- 2. 创建联合索引
CREATE INDEX idx_name_age ON test_index(name, age);

-- 3. 验证最左前缀
EXPLAIN SELECT * FROM test_index WHERE name = '张三';
-- 预期：走索引

EXPLAIN SELECT * FROM test_index WHERE age = 25;
-- 预期：不走索引

-- 4. 验证覆盖索引

EXPLAIN SELECT * FROM test_index WHERE name LIKE '%三';

-- 预期：不走索引

-- 6. 验证 ICP

-- 预期：Extra: Using index condition

DROP TABLE test_index;
```

## 不对怎么办？

### 常见错误 1：索引列用函数

```sql
-- ❌ 错误
SELECT * FROM users WHERE YEAR(created_at) = 2024;

-- ✅ 正确：用范围替代函数
SELECT * FROM users WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
```

### 常见错误 2：联合索引不遵循最左前缀

```sql
-- 联合索引 idx_name_age(name, age)

-- ❌ 错误：跳过了 name
SELECT * FROM users WHERE age = 25;

-- ✅ 正确：包含 name
SELECT * FROM users WHERE name = '张三' AND age = 25;
-- 或者：单独为 age 建索引
CREATE INDEX idx_age ON users(age);
```

### 常见错误 3：在低选择性列建索引

```sql
-- ❌ 错误：gender 只有 2-3 个值
CREATE INDEX idx_gender ON users(gender);
-- MySQL 优化器大概率不会用这个索引

-- ✅ 正确：在选择性高的列建索引
CREATE INDEX idx_email ON users(email);
```

### 常见错误 4：隐式类型转换不设防

```sql
-- 假设 phone 是 VARCHAR

-- ❌ 错误：传入数字
SELECT * FROM users WHERE phone = 13800138000;

-- ✅ 正确：传入字符串
SELECT * FROM users WHERE phone = '13800138000';
```

**最佳实践**：在应用代码中，始终用字符串类型传入字符串列的值，不要依赖语言自动转换。
