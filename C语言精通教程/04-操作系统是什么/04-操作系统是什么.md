## 我在做什么？

前三章你认识了 CPU（大脑）、内存（记忆）、硬盘（仓库）。这三个硬件都很厉害——但它们各管各的，谁也不听谁的。需要一个人来指挥它们配合——这个人就是操作系统。

这一章讲操作系统是什么，以及它和我们写的 C 程序有什么关系。读完你能理解：为什么 C 程序从 `main()` 开始，以及 `main()` 被调用之前发生了什么。

## 一、操作系统是"管家"加"翻译官"

### 1.1 没有操作系统会怎样？

想象你买了一台电脑，只有硬件，没有操作系统。你按开机键，CPU 通电了，内存也准备好了——然后呢？什么都没有。

你无法"双击"打开文件，因为"双击"这个动作需要有程序来识别。你无法在屏幕上看到任何东西，因为屏幕需要程序来驱动。你甚至无法输入——键盘需要程序来解读按键信号。

```
没有操作系统的电脑：

+----------+  +----------+  +----------+  +----------+
|   CPU    |  |   内存   |  |   硬盘   |  |   屏幕   |
+----------+  +----------+  +----------+  +----------+
     ^             ^             ^             ^
     |             |             |             |
     +------+------+------+------+
            |
       一堆硬件，没人管，什么都做不了
```

### 1.2 操作系统上场

*操作系统* *此术语见附录F* 是电脑开机后运行的第一个程序。它有两个核心角色：

**角色一：管家**——管理所有硬件资源。

```
有了操作系统的电脑：

         +---------------------+
         |    操作系统（管家）  |
         |                     |
         |  "CPU 给谁用？"      |
         |  "内存怎么分？"      |
         |  "文件谁在读？"      |
         |  "屏幕显示什么？"    |
         +--+------+------+-----+
            |      |      |
            v      v      v
         +----+ +----+ +----+
         |CPU | |内存| |硬盘|
         +----+ +----+ +----+
```

**角色二：翻译官**——程序不能直接操作硬件，必须通过操作系统"传话"。

```text
程序想要在屏幕上显示一个字符：

错误方式（程序直连硬件）：
  程序 --> 屏幕硬件  (不让！操作系统会拦下)

正确方式（通过操作系统）：
  程序 --> 操作系统 --> 屏幕驱动 --> 屏幕硬件
```

这种"传话"的机制叫 *系统调用* *此术语见附录F*——程序向操作系统发出请求，操作系统代为执行。

### 1.3 操作系统和内核

日常用语中，"操作系统"和 *内核* *此术语见附录F* 经常混用。严格来说：

- **内核**：操作系统的核心部分，直接管理硬件，运行在最高权限级别。
- **操作系统**：内核 + 外围工具（命令行、图形界面、系统服务等）。

```
        操作系统（完整）
+--------------------------------+
|  图形界面 (GUI) / 命令行 (CLI)  |  <-- 你和操作系统的交互层
+--------------------------------+
|  系统服务 (网络、打印、...)     |  <-- 基础设施
+--------------------------------+
|  内核 (Kernel)                 |  <-- 核心，直接管理硬件
|  - 进程管理                    |
|  - 内存管理                    |
|  - 文件系统                    |
|  - 设备驱动                    |
+--------------------------------+
|  硬件                          |
+--------------------------------+
```

C 语言写的程序，很多时候直接和内核打交道（通过系统调用），所以理解内核的存在很重要。

## 二、进程与线程

### 2.1 进程：程序的"运行实例"

你在硬盘上有一个可执行文件 `hello.exe`。它躺在硬盘里，只是一个文件。

当你双击它，操作系统做了一系列事情：
1. 把 `hello.exe` 的机器码从硬盘加载到内存。
2. 为它分配一块独立的内存空间。
3. 创建一个 *进程* *此术语见附录F*，告诉 CPU："开始执行这个程序吧。"

```text
硬盘上的文件（死的）            内存中的进程（活的）
+------------------+          +------------------+
| hello.exe        |  加载    | 进程 (PID: 1234) |
| 机器码：         | -------> | 代码段：机器码    |
| 10110000...      |          | 数据段：全局变量  |
| 00000101...      |          | 栈：局部变量     |
+------------------+          | 堆：动态内存     |
                              +------------------+
```

**文件是"菜谱"，进程是"正在炒菜的人"。** 同一份菜谱可以被多个厨师同时照着做（同一个 .exe 可以运行多次，产生多个进程）。

### 2.2 进程的隔离性

操作系统给每个进程一块独立的"领地"。进程 A 不能访问进程 B 的内存——这是有意设计的保护机制。

```text
进程 A 的内存空间              进程 B 的内存空间
+------------------+          +------------------+
| 地址 0x1000: 42  |          | 地址 0x1000: 99  |
| 地址 0x2000: 10  |          | 地址 0x2000: 55  |
+------------------+          +------------------+
        |                              |
        | 不能互相访问！                |
        +------------------------------+
```

这就是为什么一个程序崩溃通常不会影响其他程序——它们被"隔离"在不同的进程里。

### 2.3 线程：进程内部的"分身"

一个进程可以包含多个 *线程* *此术语见附录F*。线程是"进程中真正执行代码的单位"。

```text
进程就像一个厨房：
  - 厨房有自己的灶台、冰箱、菜刀（共享内存）
  - 线程就像厨房里的厨师（可以有多个厨师同时干活）

+------------------ 进程（厨房）------------------+
|                                                 |
|  线程 1（切菜）    线程 2（炒菜）   线程 3（摆盘）  |
|    |                  |                  |       |
|    +--------+---------+---------+--------+       |
|             |                   |                |
|       共享的灶台、冰箱、菜刀（共享内存）            |
|                                                 |
+-------------------------------------------------+
```

*并发* *此术语见附录F* 就是多个线程"同时"执行（在单核 CPU 上是快速切换，在多核 CPU 上是真正同时）。

## 三、为什么 C 程序从 main() 开始？

### 3.1 谁调用了 main()？

你写 C 程序，第一个函数永远是 `main()`：

```c
#include <stdio.h>

int main() {
    printf("Hello, World!\n");
    return 0;
}
```

但你想过一个问题吗：**谁调用了 `main()`？**

`main()` 也是一个函数，函数必须被调用才能执行。谁在调用它？

答案：**操作系统调用了你的程序，但中间有一个"引路人"——启动代码。**

### 3.2 启动代码：main() 之前的秘密

你的程序不是从 `main()` 开始执行的。在 `main()` 之前，有一段隐藏的代码在运行。这段代码的专业名称叫 *启动代码* *此术语见附录F*（也叫 *crt0* *此术语见附录F* 或 C 运行时启动例程）。

```text
程序从硬盘加载到内存后的真正执行顺序：

1. 操作系统创建进程，加载程序
         |
         v
2. 启动代码 (crt0) 开始执行：
   - 初始化栈、堆
   - 初始化全局变量
   - 设置命令行参数 (argc, argv)
   - 设置环境变量
         |
         v
3. 调用 main(argc, argv)
         |
         v
4. main() 返回，返回值传给启动代码
         |
         v
5. 启动代码把返回值传给操作系统（exit 系统调用）
         |
         v
6. 操作系统清理进程资源
```

```text
用 ASCII 图表示：

           操作系统
              |
              | 创建进程，加载程序
              v
      +----------------+
      |  启动代码 crt0  |  <-- 你看不到，但它就在那里
      |  - 初始化环境    |
      |  - 准备参数     |
      +-------+--------+
              |
              | 调用 main(argc, argv)
              v
      +----------------+
      |  int main() {  |  <-- 你写的代码
      |    ...          |
      |    return 0;    |
      |  }              |
      +-------+--------+
              |
              | 返回 0
              v
      +----------------+
      |  启动代码 crt0  |
      |  - 接收返回值    |
      |  - exit(0)      |
      +----------------+
              |
              | 系统调用 exit
              v
           操作系统
           (清理进程)
```

### 3.3 main() 的返回值去了哪里？

你写的 `return 0;` 不是没有意义的。这个返回值最终会传给操作系统，表示"我正常结束了"。

```bash
# 在 Linux/macOS 命令行中，可以用 $? 查看上一个程序的返回值
./hello
echo $?
# 输出: 0   (表示程序正常结束)

# 返回 0 通常表示成功，非 0 表示出错
```

C 标准规定 `main()` 返回 `int`，正是为了这个目的——给操作系统一个"交代"。

### 3.4 运行时库

启动代码是 *运行时库* *此术语见附录F* 的一部分。运行时库包含了程序运行所需的基础设施：

- 启动代码（crt0）
- 标准库函数（`printf`、`malloc`、`strlen` 等）
- 异常处理支持
- 栈溢出检测

当你写 `gcc hello.c` 时，编译器会自动链接运行时库。你不需要手动指定——但你写的代码，实际是在运行时库的"框架"里运行的。

## 四、C 程序如何与操作系统交互？

### 4.1 系统调用：向操作系统"求助"

C 程序不能直接操作硬件。你想在屏幕上打印文字，必须通过操作系统。

`printf("Hello")` 最终会调用操作系统的"写"系统调用，把 "Hello" 发给操作系统，操作系统再指挥屏幕显示出来。

```text
printf("Hello")
    |
    v
C 标准库 (格式化字符串)
    |
    v
write() 系统调用
    |
    v
操作系统内核 (把数据发给屏幕驱动)
    |
    v
屏幕硬件 (显示 "Hello")
```

### 4.2 标准库封装了系统调用

C 标准库（`stdio.h`、`stdlib.h`、`string.h` 等）是对系统调用的封装。它让程序员不用直接和操作系统"说方言"——可以用一套统一的接口，在不同操作系统上工作。

```c
// 你写的是标准 C
printf("Hello");

// 在 Linux 上，printf 内部调用 write() 系统调用
// 在 Windows 上，printf 内部调用 WriteConsole() 系统调用
// 你不需要关心——标准库帮你处理了差异
```

## 本章小结

| 概念 | 一句话解释 |
|------|-----------|
| 操作系统 | 管理硬件资源的"管家"和程序与硬件之间的"翻译官" |
| 内核 | 操作系统的核心，直接管理硬件，运行在最高权限 |
| 系统调用 | 程序向操作系统请求服务的接口，如读写文件、分配内存 |
| 进程 | 程序的"运行实例"，拥有独立的内存空间 |
| 线程 | 进程中实际执行代码的单位，同一进程的多个线程共享内存 |
| 并发 | 多个线程（或进程）交替或同时执行 |
| 运行时库 | 程序运行所需的基础设施，包括启动代码和标准库函数 |
| crt0 / 启动代码 | 在 main() 之前执行的隐藏代码，负责初始化环境并调用 main() |

## 我做得对不对？

验证方法——你能用自己的话回答以下问题吗？

1. **操作系统是什么？** —— 管家（管理硬件资源）+ 翻译官（程序通过它和硬件对话）。没有操作系统，硬件只是一堆没用的电路板。
2. **进程和线程的区别？** —— 进程是"程序的运行实例"，拥有独立内存空间；线程是进程内的执行单元，共享进程的内存。
3. **为什么 C 程序从 `main()` 开始？** —— 这是 C 标准规定的约定。实际上在 `main()` 之前有启动代码（crt0）在运行，它负责初始化环境，然后调用 `main()`。

## 不对怎么办？

**常见困惑 1**："我写 Python 程序也有 `if __name__ == '__main__':`，这和 C 的 `main()` 是一回事吗？"

解答：精神上相似，机制上不同。Python 的 `__main__` 是解释器层面的约定——当 Python 解释器执行一个脚本时，它把 `__name__` 设为 `'__main__'`。C 的 `main()` 是操作系统层面的约定——操作系统通过启动代码调用 `main()`。Python 的 `__main__` 是"解释器觉得这是入口"，C 的 `main()` 是"操作系统强制要求这个入口"。

**常见困惑 2**："系统调用和普通函数调用有什么区别？"

解答：普通函数调用是在你自己的代码里跳来跳去，都在用户态（低权限）。系统调用是从用户态切换到内核态（高权限），让操作系统帮你做你无权做的事——比如读写文件、分配内存。系统调用比普通函数调用慢得多，因为它涉及权限切换。

**常见困惑 3**："我写的 C 程序能在 Windows 和 Linux 上都能跑吗？"

解答：如果只用标准 C 库（`stdio.h`、`stdlib.h` 等），代码可以跨平台——但编译出来的可执行文件不能。Windows 的 .exe 不能在 Linux 上直接运行，反之亦然。你需要为每个平台分别编译。这就是本章讲的"启动代码"和"运行时库"依赖操作系统的原因：每个操作系统的系统调用接口不同，所以运行时库也不同。
