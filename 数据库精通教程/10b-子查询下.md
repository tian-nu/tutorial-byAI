## 我在做什么？

上一章你学会了"查超过平均金额的订单"（标量子查询）和"查下过单的用户"（列子查询）。但这两个只是子查询的冰山一角。

本章你会遇到三种更高级的玩法：
1. **行子查询**：一次比较多个列
2. **表子查询**：把子查询结果当作临时表用
3. **EXISTS**：只关心"有没有"，不关心"是什么"

学完这一章，你能正确判断"这里用子查询还是 JOIN"，并在面试中解释为什么 EXISTS 比 IN 更安全。

## 一、行子查询：一次比较多个列

上一章讲的标量子查询返回**一个值**，列子查询返回**一列多行**。但有时候，你需要**同时比较多个列**。

**行子查询**（Row Subquery）*此术语见附录E* 返回**一行多列**，用括号把多个列包起来比较。

```sql
-- 场景：查"和某个用户同一城市、同一年龄段的其他用户"
-- 假设 users 表有 city 和 age 列

-- ❌ 分步写法（需要两个子查询）
SELECT * FROM users
WHERE city = (SELECT city FROM users WHERE id = 1)
  AND age  = (SELECT age  FROM users WHERE id = 1);

-- ✅ 行子查询写法（一次搞定）
SELECT * FROM users
WHERE (city, age) = (SELECT city, age FROM users WHERE id = 1);
```

`(city, age) = (SELECT city, age FROM ...)` 的意思是：**city 和 age 同时匹配**。MySQL 把 `(city, age)` 当作一个"行"来比较。

### 行子查询的其他用法

```sql
-- 场景：查"订单金额和数量都最高的订单"
-- 假设 orders 表有 amount 和 quantity 列
SELECT * FROM orders
WHERE (amount, quantity) = (
    SELECT MAX(amount), MAX(quantity) FROM orders
);
-- 注意：这里 MAX(amount) 和 MAX(quantity) 可能来自不同行！
-- 这个查询找的是"amount 等于最大金额，且 quantity 等于最大数量"的行

-- 更实用的场景：找出每个用户最新的一笔订单
SELECT * FROM orders
WHERE (user_id, created_at) IN (
    SELECT user_id, MAX(created_at)
    FROM orders
    GROUP BY user_id
);
```

> **想多一点**：行子查询在"找每个分组的最大/最小记录"时特别有用。上面的例子——"每个用户的最新订单"——如果用 JOIN 写会非常绕，但用行子查询一行就解决了。这是子查询在"语义清晰"上胜过 JOIN 的典型场景。

## 二、表子查询（派生表）：FROM 中的子查询

**表子查询**（Table Subquery）*此术语见附录E* 返回**多行多列**，放在 `FROM` 子句中，当作一张临时表来用。这种临时表有个专业名字叫**派生表**（Derived Table）*此术语见附录E*。

### 2.1 基本用法

```sql
-- 场景：查"每个用户的订单总额"，然后只显示总额超过 500 的用户
-- 思路：先把"每个用户订单总额"算出来当临时表，再过滤

SELECT user_id, total_amount
FROM (
    SELECT user_id, SUM(amount) AS total_amount
    FROM orders
    GROUP BY user_id
) AS user_totals           -- ← 派生表必须给别名！
WHERE total_amount > 500;
```

**执行过程**：
1. 先执行子查询，得到一张临时表 `user_totals`（包含 user_id 和 total_amount）
2. 再对这张临时表执行外层查询的 WHERE 过滤

### 2.2 派生表必须给别名

```sql
-- ❌ 错误：派生表没有别名
SELECT * FROM (
    SELECT user_id, SUM(amount) AS total
    FROM orders GROUP BY user_id
);
-- ERROR 1248 (42000): Every derived table must have its own alias

-- ✅ 正确：给别名
SELECT * FROM (
    SELECT user_id, SUM(amount) AS total
    FROM orders GROUP BY user_id
) AS t;
```

**为什么必须给别名？** 因为派生表在 SQL 执行过程中是一张"匿名临时表"，MySQL 需要你给它一个名字，才能在后续引用它。就算你后面不用这个名字，也必须给。

### 2.3 派生表 vs 子查询 vs 直接查询

同一个需求，三种写法：

```sql
-- 需求：查"订单总额超过 500 的用户"

-- 写法 A：派生表
SELECT user_id, total_amount
FROM (
    SELECT user_id, SUM(amount) AS total_amount
    FROM orders GROUP BY user_id
) AS t
WHERE total_amount > 500;

-- 写法 B：HAVING（更简洁）
SELECT user_id, SUM(amount) AS total_amount
FROM orders
GROUP BY user_id
HAVING total_amount > 500;

-- 写法 C：HAVING + 子查询
SELECT user_id, SUM(amount) AS total_amount
FROM orders
GROUP BY user_id
HAVING SUM(amount) > 500;
```

在这个例子中，写法 B 最简洁。但派生表在**更复杂的场景**中不可替代——比如你需要对聚合结果做多次处理，或者把聚合结果再 JOIN 给其他表。

## 三、EXISTS：只关心"有没有"

**EXISTS** *此术语见附录E* 是一个特殊的关键词，它不返回任何数据，只回答一个问题：**"子查询有没有返回至少一行？"**

### 3.1 基本语法

```sql
SELECT ...
FROM table_a
WHERE EXISTS (
    SELECT 1 FROM table_b WHERE table_b.a_id = table_a.id
);
```

`EXISTS (子查询)` 返回：
- **TRUE**：子查询至少返回一行
- **FALSE**：子查询一行都没返回

注意 `SELECT 1` 而不是 `SELECT *`——因为 EXISTS 根本不关心子查询返回什么数据，它只关心**有没有行**。`SELECT 1` 是一种约定俗成的写法，告诉读代码的人"我不需要任何列，我只是在检查存在性"。

### 3.2 实战：用 EXISTS 查"下过订单的用户"

```sql
-- 场景：查"下过订单的用户"
-- 对比三种写法

-- 写法 1：IN（你上一章学的）
SELECT id, username
FROM users
WHERE id IN (SELECT DISTINCT user_id FROM orders);

-- 写法 2：EXISTS
SELECT id, username
FROM users u
WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id
);

-- 写法 3：NOT EXISTS（查"从未下过单的用户"）
SELECT id, username
FROM users u
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id
);
```

### 3.3 EXISTS vs IN：语义差异

> **注意**：本章只讲语义差异。IN 和 EXISTS 的性能对比涉及索引，留到第 20 章深入。

| 维度 | IN | EXISTS |
|------|-----|--------|
| **语义** | "某个值在不在一组值里面" | "有没有匹配的行" |
| **NULL 安全性** | NOT IN 有 NULL 陷阱（见 10a 章） | NOT EXISTS 天然安全，不受 NULL 影响 |
| **可读性** | 简单场景更直观 | 复杂条件（多列匹配）更灵活 |
| **典型场景** | 查"在某个列表中" | 查"存在关联记录" |

**选型建议**：
- 简单场景（单列匹配）：IN 更直观
- 需要多列匹配：EXISTS 更灵活
- 涉及 NOT 逻辑：**优先用 NOT EXISTS**，避免 NULL 陷阱

```sql
-- EXISTS 可以轻松处理多列匹配
SELECT * FROM users u
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.user_id = u.id
      AND o.amount > 1000
      AND o.status = 'paid'
);
```

## 四、相关子查询 vs 非相关子查询

这是子查询的一个重要分类维度，决定了**子查询的执行次数**。

### 4.1 非相关子查询：只执行一次

**非相关子查询**（Non-correlated Subquery）的特点是：**子查询不依赖外层查询的任何列**。它只需要执行一次，结果用于外层查询的每一行。

```sql
-- 非相关子查询：子查询独立于外层
SELECT * FROM orders
WHERE amount > (SELECT AVG(amount) FROM orders);
--              ↑ 子查询不引用外层任何列，独立执行一次
```

### 4.2 相关子查询：每行执行一次

**相关子查询**（Correlated Subquery）*此术语见附录E* 的特点是：**子查询引用了外层查询的列**。这意味着外层查询的每一行，都要重新执行一次子查询。

```sql
-- 相关子查询：子查询依赖外层 u.id
SELECT id, username,
    (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS order_count
FROM users u;
--  ↑ 子查询引用了外层的 u.id
```

**执行过程**：
1. 外层取出 users 第一行（id=1）
2. 子查询用 `u.id=1` 执行：`SELECT COUNT(*) FROM orders WHERE user_id = 1`
3. 得到结果，继续外层下一行
4. 外层取出 users 第二行（id=2）
5. 子查询用 `u.id=2` 执行：`SELECT COUNT(*) FROM orders WHERE user_id = 2`
6. ... 重复直到 users 表的所有行都处理完

**如果有 1000 个用户，子查询就要执行 1000 次！**

> **想多一点**：相关子查询就像一个"每行都要问一次"的问题。非相关子查询是"问一次，答案通用"。相关子查询在数据量小的时候问题不大，但数据量大时，1000 次查询就是 1000 次数据库交互。这就是为什么我们常说"相关子查询效率低"——不是因为子查询本身慢，而是因为**执行次数多**。

### 4.3 何时用相关子查询？

相关子查询的典型场景：**需要根据外层每一行的值去做不同的查询**。

```sql
-- 场景：查"每个用户订单金额超过他本人平均订单金额的订单"
SELECT o.id, o.user_id, o.amount
FROM orders o
WHERE o.amount > (
    SELECT AVG(amount)
    FROM orders
    WHERE user_id = o.user_id
);
--  ↑ 子查询引用了外层 o.user_id，是相关子查询
```

这种需求，不用相关子查询很难写出来。但如果你发现相关子查询可以用 JOIN 替代，通常选 JOIN 会更快（原因见下一节）。

## 五、子查询 vs JOIN：何时用哪个？

这是实际工作中最常见的决策之一。同一个需求，往往既可以用子查询，也可以用 JOIN。

### 5.1 对比示例

```sql
-- 需求：查"下过订单的用户名和邮箱"

-- 方案 A：子查询 + IN
SELECT username, email
FROM users
WHERE id IN (SELECT DISTINCT user_id FROM orders);

-- 方案 B：JOIN
SELECT DISTINCT u.username, u.email
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- 方案 C：EXISTS
SELECT username, email
FROM users u
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);
```

三个方案都能得到相同结果。选哪个？

### 5.2 决策指南

| 场景 | 推荐 | 原因 |
|------|------|------|
| 只查主表数据，子表只用于筛选 | 子查询（IN / EXISTS） | 语义清晰：我要查用户，条件是"下过单" |
| 需要同时显示两表的列 | JOIN | 子查询做不到同时显示两表数据 |
| 需要聚合子表数据后筛选 | 子查询 / 派生表 | 比如"查总额超过 500 的用户" |
| NOT 逻辑（不存在） | NOT EXISTS | 安全，避免 NULL 陷阱 |
| 数据量大，子表有索引 | JOIN | 性能通常更好（第 20 章详解） |
| 查询"每组最大/最小" | 行子查询 | 语义最清晰 |

### 5.3 关键原则

> **子查询用于"筛选"，JOIN 用于"关联查询"**。

- 如果你只需要 A 表的数据，B 表只是用来过滤条件 → 用子查询
- 如果你需要 A 表和 B 表的数据一起展示 → 用 JOIN

```sql
-- 子查询：只需要 users 的数据，orders 只是用来筛选
SELECT * FROM users WHERE id IN (SELECT user_id FROM orders);

-- JOIN：需要 users 和 orders 两表的数据一起展示
SELECT u.username, o.amount, o.created_at
FROM users u
JOIN orders o ON u.id = o.user_id;
```

## 我做得对不对？

### 验证方法

```sql
-- 1. 行子查询：验证 (city, age) = (SELECT ...)
-- 先建一个测试表
CREATE TABLE test_users (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    city VARCHAR(50),
    age INT
);
INSERT INTO test_users VALUES
(1, 'Alice', '北京', 25),
(2, 'Bob',   '上海', 30),
(3, 'Carol', '北京', 25),
(4, 'Dave',  '北京', 35);

-- 查"和 Alice 同一城市同一年龄的人"
SELECT * FROM test_users
WHERE (city, age) = (SELECT city, age FROM test_users WHERE id = 1);
-- 预期：返回 id=1(Alice) 和 id=3(Carol)，都是北京+25

-- 2. 派生表：验证别名必须
-- 这个会报错
SELECT * FROM (SELECT 1 AS n) AS t;
-- 预期：返回 1

-- 这个会报错
-- SELECT * FROM (SELECT 1 AS n);
-- 去掉注释试一下，会报 ERROR 1248

-- 3. EXISTS：验证
SELECT * FROM test_users u
WHERE EXISTS (SELECT 1 FROM test_users WHERE city = u.city AND id != u.id);
-- 预期：返回 Alice(有 Carol 同城) 和 Carol(有 Alice 同城)

-- 4. 清理
DROP TABLE test_users;
```

## 不对怎么办？

### 常见错误 1：派生表忘记给别名

```sql
-- ❌ 错误
SELECT * FROM (
    SELECT user_id, COUNT(*) AS cnt FROM orders GROUP BY user_id
);
-- ERROR 1248 (42000): Every derived table must have its own alias

-- ✅ 正确
SELECT * FROM (
    SELECT user_id, COUNT(*) AS cnt FROM orders GROUP BY user_id
) AS t;
```

### 常见错误 2：EXISTS 子查询中忘了关联条件

```sql
-- ❌ 错误：子查询没有关联条件，EXISTS 永远为 TRUE
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM orders);
-- 只要 orders 表有任何一行，所有 users 都被返回！
-- 这不是"下过单的用户"，而是"所有用户（只要有人下过单）"

-- ✅ 正确：加上关联条件
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);
```

### 常见错误 3：相关子查询用在不必要的场景

```sql
-- ❌ 过度使用相关子查询（每个用户执行一次子查询）
SELECT u.*,
    (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS cnt
FROM users u;

-- ✅ 更好的写法：用 JOIN + GROUP BY（一次查询完成）
SELECT u.*, COUNT(o.id) AS cnt
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id;
```

### 常见错误 4：行子查询列数不匹配

```sql
-- ❌ 错误：左边 2 列，右边子查询返回 3 列
SELECT * FROM users
WHERE (city, age) = (SELECT city, age, gender FROM users WHERE id = 1);
-- ERROR 1241 (21000): Operand should contain 2 column(s)

-- ✅ 正确：列数必须一致
SELECT * FROM users
WHERE (city, age) = (SELECT city, age FROM users WHERE id = 1);
```

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 行子查询 | `(a, b) = (SELECT a, b FROM ...)` | 左边列数必须等于子查询返回列数；适合"每组最大"场景 |
| 表子查询（派生表） | FROM 中的子查询，必须给别名 | 不写别名报错 ERROR 1248；适合对聚合结果做二次处理 |
| EXISTS | `WHERE EXISTS (SELECT 1 FROM ...)` | 只检查"有没有行"，不关心数据；SELECT 1 是约定俗成 |
| EXISTS vs IN | EXISTS 语义更灵活，NOT EXISTS 无 NULL 陷阱 | 性能对比见第 20 章；NOT 逻辑优先用 NOT EXISTS |
| 相关子查询 | 子查询引用外层列，每行执行一次 | 数据量大时效率低；能用 JOIN 替代时优先用 JOIN |
| 非相关子查询 | 子查询独立于外层，只执行一次 | 效率高；10a 章的标量和列子查询大多是非相关的 |
| 子查询 vs JOIN | 子查询用于筛选，JOIN 用于关联查询 | 需要两表数据同时展示时，必须用 JOIN |
