## 19.1 日志体系

### 19.1.1 Java日志框架演进简史

Java的日志体系可能是整个生态中最混乱的部分——你会在项目中同时看到`java.util.logging`、`log4j`、`slf4j`、`logback`等名词。要理解今天的日志体系，必须从它的演化史讲起：

```
年代      事件                          问题
─────────────────────────────────────────────────────────
2002    JDK 1.4 内置 JUL            太简陋，功能不足
       (java.util.logging)
─────────────────────────────────────────────────────────
2003    Apache Log4j 1.x 流行        性能一般，代码停止维护
─────────────────────────────────────────────────────────
2005    Apache JCL (commons-logging) 门面模式萌芽，但运行时绑定有bug
─────────────────────────────────────────────────────────
2006    SLF4J 诞生（Ceki Gülcü）     真正的门面模式
       Logback 诞生（同上作者）       Log4j 的接班人，性能优异
─────────────────────────────────────────────────────────
2014    Apache Log4j 2.x 发布        借鉴Logback，异步性能更好
─────────────────────────────────────────────────────────
2021    Log4j 2.x "Log4Shell"漏洞    全球安全地震级事件
─────────────────────────────────────────────────────────
现在    Spring Boot默认：            SLF4J（门面）+ Logback（实现）
```

一句话总结：**SLF4J是接口，Logback是Spring Boot默认的实现。** 你写代码时面对的是SLF4J的API，背后是哪套实现在运行时决定——换日志框架只需换jar包，代码零改动。这就是门面模式的威力。

### 19.1.2 SLF4J门面模式

```
你的代码
    │
    ▼
┌──────────┐     调用          ┌─────────────┐
│  SLF4J   │ ───────────────── │   Logback   │  ← Spring Boot默认
│  API     │                   │   (实现1)   │
│ (接口层) │ ───────────────── │   Log4j2    │  ← 高性能替代
└──────────┘                   │   (实现2)   │
                               │   JUL       │
                               │   (实现3)   │
                               └─────────────┘
```

门面模式的本质是"依赖倒置"：高层模块（你的代码）不依赖低层模块（具体日志框架），而是共同依赖抽象（SLF4J API）。

### 19.1.3 Lombok @Slf4j — 一行注解搞定日志对象

传统写法：

```java
public class EmployeeService {
    private static final Logger log = LoggerFactory.getLogger(EmployeeService.class);
    
    public void doSomething() {
        log.info("doing something");
    }
}
```

Lombok `@Slf4j`写法：

```java
@Slf4j  // ← 编译时自动生成上面那行 private static final Logger log = ...
public class EmployeeService {
    
    public void doSomething() {
        log.info("doing something");
    }
}
```

`@Slf4j`在编译时自动生成 `private static final org.slf4j.Logger log = org.slf4j.LoggerFactory.getLogger(EmployeeService.class);`，你写的代码和传统写法完全等价，但少了一行样板代码。项目中有多少个类，就省了多少行。

Lombok还提供了其他日志门面的注解：

| 注解 | 底层日志框架 |
|------|-------------|
| `@Slf4j` | SLF4J（最常见） |
| `@Log4j2` | Apache Log4j 2 |
| `@Log` | Apache Commons Logging |
| `@Log4j` | Apache Log4j 1.x |
| `@XSlf4j` | SLF4J扩展（MDC支持） |

### 19.1.4 日志级别

日志级别从低到高：

```
TRACE < DEBUG < INFO < WARN < ERROR
```

| 级别 | 含义 | 何时使用 |
|------|------|---------|
| **TRACE** | 最详细的追踪信息 | 几乎不用，仅极端调试如查看方法内部变量值 |
| **DEBUG** | 调试信息 | 开发环境查看变量值、SQL参数、方法进出 |
| **INFO** | 关键业务流程节点 | 请求到达、业务操作完成、定时任务执行等 |
| **WARN** | 警告（可能有问题） | 降级处理、重试、配置缺失但有默认值 |
| **ERROR** | 错误（必须处理） | 异常捕获、数据库连接失败、第三方服务不可用 |

**设置规则**：设置了`INFO`级别 → 输出INFO、WARN、ERROR（≥INFO的都输出），DEBUG和TRACE被屏蔽。

选择指南：
- 在**你自己的业务包**（如`com.example.ems`）设`DEBUG`（开发）或`INFO`（生产）
- **框架包**（`org.springframework`）设`INFO`
- **MyBatis Mapper**包设`DEBUG`——这是查看SQL的最简单方式
- **生产环境**全局默认为`WARN`，只记录值得关注的异常

### 19.1.5 logback-spring.xml 完整配置

Spring Boot默认的日志配置比较简单。生产级应用通常需要一个定制的`logback-spring.xml`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <!-- ==================== 属性定义 ==================== -->
    <!-- 日志文件存放路径（可通过启动参数覆盖） -->
    <springProperty scope="context" name="LOG_PATH"
                    source="logging.file.path" defaultValue="./logs"/>
    <!-- 应用名称 -->
    <springProperty scope="context" name="APP_NAME"
                    source="spring.application.name" defaultValue="ems"/>

    <!-- ==================== 控制台输出（所有环境都需要） ==================== -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <!-- 日志格式：时间 [线程] 级别 类名 : 消息 \n -->
            <pattern>
                %d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{50} : %msg%n
            </pattern>
            <charset>UTF-8</charset>
        </encoder>
    </appender>

    <!-- ==================== 文件输出（按日期 + 大小滚动） ==================== -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <!-- 当前日志文件 -->
        <file>${LOG_PATH}/${APP_NAME}.log</file>
        
        <!-- 滚动策略：按日期 + 按大小 -->
        <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
            <!-- 滚动后的文件名格式 -->
            <fileNamePattern>
                ${LOG_PATH}/%d{yyyy-MM-dd}/${APP_NAME}-%d{yyyy-MM-dd}-%i.log.gz
            </fileNamePattern>
            <!-- 单个文件最大 100MB -->
            <maxFileSize>100MB</maxFileSize>
            <!-- 保留最近 30 天的日志 -->
            <maxHistory>30</maxHistory>
            <!-- 所有日志文件总大小上限 10GB -->
            <totalSizeCap>10GB</totalSizeCap>
        </rollingPolicy>
        
        <encoder>
            <pattern>
                %d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{50} : %msg%n
            </pattern>
            <charset>UTF-8</charset>
        </encoder>
    </appender>

    <!-- ==================== 错误日志单独输出 ==================== -->
    <appender name="ERROR_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <!-- 只记录 ERROR 级别 -->
        <filter class="ch.qos.logback.classic.filter.LevelFilter">
            <level>ERROR</level>
            <onMatch>ACCEPT</onMatch>
            <onMismatch>DENY</onMismatch>
        </filter>
        <file>${LOG_PATH}/${APP_NAME}-error.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
            <fileNamePattern>
                ${LOG_PATH}/%d{yyyy-MM-dd}/${APP_NAME}-error-%d{yyyy-MM-dd}-%i.log.gz
            </fileNamePattern>
            <maxFileSize>100MB</maxFileSize>
            <maxHistory>90</maxHistory>
        </rollingPolicy>
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{50} : %msg%n</pattern>
            <charset>UTF-8</charset>
        </encoder>
    </appender>

    <!-- ==================== 异步日志（提升性能） ==================== -->
    <appender name="ASYNC_FILE" class="ch.qos.logback.classic.AsyncAppender">
        <!-- 不丢弃日志，队列满时阻塞等待 -->
        <discardingThreshold>0</discardingThreshold>
        <queueSize>512</queueSize>
        <appender-ref ref="FILE"/>
    </appender>

    <!-- ==================== 不同包的日志级别控制 ==================== -->
    <!-- Spring 框架自身 -->
    <logger name="org.springframework" level="INFO"/>
    <logger name="org.springframework.web" level="INFO"/>

    <!-- MyBatis Mapper（设为DEBUG可以看到SQL语句和参数绑定） -->
    <logger name="com.example.ems.mapper" level="DEBUG" additivity="false">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="ASYNC_FILE"/>
    </logger>

    <!-- 你的业务代码 -->
    <logger name="com.example.ems" level="DEBUG"/>

    <!-- Hibernate SQL输出（如果使用JPA） -->
    <logger name="org.hibernate.SQL" level="DEBUG"/>
    <logger name="org.hibernate.type.descriptor.sql.BasicBinder" level="TRACE"/>

    <!-- 屏蔽某些包的噪音日志 -->
    <logger name="org.apache.http" level="WARN"/>
    <logger name="com.zaxxer.hikari" level="WARN"/>

    <!-- ==================== 根日志配置 ==================== -->
    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="ASYNC_FILE"/>
    </root>
</configuration>
```

这个配置的关键特性：

- **`<springProperty>`**：在logback中引用Spring Boot的配置值，实现一处配置多处生效
- **控制台 + 文件双输出**：开发时看控制台，生产时查文件
- **按日期和大小双重滚动**：避免单个日志文件过大，同时按日期归档便于查找
- **ERROR单独文件**：运维可以只关注error日志快速定位问题
- **异步日志（AsyncAppender）**：日志写入不阻塞业务线程，提升性能
- **MyBatis Mapper DEBUG**：开发和测试环境可以看到完整的SQL语句及参数
- **`additivity="false"`**：阻止日志向上传播到root，避免重复输出

### 19.1.6 日志冲突排查

> 🚨 **坑点：Class path contains multiple SLF4J bindings**
> 
> 这是Spring Boot中最常见的日志警告，完整信息如下：
> ```
> SLF4J: Class path contains multiple SLF4J bindings.
> SLF4J: Found binding in [jar:file:/.../logback-classic-1.4.12.jar!/.../StaticLoggerBinder.class]
> SLF4J: Found binding in [jar:file:/.../log4j-slf4j-impl-2.20.0.jar!/.../StaticLoggerBinder.class]
> SLF4J: See http://www.slf4j.org/codes.html#multiple_bindings for an explanation.
> ```
>
> 原因：classpath中同时存在多个SLF4J实现（如logback和log4j-slf4j-impl），SLF4J不知道用哪个。
>
> **排查步骤**：
> 1. 用Maven命令查看依赖树：
>    ```bash
>    mvn dependency:tree | findstr "slf4j logback log4j"
>    ```
> 2. 找到引入多余日志框架的依赖（通常是某个第三方库传递依赖了log4j-slf4j-impl）
> 3. 用`<exclusion>`排除：
>    ```xml
>    <dependency>
>        <groupId>某个第三方库</groupId>
>        <artifactId>某个artifact</artifactId>
>        <exclusions>
>            <exclusion>
>                <groupId>org.apache.logging.log4j</groupId>
>                <artifactId>log4j-slf4j-impl</artifactId>
>            </exclusion>
>        </exclusions>
>    </dependency>
>    ```

### 19.1.7 日志最佳实践与常见坑

> 🚨 **坑点：生产环境用DEBUG级别 → 日志爆炸 → 磁盘写满 → 系统崩溃**
> - DEBUG级别下，一个简单的HTTP请求可能产生几十行日志（请求参数、SQL、响应结果）
> - 假设每秒1000个请求，每条请求50行DEBUG日志 → 每秒5万行 → 每分钟300万行 → 几小时就能填满磁盘
> - **原则**：生产环境默认INFO或WARN，仅临时排查问题时动态调整为DEBUG（见19.2节的Actuator动态调整）

> 🚨 **坑点：不要用System.out.println调试！**
> - `System.out.println` 输出到标准输出流，不受日志框架管理 → 无级别、无格式、无文件记录
> - 上线后忘记删除 → 生产环境控制台输出大量无用信息
> - 高并发下synchronized导致的性能瓶颈
> - **结论**：从今天开始，在你的项目中搜索`System.out.println`并全部替换为`log.info()`

**日志最佳实践速查**：

| 场景 | 应该做的 | 不应该做的 |
|------|---------|-----------|
| 方法入口 | `log.info("开始处理订单，orderId={}", orderId)` | `log.info("开始处理订单")` — 缺少关键标识 |
| 异常捕获 | `log.error("处理订单失败，orderId={}", orderId, e)` | `e.printStackTrace()` — 不受日志框架管理 |
| 循环内 | 只在循环前后各打一行 | 循环内每行都打 — 日志量爆炸 |
| 敏感信息 | `log.info("用户登录，userId={}", userId)` | `log.info("密码={}", password)` — 密码绝不打日志 |
| 大对象 | `log.debug("请求参数: {}", toJsonSafely(dto))` | `log.debug("请求参数: " + dto.toString())` — 字符串拼接永远执行 |

## 19.2 Actuator监控

### 19.2.1 Actuator是什么？

Spring Boot Actuator是Spring Boot的"运维工具包"。它提供了一系列内置端点（Endpoint），让你能实时查看应用的内部状态：

- 应用健康吗？（health）
- 有哪些Bean？（beans）
- 用了哪些配置？（env/configprops）
- 内存/CPU怎么样？（metrics）
- 最近有哪些HTTP请求？（httptrace）

> **类比**：Actuator就像汽车仪表盘——你不看也能开车，但有了它能及时发现油量不足、水温过高等问题。

### 19.2.2 添加依赖

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

添加后重启应用，访问 `http://localhost:8080/api/actuator`（注意你的`context-path`），你会看到：

```json
{
  "_links": {
    "self": { "href": "http://localhost:8080/api/actuator", "templated": false },
    "health": { "href": "http://localhost:8080/api/actuator/health", "templated": false },
    "health-path": { "href": "http://localhost:8080/api/actuator/health/{*path}", "templated": true }
  }
}
```

默认只暴露了`health`端点。要查看更多，需要配置。

### 19.2.3 常用端点速查表

| 端点 | 功能 | 生产环境建议 |
|------|------|-------------|
| `health` | 应用健康状态（DB/Redis连接是否正常） | ✅ 必暴露 |
| `info` | 应用基本信息（版本、描述） | ✅ 可暴露 |
| `metrics` | JVM内存、CPU、HTTP请求统计 | ⚠️ 需授权 |
| `env` | 所有环境变量和配置属性 | 🚫 **禁止公网暴露！** |
| `configprops` | 所有`@ConfigurationProperties` Bean | 🚫 **禁止公网暴露！** |
| `beans` | Spring容器中所有Bean的列表 | 🚫 内部使用 |
| `loggers` | 运行时动态修改日志级别 | ⚠️ 需授权 |
| `mappings` | 所有`@RequestMapping`路径 | ⚠️ 内部使用 |
| `threaddump` | 线程快照（排查死锁、线程阻塞） | ⚠️ 需授权 |
| `heapdump` | 堆内存快照（排查内存泄漏） | 🚫 **禁止公网暴露！** |
| `scheduledtasks` | 所有定时任务 | ⚠️ 内部使用 |
| `caches` | 缓存信息 | ⚠️ 内部使用 |

> 🚨 **安全警告：绝对不要将所有端点到暴露到公网！**
> 
> - `env`端点会泄露**所有环境变量**（包括数据库密码、API密钥等敏感信息）
> - `heapdump`端点下载的文件包含**内存中所有对象**（包括解密后的密码明文、JWT密钥等）
> - `configprops`端点显示**所有配置属性**（包括数据库连接字符串等）
> 
> **生产环境最佳实践**：
> 1. 只暴露`health`和`info`
> 2. 其他端点通过Spring Security保护（见第23章）
> 3. 或者直接修改`management.server.port`使用不同端口（不对外暴露）
> 4. 结合Prometheus + Grafana间接获取metrics，而非直接暴露端点

### 19.2.4 端点暴露配置

**开发环境（可以全开）**：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: "*"           # 暴露所有端点
  endpoint:
    health:
      show-details: always     # 显示健康检查的详细信息
```

**生产环境（最小暴露）**：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info   # 只暴露健康检查和基本信息
  endpoint:
    health:
      show-details: when-authorized  # 有权限才显示详情
  server:
    port: 8081                # 用不同端口（可选）
```

### 19.2.5 自定义健康检查

Actuator默认的健康检查会检测数据库连接、磁盘空间等。你可以添加自定义的健康检查：

```java
package com.example.ems.health;

import org.springframework.boot.actuate.health.Health;
import org.springframework.boot.actuate.health.HealthIndicator;
import org.springframework.stereotype.Component;

import java.net.InetSocketAddress;
import java.net.Socket;

@Component
public class ExternalServiceHealthIndicator implements HealthIndicator {

    @Override
    public Health health() {
        // 检查外部支付服务是否可达
        if (isServiceReachable("payment.internal.com", 8443)) {
            return Health.up()
                    .withDetail("service", "payment-service")
                    .withDetail("host", "payment.internal.com:8443")
                    .build();
        } else {
            return Health.down()
                    .withDetail("service", "payment-service")
                    .withDetail("error", "连接超时")
                    .build();
        }
    }

    private boolean isServiceReachable(String host, int port) {
        try (Socket socket = new Socket()) {
            socket.connect(new InetSocketAddress(host, port), 3000);
            return true;
        } catch (Exception e) {
            return false;
        }
    }
}
```

访问`/actuator/health`时就能看到自定义的健康状态：

```json
{
  "status": "UP",
  "components": {
    "db": { "status": "UP" },
    "externalService": {
      "status": "UP",
      "details": {
        "service": "payment-service",
        "host": "payment.internal.com:8443"
      }
    }
  }
}
```

### 19.2.6 自定义Info信息

**方式一：application.yml**：

```yaml
management:
  info:
    env:
      enabled: true   # 允许通过环境变量覆盖info

info:
  app:
    name: EMS-员工管理系统
    version: 1.0.0
    description: 企业员工管理后台系统
  contact:
    email: admin@example.com
    team: 基础架构组
```

**方式二：实现InfoContributor**：

```java
@Component
public class CustomInfoContributor implements InfoContributor {

    @Override
    public void contribute(Info.Builder builder) {
        builder
            .withDetail("java-version", System.getProperty("java.version"))
            .withDetail("os", System.getProperty("os.name"))
            .withDetail("startup-time", LocalDateTime.now().toString());
    }
}
```

访问`/actuator/info`时两种方式会合并输出。

### 19.2.7 运行时动态修改日志级别

你不需要重启应用就能修改日志级别——运维排查问题时的利器：

```bash
# 查看com.example.ems包的日志级别
GET /actuator/loggers/com.example.ems

# 将com.example.ems包的级别改为DEBUG
POST /actuator/loggers/com.example.ems
Content-Type: application/json

{ "configuredLevel": "DEBUG" }
```

应用处理完毕后改回去：

```json
{ "configuredLevel": "INFO" }
```

这个功能让你在生产环境临时排错时避免全局开启DEBUG导致的日志爆炸。

### 19.2.8 Prometheus + Grafana集成概述

> 本节为概述，不涉及详细搭建步骤。

Actuator配合Micrometer可以暴露Prometheus格式的metrics：

```xml
<dependency>

    <artifactId>micrometer-registry-prometheus</artifactId>

```


然后Prometheus定期拉取`/actuator/prometheus`端点数据，在Grafana中可视化展示。这是云原生时代的标准监控方案。

### 19.2.9 独立管理端口

将Actuator端口与应用端口分离，在防火墙上屏蔽管理端口：

```yaml
    address: 127.0.0.1   # 只监听本地，外部无法访问
```

这样暴露所有端点也不怕，因为只有服务器本地能访问。运维通过SSH端口转发或跳板机访问。

## 19.3 JUnit 5 测试框架

### 19.3.1 JUnit 5 架构

JUnit 5不是JUnit 4的简单升级，而是完全重新设计的模块化架构：

```
┌────────────────────────────────────────────┐
│              JUnit 5 Platform              │ ← 启动测试框架的基础平台
│         (TestEngine API + Launcher)        │    支持IDE、Maven、Gradle
├──────────────────┬─────────────────────────┤
│   JUnit Jupiter  │   JUnit Vintage         │
│   (新测试引擎)    │   (兼容JUnit 3/4)       │
│   @Test 等新注解  │   跑老测试代码           │
└──────────────────┴─────────────────────────┘
```

核心依赖：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>
```

这个starter已经包含了JUnit 5、Mockito、AssertJ等测试所需的一切。

### 19.3.2 核心注解

```java
import org.junit.jupiter.api.*;

public class CalculatorTest {

    private Calculator calculator;

    @BeforeAll
    static void initAll() {
        // 所有测试方法执行前执行一次（必须是static！）
        System.out.println("开始测试...");
    }

    @BeforeEach
    void init() {
        // 每个测试方法执行前都执行一次
        calculator = new Calculator();
    }

    @Test
    @DisplayName("测试 1 + 1 = 2")
    void testAdd() {
        assertEquals(2, calculator.add(1, 1), "1 + 1 应该等于 2");
    }

    @Test
    @DisplayName("测试除法：被除数为0应抛异常")
    void testDivideByZero() {
        assertThrows(ArithmeticException.class,
                () -> calculator.divide(10, 0));
    }

    @AfterEach
    void tearDown() {
        // 每个测试方法执行后都执行一次
        calculator = null;
    }

    @AfterAll
    static void tearDownAll() {
        // 所有测试方法执行后执行一次（必须是static！）
        System.out.println("测试结束。");
    }
}
```

> 🚨 **坑点：@Test必须是org.junit.jupiter.api.Test，不要导入JUnit4的@Test！**
> - JUnit 4的`@Test`是`org.junit.Test`
> - JUnit 5的`@Test`是`org.junit.jupiter.api.Test`
> - 如果导入了JUnit 4的`@Test`，测试方法**不会被执行**（JUnit Vintage引擎可能不生效）
> - **快速检查**：看import语句。如果看到`org.junit.Test`，立刻改成`org.junit.jupiter.api.Test`
> - 如果你使用了`spring-boot-starter-test`，它默认排除JUnit Vintage，所以JUnit 4的注解完全不生效

### 19.3.3 断言API

```java
@Slf4j
public class AssertionDemo {

    @Test
    @DisplayName("基本断言")
    void testBasicAssertions() {
        // 相等断言
        assertEquals(4, 2 + 2);
        assertEquals(0.001, 1.0 / 3.0, 0.01, "误差在0.01内");  // 浮点数带容差

        // 真假断言
        assertTrue("hello".startsWith("he"));
        assertFalse("hello".isEmpty());

        // 空/非空
        assertNull(null);
        assertNotNull("hello");

        // 引用相等
        String a = new String("hello");
        String b = new String("hello");
        assertNotSame(a, b);       // 不同对象
        assertEquals(a, b);        // 但内容相同
    }

    @Test
    @DisplayName("异常断言")
    void testExceptionAssertion() {
        // 验证异常类型
        ArithmeticException exception = assertThrows(ArithmeticException.class, () -> {
            int result = 10 / 0;
        });
        // 还可以验证异常信息
        assertEquals("/ by zero", exception.getMessage());
    }

    @Test
    @DisplayName("批量断言（一个失败不影响其他）")
    void testAssertAll() {
        Employee emp = new Employee("张三", 25, "技术部", 15000.0);

        assertAll("员工信息校验",
            () -> assertEquals("张三", emp.getName()),
            () -> assertEquals(25, emp.getAge()),
            () -> assertEquals("技术部", emp.getDepartment()),
            () -> assertTrue(emp.getSalary() > 0)
        );
        // assertAll会执行所有断言，汇总所有失败，而不是遇到第一个失败就停止
    }
}
```

`assertAll`是JUnit 5最有价值的改进之一。在JUnit 4中，第一个断言失败后面的断言不会执行——你修好了第一个，运行才知道第二个也失败。`assertAll`一次告诉你所有失败。

### 19.3.4 参数化测试

当同一逻辑需要测试多组数据时，不要写多个几乎一模一样的`@Test`方法：

```java
public class ParameterizedDemo {

    @ParameterizedTest
    @ValueSource(ints = {1, 3, 5, -3, 15})
    @DisplayName("测试奇数判断")
    void testIsOdd(int number) {
        assertTrue(MathUtils.isOdd(number),
                number + " 应该是奇数");
    }

    @ParameterizedTest
    @CsvSource({
        "1, 1, 2",
        "0, 0, 0",
        "-1, 5, 4",
        "100, 200, 300"
    })
    @DisplayName("测试加法（多组数据）")
    void testAdd(int a, int b, int expected) {
        assertEquals(expected, calculator.add(a, b));
    }

    @ParameterizedTest
    @CsvSource({
        "admin, 123456, true",
        "admin, wrong, false",
        "unknown, 123456, false"
    })
    @DisplayName("测试登录（用户名+密码+期望结果）")
    void testLogin(String username, String password, boolean expected) {
        assertEquals(expected, authService.login(username, password));
    }

    @ParameterizedTest
    @EnumSource(Department.class)
    @DisplayName("所有部门都应该有对应代码")
    void testDepartmentCode(Department dept) {
        assertNotNull(dept.getCode());
        assertTrue(dept.getCode().length() > 0);
    }

    @ParameterizedTest
    @MethodSource("provideStringsForConcat")
    @DisplayName("测试字符串拼接（方法数据源）")
    void testConcat(String a, String b, String expected) {
        assertEquals(expected, a + b);
    }

    static Stream<Arguments> provideStringsForConcat() {
        return Stream.of(
            Arguments.of("Hello", "World", "HelloWorld"),
            Arguments.of("", "test", "test"),
            Arguments.of("a", "", "a")
        );
    }
}
```

## 19.4 Spring Boot 测试

### 19.4.1 @SpringBootTest — 完整容器测试

```java
@SpringBootTest
class EmployeeServiceIntegrationTest {

    @Autowired
    private EmployeeService employeeService;

    @Test
    @DisplayName("集成测试：创建员工并查询")
    void testCreateAndFindEmployee() {
        EmployeeDTO dto = new EmployeeDTO("张三", 25, "技术部", 15000.0, "zhangsan@test.com");
        Long id = employeeService.create(dto);
        
        EmployeeVO vo = employeeService.findById(id);
        assertNotNull(vo);
        assertEquals("张三", vo.getName());
    }
}
```

`@SpringBootTest`默认会启动**完整的Spring容器**（包括数据库连接、所有Bean等）。这最接近真实环境，但代价是**慢**。

`webEnvironment`属性控制Web环境：

| 值 | 说明 |
|----|------|
| `MOCK`（默认） | 不启动Web服务器，模拟Servlet环境 |
| `RANDOM_PORT` | 启动真实Web服务器，使用随机端口 |
| `DEFINED_PORT` | 启动真实Web服务器，使用配置的端口 |
| `NONE` | 不启动任何Web环境 |

> 🚨 **坑点：一个模块几十个@SpringBootTest → CI流水线跑十几分钟**
> - 每个`@SpringBootTest`都需要启动完整Spring容器，启动一次5-30秒
> - 如果有40个`@SpringBootTest`类 → 启动40次容器 → 至少200秒 → 3分钟以上
> - 解决方案见下一节：**测试切片**

### 19.4.2 测试切片 — 只加载需要的Bean（快！）

Spring Boot提供了一系列"测试切片"注解，每个只加载特定层的Bean，速度比`@SpringBootTest`快10倍以上：

| 切片注解 | 加载的Bean | 不加载的Bean | 典型用途 |
|---------|-----------|-------------|---------|
| `@WebMvcTest` | Controller、MockMvc | Service、Repository | 测试Controller层 |
| `@DataJpaTest` | Repository、EntityManager | Controller、Service | 测试JPA数据层 |
| `@MybatisTest` | Mapper、SqlSession | Controller、Service | 测试MyBatis数据层 |
| `@JsonTest` | JSON序列化/反序列化 | 几乎所有 | 测试JSON映射 |
| `@RestClientTest` | RestTemplate | 大部分 | 测试REST客户端 |
| `@WebFluxTest` | WebFlux Controller | Service等 | 测试响应式Controller |

### 19.4.3 @WebMvcTest — Controller 层测试（独立完整示例）

只测试Controller层，Service等依赖用Mockito模拟：

```java
@WebMvcTest(EmployeeController.class)
@DisplayName("EmployeeController 单元测试")
class EmployeeControllerTest {

    @Autowired
    private MockMvc mockMvc;  // 模拟HTTP请求

    @MockBean  // ← 替换Spring容器中的EmployeeService为Mock
    private EmployeeService employeeService;

    @Test
    @DisplayName("GET /api/employees/{id} — 查询员工成功")
    void testFindById_Success() throws Exception {
        // 准备Mock行为
        EmployeeVO mockVO = new EmployeeVO();
        mockVO.setId(1L);
        mockVO.setName("张三");
        mockVO.setDepartment("技术部");

        given(employeeService.findById(1L)).willReturn(mockVO);

        // 发送HTTP请求并验证
        mockMvc.perform(get("/api/employees/1")
                        .contentType(MediaType.APPLICATION_JSON))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.code").value(200))
                .andExpect(jsonPath("$.data.name").value("张三"))
                .andExpect(jsonPath("$.data.department").value("技术部"));

        // 验证service.findById(1L)确实被调用了
        verify(employeeService).findById(1L);
    }

    @Test
    @DisplayName("GET /api/employees/999 — 查询不存在的员工返回404")
    void testFindById_NotFound() throws Exception {
        given(employeeService.findById(999L))
                .willThrow(new BusinessException(404, "员工不存在"));

        mockMvc.perform(get("/api/employees/999"))
                .andExpect(status().isNotFound())
                .andExpect(jsonPath("$.code").value(404))
                .andExpect(jsonPath("$.message").value("员工不存在"));
    }

    @Test
    @DisplayName("POST /api/employees — 创建员工：参数校验失败返回400")
    void testCreate_ValidationFail() throws Exception {
        // 发送不合法数据（name为空）
        String requestBody = """
            {
                "name": "",
                "age": 25,
                "department": "技术部",
                "salary": 15000,
                "email": "test@example.com"
            }
            """;

        mockMvc.perform(post("/api/employees")
                        .contentType(MediaType.APPLICATION_JSON)
                        .content(requestBody))
                .andExpect(status().isBadRequest())
                .andExpect(jsonPath("$.code").value(400));
    }
}
```

> 🚨 **坑点：@WebMvcTest不会加载完整的Spring Security配置**
> - `@WebMvcTest`只加载Controller相关的Bean，不包含`SecurityFilterChain`
> - 如果你的Controller有`@PreAuthorize`等权限注解，在`@WebMvcTest`中不会生效
> - 如果需要测试安全相关的行为，使用`@SpringBootTest` + `@AutoConfigureMockMvc`
> - 或者使用`@WebMvcTest` + `@Import(SecurityConfig.class)`手动添加安全配置

### 19.4.4 @DataJpaTest — JPA 数据层测试（独立完整示例）

只加载JPA相关Bean（EntityManager、Repository等），自动使用H2内存数据库：

```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
// ↑ 不使用内嵌数据库，使用真实MySQL的话加这一行
@DisplayName("EmployeeRepository JPA测试")
class EmployeeRepositoryTest {

    @Autowired
    private TestEntityManager entityManager;  // 测试专用的EntityManager

    @Autowired
    private EmployeeRepository employeeRepository;

    @Test
    @DisplayName("findByDepartment — 按部门查询员工")
    void testFindByDepartment() {
        // 插入测试数据
        Employee emp1 = new Employee("张三", 25, "技术部", 15000.0, "zhangsan@test.com");
        Employee emp2 = new Employee("李四", 30, "技术部", 18000.0, "lisi@test.com");
        Employee emp3 = new Employee("王五", 28, "产品部", 17000.0, "wangwu@test.com");
        entityManager.persist(emp1);
        entityManager.persist(emp2);
        entityManager.persist(emp3);
        entityManager.flush();

        // 查询
        List<Employee> techEmployees = employeeRepository.findByDepartment("技术部");
        assertEquals(2, techEmployees.size());
    }

    @Test
    @DisplayName("findBySalaryGreaterThan — 按薪资查询")
    void testFindBySalaryGreaterThan() {
        entityManager.persist(new Employee("张三", 25, "技术部", 15000.0, "zhangsan@test.com"));
        entityManager.persist(new Employee("李四", 30, "技术部", 18000.0, "lisi@test.com"));
        entityManager.flush();

        List<Employee> rich = employeeRepository.findBySalaryGreaterThan(16000.0);
        assertEquals(1, rich.size());
        assertEquals("李四", rich.get(0).getName());
    }
}
```

> 🚨 **坑点：H2和MySQL语法不完全兼容**
> - `@DataJpaTest`默认使用H2内存数据库（因为不需要外部依赖，测试速度快）
> - 但H2和MySQL在以下方面有差异：
>   - `AUTO_INCREMENT` → H2用`AUTO_INCREMENT`可以，但`IDENTITY`更保险
>   - MySQL的`FULLTEXT INDEX` → H2不支持
>   - MySQL的`ON UPDATE CURRENT_TIMESTAMP` → H2不支持
>   - MySQL的某些SQL函数（`GROUP_CONCAT`等） → H2不支持或语法不同
> - **解决方案**：如果项目大量使用MySQL特有语法，用`@AutoConfigureTestDatabase(replace = Replace.NONE)`指定使用真实MySQL（配合Testcontainers更好）

### 19.4.5 Mockito 整合

Mockito是Java生态中最流行的Mock框架。在Spring Boot测试中，它通过`@MockBean`与Spring容器深度整合：

```java
@ExtendWith(MockitoExtension.class)  // 纯Mockito（不启动Spring容器）
@DisplayName("EmployeeService 纯Mockito测试")
class EmployeeServiceMockitoTest {

    @Mock  // 创建一个Mock对象（不会替换Spring容器中的Bean）
    private EmployeeRepository employeeRepository;

    @InjectMocks  // 将上面两个Mock注入到这个对象中
    private EmployeeService employeeService;

    @Test
    @DisplayName("findById — 正常查询")
    void testFindById_Success() {
        Employee mockEmployee = new Employee("张三", 25, "技术部", 15000.0, "zhangsan@test.com");
        given(employeeRepository.findById(1L)).willReturn(Optional.of(mockEmployee));

        EmployeeVO result = employeeService.findById(1L);

        assertNotNull(result);
        assertEquals("张三", result.getName());
        verify(employeeRepository).findById(1L);  // 验证方法确实被调用了
    }

    @Test
    @DisplayName("findById — 查不到抛异常")
    void testFindById_NotFound() {
        given(employeeRepository.findById(999L)).willReturn(Optional.empty());

        assertThrows(BusinessException.class, () -> employeeService.findById(999L));
    }
}
```

**关键概念对比**：

| 注解 | 范围 | 是否启动Spring容器 |
|------|------|-------------------|
| `@Mock` + `@InjectMocks` | 纯Mockito，不启动容器 | ❌ 不启动，非常快 |
| `@MockBean` | Spring Boot测试，替换容器中的Bean | ✅ 启动容器 |

`@MockBean`会替换Spring容器中**同类型的真实Bean**为Mock对象。这是它与`@Mock`的最大区别。

> 🚨 **坑点：@MockBean污染Spring容器 → 多个测试类间相互影响 → 用@DirtiesContext清理**
> - `@MockBean`会把Spring容器中的真实Bean替换为Mock
> - 如果测试类A中把`EmployeeService` Mock了，测试类B执行时可能拿到的是Mock而非真实Service
> - Spring Boot测试默认会**复用**Spring容器以提升速度（缓存ApplicationContext）
> - **解决方案1**：如果Mock替换了关键Bean，在测试类上加`@DirtiesContext`，告诉Spring这个测试类"弄脏了"容器，下一个测试重建
> - **解决方案2**：优先使用测试切片（`@WebMvcTest`等），它们每次只加载需要的Bean，容器互不影响

## 19.5 测试覆盖率 — JaCoCo

### 19.5.1 为什么需要覆盖率？

测试覆盖率回答了一个朴素的问题：**你的代码有多少被测试了？** 但覆盖率不是银弹——100%覆盖率不等于没有Bug。覆盖率的价值在于：告诉你在代码的哪片区域完全没有测试。

### 19.5.2 JaCoCo 配置

```xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.11</version>
    <executions>
        <execution>
            <goals>
                <goal>prepare-agent</goal>  <!-- 测试时启动JaCoCo代理 -->
            </goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>test</phase>
            <goals>
                <goal>report</goal>         <!-- test阶段结束后生成报告 -->
            </goals>
        </execution>
    </executions>
</plugin>
```

运行测试并生成报告：

```bash
mvn clean test
```

报告位置：`target/site/jacoco/index.html`，用浏览器打开。

### 19.5.3 覆盖率指标解读

JaCoCo报告中有两个关键指标：

| 指标 | 含义 | 示例 |
|------|------|------|
| **行覆盖率（Line Coverage）** | 哪些行被执行过 | 100行代码中有80行被测试走到了 = 80% |
| **分支覆盖率（Branch Coverage）** | if-else、switch等分支是否都被测试 | 4个分支测试了3个 = 75% |

分支覆盖率比行覆盖率更难提升——因为即使一行代码被执行了，其中的if-else可能只走了一半。

### 19.5.4 覆盖率目标

不是所有代码都必须100%覆盖。合理的目标：

| 代码类型 | 覆盖率目标 | 原因 |
|---------|-----------|------|
| **核心业务逻辑（Service）** | ≥80% | 业务规则多，易出Bug |
| **Controller** | ≥70% | 主要测参数校验和异常分支 |
| **工具类（Utils）** | ≥80% | 被多处调用，影响面大 |
| **Repository/DAO** | ≥60% | SQL本身由数据库保证 |
| **配置类、POJO** | 可不测 | getter/setter/配置绑定，框架已保证 |
| **启动类** | 可不测 | 只有一个run方法 |

**核心原则**：测试有价值的地方，而非追求100%的数字。

**第1题**：

可能原因：某个`<logger>`标签中缺少`additivity="false"`。

例如：
```xml
<logger name="com.example.ems" level="DEBUG">
    <appender-ref ref="CONSOLE"/>
</logger>
```

此logger的日志会先被CONSOLE appender输出一次，然后因为`additivity`默认为`true`，向上传播到`<root>`，又被root中引用的CONSOLE appender输出第二次。

修正方案：添加`additivity="false"`：
```xml
<logger name="com.example.ems" level="DEBUG" additivity="false">
    <appender-ref ref="CONSOLE"/>
</logger>
```

**第2题**：

使用Actuator的loggers端点：

```bash
# 1. 查看当前Mapper包的日志级别
curl http://服务器:8080/api/actuator/loggers/com.example.ems.mapper

# 2. 临时改为DEBUG
curl -X POST http://服务器:8080/api/actuator/loggers/com.example.ems.mapper \
  -H "Content-Type: application/json" \
  -d '{"configuredLevel": "DEBUG"}'

# 3. 观察日志定位问题

# 4. 问题解决后恢复INFO
curl -X POST http://服务器:8080/api/actuator/loggers/com.example.ems.mapper \
  -H "Content-Type: application/json" \
  -d '{"configuredLevel": "INFO"}'
```

当然前提是生产环境的Actuator配置了这个端点的访问权限（通过Spring Security或管理端口隔离）。

**第3题**：

错误1：导入了JUnit 4的`@Test`（`org.junit.Test`），不是JUnit 5的`@Test`（`org.junit.jupiter.api.Test`）。应改为`org.junit.jupiter.api.Test`。

错误2：期望`calculator.divide(10, 0)`抛出异常，但没有用`assertThrows`包裹。正确的写法：

```java
@Test
@DisplayName("测试除以0抛出异常")
void testDivideByZero() {
    int result = calculator.divide(10, 2);
    assertEquals(5, result);

    // 用 assertThrows 验证异常
    assertThrows(ArithmeticException.class,
        () -> calculator.divide(10, 0));
}
```

原代码中`calculator.divide(10, 0)`如果抛异常，测试会直接失败（因为没有用`assertThrows`声明期望该异常），而不会"期望抛出异常"。

**第4题**：

优化方案（组合使用，效果最佳）：

1. **按需使用测试切片**：
   - Controller层 → `@WebMvcTest`（启动时间从10秒降到2秒）
   - JPA Repository → `@DataJpaTest`（启动时间从10秒降到1秒）
   - 只有跨层集成测试（如Service+Repository+Controller联动）才用`@SpringBootTest`

2. **分类测试**：
   - 快速测试（纯单元测试+测试切片）：30秒内完成 → 每次push都跑
   - 慢速测试（@SpringBootTest集成测试）：保留5-8个关键业务流程测试 → 仅在合并到主分支前跑

3. **使用@DirtiesContext谨慎标记**：Spring Boot默认复用容器，避免不必要的容器重建

4. **拆分为多个Maven模块**：每个模块独立测试，Maven并行执行

预期效果：构建时间从15分钟降到3-5分钟。

**第5题**：

主要区别：

| 维度 | `@WebMvcTest` | `@SpringBootTest(WebEnvironment.MOCK)` |
|------|--------------|--------------------------------------|
| **加载Bean范围** | 只加载Controller层相关Bean | 加载**全部**Spring Bean |
| **Service/Repository** | **不加载**，需`@MockBean` | 完全加载 |
| **启动时间** | 1-3秒 | 5-15秒 |
| **适用场景** | 测试Controller的请求参数验证、HTTP响应格式 | 测试Controller→Service→Repository的完整链路 |

**使用建议**：
- `@WebMvcTest`：测试Controller的参数校验、MockMvc请求/响应、异常处理映射（如第16章全局异常处理），这些不需要真实Service
- `@SpringBootTest`：端到端集成测试，验证整个请求链路（Controller→Service→Repository→DB）正确工作

**第6题**：

问题：`@DataJpaTest`默认使用H2内存数据库，H2不支持MySQL的`JSON_EXTRACT`、`FULLTEXT INDEX`等特有函数和语法。如果用H2跑包含这些语法的测试，SQL解析会直接报错。

解决方案（按推荐度排序）：

**方案1（最推荐）— Testcontainers**：在测试中启动真实的MySQL Docker容器
```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
@Testcontainers
class EmployeeRepositoryTest {
    @Container
    static MySQLContainer<?> mysql = new MySQLContainer<>("mysql:8.0")
        .withDatabaseName("testdb")
        .withUsername("test")
        .withPassword("test");
    
    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", mysql::getJdbcUrl);
        registry.add("spring.datasource.username", mysql::getUsername);
        registry.add("spring.datasource.password", mysql::getPassword);
    }
}
```
这是最接近真实环境的方式，唯一的代价是需要安装Docker。

**方案2 — 使用真实MySQL**：
```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
```
测试将直接连接`application-test.yml`中配置的数据库。缺点是需要外部数据库，CI需额外配置。

**方案3 — 避免MySQL特有语法**：尽量用标准SQL，MySQL特有函数通过Service层封装（Mock Service层的特殊逻辑），Repository层只用标准JPA/Hibernate语法。
