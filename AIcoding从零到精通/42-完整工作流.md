## 42.1 AI编程不是"打开AI说句话就能用"

很多新手以为AI编程是这样的：

1. 打开Cursor
2. 说一句"帮我做个XXX"
3. 代码出来
4. 跑起来
5. 完事

**现实是**：第3步和第4步之间，往往隔着**无数次报错、调试、返工**。如果你没有一套清晰的工作流，你会陷入"修了这里坏了那里"的泥潭，最后发现整个项目是一团乱麻。

这一章，我们建立一套**完整的AI编程工作流**。它不是让你写更多代码，而是让你**写更少返工代码**。

---

## 42.2 什么是工作流

> **📖 术语：工作流（Workflow）**——把一件复杂的事情，拆成一系列有顺序的步骤，每一步有明确的输入、输出和完成标准。就像工厂流水线：第一个人拧螺丝，第二个人装外壳，第三个人质检——每个人只做一件事，但整体产出是完整的。

AI编程的工作流，就是把"从想法到成品"拆成六个步骤：

```
需求 → 设计 → 编码 → 测试 → 调试 → 部署
```

**每一步，AI能干一部分，人必须干一部分。** 如果你以为AI能帮你干全部，那你就等着返工吧。

---

## 42.3 六步工作流详解

### 第一步：需求（Requirement）——"你到底想要什么？"

这是**最容易被跳过、但最重要的一步**。

很多人打开AI就开始说："帮我写一个博客系统。"AI问："你要什么功能？"他答："就一个博客系统啊，能写文章、能看文章就行。"

**这种模糊需求，AI写出来的代码大概率不是你想要的。** 因为"博客系统"这四个字，在AI脑子里可能有100种不同的实现方式。

#### ✅ 正确做法：先写需求文档

在让AI写任何代码之前，先自己回答这些问题（可以在纸上，也可以在Cursor里开一个文档）：

1. **这个项目是干什么的？**（一句话说清楚）
2. **核心功能有哪些？**（列出来，不要超过5个）
3. **谁会用到这个项目？**（只有你自己？还是别人也会用？）
4. **数据长什么样？**（用户信息？文章？评论？每个有什么字段？）
5. **界面大概长什么样？**（不用画，用文字描述就行）

**写完之后，把这段需求发给AI，让AI帮你检查遗漏**：

```
我正在规划一个[项目名]，核心需求如下：
[粘贴你的需求]

请帮我检查：
1. 有没有遗漏的重要功能？
2. 有没有逻辑矛盾的地方？
3. 有没有我不知道但应该知道的技术挑战？
```

#### AI能帮你做什么？人必须做什么？

| AI能做 | 人必须做 |
|--------|----------|
| 帮你检查需求是否有遗漏 | 定义核心目标（只有你知道你要什么） |
| 帮你补充技术细节（比如"你需要一个数据库存文章"） | 判断优先级（哪个功能先做，哪个后做） |
| 帮你把模糊需求变成具体功能点 | 确认最终需求（拍板权在你手里） |

---

### 第二步：设计（Design）——"整体框架长什么样？"

需求清楚了，第2步不是写代码，是**设计**。

> **设计不是让你画UI图**。对于AI编程，设计主要指：
> 1. **技术选型**：用什么语言？什么框架？什么数据库？
> 2. **文件结构**：项目有哪些文件？怎么组织？
> 3. **数据流**：数据从哪来，到哪去，怎么存？

#### 怎么让AI帮你设计？

把需求文档发给AI，加上这句话：

```
基于以上需求，帮我做技术设计：
1. 推荐技术栈（语言、框架、数据库），说明为什么选这个
2. 规划项目文件结构（每个文件干什么）
3. 设计数据库表结构（列出表和字段）
4. 如果有前端页面，列出有哪些页面及每个页面的功能
```

AI会给你一份完整的设计方案。

#### ⚠️ 关键：你拍板，不是AI拍板

AI推荐的技术栈，你**必须**自己判断一下：

- 你熟悉这个语言/框架吗？如果不熟悉，学起来要多久？
- 这个技术栈适合你的项目规模吗？如果只是一个小工具，不需要Spring Boot这种重型框架。
- 如果AI推荐了付费服务（比如某个云服务），你有预算吗？

**比如AI说"用React + Node.js + MongoDB"**——如果你只会Python，那就告诉AI："我只会Python，请用Python Flask + SQLite重新设计。"

#### AI能帮你做什么？人必须做什么？

| AI能做 | 人必须做 |
|--------|----------|
| 推荐技术栈并说明理由 | 根据自身情况判断是否采纳 |
| 设计文件结构和数据库表 | 拍板最终方案 |
| 发现设计中的潜在问题 | 确认设计是否满足所有需求 |

---

### 第三步：编码（Coding）——"分文件，一步一步来"

现在才开始写代码。这也是很多人最期待的环节，但也是最容易出乱子的环节。

#### 核心原则：一次只让AI写一个文件

**❌ 错误做法**：

```
帮我写一个完整的博客系统，包括前端、后端、数据库。
```

AI会给你一大堆代码，但大概率：
- 几个文件之间接口对不上
- 引用了不存在的函数
- 你复制粘贴都嫌累

**✅ 正确做法**：按设计好的文件结构，一个文件一个文件来。

```
第一步：先帮我创建数据库模型文件 models.py，包含User和Post两个表。
第二步：（等AI写完并确认后）好，现在帮我写用户认证的 auth.py，使用刚才的User模型。
第三步：（再确认后）现在帮我写文章管理的 routes.py，关联Post模型。
```

**每一步写完，停下来检查一下**——文件能不能正常运行？和上一个文件对得上吗？

#### 编码阶段的"铁律"

1. **一个文件一个文件来**：不要一次让AI生成5个文件，它hold不住
2. **每写完一个文件就验证**：至少跑一下，别全写完才发现第一个文件就错了
3. **保持对话上下文**：在同一个Chat会话里继续，AI记得前面的文件
4. **如果上下文太长，开新会话**：但要把之前的关键信息（表结构、函数签名）重新贴给AI

#### AI能帮你做什么？人必须做什么？

| AI能做 | 人必须做 |
|--------|----------|
| 按设计生成每个文件的代码 | 决定先写哪个文件、后写哪个 |
| 处理语法细节、API调用 | 检查代码逻辑是否符合预期 |
| 解释代码的含义 | 运行代码，确认不报错 |

---

### 第四步：测试（Testing）——"跑起来，测功能对不对"

代码写完了，下一步是**测试**——不是只跑一遍不报错就完事，而是**验证功能对不对**。

#### 最小测试步骤

1. **启动项目**：跑起来，能启动就是第一步胜利
2. **核心功能逐个测试**：按需求文档，一个功能一个功能试
3. **边界情况测试**：故意输入错误的东西，看程序会不会崩溃
4. **让AI帮你写测试**：这是AI最擅长的事（详见第44章）

#### 测试发现Bug怎么办？

回到第43章（AI辅助调试）的流程，不要慌、不要自己瞎猜。

#### AI能帮你做什么？人必须做什么？

| AI能做 | 人必须做 |
|--------|----------|
| 生成测试用例 | 手工执行核心功能测试 |
| 分析测试失败的原因 | 判断Bug的严重程度 |
| 写自动化测试代码 | 确认测试是否覆盖了所有需求 |

---

### 第五步：调试（Debugging）——"不对让AI帮你修"

测试发现Bug，就进入调试阶段。这个阶段在第43章有详细讲解，这里只强调核心原则：

> **不要自己瞎猜。把完整错误信息复制给AI。**

另外，AI帮你修了Bug之后，**重新跑一遍所有测试**——修A时可能引入Bug B。

#### AI能帮你做什么？人必须做什么？

| AI能做 | 人必须做 |
|--------|----------|
| 分析错误日志，定位根因 | 把完整错误信息喂给AI |
| 提出修复方案并生成代码 | 判断修复方案是否合理 |
| 解释为什么出错 | 验证修复后功能正常 |

---

### 第六步：部署（Deployment）——"放到服务器让别人用"

部署是可选的——如果你的项目只是在本地跑给自己看，这一步可以跳过。

但如果想让别人也能用（比如你的个人主页、Todo应用），就需要部署。本书的实战项目篇（第48-51章）会涉及具体部署，这里只讲概念：

- **部署就是把代码放到一台"永远开着"的电脑上**（服务器）
- 别人通过网址就能访问你的项目
- 最简单的部署方式：用GitHub Pages（静态网页）或免费云服务（如Railway、Vercel）
- **让AI帮你写部署配置**：Dockerfile、部署脚本，AI都能写

#### AI能帮你做什么？人必须做什么？

| AI能做 | 人必须做 |
|--------|----------|
| 生成部署配置文件 | 注册云服务账号、配置域名 |
| 写部署脚本 | 提供服务器信息（不要发给AI） |
| 检查部署配置是否正确 | 执行部署，确认服务正常运行 |

## 42.4 工作流全景总结

一张表，看清楚每一步AI和人的分工：

| 步骤 | 主要产出 | AI做什么 | 人做什么 |
|------|----------|----------|----------|
| 1. 需求 | 需求文档 | 检查遗漏、补充技术细节 | 定义目标、排序优先级、拍板 |
| 2. 设计 | 技术方案、文件结构 | 推荐技术栈、设计表结构 | 判断是否合适、拍板最终方案 |
| 3. 编码 | 每个文件的代码 | 按设计生成代码 | 决定顺序、运行验证 |
| 4. 测试 | 测试用例、测试报告 | 生成测试用例 | 手工测试核心功能 |
| 5. 调试 | 修复后的代码 | 分析错误、提出修复方案 | 喂错误信息、验证修复 |
| 6. 部署 | 线上可访问的项目 | 写部署配置和脚本 | 配置服务器、执行部署 |

## 42.5 一句话记住本章

> **AI干活，人决策。** AI负责执行层面的工作（写代码、生成测试、分析错误），人负责决策层面的工作（定义需求、判断方案、拍板确认）。你不需要会写每一行代码，但你必须知道每一步要做什么决定。

## 自测

1. 为什么不能让AI一次性生成所有文件？会导致什么问题？
2. 六步工作流中，哪一步是"最容易被跳过但最重要"的？为什么？
3. 设计阶段，AI推荐了技术栈，你应该怎么做？可以直接照搬吗？

> **答案**：
> 1. AI一次生成多个文件时，文件之间的接口可能对不上，引用不存在的函数，而且复制粘贴也容易出错。正确做法是一个文件一个文件来，每写完一个就验证。
> 2. 需求（第一步）。很多人跳过需求直接让AI写代码，结果写出来的不是自己想要的，大量返工。需求越清楚，AI写得越准。
> 3. 不能直接照搬。你需要判断：这个技术栈你熟悉吗？适合你的项目规模吗？有没有付费服务？如果不合适，告诉AI你的约束条件，让它重新推荐。
