# 第55章 · 索引原理

> "想象一本800页的大学教材，没有目录、没有索引。你要找'贝叶斯定理在第几页'，只能从第1页一页一页翻到第800页——运气好第5页就找到了，运气不好翻到最后一页。这只是一个人的情况。如果1000个人同时来翻，整本书的纸张都会被翻烂。索引（数据库里的目录，帮查询快速定位数据，详见附录I）就是那本教材的目录——翻两下就能定位到页数，然后直接翻到目标位置。这是整个数据库篇最重要的一章，没有之一。"

---

## 55.1 索引是什么

没有索引时，`SELECT * FROM users WHERE email = 'test@example.com'` 的执行过程：

1. 从硬盘读取users表的第一行数据。
2. 检查email是否等于 'test@example.com'。
3. 不是→读下一行。
4. 重复1~3，直到找到目标行或遍历完全表。
5. 如果users表有100万行，最坏情况要读100万次硬盘。

这就是**全表扫描**。在数据量小时还好（几毫秒的事），但到了百万级，查询要几十秒——网站早就超时了。

加上索引后：

1. 在email索引中查找 'test@example.com'。
2. 索引告诉你：这条记录在数据文件的第12345号位置。
3. 直接跳到第12345号位置，读完。

索引查找不是一行一行看——它用的是B+树数据结构，100万行数据只需约**20次比较**就能定位目标。就像翻字典找"zebra"这个单词，你不会从第一页"A"开始翻——你直接翻到最后几页。

## 55.2 B+树原理

InnoDB存储引擎的索引底层是**B+树**（一种多叉平衡树，所有数据存在叶子节点，详见附录I）。这个名字让人望而却步，但原理其实很直观。

### 为什么不用二叉树

二叉搜索树的查找效率是 O(log N)，理论上不错。但有个致命问题：数据的插入顺序决定了树的形状。如果按id=1,2,3,4,5...顺序插入，二叉树退化成：

```
1
 \
  2
   \
    3
     \
      4
       \
        5
```

这已经不是树了——变成了一条链表。查询id=500000要比较50万次。

### B+树长什么样

B+树是一棵**多叉平衡树**。每个节点不只存一个值，而是存多个值——像一个"迷你数组"。高度控制在3~4层。

```
                    [100  |  200  |  300]          ← 根节点（第1层）
                   /      |       |      \
          [10|20|30]  [110|150] [210|250] [310|400] ← 内部节点（第2层）
          / / | \ \     ...       ...       ...
        [1][11][21][31]                           ← 叶子节点（第3层）
                   ↔  ↔  ↔                         ← 叶子节点之间用双向链表连接
```

**B+树的核心特征**：

1. **所有数据存在叶子节点**。上面的非叶子节点只存索引键和指针——不存完整数据。
2. **叶子节点之间用双向链表连接**，形成一个有序链表。这对于范围查询（`WHERE id BETWEEN 100 AND 200`）极其高效——定位到100后，顺着链表往后读就行了，不需要再次从根节点往下找。
3. **每个节点存储在一页中**（InnoDB默认页大小16KB）。一个节点能存几百到上千个键值，所以100万行数据只需要3层：根节点→内部节点→叶子节点。3次IO就能找到任意数据。
4. **自平衡**：插入和删除时自动分裂或合并节点，保证树始终"矮胖"。

### B+树查找过程（动画版）

在 `users` 表的 `age` 字段上有B+树索引，查找 `age = 25`：

```
第一步：从根节点开始
根节点：[20, 30, 40]
25 在 20 和 30 之间 → 走第二个孩子指针

第二步：进入内部节点
内部节点：[22, 25, 28]
找到了25 → 走对应的孩子指针

第三步：进入叶子节点
叶子节点：[..., 25 → (数据位置), ...]
找到25的记录，读取完整行数据

总计：3次IO操作（每个节点一次读盘）
```

如果不用索引，全表扫描一个100万行的表，按每页16KB计算（InnoDB页大小），大约需要几十万次IO——差距巨大。

### B+树 vs B树

B树的数据分布在所有节点上（非叶子节点也存数据），B+树只在叶子节点存数据。这意味着：

- B+树非叶子节点能存更多索引键（因为没有数据占地方），树更矮。
- B+树的叶子节点通过链表相连，范围查询极快。
- InnoDB选择了B+树——这不是偶然的。

🤔 **想多一点**：有人问"那为什么不用哈希表？哈希表 O(1) 不是更快吗？"哈希表确实等值查询快，但它不支持范围查询（`BETWEEN`、`>`）、不支持排序（`ORDER BY`）、不支持前缀匹配（`LIKE 'abc%'`）。而且哈希冲突、扩容等问题在持久化存储上不好解决。数据库索引要覆盖的场景比纯内存结构复杂得多。

## 55.3 聚簇索引 vs 非聚簇索引

这是理解MySQL索引的关键分水岭。

### 聚簇索引（Clustered Index）

InnoDB中，**主键索引 = 聚簇索引**。叶子节点上直接挂着**整行数据**。

```
B+树（按主键id组织）：
         [50 | 100]
        /          \
   [10|20|30]    [60|70|80]
   /  |  |  \    /  |  |  \
 [1] [10] [20] [30] [50] [60] [70] [80]  ← 叶子节点上挂着完整行数据
  |    |    |    |
(id=1 (id=10(id=20(id=30
 name   name  name  name
 email  email email email
 ...)   ...)  ...)  ...)
```

查询 `WHERE id = 20`：顺着B+树找到叶子节点20，**整行数据就在那里**，不需要再去别处找。

**一张表只能有一个聚簇索引**——因为数据物理上只能按一种顺序排列。这就好比一本字典只能按拼音排序，不可能同时按拼音和按部首排序。

### 非聚簇索引（二级索引 / Secondary Index）

你在 `email` 字段上建的索引就是非聚簇索引：

```
B+树（按email组织）：
       ['k@x.com' | 'm@x.com']
       /                      \
['a@x.com']               ['n@x.com'|'z@x.com']
    |
叶子节点：['a@x.com' → 主键id=30]
```

**叶子节点里存的不是完整行数据，而是主键id**。

查询 `WHERE email = 'a@x.com'` 的执行过程：

1. 在email索引的B+树中找到 'a@x.com' → 获取主键id=30。
2. **回表**：拿着id=30去聚簇索引的B+树中，再查一次，获取完整行数据。

这就是"回表"——一次查询要走两棵B+树。因此，通过非聚簇索引查询数据，比聚簇索引多一次IO。

**为什么非聚簇索引不直接存完整行数据？**
- 如果每个索引都存一份完整数据，存储开销爆炸（一张表有5个索引就存5份完整数据）。
- 数据更新时，所有索引副本都要同步更新——性能灾难。

用主键作为"指针"是最聪明的设计：数据只存一份（在聚簇索引里），其他索引只记主键。

### 一个形象比喻

- **聚簇索引**：新华字典按拼音编排正文。你要查"大"字——直接翻到da那页，读音、释义、例句全在上面。
- **非聚簇索引**：字典后面的偏旁部首检字表。查"煌"字——部首检字表告诉你"煌=页545"，然后翻到545页看完整释义。

## 55.4 联合索引与最左前缀

### 联合索引是什么

联合索引是在**多个列**上建立的索引：

```sql
CREATE INDEX idx_name_age ON users(name, age);
```

这棵B+树的排序规则：**先按name排序，name相同的再按age排序**。

```
联合索引树：
        ['wang', 25 | 'zhang', 30]
       /                            \
['li',20 | 'liu',22]          ['zhao',18 | 'zhou',40]
       |
叶子节点：
['li',20 → id=5]
['liu',22 → id=8]
['wang',25 → id=1]
['wang',25 → id=15]
['wang',30 → id=3]
['zhang',30 → id=2]
['zhao',18 → id=10]
['zhou',40 → id=7]
```

注意：name相同的行按age排序。

### 最左前缀原则

这是索引使用中**最重要的规则**，面试必考。

联合索引 `(name, age, city)` 相当于创建了三个索引：

1. `(name)` ——最左字段独立查询能用
2. `(name, age)` ——前两个字段联合查询能用
3. `(name, age, city)` ——全部三个字段查询能用

但**不能**直接使用 `(age)` 或 `(city)` 或 `(age, city)` 作为索引——因为B+树是先按name排序的，跳过name就没法用。

### 能用到索引的查询

```sql
SELECT * FROM users WHERE name = 'zhangsan';
```

```sql
SELECT * FROM users WHERE name = 'zhangsan' AND age = 25;
```

```sql
SELECT * FROM users WHERE name = 'zhangsan' AND age = 25 AND city = 'Beijing';
```

```sql
SELECT * FROM users WHERE name = 'zhangsan' AND city = 'Beijing';
```

**关键**：即使age在中间被跳过，**只要最左列（name）在WHERE中，索引就能用**。MySQL优化器会自动调整WHERE条件的顺序。但age之后的条件停用——上面的例子中city条件不走索引。

### 完全不能用到索引的查询

```sql
SELECT * FROM users WHERE age = 25;
```

最左列name不在WHERE中——索引报废，全表扫描。

```sql
SELECT * FROM users WHERE city = 'Beijing';
```

同理，跳过了name和age。

### 范围查询截断

```sql
SELECT * FROM users WHERE name = 'zhangsan' AND age > 20 AND city = 'Beijing';
```

- name：等值匹配，走索引 ✓
- age：范围匹配，走索引 ✓
- city：**不走索引** ✗（因为age用了范围条件，age之后的索引列失效）

这就是"范围查询截断"——范围条件（`>`、`<`、`BETWEEN`、`LIKE`）会中断后续字段对索引的使用。`=` 和 `IN` 不会截断。

🤔 **想多一点**：面试爱问"联合索引 (a, b, c)，WHERE a=1 AND c=3 能用索引吗？"答案是：能用，但只用到a列。b被跳过了，c没法续上。MySQL 5.6引入了**索引下推**（ICP），会尽量在索引中过滤c的条件，但不走完整的索引树搜索。

## 55.5 EXPLAIN详解

EXPLAIN是索引优化的罗盘——没有它，你就是在黑暗中乱撞。

```sql
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';
```

结果：

```
+----+-------------+-------+------+---------------+----------+---------+-------+------+-------+
| id | select_type | table | type | possible_keys | key      | key_len | ref   | rows | Extra |
+----+-------------+-------+------+---------------+----------+---------+-------+------+-------+
|  1 | SIMPLE      | users | ref  | uk_email      | uk_email | 402     | const |    1 | NULL  |
+----+-------------+-------+------+---------------+----------+---------+-------+------+-------+
```

我们来逐个字段解读：

### type（访问类型）—最重要的字段

从好到坏排序：

| type值 | 说明 | 比喻 |
|--------|------|------|
| system | 系统表，只有一行 | 查一张纸上的一个字 |
| const | 常数值：PRIMARY KEY或UNIQUE索引的等值查询 | 字典按拼音精确查 "da"，直接翻到 |
| eq_ref | JOIN时用PRIMARY KEY或UNIQUE关联，每个值只匹配一行 | 订单表通过唯一主键关联用户表 |
| ref | 非唯一索引等值查询，可能返回多行 | 查年龄=25的人，可能多个 |
| range | 索引范围扫描：`>`、`<`、`IN`、`BETWEEN` | 查年龄在20~30的人 |
| index | 全索引扫描（扫描整个索引树，不扫描数据行） | 虽然扫了整个索引，但比全表扫描快 |
| ALL | 全表扫描——**最差** | 没有索引，一页页翻 |

**目标**：至少达到 `range` 级别，最好 `ref` 以上。看到 `ALL` 就要警觉。

### possible_keys vs key

- `possible_keys`：MySQL**认为**可能用到的索引（不是一定会用）。
- `key`：MySQL**实际**使用的索引。

如果 `possible_keys` 有索引但 `key` 是NULL——说明优化器觉得用索引不如全表扫描快（当结果行数占比很大时，索引反而慢）。

### key_len

使用的索引字节数。可以推断出联合索引中实际使用了多少列。比如联合索引 `(name VARCHAR(50), age INT)` 的key_len在utf8mb4下为202（VARCHAR(50) × 4 = 200 + 2字节长度标记）。如果key_len正好是202，说明只用了name列。

### rows

MySQL**预估**需要扫描的行数。不是精确值，但能反映查询成本。rows越小越好。

### Extra

额外信息，重点关注：

| Extra值 | 说明 |
|---------|------|
| Using index | **覆盖索引**——查询列全在索引里，不需要回表。最优！ |
| Using where | 存储引擎返回行后，由服务层做WHERE过滤。 |
| Using index condition | 使用了索引下推（ICP）。 |
| Using temporary | 需要临时表（常见于GROUP BY和DISTINCT）——性能较差。 |
| Using filesort | 需要额外排序（ORDER BY字段没有索引）——性能差。 |
| Using join buffer | JOIN时缓冲不够，使用了额外缓冲区。 |

**红色警报**：`Using filesort` 和 `Using temporary` 是两个性能杀手。遇到就要想办法优化。

## 55.6 索引失效场景

建了索引不代表查询一定能用上。以下是索引失效的经典场景——面试连珠炮。

### 1. LIKE 以 % 开头

```sql
SELECT * FROM users WHERE name LIKE '%zhang';
```

索引**彻底失效**——因为B+树是按从左到右的顺序组织的，`%zhang` 不知道从哪开始匹配。

```sql
SELECT * FROM users WHERE name LIKE 'zhang%';
```

索引**有效**——因为可以从'zhang'开始定位。

**需要前后模糊匹配**的场景（如搜索框），用 Elasticsearch 或 MySQL 的全文索引（FULLTEXT），不要硬用LIKE。

### 2. 对索引字段使用函数

```sql
SELECT * FROM users WHERE YEAR(created_at) = 2024;
```

索引失效——因为函数会改变字段的值，索引存的是原始值。

```sql
SELECT * FROM users WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
```

索引有效——用范围替代函数。

### 3. 隐式类型转换

```sql
SELECT * FROM users WHERE phone = 13800001111;
```

如果 `phone` 字段是 `CHAR(11)`（字符串类型），而你传入的是整数 `13800001111`——MySQL会把phone的值**隐式转换**为数字再比较。这个转换触发了"对索引字段使用函数"，索引失效。

```sql
SELECT * FROM users WHERE phone = '13800001111';
```

字符串对字符串——索引有效。

### 4. OR 条件

```sql
SELECT * FROM users WHERE name = 'zhangsan' OR age = 25;
```

如果name有索引但age没有——**索引失效**，因为MySQL不知道用哪个索引（或认为全表扫描更划算）。

```sql
SELECT * FROM users WHERE name = 'zhangsan'
UNION
SELECT * FROM users WHERE age = 25;
```

拆成两个独立的索引查询再合并——各自走各自的索引。

### 5. 不等于（<> / !=）

```sql
SELECT * FROM users WHERE age <> 25;
```

大多数情况下索引失效——因为不等于意味着"除了25之外的所有值"，需要扫描大部分数据，优化器判断全表扫描更划算。但如果表里age=25的数据占绝大多数（比如99%都是25岁），不等于反而走索引更快。

### 6. IS NULL / IS NOT NULL

```sql
SELECT * FROM users WHERE phone IS NULL;
```

如果大部分行的phone都是NULL——走索引。如果只有少数是NULL——全表扫描可能更快。取决于数据分布。

### 7. NOT IN / NOT EXISTS

几乎必定不走索引——因为要检查"不在集合里"等于要把整个索引树遍历一遍。

### 快速自查法

查询慢的时候，问自己两个问题：
1. WHERE条件里的字段有没有索引？
2. 如果有，索引有没有被"破坏"（函数、类型转换、前导%、OR）？

用 EXPLAIN 验证——不看执行计划的优化都是瞎蒙。

## 55.7 覆盖索引

覆盖索引是索引优化的"银弹"——查询所需的所有列都在索引里，**不需要回表**。

前文的 `idx_name_age (name, age)`。执行：

```sql
SELECT name, age FROM users WHERE name = 'zhangsan';
```

1. 在联合索引中找到 'zhangsan'。
2. name和age都在索引的叶子节点上。
3. 直接返回，**不需要去聚簇索引回表**。

EXPLAIN 的 Extra 列会显示 `Using index`——高性能的标志。

### 一个实战优化

原查询（name上有索引，但还要取age，需要回表）：

```sql
SELECT name, age FROM users WHERE name = 'zhangsan';
```

建一个 `(name, age)` 的联合索引——秒变覆盖索引，查询性能提升一个数量级。

**覆盖索引的代价**：索引变大，写操作（INSERT/UPDATE/DELETE）多维护一棵更重的B+树。所以不要在索引里塞太多字段——权衡读写。

## 55.8 索引设计原则

经过以上知识，总结设计索引的黄金法则：

### 1. 为WHERE、JOIN、ORDER BY字段建索引

这三个地方是查询的性能瓶颈。

### 2. 选择性高的字段优先

**选择性 = 不重复值的数量 / 总行数**

- `gender`（男/女）：选择性 ≈ 0.5——非常低，建索引意义不大（还要回表）。
- `email`：选择性 ≈ 1.0——非常高，每个值基本唯一，建索引效果极好。
- `status`（0/1/2/3）：选择性0.01？要看分布。如果99%是status=1，查status=2时索引才有效。

### 3. 联合索引把选择性高的字段放前面

联合索引 `(status, email)`：status选择性低，第一个字段就不精准，索引效率低。

联合索引 `(email, status)`：email选择性高，精确命中后，status能迅速过滤——这才是正确的顺序。

但注意：**最左前缀原则优先于选择性**。如果几乎所有查询都带 `user_id`（选择性不一定最高但必查），那就把 `user_id` 放第一位。

### 4. 避免过多索引

一张表3~5个索引是合理的。10个以上要反思——插入、更新、删除每条语句都要维护10个索引，写入性能严重下降。

### 5. 定期分析索引使用情况

```sql
SELECT * FROM sys.schema_unused_indexes;
```

MySQL 8.0的 `sys` 库提供了未使用索引列表。定期清理没人用的索引，减小写入负担。

### 6. 字符串字段用前缀索引

```sql
CREATE INDEX idx_email_prefix ON users(email(10));
```

只索引email的前10个字符。适合特别长的字符串字段（如URL、长文本描述）。前缀越长选择性越高，但索引越大——取个平衡值。

| 聚簇索引 | 主键索引，叶子节点存**完整行数据**，一张表只有一个 |
| 非聚簇索引 | 二级索引，叶子节点存**主键值**，查数据需回表 |
| 联合索引 | 多列组合索引，遵循最左前缀原则 |
| 最左前缀 | `(a,b,c)` = `(a)` + `(a,b)` + `(a,b,c)`；跳过a则全部失效 |
| EXPLAIN | 关键看type(ALL最差)、key、rows(扫描行数)、Extra(filesort/temporary差) |
| 索引失效 | LIKE'%x'、函数包裹、隐式转换、OR、不等于、NOT IN |
| 覆盖索引 | 查询列全在索引里，Extra显示Using index，无需回表 |
| 设计原则 | WHERE/JOIN/ORDER BY加索引、选择性高优先、3~5个索引合理、定期清理 |
