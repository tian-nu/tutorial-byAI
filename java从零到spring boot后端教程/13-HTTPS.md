# 第13章 · HTTPS

> "HTTP就像寄明信片——路上任何一个邮递员都能翻开看内容。HTTPS就像把信装进了带锁的铁盒里——邮递员能送，但打不开。"

## 13.1 为什么需要HTTPS

HTTP是明文传输。任何在中间环节（路由器、交换机、WiFi热点）截获数据包的人，都能直接看到内容：

```
HTTP（明文）:
  用户 → 网络 → POST /login
                    username=zhangsan&password=123456  ← 任何人能看到！
                    → 服务端

HTTPS（加密）:
  用户 → 网络 → POST /login
                ████████████████████████████████████  ← 加密后的乱码
                    → 服务端（只有服务端能解密）
```

**HTTPS = HTTP + TLS（Transport Layer Security，传输层安全协议）**。TLS提供了三个保证：
- **加密**：数据内容只有通信双方能看懂
- **完整性**：数据在传输中没有被篡改
- **身份认证**：确认你正在和真正的服务器通信，而非假冒的

## 13.2 对称加密 vs 非对称加密

HTTPS混合使用了两种加密方式。

### 对称加密

加密和解密用**同一把密钥**：

```
明文 "hello" ──→ [加密: 密钥K] ──→ 密文 "ifmmp" ──→ [解密: 密钥K] ──→ 明文 "hello"
```

- 优点：快
- 缺点：怎么安全地把密钥K给对方？如果通过网络传密钥K，那K本身就可能被截获

### 非对称加密

有两把钥匙：**公钥**和**私钥**。

- 公钥加密的内容，只有私钥能解密
- 私钥加密的内容，只有公钥能解密
- 公钥可以公开，私钥必须保密

```
公钥加密 → 私钥解密：
  明文 "hello" ──→ [加密: 公钥] ──→ 密文 ──→ [解密: 私钥] ──→ 明文 "hello"

私钥加密 → 公钥解密（数字签名）：
  摘要 "abc" ──→ [加密: 私钥] ──→ 签名 ──→ [解密: 公钥] ──→ 摘要 "abc"
```

- 优点：公钥可以放心传播，解决了密钥分发问题
- 缺点：慢，比对称加密慢很多

## 13.3 HTTPS握手：两者混用

HTTPS握手的过程巧妙结合了两者：

```
客户端                                    服务端
  │                                         │
  │ ──① ClientHello ──────────────────────→  │
  │    "我支持的加密算法: AES, ChaCha20..."     │
  │                                         │
  │ ←──② ServerHello + 证书 ───────────────  │
  │    "我们用AES吧"                          │
  │    证书包含: 域名 + 公钥 + CA签名           │
  │                                         │
  │ ③ 验证证书:                               │
  │   - 域名匹配?                             │
  │   - CA签名有效?                           │
  │   - 证书没过期?                           │
  │                                         │
  │ ──④ 用公钥加密"会话密钥"──────────────→  │
  │    只有拥有私钥的服务端能解密               │
  │                                         │
  │ ⑤ 双方都得到了同一个"会话密钥"              │
  │                                         │
  │ ═══ 之后用对称加密(AES) + 会话密钥通信 ═══  │
  │    （快！）                               │
```

整个过程：
1. 用**非对称加密**安全地交换"会话密钥"（慢，但只做一次）
2. 之后用**对称加密** + 会话密钥通信（快）

> 🤔 想多一点：为什么需要CA（证书颁发机构）？因为中间人攻击——你访问 `https://bank.com`，但有个黑客劫持了你的网络，伪造了一个假服务器冒充bank.com。没有CA的证书签名验证，你无法区分真假。CA就像一个你信任的第三方——它在证书上盖了章："我验证过了，这确实是bank.com的公钥"。

## 13.4 HTTPS在Spring Boot中的配置

### 开发环境
开发时通常不用HTTPS——`http://localhost:8080` 就够了。

### 生产环境
生产环境用HTTPS有两种方式：

**方式一：Nginx代理（推荐）**

```
用户 [HTTPS] → Nginx（处理TLS）→ [HTTP] → Spring Boot:8080
```

Nginx负责加密解密，Spring Boot只管处理业务逻辑。这是最常见的方式——SSL证书在Nginx上配置，Spring Boot不需要关心。

**方式二：Spring Boot直接配置**

```yaml
# application.yml
server:
  port: 8443
  ssl:
    key-store: classpath:keystore.p12
    key-store-password: your_keystore_password
    key-store-type: PKCS12
    key-alias: tomcat
```

生产环境推荐方式一（Nginx代理），更安全、更灵活。

## 13.5 中间人攻击（MITM）

假设你在咖啡厅连了公共WiFi。黑客在同一个WiFi里：

```
你 ──→ 黑客（伪装成 bank.com）──→ 真正的 bank.com
          ↑
     黑客能看到你所有数据！
```

HTTPS如何防御？
- 黑客可以截获你的请求，但因为请求是用bank.com的公钥加密的，黑客没有私钥，无法解密
- 黑客可以伪造一个假证书，但他的假证书没有CA签名，浏览器会警告"此网站的安全证书不受信任"

> ⚠️ 如果浏览器弹出"证书不受信任"的警告，**不要点"继续访问"**——这可能就是中间人攻击。

## 本章小结

| 学了什么 | 一句话说明 |
|----------|-----------|
| HTTPS | HTTP + TLS，提供加密、完整性、身份认证 |
| 对称加密 | 同一把密钥加解密，快但密钥分发困难 |
| 非对称加密 | 公钥加密私钥解密，解决密钥分发但慢 |
| HTTPS握手 | 非对称加密交换会话密钥，之后对称加密通信 |
| 证书 | CA签名的域名+公钥凭据，防中间人攻击 |
| Spring Boot HTTPS | 推荐Nginx代理方式，开发环境不用 |
