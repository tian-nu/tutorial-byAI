# 第33章 · 包与模块

## 33.1 包是什么

### 包 = 一个目录 + 一个package声明

Go的代码以"包"为单位组织。一个包就是一个目录，目录下的所有 `.go` 文件的第一行必须是相同的 `package` 声明。

```
myapp/
├── main.go           → package main
├── user/
│   ├── user.go       → package user
│   └── validator.go  → package user
└── order/
    └── order.go      → package order
```

同一个包内的所有文件共享一切——你在这个文件定义的变量，那个文件直接用，不需要导入。

### 比喻：公司部门

你的代码是公司。公司分部门：人事部、财务部、研发部。每个部门（包）有自己的工作，部门之间有明确的接口（公开的函数/类型）。

研发部内部的事情（比如具体怎么实现的）不需要让财务部知道——这就是包的"封装"。

## 33.2 导出规则：大写公开，小写私有

这是Go最简单也最重要的规则：

```go
package user

var publicVar = "所有人都能看到我"
var privateVar = "只有user包内部能看到"

func PublicFunc() {}
func privateFunc() {}
```

| 首字母 | 可见性 | 含义 |
|--------|--------|------|
| 大写 | 公开（exported） | 其他包可以访问 |
| 小写 | 私有（unexported） | 只能在当前包内访问 |

没有 `public`、`private`、`protected` 关键字——大小写就是一切。

```go
package main

import "myapp/user"

func main() {
    user.PublicFunc()

    user.privateFunc()
}
```

### 不只是函数和变量

这条规则适用于**一切**顶层名字：类型、常量、结构体字段（是的，结构体字段也分公开和私有）：

```go
type User struct {
    Name    string
    age     int
    email   string
}

u := User{Name: "张三", age: 25}
```

`Name` 能被其他包设置，`age` 和 `email` 不能。

为什么字段也要控制可见性？因为你不希望其他包绕过你的方法直接修改内部状态。比如你有个 `SetAge()` 方法会校验年龄范围，如果 `age` 字段公开，别人就能 `u.age = -1` 绕过你的校验。

## 33.3 init函数：自动执行的初始化

每个Go文件可以有一个（或多个）特殊的 `init` 函数：

```go
package config

var AppName string

func init() {
    AppName = "我的应用"
    fmt.Println("config包初始化完成")
}
```

`init` 函数不能手动调用——它在包被导入时**自动执行**。

### init的执行顺序

1. 先初始化包的全局变量（按声明顺序）
2. 再执行 `init` 函数
3. 如果包A导入了包B，先执行B的init，再执行A的init
4. `main` 包的 `init` 最后执行
5. `main()` 函数最后最后执行

```
导入链: main → A → B

执行顺序:
1. B的全局变量初始化
2. B的init()
3. A的全局变量初始化
4. A的init()
5. main的全局变量初始化
6. main的init()
7. main()
```

### init的用途

- 初始化全局配置
- 注册驱动（如数据库驱动）
- 验证环境变量

```go
package main

import (
    "database/sql"
    _ "github.com/go-sql-driver/mysql"
)
```

注意那个 `_ "..."` ——这叫"匿名导入"。只执行该包的 `init` 函数，不直接使用包里的任何东西。MySQL驱动就是用 `init` 自动注册自己的。（数据库操作详见第50章起）

### init的注意事项

- 不要滥用 `init`。依赖init的执行顺序会让代码变得脆弱
- init里做太多事情会让包很"重"，不方便测试
- 如果你的包被很多地方导入，init的执行时机很难预料

## 33.4 Go Modules：依赖管理

在Go Modules（Go的依赖管理系统，通过go.mod文件管理项目依赖，详见附录I）之前，Go的依赖管理是一团乱麻（GOPATH模式）。Go 1.11引入了Go Modules，Go 1.16后成为默认模式。

### 比喻：购物清单

你写一个程序，用了别人写的库（比如Gin框架）。你需要记录：
- 用了哪些库 → `go.mod`
- 每个库的版本 → `go.mod`
- 每个库的校验和（防止被篡改）→ `go.sum`

这三项就像你装修房子的购物清单——水泥（Gin v1.9）、油漆（Redis v9.0）、地板（GORM v1.25）。同一个项目下次装修，照单买就行。

### go.mod

```go
module github.com/你的用户名/项目名

go 1.21

require (
    github.com/gin-gonic/gin v1.9.1
    github.com/go-sql-driver/mysql v1.7.1
)
```

`module` 行定义了你的**模块路径**——这是你项目的"全名"，别人导入你的包时用这个路径。

`require` 列出了所有直接依赖。

### go mod init：初始化

```bash
go mod init github.com/zhangsan/myapp
```

执行后生成 `go.mod` 文件。

### go mod tidy：整理依赖

```bash
go mod tidy
```

这个命令做了两件事：
1. 扫描代码，找到所有 `import`，把缺失的依赖加入 `go.mod`
2. 删除 `go.mod` 中已经不再使用的依赖

每次你添加或删除 `import` 后，记得跑一次 `go mod tidy`。

### go get：获取依赖

```bash
go get github.com/gin-gonic/gin
```

下载最新版本并更新 `go.mod`。也可以指定版本：

```bash
go get github.com/gin-gonic/gin@v1.9.1
go get github.com/gin-gonic/gin@latest
```

### go.sum：校验文件

`go.sum` 记录了每个依赖的哈希值，确保下载的代码和原作者发布的一致。**`go.sum` 应该提交到版本控制。**

### 间接依赖

```
你的项目 → Gin v1.9.1 → httprouter → ...
```

`go.mod` 里会列出直接依赖，`go.sum` 里会列出所有依赖（含间接依赖）的哈希值。

### vendor：离线模式

```bash
go mod vendor
```

把依赖的源码下载到项目的 `vendor/` 目录下。这在某些场景有用（比如公司内网CI环境不能访问外部网络）。

## 33.5 包的导入

### 普通导入

```go
import "fmt"
import "os"
```

或者分组：

```go
import (
    "fmt"
    "os"
)
```

### 别名导入

包名冲突或太长时用别名：

```go
import (
    myfmt "mylib/fmt"
    jsoniter "github.com/json-iterator/go"
)
```

### 点导入（不推荐）

```go
import (
    . "fmt"
)

func main() {
    Println("不需要写fmt.")
}
```

`.` 把包里的公开名字直接导入当前包——不推荐，因为看不出函数来自哪个包。

## 🤔 想多一点：包的组织哲学

Go社区有一个共识：**包名应该是名词，而不是动词**。

- ✅ `package user`、`package http`、`package json`
- ❌ `package userservice`、`package httpclient`

包名应该反映它"提供什么"，而不是"做什么"。

另外，包不应该太大。如果一个包包含了20个文件、30个公开函数、10个公开类型——它可能应该被拆分成多个更聚焦的包。

一个实用的判断标准：**你能用一句话描述这个包是干什么的吗？** 如果描述太长或太模糊，就该拆分了。
