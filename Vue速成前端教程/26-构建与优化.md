## 一、目标与完成效果

**一句话目标**：学会 `npm run build` 构建生产包，理解产物结构，配置环境变量分离开发/生产环境，并移除调试代码。

**完成后你将看到**：
- 一个 `dist/` 文件夹，里面是压缩混淆后的 HTML + JS + CSS + 图片。
- 用 `npm run preview` 可以在本地模拟生产环境预览。
- 开发环境请求 `http://localhost:8000/api`，生产环境自动切换到 `https://your-domain.com/api`。
- 生产包中所有 `console.log` 都被自动移除。

## 三、分步操作

### 步骤 1：`npm run build` — 生成生产包

**我在做什么？**
在项目根目录执行一条命令，Vite 会把你的 `.vue`、`.js`、`.css` 文件全部打包压缩，输出到 `dist/` 文件夹。这个文件夹里的东西就是最终部署到服务器上的"成品"。

```bash
npm run build
```

**输出示例**：
```
vite v5.x.x building for production...
✓ 42 modules transformed.
dist/index.html                   0.45 kB │ gzip: 0.30 kB
dist/assets/index-C8fG7H3k.css   2.31 kB │ gzip: 0.98 kB
dist/assets/index-DkL9mP2q.js   52.18 kB │ gzip: 18.74 kB
dist/assets/Home-CkL9mP2q.js    12.34 kB │ gzip:  4.56 kB
dist/assets/Login-BrT8xY1n.js    8.21 kB │ gzip:  3.12 kB
```

**重点关注**：
- `gzip` 列是实际传输到用户浏览器的大小（服务器通常开启 gzip 压缩）。
- 每个文件名后面都有一串随机字符（如 `C8fG7H3k`）——这是**内容哈希**，文件内容变了哈希就变，用于浏览器缓存策略。

🤔 **想多一点**：为什么文件名里有随机字符？假设你的 `index.js` 更新了，但文件名还是 `index.js`，浏览器可能用缓存里的旧版本，用户就看不到新功能了。加上哈希后，内容一变文件名就变，浏览器当成新文件重新下载。这叫"缓存破坏（Cache Busting）"。

### 步骤 2：看构建产物 — dist/ 里有什么

**我在做什么？**
打开 `dist/` 文件夹，理解每个文件/文件夹的作用。

**典型 dist/ 结构**：

```
dist/
├── index.html              # 入口 HTML（Vite 自动注入 <script> 和 <link>）
├── assets/
│   ├── index-C8fG7H3k.css  # 全局 CSS（压缩成一行）
│   ├── index-DkL9mP2q.js   # 入口 JS（压缩混淆）
│   ├── Home-CkL9mP2q.js    # 首页懒加载 JS（教程 25 的成果）
│   ├── Login-BrT8xY1n.js   # 登录页懒加载 JS
│   └── logo-aB3dE5fG.png   # 小图片（base64 内联到 JS 中）
└── favicon.ico              # 网站图标
```

**打开 `index.html` 看看**：

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8" />
  <link rel="icon" href="/favicon.ico" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>我的博客</title>
  <script type="module" crossorigin src="/assets/index-DkL9mP2q.js"></script>
  <link rel="stylesheet" crossorigin href="/assets/index-C8fG7H3k.css">
</head>
<body>
  <div id="app"></div>
</body>
</html>
```

注意 Vite 自动注入了 `<script>` 和 `<link>` 标签，你不需要手动写。

**打开一个 JS 文件看看**（不建议细读，但扫一眼即可）：

```js
// 原本你写的：
const posts = ref([])

// 构建后变成了：
const o=H([])
```

这就是**压缩混淆（Minification + Uglification）**：变量名缩短、空格删除、注释移除。人类很难读，但机器执行完全一样，体积只有原来的 1/3 ~ 1/5。

### 步骤 3：`npm run preview` — 本地预览生产包

**我在做什么？**
`npm run dev` 启动的是开发服务器（带热更新，代码不压缩）。`npm run preview` 启动的是**生产预览服务器**——它会读取 `dist/` 文件夹的内容，模拟真实部署后的行为。

```bash
npm run preview
```

输出：
```
  ➜  Local:   http://localhost:4173/
  ➜  Network: use --host to expose
```

打开 `http://localhost:4173`，你应该看到和 `npm run dev` 一样的页面，但：
- 代码是压缩的（右键 → 查看源代码，只有几行）
- 没有 Vue DevTools 的提示（生产环境默认关闭）
- 如果你在教程 25 做了懒加载，可以在 Network 中看到按需加载的 JS 文件

**开发 vs 生产对比**：

| 特性 | `npm run dev` | `npm run preview` |
|------|:---:|:---:|
| 代码压缩 | ❌ | ✅ |
| 热更新 (HMR) | ✅ | ❌ |
| Vue DevTools | ✅ | ❌（默认关闭） |
| Source Map | ✅ 完整 | ❌（默认关闭） |
| 加载速度 | 慢（大量未压缩代码） | 快 |

### 步骤 4：环境变量 — 开发/生产用不同 API 地址

**我在做什么？**
开发时，后端 API 跑在 `http://localhost:8000`。部署到生产环境后，API 地址变成 `https://api.your-blog.com`。你不可能每次部署前手动改代码里的 URL——环境变量就是来解决这个问题的。

**Vite 环境变量规则**：
- 只有 `VITE_` 开头的变量才会暴露给前端代码（安全考虑）。
- `.env` 文件：所有环境都加载。
- `.env.development` 文件：只在 `npm run dev` 时加载。
- `.env.production` 文件：只在 `npm run build` 时加载。

**创建三个文件**：

`.env`（通用配置）：
```
VITE_APP_TITLE=我的博客
```

`.env.development`（开发环境）：
```
VITE_API_BASE_URL=http://localhost:8000/api
```

`.env.production`（生产环境）：
```
VITE_API_BASE_URL=https://api.your-blog.com/api
```

**在代码中使用**：

```js
// 任何 .vue 或 .js 文件中
const API_BASE = import.meta.env.VITE_API_BASE_URL

async function fetchPosts() {
  const res = await fetch(`${API_BASE}/posts`)
  // ...
}
```

🤔 **想多一点**：为什么必须是 `VITE_` 前缀？这是 Vite 的安全机制——防止你无意中把敏感变量（如数据库密码、API 密钥）暴露给前端。如果你在 `.env` 里写了 `DB_PASSWORD=secret`，前端代码中 `import.meta.env.DB_PASSWORD` 是 `undefined`。只有 `VITE_DB_PASSWORD=secret` 才能被前端读到——但**你绝对不应该把数据库密码暴露给前端**，所以 Vite 用前缀提醒你："这个变量会被打包进前端代码，任何人都能在浏览器里看到，请三思。"

**注意**：`.env` 文件修改后需要重启开发服务器（`npm run dev`）才能生效。

**我做得对不对？**
在组件中 `console.log(import.meta.env.VITE_API_BASE_URL)`：
- `npm run dev` 时输出 `http://localhost:8000/api`
- `npm run build && npm run preview` 时输出 `https://api.your-blog.com/api`

### 步骤 5：移除 console.log — 生产环境不留调试痕迹

**我在做什么？**
开发时写的 `console.log` 在调试时很有用，但部署到生产环境后不应该留在代码里——它们会暴露内部逻辑，还可能泄露敏感数据。

**Vite 自动移除**：在 `vite.config.js` 中配置即可：

```js
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  plugins: [vue()],
  build: {
    minify: 'terser',         // 使用 terser 压缩（比默认的 esbuild 功能更强）
    terserOptions: {
      compress: {
        drop_console: true,   // 🎯 移除所有 console.*
        drop_debugger: true   // 🎯 移除所有 debugger
      }
    }
  }
})
```

**如果你只想移除 `console.log` 但保留 `console.error` 和 `console.warn`**：

```js
terserOptions: {
  compress: {
    pure_funcs: ['console.log'],  // 只移除 console.log
    drop_debugger: true
  }
}
```

**验证**：`npm run build` 后，在 `dist/assets/` 中搜索 `console.log`，应该搜不到。

❌ **错误示例**：手动在代码里删 `console.log`，部署完再手动加回来。
✅ **正确示例**：代码里保留 `console.log`，让构建工具自动移除。

## 七、术语附录

| 术语 | 解释 |
|------|------|
| **构建 (Build)** | 将源代码（`.vue`、`.js`、`.css`）编译、打包、压缩为可在浏览器中高效运行的生产文件的过程。 |
| **dist/ (Distribution)** | 构建产物的输出目录，里面的文件就是最终部署到服务器上的成品。 |
| **环境变量 (Environment Variable)** | 在应用外部定义的配置变量，用于在不同环境（开发/测试/生产）中切换行为，如 API 地址。Vite 中只有 `VITE_` 前缀的变量暴露给前端。 |
| **压缩混淆 (Minification / Uglification)** | 移除代码中的空格、注释，缩短变量名，使文件体积减小但功能不变。同时也让代码难以被人类阅读（反爬、反逆向的附加效果）。 |
| **Content Hash** | 基于文件内容计算的哈希值，放在文件名中。内容变了哈希就变，用于浏览器缓存管理。 |
| **Terser** | JavaScript 压缩工具，Vite 的可选压缩引擎。与默认的 esbuild 相比，Terser 支持更多的压缩选项（如 `drop_console`）。 |

## 八、已知坑点与禁止事项

- **禁止把 `.env` 文件提交到 Git**：`.env.production` 可能包含真实 API 地址，虽然不是密码，但仍是内部信息。Vite 项目模板默认已在 `.gitignore` 中排除 `.env` 文件，但 `.env.development` 和 `.env.production` 不一定。检查 `.gitignore`：
  ```
  .env
  .env.local
  .env.*.local
  ```
  如果你希望 `.env.development` 被提交（团队成员共享开发配置），确保里面没有敏感信息。
- **禁止在 `vite.config.js` 中直接 `import` 环境变量**：`vite.config.js` 运行在 Node.js 环境，用 `process.env` 而非 `import.meta.env`。
- **`dist/` 不要提交到 Git**：这是构建产物，应该由 CI/CD 在部署时生成。检查 `.gitignore` 是否包含 `dist/`。
- **`drop_console` 注意**：如果你依赖 `console.error` 做错误上报，用 `pure_funcs: ['console.log']` 代替 `drop_console: true`。
