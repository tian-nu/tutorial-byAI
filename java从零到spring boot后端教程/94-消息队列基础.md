# 94-消息队列基础

> 💡 你去银行柜台办业务。在 1990 年，你得排长队，柜员一个个处理——这就是同步调用。到了 2025 年，你取号、坐下、玩手机，轮到你了叫号——业务请求进了"排队系统"，柜员不慌不忙一个个处理。消息队列就是软件系统中的"叫号系统"——它让你把一个请求"扔进去"，然后继续干别的，等有空了再回来拿结果。

## 94.1 没有消息队列的世界——三个痛点

### 痛点一：紧耦合

```
用户注册 → 写数据库 → 发短信 → 发邮件 → 送优惠券 → 返回"注册成功"
```

问题：发短信服务挂了 → 整个注册流程失败。一个上游功能依赖一堆下游功能，下游任何一个出故障都会阻塞上游。

### 痛点二：流量冲击

秒杀活动开始，100 万人同时下单。数据库直接被打死。

### 痛点三：同步等待

用户提交一个"生成年度报表"的请求，后端要处理 30 秒。用户看着页面转圈——体验极差。

---

## 94.2 消息队列的三个核心能力

### ① 解耦

```
改造前：注册 → 写数据库 → 发短信 → 发邮件 → 送优惠券（全串行）

改造后：
注册 → 写数据库 → 发一条"用户已注册"消息到 MQ → 返回"注册成功"
                          ├── 短信服务（独立部署）消费这条消息 → 发短信
                          ├── 邮件服务（独立部署）消费这条消息 → 发邮件
                          └── 优惠券服务（独立部署）消费这条消息 → 送优惠券
```

上游只发消息，不管谁消费。下游独立部署、独立扩缩容、挂了也不影响上游。

### ② 削峰

```
秒杀请求峰值：100万 QPS
                     │
                     ▼
              ┌─────────────┐
              │   消息队列    │  ← 所有请求先进队列（100万 QPS 接收能力）
              └──────┬──────┘
                     │
                     ▼
              ┌─────────────┐
              │  订单处理服务  │  ← 按自己能力消费（比如 5000 QPS）
              └─────────────┘
```

消息队列像水库，把洪峰来的水存起来，按下游处理能力匀速放水。

### ③ 异步

```
同步：用户请求 → 开始处理 → 等待30秒 → 返回结果（用户一直等）

异步：用户请求 → 发消息到 MQ → 立刻返回"任务已提交，处理完成后通知你"
                   ↓
              消费者慢慢处理30秒 → 处理完通知用户（短信/WebSocket）
```

## 94.3 消息队列核心概念

```
┌──────────┐                  ┌──────────────┐                  ┌──────────┐
│ Producer │ ──发送消息──────→│    Broker     │──消费消息──────→│ Consumer │
│  (生产者) │                  │  (消息代理)    │                  │  (消费者) │
└──────────┘                  │              │                  └──────────┘
                              │ ┌──────────┐ │
                              │ │  Queue   │ │  ← 存储消息的地方
                              │ │ /Topic   │ │
                              │ └──────────┘ │
                              └──────────────┘
```

| 概念 | 说明 | 类比 |
|------|------|------|
| **Producer** | 发送消息的应用 | 快递寄件人 |
| **Consumer** | 接收并处理消息的应用 | 快递收件人 |
| **Broker** | 消息队列服务器本身 | 快递中转站 |
| **Queue** | 点对点消息队列，一条消息只被一个消费者消费 | 一个快递只给一个人 |
| **Topic** | 发布/订阅，一条消息被所有订阅者消费 | 公众号推送给所有粉丝 |
| **Message** | 传输的数据体 | 快递包裹 |

> 📊 上述解耦、削峰、异步三者的交互流程见 [java_94_mq_visual.html](java_94_mq_visual.html) 中的动画演示。

## 94.4 两种消息模型

### 点对点（Queue）

```
Producer → Queue → Consumer A   ← 只有 A 收到
                → Consumer B   ← 不会收到同一条消息
```

每个消息只被一个消费者处理。适用于**任务分发**——多个 Worker 抢任务，一条任务只被一个人做。

### 发布/订阅（Topic）

```
Producer → Topic → Consumer A   ← 收到
                → Consumer B   ← 也收到
                → Consumer C   ← 也收到
```

每个消息被所有订阅者收到。适用于**事件广播**——用户注册后，短信和邮件都要知道。

## 94.5 主流消息队列对比

| | RabbitMQ | Kafka | RocketMQ |
|------|------|------|------|
| **开发语言** | Erlang | Java/Scala | Java |
| **吞吐量** | 万级 | 百万级 | 十万级 |
| **延迟** | 微秒级 | 毫秒级 | 毫秒级 |
| **消息可靠性** | ⭐⭐⭐ 很高 | ⭐⭐⭐ 很高 | ⭐⭐⭐ 很高 |
| **核心模型** | Queue + Exchange | Topic + Partition | Topic + Queue |
| **适用场景** | 业务解耦、RPC 调用 | 大数据流、日志收集、事件溯源 | 电商交易、金融支付 |
| **运维复杂度** | 低 | 高 | 中 |
| **社区生态** | 成熟 | 极其成熟 | 阿里主导 |

### 一句话选型

- **公司业务系统、微服务间通信** → RabbitMQ（简单、可靠、够用）
- **大数据、日志、流处理、事件溯源** → Kafka（高吞吐、持久化）
- **阿里云环境、金融级事务消息** → RocketMQ

## 94.6 什么时候不该用消息队列？

| 场景 | 为什么不合适 |
|------|------|
| 简单的同步查询（查用户信息） | 用 REST API 直接调更快 |
| 对实时性要求极高（<1ms） | MQ 引入额外延迟 |
| 数据量极小、调用方极少 | MQ 的运维成本大于收益 |
| 必须同步返回结果的请求 | MQ 天然异步，不适合 |

> 🤔 想多一点：消息队列不是银弹。每引入一个中间件，就多一个可能挂掉的服务、多一个需要监控的指标、多一个面试要被问的组件。在不需要解耦/削峰/异步的场景下，不要为了"有技术含量"而上 MQ。

## 94.7 消息队列的可靠性保障

| 问题 | RabbitMQ 方案 | Kafka 方案 |
|------|------|------|
| 消息丢失（生产者侧） | Publisher Confirm | `acks=all` + `min.insync.replicas` |
| 消息丢失（Broker 侧） | 持久化队列 + 持久化消息 | 多副本（Replication Factor） |
| 消息丢失（消费者侧） | 手动 ACK | 手动提交 offset |
| 重复消费 | 消费者做幂等（唯一 ID 去重） | 消费者做幂等 |
| 消息积压 | 增加消费者、设置 TTL | 增加分区和消费者 |

## 94.9 自测题

**1. 消息队列解决的三个核心问题是什么？请各举一个具体的业务场景。**

**2. Queue 模型和 Topic 模型的区别是什么？每个用户注册后要发短信和发邮件，该用哪种？**

**3. 消息被消费者处理过程中，消费者崩溃了，消息会丢失吗？如果不会，消息队列是怎么保证不丢的？**

---

**答案提示**：1→解耦（注册后下游多个服务独立消费）、削峰（秒杀请求先入队列再匀速处理）、异步（耗时任务提交后立即返回，后台处理完成后通知）。2→Queue 是一对一（一条消息一个消费者），Topic 是一对多（一条消息所有订阅者）。发短信和发邮件用 Topic，两个消费者各自订阅。3→不会丢失。消费者必须显式发送 ACK（确认），消息队列才会删除该消息。如果消费者崩溃没发 ACK，Broker 会把消息重新投递给其他消费者。下一章——RabbitMQ 入门。
