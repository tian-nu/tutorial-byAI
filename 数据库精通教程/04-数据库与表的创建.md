## 我在做什么？

第 03 章你装好了 MySQL，但此刻你的数据库里空空如也——就像搬进了一栋新房子，墙是白的，房间是空的，什么都没有。

这一章，我们要开始"装修"：先**建一个数据库**（相当于给房子划出一个房间），然后在这个数据库里**建表**（相当于在房间里放家具），最后学会**修改**和**删除**表结构。

你写的所有 SQL 语句都属于 **DDL（Data Definition Language，数据定义语言）** *此术语见附录E*——SQL 的 5 种子语言中负责"定义数据结构"的那一种。

学完这一章，你会拥有一个名为 `my_shop` 的数据库，里面有两张表（`users` 和 `orders`），并且你能用命令查看它们的结构。

## 一、CREATE DATABASE：创建数据库

### 第一步：连接到 MySQL

打开终端，用 root 用户登录：

```bash
mysql -u root -p
```

输入密码后，你就进入了 MySQL 的命令行界面（`mysql>` 提示符）。

### 第二步：创建数据库

```sql
CREATE DATABASE my_shop
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;
```

这条命令做什么？

- `CREATE DATABASE my_shop`：创建一个名叫 `my_shop` 的数据库（数据库名用小写+下划线，我们后面会讲命名规范）。
- `CHARACTER SET utf8mb4`：指定字符集 *此术语见附录E* 为 `utf8mb4`。简单说：选 `utf8mb4` 是因为它能存 emoji 表情 😀，而老旧的 `utf8`（MySQL 里其实是 `utf8mb3`）不能。字符集原理不在这里展开，第 12 章会详细讲。
- `COLLATE utf8mb4_unicode_ci`：指定排序规则 *此术语见附录E*。`utf8mb4_unicode_ci` 是通用排序规则，`ci` 表示 case-insensitive（不区分大小写）。

> **想多一点**：为什么 MySQL 里的 `utf8` 其实不是真正的 UTF-8？因为 MySQL 早期为了性能，实现了一个最多 3 字节的"阉割版" UTF-8（`utf8mb3`），存不了 emoji 和部分生僻字。真正的 UTF-8 最多 4 字节，所以后来补了一个 `utf8mb4`。这个坑不知道坑了多少人——你存了一个 emoji，结果数据库报错 `Incorrect string value`。**记住：永远用 `utf8mb4`，不要用 `utf8`。**

### 第三步：切换到你的数据库

```sql
USE my_shop;
```

执行后，MySQL 提示符不变，但你的"当前工作数据库"已经切换为 `my_shop`。后续所有操作（建表、插入数据）都会在这个数据库里执行。

### 验证

```sql
SELECT DATABASE();
```

输出：

```
+------------+
| DATABASE() |
+------------+
| my_shop    |
+------------+
```

这说明你当前在 `my_shop` 数据库里。如果输出 `NULL`，说明你还没选数据库，执行 `USE my_shop;`。

## 二、CREATE TABLE：创建表

数据库有了，接下来建表。表是真正存数据的地方。

### 先建用户表（users）

```sql
CREATE TABLE users (
    id         INT          NOT NULL AUTO_INCREMENT COMMENT '用户ID，主键',
    username   VARCHAR(50)  NOT NULL              COMMENT '用户名',
    email      VARCHAR(100) NOT NULL              COMMENT '邮箱',
    password_hash VARCHAR(255) NOT NULL           COMMENT '密码哈希值',
    created_at DATETIME     DEFAULT CURRENT_TIMESTAMP COMMENT '注册时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_email (email)
) COMMENT='用户表';
```

### 逐行拆解

整条语句的结构是：

```
CREATE TABLE 表名 (
    列名  数据类型  约束  COMMENT '注释',
    列名  数据类型  约束  COMMENT '注释',
    ...
    表级约束
) COMMENT='表注释';
```

我们来逐行看：

#### 第 1 列：`id`

```sql
id INT NOT NULL AUTO_INCREMENT COMMENT '用户ID，主键',
```

- `INT`：整数类型，范围约 -21 亿到 21 亿。对于用户 ID 足够了。
- `NOT NULL`：这列不能为空。每个用户必须有一个 ID。
- `AUTO_INCREMENT`：自增——每插入一行，这列的值自动加 1。你不需要手动指定 ID 值。
- `COMMENT '用户ID，主键'`：给这列加注释，方便以后维护。

#### 第 2 列：`username`

```sql
username VARCHAR(50) NOT NULL COMMENT '用户名',
```

- `VARCHAR(50)`：可变长度字符串，最大 50 个字符。`VARCHAR` 是"能伸能缩"的类型——存 "alice" 只占 5 个字符的空间，不会浪费。
- 为什么是 50？这是一个经验值，大多数用户名不会超过 50 个字符。你可以根据实际需求调整。

#### 第 3 列：`email`

```sql
email VARCHAR(100) NOT NULL COMMENT '邮箱',
```

- `VARCHAR(100)`：邮箱可能比较长，给 100 个字符。

#### 第 4 列：`password_hash`

```sql
password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希值',
```

- `VARCHAR(255)`：存密码的**哈希值**（不是明文密码！）。哈希值通常比较长，255 是常见选择。
- ⚠️ **安全提醒**：永远不要把用户密码明文存储。这里存的是哈希值（如 bcrypt 的结果），不是原始密码。

#### 第 5 列：`created_at`

```sql
created_at DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '注册时间',
```

- `DATETIME`：日期 + 时间类型，如 `2025-06-05 14:30:00`。
- `DEFAULT CURRENT_TIMESTAMP`：如果你插入时不指定这列的值，数据库自动填当前时间。这是一个非常实用的技巧。

#### 表级约束

```sql
PRIMARY KEY (id),
UNIQUE KEY uk_email (email)
```

- `PRIMARY KEY (id)`：指定 `id` 为主键。主键自动获得两个特性：NOT NULL + UNIQUE。
- `UNIQUE KEY uk_email (email)`：邮箱列加唯一约束，防止两个用户注册同一个邮箱。`uk_email` 是这个约束的名字，`uk` 前缀表示 unique key。

> **想多一点**：为什么存 `password_hash` 而不是 `password`？因为如果数据库被攻破（或被内部人员泄露），攻击者拿到的是哈希值，无法直接登录。哈希是单向的——你可以从密码算出哈希，但无法从哈希反推出密码。这是安全领域的常识，但很多初学者建表时直接写 `password VARCHAR(100)`，这是灾难性的错误。

### 再建订单表（orders）

```sql
CREATE TABLE orders (
    id          INT           NOT NULL AUTO_INCREMENT COMMENT '订单ID，主键',
    user_id     INT           NOT NULL              COMMENT '下单用户ID，外键关联 users.id',
    product     VARCHAR(200)  NOT NULL              COMMENT '商品名称',
    amount      DECIMAL(10,2) NOT NULL              COMMENT '订单金额',
    created_at  DATETIME      DEFAULT CURRENT_TIMESTAMP COMMENT '下单时间',
    PRIMARY KEY (id),
    KEY idx_user_id (user_id)
) COMMENT='订单表';
```

#### 新增数据类型

`amount DECIMAL(10,2)`：**精确小数**，专门用来存金额。

- `DECIMAL(M,D)`：M 是总位数（整数部分 + 小数部分），D 是小数位数。
- `DECIMAL(10,2)` 表示最多 8 位整数 + 2 位小数，如 `12345678.99`。
- **为什么不用 FLOAT？** 因为浮点数不精确！在计算机里，`0.1 + 0.2` 用 FLOAT 计算的结果是 `0.30000000000000004`，不是 `0.3`。这种误差在金额计算中是不可接受的——你肯定不想给用户多算一分钱或少算一分钱。DECIMAL 是定点数，精度完全准确。

#### 新增索引

```sql
KEY idx_user_id (user_id)
```

给 `user_id` 列建一个索引。因为以后会经常"查某个用户的所有订单"（`WHERE user_id = ?`），建索引能加速这种查询。`idx_user_id` 是索引名，`idx` 前缀表示 index。

> 注意：这里没有加 `FOREIGN KEY` 外键约束。外键约束本身是可选的高级特性，我们将在第 16 章详细讨论。现在先记住：`orders.user_id` 存的是 `users.id` 的值，这是一种**逻辑上的外键关系**。

## 三、数据类型初识：每种类型的选型直觉

上面我们用了 6 种数据类型。以下是每种类型的"选型直觉"——当你面对一个字段时，怎么选类型：

| 数据类型 | 选型直觉 | 示例 |
|---------|---------|------|
| **INT** | 存整数。范围约 -21 亿~21 亿。如果不够（如全国人口），用 BIGINT。 | `id`, `age`, `count` |
| **VARCHAR(n)** | 可变长度字符串。n 是最大字符数。适合"有上限但长度不固定"的文本。 | `username`, `email`, `phone` |
| **TEXT** | 长文本。不用指定长度。适合存文章、简介等。 | `bio`, `content` |
| **DECIMAL(M,D)** | 精确小数，金额专用。M 是总位数，D 是小数位数。 | `price`, `amount`, `balance` |
| **DATE** | 只有日期，没有时间。如 `2025-06-05`。 | `birthday` |
| **DATETIME** | 日期 + 时间。如 `2025-06-05 14:30:00`。 | `created_at`, `updated_at` |
| **TINYINT(1)** | 存 0 或 1，适合布尔值。实际上存的是 1 字节整数，但习惯上 `TINYINT(1)` 表示"是/否"。 | `is_active`, `is_deleted` |

> 第 12 章会深入讲解所有数据类型，包括每个类型的存储大小、边界值、性能差异。现在你只需要记住上面的"选型直觉"就够了。

## 四、查看表结构

表建好了，怎么确认它长什么样？有两种方法：

### 方法 1：DESC（简洁版）

```sql
DESC users;
```

输出：

```
+---------------+--------------+------+-----+-------------------+-------------------+
| Field         | Type         | Null | Key | Default           | Extra             |
+---------------+--------------+------+-----+-------------------+-------------------+
| id            | int          | NO   | PRI | NULL              | auto_increment    |
| username      | varchar(50)  | NO   |     | NULL              |                   |
| email         | varchar(100) | NO   | UNI | NULL              |                   |
| password_hash | varchar(255) | NO   |     | NULL              |                   |
| created_at    | datetime     | YES  |     | CURRENT_TIMESTAMP | DEFAULT_GENERATED |
+---------------+--------------+------+-----+-------------------+-------------------+
```

这张表的含义：
- `Field`：列名
- `Type`：数据类型
- `Null`：是否允许为空（`NO` = 不允许，`YES` = 允许）
- `Key`：`PRI` = 主键，`UNI` = 唯一约束，空 = 普通列
- `Default`：默认值
- `Extra`：额外信息（`auto_increment` = 自增）

### 方法 2：SHOW CREATE TABLE（完整版）

```sql
SHOW CREATE TABLE users\G
```

注意末尾是 `\G` 而不是 `;`。`\G` 让输出按列纵向排列，更易读。

输出：

```
*************************** 1. row ***************************
       Table: users
Create Table: CREATE TABLE `users` (
  `id` int NOT NULL AUTO_INCREMENT COMMENT '用户ID，主键',
  `username` varchar(50) NOT NULL COMMENT '用户名',
  `email` varchar(100) NOT NULL COMMENT '邮箱',
  `password_hash` varchar(255) NOT NULL COMMENT '密码哈希值',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP COMMENT '注册时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_email` (`email`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户表'
```

这个方法会输出完整的建表语句——包括了你在 CREATE TABLE 时没有显式指定的默认值（如 ENGINE=InnoDB）。这是"如果我想重建这张表，该执行什么 SQL"的精确答案。

### 验证

```sql
SHOW TABLES;
```

你应该看到：

```
+-------------------+
| Tables_in_my_shop |
+-------------------+
| orders            |
| users             |
+-------------------+
```

## 五、ALTER TABLE：修改表结构

表建好了，但需求变了——很正常。`ALTER TABLE` *此术语见附录E* 让你修改已有的表结构，而不需要删表重建。

### 添加列

```sql
ALTER TABLE users ADD COLUMN phone VARCHAR(20) COMMENT '手机号';
```

### 修改列（改数据类型或约束）

```sql
ALTER TABLE users MODIFY COLUMN phone VARCHAR(30) COMMENT '手机号（扩展长度）';
```

### 删除列

```sql
ALTER TABLE users DROP COLUMN phone;
```

> ⚠️ **警告**：删除列会**永久丢失**该列的所有数据。执行前请确认。

### 添加索引

```sql
ALTER TABLE users ADD INDEX idx_username (username);
```

### 验证

执行 `DESC users;` 查看修改后的表结构。

## 六、DROP TABLE / DROP DATABASE：危险操作警告

`DROP` *此术语见附录E* 用来删除数据库对象。它**不可逆**——删了就是删了，不像操作系统的"回收站"可以恢复。

### 删除表

```sql
DROP TABLE users;
```

执行后，`users` 表及其所有数据**永久消失**。

### 删除数据库

```sql
DROP DATABASE my_shop;
```

执行后，`my_shop` 数据库及其所有表、所有数据**永久消失**。

> ⚠️ **危险操作确认框**：
> 
> 在执行任何 DROP 命令前，请三连问自己：
> 1. 我真的要删这个吗？
> 2. 备份了吗？（`mysqldump` 先备份）
> 3. 现在在哪个数据库？（`SELECT DATABASE();` 确认）
>
> 特别提醒：**不要在生产环境（线上服务器）执行 DROP 命令**，除非你完全清楚后果。第 25 章会讲备份与恢复策略。

## 七、命名规范

好的命名让数据库"自文档化"——别人看表名就知道里面存什么。以下是推荐的命名规范：

| 规则 | 示例 | 说明 |
|------|------|------|
| **小写** | `users` 而非 `Users` 或 `USERS` | 跨操作系统兼容（Windows 不区分大小写，Linux 区分） |
| **下划线分隔** | `order_items` 而非 `orderItems` 或 `OrderItems` | 驼峰命名在 SQL 里不推荐 |
| **表名用复数** | `users`、`orders`、`products` | 本教程统一使用复数（MySQL 社区习惯） |
| **主键列名** | `id` | 简单明了 |
| **外键列名** | `user_id`、`order_id` | 表名单数 + `_id` |
| **时间列名** | `created_at`、`updated_at` | `_at` 后缀表示时间点 |
| **索引名** | `idx_列名` | 如 `idx_user_id` |
| **唯一约束名** | `uk_列名` | 如 `uk_email` |

> 为什么小写？因为 MySQL 在 Linux 上区分表名大小写，在 Windows 上不区分。如果你在 Windows 开发时写了 `Users`，部署到 Linux 后可能有奇怪的问题。**统一小写，一劳永逸。**

## 我做得对不对？

### 验证方法

执行以下命令，逐一确认：

```bash
# 1. 登录 MySQL
mysql -u root -p

# 2. 查看所有数据库，确认 my_shop 存在
SHOW DATABASES;

# 3. 切换数据库
USE my_shop;

# 4. 查看所有表，确认 users 和 orders 存在
SHOW TABLES;

# 5. 查看 users 表结构，确认所有列和约束正确
DESC users;

# 6. 查看 orders 表结构，确认 amount 列是 DECIMAL(10,2)
DESC orders;
```

如果你的输出与上面一致，恭喜——你成功创建了人生中第一个数据库和第一张表！

## 不对怎么办？

### 常见错误 1：忘记设置字符集，中文乱码

❌ 错误做法：

```sql
CREATE DATABASE my_shop;  -- 没有指定字符集，使用默认值（可能是 latin1）
```

然后插入中文数据，查出来全是 `???`。

✅ 正确做法：

```sql
CREATE DATABASE my_shop
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;
```

如果已经建了库但没设字符集，补救：

```sql
ALTER DATABASE my_shop CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

但这只影响**之后**创建的表。已有的表需要逐表修改：

```sql
ALTER TABLE users CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

### 常见错误 2：VARCHAR 不给长度

❌ 错误做法：

```sql
CREATE TABLE users (
    username VARCHAR NOT NULL  -- 缺少长度！
);
```

MySQL 报错：`ERROR 1064 (42000): You have an error in your SQL syntax ...`

✅ 正确做法：

```sql
username VARCHAR(50) NOT NULL
```

### 常见错误 3：数据库已存在，重复创建

❌ 错误做法：

```sql
CREATE DATABASE my_shop;  -- 如果 my_shop 已经存在，会报错
```

MySQL 报错：`ERROR 1007 (HY000): Can't create database 'my_shop'; database exists`

✅ 正确做法（加 IF NOT EXISTS）：

```sql
CREATE DATABASE IF NOT EXISTS my_shop
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;
```

同样，创建表也可以用：

```sql
CREATE TABLE IF NOT EXISTS users (...);
```

### 常见错误 4：直接 DROP 不备份

❌ 错误做法：

```sql
-- 脑子一热就执行了
DROP TABLE users;
```

✅ 正确做法：在 DROP 之前先备份！

```bash
# 在终端执行（不是在 MySQL 里）
mysqldump -u root -p my_shop users > users_backup.sql
```

万一删错了，可以恢复：

```bash
mysql -u root -p my_shop < users_backup.sql
```

### 常见错误 5：表名或列名用了关键字

❌ 错误做法：

```sql
CREATE TABLE order (  -- "order" 是 SQL 关键字（ORDER BY）！
    id INT
);
```

MySQL 报错或者产生奇怪行为。

✅ 正确做法：避免用关键字做表名，或使用反引号包裹：

```sql
CREATE TABLE `order` (  -- 反引号强制使用关键字
    id INT
);
```

但更好的做法是换个名字，如 `orders`。

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 创建数据库 | `CREATE DATABASE ... CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;` | 永远用 utf8mb4，不要用 utf8；`IF NOT EXISTS` 防重复 |
| 切换数据库 | `USE my_shop;` | `SELECT DATABASE();` 确认当前在哪个库 |
| 创建表 | `CREATE TABLE 表名 (列定义...);` | 每列指定数据类型、约束、注释；表级约束（PRIMARY KEY、UNIQUE）单独写 |
| 数据类型（7 种） | INT、VARCHAR(n)、TEXT、DECIMAL(M,D)、DATE、DATETIME、TINYINT(1) | 金额用 DECIMAL 不用 FLOAT；VARCHAR 必须给长度 |
| 主键与自增 | `PRIMARY KEY (id)` + `AUTO_INCREMENT` | 主键自动 NOT NULL + UNIQUE；自增意味着插入时不用手动指定 ID |
| 唯一约束 | `UNIQUE KEY uk_email (email)` | 防止重复数据，如两个用户同一邮箱 |
| 默认值 | `DEFAULT CURRENT_TIMESTAMP` | 插入时不指定该列，数据库自动填值 |
| 查看表结构 | `DESC 表名;` / `SHOW CREATE TABLE 表名\G` | DESC 简洁，SHOW CREATE TABLE 完整（含建表 SQL） |
| 修改表结构 | `ALTER TABLE ... ADD/MODIFY/DROP COLUMN` | 修改列会丢失数据（如缩短 VARCHAR 长度），谨慎操作 |
| 删除表/数据库 | `DROP TABLE/DATABASE` | **不可逆！** 先备份再执行；生产环境禁用 |
| 命名规范 | 小写+下划线，表名复数，`_id` 外键，`_at` 时间，`idx_` 索引 | 避免关键字；统一风格让数据库自文档化 |
