## 三、分步操作

### 步骤 1：问题引入——为什么需要"参数"？

到目前为止，你写的接口都很"死板"——不管谁来访问，返回的都是同一句话。就像一家餐厅不管客人点什么菜，永远只端上一盘宫保鸡丁。

**真实世界里，后端需要根据请求的不同返回不同数据**。比如：
- 用户想看第 1 篇文章 → 返回文章 1 的内容
- 用户想看第 2 篇文章 → 返回文章 2 的内容
- 用户想搜"Python"相关的文章 → 返回包含"Python"的文章列表
- 用户想看第 3 页、每页 10 条 → 返回第 3 页的 10 条数据

**这些"不同"靠什么来区分？就是参数。**

FastAPI 提供了两种传参方式，分别对应两种不同的"我要什么"：

| 方式 | URL 长什么样 | 比喻 | 用来干什么 |
|------|------------|------|-----------|
| **路径参数** | `/blog/1`、`/blog/2` | 图书馆的书架编号——你要找第几层的第几本书 | 指定"哪一个"——定位一个具体资源 |
| **查询参数** | `/blog?page=2&size=10` | 筛选条件——你要什么分类、第几页、排序方式 | 指定"怎么筛选"——过滤、排序、分页 |

下面我们逐一击破。

### 步骤 2：路径参数——让 URL 里出现"变量"

#### 比喻：图书馆的书架编号

你去图书馆找一本书。管理员告诉你："这本书在 `文学区 → 第 3 排 → 第 12 本`"。

这个编号 `3-12` 就是**路径参数**——它嵌在路径里，唯一标识你要找的东西。换一个编号，就是另一本书。

在 URL 里，路径参数长这样：

```
/blog/1    →  第 1 篇文章
/blog/2    →  第 2 篇文章
/blog/99   →  第 99 篇文章
```

`1`、`2`、`99` 是变化的，但 `/blog/` 这个前缀是固定的。FastAPI 用 `{ }` 花括号来标记"这里是一个变量"。

#### 动手写代码

打开 `blog_backend/main.py`，在现有代码后面追加以下内容：

```python
# === 新增：路径参数示例 ===
@app.get("/blog/{blog_id}")
def get_blog(blog_id: int):
    return {"blog_id": blog_id, "title": f"文章 #{blog_id}", "content": f"这是第 {blog_id} 篇文章的内容"}
```

保存文件。因为开启了 `--reload`，服务器会自动重启。

#### 逐行理解

```python
@app.get("/blog/{blog_id}")
```

`{blog_id}` 就是路径参数。花括号里的名字（`blog_id`）**必须**和下面函数参数的名字一模一样。FastAPI 就是靠名字来匹配的——"路径里有个叫 `blog_id` 的变量，函数参数里也有个叫 `blog_id` 的变量，好，我把路径里的值传给你。"

```python
def get_blog(blog_id: int):
```

`blog_id: int` 里的 `: int` 是 Python 的**类型注解**（type hint）。它告诉 FastAPI 两件事：
1. **我期望 `blog_id` 是一个整数**
2. **如果不是整数，自动拒绝请求，返回 422 错误**

> 这就是 FastAPI 最强大的特性之一：你只需要在参数后面加个 `: int`，FastAPI 就帮你做完类型检查、类型转换、错误提示。其他框架可能要写好几行代码才能做到同样的事。

#### 验证

打开浏览器，依次访问：

```
http://127.0.0.1:8000/blog/1
http://127.0.0.1:8000/blog/2
http://127.0.0.1:8000/blog/99
```

你会分别看到：

```json
{"blog_id":1,"title":"文章 #1","content":"这是第 1 篇文章的内容"}
{"blog_id":2,"title":"文章 #2","content":"这是第 2 篇文章的内容"}
{"blog_id":99,"title":"文章 #99","content":"这是第 99 篇文章的内容"}
```

**同一个接口，不同的参数，返回不同的数据。** 这就是路径参数的价值。

### 步骤 3：FastAPI 自动类型转换——传错类型会怎样？

步骤 2 里我们写了 `blog_id: int`。现在来测试：**如果传的不是整数会怎样？**

在浏览器访问：

```
http://127.0.0.1:8000/blog/abc
```

你会看到 FastAPI 返回了一个错误页面（或 JSON 错误信息），核心内容是：

```json
{
  "detail": [
    {
      "type": "int_parsing",
      "loc": ["path", "blog_id"],
      "msg": "Input should be a valid integer, unable to parse string as an integer",
      "input": "abc"
    }
  ]
}
```

HTTP 状态码是 **422 Unprocessable Entity**（无法处理的实体）。

**翻译一下**：FastAPI 说："你说 `blog_id` 应该是 `int`，但用户传了个 `abc`（字符串），我转不了，所以我拒绝了这个请求。"

#### 为什么是 422 而不是 400？

- **400 Bad Request**：请求格式本身有问题（比如 JSON 格式错误）。
- **422 Unprocessable Entity**：请求格式没问题，但**数据内容不合法**（比如你要求整数，对方给了字符串）。

FastAPI 用了更精确的 422，这样调用方（前端、App）可以区分"我格式写错了"和"我值写错了"。

#### 这是好事还是坏事？

**是好事，而且是天大的好事。** 想象一下如果没有这个自动检查：

- 你的代码里 `blog_id` 会被当成字符串 `"abc"` 传进来
- 你后面可能会用 `blog_id` 去数据库里查 `WHERE id = 'abc'`
- 数据库报错，你的程序崩溃，返回 500
- 你还要手动写一堆 `if not isinstance(blog_id, int)` 的检查代码

**FastAPI 帮你把脏活累活全干了。** 你只需要声明类型，它负责挡掉所有类型不对的请求。这种机制叫做**类型驱动开发**——你定义类型，框架自动验证。

> 📊 这个知识点很重要，后面学请求体（POST）时还会用到。到时候你会看到，FastAPI 甚至能自动验证"这个字段必填""那个字段最多 100 个字符"——全部靠类型注解。

---

### 🤔 想多一点：为什么 FastAPI 要这么强调类型？

Python 是一门**动态类型**语言——你写 `x = 1` 和 `x = "hello"`，Python 都不拦你。这在写小脚本时很方便，但在写后端时是个巨大的隐患。

**没有类型检查的后端就像没有安检的机场**——任何人都可以带任何东西上飞机（传任何数据进你的函数）。你只能在函数内部一个一个检查："你是整数吗？你是字符串吗？你长度合规吗？" 这种代码叫"防御性编程"，写起来又臭又长。

FastAPI 的思路是：**把安检口放在机场大门（框架层），而不是每个登机口（每个函数内部）。** 你只需要在函数参数上声明类型，FastAPI 在请求到达你的函数之前就把类型检查做了。不合规的请求根本进不了你的代码。

**这就是为什么 FastAPI 是 Python 最快的 Web 框架之一**——不是因为 Python 快，而是因为它把大量验证工作交给了类型系统，减少了运行时检查。

### 步骤 4：查询参数——URL 问号后面的"筛选条件"

#### 比喻：筛选条件

路径参数是"你要找第几本书"，是**定位**。但如果你说："我要找所有 Python 相关的书，按出版时间排序，只看第 2 页"，这就不是定位某一本书了，而是**筛选 + 排序 + 分页**。

这种"附加条件"在 URL 里长这样：

```
/blogs?page=2&size=10&sort=latest
```

`?` 之后的部分就是**查询参数**（query parameters）。多个参数之间用 `&` 连接。拆开看：

| 部分 | 含义 |
|------|------|
| `?` | "下面开始是查询参数" |
| `page=2` | 第 2 页 |
| `&` | "还有一个参数" |
| `size=10` | 每页 10 条 |
| `&` | "还有一个参数" |
| `sort=latest` | 按最新排序 |

#### 动手写代码

在 `main.py` 末尾追加：

```python
# === 新增：查询参数示例 ===
@app.get("/blogs")
def list_blogs(page: int = 1, size: int = 10):
    return {
        "page": page,
        "size": size,
        "message": f"返回第 {page} 页，每页 {size} 条数据"
    }
```

保存文件。

#### 逐行理解

```python
@app.get("/blogs")
```

注意路径是 `/blogs`，里面**没有**花括号。查询参数不在路径里，在 `?` 后面，所以路径本身不需要任何特殊标记。

```python
def list_blogs(page: int = 1, size: int = 10):
```

`page: int = 1` 的意思是：
- `page` 是一个查询参数
- 类型是 `int`（FastAPI 会自动把 URL 里的字符串 `"2"` 转成整数 `2`）
- `= 1` 是**默认值**——如果用户不传 `page`，就用 `1`

**关键规则**：FastAPI 判断一个参数是"路径参数"还是"查询参数"的方法很简单——**看这个参数有没有出现在路径的花括号里**。出现在路径里的就是路径参数，没出现在路径里的就是查询参数。

#### 验证

用浏览器访问以下 URL，观察返回结果：

**不传参数（用默认值）**：
```
http://127.0.0.1:8000/blogs
```
返回：
```json
{"page":1,"size":10,"message":"返回第 1 页，每页 10 条数据"}
```

**传一个参数**：
```
http://127.0.0.1:8000/blogs?page=3
```
返回：
```json
{"page":3,"size":10,"message":"返回第 3 页，每页 10 条数据"}
```
注意 `size` 没传，用了默认值 `10`。

**传两个参数**：
```
http://127.0.0.1:8000/blogs?page=2&size=5
```
返回：
```json
{"page":2,"size":5,"message":"返回第 2 页，每页 5 条数据"}
```

**传错类型**：
```
http://127.0.0.1:8000/blogs?page=abc
```
返回 422 错误——和路径参数一样，FastAPI 自动做了类型检查。

### 步骤 5：可选查询参数——允许用户"不传"

有时候，一个查询参数不是必须的。比如搜索功能——用户可能搜关键词，也可能不搜（直接看全部）。

#### 动手写代码

在 `main.py` 末尾追加：

```python
# === 新增：可选查询参数示例 ===
@app.get("/search")
def search_blogs(keyword: str | None = None):
    if keyword:
        return {"keyword": keyword, "message": f"搜索关键词：{keyword}"}
    else:
        return {"keyword": None, "message": "没有提供关键词，显示全部内容"}
```

保存文件。

#### 逐行理解

```python
keyword: str | None = None
```

这行有三个信息：

| 部分 | 含义 |
|------|------|
| `str` | 如果传了，必须是字符串 |
| `\| None` | 如果不传，可以是 `None` |
| `= None` | 默认值是 `None` |

合起来就是：**"keyword 是一个可选的字符串参数。传了就按字符串处理，不传就是 `None`。"**

> **`str | None` 语法**：这是 Python 3.10+ 的新语法。如果你用的是 Python 3.9 或更早版本，需要写成 `Optional[str]`（需要 `from typing import Optional`）。本教程假设 Python 3.10+，直接用 `str | None`。

#### 验证

**不传 keyword**：
```
http://127.0.0.1:8000/search
```
返回：
```json
{"keyword":null,"message":"没有提供关键词，显示全部内容"}
```

**传 keyword**：
```
http://127.0.0.1:8000/search?keyword=python
```
返回：
```json
{"keyword":"python","message":"搜索关键词：python"}
```

**传中文 keyword**（URL 会自动编码，浏览器会帮你处理）：
```
http://127.0.0.1:8000/search?keyword=后端开发
```
返回：
```json
{"keyword":"后端开发","message":"搜索关键词：后端开发"}
```

### 步骤 6：路径参数 vs 查询参数——什么时候用哪个？

学到现在，你可能有一个疑问：**同样都是传参数，路径参数和查询参数到底有什么区别？我什么时候该用哪个？**

这个问题问得好。答案是：**看你要定位的是"一个东西"还是"一堆东西的筛选条件"。**

| 对比维度 | 路径参数 | 查询参数 |
|----------|----------|----------|
| URL 位置 | `/blog/{id}` 路径里 | `/blog?page=1` 问号后 |
| 是否必填 | 必填（路径里写了就必须传） | 可选（有默认值就可以不传） |
| 用途 | 指定**哪一个**资源 | 筛选、排序、分页 |
| 比喻 | 图书馆的"第 3 排第 12 本" | 图书馆的"Python 分类、按出版时间排序、第 2 页" |
| 典型场景 | `/user/5`（第 5 号用户）、`/article/99`（第 99 篇文章） | `/articles?page=2&size=10&sort=latest` |
| 是否影响"资源定位" | 是——换了参数就是另一个资源 | 否——换了参数只是换了同一批资源的展示方式 |
| 搜索引擎如何看待 | 视为不同页面 | 视为同一页面的不同变体 |

**一个简单的判断口诀**：

> 如果你问的是"哪一个？" → 用路径参数。
> 如果你问的是"怎么筛选？" → 用查询参数。

**具体例子**：

```python
# ✅ 正确：路径参数指定"哪篇文章"，查询参数指定"怎么筛选评论"
@app.get("/blog/{blog_id}/comments")            # blog_id 是路径参数
def get_comments(blog_id: int, page: int = 1):  # page 是查询参数
    pass

# ✅ 正确：路径参数指定"哪个用户"
@app.get("/user/{user_id}")
def get_user(user_id: int):
    pass

# ✅ 正确：查询参数指定"怎么排列文章列表"
@app.get("/articles")
def list_articles(category: str | None = None, sort: str = "latest"):
    pass
```

### 步骤 7：路径参数和查询参数可以一起用

一个接口可以**同时**有路径参数和查询参数。这在真实项目中非常常见。

#### 动手写代码

在 `main.py` 末尾追加：

```python
# === 新增：路径参数 + 查询参数 组合 ===
@app.get("/blog/{blog_id}/comments")
def get_blog_comments(blog_id: int, page: int = 1, size: int = 10):
    return {
        "blog_id": blog_id,
        "page": page,
        "size": size,
        "message": f"返回文章 #{blog_id} 的第 {page} 页评论，每页 {size} 条"
    }
```

保存文件。

#### 验证

```
http://127.0.0.1:8000/blog/5/comments?page=1&size=3
```

返回：

```json
{"blog_id":5,"page":1,"size":3,"message":"返回文章 #5 的第 1 页评论，每页 3 条"}
```

**一句话总结**：路径参数决定"对谁操作"，查询参数决定"怎么操作"。

### 步骤 8：❌ 常见错误——路由冲突

这是一个新手非常容易踩的坑。假设你有两个接口：

```python
@app.get("/blog/latest")
def get_latest_blog():
    return {"title": "最新文章"}

@app.get("/blog/{blog_id}")
def get_blog(blog_id: int):
    return {"blog_id": blog_id}
```

当你访问 `http://127.0.0.1:8000/blog/latest` 时，你期望返回"最新文章"。但实际会发生什么？

**FastAPI 会按路由定义的顺序匹配。** 如果 `/blog/{blog_id}` 定义在 `/blog/latest` 之前，FastAPI 会把 `latest` 当成 `blog_id` 的值，然后尝试把它转成 `int`——`latest` 不是整数，所以返回 **422 错误**。

#### ❌ 错误示例

```python
# ❌ 错误：/blog/{blog_id} 定义在前面，/blog/latest 永远匹配不到
@app.get("/blog/{blog_id}")
def get_blog(blog_id: int):
    return {"blog_id": blog_id}

@app.get("/blog/latest")
def get_latest_blog():
    return {"title": "最新文章"}
```

访问 `/blog/latest` → 返回 422（因为 `latest` 不能转成 `int`）

#### ✅ 正确示例

```python
# ✅ 正确：固定路径定义在前面，变量路径定义在后面
@app.get("/blog/latest")
def get_latest_blog():
    return {"title": "最新文章"}

@app.get("/blog/{blog_id}")
def get_blog(blog_id: int):
    return {"blog_id": blog_id}
```

访问 `/blog/latest` → 返回 `{"title": "最新文章"}` ✅
访问 `/blog/5` → 返回 `{"blog_id": 5}` ✅

**规则**：**固定路径永远写在变量路径前面。** 就像停车场——固定车位（`/blog/latest`）先安排，灵活车位（`/blog/{blog_id}`）放后面。

---

### 🤔 想多一点：为什么路由顺序这么重要？

FastAPI 的路由匹配机制是**从上到下、先到先得**。它不会"聪明地"判断"latest 看起来像字符串，应该匹配固定路径"——它只是机械地按顺序尝试。

这个设计不是 FastAPI 的缺陷，而是**刻意为之的简单性**。如果框架要"智能匹配"，它需要理解每个路径的语义——"这个参数是 int，所以 latest 不是 int，跳过"——这会让路由匹配变得不可预测。你永远不知道哪个路由会被匹配到，因为规则太复杂了。

**简单规则 = 可预测行为 = 易调试。** 这就是 FastAPI 的设计哲学：规则越少越好，但每条规则都严格执行。

## 四、完整代码清单

### `blog_backend/main.py`（本教程最终版本）

```python
from fastapi import FastAPI
from fastapi.responses import JSONResponse

app = FastAPI()


@app.get("/")
def read_root():
    return {"message": "Hello World"}


@app.get("/secret")
def secret_page():
    return JSONResponse(
        status_code=404,
        content={"error": "你找的秘密页面不存在！", "hint": "试试别的路径，比如 /"}
    )


# === 路径参数：定位"哪一个"资源 ===
@app.get("/blog/{blog_id}")
def get_blog(blog_id: int):
    return {
        "blog_id": blog_id,
        "title": f"文章 #{blog_id}",
        "content": f"这是第 {blog_id} 篇文章的内容"
    }


# === 固定路径必须写在变量路径前面 ===
@app.get("/blog/latest")
def get_latest_blog():
    return {"title": "最新文章", "content": "这是最新的一篇文章"}


# === 查询参数：筛选条件 ===
@app.get("/blogs")
def list_blogs(page: int = 1, size: int = 10):
    return {
        "page": page,
        "size": size,
        "message": f"返回第 {page} 页，每页 {size} 条数据"
    }


# === 可选查询参数：允许不传 ===
@app.get("/search")
def search_blogs(keyword: str | None = None):
    if keyword:
        return {"keyword": keyword, "message": f"搜索关键词：{keyword}"}
    else:
        return {"keyword": None, "message": "没有提供关键词，显示全部内容"}


# === 路径参数 + 查询参数 组合 ===
@app.get("/blog/{blog_id}/comments")
def get_blog_comments(blog_id: int, page: int = 1, size: int = 10):
    return {
        "blog_id": blog_id,
        "page": page,
        "size": size,
        "message": f"返回文章 #{blog_id} 的第 {page} 页评论，每页 {size} 条"
    }
```

## 六、小结

| 你学到了什么 | 一句话总结 |
|--------------|-----------|
| 路径参数 | URL 路径里用 `{变量名}` 表示，用来定位"哪一个"资源，如 `/blog/{blog_id}` |
| 查询参数 | URL 问号 `?` 后面 `key=value` 格式，用来筛选/排序/分页，如 `?page=1&size=10` |
| 类型注解 `: int` | 告诉 FastAPI 参数的期望类型，框架自动做类型转换和校验 |
| 默认值 `= 1` | 查询参数不传时使用的值，让参数变成可选的 |
| 可选参数 `str \| None = None` | 参数可以不传，值为 `None`，适合搜索等场景 |
| 路由顺序 | 固定路径（`/blog/latest`）必须写在变量路径（`/blog/{blog_id}`）前面 |
| 422 错误 | 类型转换失败时 FastAPI 自动返回的错误码，帮你挡掉无效请求 |
| 路径参数 + 查询参数组合 | 可以同时使用，路径参数决定"对谁操作"，查询参数决定"怎么操作" |

## 七、术语附录

| 术语 | 英文 | 通俗解释 | 本章出现位置 |
|------|------|----------|-------------|
| 路径参数 | path parameter | URL 路径中 `{ }` 包裹的变量，如 `/blog/{blog_id}` 里的 `blog_id`。用来唯一标识一个资源。**字面陷阱**：它叫"路径参数"但不是"路径的参数"，而是"路径本身的一部分"。 | 步骤 2 |
| 查询参数 | query parameter | URL 问号 `?` 后面的 `key=value` 对，如 `?page=1&size=10`。用来筛选、排序、分页。**字面陷阱**：叫"查询"但不是只能用于搜索，任何附加条件都是查询参数。 | 步骤 4 |
| 类型注解 | type hint / type annotation | Python 3.5+ 的语法，在变量或参数后面加 `: 类型`，如 `blog_id: int`。FastAPI 利用它做自动校验和转换。 | 步骤 2 |
| 类型转换 | type coercion / parsing | 把一种类型的数据转成另一种类型。比如 URL 里传过来的都是字符串，FastAPI 自动把 `"1"` 转成整数 `1`。 | 步骤 3 |
| 路由冲突 | route conflict | 两个路由模式都能匹配同一个 URL 时发生的问题。固定路径和变量路径写反顺序是最常见的路由冲突场景。 | 步骤 8 |
| 422 Unprocessable Entity | — | HTTP 状态码，表示"请求格式没问题，但数据内容不合法"。FastAPI 在类型转换失败时返回此状态码。 | 步骤 3 |
| 默认值 | default value | 函数参数 `= 值` 的部分，当调用方不传这个参数时使用的值。 | 步骤 4 |
| Optional / `\| None` | — | 表示一个参数可以传值也可以不传（值为 `None`）。Python 3.10+ 用 `str \| None`，旧版本用 `Optional[str]`。 | 步骤 5 |

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| 路径参数花括号里名字和函数参数名不一致 | 服务器启动报错或返回异常 | FastAPI 靠名字匹配路径参数和函数参数，名字必须一模一样 | 确保 `{blog_id}` 和 `def get_blog(blog_id: int)` 里的名字一致 |
| 固定路径写在变量路径后面 | 访问固定路径返回 422 或变量路径的内容 | FastAPI 从上到下匹配，`/blog/{blog_id}` 会抢先把 `/blog/latest` 匹配掉 | **固定路径永远写在变量路径前面**（见步骤 8） |
| 查询参数 URL 里拼错参数名 | 参数用默认值，感觉"没生效" | 你传了 `?pgae=3`（拼错了），FastAPI 不知道 `pgae` 是什么，忽略了它，`page` 用默认值 `1` | 仔细检查参数名拼写，和函数参数名对照 |
| 忘记给查询参数加默认值 | 访问不带参数的 URL 返回 422 | 没有默认值的查询参数被视为必填 | 如果参数是可选的，加上 `= 默认值` 或 `= None` |
| 类型注解写错 | 访问时报 422 或返回异常数据 | 比如 `blog_id: str` 写了 `blog_id: int` 但实际想要字符串 | 确认参数的实际类型，写对应的类型注解 |
| curl 里 URL 含 `&` 被终端截断 | 只传了第一个参数 | 终端把 `&` 解释为"后台运行" | 用双引号包裹 URL：`curl "http://127.0.0.1:8000/blogs?page=1&size=5"` |
| 路径参数写了但函数参数没写 | 服务器启动报错 | FastAPI 在路径里发现了 `{blog_id}`，但在函数参数里找不到同名的 | 确保函数参数包含路径里所有花括号变量 |
