## 我在做什么？

前几章你学会了筛选、排序、分页——但这些操作都是"一行一行"地看数据。现实中的需求常常是"统计"：有多少用户？平均订单金额是多少？每个城市的用户数？这些都需要**聚合函数**和**分组**。

这一章，我们学 SQL 中将多行数据"压缩"成汇总信息的操作：`COUNT`、`SUM`、`AVG`、`MAX`、`MIN`，以及 `GROUP BY` 和 `HAVING`。最后，我们会画一张 **SQL 执行顺序图**——这是贯穿全书的核心知识，理解它之后，很多 SQL 问题会变得一目了然。

## 一、聚合函数：把多行"压缩"成一行

**聚合函数（Aggregate Function）** *此术语见附录E* 的作用是：输入多行数据，输出一个汇总值。

### 1.1 COUNT：统计行数

```sql
-- 统计总用户数
SELECT COUNT(*) FROM users;
```

输出：

```
+----------+
| COUNT(*) |
+----------+
|        4 |
+----------+
```

### 1.2 SUM：求和

```sql
-- 统计所有订单的总金额
SELECT SUM(amount) FROM orders;
```

输出：

```
+-------------+
| SUM(amount) |
+-------------+
|    28395.00 |
+-------------+
```

### 1.3 AVG：平均值

```sql
-- 统计平均订单金额
SELECT AVG(amount) FROM orders;
```

输出：

```
+-------------+
| AVG(amount) |
+-------------+
| 5679.000000 |
+-------------+
```

### 1.4 MAX / MIN：最大值和最小值

```sql
-- 最高和最低订单金额
SELECT MAX(amount) AS 最高金额, MIN(amount) AS 最低金额
FROM orders;
```

输出：

```
+----------+----------+
| 最高金额  | 最低金额  |
+----------+----------+
| 12999.00 |  1899.00 |
+----------+----------+
```

### 1.5 聚合函数一览

| 函数 | 作用 | 示例 |
|------|------|------|
| `COUNT(*)` | 统计总行数（含 NULL） | `SELECT COUNT(*) FROM users;` |
| `COUNT(col)` | 统计指定列非 NULL 的行数 | `SELECT COUNT(phone) FROM users;` |
| `SUM(col)` | 求和 | `SELECT SUM(amount) FROM orders;` |
| `AVG(col)` | 平均值 | `SELECT AVG(amount) FROM orders;` |
| `MAX(col)` | 最大值 | `SELECT MAX(amount) FROM orders;` |
| `MIN(col)` | 最小值 | `SELECT MIN(created_at) FROM users;` |

## 二、COUNT(*) vs COUNT(col)：一个关键区别

```sql
-- 假设 users 表中有 4 行，其中 2 行的 phone 是 NULL
SELECT COUNT(*) FROM users;       -- 返回 4（所有行）
SELECT COUNT(phone) FROM users;   -- 返回 2（只统计 phone 非 NULL 的行）
```

`COUNT(*)` 统计所有行，不管列值是否 NULL。`COUNT(col)` 只统计该列**非 NULL** 的行。

> **想多一点**：有人问 `COUNT(1)` 和 `COUNT(*)` 有什么区别？在 MySQL 中，**没有区别**。MySQL 对 `COUNT(*)` 做了专门优化，和 `COUNT(1)` 一样快。所以你不需要纠结到底用 `COUNT(*)`、`COUNT(1)` 还是 `COUNT(id)`——直接用 `COUNT(*)` 就好，语义最清晰。

## 三、GROUP BY：分组聚合

`GROUP BY` *此术语见附录E* 是聚合函数的"搭档"——它把数据按某列的值分成多个组，然后对**每个组**分别执行聚合。

### 3.1 基本用法

```sql
-- 统计每个用户有多少订单
SELECT user_id, COUNT(*) AS 订单数
FROM orders
GROUP BY user_id;
```

输出：

```
+---------+--------+
| user_id | 订单数  |
+---------+--------+
|       1 |      2 |
|       2 |      1 |
|       3 |      2 |
+---------+--------+
```

这条 SQL 的逻辑是：
1. 把 `orders` 表按 `user_id` 分组 → 用户 1 的订单放一组，用户 2 的放一组，用户 3 的放一组
2. 对每组分别统计 `COUNT(*)` → 用户 1 有 2 条，用户 2 有 1 条，用户 3 有 2 条
3. 每组输出一行

### 3.2 多列分组

```sql
-- 按 user_id 和 product 分组
SELECT user_id, product, COUNT(*) AS 数量
FROM orders
GROUP BY user_id, product;
```

按 `(user_id, product)` 的组合分组——只有当两列都相同时，才归入同一组。

### 3.3 分组后排序

```sql
-- 统计每个用户的订单数和总金额，按订单数降序
SELECT
    user_id,
    COUNT(*)     AS 订单数,
    SUM(amount)  AS 总金额
FROM orders
GROUP BY user_id
ORDER BY 订单数 DESC;
```

## 四、HAVING：过滤分组结果

`HAVING` *此术语见附录E* 和 `WHERE` 很像，但它们的过滤对象不同：
- `WHERE`：过滤**原始行**（在分组之前）
- `HAVING`：过滤**分组后的结果**（在分组之后）

### 4.1 基本用法

```sql
-- 找出订单数超过 1 个的用户
SELECT user_id, COUNT(*) AS 订单数
FROM orders
GROUP BY user_id
HAVING COUNT(*) > 1;
```

### 4.2 WHERE 和 HAVING 同时使用

```sql
-- 先过滤掉金额小于 2000 的订单，再统计每个用户的订单数，最后只保留订单数 > 1 的用户
SELECT
    user_id,
    COUNT(*)    AS 订单数,
    SUM(amount) AS 总金额
FROM orders
WHERE amount > 2000           -- 第 1 步：过滤原始行
GROUP BY user_id              -- 第 2 步：分组
HAVING COUNT(*) > 1;          -- 第 3 步：过滤分组结果
```

> **记住**：WHERE 在 GROUP BY 之前，HAVING 在 GROUP BY 之后。WHERE 过滤行，HAVING 过滤组。

## 五、SQL 执行顺序（全书核心知识点）

这是一张你需要在心里记住的图。SQL 语句中各部分的执行顺序，**不是你写的顺序**：

```
你写的顺序：                         实际执行顺序：
SELECT ...          （第 5 步）       FROM        （第 1 步）
FROM ...            （第 1 步）       WHERE       （第 2 步）
WHERE ...           （第 2 步）       GROUP BY    （第 3 步）
GROUP BY ...        （第 3 步）       HAVING      （第 4 步）
HAVING ...          （第 4 步）       SELECT      （第 5 步）
ORDER BY ...        （第 6 步）       ORDER BY    （第 6 步）
LIMIT ...           （第 7 步）       LIMIT       （第 7 步）
```

### 用一张流程图来理解

```
┌─────────────────────────────────────────────────┐
│                    SQL 执行顺序                    │
├─────────────────────────────────────────────────┤
│                                                   │
│  FROM         找到数据来源（哪张表）                │
│    │                                              │
│    ▼                                              │
│  WHERE        过滤原始行                           │
│    │                                              │
│    ▼                                              │
│  GROUP BY     分组                                 │
│    │                                              │
│    ▼                                              │
│  HAVING       过滤分组结果                         │
│    │                                              │
│    ▼                                              │
│  SELECT       选择要输出的列（计算表达式、别名）     │
│    │                                              │
│    ▼                                              │
│  ORDER BY     排序（此时才能用 SELECT 中定义的别名） │
│    │                                              │
│    ▼                                              │
│  LIMIT        限制返回行数                         │
│                                                   │
└─────────────────────────────────────────────────┘
```

### 为什么执行顺序很重要？

**原因 1：别名可用性**

```sql
-- ❌ 错误：WHERE 不能使用 SELECT 中定义的别名
SELECT user_id, COUNT(*) AS 订单数
FROM orders
WHERE 订单数 > 1   -- 报错！WHERE 执行时，别名"订单数"还不存在
GROUP BY user_id;

-- ✅ 正确：HAVING 和 ORDER BY 可以使用别名
SELECT user_id, COUNT(*) AS 订单数
FROM orders
GROUP BY user_id
HAVING 订单数 > 1;   -- 正确！HAVING 在 SELECT 之后执行
```

**原因 2：聚合函数的使用位置**

```sql
-- ❌ 错误：WHERE 中不能使用聚合函数
SELECT user_id, COUNT(*) AS 订单数
FROM orders
WHERE COUNT(*) > 1     -- 报错！WHERE 执行时还没有分组，无法聚合
GROUP BY user_id;

-- ✅ 正确：HAVING 中可以使用聚合函数
SELECT user_id, COUNT(*) AS 订单数
FROM orders
GROUP BY user_id
HAVING COUNT(*) > 1;   -- 正确！HAVING 在 GROUP BY 之后执行
```

> **想多一点**：把执行顺序记在心里后，很多 SQL 问题都不需要查资料了。比如"为什么 WHERE 不能用别名？"——因为 WHERE 在第 2 步执行，而别名是在第 5 步 SELECT 中定义的，第 2 步时还不存在。这个逻辑推理比死记硬背好用得多。

## 六、ONLY_FULL_GROUP_BY：一个容易踩的坑

### 6.1 什么是 SQL Mode？

**SQL Mode** *此术语见附录E* 是 MySQL 的"行为开关"——它决定了 MySQL 如何处理某些边界情况。你可以把它理解为 MySQL 的"性格设置"。

查看当前 SQL Mode：

```sql
SELECT @@sql_mode;
```

输出可能包含很多标志，其中一个是 `ONLY_FULL_GROUP_BY`。

### 6.2 ONLY_FULL_GROUP_BY 是什么？

`ONLY_FULL_GROUP_BY` *此术语见附录E* 是 MySQL 5.7.5+ 默认开启的一个 SQL Mode。它要求：**SELECT 中的非聚合列，必须出现在 GROUP BY 中。**

❌ 违反 ONLY_FULL_GROUP_BY 的写法：

```sql
-- user_id 在 GROUP BY 中，但 username 不在
SELECT user_id, username, COUNT(*) AS 订单数
FROM orders
JOIN users ON orders.user_id = users.id
GROUP BY user_id;
```

MySQL 报错：

```
ERROR 1055 (42000): Expression #2 of SELECT list is not in GROUP BY clause
and contains nonaggregated column 'my_shop.users.username' which is not
functionally dependent on columns in GROUP BY clause; this is incompatible
with sql_mode=only_full_group_by
```

✅ 正确写法：

```sql
-- 要么把 username 也加入 GROUP BY
SELECT user_id, username, COUNT(*) AS 订单数
FROM orders
JOIN users ON orders.user_id = users.id
GROUP BY user_id, username;

-- 要么去掉 username（如果不需要）
SELECT user_id, COUNT(*) AS 订单数
FROM orders
GROUP BY user_id;
```

### 6.3 为什么这个规则存在？

想象一下：你有 2 个订单，都属于 user_id=1。但有一条是 `username='alice'`，另一条是 `username='alice_wang'`（用户名被改过）。如果 `GROUP BY user_id` 只输出一行，username 该显示 alice 还是 alice_wang？

数据库无法决定，所以它要求你**要么把 username 也放入 GROUP BY（确保唯一），要么不要 SELECT username**。

## 我做得对不对？

### 验证方法

```sql
-- 1. 测试 COUNT
SELECT COUNT(*) FROM users;
-- 应该返回 4（假设你没有额外删数据）

-- 2. 测试 SUM 和 AVG
SELECT SUM(amount) AS 总金额, AVG(amount) AS 平均金额 FROM orders;
-- 手动加总验算：6999+3499+12999+1899+2999 = 28395 ✅
-- 手动平均：28395/5 = 5679 ✅

-- 3. 测试 GROUP BY
SELECT user_id, COUNT(*) AS 订单数 FROM orders GROUP BY user_id;
-- user_id=1: 2 条，user_id=2: 1 条，user_id=3: 2 条 ✅

-- 4. 测试 HAVING
SELECT user_id, COUNT(*) AS 订单数
FROM orders
GROUP BY user_id
HAVING COUNT(*) > 1;
-- 应该只返回 user_id=1 和 user_id=3（订单数 > 1）

-- 5. 测试 SQL 执行顺序理解
-- 对比以下两条语句，看哪个报错
SELECT user_id, COUNT(*) AS cnt FROM orders WHERE cnt > 1 GROUP BY user_id;  -- 报错！
SELECT user_id, COUNT(*) AS cnt FROM orders GROUP BY user_id HAVING cnt > 1; -- 正确！
-- 原因：WHERE 在第 2 步执行，此时别名 cnt 还不存在；HAVING 在第 4 步，别名已存在
```

## 不对怎么办？

### 常见错误 1：SELECT 中放了非聚合列但不在 GROUP BY 中

❌ 错误做法：

```sql
SELECT user_id, product, COUNT(*) AS 数量
FROM orders
GROUP BY user_id;  -- product 不在 GROUP BY 中！
```

如果 `ONLY_FULL_GROUP_BY` 开启，MySQL 会报错。

✅ 正确做法：

```sql
-- 要么把 product 也加入 GROUP BY
SELECT user_id, product, COUNT(*) AS 数量
FROM orders
GROUP BY user_id, product;

-- 要么不 SELECT product
SELECT user_id, COUNT(*) AS 数量
FROM orders
GROUP BY user_id;
```

### 常见错误 2：WHERE 和 HAVING 混用

❌ 错误做法：

```sql
-- 试图用 WHERE 过滤聚合结果
SELECT user_id, COUNT(*) AS 订单数
FROM orders
WHERE COUNT(*) > 1   -- 报错！WHERE 中不能用聚合函数
GROUP BY user_id;
```

✅ 正确做法：

```sql
-- 过滤聚合结果用 HAVING
SELECT user_id, COUNT(*) AS 订单数
FROM orders
GROUP BY user_id
HAVING COUNT(*) > 1;

-- 过滤原始行用 WHERE
SELECT user_id, COUNT(*) AS 订单数
FROM orders
WHERE amount > 2000     -- 先过滤掉金额 <= 2000 的行
GROUP BY user_id
HAVING COUNT(*) > 1;    -- 再过滤分组结果
```

**记住口诀**：WHERE 过滤行（分组前），HAVING 过滤组（分组后）。WHERE 里不能用聚合函数，HAVING 里可以。

### 常见错误 3：WHERE 中使用了 SELECT 的别名

❌ 错误做法：

```sql
SELECT user_id, COUNT(*) AS 订单数
FROM orders
WHERE 订单数 > 1   -- 报错！WHERE 执行时别名"订单数"还不存在
GROUP BY user_id;
```

✅ 正确做法：在 HAVING 或 ORDER BY 中使用别名：

```sql
SELECT user_id, COUNT(*) AS 订单数
FROM orders
GROUP BY user_id
HAVING 订单数 > 1;  -- 正确！HAVING 在 SELECT 之后执行
```

### 常见错误 4：COUNT 和 SUM 混淆

❌ 错误做法：

```sql
-- 想统计订单总数，但写成了 SUM
SELECT SUM(*) FROM orders;  -- 语法错误！SUM 不能接 *
```

✅ 正确做法：

```sql
SELECT COUNT(*) FROM orders;  -- 统计行数
SELECT SUM(amount) FROM orders;  -- 对某列求和
```

`COUNT(*)` 统计行数，`SUM(col)` 对某列的值求和。两者不能互换。

## 最可能出错的地方及原因

以下是第二篇上半（04-09 章）中读者最容易出错的地方，按"灾难性"从高到低排列：

### 1. UPDATE / DELETE 忘记 WHERE → 全表数据被改/删
**原因**：写 SQL 时习惯先写 SET 再写 WHERE，写到一半忘了补 WHERE。  
**预防**：养成习惯——先写 WHERE 再写 SET 或 DELETE；操作前先 SELECT 确认范围。

### 2. `= NULL` 而不是 `IS NULL` → 查询结果永远为空
**原因**：NULL 的三值逻辑和直觉相反——直觉上"等于 NULL"应该是"为空"，但 SQL 中 `= NULL` 永远返回 NULL（未知），被 WHERE 过滤掉。  
**预防**：记住"NULL 不能比较，只能 IS NULL / IS NOT NULL"。

### 3. AND 和 OR 优先级混淆 → 查询结果不符合预期
**原因**：AND 优先级高于 OR，类似 `*` 高于 `+`。不加括号时条件被意外分组。  
**预防**：混合使用 AND 和 OR 时，永远加括号。

### 4. WHERE 和 HAVING 混用 → 报错或结果错误
**原因**：不清楚 SQL 执行顺序——WHERE 在分组前执行，不能用聚合函数；HAVING 在分组后执行。  
**预防**：记住口诀"WHERE 过滤行，HAVING 过滤组"。

### 5. GROUP BY 中漏了 SELECT 的非聚合列 → ONLY_FULL_GROUP_BY 报错
**原因**：MySQL 5.7+ 默认开启此模式，但很多老教程没有提到。  
**预防**：要么把 SELECT 中的非聚合列都加入 GROUP BY，要么去掉不必要的列。

### 6. 深分页直接用 LIMIT 大 OFFSET → 查询越来越慢
**原因**：不理解 MySQL 的 LIMIT 实现——偏移量越大，需要跳过的行越多，成本线性增长。  
**预防**：大数据量用游标分页（`WHERE id > last_id`），不要用大 OFFSET。
