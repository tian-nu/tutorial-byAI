## 我在做什么？

第 04 章你建好了房子（数据库）和家具（表），但房间是空的——没有数据。这一章，我们往里面**放东西**：插入数据、修改数据、删除数据。

这些操作都属于 **DML（Data Manipulation Language，数据操作语言）** *此术语见附录E*——SQL 的 5 种子语言中负责"增删改数据"的那一种。

学完这一章，你的 `users` 表里会有 5 个用户，`orders` 表里会有几条订单，你能随心所欲地修改和删除它们。

> ⚠️ **安全提醒**：本章所有密码示例使用 `your_password_here` 占位，请替换为你自己的密码。插入数据时，密码列存的是哈希值（不是明文密码）。

## 一、INSERT：插入数据

`INSERT` *此术语见附录E* 是"往表里添加新行"的命令。每执行一次 INSERT，表中就多一行数据。

### 1.1 单行插入

```sql
INSERT INTO users (username, email, password_hash)
VALUES ('alice', 'alice@example.com', 'hashed_password_alice_123');
```

拆解这条语句：

- `INSERT INTO users`：往 `users` 表里插入
- `(username, email, password_hash)`：指定要填哪些列。注意：**没有写 `id` 和 `created_at`**——因为 `id` 是自增的（数据库自动填），`created_at` 有默认值（数据库自动填当前时间）。
- `VALUES (...)`：对应上面列的**值**，顺序必须一致。

执行后，MySQL 输出：

```
Query OK, 1 row affected (0.01 sec)
```

### 1.2 批量插入（一次插入多行）

```sql
INSERT INTO users (username, email, password_hash) VALUES
    ('bob',   'bob@gmail.com',       'hashed_password_bob_456'),
    ('charlie', 'charlie@outlook.com', 'hashed_password_charlie_789'),
    ('diana', 'diana@company.com',   'hashed_password_diana_012'),
    ('eve',   'eve@example.com',     'hashed_password_eve_345');
```

关键点：多组 `VALUES` 用逗号分隔，一次性插入 4 行。这比执行 4 次单行 INSERT 快得多——数据库只需要一次通信、一次解析、一次提交。

> **想多一点**：为什么批量插入更快？因为每次 INSERT 都会触发：网络传输 → SQL 解析 → 磁盘写入 → 索引更新。如果你插入 1000 行，逐条插入就是 1000 次以上过程。而批量插入只需要 1 次。第 20 章会深入讨论这个优化技巧。

### 1.3 INSERT INTO ... SELECT（从其他表复制数据）

```sql
-- 假设有一个 vip_users 表，把所有 VIP 用户的邮箱复制到另一个通知表
INSERT INTO email_notifications (email, message)
SELECT email, '尊敬的VIP用户，您好！'
FROM vip_users
WHERE vip_level >= 3;
```

这个语法让你把 SELECT 查询的结果直接插入到另一张表中，非常适合数据迁移。

### 1.4 插入订单数据

有了用户，我们再来插入几条订单：

```sql
INSERT INTO orders (user_id, product, amount) VALUES
    (1, 'iPhone 15',   6999.00),
    (1, 'iPad Air',    3499.00),
    (2, 'MacBook Pro', 12999.00),
    (3, 'AirPods Pro', 1899.00),
    (3, 'Apple Watch', 2999.00);
```

注意：`user_id` 的值对应 `users.id` 的值。`user_id = 1` 表示"这个订单属于 alice"。

## 二、UPDATE：修改数据

`UPDATE` *此术语见附录E* 用来修改表中已有的数据。

### 2.1 基本用法

```sql
UPDATE users
SET email = 'alice_new@example.com'
WHERE id = 1;
```

拆解：
- `UPDATE users`：修改 `users` 表
- `SET email = 'alice_new@example.com'`：把 `email` 列改成新值
- `WHERE id = 1`：**只修改 `id = 1` 的那一行**

### 2.2 同时修改多列

```sql
UPDATE users
SET username = 'alice_wang',
    email = 'alice_wang@example.com'
WHERE id = 1;
```

多列之间用逗号分隔。

### 2.3 WHERE 的重要性：忘写 WHERE 的灾难

❌ 错误做法——忘写 WHERE：

```sql
UPDATE users
SET email = 'everyone@example.com';
-- 忘了 WHERE！整张表的所有行的 email 都被改了！
```

这条 SQL 会把**所有用户**的邮箱都改成 `everyone@example.com`。如果有 100 万用户，100 万条数据全部被改，而且你很难恢复（除非有备份）。

✅ 正确做法——永远先写 WHERE 再写 SET：

```sql
-- 第一步：先写 WHERE，确认目标行
SELECT * FROM users WHERE id = 1;

-- 第二步：确认无误后，再写 UPDATE
UPDATE users
SET email = 'alice_new@example.com'
WHERE id = 1;
```

> **防呆技巧**：写 UPDATE 时，**先写 WHERE 子句，再写 SET 子句**。这样你就不会在写了一半时忘记 WHERE。

### 2.4 基于条件批量更新

```sql
-- 把所有 Gmail 用户的邮箱改成 @gmail.com 新域名
UPDATE users
SET email = REPLACE(email, '@gmail.com', '@googlemail.com')
WHERE email LIKE '%@gmail.com';
```

这里用了 `REPLACE()` 函数——在 SQL 里也可以做字符串替换。

## 三、DELETE：删除数据

`DELETE` *此术语见附录E* 用来删除表中的行。

### 3.1 基本用法

```sql
DELETE FROM users
WHERE id = 5;
```

删除 `id = 5` 的那一行。

### 3.2 WHERE 同样至关重要

❌ 错误做法——忘写 WHERE：

```sql
DELETE FROM users;
-- 忘了 WHERE！整张表的所有数据被清空！
```

✅ 正确做法——先 SELECT 确认再 DELETE：

```sql
-- 第一步：确认要删的行
SELECT * FROM users WHERE id = 5;

-- 第二步：确认无误后，再 DELETE
DELETE FROM users WHERE id = 5;
```

### 3.3 MySQL 的安全模式（SQL_SAFE_UPDATES）

MySQL 有一个保护机制：如果你执行 UPDATE 或 DELETE 时**没有用 WHERE 条件**（或者 WHERE 条件没用到索引），MySQL 会拒绝执行，报错：

```
ERROR 1175 (HY000): You are using safe update mode and you tried to update a table
without a WHERE that uses a KEY column.
```

这是 MySQL Workbench 和某些客户端的默认设置。如果你确实需要全表更新/删除，可以临时关闭安全模式：

```sql
SET SQL_SAFE_UPDATES = 0;
-- 执行你的 UPDATE 或 DELETE
SET SQL_SAFE_UPDATES = 1;  -- 操作完立刻恢复
```

> **强烈建议**：操作完立刻把安全模式恢复为 1。这个功能是你的"安全气囊"——你永远不希望它在你需要的时候是关闭的。

## 四、TRUNCATE vs DELETE：全表清空的两种方式

如果你想清空一张表的所有数据，有两种选择：

### 方式 1：DELETE

```sql
DELETE FROM users;
```

- **逐行删除**：每删一行，写一次日志。
- **可回滚**：如果在事务中，可以 ROLLBACK 恢复。
- **不重置自增 ID**：删除后插入新数据，ID 从上次停下的地方继续。

### 方式 2：TRUNCATE

```sql
TRUNCATE TABLE users;
```

- **整表删除**：直接释放整个表的存储空间，不逐行操作。
- **不可回滚**：TRUNCATE 是 DDL 操作（不是 DML），无法在事务中回滚。
- **重置自增 ID**：删除后插入新数据，ID 从 1 重新开始。

| 对比维度 | DELETE FROM | TRUNCATE TABLE |
|---------|-------------|----------------|
| 删除方式 | 逐行删除 | 释放整个表空间 |
| 速度 | 慢（大表很慢） | 快（大表秒删） |
| 可回滚 | ✅ 事务中可回滚 | ❌ 不可回滚 |
| 自增 ID | 不重置 | 重置为 1 |
| 触发触发器 | ✅ 触发 | ❌ 不触发 |
| 属于哪类 SQL | DML | DDL |

> **什么时候用哪个？**
> - 测试环境、想快速清空数据重新开始 → TRUNCATE
> - 生产环境、可能还需要恢复 → DELETE（在事务中执行，确认无误再 COMMIT）
> - 只想删部分数据 → DELETE（TRUNCATE 只能全删）

## 五、自增 ID 的坑

`AUTO_INCREMENT`（自增 ID）*此术语见附录E* 用起来很方便，但有几个坑你必须知道。

### 坑 1：DELETE 后 ID 不回收

```sql
-- 假设 users 表目前有 id=1,2,3,4,5 五条数据
DELETE FROM users WHERE id = 5;

-- 再插入一条新数据
INSERT INTO users (username, email, password_hash)
VALUES ('frank', 'frank@example.com', 'hashed_password_frank');

-- 新数据的 id 是 6，不是 5！
```

为什么？因为自增 ID 是一个**单调递增的计数器**，它只往前走，不回头。即使你删了 id=5，计数器已经跳到了 6，下次插入还是 6。

**这意味着什么？** ID 会不连续。`1, 2, 3, 4, 6, 7, ...` 这是正常的。不要指望 ID 是连续的——把它只当作"唯一标识符"，不要赋予它"序号"的含义。

### 坑 2：并发插入时 ID 可能不连续

当多个客户端同时插入数据时，MySQL 分配自增 ID 的方式可能导致 ID 出现"空隙"：

```sql
-- 客户端 A：INSERT INTO users ... -- 分配 id=10
-- 客户端 B：INSERT INTO users ... -- 分配 id=11
-- 客户端 A：ROLLBACK（回滚了！）
-- 客户端 C：INSERT INTO users ... -- 分配 id=12
```

结果：id=10 被回滚了，表中没有 id=10 的行，但 id=11 和 id=12 存在。这就是 ID 不连续的另一个原因。

这种行为是 InnoDB 存储引擎的 AUTO-INC 锁机制决定的——为了性能，它不会在回滚时回收已分配的 ID。

### 对待自增 ID 的正确态度

1. **ID 就是 ID**：它的唯一职责是"唯一标识一行"。不要用 ID 做业务逻辑（如"ID 越小表示越早注册"）。
2. **不连续很正常**：不要看到 ID 跳了就觉得出了 Bug。
3. **不要手动修改 ID**：除非你完全清楚后果。

## 我做得对不对？

### 验证方法

```sql
-- 1. 确认用户数据已插入（应该有 5 行）
SELECT * FROM users;

-- 2. 确认订单数据已插入（应该有 5 行）
SELECT * FROM orders;

-- 3. 确认 alice 的邮箱已更新
SELECT username, email FROM users WHERE id = 1;

-- 4. 确认 id=5 已被删除
SELECT * FROM users WHERE id = 5;
-- 应该返回 Empty set

-- 5. 确认自增 ID 不回收
SELECT * FROM users ORDER BY id;
-- 观察：id 序列是 1, 2, 3, 6（4 和 5 被删了，新插入的是 6）
```

如果以上结果符合预期，恭喜——你掌握了增删改数据的基本操作！

## 不对怎么办？

### 常见错误 1：UPDATE / DELETE 忘记 WHERE

❌ 错误做法：

```sql
UPDATE users SET email = 'oops@example.com';  -- 全表更新！
DELETE FROM users;                             -- 全表删除！
```

✅ 正确做法：

```sql
-- 先 SELECT 确认范围
SELECT * FROM users WHERE id = 1;
-- 确认无误后再操作
UPDATE users SET email = 'new@example.com' WHERE id = 1;
DELETE FROM users WHERE id = 1;
```

### 常见错误 2：INSERT 时列数不匹配

❌ 错误做法：

```sql
-- 声明了 3 列，但只给了 2 个值
INSERT INTO users (username, email, password_hash)
VALUES ('alice', 'alice@example.com');
```

MySQL 报错：`ERROR 1136 (21S01): Column count doesn't match value count at row 1`

✅ 正确做法：

```sql
INSERT INTO users (username, email, password_hash)
VALUES ('alice', 'alice@example.com', 'hashed_password_alice');
-- 列数和值数必须一致
```

### 常见错误 3：字符串值忘记加引号

❌ 错误做法：

```sql
INSERT INTO users (username, email, password_hash)
VALUES (alice, alice@example.com, hashed_password);
-- 字符串没加引号！
```

MySQL 报错：`ERROR 1054 (42S22): Unknown column 'alice' in 'field list'`

✅ 正确做法：

```sql
INSERT INTO users (username, email, password_hash)
VALUES ('alice', 'alice@example.com', 'hashed_password');
-- 字符串值必须用单引号包裹
```

### 常见错误 4：违反唯一约束

❌ 错误做法：

```sql
-- 插入一个已存在的邮箱（如果 email 列有 UNIQUE 约束）
INSERT INTO users (username, email, password_hash)
VALUES ('alice2', 'alice@example.com', 'hashed_password');
```

MySQL 报错：`ERROR 1062 (23000): Duplicate entry 'alice@example.com' for key 'users.uk_email'`

✅ 正确做法：先检查邮箱是否已存在，或使用 `INSERT IGNORE`（插入时忽略重复错误）：

```sql
INSERT IGNORE INTO users (username, email, password_hash)
VALUES ('alice2', 'alice@example.com', 'hashed_password');
```

但 `INSERT IGNORE` 会静默跳过，可能让你不知道插入失败了。更推荐在应用层先检查。

### 常见错误 5：TRUNCATE 后想回滚

❌ 错误做法：

```sql
BEGIN;
TRUNCATE TABLE users;
ROLLBACK;  -- 数据不会回来！TRUNCATE 是 DDL，不能回滚
```

✅ 正确做法：如果可能还需要恢复，用 DELETE 而非 TRUNCATE：

```sql
BEGIN;
DELETE FROM users;
ROLLBACK;  -- 数据回来了！
```

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 单行插入 | `INSERT INTO 表名 (列...) VALUES (值...);` | 列的顺序和值的顺序必须一致 |
| 批量插入 | `INSERT INTO 表名 (列...) VALUES (值1), (值2), ...;` | 比逐条插入快得多，一次性提交 |
| INSERT ... SELECT | `INSERT INTO 目标表 SELECT ... FROM 源表;` | 适合数据迁移和复制 |
| 更新数据 | `UPDATE 表名 SET 列=值 WHERE 条件;` | **永远先写 WHERE 再写 SET！** 忘写 WHERE 导致全表更新 |
| 删除数据 | `DELETE FROM 表名 WHERE 条件;` | **永远先 SELECT 确认再 DELETE！** 忘写 WHERE 导致全表删除 |
| 安全模式 | `SET SQL_SAFE_UPDATES = 0/1;` | 操作完立刻恢复为 1，这是你的安全气囊 |
| TRUNCATE vs DELETE | TRUNCATE 不可回滚、重置自增 ID、速度快 | 测试环境用 TRUNCATE，生产环境用 DELETE（事务中） |
| 自增 ID 的坑 | DELETE 后 ID 不回收；并发插入 ID 不连续 | ID 的唯一职责是"唯一标识"，不要赋予它"序号"含义 |
| 违反约束 | UNIQUE 约束冲突时报 `Duplicate entry` | 插入前先检查或使用 `INSERT IGNORE` |
