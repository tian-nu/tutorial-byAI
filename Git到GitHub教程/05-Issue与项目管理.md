# 步骤5 · Issue：给项目贴便利贴

> 你在一个白板上写项目，突然发现有个功能坏了。你想修，但手头正忙别的事。你告诉自己"等下再修"——然后过两天就忘了。Issue 就是 GitHub 给你的便利贴：你可以随时在项目墙上贴一张纸条"这里有个Bug"，有空了再回来修。修好之后把纸条撕掉，干干净净。

---

## 5.1 Issue 到底是什么——用便利贴来理解

### 你办公桌上的便利贴

想象你在桌前工作。桌子前面有一块大白板，上面写着你正在做的项目。突然你发现登录按钮的颜色不对——你想修，但现在正在写别的功能，不能分心。

这时候你干吗？抓起一张黄色的小便利贴，写上"登录按钮颜色不对，待修"，啪一下贴在显示器旁边。等手头的活干完了，一抬头看到便利贴，哦对，该修这个了。修完了，便利贴撕下来扔掉。

**Issue 就是 GitHub 项目白板上的电子便利贴。** 它比纸质便利贴厉害的地方是：

- **不会丢**：纸质便利贴会掉、会黄、你妈会帮你丢掉。电子便利贴永远在。
- **能搜索**：不用翻便利贴堆。在搜索框里打几个字就出来了。
- **能分类**：你可以给不同颜色的便利贴——红的Bug、蓝的新功能、紫的文档。
- **能讨论**：别人可以在便利贴下面留言，像论坛帖子一样。
- **修好了自动撕掉**：等你写好代码、合并进项目，便利贴自动关掉。

> **此术语需进附录：Issue**——GitHub 项目中的"电子便利贴"，用来记录 Bug、功能请求或待办事项。每张便利贴有编号、可以讨论、可以被代码合并自动关闭。

### Issue 的三种类型

在 GitHub 项目里，你贴的便利贴通常有三种：

| 类型 | 什么时候用 | 标题举例 |
|------|-----------|---------|
| **Bug 报告**（此术语需进附录） | 程序出了问题，不是按你设计的样子在跑 | "运行 `go run main.go` 后输出乱码" |
| **功能请求**（此术语需进附录） | 你想给项目加一个新功能 | "添加一个显示当前时间的功能" |
| **待办任务** | 有一件事需要做，不着急，但不记下来会忘 | "给 README.md 补充使用说明" |

### Issue 和 Pull Request 的关系：一条龙

这是 GitHub 最聪明的设计——Issue 和 Pull Request（PR）不是两个东西，而是一条流水线上的前后两个环节：

```
① 有人发现问题 → ② 贴一张Issue便利贴 → ③ 有人写代码修理 → ④ 修理代码发起PR → ⑤ PR合并 → ⑥ Issue便利贴自动撕掉
```

简单说：**Issue 管"要做什么"，PR 管"怎么做"。** 先有便利贴说"这里坏了"，再有人写代码修好，修好之后便利贴自动关掉。不需要你手动去撕。

### 🤔 想多一点：我一个人的项目也需要 Issue 吗？

你可能会想："反正就我一个人在写，我脑子里记着就行了吧？"

人的短期记忆非常不靠谱。你此时此刻觉得"我肯定不会忘"——但一周之后呢？一个月之后呢？到时候用户报 Bug 了，你甚至不记得自己曾经发现过这个问题。

**Issue 是你的"外部大脑"。** 对于一个人的项目，Issue 帮你做这些事：

- 周五下班前把下周要做的事写成 Issue，周一打开一看，清清楚楚
- 写着写着突然想到一个好功能——马上开一个 Issue 记下来，别让灵感跑了
- 几个月后翻看 Issue 列表，看到自己一路解决了哪些问题，很有成就感

## 5.2 实操：创建你的第一张 Issue 便利贴

### 进入 Issue 页面

你需要用一个叫"浏览器"的软件（比如 Edge、Chrome、Firefox——就是你平时上网看网页的那个程序）。

**第1步：打开你的 GitHub 仓库页面**

在浏览器的地址栏（最顶上那个白色长条）里输入：

```
https://github.com/<你的用户名>/my-first-project
```

然后按键盘上的 **Enter 键**（就是回车键）。页面加载出来后，你看到的是你的仓库首页。这个页面你应该已经很熟悉了——就是我们前面几个步骤一直在用的页面。

**第2步：找到"Issues"标签**

看页面的上半部分，有一排灰色的文字标签，大概在仓库名字下面。这排标签从左到右依次是：

```
Code   Issues   Pull requests   Actions   Projects   Wiki   Security   Insights   Settings
```

用你的鼠标（或者触摸板）移动屏幕上的白色小箭头，移动到写着 **"Issues"** 的那个标签上面。然后**按一下鼠标左键**（就是点一下）。

> 如果你找不到这排标签在哪里：把你的视线放在 "<你的用户名> / my-first-project" 这个大字下面大约一厘米的位置。它们就在那里。

**第3步：看 Issues 页面**

点击"Issues"之后，浏览器里的内容变了。你现在看到的是 Issues 列表页面。因为这是你第一次使用，这个页面**应该是空的**——中间可能显示一行文字 "There aren't any open issues"（意思是没有打开的问题），或者 "Welcome to Issues!" 之类的欢迎字样。

在这个空白页面的右上角，有一个**绿色的长方形按钮**，上面写着白色的字 **"New issue"**。

### 创建你的第一张 Issue

**第4步：点击"New issue"**

把鼠标移动到右上角那个绿色按钮上面，按一下鼠标左键。

页面会跳转到一个新的页面。现在你看到的是 Issue 编辑页面。这个页面分成左右两半：

**左边（大的那片）：**
- 最顶上有一个**白色长条输入框**，里面有灰色的提示文字"Title"——这是写标题的地方
- 下面是一个**更大的白色矩形区域**，里面有灰色提示文字"Leave a comment"——这是写详细描述的地方

**右边（窄的那片）：**
- 有一排文字：Assignees（负责人）、Labels（标签）、Projects（项目）、Milestone（里程碑）
- 每个文字旁边有一个齿轮图标 ⚙️

**第5步：写 Bug 报告的标题**

Bug 就是程序出了毛病，和你想要的效果不一样。

把你的鼠标移动到左边那个"Title"白色长条上，按一下鼠标左键。你会看到里面有一个一闪一闪的竖线——这叫"光标"。现在你可以在键盘上打字了。

把标题写成这样（照着敲进去）：

```
修复 main.go 运行时输出乱码
```

**第6步：写 Bug 报告的详细描述**

把鼠标移动到标题下面那个大的白色矩形区域上，按一下鼠标左键。光标会开始在里面一闪一闪。

现在你需要把下面这段文字**完整打进去**。不要只复制粘贴——自己打一遍，你会更熟悉这个过程。

```
## 复现步骤
1. 打开终端
2. 输入 go run main.go 然后按回车
3. 看屏幕上出现什么字

## 期望看到的
屏幕上出现 "Hello, GitHub!"

## 实际看到的
屏幕上出现了我看不懂的乱码符号

## 我的环境
- 操作系统：Windows 11
- Go 版本：1.22
```

解释一下为什么 Bug 报告要这样写：

- **复现步骤**：你要像写"做菜步骤"一样，把从零开始触发这个 Bug 的每一步都写下来。这样别人（包括三个月后的你自己）才能一模一样地重现问题。没有复现步骤的 Bug 报告等于废纸——别人没法确认问题是不是真的存在。
- **期望 vs 实际**：这是 Bug 报告的心脏。你必须要说清楚两件事：①我本来希望发生什么（期望）；②实际上发生了什么（实际）。不说清楚这两点，别人连"这到底是不是 Bug"都判断不了。
- **环境信息**：很多 Bug 只在特定环境下才出现——比如 Windows 上正常、Mac 上出错；Go 1.21 正常、1.22 出错。不写环境信息，修 Bug 的人要花大量时间先问清楚这些。

### Bug 报告模板（直接复制用）

贴心的我给你准备了一个 Bug 报告模板。下次你要报 Bug，把下面这段文字复制到描述框里，填上空白的部分就行：

```
## 复现步骤
1. 
2. 
3. 

## 期望看到的


## 实际看到的


## 我的环境
- 操作系统：
- 软件版本：
```

### 功能请求模板（直接复制用）

如果你不是报 Bug，而是想提一个"加新功能"的 Idea，用下面这个模板：

```
## 我想要的功能


## 为什么需要


## 我的建议实现方式

```

**第7步：提交这张 Issue**

写完标题和描述之后，看页面的**最底部**。在描述框下面，有一个**绿色的长方形按钮**，上面写着 **"Submit new issue"**。

把鼠标移过去，按一下鼠标左键。

过一两秒，页面会跳转到一个新页面。你现在看到的是**你刚创建的 Issue 的详情页**。看浏览器顶部的地址栏——网址应该类似于：

```
https://github.com/<你的用户名>/my-first-project/issues/1
```

看到这个网址，说明你的第一张 Issue 便利贴已经贴上去了！

`#1` 是这张便利贴的编号。**记住这个编号**——后面你会用到它来让 PR 自动关掉这张便利贴。

### ❌ 错误写法 vs ✅ 正确写法

**Bug 报告：**

```
❌ 标题："有个Bug"
   描述："运行的时候报错了"
   为什么不行：不知道是哪个功能报错、怎么触发的、期望看到什么

✅ 标题："运行 main.go 时终端输出乱码"
   描述：包含复现步骤、期望行为、实际行为、环境信息
   为什么好：任何人拿着这个 Issue 就能开始排查
```

**功能请求：**

```
❌ 标题："加个功能吧"
   描述："感觉加个功能会更好"
   为什么不行：完全没说清楚要什么、为什么需要

✅ 标题："添加显示当前时间的功能"
   描述：明确写了想要什么功能、为什么需要、建议怎么做
   为什么好：维护者可以直接评估可行性和优先级
```

### Issue 的"开关"：Open 和 Closed

你刚创建的 Issue，现在是 **Open**（打开）状态。你可以看到页面上有一个紫色的标签写着一个圆圈里面有 "Open" 字样。

**关闭 Issue：**

在你刚创建的 Issue 详情页的**底部**，有一个**灰色的长方形按钮**，上面写着 **"Close issue"**。点了它，Issue 就关了。

关闭后你会发现：
- 那个紫色 Open 标签变成了**紫色 Closed 标签**（带一个 ✓ 标志）
- 页面中间出现一条横幅说 "<你的用户名> closed this as completed"
- 这张 Issue 默认不在列表里显示了——就像你把便利贴从白板上撕掉了

**重新打开 Issue：**

如果关错了怎么办？关闭之后，原来的"Close issue"按钮变成了 **"Reopen issue"**。点它，便利贴又贴回去了。

## 5.3 给便利贴分类：Label、Milestone、Assignee

当你的便利贴只有一两张时，用不着分类。但当你贴了十几张之后，就需要区分"哪些是 Bug"、"哪些是新功能"、"哪些是文档问题"。

GitHub 提供了三个管理工具。我们现在一个个看。

### Labels（标签）：给便利贴涂颜色

> **此术语需进附录：Label**——Issue 的彩色分类标签。比如红色=bug、蓝色=新功能、紫色=文档。一眼就能看出这张便利贴是什么类型。

**在哪里操作：**

打开你刚才创建的 Issue #1 页面。看**右侧边栏**（页面的右半部分）。在右侧边栏往下找，你会看到一行文字 **"Labels"**，旁边有一个齿轮图标 ⚙️。

**怎么添加标签：**

1. 把鼠标移到 "Labels" 那个齿轮图标上，按一下鼠标左键
2. 会弹出一个小窗口，里面列出了一串标签。每个标签是一个彩色的小方块，上面有字
3. 这些是 GitHub 自带的标签，常见的包括：

| 你在屏幕上看到的文字 | 颜色 | 什么意思 |
|------|------|------|
| `bug` | 红色底 | 这是一个Bug（错误） |
| `enhancement` | 浅蓝色底 | 这是一个新功能请求 |
| `documentation` | 紫色底 | 和文档有关 |
| `help wanted` | 深绿色底 | 需要别人帮忙 |
| `good first issue` | 深蓝色底 | 适合新手入门的简单任务 |
| `question` | 粉色底 | 需要讨论的问题 |

4. 把鼠标移到你想选的标签上，按一下鼠标左键。选中的标签会打一个勾 ✓
5. 对于你刚才的 Bug Issue，选 **`bug`** 标签
6. 选好后，把鼠标移到弹出窗口外面任意空白处，按一下鼠标左键（窗口自动关闭）

现在你看 Issue 页面，会看到标题旁边多了一个红色小方块，上面写着"bug"。这张便利贴被标记为 Bug 了。

**至少记住两个标签：** `bug`（Bug）和 `enhancement`（新功能）。每一个 Issue 你至少选其中一个贴上。

**怎么创建自己的标签：**

在 Issues 列表页面（不是单个 Issue 里面），顶部有一个灰色按钮写着 **"Labels"**。点它 → 看到 **"New label"** 绿色按钮 → 点它 → 输入名字、描述、选颜色 → 点 **"Create label"**。

### Milestones（里程碑）：给便利贴定一个小目标

> **此术语需进附录：Milestone**——里程碑。把一组 Issue 归到一个版本目标下。比如"v1.0 发布"这个里程碑下，挂着所有 v1.0 必须修完的 Issue。

**什么时候需要用里程碑：**

如果你只是想先体验 GitHub，**可以跳过 Milestone**。对于一个人的项目，Milestone 是选填的。当你 Issue 数量超过 10 个、开始分"这周要做"和"以后再说"的时候，再回来看这个功能。

**怎么创建里程碑：**

1. 在 Issues 列表页面（点击仓库顶部 "Issues" 标签回去），找 **"Milestones"** 按钮（在搜索框右边附近）
2. 点它 → 看到 **"New milestone"** 绿色按钮 → 点它
3. 在弹出的页面里：
   - **Title**（标题）输入框：写上这个里程碑的名字，比如 `v1.0 正式发布`
   - **Due date**（截止日期）：一个日历选择器，可以选一个目标日期，也可以不选
   - **Description**（描述）：大的文本框，写 `包含所有 v1.0 必须完成的功能和修复`
4. 点绿色的 **"Create milestone"** 按钮

创建后，回到你的 Issue 详情页，在右侧边栏找到 **"Milestone"**，点旁边的齿轮图标 → 在弹出的列表里选你刚创建的里程碑。

### Assignees（负责人）：便利贴交给谁处理

> **此术语需进附录：Assignee**——Issue 的指定处理人。告诉所有人"这件事归谁做"。

在 Issue 详情页右侧边栏找到 **"Assignees"**，点旁边的齿轮图标 → 在弹出的列表里选你自己（你的 GitHub 用户名）。

对于你一个人的项目，Assignee 填不填都行——反正所有事都是你做。但养成习惯填上自己，对你的编程素养有好处。到了团队项目中，这个字段极其重要——一眼看出谁在干什么，避免两个人抢同一件事。

## 5.4 魔法时刻：PR 合并后 Issue 自动关闭

这是 GitHub 最优雅的功能。你在写 Pull Request（PR）描述时，只要写上几个特定的"魔法单词"加上 Issue 的编号，**PR 被合并后，那张 Issue 便利贴就会自动撕掉**。

### 魔法单词列表（大小写都行）

以下任意一个单词都可以：

| 可以用 | 也可以用 | 也可以用 |
|--------|----------|----------|
| close | closes | closed |
| fix | fixes | fixed |
| resolve | resolves | resolved |

**写法：** 魔法单词 + 空格 + `#` + Issue 编号

比如，如果你之前创建的 Issue 编号是 `#1`，在 PR 的描述里写上：

```
Closes #1
```

PR 合并之后，Issue #1 就会自动关闭。

### 实操演示：完整的一条龙

我们快速模拟一遍完整流程：

**场景：** 你发现项目里有一个 Bug——"运行 main.go 输出乱码"。你已经创建了 Issue #1 来记录它。

**① 创建修复分支：**

```bash
git checkout -b fix/output-garbled
```

**② 修改代码、提交：**

```bash
git add main.go
git commit -m "fix: 修复main.go输出乱码"
```

**③ 推送到 GitHub：**

```bash
git push origin fix/output-garbled
```

**④ 在 GitHub 上发起 PR。**

在 PR 描述框里，除了说明你做了什么修改之外，**一定要写上**：

```
Closes #1
```

**⑤ 合并 PR。**

合并之后，打开浏览器访问 `https://github.com/<你的用户名>/my-first-project/issues/1`。你会看到：

- Issue 状态变成了紫色大标签 **"Closed as completed"**（带 ✓ 标志）
- 在描述下方，GitHub 自动添加了一条绿色注释：

> `<你的用户名> closed this as completed in #2 2 minutes ago`

- 点击那个蓝色的 **#2** 链接，可以直接跳到对应的 PR

从 Issue 到 PR 到代码改动，全部可以追溯。这就是 GitHub 的一条龙工作流。

### 重要注意事项

1. **魔法单词 + 编号必须写在 PR 描述正文中**，写在 PR 标题里是无效的
2. **只在 PR 合并到默认分支时才会自动关闭**（默认分支通常是 `main`）。合并到其他分支（比如 `develop`）不会触发自动关闭
3. **一个 PR 可以关闭多个 Issue**：

   ```
   Closes #1, fixes #3, resolves #7
   ```

4. **如果没自动关闭怎么办？** 有时候 GitHub 抽风了没触发自动关闭。别慌——手动去 Issue 页面点 "Close issue" 按钮就行。不是什么大事。

## 本章小结

| 知识点 | 一句话要点 |
|--------|-----------|
| Issue 是什么 | 贴在项目白板上的电子便利贴，记录 Bug、功能请求或待办事项 |
| Issue 三种类型 | Bug 报告（出毛病了）、功能请求（加新功能）、待办任务（要做的事） |
| Issue vs PR | Issue 管"要做什么"，PR 管"怎么做"，合并后自动闭环 |
| 创建 Issue 的入口 | 仓库页面 → 顶部"Issues"标签 → 右上角绿色"New issue"按钮 |
| 标题栏在哪 | Issue 编辑页面左上方的白色长条，里面有灰色"Title"提示 |
| 描述框在哪 | 标题下面的大白色矩形区域，支持 Markdown 格式 |
| Bug 报告怎么写 | 复现步骤 + 期望行为 + 实际行为 + 环境信息，缺一不可 |
| 功能请求怎么写 | 想要什么 + 为什么需要 + 建议怎么实现 |
| 提交按钮在哪 | 描述框正下方，绿色长方形按钮"Submit new issue" |
| Issue 两种状态 | Open（打开=便利贴还在白板上）、Closed（关闭=便利贴撕掉了） |
| Label（标签） | 彩色分类标签；至少记住 bug（红色）和 enhancement（浅蓝色） |
| 添加标签的位置 | Issue 详情页 → 右侧边栏 → "Labels" 齿轮图标 |
| Milestone（里程碑） | 把一组 Issue 归到一个版本目标下；个人项目选填 |
| Assignee（负责人） | 指定谁处理；个人项目选填，但要养成习惯 |
| PR 自动关闭 Issue 的魔法单词 | close/closes/closed, fix/fixes/fixed, resolve/resolves/resolved |
| 自动关闭写法 | 在 PR 描述中写 `Closes #编号`，合并后 Issue 自动关闭 |
| 为什么单人也要用 Issue | 人类短期记忆不可靠，Issue 是你的"外部大脑" |

> 🚀 下一步：你已经会创建 Issue、贴分类标签、用 PR 自动关闭 Issue 了。但到目前为止，你一直在操作自己的仓库。如果你找到一个别人写的好项目，想帮忙修 Bug——你没有那个项目仓库的写权限，怎么把修改贡献回去？这就叫 **Fork**，第6步要讲的内容。
