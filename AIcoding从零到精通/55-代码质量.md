## AI写的代码，到底能不能直接用？

**直接回答：能，但你不能不检查就上线。**

AI写的代码有三类：

| 类型 | 例子 | 能直接用吗？ |
|------|------|------------|
| 样板代码 | 搭一个Flask项目骨架、写一个通用的CRUD接口 | ✅ 基本可以，但还是要跑一遍 |
| 业务逻辑 | 用户登录验证、订单状态流转、支付计算 | ⚠️ 必须审查，可能有逻辑漏洞 |
| 安全/性能敏感 | 密码加密、权限控制、数据库查询优化 | ❌ 绝对不能直接用，必须人工审查 |

## AI代码的五大常见质量问题

### 问题一：缺少错误处理

AI写的代码，往往假设"一切顺利"。但真实世界不是这样的。

```python
# ❌ AI写的代码：没有错误处理
def read_file(path):
    with open(path, "r") as f:
        return f.read()

# 如果文件不存在 → 程序崩溃
# 如果文件是乱码 → 程序崩溃
# 如果文件太大 → 程序崩溃

# ✅ 应该有的样子：
def read_file(path):
    try:
        with open(path, "r", encoding="utf-8") as f:
            return f.read()
    except FileNotFoundError:
        print(f"错误：文件 {path} 不存在")
        return None
    except UnicodeDecodeError:
        print(f"错误：文件 {path} 编码不正确，请确认是UTF-8格式")
        return None
    except PermissionError:
        print(f"错误：没有权限读取文件 {path}")
        return None
```

**AI不会自动加错误处理。** 它只会给你"理想情况下的代码"。你需要自己思考：如果这里出错了，会发生什么？

### 问题二：硬编码

📖 **硬编码（Hardcoding）** = 把应该配置的东西直接写死在代码里。比如把密码、文件路径、服务器地址直接写在代码中。

```python
# ❌ AI写的代码：硬编码
def connect_database():
    return mysql.connector.connect(
        host="192.168.1.100",          # 硬编码IP
        user="admin",                  # 硬编码用户名
        password="MySecret123",        # 硬编码密码 ← 这是最危险的！
        database="production_db"       # 硬编码数据库名
    )

# ✅ 应该有的样子：使用配置文件或环境变量
import os
import json

def connect_database():
    config = json.load(open("config.json"))  # 配置从文件读取
    return mysql.connector.connect(
        host=os.environ.get("DB_HOST", "localhost"),
        user=os.environ.get("DB_USER", "root"),
        password=os.environ.get("DB_PASSWORD"),  # 密码从环境变量读取
        database=os.environ.get("DB_NAME", "mydb")
    )
```

**不仅是安全问题，硬编码还会让代码难以维护。** 换个服务器就要改代码——这是1980年代的编程风格。

### 问题三：性能问题

AI会写出"能跑但慢"的代码。

```python
# ❌ AI写的代码：性能差
# 在10000个用户中查找所有vip用户
users = get_all_users()  # 假设有10000条
vip_users = []
for user in users:
    if user["is_vip"]:
        vip_users.append(user)

# 然后又查一遍
for user in users:        # ← 又遍历了一遍！
    if user["level"] > 5:
        send_notification(user)

# 问题：同一个列表遍历了两遍，每遍10000次
# 如果数据量是100万，你就能感觉到慢了

# ✅ 更好的做法：合并遍历
vip_users = []
high_level_users = []
for user in users:
    if user["is_vip"]:
        vip_users.append(user)
    if user["level"] > 5:
        high_level_users.append(user)
# 遍历一遍，干两件事
```

另一个常见性能问题：AI喜欢写**嵌套循环**（循环里面套循环），这在数据量大的时候是性能杀手。

```python
# ❌ AI写的嵌套循环：O(n²) 复杂度
# 假设两个列表各有10000条数据
for order in orders:           # 10000次
    for product in products:   # 10000次 × 10000次 = 1亿次！
        if order["product_id"] == product["id"]:
            # 处理逻辑
            pass

# ✅ 应该用字典加速：O(n) 复杂度
product_dict = {p["id"]: p for p in products}  # 先建索引
for order in orders:  # 10000次
    product = product_dict.get(order["product_id"])  # 一次查找
    if product:
        # 处理逻辑
        pass
```

### 问题四：安全漏洞

这是最严重的问题。AI不会主动防御安全攻击。

**SQL注入（📖 SQL Injection）** = 攻击者在输入框里写SQL语句，篡改你的数据库查询，从而窃取或破坏数据。这是最古老、最常见、最危险的Web安全漏洞之一。

```python
# ❌ AI写的代码：SQL注入风险
user_input = request.args.get('username')  # 用户输入：' OR '1'='1
query = f"SELECT * FROM users WHERE username = '{user_input}'"
cursor.execute(query)
# 实际执行的SQL变成：
# SELECT * FROM users WHERE username = '' OR '1'='1'
# → 返回所有用户！攻击者拿到了全部数据！

# ✅ 安全做法：用参数化查询
user_input = request.args.get('username')
cursor.execute("SELECT * FROM users WHERE username = %s", (user_input,))
# 即使用户输入恶意内容，也会被当作普通字符串处理，不会执行
```

**XSS（📖 Cross-Site Scripting，跨站脚本攻击）** = 攻击者在网页上注入恶意JavaScript代码，当其他用户访问时，代码在浏览器中执行。可以窃取用户Cookie、重定向到钓鱼网站等。

```python
# ❌ AI写的代码：XSS风险
from flask import Flask, request

@app.route('/hello')
def hello():
    name = request.args.get('name', '')
    return f"<h1>你好, {name}!</h1>"
# 用户输入：<script>alert('你被黑了')</script>
# → 页面会执行这段JS代码！

# ✅ 安全做法：转义HTML
from flask import Flask, request, escape

@app.route('/hello')
def hello():
    name = request.args.get('name', '')
    return f"<h1>你好, {escape(name)}!</h1>"
# <script> 会被转义成 &lt;script&gt;，变成普通文本，不会执行
```

### 问题五：命名混乱、缺少注释

AI写的代码有时会使用它自己"觉得对"的变量名，但你看不懂。

```python
# ❌ AI写的代码：命名让人困惑
def f(x, y):
    z = x + y
    return z * 2

# f是什么意思？x和y是什么？z是什么？为什么不直接说清楚？

# ✅ 好的命名：看名字就知道在干什么
def calculate_double_sum(first_number, second_number):
    total = first_number + second_number
    return total * 2
```

## 代码质量检查清单（提交/部署前必查）

每次AI写完代码，你要上线之前，逐条检查这7项。全部打勾，才能上线。

```
1. [ ] 能跑通，功能正确
   → 运行代码，用几个测试用例验证结果

2. [ ] 没有硬编码的密钥
   → 搜索代码中的 key、token、password、secret 等关键词
   → 确认全是占位符或环境变量

3. [ ] 关键路径有错误处理
   → 数据库连接失败怎么办？网络超时怎么办？
   → 文件不存在怎么办？用户输入为空怎么办？
   → 每个"可能出错的地方"都有 try/except 或类似处理

4. [ ] 变量名、函数名命名清晰
   → 函数名：用动词开头（get_user, calculate_total, send_email）
   → 变量名：用名词（user_count, total_price, file_path）
   → 不要用：x, y, temp, data, info 这种模糊的名字

5. [ ] 有基本注释
   → 复杂逻辑有注释解释"为什么这么写"
   → 不写"做什么"（代码本身已经说明了），写"为什么"

6. [ ] 没有明显的性能问题
   → 有没有不必要的嵌套循环？
   → 有没有重复遍历同一个大列表？
   → 有没有在循环里做数据库查询？

7. [ ] 安全：没有已知漏洞
   → 有没有SQL注入风险？（搜索 f"SELECT ... {user_input}"）
   → 有没有XSS风险？（用户输入直接拼到HTML里？）
   → 有没有未验证的用户输入？（文件上传、表单提交等）
```

## 何时可以信任AI代码，何时必须人工审查

### ✅ 可以信任AI的场景

| 场景 | 为什么可以信任 |
|------|---------------|
| 样板代码（脚手架） | 结构固定，AI见过的模式非常多，不容易出错 |
| 单元测试 | 测试的结构是固定的，AI很擅长写测试 |
| 文档/注释 | AI善于用自然语言描述代码 |
| 数据格式转换 | JSON转CSV、XML转JSON等——模式固定 |
| 正则表达式 | AI写正则的能力很强，但还是要跑一下验证 |

### ⚠️ 必须人工审查的场景

| 场景 | 为什么必须审查 |
|------|---------------|
| 安全相关代码 | 密码加密、权限控制、认证流程——AI不懂安全 |
| 核心业务逻辑 | 支付计算、订单状态机——错了就是真金白银 |
| 性能敏感代码 | 大数据处理、高并发场景——AI不会考虑性能 |
| 数据库操作 | SQL注入风险、慢查询——AI不会帮你优化 |
| 与外部API交互 | 错误处理、重试逻辑、超时——AI考虑不周全 |

## 自测

1. 你用AI写了一个用户注册接口，准备上线。请用7项检查清单逐项检查，每项至少说出一条具体的检查内容。

2. 下面这段AI生成的代码有什么问题？至少找出3个。

```python
def get_user(name):
    db = connect("mydb", "admin", "pass123")
    query = "SELECT * FROM users WHERE name = '" + name + "'"
    return db.execute(query)
```

3. "AI写的代码看起来都很合理，所以不需要检查"——用你自己的话，反驳这个观点。

> **答案**：
> 1. ①能跑通：用Postman或curl测试注册接口，传正常数据和异常数据各测一次。②无硬编码密钥：搜索代码中是否有 password、token、secret 等字样。③错误处理：用户名重复怎么处理？邮箱格式不正确怎么处理？④命名清晰：函数名是否是 register_user 而不是 f 或 do？⑤有注释：复杂验证逻辑是否有注释说明规则。⑥无性能问题：密码加密是否用了bcrypt（而不是慢的算法）。⑦安全：是否有SQL注入风险？密码是否明文存储？是否有XSS风险？
> 2. 至少三个问题：①硬编码——数据库密码 `"pass123"` 直接写在代码里。②SQL注入——用字符串拼接SQL，`name` 参数未经任何过滤。③缺少错误处理——数据库连接失败怎么办？查询失败怎么办？④命名不清晰——函数名 `get_user` 和参数 `name` 尚可，但 `db` 变量名模糊。
> 3. "看起来合理"不等于"真的对"。AI的核心是"猜下一个字"，它输出的代码是"看起来最像正确代码的代码"，而不是"经过验证的正确代码"。AI不懂你的业务逻辑，不懂安全规范，不懂性能要求。你必须审查——因为代码出问题，负责的是你，不是AI。
