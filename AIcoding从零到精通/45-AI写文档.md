## 45.1 程序员最讨厌什么？写文档

如果给程序员做一个"最讨厌的工作"排行榜，**写文档**大概率排前三。

原因很简单：写代码是有创造性的、有趣的；写文档是重复性的、无聊的。而且写完之后，代码一改，文档就过时了——又要重新写。

**但AI编程时代，这不再是问题。** 因为AI最擅长做的，就是这种"无聊但需要条理"的工作。

## 45.2 代码注释：让AI帮你解释代码

> **📖 术语：注释（Comment）**——代码中不会被执行，但给人看的文字。Python中用 `#` 开头，可以写在任何地方。好的注释解释"为什么这么写"，而不是"这行代码做了什么"（后者看代码本身就能知道）。

### 场景1：给一段代码加注释

选中代码，发给AI：

```
给下面这段代码加上清晰的中文注释，解释每一部分的逻辑：
```

```python
def process_orders(orders):
    result = {}
    for order in orders:
        status = order.get('status', 'unknown')
        if status == 'paid':
            if order['user_id'] not in result:
                result[order['user_id']] = []
            result[order['user_id']].append(order['amount'])
    return result
```

AI会生成：

```python
def process_orders(orders):
    """处理订单列表，按用户ID分组已支付的订单金额"""
    # 创建一个空字典，用于存储结果
    # key: 用户ID, value: 该用户所有已支付订单的金额列表
    result = {}
    
    # 遍历每个订单
    for order in orders:
        # 获取订单状态，如果订单中没有status字段，默认为'unknown'
        status = order.get('status', 'unknown')
        
        # 只处理已支付的订单
        if status == 'paid':
            # 如果这个用户还没在结果中，先创建一个空列表
            if order['user_id'] not in result:
                result[order['user_id']] = []
            # 把订单金额加到该用户的列表中
            result[order['user_id']].append(order['amount'])
    
    return result
```

### 场景2：看不懂一段代码，让AI解释

选中代码，在Chat里输入：

```
这段代码是做什么的？用大白话解释一下，假设我是一个刚学编程的人。
```

AI会用通俗的语言解释这段代码的逻辑，比你看注释快得多。

## 45.3 README生成：让AI帮你写项目介绍

> **📖 术语：README**——项目根目录下的 `README.md` 文件，通常是别人打开你的项目时看到的第一个文件。包含项目介绍、安装方法、使用说明、示例等。名字来源于"read me"（读我），相当于项目的"使用说明书"。

### 一键生成README

在你的项目文件夹里，打开AI对话，输入：

```
帮我给这个项目写一个 README.md，包含以下内容：
1. 项目名称和一句话介绍
2. 功能列表
3. 安装步骤（包括依赖）
4. 快速开始（一个最小使用示例）
5. 项目结构说明
6. 技术栈

请先扫描项目文件，了解项目结构后再写。
```

AI会读取你的项目文件，理解项目结构，然后生成一份完整的README。

### 案例：一个简单的天气查询工具

假设你的项目是一个天气查询脚本，只有一个 `weather.py` 文件。

AI生成的README大概长这样：

```markdown
# Weather Query Tool

一个简单的命令行天气查询工具，输入城市名，返回当前天气。

## 功能

- 查询任意城市的实时天气
- 显示温度、湿度、天气描述
- 支持中英文城市名

## 安装

```bash
git clone https://github.com/yourname/weather-tool.git
cd weather-tool
pip install -r requirements.txt
```

## 快速开始

```bash
python weather.py Beijing
```

输出示例：
```
北京当前天气：晴
温度：25°C
湿度：45%
```

## 项目结构

```
weather-tool/
├── weather.py          # 主程序
├── requirements.txt    # 依赖列表
└── README.md          # 本文件
```

## 技术栈

- Python 3.12
- requests 库（HTTP请求）
- OpenWeatherMap API
```

**你不需要写一个字，但你需要审核**：项目名称对吗？安装步骤能跑通吗？示例输出和实际一致吗？

## 45.4 API文档：让AI帮你写接口文档

如果你的项目有API接口（比如第49章会做的Todo应用后端），AI可以帮你写API文档。

### Prompt模板

```
帮我给下面这些API接口写文档，每个接口包含：
1. 接口路径和方法（GET/POST/PUT/DELETE）
2. 请求参数（参数名、类型、是否必填、说明）
3. 返回示例（JSON格式）
4. 可能的错误码

接口代码如下：
[粘贴你的API代码]
```

AI会生成一份结构清晰的API文档，比你手写快10倍。

## 45.5 文档维护：代码改了，文档也要同步

**这是AI写文档最大的优势**：传统编程中，代码改了，文档经常忘记更新。AI编程中，你可以：

```
我刚刚修改了 weather.py 的 get_weather 函数，现在它多返回一个 humidity 字段。请帮我更新 README.md 中的"输出示例"部分。
```

AI会找到README中对应的部分，更新为新的输出格式。**代码和文档永远不会"脱节"**——因为你随时可以让AI重新同步。

---

## 45.6 文档不是越多越好

记住一个原则：**好的文档是"刚好够用"，不是"面面俱到"。**

| 应该写文档 | 可以不写文档 |
|-----------|-------------|
| 项目的安装和使用方法 | 每个函数的内部实现细节 |
| API接口的输入输出格式 | 显而易见的代码逻辑 |
| 配置说明（环境变量、依赖） | 会频繁变化的部分 |
| 常见问题和排错指南 | 只有你自己用的内部工具 |

**一句话**：写别人会问的问题的答案，不要写别人不会看的废话。

## 自测

1. 好的注释解释"为什么这么写"还是"这行代码做了什么"？为什么？
2. 让AI生成README时，为什么最好让它先扫描项目文件？
3. 代码改了之后，文档怎么同步更新？AI编程时代有什么优势？

> **答案**：
> 1. 解释"为什么这么写"。因为"这行代码做了什么"在代码本身就能看到（比如 `total += score` 就是累加），不需要注释重复一遍。但"为什么跳过非数字项"这种设计意图，看代码不知道，需要注释说明。
> 2. 因为AI需要了解项目结构、文件内容、依赖关系，才能写出准确的README。如果它不知道项目里有什么文件，可能会编造内容。
> 3. 把修改后的代码和需要更新的文档部分一起发给AI，让它同步更新。AI编程时代的优势是：你随时可以重新生成文档，不用手动维护，代码和文档永远不会脱节。
