# 第01章 · 计算机的大脑——CPU

> "你写的每一行Java代码，最终都会变成CPU能听懂的'嘀嗒声'——1和0。这一章，我们看看这个翻译过程是怎么发生的。"

## 01.1 CPU是什么

CPU（Central Processing Unit，中央处理器）是计算机的"大脑"。它只做一件事：**执行指令**。一秒能执行几十亿条。

你的电脑里有一颗CPU，你的手机里也有一颗，AWS云服务器上那颗也是。对Java程序员来说，CPU是最终运行你代码的硬件——你的 `.java` 文件经历了层层翻译，最终变成CPU认识的机器码。

## 01.2 从.java到机器码：Java的编译链

这是Java最独特的特性之一。我们来看一段最简单的Java代码经历了什么：

```java
// 文件名：Hello.java
public class Hello {
    public static void main(String[] args) {
        System.out.println("你好，世界！");
    }
}
```

**第一步：javac编译（.java → .class）**

```bash
javac Hello.java    # 生成 Hello.class 文件
```

`javac` 是Java编译器。它把人类能读懂的Java源码编译成**字节码（bytecode）**。字节码不是机器码——CPU不认识它。

如果你用十六进制编辑器打开 `Hello.class`，会看到以 `CA FE BA BE`（咖啡宝贝）开头的二进制数据。这四个字节是Java字节码的"魔数"，JVM用它来识别"这是一个合法的class文件"。

**第二步：JVM解释/编译执行（.class → 机器码）**

```bash
java Hello    # JVM加载Hello.class并执行
```

JVM（Java Virtual Machine，Java虚拟机）才是真正跟CPU打交道的东西。它读取字节码，翻译成当前操作系统和CPU能理解的机器码，然后交给CPU执行。

```
.java 源文件
    │  javac 编译
    ▼
.class 字节码（平台无关）
    │  JVM 加载并执行
    ▼
机器码（平台相关）
    │  CPU 执行
    ▼
"你好，世界！"
```

## 01.3 为什么Java要"绕一圈"？

其他语言（比如C语言）是直接编译成机器码的：

```
C源码 → gcc编译 → 机器码（只能在当前OS/CPU上运行）
```

Java多了一层字节码和JVM。这不是"绕远路"，而是一个精妙的设计：

| | C语言 | Java |
|---|------|------|
| 编译产物 | 平台相关的机器码 | 平台无关的字节码 |
| 跨平台 | 需要为每个平台重新编译 | 一次编译，到处运行 |
| 执行方式 | CPU直接执行机器码 | JVM解释/编译字节码 |
| 性能 | 启动快，无中间层 | JIT编译后可达相近性能 |

> 🤔 想多一点：Java的口号"Write Once, Run Anywhere"（一次编写，到处运行），核心秘密就在这里。你在Windows上编译出的 `Hello.class`，可以直接复制到Linux服务器上运行——只要那台服务器装了对应版本的JVM。部署时你不需要重新编译。

## 01.4 JVM里的JIT编译

JVM不是单纯地"逐条解释"字节码。它有一个叫 **JIT（Just-In-Time，即时编译）** 的组件。

JIT的工作方式像一个聪明的翻译：
- 发现某段代码被频繁执行（比如一个循环跑了1万次），JIT就把这段字节码直接编译成机器码缓存起来
- 下次再遇到这段代码，直接用缓存的机器码，不再逐条解释
- 这就是为什么Java程序经常"越跑越快"

```java
// 这段代码的循环体会被JIT编译优化
for (int i = 0; i < 10_000; i++) {
    processItem(i);  // JIT发现这个循环跑了很多次，编译成机器码
}
```

## 01.5 对Java程序员意味着什么？

你不用直接跟CPU打交道，但理解CPU对Java开发有帮助：

1. **了解JVM的内存模型**（后面章节会讲）需要理解CPU缓存
2. **多线程编程**的本质是充分利用多核CPU
3. **性能调优**时，你需要知道你的代码在CPU层面到底在做什么

举个简单例子——你的Spring Boot服务部署在一台4核服务器上：

```yaml
# application.yml
server:
  tomcat:
    threads:
      max: 200    # Tomcat线程池最大200个线程
```

这200个线程并不是同时运行的。4核CPU同一时刻只能真正并行执行4个线程。其余196个线程在排队等待操作系统调度。理解这一点，你就不会盲目把线程数调得很大。
