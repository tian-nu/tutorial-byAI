## 一、外键（Foreign Key）：建立表之间的"关联字段"

上一章我们看到了 `orders.user_id` 如何关联到 `users.id`。在数据库里，这个"关联字段"有一个正式的名字：**外键（Foreign Key）** *此术语见附录E*。

### 外键是什么？

外键就是**一张表里的一列，它存的是另一张表的主键值**。

```
users 表（主键）              orders 表（外键）
┌────┬──────────┐            ┌────┬─────────┬──────────┐
│ id │ username │            │ id │ user_id │ product  │
├────┼──────────┤            ├────┼─────────┼──────────┤
│ 1  │ alice    │◄───────────│ 1  │    1    │ iPhone   │
│ 2  │ bob      │            │ 2  │    1    │ iPad     │
│ 3  │ charlie  │            │ 3  │    2    │ MacBook  │
└────┴──────────┘            └────┴─────────┴──────────┘
  主键 (Primary Key)            外键 (Foreign Key)
  users.id                       orders.user_id
```

- `users.id` 是**主键**：在 users 表内部唯一标识每一行。
- `orders.user_id` 是**外键**：在 orders 表里，它存的是 `users.id` 的值，用来"引用"（关联）到 users 表。

### 外键的作用

外键帮你回答"这个订单是谁下的？"：

```
订单 id=1 的 user_id=1 → 去 users 表找 id=1 → alice
订单 id=3 的 user_id=2 → 去 users 表找 id=2 → bob
```

如果把主键比作"身份证号"，那外键就是"紧急联系人"那一栏——它不存那个人的全部信息，只存他的身份证号，需要时可以通过身份证号找到他。

### 外键约束（一个预告）

数据库可以设置一个规则："如果 users 表里没有 id=999 这个人，那么 orders 表里也不能有 user_id=999 的订单"。这叫**外键约束**——保证你不会引用一个不存在的人。第 16 章会详细讲。

## 二、索引（Index）：加速查找的"目录"

### 先给你一个直观感受

假设 users 表有 1000 万行数据。执行以下两条查询：

```sql
-- 查询 A：按主键查找
SELECT * FROM users WHERE id = 5000000;
-- 结果：瞬间返回（毫秒级）

-- 查询 B：按用户名查找
SELECT * FROM users WHERE username = 'zhangsan';
-- 结果：等了好几秒才返回
```

为什么查询 A 这么快，查询 B 这么慢？

因为 `id` 是主键，数据库自动为主键创建了**索引（Index）** *此术语见附录E*。而 `username` 列默认没有索引，所以数据库不得不**一行一行地扫描**（全表扫描），直到找到 `username = 'zhangsan'` 的那一行。

### 索引是什么？

**索引就像一本书的目录**。你想找"第 5 章"的内容：
- 没有目录：从第 1 页翻到第 500 页，一行一行找。
- 有目录：翻到目录，看到"第 5 章 → 第 120 页"，直接跳到第 120 页。

数据库的索引也是这个原理：它维护了一个"键值 → 位置"的映射表，让你能快速定位数据。

### 索引的代价（先知道有这回事）

索引不是免费的：
- **占空间**：索引本身也要占磁盘空间。
- **拖慢写入**：每次插入、更新、删除数据时，索引也要同步更新。

所以不是"给所有列都建索引更好"——你只在**经常用于查询的列**上建索引。

> **想多一点**：索引的内部实现是一棵 B+ 树，这是数据库最精妙的设计之一。第 17 章会给你拆开讲。现在你只需要记住一个直觉：**主键自动有索引，所以按主键查最快；其他列默认没有索引，查起来慢。**

## 三、约束（Constraint）：数据的"交通规则"

**约束（Constraint）** *此术语见附录E* 是数据库层面的"规则"，用来保证数据质量。就像交通规则防止车祸，约束防止"脏数据"进入数据库。

常见的约束有：

| 约束类型 | 作用 | 例子 |
|---------|------|------|
| **NOT NULL** | 这列不能为空 | `email` 不能是空的，否则怎么联系用户？ |
| **UNIQUE** | 这列的值不能重复 | 两个用户不能用同一个 `email` |
| **PRIMARY KEY** | 主键 = NOT NULL + UNIQUE | `id` 不能空、不能重复 |
| **FOREIGN KEY** | 外键值必须在另一张表的主键中存在 | `orders.user_id` 必须在 `users.id` 里存在 |
| **CHECK** | 自定义条件 | `age >= 0 AND age <= 150`（年龄不能是负数） |
| **DEFAULT** | 默认值 | `created_at` 默认填当前时间 |

### 约束是多余的吗？

❌ 错误认知："约束是多余的，我在代码里判断就行了。"

✅ 正确认知：代码层面的判断可以绕过。比如：
- 你写了一个 API 接口，做了参数校验。
- 但有人直接连上数据库，手动插入了一条 `email = NULL` 的用户。
- 如果有 `NOT NULL` 约束，数据库会直接拒绝这条插入，无论你用什么方式连接。

**约束是数据库的最后一道防线。** 代码会变、会出 Bug，但约束永远生效。

## 四、SQL：与数据库对话的语言

**SQL（Structured Query Language，结构化查询语言）** *此术语见附录E* 是你和数据库沟通的语言。你想做什么，就用 SQL 写出来，数据库执行。

SQL 不是一种编程语言（你不能用它写后端逻辑），它只做一件事：**操作数据**。

### SQL 的 5 个子语言（顶层地图）

所有 SQL 语句都可以归入以下 5 类。把这张图记住——它是后续所有 SQL 章节的导航图：

```
                           SQL
                            │
        ┌───────┬───────────┼───────────┬──────────┐
        │       │           │           │          │
      DDL     DML         DQL         DCL        TCL
    数据定义  数据操作     数据查询     数据控制    事务控制
        │       │           │           │          │
   CREATE    INSERT      SELECT      GRANT      BEGIN
   ALTER     UPDATE                  REVOKE     COMMIT
   DROP      DELETE                             ROLLBACK
```

以下是每一类的详细说明：

### 1. DDL（Data Definition Language，数据定义语言）*此术语见附录E*

**做什么**：定义数据库的"骨架"——建库、建表、改表结构、删表。

```sql
CREATE DATABASE my_shop;            -- 创建数据库
CREATE TABLE users (id INT, ...);   -- 创建表
ALTER TABLE users ADD COLUMN age;   -- 修改表结构（加一列）
DROP TABLE users;                   -- 删除表（危险！）
```

**关键词**：CREATE、ALTER、DROP。

### 2. DML（Data Manipulation Language，数据操作语言）*此术语见附录E*

**做什么**：操作表里的数据——增、删、改。

```sql
INSERT INTO users (username, email) VALUES ('alice', 'alice@example.com');  -- 插入
UPDATE users SET email = 'new@example.com' WHERE id = 1;                     -- 更新
DELETE FROM users WHERE id = 1;                                              -- 删除
```

**关键词**：INSERT、UPDATE、DELETE。

### 3. DQL（Data Query Language，数据查询语言）*此术语见附录E*

**做什么**：查询数据——这是你用得最多的操作。

```sql
SELECT * FROM users;                          -- 查所有用户
SELECT username, email FROM users WHERE id = 1;  -- 查指定用户
```

**关键词**：SELECT（只有一个，但它的用法极其丰富）。

### 4. DCL（Data Control Language，数据控制语言）*此术语见附录E*

**做什么**：控制谁能访问什么数据——权限管理。

```sql
GRANT SELECT ON my_shop.* TO 'alice'@'localhost';  -- 授予权限
REVOKE INSERT ON my_shop.* FROM 'alice'@'localhost'; -- 收回权限
```

**关键词**：GRANT、REVOKE。

### 5. TCL（Transaction Control Language，事务控制语言）*此术语见附录E*

**做什么**：控制事务——一组操作要么全成功、要么全失败。

```sql
BEGIN;                    -- 或 START TRANSACTION; 开始事务
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- 扣钱
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- 加钱
COMMIT;                   -- 确认：都成功了，保存
-- 或 ROLLBACK;           -- 撤销：出错了，全部回滚
```

**关键词**：BEGIN / START TRANSACTION、COMMIT、ROLLBACK。

> **想多一点**：很多人学 SQL 时，只意识到 DDL、DML、DQL 的存在，忽略了 DCL 和 TCL。但实际工作中，DCL（权限管理）和 TCL（事务）是生产环境的关键。一个没有 TCL 的数据库就像没有刹车系统的车——一次操作失误就可能造成灾难。

## 不对怎么办？

### 常见错误 1：外键和索引混淆

❌ 错误认知：外键就是索引，加了外键查询就快了。

✅ 正确认知：外键和索引是两个**完全独立**的概念。
- **外键**：逻辑概念，表示"这列的值对应另一张表的主键"。它管的是**数据关系**。
- **索引**：物理概念，表示"为这列建一个快速查找结构"。它管的是**查询速度**。

比如，`orders.user_id` 是外键（它关联到 users 表），但如果没给 `user_id` 建索引，按 `user_id` 查订单仍然很慢。**外键不自动创建索引**（MySQL 的 InnoDB 引擎实际上会为外键自动创建索引，但这是引擎实现细节，逻辑上它们仍是两个概念）。

### 常见错误 2：认为约束是多余的

❌ 错误认知："我在代码里判断了 email 不为空，数据库再加 NOT NULL 约束多余。"

✅ 正确认知：代码层面的校验和数据库层面的约束是**两层防线**。
- 代码校验：对用户友好，能在前端就提示错误。
- 数据库约束：对数据安全负责，任何代码绕过它都不行（比如数据库管理员手动操作、另一个服务直接连接数据库、Bug 导致的校验遗漏）。

**你的代码是你写的，数据库约束是数据库的，两者互不替代。**

### 常见错误 3：SQL 分类记不住

❌ 错误认知："我只要会写 SELECT 就行了，管它叫什么 DQL。"

✅ 正确认知：知道分类不是"背名词"，而是帮你**建立知识框架**。当你遇到一个新需求时：
- "我要建表" → 翻 DDL 章节
- "我要批量更新" → 翻 DML 章节
- "我要复杂查询" → 翻 DQL 章节
- "我要权限管理" → 翻 DCL 章节
- "我要保证数据一致性" → 翻 TCL 章节

分类让你知道"去哪里找答案"。
