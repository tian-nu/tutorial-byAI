## F.1 计算机底层

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **CPU** | 计算机的"大脑"，只会做算术和逻辑判断，但做得极快（每秒几十亿次） | 第01章 |
| **寄存器** | CPU内部的超快存储单元，比内存快几百倍，但数量极少（几十个），相当于你办公桌上的便签纸 | 第01章 |
| **二进制** | 只有0和1的数字系统。计算机底层一切——文字、图片、视频——本质上都是一串0和1 | 第01章 |
| **汇编语言** | 比机器码（纯0101）可读一点的语言，每条指令直接对应CPU的一个操作，是"最低级"的编程语言 | 第01章 |
| **机器码** | CPU能直接"吃"的指令，纯二进制。你写的任何高级语言代码，最终都要变成机器码才能运行 | 第01章 |
| **编译** | 把人类能看懂的代码翻译成机器能看懂的机器码，这个过程叫编译。Java的编译分两步：javac把.java变成.class（字节码），JVM再把字节码变成机器码 | 第01章 |
| **编译器** | 执行"编译"这个翻译工作的工具。Java的编译器叫`javac`，C的编译器叫`gcc` | 第01章 |
| **内存（RAM）** | 计算机的"临时工作台"，运行中的程序和数据都放在这里。断电就清空，速度快 | 第02章 |
| **内存地址** | RAM里每个存储位置的编号，从0开始。就像小区里每套房子的门牌号 | 第02章 |
| **堆（Heap）** | 内存中一块"公共区域"，所有对象都存在这里。Java中用`new`创建的对象都在堆上，由GC自动回收 | 第02章 |
| **栈（Stack）** | 内存中一块"临时便签区"，方法调用时自动分配局部变量，方法返回自动释放。每个线程有自己的栈 | 第02章 |
| **硬盘（HDD/SSD）** | 计算机的"仓库"，存文件、照片、数据库。断电数据还在，但比内存慢几百到几千倍 | 第03章 |
| **文件系统** | 硬盘上组织数据的方式。Windows用C:\这样的盘符，Linux/Unix用/这样的树形结构 | 第03章 |
| **磁盘IO** | 程序读写硬盘的过程。比内存读写慢很多，所以有了缓存、索引等优化手段 | 第03章 |
| **操作系统（OS）** | 硬件和软件之间的"翻译官+管家"。管理CPU、内存、硬盘，提供程序运行的环境。你用的Windows、macOS、Linux都是操作系统 | 第04章 |
| **进程** | 一个正在运行的程序实例。每个进程有自己独立的内存空间，互不干扰。启动一个Java程序就是启动一个JVM进程 | 第04章 |
| **线程** | 进程内部的一条"执行流"。一个进程可以有多个线程，共享同一块内存。相当于一个厨房里的多个厨师 | 第04章 |
| **并发** | 一个人同时处理多件事——快速切换，看起来像同时。比如烧水的同时泡面，来回切换 | 第04章 |
| **并行** | 多个人同时处理多件事——真正的同时。比如两个人分别烧水和泡面 | 第04章 |
| **上下文切换** | CPU从一个线程切换到另一个线程时，要保存当前状态、加载新状态，这个开销叫上下文切换。切换太多会变慢 | 第04章 |
| **内核** | 操作系统的核心。负责跟硬件直接打交道的最底层部分。你写的代码不能直接访问硬件，必须通过内核 | 第04章 |
| **系统调用** | 你的程序请求内核办事的方式。比如"我要读这个文件"——程序发起系统调用，内核去操作硬盘 | 第108章 |
| **32位/64位** | CPU一次能处理的数据位数。32位CPU最多只能访问4GB内存，所以现在都用64位。JDK要选64位版本 | 第01章 |

## F.2 Java语言核心概念

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **JVM** | Java虚拟机，Java程序的"翻译官+管家"。把字节码翻译成机器码然后运行，同时负责内存管理（GC） | 第15章 |
| **JDK** | Java开发工具包，包含JVM、核心类库、编译器等开发工具。你要写Java代码，就必须装JDK | 第15章 |
| **JRE** | Java运行时环境，只包含JVM和核心类库，不含编译器。用户只运行你的程序就装JRE就够了。JDK包含JRE | 第15章 |
| **字节码（bytecode）** | .class文件的内容。它不是机器码，而是JVM的"中间语言"——比.java更接近机器，但还需要JVM翻译才能执行 | 第15章 |
| **类（class）** | Java程序的基本组织单位。一切代码都在类里面。类就像"蓝图"——定义了对象长什么样、能做什么事 | 第22章 |
| **对象（object）** | 根据类（蓝图）造出来的"实例"。`new Car()`造出来的那辆具体的车就是对象 | 第22章 |
| **构造器（constructor）** | 创建对象时自动调用的特殊方法，和类同名。`new Car("红色")`里的`Car("红色")`就是在调用构造器，负责初始化对象 | 第22章 |
| **封装（encapsulation）** | 把数据（字段）和操作数据的方法打包在一起，用`private`隐藏内部细节，只暴露必要的API。就像遥控器——你按按钮就行，不用知道里面电路怎么走 | 第22章 |
| **继承（inheritance）** | 一个类可以"继承"另一个类的字段和方法。`class Dog extends Animal`——Dog自动拥有Animal的所有能力，还能添加自己的特色 | 第23章 |
| **多态（polymorphism）** | 同一个方法名，不同的对象表现出不同的行为。`Animal a = new Dog(); a.speak()`——虽然a的类型是Animal，但实际调用的是Dog的speak方法 | 第24章 |
| **抽象类（abstract class）** | 不能直接`new`的类，只能被继承。它定义"骨架"，子类填充"血肉"。比如"动物"太抽象了，不能直接造一个动物，但可以造猫和狗 | 第25章 |
| **接口（interface）** | 一份"合同"——定义了一组方法签名，实现它的类必须提供这些方法的具体实现。`class Dog implements Runnable`——Dog必须实现run方法 | 第25章 |
| **方法重写（override）** | 子类重新定义父类已有的方法。`@Override`注解标记，方法签名必须完全一致 | 第23章 |
| **方法重载（overload）** | 同一个类里多个同名方法，但参数不同（个数/类型/顺序）。编译器根据你传的参数自动选择调用哪个 | 第22章 |
| **枚举（enum）** | 限定取值范围的特殊类。`enum Season { SPRING, SUMMER, AUTUMN, WINTER }`——Season只能是这四个之一 | 第26章 |
| **record** | Java 14新增的"数据类"，专门用来存数据。一行代码自动生成构造器、getter、equals、hashCode、toString | 第26章 |
| **包装类（wrapper）** | 为基本类型（int/double等）提供对应的对象类型。`int → Integer`，`double → Double`。因为集合（ArrayList等）只能装对象，不能装基本类型 | 第28章 |
| **自动装箱/拆箱** | Java自动在基本类型和包装类之间转换。`Integer x = 5`自动把int 5装箱成Integer对象；`int y = x`自动拆箱 | 第28章 |
| **异常（exception）** | 程序运行时出了"意外"。比如文件不存在、网络断了、数组越界。用try-catch来"兜底"处理，不让程序直接崩溃 | 第29章 |
| **泛型（generics）** | 让类和方法可以处理"任意类型"而不丢失类型安全。`ArrayList<String>`——这个列表只能放String，放别的类型编译器会报错 | 第30章 |
| **类型擦除（type erasure）** | Java泛型的实现方式——编译后`ArrayList<String>`和`ArrayList<Integer>`在字节码层是一样的，泛型信息被"擦除"了。这是为了兼容老版本Java | 第30章 |
| **Lambda表达式** | 一种简洁的写法，把"一段代码"当作参数传递。`(x, y) -> x + y`就是一个Lambda。它让你可以像传递数据一样传递行为 | 第35章 |
| **方法引用** | Lambda的简化写法。`String::length`等价于`s -> s.length()`。用于当Lambda只是调用一个已有方法时 | 第35章 |
| **函数式接口** | 只有一个抽象方法的接口，可以用Lambda实现。`Runnable`、`Comparator`、`Predicate`都是函数式接口 | 第25章 |
| **Stream API** | 像"流水线"一样处理数据集合。`list.stream().filter(...).map(...).collect(...)`——数据从一头流进去，经过过滤、转换、收集，从另一头流出来 | 第36章 |
| **Optional** | 一个"盒子"——里面可能有值也可能是空的。用来代替null，避免NullPointerException。强迫你显式处理"空"的情况 | 第36章 |
| **注解（annotation）** | 给代码贴"标签"。`@Override`告诉编译器"我在重写方法，帮我检查"；`@Autowired`告诉Spring"帮我注入这个依赖"。注解本身不改变代码行为，是工具或框架读取注解后做出反应 | 第43章 |
| **反射（reflection）** | 程序在运行时"照镜子"——检查自己的结构。"我这个类有哪些方法？哪些字段？"强大但慢，框架（Spring、JUnit）大量使用 | 第43章 |
| **动态代理** | 在运行时动态创建"代理对象"，拦截方法调用。AOP的底层就是动态代理——Spring在你不注意的时候，偷偷给你的Bean包了一层代理 | 第43章 |
| **垃圾回收（GC）** | JVM自动清理不再使用的对象，释放内存。你只管new对象，不用管什么时候删除——JVM会定期"扫垃圾" | 第42章 |
| **Maven** | Java项目的构建工具。管理依赖（你的项目需要哪些第三方库）、编译、测试、打包（打成jar文件）——一条龙服务 | 第15章 |
| **pom.xml** | Maven项目的配置文件。写清楚项目叫什么、依赖哪些库、用什么Java版本——都在这个文件里 | 第15章 |
| **static** | 表示"属于类，不属于某个对象"。`Math.PI`就是static的——所有对象共享同一个PI值，不需要创建对象就能用 | 第16章 |
| **final** | 表示"不可变"。`final int x = 5`——x的值不能改；`final class`——这个类不能被继承；`final`方法——子类不能重写 | 第17章 |
| **synchronized** | Java的"锁"，保证同一时刻只有一个线程能执行被它标记的代码。防止多线程同时修改数据导致混乱 | 第39章 |
| **volatile** | 保证多线程之间变量的可见性——一个线程修改了volatile变量，其他线程立刻能看到新值。但只保证可见性，不保证原子性 | 第39章 |
| **线程池** | 预先创建好一批线程放着，有任务来了直接拿一个用，用完还回去。避免反复创建销毁线程的开销 | 第40章 |
| **虚拟线程（Virtual Thread）** | Java 21的重磅特性。一种"轻量级线程"，创建成本极低，可以同时跑几十万个。由JVM调度，不直接映射到操作系统线程 | 第41章 |
| **内部类** | 定义在另一个类内部的类。静态内部类、成员内部类、局部内部类、匿名内部类——四种形式，各有各的用途 | 第26章 |
| **匿名内部类** | 没有名字的"一次性"类，创建时当场定义。`new Runnable() { public void run() {...} }`。Lambda出现后，很多场景用Lambda替代了匿名内部类 | 第26章 |

## F.3 Spring生态

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **IoC（控制反转）** | "不要自己找依赖，等着别人给你"。你不再自己new对象，而是告诉Spring"我需要一个UserService"，Spring帮你创建并注入。控制权从"你"反转到了"Spring" | 第71章 |
| **DI（依赖注入）** | IoC的实现方式。Spring把对象需要的依赖（比如UserService需要UserRepository）自动"注入"进去。你只管声明，Spring负责快递到家 | 第73章 |
| **AOP（面向切面编程）** | 把"横切"的关注点（日志、事务、权限检查）从业务代码中抽离出来。比如给所有Service方法自动加日志——不用在每个方法里写`log.info(...)`，写一次就够了 | 第79章 |
| **Bean** | Spring容器管理的对象。你写的`@Service`、`@Repository`、`@Controller`标记的类，Spring都会创建它们的实例（Bean）并管理它们的整个生命周期 | 第73章 |
| **@Component** | 告诉Spring"这个类给我管，帮我创建Bean"。`@Service`、`@Repository`、`@Controller`都是`@Component`的"特殊版本"，语义更明确 | 第73章 |
| **@Service** | 标记业务逻辑层的Bean。`@Component`的语义化子类，告诉读代码的人"这个类负责业务逻辑" | 第73章 |
| **@Repository** | 标记数据访问层的Bean。Spring会自动把数据库相关异常翻译成Spring的`DataAccessException` | 第73章 |
| **@Controller** | 标记控制器层的Bean，处理HTTP请求。配合`@RequestMapping`使用 | 第75章 |
| **@RestController** | `@Controller` + `@ResponseBody`的合体。每个方法的返回值直接作为JSON写入HTTP响应体——这是REST API最常用的注解 | 第75章 |
| **@Autowired** | "帮我注入这个Bean"。Spring会自动找到匹配类型的Bean，注入到被标记的字段或构造器 | 第73章 |
| **@RequestMapping** | 把HTTP请求映射到Java方法。`@RequestMapping("/users")`表示"访问/users时调用这个方法" | 第75章 |
| **@GetMapping / @PostMapping** | `@RequestMapping`的简写版本。`@GetMapping("/users")`等价于`@RequestMapping(method=GET, path="/users")` | 第75章 |
| **Spring MVC** | Spring的Web框架，基于MVC（Model-View-Controller）模式。你把请求发给Controller，Controller操作Model（数据），返回View（或JSON）给客户端 | 第75章 |
| **Filter** | Java Servlet的"守门员"，在请求到达Controller之前和响应返回客户端之前执行。Spring Security就是一堆Filter组成的 | 第78章 |
| **Interceptor** | Spring的"拦截器"，比Filter更贴近Spring生态。在Controller方法执行前后做处理，可以拿到Handler信息 | 第78章 |
| **@Transactional** | 声明式事务管理。标了这个注解的方法，Spring会自动开启事务、提交或回滚。你不需要手动写`begin/commit/rollback` | 第82章 |
| **JPA** | Java持久化API，一套ORM标准。定义了一套接口，Hibernate是这个标准最流行的实现 | 第67章 |
| **Entity** | JPA的"实体类"，对应数据库中的一张表。`@Entity class User`——这个User类的实例就对应users表中的一行 | 第67章 |
| **Spring Security** | Spring的安全框架。处理认证（你是谁）和授权（你能做什么）。它实际上是一串Filter，拦截每个请求做安全检查 | 第87章 |
| **JWT** | JSON Web Token，一种"自包含"的令牌。用户登录后服务器发一个JWT，之后每次请求带着它就行——服务器不用查数据库就知道你是谁。结构：Header.Payload.Signature | 第88章 |
| **OAuth 2.0** | 授权协议标准。"用你的GitHub账号登录我的网站"——就是OAuth 2.0在起作用。四种授权模式：授权码、隐式、密码、客户端凭证 | 第89章 |
| **RBAC** | 基于角色的访问控制。不直接给用户分配权限，而是给用户分配"角色"（如管理员、编辑、访客），角色再关联权限。管理起来更灵活 | 第90章 |
| **Spring Boot** | Spring的"一键启动"版本。帮你自动配置了大部分东西，内置了Tomcat——你不需要部署到外部服务器，直接`java -jar`就能跑 | 第72章 |
| **Spring Cloud** | 微服务工具集。提供服务注册与发现、配置管理、负载均衡、断路器、网关等功能——一套做微服务的全家桶 | 第102章 |
| **application.yml** | Spring Boot的主配置文件。数据库连接、端口号、Redis地址、日志级别——所有配置都写在这里 | 第74章 |

## F.4 数据库与缓存

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **数据库** | 有组织地存储数据的地方。不是简单的文件——能高效查询、事务保证安全、多人同时访问不冲突 | 第54章 |
| **SQL** | 和数据库"对话"的语言。`SELECT * FROM users WHERE age > 18`——英文一样的语法 | 第55章 |
| **索引** | 数据库里的"目录"。给常用查询的列建索引，就像给书加目录——不用从头翻到尾 | 第59章 |
| **B+树** | MySQL InnoDB引擎索引的底层数据结构。一种自平衡的树，所有数据都存在叶子节点，叶子节点之间用链表连接——适合范围查询 | 第59章 |
| **事务** | 一组操作打包执行——要么全成功，要么全失败。就像转账：A扣钱和B加钱必须同时成功，不能只成功一半 | 第61章 |
| **ACID** | 事务的四个特性：原子性（全或无）、一致性（规则不破）、隔离性（互不干扰）、持久性（写入不丢） | 第61章 |
| **MVCC** | 多版本并发控制。InnoDB实现事务隔离的核心机制——每行数据有多个版本，不同事务看到不同版本，读写不冲突 | 第61章 |
| **隔离级别** | 事务之间的"互不干扰"程度。读未提交→读已提交→可重复读→串行化，级别越高越安全但越慢。MySQL默认是可重复读 | 第61章 |
| **死锁** | 两个事务互相等待对方释放锁，谁都动不了。就像两个人面对面过独木桥，谁也不让谁 | 第62章 |
| **连接池** | 预先建好一批数据库连接放着。请求来了直接拿一个用，用完还回去——避免反复"拨号"的开销。HikariCP是Spring Boot默认的连接池 | 第65章 |
| **ORM** | 对象关系映射。把数据库表映射成Java类，表的行对应对象，表的列对应字段。你不用写SQL，直接操作Java对象。JPA/Hibernate就是ORM框架 | 第67章 |
| **JDBC** | Java数据库连接。Java操作数据库的最底层API——`Connection`、`Statement`、`ResultSet`。日常开发很少直接用，而是用JPA或MyBatis封装 | 第64章 |
| **缓存** | 把经常要用的数据放在"伸手就能够到"的位置。Redis就是最流行的缓存工具 | 第68章 |
| **Redis** | 数据存在内存里的"超快存储"。不像MySQL存硬盘那么慢，但内存容量有限，一般当缓存用 | 第68章 |
| **缓存穿透** | 查一个数据库里根本不存在的key，每次都穿透缓存打到数据库。恶意攻击常用这招。解决方案：缓存空值或布隆过滤器 | 第69章 |
| **缓存击穿** | 一个热点key在过期的瞬间，大量请求同时打到数据库。就像热门商品秒杀瞬间的流量洪峰。解决方案：互斥锁或永不过期 | 第69章 |
| **缓存雪崩** | 大量缓存在同一时间过期，数据库瞬间承受所有请求——像雪崩一样压垮系统。解决方案：过期时间加随机值、多级缓存 | 第69章 |

## F.5 网络与协议

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **协议** | 通信双方约定好的"规矩"。就像写信要写邮政编码、收件人——HTTP协议规定了请求必须包含方法（GET/POST）、路径、头部等 | 第06章 |
| **IP地址** | 互联网上每台设备的"门牌号"，格式如`192.168.1.1`。IPv4大约43亿个地址，已经不够用了 | 第06章 |
| **DNS** | 互联网的电话号码簿——把你输入的`www.baidu.com`翻译成IP地址`110.242.68.66` | 第07章 |
| **TCP** | 可靠的"快递员"协议——保证数据不丢、不乱、不重复，按顺序送达。三次握手建立连接，四次挥手断开 | 第09章 |
| **UDP** | 不可靠的"明信片"协议——发出去不管，丢了就丢了。快但没有保证。适合视频通话、游戏等 | 第10章 |
| **HTTP** | 浏览器和服务器之间"对话"的语言。你每次打开网页，都在发HTTP请求。请求=方法+URL+头部+可选的请求体 | 第11章 |
| **HTTPS** | HTTP的"加密版"。数据在传输过程中被加密，中间人看不到内容。网址前的🔒小锁就是HTTPS | 第13章 |
| **TLS/SSL** | HTTPS用到的加密协议。在HTTP和TCP之间加了一层加密，就像给普通信封加上了密码锁 | 第13章 |
| **WebSocket** | HTTP的"升级版"——建立一次连接后，双方可以随时互发消息，不需要客户端反复问"有消息吗" | 第14章 |
| **API** | 程序之间"对话"的接口。后端写API，前端调用API——就像餐厅的菜单：告诉你有什么菜、怎么点 | 第00章 |
| **RESTful** | 一种设计API的风格——用URL表示"资源"（如`/users/1`），用HTTP方法表示"操作"（GET查、POST增、PUT改、DELETE删） | 第75章 |
| **JSON** | 一种轻量级数据格式，人和机器都能轻松读。后端API最常用的数据交换格式。`{"name": "张三", "age": 25}` | 第77章 |
| **CORS** | 跨域资源共享。浏览器安全策略——一个域名的网页默认不能请求另一个域名的API。后端需要配置CORS允许跨域 | 第12章 |

## F.6 架构与运维

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **中间件** | 在请求处理链中的"插件"。日志记录、权限检查、跨域处理——每个中间件做一件事。Spring里的Filter和Interceptor就是中间件 | 第78章 |
| **微服务** | 把一个大应用拆成多个独立小服务。每个服务有自己的数据库，通过API通信。就像超市拆成专卖店——每个店独立运营，但合起来提供完整服务 | 第102章 |
| **消息队列（MQ）** | 异步通信的"中间人"。生产者扔消息进去，消费者慢慢取出来处理——各干各的，互不打扰。典型用途：解耦、削峰、异步 | 第94章 |
| **负载均衡** | 把请求分给多台服务器，压力分散。Nginx是最常用的反向代理+负载均衡工具 | 第105章 |
| **反向代理** | 站在服务器前面"接客"。客户端以为在跟服务器说话，其实是在跟代理说话——代理再把请求转发给后端的服务器。Nginx就是典型反向代理 | 第105章 |
| **容器（Docker）** | 把应用和环境打包在一起，到哪都能跑。就像一个标准集装箱——不管装什么货，运输体系都一样 | 第112章 |
| **CI/CD** | 持续集成+持续部署。代码提交后自动测试→自动打包→自动部署，一条流水线全自动。GitHub Actions是常见的CI/CD工具 | 第113章 |
| **熔断** | 当一个服务频繁出错时，暂时"切断"对它的调用，直接返回兜底结果。就像电路过载跳闸保护 | 第106章 |
| **降级** | 系统压力大时，主动关掉非核心功能，保证核心功能正常。就像春运时火车站暂停售票厅的大屏幕以保障售票系统 | 第106章 |
| **限流** | 控制请求速率，"一秒最多1000个请求"。超过的直接拒绝——保护系统不被冲垮。常用算法：令牌桶、漏桶 | 第106章 |
| **服务发现** | 微服务架构中，服务实例的地址是动态变化的。服务发现自动跟踪"现在有哪些实例、IP和端口是什么" | 第102章 |
| **SOLID原则** | 面向对象设计的五个原则：单一职责、开闭原则、里氏替换、接口隔离、依赖反转。简而言之——让代码好改、好测、好理解 | 第101章 |
| **设计模式** | 反复出现的问题的"标准解法"。单例（全局唯一）、工厂（按需生产对象）、策略（算法可替换）——都是前人踩坑总结的套路 | 第100章 |

## F.7 测试

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **单元测试** | 测试单个方法/函数的正确性。给定输入，检查输出是否和预期一致。JUnit是Java最常用的单元测试框架 | 第115章 |
| **Mock** | 测试用的"替身"。测试发邮件功能时，不用真的发邮件——用Mock假装发了，只验证"发邮件这个方法被调用了" | 第116章 |
| **集成测试** | 多个模块一起测。确保数据库、API、中间件能配合工作，而不是各自孤立地正确。Spring Boot用`@SpringBootTest`启动完整应用上下文 | 第117章 |
| **性能测试** | 测系统的速度和负载能力。2000人同时刷首页，响应时间还在200毫秒内吗？JMeter是常用的压测工具 | 第118章 |

## F.8 编程语言通识

| 术语 | 大白话解释 | 书中首见 |
|------|-----------|----------|
| **C语言** | 1972年诞生，几乎所有现代操作系统和数据库都用C写的。它接近底层但还算人类可读，JVM本身也是用C/C++写的 | 第01章 |
| **高级语言** | 离人类语言近、离机器语言远的编程语言。Java、Python、JavaScript都是高级语言 | 第01章 |
| **低级语言** | 离机器语言近的编程语言。"低级"不是贬义——不是说它差，是它更接近硬件。汇编和C都属于低级语言 | 第01章 |
| **编译型语言** | 写完代码后需要先"编译"成可执行文件才能运行的语言。Java、C、C++、Rust是编译型 | 第15章 |
| **解释型语言** | 不需要提前编译，运行时一行一行翻译执行。Python、JavaScript是解释型 | 第15章 |
| **静态类型** | 变量在代码写好时就确定了类型，编译时检查。比如`int x = 5`——x这辈子只能是int。Java是静态类型 | 第17章 |
| **动态类型** | 变量类型在运行时才确定，同一个变量可以先存数字再存字符串。Python和JavaScript是动态类型 | 第17章 |
| **强类型** | 不同类型不能混用，必须显式转换。`"hello" + 3`会报错（Java里不行，但`"hello" + 3`会变成`"hello3"`因为Java自动把3转成字符串）。Java和Python都是强类型 | 第17章 |
| **弱类型** | 不同类型可以自动混用。JavaScript是弱类型 | 第17章 |
| **垃圾回收（GC）** | 自动清理不再使用的内存。你只需要分配内存，不用手动释放——像有清洁工定期扫垃圾。Java的JVM自带GC | 第15章 |
| **闭包** | 一个函数"记住"了它出生时的外部变量。即使外部方法已经返回，闭包还能访问那些变量。Java的Lambda表达式本质上就是闭包 | 第35章 |
| **序列化** | 把程序里的对象转成能在网络上传输或存文件的格式（JSON、XML等）。Spring Boot返回JSON给前端就是自动序列化 | 第77章 |
| **反序列化** | 序列化的逆过程——把JSON字符串还原成程序里的Java对象 | 第77章 |
