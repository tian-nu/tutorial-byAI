## 我在做什么？

前面三章你已经学了预处理、条件编译、多文件项目。但 `gcc main.c -o main` 这行命令背后到底发生了什么？C 语言从源代码到可执行文件，不是一步到位的，而是经历了四个独立的阶段：**预处理 → 编译 → 汇编 → 链接**。理解这四个阶段，不仅能帮你理解之前学过的宏、头文件、链接错误，还能让你在遇到奇怪的编译错误时知道去哪一阶段排查。

读完本章你能：用 gcc 的分步选项手动执行编译的每个阶段，理解每个阶段的输入、输出和职责，看懂 `.i`、`.s`、`.o` 文件的意义，以及理解链接器在最后阶段做了什么。

## 一、编译的四个阶段全貌

### 1.1 一张图看清全过程

```text
hello.c                hello.i                hello.s                hello.o                hello (可执行文件)
(源代码)              (预处理后)              (汇编代码)              (目标文件)
    |                     |                      |                      |                      |
    |  预处理器            |   编译器              |   汇编器              |   链接器              |
    |  (cpp)              |   (cc1)              |   (as)               |   (ld)               |
    |  gcc -E             |   gcc -S             |   gcc -c             |   gcc -o             |
    v                     v                      v                      v                      v
┌─────────┐         ┌──────────┐          ┌──────────┐          ┌──────────┐          ┌──────────┐
│ 展开    │         │ C → 汇编 │          │ 汇编 →   │          │ 合并所有 │          │ 可执行   │
│ #include│  ────>  │ 语法分析 │  ────>   │ 机器码   │  ────>   │ .o文件   │  ────>   │ 文件     │
│ #define │         │ 生成 .s  │          │ 生成 .o  │          │ 解析符号 │          │          │
└─────────┘         └──────────┘          └──────────┘          └──────────┘          └──────────┘
```

gcc 在背后自动串联了这四个阶段，但你可以用选项让它在任意阶段停下来，查看中间产物。

### 1.2 四个阶段的 gcc 选项一览

| 阶段 | 输入 | 输出 | gcc 选项 | 输出文件 |
|------|------|------|----------|---------|
| 1. 预处理 | `.c` | 预处理后的 `.i` | `gcc -E` | `hello.i` |
| 2. 编译 | `.i` | 汇编代码 `.s` | `gcc -S` | `hello.s` |
| 3. 汇编 | `.s` | 目标文件 `.o` | `gcc -c` | `hello.o` |
| 4. 链接 | `.o` | 可执行文件 | `gcc -o` | `hello`（或 `hello.exe`） |

## 二、阶段一：预处理（Preprocessing）

### 2.1 预处理器做什么

预处理器（`cpp`）的任务：**文本处理**。它不理解 C 语言语法，只做纯文本替换。

```c
// hello.c — 输入
#include <stdio.h>
#define PI 3.14159
#define SQUARE(x) ((x) * (x))

int main() {
    printf("PI = %f\n", PI);
    printf("5^2 = %d\n", SQUARE(5));
    return 0;
}
```

### 2.2 手动执行预处理

```bash
gcc -E hello.c -o hello.i
```

- `-E`：只做预处理，不编译
- `-o hello.i`：输出到 `hello.i` 文件

打开 `hello.i`（通常几千行），你会看到：

```c
// ... 几千行 stdio.h 的内容被复制进来了 ...

int main() {
    printf("PI = %f\n", 3.14159);       // #define PI 被替换
    printf("5^2 = %d\n", ((5) * (5)));  // 宏函数被展开
    return 0;
}
```

### 2.3 预处理阶段做了什么

| 操作 | 指令 | 结果 |
|------|------|------|
| 头文件展开 | `#include <stdio.h>` | 把 `stdio.h` 的全部内容复制到当前位置 |
| 宏替换 | `#define PI 3.14159` | 所有 `PI` 替换为 `3.14159` |
| 条件编译 | `#ifdef DEBUG` | 根据条件保留或删除代码块 |
| 删除注释 | `// 注释` 或 `/* 注释 */` | 注释被替换为空格 |
| 行号标记 | 自动 | 插入 `#line` 指令，帮助编译器报错时定位原始行号 |

### 2.4 查看预处理结果的好处

当你遇到"宏展开后不是我想要的结果"时，`gcc -E` 是最直接的调试工具——它让你看到编译器实际收到的代码。

## 三、阶段二：编译（Compilation）

### 3.1 编译器做什么

编译器（`cc1`）的任务：**将预处理后的 C 代码翻译成汇编语言**。汇编语言是介于 C 和机器码之间的"低级语言"，每条汇编指令对应一条机器指令。

### 3.2 手动执行编译

```bash
# 从 .c 直接生成汇编
gcc -S hello.c -o hello.s

# 或者从 .i 生成汇编
gcc -S hello.i -o hello.s
```

- `-S`：编译生成汇编代码，不汇编

### 3.3 汇编代码是什么样

```bash
gcc -S hello.c -o hello.s
```

打开 `hello.s`（部分内容）：

```asm
    .section    .rodata
.LC0:
    .string "PI = %f\n"
.LC1:
    .string "5^2 = %d\n"
    .text
    .globl  main
    .type   main, @function
main:
    pushq   %rbp
    movq    %rsp, %rbp
    movsd   .LC2(%rip), %xmm0
    leaq    .LC0(%rip), %rdi
    movl    $1, %eax
    call    printf
    movl    $25, %esi
    leaq    .LC1(%rip), %rdi
    movl    $0, %eax
    call    printf
    movl    $0, %eax
    popq    %rbp
    ret
```

这个阶段做了：
- **词法分析**：把源代码拆成一个个 token（关键字、标识符、运算符、字面量）。
- **语法分析**：检查 token 序列是否符合 C 语法规则，构建抽象语法树（AST）。
- **语义分析**：检查类型是否匹配、变量是否声明等。
- **中间代码生成**：生成编译器内部的中间表示。
- **优化**：对中间表示做优化（如常量折叠、死代码删除）。
- **目标代码生成**：将中间表示翻译成汇编代码。

### 3.4 注意：常量已经计算好了

在汇编代码中，`SQUARE(5)` 展开后是 `((5) * (5))`，编译器在编译阶段就计算出结果是 `25`，直接生成 `movl $25, %esi`。这就是**常量折叠**优化——编译器在编译时就计算了常量表达式，而不是等到运行时。

## 四、阶段三：汇编（Assembly）

### 4.1 汇编器做什么

汇编器（`as`）的任务：**将汇编代码翻译成机器码（二进制指令）**，生成目标文件（`.o`）。

```text
汇编代码 (.s)           →  目标文件 (.o)
  movl  $25, %esi          1011 0100 0001 1001 ...  (二进制机器码)
  call  printf             1110 1000 0000 0000 ...  (二进制机器码)
```

### 4.2 手动执行汇编

```bash
# 从 .s 生成 .o
gcc -c hello.s -o hello.o

# 或者从 .c 直接生成 .o（跳过中间文件）
gcc -c hello.c -o hello.o
```

- `-c`：编译和汇编，但不链接

### 4.3 目标文件里有什么

目标文件（`.o`）包含：
- **机器码**：函数的二进制指令
- **数据**：全局变量和静态变量的初始值
- **符号表**：这个文件定义了哪些符号（函数、变量），需要哪些外部符号
- **重定位信息**：哪些地址需要在链接时修正

```bash
# 查看目标文件的符号表
nm hello.o

# 查看目标文件的详细结构
objdump -h hello.o     # 查看段（section）
objdump -d hello.o     # 反汇编（查看机器码对应的汇编）
objdump -t hello.o     # 查看符号表
```

### 4.4 .o 文件是可读的吗？

```bash
# 用 hexdump 查看二进制内容
xxd hello.o | head -20
```

输出类似：

```
00000000: 7f45 4c46 0201 0100 0000 0000 0000 0000  .ELF............
00000010: 0100 3e00 0100 0000 0000 0000 0000 0000  ..>.............
```

`.o` 文件是 ELF（Linux）或 PE/COFF（Windows）格式的二进制文件，人类无法直接阅读。

## 五、阶段四：链接（Linking）

### 5.1 链接器做什么

链接器（`ld`）的任务：**将多个 `.o` 文件和库文件合并成一个可执行文件**。

```text
main.o ──────┐
             │
math_ops.o ──┼──→ 链接器 (ld) ──→ 可执行文件 (hello)
             │
libc.a ──────┘  (标准 C 库)
```

链接器解决的核心问题：**符号解析**和**重定位**。

### 5.2 符号解析

每个 `.o` 文件都记录了：
- 它**定义了**哪些符号（函数、变量）
- 它**需要**哪些外部符号

```text
main.o:
  定义: main
  需要: printf, add, operation_count

math_ops.o:
  定义: add, subtract, multiply, divide, operation_count
  需要: printf

libc.a:
  定义: printf, scanf, malloc, ...
```

链接器遍历所有 `.o` 文件，为每个"需要"的符号找到对应的"定义"。如果找不到，就报 `undefined reference`。

### 5.3 重定位

每个 `.o` 文件中的地址都是**临时的**（从 0 开始），链接器需要把它们合并，计算最终的地址。

```text
main.o 中:
  0x0000: main 函数的代码
  0x0050: 调用 printf（地址待定）

math_ops.o 中:
  0x0000: add 函数的代码
  0x0020: 调用 printf（地址待定）

链接后:
  0x1000: main 函数的代码
  0x1050: 调用 printf → 修正为 printf 的实际地址 0x2000
  0x1100: add 函数的代码
  0x1120: 调用 printf → 修正为 printf 的实际地址 0x2000
  0x2000: printf 函数的代码（来自 libc）
```

### 5.4 手动执行链接

```bash
# 先生成所有 .o 文件
gcc -c main.c
gcc -c math_ops.c

# 链接
gcc main.o math_ops.o -o program

# 等价于直接调用 ld（但 gcc 会自动添加标准库等，更推荐用 gcc 做链接）
```

### 5.5 静态链接 vs 动态链接

| 特性 | 静态链接 | 动态链接 |
|------|----------|----------|
| 库文件 | `.a`（Linux）/ `.lib`（Windows） | `.so`（Linux）/ `.dll`（Windows） |
| 链接时机 | 编译时合并到可执行文件 | 运行时由操作系统加载 |
| 可执行文件大小 | 较大（包含库代码） | 较小（只记录引用） |
| 库更新 | 需要重新编译 | 替换库文件即可 |
| 依赖 | 无外部依赖 | 运行时必须有对应的 `.so`/`.dll` |

默认情况下，gcc 使用动态链接。可以用 `-static` 强制静态链接：

```bash
gcc -static main.c math_ops.c -o program
```

## 六、分步手动执行全过程

### 6.1 准备源文件

```c
// hello.c
#include <stdio.h>

#define GREETING "Hello, World!"

int main() {
    printf("%s\n", GREETING);
    return 0;
}
```

### 6.2 第一步：预处理

```bash
gcc -E hello.c -o hello.i
```

查看 `hello.i` 的末尾几行：

```bash
tail -10 hello.i
```

你会看到 `printf("%s\n", "Hello, World!");`——`GREETING` 已被替换。

### 6.3 第二步：编译

```bash
gcc -S hello.i -o hello.s
```

查看 `hello.s`：

```bash
cat hello.s
```

你会看到 `main:` 标签下的汇编指令。

### 6.4 第三步：汇编

```bash
gcc -c hello.s -o hello.o
```

查看符号表：

```bash
nm hello.o
```

输出类似：

```
                 U _GLOBAL_OFFSET_TABLE_
0000000000000000 T main
                 U printf
```

- `T main`：`main` 函数在这个文件中定义
- `U printf`：`printf` 需要从外部链接

### 6.5 第四步：链接

```bash
gcc hello.o -o hello
```

运行：

```bash
./hello
```

输出：

```
Hello, World!
```

### 6.6 一步到位 vs 分步

```bash
# 一步到位
gcc hello.c -o hello

# 等价的分步操作
gcc -E hello.c -o hello.i
gcc -S hello.i -o hello.s
gcc -c hello.s -o hello.o
gcc hello.o -o hello
```

## 七、中间文件的作用

### 7.1 .i 文件（预处理后）

- 用途：调试宏展开、查看头文件引入的内容
- 大小：通常很大（因为 `#include` 展开）
- 补充：包含 `#line` 指令，帮助编译器在报错时定位原始行号

```bash
# 生成 .i 并保留注释（默认 -E 会删除注释）
gcc -E -C hello.c -o hello.i
```

### 7.2 .s 文件（汇编代码）

- 用途：学习汇编、性能分析、调试底层问题
- 大小：中等
- 补充：可以用 `-masm=intel` 生成 Intel 语法汇编（默认是 AT&T 语法）

```bash
# 生成 Intel 语法的汇编代码
gcc -S -masm=intel hello.c -o hello.s
```

### 7.3 .o 文件（目标文件）

- 用途：模块化编译、增量编译（只重编译修改过的文件）
- 大小：较小
- 补充：可以用 `ar` 工具打包成静态库 `.a`

```bash
# 打包多个 .o 文件为静态库
ar rcs libmath.a math_ops.o advanced_math.o

# 使用静态库
gcc main.c -L. -lmath -o program
```

### 7.4 编译优化级别

gcc 提供不同的优化级别，影响编译阶段生成的代码质量：

```bash
gcc -O0 hello.c -o hello     # 无优化（默认），编译快，运行慢
gcc -O1 hello.c -o hello     # 基本优化
gcc -O2 hello.c -o hello     # 标准优化（推荐发布用）
gcc -O3 hello.c -o hello     # 激进优化（可能增大代码体积）
gcc -Os hello.c -o hello     # 优化代码体积
```

你可以对比不同优化级别生成的 `.s` 文件，观察汇编代码的差异：

```bash
gcc -S -O0 hello.c -o hello_O0.s
gcc -S -O2 hello.c -o hello_O2.s
diff hello_O0.s hello_O2.s
```

## 八、常见错误与对应阶段

### 8.1 按阶段排查错误

```text
错误类型                        → 发生在哪个阶段
─────────────────────────────────────────────────
预处理指令错误（#include 找不到）  → 预处理阶段
语法错误（缺少分号、括号不匹配）   → 编译阶段
类型错误（类型不匹配）            → 编译阶段
未定义变量/函数（在单个文件中）    → 编译阶段
undefined reference               → 链接阶段
multiple definition               → 链接阶段
段错误（运行时）                  → 运行时（不在编译阶段）
```

### 8.2 常见编译错误定位

```bash
# 只做预处理和编译，停在第 2 阶段后
gcc -S hello.c
# 如果报错，说明问题在预处理或编译阶段

# 如果 -S 通过了但 -c 失败
gcc -c hello.c
# 问题在汇编阶段（极少见）

# 如果 -c 通过了但链接失败
gcc hello.o -o hello
# 问题在链接阶段（undefined reference 等）
```

## 九、我做得对不对？

### 9.1 分步编译验证

创建以下源文件，然后分步执行：

```c
// test.c
#include <stdio.h>

#define NUM 42
#define DOUBLE(x) ((x) * 2)

int main() {
    int result = DOUBLE(NUM);
    printf("Result: %d\n", result);
    return 0;
}
```

```bash
# 步骤 1：预处理
gcc -E test.c -o test.i
# 查看 test.i 末尾，确认 NUM 被替换为 42，DOUBLE 被展开

# 步骤 2：编译
gcc -S test.i -o test.s
# 查看 test.s，找到 main 函数，观察汇编代码

# 步骤 3：汇编
gcc -c test.s -o test.o
nm test.o
# 确认输出中包含 T main 和 U printf

# 步骤 4：链接
gcc test.o -o test
./test
# 预期输出: Result: 84
```

### 9.2 对比优化级别

```bash
gcc -S -O0 test.c -o test_O0.s
gcc -S -O2 test.c -o test_O2.s
```

打开两个 `.s` 文件，对比 `main` 函数。你会发现 `-O2` 版本可能直接计算了 `84`，连 `DOUBLE` 宏的展开都被优化掉了。

### 9.3 验证中间文件大小

```bash
gcc -E test.c -o test.i
gcc -S test.c -o test.s
gcc -c test.c -o test.o
gcc test.c -o test

ls -lh test.i test.s test.o test
```

一般来说：`test.i`（最大） > `test.s` > `test.o` > `test`（可执行文件可能比 .o 大，因为包含了库代码）。

## 十、不对怎么办？

| 现象 | 可能原因 | 解决 |
|------|----------|------|
| `gcc -E` 报错 | `#include` 的文件找不到 | 检查头文件路径，用 `-I` 指定 |
| `gcc -S` 报语法错误 | 源代码有语法错误 | 根据错误信息修改源代码 |
| `gcc -c` 报错 | 汇编代码有问题（极少见） | 检查是否生成了正确的 `.s` 文件 |
| 链接时报 `undefined reference` | 缺少 `.o` 文件或库 | 确保所有 `.o` 文件都在链接命令中 |
| 链接时报 `multiple definition` | 同一个符号在多个 `.o` 中定义了 | 检查头文件中是否有函数定义 |
| 预处理后 `.i` 文件中奇怪的内容 | 正常的——`#include` 展开了所有头文件 | 关注文件末尾你自己的代码 |
| 汇编代码看不懂 | 正常——汇编语言需要专门学习 | 不需要完全看懂，重点是理解这是C到机器码的中间步骤 |
| `.o` 文件不能直接运行 | `.o` 不是可执行文件 | 需要通过链接阶段生成可执行文件 |

## 本章小结

| 阶段 | 命令 | 输入 | 输出 | 做什么 |
|------|------|------|------|--------|
| 预处理 | `gcc -E` | `.c` | `.i` | 展开 `#include`、替换 `#define`、处理条件编译、删除注释 |
| 编译 | `gcc -S` | `.i` 或 `.c` | `.s` | C 代码翻译为汇编代码，做语法/语义分析、优化 |
| 汇编 | `gcc -c` | `.s` 或 `.c` | `.o` | 汇编代码翻译为机器码，生成目标文件 |
| 链接 | `gcc -o` | `.o` | 可执行文件 | 合并所有 `.o` 和库，解析符号，重定位地址 |

## 练习题

1. 创建一个包含宏定义和条件编译的源文件，然后分步执行编译四阶段。在每一步之后检查中间文件的内容，确认你理解了每个阶段做了什么。

2. 使用 `nm` 分别查看 `main.o` 和 `math_ops.o`（来自第 40 章的示例）的符号表，确认：
   - `main.o` 定义了哪些符号？需要哪些符号？
   - `math_ops.o` 定义了哪些符号？需要哪些符号？
   - 链接时，哪些符号被"配对"了？

3. 编译以下程序，对比 `-O0` 和 `-O2` 生成的汇编代码：
   ```c
   int sum_to_n(int n) {
       int sum = 0;
       for (int i = 1; i <= n; i++) {
           sum += i;
       }
       return sum;
   }

   int main() {
       return sum_to_n(100);
   }
   ```
   用 `gcc -S -O0` 和 `gcc -S -O2` 分别生成汇编，比较 `sum_to_n` 函数的差异。你能发现 `-O2` 做了什么优化吗？

4. 故意制造一个链接错误：在一个 `.c` 文件中声明一个函数，在另一个 `.c` 文件中实现它，但故意把函数名写错（大小写不同）。然后分步编译，观察哪一步报错，报的是什么信息。
