## 一、目标与完成效果

**一句话目标**：给 `GET /posts` 加上分页功能——不再一次返回所有文章，而是按页返回，每页固定条数。

**完成后的可观测效果**：
- `GET /posts` 默认返回第 1 页，每页 10 条。
- `GET /posts?page=2&size=5` 返回第 2 页，每页 5 条。
- 返回的 JSON 里除了文章列表，还有 `total`（总条数）、`page`（当前页）、`size`（每页条数）、`pages`（总页数）。
- 请求一个超出范围的页码（比如总共 3 页你请求第 100 页），返回空列表 `items: []`，而不是报错。
- 你能跟朋友解释清楚：为什么分页不是"把数据切好发给前端"，而是"前端要第几页，后端从数据库里挑对应的几条返回"。

## 三、分步操作

### 步骤 1：问题——1000 篇文章一次返回有多糟糕？

假设你的博客火了，数据库里有 1000 篇文章。当前 `GET /posts` 接口的行为是：

```python
posts = db.query(Post).all()  # 一次查出全部文章
return posts
```

**这意味着什么？**

- 数据库要扫描 1000 行数据，全部加载到内存。
- Python 要把 1000 个 ORM 对象序列化成 JSON。
- 网络要传输这个巨大的 JSON 响应体（可能好几 MB）。
- 前端要渲染 1000 篇文章的列表——用户根本看不完，但浏览器卡死了。

**比喻**：你开了一家图书馆，有 10 万本书。一个读者进门说"我要看所有书"，你把 10 万本书全部从仓库搬出来、堆在他面前——他根本看不过来，但你搬书的力气和仓库的磨损是实实在在的。

**分页**就是图书馆的"每次只借 10 本，看完再来换"——读者说"给我第 3 页，每页 10 本"，你只搬第 21~30 本出来，又快又省力。

> 🤔 想多一点：为什么不在前端做分页？

**绝对不行。** 如果你把 1000 条数据全部返回给前端，然后在浏览器里用 JavaScript 分页，看起来方便了，但：
- 数据库仍然要查 1000 条（慢）。
- 网络仍然要传 1000 条（大）。
- 前端仍然要加载 1000 条数据到内存（卡）。

**分页必须在后端做**——数据库只查你要的那几条，网络只传你要的那几条，前端只渲染你要的那几条。这才是真正的"省力"。

### 步骤 2：分页的核心参数——`page` 和 `size`

分页需要两个数字：

| 参数 | 含义 | 示例 | 默认值 |
|------|------|------|--------|
| `page` | 第几页，从 1 开始 | `page=3` 表示第 3 页 | 1 |
| `size` | 每页多少条 | `size=10` 表示每页 10 条 | 10 |

**怎么算"跳过多少条"？**

假设每页 10 条：
- 第 1 页：跳过 0 条，取前 10 条 → 文章 1~10
- 第 2 页：跳过 10 条，取 10 条 → 文章 11~20
- 第 3 页：跳过 20 条，取 10 条 → 文章 21~30

**公式**：`跳过的条数 = (page - 1) × size`

**比喻**：一本 300 页的书，每页 30 行。你想看第 5 页——需要先翻过前 4 页（跳过 4 × 30 = 120 行），然后读 30 行。`page=5, size=30 → skip=120, limit=30`。

### 步骤 3：SQL 层面——`LIMIT` 和 `OFFSET`

在 SQL 中，分页对应两个关键字：

| SQL 关键字 | 含义 | 对应公式 |
|-----------|------|----------|
| `LIMIT n` | 只返回 n 条 | `LIMIT {size}` |
| `OFFSET m` | 跳过前 m 条 | `OFFSET {(page - 1) × size}` |

```sql
-- 查询第 2 页，每页 10 条
SELECT * FROM posts
LIMIT 10 OFFSET 10;
```

**逐行解释**：
- `LIMIT 10`：最多返回 10 条记录。
- `OFFSET 10`：跳过前 10 条（第 1 页的 10 条），从第 11 条开始取。

**比喻**：`OFFSET` 是你翻书时跳过的页数，`LIMIT` 是你翻到目标页后要读的行数。`OFFSET 10 LIMIT 10` = "翻过前 10 行，然后读 10 行"。

---

### 步骤 4：SQLAlchemy 层面——`offset()` 和 `limit()`

SQLAlchemy 把 SQL 的 `OFFSET` 和 `LIMIT` 翻译成了 Python 方法：

```python
# 查询第 2 页，每页 10 条
page = 2
size = 10
skip = (page - 1) * size  # 跳过 10 条

posts = db.query(Post).offset(skip).limit(size).all()
```

**逐行解释**：
- `.offset(skip)`：对应 SQL 的 `OFFSET`，跳过前 `skip` 条。
- `.limit(size)`：对应 SQL 的 `LIMIT`，最多返回 `size` 条。
- `.all()`：执行查询，返回列表。

**链式调用**：`db.query(Post)` 返回一个查询对象，你可以连续调用 `.filter()`、`.offset()`、`.limit()`、`.order_by()` 等方法，最后 `.all()` 才真正执行。这就像拼乐高——先搭好查询条件，最后一个 `.all()` 相当于"按图施工"。

❌ 常见错误 → ✅ 正确示例：

❌ 错误：`offset` 和 `limit` 顺序写反

```python
# 错误：先 limit 再 offset 是没意义的，但不会报错
posts = db.query(Post).limit(size).offset(skip).all()
```

实际上 SQLAlchemy 中 `.limit()` 和 `.offset()` 的调用顺序不影响最终 SQL——但为了代码可读性，建议按 SQL 习惯：先 offset 再 limit。

✅ 正确：按习惯写，先 offset 再 limit

```python
posts = db.query(Post).offset(skip).limit(size).all()
```

### 步骤 5：返回格式设计——不止返回文章列表

分页接口不应该只返回文章列表。前端需要知道"总共有多少页"，才能渲染"上一页 / 下一页"按钮。

**返回格式**：

```json
{
  "items": [
    { "id": 1, "title": "...", ... },
    { "id": 2, "title": "...", ... }
  ],
  "total": 1000,
  "page": 2,
  "size": 10,
  "pages": 100
}
```

| 字段 | 含义 | 前端用在哪 |
|------|------|-----------|
| `items` | 当前页的文章列表 | 渲染文章卡片 |
| `total` | 数据库里一共有多少篇文章 | 显示"共 1000 篇" |
| `page` | 当前是第几页 | 高亮当前页码按钮 |
| `size` | 每页多少条 | 前端知道每页条数 |
| `pages` | 总页数 | 渲染"第 2/100 页"和页码按钮 |

**`pages` 怎么算？** `总页数 = ceil(总条数 / 每页条数)`。比如 1000 篇文章，每页 10 条：`1000 / 10 = 100` 页。如果 25 篇文章，每页 10 条：`25 / 10 = 2.5`，向上取整 → 3 页。

**比喻**：`items` 是你当前手上的书，`total` 是图书馆的总藏书量，`pages` 是"你一共需要借多少次才能看完"。

### 步骤 6：定义分页响应模型（`schemas.py`）

我们先把返回格式定义成一个 Pydantic 模型，这样 FastAPI 知道返回什么，Swagger 文档也能自动生成。

打开 `blog_backend/schemas.py`，在文件末尾追加：

```python
# blog_backend/schemas.py —— 在文件末尾追加

from typing import Generic, TypeVar
from pydantic import BaseModel

# 泛型变量：T 可以是任何 Pydantic 模型（文章、评论、用户……）
T = TypeVar("T")


class PaginatedResponse(BaseModel, Generic[T]):
    """分页响应通用模型。可用于任何需要分页的接口。"""
    items: list[T]       # 当前页的数据列表
    total: int           # 总条数
    page: int            # 当前页码
    size: int            # 每页条数
    pages: int           # 总页数
```

**逐行解释**：
- `T = TypeVar("T")`：定义一个"占位符类型"，就像函数参数里的 `x`——调用时你才告诉它具体是什么。此术语需进附录：泛型。
- `class PaginatedResponse(BaseModel, Generic[T])`：一个泛型 Pydantic 模型。`Generic[T]` 的意思是"这个模型里有一个类型是不确定的，用的时候再说"。
- `items: list[T]`：当前页的数据列表，`T` 可以是 `BlogResponse`（文章）、`CommentResponse`（评论）等任何模型。
- `total, page, size, pages`：分页元信息，全是整数。

**为什么用泛型？** 因为分页是一个通用模式——文章列表要分页，评论列表也要分页，用户列表也要分页。写一个 `PaginatedResponse`，所有地方都能用，不用重复写 `PostPaginatedResponse`、`CommentPaginatedResponse`……

**比喻**：`PaginatedResponse` 就像一个"快递盒"——不管里面装的是书、衣服还是电子产品，盒子本身的结构（收件人、地址、重量）都一样。`Generic[T]` 就是说"盒子里装什么你自己定，但盒子本身的样子不变"。

### 步骤 7：改造 `GET /posts`——加上分页参数

打开 `blog_backend/main.py`，找到 `get_all_posts` 函数，把它改造成分页版。

**改造前**：

```python
@app.get("/posts")
def get_all_posts(db: Session = Depends(get_db)):
    posts = db.query(Post).all()
    return posts
```

**改造后**：

```python
# blog_backend/main.py —— 在文件顶部追加导入
import math
from blog_backend.schemas import PaginatedResponse


@app.get("/posts", response_model=PaginatedResponse[BlogResponse])
def get_all_posts(
    page: int = 1,
    size: int = 10,
    db: Session = Depends(get_db),
):
    # 1. 查询总条数
    total = db.query(Post).count()

    # 2. 计算跳过的条数
    skip = (page - 1) * size

    # 3. 分页查询
    posts = db.query(Post).offset(skip).limit(size).all()

    # 4. 计算总页数
    pages = math.ceil(total / size) if total > 0 else 0

    # 5. 返回分页响应
    return {
        "items": posts,
        "total": total,
        "page": page,
        "size": size,
        "pages": pages,
    }
```

**逐行解释**：

- `page: int = 1, size: int = 10`：FastAPI 的查询参数，默认值分别是 1 和 10。用户不传参数时，`GET /posts` 等价于 `GET /posts?page=1&size=10`。
- `response_model=PaginatedResponse[BlogResponse]`：告诉 FastAPI 返回格式是分页响应，里面的 `items` 类型是 `BlogResponse`。Swagger 文档会自动生成对应的 Schema。
- `db.query(Post).count()`：`COUNT(*)` 的 SQLAlchemy 写法，返回数据库里一共有多少篇文章。**注意**：`count()` 会执行一条 `SELECT COUNT(*) FROM posts`，和分页查询是两条 SQL。
- `skip = (page - 1) * size`：用步骤 2 的公式计算跳过的条数。
- `db.query(Post).offset(skip).limit(size).all()`：分页查询。
- `pages = math.ceil(total / size) if total > 0 else 0`：总页数 = 总条数 ÷ 每页条数，向上取整。`math.ceil(2.3)` = 3。`total` 为 0 时总页数为 0（避免除零和空列表的歧义）。

**你可能注意到 `BlogResponse` 还没定义**——确实，我们在之前的教程中一直直接返回 ORM 对象。为了分页响应模型能正常工作，我们需要在 `schemas.py` 中加一个 `BlogResponse`：

```python
# blog_backend/schemas.py —— 追加（如果还没有的话）

from datetime import datetime


class BlogResponse(BaseModel):
    """文章响应模型（从数据库返回给客户端）"""
    id: int
    title: str
    content: str
    published: bool
    summary: str | None = None
    author_id: int | None = None
    created_at: datetime
    updated_at: datetime

    class Config:
        from_attributes = True  # 允许从 ORM 对象自动转换
```

**`from_attributes = True`**（旧版 Pydantic 叫 `orm_mode = True`）的意思是：你可以直接把 SQLAlchemy 的 ORM 对象（`Post` 实例）传给 `BlogResponse`，Pydantic 会自动从对象的属性中取值。**不加这个配置，FastAPI 无法把 ORM 对象序列化成 JSON。**

**比喻**：`BlogResponse` 是"对外展示的身份证"——只包含你愿意让外人看到的信息。ORM 模型 `Post` 可能包含更多内部字段，但给客户端看的时候，只暴露 `BlogResponse` 里的字段。

### 步骤 8：边界处理——page 超出范围怎么办？

当前的代码有一个潜在问题：如果用户请求 `page=100`，但总共只有 3 页，会发生什么？

```python
page = 100
size = 10
skip = (100 - 1) * 10  # = 990
posts = db.query(Post).offset(990).limit(10).all()
```

**实际上不会报错**——SQL 的 `OFFSET` 超出数据范围时，返回空结果集。`posts` 会是 `[]`，`pages` 会是 3（因为 `total` 不变），响应会是：

```json
{
  "items": [],
  "total": 25,
  "page": 100,
  "size": 10,
  "pages": 3
}
```

**这是正确的行为**——分页超出范围不报错，返回空列表。前端看到 `page=100 > pages=3`，就知道用户翻过头了，可以显示"没有更多数据"。

**但有一种边界情况需要注意**：`page` 为 0 或负数。`page=0` 时 `skip = -10`，SQLite 会把负数的 `OFFSET` 当作 0 处理（不会报错但行为不符合预期）。

**健壮处理**：加一个简单的边界检查：

```python
@app.get("/posts", response_model=PaginatedResponse[BlogResponse])
def get_all_posts(
    page: int = 1,
    size: int = 10,
    db: Session = Depends(get_db),
):
    # 边界保护：page 最小为 1，size 限制在 1~100 之间
    if page < 1:
        page = 1
    if size < 1:
        size = 1
    if size > 100:
        size = 100  # 防止有人一次请求 10000 条，把服务器拖垮

    total = db.query(Post).count()
    skip = (page - 1) * size
    posts = db.query(Post).offset(skip).limit(size).all()
    pages = math.ceil(total / size) if total > 0 else 0

    return {
        "items": posts,
        "total": total,
        "page": page,
        "size": size,
        "pages": pages,
    }
```

**`size` 上限 100**：这是一个防御性设计——如果有人恶意请求 `?size=9999999`，数据库会试图加载全部数据，可能导致服务器内存耗尽。限制 `size` 最大值，就像图书馆规定"一次最多借 20 本"——多了不给。

### 步骤 10：🤔 想多一点——分页有哪些常见变体？

你刚刚实现的是**偏移分页**（Offset-based Pagination），用 `OFFSET` + `LIMIT`。这是最常用的分页方式，但还有其他方案：

| 分页方式 | 原理 | 优点 | 缺点 | 适用场景 |
|----------|------|------|------|----------|
| 偏移分页 | `OFFSET + LIMIT` | 简单，前端容易实现"跳转到第 N 页" | 大 OFFSET 时性能差（数据库要扫描并丢弃前 N 条） | 数据量小、需要页码跳转 |
| 游标分页 | 用上一页最后一条的 ID 或时间戳作为游标，`WHERE id > cursor LIMIT n` | 性能稳定，不受页数影响 | 不能跳页，只能"上一页/下一页" | 数据量大、无限滚动（微博、Twitter） |
| 键集分页 | 类似游标分页，但游标是一个排序字段的组合 | 性能最好，适合实时数据流 | 实现复杂，需要索引配合 | 高并发、实时流（聊天消息） |

**本教程用的是偏移分页**——对于博客系统来说，文章数量通常不会到百万级别，`OFFSET` 的性能问题不明显。如果你的博客以后火到有 100 万篇文章……那是幸福的烦恼，届时再升级到游标分页也不迟。

import math
from fastapi import FastAPI, Depends
from fastapi.responses import JSONResponse
from sqlalchemy.orm import Session
from blog_backend.schemas import BlogCreate, BlogResponse, PaginatedResponse
from blog_backend.models import Post, User
from blog_backend.database import get_db
from blog_backend.auth import get_current_user, check_owner

app = FastAPI()

# ... 其他接口保持不变 ...


# === 获取全部文章（分页版） ===
@app.get("/posts", response_model=PaginatedResponse[BlogResponse])
def get_all_posts(
    page: int = 1,
    size: int = 10,
    db: Session = Depends(get_db),
):
    # 边界保护
    if page < 1:
        page = 1
    if size < 1:
        size = 1
    if size > 100:
        size = 100

    total = db.query(Post).count()
    skip = (page - 1) * size
    posts = db.query(Post).offset(skip).limit(size).all()
    pages = math.ceil(total / size) if total > 0 else 0

    return {
        "items": posts,
        "total": total,
        "page": page,
        "size": size,
        "pages": pages,
    }

## 六、术语附录

| 术语 | 英文 | 通俗解释 | 本章出现位置 |
|------|------|----------|-------------|
| 分页 | Pagination | 把大量数据分成一页一页展示，每次只取一页，而不是全部加载。核心公式：`skip = (page-1) × size`。 | 全文 |
| LIMIT | — | SQL 关键字，限制查询返回的最大行数。`LIMIT 10` 表示最多返回 10 条。 | 步骤 3 |
| OFFSET | — | SQL 关键字，跳过前 N 行。`OFFSET 10` 表示从第 11 行开始返回。配合 `LIMIT` 实现分页。 | 步骤 3 |
| skip | — | 在 SQLAlchemy 中对应 `OFFSET`，调用 `.offset(skip)` 跳过前 N 条记录。 | 步骤 4、7 |
| 边界处理 | Edge Case Handling | 对非正常输入（page=0、page=超大值、size=超大值）的处理逻辑。好的边界处理保证接口不崩溃、不返回奇怪结果。 | 步骤 8 |
| 泛型 | Generic | 允许一个类/函数处理多种类型，而不需要为每种类型单独写一份代码。`PaginatedResponse[T]` 中的 `T` 就是泛型参数。 | 步骤 6 |
| `from_attributes` | — | Pydantic 配置项（旧版叫 `orm_mode`），允许直接从 ORM 对象属性中取值，而不需要手动 `.dict()` 转换。 | 步骤 7 |
| 偏移分页 | Offset-based Pagination | 用 `OFFSET` + `LIMIT` 实现的分页方式。优点是简单、支持跳页，缺点是大 `OFFSET` 时性能差。 | 步骤 10 |
| 游标分页 | Cursor-based Pagination | 用上一页最后一条的 ID 或时间戳作为游标，`WHERE id > cursor LIMIT n`。性能稳定但不能跳页。 | 步骤 10 |

## 七、小结

| 你学到了什么 | 一句话总结 |
|--------------|-----------|
| 分页的必要性 | 1000 条数据一次返回 = 慢、卡、浪费。分页只返回当前页需要的几条。 |
| 分页核心公式 | `skip = (page - 1) × size`，SQL 里是 `OFFSET skip LIMIT size` |
| SQLAlchemy 分页 | `.offset(skip).limit(size).all()` |
| 返回格式 | 不止 `items`，还要 `total`、`page`、`size`、`pages`——前端需要这些信息渲染分页导航 |
| 泛型分页模型 | `PaginatedResponse[T]` 一个模型，所有分页接口都能用 |
| 边界处理 | `page < 1` 修正为 1，`size > 100` 限制上限，超出页码范围返回空列表不报错 |
| `from_attributes` | Pydantic 配置，让 ORM 对象能直接序列化为 JSON |
| 分页的变体 | 偏移分页（简单，支持跳页）、游标分页（性能好，不能跳页） |

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| 忘记 `from_attributes = True` | 返回 500 内部错误，或返回空对象 | Pydantic 不知道如何从 ORM 对象取值 | 在 `BlogResponse` 的 `Config` 类中加 `from_attributes = True` |
| 忘记 `response_model` 参数 | 返回值没有被校验/序列化，可能暴露多余字段 | 没有指定 `response_model`，FastAPI 不会用 Pydantic 模型过滤输出 | 加上 `response_model=PaginatedResponse[BlogResponse]` |
| 让前端传 `skip` 而不是 `page` | 前端要自己算 `skip`，公式耦合到前端 | 后端应该隐藏"跳过多少条"的实现细节，只暴露"第几页"这个业务概念 | 接口参数用 `page`，后端内部算 `skip` |
| `size` 不设上限 | 请求 `?size=9999999` 导致数据库全表扫描，内存爆满 | 没有限制 `size` 最大值 | 加 `if size > 100: size = 100` |
| 分页后 `GET /posts` 不再返回简单数组 | 前端代码 `response.data[0]` 报错 | 前端可能还在用数组方式访问，现在返回的是 `{items: [...], total: ...}` | 前端需要同步更新，改为 `response.data.items[0]` |
