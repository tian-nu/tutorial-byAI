## 我在做什么？

到目前为止，你的容器都是孤独的——它们各自运行，互不通信。但真正的应用从来不是孤岛：Web 服务器要连数据库，微服务要互相调用，日志收集器要监听多个容器的输出。

要让容器之间通信，你不需要学 TCP/IP 协议栈，也不需要考网络工程师证书。Docker 的网络模型只有三层，理解这三层你就能搭建任意复杂的容器网络。

学完这一章，你能：
- 说出 Docker 三种网络模式*此术语见附录C*（bridge、host、none）的区别
- 理解 bridge 网络*此术语见附录C*的工作原理——docker0 虚拟网桥*此术语见附录C*是什么
- 用 `docker network ls` 和 `docker network inspect` 查看网络拓扑
- 在 host 模式下运行容器，直接使用宿主机网络栈
- 判断什么时候该用哪种网络模式

## 一、先看看你的 Docker 默认有哪些网络

```bash
docker network ls
# NETWORK ID     NAME      DRIVER    SCOPE
# abc123def456   bridge    bridge    local
# def456abc789   host      host      local
# ghi789jkl012   none      null      local
```

三个默认网络，刚好对应 Docker 的三种网络模式。我们一个一个来。

## 二、bridge 模式——默认的"公寓楼"

### 比喻

想象一栋公寓楼：
- 每个房间（容器）有独立的门牌号（IP 地址）
- 楼里有内部的走廊（docker0 虚拟网桥），房间里的人可以互相串门
- 对外只有一个大门（宿主机网卡），访客要进来必须通过门卫（端口映射）

这就是 bridge 模式。**99% 的容器都用这种模式。**

### 工作原理

当你启动一个容器而不指定 `--network` 时，Docker 默认把它放进 `bridge` 网络：

```bash
docker run -d --name web1 nginx:alpine
```

发生了什么？Docker 做了三件事：
1. 在宿主机上创建一对虚拟网卡（veth pair），一端插进容器（eth0），一端插在 docker0 网桥上
2. 给容器分配一个 172.17.0.0/16 网段的 IP
3. 容器通过 docker0 网桥 → 宿主机网卡 → 外部网络，实现 NAT 上网

### 查看 bridge 网络内部

```bash
docker network inspect bridge
```

输出很长，关键字段：

```json
[
    {
        "Name": "bridge",
        "Driver": "bridge",
        "Subnet": "172.17.0.0/16",
        "Gateway": "172.17.0.1",
        "Containers": {
            "abc123...": {
                "Name": "web1",
                "IPv4Address": "172.17.0.2/16"
            }
        }
    }
]
```

解读：
- `Subnet`：这个网络里所有容器的 IP 都在 172.17.x.x 范围内
- `Gateway`：172.17.0.1 是 docker0 网桥的 IP，也是容器的默认网关
- `Containers`：当前连在这个网络上的容器及其 IP

### 默认 bridge 的限制

在默认 bridge 网络里，容器之间**只能用 IP 通信，不能用容器名通信**：

```bash
# 启动两个容器
docker run -d --name app1 nginx:alpine
docker run -d --name app2 nginx:alpine

# 从 app1 ping app2 通过容器名
docker exec app1 ping app2
# ping: bad address 'app2'
#     ──────┬──────
#     容器名解析不了！
```

这不是 bug，这是默认 bridge 网络的设计——它不提供 DNS 解析*此术语见附录C*。我们下一章会解决这个问题。

## 三、host 模式——"裸奔"模式

### 比喻

这次你不住公寓了，你直接睡在宿主机的大街上。你没有自己的房间号，你用的是宿主机自己的 IP 地址。

### 怎么用

```bash
docker run --network host nginx:alpine
```

加了 `--network host`*此术语见附录C*，容器就**直接使用宿主机的网络栈**——没有独立的 IP，没有 NAT 转换，容器里监听的端口直接暴露在宿主机上。

### 验证

```bash
# 启动一个 host 模式的 Nginx
docker run -d --name nginx-host --network host nginx:alpine

# 直接在宿主机上访问
curl http://localhost
# 返回 Nginx 欢迎页
```

注意：你**没有**指定 `-p 80:80`，但 Nginx 的 80 端口直接可以在宿主机上访问。因为 host 模式下，容器和宿主机共享同一个网络栈，容器的 80 端口就是宿主机的 80 端口。

### 什么时候用 host 模式？

- 需要极致网络性能（跳过 Docker 的 NAT 转发层）
- 容器需要监听大量端口（host 模式不需要 `-p` 逐端口映射）
- 容器需要访问宿主机的网络接口（如抓包工具）

### 什么时候不能用？

- 端口冲突：两个容器不能同时监听 80 端口
- 安全隔离差：容器可以直接访问宿主机的所有网络接口
- 不可移植：在 Windows 和 macOS 上 host 模式行为不同

## 四、none 模式——"孤岛"模式

```bash
docker run --network none alpine:latest ip addr
```

输出：

```
1: lo: <LOOPBACK,UP,LOWER_UP> ...
    inet 127.0.0.1/8 scope host lo
```

只有 `lo`（回环接口），没有 `eth0`。这个容器**完全与外界隔离**——不能上网，也不能和其他容器通信。

### 什么时候用？

- 安全敏感的数据处理任务（批处理、密钥生成）
- 你准备自己用第三方网络插件接管网络（如 Calico、Weave）时，先用 none 兜底

---

## 五、三种模式对比

| 特性 | bridge | host | none |
|------|--------|------|------|
| 独立 IP | ✅ 172.17.x.x | ❌ 用宿主机 IP | ❌ 只有 127.0.0.1 |
| 端口映射 | 需要 `-p` | 自动暴露 | 无法映射 |
| 容器间通信 | ✅ 通过 IP | ✅ 通过 localhost | ❌ 完全隔离 |
| 网络性能 | 有 NAT 损耗 | 接近原生 | N/A |
| 安全隔离 | 好 | 差 | 最好 |
| 默认 | ✅ | ❌ | ❌ |

## 六、查看网络详情

### `docker network ls`——列出所有网络

```bash
docker network ls
# NETWORK ID     NAME      DRIVER    SCOPE
# abc123def456   bridge    bridge    local
# def456abc789   host      host      local
# ghi789jkl012   none      null      local
```

### `docker network inspect`——深入一个网络

```bash
docker network inspect bridge
```

关键字段解读：

| 字段 | 含义 | 你的关注点 |
|------|------|-----------|
| `Subnet` | 子网范围 | 容器 IP 都在这个网段内 |
| `Gateway` | 网关地址 | 也是 docker0 网桥的 IP |
| `Containers` | 连接到此网络的容器 | 可以看每个容器的 IP 和 MAC 地址 |
| `IPAM.Config` | IP 地址管理配置 | 可以自定义子网和网关 |

## 七、常见错误

### 错误 1：host 模式下端口冲突

```bash
# ❌ 启动一个 host 模式的 Nginx
docker run -d --name nginx1 --network host nginx:alpine

# ❌ 再启动第二个 host 模式的 Nginx
docker run -d --name nginx2 --network host nginx:alpine
# 第二个 Nginx 启动失败：80 端口已被占用
# 因为它们共享宿主机网络栈，80 端口只能被一个进程监听
```

✅ 解决办法：host 模式下，每个端口只能有一个容器使用。如果确实需要多个 Nginx，用 bridge 模式 + 不同端口映射。

### 错误 2：在默认 bridge 网络中尝试用容器名通信

```bash
# ❌ 默认 bridge 不支持容器名 DNS
docker exec app1 ping app2
# ping: bad address 'app2'
```

✅ 解决办法：下一章讲的自定义网络。或者用 IP 地址：

```bash
# 先查出 app2 的 IP
docker inspect app2 -f '{{ .NetworkSettings.IPAddress }}'
# 172.17.0.3

# 用 IP ping
docker exec app1 ping 172.17.0.3
# 通了
```

### 错误 3：以为 host 模式在 Windows/macOS 上和 Linux 一样

```bash
# 在 Windows 上运行
docker run --network host nginx:alpine
# 访问 http://localhost 可能不工作！
```

❌ 原因：Docker Desktop 在 Windows 和 macOS 上通过一个虚拟机运行 Docker 守护进程。host 模式共享的是**虚拟机**的网络栈，不是宿主机（Windows/macOS）的网络栈。你访问 `localhost` 是访问宿主机，不是虚拟机。

✅ 在 Windows/macOS 上，host 模式几乎不可用。用 bridge 模式 + 端口映射代替。

### 错误 4：不理解 bridge 模式的 IP 分配

```bash
# 你启动容器，看到 IP 是 172.17.0.2
# 然后停了又启动，IP 变成了 172.17.0.3
# 你问：为什么 IP 变了？
```

❌ 原因：默认 bridge 网络的 IP 分配是**动态的**，容器重启后 IP 可能变化。这是容器化设计的一部分——你不应该依赖容器 IP。

✅ 解决方案：用自定义网络（下一章讲），自定义网络里的容器有稳定的 DNS 名称。

## 八、我做得对不对？

### 验证方法

```bash
# 1. 查看默认网络
docker network ls
# 预期：看到 bridge、host、none

# 2. 检查 bridge 网络细节
docker network inspect bridge
# 预期：看到 Subnet、Gateway 字段

# 3. 启动一个 bridge 模式的容器，查看其 IP
docker run -d --name net-test nginx:alpine
docker inspect net-test -f '{{ .NetworkSettings.IPAddress }}'
# 预期：172.17.0.x

# 4. host 模式验证（仅 Linux）
docker run --rm --network host alpine:latest ip addr show eth0
# 预期：看到宿主机的 IP 地址（不是 172.17.0.x）

# 5. none 模式验证
docker run --rm --network none alpine:latest ip addr
# 预期：只有 lo，没有 eth0

# 清理
docker rm -f net-test
```

## 本篇最可能出错的地方及原因

### 1. host 模式在 Windows/macOS 上"不工作"

这是最让人困惑的跨平台差异。你在 Linux 上 `--network host` 跑得好好的，到 Windows 或 macOS 上就不行。原因是 Docker Desktop 在虚拟机上运行，host 模式绑定的是虚拟机的网络，不是宿主机的。**Windows/macOS 用户请远离 host 模式。**

### 2. 默认 bridge 网络不能用容器名 ping

```bash
docker exec container-a ping container-b
# ping: bad address 'container-b'
```

然后你搜了一圈，发现有人告诉你用 `--link`，有人告诉你用自定义网络。正确的答案是：**自定义网络**（下一章）。`--link` 已废弃。

### 3. 以为 `docker network inspect bridge` 里的 IP 是宿主机 IP

`docker network inspect bridge` 看到的 `Gateway: 172.17.0.1` 是 docker0 网桥的 IP，**不是**宿主机的 IP。宿主机和 docker0 之间通过 NAT 通信。外部网络无法直接访问 172.17.0.x 地址。

### 4. host 模式下多个容器端口冲突

host 模式容器共享宿主机端口空间。你启动两个 Nginx 都试图监听 80 端口，第二个必然失败。如果你需要运行多个同类服务，必须用 bridge 模式 + 不同端口映射。

### 5. 混淆 bridge 网络和 docker0 网桥

bridge 是 Docker 网络驱动的名字，docker0 是默认 bridge 网络在宿主机上创建的虚拟网桥设备名。它们是同一个东西的不同层面：bridge 是逻辑概念，docker0 是物理实现。你可以在宿主机上执行 `ip link show docker0` 看到它。
