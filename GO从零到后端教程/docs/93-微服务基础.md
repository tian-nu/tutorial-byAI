# 第93章 · 微服务基础

> "前面学的一切——HTTP服务、数据库、缓存、消息队列、设计模式、SOLID——都是让你成为一个合格的'单体应用'工程师。但从这一章开始，你要学习如何用这些积木搭建一座城市，而不只是一栋楼。微服务不是银弹，更不是'把你代码拆散'那么简单。在开始拆之前，先学会问自己：真的需要拆吗？"

---

## 93.1 单体 vs 微服务

### 小餐馆 vs 美食广场

**单体应用**就像一家**小餐馆**：
- 厨房、收银、采购、洗碗全在一个屋檐下
- 老板一个人管所有事，沟通成本为零
- 但客流一上来——厨房忙不过来了，收银台排长队了——加人？厨房就那么大，站不下了

**微服务**（将应用拆分为多个独立小服务的架构风格，详见附录I）就像一个**美食广场**（Food Court）：
- 泰餐档口管泰餐，日料档口管日料，奶茶档口管奶茶
- 每个档口独立运营：自己进货、自己定价、自己招人
- 客人流量大了——泰餐火就多开一个泰餐档口，日料不受影响

### 用代码说话

**单体应用**：

```
my-app/
├── main.go            （一个入口）
├── handlers/          （用户、订单、商品处理全在一起）
├── models/            （所有数据模型）
├── services/          （所有业务逻辑）
└── go.mod             （所有依赖）
```

所有东西在一个进程里。`userHandler` 可以直接调用 `orderService.CreateOrder()`——不需要网络调用，没有序列化开销，快得飞起。

**微服务**：

```
├── user-service/      （用户：注册/登录/个人资料）
├── order-service/     （订单：创建/支付/取消）
├── product-service/   （商品：上架/搜索/库存）
├── payment-service/   （支付：微信/支付宝/退款）
└── notification-service/（通知：短信/邮件/Push）
```

每个服务独立的代码仓库、独立的数据库、独立的部署流水线。`order-service` 创建订单时需要用户信息？不能直接调 `user-service` 的代码，要通过HTTP/gRPC请求。

### 对比表

| 维度 | 单体 | 微服务 |
|------|------|--------|
| 开发速度 | 初期快（一个项目） | 初期慢（多个项目+基础设施） |
| 部署 | 一次部署全部 | 独立部署，互不影响 |
| 扩容 | 整体扩容（用不到的功能也扩了） | 按需扩容（只扩压力大的服务） |
| 故障隔离 | 一个Bug全站崩溃 | 一个服务挂了其他继续工作 |
| 数据管理 | 一个数据库，JOIN查一切 | 每个服务自己的数据库，不能直接JOIN |
| 调试 | 简单（一个进程） | 复杂（跨服务追踪） |
| 技术栈 | 统一（全部Go） | 异构（订单Go、推荐Python、搜索Java） |

## 93.2 何时拆

### 三个信号告诉你"该拆了"

**信号一：团队变大了**

3个人的团队管一个代码仓库——沟通成本几乎为零，一个人改了啥另外两个人心里有数。

15个人的团队还在一个代码仓库——A改了B依赖的接口，B不知道→部署后崩溃。张三和李四同时在改同一个文件→合并冲突地狱。每次部署都要15个人协调——"你那边好了没？我能不能发了？"

**拆分建议**：当一个代码仓库被超过5个人同时维护，开始考虑按团队边界拆分。

**信号二：模块的独立性自然显现**

你的系统里，用户模块和订单模块有没有以下特征：
- 开发节奏不同（用户模块几个月才改一次，订单每周都在迭代）
- 压力不同（用户模块QPS 100，订单模块QPS 10000）
- 数据模型不同（用户表10个字段，订单表50个字段）
- 由不同的人负责

如果有——这些模块已经在"精神上独立"了，给它配上独立的数据库和部署流水线只是形式上的确认。

**信号三：扩展需求不同**

做活动促销→订单服务压力暴增→需要加10台服务器。但单体部署下，你只能加10台"整个应用"的服务器——用户服务根本不需要扩容，也被迫扩了。每台服务器的成本中有80%花在了不需要的模块上。

微服务下你可以只给订单服务加10台，其他服务保持现状。

### 不要过早拆分

⚠️ **最重要的警告：不要在项目初期就开始微服务。**

一个只有一个用户、每天10个请求的创业项目，拆成5个微服务——你花在基础设施上的时间（服务发现、配置中心、链路追踪、CI/CD管道）比写业务代码的时间还多。

**Martin Fowler的建议**：从单体开始，当单体带来的痛苦大到无法忍受时，再开始拆分。永远不要为了"未来可能需要"而提前拆分。

## 93.3 拆分原则

### 按业务领域拆分（DDD限界上下文）

拆分的核心原则：**不是按技术分层拆（前端/后端/数据库），而是按业务领域拆（用户/订单/商品）。**

```
❌ 错误拆分（按技术分层）：
  ├── controller-service（所有API）
  ├── business-service（所有业务逻辑）
  └── data-service（所有数据库操作）

✅ 正确拆分（按业务领域）：
  ├── user-service（用户注册、登录、个人资料）
  ├── order-service（下单、支付、退款）
  ├── product-service（商品上架、搜索、库存）
  └── notification-service（短信、邮件、推送）
```

为什么按业务拆分？因为：
- 同一个业务领域的代码天然高内聚
- 不同业务领域的代码天然低耦合
- 一个业务需求改动通常只影响一个领域

### 高内聚低耦合

在拆分时不断问自己两个问题：

1. **高内聚**："把这两个功能放在同一个服务里，是让它们更容易协调还是相互拖累？"

比如"创建订单"和"扣减库存"：创建订单必然要扣库存，两者紧密相关。但它们属于不同的业务领域（订单 vs 商品）。如果在同一个服务→高内聚但违反了领域边界。如果分开→低耦合但需要分布式事务。

这里没有"标准答案"。对于大多数电商系统，选择是：**订单创建时同步扣库存（放在一起或紧密调用），而不是异步**。因为"用户看到下单成功但库存不够"比"库存扣了订单没创建"更糟糕。

2. **低耦合**："修改这个服务时，我需要同时修改另一个服务吗？"

如果需要→耦合太高→考虑合并或重新定义边界。

### 每个服务应有独立数据库

这是微服务最"贵"的设计决策之一：

```
单体：
  user-service ─┐
  order-service ─┼──→ 同一个MySQL数据库
  product-service┘

微服务：
  user-service ──→ user-db
  order-service ──→ order-db
  product-service┘→ product-db
```

**为什么？** 共享数据库意味着一个服务的表结构修改会影响其他服务。你动了 `users` 表→订单服务的查询可能挂了。从根本上消除了"独立部署"的可能性。

**代价是什么？** 不再能跨服务JOIN。要查"用户'张三'的所有订单"→不能 `SELECT * FROM orders JOIN users` →需要调两次API（先查用户ID，再查订单）。

## 93.4 服务发现

### 问题：服务怎么找到彼此

微服务架构中，一个服务可能随时新增实例（扩容）、下线实例（缩容）、甚至整台机器挂掉。如果 `order-service` 用硬编码 `user-service: http://10.0.1.5:8080`——10.0.1.5挂了呢？

**服务发现**解决的就是这个问题：服务启动时**注册**自己，调用方通过名字**发现**实例。

### Consul

Consul是HashiCorp开源的服务发现工具，自带健康检查。

**架构**：

```
                    ┌──────────────┐
                    │   Consul     │
                    │  (注册中心)    │
                    │ 1. user-service│──→ 注册（ip:port + 健康状态）
                    │ 2. order-service│─→ 查询user-service的地址
                    └──────┬───────┘
                           │
    ┌──────────────────────┼──────────────────────┐
    │                      │                      │
    ▼                      ▼                      ▼
 user-service         order-service         product-service
 (启动时注册)         (查询并调用)            (启动时注册)
```

**Docker启动Consul**：

```bash
docker run -d \
  --name consul \
  -p 8500:8500 \
  consul:1.15
```

**Go服务注册到Consul**：

```go
config := consulapi.DefaultConfig()
config.Address = "localhost:8500"
client, _ := consulapi.NewClient(config)

registration := &consulapi.AgentServiceRegistration{
    ID:      "user-service-1",
    Name:    "user-service",
    Port:    8080,
    Address: getLocalIP(),
    Check: &consulapi.AgentServiceCheck{
        HTTP:     fmt.Sprintf("http://%s:%d/health", getLocalIP(), 8080),
        Interval: "10s",
        Timeout:  "2s",
    },
}
client.Agent().ServiceRegister(registration)
```

**Go服务发现其他服务**：

```go
services, _, err := client.Health().Service("order-service", "", true, nil)
if err != nil || len(services) == 0 {
    log.Fatal("order-service不可用")
}
target := services[0].Service
url := fmt.Sprintf("http://%s:%d/api/orders", target.Address, target.Port)
```

### Etcd

Etcd是CoreOS开源的一个高可用KV存储，被Kubernetes用作后端存储。

```bash
docker run -d \
  --name etcd \
  -p 2379:2379 \
  -e ALLOW_NONE_AUTHENTICATION=yes \
  bitnami/etcd:3.5
```

**Go操作Etcd做服务发现**：

```go
cli, _ := clientv3.New(clientv3.Config{
    Endpoints:   []string{"localhost:2379"},
    DialTimeout: 5 * time.Second,
})

lease, _ := cli.Grant(context.Background(), 10)
cli.Put(context.Background(), "/services/user-service/inst1", "10.0.1.5:8080",
    clientv3.WithLease(lease.ID))

keepAlive, _ := cli.KeepAlive(context.Background(), lease.ID)
go func() {
    for range keepAlive {
    }
}()

resp, _ := cli.Get(context.Background(), "/services/order-service",
    clientv3.WithPrefix())
for _, kv := range resp.Kvs {
    fmt.Printf("发现服务: %s = %s\n", kv.Key, kv.Value)
}
```

Etcd的TTL机制：服务的注册信息带一个过期时间，服务需要定期"心跳续约"。如果服务挂了→续约停止→TTL过期→注册信息自动删除。这就是用TTL实现了健康检查。

### Consul vs Etcd

| 维度 | Consul | Etcd |
|------|--------|------|
| 功能 | 服务发现 + 健康检查 + KV + DNS | 纯KV存储 |
| 协议 | HTTP/DNS | gRPC |
| 健康检查 | 内置（HTTP/TCP/Script） | 需要自己实现 |
| 易用性 | 高（管理界面、DNS接口） | 中（命令行为主） |
| 生态 | HashiCorp全家桶 | Kubernetes标准组件 |

大多数中小团队选Consul（功能全、开箱即用），深度K8s用户选Etcd（天然集成）。

## 93.5 API网关

### 问题：客户端怎么调用多个微服务

如果不加网关，客户端（浏览器/App）需要直接面对所有微服务：

```
浏览器 ────→ user-service:8080 （登录）
      │
      ├────→ order-service:8081（下单）
      │
      └────→ product-service:8082（浏览商品）
```

问题：
1. 客户端需要知道每个服务的地址——服务地址一变，客户端要更新
2. 每个服务都要自己实现认证、限流、日志——重复造轮子
3. 跨域（CORS）问题复杂

### 网关的作用

API网关是微服务的**统一入口**——像大楼的前台。所有外部请求都走这里，网关负责：
- **路由**：把 `/api/users` 转发到 `user-service`，把 `/api/orders` 转发到 `order-service`
- **认证**：统一验证JWT Token，不需要每个服务自己验证
- **限流**：限制单个IP的请求频率
- **日志/监控**：记录所有请求的耗时和状态码

```
浏览器 ──→ API网关 ──→ user-service
    (统一入口)   ├──→ order-service
                 └──→ product-service
```

### Traefik

云原生API网关，自动发现Docker/K8s中的服务。配置简单到令人感动。

**Docker Compose启动Traefik**：

```yaml
version: '3.8'
services:
  traefik:
    image: traefik:v2.10
    ports:
      - "80:80"
      - "8080:8080"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    command:
      - --api.insecure=true
      - --providers.docker=true
```

启动服务时加Docker label，Traefik自动发现：

```yaml
user-service:
  image: myapp/user-service
  labels:
    - "traefik.http.routers.users.rule=Host(`api.example.com`) && PathPrefix(`/api/users`)"
```

### Kong

插件生态最丰富的API网关。认证、限流、日志、转换……Kong有上百个插件。

**Docker启动Kong（需要PostgreSQL）**：

```yaml
services:
  kong-db:
    image: postgres:15
    environment:
      POSTGRES_DB: kong
      POSTGRES_USER: kong
      POSTGRES_PASSWORD: kongpass

  kong:
    image: kong:3.4
    environment:
      KONG_DATABASE: postgres
      KONG_PG_HOST: kong-db
      KONG_PG_PASSWORD: kongpass
      KONG_ADMIN_LISTEN: "0.0.0.0:8001"
    ports:
      - "8000:8000"
      - "8001:8001"
```

通过Admin API配置路由：

```bash
curl -X POST http://localhost:8001/services \
  -d "name=user-service" \
  -d "url=http://user-service:8080"

curl -X POST http://localhost:8001/services/user-service/routes \
  -d "paths[]=/api/users"
```

🤔 **想多一点**：微服务的"隐性成本"

很多人都看到微服务的"好处"——独立部署、技术异构、故障隔离。但很少有人提到它的"隐性成本"：

1. **网络开销**：单体里一个函数调用是纳秒级的，微服务里一个HTTP调用是毫秒级的——慢了1000倍。
2. **分布式事务**：之前一个 `BEGIN/COMMIT` 解决的问题，现在需要分布式事务/补偿机制。
3. **调试地狱**：之前一个日志文件看全链路，现在要跨5个服务缝合日志。
4. **运维复杂度**：之前管一个应用，现在管10个+K8s+服务网格+可观测性平台。
5. **数据一致性**：跨服务的"最终一致性"不是你想要的"最终"，而是你没办法的"最终"。

这些成本对大型团队和大流量系统来说，值得支付——换来的是独立部署和弹性伸缩。但对小团队和小项目来说，这些成本纯粹是浪费。

**记住：微服务解决的问题是"组织规模的瓶颈"和"流量的瓶颈"，而不是"代码质量的瓶颈"。** 如果你的瓶颈是代码写得烂，拆成微服务只会让你有10份烂代码。
