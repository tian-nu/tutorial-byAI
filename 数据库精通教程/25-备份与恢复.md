## 一、备份分类：逻辑 vs 物理，该用哪个？

先搞懂两个最基本的概念，别上来就敲命令。

### 1.1 逻辑备份（Logical Backup）

逻辑备份就是把数据库里的数据，**一条条导出成 SQL 文本**。就像你把书里的内容，逐字抄写到另一个本子上。

特点：
- ✅ 结果是纯文本 SQL 文件，可读、可编辑、可修改
- ✅ 跨版本、跨平台兼容性好，恢复简单
- ❌ 速度慢，大数据库（100GB+）备份恢复都很慢
- ❌ 恢复后需要重新建索引，占用更多空间

**常用工具**：`mysqldump`（MySQL 自带）

**适用场景**：中小数据库（几十GB以内）、需要随时查看修改备份内容。

### 1.2 物理备份（Physical Backup）

物理备份就是直接**复制数据文件**。就像整本书直接影印，一页不差。

特点：
- ✅ 速度快，直接拷贝文件，大数据库也快
- ✅ 恢复就是拷贝回去，不用重建索引
- ❌ 二进制文件，不可读不可编辑
- ❌ 兼容性依赖存储引擎和版本，跨平台可能出问题

**常用工具**：Xtrabackup（Percona 开源）、直接 cp 数据目录

**适用场景**：大数据库（100GB+）、追求备份恢复速度。

### 一句话总结

| 对比项 | 逻辑备份（mysqldump） | 物理备份（Xtrabackup） |
|--------|----------------------|-----------------------|
| 速度 | 慢 | 快 |
| 文件大小 | 小 | 大（和原数据差不多） |
| 可读性 | 可编辑 | 不可读 |
| 兼容性 | 好 | 一般 |
| 推荐场景 | < 50GB | > 100GB |

新手从 `mysqldump` 开始玩，足够用了。

## 二、mysqldump 最常用备份

`mysqldump` 是 MySQL 自带的工具，不用额外安装，开箱即用。

### 2.1 最基本用法

备份单个数据库：

```bash
# 语法：mysqldump -u 用户名 -p 数据库名 > 备份文件.sql
mysqldump -u root -p my_shop > my_shop_20240605.sql
```

回车后会提示输入密码，输入正确就开始备份。

> 💡 提示：密码也可以写在命令行，但**绝对不要在生产环境这么做**，会留在 shell 历史里！
> ```bash
> # ❌ 危险！密码泄露给所有能看进程的人
> mysqldump -u root -p123456 my_shop > backup.sql
>
> # ✅ 安全做法：只写 -p，让它提示输入
> mysqldump -u root -p my_shop > backup.sql
> ```

### 2.2 必须记住的三个参数

这三个参数 90% 的场景你都应该加上，少一个都可能出问题。

```bash
# InnoDB 推荐完整命令
mysqldump -u root -p \
  --single-transaction \
  --routines \
  --triggers \
  my_shop > my_shop_backup.sql
```

逐个解释：

| 参数 | 作用 | 为什么必须加？ |
|------|------|---------------|
| `--single-transaction` | 在一个事务中导出，利用 MVCC 不锁表 | InnoDB 如果不加，会**锁整个表**，备份期间所有写入都卡住！ |
| `--routines` | 导出存储过程和函数 | 默认不导，如果你用了存储过程，恢复后发现全没了！ |
| `--triggers` | 导出触发器 | 默认虽然导，但某些旧版本不导，显式加上保险 |

**想多一点：为什么 `--single-transaction` 能不锁表？**

因为 InnoDB 支持 MVCC（第 22 章学过），导出开始时拿一个一致性快照，整个导出过程中，别的事务还能正常读写。对业务几乎没有影响。

MyISAM 不支持 MVCC，只能用 `--lock-tables` 锁表导出。所以生产环境真的别用 MyISAM 了。

### 2.3 其他常用场景

**备份多个数据库：**

```bash
# --databases 后面跟多个库名
mysqldump -u root -p --single-transaction --databases db1 db2 db3 > multi_backup.sql
```

**备份所有数据库：**

```bash
mysqldump -u root -p --single-transaction --all-databases > all_db_backup.sql
```

**只备份表结构不备份数据：**

```bash
# --no-data 只导结构
mysqldump -u root -p --no-data my_shop > my_shop_schema.sql
```

这招很有用！比如你要把生产库的表结构同步到测试库，只导结构就行。

### 2.4 恢复：把备份导回去

恢复比备份还简单：

```bash
# 先创建空数据库（如果还没有）
mysql -u root -p -e "CREATE DATABASE IF NOT EXISTS my_shop CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"

# 恢复
mysql -u root -p my_shop < my_shop_backup.sql
```

就是这么直接。如果你备份的时候用了 `--all-databases`，恢复的时候直接：

```bash
mysql -u root -p < all_db_backup.sql
```

它会自动创建库，不用你提前建。

## 三、binlog 点时间恢复：误删表救回数据

这是 DBA 必备技能。假设：

- 你每天凌晨 2 点做全量备份
- 今天上午 10 点，有人手滑 `DROP TABLE orders;`
- 全量备份是凌晨 2 点，恢复的话会丢 8 小时数据

怎么办？**用 binlog 恢复到删表前一刻！**

### 3.1 确认 binlog 已开启

先检查你的 MySQL 有没有开 binlog：

```sql
-- 查看 binlog 配置
SHOW VARIABLES LIKE 'log_bin';
-- 应该返回 ON，如果是 OFF 需要先开启
```

开启 binlog 需要在 `my.cnf` / `my.ini` 中配置：

```ini
# my.cnf 添加这两行
log_bin = /var/log/mysql/mysql-bin.log
server-id = 1
expire_logs_days = 7  # 7 天自动清理旧 binlog
```

改完配置重启 MySQL 生效。**生产环境强烈建议开启 binlog，这是最后一道救命防线。**

### 3.2 找到误删时间点

方法一：看 binlog 事件

```bash
# 查看当前所有 binlog 文件
mysql -u root -p -e "SHOW BINARY LOGS;"
```

输出大概这样：

```
+------------------+-----------+
| Log_name         | File_size |
+------------------+-----------+
| mysql-bin.000001 | 123456    |
| mysql-bin.000002 |  78901    |
| mysql-bin.000003 | 456789    |  <-- 当前在用的就是这个
+------------------+-----------+
```

查看 binlog 内容找 `DROP TABLE`：

```bash
mysqlbinlog mysql-bin.000003 | grep -n "DROP TABLE"
```

找到行号，看一下时间：

```bash
# 看附近的内容，找到确切时间
mysqlbinlog mysql-bin.000003 | head -n 200 | tail -n 50
```

方法二：如果你记得大概时间（比如上午 10:15 误删），直接用时间点恢复。

### 3.3 基于时间点恢复

```bash
# --stop-datetime：恢复到这个时间点之前（不包含这个时间点）
# 格式：YYYY-MM-DD HH:MM:SS
# 假设误删时间是 2024-06-05 10:15:30，我们恢复到 10:15:29

# 第一步：先恢复全量备份（凌晨 2 点的）
mysql -u root -p my_shop < full_backup_20240605.sql

# 第二步：应用 binlog 从全量备份时间到误删前一刻
mysqlbinlog \
  --stop-datetime="2024-06-05 10:15:29" \
  /var/log/mysql/mysql-bin.000003 \
  | mysql -u root -p my_shop
```

完美！现在数据恢复到了删表前一秒，只丢了最后 1 秒的数据，大部分场景完全可以接受。

如果你只想恢复中间一段（从 A 时间到 B 时间），加上 `--start-datetime`：

```bash
mysqlbinlog \
  --start-datetime="2024-06-05 02:00:00" \
  --stop-datetime="2024-06-05 10:15:29" \
  /var/log/mysql/mysql-bin.000003 \
  | mysql -u root -p my_shop
```

### 3.4 完整演练：从误删到恢复

我们来做一遍完整实战，确保你真的会。

```bash
# 1. 当前已有数据库，做个全量备份
mysqldump -u root -p --single-transaction --routines --triggers my_shop > full_backup.sql

# 2. 查看当前 binlog 文件
mysql -u root -p -e "SHOW MASTER STATUS;"
```

`SHOW MASTER STATUS` 输出类似：

```
+------------------+----------+--------------+------------------+
| File             | Position | Binlog_Do_DB  | Binlog_Ignore_DB |
+------------------+----------+--------------+------------------+
| mysql-bin.000001 |      156 |              |                  |
+------------------+----------+--------------+------------------+
```

记住当前的 File 和 Position，后面要用。

```sql
-- 3. 插入一些测试数据
USE my_shop;
INSERT INTO users (username, email) VALUES ('alice', 'alice@test.com');
INSERT INTO users (username, email) VALUES ('bob', 'bob@test.com');
INSERT INTO users (username, email) VALUES ('charlie', 'charlie@test.com');
SELECT COUNT(*) FROM users;  -- 假设现在是 3 条
```

现在，有人**手滑误删表**：

```sql
-- ❌ 灾难！
DROP TABLE users;
SELECT COUNT(*) FROM users;  -- 报错 Table doesn't exist
```

别慌，开始恢复：

```bash
# 4. 先恢复全量备份（这个备份里 users 表还在，但只有 0 条或旧数据）
mysql -u root -p my_shop < full_backup.sql

# 5. 应用 binlog 从备份位置到删表前
# 假设备份后 binlog 从 Position 156 开始，我们现在找到误删时间点
# 用 mysqlbinlog 查看找到确切位置，或者直接用时间

mysqlbinlog \
  --start-position=156 \
  --stop-datetime="2024-06-05 10:15:00" \
  /var/log/mysql/mysql-bin.000001 \
  | mysql -u root -p my_shop
```

验证恢复结果：

```sql
USE my_shop;
SELECT COUNT(*) FROM users;  -- 应该是 3 条，三条记录都回来了！
SELECT * FROM users;  -- 数据完整
```

就是这么神奇！全量备份 + binlog，能把数据恢复到任意时间点。

## 四、Xtrabackup：大数据库热备份

如果你数据库有几百GB甚至上TB，`mysqldump` 导几个小时，恢复也要几个小时，业务扛不住怎么办？

用 **Xtrabackup** *此术语见附录E*，这是 Percona 开源的物理备份工具，**热备份不锁表**，速度比 mysqldump 快好几倍。

### 4.1 适用场景

- ✅ 数据库 > 100GB
- ✅ 需要不锁表备份（业务不能停）
- ✅ 要求恢复快

不适用：

- ❌ < 50GB 的小数据库，mysqldump 足够，更简单
- ❌ 非 InnoDB 引擎（Xtrabackup 对 InnoDB 支持最好）

### 4.2 安装

**Debian/Ubuntu:**

```bash
wget https://repo.percona.com/apt/percona-release_latest.generic_all.deb
dpkg -i percona-release_latest.generic_all.deb
apt update
apt install percona-xtrabackup-80
```

**CentOS/RHEL:**

```bash
yum install https://repo.percona.com/yum/percona-release-latest.noarch.rpm
yum install percona-xtrabackup-80
```

**macOS (Homebrew):**

```bash
brew install percona-xtrabackup
```

### 4.3 完整备份

```bash
# 创建备份目录
mkdir -p /backup/mysql

# 全量备份
xtrabackup \
  --user=root \
  --password=your_password_here \
  --backup \
  --target-dir=/backup/mysql/full_20240605
```

- `--backup`：备份模式
- `--target-dir`：备份输出目录

备份完成会输出 `completed OK!` 就成功了。

### 4.4 准备备份（prepare）

恢复前必须先 prepare：

```bash
xtrabackup \
  --prepare \
  --target-dir=/backup/mysql/full_20240605
```

这一步会把备份过程中产生的日志应用进去，让数据文件一致。prepare 完成才能恢复。

### 4.5 恢复

```bash
# 停止 MySQL 服务
systemctl stop mysql

# 清空旧数据目录（注意：先备份！）
rm -rf /var/lib/mysql/*

# 恢复
xtrabackup \
  --copy-back \
  --target-dir=/backup/mysql/full_20240605 \
  --datadir=/var/lib/mysql

# 修复权限
chown -R mysql:mysql /var/lib/mysql

# 启动 MySQL
systemctl start mysql
```

就是这么直接，物理拷贝比导入 SQL 快太多了。

## 五、备份策略：怎么设计才靠谱？

再牛的工具，策略错了一样死得很惨。分享一个生产环境用了很多年的成熟策略：

### 5.1 推荐策略：全量 + binlog + 定期演练

```
每天凌晨 2 点：全量备份（mysqldump 或 Xtrabackup）
实时：binlog 自动记录（本来就开着）
保留：最近 7 天的全量备份，binlog 保留 7 天
每周日：随机拿一个备份做恢复演练，确认能正常恢复
备份文件同步到异地存储（OSS、另一个服务器）
```

### 5.2 增量备份？

很多人一听就想"我要做增量备份省空间"，实际上：

- 优点：省空间，备份快
- 缺点：恢复麻烦，任何一个增量坏了全链都废了
- 如果磁盘空间够，每天全量备份最简单可靠

**一句话：能每天全量就每天全量，空间不够再考虑增量。**

### 5.3 备份验证！备份验证！备份验证！

重要的事说三遍。

你以为备份了，其实备份文件损坏了，真出事了恢复不出来，等于没备。

```bash
# 每次备份完，抽一天找个测试库恢复一下
mysql -u root -p test_db < backup.sql

# 恢复完数行数
mysql -u root -p test_db -e "SELECT COUNT(*) FROM users;"
```

对比原库行数，对上了才算成功。

## 我做得对不对？

### 验证方法

```bash
# 1. 备份测试：用 mysqldump 备份 my_shop
mysqldump -u root -p --single-transaction --routines --triggers my_shop > test_backup.sql

# 2. 检查备份文件大小
ls -lh test_backup.sql
# 应该和你的数据库大小匹配，不会是 0 字节

# 3. 恢复到新数据库测试
mysql -u root -p -e "CREATE DATABASE my_shop_restore CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
mysql -u root -p my_shop_restore < test_backup.sql

# 4. 验证数据行数
mysql -u root -p my_shop_restore -e "SELECT COUNT(*) FROM users;"
# 和原库对比：
mysql -u root -p my_shop -e "SELECT COUNT(*) FROM users;"
# 两个数字一致 → ✅ 成功

# 5. 清理测试库
mysql -u root -p -e "DROP DATABASE my_shop_restore;"
rm test_backup.sql
```

### binlog 恢复验证

```bash
# 按照本章 3.4 节完整演练一遍
# 插入数据 → 误删 → 全量恢复 → binlog 恢复
# 最终数据行数正确 → ✅ 成功
```

## 不对怎么办？

### 常见错误 1：备份 InnoDB 不加 `--single-transaction` 导致锁表

❌ 错误：
```bash
# 什么参数都不加，InnoDB 会锁表
mysqldump -u root -p my_shop > backup.sql
# 备份期间，所有写操作都卡住，业务停了！
```

✅ 正确：
```bash
mysqldump -u root -p --single-transaction --routines --triggers my_shop > backup.sql
```

排错：备份完检查一下，如果你看到 `FLUSH TABLES WITH READ LOCK` 在进程列表里，说明锁表了。下次记得加参数。

### 常见错误 2：备份文件从来不验证，出事了才发现坏了

现象：备份文件大小是 0，或者 corrupted，恢复不出来。

原因：磁盘满了、权限不够、备份进程被 killed。

解决：**每次备份完自动校验**，写个简单的 shell 脚本：

```bash
#!/bin/bash
# 备份脚本示例
DATE=$(date +%Y%m%d)
mysqldump -u root -pyour_password_here --single-transaction --routines --triggers my_shop > /backup/my_shop_$DATE.sql

# 检查文件大小
FILE_SIZE=$(stat -c%s /backup/my_shop_$DATE.sql)
if [ $FILE_SIZE -lt 1024 ]; then
  echo "备份失败！文件太小，通知管理员！"
  # 发邮件/钉钉报警
  exit 1
fi
echo "备份成功，大小: $FILE_SIZE bytes"
```

### 常见错误 3：binlog 开了，但设置 `expire_logs_days` 太小，旧 binlog 被清了

现象：想恢复到一周前，发现 binlog 早就自动删了。

解决：根据你的全量备份频率设置保留时间：
- 每天全量 → 保留 2 天足够
- 每周全量 → 保留 8 天
- 追求安全 → 保留 7-15 天

```ini
# my.cnf 配置
expire_logs_days = 7
```

### 常见错误 4：Xtrabackup 恢复完启动失败，权限不对

现象：启动 MySQL 报错 `Permission denied`。

原因：`--copy-back` 过来的文件所有者是 root，不是 mysql。

✅ 解决：必须记得改权限：

```bash
chown -R mysql:mysql /var/lib/mysql
```

### 常见错误 5：mysqldump 备份完，存储过程和触发器丢了

现象：恢复完，能查到数据，但找不到存储过程。

原因：默认没加 `--routines` 参数。

✅ 解决：永远加上三个参数：`--single-transaction --routines --triggers`。

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 逻辑备份 | mysqldump，导出 SQL 文本 | 中小数据库适用，可读可编辑 |
| 物理备份 | Xtrabackup，复制数据文件 | 大数据库适用，速度快 |
| mysqldump 基础备份 | `mysqldump -u root -p db > backup.sql` | 密码不要写在命令行 |
| 必加参数 | `--single-transaction --routines --triggers` | InnoDB 不加锁表 |
| 恢复备份 | `mysql -u root -p db < backup.sql` | 提前建库（除非 --all-databases） |
| 点时间恢复 | `mysqlbinlog --stop-datetime ... | mysql` | 误删后能恢复到删前一刻 |
| binlog 开启 | `log_bin` + `server-id` 写入 my.cnf | 生产必须开，救命用 |
| Xtrabackup 流程 | backup → prepare → copy-back | 恢复后改权限 |
| 备份策略 | 每天全量 + binlog + 异地存储 + 定期演练 | 最简单最可靠 |
| 备份验证 | 恢复到测试库数行数 | 不验证等于没备份 |

---

> 本章需要记住一句话：**有备份没验证 = 没备份**。定期恢复演练，比什么都重要。

---

## 最可能出错的地方及原因

1. **忘记 `--single-transaction`**：对 InnoDB 来说会锁表，导致业务长时间卡住，原因是不知道这个参数的作用。
2. **密码写在命令行**：会被 `ps` 命令看到，泄露给其他用户，原因是图方便。
3. **从来不验证备份**：备份文件损坏了也不知道，出事才发现恢复不了，原因是侥幸心理。
4. **binlog 保留时间太短**：需要恢复时发现旧日志已经被清了，原因是配置时没考虑全量备份周期。
5. **Xtrabackup 恢复后忘记改权限**：MySQL 启动失败，原因是不了解 Linux 文件权限机制。
