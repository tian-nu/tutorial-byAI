# 第113章 · 项目四：电商秒杀系统

> "如果前面的项目是开普通汽车，这一章就是开F1赛车。秒杀（限时限量超低价抢购活动，瞬时高并发场景的典型代表，详见附录I）是后端面试中最常考的高并发场景——100万人同时抢100件商品，你的代码不能有任何的疏忽。一个微小的竞态条件，就可能导致超卖：只准备了100件货，却卖出了200单。本章会让你真正理解什么是并发安全、什么是原子操作、以及Redis在其中的关键角色。"

## 113.1 场景复现：秒杀到底难在哪

### 一个典型的秒杀场景

某电商平台宣布：今晚8点，iPhone 15 限量100台，五折秒杀。结果：

- 8:00:00 瞬间涌入50万并发请求
- 数据库的库存字段是这样更新的：
  ```sql
  UPDATE products SET stock = stock - 1 WHERE id = 1 AND stock > 0;
  ```
- 如果100个请求**同时**读到 `stock = 1`，每个都会觉得"还有库存"，然后各自减1——最终库存变成-99

这就是**超卖**（库存卖出数量超过实际数量，并发读写竞态条件导致，详见附录I）的根源：读和写之间有缝隙。就像100个人同时看到"还剩1间房"，然后同时付款——酒店只能给一个人住，但99个人都付了钱。

### 秒杀系统的四大核心挑战

| 挑战 | 描述 | 解决方案 |
|------|------|---------|
| 防止超卖 | 100件货不能卖出101件 | Redis预减库存+Lua脚本原子操作 |
| 高并发处理 | 瞬间50万QPS | 消息队列削峰填谷，异步下单 |
| 缓存策略 | 数据库扛不住 | Redis缓存商品信息，减少数据库压力 |
| 限流防刷 | 有人用脚本刷接口 | 令牌桶算法限流 |

```
    │  ┌──────────┐  ┌──────────┐  ┌────────┐  │
    │  │ 商品Handler│  │ 秒杀Handler│  │订单Handler│
    │  └─────┬────┘  └────┬─────┘  └───┬────┘  │
    └────────┼────────────┼─────────────┼───────┘
             │            │             │
    ┌────────┼────────────┼─────────────┼───────┐
    │        v            v             v       │
    │  ┌──────────┐  ┌──────────────┐ ┌──────┐  │
    │  │  MySQL    │  │    Redis     │ │ 消息  │  │
    │  │ 持久化存储  │  │  缓存+库存    │ │ 队列  │  │
    │  └──────────┘  └──────────────┘ └──────┘  │
    └──────────────────────────────────────────┘
```

请求进来后：
1. 先过限流——令牌桶判断能否放行
2. 再过认证——确认你是谁
3. 秒杀请求到Redis预减库存
4. 预减成功返回"抢到了"，异步发消息到队列
5. 队列消费者慢慢写MySQL订单

这就是"先到先得，后方记账"的策略——就像餐厅收银台先给你个号牌，后厨再慢慢做菜。

### 字段说明

- `products.stock`：商品的常规库存（非秒杀场景用）
- `seckill_events.stock`：秒杀活动的库存，独立于常规库存。秒杀活动有独立的开始和结束时间
- `orders.status`：0=待支付，1=已支付，2=已取消，3=已退款
- `orders.order_no`：订单号，唯一索引防止重复下单
- `seckill_events.status`：0=未开始，1=进行中，2=已结束

秒杀库存和常规库存分开——就像超市的促销区有独立的库存，和正常货架的库存互不影响。

## 113.5 核心难点一：防止超卖

### 方案对比

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| MySQL行锁 | `SELECT ... FOR UPDATE` | 可靠但QPS低（~1000），秒杀场景直接挂 |
| MySQL乐观锁 | `UPDATE ... WHERE stock > 0 AND version = ?` | 比行锁好但仍有数据库压力 |
| Redis预减库存 | 先减Redis，异步写MySQL | 高性能（10万+QPS），但需要处理一致性 |
| Redis+Lua | Lua脚本保证原子性 | **推荐方案**——兼顾性能和数据一致性 |

### Redis+Lua原子预减库存

核心思想：把"判断库存→扣减库存"这两个操作打包成一个Lua脚本，让Redis一次性执行完。就像你去ATM取钱——你不会希望"查余额"和"扣款"之间有人插队。

```lua
local key = KEYS[1]
local user_id = ARGV[1]
local stock = tonumber(redis.call('get', key))

if stock == nil then
    return -1
end

if stock <= 0 then
    return -2
end

local bought_key = key .. ':bought:' .. user_id
local bought = redis.call('exists', bought_key)
if bought == 1 then
    return -3
end

redis.call('decr', key)
redis.call('set', bought_key, 1)

return stock - 1
```

这个Lua脚本做了四件事（全部原子执行）：
1. 检查库存是否存在（-1：活动不存在）
2. 检查库存是否大于0（-2：已售罄）
3. 检查用户是否已购买（-3：不能重复购买）
4. 减库存 + 标记用户已购买

返回值的含义：
- `>= 0`：秒杀成功，返回值是剩余库存
- `-1`：秒杀活动不存在
- `-2`：库存不足，已售罄
- `-3`：你已经抢过了

### Go中执行Lua脚本

```go
package model

import (
    "context"
    "fmt"
    "github.com/redis/go-redis/v9"
)

const luaScript = `
local key = KEYS[1]
local user_id = ARGV[1]
local stock = tonumber(redis.call('get', key))
if stock == nil then
    return -1
end
if stock <= 0 then
    return -2
end
local bought_key = key .. ':bought:' .. user_id
local bought = redis.call('exists', bought_key)
if bought == 1 then
    return -3
end
redis.call('decr', key)
redis.call('set', bought_key, 1)
return stock - 1
`

func TrySeckill(ctx context.Context, rdb *redis.Client, eventID, userID uint64) (int64, error) {
    key := fmt.Sprintf("seckill:stock:%d", eventID)
    result, err := rdb.Eval(ctx, luaScript, []string{key}, userID).Result()
    if err != nil {
        return 0, err
    }
    return result.(int64), nil
}

func InitSeckillStock(ctx context.Context, rdb *redis.Client, eventID uint64, stock int) error {
    key := fmt.Sprintf("seckill:stock:%d", eventID)
    return rdb.Set(ctx, key, stock, 0).Err()
}
```

秒杀开始前，把活动库存加载到Redis：
- `seckill:stock:1` → 100（活动1的库存100件）
- 秒杀时执行Lua脚本原子减库存
- `seckill:stock:1:bought:123` → 标记用户123已购买

## 113.6 核心难点二：消息队列异步下单

秒杀成功后，不能直接在HTTP请求里写MySQL——50万并发请求同时写数据库，数据库瞬间崩溃。解决方案：**消息队列削峰**。

```
用户秒杀请求 ──> Redis原子减库存成功 ──> 发送消息到队列 ──> 返回"抢到了"
                                                    │
                                                    v
                                        消费者慢慢处理 ──> 写入MySQL订单
```

消息就像排队买奶茶——店员先给你一个取餐号（秒杀成功），然后你拿着号等着（异步），后厨做好一杯叫一杯（消费者处理）。

### 使用Go Channel实现消息队列

```go
package mq

import "log"

type SeckillMessage struct {
    UserID      uint64
    EventID     uint64
    ProductID   uint64
    ProductName string
    Amount      float64
    OrderNo     string
}

type SeckillMQ struct {
    queue chan SeckillMessage
}

func NewSeckillMQ(bufferSize int) *SeckillMQ {
    return &SeckillMQ{
        queue: make(chan SeckillMessage, bufferSize),
    }
}

func (mq *SeckillMQ) Publish(msg SeckillMessage) {
    mq.queue <- msg
}

func (mq *SeckillMQ) Consume(handler func(SeckillMessage) error) {
    for msg := range mq.queue {
        if err := handler(msg); err != nil {
            log.Printf("处理秒杀消息失败: %v", err)
        }
    }
}
```

`channel` 是Go内置的并发安全队列——多个goroutine可以安全地往里面发消息。bufferSize=10000意味着可以缓存1万条消息，瞬间流量再大也不会丢失。

### 消费者：写MySQL订单

```go
func StartSeckillConsumer(db *gorm.DB, mq *SeckillMQ) {
    mq.Consume(func(msg SeckillMessage) error {
        order := &Order{
            OrderNo:         msg.OrderNo,
            UserID:          msg.UserID,
            ProductID:       msg.ProductID,
            ProductName:     msg.ProductName,
            Amount:          msg.Amount,
            SeckillEventID:  msg.EventID,
            Status:          0,
        }

        err := db.Transaction(func(tx *gorm.DB) error {
            if err := tx.Create(order).Error; err != nil {
                return err
            }

            result := tx.Model(&SeckillEvent{}).
                Where("id = ? AND stock > 0", msg.EventID).
                UpdateColumn("stock", gorm.Expr("stock - 1"))
            if result.RowsAffected == 0 {
                return fmt.Errorf("MySQL库存扣减失败")
            }

            return nil
        })
        return err
    })
}
```

注意：即使Redis已经减了库存，MySQL还得再减一次——这叫**最终一致性**。Redis的减库存是"第一道防线"（防止超卖），MySQL的减库存是"持久化记录"（真实的库存数据）。

## 113.7 核心难点三：Redis缓存策略

### 缓存什么

- 商品信息：经常被查询，变化不频繁
- 秒杀活动信息：秒杀期间被疯狂查询
- 活动库存：这是最关键的热数据

### 缓存模式：Cache Aside

```
读数据：
  查Redis → 有就返回
         → 没有就查MySQL → 写入Redis → 返回

写数据：
  更新MySQL → 删除Redis缓存（让下次读重新加载）
```

为什么不直接更新Redis而是删除？因为并发时可能产生"写覆盖"问题——A先更新Redis，B后更新Redis，如果B的MySQL数据更新在A之前，Redis里的就是A的旧数据。删除缓存让下次读取时重新从MySQL加载最新数据，更安全。

```go
func GetProductCache(ctx context.Context, rdb *redis.Client, db *gorm.DB, productID uint64) (*Product, error) {
    cacheKey := fmt.Sprintf("product:%d", productID)

    cached, err := rdb.Get(ctx, cacheKey).Result()
    if err == nil {
        var product Product
        json.Unmarshal([]byte(cached), &product)
        return &product, nil
    }

    var product Product
    if err := db.First(&product, productID).Error; err != nil {
        return nil, err
    }

    data, _ := json.Marshal(product)
    rdb.Set(ctx, cacheKey, data, 10*time.Minute)

    return &product, nil
}

func InvalidateProductCache(ctx context.Context, rdb *redis.Client, productID uint64) {
    cacheKey := fmt.Sprintf("product:%d", productID)
    rdb.Del(ctx, cacheKey)
}
```

## 113.8 核心难点四：令牌桶限流

秒杀场景下，有人会用脚本疯狂刷接口。限流的目标不是阻止所有人，而是**保护系统不被冲垮**——就像商场限流，"满5000人暂停进场"，不是不让进，而是排队。

### 令牌桶算法

想象一个桶：
- 以固定速率（如每秒100个）往桶里放令牌
- 桶有最大容量（如200个），满了就溢出
- 每个请求必须拿到一个令牌才能处理
- 桶里没令牌时，请求被拒绝

这个算法天然允许**突发流量**：如果桶里积累了200个令牌，瞬间可以处理200个并发请求，然后恢复为每秒100个的匀速。

### Go实现令牌桶限流中间件

```go
package middleware

import (
    "net/http"
    "sync"
    "time"

    "github.com/gin-gonic/gin"
)

type TokenBucket struct {
    rate       float64
    capacity   float64
    tokens     float64
    lastRefill time.Time
    mu         sync.Mutex
}

func NewTokenBucket(rate, capacity float64) *TokenBucket {
    return &TokenBucket{
        rate:       rate,
        capacity:   capacity,
        tokens:     capacity,
        lastRefill: time.Now(),
    }
}

func (tb *TokenBucket) Allow() bool {
    tb.mu.Lock()
    defer tb.mu.Unlock()

    now := time.Now()
    elapsed := now.Sub(tb.lastRefill).Seconds()
    tb.tokens += elapsed * tb.rate
    if tb.tokens > tb.capacity {
        tb.tokens = tb.capacity
    }
    tb.lastRefill = now

    if tb.tokens >= 1 {
        tb.tokens--
        return true
    }
    return false
}

func RateLimit(rate, capacity float64) gin.HandlerFunc {
    bucket := NewTokenBucket(rate, capacity)
    return func(c *gin.Context) {
        if !bucket.Allow() {
            c.JSON(http.StatusTooManyRequests, gin.H{
                "code":    429,
                "message": "请求过于频繁，请稍后再试",
                "data":    nil,
            })
            c.Abort()
            return
        }
        c.Next()
    }
}
```

限流参数配置：
- 普通接口：`RateLimit(100, 200)`——每秒100个，突发200个
- 秒杀接口：根据实际库存调小，防止刷单——`RateLimit(50, 100)`

### handler/seckill.go — 秒杀Handler

```go
package handler

import (
    "fmt"
    "net/http"
    "strconv"
    "time"

    "seckill-api/model"
    "seckill-api/mq"

    "github.com/gin-gonic/gin"
    "github.com/google/uuid"
    "github.com/redis/go-redis/v9"
    "gorm.io/gorm"
)

type SeckillHandler struct {
    DB  *gorm.DB
    RDB *redis.Client
    MQ  *mq.SeckillMQ
}

func NewSeckillHandler(db *gorm.DB, rdb *redis.Client, m *mq.SeckillMQ) *SeckillHandler {
    return &SeckillHandler{DB: db, RDB: rdb, MQ: m}
}

func (h *SeckillHandler) DoSeckill(c *gin.Context) {
    userID := c.GetUint64("user_id")
    eventID, _ := strconv.ParseUint(c.Param("event_id"), 10, 64)

    var event model.SeckillEvent
    if err := h.DB.Preload("Product").First(&event, eventID).Error; err != nil {
        c.JSON(http.StatusNotFound, gin.H{
            "code": 404, "message": "秒杀活动不存在", "data": nil,
        })
        return
    }

    now := time.Now()
    if now.Before(event.StartTime) {
        c.JSON(http.StatusBadRequest, gin.H{
            "code": 400, "message": "秒杀尚未开始", "data": nil,
        })
        return
    }
    if now.After(event.EndTime) {
        c.JSON(http.StatusBadRequest, gin.H{
            "code": 400, "message": "秒杀已结束", "data": nil,
        })
        return
    }

    result, err := model.TrySeckill(c.Request.Context(), h.RDB, eventID, userID)
    if err != nil {
        c.JSON(http.StatusInternalServerError, gin.H{
            "code": 500, "message": "系统繁忙，请稍后再试", "data": nil,
        })
        return
    }

    switch result {
    case -1:
        c.JSON(http.StatusNotFound, gin.H{
            "code": 404, "message": "秒杀活动不存在", "data": nil,
        })
    case -2:
        c.JSON(http.StatusOK, gin.H{
            "code": 200, "message": "已售罄", "data": gin.H{
                "result": "sold_out",
            },
        })
    case -3:
        c.JSON(http.StatusOK, gin.H{
            "code": 200, "message": "您已参与过本次秒杀", "data": gin.H{
                "result": "already_bought",
            },
        })
    default:
        orderNo := fmt.Sprintf("SK%s%d", time.Now().Format("20060102150405"), userID)
        msg := mq.SeckillMessage{
            UserID:    userID,
            EventID:   eventID,
            ProductID: event.ProductID,
            OrderNo:   orderNo,
        }
        h.MQ.Publish(msg)

        c.JSON(http.StatusOK, gin.H{
            "code": 200, "message": "秒杀成功，订单处理中", "data": gin.H{
                "result":       "success",
                "order_no":     orderNo,
                "product_name": event.Product.Name,
                "price":        event.SeckillPrice,
            },
        })
    }
}
```

秒杀成功后立即返回，不等待订单写MySQL——用户看到"秒杀成功"只需要几十毫秒，而订单写入可能在几百毫秒后完成。这就是异步的魅力：**让用户感觉快，让系统稳得住。**

## 本章小结

这一章是六个项目中最硬核的。你学到了：

1. **防止超卖**：Redis预减库存 + Lua脚本原子操作，这是秒杀系统的灵魂

5. **架构设计思维**：面对高并发，不是硬扛，而是分流、削峰、降级

- "有人刷接口怎么办？" → 令牌桶限流
