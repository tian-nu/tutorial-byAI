# 第13章 · HTTPS

> "HTTP等于在大街上裸奔传银行卡密码。HTTPS给它穿上了一套盔甲——但盔甲是怎么穿上的？锁和钥匙的比喻，一次给你讲透。"

---

## 13.1 HTTP的三大致命问题

HTTP协议天生就有三个致命缺陷：

| 问题 | 后果 | 比喻 |
|------|------|------|
| **明文传输** | 任何人截获网络数据，都能直接看到内容——包括你的密码、身份证号、聊天记录 | 在大街上用明信片给人传银行卡密码 |
| **无身份验证** | 你以为你在和淘宝通信，实际上中间人可能伪造了淘宝的回应 | 你打电话给银行，但接电话的其实是骗子 |
| **可被篡改** | 中间人可以修改你接收或发送的内容而不被你发现 | 快递员把你的快递打开，换了个东西再封好寄给你 |

HTTPS = HTTP + TLS（Transport Layer Security，加密传输协议，详见附录I）。TLS在HTTP和TCP之间加了一层**加密保护**（数据转码防窃取，详见附录I）。

```
HTTP:      [HTTP明文] → 直接在TCP上传输
HTTPS:     [HTTP明文] → [TLS加密层] → [TCP传输]
```

TLS做了三件事，正好对应上面三个问题：

- **加密** → 解决"明文传输"（内容被加密，截获了也看不懂）
- **身份验证** → 解决"可能被冒充"（用数字证书证明"我真的是淘宝"）
- **完整性校验** → 解决"可能被篡改"（用MAC校验，改了就能发现）

## 13.2 对称加密

### 一把锁，两把钥匙

想象你有一个**带锁的盒子**。你用钥匙锁上它，快递给朋友。朋友用**同一把钥匙**打开它。

这就是**对称加密**：加密和解密用的是**同一把密钥**。

```
明文 ──→ [加密 + 密钥K] ──→ 密文 ──→ [解密 + 密钥K] ──→ 明文
发送方                                                  接收方
```

### 优点和致命缺点

- **优点**：速度快，算法简单（AES等算法有CPU硬件加速）
- **致命缺点**：**你怎么把钥匙安全地交给对方？**

如果你在网上和一个素未谋面的人通信，你怎么把密钥传给他？直接发过去？万一被中间人截获了，他也能解密。打电话告诉他？电话也可能被监听。见个面亲手交给他？那你还不如直接把要传的内容当面告诉对方。

这就是**密钥交换问题**（Key Exchange Problem）——对称加密本身是好的，但**密钥的传输本身需要一条安全通道**。

> 🤔 想多一点：你可能会想"那就用另一把对称密钥来加密第一把对称密钥，再用第三把来加密第二把……"——这是无限套娃，总会有一把密钥需要明文传输。这个死循环直到非对称加密的出现才被打破。

---

## 13.3 非对称加密

### 锁和钥匙的分离

非对称加密有**两把**不同的钥匙：

- **公钥（Public Key）**：可以公开给任何人。用公钥加密的数据，只能用对应的私钥解密。类比：一个**敞开的锁**——谁都可以把它扣上，但只有有钥匙的人能打开。
- **私钥（Private Key）**：只有自己知道，绝对不公开。类比：打开那把锁的**钥匙**。

```
公钥加密 → 私钥解密
私钥签名 → 公钥验签
```

### 一幅图解

```
                     公钥（锁）
                     ┌──────┐
明文 ──→ [加密+公钥] →│ 密文 │
                     └──┬───┘
                        │
                  只有私钥能打开
                        │
                     ┌──┴───┐
密文 ──→ [解密+私钥] →│ 明文 │
                     └──────┘
                     私钥（钥匙）
```

### 解决密钥交换问题的方式

有了非对称加密，之前那个"如何安全传递对称密钥"的问题终于有了答案：

```
客户端                                   服务端
   │                                        │
   │  ←── 服务端公开自己的公钥 ────────      │  （公钥可以公开传！）
   │                                        │
   │  客户端生成一个"会话密钥"               │
   │  用服务端的公钥加密这个"会话密钥"        │
   │  ──── 加密后的会话密钥 ─────────→      │
   │                                        │
   │  服务端用私钥解密，得到"会话密钥"        │
   │                                        │
   │════ 之后双方用"会话密钥"做对称加密 ════│
```

但实际上这个简单流程有中间人攻击的问题，后面会讲。

> 🤔 想多一点：非对称加密比对称加密慢得多——慢几百到上千倍。所以实际中不会全程用非对称加密，而是只用来传递一小段"会话密钥"，之后的通信都换成对称加密。这就是HTTPS的核心精妙之处：**用非对称加密的安全性和对称加密的速度，取各自所长。**

## 13.4 TLS握手 —— HTTPS最核心的过程

这是整本书最复杂的机制之一，但用比喻拆开后其实非常清晰。

### 准备工作：TCP三次握手

在TLS握手之前，TCP三次握手已经完成了。TCP连接已经建立，接下来在这个连接上进行TLS握手。

```
客户端                             服务端
  │                                  │
  │═══ TCP三次握手已完成 ═════════│
  │                                  │
  │────── TLS握手开始 ──────────→  │
```

### TLS握手的6个步骤

#### 第1步：Client Hello

客户端先开口："你好！我支持这些加密方式，你看用哪个？"

客户端发出去的信息包括：
- 支持的TLS版本（如TLS 1.2、TLS 1.3）
- 支持的加密套件列表（如 `TLS_AES_128_GCM_SHA256`）
- 一个随机数（Random1，后面用来生成会话密钥）

```
客户端："我会说中文、英文、日语，你选一个？这是我的第一个随机数"abc""
```

#### 第2步：Server Hello

服务端回应："好，咱们用中文。这是我的随机数。"

服务端回应：
- 选定的加密套件（从客户端列表中选一个）
- 一个随机数（Random2）
- **服务端的数字证书**（包含服务端的公钥）

```
服务端："好，用中文聊。这是我的第二个随机数"def"，还有我的身份证（证书）"
```

#### 第3步：客户端验证证书

客户端拿到服务端的证书，开始验证：
- 证书是不是由受信任的CA签发的？
- 证书过期了吗？
- 证书上的域名是我正在访问的域名吗？

这一步相当于：你见到一个自称是快递员的人，你看了他的工牌，打电话给快递公司确认了他是真员工。证书验证机制后面第13.5节会详细展开。

#### 第4步：客户端生成Pre-Master Secret

验证通过后，客户端生成第三个随机数——**Pre-Master Secret**。然后**用服务端的公钥加密**它，发给服务端。

```
客户端用服务端的公钥（从证书里拿的）加密：
"这是我的第三个随机数'ghi'——用你的公钥锁好了，只有你能打开！"
```

> 🤔 想多一点：这一步是整个TLS握手最精妙的设计。Pre-Master Secret是用**非对称加密**（公钥加密、私钥解密）传输的。理论上，即使整个握手过程被中间人全部监听到，只要中间人没有服务端的私钥，他就无法解密出Pre-Master Secret。而Pre-Master Secret是生成后续"会话密钥"的关键原料——拿不到它，中间人就没法解密后面的通信。

#### 第5步：双方各自计算出"会话密钥"

现在客户端和服务端手里都有三个随机数：
- Random1（客户端生成的，明文传的）
- Random2（服务端生成的，明文传的）
- Pre-Master Secret（客户端生成的，用公钥加密传的）

双方用这三个随机数，通过一个约定的算法**PRF（Pseudo-Random Function，伪随机函数）**，各自计算出**会话密钥**（Session Key）。

```
SessionKey = PRF(Random1, Random2, PreMasterSecret)
```

注意：会话密钥是**双方各自算出来的，从来没有在网络上传输过**。

> 🤔 想多一点：为什么需要三个随机数而不是一个？因为单个随机数可能不够随机——如果只用客户端随机数，万一客户端随机数生成器有缺陷，整个连接的安全性就毁了。三个来源（客户端+服务端+加密传输的），任何一个足够随机，最终结果就足够随机。这是密码学中"混合熵源"的设计思想。

#### 第6步：双方确认握手完成

双方用刚算出来的会话密钥，各发一条加密的"Finished"消息。如果对方能解密并验证，握手成功！

```
客户端 → 服务端：Finished（用会话密钥加密的）
服务端 → 客户端：Finished（用会话密钥加密的）

双方："测试消息都能解密！握手成功，开始正式通信！"
```

此后，所有HTTP数据都用这个**会话密钥**做**对称加密**。

### 完整流程图

```
客户端                              服务端
  │                                   │
  │  ① ClientHello ──────────────→   │
  │   (TLS版本, 加密套件, Random1)      │
  │                                   │
  │  ←── ② ServerHello ──────────     │
  │   (选定加密套件, Random2, 证书)      │
  │                                   │
  │  ③ 验证证书 ✓                     │
  │                                   │
  │  ④ ClientKeyExchange ────────→   │
  │   (用公钥加密的Pre-Master Secret)    │
  │                                   │
  │  ⑤ 算出会话密钥                    │  ⑤ 算出会话密钥
  │                                   │
  │  ⑥ Finished(加密) ───────────→   │
  │  ←── Finished(加密) ─────────     │
  │                                   │
  │═══ 之后全部用会话密钥对称加密 ═══│
```

## 13.5 数字证书和CA

### 13.4节的流程有一个漏洞

回到第1步和第3步之间：

```
客户端以为自己在和 baidu.com 通信。
但中间人截获了ClientHello，伪装成baidu.com回了一个ServerHello，
里面夹着自己的假证书（也写着"baidu.com"）。
客户端拿到假证书里的公钥，用它加密了Pre-Master Secret发出去。
中间人用自己的私钥解密，拿到了Pre-Master Secret。
然后中间人再用真baidu.com的公钥加密，转发给真baidu.com。
```

这样，中间人就**完全透明地窃听到了全部通信**——这就是**中间人攻击**（Man-in-the-Middle Attack）。

客户端在第3步**验证证书**就是为了防止这个。但怎么验证？一张假身份证看起来可以和真的一模一样。

### 数字证书是什么

数字证书相当于一个网站**在互联网上的身份证**。里面包含：

```
证书内容：
├── 域名：www.baidu.com
├── 公钥：xxxxxx（百度的公钥）
├── 有效期：2025-01-01 ~ 2026-01-01
├── 颁发者：GlobalSign（CA机构）
└── 数字签名：由GlobalSign用自己的私钥对以上内容签名
```

### CA：互联网的"公安局"

CA = **Certificate Authority** = 证书颁发机构

CA是受信任的第三方机构，它在互联网上的角色相当于现实中的公安局——你说你是谁不算数，得公安局给你发身份证才算数。

知名CA机构：DigiCert、GlobalSign、Let's Encrypt（免费的）、Sectigo 等。

### 证书链：信任的传递

你的操作系统和浏览器**出厂时就内置了一组"根CA证书"**——这些都是全世界公认最可信的CA。

```
根CA (Root CA) ──── 自签名，出厂内置在操作系统中
  │                   "我永远信任你"
  ↓
中间CA (Intermediate CA) ──── 由根CA签名授权
  │                           "我信任你，所以你的签名我也认"
  ↓
网站证书 (End-entity) ──── 由中间CA签名
                            "你的证书是由我签发的，我担保你的身份是真的"
```

浏览器验证时，从下往上验证：
1. 验证 `www.baidu.com` 的证书签名 → 用中间CA的公钥验证 → 通过
2. 验证中间CA的证书签名 → 用根CA的公钥验证 → 通过
3. 根CA证书在浏览器内置列表里 → 最终通过

**只要根CA可信，整条链上的所有证书都可信。** 这叫**信任链**。

### 为什么你能信任CA？

CA要是乱发证书怎么办？历史上有过：

- 2011年，荷兰CA机构DigiNotar被黑客入侵，签发了大量假证书（包括google.com的）。然后这家CA**直接破产了**。
- 2015年，中国CNNIC签发了假的google.com证书，被Chrome和Firefox拉黑。

信任是CA的生存之本。失去信任的CA，浏览器会直接从内置列表中移除，它签的证书也跟着全部作废。

### 自签名证书

你自己也可以用OpenSSL给自己签一个"证书"。但它不会被浏览器信任——因为你的CA不在浏览器的内置列表中。浏览器会显示：

```
⚠️ 您的连接不是私密连接
攻击者可能会试图窃取您的信息
NET::ERR_CERT_AUTHORITY_INVALID
```

自签名证书在开发环境或内部系统中可以用（手动把自签发的根证书加到系统信任列表），但**公网绝对不能用**。

### 用Go搭建一个HTTPS服务器

```go
package main

import (
    "fmt"
    "net/http"
)

func main() {
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprint(w, "这是通过HTTPS传输的数据！")
    })

    err := http.ListenAndServeTLS(":443", "server.crt", "server.key", nil)
    if err != nil {
        fmt.Println("HTTPS启动失败:", err)
    }
}
```

`server.crt` 是证书文件（含公钥），`server.key` 是私钥文件。这两样东西就是你的"互联网身份证"。

> 🤔 想多一点：生产环境中通常不直接用Go的 `ListenAndServeTLS` 对外服务。常见架构是：Nginx在前面处理TLS终止（TLS Termination），然后把解密后的HTTP请求转发给后面的Go应用。这样Go应用不用处理加密解密，Nginx专门优化过TLS性能。另外，现在还有一个趋势叫**自动化证书管理**——Let's Encrypt通过ACME协议实现了免费、自动化的证书签发和续期。配合Certbot或Caddy，你的服务器可以自动搞定HTTPS，零人工维护。

## 本章小结

| 知识点 | 一句话概括 |
|--------|------------|
| HTTP三大问题 | 明文、无身份验证、可被篡改——HTTPS用TLS解决全部三个 |
| 对称加密 | 同一把密钥既加密又解密，快但密钥交换困难 |
| 非对称加密 | 公钥加密私钥解密，解决了密钥交换，但慢 |
| TLS握手 | 6步：ClientHello → ServerHello → 验证书 → Pre-Master → 算密钥 → Finished |
| 会话密钥 | 用三个随机数各自算出，从未在网络上传输 |
| 精妙设计 | 非对称加密只用来传Pre-Master（一小段），之后全用对称加密 |
| 数字证书 | 网站的身份证明，由CA签名担保 |
| 证书链 | 根CA → 中间CA → 网站证书，信任一层层传递 |
| 自签名证书 | 自己签自己，浏览器不认，公网不能用 |
