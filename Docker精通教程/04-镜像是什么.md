## 我在做什么？

前面三章，你已经跑过 `docker run hello-world`，也进过 Ubuntu 容器里敲过 `apt update`。但你有没有想过一个问题：**那个 `ubuntu` 镜像到底是什么？它为什么只有几十 MB，却能提供一个完整的 Linux 环境？**

这一章，我们把"镜像"这个东西拆开来看。你不光要会用 `docker images` 看镜像列表，更要理解镜像的**分层结构**——这个设计是 Docker 能做到"启动快、占用小、传输快"的根本原因。

学完这一章，你能：
- 用 `docker pull` 拉取任意镜像，用 `docker images` 查看本地镜像
- 理解镜像的**分层存储**机制，知道为什么镜像比虚拟机小那么多
- 理解 Tag 标签的作用，知道为什么 `latest` 是个"坑"
- 用 `docker rmi` 安全删除不再需要的镜像

## 一、镜像 = 应用的打包模板

### 先回想一个场景

你写了一个 Python Web 应用，部署到服务器上需要：

```bash
# 1. 装 Python 3.11
# 2. 装 pip 依赖
# 3. 装 Nginx 做反向代理
# 4. 配 Nginx 的配置文件
# 5. 设置环境变量
# 6. 启动应用
```

每一步都可能出错。版本不对、路径不对、权限不够……运维的痛苦就来源于此。

Docker 镜像 *此术语见附录C* 的思路是：**把上面所有步骤的结果，打包成一个"模板"**。这个模板里包含了：

- 一个精简的操作系统（比如 Ubuntu 的文件系统）
- 你需要的运行时（Python 3.11）
- 你的代码和依赖
- 配置文件、环境变量……

然后你把模板交给任何人，他们只需要 `docker run`，一切就绪。

> 镜像 = 做菜的**食谱**。食谱是静态的，不会变成菜，但它包含了做菜所需的一切信息。你可以把同一本食谱复印一百份，发给一百个人，每个人照着做出来的菜都是一样的。

### 镜像 vs 容器：不能搞混

| | 镜像（Image） | 容器（Container） |
|------|------------|----------------|
| 是什么 | 静态模板，只读 | 镜像的运行实例，可读写 |
| 类比 | 类（Class） | 实例（Instance） |
| 能不能改 | 不能直接改，只能重新构建 | 可以改（但改了不影响镜像） |
| 一个能变多个吗 | 能——一个镜像可以跑出无数个容器 | 不能——一个容器只来自一个镜像 |

## 二、镜像的分层结构：Docker 最巧妙的设计

### 你不用急着理解"层"，先用一个实验来感受

```bash
# 拉取 nginx 镜像
docker pull nginx
```

输出中你会看到类似这样的内容：

```
a1b2c3d4e5f6: Pull complete
b2c3d4e5f6a1: Pull complete
c3d4e5f6a1b2: Pull complete
d4e5f6a1b2c3: Pull complete
e5f6a1b2c3d4: Pull complete
f6a1b2c3d4e5: Pull complete
Digest: sha256:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
Status: Downloaded newer image for nginx:latest
```

注意——**每一行 `Pull complete` 就是一层**。nginx 镜像由 6 层组成。

### 层是什么？—— 一本"只读连环画"

想象你在画一本连环画。每一页是透明的塑料片，你在上面画一部分内容：

```
第 1 层（最底层）：画了一个 Ubuntu 的基础文件系统
第 2 层：装了 apt 软件包
第 3 层：装了 Nginx 软件
第 4 层：复制了 Nginx 配置文件
第 5 层：设置了环境变量
第 6 层（最顶层）：设置了启动命令
```

把这 6 层透明塑料片叠在一起，从上往下看——你看到的就是一个完整的 Nginx 运行环境。

```
┌─────────────────────────────────────────┐
│ 第 6 层：CMD ["nginx", "-g", "daemon.."]│  ← 启动命令
├─────────────────────────────────────────┤
│ 第 5 层：ENV NGINX_VERSION=1.25.3       │  ← 环境变量
├─────────────────────────────────────────┤
│ 第 4 层：COPY nginx.conf /etc/nginx/    │  ← 配置文件
├─────────────────────────────────────────┤
│ 第 3 层：RUN apt-get install nginx      │  ← 安装软件
├─────────────────────────────────────────┤
│ 第 2 层：RUN apt-get update             │  ← 更新软件源
├─────────────────────────────────────────┤
│ 第 1 层：FROM debian:bookworm-slim      │  ← 基础系统
└─────────────────────────────────────────┘
```

**每一层都是只读的。** 当你 `docker run` 启动容器时，Docker 会在这些只读层上面**再加一层可写层**（容器层），你在容器里做的所有修改都在这个可写层里。镜像本身的层不会被修改。

### 分层设计的三大好处

**好处 1：节省存储空间**

你有 10 个不同的应用镜像，它们都基于 `debian:bookworm-slim`。如果每个镜像都是完整独立的，你要存 10 份 Debian 文件系统。但有了分层，`debian:bookworm-slim` 这层只存一份，10 个镜像共享。

```bash
# 你可以验证这一点
docker pull debian:bookworm-slim   # 拉取基础镜像
docker pull nginx                   # 拉取 nginx（基于 debian）
docker pull python:3.11-slim        # 拉取 python（也基于 debian）

# 观察它们的层——你会发现 debian 相关的层被共享了
```

**好处 2：加速镜像分发**

你要更新 nginx 镜像，从 1.25.3 升级到 1.25.4。如果每一层都是独立的，你需要重新下载整个镜像。但有了分层，只有**变化的层**需要重新下载，没变的层直接在本地缓存里复用。

**好处 3：构建缓存**

当你用 Dockerfile 构建镜像时（后面章节会讲），Docker 会缓存每一层。如果某一层没变，Docker 就跳过重建，直接用缓存。这让构建速度大幅提升。

> **想多一点**：你可能会问："分层的粒度是怎么决定的？"答案是：**Dockerfile 里的每一条指令（FROM、RUN、COPY、ENV 等）都会生成一层。** 所以写 Dockerfile 时，把不常变的指令放前面（比如 `apt-get update`），常变的放后面（比如 `COPY . /app`），可以最大化缓存命中率。这个技巧会在后面的 Dockerfile 章节详细展开。

## 三、Docker Hub：全球最大的镜像仓库

### 它是什么？

**Docker Hub** *此术语见附录C*（hub.docker.com）是 Docker 官方维护的公共镜像仓库 *此术语见附录C*。你可以把它理解成"Docker 镜像的 GitHub"——上面有成千上万个官方和社区维护的镜像，任何人都可以免费拉取。

### 官方镜像 vs 社区镜像

```bash
# 官方镜像：由 Docker 官方或软件厂商维护，质量有保障
docker pull nginx          # Docker 官方维护
docker pull ubuntu         # Canonical 官方维护
docker pull mysql          # Oracle 官方维护
docker pull python         # Python 官方维护

# 社区镜像：由个人或组织维护，质量参差不齐
docker pull someuser/someapp:latest
```

> 选镜像的金科玉律：**优先选官方镜像**。官方镜像通常有 100M+ 的下载量，说明经过大量用户验证。社区镜像要仔细看 README 和下载量，有些可能已经两年没更新了。

### 拉取镜像：`docker pull`

```bash
# 完整格式
docker pull [仓库地址/]镜像名[:标签]

# 最简格式（默认从 Docker Hub 拉取，默认 latest 标签）
docker pull nginx

# 等价于
docker pull docker.io/library/nginx:latest
```

各部分含义：

```
docker pull nginx:1.25
│      │    │     │
│      │    │     └── 标签（Tag），指定版本
│      │    └── 镜像名
│      └── 子命令
└── Docker 客户端
```

### `docker pull` 背后的流程

```
你敲下 docker pull nginx:1.25
           │
           ▼
┌─────────────────────────────────────────────┐
│ 步骤 1：Docker 客户端联系 Docker Hub        │
│         问："有没有 nginx:1.25 这个镜像？"    │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ 步骤 2：Docker Hub 返回镜像的元数据         │
│         包括：层列表、每层的哈希值、大小      │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ 步骤 3：Docker 检查本地是否已有这些层        │
│         有 → 跳过，不重复下载                │
│         没有 → 下载这一层                    │
└──────────────────────┬──────────────────────┘
                       ▼
┌─────────────────────────────────────────────┐
│ 步骤 4：所有层下载完成后，组装成完整镜像     │
│         存入本地镜像仓库                    │
└─────────────────────────────────────────────┘
```

> 注意步骤 3：这就是分层结构的威力。如果你本地已经有 `debian:bookworm-slim`，再拉取 `nginx` 时，debian 相关的层就不需要重新下载了。

## 四、查看本地镜像：`docker images`

### 基本用法

```bash
docker images
```

输出示例：

```
REPOSITORY   TAG       IMAGE ID       CREATED        SIZE
nginx        latest    abc123def456   2 weeks ago    187MB
ubuntu       22.04     def456abc789   4 weeks ago    77.8MB
alpine       latest    ghi789jkl012   6 weeks ago    7.05MB
hello-world  latest    jkl012mno345   8 weeks ago    13.3kB
```

### 每列的含义

| 列名 | 含义 | 怎么看 |
|------|------|--------|
| **REPOSITORY** | 镜像名（仓库名） | 同一个 REPOSITORY 可以有多个 TAG |
| **TAG** | 标签，用于区分版本 | `latest` 是最新稳定版，`22.04` 是具体版本号 |
| **IMAGE ID** | 镜像的唯一标识符（12 位十六进制） | 每个镜像有唯一的 ID，不同 TAG 可以指向同一个 ID |
| **CREATED** | 镜像创建时间 | 看镜像的新旧程度 |
| **SIZE** | 镜像大小 | 注意：这是"逻辑大小"，实际占用可能更少（因为层共享） |

### 常用组合

```bash
# 只看某个镜像
docker images nginx

# 只显示镜像 ID（脚本中常用）
docker images -q

# 按镜像名过滤
docker images --filter "reference=nginx"

# 显示完整的镜像 ID（而非截断的 12 位）
docker images --no-trunc
```

### 一行命令对比：`docker images` vs `docker ps`

很多新手会搞混这两个命令。记住一句口诀：

```
docker images  → 看"模板"（镜像，静态，存在硬盘上）
docker ps      → 看"实例"（容器，动态，正在运行）
```

## 五、Tag 标签：版本管理的命门

### Tag 是什么？

**Tag** *此术语见附录C* 就是镜像的版本标签。它让你能精确指定"我要用哪个版本的镜像"。

```bash
# 同一个镜像名，不同标签
nginx:latest      # 最新稳定版（当前是 1.27.x）
nginx:1.27        # 1.27.x 的最新版本
nginx:1.27.4      # 精确到补丁版本
nginx:mainline    # 主线开发版（比 stable 新但不一定稳定）
nginx:alpine      # 基于 Alpine Linux 的精简版（只有 ~40MB）
```

### Tag 的命名规则

没有强制规则，但社区有一些约定俗成：

| 格式 | 示例 | 含义 |
|------|------|------|
| `latest` | `nginx:latest` | 最新稳定版（默认标签） |
| `主版本号` | `ubuntu:22` | 该大版本的最新版 |
| `主.次版本号` | `python:3.11` | 该次版本的最新版 |
| `主.次.补丁` | `nginx:1.27.4` | 精确版本 |
| `-slim` 后缀 | `python:3.11-slim` | 精简版（去掉了编译工具和文档） |
| `-alpine` 后缀 | `node:20-alpine` | 基于 Alpine Linux 的超精简版 |

### `latest` 标签的陷阱（重要！）

❌ 新手的错误认知：

> "`latest` 就是最新的稳定版本，我永远用 `latest` 就行了。"

✅ 正确认知：

> "`latest` 只是一个**默认标签**，它不一定指向最新版本，也不一定指向稳定版本。它只是镜像维护者给某个版本打的标签，**随时可能变化**。"

场景：你三个月前 `docker pull nginx:latest`，当时 `latest` 指向 `1.25.3`。三个月后你在另一台机器上 `docker pull nginx:latest`，此时 `latest` 已经指向 `1.27.1`。两台机器上的 nginx 版本不同，行为可能也不同。

**正确做法**：生产环境**永远不要用 `latest`**，指定精确版本号。

```bash
# ❌ 生产环境不推荐
docker run -d nginx:latest

# ✅ 生产环境推荐
docker run -d nginx:1.27.4
```

> **想多一点**：`latest` 这个标签的名字起得不好。它应该叫 `default`——因为它就是"你不指定标签时的默认值"，和"最新"没有必然关系。但历史原因已经没法改了，你只能记住这个坑。

## 六、删除镜像：`docker rmi`

### 基本用法

```bash
# 用镜像名:标签删除
docker rmi nginx:latest

# 用镜像 ID 删除（可以用前几位）
docker rmi abc123def456

# 强制删除（即使有容器在用这个镜像）
docker rmi -f abc123def456
```

### 删除镜像的常见情况

**情况 1：镜像正在被容器使用**

```bash
docker rmi nginx:latest
# 报错：Error: conflict: unable to remove repository reference "nginx:latest"
#       (must force) - container abc123 is using its referenced image def456
```

解决方法：先删除使用该镜像的容器，再删除镜像。

```bash
docker rm abc123            # 删除容器
docker rmi nginx:latest     # 再删除镜像
```

**情况 2：多个 Tag 指向同一个镜像**

```bash
docker tag nginx:latest nginx:my-tag     # 给镜像打两个标签
docker rmi nginx:latest                  # 删除一个标签，镜像还在
docker rmi nginx:my-tag                  # 删除最后一个标签，镜像才真正删除
```

**情况 3：批量清理无用镜像**

```bash
# 删除所有未使用的镜像（没有被任何容器引用的）
docker image prune

# 删除所有未使用的镜像（包括被已停止容器引用的）
docker image prune -a

# 强制删除，不询问确认
docker image prune -a -f
```

### 不清理旧镜像的后果

每次 `docker pull` 一个新的 `latest`，旧的 `latest` 标签会从旧镜像上**脱钩**，变成 `<none>:<none>`（也叫"悬空镜像"或 dangling image）。

```bash
docker images
# 你会看到这样的行：
# REPOSITORY   TAG       IMAGE ID       CREATED        SIZE
# <none>       <none>    xyz789abc012   3 months ago   186MB
```

这些 `<none>:<none>` 镜像**不再被任何标签引用，但还占用磁盘空间**。如果不定期清理，你的硬盘会被悄悄塞满。

```bash
# 清理所有悬空镜像
docker image prune
```

## 七、实战验证：从拉取到删除走一遍

现在，我们用 `alpine` 镜像（一个超轻量的 Linux 发行版，只有 ~7MB）完整走一遍流程：

```bash
# 1. 拉取 alpine 镜像
docker pull alpine

# 2. 查看本地镜像，确认 alpine 在列表中
docker images

# 3. 只查看 alpine 镜像
docker images alpine

# 4. 记录 alpine 的 IMAGE ID（前 12 位）

# 5. 给 alpine 打个自定义标签
docker tag alpine:latest alpine:my-test

# 6. 再次查看，发现 alpine 和 alpine:my-test 指向同一个 IMAGE ID
docker images alpine

# 7. 删除自定义标签（镜像本身还在）
docker rmi alpine:my-test

# 8. 确认 alpine:latest 还在
docker images alpine

# 9. 删除 alpine 镜像
docker rmi alpine:latest

# 10. 确认已删除
docker images | grep alpine
# 应该没有输出
```

## 我做得对不对？

### 验证方法

请按顺序执行以下操作，**全部成功**才算通过：

```bash
# 1. 拉取 alpine 镜像
docker pull alpine

# 2. 查看本地镜像，确认 alpine 在列表中
docker images | grep alpine

# 3. 查看 alpine 镜像的详细信息（IMAGE ID、SIZE）
docker images alpine

# 4. 用 alpine 镜像启动一个容器，执行 echo 命令后退出
docker run --rm alpine echo "Hello from Alpine!"
# 预期输出：Hello from Alpine!

# 5. 清理：删除 alpine 镜像
docker rmi alpine:latest

# 6. 确认已删除
docker images | grep alpine
# 应该没有输出
```

**预期结果**：
- 步骤 1：看到 `Pull complete` 等输出，最后显示 `Downloaded newer image`
- 步骤 2-3：`docker images` 中能看到 `alpine`，SIZE 约 7MB
- 步骤 4：输出 `Hello from Alpine!`
- 步骤 5-6：`docker images | grep alpine` 没有输出

## 不对怎么办？

### 常见错误 1：`docker pull` 拉取超时或极慢

```
Error response from daemon: Get https://registry-1.docker.io/v2/: net/http: request canceled
```

这和第 02 章的问题一样，需要配置镜像加速器。请回顾 02 章"排错预案 2"。

### 常见错误 2：`latest` 标签导致版本不一致

❌ 错误做法：不同机器上使用 `nginx:latest`，导致版本不同，出现诡异 bug。

✅ 正确做法：

```bash
# 生产环境指定精确版本
docker pull nginx:1.27.4
docker run -d --name my-nginx nginx:1.27.4
```

> 为什么这个错误很隐蔽？因为 `latest` 在拉取时**不会告诉你它具体是哪个版本**。你只能通过 `docker inspect nginx:latest` 查看详细信息才能知道。所以建议养成习惯：**拉取镜像时永远指定版本号**。

### 常见错误 3：`docker rmi` 删不掉镜像

```bash
docker rmi nginx:latest
# Error: conflict: unable to remove repository reference "nginx:latest"
#       (must force) - container abc123 is using its referenced image
```

**原因**：有容器（包括已停止的）引用了这个镜像。

**解决**：

```bash
# 先查看哪些容器在使用这个镜像
docker ps -a --filter "ancestor=nginx"

# 删除这些容器
docker rm abc123

# 再删除镜像
docker rmi nginx:latest
```

### 常见错误 4：不清理旧镜像，磁盘被占满

**现象**：`docker images` 看到很多 `<none>:<none>` 的镜像，磁盘空间越来越少。

**原因**：每次 `docker pull` 新版本的 `latest` 标签，旧镜像就变成了悬空镜像，但不会自动删除。

**解决**：

```bash
# 查看磁盘使用情况
docker system df

# 清理所有未使用的镜像、容器、网络、构建缓存
docker system prune -a

# 如果只想清理悬空镜像
docker image prune
```

> 建议每隔几周执行一次 `docker system prune`，保持 Docker 环境的整洁。

### 常见错误 5：把镜像大小当真

**现象**：`docker images` 显示 nginx 是 187MB，python 是 1GB。加起来 1.2GB，但你发现硬盘只用了 700MB。

**原因**：`docker images` 显示的是**逻辑大小**（每层单独计算），实际物理占用更少，因为层被共享了。

**验证**：

```bash
# 查看实际磁盘占用
docker system df

# 输出示例：
# TYPE            TOTAL     ACTIVE    SIZE      RECLAIMABLE
# Images          5         2         1.2GB     800MB (66%)
# Containers      3         1         50MB      30MB (60%)
# Local Volumes   2         1         100MB     50MB (50%)
# Build Cache     0         0         0B        0B
```

`SIZE` 列是逻辑大小，`RECLAIMABLE` 列是清理后能释放的空间。

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 镜像的定义 | 镜像 = 应用 + 依赖 + 运行环境的只读模板 | 镜像本身不运行，是静态的；容器是镜像的运行时实例 |
| 镜像分层结构 | 每层是一个只读文件系统，叠加成完整镜像 | 分层是 Docker 轻量、快速、省空间的根本原因 |
| Docker Hub | 公共镜像仓库，hub.docker.com | 优先选官方镜像，检查最新更新时间 |
| 拉取镜像 | `docker pull nginx:1.27.4` | 自动检查本地缓存，已存在的层不重复下载 |
| 查看本地镜像 | `docker images` / `docker images nginx` | SIZE 列是逻辑大小，实际占用更少（层共享） |
| Tag 标签 | `nginx:1.27.4`、`nginx:alpine` | 生产环境永远不要用 `latest`，指定精确版本号 |
| `latest` 标签的陷阱 | `latest` 只是默认标签，不一定是最新稳定版 | 每次 pull 可能指向不同版本，导致环境不一致 |
| 删除镜像 | `docker rmi` 镜像名:标签 | 有容器引用时删不掉，先删容器再删镜像 |
| 清理无用镜像 | `docker image prune` / `docker system prune -a` | 定期清理，防止磁盘被 `<none>:<none>` 镜像占满 |
| 镜像的实际大小 | `docker system df` | 查看 Docker 实际磁盘占用，而非逻辑大小 |

## 本篇最可能出错的地方及原因

### 1. 生产环境用 `latest` 标签（最隐蔽的坑）

**原因**：`latest` 听起来很安全，新手会以为它是最新稳定版。实际上它只是一个可以随时被重新指向的标签。今天 `latest` 指向 `1.27.4`，下个月可能指向 `1.28.0`，而 `1.28.0` 可能不兼容你的配置。

**排查**：`docker inspect nginx:latest | grep -A 5 "Labels"` 查看当前 latest 实际指向的版本。

**解决**：养成习惯，`docker pull` 和 `docker run` 时永远指定精确版本号。

### 2. 不清理旧镜像，磁盘悄悄被占满

**原因**：`docker pull` 新版本后，旧版本的层不会自动删除。尤其 CI/CD 流水线中频繁拉取镜像，几周就能积攒几十 GB。

**排查**：`docker system df` 查看实际磁盘占用。

**解决**：定期执行 `docker image prune -a` 或在 CI/CD 脚本末尾加清理命令。

### 3. 混淆 `docker images` 的 SIZE 和实际磁盘占用

**原因**：`docker images` 显示的 SIZE 是每层独立计算的总和，不考虑层共享。如果你有 10 个基于 `debian` 的镜像，`docker images` 会显示 10 个 Debian 的大小，但实际只存了一份。

**排查**：`docker system df` 查看真实磁盘占用。

**解决**：关注磁盘空间时用 `docker system df`，不要信 `docker images` 的 SIZE 列。
