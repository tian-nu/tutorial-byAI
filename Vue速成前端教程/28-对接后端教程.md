## 一、目标与完成效果

**一句话目标**：把你的 Vue 前端和三种主流后端框架之一打通，实现完整的注册→登录→发文章→前端展示流程。

**完成后你将看到**：
- 前端注册页填写信息 → 数据存入后端数据库 → 返回成功。
- 前端登录 → 后端返回 token → 前端存入 Pinia store → 导航栏显示已登录。
- 前端发文章 → 请求带 token → 后端验证身份 → 文章存入数据库。
- 前端首页 → 从后端拉取文章列表 → 展示。

## 三、分步操作

### 步骤 1：三种后端方案概览

**我在做什么？**
根据你的后端技术栈，选择合适的对接方案。三种方案本质上做的事情一样：接收 HTTP 请求 → 处理逻辑 → 返回 JSON 数据。区别只是语法和项目结构。

| 后端 | 典型端口 | API 文档地址 | 适合人群 |
|------|:---:|------|----------|
| **Python FastAPI** | 8000 | `http://localhost:8000/docs` | Python 开发者、数据/ML 工程师 |
| **Java Spring Boot** | 8080 | 需集成 Swagger | Java 企业级开发者 |
| **Go Gin** | 8080 | 需手动写文档 | 追求高性能的后端开发者 |

**统一的通信格式**（三种后端都用同样的 JSON 格式）：

```json
// 请求（注册）
POST /api/register
Content-Type: application/json
{
  "username": "zhangsan",
  "email": "zhangsan@example.com",
  "password": "MyP@ssw0rd"
}

// 响应（成功）
{
  "code": 200,
  "message": "注册成功"
}

// 响应（登录成功）
{
  "code": 200,
  "data": {
    "token": "eyJhbGciOi...",
    "user": {
      "id": 1,
      "username": "zhangsan",
      "email": "zhangsan@example.com"
    }
  }
}
```

### 步骤 2：配置 VITE_API_BASE_URL

**我在做什么？**
前端需要知道后端在哪。开发时后端在 `localhost`，生产环境后端在公网服务器上。

**`.env.development`**（开发环境）：
```
VITE_API_BASE_URL=http://localhost:8000/api
```
> 端口改为你后端的实际端口：FastAPI 用 8000，Spring Boot 用 8080，Go Gin 用 8080。

**`.env.production`**（生产环境）：
```
VITE_API_BASE_URL=https://api.your-blog.com/api
```
> 你的后端部署后的公网地址。

**前端统一请求封装**（创建 `src/utils/api.js`）：

```js
const API_BASE = import.meta.env.VITE_API_BASE_URL

export async function apiRequest(endpoint, options = {}) {
  const url = `${API_BASE}${endpoint}`

  const defaultHeaders = {
    'Content-Type': 'application/json'
  }

  // 如果有 token，自动带上
  // 注意：这里需要特殊处理，因为不能在模块顶层调用 useUserStore()
  // 解决方案：让调用方传入 token 参数
  if (options.token) {
    defaultHeaders['Authorization'] = `Bearer ${options.token}`
  }

  const res = await fetch(url, {
    ...options,
    headers: {
      ...defaultHeaders,
      ...options.headers
    }
  })

  if (!res.ok) {
    const errorData = await res.json().catch(() => ({}))
    throw new Error(errorData.message || `请求失败 (${res.status})`)
  }

  return res.json()
}
```

**在组件中使用**（以 Login.vue 为例）：

```js
import { apiRequest } from '@/utils/api'
import { useUserStore } from '@/stores/user'

const userStore = useUserStore()

async function handleSubmit() {
  // ... 校验逻辑 ...
  try {
    const data = await apiRequest('/login', {
      method: 'POST',
      body: JSON.stringify(form)
    })
    // data = { code: 200, data: { token: "...", user: {...} } }
    userStore.login(data.data.token, data.data.user)
    // 跳转到首页
    router.push('/')
  } catch (e) {
    toast.message = e.message
    toast.type = 'error'
    toast.show = true
  }
}
```

### 步骤 3：CORS 配置 — 后端允许前端跨域访问

**我在做什么？**
浏览器有一个安全策略叫**同源策略（Same-Origin Policy）**：如果前端在 `http://localhost:5173`（Vite 开发服务器），后端在 `http://localhost:8000`，浏览器会阻止前端发请求到后端，因为"端口不同"。

**比喻**：同源策略就像小区门禁——不是本小区的车不让进。CORS 就是"给特定车辆发放通行证"。

**解决方案**：在后端加上 CORS 中间件，允许前端域名。

**Python FastAPI**（`main.py`）：
```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

# 🎯 CORS 配置
app.add_middleware(
    CORSMiddleware,
    allow_origins=[
        "http://localhost:5173",         # Vite 开发服务器
        "https://你的项目名.vercel.app",  # 生产环境前端
    ],
    allow_credentials=True,
    allow_methods=["*"],   # 允许所有 HTTP 方法
    allow_headers=["*"],   # 允许所有请求头
)
```

**Java Spring Boot**（添加配置类）：
```java
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
            .allowedOrigins(
                "http://localhost:5173",
                "https://你的项目名.vercel.app"
            )
            .allowedMethods("*")
            .allowedHeaders("*")
            .allowCredentials(true);
    }
}
```

**Go Gin**（`main.go`）：
```go
import (
    "github.com/gin-contrib/cors"
    "github.com/gin-gonic/gin"
)

func main() {
    r := gin.Default()

    // 🎯 CORS 配置
    r.Use(cors.New(cors.Config{
        AllowOrigins: []string{
            "http://localhost:5173",
            "https://你的项目名.vercel.app",
        },
        AllowMethods:  []string{"GET", "POST", "PUT", "DELETE", "OPTIONS"},
        AllowHeaders:  []string{"Origin", "Content-Type", "Authorization"},
        AllowCredentials: true,
    }))

    // ... 路由注册 ...
    r.Run(":8080")
}
```

❌ **错误示例**：CORS 配置 `allow_origins: ["*"]`（允许所有来源）。开发时方便，但生产环境不安全——任何网站都能调用你的 API。
✅ **正确示例**：明确列出前端域名，只有这些域名可以跨域访问。

**我做得对不对？**
在浏览器中打开前端页面，尝试登录。如果控制台没有出现 CORS 相关错误（关键字 `blocked by CORS policy`），说明配置成功。

**不对怎么办？**
- 报错 `blocked by CORS policy: No 'Access-Control-Allow-Origin' header` → 检查后端 CORS 配置是否生效，域名是否写对（包括 `http://` 前缀和端口号）。
- 报错 `blocked by CORS policy: Response to preflight` → 检查后端是否允许 `OPTIONS` 方法（preflight 请求）。

### 步骤 4：完整流程演示 — 注册→登录→发文章→展示

**我在做什么？**
从头到尾走一遍完整的用户流程，验证前后端是否真正打通。

**流程 1：注册**

```
用户填写注册表单 → 前端校验通过 → POST /api/register
→ 后端验证用户名/邮箱唯一性 → 密码哈希存储 → 返回成功
→ 前端收到响应 → Toast 提示"注册成功" → 跳转到登录页
```

**前端代码**（Register.vue 的 `handleSubmit`）：
```js
const data = await apiRequest('/register', {
  method: 'POST',
  body: JSON.stringify({
    username: form.username,
    email: form.email,
    password: form.password
  })
})
// 成功后自动跳转登录页
router.push('/login')
```

**流程 2：登录**

```
用户填写邮箱密码 → POST /api/login
→ 后端验证密码 → 生成 JWT token → 返回 token + userInfo
→ 前端 userStore.login(token, userInfo) → 导航栏自动更新
```

**流程 3：发文章**

```
用户填写标题和内容 → POST /api/posts（Header 带 Authorization: Bearer xxx）
→ 后端验证 token → 解析用户 ID → 文章存入数据库 → 返回文章详情
→ 前端跳转到文章详情页
```

**前端代码**（CreatePost.vue 的 `handleSubmit`）：
```js
const data = await apiRequest('/posts', {
  method: 'POST',
  body: JSON.stringify({
    title: form.title,
    content: form.content
  }),
  token: userStore.token  // 🎯 从 Pinia store 取 token
})
router.push(`/post/${data.data.id}`)
```

**流程 4：文章展示**

```
用户访问首页 → GET /api/posts
→ 后端查询文章列表 → 返回 JSON 数组
→ 前端渲染文章卡片
```

### 步骤 5：总结 — 前后端打通

**回顾整个数据流**：

```
浏览器（Vue 前端）
  │
  │  HTTP 请求（JSON）
  │  Authorization: Bearer <token>
  ▼
后端服务器（FastAPI / Spring Boot / Go Gin）
  │
  │  验证 token → 查询数据库 → 处理业务逻辑
  ▼
数据库（MySQL / PostgreSQL / SQLite）
  │
  │  返回数据
  ▼
后端 → JSON 响应 → 前端渲染
```

**关键点回顾**：

| 要点 | 前端做什么 | 后端做什么 |
|------|-----------|-----------|
| **API 地址** | `.env` 中配置 `VITE_API_BASE_URL` | 监听对应端口 |
| **CORS** | 不做什么（浏览器的限制） | 在 CORS 中间件中允许前端域名 |
| **Token 传递** | `Authorization: Bearer <token>` 请求头 | 从请求头中提取并验证 token |
| **错误处理** | `try/catch` 捕获，显示 toast | 返回合适的 HTTP 状态码和错误信息 |

## 五、已知坑点与禁止事项

- **禁止将 `allow_origins` 设置为 `["*"]`**：生产环境必须明确列出允许的域名。
- **禁止在前端硬编码 API 地址**：始终用 `import.meta.env.VITE_API_BASE_URL`。
- **CORS 问题排查顺序**：先确认后端 CORS 配置 → 再检查域名拼写（含协议和端口）→ 最后检查浏览器 DevTools 的 Network 面板。
- **Token 安全**：JWT token 存在 `localStorage` 中有 XSS 风险。生产环境建议用 httpOnly cookie（需要后端配合）。
