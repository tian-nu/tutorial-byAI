## 三、分步操作

### 步骤1：画出博客系统的页面结构图

#### 我在做什么？

在开始敲代码之前，先把博客系统需要哪些「页面」列出来。就像盖房子之前画户型图——你得先知道有几个房间、每个房间干什么用。

我们的博客系统需要6个页面：

```
┌──────────────────────────────────────────────────┐
│                    博客系统                        │
│                                                   │
│  / (首页)      →  文章列表（所有文章）              │
│  /post/:id     →  文章详情页                       │
│  /login        →  登录页                           │
│  /register     →  注册页                           │
│  /create       →  发布文章页                        │
│  /edit/:id     →  编辑文章页                        │
└──────────────────────────────────────────────────┘
```

页面之间的跳转关系用箭头表示：

```
首页 ──点击文章──→ 文章详情页
                   ↑
登录页 ←── 点击「写文章」── 首页
  │                          │
  └── 登录成功 ──→ 首页      │
                   │         编辑页 ←── 点击「编辑」── 文章详情页
                   │          │                        │
注册页 ←── 点击「注册」       └── 保存后 ──→ 首页      └── 删除后 ──→ 首页
  │
  └── 注册成功 ──→ 登录页
```

#### 我做得对不对？

- [ ] 数一下，正好6个页面？
- [ ] 每个页面都有一个URL路径（`/xxx` 或 `/xxx/:id`）？
- [ ] `/post/:id` 和 `/edit/:id` 中的 `:id` 表示动态参数，你理解了吗？（不同文章用不同id）

#### 不对怎么办？

- 如果你画出来超过6个页面，停下来想：是不是把「弹窗」「侧边栏」当成了页面？页面 = 浏览器地址栏里的URL会变的那个。
- 如果少于6个，检查是否漏掉了注册页或编辑页。

🤔 **想多一点**：为什么教程选择这6个页面而不是更多？因为这6个页面覆盖了前端80%的常见模式：列表页（首页）、详情页（/post/:id）、表单页（注册/登录/发布/编辑）。学完这6种，你就能套用到任何CRUD项目。

### 步骤2：画出组件树

#### 我在做什么？

「组件树」就是把每个页面拆成一个个组件，再把这些组件按父子关系画成一棵树。树根是 `App.vue`，从根往下长的每一层都是子组件。

想象一棵真实的树：树干是 App.vue，主要的枝干是各个页面，页面上的小组件是更细的树枝和叶子。

```
App.vue (树根)
│
├── BlogHeader (公共组件，每个页面都有)
│   ├── Logo
│   ├── 导航链接：首页 / 写文章 / 登录
│   └── 用户头像
│
├── RouterView (路由出口，根据URL显示不同页面)
│   │
│   ├── Home.vue (首页 /)
│   │   ├── BlogCard.vue (文章卡片) × N
│   │   └── BlogCard.vue ...
│   │
│   ├── PostDetail.vue (文章详情 /post/:id)
│   │   └── CommentList.vue (评论列表)
│   │       └── CommentItem.vue (单条评论) × N
│   │
│   ├── Login.vue (登录 /login)
│   │
│   ├── Register.vue (注册 /register)
│   │
│   ├── CreatePost.vue (发布文章 /create)
│   │
│   └── EditPost.vue (编辑文章 /edit/:id)
│
└── Footer (可选的页脚公共组件)
```

#### 关键概念区分

| 概念 | 比喻 | 在代码里 |
|------|------|----------|
| **页面组件** | 一整页纸 | 放在 `src/views/` 下，通常对应一个URL路径 |
| **公共组件** | 便利贴，贴在哪页都能用 | 放在 `src/components/` 下，可被多个页面复用 |
| **组件树** | 公司的组织架构图 | App.vue 是CEO，各个子组件是各级员工 |

> 📌 **术语标记**：「组件树」「页面组件」「公共组件」将收录于术语附录。

#### 我做得对不对？

- [ ] 组件树的根是 `App.vue`？
- [ ] `BlogHeader`（导航栏）在 `RouterView` 的外面（因为每页都要显示）？
- [ ] 每个 `views/` 下的页面都挂在 `RouterView` 下面（因为根据URL切换显示）？

#### 不对怎么办？

常见错误：把 `BlogHeader` 放在了 `RouterView` 里面。这会导致导航栏只在某个页面出现。

❌ **错误**：RouterView 里面放 BlogHeader
```
RouterView
  ├── BlogHeader
  ├── Home.vue
  └── ...
```
导航栏变成了「首页的一部分」，切换到登录页就没了。

✅ **正确**：BlogHeader 和 RouterView 是兄弟关系
```
App.vue
  ├── BlogHeader        ← 始终显示
  ├── RouterView        ← 根据URL切换
  └── Footer            ← 始终显示
```

🤔 **想多一点**：「兄弟关系」在Vue里怎么写？在 `App.vue` 的 `<template>` 里，`<BlogHeader />` 和 `<RouterView />` 是并列的两个标签，谁也不包含谁。

### 步骤3：设计文件组织

#### 我在做什么？

把组件树翻译成磁盘上的文件夹和文件。就像你把衣柜分成「上衣区」「裤子区」「配饰区」——每个文件夹有明确的职责。

```
src/
├── components/          ← 公共组件（可复用）
│   ├── BlogHeader.vue   ← 导航栏
│   ├── BlogCard.vue     ← 文章卡片
│   ├── CommentList.vue  ← 评论列表
│   └── CommentItem.vue  ← 单条评论
│
├── views/               ← 页面组件（对应URL）
│   ├── Home.vue         ← 首页 /
│   ├── PostDetail.vue   ← 文章详情 /post/:id
│   ├── Login.vue        ← 登录 /login
│   ├── Register.vue     ← 注册 /register
│   ├── CreatePost.vue   ← 发布文章 /create
│   └── EditPost.vue     ← 编辑文章 /edit/:id
│
├── router/              ← 路由配置
│   └── index.js         ← Vue Router 路由表
│
├── stores/              ← 状态管理（Pinia）
│   └── auth.js          ← 用户登录状态
│
├── api/                 ← 后端接口封装
│   └── index.js         ← fetch 请求封装
│
├── mock/                ← 模拟数据（开发阶段用）
│   └── posts.js         ← 假文章数据
│
├── App.vue              ← 根组件
└── main.js              ← 入口文件
```

**职责边界表**：

| 文件夹 | 放什么 | 不放什么 |
|--------|--------|----------|
| `components/` | 可被多个页面复用的组件 | 只被一个页面用的组件（放 `views/` 里） |
| `views/` | 对应URL路径的页面组件 | 公共组件、工具函数 |
| `router/` | 路由配置 | 页面组件本身 |
| `stores/` | 全局状态（用户信息等） | 组件内部状态 |
| `api/` | fetch 封装 | 业务逻辑 |
| `mock/` | 假数据 | 生产数据 |

#### 我做得对不对？

在项目根目录执行：
```bash
ls src/components/ src/views/ src/router/ src/stores/ src/api/ src/mock/
```

每个文件夹都存在即正确。

#### 不对怎么办？

- 如果某个文件夹不存在，用 `mkdir` 创建。例如：`mkdir -p src/api src/mock src/stores`
- 不要把所有文件都扔在 `src/` 根目录——就像不要把袜子、衬衫、裤子全塞一个抽屉。

🤔 **想多一点**：为什么 `BlogHeader` 放 `components/` 而不是 `views/`？因为它出现在「每一个页面」的顶部，而不是只在一个页面里。判断标准是：**如果一个组件被 ≥2 个页面使用，它就是公共组件，该放 `components/`**。

## 四、术语附录

| 术语 | 解释 | 是否项目特有 |
|------|------|:---:|
| **组件树** | Vue应用中所有组件的父子关系图，根是 App.vue，叶子是最内层的小组件。想象成一棵倒过来的树。 | 否 |
| **页面组件** | 对应一个完整URL路径的组件，放在 `views/` 下。相当于一本书的一整页。 | 否 |
| **公共组件** | 可在多个页面中复用的组件，放在 `components/` 下。相当于便利贴，贴哪儿都能用。 | 否 |
