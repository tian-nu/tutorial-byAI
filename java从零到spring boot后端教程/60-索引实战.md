## 60.1 EXPLAIN：SQL的诊断仪

在任意SELECT语句前加 `EXPLAIN`，MySQL会告诉你它打算怎么执行这条查询，而不是真正去跑。

```sql
EXPLAIN SELECT * FROM users WHERE username = 'zhangsan';
```

输出（简化版）：

```
+----+-------------+-------+------+---------------+---------+-------+------+-------+
| id | select_type | table | type | possible_keys | key     | ref   | rows | Extra |
+----+-------------+-------+------+---------------+---------+-------+------+-------+
|  1 | SIMPLE      | users | ref  | idx_username  | idx_... | const |    1 | NULL  |
+----+-------------+-------+------+---------------+---------+-------+------+-------+
```

### 60.1.1 核心列详解

| 列 | 含义 | 重点关注 |
|----|------|---------|
| **type** | 访问类型，从好到差 | **最关键的列** |
| **key** | 实际使用的索引 | NULL表示没用到索引 |
| **rows** | MySQL预估需要扫描的行数 | 越小越好 |
| **Extra** | 额外信息 | Using filesort, Using temporary 是红色警报 |
| possible_keys | 可能用到的索引 | 参考对比key来看 |
| key_len | 使用索引的字节数 | 判断联合索引用了几个列 |

### 60.1.2 type从好到差

```
NULL > system > const > eq_ref > ref > range > index > ALL
 └──────── 好 ────────┘    └───── 一般 ─────┘  └── 差 ──┘
```

| type | 含义 | 示例 |
|------|------|------|
| **const** | 主键或唯一索引的等值查询，最多一条结果 | `WHERE id = 1` |
| **eq_ref** | JOIN时，前表的每行在后表中最多匹配一行 | `JOIN ON a.id = b.a_id`（b.a_id 有唯一索引） |
| **ref** | 非唯一索引的等值查询 | `WHERE username = 'zhangsan'` |
| **range** | 索引范围扫描 | `WHERE age BETWEEN 20 AND 30` |
| **index** | 全索引扫描（遍历整个索引树） | 比ALL好但仍是全扫描 |
| **ALL** | 全表扫描 | 没有可用索引时的噩梦 |

> 🚨 **type=ALL** 且 rows 很大的查询，就是你需要建索引的目标。

## 60.2 Extra深度解读

| Extra值 | 含义 | 严重度 |
|---------|------|--------|
| **Using index** | 覆盖索引，不需要回表 | ✅ 最优 |
| **Using where** | 在存储引擎返回行后，在Server层做了WHERE过滤 | ⚠️ 一般 |
| **Using index condition** | 使用了索引下推（ICP，5.6+特性） | ✅ 好 |
| **Using filesort** | 需要额外排序（内存或磁盘） | 🚨 危险，说明排序没用到索引 |
| **Using temporary** | 需要临时表（常见于GROUP BY和DISTINCT） | 🚨 危险，消耗内存/磁盘 |
| **Using join buffer** | JOIN时使用了连接缓冲区 | ⚠️ 可能意味着JOIN列没索引 |

> 🚨 `Using filesort` 和 `Using temporary` 是大查询的头号杀手。解决方案：确保 ORDER BY 和 GROUP BY 的列有合适的索引。

## 60.3 索引失效的经典场景

建了索引不等于就会用。以下是索引**失效**的经典场景（建了索引但EXPLAIN显示没用，或只用了部分）：

### 场景1：对索引列使用函数

```sql
-- ❌ 索引失效
SELECT * FROM users WHERE YEAR(created_at) = 2026;

-- ✅ 改为范围查询
SELECT * FROM users WHERE created_at >= '2026-01-01' AND created_at < '2027-01-01';
```

### 场景2：对索引列做运算

```sql
-- ❌ 索引失效
SELECT * FROM products WHERE price + 10 > 100;

-- ✅ 把运算移到右边
SELECT * FROM products WHERE price > 90;
```

### 场景3：隐式类型转换

```sql
-- phone是VARCHAR类型
-- ❌ 索引失效（MySQL把字符串转为数字进行比较）
SELECT * FROM users WHERE phone = 13800138000;

-- ✅ 加引号
SELECT * FROM users WHERE phone = '13800138000';
```

> 🐞 这是生产环境最常见的索引失效原因。URL参数传进来的数字没有引号包裹，开发者完全没意识到。

### 场景4：LIKE以%开头

```sql
-- ❌ 索引失效（不知道从哪开始匹配）
SELECT * FROM users WHERE username LIKE '%zhang';

-- ✅ 前缀匹配可以使用索引
SELECT * FROM users WHERE username LIKE 'zhang%';

-- ❌ 前后都有%，索引也失效
SELECT * FROM users WHERE username LIKE '%zhang%';
```

### 场景5：OR连接非索引列

```sql
-- 假设name有索引，age没有索引
-- ❌ 整体索引失效（因为age要走全表扫描，不如一开始就全表扫描）
SELECT * FROM users WHERE name = '张三' OR age = 25;

-- ✅ 改用UNION
SELECT * FROM users WHERE name = '张三'
UNION
SELECT * FROM users WHERE age = 25;
```

### 场景6：NOT IN 和 !=

```sql
-- ❌ 大部分情况下不使用索引
SELECT * FROM users WHERE status != 'deleted';
SELECT * FROM users WHERE id NOT IN (1, 2, 3);

-- ✅ 如果可以，用IN代替NOT IN
SELECT * FROM users WHERE status IN ('active', 'pending');
```

### 场景7：联合索引跳过最左列

```sql
-- 索引 (a, b, c)
-- ❌ 跳过a
SELECT * FROM t WHERE b = 1 AND c = 2;

-- ✅ 包含a
SELECT * FROM t WHERE a = 1 AND b = 1 AND c = 2;
```

## 60.4 覆盖索引：不回表的魔法

当SELECT的所有列都在索引中时，MySQL直接从索引取数据，不需要回表——这叫覆盖索引（Covering Index）。

```sql
-- 假设 idx_username_email 是 (username, email) 联合索引

-- ✅ 覆盖索引：查询的列都在索引里
EXPLAIN SELECT username, email FROM users WHERE username = 'zhangsan';
-- Extra: Using index  ← 不回表！

-- ❌ 非覆盖索引：age不在索引里，需要回表
EXPLAIN SELECT username, email, age FROM users WHERE username = 'zhangsan';
-- Extra: NULL  ← 走了回表
```

> 💡 覆盖索引的EXPLAIN中Extra显示 `Using index`，这是高性能的标志。

### 实战：用覆盖索引优化分页

```sql
-- 慢：OFFSET很大时，需要扫描并丢弃大量行，还要回表
SELECT * FROM users ORDER BY id LIMIT 100000, 20;

-- 快：先用覆盖索引定位ID，再取数据
SELECT * FROM users
WHERE id >= (
    SELECT id FROM users ORDER BY id LIMIT 100000, 1
)
ORDER BY id LIMIT 20;
```

## 60.5 创建索引的实战决策

### 60.5.1 哪些列该建索引

| 必建索引 | 谨慎建索引 |
|----------|-----------|
| WHERE条件的列 | 性别（只有男女，区分度极低） |
| JOIN的ON条件列 | 状态码只有2-3个值的列 |
| ORDER BY的列 | 频繁更新的列 |
| GROUP BY的列 | 很长的字符串列（TEXT、VARCHAR(500)） |
| 外键列 | 表数据很少（< 1000行） |

### 60.5.2 选择性（Cardinality）

```sql
SHOW INDEX FROM users;
```

查看 `Cardinality` 列——它表示该索引列有多少个不同的值。Cardinality 越接近总行数，索引区分度越高。

```sql
-- 区分度高，适合建索引
SELECT COUNT(DISTINCT username) / COUNT(*) FROM users;  -- 接近 1.0

-- 区分度低，不适合建索引
SELECT COUNT(DISTINCT gender) / COUNT(*) FROM users;    -- 可能只有 0.01
```

### 60.5.3 索引命名规范

```sql
-- 普通索引
CREATE INDEX idx_列名 ON 表名(列名);
CREATE INDEX idx_users_username ON users(username);

-- 唯一索引
CREATE UNIQUE INDEX uk_列名 ON 表名(列名);
CREATE UNIQUE INDEX uk_users_email ON users(email);

-- 联合索引
CREATE INDEX idx_表名_列1_列2 ON 表名(列1, 列2);
CREATE INDEX idx_orders_user_status ON orders(user_id, status);
```

## 60.6 索引实战调试流程

线上一个慢SQL，按以下步骤排查：

```
1. 拿到慢SQL → EXPLAIN 看一下
                │
2. type=ALL？  ──── 是 ──→  建索引（从WHERE和JOIN条件列下手）
    │
   否
    │
3. Extra有 Using filesort / Using temporary？
    │                          │
   是                          否
    │                          │
4. 给ORDER BY/GROUP BY         │
   列建索引                    │
                               │
5. rows还是很大？── 是 ──→  尝试覆盖索引
    │
   否
    │
6. 完成。实测验证。
```

```sql
-- 第1步：EXPLAIN
EXPLAIN SELECT * FROM orders WHERE user_id = 1 ORDER BY created_at DESC LIMIT 10;

-- 假设 type=ALL, rows=5000000

-- 第3步：建索引
CREATE INDEX idx_orders_user_created ON orders(user_id, created_at);

-- 第5步：再EXPLAIN验证
EXPLAIN SELECT * FROM orders WHERE user_id = 1 ORDER BY created_at DESC LIMIT 10;
-- type=ref, rows=50, Extra=Using index condition  ✅
```

## 60.7 前缀索引

对于很长的VARCHAR列（如 `description VARCHAR(1000)`），可以只对前N个字符建索引：

```sql
-- 对email的前10个字符建索引
CREATE INDEX idx_email_prefix ON users(email(10));
```

选择N的原则：

```sql
-- 看不同前缀长度的选择性
SELECT
    COUNT(DISTINCT LEFT(email, 4))  / COUNT(*) AS sel_4,
    COUNT(DISTINCT LEFT(email, 6))  / COUNT(*) AS sel_6,
    COUNT(DISTINCT LEFT(email, 8))  / COUNT(*) AS sel_8,
    COUNT(DISTINCT LEFT(email, 10)) / COUNT(*) AS sel_10
FROM users;
```

选择性接近1.0且N尽量小——这就是最优前缀长度。

> ⚠️ 前缀索引不能用于ORDER BY和GROUP BY，也不能作为覆盖索引（因为索引里只有前缀，没有完整值）。

## 本章小结

| 概念 | 要点 |
|------|------|
| EXPLAIN | SQL诊断工具，核心看type/key/rows/Extra |
| type | ALL < index < range < ref < eq_ref < const |
| Extra | Using filesort和Using temporary是危险信号 |
| 索引失效 | 函数/运算/类型转换/LIKE '%开头'/OR混合/跳过最左前缀 |
| 覆盖索引 | SELECT列全在索引中，不需要回表，Extra显示Using index |
| 选择性 | Cardinality/总行数，越高越适合建索引 |
| 前缀索引 | 只对前N个字符建索引，节省空间，但不能用于排序和覆盖 |

## 自测题

1. **`EXPLAIN SELECT * FROM users WHERE YEAR(birthday) = 2000;` 发现 type=ALL，rows=2000000。如何优化？**

2. **联合索引 `(status, created_at)`，以下SQL的索引使用情况分别是什么？**
   ```sql
   A: SELECT * FROM orders WHERE status = '待支付' AND created_at > '2026-01-01';
   B: SELECT * FROM orders WHERE created_at > '2026-01-01';
   C: SELECT * FROM orders WHERE status = '已完成' ORDER BY created_at DESC LIMIT 10;
   ```

3. **Extra中出现了 `Using filesort`，说明什么问题？如何解决？**

<details>
<summary>参考答案（做完再看）</summary>

1. 索引失效是因为对 `birthday` 使用了 `YEAR()` 函数。改为范围查询：`WHERE birthday >= '2000-01-01' AND birthday < '2001-01-01'`，然后在 `birthday` 列上建索引即可。

2. A：使用了完整的 `(status, created_at)` 索引，type=range。B：跳过status，索引失效，type=ALL。C：使用了完整索引，status等值匹配后按created_at排序（索引本身有序），Extra不会有Using filesort。

3. Using filesort 表示MySQL无法用索引完成排序，需要额外的排序操作（内存排序或磁盘排序），对性能影响大。原因通常是ORDER BY的列没有合适的索引，或者索引列顺序和ORDER BY顺序不一致。解决方案：创建包含ORDER BY列的联合索引，且WHERE条件中的列排在ORDER BY列前面。
</details>
