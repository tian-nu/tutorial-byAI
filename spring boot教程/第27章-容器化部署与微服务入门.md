## 27.1 Docker核心概念

### 27.1.1 Docker解决了什么

每个开发者都经历过这种痛苦：

> "在我机器上能跑啊！"

你的Spring Boot项目在本地运行完美，但到了服务器上：
- JDK版本不一致 → 启动报错
- MySQL字符集配置不同 → 中文乱码
- Redis没装 → 连接失败
- 操作系统不同 → 路径分隔符、编码都不对

**根本原因**：开发环境和运行环境不一致。

Docker的解决方案：**将应用和它的整个运行环境一起打包**——操作系统、JDK、依赖库、配置文件、应用代码，全部封装成一个标准化的**镜像**。无论在谁的机器上，只要装了Docker，运行这个镜像就能得到完全一致的运行环境。

```
传统方式：
你的代码 → 服务器上装JDK → 装MySQL → 装Redis → 改配置 → 部署jar → 各种环境问题

Docker方式：
你的代码 → 打包成镜像 → docker run → 完美运行！
```

### 27.1.2 Docker vs 虚拟机

Docker经常被拿来和虚拟机比较，但它们有本质区别：

```
┌─────────────────────────────────────────────────────┐
│ 虚拟机 (VM)                                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │  App A   │  │  App B   │  │  App C   │          │
│  │ Bins/Lib │  │ Bins/Lib │  │ Bins/Lib │          │
│  │ Guest OS │  │ Guest OS │  │ Guest OS │  ← 每个VM都有完整的操作系统 │
│  └──────────┘  └──────────┘  └──────────┘          │
│  ┌──────────────────────────────────────┐           │
│  │          Hypervisor (虚拟化层)         │           │
│  └──────────────────────────────────────┘           │
│  ┌──────────────────────────────────────┐           │
│  │            Host OS (宿主操作系统)       │           │
│  └──────────────────────────────────────┘           │
│  ┌──────────────────────────────────────┐           │
│  │            Infrastructure (硬件)       │           │
│  └──────────────────────────────────────┘           │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│ Docker容器                                           │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │  App A   │  │  App B   │  │  App C   │          │
│  │ Bins/Lib │  │ Bins/Lib │  │ Bins/Lib │  ← 共享宿主内核，无需Guest OS │
│  └──────────┘  └──────────┘  └──────────┘          │
│  ┌──────────────────────────────────────┐           │
│  │        Docker Engine (容器引擎)        │           │
│  └──────────────────────────────────────┘           │
│  ┌──────────────────────────────────────┐           │
│  │            Host OS (宿主操作系统)       │           │
│  └──────────────────────────────────────┘           │
│  ┌──────────────────────────────────────┐           │
│  │            Infrastructure (硬件)       │           │
│  └──────────────────────────────────────┘           │
└─────────────────────────────────────────────────────┘
```

| 维度 | 虚拟机 | Docker容器 |
|------|--------|-----------|
| **隔离级别** | 硬件级（独立OS） | 进程级（共享内核） |
| **启动速度** | 分钟级 | 秒级 |
| **镜像大小** | GB级 | MB级 |
| **资源占用** | 每个VM需分配固定内存 | 按需使用，共享宿主资源 |
| **性能** | 有虚拟化开销 | 接近原生 |
| **隔离性** | 彻底隔离 | 共享内核，隔离稍弱 |
| **适用场景** | 运行不同操作系统 | 同一OS下的应用隔离 |

**关键理解**：Docker容器不是微型虚拟机，它本质是**宿主操作系统上的一个特殊进程**，通过Linux内核的Namespace（命名空间隔离）和Cgroup（资源限制）技术实现隔离。

### 27.1.3 三大核心概念

**镜像（Image）—— 只读模板**

镜像是创建容器的模板，包含运行应用所需的一切：操作系统、运行时、代码、依赖。镜像是分层的、只读的。

```
镜像层（从下到上）：
┌──────────────────────┐
│  应用代码 + 配置文件    │  ← 你的应用层（可变）
├──────────────────────┤
│  JDK 17              │  ← 运行时层
├──────────────────────┤
│  Alpine Linux        │  ← 基础OS层
└──────────────────────┘
```

**容器（Container）—— 运行实例**

容器是镜像的运行实例。一个镜像可以创建多个容器，每个容器都是隔离的运行环境。容器在镜像之上添加了一个可写层，所有运行时的修改都写在这个层中。

```bash
# 类比：镜像是类（Class），容器是对象（Instance）
# 镜像 = 面向对象中的类定义
# 容器 = new 出来的对象
```

**仓库（Registry）—— 存储镜像**

仓库用于存储和分发镜像。最知名的是Docker Hub（hub.docker.com），国内有阿里云容器镜像服务。

```bash
# 从仓库拉取镜像
docker pull mysql:8.0
docker pull redis:7-alpine
```

> 🚨 **坑点：Windows Docker Desktop需开启虚拟化**
> - Windows上运行Docker Desktop需要硬件虚拟化支持
> - 在BIOS中启用Intel VT-x或AMD-V
> - 选择WSL2后端（推荐）或Hyper-V后端
> - 如果启动报错"Hardware assisted virtualization and data execution protection must be enabled in BIOS"，进BIOS开启虚拟化

> 🚨 **坑点：国内镜像加速器配置**
> - Docker Hub服务器在国外，国内拉取镜像经常超时
> - 配置阿里云镜像加速器：
>
> ```bash
> # Linux: 编辑 /etc/docker/daemon.json
> sudo vim /etc/docker/daemon.json
> ```
>
> ```json
> {
>   "registry-mirrors": [
>     "https://你的加速器地址.mirror.aliyuncs.com"
>   ]
> }
> ```
>
> ```bash
> sudo systemctl daemon-reload
> sudo systemctl restart docker
> ```
>
> - 加速器地址在阿里云控制台"容器镜像服务 → 镜像工具 → 镜像加速器"中获取

## 27.2 Docker命令实战

### 27.2.1 镜像命令

```bash
# 拉取镜像（默认从Docker Hub）
docker pull openjdk:17-jdk-slim
docker pull mysql:8.0
docker pull redis:7-alpine

# 查看本地镜像
docker images
# REPOSITORY   TAG           IMAGE ID       CREATED        SIZE
# mysql        8.0           abc123def456   2 weeks ago    580MB
# redis        7-alpine      789ghi012jkl   3 days ago     32MB

# 删除镜像
docker rmi mysql:8.0

# 构建镜像（根据Dockerfile）
docker build -t ems:latest .
```

### 27.2.2 容器命令

**运行容器**：

```bash
docker run -d \
    --name ems \
    -p 8080:8080 \
    -v /opt/ems/config:/app/config \
    -e SPRING_PROFILES_ACTIVE=prod \
    -e TZ=Asia/Shanghai \
    --restart unless-stopped \
    ems:latest
```

`docker run`参数详解：

| 参数 | 含义 | 示例 |
|------|------|------|
| `-d` | 后台运行（detached模式） | `-d` |
| `--name` | 容器名称 | `--name ems` |
| `-p` | 端口映射（宿主:容器） | `-p 8080:8080` |
| `-v` | 数据卷挂载（宿主:容器） | `-v /opt/data:/var/lib/mysql` |
| `-e` | 环境变量 | `-e MYSQL_ROOT_PASSWORD=xxx` |
| `--restart` | 重启策略 | `--restart unless-stopped` |
| `--network` | 网络 | `--network ems-net` |

**重启策略**：

| 策略 | 行为 |
|------|------|
| `no` | 不自动重启（默认） |
| `on-failure` | 非正常退出时重启 |
| `always` | 总是重启（包括手动停止后Docker重启时） |
| `unless-stopped` | 类似always，但手动停止后Docker重启时不会自动启动 |

**容器生命周期管理**：

```bash
# 查看运行中的容器
docker ps

# 查看所有容器（包括已停止的）
docker ps -a

# 停止容器
docker stop ems

# 启动已停止的容器
docker start ems

# 重启容器
docker restart ems

# 删除容器（必须先停止）
docker rm ems

# 强制删除运行中的容器
docker rm -f ems
```

**查看日志和调试**：

```bash
# 查看容器日志
docker logs ems
docker logs -f ems              # 实时跟踪
docker logs --tail 100 ems      # 最后100行

# 进入容器内部（调试用）
docker exec -it ems sh          # Alpine镜像用sh
docker exec -it ems bash        # Ubuntu/CentOS镜像用bash

# 在容器内执行命令
docker exec ems cat /app/config/application.yml
```

> 🚨 **坑点：docker run -d后容器秒退**
> - 现象：`docker ps`看不到容器，`docker ps -a`显示状态为`Exited`
> - 原因：容器必须有一个前台进程在运行，否则Docker认为任务完成就退出
> - 排查：`docker logs 容器名`查看退出原因
> - Spring Boot应用不会秒退（它有前台进程），但如果配置错误导致启动失败就会退出

> 🚨 **坑点：数据不挂载volumes → 容器删除后数据全丢！**
> - 容器的文件系统是临时的，`docker rm`后所有数据消失
> - MySQL的数据、Redis的持久化文件、应用的日志——都必须挂载到宿主机
> - **黄金法则**：任何需要持久化的数据，必须用`-v`挂载到宿主机

### 27.2.3 清理命令

```bash
# 删除所有已停止的容器
docker container prune

# 删除所有未使用的镜像
docker image prune -a

# 一键清理（删除所有未使用的容器、网络、镜像）
docker system prune -a

# 查看磁盘占用
docker system df
```

## 27.3 Dockerfile编写

### 27.3.1 常用指令详解

Dockerfile是构建镜像的"脚本"，每一行指令对应镜像的一层。

**FROM —— 基础镜像**

```dockerfile
FROM eclipse-temurin:17-jre-alpine
```

- 所有Dockerfile必须以FROM开头
- 选择基础镜像的原则：**够用就行，越小越好**
- `alpine`版本基于Alpine Linux，只有5MB；`slim`版本基于Debian slim，约80MB
- Alpine用`apk`包管理器，Debian用`apt`

**WORKDIR —— 工作目录**

```dockerfile
WORKDIR /app
```

- 设置后续指令的工作目录，相当于`cd`
- 如果目录不存在会自动创建

**COPY vs ADD —— 复制文件**

```dockerfile
COPY target/*.jar app.jar
ADD app.tar.gz /opt/           # ADD会自动解压tar文件
ADD http://example.com/file.txt /tmp/   # ADD可以从URL下载
```

| 维度 | COPY | ADD |
|------|------|-----|
| 复制文件 | ✅ | ✅ |
| 自动解压tar | ❌ | ✅ |
| 从URL下载 | ❌ | ✅ |
| **推荐** | ✅ 优先使用 | 仅在需要解压时使用 |

**RUN —— 构建时执行命令**

```dockerfile
# ❌ 每个RUN创建一个新层（镜像变大）
RUN apk add --no-cache curl
RUN apk add --no-cache vim

# ✅ 合并RUN减少层数
RUN apk add --no-cache curl vim
```

> 🚨 **坑点：每个RUN新建一层 → 合并减少层数**
> - Docker镜像采用分层存储，每个RUN指令创建一个新层
> - 即使后续RUN删除了前面的文件，那一层仍然占用空间
> - 最佳实践：用`&&`合并多个命令为一行
>
> ```dockerfile
> # ❌ 不好：三层
> RUN apt-get update
> RUN apt-get install -y curl
> RUN rm -rf /var/lib/apt/lists/*
>
> # ✅ 好：一层
> RUN apt-get update && \
>     apt-get install -y curl && \
>     rm -rf /var/lib/apt/lists/*
> ```

**EXPOSE —— 声明端口**

```dockerfile
EXPOSE 8080
```

- 纯文档作用，告诉使用者这个镜像会使用哪个端口
- 实际端口映射还是要在`docker run -p`中指定
- 不写EXPOSE也不影响功能，但写了更规范

**CMD vs ENTRYPOINT —— 启动命令**

```dockerfile
# CMD：默认启动命令，可以被docker run的参数覆盖
CMD ["java", "-jar", "app.jar"]

# ENTRYPOINT：固定入口，docker run的参数会追加到ENTRYPOINT后面
ENTRYPOINT ["java", "-jar", "app.jar"]
```

| 维度 | CMD | ENTRYPOINT |
|------|-----|-------------|
| 用途 | 提供默认命令 | 固定入口点 |
| docker run参数 | **覆盖**CMD | **追加**到ENTRYPOINT |
| 可组合使用 | ✅ | ✅ |

组合使用示例：

```dockerfile
ENTRYPOINT ["java", "-jar", "app.jar"]
CMD ["--spring.profiles.active=prod"]
# docker run ems → java -jar app.jar --spring.profiles.active=prod
# docker run ems --spring.profiles.active=dev → java -jar app.jar --spring.profiles.active=dev
```

### 27.3.2 Spring Boot Dockerfile（多阶段构建）

传统方式先在本地`mvn package`，再`COPY`jar到镜像中。但这要求构建环境有Maven和JDK。**多阶段构建**将编译和运行分离在同一个Dockerfile中，最终镜像只包含运行所需的最小内容：

```dockerfile
# ============ 第一阶段：构建 ============
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app

# 先复制pom.xml，下载依赖（利用Docker缓存层，依赖不变时无需重新下载）
COPY pom.xml .
RUN mvn dependency:go-offline

# 再复制源码，编译打包
COPY src ./src
RUN mvn package -DskipTests

# ============ 第二阶段：运行 ============
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app

# 设置时区
ENV TZ=Asia/Shanghai
RUN apk add --no-cache tzdata && \
    cp /usr/share/zoneinfo/$TZ /etc/localtime && \
    echo $TZ > /etc/timezone && \
    apk del tzdata

# 只从第一阶段复制jar包
COPY --from=builder /app/target/*.jar app.jar

EXPOSE 8080

ENTRYPOINT ["java", "-jar", "app.jar"]
```

**多阶段构建的优势**：

| 维度 | 单阶段 | 多阶段 |
|------|--------|--------|
| 镜像大小 | ~800MB（含Maven+JDK） | ~200MB（只有JRE） |
| 安全性 | 包含构建工具和源码 | 只含运行时 |
| 构建环境 | 需要本地有Maven | Docker内完成，无需本地Maven |

> 🚨 **坑点：容器内时区不是CST**
> - 默认Docker容器时区是UTC（+0000），比北京时间晚8小时
> - 日志时间、数据库时间全部对不上
> - **解决方案**：在Dockerfile中设置时区环境变量
>
> ```dockerfile
> ENV TZ=Asia/Shanghai
> ```
>
> 或在`docker run`时通过`-e`传入：
>
> ```bash
> docker run -e TZ=Asia/Shanghai ems:latest
> ```

### 27.3.3 .dockerignore

类似`.gitignore`，`.dockerignore`指定哪些文件不复制到构建上下文中：

```
# .dockerignore
target/
!.target/*.jar       # 排除target目录，但保留jar（如果不用多阶段构建）
.git/
.idea/
*.md
node_modules/
.env
```

构建上下文越小，`docker build`越快。特别是`target/`目录可能几百MB，不排除的话每次构建都要上传这些文件。

## 27.4 Docker Compose

### 27.4.1 为什么需要Compose

一个完整的EMS应用需要三个服务：MySQL、Redis、Spring Boot应用。用`docker run`一个个启动：

```bash
docker run -d --name mysql -e MYSQL_ROOT_PASSWORD=xxx mysql:8.0
docker run -d --name redis redis:7-alpine
docker run -d --name ems -p 8080:8080 --link mysql --link redis ems:latest
```

问题：
- 启动顺序怎么保证？MySQL还没就绪，应用就启动了 → 连接失败
- 停止时要一个个停
- 配置散落在命令行参数中，难以维护
- `--link`已废弃

**Docker Compose**用一个YAML文件定义所有服务，一条命令启停全部：

```bash
docker compose up -d      # 一键启动所有服务
docker compose down        # 一键停止并删除所有容器
docker compose logs -f     # 查看所有服务日志
```

### 27.4.2 完整的docker-compose.yml

```yaml
version: '3.8'

services:
  mysql:
    image: mysql:8.0
    container_name: ems-mysql
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
      MYSQL_DATABASE: ems
      MYSQL_USER: ems
      MYSQL_PASSWORD: ${MYSQL_PASSWORD}
      TZ: Asia/Shanghai
    volumes:
      - mysql_data:/var/lib/mysql
      - ./sql:/docker-entrypoint-initdb.d    # 初始化SQL脚本
    ports:
      - "3306:3306"
    command: --character-set-server=utf8mb4 --collation-server=utf8mb4_unicode_ci
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 10
      start_period: 30s
    restart: unless-stopped
    networks:
      - ems-network

  redis:
    image: redis:7-alpine
    container_name: ems-redis
    command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
    volumes:
      - redis_data:/data
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "-a", "${REDIS_PASSWORD}", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped
    networks:
      - ems-network

  app:
    build: .
    container_name: ems-app
    ports:
      - "8080:8080"
    depends_on:
      mysql:
        condition: service_healthy
      redis:
        condition: service_healthy
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/ems?useSSL=false&serverTimezone=Asia/Shanghai&characterEncoding=utf8mb4
      SPRING_DATASOURCE_USERNAME: ems
      SPRING_DATASOURCE_PASSWORD: ${MYSQL_PASSWORD}
      SPRING_DATA_REDIS_HOST: redis
      SPRING_DATA_REDIS_PORT: 6379
      SPRING_DATA_REDIS_PASSWORD: ${REDIS_PASSWORD}
      SPRING_PROFILES_ACTIVE: prod
      TZ: Asia/Shanghai
    restart: unless-stopped
    networks:
      - ems-network

volumes:
  mysql_data:
  redis_data:

networks:
  ems-network:
    driver: bridge
```

**关键配置解析**：

**服务间通信**：Compose会自动创建一个网络（`ems-network`），服务间通过**服务名**互相访问。注意`app`服务中数据库URL的host是`mysql`（服务名），不是`localhost`或IP地址。

> 🚨 **坑点：depends_on只等容器启动，不等服务就绪**
> - `depends_on: mysql`只保证MySQL容器先启动，不保证MySQL服务已准备好接受连接
> - MySQL启动后还需要几秒到几十秒的初始化时间
> - 如果app在MySQL未就绪时启动 → 连接失败 → 启动失败
> - **解决方案**：加`healthcheck` + `condition: service_healthy`
> - `healthcheck`定义如何检查服务是否就绪，`condition: service_healthy`确保依赖服务健康后才启动

**数据持久化**：

```yaml
volumes:
  - mysql_data:/var/lib/mysql    # 命名卷（Docker管理）
  - ./sql:/docker-entrypoint-initdb.d  # 绑定挂载（映射宿主目录）
```

| 类型 | 语法 | 特点 |
|------|------|------|
| 命名卷 | `volume_name:/container/path` | Docker管理，数据安全，推荐 |
| 绑定挂载 | `./host/path:/container/path` | 映射宿主目录，方便编辑和查看 |

> 🚨 **坑点：compose中密码明文 → 用.env文件**
> - `docker-compose.yml`中的密码直接写在文件里 → 提交到Git → 泄露
> - **解决方案**：使用`.env`文件
>
> ```bash
> # .env（不要提交到Git！加入.gitignore）
> MYSQL_ROOT_PASSWORD=YourRootPassword123!
> MYSQL_PASSWORD=YourEmsPassword123!
> REDIS_PASSWORD=YourRedisPassword123!
> ```
>
> Compose会自动读取同目录下的`.env`文件，用`${变量名}`引用。

### 27.4.3 Compose常用命令

```bash
# 启动所有服务（后台运行）
docker compose up -d

# 启动并重新构建镜像
docker compose up -d --build

# 停止并删除所有容器、网络
docker compose down

# 停止并删除容器、网络、数据卷（⚠️ 数据会丢失！）
docker compose down -v

# 查看服务状态
docker compose ps

# 查看日志
docker compose logs -f              # 所有服务
docker compose logs -f app          # 只看app服务

# 进入容器
docker compose exec app sh

# 重启单个服务
docker compose restart app

# 重新构建并重启单个服务
docker compose up -d --build app
```

## 27.5 微服务概念入门

### 27.5.1 从单体到微服务的演进

**单体架构**（Monolithic）——我们EMS项目目前的样子：

```
┌──────────────────────────────────────────┐
│              EMS 单体应用                  │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │ 员工模块  │ │ 部门模块  │ │ 考勤模块  │ │
│  └──────────┘ └──────────┘ └──────────┘ │
│  ┌──────────────────────────────────────┐│
│  │        共享数据库（MySQL）              ││
│  └──────────────────────────────────────┘│
└──────────────────────────────────────────┘
```

单体架构在项目初期很美好：开发简单、调试方便、部署容易（一个jar包搞定）。但随着业务增长，问题逐渐暴露：

| 问题 | 表现 |
|------|------|
| **代码膨胀** | 几十万行代码，IDE打开都卡 |
| **协同困难** | 多人改同一代码库，冲突不断 |
| **扩展受限** | 只有一个模块需要扩容，却要扩容整个应用 |
| **故障连锁** | 一个模块OOM，整个应用挂掉 |
| **技术债务** | 想升级某个模块的技术栈，影响全局 |

**微服务架构**——将单体拆分为多个独立服务：

```
┌────────────┐  ┌────────────┐  ┌────────────┐
│  员工服务    │  │  部门服务    │  │  考勤服务    │
│  (独立部署)  │  │  (独立部署)  │  │  (独立部署)  │
│  (独立数据库) │  │  (独立数据库) │  │  (独立数据库) │
└──────┬─────┘  └──────┬─────┘  └──────┬─────┘
       │               │               │
       └───────────────┼───────────────┘
                       │
              ┌────────┴────────┐
              │    API 网关      │
              │  (统一入口)      │
              └────────┬────────┘
                       │
                  用户请求
```

### 27.5.2 拆分原则

**高内聚、低耦合**是微服务拆分的核心原则：

- **高内聚**：一个服务内部的业务逻辑紧密相关，变更原因单一
- **低耦合**：服务之间通过明确的API通信，不直接访问对方的数据库

**按业务领域拆分**（DDD思想）：

| 拆分方式 | 示例 | 评价 |
|----------|------|------|
| 按技术层 | Controller服务、Service服务、DAO服务 | ❌ 违反高内聚 |
| 按业务领域 | 员工服务、部门服务、考勤服务 | ✅ 推荐 |
| 按子域 | 用户子域、认证子域、核心业务子域 | ✅ DDD推荐 |

**拆分粒度的判断标准**：

- 一个服务能被一个小团队（6-8人）独立开发和维护
- 一个服务有独立的业务边界和数据边界
- 一个服务可以独立部署，不影响其他服务

### 27.5.3 微服务带来的挑战

> 🚨 **坑点：不要为了微服务而微服务**
> - 微服务不是银弹，它解决了一些问题，但引入了更多复杂性
> - 创业公司、小团队、业务未验证的项目 → **先单体验证**
> - 等业务增长到单体确实扛不住了，再逐步拆分

> 🚨 **坑点：拆分过细 → 运维地狱**
> - 一个简单的CRUD应用拆成10个微服务 → 调试一个请求要跨5个服务
> - 每个服务都要：独立部署、独立监控、独立数据库、独立配置
> - 10个服务 × 3个环境 = 30个部署单元
> - **经验法则**：大多数项目3-8个服务就足够了

微服务引入的挑战和对应解决方案：

| 挑战 | 解决方案 | 技术选型 |
|------|----------|----------|
| 服务发现 | 注册中心 | Nacos / Eureka / Consul |
| 配置管理 | 配置中心 | Nacos / Apollo / Spring Cloud Config |
| 服务通信 | REST / RPC | OpenFeign / Dubbo / gRPC |
| 链路追踪 | 分布式追踪 | SkyWalking / Zipkin |
| 熔断限流 | 服务容错 | Sentinel / Resilience4j |
| 分布式事务 | 一致性保障 | Seata |
| API网关 | 统一入口 | Spring Cloud Gateway |
| 日志聚合 | 集中日志 | ELK（Elasticsearch+Logstash+Kibana） |

## 27.6 Spring Cloud Gateway

### 27.6.1 API网关的作用

微服务架构下，客户端直接调用各个服务会面临很多问题：每个服务都要处理认证、跨域、限流等公共逻辑；服务地址变化时客户端要跟着改；内部服务结构暴露给外部。

**API网关是微服务的统一入口**，承担四项核心职责：

```
客户端请求
    │
    ▼
┌─────────────────────────────────────┐
│           API 网关 (Gateway)          │
│                                      │
│  1. 路由：将请求转发到正确的微服务       │
│  2. 鉴权：统一JWT验证，不合法直接拒绝    │
│  3. 限流：防止恶意请求压垮服务          │
│  4. 日志：统一记录请求日志              │
│                                      │
└──────┬──────────┬──────────┬────────┘
       │          │          │
   ┌───┴───┐  ┌──┴───┐  ┌──┴───┐
   │员工服务│  │部门服务│  │考勤服务│
   └───────┘  └──────┘  └──────┘
```

### 27.6.2 Gateway vs Zuul

| 维度 | Zuul 1.x | Spring Cloud Gateway |
|------|----------|---------------------|
| **模型** | Servlet同步阻塞 | WebFlux异步非阻塞 |
| **底层** | Tomcat | Netty |
| **性能** | 一般 | 优秀（异步非阻塞） |
| **维护** | 已停止维护 | Spring官方维护 |
| **Spring Cloud** | ❌ 不再推荐 | ✅ 官方推荐 |

### 27.6.3 快速搭建Gateway

**第一步：创建Gateway项目**

在Spring Initializr中选择：
- Spring Cloud Gateway
- Nacos Service Discovery（可选）

**第二步：pom.xml**

```xml
<dependencies>
    <!-- Gateway核心（注意：不是spring-boot-starter-web！） -->
    <dependency>
        <groupId>org.springframework.cloud</groupId>
        <artifactId>spring-cloud-starter-gateway</artifactId>
    </dependency>

    <!-- Nacos服务发现（可选） -->
    <dependency>
        <groupId>com.alibaba.cloud</groupId>
        <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
    </dependency>
</dependencies>

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-dependencies</artifactId>
            <version>2023.0.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
        <dependency>
            <groupId>com.alibaba.cloud</groupId>
            <artifactId>spring-cloud-alibaba-dependencies</artifactId>
            <version>2023.0.0.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

> 🚨 **坑点：Gateway基于WebFlux（Netty），不能和spring-boot-starter-web共存！**
> - Gateway底层使用Spring WebFlux（基于Reactor + Netty），是异步非阻塞模型
> - `spring-boot-starter-web`使用Servlet + Tomcat，是同步阻塞模型
> - 两者不能同时引入，否则启动报错
> - **如果你的Gateway项目不小心引入了web starter，必须排除**
> - 排除方式：不要勾选Spring Web依赖，或在pom中排除

**第三步：路由配置**

`application.yml`：

```yaml
server:
  port: 9000

spring:
  application:
    name: ems-gateway
  cloud:
    gateway:
      routes:
        - id: ems-employee
          uri: http://localhost:8081    # 直连模式
          predicates:
            - Path=/api/employees/**
          filters:
            - StripPrefix=0

        - id: ems-department
          uri: http://localhost:8082
          predicates:
            - Path=/api/departments/**
          filters:
            - StripPrefix=0

        - id: ems-attendance
          uri: http://localhost:8083
          predicates:
            - Path=/api/attendance/**
          filters:
            - StripPrefix=0
```

路由配置解析：

| 字段 | 含义 |
|------|------|
| `id` | 路由唯一标识 |
| `uri` | 目标服务地址（直连模式写URL，服务发现模式写`lb://服务名`） |
| `predicates` | 断言（匹配条件），满足才路由 |
| `filters` | 过滤器（请求/响应处理） |

**常用断言（Predicate）**：

| 断言 | 说明 | 示例 |
|------|------|------|
| `Path` | 路径匹配 | `Path=/api/**` |
| `Method` | HTTP方法 | `Method=GET,POST` |
| `Header` | 请求头匹配 | `Header=X-Token, \d+` |
| `Query` | 查询参数 | `Query=name, abc` |
| `After` | 时间之后 | `After=2026-01-01T00:00:00+08:00[Asia/Shanghai]` |

**结合Nacos服务发现的路由**（推荐）：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: ems-employee
          uri: lb://ems-employee-service    # lb://表示负载均衡，后面是Nacos中的服务名
          predicates:
            - Path=/api/employees/**
```

### 27.6.4 Gateway统一鉴权

在Gateway中添加JWT验证过滤器，所有请求先验证Token再路由：

```java
@Component
public class AuthGatewayFilter implements GlobalFilter, Ordered {

    private static final List<String> WHITE_LIST = List.of(
            "/api/auth/login",
            "/api/auth/register"
    );

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        String path = request.getURI().getPath();

        if (WHITE_LIST.stream().anyMatch(path::startsWith)) {
            return chain.filter(exchange);
        }

        String token = request.getHeaders().getFirst("Authorization");
        if (token == null || !token.startsWith("Bearer ")) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        try {
            String jwt = token.substring(7);
            Claims claims = Jwts.parserBuilder()
                    .setSigningKey(Keys.hmacShaKeyFor("YourSecretKeyWithAtLeast256BitsForHS256Algorithm".getBytes()))
                    .build()
                    .parseClaimsJws(jwt)
                    .getBody();

            String userId = claims.getSubject();
            ServerHttpRequest mutatedRequest = request.mutate()
                    .header("X-User-Id", userId)
                    .build();
            return chain.filter(exchange.mutate().request(mutatedRequest).build());
        } catch (Exception e) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
    }

    @Override
    public int getOrder() {
        return -1;
    }
}
```

这样，所有经过Gateway的请求都会先验证JWT，只有白名单中的登录/注册接口可以跳过验证。验证通过后，将用户ID通过请求头传递给下游微服务。

## 27.7 Nacos入门

### 27.7.1 Nacos是什么

Nacos = **Na**ming + **Co**nfiguration + **S**ervice，是阿里巴巴开源的**注册中心 + 配置中心**二合一组件。

```
┌─────────────────────────────────────────────┐
│                  Nacos                        │
│                                               │
│  ┌─────────────────┐  ┌──────────────────┐  │
│  │   注册中心        │  │    配置中心        │  │
│  │                   │  │                    │  │
│  │ 服务注册          │  │ 配置集中管理        │  │
│  │ 服务发现          │  │ 动态配置刷新        │  │
│  │ 健康检查          │  │ 配置版本管理        │  │
│  └─────────────────┘  └──────────────────┘  │
└─────────────────────────────────────────────┘
```

**注册中心**解决"服务之间怎么找到对方"的问题：

```
员工服务启动 → 向Nacos注册自己的地址（192.168.1.100:8081）
部门服务启动 → 向Nacos注册自己的地址（192.168.1.101:8082）
Gateway启动  → 从Nacos获取所有服务地址 → 负载均衡转发
```

**配置中心**解决"几十个服务的配置怎么统一管理"的问题：

```
修改Nacos中的Redis密码 → 所有服务自动刷新 → 无需重启
```

### 27.7.2 Nacos安装

最简单的方式——Docker：

```bash
docker run -d \
    --name nacos \
    -e MODE=standalone \
    -p 8848:8848 \
    -p 9848:9848 \
    nacos/nacos-server:v2.3.0
```

访问`http://localhost:8848/nacos`，默认用户名密码都是`nacos`。

> 🚨 **坑点：Nacos 2.x需要开放gRPC端口（9848/9849），不仅8848**
> - Nacos 1.x只需要8848端口（HTTP通信）
> - Nacos 2.x新增了gRPC通信，主端口8848偏移1000得到gRPC端口9848
> - 如果只开放8848，服务注册会失败，日志报连接超时
> - **规则**：主端口 + 1000 = 客户端gRPC端口，主端口 + 1001 = 服务端gRPC端口
> - 即8848对应9848和9849

### 27.7.3 服务注册

**第一步：添加依赖**

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

**第二步：配置Nacos地址**

`bootstrap.yml`（注意：Nacos配置必须在bootstrap阶段加载，不能用application.yml）：

```yaml
spring:
  application:
    name: ems-employee-service
  cloud:
    nacos:
      discovery:
        server-addr: 192.168.1.100:8848
        namespace: ems-dev
```

> **为什么用bootstrap.yml？**
> - Spring Cloud的配置加载分两个阶段：bootstrap（引导阶段）和application（应用阶段）
> - Nacos地址需要在bootstrap阶段就确定，才能在application阶段从Nacos拉取配置
> - Spring Boot 2.4+默认不再自动加载bootstrap.yml，需要额外引入`spring-cloud-starter-bootstrap`依赖

启动服务后，在Nacos控制台的"服务列表"中可以看到注册的服务。

### 27.7.4 配置中心

**第一步：添加依赖**

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-config</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-bootstrap</artifactId>
</dependency>
```

**第二步：bootstrap.yml配置**

```yaml
spring:
  application:
    name: ems-employee-service
  profiles:
    active: dev
  cloud:
    nacos:
      server-addr: 192.168.1.100:8848
      config:
        namespace: ems-dev
        file-extension: yaml
        shared-configs:
          - data-id: common-redis.yml
            group: DEFAULT_GROUP
            refresh: true
          - data-id: common-mysql.yml
            group: DEFAULT_GROUP
            refresh: true
```

**第三步：在Nacos控制台创建配置**

在Nacos控制台"配置管理 → 配置列表"中创建：

- **Data ID**：`ems-employee-service-dev.yml`（格式：`服务名-环境.后缀`）
- **Group**：`DEFAULT_GROUP`
- **配置内容**：

```yaml
server:
  port: 8081

spring:
  datasource:
    url: jdbc:mysql://mysql:3306/ems_employee?useSSL=false&serverTimezone=Asia/Shanghai
    username: ems
    password: ${MYSQL_PASSWORD}
```

**第四步：动态刷新**

在需要动态刷新的类上加`@RefreshScope`注解：

```java
@RestController
@RefreshScope
public class ConfigController {

    @Value("${app.max-employees:100}")
    private int maxEmployees;

    @GetMapping("/config/max-employees")
    public int getMaxEmployees() {
        return maxEmployees;
    }
}
```

在Nacos控制台修改`app.max-employees`的值，服务会自动刷新，无需重启。

### 27.8.3 进阶方向

以下是5个值得深入的方向，每个方向都附上推荐的学习路径和理由：

---

**方向一：消息队列 —— 系统解耦与异步处理**

为什么学：微服务之间不能只有同步调用（REST），异步消息是解耦、削峰、最终一致性的基础。

| 阶段 | 技术 | 理由 |
|------|------|------|
| 入门 | RabbitMQ | Erlang实现，AMQP协议，管理界面友好，社区活跃，适合入门学习消息模型 |
| 进阶 | RocketMQ | 阿里开源，Java实现，支持事务消息、顺序消息，国内大厂广泛使用 |
| 高阶 | Kafka | Scala/Java实现，超高吞吐量，大数据生态标配，适合日志收集和流处理 |

学习路径：先理解消息模型（点对点、发布订阅）→ 死信队列 → 消息幂等性 → 事务消息 → 顺序消息

---

**方向二：搜索引擎 —— 全文检索与数据分析**

为什么学：MySQL的LIKE模糊查询在大数据量下性能极差，Elasticsearch是全文检索的事实标准。

| 技术 | 用途 |
|------|------|
| Elasticsearch | 分布式搜索引擎，支持全文检索、聚合分析、近实时搜索 |
| Kibana | 可视化界面，日志分析、数据看板 |
| Logstash | 数据采集管道，将日志/数据库数据导入ES |

学习路径：倒排索引原理 → 索引映射设计 → DSL查询语法 → Spring Data Elasticsearch集成 → ELK日志平台搭建

---

**方向三：分布式系统 —— 高可用与一致性保障**

为什么学：微服务拆分后，服务间调用可能失败、超时、重试，需要熔断、限流、分布式事务等保障机制。

| 技术 | 解决的问题 |
|------|-----------|
| Sentinel | 流量控制、熔断降级、热点限流（阿里巴巴开源） |
| Resilience4j | 轻量级熔断器，Spring Cloud官方推荐 |
| Seata | 分布式事务解决方案（AT/TCC/Saga模式） |
| SkyWalking | 分布式链路追踪，可视化调用链 |

学习路径：CAP定理与BASE理论 → 熔断器模式 → 限流算法（令牌桶/滑动窗口） → 分布式事务 → 链路追踪

---

**方向四：DevOps —— 自动化交付流水线**

为什么学：手动部署容易出错、效率低，CI/CD流水线让代码从提交到上线全自动。

| 技术 | 用途 |
|------|------|
| Jenkins | 老牌CI/CD工具，插件丰富 |
| GitLab CI | 与GitLab深度集成，配置即代码 |
| GitHub Actions | GitHub原生CI/CD，免费额度充足 |
| Kubernetes | 容器编排平台，自动扩缩容、滚动更新、服务发现 |

学习路径：Git工作流（GitFlow/Trunk Based） → CI流水线（自动构建+测试） → CD流水线（自动部署） → K8s入门（Pod/Service/Deployment） → Helm包管理

**方向五：系统设计 —— 架构师思维**

为什么学：技术会过时，但设计思维不会。掌握系统设计能力，才能从"会用框架"升级到"能设计系统"。

| DDD（领域驱动设计） | 限界上下文、聚合根、领域事件、CQRS |
| 整洁架构 | 依赖规则、用例层、接口适配层、框架层 |
| 系统设计面试 | 高并发、高可用、可扩展系统的设计方法论 |

学习路径：设计模式（结合项目实践） → DDD战略设计 → 整洁架构 → 分布式系统设计模式 → 系统设计案例分析

## 本章小结

1. **Docker核心**：镜像（只读模板）、容器（运行实例）、仓库（存储分发），容器不是微型虚拟机而是隔离的进程
2. **Docker命令**：pull/run/ps/stop/rm/logs/exec是日常高频命令，`-d`后台运行、`-p`端口映射、`-v`数据挂载、`-e`环境变量
3. **Dockerfile**：多阶段构建将编译和运行分离，最终镜像只有JRE+应用，约200MB；注意合并RUN减少层数、设置时区
4. **Docker Compose**：用YAML编排多服务，`healthcheck + condition: service_healthy`解决服务启动顺序问题，`.env`文件管理敏感信息
5. **微服务**：按业务领域拆分，高内聚低耦合；不要为了微服务而微服务，先单体验证再拆分
6. **Gateway**：API网关四合一（路由+鉴权+限流+日志），基于WebFlux不能和spring-boot-starter-web共存
7. **Nacos**：注册中心+配置中心二合一，2.x版本需开放9848/9849端口，配置动态刷新用`@RefreshScope`

### Docker Compose编排（单体版）

`docker-compose.yml`：

```yaml
version: '3.8'

services:
  mysql:
    image: mysql:8.0
    container_name: ems-mysql
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
      MYSQL_DATABASE: ems
      MYSQL_USER: ems
      MYSQL_PASSWORD: ${MYSQL_PASSWORD}
      TZ: Asia/Shanghai
    volumes:
      - mysql_data:/var/lib/mysql
      - ./sql:/docker-entrypoint-initdb.d
    ports:
      - "3306:3306"
    command: --character-set-server=utf8mb4 --collation-server=utf8mb4_unicode_ci
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 10
      start_period: 30s
    restart: unless-stopped
    networks:
      - ems-network

  redis:
    image: redis:7-alpine
    container_name: ems-redis
    command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
    volumes:
      - redis_data:/data
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "-a", "${REDIS_PASSWORD}", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped
    networks:
      - ems-network

  app:
    build:
      context: ./ems-app
      dockerfile: Dockerfile
    container_name: ems-app
    ports:
      - "8080:8080"
    depends_on:
      mysql:
        condition: service_healthy
      redis:
        condition: service_healthy
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/ems?useSSL=false&serverTimezone=Asia/Shanghai&characterEncoding=utf8mb4
      SPRING_DATASOURCE_USERNAME: ems
      SPRING_DATASOURCE_PASSWORD: ${MYSQL_PASSWORD}
      SPRING_DATA_REDIS_HOST: redis
      SPRING_DATA_REDIS_PORT: 6379
      SPRING_DATA_REDIS_PASSWORD: ${REDIS_PASSWORD}
      SPRING_PROFILES_ACTIVE: prod
      TZ: Asia/Shanghai
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/actuator/health"]
      interval: 15s
      timeout: 5s
      retries: 5
      start_period: 60s
    restart: unless-stopped
    networks:
      - ems-network

  nginx:
    image: nginx:alpine
    container_name: ems-nginx
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/ems.conf:/etc/nginx/conf.d/default.conf:ro
      - ./nginx/ssl:/etc/nginx/ssl:ro
    depends_on:
      app:
        condition: service_healthy
    restart: unless-stopped
    networks:
      - ems-network

volumes:
  mysql_data:
  redis_data:

networks:
  ems-network:
    driver: bridge
```

### 应用Dockerfile

```dockerfile
FROM maven:3.9-eclipse-temurin-17 AS builder

FROM eclipse-temurin:17-jre-alpine

### 微服务版Docker Compose（可选进阶）

当需要从单体拆分为微服务时，替换`docker-compose.yml`中的`app`服务为多个微服务：

```yaml
  nacos:
    image: nacos/nacos-server:v2.3.0
    container_name: ems-nacos
    environment:
      MODE: standalone
      TZ: Asia/Shanghai
    ports:
      - "8848:8848"
      - "9848:9848"
    restart: unless-stopped
    networks:
      - ems-network

  gateway:
    build:
      context: ./ems-gateway
      dockerfile: Dockerfile
    container_name: ems-gateway
    ports:
      - "8080:8080"
    depends_on:
      nacos:
        condition: service_started
    environment:
      NACOS_SERVER_ADDR: nacos:8848
      TZ: Asia/Shanghai
    restart: unless-stopped
    networks:
      - ems-network

  employee-service:
    build:
      context: ./ems-employee-service
      dockerfile: Dockerfile
    container_name: ems-employee-service
    depends_on:
      mysql:
        condition: service_healthy
      nacos:
        condition: service_started
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/ems_employee?useSSL=false&serverTimezone=Asia/Shanghai
      SPRING_DATASOURCE_USERNAME: ems
      SPRING_DATASOURCE_PASSWORD: ${MYSQL_PASSWORD}
      NACOS_SERVER_ADDR: nacos:8848
      TZ: Asia/Shanghai
    restart: unless-stopped
    networks:
      - ems-network

  department-service:
    build:
      context: ./ems-department-service
      dockerfile: Dockerfile
    container_name: ems-department-service
    depends_on:
      mysql:
        condition: service_healthy
      nacos:
        condition: service_started
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/ems_department?useSSL=false&serverTimezone=Asia/Shanghai
      SPRING_DATASOURCE_USERNAME: ems

> **从单体到微服务的迁移建议**：
> 1. 先确保单体版在Docker Compose下稳定运行
> 2. 按业务领域拆分模块：先拆出最独立的模块（如部门服务）
> 3. 每拆一个服务，都要确保原有功能不受影响
> 4. 引入Nacos做服务注册，Gateway做统一入口
> 5. 逐步替换，不要一次性全部拆分
