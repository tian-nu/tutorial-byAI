## 我在做什么？

想象你是一个图书管理员。图书馆里有10万本书，读者来借书，你得从书堆里翻找——如果书是随意堆放的，你每次都要从头到尾翻一遍，这得翻到什么时候？

聪明的管理员会把书**按某种规则组织起来**：按分类号排列、建立索引卡片、或者用计算机管理。不同的组织方式，决定了你找书的快慢。

**数据结构**就是程序员的"图书管理方法"——它对内存中的数据进行特定方式的组织，让我们能用**更少的CPU时间**和**更少的内存**去完成同一件事。

`*此术语见附录F*`：数据结构、抽象数据类型（ADT）。

打个比方：
- **数组**就像一排编号的储物柜，你知道第5号柜子在哪，直接走过去就能打开。
- **链表**就像寻宝游戏，每个宝箱里放着一张纸条指向下一个宝箱的位置，你得按纸条一张张找下去。

这一章不讲具体的数据结构怎么写，而是回答"为什么我要学数据结构"这个根本问题。理解了这个，后面六章的精力才不会白费。

## 42.1 同样的问题，不同的解法

假设你有一个任务：存储100个学生的考试成绩，然后能够查询某个学生的分数。

### 解法A：用数组

```c
int scores[100];
// 第i个学生的成绩就是 scores[i]
// 查询第50个学生的成绩：scores[49]  —— 一步到位！
```

### 解法B：用链表

```c
struct Student {
    int score;
    struct Student *next;
};
// 查询第50个学生：需要从第1个开始，顺着 next 指针数49次才能找到
```

### 解法C：用二叉搜索树

```c
struct TreeNode {
    int id;          // 学号
    int score;
    struct TreeNode *left;
    struct TreeNode *right;
};
// 按学号组织成二叉搜索树，查询时每次排除一半可能
```

| 操作 | 数组 | 链表 | 二叉搜索树 |
|------|------|------|------------|
| 按位置访问第i个 | O(1) | O(n) | O(n) |
| 按值查找 | O(n) | O(n) | O(log n) 平均 |
| 在中间插入新元素 | O(n) | O(1) | O(log n) 平均 |
| 删除中间元素 | O(n) | O(1) | O(log n) 平均 |
| 内存占用 | 连续固定 | 分散+指针 | 分散+两个指针 |

> **核心结论**：没有万能的数据结构。选择哪种结构，取决于你的程序**最常做哪种操作**。

## 42.2 数组 vs 链表：经典对决

这是学习数据结构时最重要的对比。用ASCII图画出来，一目了然：

### 数组：一片连续内存

```
内存地址:  1000   1004   1008   1012   1016
          +------+------+------+------+------+
          |  10  |  20  |  30  |  40  |  50  |
          +------+------+------+------+------+
索引:        0      1      2      3      4
```

- 优点：内存连续，通过 `base + index * sizeof(element)` 直接计算地址，O(1) 随机访问。
- 缺点：
  - 大小固定（静态数组），不能动态增长。
  - 在中间插入元素，需要把后面所有元素往后挪。

```
插入前: [10] [20] [30] [40] [50]
              ^ 在这里插入 25

步骤1: 所有元素后移
       [10] [20] [  ] [30] [40] [50]
                  \    \    \    \
步骤2: 填入 25
       [10] [20] [25] [30] [40] [50]
       
—— 移动了 3 个元素！如果数组有 100 万个元素，移动量可能非常巨大。
```

### 链表：分散的节点

```
        +------+------+    +------+------+    +------+------+
head--> |  10  |  *---+--->|  20  |  *---+--->|  30  | NULL |
        +------+------+    +------+------+    +------+------+
        数据    指针        数据    指针        数据    指针
        (节点1)             (节点2)             (节点3)
```

- 优点：
  - 动态大小，随时添加新节点。
  - 在任意位置插入/删除只需修改指针，O(1)。
- 缺点：
  - 不能随机访问，找第n个元素需要遍历n次。
  - 每个节点额外存储指针，空间开销大。
  - 内存不连续，缓存友好度差。

```c
/*
 * 数组：空间换时间（但空间是固定的）
 * 链表：时间换灵活（但时间代价可能很大）
 *
 * 选择建议：
 *   - 频繁随机访问 -> 数组
 *   - 频繁增删中间元素 -> 链表
 *   - 内存不确定 -> 链表
 *   - 数据量确定且不大 -> 数组
 */
```

## 42.3 C语言没有内置数据结构

Java有 `ArrayList`、`HashMap`，Python有 `list`、`dict`，C++有 STL。但C语言标准库里**几乎没有提供任何数据结构**。

你能直接使用的只有：
- 数组（静态分配）
- `malloc`/`free` 动态内存
- 结构体和指针

**这意味着你既是"需求方"也是"施工方"**——你要用最基础的砖瓦（结构体、指针、malloc）自己搭建高楼（链表、栈、队列、树、哈希表）。

这既是C语言的难点，也是C语言的魅力。亲手实现一遍，你将对计算机如何组织数据有深刻的理解。这也是为什么很多高校用C语言教授数据结构课程。

## 42.4 抽象数据类型（ADT）

`*此术语见附录F*`

**抽象数据类型（Abstract Data Type，ADT）** 是一个非常重要的概念。它的意思是：**把"数据长什么样"和"数据能做什么"分开**。

举个例子——你在用遥控器控制电视：
- 你只需要知道按"音量+"按钮能让声音变大（**能做什么**）
- 你不需要知道遥控器内部电路如何编码红外信号（**长什么样**）

在数据结构中，ADT定义了：
1. **数据之间的逻辑关系**（如"栈是一端进一端出"）
2. **可以执行的操作**（如 push、pop）

而**不关心**底层是用数组还是链表实现的。

```
┌─────────────────────────────────────┐
│        栈 ADT（抽象层）              │
│  ────────────────────────────────   │
│  操作：push(入栈)、pop(出栈)、       │
│        peek(查看栈顶)、              │
│        isEmpty(判空)                │
│                                     │
│  用户只调用这些函数，不在乎底层。     │
└──────────────┬──────────────────────┘
               │ "实现"关系
    ┌──────────┴──────────┐
    │                     │
    ▼                     ▼
┌───────────────┐  ┌───────────────┐
│ 数组实现       │  │ 链表实现       │
│ (第46章会讲)   │  │ (第46章会讲)   │
└───────────────┘  └───────────────┘
```

在C语言中，ADT通常这样组织代码：

```c
/* ============ stack.h (头文件——声明"能做什么") ============ */
#ifndef STACK_H
#define STACK_H

typedef struct Stack Stack;  /* 不暴露内部结构！ */

Stack* stack_create(void);
void   stack_destroy(Stack *s);
int    stack_push(Stack *s, int value);
int    stack_pop(Stack *s, int *out);
int    stack_peek(const Stack *s, int *out);
int    stack_is_empty(const Stack *s);

#endif

/* ============ stack.c (实现文件——定义"长什么样") ============ */
#include "stack.h"
#include <stdlib.h>

struct Stack {
    int *data;
    int  top;
    int  capacity;
};
/* ... 函数实现 ... */

/* ============ main.c (使用者——只调用接口) ============ */
#include "stack.h"

int main(void) {
    Stack *s = stack_create();
    stack_push(s, 42);
    int val;
    stack_pop(s, &val);
    stack_destroy(s);
    return 0;
}
```

**ADT的好处**：
1. **封装**：调用者不知道内部实现，改了实现（比如从数组换成链表）不影响调用者。
2. **可维护性**：接口稳定，实现可以优化。
3. **代码可读性**：`stack_push(s, 10)` 比手动操作数组清晰得多。

## 42.6 复杂度速查

在后续章节中，我们会频繁提到**时间复杂度**。这里给出快速参考：

| 常见复杂度 | 含义 | 典型例子 |
|-----------|------|----------|
| O(1) | 常数时间，与数据量无关 | 数组按下标访问 |
| O(log n) | 每次排除一半 | 二分查找、平衡树查找 |
| O(n) | 遍历全部元素 | 链表查找、数组插入 |
| O(n log n) | 比O(n)慢，比O(n²)快 | 归并排序 |
| O(n²) | 嵌套循环 | 冒泡排序 |
| O(2ⁿ) | 指数爆炸 | 穷举子集 |

## 不对怎么办？

**常见误区一："数据结构就是链表、树这些东西"**

❌ 错误理解：数据结构 = 特定的几种形态。

✅ 正确理解：数据结构是一种**组织思想**。数组也是一种数据结构，文件目录也是一种数据结构。任何"按某种规则把数据排列起来方便使用"的方式，都是数据结构。

**常见误区二："C语言没有数据结构，没法用"**

❌ 错误想法：C语言太原始了，不如用Python。

✅ 正确理解：C语言是给了你**最大自由度**去设计数据结构。你写的链表，可以精确控制每个字节的内存布局。这种能力在嵌入式系统、操作系统内核中不可替代。

**常见误区三："学了数据结构也没什么用"**

❌ 错误想法：实际工作中又不会从头写链表。

✅ 正确理解：你大概率不会从零写链表（除非做底层开发），但理解链表原理让你能：
- 理解 `Linux` 内核中的链表实现
- 看懂数据库 B+ 树索引为什么快
- 面试中不被算法题卡住
- 设计自己的数据存储方案时有更多选择

```c
/* counter.h —— 当前有问题的版本 */
#ifndef COUNTER_H
#define COUNTER_H

typedef struct {
    int value;
} Counter;

#endif
```

要求：
- `counter.h` 只暴露操作函数，不暴露 `value` 字段。
