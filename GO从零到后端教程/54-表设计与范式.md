# 第54章 · 表设计与范式

> "不会建表的后端就像不会画图纸的建筑师——地基打好就开始垒墙，结果砌到一半发现承重墙位置不对，推倒重来。数据库范式就是这么一套建筑规范，告诉你怎么设计表结构才不会在后期发现这里数据重复、那里更新漏改、删掉一条记录连带着丢了一堆不该丢的东西。"

---

## 54.1 第一范式（1NF）：字段不可再分

**定义**：表的每个字段必须是**原子性**的——不可再拆分的最小数据单元。

### 违反1NF的例子

```
students:
id | name   | phone_numbers
1  | 张三   | 13800001111, 13900002222
2  | 李四   | 13700003333
```

`phone_numbers` 字段里塞了多个值，用逗号分隔——罪大恶极。后果：

- 查"手机号13800001111是谁的？"：`WHERE phone_numbers LIKE '%13800001111%'`，没法用索引，全表扫描。
- 更新"把13800001111改成13600000000"：你得用字符串替换，容易误伤（比如138000011112也会被匹配）。
- 统计"每个人有几个手机号"：要去数逗号，SQL根本没法优雅实现。

### 符合1NF

拆成两张表：

```sql
CREATE TABLE students (
    id   BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    name VARCHAR(50)     NOT NULL,
    PRIMARY KEY (id)
);

CREATE TABLE student_phones (
    id         BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    student_id BIGINT UNSIGNED NOT NULL,
    phone      CHAR(11)        NOT NULL,
    PRIMARY KEY (id),
    INDEX idx_student_id (student_id)
);
```

现在一个学生可以有多个手机号，每个手机号一行。查询、更新、统计都清晰明了。

1NF是最基本的底线。违反1NF的表基本没法写正常的SQL，所有操作都要靠字符串处理硬撑。

## 54.2 第二范式（2NF）：消除部分依赖

**定义**：在满足1NF的基础上，每个非主键列必须**完全依赖于主键**（不能只依赖于主键的一部分）。这个定义针对的是**有复合主键**的表。

### 违反2NF的例子

一张学生选课表，主键是 `(student_id, course_id)` 复合主键：

```
student_courses:
student_id | course_id | course_name    | teacher    | score
1          | 101       | 高等数学       | 王教授     | 85
1          | 102       | 大学英语       | 李教授     | 90
2          | 101       | 高等数学       | 王教授     | 78
```

主键是 `(student_id, course_id)` 的组合。问题出在哪？

- `course_name`：只依赖 `course_id`，不依赖 `student_id`。
- `teacher`：也只依赖 `course_id`。

这意味着 `course_name` 和 `teacher` 只**部分依赖**于主键——违反了2NF。

**后果**："高等数学"的课程名和王教授的名字被重复存了多次（每个选这门课的学生都存一份）。如果王教授换人了，你要在几十条记录里修改——一漏就错。

### 符合2NF

拆成两张表：

```sql
CREATE TABLE courses (
    id    BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    name  VARCHAR(100)    NOT NULL,
    teacher VARCHAR(50)   NOT NULL,
    PRIMARY KEY (id)
);

CREATE TABLE student_scores (
    student_id BIGINT UNSIGNED NOT NULL,
    course_id  BIGINT UNSIGNED NOT NULL,
    score      TINYINT UNSIGNED DEFAULT 0,
    PRIMARY KEY (student_id, course_id)
);
```

现在课程信息只存一份。`score` 依赖的是整个复合主键（哪个学生在哪门课的分数），符合2NF。

**简单记忆**：如果你的表没有复合主键（只有一个主键字段），那它**自动满足2NF**。2NF只在复合主键场景下才可能被违反。

## 54.3 第三范式（3NF）：消除传递依赖

**定义**：在满足2NF的基础上，非主键列不能依赖于**另一个非主键列**——每条非主键信息都只能直接依赖主键，不能绕路。

### 违反3NF的例子

```
employees:
id | name   | dept_id | dept_name | dept_location
1  | 张三   | 1       | 技术部    | 3楼
2  | 李四   | 1       | 技术部    | 3楼
3  | 王五   | 2       | 市场部    | 5楼
```

依赖链条：
- `dept_name` 依赖于 `dept_id`（不是主键id）
- `dept_location` 依赖于 `dept_id`
- 即：主键id → dept_id → dept_name / dept_location

这是**传递依赖**——非主键字段通过 `dept_id` 间接依赖主键。

**后果**：技术部搬到了6楼，你得改张三和李四两条记录。改漏了就是数据不一致——张三看是6楼，李四看还是3楼。

### 符合3NF

```sql
CREATE TABLE departments (
    id       BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    name     VARCHAR(50)     NOT NULL,
    location VARCHAR(100)    NOT NULL,
    PRIMARY KEY (id)
);

CREATE TABLE employees (
    id      BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    name    VARCHAR(50)     NOT NULL,
    dept_id BIGINT UNSIGNED NOT NULL,
    PRIMARY KEY (id),
    INDEX idx_dept_id (dept_id)
);
```

现在部门信息只存一份。要查"张三在哪个部门、在几楼"，用JOIN：

```sql
SELECT e.name, d.name, d.location
FROM employees e
JOIN departments d ON e.dept_id = d.id;
```

### 三范式速查口诀

| 范式 | 口诀 | 通俗说法 |
|------|------|----------|
| 1NF | 列不可分 | 一个格子里不能塞多个值 |
| 2NF | 完全依赖主键 | 不能因为主键的一部分就确定某个字段的值 |
| 3NF | 不依赖非主键 | 不能因为另一个不是主键的字段就确定某个字段的值 |

🤔 **想多一点**："既然有范式规则，是不是所有表都要严格遵守3NF？"这是一个好问题。答案是：**理论上应该，实践中不一定**。范式保证数据不冗余，但也意味着每次查询要JOIN很多表，性能下降。所以才有了"反范式化"。

## 54.4 反范式化：当规则可以被打破

**场景**：一个电商订单表，下单时要显示用户收货地址。按3NF设计，地址应该在用户表里。但每次查订单都要JOIN用户表——订单量大了以后，这个JOIN就是性能瓶颈。

**反范式化做法**：在订单表里直接冗余一个地址字段：

```sql
CREATE TABLE orders (
    id            BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    user_id       BIGINT UNSIGNED NOT NULL,
    address_snapshot VARCHAR(500) NOT NULL,
    amount        DECIMAL(10, 2)  NOT NULL,
    created_at    DATETIME        NOT NULL,
    PRIMARY KEY (id)
);
```

下单时把用户的当前地址**快照**存进订单。即使后来用户改了地址，历史订单的地址也不会变——这反而**更合理**（订单上的地址应该是下单时那一刻的地址）。

### 何时反范式化

| 反范式化的理由 | 说明 |
|---------------|------|
| 读多写少 | 查询频率远高于更新频率，冗余加速读取 |
| 历史快照需求 | 订单地址、商品快照——需要记录"当时的"信息 |
| JOIN过多性能差 | 多表JOIN太慢，冗余一个字段减少JOIN |
| 统计报表 | 单独的统计表提前算好数据，而不是实时聚合 |

### 何时坚守范式

| 坚守范式的理由 | 说明 |
|---------------|------|
| 写多读少 | 数据频繁更新，冗余意味着每次更新要改多处 |
| 强一致性要求 | 金融、库存——"一个数字两种表示"是致命错误 |
| 小型项目 | 数据量不大，JOIN性能无压力，范式让代码更干净 |

**Golden Rule**：先按3NF设计，确认无误。然后在性能瓶颈出现时，有意识地、有文档记录地反范式化。

## 54.5 ER图入门

ER图（Entity-Relationship Diagram）是数据库设计的可视化工具。你不需要用UML工具画——手绘或用draw.io就够了。下面是一个简单的电商ER图：

### 实体（Entity）

矩形表示实体——对应一张表。

### 属性（Attribute）

椭圆表示属性——对应表的字段。

### 关系（Relationship）

菱形表示关系——两个实体之间的联系。

### 基数（Cardinality）

标记关系的数量：

- **一对一（1:1）**：一个用户有一个身份证号。
- **一对多（1:N）**：一个用户有多笔订单。
- **多对多（M:N）**：一个学生选多门课，一门课有多个学生。

**文字版ER图示例（电商系统）**：

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  users   │ 1     N │  orders  │ N     1 │ products │
│──────────│─────────│──────────│─────────│──────────│
│ id (PK)  │         │ id (PK)  │         │ id (PK)  │
│ username │         │ user_id  │         │ name     │
│ email    │         │ amount   │         │ price    │
│ phone    │         │ status   │         │ stock    │
└──────────┘         │ created_at│        └──────────┘
                     └──────────┘
                          │
                          │ N
                     ┌──────────────┐
                     │ order_items  │
                     │──────────────│
                     │ order_id(PK) │
                     │ product_id(PK)│
                     │ quantity     │
                     │ price_snapshot│
                     └──────────────┘
```

- users 和 orders：一对多（1:N）——一个用户多个订单。
- products 和 orders：多对多（M:N）——通过 order_items 中间表实现。
- order_items：复合主键 `(order_id, product_id)`。

### 实际工作中的设计流程

1. **需求梳理**：和产品经理确认业务范围。
2. **画ER图**：列出实体、属性、关系。
3. **转表结构**：实体→表，1:N加外键，M:N加中间表。
4. **检查范式**：确保至少满足3NF。
5. **反范式化评估**：判断哪些地方需要冗余。
6. **编写建表SQL**。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 1NF | 字段原子性，一个格子里不能塞逗号分隔的多个值 |
| 2NF | 消除部分依赖，非主键列必须依赖**整个**主键（针对复合主键） |
| 3NF | 消除传递依赖，非主键列不能依赖另一个非主键列 |
| 范式好处 | 减少冗余、保证一致性、更新安全 |
| 反范式化 | 读多写少时适当冗余，历史快照场景必须冗余 |
| ER图 | 实体（矩形）、属性（椭圆）、关系（菱形）、基数（1:1/1:N/M:N） |
| M:N处理 | 必须引入中间表（如order_items），拆成两个1:N关系 |
| 设计流程 | 需求→ER图→转表→检查范式→反范式评估→建表SQL |
