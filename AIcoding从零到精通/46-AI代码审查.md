## 46.1 代码审查是什么

> **📖 术语：代码审查（Code Review）**——写完代码后，找另一个人帮你看看，检查有没有Bug、风格是否规范、有没有更好的写法。就像你写作文后，找同桌帮你批改一遍——多一双眼睛，多发现一些你没注意到的问题。

在传统软件公司，代码审查是**强制流程**：任何代码必须经过至少一个人审查，才能上线。因为一个人的脑容量有限，写出来的代码总有自己注意不到的问题。

**AI编程时代，你可以让AI帮你做代码审查**——免费、即时、不会累。

---

## 46.2 为什么AI特别适合做代码审查

代码审查的本质是**找模式**：

- 这个变量命名不符合规范 → 模式识别
- 这段代码有SQL注入风险 → 模式识别
- 这个循环可以优化 → 模式识别

**AI最擅长的就是模式识别。** 它看过几亿行代码，知道什么写法是"好的"、什么写法是"差的"、什么写法是"危险的"。

而且AI不会：
- 觉得"算了，这代码虽然不好但也能跑，不说了"（AI会如实报告）
- 因为和你是朋友就放水（AI没有感情）
- 纠结"这么说会不会太直接"（AI不会顾虑人际）

## 46.3 完整审查流程

### 五步审查法

```
① 把代码给AI
② 让AI按五个维度审查
③ AI列出问题清单
④ 你自己判断每个问题
⑤ 选择性修改
```

---

### 步骤①：把代码给AI

选中你要审查的代码（可以是一个文件，也可以是多个文件），打开Chat。

### 步骤②：让AI按五个维度审查

发送以下Prompt：

```
请帮我审查这段代码，从以下五个维度分析：

1. **Bug**：有没有逻辑错误？边界情况处理了吗？
2. **风格**：命名规范吗？代码格式一致吗？
3. **性能**：有没有明显的性能问题？循环嵌套太多？
4. **安全**：有没有SQL注入、XSS、敏感信息泄露等风险？
5. **可读性**：有没有复杂到看不懂的地方？函数是不是太长了？

对于每个问题，请说明：
- 问题类型（Bug/风格/性能/安全/可读性）
- 严重程度（🔴严重 / 🟡中等 / 🟢建议）
- 具体位置（哪一行或哪段代码）
- 问题描述
- 修改建议
```

---

### 步骤③：AI列出问题清单

AI会给出一个结构化的审查报告。

### 步骤④：你自己判断——这是最关键的一步

**AI的审查意见，不是圣旨。** 你必须自己判断：

- 🔴 严重Bug → 必须修
- 🟡 中等建议 → 大多数应该修
- 🟢 风格建议 → 可修可不修，看你的偏好

**特别要注意**：AI可能在审查中提出"错误的建议"——比如它建议你改成某种写法，但那种写法在你的场景下不适用。AI不知道你的全部业务逻辑，它的建议是基于"通用最佳实践"的。

---

### 步骤⑤：选择性修改

把你想修的问题，重新发给AI：

```
请帮我修复上面审查报告中的问题 #1、#3、#5，保持其他部分不变。
```

---

## 46.4 完整案例：审查一个用户注册函数

### 原始代码

```python
import sqlite3

def register(username, password, email):
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    cursor.execute(f"INSERT INTO users (username, password, email) VALUES ('{username}', '{password}', '{email}')")
    conn.commit()
    conn.close()
    return "注册成功！"
```

### 发给AI的审查Prompt

```
请帮我审查这段代码，从Bug、风格、性能、安全、可读性五个维度分析：

```python
import sqlite3

def register(username, password, email):
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    cursor.execute(f"INSERT INTO users (username, password, email) VALUES ('{username}', '{password}', '{email}')")
    conn.commit()
    conn.close()
    return "注册成功！"
```

对于每个问题，请说明严重程度、具体位置、问题描述和修改建议。
```

### AI的审查报告（预期）

> ## 代码审查报告
>
> ### 🔴 严重问题
>
> **1. SQL注入风险（安全）**
> - 位置：第5行 `cursor.execute(f"INSERT INTO ... VALUES ('{username}', '{password}', '{email}')")`
> - 问题：使用 f-string 拼接SQL语句，如果用户输入包含 `' OR '1'='1` 等恶意内容，可以绕过认证或删除数据
> - 建议：使用参数化查询 `cursor.execute("INSERT INTO ... VALUES (?, ?, ?)", (username, password, email))`
>
> **2. 密码明文存储（安全）**
> - 位置：第5行，密码直接存入数据库
> - 问题：密码以明文形式存储，数据库泄露 = 所有用户密码泄露
> - 建议：使用 `bcrypt` 或 `hashlib` 对密码进行哈希后再存储
>
> ### 🟡 中等问题
>
> **3. 缺少输入验证（Bug）**
> - 位置：整个函数
> - 问题：没有检查用户名是否为空、密码长度是否足够、邮箱格式是否正确
> - 建议：在操作数据库前，先验证所有输入
>
> **4. 缺少错误处理（Bug）**
> - 位置：整个函数
> - 问题：如果数据库连接失败、用户名已存在等情况，程序会直接崩溃
> - 建议：添加 `try/except` 捕获异常，返回友好的错误信息
>
> **5. 缺少重复用户名检查（Bug）**
> - 位置：第5行
> - 问题：没有检查用户名是否已存在，可能插入重复数据
> - 建议：先查询用户名是否存在，再决定是否插入
>
> ### 🟢 建议
>
> **6. 硬编码数据库路径（风格）**
> - 位置：第3行 `'users.db'`
> - 问题：数据库路径写死在代码里，不方便修改
> - 建议：提取为常量或配置项

**这就是AI审查的价值**——你写了一个25行的函数，AI找出了6个问题，其中2个是严重安全漏洞。这些问题如果等到上线才发现，后果不堪设想。

## 46.5 AI审查的局限性：哪些问题AI看不出来

**AI不是万能的。** 以下三类问题，AI的审查可能"漏掉"甚至"误判"：

### 1. 业务逻辑对不对

AI不知道你的业务规则。比如"VIP用户享受8折"——这个逻辑对不对，只有你知道。AI看到 `if user_level == 'vip': price *= 0.8` 只会说"代码风格没问题"，但如果你本意是"VIP享受7折"，AI发现不了。

### 2. 安全漏洞可能看不出来（是的，这很矛盾）

AI能发现**常见的安全漏洞**（SQL注入、XSS、明文密码），但**复杂的、业务特定的安全漏洞**，AI可能看不出来。比如：

- 权限控制：AI不知道"普通用户不应该能访问管理后台"
- 敏感数据泄露：AI不知道哪些字段是"敏感的"
- 逻辑漏洞：AI不知道"这个API应该限制每分钟调用次数"

### 3. AI可能"过度审查"

AI有时候会提出一些"技术上对但实际没必要"的建议。比如建议你给一个5行的函数写单元测试——技术上没错，但投入产出比太低。

**结论**：AI审查是**初筛**，不是**终审**。AI列出问题清单，你来做最终判断。

## 46.6 人机协作：AI初筛，人做判断

| 环节 | AI做 | 人做 |
|------|------|------|
| 发现常见Bug | ✅ 几乎不漏 | - |
| 检查代码风格 | ✅ 比人严格 | 判断是否采纳 |
| 发现安全漏洞 | ✅ 常见漏洞 | 判断业务安全逻辑 |
| 发现性能问题 | ✅ 明显问题 | 判断是否真的需要优化 |
| 业务逻辑正确性 | ❌ 看不出来 | ✅ 只有你知道 |
| 最终判断 | - | ✅ 你拍板改不改 |

**把AI当作"免费的代码审查助手"**，它会帮你发现90%的问题，剩下10%还是需要你自己的判断。
