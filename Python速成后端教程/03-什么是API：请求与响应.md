## 三、分步操作

### 步骤 1：API 到底是什么？——餐厅服务员的故事

在上一章，你写了一个后端程序，浏览器访问它就能拿到数据。但你有没有想过：**浏览器和你的 Python 程序之间，到底是怎么"对话"的？**

它们说的就是 **API** 这门语言。

#### 比喻：餐厅里的服务员

你去一家餐厅吃饭。流程是这样的：

```
你（顾客）          服务员            厨房（厨师）
   │                 │                  │
   │──"来份宫保鸡丁"──→│                  │
   │                 │──"宫保鸡丁一份！"──→│
   │                 │                  │
   │                 │              （炒菜中...）
   │                 │                  │
   │                 │←──🍲 宫保鸡丁 ────│
   │←──🍲 宫保鸡丁 ────│                  │
```

你**不需要**知道厨房里有什么锅、用什么火候、厨师怎么颠勺。你只需要知道：
1. **你要什么**（宫保鸡丁）
2. **怎么点**（跟服务员说）
3. **你会得到什么**（一盘宫保鸡丁）

**API 就是这个服务员。**

- **你（顾客）= 客户端**（浏览器、手机 App、curl 命令行，任何一个发请求的程序）
- **厨房（厨师）= 服务器**（你的 `main.py`，它负责"做菜"——处理数据、查数据库、算结果）
- **服务员 = API**（Application Programming Interface，应用程序编程接口）

`API` 这个词拆开看：
- **Application（应用程序）**：你的浏览器是一个程序，你的 Python 后端是另一个程序。
- **Programming（编程）**：两个程序之间不是"随便聊聊"，而是按**固定规则**对话。
- **Interface（接口）**：一个"接触面"——你不能直接把手伸进厨房，只能通过服务员这个"接触面"来点菜。

> 📊 可视化演示见 [python_03_http_visual.html](python_03_http_visual.html)（步骤 1-2：客户端-服务器模型）

**所以，你上一章写的 `@app.get("/")` 本质上就是在说："服务员，当有人点'首页'这道菜时，把 `{"message": "Hello World"}` 端上去。"**

### 步骤 3：拆解 HTTP 请求——浏览器说了什么？

一次 HTTP 请求就像你给服务员下的一张"点菜单"。它有四个部分：

#### 3.1 URL（统一资源定位符）——你点的是哪道菜

```
http://127.0.0.1:8000/
```

拆开看：

| 部分 | 值 | 含义 |
|------|-----|------|
| 协议（scheme） | `http` | 用 HTTP 协议通信（就像餐厅用"普通话"点菜） |
| 主机（host） | `127.0.0.1` | 厨房在哪台电脑上（`127.0.0.1` = 你自己的电脑） |
| 端口（port） | `8000` | 厨房的哪个窗口（一台电脑可以同时跑很多程序，用端口号区分） |
| 路径（path） | `/` | 点哪道菜（`/` = 首页，`/users` = 用户列表，`/posts/5` = 第 5 号文章） |

**比喻总结**：`http://127.0.0.1:8000/` = "用普通话（HTTP），在我自己电脑的 8000 号窗口，点一道叫'首页'的菜。"

#### 3.2 请求方法（Request Method）——你怎么点

在 Network 面板里，你会看到 `GET` 字样。这就是**请求方法**。

常见的请求方法对应餐厅里的动作：

| 方法 | 餐厅比喻 | 含义 |
|------|----------|------|
| **GET** | "给我看看菜单上的这道菜" | 获取数据，不修改任何东西 |
| **POST** | "我要点一道新菜（菜单上没有的）" | 创建新数据 |
| **PUT** | "这道菜换一种做法" | 完整替换已有数据 |
| **DELETE** | "这道菜不要了，退掉" | 删除数据 |

> 你在浏览器地址栏输入网址回车，浏览器发的就是 **GET** 请求。这也是为什么你只能"看"网页，不能用地址栏"删"网页——地址栏只会发 GET。

#### 3.3 请求头（Request Headers）——额外的备注信息

在 Network 面板里往下翻，找到 **Request Headers** 区域。你会看到类似这些：

```
GET / HTTP/1.1
Host: 127.0.0.1:8000
Accept: text/html,application/json
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) ...
Accept-Encoding: gzip, deflate
Accept-Language: zh-CN,zh;q=0.9
```

**请求头就是点菜时附带的备注**：
- `Host`：告诉服务员"我找的是这个厨房"（一台服务器可能托管多个网站）。
- `Accept`：告诉厨房"我能接受的菜式"（浏览器说：我能吃 HTML 和 JSON）。
- `User-Agent`：告诉厨房"我是谁"（我是 Chrome 浏览器，Windows 系统）。
- `Accept-Language`：告诉厨房"我偏好什么语言"（中文优先）。

#### 3.4 请求体（Request Body）——附带的数据

**GET 请求通常没有请求体。** 就像你问服务员"今天有什么推荐菜？"——你不需要额外递一张纸过去。

POST 和 PUT 请求才会有请求体，里面装着你要提交的数据。我们会在第 06 章详细讲。

### 步骤 4：拆解 HTTP 响应——服务器回了什么？

浏览器发完请求后，服务器（你的 `main.py`）处理完，回了一个**响应**。响应有三个部分：

#### 4.1 状态码（Status Code）——"您的菜做好了"还是"厨房没这道菜"

在 Network 面板里，最显眼的就是那个 **200**。这就是状态码——服务器用三个数字告诉你"结果如何"。

| 状态码 | 含义 | 餐厅比喻 |
|--------|------|----------|
| **200** | OK，成功 | "您的菜好了，请慢用！" |
| **201** | Created，创建成功 | "您点的定制菜已经做好了！" |
| **400** | Bad Request，请求有误 | "您点的'红烧橡皮擦'我们做不了" |
| **404** | Not Found，找不到 | "抱歉，菜单上没有这道菜" |
| **500** | Internal Server Error，服务器内部错误 | "厨房着火了！" |

> 状态码的第一位数字代表大类：
> - `2xx` = 成功（一切正常）
> - `3xx` = 重定向（"这道菜搬到另一桌了，请移步"）
> - `4xx` = 客户端错误（你点菜的方式有问题）
> - `5xx` = 服务器错误（厨房出问题了，跟你没关系）

#### 4.2 响应头（Response Headers）——附带的说明

在 Network 面板里找到 **Response Headers**：

```
HTTP/1.1 200 OK
content-type: application/json
content-length: 27
server: uvicorn
```

- `content-type: application/json`：告诉浏览器"我返回的是 JSON 格式的数据，请按 JSON 来解析"。
- `content-length: 27`：返回的数据有 27 个字节（`{"message":"Hello World"}` 的长度）。
- `server: uvicorn`：告诉浏览器"我是用 uvicorn 这个服务器软件在处理你的请求"。

#### 4.3 响应体（Response Body）——菜本身

在 Network 面板里点击 **Response** 或 **Preview** 标签，你会看到：

```json
{"message": "Hello World"}
```

这就是响应体——服务器真正返回给你的**数据本身**。前面的状态码、响应头都是"包装"，响应体才是你点的那盘菜。

---

### 🤔 想多一点：为什么需要状态码这套设计？

**如果没有状态码会怎样？**

想象你去餐厅，服务员什么也不说，直接端上来一盘菜。或者更糟——服务员什么也不说，什么也不端，就站在那看着你。你完全不知道：
- 菜做好了没？
- 是不是没这道菜？
- 还是厨房着火了，厨师跑了？

**你现在遇到的每一行报错信息，本质上都是"状态码"的思想在起作用。** Python 里 `FileNotFoundError`、`KeyError`、`TypeError`——这些错误类型和 HTTP 状态码一样，都是在用分类的方式告诉你"出了什么问题"，让你能快速定位原因。

**状态码体系的核心价值**：让程序（而不只是人）能自动判断下一步该做什么。浏览器看到 301 会自动跳转，看到 404 会显示"找不到页面"，看到 500 会显示"服务器错误"。如果所有错误都返回 200 但内容是"呃，出错了"，那程序就没办法自动处理了——它得先读懂人类语言才知道发生了什么。

**这就是 API 设计的核心原则之一：让机器能读懂机器。**

### 步骤 5：用 curl 发请求——不用浏览器，也能跟服务器对话

浏览器是"能看到网页的客人"，但有时候我们只需要"跟服务员说句话"，不需要打开整个餐厅大门。这时候用 **curl**。

> **curl 是什么？** `curl` = "Client URL"，一个命令行工具，专门用来发 HTTP 请求。你可以把它想象成"纯文本版的浏览器"——它不渲染网页，只把服务器返回的原始数据打印出来。

#### 5.1 检查你有没有 curl

打开终端，输入：

```bash
curl --version
```

如果输出类似 `curl 8.x.x ...`，说明你有 curl，跳到 5.3。

如果提示 `'curl' 不是内部或外部命令`，看 5.2。

#### 5.2 Windows 没有 curl 怎么办？三种方案

**方案一（推荐）：用 PowerShell 的 `Invoke-WebRequest`**

Windows PowerShell 自带一个功能类似的命令：

```powershell
Invoke-WebRequest -Uri http://127.0.0.1:8000
```

输出会很长，但你能看到 `StatusCode: 200` 和 `Content: {"message":"Hello World"}`。

> 为了方便后续教程，建议安装 curl。方案二和三任选其一。

**方案二：安装 curl（Windows 10 1803+ 已内置）**

Windows 10 2018 年 4 月更新之后的版本，curl 已经内置了。在 PowerShell 里试试：

```powershell
curl.exe --version
```

如果还是找不到，去 https://curl.se/windows/ 下载，解压后把 `curl.exe` 所在目录加入系统 PATH。

**方案三：用 Git Bash**

如果你装了 Git，Git Bash 自带 curl。打开 Git Bash，直接输入 `curl --version` 就能用。

> 本章后续示例以 `curl` 为准。如果你用 PowerShell 的 `Invoke-WebRequest`，请自行替换对应命令。核心观察点（状态码、响应体）是一样的。

#### 5.3 用 curl 发第一个请求

确保服务器在跑，然后新开一个终端窗口，输入：

```bash
curl http://127.0.0.1:8000
```

输出：

```
{"message":"Hello World"}
```

就这么简单。`curl` 默认发 GET 请求，把响应体打印出来。

#### 5.4 加上 `-v`，看完整过程

现在加上 `-v`（verbose，啰嗦模式），让 curl 把请求和响应的**所有细节**都打印出来：

```bash
curl -v http://127.0.0.1:8000
```

输出会很长。我们**逐行拆解**：

```
*   Trying 127.0.0.1:8000...
```
curl 在尝试连接你电脑的 8000 端口。`*` 开头的是 curl 自己的调试信息。

```
* Connected to 127.0.0.1 (127.0.0.1) port 8000
```
连接成功！curl 已经跟你的服务器"握上手"了。

```
> GET / HTTP/1.1
> Host: 127.0.0.1:8000
> User-Agent: curl/8.x.x
> Accept: */*
>
```
`>` 开头的，是 curl **发出去**的请求。这就是我们步骤 3 讲的请求内容：
- 第一行：`GET / HTTP/1.1`——用 GET 方法，请求路径 `/`，用 HTTP/1.1 协议。
- 后面几行是请求头：`Host`、`User-Agent`、`Accept`。

```
< HTTP/1.1 200 OK
< date: Sun, 01 Jan 2026 12:00:00 GMT
< server: uvicorn
< content-type: application/json
< content-length: 27
<
```
`<` 开头的，是服务器**返回**的响应。这就是我们步骤 4 讲的响应内容：
- 第一行：`HTTP/1.1 200 OK`——状态码 200，成功。
- 后面几行是响应头：`server: uvicorn`、`content-type: application/json` 等。
- 最后一个空行 `<` 表示"响应头结束，下面是响应体"。

```
{"message":"Hello World"}
```
响应体——你的那盘菜。

```
* Connection #0 to host 127.0.0.1 left intact
```
连接关闭，curl 完成任务。

> **如果 `curl -v` 输出太多看不懂**：记住这个口诀——`>` 是"我说的"，`<` 是"它回的"。`*` 是 curl 自己的碎碎念，可以忽略。

### 步骤 6：常见状态码速查——一张表搞定

前面你已经见过 200 了。现在一次性把最常见的 5 个状态码认全：

| 状态码 | 英文名 | 含义 | 餐厅比喻 | 你会遇到的场景 |
|--------|--------|------|----------|----------------|
| **200** | OK | 请求成功 | "您的菜好了！" | 正常访问任何页面 |
| **201** | Created | 创建成功 | "您定制的新菜已加入菜单！" | 用 POST 创建了一篇文章 |
| **400** | Bad Request | 请求有误 | "您点的'红烧橡皮擦'我们做不了" | 你传了错误格式的数据 |
| **404** | Not Found | 资源不存在 | "抱歉，菜单上没有这道菜" | 访问了一个不存在的路径 |
| **500** | Internal Server Error | 服务器内部错误 | "厨房着火了！" | 你的代码里有 bug，崩溃了 |

> **记忆技巧**：`2` 开头的你开心，`4` 开头的你错了，`5` 开头的服务器错了。

### 步骤 7：动手改代码——加一个返回 404 的接口

知道 404 是什么意思了，我们来亲手制造一个 404 响应。

#### 7.1 修改 `main.py`

打开 `blog_backend/main.py`，在现有代码后面追加以下内容：

```python
from fastapi import FastAPI
from fastapi.responses import JSONResponse

app = FastAPI()


@app.get("/")
def read_root():
    return {"message": "Hello World"}


# === 新增：一个总是返回 404 的接口 ===
@app.get("/secret")
def secret_page():
    return JSONResponse(
        status_code=404,
        content={"error": "你找的秘密页面不存在！", "hint": "试试别的路径，比如 /"}
    )
```

保存文件（`Ctrl+S`）。因为开启了 `--reload`，服务器会自动重启。

#### 7.2 用浏览器观察

在浏览器访问 `http://127.0.0.1:8000/secret`：

你会看到浏览器返回了 JSON 数据 `{"error":"你找的秘密页面不存在！","hint":"试试别的路径，比如 /"}`。

但等等——浏览器可能没有显示明显的"404 错误页面"。这是因为浏览器**确实收到了数据**（响应体），只是状态码是 404。有些浏览器对 JSON 格式的 404 响应不会显示错误页面，而是直接展示 JSON 内容。

**用 F12 验证**：按 F12 → Network 标签 → 刷新 `http://127.0.0.1:8000/secret` → 点击这条记录 → 你会看到 **Status: 404**。状态码确认了这是 404，不管浏览器怎么显示。

#### 7.3 用 curl 观察

```bash
curl -v http://127.0.0.1:8000/secret
```

输出：

```
< HTTP/1.1 404 Not Found
< content-type: application/json
<
{"error":"你找的秘密页面不存在！","hint":"试试别的路径，比如 /"}
```

注意第一行：`404 Not Found`。curl 明明白白告诉你这是 404，不管响应体里写了什么。

#### 7.4 对比：正常路径 vs 不存在的路径

```bash
# 正常路径 — 200
curl -v http://127.0.0.1:8000/

# 你手动返回 404 的路径 — 404
curl -v http://127.0.0.1:8000/secret

# FastAPI 自动返回 404 的路径（你没定义 /hello） — 404
curl -v http://127.0.0.1:8000/hello
```

三条命令，三种情况：
- `/`：你在 `main.py` 里定义了，返回 200。
- `/secret`：你在 `main.py` 里定义了，但**手动返回了 404**。说明状态码是你可以控制的——你想让它返回什么状态码就返回什么。
- `/hello`：你**没定义**这个路径，FastAPI 自动帮你返回了 404。这说明 FastAPI 自带"不存在的路径自动 404"的功能。

### 🤔 想多一点：为什么浏览器看到 404 可能不会"崩溃"？

你可能注意到，访问 `/secret` 时，浏览器平静地显示了 JSON，并没有像访问某些网站时那样跳出一个大大的"404 页面不存在"错误页。

**原因是**：浏览器对 HTML 页面和 JSON 数据的处理方式不同。
- 如果服务器返回的是 `text/html`（网页），浏览器会渲染它，大多数 Web 框架对 404 自动返回一个友好的 HTML 错误页。
- 如果服务器返回的是 `application/json`（API 数据），浏览器只负责显示数据本身，把"是不是错误"的判断留给你（调用方）。

**API 的设计哲学**：API 返回的是**数据**，不是**页面**。一个 API 调用者（比如手机 App）收到 404 后，它应该自己决定怎么处理——是弹窗提示用户，还是静默跳转，还是记录日志。你的 API 只需要诚实地告诉它"状态码是 404"就够了。

## 六、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| curl 不存在 | `'curl' 不是内部或外部命令` | Windows 未安装 curl | 用 PowerShell 的 `Invoke-WebRequest`，或安装 curl（见步骤 5.2） |
| F12 Network 空白 | 打开 Network 标签后什么都看不到 | 在打开 F12 **之前**发起的请求不会被记录 | 打开 F12 后再刷新页面 |
| `curl -v` 输出看不懂 | 一大堆 `*` `>` `<` 符号 | 不熟悉 curl 的 verbose 格式 | 记住口诀：`>` 是"我说的"，`<` 是"它回的"，`*` 是 curl 的碎碎念 |
| 返回 404 但浏览器没报错 | 浏览器正常显示了 JSON | 浏览器对 JSON 的 404 不显示错误页 | 看 F12 Network 里的 Status 列确认状态码 |
| 改了代码没生效 | 访问 `/secret` 返回 404（FastAPI 自动的，不是你的 JSON 内容） | 服务器没重启，或 `--reload` 没开 | 确认终端里看到 `WatchFiles detected changes`，或手动 `Ctrl+C` 重启 |

## 七、小结

| 你学到了什么 | 一句话总结 |
|--------------|-----------|
| API 是什么 | 客户端（浏览器）和服务器（你的 Python 程序）之间的"服务员"——传话、端菜、反馈结果 |
| HTTP 请求的四个部分 | URL（点哪道菜）+ 方法（怎么点）+ 头部（备注）+ 体（附带数据） |
| HTTP 响应的三个部分 | 状态码（做没做成）+ 响应头（额外说明）+ 响应体（菜本身） |
| 常见状态码 | 200 成功 / 201 创建成功 / 400 你错了 / 404 找不到 / 500 服务器崩了 |
| 浏览器 F12 Network | 能看到每一次请求的完整"解剖图"——请求头、响应头、状态码、响应体 |
| curl 命令 | 不用浏览器也能跟服务器对话，`-v` 查看完整过程 |
| 手动返回 404 | 用 `JSONResponse(status_code=404, ...)` 来精确控制接口返回的状态码 |
