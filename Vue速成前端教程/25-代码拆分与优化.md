## 一、目标与完成效果

**一句话目标**：用路由懒加载、异步组件和图片懒加载把博客首页的加载速度提升一倍以上。

**完成后你将看到**：
- 首次打开博客时，浏览器只下载当前页面的代码（而不是整个网站的所有页面）。
- 图片滚动到可视区域时才加载，首屏加载的图片数量大幅减少。
- 用 Lighthouse 跑分，Performance 分数比优化前明显提升。

## 三、分步操作

### 步骤 1：路由懒加载 — 按需加载页面

**我在做什么？**
目前的 `router/index.js` 大概是这样的：

```js
import Home from '@/views/Home.vue'
import Login from '@/views/Login.vue'
import Register from '@/views/Register.vue'
import CreatePost from '@/views/CreatePost.vue'
import PostDetail from '@/views/PostDetail.vue'

const routes = [
  { path: '/', component: Home },
  { path: '/login', component: Login },
  { path: '/register', component: Register },
  { path: '/create', component: CreatePost },
  { path: '/post/:id', component: PostDetail },
]
```

**问题**：所有 `import` 都在文件顶部，意味着浏览器加载 `router/index.js` 时，会**一次性把所有页面组件都下载下来**——哪怕用户今天只看首页，登录页、注册页的代码也照下不误。

**解决方案**：路由懒加载。把 `import` 从顶部移到 `component` 的定义处，用动态导入语法：

```js
const routes = [
  {
    path: '/',
    component: () => import('@/views/Home.vue')       // 🎯 懒加载
  },
  {
    path: '/login',
    component: () => import('@/views/Login.vue')
  },
  {
    path: '/register',
    component: () => import('@/views/Register.vue')
  },
  {
    path: '/create',
    component: () => import('@/views/CreatePost.vue')
  },
  {
    path: '/post/:id',
    component: () => import('@/views/PostDetail.vue')
  }
]
```

**发生了什么？**
- `() => import('@/views/Home.vue')` 返回一个**函数**，而不是直接导入组件。
- Vue Router 会在用户**第一次访问该路由时**才调用这个函数，下载对应的 JS 文件。
- 每个页面会被打包成独立的 `.js` 文件（如 `Home-abc123.js`），按需加载。

**比喻**：这就好比你去餐厅，以前是服务员不管你点什么，先把所有菜都端上来。现在是你点什么，厨房才做什么。

❌ **旧写法**：`import Home from '@/views/Home.vue'`（顶部静态导入）
✅ **新写法**：`component: () => import('@/views/Home.vue')`（路由懒加载）

**我做得对不对？**
1. 清空浏览器缓存，打开 DevTools Network 面板。
2. 访问首页 → 看 JS 文件列表，应该只有 `index.js` + `Home-xxx.js`，没有 `Login-xxx.js`。
3. 点击"登录"链接 → Network 面板新增一个 `Login-xxx.js` 文件。

**不对怎么办？**
- 所有页面仍然打包在一起 → 检查 `import()` 语法是否正确，注意是 `() => import(...)` 不是 `import(...)`。
- 报错 "Failed to fetch dynamically imported module" → 检查文件路径是否正确，`@` 别名是否在 `vite.config.js` 中配置。

### 步骤 2：异步组件 — 非路由场景的懒加载

**我在做什么？**
路由懒加载解决了"页面级"的拆分。但有些场景不是路由——比如一个很重的 Markdown 编辑器组件，只在"写文章"页面里用，但你不希望它拖慢"写文章"页面本身的首次渲染。

**用 `defineAsyncComponent`**：

```html
<script setup>
import { defineAsyncComponent } from 'vue'

// 🎯 异步加载：只在真正需要渲染时才下载
const MarkdownEditor = defineAsyncComponent(() =>
  import('@/components/MarkdownEditor.vue')
)

// 带 loading 状态的异步组件
const MarkdownEditorWithLoading = defineAsyncComponent({
  loader: () => import('@/components/MarkdownEditor.vue'),
  loadingComponent: () => import('@/components/LoadingSpinner.vue'),  // 加载中显示
  delay: 200,         // 200ms 后才显示 loading（避免闪烁）
  timeout: 10000      // 10 秒超时
})
</script>

<template>
  <!-- 直接使用，和普通组件别无二致 -->
  <MarkdownEditor v-model="content" />
</template>
```

**何时用异步组件？**
- 组件体积大（如富文本编辑器、图表库、代码高亮）。
- 组件不是首屏必需的（如模态框、设置面板）。
- 组件在特定条件下才渲染（如 `v-if="showAdvanced"`）。

🤔 **想多一点**：`defineAsyncComponent` 和路由懒加载本质上是同一件事——都是 `() => import()` 的封装。区别在于路由懒加载由 Vue Router 管理时机（"用户访问这个路径时"），异步组件由你自己管理时机（"v-if 变成 true 时"或"组件挂载时"）。

### 步骤 3：图片懒加载 — 看到才加载

**我在做什么？**
博客首页可能有 30 篇文章，每篇都有一张封面图。如果浏览器一口气加载 30 张图，用户可能只看前 3 篇就关掉了，后面 27 张图白下了。图片懒加载就是"图片滚到屏幕里再开始下载"。

**HTML 原生方式（最简单）**：

```html
<!-- 只在图片即将进入视口时才加载 -->
<img src="placeholder.jpg" data-src="real-image.jpg" loading="lazy" />
```

`loading="lazy"` 是 HTML 标准属性，浏览器原生支持，**不需要任何 JavaScript**。但注意：`loading="lazy"` 只对 `<img>` 的 `src` 属性生效。

**如果你想更精细的控制**（比如先显示模糊占位图，加载完成后渐变清晰），可以用 Intersection Observer：

```js
// src/composables/useLazyLoad.js
export function useLazyLoad() {
  function observeImages() {
    const images = document.querySelectorAll('img[data-src]')
    const observer = new IntersectionObserver((entries) => {
      entries.forEach(entry => {
        if (entry.isIntersecting) {
          const img = entry.target
          img.src = img.dataset.src         // 替换为真实图片
          img.onload = () => img.classList.add('loaded')
          observer.unobserve(img)           // 加载完就停止观察
        }
      })
    }, {
      rootMargin: '100px'  // 提前 100px 开始加载（用户快滚到时就触发）
    })

    images.forEach(img => observer.observe(img))
  }

  return { observeImages }
}
```

**在组件中使用**：

```html
<script setup>
import { onMounted } from 'vue'
import { useLazyLoad } from '@/composables/useLazyLoad'

const { observeImages } = useLazyLoad()
onMounted(() => observeImages())
</script>

<template>
  <img
    data-src="https://example.com/cover.jpg"
    src="data:image/svg+xml,..."  <!-- 占位图（超小 SVG） -->
    loading="lazy"
    class="lazy-img"
  />
</template>

<style scoped>
.lazy-img {
  opacity: 0;
  transition: opacity 0.3s;
}
.lazy-img.loaded {
  opacity: 1;
}
</style>
```

❌ **错误示例**：所有图片用 `src` 直接加载，首页 30 张图一起下。
✅ **正确示例**：用 `loading="lazy"` 或 Intersection Observer，只加载可见区域的图片。

### 步骤 4：Lighthouse 性能分析

**我在做什么？**
优化完了，用数据说话。Lighthouse 是 Chrome 内置的性能审计工具，会给你的网站打分（0-100），并给出优化建议。

**操作步骤**：
1. 在 Chrome 中打开你的博客（`npm run dev` 后访问）。
2. 按 F12 → 切换到 **Lighthouse** 标签（如果没有，点击 `>>` 在更多工具里找）。
3. Mode 选 **Navigation**，Device 选 **Mobile**，Categories 勾选 **Performance**。
4. 点击 **Analyze page load**。
5. 等待 30 秒左右，得到一份报告。

**重点看三个指标**：

| 指标 | 含义 | 好（绿色）| 需优化（黄色）| 差（红色）|
|------|------|:---:|:---:|:---:|
| **FCP** (First Contentful Paint) | 用户看到第一个内容的时间 | < 1.8s | 1.8-3s | > 3s |
| **LCP** (Largest Contentful Paint) | 最大内容（通常是首屏大图）渲染完成 | < 2.5s | 2.5-4s | > 4s |
| **TBT** (Total Blocking Time) | 主线程被阻塞的总时间 | < 200ms | 200-600ms | > 600ms |

**优化前后对比**（预期）：

| 指标 | 优化前 | 优化后（懒加载+拆分） |
|------|--------|----------------------|
| Performance 总分 | 60-70 | 85-95 |
| 首次 JS 下载量 | 全量（~500KB） | 按需（~150KB） |
| 首屏图片加载 | 全部加载 | 仅可见区域 |

🤔 **想多一点**：Lighthouse 的分数不是绝对的。在本地开发环境（`npm run dev`）跑分会比生产环境（`npm run build` 后）低很多，因为开发模式下代码没有压缩、没有 tree-shaking，还夹带了 HMR（热更新）的 WebSocket 代码。真正有参考价值的分数应该在 `npm run preview` 下测。

### 步骤 5：小结 — 优化是工程师的修养

优化这件事，不是"做完功能再说"的可选项，而是工程师的修养。就像你写完一篇文章会检查错别字一样，写完代码应该检查性能。

**优化清单**：

| 优化项 | 做了什么 | 收益 |
|--------|----------|------|
| 路由懒加载 | `() => import(...)` | 首屏 JS 体积减少 60-80% |
| 异步组件 | `defineAsyncComponent` | 非首屏组件延迟加载 |
| 图片懒加载 | `loading="lazy"` | 首屏图片请求减少 70-90% |
| Lighthouse 审计 | 跑分 + 按建议优化 | 有数据、有方向 |

**优化不是一次性的事**。每次加新功能、新页面、新依赖，都应该重新跑一次 Lighthouse，看看有没有拖慢速度。

## 七、术语附录

| 术语 | 解释 |
|------|------|
| **懒加载 (Lazy Loading)** | 延迟加载非关键资源（JS、图片等），直到用户真正需要它们时才下载。减少首屏加载时间，节省带宽。 |
| **异步组件 (Async Component)** | 在 Vue 中通过 `defineAsyncComponent` 定义的组件，其代码在需要渲染时才动态下载。适合体积大或不常使用的组件。 |
| **Lighthouse** | Google 开发的开源网站性能审计工具，内置于 Chrome DevTools。从性能、可访问性、最佳实践、SEO 四个维度打分。 |
| **Intersection Observer** | 浏览器 API，用于监测元素是否进入视口（可视区域）。常用于图片懒加载、无限滚动、曝光埋点等场景。 |
| **Code Splitting (代码拆分)** | 将应用代码拆分为多个较小的文件（chunk），按需加载，而不是一次性下载全部代码。Vite/Webpack 在构建时自动完成。 |

## 八、已知坑点与禁止事项

- **禁止所有路由都用懒加载后又全量预加载**：`() => import(...)` 后不要在路由守卫里手动 `import()` 所有页面，那就白拆了。
- **`loading="lazy"` 不要用在首屏大图（LCP 元素）上**：首屏最关键的那张图应该立即加载，延迟加载反而拖慢 LCP 分数。
- **异步组件不要包在 `v-if` 里又套 `v-if`**：`defineAsyncComponent` 本身已经处理了加载时机，外层再包 `v-if="false"` 会导致组件永远不加载。
- **Lighthouse 在开发模式下跑分无意义**：必须在 `npm run build` + `npm run preview` 的生产模式下跑。
