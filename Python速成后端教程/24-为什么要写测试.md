## 三、分步操作

### 步骤 1：痛点回顾——你现在是怎么测试的？

回想一下，从教程 07 到现在，你每次写完或改完一个接口，都是怎么验证的？

**你的日常测试流程**：

1. 改了几行代码。
2. 切到终端，手敲 `curl -X POST http://127.0.0.1:8000/login -H "Content-Type: application/json" -d "{\"username\": \"alice\", \"password\": \"alice123\"}"`。
3. 看返回结果，脑子判断"嗯，返回了 token，没问题"。
4. 再测试下一个接口：`curl -X POST http://127.0.0.1:8000/posts -H "Authorization: Bearer <刚才的token>" -H "Content-Type: application/json" -d "{\"title\": \"测试\", \"content\": \"测试内容\"}"`。
5. 看返回结果，再判断"嗯，返回了文章，没问题"。
6. 再测试删除……分页……搜索……
7. 5 个接口要手动 curl 5 次，每次还要复制粘贴 token，复制粘贴 JSON。

**这有什么问题？**

| 问题 | 表现 |
|------|------|
| **烦** | 每次改代码都要手动 curl 一遍，手指头酸痛 |
| **慢** | 5 个接口，curl + 肉眼看结果，至少 2 分钟 |
| **容易漏** | 你只测了"改动的那个接口"，其他接口有没有被搞坏？不知道 |
| **不可重复** | 下周一你改了代码，能记得上周测了哪些边界情况吗？不能 |
| **别人没法测** | 同事接手你的代码，他怎么知道"改完要测哪些接口"？ |

**比喻**：你现在就像一个"手工质检员"——每次工厂（代码）改了一条生产线，你都要亲自去每条生产线转一圈，看看有没有出问题。今天是 5 个接口，你能手动测。明天是 50 个接口呢？500 个呢？

### 步骤 2：测试的本质——用代码验证代码

**测试的本质就是一句话**：写一段代码，用这段代码去验证另一段代码对不对。

你不需要手动 curl，不需要肉眼判断，不需要记住"上次测了哪些"。你只需要：

```bash
pytest -v
```

一条命令，跑完所有检查。全绿 → 放心了。有红 → 定位问题，修。

**比喻**：测试就像给代码装了一个"自动体检仪"。以前是每次自己量体温、测血压、验血（手动 curl），现在是往机器上一站，3 秒出报告（pytest）。机器不会忘，不会累，不会漏。

**再比喻**：测试是"给自己买保险"。你改代码时，可能不小心搞坏了另一个功能——就像装修客厅，结果把卧室的墙打穿了。测试就是你的"墙体检员"——改完代码跑一遍，他告诉你："客厅装修完毕，卧室墙完好，阳台墙完好，厨房墙完好。"

### 步骤 3：测试金字塔——不是所有测试都一样

在写测试之前，先了解一个概念：**测试金字塔**。此术语需进附录。

```
        /\
       /E2E\          ← 少（端到端：模拟真实用户操作）
      /------\
     / 集成测试 \      ← 中（多个模块一起测）
    /----------\
   /   单元测试   \    ← 多（单个函数/类独立测）
  /--------------\
```

**三层测试的含义**：

| 层级 | 测什么 | 例子 | 速度 | 数量 |
|------|--------|------|------|------|
| **单元测试** | 单个函数/方法的逻辑 | 测试 `hash_password("123")` 的返回值长度 | 毫秒级 | 最多 |
| **集成测试** | 多个模块协作 | 测试 `POST /posts` 接口 → 数据库有记录 → 返回 201 | 秒级 | 中等 |
| **E2E 测试** | 完整用户操作流程 | 模拟浏览器：打开页面 → 登录 → 写文章 → 发布 | 分钟级 | 最少 |

**本教程写的是哪一层？** 主要是**集成测试**——用 httpx 发 HTTP 请求到 FastAPI 应用，验证整个请求-响应链路。这比单元测试覆盖更全面，又比 E2E 测试快得多。

**比喻**：测试金字塔就像一个体检套餐：
- 单元测试 = 每天量体温（快、频繁、基础）
- 集成测试 = 每月体检（查几个关键指标）
- E2E 测试 = 每年全身大体检（慢、全面、昂贵）

你不会每天都做全身大体检，但每天量体温能快速发现异常。

### 步骤 4：安装依赖——pytest 和 httpx

我们需要两个工具：

- **pytest**：Python 最流行的测试框架。此术语需进附录。它能自动发现 `test_*.py` 文件，运行里面的 `test_*` 函数，告诉你哪些通过、哪些失败。
- **httpx**：一个支持异步的 HTTP 客户端库。此术语需进附录。它在测试中的作用是"假装自己是浏览器"——向我们的 FastAPI 应用发 HTTP 请求，拿到响应，让我们检查响应对不对。

```powershell
pip install pytest httpx
```

**验证安装**：

```powershell
pytest --version
```

```powershell
python -c "import httpx; print('httpx 安装成功')"
```

**为什么用 httpx 而不是 requests？** 因为 FastAPI 是异步框架，`httpx` 原生支持异步，能和 FastAPI 的 `TestClient` 无缝配合。`requests` 是同步库，在 FastAPI 测试中会有兼容问题。

**比喻**：pytest 是"考试系统"——负责出题、收卷、打分。httpx 是"答题笔"——负责在试卷上写字（发 HTTP 请求）。两者配合，自动化考试一条龙。

### 步骤 5：写第一个测试——`test_create_post`

现在动手写第一个测试。新建文件 `blog_backend/test_main.py`：

```python
# blog_backend/test_main.py —— 第一个测试：创建文章

from fastapi.testclient import TestClient
from blog_backend.main import app

# 创建一个测试客户端——它"假装"是浏览器，向我们的应用发请求
client = TestClient(app)


def test_create_post():
    """
    测试：创建一篇新文章
    预期：返回 201 状态码，响应 JSON 中包含文章标题
    """
    # 发送 POST 请求到 /posts
    response = client.post(
        "/posts",
        json={
            "title": "测试文章",
            "content": "这是一篇测试文章的内容。",
            "published": True,
        },
    )

    # 断言：状态码应该是 201（创建成功）
    assert response.status_code == 201

    # 断言：返回的 JSON 中 title 应该等于我们发送的值
    data = response.json()
    assert data["title"] == "测试文章"
    assert data["content"] == "这是一篇测试文章的内容。"
```

**逐行解释**：

- `from fastapi.testclient import TestClient`：FastAPI 自带的测试客户端。此术语需进附录：TestClient。它不需要你真的启动服务器（不需要 `uvicorn`），直接在内存中模拟 HTTP 请求。
- `client = TestClient(app)`：创建一个测试客户端，绑定到我们的 FastAPI 应用 `app`。
- `response = client.post("/posts", json={...})`：向 `/posts` 发 POST 请求，请求体是 JSON。`client.post()` 返回一个响应对象。
- `assert response.status_code == 201`：**断言**。此术语需进附录。意思是"我断言这件事一定为真，如果为假，测试失败"。`assert` 是 Python 关键字，后面跟一个布尔表达式。如果表达式为 `True`，测试通过；如果为 `False`，测试失败。
- `data = response.json()`：把响应体的 JSON 解析成 Python 字典。
- `assert data["title"] == "测试文章"`：断言返回的文章标题和我们发过去的一致。

**比喻**：`TestClient` 就像一个"内部测试员"——他不需要真的从外面打电话（HTTP 请求），而是直接走到服务器内部问："嘿，如果现在有人发一个 POST /posts，你会怎么回答？"服务器回答后，测试员检查答案对不对。

❌ 常见错误 → ✅ 正确示例：

❌ 错误：测试时启动 uvicorn 服务器

```python
# 错误：不需要启动服务器！
# 你不需要在另一个终端运行 uvicorn
# TestClient 在内存中直接与 app 交互
```

✅ 正确：直接用 TestClient，不需要启动服务器

```python
from fastapi.testclient import TestClient
from blog_backend.main import app

client = TestClient(app)  # 直接绑定 app，不需要服务器
```

### 步骤 7：测试失败的样子——故意让断言失败

你可能会想："测试通过了当然好，但如果失败了，输出长什么样？我能看懂吗？"

**只有见过失败，你才知道怎么排查。** 现在故意把断言写错，看看 pytest 的错误输出。

在 `test_create_post` 里，把 `assert response.status_code == 201` 改成 `assert response.status_code == 200`（故意写错）：

```python
def test_create_post():
    response = client.post(
        "/posts",
        json={
            "title": "测试文章",
            "content": "这是一篇测试文章的内容。",
            "published": True,
        },
    )

    # 故意写错：创建成功通常返回 201，不是 200
    assert response.status_code == 200  # ❌ 这里会失败！
```

再次运行：

```bash
pytest -v
```

**预期输出**：

```
============================= test session starts =============================
platform win32 -- Python 3.10.x, pytest-7.x.x, pluggy-1.x.x
rootdir: D:\Project\...
collected 1 item

blog_backend\test_main.py::test_create_post FAILED                        [100%]

================================== FAILURES ===================================
_____________________________ test_create_post _______________________________

    def test_create_post():
        response = client.post(
            "/posts",
            json={...},
        )
>       assert response.status_code == 200
E       assert 201 == 200
E        +  where 201 = <Response [201 Created]>.status_code

blog_backend\test_main.py:18: AssertionError
=========================== short test summary info ===========================
FAILED blog_backend/test_main.py::test_create_post - assert 201 == 200
============================== 1 failed in 0.xxs ==============================
```

**逐行解读错误输出**：

- `>       assert response.status_code == 200`：`>` 箭头指向**失败的那一行代码**。
- `E       assert 201 == 200`：pytest 告诉你——实际值是 201，你期望是 200，不匹配。
- `E        +  where 201 = <Response [201 Created]>.status_code`：pytest 进一步解释——这个 201 是从 `response.status_code` 来的，而 `response` 是一个 `201 Created` 响应。
- `FAILED blog_backend/test_main.py::test_create_post - assert 201 == 200`：最下面的总结——哪个文件、哪个函数、失败原因。

**你能从这段输出中定位问题吗？** 能。pytest 告诉了你：
1. 哪个测试失败了（`test_create_post`）
2. 哪一行失败了（`test_main.py:18`）
3. 实际值是什么（201）
4. 期望值是什么（200）
5. 实际值是从哪来的（`response.status_code`）

**把断言改回正确的**：`assert response.status_code == 201`，再次运行 `pytest -v`，应该又绿了。

**比喻**：pytest 的失败输出就像医生开的化验单——不仅告诉你"有问题"，还告诉你"哪个指标（哪一行）""实际值是多少（201）""正常范围是多少（200）"。你不需猜，不需翻日志，错误信息直接指出问题所在。

### 步骤 8：测试命名规范——`test_<功能>_<场景>`

写完第一个测试后，我们要建立命名规范，让测试文件一目了然。

**命名规范**：`test_<功能>_<场景>`

| 命名 | 含义 |
|------|------|
| `test_create_post` | 测试"创建文章"这个功能 |
| `test_create_post_without_auth` | 测试"创建文章"→"不带认证"这个场景 |
| `test_register_user` | 测试"注册用户"这个功能 |
| `test_register_duplicate_user` | 测试"注册用户"→"重复注册"这个场景 |
| `test_login_success` | 测试"登录"→"成功"这个场景 |
| `test_login_wrong_password` | 测试"登录"→"密码错误"这个场景 |

**规则**：
1. 函数名必须以 `test_` 开头（pytest 的发现规则）。
2. 用下划线分隔单词，全小写（Python 命名惯例）。
3. 名字要能看出"测什么功能"和"什么场景"——不要叫 `test_1`、`test_a`。

**为什么命名很重要？** 当 `pytest -v` 输出 `FAILED blog_backend/test_main.py::test_login_wrong_password` 时，你不需要打开文件就知道是"登录 + 密码错误"这个场景出了问题。

### 步骤 9：🤔 想多一点——写测试是不是浪费时间？

你可能会想："我写这个测试函数花了 10 分钟，我自己手动 curl 一下只要 10 秒。写测试不是更慢吗？"

**短期看，手动的确更快。但长期看，测试能帮你省下大量时间。**

| 场景 | 手动 curl | 有测试 |
|------|-----------|--------|
| 第 1 次验证 | 10 秒 | 10 分钟（写测试） |
| 第 2 次验证（改了别的代码） | 10 秒 × 5 接口 = 50 秒 | 2 秒（`pytest -v`） |
| 第 10 次验证 | 50 秒，且可能漏测 | 2 秒，100% 覆盖 |
| 第 100 次验证 | 50 秒，消磨耐心，开始跳过 | 2 秒，从不跳过 |
| 凌晨 2 点改了代码 | 困，可能漏测，上线出 bug | `pytest -v`，全绿，安心睡觉 |
| 同事改了你的代码 | 他不知道要测哪些接口 | `pytest -v`，自动跑了你写的所有测试 |

**关键洞察**：写测试是一次性的时间投资，但它能在未来**每次代码变更时**自动保护你。你写了一个测试，它会帮你在未来 100 次、1000 次代码变更中把关。

**比喻**：写测试就像买保险——你每年交保费（写测试花时间），但出事时（改代码引入 bug）保险公司赔你（pytest 第一时间发现）。不买保险的人，出事了自己扛（半夜爬起来修线上 bug）。

**再比喻**：写测试就像给代码装"刹车"——没有刹车的车也能开，但你敢开快吗？有了刹车，你才敢大胆改代码，因为你知道：改坏了，刹车（测试）会告诉你。

## 六、术语附录

| 术语 | 英文 | 通俗解释 | 本章出现位置 |
|------|------|----------|-------------|
| 测试金字塔 | Test Pyramid | 测试分层模型：底层单元测试（多、快），中层集成测试（中），顶层 E2E 测试（少、慢）。指导我们写测试时的比例分配。 | 步骤 3 |
| 单元测试 | Unit Test | 测试单个函数/方法的逻辑，不涉及外部依赖（数据库、网络等）。快、多、稳定。 | 步骤 3 |
| 集成测试 | Integration Test | 测试多个模块协作的完整链路，如发 HTTP 请求 → 路由 → 数据库 → 返回响应。 | 步骤 3 |
| E2E 测试 | End-to-End Test | 端到端测试，模拟真实用户操作整个应用。最慢，最接近真实场景。 | 步骤 3 |
| pytest | — | Python 最流行的测试框架。自动发现测试文件/函数，运行并输出结果。支持丰富的插件生态。 | 步骤 4 |
| httpx | — | 支持异步的 Python HTTP 客户端库。在测试中用于向 FastAPI 应用发 HTTP 请求。 | 步骤 4 |
| TestClient | — | FastAPI 自带的测试客户端。不需要启动真实服务器，在内存中直接与 FastAPI app 交互。 | 步骤 5 |
| 断言 | Assertion | 程序中的"我认为某件事一定为真"的声明。`assert a == b` 表示断言 a 等于 b，不成立则测试失败。 | 步骤 5 |
| `assert` | — | Python 关键字，用于断言。`assert 条件` 如果条件为 `False`，抛出 `AssertionError`，测试失败。 | 步骤 5 |

## 七、小结

| 你学到了什么 | 一句话总结 |
|--------------|-----------|
| 手动测试的痛点 | 烦、慢、容易漏、不可重复——手工 curl 5 个接口，未来 50 个怎么办？ |
| 测试的本质 | 用代码验证代码，一条 `pytest -v` 跑完所有检查 |
| 测试金字塔 | 单元测试（多）→ 集成测试（中）→ E2E 测试（少），本教程写集成测试 |
| pytest 安装 | `pip install pytest httpx`，pytest 自动发现 `test_*.py` 中的 `test_*` 函数 |
| TestClient | FastAPI 内置测试客户端，不需要启动服务器，直接在内存中模拟 HTTP 请求 |
| 第一个测试 | `test_create_post` → 发 POST → `assert` 状态码和返回值 |
| 运行测试 | `pytest -v`，绿色 PASSED = 通过，红色 FAILED = 失败 |
| 失败输出 | pytest 告诉你：哪个文件、哪一行、实际值、期望值——不需要猜 |
| 命名规范 | `test_<功能>_<场景>`，如 `test_create_post`、`test_login_wrong_password` |
| 测试的价值 | 一次投入，终身保护——每次改代码后跑一遍，全绿就放心 |

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| 测试时启动 uvicorn | 测试跑不起来或报端口占用 | `TestClient` 不需要启动服务器，它在内存中直接与 app 交互 | 不需要 `uvicorn`，直接 `pytest -v` |
| `assert response.status_code == 200` 测试创建文章 | 测试失败，实际是 201 | 创建资源（POST）通常返回 201 Created，不是 200 OK | 用 `assert response.status_code == 201` |
| 忘记导入 `TestClient` | `NameError: name 'TestClient' is not defined` | 没写 `from fastapi.testclient import TestClient` | 在文件顶部加上导入 |
| 函数名不以 `test_` 开头 | pytest 找不到你的测试函数 | pytest 只发现 `test_` 开头的函数 | 重命名为 `test_xxx` |
| 测试文件不在正确位置 | pytest 找不到测试文件 | pytest 扫描当前目录及子目录，文件名必须匹配 `test_*.py` | 放在项目根目录或 `blog_backend/` 下，文件名以 `test_` 开头 |
| 测试中修改了真实数据库 | 测试数据污染了开发数据库 | 测试直接用了 `blog.db` | 下一章（教程 25）会讲如何用独立的测试数据库。本章先了解概念。 |

## 九、下一步建议

你写完了第一个测试，理解了测试的意义。但这只是一个开始——你的博客系统有十几个接口，需要为每个接口写测试。而且当前测试有一个问题：它直接用了你的开发数据库 `blog.db`，测试数据会污染你的真实数据。

**下一章（教程 25：为博客 API 写测试）**，你将：
- 创建独立的测试数据库，不污染开发数据。
- 用 pytest fixture 自动准备测试环境。
- 为注册、登录、创建文章、评论、分页、搜索等全部接口写测试。
- 运行 `pytest -v`，看到一片绿色的 ✓。
- 用 `pytest-cov` 看测试覆盖率。
