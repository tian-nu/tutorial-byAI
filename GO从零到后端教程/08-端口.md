# 第8章 · 端口

> "你家小区只有一个大门，但快递员怎么知道该把包裹送到你家里而不是邻居家？——靠的是门牌号。对服务器来说，这个'门牌号'就是端口。"

---

## 8.1 端口的比喻

### IP = 小区地址，端口 = 房间号

想象你住在一个巨型公寓大楼里——

- **公寓大楼的门牌号**（比如"长安街100号"）= **IP地址**，它让外面的世界能找到这栋楼。
- **你家的房间号**（比如"1801室"）= **端口号**，它让进了大楼的人能找到特定的人。

快递员送快递时，他既需要**地址**（找到楼），也需要**房间号**（找到你）。

服务器的世界一模一样。一个服务器只有一个IP地址，但它上面跑着几十个程序——网站、数据库、文件服务、邮件服务……操作系统怎么知道"这条网络数据该交给哪个程序"？答案就是**端口号**。

```
互联网 → 服务器 (IP: 123.456.789.0)
                  │
                  ├── :80   →  Nginx（网站程序）
                  ├── :443  →  Nginx（HTTPS）
                  ├── :22   →  SSH服务（远程登录）
                  ├── :3306 →  MySQL数据库
                  ├── :6379 →  Redis缓存
                  └── :5432 →  PostgreSQL数据库
```

每一条网络数据到达服务器时，都带着一个目标端口号。操作系统一看："哦，是发给80端口的，那交给Nginx处理。"

### 端口号的数值范围

端口号是一个16位的整数，所以范围是 **0 ~ 65535**，一共65536个。

这65536个端口被分成三组：

| 范围 | 名称 | 说明 |
|------|------|------|
| 0 ~ 1023 | **知名端口**（Well-known Ports） | 被IANA分配给了公认的服务，如80=HTTP、443=HTTPS。普通程序不能用，需要管理员权限才能监听。 |
| 1024 ~ 49151 | **注册端口**（Registered Ports） | 被各种软件厂商注册使用的端口，如3306=MySQL、6379=Redis。普通程序可以用。 |
| 49152 ~ 65535 | **动态/私有端口**（Dynamic/Private Ports） | 操作系统随机分配给客户端程序的临时端口。 |

> 🤔 想多一点：为什么知名端口需要管理员权限？假设任何人写的程序都能监听80端口，那你装个恶意软件，它直接在你电脑上开个假的淘宝网站，你输入密码就被盗了。所以操作系统规定：小于1024的端口，只有root/管理员才能监听。这是一种**安全机制**——"这个端口号小，说明它服务的程序是系统级的重要程序，值得信任。"

## 8.2 常见端口一览

以下是后端开发中最常打交道的端口，混个脸熟：

| 端口 | 服务 | 通俗解释 |
|------|------|----------|
| **80** | HTTP | 网站的"大门"，浏览器默认访问的就是这个端口 |
| **443** | HTTPS | HTTP的加密版，带一把锁 |
| **22** | SSH | 远程登录服务器的"秘密通道" |
| **3306** | MySQL | 最流行的关系型数据库 |
| **5432** | PostgreSQL | 另一个强大的关系型数据库 |
| **6379** | Redis | 内存缓存，快得飞起 |
| **27017** | MongoDB | 最流行的文档型NoSQL数据库 |
| **8080** | HTTP备用 | 开发调试时常用，因为80需要管理员权限 |
| **3000** | Node.js开发 | Node.js项目的默认开发端口 |
| **5000** | Flask开发 | Python Flask框架的默认开发端口 |
| **9200** | Elasticsearch | 搜索引擎 |
| **9092** | Kafka | 消息队列 |
| **25** | SMTP | 发送邮件 |

> 🤔 想多一点：你会发现很多开发框架默认用8080或者3000而不是80。为什么？因为开发时你通常没有管理员权限，而1024以下的端口需要管理员权限。所以约定俗成地用大于1024的"高编号端口"来开发，上线后再用Nginx反向代理转发到80/443。

### 端口冲突

如果你启动一个程序想监听3000端口，但这个端口已经被另一个程序占了，会发生什么？

```go
package main

import "net/http"

func main() {
    http.ListenAndServe(":3000", nil)
}
```

你会得到类似这样的错误：

```
listen tcp :3000: bind: address already in use
```

这就是**端口冲突**——一个端口在同一时间只能被一个程序占用。就好像一扇门不能同时被两个人堵着。

解决方式：
- 换个端口
- 或者把占用端口的程序杀掉：

```bash
lsof -i :3000
kill -9 <PID>
```

## 8.3 一个服务器能跑多少个服务

### 理论极限

端口号是16位，所以理论上一个IP地址可以开 **65536个服务**。但实际上远远达不到：

1. **每个TCP连接要占内存**。一个TCP连接大约占几KB内存（包括内核缓冲区），65535个连接就是几百MB。
2. **文件描述符上限**。Linux默认一个进程最多打开1024个文件描述符（可以调大）。每个TCP连接占一个文件描述符。
3. **CPU处理能力**。就算内存够，CPU也处理不过来这么多并发连接。

> 🤔 想多一点：那你可能听到过"C10K问题"——即"单个服务器如何同时处理1万个客户端连接"。这曾经是2000年代的一个技术难题，催生了epoll、kqueue等高性能I/O模型。后来又有了"C10M问题"（1000万并发）。这些问题的核心不是端口号不够，而是**操作系统和程序的架构能不能高效处理这么多连接**。

### 客户端端口 vs 服务器端口

这里有一个非常重要的区分：

- **服务器端口**：固定的、公开的。比如Nginx固定在80端口上监听。
- **客户端端口**：临时的、随机的。比如你的Chrome浏览器访问百度时，操作系统随便给它分配了一个端口（比如52341）。

```
你的浏览器（客户端）                  百度服务器
IP: 192.168.1.5                      IP: 110.242.68.66
端口: 52341  ──────TCP连接──────→    端口: 80
```

当服务器返回数据时：

```
源：110.242.68.66:80  ──→  目标：192.168.1.5:52341
```

操作系统看到数据是发给52341端口的，就知道"哦，这是给Chrome的"。**这就是为什么你可以同时打开100个浏览器标签页访问不同的网站——每个连接在客户端用不同的临时端口区分。**

一个连接由四个东西唯一确定：

```
{源IP, 源端口, 目标IP, 目标端口}
```

这叫**四元组**。只要四元组中任何一个不同，就是不同的连接。所以理论上同一个客户端可以同时建立65535个连接到同一个服务器。

### 生活中的类比

把服务器想象成一家**客服中心**：

- 客服中心的**总机号码**（400-xxx-xxxx）= **服务器的IP + 知名端口**
- 客服中心的**各个部门分机号** = **不同的端口**（投诉按1，咨询按2，技术支持按3……）
- 客服手中的**临时通话线路** = **临时端口**，通话结束就释放

你打进去时，总机给你分配一条临时线路（比如线路编号#9527）。这条线路在通话期间属于你，挂断后就还给系统了。

## 本章小结

| 知识点 | 一句话概括 |
|--------|------------|
| 端口是什么 | 操作系统用来区分不同网络程序的编号（0~65535） |
| IP vs 端口 | IP找服务器，端口找服务器上的哪个程序 |
| 知名端口 | 0~1023，被IANA分配，需要管理员权限 |
| 端口冲突 | 一个端口同一时刻只能被一个程序占用 |
| 四元组 | {源IP, 源端口, 目标IP, 目标端口} 唯一确定一个连接 |
| 客户端端口 | 操作系统随机分配的临时端口，连接结束后释放 |
