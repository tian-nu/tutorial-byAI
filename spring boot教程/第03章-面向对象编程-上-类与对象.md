## 3.1 从面向过程到面向对象

### 3.1.1 一个场景，两种写法

假设我们要管理学生的成绩：存储每个学生的姓名和分数，然后打印所有及格学生的信息。

**面向过程写法**（类似C语言的思路）：

```java
public class ProceduralDemo {
    public static void main(String[] args) {
        // 数据：用多个数组存储
        String[] names = {"张三", "李四", "王五", "赵六"};
        int[] scores = {58, 92, 76, 45};
        
        // 操作：函数驱动
        printPassedStudents(names, scores);
    }
    
    // 函数：打印及格学生
    public static void printPassedStudents(String[] names, int[] scores) {
        for (int i = 0; i < names.length; i++) {
            if (scores[i] >= 60) {
                System.out.println(names[i] + "：" + scores[i] + "分，及格");
            }
        }
    }
    
    // 如果还需要计算平均分、排名、查找最高分……
    // 你会写一堆接收数组参数的函数
    public static double calculateAverage(int[] scores) {
        int sum = 0;
        for (int s : scores) sum += s;
        return (double) sum / scores.length;
    }
    
    public static int findMaxScore(int[] scores) {
        int max = scores[0];
        for (int s : scores) {
            if (s > max) max = s;
        }
        return max;
    }
}
```

在这个写法中，**数据（`names` 和 `scores` 数组）与操作数据的函数是分离的**。修改一个人的名字需要同时记住两个数组的索引，极易出错。更重要的是，这些函数可以被任何地方调用，没有"谁的数据归谁管"的概念。

**面向对象写法**：

```java
public class Student {
    // 数据：封装在类内部
    private String name;
    private int score;
    
    // 构造方法
    public Student(String name, int score) {
        this.name = name;
        this.score = score;
    }
    
    // 行为：数据自带的操作
    public boolean isPassed() {
        return this.score >= 60;
    }
    
    public String getInfo() {
        return this.name + "：" + this.score + "分，" + (isPassed() ? "及格" : "不及格");
    }
    
    // getter/setter
    public String getName() { return name; }
    public int getScore() { return score; }
    public void setScore(int score) { this.score = score; }
}

public class OOPDemo {
    public static void main(String[] args) {
        // 数据：每个学生是一个独立的对象
        Student[] students = {
            new Student("张三", 58),
            new Student("李四", 92),
            new Student("王五", 76),
            new Student("赵六", 45)
        };
        
        // 操作：对象自己知道怎么判断是否及格
        for (Student s : students) {
            if (s.isPassed()) {
                System.out.println(s.getInfo());
            }
        }
    }
}
```

在这个写法中，**数据和操作数据的逻辑被封装在同一个类中**。一个 `Student` 对象自己知道自己的名字、分数，也自己知道如何判断是否及格。"学生"这个**对象**成了最小的完整单元：既有属性（名字、分数），又有行为（判断及格、返回信息）。

### 3.1.2 面向过程 vs 面向对象

| 对比维度 | 面向过程 | 面向对象 |
|----------|----------|----------|
| 核心单位 | 函数 | 对象（类） |
| 数据与操作 | 分离存储 | 封装在一起 |
| 思维方式 | "先做什么，再做什么" | "有哪些实体，它们能做什么" |
| 代码组织 | 按功能分组函数 | 按实体组织类 |
| 扩展方式 | 增加新函数，可能影响已有数据 | 增加新类或修改现有类，影响范围可控 |
| 代码复用 | 复制函数或传参 | 继承、组合 |
| 典型语言 | C、Pascal | Java、C++、C#、Python |

**为什么OOP更好？** 当你需要新增一个"按等级评优"的功能时，面向过程需要再写一个新函数遍历数组；而面向对象只需在 `Student` 类中增加一个 `getGrade()` 方法。当系统有100个实体、每个实体有20个属性时，面向过程的维护噩梦就开始了——而面向对象的封装能让你清晰地管理复杂度。

### 3.1.3 三大特性总览

面向对象有三大基石特性，我们将分两章深入学习：

| 特性 | 含义 | 解决什么问题 | 所在章节 |
|------|------|-------------|----------|
| **封装** | 隐藏内部细节，仅暴露必要接口 | 数据安全、降低耦合 | 本章（3.4节） |
| **继承** | 子类复用父类的属性和方法 | 代码复用、建立层次体系 | 第4章 |
| **多态** | 同一接口在不同对象上表现不同行为 | 灵活扩展、解耦 | 第4章 |

### 3.1.4 类与对象的关系：一个经典比喻

> **类（Class）= 建筑图纸，对象（Object）= 建好的房子**

- 一张图纸可以盖出无数栋房子，就像一个类可以 `new` 出无数个对象
- 房子需要施工才能存在（`new`），图纸只是描述
- 每栋房子有自己的涂料颜色（属性值），内部装修可以不同
- 图纸上标注了"这里是卧室"（成员变量定义），但每栋房子的卧室大小可以不同（成员变量值）

```java
Student s1 = new Student("张三", 90);  // 用图纸（Student类）盖了第1栋房
Student s2 = new Student("李四", 85);  // 同一张图纸盖了第2栋房
// s1 和 s2 是独立的两个对象，各自拥有自己的 name 和 score
```

## 3.2 类的定义

### 3.2.1 完整的类语法结构

一个完整的Java类由以下部分构成：

```java
[修饰符] class 类名 {
    // 1. 属性（成员变量 / 字段）
    
    // 2. 构造方法（可重载多个）
    
    // 3. 普通方法（行为）
    
    // 4. 代码块（静态 / 非静态）
    
    // 5. 内部类
}
```

用一个完整的例子来说明：

```java
public class Person {
    // --- 1. 属性（成员变量）---
    private String name;       // 姓名
    private int age;           // 年龄
    private String idCard;     // 身份证号
    
    // --- 2. 构造方法 ---
    public Person() {
        // 无参构造方法：创建"空白"Person
    }
    
    public Person(String name, int age, String idCard) {
        this.name = name;
        this.age = age;
        this.idCard = idCard;
    }
    
    // --- 3. 普通方法 ---
    public void introduce() {
        System.out.println("我叫" + name + "，今年" + age + "岁");
    }
    
    public boolean isAdult() {
        return age >= 18;
    }
    
    // getter / setter
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public int getAge() { return age; }
    public void setAge(int age) { this.age = age; }
}
```

### 3.2.2 成员变量 vs 局部变量

这是初学者最容易混淆的概念，务必记住以下对比：

| 对比项 | 成员变量（实例变量） | 局部变量 |
|--------|---------------------|----------|
| **定义位置** | 类中、方法外 | 方法内、代码块内、方法参数 |
| **默认值** | **有**（int=0, double=0.0, boolean=false, 引用类型=null） | **无**，必须手动初始化才能使用 |
| **生命周期** | 随对象创建而诞生，随对象被GC回收而消亡 | 随方法调用而诞生，随方法结束而消亡 |
| **存储位置** | **堆内存**（Heap） | **栈内存**（Stack） |
| **访问修饰符** | 可以加 public/private/protected | 不能加访问修饰符 |
| **`static` 修饰** | 可以（变为类变量） | 不可以 |

代码验证：

```java
public class VariableDemo {
    String memberVar;    // 成员变量：默认值为 null
    int memberInt;       // 成员变量：默认值为 0
    
    public void test() {
        String localVar; // 局部变量：无默认值！
        // System.out.println(localVar);  // ⛔ 编译错误！未初始化
        
        int localInt = 0; // 局部变量必须显式初始化
        System.out.println(localInt);   // ✅ 正确
        
        System.out.println(memberVar);  // ✅ 输出 null
        System.out.println(memberInt);  // ✅ 输出 0
    }
}
```

> 🚨 **坑点：局部变量不初始化就使用 → 编译错误**
> 
> **原因**：Java设计者认为，局部变量忘记初始化通常是程序员的疏漏，编译器应该强制提醒。而成员变量有默认值是考虑到对象创建时JVM统一清零更高效。
> 
> **错误示例**：
> ```java
> public void badMethod() {
>     int x;
>     System.out.println(x);  // 编译错误：variable x might not have been initialized
> }
> ```
> 
> **正确做法**：
> ```java
> public void goodMethod() {
>     int x = 0;              // 显式初始化
>     System.out.println(x);  // ✅ 0
> }
> ```

### 3.2.3 访问修饰符速查表

Java有四种访问权限，它们控制着类、方法、属性在多大范围内可以被访问：

| 修饰符 | 同类中 | 同包下（子类/非子类） | 不同包的子类 | 不同包的非子类 |
|--------|--------|----------------------|-------------|---------------|
| `private` | ✅ | ❌ | ❌ | ❌ |
| **default**（不写） | ✅ | ✅ | ❌ | ❌ |
| `protected` | ✅ | ✅ | ✅ | ❌ |
| `public` | ✅ | ✅ | ✅ | ✅ |

记忆口诀：**"私有的只有自己，默认的同包兄弟，受保护的儿子也能看，公开的谁都不拦"**。

代码验证（需要两个不同的包来测试，这里仅示意）：

```java
package com.example.a;
public class Parent {
    private   int a = 1;  // 只有本类可见
              int b = 2;  // default：同包可见
    protected int c = 3;  // 同包 + 子类可见
    public    int d = 4;  // 所有地方可见
    
    public void show() {
        System.out.println(a); // ✅ 同类：全可见
        System.out.println(b);
        System.out.println(c);
        System.out.println(d);
    }
}

// 同包下的另一个类
package com.example.a;
public class SamePackage {
    public void show() {
        Parent p = new Parent();
        // System.out.println(p.a); // ❌ private 不可见
        System.out.println(p.b);    // ✅ default 同包可见
        System.out.println(p.c);    // ✅ protected 同包可见
        System.out.println(p.d);    // ✅ public 可见
    }
}

// 不同包下的子类
package com.example.b;
import com.example.a.Parent;
public class Child extends Parent {
    public void show() {
        // System.out.println(a); // ❌ private 不可见
        // System.out.println(b); // ❌ default 不同包不可见
        System.out.println(c);    // ✅ protected 子类可见
        System.out.println(d);    // ✅ public 可见
    }
}
```

### 3.2.4 方法的完整定义

方法定义的完整语法：

```
[修饰符] 返回类型 方法名([参数类型 参数名, ...]) [throws 异常类型] {
    // 方法体
    [return 返回值;]
}
```

```java
public class MethodDemo {
    // 1. 无参无返回值
    public void sayHello() {
        System.out.println("Hello");
    }
    
    // 2. 有参有返回值
    public int add(int a, int b) {
        return a + b;
    }
    
    // 3. 有参无返回值
    public void printSum(int a, int b) {
        System.out.println("和：" + (a + b));
    }
    
    // 4. 可变参数（实际上是数组）
    public int sumAll(int... numbers) {
        int total = 0;
        for (int n : numbers) {
            total += n;
        }
        return total;
    }
}
```

> 🚨 **坑点：方法声明了非void返回类型，但不是所有执行分支都有return → 编译错误**
> 
> **原因**：编译器会分析方法的每一条执行路径，如果存在某条路径没有返回值，就会报错（因为调用方期望得到一个值，但可能得不到）。
> 
> **错误示例**：
> ```java
> public int getGrade(int score) {
>     if (score >= 90) { return 5; }
>     if (score >= 80) { return 4; }
>     // score < 80 时，没有return！编译错误：missing return statement
> }
> ```
> 
> **正确做法**：
> ```java
> public int getGrade(int score) {
>     if (score >= 90) return 5;
>     if (score >= 80) return 4;
>     if (score >= 70) return 3;
>     return 0;  // 兜底返回
> }
> ```

## 3.3 对象创建与构造方法

### 3.3.1 `new` 关键字背后发生了什么

当你写下 `Student s = new Student("张三", 58);` 这行代码时，JVM在背后完成了四件大事：

```
1. 堆中分配内存
   ↓
2. 成员变量默认初始化（int→0, String→null, boolean→false）
   ↓
3. 执行构造方法中的代码（显式赋值）
   ↓
4. 将堆中对象的引用地址返回，赋给栈中的变量 s
```

图示：

```
栈内存（Stack）                    堆内存（Heap）
┌─────────────┐                 ┌────────────────────┐
│ s = 0x3F8A  │ ─────引用────→ │ Student对象         │
│             │                 │ name: null → "张三" │  ← 步骤3
│             │                 │ score: 0 → 58       │  ← 步骤2→3
│             │                 └────────────────────┘
└─────────────┘
```

```java
public class NewKeywordDemo {
    public static void main(String[] args) {
        // 这一行代码触发了上述四步
        Student s = new Student("张三", 58);
        
        // s 存储的不是对象本身，而是对象在堆中的地址（引用）
        // Java中你不能直接操作内存地址，这个概念是透明的
    }
}
```

### 3.3.2 构造方法的语法

构造方法是一种特殊的方法，它的特征非常明显：

```java
public class Student {
    private String name;
    private int score;
    
    // 构造方法：与类同名，没有返回类型（连void都不写）
    public Student(String name, int score) {
        this.name = name;   // 将参数值赋给成员变量
        this.score = score;
    }
    
    // 不是构造方法！因为有返回类型void，这是普通方法！
    // public void Student(String name) { }  ← 这是普通方法，不是构造方法
}
```

构造方法的硬性规则：
- **方法名必须与类名完全相同**（包括大小写）
- **不能写返回类型**（`void` 也不能写）
- 使用 `new` 关键字自动调用
- 可以有多个构造方法（重载）

### 3.3.3 默认构造方法

如果你没有写任何构造方法，Java编译器会自动为你生成一个**默认的无参构造方法**：

```java
// 你写的代码：
public class Person {
    private String name;
}

// 编译后等价于：
public class Person {
    private String name;
    
    // 编译器自动生成的无参构造方法
    public Person() {
        super();  // 调用Object的构造方法
    }
}
```

> 🚨 **坑点：一旦你自定义了任何构造方法（有参的），默认的无参构造方法就会立即消失！**
> 
> **原因**：编译器认为你既然手动写了构造方法，说明你对对象的创建有特殊要求，就不再画蛇添足。但这个行为会引发一系列连锁问题。
> 
> **后果**：
> 1. `new Person()` 直接报编译错误
> 2. 子类继承时，子类构造方法默认调用 `super()`，父类没有无参构造 → 子类编译失败
> 3. 框架（Spring、MyBatis等）反射创建对象时通常依赖无参构造 → 运行时崩溃
> 
> **错误示例**：
> ```java
> public class Person {
>     private String name;
>     
>     public Person(String name) {  // 自定义了有参构造
>         this.name = name;
>     }
>     // 此时默认无参构造消失了！
> }
> 
> public class Test {
>     public static void main(String[] args) {
>         Person p = new Person();  // ❌ 编译错误！找不到无参构造方法
>     }
> }
> ```
> 
> **正确做法**：**只要你写了有参构造方法，就同时显式写一个无参构造方法！**
> ```java
> public class Person {
>     private String name;
>     
>     public Person() {}  // 显式保留无参构造
>     
>     public Person(String name) {
>         this.name = name;
>     }
> }
> ```

### 3.3.4 构造方法重载

与普通方法一样，构造方法也支持重载——多个构造方法，参数列表不同：

```java
public class Employee {
    private String name;
    private int age;
    private String department;
    
    // 无参构造
    public Employee() {
        this.department = "未分配";  // 默认部门
    }
    
    // 只需姓名
    public Employee(String name) {
        this();              // 调用无参构造，设置默认department
        this.name = name;
    }
    
    // 姓名 + 年龄
    public Employee(String name, int age) {
        this(name);          // 调用上一个构造方法
        this.age = age;
    }
    
    // 全部属性
    public Employee(String name, int age, String department) {
        this(name, age);     // 调用上一个构造方法
        this.department = department;
    }
}
```

通过 `this(...)` 逐级调用，构造方法的逻辑得以复用，避免了每个构造方法都要重复写一堆赋值语句。

### 3.3.5 `this` 关键字的三种用法

`this` 代表**当前对象的引用**。它在三种场景下有不同的用法：

**用法1：`this.属性` — 区分成员变量和局部变量（最常用）**

```java
public class Student {
    private String name;
    
    // 参数名和成员变量名相同 → 必须用this区分
    public void setName(String name) {
        this.name = name;  // this.name 是成员变量，name 是参数
    }
    
    // 参数名和成员变量名不同 → 不需要this（但不推荐这样写）
    public void setNameBad(String n) {
        name = n;  // 能用，但不够直观
    }
}
```

**用法2：`this.方法名()` — 调用本类的另一个方法**

```java
public class Calculator {
    public int add(int a, int b) {
        return a + b;
    }
    
    public int addThree(int a, int b, int c) {
        return this.add(this.add(a, b), c);  // 调用本类的add方法
        // 这里 this. 可以省略，但加上更清晰
    }
}
```

**用法3：`this(参数)` — 调用本类的另一个构造方法（必须在第一行）**

```java
public class Person {
    private String name;
    private int age;
    
    public Person() {
        this("未知", 0);  // 调用另一个构造方法
        // 注意：这行必须在构造方法的第一行！
        System.out.println("无参构造完成");
    }
    
    public Person(String name, int age) {
        this.name = name;
        this.age = age;
        System.out.println("有参构造完成");
    }
}

// 测试：
public class Test {
    public static void main(String[] args) {
        Person p = new Person();
        // 输出：
        // 有参构造完成      ← 先执行被调用的构造方法
        // 无参构造完成      ← 再执行当前构造方法的剩余部分
    }
}
```

> 🚨 **坑点：`this()` 和 `super()` 不能同时出现在一个构造方法中**
> 
> **原因**：`this()` 会间接调用另一个构造方法，而另一个构造方法的第一行必然有 `super()`（显式或隐式）。如果在同一个构造方法中同时写 `this()` 和 `super()`，相当于父类构造被调用了两次，这是不合理的。Java规定构造方法的第一行要么是 `this(...)`，要么是 `super(...)`，二者只能选其一。

## 3.4 封装

### 3.4.1 为什么需要封装——三步递进式解释

**步骤1：直接用 `public` 属性——最原始的方式**

```java
public class BadStudent {
    public String name;
    public int age;
}
```

使用起来很简单：

```java
BadStudent bs = new BadStudent();
bs.name = "张三";
bs.age = -1;  // ⚠️ 年龄设为-1，显然不合理，但没有任何检查！
```

**步骤2：问题暴露——数据可能被任意污染**

`age = -1` 违反了业务规则（年龄不能为负数），但由于属性是公开的，外部代码可以随意设置任何值，你无法阻止。如果这是一个工资管理系统，`salary = -5000` 会带来严重后果。

**步骤3：解决方案——`private` + getter/setter**

```java
public class GoodStudent {
    private String name;
    private int age;
    
    public void setAge(int age) {
        if (age >= 0 && age <= 150) {
            this.age = age;
        } else {
            throw new IllegalArgumentException("年龄必须在0~150之间，你传的是：" + age);
        }
    }
    
    public int getAge() {
        return this.age;
    }
    
    public void setName(String name) {
        if (name != null && !name.trim().isEmpty()) {
            this.name = name;
        } else {
            throw new IllegalArgumentException("姓名不能为空");
        }
    }
    
    public String getName() {
        return this.name;
    }
}
```

现在，任何试图设置非法年龄的操作都会被拦截：

```java
GoodStudent gs = new GoodStudent();
gs.setAge(-1);  // 抛出异常：IllegalArgumentException: 年龄必须在0~150之间，你传的是：-1
gs.setAge(20);  // ✅ 正常设置
```

**封装的核心价值**：
1. **数据安全**：通过setter中的校验逻辑，确保对象状态始终合法
2. **实现隐藏**：外部不知道内部如何存储年龄（可能是int可能是long），只需要知道getter/setter
3. **灵活修改**：将来如果年龄要改为从身份证号自动计算，只需修改getter内部逻辑，外部调用者无感知
4. **调试方便**：在所有setter处加日志，可以追踪到每一次数据修改的来源

### 3.4.2 JavaBean规范

JavaBean是Java世界约定俗成的可重用组件规范，Spring、MyBatis等几乎所有框架都依赖这个规范：

```java
public class Employee implements java.io.Serializable {
    // 1. 所有属性用 private
    private Long id;
    private String name;
    private Integer age;
    private Double salary;
    private String department;
    
    // 2. 必须有 public 无参构造方法（框架反射需要）
    public Employee() {}
    
    // 3. 每个属性提供 public 的 getter 和 setter
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    
    public Integer getAge() { return age; }
    public void setAge(Integer age) { this.age = age; }
    
    public Double getSalary() { return salary; }
    public void setSalary(Double salary) { this.salary = salary; }
    
    public String getDepartment() { return department; }
    public void setDepartment(String department) { this.department = department; }
}
```

**一条铁律：所有框架中的实体类都要求遵循JavaBean规范**。如果缺少无参构造方法，MyBatis的结果映射会失败；如果缺了某个setter，前端传过来的JSON就无法正确绑定。

### 3.4.3 Lombok `@Data` — 一行注解解放双手

手写这么多getter/setter不仅枯燥，而且容易出错（比如改了属性名忘记改getter）。Lombok是Java世界的一大利器：

**传统写法（约30行）：**

```java
public class Employee {
    private Long id;
    private String name;
    private Integer age;
    private Double salary;
    
    public Employee() {}
    
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public Integer getAge() { return age; }
    public void setAge(Integer age) { this.age = age; }
    public Double getSalary() { return salary; }
    public void setSalary(Double salary) { this.salary = salary; }
    
    @Override
    public String toString() { /* ... 几十行 */ }
    
    @Override
    public boolean equals(Object o) { /* ... 几十行 */ }
    
    @Override
    public int hashCode() { /* ... */ }
}
```

**Lombok写法（约5行）：**

```java
import lombok.Data;
import lombok.NoArgsConstructor;
import lombok.AllArgsConstructor;

@Data                    // 生成 getter/setter/toString/equals/hashCode
@NoArgsConstructor       // 生成无参构造
@AllArgsConstructor      // 生成全参构造
public class Employee {
    private Long id;
    private String name;
    private Integer age;
    private Double salary;
}
```

`@Data` 帮你自动生成了所有getter/setter、`toString()`、`equals()` 和 `hashCode()` 方法。这些方法在编译期生成，不影响运行时性能。你的IDE（安装了Lombok插件后）也能正确提示这些方法。

> 🚨 **坑点：`boolean` 类型的 getter 是 `isXxx()` 而不是 `getXxx()`**
> 
> **原因**：这是JavaBean规范的硬性规定——基本类型 `boolean` 的getter方法名应以 `is` 开头。但**包装类型 `Boolean`** 的getter仍然是 `getXxx()`。这个差异在某些序列化框架中会导致问题。
> 
> ```java
> public class User {
>     private boolean active;   // 基本类型
>     private Boolean deleted;  // 包装类型
>     
>     // Lombok生成：
>     public boolean isActive() { return active; }    // ✅ isActive，不是 getActive
>     public Boolean getDeleted() { return deleted; } // ✅ getDeleted，不是 isDeleted
> }
> ```
> 
> **陷阱**：如果你手动写 `getActive()` 方法，一些框架（如Jackson）在序列化时会去找 `isActive()`，导致出现两个 "active" 属性。如果你同时写了 `getActive()` 和 `isActive()`，行为将变得不可预测。**交给Lombok自动生成即可避免这个问题。**

## 3.5 `static` 关键字

### 3.5.1 `static` 变量（类变量）

`static` 修饰的变量属于**类本身**，而不属于任何实例。所有实例共享同一份 `static` 变量：

```java
public class Counter {
    // 实例变量：每个对象一份
    private String name;
    
    // 静态变量：只有一份，被所有对象共享
    private static int totalCount = 0;
    
    public Counter(String name) {
        this.name = name;
        totalCount++;  // 每创建一个对象，计数器+1
    }
    
    public static int getTotalCount() {
        return totalCount;
    }
    
    public String getName() {
        return name;
    }
}

// 测试：
public class Test {
    public static void main(String[] args) {
        System.out.println(Counter.getTotalCount()); // 0（还没创建对象时就可以访问）
        
        Counter c1 = new Counter("一号");
        Counter c2 = new Counter("二号");
        Counter c3 = new Counter("三号");
        
        System.out.println(Counter.getTotalCount()); // 3
        System.out.println(c1.getName());             // "一号"（实例变量各不同）
    }
}
```

**内存视角解析**：

```
方法区/元空间（JDK 8+）                堆内存（Heap）
┌──────────────────────┐           ┌──────────────┐
│ Counter.class        │           │ c1对象       │
│   totalCount = 3     │ ← 共享    │ name = "一号" │
│   getTotalCount()    │           └──────────────┘
└──────────────────────┘           ┌──────────────┐
                                   │ c2对象       │
                                   │ name = "二号" │
                                   └──────────────┘
```

`static` 变量在类加载时初始化，比任何对象创建都要早。这也是为什么可以通过 `类名.静态变量` 直接访问。

### 3.5.2 `static` 方法（类方法）

```java
public class MathUtils {
    // 静态方法：不需要创建对象就能调用
    public static int max(int a, int b) {
        return a > b ? a : b;
    }
    
    public static double average(int... numbers) {
        if (numbers.length == 0) return 0;
        int sum = 0;
        for (int n : numbers) sum += n;
        return (double) sum / numbers.length;
    }
    
    // 实例方法：必须先创建对象
    public int min(int a, int b) {
        return a < b ? a : b;
    }
}

// 调用：
public class Test {
    public static void main(String[] args) {
        // 静态方法：类名直接调用
        int m = MathUtils.max(10, 20);  // ✅
        System.out.println(MathUtils.average(1, 2, 3, 4, 5)); // 3.0
        
        // 实例方法：需要对象
        MathUtils utils = new MathUtils();
        int n = utils.min(10, 20);  // ✅
    }
}
```

> 🚨 **坑点：`static` 方法中不能直接使用 `this` 关键字**
> 
> **原因**：`this` 指代当前对象，但 `static` 方法可以通过类名直接调用，此时根本没有任何对象。编译器直接报错。
> 
> ```java
> public class Demo {
>     public static void staticMethod() {
>         // System.out.println(this);  // ❌ 编译错误！静态方法中没有this
>     }
> }
> ```

> 🚨 **坑点：`static` 方法中不能直接访问非 `static` 成员（实例变量和实例方法）**
> 
> **原因**：`static` 方法属于类，调用时对象可能不存在。而非 `static` 成员必须依附于具体的对象。编译器无法知道你要操作哪个对象。
> 
> **错误示例**：
> ```java
> public class Demo {
>     private String name = "hello";           // 非static
>     
>     public static void printName() {
>         System.out.println(name);            // ❌ 编译错误！
>     }
>     
>     public void instanceMethod() {
>         System.out.println("实例方法");
>     }
>     
>     public static void callInstance() {
>         instanceMethod();                    // ❌ 编译错误！
>     }
> }
> ```
> 
> **正确做法**：
> ```java
> public class Demo {
>     private String name = "hello";
>     
>     public static void printName() {
>         Demo d = new Demo();                 // 先创建对象
>         System.out.println(d.name);          // ✅ 通过对象访问
>         d.instanceMethod();                  // ✅ 通过对象调用
>     }
>     
>     public void instanceMethod() {
>         System.out.println("实例方法");
>     }
> }
> ```

### 3.5.3 `static` 代码块

`static` 代码块在**类加载时**执行，且只执行一次。常用于加载配置文件、初始化数据库连接池、预先准备静态数据等场景：

```java
public class Config {
    public static String appName;
    public static int maxConnections;
    public static List<String> allowedOrigins;
    
    // 静态代码块：类加载时执行，只执行一次
    static {
        System.out.println("正在初始化配置...");
        appName = "员工管理系统";
        maxConnections = 100;
        allowedOrigins = new ArrayList<>();
        allowedOrigins.add("http://localhost:3000");
        allowedOrigins.add("http://localhost:8080");
        System.out.println("配置初始化完成");
    }
    
    public static void main(String[] args) {
        System.out.println(Config.appName);    // "员工管理系统"
        // "正在初始化配置..." 只会在类首次加载时打印一次
    }
}
```

**执行顺序测试**（非常重要！面试常考）：

```java
public class InitOrder {
    static {
        System.out.println("1. 静态代码块");
    }
    
    {
        System.out.println("3. 实例代码块（每次new都执行）");
    }
    
    public InitOrder() {
        System.out.println("4. 构造方法");
    }
    
    static {
        System.out.println("2. 第二个静态代码块");
    }
    
    public static void main(String[] args) {
        System.out.println("--- main开始 ---");
        new InitOrder();
        System.out.println("--- 第一个对象创建完毕 ---");
        new InitOrder();
        System.out.println("--- 第二个对象创建完毕 ---");
    }
}
```

输出：
```
1. 静态代码块
2. 第二个静态代码块
--- main开始 ---
3. 实例代码块
4. 构造方法
--- 第一个对象创建完毕 ---
3. 实例代码块
4. 构造方法
--- 第二个对象创建完毕 ---
```

结论：**静态代码块仅执行一次（类加载时），实例代码块每次 `new` 都执行，在构造方法之前。**

### 3.5.4 静态导入（`import static`）

当你频繁使用某个类的静态方法或常量时，静态导入可以省去类名前缀：

```java
// 不使用静态导入：
public class WithoutStaticImport {
    public static void main(String[] args) {
        System.out.println(Math.PI);         // 每次都要写 Math.
        System.out.println(Math.sqrt(16));
        System.out.println(Math.max(10, 20));
        System.out.println(Math.random());
    }
}

// 使用静态导入：
import static java.lang.Math.PI;
import static java.lang.Math.sqrt;
import static java.lang.Math.max;
import static java.lang.Math.random;

public class WithStaticImport {
    public static void main(String[] args) {
        System.out.println(PI);              // 直接写，简洁很多
        System.out.println(sqrt(16));
        System.out.println(max(10, 20));
        System.out.println(random());
    }
}

// 更简洁：通配符导入
import static java.lang.Math.*;

public class WildcardStaticImport {
    public static void main(String[] args) {
        System.out.println(abs(-5));         // Math.abs
        System.out.println(ceil(3.14));      // Math.ceil
        System.out.println(floor(3.14));     // Math.floor
    }
}
```

> ⚠️ 静态导入虽然方便，但不要滥用。如果到处静态导入 `*`，代码可读性会下降——别人不知道 `abs` 是哪个类的。建议仅对高频使用的 `Math`、`Assertions`（测试）等使用。

### 3.5.5 单例模式入门

单例模式确保一个类在应用程序中**只有一个实例**，并提供全局访问点。Spring框架管理的Bean默认就是单例（第10章详解）。

**饿汉式（Eager）**：类加载时就创建实例——简单安全，但不灵活：

```java
public class EagerSingleton {
    // 1. 私有静态实例——类加载时就创建
    private static final EagerSingleton INSTANCE = new EagerSingleton();
    
    // 2. 私有构造方法——外部无法new
    private EagerSingleton() {
        System.out.println("饿汉式单例被创建了");
    }
    
    // 3. 公共的获取方法
    public static EagerSingleton getInstance() {
        return INSTANCE;
    }
    
    public void doSomething() {
        System.out.println("单例工作中...");
    }
}

// 使用：
EagerSingleton s1 = EagerSingleton.getInstance();
EagerSingleton s2 = EagerSingleton.getInstance();
System.out.println(s1 == s2);  // true —— 同一个实例
```

**懒汉式（Lazy）**：第一次使用时才创建——省资源，但需要处理线程安全（第7章详讲）：

```java
public class LazySingleton {
    private static LazySingleton instance;
    
    private LazySingleton() {
        System.out.println("懒汉式单例被创建了");
    }
    
    // synchronized 保证线程安全 —— 但每次获取都加锁，性能差
    public static synchronized LazySingleton getInstance() {
        if (instance == null) {
            instance = new LazySingleton();
        }
        return instance;
    }
}
```

> 📌 **预告**：到第10章你会看到，Spring IoC容器默认将每个Bean创建为单例。理解了这里的单例思想，再去学Spring的 `@Scope("singleton")` 就易如反掌了。

## 3.6 包（`package`）

### 3.6.1 包的三个作用

1. **命名空间**：防止类名冲突。你写的 `User` 类和第三方库的 `User` 类可以同名但不同包。
2. **访问控制**：`default` 访问权限就是靠包来界定的——同包可见，不同包不可见。
3. **代码组织**：将功能相关的类放在同一个包下，便于管理和查找。

### 3.6.2 命名规范

Java生态广泛遵循以下命名约定：

```
com.<公司名>.<项目名>.<模块名>
```

示例：
```
com.example.ems.controller   → 存放Controller（控制器）
com.example.ems.service      → 存放Service（业务逻辑）
com.example.ems.repository   → 存放Repository（数据访问）
com.example.ems.entity       → 存放Entity（实体类）
com.example.ems.config       → 存放Config（配置类）
com.example.ems.util         → 存放Util（工具类）
```

### 3.6.3 `import` — 引入外部类

```java
// 声明本类所属的包（必须写在文件第一行）
package com.example.ems.entity;

// 导入需要使用的其他类
import java.util.List;           // 导入单个类
import java.util.ArrayList;      // 导入单个类
import java.util.*;              // 导入 java.util 包下所有类（不包括子包的类）

// java.lang 包下的类自动导入，无需手动写 import：
// String、Math、Integer、System、Object 等都在 java.lang 中
```

> 🚨 **坑点：`import java.util.*` 不会导入子包的类**
> 
> **原因**：Java的 `import` 只导入指定包中的类，不会递归导入子包。`java.util.*` 只能导入 `java.util` 包下的类，不能导入 `java.util.concurrent` 或 `java.util.stream` 下的类。
> 
> ```java
> import java.util.*;       // 导入了 List、Map、Set 等
> // 但下面的类仍然需要单独导入：
> import java.util.concurrent.ConcurrentHashMap;  // concurrent 是 util 的子包
> import java.util.stream.Collectors;              // stream 是 util 的子包
> ```

## 本章小结

本章带你完成了从"面向过程"到"面向对象"的心智转变。让我们回顾核心知识点：

- **类与对象**：类是图纸，对象是房子。`new` 关键字完成"分配内存→默认初始化→构造初始化→返回引用"四步。
- **成员变量 vs 局部变量**：成员变量有默认值，存储在堆中；局部变量必须显式初始化，存储在栈中。
- **构造方法**：与类同名、无返回类型。一旦自定义有参构造，默认无参构造消失——**必须显式补一个无参构造**，否则框架反射时会崩溃。
- **`this` 三种用法**：`this.属性` 区分变量、`this.方法()` 链式调用、`this(参数)` 调用其他构造方法（必须在第一行）。
- **封装**：`private` 属性 + 公开 getter/setter，保护数据不被随意修改。JavaBean规范是所有框架的基石。
- **Lombok `@Data`**：一行注解替代几十行模板代码，但注意 `boolean` 的 getter 是 `isXxx()`。
- **`static`**：属于类而非对象。静态方法中不能使用 `this` 也不能直接访问实例成员。静态代码块在类加载时执行一次。
- **单例模式**：确保全局只有一个实例，Spring的Bean默认就是单例。
- **`package`**：三层作用（命名空间、访问控制、代码组织）。`import` 不会递归导入子包。

**答案5**：

```java
public class BankAccount {
    private String accountNo;
    private String ownerName;
    private double balance;
    private static int totalAccounts = 0;
    
    public BankAccount() {
        totalAccounts++;
    }
    
    public BankAccount(String accountNo, String ownerName, double balance) {
        this.accountNo = accountNo;
        this.ownerName = ownerName;
        this.setBalance(balance);  // 走setter校验
        totalAccounts++;
    }
    
    public void deposit(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("存款金额必须大于0");
        }
        this.balance += amount;
    }
    
    public void withdraw(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("取款金额必须大于0");
        }
        if (amount > this.balance) {
            throw new IllegalArgumentException("余额不足，当前余额：" + this.balance);
        }
        this.balance -= amount;
    }
    
    // getters & setters
    public String getAccountNo() { return accountNo; }
    public void setAccountNo(String accountNo) { this.accountNo = accountNo; }
    public String getOwnerName() { return ownerName; }
    public void setOwnerName(String ownerName) { this.ownerName = ownerName; }
    public double getBalance() { return balance; }
    public void setBalance(double balance) {
        if (balance < 0) {
            throw new IllegalArgumentException("余额不能为负数");
        }
        this.balance = balance;
    }
    public static int getTotalAccounts() { return totalAccounts; }
}
