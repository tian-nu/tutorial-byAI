# 步骤3 · 把本地的代码推送到你自己的 GitHub 仓库

> "你手里有一堆文件，GitHub 上有一个空仓库——现在让它们'结婚'。Git 做媒人，把本地代码送到远程仓库，这样全世界的人都能看到你的代码。这个动作叫 push（推送），就像快递员把你打包好的箱子送到收件人门口。第一次 push 有点像搬家：你得先把东西打包、找到目的地地址、运过去，还得确认有没有漏掉什么。来，一步一步来。"

> 目标仓库：你在第2章亲手创建的仓库——`git@github.com:<你的用户名>/my-first-project.git`

---

## 3.1 在电脑上创建一个项目文件夹

### 我在做什么？

在你的电脑桌面上新建一个文件夹，往里面放三个文件，模拟一个真实的小项目。这个文件夹就是你的"工作区"——你写代码的地方。

### 第一步：打开文件资源管理器

**Windows 用户：**
看看你屏幕最下方那条横着的长条（这叫"任务栏"），上面有一排小图标。找到一个**黄色的文件夹图标**——这就是"文件资源管理器"。把鼠标移上去，左键单击一下，一个窗口就弹出来了。

**Mac 用户：**
看看你屏幕最下方那条横着的长条（这叫"Dock 栏"），上面有一排图标。找到一个**蓝白相间的笑脸图标**——这就是"访达（Finder）"。把鼠标移上去，左键单击一下。

### 第二步：导航到桌面

弹出来的窗口左边有一列名字。在这一列里找到**"桌面"**（Windows）或**"Desktop"**（Mac）这几个字，鼠标左键单击它。现在窗口中间显示的就是你桌面上的所有文件和文件夹。

### 第三步：新建文件夹

在窗口中间**空白的地方**（不要点到任何文件或文件夹），**鼠标右键单击**，会弹出一个菜单。

把鼠标移到"**新建**"（Windows）或"**新建文件夹**"（Mac）上——这个菜单项旁边可能有一个小三角，说明它下面还有更多选项。不要急，鼠标不要移开，等子菜单出来。在子菜单里找到**"文件夹"**（Windows）或直接点"新建文件夹"（Mac），**鼠标左键单击**。

这时窗口里会出现一个新文件夹，名字是蓝色高亮的"新建文件夹"几个字。不用鼠标点别的地方，**直接在键盘上打字**：`my-first-project`。然后按键盘上的 **Enter（回车键）**。

> Windows 用户注意：打完字之后名字还在蓝色高亮状态——这就对了，直接按 Enter 就好。如果你在打字之前不小心在别的地方点了一下鼠标，蓝色高亮消失了，就**右键单击这个文件夹 → 重命名 → 输入 `my-first-project` → 按 Enter**。

### 第四步：进入文件夹

**鼠标左键双击**这个刚建好的 `my-first-project` 文件夹。窗口内容变空了——因为你还没往里面放任何东西。保持这个窗口开着，下面我们要在里面创建三个文件。

---

### 创建文件1：`README.md`

**步骤1：新建文件**

在 `my-first-project` 文件夹窗口的空白处**鼠标右键单击** → **新建** → **文本文档**。

出现一个名叫"新建文本文档.txt"的文件，名字处于蓝色高亮状态。**直接打字**：`README.md`。

> ⚠️ Windows 会弹出一个警告框，上面写着："**如果改变文件扩展名，可能会导致文件不可用。确实要更改吗？**" 别被吓到——这句话的意思是"你确定要把 .txt 改成 .md 吗？"我们就是要改扩展名。**鼠标左键单击"是"按钮**。

现在文件名变成了 `README.md`，文件图标也可能变了——这是正常现象。`.md` 的意思是 Markdown 文件，一种写文档的格式。

**步骤2：打开文件**

**鼠标左键双击** `README.md` 文件。Windows 会用"记事本"程序打开它（一个白色背景的简单文字编辑器）。Mac 会用"文本编辑"打开。

**步骤3：粘贴内容**

下面这个灰色框里的内容，就是你要放进文件里的文字。怎么复制？

- 把鼠标移到下面灰色框的第一行文字开头，**按住鼠标左键不放**，往下拖动鼠标，直到灰色框里的所有文字都被选中（文字背景变成蓝色）
- 松开鼠标左键
- 在键盘上**同时按下 Ctrl 键和 C 键**（Mac 用户按 Cmd 键和 C 键）——这叫"复制"。你选的文字已经存到了电脑的"剪贴板"里

```markdown
# My First Project

这是我的第一个GitHub项目！

## 功能

- 输出一句问候语
- 运行 `go run main.go`
```

现在回到记事本窗口。在记事本的白色区域里**鼠标左键单击一下**，确保光标在闪。然后在键盘上**同时按下 Ctrl 键和 V 键**（Mac 用户按 Cmd 键和 V 键）——这叫"粘贴"。你刚才复制的文字就出现在记事本里了。

**步骤4：保存并关闭**

在记事本窗口的**左上角**，找到"**文件**"两个字，鼠标左键单击 → 在弹出菜单里找到"**保存**"，鼠标左键单击。然后点击窗口**右上角的 × 按钮**，关掉记事本。

✅ 第一个文件完成！

---

### 创建文件2：`.gitignore`

这个文件名有点特殊——它**以一个点 `.` 开头**。Windows 系统对这种文件名比较挑剔。

**步骤1：新建文件**

和创建 README.md 一样，在文件夹空白处**右键单击 → 新建 → 文本文档**。

**步骤2：命名（Windows 特殊技巧）**

名字高亮时，鼠标**不要点别的地方**，直接打字：`.gitignore.`

注意！最后面**多打了一个点**！完整输入的是：`.gitignore.`

按键盘上的 **Enter 键**。Windows 会**自动去掉末尾那个多余的点**，最终文件名变成 `.gitignore`。

> 🤔 为什么要在末尾多加一个点？Windows 不允许文件以点开头、以点结尾（它会认为"你没有输入文件名"），但它允许以点开头、后面有内容，再以点结尾——然后自动把结尾的点去掉。这就是绕过 Windows 限制的"作弊手法"。

如果上面的方法没有成功（文件名还是带 .txt），还有备用方法：
- 打开记事本 → 鼠标单击"文件" → "另存为" → 在文件名框里输入 `.gitignore.` → 保存类型选"所有文件" → 保存

**步骤3：粘贴内容**

**鼠标左键双击** `.gitignore` 文件 → 打开记事本。把下面灰色框里的内容复制粘贴进去：

```
*.exe
*.test
*.out

vendor/
```

**步骤4：保存并关闭**

点"文件"→"保存"→ 关掉窗口。

### 创建文件3：`main.go`

**步骤1：新建文件**

在文件夹空白处**右键单击 → 新建 → 文本文档**。

**步骤2：命名**

名字高亮时，直接打字：`main.go`，按 **Enter**。

Windows 又会弹出"改变扩展名可能导致文件不可用"的警告——**点"是"**。

**步骤3：粘贴内容**

双击打开 `main.go`，把下面灰色框里的内容完整复制粘贴进去：

```go
package main

import "fmt"

func main() {
	fmt.Println("Hello, GitHub!")
	fmt.Println("这是我的第一个GitHub项目！")
}
```

**步骤4：保存并关闭**

点"文件"→"保存"→ 关掉窗口。

### 我做得对不对？

你的 `my-first-project` 文件夹窗口里现在应该有**三个文件**：

```
.gitignore
main.go
README.md
```

> ⚠️ `.gitignore` 在某些情况下可能"看不见"。Windows 默认会隐藏以点开头的文件。如果窗口里只看到两个文件，在窗口顶部的菜单栏找到"**查看**"→ 把"**隐藏的项目**"前面的勾打上（鼠标左键单击），`.gitignore` 就出现了。

三个文件的内容你也检查一下：依次双击打开每个文件，确认里面的文字和上面灰色框里的一模一样。

### 不对怎么办？

❌ **文件1：README.md 文件图标变成了浏览器的图标**
这是正常的。`.md` 是 Markdown 文件的扩展名，如果你的电脑装了能打开 Markdown 的软件（比如 Typora、VS Code），图标会变。**内容没变就行**。

❌ **文件2：`.gitignore` 创建失败，文件名还是 .txt**
用备用方法：打开记事本 → 文件 → 另存为 → 文件名输入 `.gitignore.` → 保存类型选"所有文件"（不是"文本文档"）→ 保存位置选 `my-first-project` 文件夹 → 保存。

❌ **文件3：双击 `main.go` 打开的不是记事本**
如果你的电脑装了 Go 开发工具，`.go` 文件可能会用别的程序打开。没关系——只要能让鼠标右键单击文件 → "打开方式" → "记事本"，也一样能查看和编辑内容。

## 3.2 让 Git 接管这个文件夹

### 我在做什么？

现在你有一个文件夹和三个文件。接下来要让 **Git** 这个程序来"管理"这个文件夹。Git 就是代码的"时光机"——它能记住你每次修改了什么、谁改的、什么时候改的。

### 工具准备：打开终端

**终端（Terminal）** 是一个黑底白字的窗口，你可以在里面打字向电脑发号施令。这就像你和电脑之间有一个"文字聊天窗口"——你打一句命令，电脑执行并回复结果。

**Windows 用户：打开 PowerShell**

（如果你还记得第0章教的方法，直接用那个就行。如果忘了，这里再讲一遍：）

按下键盘左下角的 **Windows 键**（四个小方块的图案），屏幕左下角会弹出一个菜单。不用管菜单，**直接在键盘上打字**：`PowerShell`。你会看到菜单上方出现一个搜索框，里面显示着你打的字，下方出现一个蓝色图标的"Windows PowerShell"。用鼠标点击它，一个蓝色或黑色背景的窗口就弹出来了——这就是 PowerShell 终端。

**Mac 用户：打开终端**

（如果你还记得第0章教的方法，直接用那个就行。如果忘了：）

同时按下键盘上的 **Command 键（⌘）** 和**空格键**，然后松开。屏幕中央会出现一个搜索框。在搜索框里打字：`Terminal`，然后按 **Enter 键**。一个白色背景（深色模式下是黑色）的窗口弹出来——这就是终端。

---

### 操作1：让终端进入你的文件夹（`cd` 命令）

现在终端窗口里有一行字，最后是一个闪烁的光标。这就是电脑在等你发号施令。

**Windows 用户**，在终端里用键盘打字（注意大小写，不要打错）：
```
cd $env:USERPROFILE\Desktop\my-first-project
```
然后按 **Enter 键**。

**Mac 用户**，在终端里打字：
```
cd ~/Desktop/my-first-project
```
然后按 **Enter 键**。

**这条命令是什么意思？逐词拆开看：**

- `cd` = **c**hange **d**irectory（切换目录）。"目录"就是"文件夹"的专业叫法。这条命令的意思是："电脑，我要去另一个文件夹。"
- `$env:USERPROFILE\Desktop\my-first-project`（Windows）或 `~/Desktop/my-first-project`（Mac）= 你要去的**地址**。这就像告诉快递员"请去这个地址"。
  - `$env:USERPROFILE`（Windows）或 `~`（Mac）= 你的**用户主目录**。每个人的电脑上这个位置都不同，但电脑知道它在哪里。它大概等于 `C:\Users\你的用户名`（Windows）或 `/Users/你的用户名`（Mac）。
  - `Desktop` = 桌面文件夹
  - `my-first-project` = 你刚刚创建的那个项目文件夹

**这条命令连起来读就是**："电脑，从当前位置出发，先去我的用户主目录，再进 Desktop（桌面），再进 my-first-project 文件夹。"

怎么判断命令执行成功了？看光标前面那行字。执行前它可能长这样：
```
PS C:\Users\你的用户名>
```
执行后应该变成这样：
```
PS C:\Users\你的用户名\Desktop\my-first-project>
```
最后那个 `my-first-project` 出现了——说明你已经"站"在这个文件夹里了。

---

### 操作2：初始化 Git 仓库（`git init`）

在终端里打字（一样，注意大小写和空格）：
```
git init
```
按 **Enter 键**。

**逐词拆解：**
- `git` = 你在叫 Git 这个程序
- `init` = **init**ialize（初始化）的缩写。意思是"Git，请开始管理当前这个文件夹"

**你该看到什么？**

终端会输出类似下面这样的文字（路径是你的实际路径，不用一模一样）：
```
Initialized empty Git repository in C:/Users/你的用户名/Desktop/my-first-project/.git/
```

翻译成人话："已在 `C:/Users/.../my-first-project/.git/` 初始化了一个空的 Git 仓库。"

关注点：最后那个 `.git`——Git 在你文件夹里悄悄建了一个**隐藏文件夹**叫 `.git`。你所有的版本历史、每一次修改的记录都存在这里。如果把 `.git` 文件夹删了，Git 就不认识这个项目了（但你的 `README.md`、`main.go` 这些代码文件不会丢）。

---

### 操作3：把文件加入暂存区（`git add .`）

在终端里打字：
```
git add .
```
按 **Enter 键**。

**逐词拆解：**
- `git` = 叫 Git
- `add` = 添加。意思是"Git，注意这些文件，我要把它们纳入管理"
- `.` = 一个点。在计算机里，**一个点**代表"**当前文件夹里的所有东西**"。所以 `git add .` 翻译成人话就是："Git，把我当前文件夹里所有文件都加入暂存区。"

> 🤔 暂存区是什么？（此术语需进附录）
>
> 把 Git 的工作方式想象成**寄快递**：
> 1. 你写好代码 → 相当于你把东西放在桌上
> 2. `git add` → 相当于你把东西装进快递盒里、封好口
> 3. `git commit`（下一步）→ 相当于你在快递单上写下"这批东西是什么"、贴上单子，交给快递公司
>
> "暂存区"就是那个你装好、封好口、但还没写快递单的快递盒。

为什么需要暂存区？因为有时你改了 10 个文件，但只想提交其中 3 个——你可以只把要提交的 3 个 `git add` 进去。暂存区让你**精确控制每次提交包含哪些修改**。

---

### 操作4：提交到本地仓库（`git commit -m`）

在终端里打字（**引号不要漏掉，而且是英文引号不是中文引号！**）：
```
git commit -m "feat: 初始化项目"
```
按 **Enter 键**。

**逐词拆解：**
- `git` = 叫 Git
- `commit` = 提交。意思是"Git，把暂存区里的东西正式记录到版本历史里"
- `-m` = 这个参数告诉 Git "我接下来要把提交信息直接写在命令里"。`-m` 是 `--message` 的缩写。（此术语需进附录：Commit 提交）
- `"feat: 初始化项目"` = 提交信息（commit message）。每次提交都要写一句话，说明"这次改了什么"。`feat:` 是一个约定俗成的前缀，表示这是一个**新功能**（feature）。

**你该看到什么？**

终端会输出类似下面这样的文字：
```
[main (root-commit) abc1234] feat: 初始化项目
 3 files changed, 12 insertions(+)
 create mode 100644 .gitignore
 create mode 100644 README.md
 create mode 100644 main.go
```

翻译成人话：
- `[main (root-commit) abc1234]` = 这是在 `main` 分支上的第一次提交（root-commit），编号是 `abc1234`。你的编号会是一串不同的随机字符——这叫"哈希值"，相当于这次提交的身份证号。（此术语需进附录：哈希值 / Hash / 提交ID）
- `3 files changed, 12 insertions(+)` = 改了 3 个文件，新增了 12 行代码
- 下面三行列出了每个文件：都是"create mode"（新建），说明这是它们第一次被 Git 记录

### 我做得对不对？——验证

在终端里打字，做**两个验证**：

**验证1：`git status`**

打字：
```
git status
```
按 **Enter 键**。

**你该看到什么？**

```
On branch main
nothing to commit, working tree clean
```

翻译：
- `On branch main` = 你当前在 main 分支上（分支的事后面会讲）
- `nothing to commit, working tree clean` = "没有需要提交的东西，工作区是干净的"
- **干净 = 完美**。这说明你刚才的三个操作（init → add → commit）全部成功。

**验证2：`git log --oneline`**

打字：
```
git log --oneline
```
按 **Enter 键**。

**你该看到什么？**

类似这样：
```
abc1234 feat: 初始化项目
```

你的那串字符和 `abc1234` 不一样——这是正常的，每个人每次提交的编号都是唯一的。

`--oneline` 参数的作用是让日志**一行一条**，简洁清晰。不加这个参数的话，Git 会输出很详细的信息（谁提交的、什么时间、完整哈希值等），对新手来说信息量太大。

### 不对怎么办？

❌ **错误1：`git status` 显示 `fatal: not a git repository`**
完整报错：
```
fatal: not a git repository (or any of the parent directories): .git
```

**为什么？** 你还没执行 `git init`，或者你不在 `my-first-project` 文件夹里。没有 `.git` 隐藏文件夹，Git 不认识这个目录。

**怎么修？**
1. 打字 `pwd`（Mac）或 `Get-Location`（Windows PowerShell），按 Enter，看看当前在哪个目录
2. 如果不在 `my-first-project` 文件夹里，重新执行 `cd` 命令进入
3. 确认进入了再执行 `git init`

❌ **错误2：`git status` 显示文件是红色的 "Untracked files"**
完整输出类似：
```
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .gitignore
        README.md
        main.go
```

**为什么？** 你执行了 `git init`，但没有执行 `git add .`。文件还没有被加入暂存区。

**怎么修？** 执行 `git add .`，然后再 `git commit -m "feat: 初始化项目"`。

❌ **错误3：`git commit -m` 报错 "nothing to commit"**
完整报错：
```
nothing to commit, working tree clean
```

**为什么？** 你可能已经提交过一次了，或者没有文件被加入暂存区。

**怎么修？** 先 `git status` 看看状态。如果显示 `nothing to commit`，说明已经提交过了，不需要再提交。检查 `git log --oneline` 看有没有提交记录。

❌ **错误4：`git commit` 报错 "Please tell me who you are"**
完整报错：
```
*** Please tell me who you are.
Run
  git config --global user.email "you@example.com"
  git config --global user.name "Your Name"
```

**为什么？** 你是第一次在这台电脑上用 Git，Git 不知道你是谁。每条提交记录都要写上"谁提交的"。

**怎么修？** 逐行执行这两条命令（把邮箱和名字换成你自己的）：
```
git config --global user.name "你的名字"
git config --global user.email "你的GitHub注册邮箱"
```
然后再重新执行 `git commit -m "feat: 初始化项目"`。

## 3.3 关联远程仓库并推送

### 我在做什么？

现在你的代码还只在你自己的电脑里。你要把它们送到 GitHub 上你第2章创建的那个仓库里去。需要三步：①获取你自己仓库的 SSH 地址；②告诉本地 Git"远程仓库在哪里"；③确认你的分支名字和远程一致；④推送。

### 第1步：获取你自己仓库的 SSH 地址

**1.** 打开你的浏览器。在浏览器窗口**最顶部的那一长条白色输入框**（这叫"地址栏"）里，用键盘打字：`github.com/<你的用户名>/my-first-project`，然后按 **Enter 键**。你会进入你自己在第2章创建的仓库页面。

> 如果你在第2章给仓库取了别的名字（不是 `my-first-project`），这里输入你自己取的那个名字。

**2.** 页面加载出来后，找到页面中上偏右位置的**绿色长方形按钮**，上面写着"**▾ Code**"（▾ 是一个向下的三角箭头）。鼠标左键单击它——会弹出一个白色的小面板。

**3.** 小面板里面顶部有三个选项标签：HTTPS、SSH、GitHub CLI。鼠标左键单击"**SSH**"——选中后它的背景变成蓝色或有一个下划线。

**4.** SSH 标签下方有一个白色的长条框，里面有一行灰色文字，内容类似：
```
git@github.com:<你的用户名>/my-first-project.git
```
在这行灰色文字的右边，有一个**小小的方形按钮**（两个重叠的方块图标，这是"复制"按钮）。鼠标移上去，左键单击。按钮会短暂变成"✓"或显示"Copied!"——说明地址已经复制到剪贴板了。

> 如果你按了复制按钮但不确定是否成功，也可以用老办法：鼠标选中那行灰色文字（按住左键拖动），然后按键盘上的 **Ctrl + C**（Mac：Cmd + C）。

**5.** 别关浏览器，等会儿还要回来查看结果。

---

### 第2步：把远程地址告诉本地 Git（`git remote add`）

回到你的终端窗口。确保光标还在闪。打字（**直接把上一步复制的地址粘贴进去**——按 **Ctrl + V**（Mac：Cmd + V））：
```
git remote add origin git@github.com:<你的用户名>/my-first-project.git
```
按 **Enter 键**。

**逐词拆解：**
- `git` = 叫 Git
- `remote` = 远程。告诉 Git 我们要操作远程仓库相关的东西。（此术语需进附录：Remote 远程仓库）
- `add` = 添加。我们要添加一个新的远程仓库地址。
- `origin` = 这是你给这个远程地址起的**别名**（此术语需进附录：Origin）。你可以把它想成"那个远程仓库的快捷方式"。为什么叫 origin？因为大多数项目的代码都来自一个"源头"（origin），这是社区约定俗成的名字。你完全可以给它起别的名字，比如 `github`——但用 `origin` 的话，全世界开发者都能一眼看懂。
- `git@github.com:<你的用户名>/my-first-project.git` = 你自己仓库的 SSH 地址。Git 通过这个地址找到 GitHub 上你第2章创建的那个仓库。

> 🤔 想象一下：GitHub 上有几百万个仓库，你的电脑怎么知道要把代码送到哪个仓库去？`git remote add` 就是在回答这个问题——"看到 `origin` 这个名字了吗？它对应的是 `git@github.com:<你的用户名>/my-first-project.git` 这个地址。以后我说 `origin`，你就去这个地址。"

---

### 第3步：统一分支名为 `main`（`git branch -M main`）

在终端里打字：
```
git branch -M main
```
按 **Enter 键**。

**逐词拆解：**
- `git` = 叫 Git
- `branch` = 分支。我们要操作分支。（此术语需进附录：Branch 分支）
- `-M` = 强制重命名分支。**大写** M 表示"强制"——即使目标名字已经存在，也强行改过去。如果是小写 `-m`，遇到冲突会报错。
- `main` = 新名字。我们要把当前分支改名为 `main`。

**为什么需要这一步？** 故事是这样的：Git 刚诞生的时候，默认分支名叫 `master`。2020 年，GitHub 把默认分支名改成了 `main`。如果你的 Git 版本比较新（2.28 以上），`git init` 出来的默认分支已经是 `main` 了；如果是旧版本，还是 `master`。`git branch -M main` 这条命令**不管你的分支现在叫 master 还是 main，都会让它最终叫 main**——两边名字统一了，后面推送就不会出错。

---

### 第4步：推送！（`git push -u origin main`）

在终端里打字：
```
git push -u origin main
```
按 **Enter 键**。

**逐词拆解：**
- `git` = 叫 Git
- `push` = 推送。把你的本地提交"推"到远程仓库。（此术语需进附录：Push 推送）
- `-u` = `--set-upstream` 的缩写。意思是"设置上游分支"。（此术语需进附录：Upstream 上游）设好之后，Git 会记住"当前本地分支对应远程的 origin/main"。以后在这个分支上你只需要打 `git push`，不用再写一长串了。这就是"配置一次，省事一辈子"。
- `origin` = 推送到哪个远程仓库（就是第2步设的那个别名）
- `main` = 推送哪个本地分支

**你该看到什么？**

终端会输出一大段文字，类似这样（你的用户名会出现在地址中）：
```
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Delta compression using up to 8 threads
Compressing objects: 100% (3/3), done.
Writing objects: 100% (5/5), 432 bytes | 432.00 KiB/s, done.
Total 5 (delta 0), reused 0 (delta 0), pack-reused 0
To git@github.com:<你的用户名>/my-first-project.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

你不需要理解每一行。只需要看**最后两行**：

- `[new branch] main -> main` = 在远程仓库上新建了一个叫 `main` 的分支，把本地的 `main` 分支内容送过去了
- `branch 'main' set up to track 'origin/main'` = 上游分支设置成功，以后 `git push` 就知道往哪推了

**没有看到 `error` 或 `fatal` 字样 = 成功了！**

---

### 第5步：回到浏览器验证

回到你的浏览器，在 `github.com/<你的用户名>/my-first-project` 这个页面上，**按键盘上的 F5 键**（或者浏览器地址栏左边那个圆形箭头图标——"刷新"按钮）。页面重新加载后，你应该看到：

- 页面中间不再是"Quick setup"引导界面了
- 出现了一个文件列表，里面有**三个文件**：`.gitignore`、`main.go`、`README.md`
- `README.md` 的内容被渲染显示在文件列表下方——你刚才写的标题"# My First Project"显示成了大号加粗文字
- `main.go` 的代码带有**语法高亮**（关键字比如 `package`、`func` 有不同颜色）

**🎉 恭喜！你的代码已经成功推送到 GitHub 了！** 互联网上任何一个人都可以通过 `github.com/<你的用户名>/my-first-project` 看到你的代码——这是属于你自己的仓库！

---

### 我做得对不对？——快速验证

- 终端在执行 `git push` 后没有出现 `error` 或 `fatal` 字样
- 浏览器刷新仓库页面后，能看到三个文件
- 点击 `main.go` 文件名，进去能看到代码内容

三条都满足 = 完美。

## 3.4 常见错误与排错

以下是最容易遇到的 5 种错误。每种都包含：完整的报错文字、通俗解释、以及修复步骤。

---

### ❌ 错误1：`remote origin already exists`

**完整报错：**
```
error: remote origin already exists.
```

**通俗解释：** 你已经添加过一个叫 `origin` 的远程地址了。可能是之前试过一次 `git remote add`，或者是克隆别人的仓库带进来的。Git 不允许两个远程仓库用同一个别名。

**怎么修？**

先看看现在的 origin 指向哪里——在终端里打字：
```
git remote -v
```
按 **Enter 键**。

你会看到类似：
```
origin  git@github.com:某某/某某.git (fetch)
origin  git@github.com:某某/某某.git (push)
```

如果显示的地址不是你自己仓库的地址（`git@github.com:<你的用户名>/my-first-project.git`），那就改掉它。打字：
```
git remote set-url origin git@github.com:<你的用户名>/my-first-project.git
```
按 **Enter 键**。

再执行 `git remote -v` 确认地址已经改了，然后继续推送。

如果嫌麻烦，也可以先删掉再重建。打字：
```
git remote remove origin
```
按 Enter，然后再：
```
git remote add origin git@github.com:<你的用户名>/my-first-project.git
```
按 Enter。

---

### ❌ 错误2：`failed to push some refs`（远程有本地没有的内容）

**完整报错：**
```
To git@github.com:<你的用户名>/my-first-project.git
 ! [rejected]        main -> main (fetch first)
error: failed to push some refs to 'git@github.com:<你的用户名>/my-first-project.git'
hint: Updates were rejected because the remote contains work that you do
hint: not have locally.
```

**通俗解释：** 远程仓库里有内容（比如 README.md、.gitignore），而你的本地没有这些内容。Git 拒绝直接覆盖——就像一个快递员不能把快递丢到已经有东西的柜子里（会压坏别人的东西）。这种情况最常见的原因是你第2章创建仓库时不小心勾选了某个初始化复选框（状态B或状态C）。

**怎么修？** 先把远程的内容"拉"到本地合并，再推送。

在终端里打字：
```
git pull origin main --rebase
```
按 **Enter 键**。

- `git pull` = 拉取。（此术语需进附录：Pull 拉取）和 push 相反——把远程的代码"拉"到本地
- `origin main` = 从 origin 这个远程仓库的 main 分支拉取
- `--rebase` = 把你的本地提交"接"在远程提交后面，保持历史是一条直线（此术语需进附录：Rebase 变基）

执行完后，再推送：
```
git push -u origin main
```
按 **Enter 键**。

如果 `git pull --rebase` 过程中出现了冲突（conflict），终端会提示哪些文件冲突了。这种情况在第一次推送时很少见。如果发生了，打开冲突文件，手动修改冲突标记（`<<<<<<<`、`=======`、`>>>>>>>`），保存后执行：
```
git add .
git rebase --continue
git push -u origin main
```

---

### ❌ 错误3：`Permission denied (publickey)`（SSH 密钥问题）

**完整报错：**
```
git@github.com: Permission denied (publickey).
fatal: Could not read from remote repository.

Please make sure you have the correct access rights
and the repository exists.
```

**通俗解释：** GitHub 不认识你的电脑。就像你拿了一把别人的钥匙去开自己家的门——锁不认这把钥匙。

**怎么修？** 回到[步骤1 · 环境准备与SSH密钥配置](01-环境准备与SSH.md)，重新检查：

1. 在终端里打字 `ssh -T git@github.com`，按 Enter。如果输出 `Hi <你的用户名>! You've successfully authenticated...`，说明密钥没问题——可能是仓库地址写错了。如果不成功，继续往下。
2. 打开浏览器，进入 GitHub → 右上角头像 → Settings → 左侧 "SSH and GPG keys" → 看看列表里有没有你的公钥。如果没有，回到步骤1重新添加。
3. 如果公钥在但连接还是失败，在终端里打字 `ssh -vT git@github.com`，按 Enter。`-v` 参数会打印详细调试信息，帮助定位问题。

---

### ❌ 错误4：`src refspec main does not match any`

**完整报错：**
```
error: src refspec main does not match any
error: failed to push some refs to 'git@github.com:<你的用户名>/my-first-project.git'
```

**通俗解释：** 你在命令里写了 `main`，但你的本地分支不叫 `main`（可能叫 `master` 或其他名字）。Git 找不到叫 `main` 的分支。

**怎么修？** 先确认你当前的分支叫什么：

打字 `git branch`，按 **Enter 键**。

输出类似：
```
* master
```
（前面带 `*` 号的是当前分支）

如果显示 `* master`，执行改名：
```
git branch -M main
```
按 **Enter 键**。

再确认一下：
```
git branch
```
现在应该显示 `* main`。然后重新推送：
```
git push -u origin main
```
按 **Enter 键**。

---

### ❌ 错误5：`403 Forbidden`（SSH 密钥对应了不同的 GitHub 账号）

**完整报错（HTTPS 方式）：**
```
remote: Permission to <你的用户名>/my-first-project.git denied to 其他用户名.
fatal: unable to access 'https://github.com/<你的用户名>/my-first-project.git/': The requested URL returned error: 403
```
**或者（SSH 方式下的类似错误）：**
```
ERROR: Permission to <你的用户名>/my-first-project.git denied to 其他用户名.
fatal: Could not read from remote repository.
```

**通俗解释：** 你的电脑上有多个 GitHub 账号的 SSH 密钥，Git 使用的那把钥匙对应的是账号 A，但你要推送的仓库属于账号 B，而账号 A 没有写入权限。

**怎么修？**

1. 先在终端里打字 `ssh -T git@github.com`，按 Enter。看输出的那句话是 `Hi 哪个用户名!`。如果这个用户名不是你的，说明连错账号了。
2. 检查你的 SSH 密钥配置。打字 `ls ~/.ssh/`，看看有几个密钥文件。如果创建了多个，可能需要配置 SSH config 文件来指定不同 GitHub 账号用不同密钥（这部分稍微高级，可以搜索"GitHub 多账号 SSH 配置"）。

---

## 🤔 想多一点：`origin` 为什么叫 `origin`？

`origin` 不是 Git 的保留关键字——它只是一个**约定俗成的名字**。你完全可以把它改成任何名字：

```
git remote add github git@github.com:<你的用户名>/my-first-project.git
git push -u github main
```
这两行完全合法，功能一模一样。

那为什么大家都用 `origin`？因为当你用 `git clone` 克隆一个仓库时，Git **自动**把克隆来源命名为 `origin`。于是整个 Git 社区就形成默契：`origin` = "这是我代码的来源，也是我要推送回去的目标"。

那 `upstream` 又是什么？当你 fork（复刻，步骤6会讲到）了别人的开源项目，你会把原作者的仓库加为 `upstream`：
```
git remote add upstream git@github.com:原作者/原仓库.git
```
这样 `origin` 指向你的复刻，`upstream` 指向原仓库。你从 upstream 拉取更新，推到自己 origin——两条线互不干扰。
