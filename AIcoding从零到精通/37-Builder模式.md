## 37.1 Builder到底是什么？——一句话的威力

回到那个比喻：

> **Builder是建筑师**——你告诉它"我要一栋三层小楼，有车库、有花园、三室两厅"，它给你画图纸、打地基、搭框架。

但Builder实际做了什么呢？让我们拆开来看：

**Builder = 需求理解 + 项目规划 + 代码生成 + 文件管理**

| 阶段 | 你做什么 | Builder做什么 |
|------|---------|-------------|
| ① 需求理解 | 输入一句话描述 | 用自己的话复述需求，确认理解正确 |
| ② 项目规划 | 确认或修改规划 | 列出项目结构、技术选型、文件清单 |
| ③ 交互确认 | 回复"可以"或"不对，改一下" | 根据你的反馈调整规划 |
| ④ 代码生成 | 坐等 | 自动创建所有文件，写入代码 |
| ⑤ 预览/运行 | 运行项目 | 告诉你如何运行，哪里需要注意 |

**和Cursor的Composer最大的区别**：Cursor的Composer是"你描述 → AI直接生成代码"，没有中间的"规划确认"环节。Builder多了一个**"让你审查规划"**的步骤——这很重要，因为一旦方向错了，后面全错。

## 37.2 Builder的完整工作流程（拆解版）

让我们用一个真实例子，一步步拆解Builder的工作流程。

### 场景：用Builder做一个"个人记账工具"

**第1步：打开Builder，输入需求**

在Trae的AI面板，点击"Builder"标签，输入：

```
帮我做一个个人记账工具，Web版。
后端用Python Flask，前端用HTML+CSS+JS。
功能：记录收支、查看月度统计、分类管理（餐饮/交通/购物/其他）。
数据保存在本地SQLite数据库。
界面要好看一点，简洁风格。
```

> **📖 术语：Flask**——Python的一个轻量级Web框架。你可以把它理解为"搭网站用的积木套件"——它帮你处理HTTP请求、路由、模板这些繁琐的事情，你只需要关心你的业务逻辑。类比：盖房子用的脚手架，Flask就是搭网站用的脚手架。

**第2步：Builder复述需求**

Builder会回复类似这样的内容：

```
我理解你的需求：
- 一个Web版的个人记账工具
- 后端：Python Flask
- 前端：HTML + CSS + JS
- 功能：记录收支、月度统计、分类管理
- 数据库：SQLite

我计划创建以下项目结构：
[项目文件列表...]

这个结构可以吗？
```

**第3步：审查规划（关键步骤！）**

⚠️ **这是Builder最有价值的地方，也是最容易被忽略的地方。**

很多人看到Builder的规划，扫一眼就点"确认"——然后生成的项目和预期不一样，又回来骂Builder不行。

**正确的做法**：仔细看Builder列出的文件结构，问自己三个问题：
- 功能全了吗？（我有没有漏掉什么需求？）
- 结构合理吗？（每个文件负责的职责清楚吗？）
- 技术选型对吗？（Flask做后端合适吗？）

如果发现问题，现在就说：
```
不对，我忘了说还需要"导出CSV报表"功能，请加上。
```
或者：
```
数据库用SQLite可以，但能不能再加一个JSON备份功能？
```

**第4步：确认后，Builder自动生成所有文件**

你确认后，Builder会：
1. 创建项目文件夹
2. 创建所有 `.py`、`.html`、`.css`、`.js` 文件
3. 写入代码（不是空壳，是真正的、能运行的代码）
4. 创建数据库初始化脚本
5. 告诉你如何安装依赖、如何运行

**第5步：运行验证**

```bash
cd 项目文件夹
pip install flask        # 安装Flask
python app.py            # 启动应用
```

打开浏览器，访问 `http://localhost:5000`，你应该看到一个能用的记账工具。

## 37.3 Builder的四大使用技巧

### 技巧一：描述越具体，结果越准

这是Builder最重要的使用原则。对比一下：

| 描述方式 | 效果 |
|---------|------|
| ❌ "帮我做一个博客" | Builder不知道你要什么类型的博客、用什么技术、有哪些功能 |
| ✅ "帮我做一个博客网站，Python Flask后端，HTML前端，支持发布文章、文章分类、评论功能，界面简洁" | Builder清楚知道技术栈、功能列表、风格偏好 |

**描述公式**：**做什么 + 用什么技术 + 有哪些功能 + 有什么特殊要求**

```
帮我做一个[项目类型]，[技术栈]，[功能列表]，[风格/其他要求]
```

### 技巧二：利用"规划审查"环节

Builder问"这个结构可以吗"的时候，不要急着确认。这是你**唯一一次免费改需求的机会**。一旦确认，Builder就开始生成代码了，再改需求就得从头来。

**审查清单**：
- [ ] 文件数量合理吗？（3-8个文件是正常的，太多或太少都不好）
- [ ] 每个文件职责清楚吗？（一个文件不应该负责太多事情）
- [ ] 技术栈是你想要的吗？（Builder有时候会默认选它熟悉的技术）
- [ ] 功能列表完整吗？（有没有漏掉你想要的？）

### 技巧三：Builder擅长什么，不擅长什么

| Builder擅长 | Builder不擅长 |
|------------|-------------|
| 从零新建项目 | 修改已有项目（改一两行代码） |
| 标准项目结构（Web应用、CLI工具、API服务） | 非常规架构（特殊框架、老旧技术栈） |
| 小型到中型项目（10-30个文件） | 大型项目（50+文件，Builder会力不从心） |
| 快速原型（想看看效果） | 生产级代码（需要精细的错误处理、安全加固） |

**如果你要改一个已有的大型项目**，不要用Builder。用Cursor的Chat或Agent模式。Builder是"从零建新房子"的，不是"装修老房子"的。

### 技巧四：Builder生成的项目跑不通？用Chat修复

这是Trae的"自愈循环"（第36章已经提过）：

```
Builder生成 → 跑不通 → 切到Chat → 粘贴错误 → Chat修复 → 再跑 → 直到成功
```

你不需要自己Debug。Builder生成的项目，如果跑不通，**把错误信息直接给Trae Chat**，它会自己修自己写出来的代码。

## 37.4 Builder vs Cursor Composer：谁更擅长什么

Cursor也有一个类似Builder的功能，叫 **Composer**（`Ctrl + I`）。它们的思路相似但实现不同：

| 维度 | Trae Builder | Cursor Composer |
|------|-------------|-----------------|
| **入口** | AI面板 → Builder标签页 | `Ctrl + I` 快捷键 |
| **交互方式** | 先规划 → 你确认 → 再生成 | 你描述 → 直接生成，没有规划确认环节 |
| **项目感知** | 默认读所有文件 | 需要手动@文件 |
| **生成方式** | 一次性生成所有文件 | 逐个文件生成，你可以看到每个文件的diff |
| **修改能力** | 弱（主要做新建） | 强（可以新建，也可以修改现有代码） |
| **适合场景** | 从零新建项目 | 新建项目 + 修改现有项目 |
| **中文支持** | 母语级 | 很好 |
| **速度** | 快（规划一次，生成全部） | 中等（逐个文件生成，可以看到细节） |

**简单总结**：
- **从零新建项目** → 用Trae Builder（快速、省心、中文好）
- **修改现有项目、加新功能** → 用Cursor Composer（精细、可控、能看到每一步改动）

## 37.5 Builder实战：三个典型场景

### 场景一：快速原型——"我想看一个东西长什么样"

**需求**：你想做一个"随机名言生成器"网页，用户点一下按钮，随机显示一条名言。

**用Builder**：
```
帮我做一个随机名言生成器网页。
单页HTML，不需要后端。
包含：一个显示名言的区域、一个"换一条"按钮、名言从预设的20条里随机选。
界面要好看，简洁风格，居中布局。
```

**结果**：Builder生成一个 `index.html`（包含CSS和JS），双击打开就能用。**全程不到2分钟。**

### 场景二：小工具——"帮我做一个实用的小程序"

**需求**：你想做一个"文件批量重命名"工具，Python命令行。

**用Builder**：
```
帮我做一个Python命令行工具，功能是批量重命名文件。
支持：指定文件夹路径、添加前缀、添加后缀、替换文件名中的某个字符串。
要有详细的命令行参数说明（用argparse）。
加上错误处理：文件夹不存在、文件重名等情况。
```

**结果**：Builder生成一个Python脚本，包含 `argparse` 参数解析、文件遍历、重命名逻辑、错误处理。**可以直接用。**

### 场景三：学习项目——"我想看一个完整项目的代码结构"

**需求**：你想学Flask，想看看一个完整的Flask项目长什么样。

**用Builder**：
```
帮我做一个Flask学习项目：一个简单的图书管理系统。
功能：添加图书、查看全部图书、搜索图书、删除图书。
数据库用SQLite，ORM用SQLAlchemy。
项目结构要标准：app.py主入口、models.py数据模型、routes.py路由、templates文件夹放HTML模板。
请加上详细的中文注释，方便我学习。
```

**结果**：Builder生成一个标准Flask项目，包含完整的MVC结构。你打开每个文件，看代码+注释，就能理解Flask项目是怎么组织的。

## 37.6 ⚠️ 如果出错了怎么办

### 🔴 问题1：Builder生成的项目跑不起来

**症状**：`ModuleNotFoundError: No module named 'flask'` 或类似的依赖缺失错误。

**原因**：Builder生成的代码引用了第三方库，但你的电脑上还没装。

**解决方案**：
1. 看Builder生成项目时，有没有提示"需要安装的依赖"。通常它会列出来
2. 如果没有，看代码里 `import` 了什么，手动安装：
   ```bash
   pip install flask        # 如果代码里有 import flask
   pip install requests     # 如果代码里有 import requests
   ```
3. 如果Builder生成了 `requirements.txt` 文件，直接：
   ```bash
   pip install -r requirements.txt
   ```
4. 装完依赖后再运行

### 🔴 问题2：Builder生成的项目结构不合理

**症状**：所有代码堆在一个文件里，或者文件太多太碎。

**原因**：Builder的规划阶段你没仔细审查，直接点了确认。

**解决方案**：
1. **在规划阶段就纠正**：Builder问"这个结构可以吗"的时候，告诉它：
   ```
   结构太简单了，请把数据模型、业务逻辑、界面分开放在不同文件里
   ```
   或者：
   ```
   文件太多了，请合并一些，保持在5个文件以内
   ```
2. **如果已经生成了**：在Chat里告诉AI：
   ```
   这个项目的文件结构不太合理，请帮我重构一下，把[具体问题]改成[你想要的]
   ```

### 🔴 问题3：Builder理解错了我的需求，生成的和我想的不一样

**症状**：你让它做"命令行工具"，它生成了"网页版"；你让它做"记账工具"，它做了"待办事项"。

**原因**：需求描述不够精确，或者关键词有歧义。

**解决方案**：
1. **在规划阶段就发现**：Builder复述需求时，如果不对，立刻纠正
2. **如果已经生成了**：重新开始。在Builder输入框里，**换一种方式描述需求**：
   - ❌ "帮我做一个记账工具"（上次这个描述被误解了）
   - ✅ "帮我做一个个人财务记账Web应用，用户可以输入收入和支出金额、选择分类（餐饮/交通/购物/其他）、查看月度收支统计。用Python Flask做后端，HTML页面做前端。**不是待办事项工具，是记账工具。**"
3. **加上否定句**：明确告诉Builder"不要什么"，有时候比"要什么"更有效

### 🔴 问题4：Builder生成的代码质量太差

**症状**：代码能跑，但写得很乱——变量名随便起、没有注释、逻辑混乱。

**原因**：Builder的目标是"快速生成能跑的代码"，不是"生成高质量代码"。这是设计取舍。

**解决方案**：
1. **在需求里明确要求代码质量**：
   ```
   请写规范的代码：变量名要有意义、函数要有注释、不要有重复代码
   ```
2. **生成后，切到Cursor进行代码审查和优化**（这就是双工具协作的价值）
3. **或者用Trae Chat**：把代码粘贴给Chat，让它"优化这段代码，加上注释，改善变量名"

## 小结

| 你学到了什么 | 一句话 |
|-------------|--------|
| Builder是什么 | 需求理解 → 项目规划 → 代码生成 → 文件管理，四合一的项目生成引擎 |
| 工作流程 | 描述需求 → Builder复述 → 你审查规划 → 确认 → 自动生成所有文件 |
| 规划审查 | Builder最有价值但最容易被忽略的环节，方向错了全错 |
| 描述公式 | 做什么 + 用什么技术 + 有哪些功能 + 有什么特殊要求 |
| Builder擅长 | 从零新建项目、快速原型、标准项目结构 |
| Builder不擅长 | 修改已有项目、大型项目、非常规架构 |
| 自愈循环 | Builder生成 → 跑不通 → Chat修复 → 再跑 |
| vs Cursor Composer | Builder先规划后生成，Composer直接生成；Builder做新建，Composer能新建也能改 |
