## 一、目标与完成效果

**一句话目标**：给博客接口加上"门禁"——创建、修改、删除文章必须登录，查看文章不需要登录。用 FastAPI 的依赖注入（`Depends`）实现，而不是全局中间件。

**完成后的可观测效果**：
- 你不带 token 直接 `curl -X POST /posts`，返回 **401 Unauthorized**——被拒之门外。
- 你先登录拿到 token，再带 token 发 `curl -X POST /posts`，成功创建文章。
- 你更新和删除文章也需要 token，没有 token 一样返回 401。
- 但 `GET /posts` 和 `GET /posts/{id}` 不用 token 也能正常访问——查看文章对所有人开放。
- 你能给朋友讲清楚：为什么用依赖注入而不是中间件？因为中间件对**所有请求**生效，但查看文章不需要登录——依赖注入让每个接口自己决定要不要保护。

## 三、分步操作

### 步骤 1：需求明确——哪些接口要保护？

在上一章（教程 15），我们实现了登录功能：用户发用户名和密码，服务器验证通过后返回一个 JWT token。但光有 token 没有用——你得在实际接口里**检查**这个 token 才叫"保护"。

**我们博客系统的权限规则**：

| 接口 | 操作 | 是否需要登录 |
|------|------|-------------|
| `GET /posts` | 查看文章列表 | ❌ 不需要（谁都能看） |
| `GET /posts/{id}` | 查看单篇文章 | ❌ 不需要（谁都能看） |
| `POST /posts` | 创建文章 | ✅ 需要登录 |
| `PUT /posts/{id}` | 更新文章 | ✅ 需要登录 |
| `DELETE /posts/{id}` | 删除文章 | ✅ 需要登录 |

**比喻**：你的博客就像一家图书馆——看书（GET）随便进，但要写书、改书、撕书（POST/PUT/DELETE），必须出示借书证（token），证明你是注册用户。

> 🤔 想多一点：为什么查看文章不限制？这不是"随便让人看"吗？

**因为博客的天然属性就是"公开阅读"**。如果连看都要登录，那就不叫博客，叫"内部文档系统"了。权限保护的粒度取决于你的业务需求——不是所有接口都要保护，只保护需要保护的那些。

### 步骤 2：`OAuth2PasswordBearer` —— 门禁刷卡器

我们要保护接口，第一步得从 HTTP 请求中**提取 token**。token 在哪？在 HTTP 请求头 `Authorization` 里，格式是 `Bearer <token>`。

**比喻**：`OAuth2PasswordBearer` 就像一个**门禁刷卡器**。每个请求进来，它自动从请求中"刷"出 token。如果请求里没带 token，它直接返回 401，根本不让你进到后面的逻辑。

我们先在 `blog_backend/auth.py` 中创建这个"刷卡器"：

```python
# blog_backend/auth.py —— 在文件末尾追加以下内容

from fastapi.security import OAuth2PasswordBearer  # 新增导入

# 门禁刷卡器：告诉 FastAPI "token 在 Authorization Header 里，去找 /login 接口拿"
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/login")  # 此术语需进附录：OAuth2PasswordBearer
```

**逐行解释**：

- `from fastapi.security import OAuth2PasswordBearer`：FastAPI 内置的安全工具，专门用来从请求中提取 Bearer token。
- `OAuth2PasswordBearer(tokenUrl="/login")`：创建一个"刷卡器"实例。`tokenUrl="/login"` 的意思是：如果用户没带 token 就访问受保护的接口，FastAPI 会在 401 响应里告诉客户端"去 `/login` 登录拿 token"。
- 这个 `tokenUrl` 参数只是**提示信息**，不会自动跳转，只影响 401 响应体的内容。

**`oauth2_scheme` 怎么用？**

在任何一个需要 token 的函数里，声明一个参数 `token: str = Depends(oauth2_scheme)`，FastAPI 就会自动从请求头里提取 token 给你：

```python
# 伪代码：任何需要 token 的地方
async def some_protected_function(token: str = Depends(oauth2_scheme)):
    # token 已经是解码后的字符串了，比如 "eyJhbGciOi..."
    # 你不需要手动解析 HTTP 请求头
    pass
```

**比喻回顾**：`OAuth2PasswordBearer` 就是写字楼门口的刷卡闸机。你没带卡（token），闸机不开（401）。你带了卡，闸机把卡号（token 字符串）递给后面的保安（下一步的 `get_current_user`），保安再查你是不是本写字楼的员工（查数据库）。

### 步骤 3：`get_current_user` —— 保安查工牌

有了 token 字符串还不够——我们得验证这个 token 是谁的、是否有效。这就是 `get_current_user` 函数的工作。

**比喻**：`get_current_user` 是写字楼大堂的**保安**。刷卡器（`oauth2_scheme`）把卡号（token）递给他，他做三件事：
1. 验卡的真伪（解码 JWT，检查签名是否有效、是否过期）
2. 从卡里读出员工姓名（从 JWT 的 payload 中提取 `username`）
3. 查员工花名册，确认这个人确实是在职员工（查数据库，看用户是否存在）

在 `blog_backend/auth.py` 中追加 `get_current_user` 函数：

```python
# blog_backend/auth.py —— 继续追加

from fastapi import Depends, HTTPException, status  # 新增导入
from sqlalchemy.orm import Session  # 新增导入
from blog_backend.database import get_db  # 新增导入
from blog_backend.models import User  # 新增导入

# 这个文件顶部的导入汇总（完整版）：
# from datetime import datetime, timedelta
# from jose import JWTError, jwt
# from passlib.context import CryptContext
# from fastapi.security import OAuth2PasswordBearer
# from fastapi import Depends, HTTPException, status
# from sqlalchemy.orm import Session
# from blog_backend.database import get_db
# from blog_backend.models import User


# 保安：验证 token → 查数据库 → 返回当前用户
async def get_current_user(
    token: str = Depends(oauth2_scheme),  # 刷卡器提取 token
    db: Session = Depends(get_db),        # 数据库会话
) -> User:
    """从请求的 token 中解析出当前登录用户。token 无效或用户不存在则返回 401。"""
    # 第一步：解码 token
    payload = decode_access_token(token)  # 这在教程 15 中已经写好了
    if payload is None:
        # token 无效（过期、签名错误、被篡改）
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,  # 此术语需进附录：401 Unauthorized
            detail="token 无效或已过期，请重新登录",
            headers={"WWW-Authenticate": "Bearer"},
        )

    # 第二步：从 payload 中取出 username
    username: str = payload.get("sub")  # "sub" 是 JWT 标准字段，放的是用户名
    if username is None:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="token 中缺少用户标识",
            headers={"WWW-Authenticate": "Bearer"},
        )

    # 第三步：查数据库，确认用户存在
    user = db.query(User).filter(User.username == username).first()
    if user is None:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="用户不存在或已被删除",
            headers={"WWW-Authenticate": "Bearer"},
        )

    # 第四步：全部通过，返回 User 对象
    return user
```

**逐行解释**：

- `token: str = Depends(oauth2_scheme)`：依赖注入链——FastAPI 先调用 `oauth2_scheme` 从请求头提取 token 字符串，再把结果传给 `get_current_user`。
- `db: Session = Depends(get_db)`：同时注入数据库会话，用来查用户。
- `payload = decode_access_token(token)`：调用教程 15 中写好的 JWT 解码函数。如果 token 无效（过期、签名不对、被篡改），`decode_access_token` 返回 `None`。
- `payload.get("sub")`：JWT 的 `sub`（subject）字段，我们在教程 15 创建 token 时把用户名存到了 `sub` 里。这里把它取出来。
- `db.query(User).filter(User.username == username).first()`：按用户名查数据库，确认用户确实存在（没有被删号）。
- 三个 `if` 判断，任何一个失败都 `raise HTTPException(status_code=401)`——**401 Unauthorized**，意思是"你没权限，不能进来"。此术语需进附录。

**❌ 常见错误 → ✅ 正确示例**：

❌ 错误：把 `get_current_user` 写成同步函数（没有 `async`）

```python
# 错误：同步函数
def get_current_user(token: str = Depends(oauth2_scheme), db: Session = Depends(get_db)):
    ...
```

✅ 正确：`async def`，因为 `oauth2_scheme` 内部是异步的

```python
# 正确：异步函数
async def get_current_user(token: str = Depends(oauth2_scheme), db: Session = Depends(get_db)):
    ...
```

> 🤔 想多一点：`decode_access_token` 返回 `None` 时报错，和直接让异常抛出去，有什么区别？

`decode_access_token` 内部可能抛出 `JWTError`（比如 token 格式不对、签名不匹配），也可能返回 `None`（比如 token 过期）。我们统一用 `if payload is None` 兜底，把所有 token 无效的情况都转成 `401 Unauthorized`——调用方不需要知道具体是"过期了"还是"签名不对"，只需要知道"你的 token 不行，重新登录"。

### 步骤 4：保护接口——给三个"写"操作加上门禁

现在 `get_current_user` 已经写好了，接下来就是把它加到需要保护的接口上。

**修改 `blog_backend/main.py`**，在需要保护的接口参数中加上 `current_user: User = Depends(get_current_user)`。

**受保护的接口**（POST、PUT、DELETE）：

```python
# blog_backend/main.py —— 在文件顶部追加导入

from blog_backend.auth import get_current_user  # 新增导入
from blog_backend.models import User  # 新增导入（如果之前没导入的话）


# === 创建文章（需要登录） ===
@app.post("/posts")
def create_post(
    blog: BlogCreate,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user),  # 新增：必须登录
):
    new_post = Post(
        title=blog.title,
        content=blog.content,
        published=blog.published,
        summary=blog.summary,
    )
    db.add(new_post)
    db.commit()
    db.refresh(new_post)
    return new_post


# === 更新文章（需要登录） ===
@app.put("/posts/{post_id}")
def update_post(
    post_id: int,
    blog: BlogCreate,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user),  # 新增：必须登录
):
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法更新"}
        )
    post.title = blog.title
    post.content = blog.content
    post.published = blog.published
    post.summary = blog.summary
    db.commit()
    return post


# === 删除文章（需要登录） ===
@app.delete("/posts/{post_id}")
def delete_post(
    post_id: int,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user),  # 新增：必须登录
):
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法删除"}
        )
    db.delete(post)
    db.commit()
    return {"message": f"文章 #{post_id}「{post.title}」已删除"}
```

**注意：`GET /posts` 和 `GET /posts/{id}` 不要加 `current_user`**——查看文章不需要登录，保持原样！

**发生了什么？**

你只是在三个接口的参数列表里多加了一行 `current_user: User = Depends(get_current_user)`，FastAPI 就自动帮你做了以下事情：

1. 请求进来 → 提取 `Authorization` 头 → 拿到 token 字符串
2. 解码 JWT → 验证签名和有效期 → 拿到 username
3. 查数据库 → 确认用户存在 → 返回 `User` 对象
4. 如果任何一步失败 → 返回 401，接口函数体根本不执行
5. 如果全部通过 → `current_user` 变量里就是一个活生生的 `User` 对象，你可以在接口函数里用它

**你一行"检查权限"的代码都没写**——FastAPI 的依赖注入帮你全干了。这就是 `Depends` 的威力。此术语需进附录：依赖注入。

### 步骤 8：🤔 想多一点——为什么用依赖注入而不是中间件？

你可能在想：教程 08 学了中间件，中间件不是可以拦截所有请求吗？为什么不用中间件来做权限保护？

**中间件 vs 依赖注入，核心区别**：

| 对比维度 | 中间件（Middleware） | 依赖注入（`Depends`） |
|----------|---------------------|----------------------|
| 生效范围 | **所有请求**都经过 | 只在你加了 `Depends` 的接口上生效 |
| 粒度 | 粗——一刀切，要么全保护要么全不保护 | 细——每个接口自己决定要不要保护 |
| 适合场景 | 日志、限流、CORS——对所有请求都需要的操作 | 权限、认证——只有部分接口需要 |
| 比喻 | 写字楼大门：所有人进楼都要过安检 | 各办公室门禁：只有进 CEO 办公室才刷卡 |

**如果我用中间件做权限保护，会怎样？**

```python
# ❌ 错误思路：用中间件保护所有接口
@app.middleware("http")
async def auth_middleware(request, call_next):
    # 检查 token...
    # 如果没 token，返回 401
    response = await call_next(request)
    return response
```

**后果**：`GET /posts` 也返回 401 了——**任何人连查看文章都需要登录**。这对于一个博客来说是完全不合理的。

**所以**：中间件适合"全局生效"的操作（日志、限流），依赖注入适合"按需保护"的操作（权限、认证）。FastAPI 把这两种机制都给了你，你只需要根据业务需求选对工具。

# === 配置 ===
SECRET_KEY = "your-secret-key-here-change-in-production"
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 30

# === 密码哈希上下文 ===
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")


def hash_password(password: str) -> str:
    """对明文密码进行哈希，返回哈希后的字符串。"""
    return pwd_context.hash(password)


def verify_password(plain_password: str, hashed_password: str) -> bool:
    """验证明文密码是否与哈希值匹配。"""
    return pwd_context.verify(plain_password, hashed_password)


def create_access_token(data: dict, expires_delta: timedelta | None = None) -> str:
    """创建 JWT access token。"""
    to_encode = data.copy()
    expire = datetime.utcnow() + (expires_delta or timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES))
    to_encode.update({"exp": expire})
    encoded_jwt = jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)
    return encoded_jwt


def decode_access_token(token: str) -> dict | None:
    """解码 JWT token，返回 payload 字典。失败返回 None。"""
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        return payload
    except JWTError:
        return None


# === 门禁刷卡器 ===
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/login")


# === 保安：验证 token → 查数据库 → 返回当前用户 ===
async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: Session = Depends(get_db),
) -> User:
    """从请求的 token 中解析出当前登录用户。token 无效或用户不存在则返回 401。"""
    payload = decode_access_token(token)
    if payload is None:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="token 无效或已过期，请重新登录",
            headers={"WWW-Authenticate": "Bearer"},
        )

    username: str = payload.get("sub")
    if username is None:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="token 中缺少用户标识",
            headers={"WWW-Authenticate": "Bearer"},
        )

    user = db.query(User).filter(User.username == username).first()
    if user is None:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="用户不存在或已被删除",
            headers={"WWW-Authenticate": "Bearer"},
        )

    return user

## 五、验证方法

在终端中依次执行以下命令，全部通过则权限保护实现正确：

```bash
# 1. 不带 token 创建文章 → 应返回 401
curl -X POST http://127.0.0.1:8000/posts -H "Content-Type: application/json" -d "{\"title\": \"test\", \"content\": \"test\", \"published\": true}"

# 2. 不带 token 查看文章 → 应正常返回
curl http://127.0.0.1:8000/posts

# 3. 登录获取 token
curl -X POST http://127.0.0.1:8000/login -H "Content-Type: application/json" -d "{\"username\": \"your_username\", \"password\": \"your_password\"}"

# 4. 用步骤 3 拿到的 token 创建文章 → 应成功
curl -X POST http://127.0.0.1:8000/posts -H "Content-Type: application/json" -H "Authorization: Bearer <token>" -d "{\"title\": \"合法文章\", \"content\": \"我登录了\", \"published\": true}"

# 5. 用 token 更新文章 → 应成功
curl -X PUT http://127.0.0.1:8000/posts/<id> -H "Content-Type: application/json" -H "Authorization: Bearer <token>" -d "{\"title\": \"已更新\", \"content\": \"更新内容\", \"published\": true}"

# 6. 用 token 删除文章 → 应成功
curl -X DELETE http://127.0.0.1:8000/posts/<id> -H "Authorization: Bearer <token>"

# 7. 不带 token 查看已删除的文章 → 应返回 404（不是 401，因为 GET 不需要登录）
curl http://127.0.0.1:8000/posts/<id>
```

全部通过？恭喜你——你的博客现在有真正的"门禁系统"了：没登录的人只能看，不能写、不能改、不能删。

## 七、小结

| 你学到了什么 | 一句话总结 |
|--------------|-----------|
| 权限保护的思路 | 不是所有接口都要锁，只锁"写"操作（POST/PUT/DELETE），"读"操作（GET）对外开放 |
| `OAuth2PasswordBearer` | FastAPI 的"门禁刷卡器"，自动从请求头提取 Bearer token |
| `get_current_user` | "保安"函数：解码 JWT → 提取 username → 查数据库 → 返回 User 对象 |
| `Depends(get_current_user)` | 一行代码加在接口参数里，整个接口就受保护了 |
| 未登录测试 | 不带 token 发 POST/PUT/DELETE → 返回 401 Unauthorized |
| 已登录测试 | 先登录拿 token → 带 `Authorization: Bearer <token>` 发请求 → 成功 |
| 中间件 vs 依赖注入 | 中间件全局生效，依赖注入按需生效——权限保护应该用依赖注入，而不是中间件 |
| `Authorization` 头 | HTTP 标准的身份凭证传递方式，格式 `Bearer <token>` |

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| `get_current_user` 写成同步函数 | 运行时报错 `coroutine object is not callable` | `oauth2_scheme` 内部是异步的，调用它的函数也必须是 `async def` | 把 `def get_current_user` 改成 `async def get_current_user` |
| token 过期后还拿来用 | 返回 401 `"token 无效或已过期"` | JWT 有 `exp` 过期时间，默认 30 分钟 | 重新登录拿新 token |
| 把 `current_user` 加到了 GET 接口上 | 查看文章也需要登录了 | 依赖注入是对单个接口生效的，GET 接口里加了 `Depends(get_current_user)` 就也被保护了 | 只给 POST/PUT/DELETE 加 `current_user`，GET 接口不要加 |
| `Authorization` 头格式写错 | 返回 401，但 token 明明有效 | 格式必须是 `Bearer <token>`，中间有空格；不能写成 `Bearer:<token>` 或 `bearer <token>`（B 大写） | 检查格式：`Authorization: Bearer <token>` |
| 用中间件实现权限保护 | 所有接口都需要登录，包括 GET | 中间件对所有请求生效，没有"选择性" | 用依赖注入（`Depends`）代替中间件做权限保护 |

## 九、下一步建议

现在你的博客有了"门禁系统"——没登录的人只能看，不能写。但还有一个问题：**任何人只要登录了，就能改别人的文章**。Alice 登录后可以修改 Bob 写的文章，这不合理。

下一章（教程 17：文章归属——只能改自己的文章），我们会在 `get_current_user` 的基础上，加上"文章作者检查"——只有作者本人才能修改和删除自己的文章。这样权限体系就完整了：**没登录的不能写，登录了也只能改自己的。**
