# 第115章 · 项目六：微服务电商系统

> "前面五个项目都是单体应用——所有代码在一个进程里，所有表在一个数据库里。这种方式适合小项目和早期创业，但当团队到了50人、代码到了50万行、日活到了百万级时，单体应用的弊端就暴露了：一个人改一行代码，全团队等半小时CI；用户模块挂了，整个网站502。微服务就是解决这些问题的——把大船拆成一队小艇，各自独立航行，但通过旗语（gRPC）保持通信。"

---

## 115.1 为什么需要微服务

### 单体应用的五个痛点

| 痛点 | 场景 | 微服务怎么解决 |
|------|------|---------------|
| 部署耦合 | 改一行用户代码要部署整个应用 | 只部署用户服务，其他服务不动 |
| 扩展困难 | 只有订单模块压力大，但只能整体扩容 | 只扩容订单服务实例 |
| 技术锁定 | 全项目绑定Go和MySQL | 订单用Go，推荐用Python，各自选最适合的 |
| 故障爆炸 | 支付模块内存泄漏，全站502 | 支付服务挂了不影响商品浏览 |
| 团队阻塞 | 50人改同一代码库，天天冲突 | 每个团队维护自己的服务，独立发版 |

### 微服务的代价

微服务不是银弹——它解决了单体的问题，但带来了新的问题：

| 新问题 | 描述 | 我们在本章的解决方案 |
|--------|------|---------------------|
| 服务间通信 | 原来函数调用变网络调用 | gRPC（Protobuf定义接口） |
| 服务发现 | 服务实例动态变化，IP不固定 | Consul注册中心 |
| 分布式事务 | 跨服务的数据一致性 | Saga模式（补偿事务） |
| 调试困难 | 一个请求经过5个服务 | 链路追踪（介绍思想不实现） |
| 运维复杂 | 原来部署1个应用，现在部署N个 | Docker Compose编排 |

如果把单体比作一个大超市——所有东西都在一层楼，买东西方便但人多就挤。微服务就是把超市拆成垂直的专卖店——生鲜店、电器店、服装店——各管各的，但要买东西得跑多个店，而且需要一张地图（服务发现）知道每家店在哪。

## 115.2 服务拆分

把第113章的单体电商拆成4个微服务：

```
┌──────────────────────────────────────────────────────────┐
│                       API Gateway (Gin)                   │
│                   统一入口 / 认证 / 限流 / 路由              │
└──────┬──────────────┬──────────────┬─────────────────────┘
       │              │              │
       │ gRPC         │ gRPC         │ gRPC
       v              v              v
┌──────────────┐ ┌─────────────┐ ┌──────────────┐ ┌──────────────┐
│  User Service│ │Product Srv  │ │ Order Service│ │Payment Service│
│  用户服务     │ │ 商品服务     │ │  订单服务     │ │  支付服务     │
│  Port: 50051 │ │ Port: 50052 │ │  Port: 50053 │ │  Port: 50054 │
│              │ │              │ │              │ │              │
│    MySQL     │ │    MySQL    │ │    MySQL     │ │    MySQL     │
│  user_db     │ │ product_db  │ │  order_db    │ │ payment_db   │
└──────────────┘ └─────────────┘ └──────────────┘ └──────────────┘
                            │
                            │
                    ┌───────┴───────┐
                    │    Consul     │
                    │  服务注册中心   │
                    │  Port: 8500   │
                    └───────────────┘
```

每个微服务有自己的数据库——这是微服务的核心原则：**数据自治**。就像一个公司里，财务部有自己的账本，人事部有自己的档案柜，谁也不直接翻谁的抽屉。

### 各服务职责

| 服务 | 职责 | 核心接口 |
|------|------|---------|
| User Service | 注册、登录、用户信息 | Register, Login, GetUser, UpdateProfile |
| Product Service | 商品CRUD、库存管理 | ListProducts, GetProduct, DeductStock |
| Order Service | 下单、订单管理 | CreateOrder, GetOrder, ListOrders |
| Payment Service | 支付处理 | Pay, Refund, GetPaymentStatus |

## 115.3 gRPC + Protobuf

### 为什么用gRPC而不是REST

| 特性 | REST/HTTP+JSON | gRPC |
|------|---------------|------|
| 协议 | HTTP/1.1 + JSON | HTTP/2 + Protobuf |
| 序列化 | JSON（文本，大） | Protobuf（二进制，小） |
| 接口定义 | 口头约定或Swagger | .proto文件（强类型） |
| 性能 | 一般 | 快5-10倍 |
| 流式传输 | 不支持 | 原生支持 |
| 代码生成 | 需要手写或第三方 | 自动生成客户端和服务端 |

gRPC就像标准化快递——寄件人按标准格式填写快递单（.proto），快递公司自动处理路由和送达，收件人按标准格式收货。不需要每次打电话确认"你用的是什么格式？"

### 安装Protobuf编译器

首先需要安装 protoc 编译器本身：从 [Protobuf Releases](https://github.com/protocolbuffers/protobuf/releases) 下载对应平台的二进制文件，解压后将 `protoc` 加入 PATH。再安装 Go 插件：

```bash
go install google.golang.org/protobuf/cmd/protoc-gen-go@latest
go install google.golang.org/grpc/cmd/protoc-gen-go-grpc@latest
```

### 定义Proto文件

创建 `proto/user.proto`：

```protobuf
syntax = "proto3";

package user;

option go_package = "microshop/proto/user";

service UserService {
    rpc Register(RegisterRequest) returns (RegisterResponse);
    rpc Login(LoginRequest) returns (LoginResponse);
    rpc GetUser(GetUserRequest) returns (GetUserResponse);
}

message RegisterRequest {
    string email = 1;
    string password = 2;
    string nickname = 3;
}

message RegisterResponse {
    int32 code = 1;
    string message = 2;
    uint64 user_id = 3;
}

message LoginRequest {
    string email = 1;
    string password = 2;
}

message LoginResponse {
    int32 code = 1;
    string message = 2;
    string token = 3;
    uint64 user_id = 4;
}

message GetUserRequest {
    uint64 user_id = 1;
}

message GetUserResponse {
    int32 code = 1;
    string message = 2;
    UserInfo user = 3;
}

message UserInfo {
    uint64 id = 1;
    string email = 2;
    string nickname = 3;
    string avatar = 4;
}
```

Proto文件是gRPC的核心——它既是接口文档（一眼看清有什么方法、什么参数）又是数据格式定义（Int32/Int64/String/Message等强类型）。

每个字段后面的序号（`= 1`, `= 2`...）是字段编号，用于二进制编码时标识字段。**永远不要改已有字段的编号**——否则所有老客户端都会解析错误。添加新字段时用新的编号。

创建 `proto/product.proto`：

```protobuf
syntax = "proto3";
```

```go
package main

import (
    "fmt"
    "log"
    "net"

    pb "microshop/proto/user"

    "google.golang.org/grpc"
    "gorm.io/driver/mysql"
    "gorm.io/gorm"

func main() {

        log.Fatalf("服务启动失败: %v", err)
```

### user-service/server/user.go

```go
package server

import (
    "context"
    "time"

    pb "microshop/proto/user"

    "github.com/golang-jwt/jwt/v5"
    "golang.org/x/crypto/bcrypt"
    "gorm.io/gorm"
)

type UserServer struct {
    pb.UnimplementedUserServiceServer
    DB *gorm.DB
}

func NewUserServer(db *gorm.DB) *UserServer {
    return &UserServer{DB: db}
}

func (s *UserServer) Register(ctx context.Context, req *pb.RegisterRequest) (*pb.RegisterResponse, error) {
    var count int64
    s.DB.Model(&User{}).Where("email = ?", req.Email).Count(&count)
    if count > 0 {
        return &pb.RegisterResponse{
            Code: 409, Message: "邮箱已被注册",
        }, nil
    }

    hash, _ := bcrypt.GenerateFromPassword([]byte(req.Password), bcrypt.DefaultCost)

    user := User{
        Email:        req.Email,
        PasswordHash: string(hash),
        Nickname:     req.Nickname,
    }
    s.DB.Create(&user)

    return &pb.RegisterResponse{
        Code: 201, Message: "注册成功", UserId: user.ID,
    }, nil
}

func (s *UserServer) Login(ctx context.Context, req *pb.LoginRequest) (*pb.LoginResponse, error) {
    var user User
    if err := s.DB.Where("email = ?", req.Email).First(&user).Error; err != nil {
        return &pb.LoginResponse{
            Code: 401, Message: "邮箱或密码错误",
        }, nil
    }

    if err := bcrypt.CompareHashAndPassword([]byte(user.PasswordHash), []byte(req.Password)); err != nil {
        return &pb.LoginResponse{
            Code: 401, Message: "邮箱或密码错误",
        }, nil
    }

    token, _ := GenerateToken(user.ID, user.Email)

    return &pb.LoginResponse{
        Code: 200, Message: "登录成功", Token: token, UserId: user.ID,
    }, nil
}

func (s *UserServer) GetUser(ctx context.Context, req *pb.GetUserRequest) (*pb.GetUserResponse, error) {
    var user User
    if err := s.DB.First(&user, req.UserId).Error; err != nil {
        return &pb.GetUserResponse{
            Code: 404, Message: "用户不存在",
        }, nil
    }

    return &pb.GetUserResponse{
        Code: 200, Message: "success",
        User: &pb.UserInfo{
            Id: user.ID, Email: user.Email,
            Nickname: user.Nickname, Avatar: user.Avatar,
        },
    }, nil
}
```

```go
func GenerateToken(userID uint64, email string) (string, error) {
    claims := jwt.MapClaims{
        "user_id": userID,
        "email":   email,
        "exp":     time.Now().Add(24 * time.Hour).Unix(),
    }
    token := jwt.NewWithClaims(jwt.SigningMethodHS256, claims)
    return token.SignedString([]byte("your-secret-key"))
}
```

### product-service/server/product.go

```go
func (s *ProductServer) DeductStock(ctx context.Context, req *pb.DeductStockRequest) (*pb.DeductStockResponse, error) {
    result := s.DB.Model(&Product{}).
        Where("id = ? AND stock >= ?", req.ProductId, req.Quantity).
        UpdateColumn("stock", gorm.Expr("stock - ?", req.Quantity))

    if result.RowsAffected == 0 {
        return &pb.DeductStockResponse{
            Code: 400, Message: "库存不足", Success: false,
        }, nil
    }

    return &pb.DeductStockResponse{
        Code: 200, Message: "扣减成功", Success: true,
    }, nil
}
```

### order-service/server/order.go — 跨服务调用

订单服务在创建订单时，需要调用商品服务扣减库存。这就是微服务的核心场景——一个服务调用另一个服务：

```go
func (s *OrderServer) CreateOrder(ctx context.Context, req *pb.CreateOrderRequest) (*pb.CreateOrderResponse, error) {
    for _, item := range req.Items {
        deductResp, err := s.ProductClient.DeductStock(ctx, &productpb.DeductStockRequest{
            ProductId: item.ProductId,
            Quantity:  item.Quantity,
        })
        if err != nil || !deductResp.Success {
            return &pb.CreateOrderResponse{
                Code: 400, Message: "商品库存不足",
            }, nil
        }
    }

    orderNo := fmt.Sprintf("ORD%s%d", time.Now().Format("20060102150405"), req.UserId)
    totalAmount := 0.0
    for _, item := range req.Items {
        totalAmount += item.Price * float64(item.Quantity)
    }

    order := Order{
        OrderNo:     orderNo,
        UserID:      req.UserId,
        TotalAmount: totalAmount,
        Status:      0,
    }
    s.DB.Create(&order)

    return &pb.CreateOrderResponse{
        Code: 201, Message: "下单成功",
        OrderId: order.ID, OrderNo: order.OrderNo,
    }, nil
}
```

## 115.5 API网关

网关是微服务架构的"前台"——对外暴露统一的HTTP接口，对内把请求转发到对应的gRPC服务。

```go
package main

import (
    "log"
    "net/http"

    userpb "microshop/proto/user"
    productpb "microshop/proto/product"
    orderpb "microshop/proto/order"

    "github.com/gin-gonic/gin"
    "google.golang.org/grpc"
    "google.golang.org/grpc/credentials/insecure"
)

func main() {
    userConn, _ := grpc.Dial("localhost:50051",
        grpc.WithTransportCredentials(insecure.NewCredentials()))
    defer userConn.Close()
    userClient := userpb.NewUserServiceClient(userConn)

    productConn, _ := grpc.Dial("localhost:50052",
        grpc.WithTransportCredentials(insecure.NewCredentials()))
    defer productConn.Close()
    productClient := productpb.NewProductServiceClient(productConn)

    orderConn, _ := grpc.Dial("localhost:50053",
        grpc.WithTransportCredentials(insecure.NewCredentials()))
    defer orderConn.Close()
    orderClient := orderpb.NewOrderServiceClient(orderConn)

    r := gin.Default()

    r.POST("/api/register", func(c *gin.Context) {
        var req struct {
            Email    string `json:"email"`
            Password string `json:"password"`
            Nickname string `json:"nickname"`
        }
        c.ShouldBindJSON(&req)

        resp, _ := userClient.Register(c.Request.Context(), &userpb.RegisterRequest{
            Email: req.Email, Password: req.Password, Nickname: req.Nickname,
        })

        c.JSON(http.StatusOK, gin.H{
            "code": resp.Code, "message": resp.Message,
            "data": gin.H{"user_id": resp.UserId},
        })
    })

    r.POST("/api/login", func(c *gin.Context) {
        var req struct {
            Email    string `json:"email"`
            Password string `json:"password"`
        }
        c.ShouldBindJSON(&req)

        resp, _ := userClient.Login(c.Request.Context(), &userpb.LoginRequest{
            Email: req.Email, Password: req.Password,
        })

        c.JSON(http.StatusOK, gin.H{
            "code": resp.Code, "message": resp.Message,
            "data": gin.H{"token": resp.Token, "user_id": resp.UserId},
        })
    })

    r.GET("/api/products", func(c *gin.Context) {
        resp, _ := productClient.ListProducts(c.Request.Context(), &productpb.ListProductsRequest{})
        c.JSON(http.StatusOK, gin.H{
            "code": resp.Code, "message": resp.Message, "data": resp.Products,
        })
    })

    authorized := r.Group("/api")
    authorized.Use(AuthMiddleware())
    {
        authorized.POST("/orders", func(c *gin.Context) {
            userID := c.GetUint64("user_id")
            var req struct {
                Items []struct {
                    ProductID   uint64  `json:"product_id"`
                    ProductName string  `json:"product_name"`
                    Price       float64 `json:"price"`
                    Quantity    uint32  `json:"quantity"`
                } `json:"items"`
            }
            c.ShouldBindJSON(&req)

            items := make([]*orderpb.OrderItem, len(req.Items))
            for i, item := range req.Items {
                items[i] = &orderpb.OrderItem{
                    ProductId: item.ProductID, ProductName: item.ProductName,
                    Price: item.Price, Quantity: item.Quantity,
                }
            }

            resp, _ := orderClient.CreateOrder(c.Request.Context(), &orderpb.CreateOrderRequest{
                UserId: userID, Items: items,
            })

            c.JSON(http.StatusOK, gin.H{
                "code": resp.Code, "message": resp.Message,
                "data": gin.H{"order_id": resp.OrderId, "order_no": resp.OrderNo},
            })
        })
    }

    log.Fatal(r.Run(":8080"))
}
```

网关的职责很纯粹：HTTP → gRPC转换。它不做业务逻辑，只是协议翻译和路由转发。就像联合国翻译官——中方代表说中文，翻译官转成法语告诉法国代表，自己不加意见。

## 115.6 Consul服务发现

### 为什么需要服务发现（自动跟踪服务实例地址变化的机制，详见附录I）

在微服务架构中，服务实例会动态变化：
- 订单服务在服务器A上（IP: 10.0.1.5:50053），后来扩容到服务器B（10.0.1.6:50053）
- 支付服务重启后IP变了

如果硬编码IP地址，每次变化都要改配置重启。Consul（HashiCorp开源的服务发现与配置管理工具，详见附录I）就像一个自动更新的通讯录——服务启动时来注册，下线时自动删除，查询时总是返回最新的在线地址。

### 注册到Consul

```go
package discovery

import (
    "fmt"
    "github.com/hashicorp/consul/api"
)

func RegisterService(serviceName, serviceID, address string, port int) error {
    config := api.DefaultConfig()
    config.Address = "localhost:8500"
    client, _ := api.NewClient(config)

    registration := &api.AgentServiceRegistration{
        ID:      serviceID,
        Name:    serviceName,
        Address: address,
        Port:    port,
        Check: &api.AgentServiceCheck{
            GRPC:     fmt.Sprintf("%s:%d", address, port),
            Interval: "10s",
            Timeout:  "3s",
        },
    }

    return client.Agent().ServiceRegister(registration)
}
```

每个服务启动时调用 `RegisterService`，注册到Consul。健康检查用gRPC协议，每10秒检查一次——如果服务挂了，Consul会自动将其标记为不健康并从列表中移除。

### 从Consul发现服务

```go
func DiscoverService(serviceName string) (string, int, error) {
    config := api.DefaultConfig()
    config.Address = "localhost:8500"
    client, _ := api.NewClient(config)

    services, _, err := client.Health().Service(serviceName, "", true, nil)
    if err != nil || len(services) == 0 {
        return "", 0, fmt.Errorf("服务 %s 不可用", serviceName)
    }

    svc := services[0].Service
    return svc.Address, svc.Port, nil
}
```

有了服务发现，API网关不再硬编码服务地址：

## 115.7 分布式事务处理

### 微服务的事务难题

在单体应用里，下单是一个事务：扣库存 → 创建订单 → 扣款，任何一个步骤失败全部回滚。

在微服务中，这三个操作分别在三个不同的数据库里——MySQL的ACID事务跨不了数据库。你需要分布式事务策略。

### 方案对比

| 方案 | 原理 | 复杂度 | 适用场景 |
|------|------|--------|---------|
| 2PC（两阶段提交） | 协调者问所有参与者"准备好了吗？"，全准备好再"提交" | 高 | 强一致性要求 |
| TCC（Try-Confirm-Cancel） | 预留资源→确认→或取消 | 高 | 金融场景 |
| Saga（补偿事务） | 每个步骤有对应的补偿操作，失败时逆序执行补偿 | 中 | **电商推荐** |
| 本地消息表 | 本地事务+消息表保证最终一致性 | 低 | 异步场景 |

### Saga模式（推荐）

Saga（分布式事务的补偿模式，每步有对应回滚操作保证最终一致性，详见附录I）把一个大事务拆成一系列小事务，每个小事务有自己的补偿操作。如果中间某步失败了，就逆序执行之前成功的补偿操作。

以创建订单为例：

```
步骤1: 商品服务 → 扣减库存
  补偿: 商品服务 → 恢复库存

步骤2: 订单服务 → 创建订单
  补偿: 订单服务 → 取消订单

步骤3: 支付服务 → 扣款
  补偿: 支付服务 → 退款
```

如果步骤3扣款失败：
1. 执行步骤2的补偿：取消订单
2. 执行步骤1的补偿：恢复库存

### Go中的Saga实现骨架

```go
type SagaStep struct {
    Name     string
    Execute  func() error
    Compensate func() error
}

func ExecuteSaga(steps []SagaStep) error {
    executedSteps := []SagaStep{}

    for _, step := range steps {
        if err := step.Execute(); err != nil {
            log.Printf("步骤 %s 失败，开始补偿", step.Name)
            for i := len(executedSteps) - 1; i >= 0; i-- {
                log.Printf("补偿步骤 %s", executedSteps[i].Name)
                if compErr := executedSteps[i].Compensate(); compErr != nil {
                    log.Printf("补偿失败: %v", compErr)
                }
            }
            return err
        }
        executedSteps = append(executedSteps, step)
    }
    return nil
}

func CreateOrderSaga(productClient, orderClient, paymentClient interface{}) {
    steps := []SagaStep{
        {
            Name: "扣减库存",
            Execute: func() error {
                return DeductStock(productClient, productID, quantity)
            },
            Compensate: func() error {
                return RestoreStock(productClient, productID, quantity)
            },
        },
        {
            Name: "创建订单",
            Execute: func() error {
                return CreateOrder(orderClient, userID, items)
            },
            Compensate: func() error {
                return CancelOrder(orderClient, orderID)
            },
        },
        {
            Name: "扣款",
            Execute: func() error {
                return Pay(paymentClient, userID, amount)
            },
            Compensate: func() error {
                return Refund(paymentClient, orderID)
            },
        },
    }

    ExecuteSaga(steps)
}
```
