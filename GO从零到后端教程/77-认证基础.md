## 77.1 Authentication vs Authorization

这是两个最容易被混淆的概念，但它们问的是完全不同的两个问题。

### 用小区门禁来理解

想象你住在一个高档小区：

**认证（Authentication，验证用户身份的过程，详见附录I）** = 小区大门口保安问："你是谁？"

你掏出身份证。保安核对："嗯，是业主，请进。"这个过程就是认证——**验证你的身份**。保安确认了"你是声称的那个人"。

**授权（Authorization，确定用户能做什么的权限检查，详见附录I）** = 进了小区以后，你能去哪？

你拿着业主卡进了一楼大厅。但你想去物业办公室、想去其他楼栋、想进电梯去楼顶——每扇门都可能需要不同的权限。你只能去你被**授权**去的地方。

### Authentication（你是谁？）

英文缩写是 **AuthN**（Authentication的缩写，即身份认证，详见附录I）。它只做一件事：

> 验证"你就是你声称的那个人"

常见的验证方式：
- **你知道什么**：密码、PIN码、密保问题的答案
- **你拥有什么**：手机（验证码）、U盾、门禁卡、硬件令牌
- **你是什么**：指纹、人脸、虹膜（生物特征）
- **你在哪里**：GPS位置、IP地址（地理位置认证）

双因素认证（2FA）就是同时用两种方式，比如"密码 + 短信验证码"。即便密码泄露，没有手机也登不进去。

### Authorization（你能干什么？）

英文缩写是 **AuthZ**（Authorization的缩写，即权限检查，详见附录I）。它做的事情完全不同：

> 验证"你有没有权限做这件事"

认证通过后，系统还需要判断：
- 你能查看哪些数据？（普通用户只能看自己的订单）
- 你能执行哪些操作？（普通编辑不能删除文章）
- 你能访问哪些接口？（管理员可以看所有用户数据）

**举个例子，把一个系统中的三种人放进场景里：**

| 角色 | 认证（你是谁） | 授权（你能干什么） |
|------|---------------|-------------------|
| 游客 | 没有登录，身份未知 | 只能看公开文章 |
| 普通用户 张三 | 登录了，是张三 | 能写文章、改自己的文章、删自己的文章 |
| 管理员 李四 | 登录了，是李四 | 能删任何人的文章、封禁用户、看后台数据 |

张三和李四都通过了认证（都登录了），但授权不同（张三不能删别人的文章）。

### 在HTTP层面对应什么

```
认证失败 → HTTP 401 Unauthorized（"你要证明你是谁"）
授权失败 → HTTP 403 Forbidden（"我知道你是谁，但你不配"）
```

尽管401的状态码名叫"Unauthorized"，它在HTTP协议中的实际含义是"未认证"——这是历史遗留的命名问题，不要被名字误导。

一个更好记的口诀：

> **认证问我是不是张三，授权问张三能不能进这扇门。**

### 🤔 想多一点

为什么要把认证和授权分开？因为它们变化的原因不同。

认证逻辑相对稳定：密码登录、手机验证码、指纹——这些方式不会天天变。但授权逻辑可能剧烈变化：今天运营小妹只能看数据，明天老板说她也得能导出报表，后天又说她可以删除垃圾评论。如果认证和授权耦合在一起，改权限就可能影响登录逻辑，出Bug的风险极大。

实际上，大多数系统的认证逻辑是通用的（一套登录系统），授权逻辑是跟业务深度绑定的（不同的产品有不同的权限模型）。把它们拆开，才能各自独立演进。

## 77.2 密码安全存储：绝对不能存明文

### 明文密码=裸奔

"存明文"是什么意思？就是在数据库里直接存用户输入的原始密码：

```
用户名    密码
张三      123456
李四      password
王五      qwerty
```

如果你这么干了，会发生什么？

**场景一：数据库泄露**

某天黑客通过SQL注入拿到了你的用户表。现在他手里有所有用户的用户名和密码——而且是明文。一个用户通常在所有网站上用同一套密码。他拿着 `zhangsan / 123456` 去试邮箱、网银、社交平台……

你的数据库泄露只是一个点，但明文密码让它变成了对所有用户的**连锁灾难**。

**场景二：内部人员作案**

运维人员、DBA、有数据库权限的开发——他们每天都能看到这些明文密码。你能保证公司一百号人里没有一个坏人？或者没有一个账号被盗的内部员工？

**场景三：日志泄露**

假设某行代码不小心把请求参数打到了日志里：

```go
log.Printf("用户登录: %+v", req)
```

日志文件里就出现了 `密码=123456`。日志文件通常分散在多台服务器上，排查困难，更容易泄露。

### 哈希（Hash）：单向加密

解决办法是：**不存密码本身，存密码的哈希值**。

哈希算法有个关键特性：**不可逆**。你可以把"123456"算出哈希值，但拿着哈希值永远推算不出"123456"。

就像把一块猪肉放进绞肉机——绞成肉馅后，你无法把它恢复成原来的那块猪肉。但你可以再绞一块猪肉来比较：如果两次绞出的肉馅一样，说明是同一块肉。

Go里最基础的哈希（**仅供理解原理，生产环境绝不能用**）：

```go
import "crypto/sha256"

password := "123456"
hash := sha256.Sum256([]byte(password))
fmt.Printf("%x\n", hash)
```

输出是一串64位十六进制字符：
```
8d969eef6ecad3c29a3a629280e686cf0c3f5d5a86aff3ca12020c923adc6c92
```

现在数据库里存的是这串哈希。用户登录时，把输入的密码也哈希一下，看两个哈希值是否一致。

### 彩虹表攻击：光哈希还不够

黑客早就想到了：你们都用 `sha256("123456")`，出来的结果都一样。那我提前把常见密码和它们的哈希值列个表不就行了？

| 密码 | SHA256哈希 |
|------|-----------|
| 123456 | 8d969eef6ecad3... |
| password | 5e884898da2804... |
| qwerty | 65e84be33532fb... |

这张庞大的表就是**彩虹表**（Rainbow Table）。现在只要从泄露的数据库里看到哈希值是 `8d969eef...`，查表立刻知道密码是 `123456`。

### 加盐（Salt）：让彩虹表失效

"盐"是一段**随机的、每个用户不同的**字符串，拼接在密码后面一起哈希。

```
用户A：密码123456 + 盐"aB3$xK9" → 哈希 → "f3a8c9..."  
用户B：密码123456 + 盐"mN7@qW2" → 哈希 → "b2d1e6..."
```

现在即便两个用户的密码完全相同，哈希值也不同。黑客无法用一张彩虹表破解所有人的密码——他需要为每个盐单独生成一张彩虹表，这在计算上不可行。

盐是公开的，通常和哈希值一起存在数据库里。不用担心盐被黑客看到——盐的作用不是保密，而是**打破哈希值的确定性**。

### 慢哈希：让暴力破解成本不可接受

即使有了盐，黑客还是可以一个一个试：从 `aaaaaaaa` 试到 `zzzzzzzz`。但如果每次哈希计算需要0.3秒，那尝试一亿个密码组合需要347天——操作成本变得不可接受。

这就是"慢哈希"的意义：**故意让哈希计算变慢**，让正常用户的登录（一次计算）不受影响，让暴力破解（大量计算）变得不可行。

## 77.3 bcrypt：现代密码哈希的标准答案

### bcrypt凭什么是最好的选择？

bcrypt同时解决了三个问题：

1. **自带盐**——你不需要手动生成和管理盐，bcrypt自动为每个密码生成随机盐并嵌入最终结果
2. **故意慢**——计算一次需要零点几秒，暴力破解成本极高
3. **可调节的成本因子（cost factor）**——随着硬件越来越快，你可以调高cost让哈希变慢，抵御未来的算力增长

### bcrypt的输出格式

一行bcrypt哈希长这样：

```
$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
```

拆开来看：

| 段落 | 值 | 含义 |
|------|-----|------|
| `$2a$` | 2a | bcrypt版本号 |
| `$10$` | 10 | cost factor（2^10 = 1024次迭代） |
| `N9qo8uLOickgx2ZMRZoMye` | 22字符 | 随机生成的盐 |
| `IjZAgcfl7p92ldGxad68LJZdL17lhWy` | 31字符 | 密码+盐的哈希结果 |

注意：盐和哈希结果**都在这一个字符串里**。你不需要额外存储盐，也不需要在验证时自己提取盐——bcrypt库全帮你做了。

### 在Go中使用bcrypt

```go
import "golang.org/x/crypto/bcrypt"

func HashPassword(password string) (string, error) {
	bytes, err := bcrypt.GenerateFromPassword([]byte(password), bcrypt.DefaultCost)
	if err != nil {
		return "", err
	}
	return string(bytes), nil
}

func CheckPassword(password, hash string) bool {
	err := bcrypt.CompareHashAndPassword([]byte(hash), []byte(password))
	return err == nil
}
```

`bcrypt.DefaultCost` 等于10。在目前（2020年代）的硬件上，cost=10时一次哈希大约需要0.1~0.3秒。如果未来硬件更快，你可以把cost调到12或14。

### Cost选择指南

| Cost | 迭代次数 | 单次耗时（参考） | 适用场景 |
|------|---------|----------------|---------|
| 10 | 2^10 = 1024 | ~0.1-0.3秒 | 通用Web应用 |
| 12 | 2^12 = 4096 | ~0.4-1.2秒 | 高安全性应用 |
| 14 | 2^14 = 16384 | ~1.6-5秒 | 极敏感数据 |

**不要只设cost=4**，那太快了，相当于没防护。**也不要设cost=20**，登录一次要等一分钟，用户会骂人。cost=10~12是绝大多数场景的最佳选择。

### bcrypt的64字节截断陷阱

bcrypt最大只能处理**72字节**的输入。如果你的密码是中文或emoji，一个字符可能占3-4字节：

```
"我爱后端开发123456"  = 25字节 （安全）
"🚀🌟✨🌙🔥💻" × 10   = 40字节 （安全）
密码超过72字节         = 被截断！
```

如果你的系统允许极长密码，有两种处理方式：
- 先用SHA256把密码哈希成固定长度，再传给bcrypt
- 限制密码最大长度（如64字符，对大多数场景已经足够）

### 🤔 想多一点

很多人问：bcrypt这么慢，高并发登录不会拖垮服务器吗？

答案是：**不会**。因为登录不是高频操作。用户登录一次以后，用Token或Session维持状态，不需要每次都重新验证密码。一个日活一万的系统，登录请求可能每分钟只有几十个——bcrypt的"慢"完全够用。

真正的高频请求（比如每次API调用都验Token），用的是JWT签名验证——那是毫秒级的，跟bcrypt完全不冲突。回头在JWT章节你会看到，Token验证和密码验证用的是完全不同的技术路径。

## 77.4 注册/登录接口：完整实现

理论讲够了，来写一个能跑的用户注册和登录功能。使用Gin框架+MySQL，但你可以轻松替换成任何数据库。

### 数据库表设计

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(64) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    email VARCHAR(128) NOT NULL DEFAULT '',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_username (username)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

关键点：存 `password_hash` 而不是 `password`。255长度够放bcrypt的输出（60字符的哈希+安全余量）。

### 用户模型

```go
package model

import (
	"time"
)

type User struct {
	ID           int64     `json:"id"`
	Username     string    `json:"username"`
	PasswordHash string    `json:"-"`
	Email        string    `json:"email"`
	CreatedAt    time.Time `json:"created_at"`
	UpdatedAt    time.Time `json:"updated_at"`
}
```

注意 `PasswordHash` 的JSON tag是 `"-"`——这个字段**永远不会被序列化到API响应中**。就算你不小心把User对象当JSON返回了，密码哈希也不会泄露。

### 注册Handler

```go
package handler

import (
	"net/http"
	"strings"

	"your-project/model"
	"your-project/database"

	"github.com/gin-gonic/gin"
	"golang.org/x/crypto/bcrypt"
)

type RegisterRequest struct {
	Username string `json:"username" binding:"required,min=3,max=64"`
	Password string `json:"password" binding:"required,min=6,max=64"`
	Email    string `json:"email" binding:"omitempty,email"`
}

type RegisterResponse struct {
	ID       int64  `json:"id"`
	Username string `json:"username"`
}

func Register(c *gin.Context) {
	var req RegisterRequest
	if err := c.ShouldBindJSON(&req); err != nil {
		c.JSON(http.StatusBadRequest, gin.H{
			"code": 400,
			"msg":  "参数错误: " + err.Error(),
		})
		return
	}

	req.Username = strings.TrimSpace(req.Username)
	if req.Username == "" {
		c.JSON(http.StatusBadRequest, gin.H{
			"code": 400,
			"msg":  "用户名不能为空",
		})
		return
	}

	hashedPassword, err := bcrypt.GenerateFromPassword([]byte(req.Password), bcrypt.DefaultCost)
	if err != nil {
		c.JSON(http.StatusInternalServerError, gin.H{
			"code": 500,
			"msg":  "服务器内部错误",
		})
		return
	}

	user := model.User{
		Username:     req.Username,
		PasswordHash: string(hashedPassword),
		Email:        req.Email,
	}

	result, err := database.DB.Exec(
		"INSERT INTO users (username, password_hash, email) VALUES (?, ?, ?)",
		user.Username, user.PasswordHash, user.Email,
	)
	if err != nil {
		if strings.Contains(err.Error(), "Duplicate") {
			c.JSON(http.StatusConflict, gin.H{
				"code": 409,
				"msg":  "用户名已被注册",
			})
			return
		}
		c.JSON(http.StatusInternalServerError, gin.H{
			"code": 500,
			"msg":  "注册失败",
		})
		return
	}

	user.ID, _ = result.LastInsertId()

	c.JSON(http.StatusCreated, gin.H{
		"code": 201,
		"msg":  "注册成功",
		"data": RegisterResponse{
			ID:       user.ID,
			Username: user.Username,
		},
	})
}
```

### 登录Handler

```go
type LoginRequest struct {
	Username string `json:"username" binding:"required"`
	Password string `json:"password" binding:"required"`
}

type LoginResponse struct {
	UserID   int64  `json:"user_id"`
	Username string `json:"username"`
}

func Login(c *gin.Context) {
	var req LoginRequest
	if err := c.ShouldBindJSON(&req); err != nil {
		c.JSON(http.StatusBadRequest, gin.H{
			"code": 400,
			"msg":  "参数错误",
		})
		return
	}

	var user model.User
	err := database.DB.QueryRow(
		"SELECT id, username, password_hash FROM users WHERE username = ?",
		req.Username,
	).Scan(&user.ID, &user.Username, &user.PasswordHash)

	if err != nil {
		c.JSON(http.StatusUnauthorized, gin.H{
			"code": 401,
			"msg":  "用户名或密码错误",
		})
		return
	}

	err = bcrypt.CompareHashAndPassword([]byte(user.PasswordHash), []byte(req.Password))
	if err != nil {
		c.JSON(http.StatusUnauthorized, gin.H{
			"code": 401,
			"msg":  "用户名或密码错误",
		})
		return
	}

	c.JSON(http.StatusOK, gin.H{
		"code": 200,
		"msg":  "登录成功",
		"data": LoginResponse{
			UserID:   user.ID,
			Username: user.Username,
		},
	})
}
```

### 两个关键安全细节

**第一：模糊错误信息**

无论是"用户名不存在"还是"密码错误"，统一返回"用户名或密码错误"。

如果返回"用户名不存在"，攻击者就能用这个信息暴力枚举出哪些用户名已注册——他随便试一堆账号，返回"密码错误"的就说明这个账号存在。统一返回模糊信息，攻击者无法区分是账号不存在还是密码不对。

**第二：用 `?` 占位符，绝不拼SQL**

```go
// ✅ 正确：占位符，参数化查询
database.DB.QueryRow("SELECT id FROM users WHERE username = ?", req.Username)

// ❌ 绝对禁止：字符串拼接
database.DB.QueryRow("SELECT id FROM users WHERE username = '" + req.Username + "'")
```

如果用户输入的是 `' OR '1'='1`，第二种写法会变成：

```sql
SELECT id FROM users WHERE username = '' OR '1'='1'
```

这条SQL会返回**所有用户**——这就是SQL注入。用占位符 `?`，数据库驱动会自动转义特殊字符，从根本上杜绝注入。第82章会详细展开SQL注入的原理和防护。

### 路由注册

```go
func SetupAuthRoutes(r *gin.Engine) {
	auth := r.Group("/api/v1/auth")
	{
		auth.POST("/register", handler.Register)
		auth.POST("/login", handler.Login)
	}
}
```

## 本章小结

| 知识点 | 要点 |
|--------|------|
| Authentication（认证） | "你是谁？"——验证身份 |
| Authorization（授权） | "你能干什么？"——检查权限 |
| 401 vs 403 | 401=没认证，403=没权限 |
| 明文密码 | 绝对不能存，泄露即灾难 |
| 哈希 | 单向不可逆，SHA256等 |
| 盐（Salt） | 随机字符串，打破哈希确定性，防彩虹表 |
| 慢哈希 | 故意慢，防暴力破解 |
| bcrypt | 自带盐+慢哈希+可调cost的标准方案 |
| bcrypt.GenerateFromPassword | 生成哈希，cost=10是通用平衡点 |
| bcrypt.CompareHashAndPassword | 验证密码，内部自动提取盐 |
| 72字节限制 | bcrypt最大输入72字节，超长密码需预处理 |
| 模糊错误信息 | 登录失败统一返回"用户名或密码错误" |
| 参数化查询 | 用 `?` 占位符防SQL注入 |
