## 我在做什么？

你已经学了 MySQL、PostgreSQL、Redis、MongoDB。现在，你接到一个新项目，老板问你："我们用什么数据库？"

你不能说"哪个火用哪个"，也不能说"MySQL 最熟就用 MySQL"。你需要一个**系统化的选型方法**。

这一章，我把选型拆成 5 个维度，给你一个**场景映射表**和一个**决策树**，让你在 5 分钟内给出靠谱的数据库选型方案。

学完这一章，给定任意一个业务场景，你能独立完成数据库选型并说明理由。

## 一、选型五维度：你该问什么？

选数据库不是挑水果——看着顺眼就拿。你需要在下面五个维度上做权衡。

### 1.1 数据模型

**数据长什么样？**

| 数据形态 | 适合的数据库类型 | 具体数据库 |
|---------|----------------|-----------|
| 结构化表格，字段固定 | 关系型数据库 | MySQL, PostgreSQL |
| 半结构化数据，字段不固定 | 文档数据库 | MongoDB |
| 键值对，简单查询 | 键值存储 | Redis |
| 图数据（节点+关系） | 图数据库 | Neo4j, ArangoDB |
| 时间序列数据 | 时序数据库 | InfluxDB, TimescaleDB |
| 全文搜索 | 搜索引擎 | Elasticsearch |

**想多一点**：数据模型是选型的第一要素。如果你发现需要把 JSON 打散成 20 张表去做 JOIN，那 MongoDB 可能更合适。反过来，如果数据之间有严格的关联关系（如"用户→订单→商品→库存"），那关系型数据库更适合。

### 1.2 查询模式

**你怎么查数据？**

| 查询模式 | 适合的数据库 |
|---------|------------|
| 复杂的 JOIN、子查询、聚合 | MySQL, PostgreSQL |
| 按主键或已知 key 精确查询 | Redis |
| 按文档内容查询（嵌套字段） | MongoDB |
| 全文搜索、模糊匹配 | Elasticsearch, PostgreSQL |
| 聚合分析、大表扫描 | PostgreSQL（并行查询）, ClickHouse |

### 1.3 一致性要求

**数据能丢吗？能暂时不一致吗？**

| 一致性级别 | 含义 | 适合数据库 |
|-----------|------|-----------|
| 强一致性（ACID） | 写入后立即读到，不能丢 | MySQL, PostgreSQL |
| 最终一致性 | 暂时不一致，最终会一致 | MongoDB（默认）, Redis Cluster |
| 弱一致性 | 允许丢失最近写入 | Redis（纯缓存模式） |

**一句话**：金融、交易、账户系统 → 必须强一致性（MySQL/PostgreSQL）。社交网络、日志 → 最终一致性够用。

### 1.4 扩展性

**数据量和并发量有多大？**

| 数据量 | 方案 |
|--------|------|
| < 100GB，单机够用 | MySQL/PostgreSQL 单机 |
| 100GB ~ 1TB，需要读写分离 | MySQL 主从 + 读写分离 |
| 1TB ~ 10TB，需要分片 | MySQL 分库分表 或 MongoDB 分片 |
| > 10TB，超大规模 | 专用分布式方案（TiDB, CockroachDB） |

### 1.5 运维成本与社区生态

| 考虑因素 | 建议 |
|---------|------|
| 团队熟悉度 | 优先选团队熟悉的数据库 |
| 云服务支持 | AWS RDS 对 MySQL 最友好，PG 其次 |
| 招聘市场 | MySQL 人才最多，PG 其次，MongoDB 第三 |
| 社区活跃度 | MySQL > PG > MongoDB > Redis |
| 学习曲线 | MySQL < MongoDB < Redis 进阶 < PostgreSQL 进阶 |

## 二、场景 → 数据库映射表

这是本章最核心的一张表。把常见业务场景直接映射到推荐数据库。

| 业务场景 | 推荐数据库 | 理由 | 备选 |
|---------|-----------|------|------|
| **电商订单系统** | MySQL + Redis | 核心业务需要 ACID 事务，高并发用 Redis 缓存 | PostgreSQL |
| **社交网络（用户关系）** | MySQL + Redis + MongoDB | 关系数据用 MySQL，好友/关注用 Redis Set，动态/Feed 用 MongoDB | Neo4j（图关系） |
| **实时排行榜** | Redis（Sorted Set） | ZADD/ZREVRANGE 天然支持排行榜 | MySQL + 定时任务 |
| **日志分析** | MongoDB / Elasticsearch | 灵活 Schema，大量写入，全文搜索 | ClickHouse（OLAP） |
| **全文搜索** | Elasticsearch | 倒排索引，分词，相关性排序 | PostgreSQL 全文搜索 |
| **IoT 数据** | TimescaleDB / MongoDB | 时序数据压缩，时序查询优化 | InfluxDB |
| **内容管理系统（CMS）** | PostgreSQL（JSONB） / MongoDB | 灵活的内容结构，JSONB 支持索引 | MySQL |
| **用户画像/个性化** | MongoDB | 每个用户字段不同，嵌入式文档 | PostgreSQL JSONB |
| **缓存/会话存储** | Redis | 内存速度，自动过期 | Memcached |
| **消息队列（轻量）** | Redis（List/Stream） | 简单，快 | RabbitMQ, Kafka |
| **地理信息系统（GIS）** | PostgreSQL + PostGIS | PostGIS 是 GIS 领域事实标准 | MongoDB（GeoJSON） |
| **金融/支付系统** | PostgreSQL | 严格 ACID，强类型，精确数值（DECIMAL） | MySQL |
| **数据分析（OLAP）** | ClickHouse / PostgreSQL | 列式存储，分析查询优化 | MySQL（小规模） |
| **创业初期（最小可行产品）** | MySQL | 简单够用，别过度设计 | PostgreSQL |

## 三、混合架构：一个项目用多个数据库

**现实中的项目，很少只用一种数据库。** 每种数据库都有自己最擅长的事，把它们组合起来才是最优解。

### 3.1 经典混合架构

```
                     ┌─────────────┐
                     │   用户请求    │
                     └──────┬──────┘
                            │
              ┌─────────────┴─────────────┐
              │                           │
        ┌─────▼─────┐              ┌──────▼──────┐
        │   Redis   │              │ Elasticsearch│
        │  缓存/会话  │              │   全文搜索    │
        └─────┬─────┘              └──────┬──────┘
              │                           │
              └─────────────┬─────────────┘
                            │
                     ┌──────▼──────┐
                     │   MySQL /   │
                     │ PostgreSQL  │
                     │  核心业务数据  │
                     └──────┬──────┘
                            │
              ┌─────────────┴─────────────┐
              │                           │
        ┌─────▼─────┐              ┌──────▼──────┐
        │  MongoDB  │              │  ClickHouse  │
        │  日志/文档  │              │   数据分析    │
        └───────────┘              └─────────────┘
```

**各层职责**：
- **Redis**：热点数据缓存、用户会话、计数器、分布式锁
- **MySQL/PostgreSQL**：核心业务数据，所有需要事务的操作
- **Elasticsearch**：商品搜索、日志搜索、全文检索
- **MongoDB**：操作日志、用户行为日志、文档型数据
- **ClickHouse**：BI 报表、用户行为分析、大屏数据

### 3.2 但请注意：不要过度设计

上面这张图看起来很牛，但**创业初期不建议全部上**。

```
❌ 过度设计（日活 100 的博客系统）：
  MySQL + Redis + MongoDB + Elasticsearch + Kafka + ClickHouse
  → 运维成本爆炸，出问题没人敢动，招人要求"全栈数据库专家"月薪 5 万

✅ 合理架构（日活 100 的博客系统）：
  MySQL 单机
  → 够用，简单，好维护，一个人搞定
```

**演进路线**：
```
阶段 1：MySQL 单机（日活 < 1000）
  ↓ 搜索变慢
阶段 2：MySQL + Elasticsearch（日活 < 10000）
  ↓ 热点数据压力大
阶段 3：MySQL + Redis + Elasticsearch（日活 < 100000）
  ↓ 日志量大
阶段 4：MySQL + Redis + MongoDB + Elasticsearch（日活 > 100000）
  ↓ 分析需求
阶段 5：MySQL + Redis + MongoDB + Elasticsearch + ClickHouse
```

## 五、反模式：什么场景绝对不要用某种数据库

### 5.1 MySQL 的反模式

| 场景 | 为什么不要用 MySQL | 改用 |
|------|-------------------|------|
| 全文搜索（中文） | MySQL 全文搜索弱，中文尤其差 | Elasticsearch 或 PG |
| 大量 JSON 操作 | MySQL JSON 不如 PG JSONB 和 MongoDB | PostgreSQL 或 MongoDB |
| 高并发计数器 | UPDATE 行锁成为瓶颈 | Redis INCR |
| 海量日志 | 写入和压缩不如专用方案 | MongoDB 或 ClickHouse |
| 图关系查询 | 递归 CTE 支持有限，性能差 | Neo4j |

### 5.2 Redis 的反模式

| 场景 | 为什么不要用 Redis | 改用 |
|------|-------------------|------|
| 大数据量持久存储 | 内存贵，持久化不是强项 | MySQL/PostgreSQL |
| 复杂条件查询 | 没有 WHERE 多条件过滤 | MySQL/PostgreSQL |
| 需要 JOIN 的数据 | 不支持关联查询 | MySQL/PostgreSQL |
| 不可丢失的核心数据 | 持久化有窗口，可能丢数据 | PostgreSQL |

### 5.3 MongoDB 的反模式

| 场景 | 为什么不要用 MongoDB | 改用 |
|------|---------------------|------|
| 高度关联的数据 | JOIN 性能差，文档模型不匹配 | MySQL/PostgreSQL |
| 金融/支付系统 | 事务不如关系型数据库成熟 | PostgreSQL |
| 需要复杂事务 | 多文档事务性能不如 MySQL | MySQL/PostgreSQL |
| 需要 Schema 强约束 | MongoDB 天生无 Schema | MySQL/PostgreSQL |

## 七、常见错误

### 错误 1：追新技术——"MongoDB 火了就全用 MongoDB"

**现象**：听说 MongoDB 快，把整个电商系统都搬到 MongoDB 上，包括订单和支付。

❌ 订单和支付需要强 ACID 事务，MongoDB 的事务不如 MySQL/PostgreSQL。结果：数据不一致，用户付款后订单状态不对。

✅ **正确做法**：核心交易数据用 MySQL/PostgreSQL，日志和文档数据用 MongoDB。

### 错误 2：大材小用——"我们公司用 Elasticsearch 做所有查询"

**现象**：一个简单 CRUD 的博客系统，上了 Elasticsearch 做全文搜索，但其实用户根本没搜索需求。

❌ Elasticsearch 运维成本高，数据量小的时候 MySQL 的 LIKE 完全够用。

✅ 上了 Elasticsearch 就要有搜索需求，否则就是浪费。

### 错误 3：过度设计——"创业第一天就上微服务 + 多数据库"

**现象**：日活 100 的内部管理系统，上了 MySQL + Redis + MongoDB + Elasticsearch + Kafka。

❌ 5 个数据库组件，运维成本翻了 5 倍，出问题没人敢动，招人困难。

✅ **创业初期 MySQL 单机足够**。等你做到日活 1 万，再考虑加 Redis；日活 10 万，再考虑加 MongoDB/ES。

### 错误 4：只看功能不看团队

**现象**：选 PostgreSQL 因为"功能强大"，但团队 5 个人全是 MySQL 背景，没人懂 PG。

❌ 上线后出问题没人能修，紧急招人招不到。

✅ 选型要考虑团队能力。如果团队只熟悉 MySQL，除非有明确的需求 MySQL 无法满足，否则不要轻易换。

## 最可能出错的地方及原因

1. **过度设计**：创业初期就上 MySQL + Redis + MongoDB + ES + ClickHouse，运维成本爆炸，原因是不了解"架构演进"原则，以为"越多越好"。
2. **追新技术**：看到 MongoDB 火了就全部用 MongoDB，看到 PG 强就全迁 PG，原因是没做场景分析，只用"好不好"代替"适不适合"。
3. **忽略团队能力**：选了 PostgreSQL 但团队没人懂，上线后出问题无法修复，原因是选型时只看技术指标，忽略了"谁来维护"这个核心问题。
4. **用 MySQL 做全文搜索**：中文搜索效果差，用户体验不好，原因是不了解 MySQL 全文搜索的局限性，也没考虑引入 Elasticsearch。
5. **在不需要 ACID 的场景强行用 MySQL**：日志系统用 MySQL 存，写入性能瓶颈，原因是习惯性"什么数据都往 MySQL 塞"，没考虑 NoSQL 的优势场景。
