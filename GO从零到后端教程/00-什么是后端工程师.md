# 第0章 · 什么是后端工程师

> "后端工程师是做网站的？写代码的？修电脑的？——都不是。读完这一章，你会用一句话跟任何人讲清楚你到底在干什么。"

---

## 0.1 用餐厅比喻讲清楚前后端

你走进一家餐厅，点了宫保鸡丁，等了15分钟，菜来了。你有没有想过，这15分钟里发生了什么？

你在餐桌上做了什么：看菜单、决定要点什么、叫服务员、告诉服务员你要宫保鸡丁、等菜、买单。这就是**前端**——你直接接触的那一层，你能看到、能摸到、能交互的部分。

厨房里发生了什么：厨师接到订单、拿出鸡胸肉和花生、切菜、炒菜、调味、装盘。这就是**后端**——你看不见的那一层，它不跟你直接交互，但它决定了你能吃到什么、多久能吃到、好不好吃。

仓库里发生了什么：冰箱里存着鸡肉、花生、蔬菜，厨师需要的时候从仓库取。这就是**数据库**——存数据的地方。数据不会凭空出现，必须有个地方存着，下次有人问你"上次那个用户叫什么名字"，你才能翻出来。

互联网世界完全一样：

| 餐厅角色 | 互联网角色 | 具体举例 |
|----------|-----------|----------|
| 餐桌、菜单、服务员 | **前端** | 你的浏览器、手机App的界面 |
| 厨房、厨师 | **后端** | 服务器上跑的Go程序 |
| 冰箱、仓库 | **数据库** | MySQL、Redis |
| 点菜单 | **API请求** | HTTP请求（"我要查用户信息"） |
| 菜 | **API响应** | HTTP响应（"这是用户信息"） |

**换句话说：**
- 前端负责"好看"——界面漂不漂亮、按钮好不好点、动画炫不炫
- 后端负责"好用"——数据对不对、速度快不快、10万人同时用会不会崩
- 数据库负责"记住"——用户的信息有没有丢、订单还在不在

你作为后端工程师，你的工作就是在厨房里：设计菜谱（数据库表结构）、炒菜（写业务逻辑）、保证厨房不出事（处理错误）、高峰期不崩（高并发）。

## 0.2 一个HTTP请求的完整旅程

> 💡 下面这些代码你现在不需要看懂，先感受一下后端工程师写的东西长什么样 —— 后面每一章会一个一个拆开讲透。

说了一堆概念，现在我们跟着一个真实的请求走一遍。你在浏览器地址栏输入 `www.example.com/login`，按下回车——1秒钟不到页面就出来了。但就在这一秒里，发生了至少8件大事。

### 第1步：DNS查询——"翻译官开工"

你输入的是 `www.example.com`，但互联网上的设备只认IP地址（比如 `93.184.216.34`），不知道什么叫"域名"。

DNS（Domain Name System）就是互联网的电话号码簿。你的浏览器问DNS服务器：

> "www.example.com 的电话号码是多少？"

DNS服务器回答：

> "93.184.216.34"

这个过程叫**DNS解析**。就像你想给张三打电话，你得先翻手机通讯录找到张三的号码。

> 🤔 **想多一点**：你的浏览器不会每次都去问DNS服务器。它会缓存（记住）一段时间。比如你刚才查过百度，它就记住了，下次就不需要再翻通讯录了。这就是为什么有时候你改了一个网站的IP地址，可能要过几分钟甚至几小时才能生效——因为世界各地的人电脑里都还存着旧的"电话号码"。

### 第2步：TCP连接——"喂，听得到吗？"

现在浏览器知道了服务器的IP地址，要开始通话了。但不是说连就连的，需要先"打一个电话"确认对方在不在。

这个过程叫**TCP三次握手**：

```
你的浏览器                      服务器
   │                              │
   │──── "喂，在吗？" ──────────→  │  (SYN)
   │                              │
   │←── "在，你听得到我吗？" ────   │  (SYN + ACK)
   │                              │
   │──── "听得到，开始聊吧" ─────→  │  (ACK)
   │                              │
   │        ✅ 连接建立             │
```

为什么要三次？想象你给朋友打电话：
- 你："喂？" — 你在确认他接听了
- 他："喂，听得到吗？" — 他在确认你也能听到他
- 你："听得到" — 你确认了双方都能听到

这就是三次握手。三次完成之后，双方都知道：我能听到你，你也能听到我。

> 🤔 **想多一点**：为什么不两次？如果只有两次，可能出现这种情况：你发了一个"在吗"但因为网络延迟很久才到，你以为失败又发了一个，结果两个都到了服务器，服务器就建立了两次连接（浪费资源）。三次握手就是为了防止这种"延迟的旧请求"造成混乱。

### 第3步：TLS握手——"我们换个密码说话"

如果你访问的是 `https://www.example.com`（注意那个"s"），在TCP连接建立后，还有一步**TLS握手**，让后面的通信都加密。

简单说（后面第13章会详细讲）：
- 服务器出示"身份证"（SSL证书）
- 浏览器验证身份证是真的
- 双方协商一个"暗号"（加密密钥）
- 之后的对话用这个暗号加密

如果访问的是 `http://`（没有s），就没有这步，后面的所有数据都是明文传输的——就像你在大街上喊"我的密码是123456"，旁边的人全能听见。

### 第4步：浏览器发送HTTP请求——"来，这是我的订单"

TCP连接建立好了，TLS加密也搞定了，现在浏览器正式"点菜"：

```http
POST /login HTTP/1.1
Host: www.example.com
Content-Type: application/json
Content-Length: 45
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)

{"username":"zhangsan","password":"123456"}
```

拆开看：

- **`POST /login HTTP/1.1`**：我要用POST方法访问 /login 这个路径，用的是HTTP/1.1协议
- **`Host: www.example.com`**：我找的是这个网站（因为一台服务器可能托管多个网站）
- **`Content-Type: application/json`**：我发的数据是JSON格式
- **`Content-Length: 45`**：我的数据有45个字节长
- **`User-Agent`**：我是Windows上的Chrome浏览器
- **最下面那两行JSON**：这就是真正的数据——我要登录，用户名是zhangsan，密码是123456

### 第5步：Nginx收到请求——"前台转接电话"

请求到了服务器，第一个接电话的通常不是你的Go程序，而是**Nginx**（读作"Engine X"）。

Nginx是一个**反向代理服务器**（或者理解成"前台"）。它的工作是：
- 接收所有进来的请求
- 判断这个请求应该交给谁处理
- 如果是静态文件（图片、CSS），直接返回（不用麻烦Go程序）
- 如果是API请求（程序间通信接口，详见附录I），转发给Go程序
- 如果某个Go程序挂了，转发给另一个

想象一个公司前台：有人打电话来说"我要找财务部"，前台就转接到财务部；有人说"我要找技术部"，前台就转接到技术部。

```nginx
# Nginx配置示意
server {
    listen 80;
    server_name www.example.com;

    location / {
        proxy_pass http://127.0.0.1:8080;  # /login 这种就转给Go程序
    }
    location /static/ {
        root /var/www;  # 静态文件直接返回
    }
}
```

### 第6步：Go应用处理请求——"厨房开火"

请求终于到了你的Go程序（运行在8080端口）。Go程序要做的事情是：

**① 路由匹配**：/login 这个路径，该找哪个函数处理？

```go
r := gin.Default()
r.POST("/login", handleLogin)  // 我注册过：POST /login 由 handleLogin 处理
```

**② 解析请求**：把用户发过来的JSON变成Go能用的数据

```go
func handleLogin(c *gin.Context) {
    var req struct {
        Username string `json:"username"`
        Password string `json:"password"`
    }
    c.ShouldBindJSON(&req)  // JSON → Go结构体
    // req.Username = "zhangsan"
    // req.Password = "123456"
```

**③ 业务逻辑**：核心处理——查用户、验密码

```go
    // 去数据库查这个用户
    user, err := db.FindUserByUsername(req.Username)
    if err != nil {
        c.JSON(500, gin.H{"error": "服务器内部错误"})
        return
    }
    if user == nil {
        c.JSON(401, gin.H{"error": "用户名或密码错误"})
        return
    }
    // 验证密码（注意：存的是哈希值，不是明文！）
    if !bcrypt.CompareHashAndPassword(user.PasswordHash, []byte(req.Password)) {
        c.JSON(401, gin.H{"error": "用户名或密码错误"})
        return
    }
```

**④ 查数据库**：找到zhangsan这个用户

```sql
SELECT id, username, password_hash FROM users WHERE username = 'zhangsan';
```

MySQL返回一行数据：
```
id=1, username='zhangsan', password_hash='$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy'
```

注意：数据库里存的不是明文密码 `123456`，而是经过bcrypt加密后的乱码。为什么要这样？因为如果数据库被黑了，黑客拿到的也是一堆不可还原的乱码，而不是你的密码。

### 第7步：Go应用返回响应——"菜做好了"

密码验证通过，Go程序生成一个JWT令牌（一种"临时通行证"），返回给浏览器：

```go
    token, _ := generateJWT(user.ID, user.Username)
    c.JSON(200, gin.H{
        "code": 0,
        "msg":  "登录成功",
        "data": gin.H{
            "token": token,
            "user": gin.H{
                "id":       user.ID,
                "username": user.Username,
            },
        },
    })
```

返回的HTTP响应长这样：

```http
HTTP/1.1 200 OK
Content-Type: application/json

{
    "code": 0,
    "msg": "登录成功",
    "data": {
        "token": "eyJhbGciOiJIUzI1NiIs...",
        "user": {
            "id": 1,
            "username": "zhangsan"
        }
    }
}
```

### 第8步：浏览器收到响应——"端菜上桌"

浏览器收到JSON数据，前端JavaScript代码解析它：
- 把token存到localStorage（下次请求带着这个token，服务器就知道你是谁）
- 把页面跳转到首页
- 展示"欢迎，zhangsan"

你看，从你按下回车到看到页面，不到1秒的时间里，你的数据跑了这么一大圈。

### 完整流程图

```
你输入网址 → 按回车 → 
  ┌─ DNS解析：www.example.com → 93.184.216.34
  ├─ TCP三次握手：建立连接
  ├─ TLS握手：建立加密通道
  ├─ HTTP请求发出：POST /login ...
  ├─ Nginx转发 → Go应用（127.0.0.1:8080）
  ├─ Go处理：
  │   ├─ 路由匹配 → handleLogin
  │   ├─ 参数解析 → JSON → 结构体
  │   ├─ 数据库查询 → SELECT ... FROM users
  │   ├─ 密码验证 → bcrypt.Compare
  │   └─ 生成Token → JWT
  ├─ Go响应返回 → JSON
  ├─ Nginx转发 → 浏览器
  └─ 浏览器渲染 → 你看到页面
```

> 🤔 **想多一点**：你可能会问——"我好像从来没写过Nginx和DNS的代码啊？"没错，DNS你不用写（那是互联网基础设施），Nginx是运维配的（但你得知道它是什么）。你的工作是第6步和第7步——写Go程序处理请求、查数据库、返回结果。**但这些步骤你必须全部理解**，因为出问题的时候（比如"为什么用户登录这么慢？"），你可能需要从第1步排查到第8步。

## 0.3 后端工程师的日常真实工作

很多人以为后端工程师就是"每天对着黑窗口敲代码"。前半句对了，后半句不完全对。一个后端工程师真正的日常包括：

### 日常一：写接口（API）

这是最核心的工作。什么叫"写接口"？就是定义"外面怎么调用我的服务"。

比如老板说："做一个获取用户列表的功能"。你要做的事：
1. 设计URL：`GET /api/users?page=1&size=20`
2. 设计请求参数：page（第几页）、size（每页多少条）
3. 设计返回格式：
   ```json
   {
       "code": 0,
       "data": {
           "list": [用户列表],
           "total": 100
       }
   }
   ```
4. 写代码实现这个逻辑
5. 写上Swagger文档让别人知道怎么用

### 日常二：设计数据库表

数据不会凭空存在，数据库表就是你设计数据"怎么存"的地方。比如设计一张"用户表"：

```
你脑子里的概念：                   数据库里的表：
┌─────────────────────┐           ┌──────────────────────────────────┐
│ 张三                 │           │ id | username | email | created_at│
│ 25岁                │    →      │ 1  | zhangsan | z@x.com | 2024-01-01│
│ z@x.com             │           │ 2  | lisi     | l@x.com | 2024-01-02│
└─────────────────────┘           └──────────────────────────────────┘
```

你需要决定：
- 有哪些字段（id, username, email, password_hash...）
- 每个字段是什么类型（int, varchar, timestamp...）
- 哪个字段是主键（唯一标识一行，比如id）
- 哪些字段需要索引（加速查询）
- 表之间的关系（一个用户可以有多个订单）

### 日常三：写业务逻辑

业务逻辑就是"数据怎么处理"。比如：
- "用户下单时，扣库存、生成订单、发短信通知"——这些顺序、规则就是业务逻辑
- "满200减30，但不能和会员折扣同时使用"——这也是业务逻辑
- "密码必须包含大小写字母和数字，长度至少8位"——这也是业务逻辑

写业务逻辑就像写菜谱：先做什么、后做什么、如果缺材料怎么办、做好了怎么装盘。

### 日常四：修Bug

Bug就是程序不按预期工作。比如：
- 用户明明存在，登录却说"用户名不存在"
- 删了一个订单，但库存没加回去
- 100个人同时抢1件商品，卖出了3件

修Bug的流程：
1. 复现Bug（能稳定触发，才算找到真问题）
2. 看日志（服务器记录了什么）
3. 定位代码（哪一行出了问题）
4. 修复 + 测试（修完了要证明它真的好了）
5. Code Review（让同事看一眼你的修改没问题）

### 日常五：Code Review

写完代码不是直接上线，同事要先检查。Code Review就是"让别人看一眼你的代码有没有问题"。

检查什么：
- 逻辑对不对？
- 有没有安全漏洞？（比如SQL注入的风险）
- 性能有没有问题？（比如在循环里查数据库）
- 代码好不好读？（三个月后你自己还看得懂吗）
- 有没有写测试？

### 日常六：上线部署

代码写完、测试通过、Code Review完成，接下来是**上线**——把代码推到生产服务器上，让真实用户用。

一般流程：
1. 合并代码到主分支
2. CI/CD自动构建Docker镜像
3. 推到测试环境自动跑测试
4. 测试通过，推到生产环境
5. 观察监控看有没有异常

很多公司有专门的"发版日"（比如每周二、周四），不会随便什么时候都能上线。

### 日常七：排查线上问题

用户突然报错："登录不上了！" 这就是**线上问题**。

排查思路：
1. 看监控面板：CPU、内存、数据库连接数有没有异常？
2. 看错误日志：最近有什么error？
3. 看最近上线了什么：是不是新代码引入的问题？
4. 尝试复现：用什么操作能触发这个问题？
5. 找根因：不只是修表面问题，要找到根本原因
6. 紧急修复：如果影响大先回滚（把代码退回到上一个版本）
7. 写复盘报告：为什么出问题、怎么修、以后怎么避免

### 日常八：技术方案设计

老板说："我们要做一个秒杀功能，双11用。"

你不能直接开始写代码。你需要先设计**技术方案**：
- 秒杀的系统架构图
- 数据库怎么设计（库存表、订单表）
- 怎么防止超卖？（Redis + Lua脚本）
- 怎么应对高并发？（限流、排队）
- 消息队列（异步消息中间件，详见附录I）用什么？（RabbitMQ异步下单）
- 预估能支撑多少人同时抢？
- 需要多少台服务器？
- 开发周期多久？

技术方案写完了，还要开会讨论、评审、修改，大家都同意了才开始写代码。

### 总结：后端工程师的一天

```
09:00  看邮件、Slack，昨天上线的功能有没有问题？
09:30  站会（Standup Meeting）：昨天干了什么、今天要干什么、有没有卡住的
10:00  写代码（开发新功能）
12:00  午饭
14:00  Code Review同事的代码
15:00  开会讨论新功能的技术方案
16:00  修昨天测试报的Bug
17:00  写技术文档
18:00  看监控，确认没问题，下班
（如果有线上问题，随时被叫起来修）
```

### 学习方法建议

**1. 不要跳着看。** 这本书是严格按依赖关系排列的。你不会网络基础就学HTTP会懵，你不会Go就学Gin会懵。

**2. 每章的代码一定要自己敲一遍。** 看代码和写代码的差距，就像看别人游泳和自己跳下水的差距——看起来简单，一下水就沉。

**3. 遇到不懂的先记下来，继续往下看。** 很多概念是后面才解释的。比如你现在不理解"JWT是什么"，等学到第79章你就明白了。

**4. 学完一篇做一个对应的项目。** 学完Go语言写个小工具，学完数据库写个CRUD应用，学完Gin写个API服务。知识不用就会忘。

**5. 保持耐心。** 这本书有115章，不是一两个星期能看完的。每天学1~2章，4~6个月学完是合理的节奏。后端工程师不是速成的，但按这条路走，你学完之后会是一个**真正合格的后端工程师**——不是那种"会抄代码但不知道为什么"的码农。

## 本章小结

- **后端工程师 = 互联网餐厅的厨房**：你看不见它，但它决定了能不能出菜、好不好吃、快不快。
- **一个HTTP请求至少经过8个步骤**：DNS → TCP → TLS → 请求发出 → Nginx → Go程序 → 数据库 → 响应返回。
- **后端工程师的日常**不只是写代码，还包括设计数据库、Code Review、上线部署、排查线上问题、写技术方案。
- **这本书有115章**，从零基础到能独立开发后端系统，按顺序学不要跳。
