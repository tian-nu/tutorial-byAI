# 第61章 · 事务与隔离级别

> "你去ATM转账，输完金额点确定——屏幕上转圈三秒，然后弹出'转账成功'。但这三秒里发生了什么？你的账户扣了1000，朋友的账户加了1000，然后把这两个操作一起'盖章确认'。如果扣你的钱成功了但加朋友的钱失败了（比如服务器断电），那1000就凭空消失了。'事务'就是那个防丢按钮——要么全都完成，要么全都不做，没有中间状态。"

---

## 61.1 什么是事务

**事务（Transaction）是一组SQL操作，这些操作要么全部成功（COMMIT），要么全部失败回滚（ROLLBACK），像原子一样不可分割。**

银行转账是最经典的例子：

```sql
-- 从张三账户扣1000，给李四账户加1000
-- 这两个操作必须在同一个事务中，要么都做，要么都不做
UPDATE accounts SET balance = balance - 1000 WHERE name = '张三';
UPDATE accounts SET balance = balance + 1000 WHERE name = '李四';
```

没有事务保护的后果：
- 第1条执行成功（张三扣了1000），第2条执行前断电了 → 张三凭空少了1000块
- 这就是为什么银行系统的每个操作都要包在事务里

### 事务基本操作

```sql
-- 开启事务
START TRANSACTION;
-- 或
BEGIN;

-- 执行一组SQL...

-- 确认提交（永久生效）
COMMIT;

-- 回滚撤销（回到事务开始前的状态）
ROLLBACK;
```

### 自动提交

MySQL默认每条单独的SQL语句自动作为一个事务提交（`autocommit = ON`）。这意味着：

```sql
UPDATE users SET age = 26 WHERE id = 1;
-- 上面这条语句自动提交了，你没法ROLLBACK
```

要手动控制事务，必须先关闭自动提交或显式开启事务：

```sql
-- 方法一：显式开启事务（推荐）
START TRANSACTION;
UPDATE users SET age = 26 WHERE id = 1;
-- 此时别的连接看不到这个修改
COMMIT;  -- 或者 ROLLBACK;

-- 方法二：关闭自动提交（影响整个会话）
SET autocommit = 0;
UPDATE users SET age = 26 WHERE id = 1;
COMMIT;
```

## 61.2 ACID：事务的四大特性

| 特性 | 含义 | 生活比喻 |
|------|------|---------|
| **A**tomicity（原子性） | 事务中的操作不可分割，全做或全不做 | 结婚登记：要么两人都写进户口本，要么都不写 |
| **C**onsistency（一致性） | 事务前后，数据始终满足所有约束规则 | 转账前后，总金额不变（能量守恒） |
| **I**solation（隔离性） | 并发事务之间互不干扰 | 多个柜台同时办业务，各办各的互不影响 |
| **D**urability（持久性） | 事务一旦提交，数据永久保存，即使断电也不丢 | 结婚证印出来，撕都撕不掉 |

> 🤔 **想多一点**：ACID中，A和D由MySQL的redo log和undo log保证；C由数据库约束+应用逻辑共同保证；I由锁和MVCC保证。隔离性是最复杂的，也是本章的重点。

## 61.3 并发事务的问题

当多个事务同时操作同一数据时，会出现三种经典问题：

### 61.3.1 脏读（Dirty Read）

**一个事务读到了另一个事务尚未提交的修改。**

```
时间线：
事务A：UPDATE users SET age = 30 WHERE id = 1;  -- 但还没提交
事务B：SELECT age FROM users WHERE id = 1;       -- 读到了30
事务A：ROLLBACK;                                   -- A回滚了！age还是25
事务B：已经按age=30做了后续业务逻辑 → 数据混乱
```

### 61.3.2 不可重复读（Non-Repeatable Read）

**同一事务内，两次读同一行数据，结果不一样（被别的事务UPDATE并提交了）。**

```
时间线：
事务A：SELECT age FROM users WHERE id = 1;  -- 读到25
事务B：UPDATE users SET age = 30 WHERE id = 1; COMMIT;
事务A：SELECT age FROM users WHERE id = 1;  -- 读到30！和第一次不一样
```

### 61.3.3 幻读（Phantom Read）

**同一事务内，两次查询同一个范围，结果集的行数不一样（别的事务INSERT了新数据）。**

```
时间线：
事务A：SELECT * FROM users WHERE age > 20;  -- 返回5行
事务B：INSERT INTO users (age) VALUES (25); COMMIT;
事务A：SELECT * FROM users WHERE age > 20;  -- 返回6行！多了一行"幻影"
```

| 问题 | 区别 | 本质 |
|------|------|------|
| 脏读 | 读了未提交的数据 | 读到"脏"数据 |
| 不可重复读 | 同一行两次读不一样 | 行内容变了（UPDATE） |
| 幻读 | 同样的条件查出行数不一样 | 行数变了（INSERT/DELETE） |

## 61.4 四种隔离级别

MySQL提供了四种隔离级别来应对上述问题：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 并发性能 |
|----------|------|-----------|------|---------|
| READ UNCOMMITTED（读未提交） | ✅ 存在 | ✅ 存在 | ✅ 存在 | 最高 |
| READ COMMITTED（读已提交） | ❌ | ✅ 存在 | ✅ 存在 | 高 |
| REPEATABLE READ（可重复读） | ❌ | ❌ | 部分解决 | 中 |
| SERIALIZABLE（串行化） | ❌ | ❌ | ❌ | 最低 |

MySQL默认隔离级别是 **REPEATABLE READ**。

### 查看和设置隔离级别

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;
-- MySQL 8.0 之前用 SELECT @@tx_isolation;

-- 设置全局隔离级别
SET GLOBAL transaction_isolation = 'READ-COMMITTED';

-- 设置当前会话的隔离级别
SET SESSION transaction_isolation = 'READ-COMMITTED';
```

### READ UNCOMMITTED 演示

```sql
-- 会话A：
SET SESSION transaction_isolation = 'READ-UNCOMMITTED';
START TRANSACTION;
SELECT age FROM users WHERE id = 1;  -- 25

-- 会话B：
START TRANSACTION;
UPDATE users SET age = 30 WHERE id = 1;
-- 不提交！

-- 会话A：
SELECT age FROM users WHERE id = 1;  -- 30 ← 脏读！
```

### READ COMMITTED 演示

```sql
-- 会话A：
SET SESSION transaction_isolation = 'READ-COMMITTED';
START TRANSACTION;
SELECT age FROM users WHERE id = 1;  -- 25

-- 会话B：
UPDATE users SET age = 30 WHERE id = 1;
COMMIT;

-- 会话A：
SELECT age FROM users WHERE id = 1;  -- 30 ← 不可重复读！
```

### REPEATABLE READ 演示

```sql
-- 会话A：
SET SESSION transaction_isolation = 'REPEATABLE-READ';
START TRANSACTION;
SELECT age FROM users WHERE id = 1;  -- 25

-- 会话B：
UPDATE users SET age = 30 WHERE id = 1;
COMMIT;

-- 会话A：
SELECT age FROM users WHERE id = 1;  -- 还是25！← 可重复读
```

### SERIALIZABLE 演示

```sql
-- 会话A：
SET SESSION transaction_isolation = 'SERIALIZABLE';
START TRANSACTION;
SELECT * FROM users WHERE age > 20;

-- 会话B：
INSERT INTO users (age) VALUES (25);
-- 阻塞等待！直到会话A提交或回滚
```

## 61.5 MVCC：MySQL如何实现事务隔离

MySQL InnoDB使用**MVCC**（Multi-Version Concurrency Control，多版本并发控制）来实现高并发下的事务隔离。核心思想：**读不阻塞写，写不阻塞读**。

### 61.5.1 MVCC的核心机制

每行数据在InnoDB中都有两个隐藏列：

| 隐藏列 | 含义 |
|--------|------|
| `DB_TRX_ID`（6字节） | 最后一次修改（INSERT/UPDATE）这行的事务ID |
| `DB_ROLL_PTR`（7字节） | 回滚指针，指向undo log中的上一个版本 |

```
行数据：
┌─────┬────────┬─────┬─────────────┬──────────────┐
│ id  │  name  │ age │ DB_TRX_ID   │ DB_ROLL_PTR  │
├─────┼────────┼─────┼─────────────┼──────────────┤
│  1  │ 张三   │  30 │    100      │ ───指向undo──→ │
└─────┴────────┴─────┴─────────────┴──────┬───────┘
                                          │
                         ┌────────────────┘
                         ▼
                  ┌──────────────┐
                  │  undo log    │
                  │ name: 张三   │
                  │ age:  25     │  ← 事务99之前的版本
                  │ trx_id: 99   │
                  └──────┬───────┘
                         │
                         ▼
                  ┌──────────────┐
                  │  undo log    │
                  │ name: 张三   │
                  │ age:  20     │  ← 更早的版本
                  │ trx_id: 80   │
                  └──────────────┘
```

### 61.5.2 ReadView：决定看到哪个版本

当事务执行SELECT时，InnoDB会生成一个**ReadView**（读视图），包含：

- 当前系统中活跃（未提交）的事务ID列表
- 最小活跃事务ID
- 下一个要分配的事务ID

对于每一行数据，判断规则：
1. 如果 `DB_TRX_ID` < 最小活跃事务ID → 这个版本在当前事务开始前已提交，**可见**
2. 如果 `DB_TRX_ID` 在活跃事务列表中 → 这个版本是未提交事务修改的，**不可见**，沿着 `DB_ROLL_PTR` 找上一个版本
3. 如果 `DB_TRX_ID` ≥ 下一个事务ID → 这个版本是"未来的"，**不可见**，找上一个版本

### 61.5.3 READ COMMITTED vs REPEATABLE READ 的MVCC差异

两者的实现几乎一样，只有一个区别：

| 隔离级别 | ReadView生成时机 | 效果 |
|----------|-----------------|------|
| READ COMMITTED | 每次SELECT都生成新的ReadView | 能看到别的事务提交的修改 → 不可重复读 |
| REPEATABLE READ | 事务第一次SELECT时生成，之后复用 | 整个事务看到的都是同一份快照 → 可重复读 |

## 61.6 实战：Spring中的事务管理（预览）

到Spring Boot章节才会详细展开，这里先看一眼事务在Java代码中的样子：

```java
@Service
public class AccountService {

    @Autowired
    private AccountMapper accountMapper;

    @Transactional  // 这个方法中的所有数据库操作在一个事务中
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        accountMapper.decreaseBalance(fromId, amount);
        // 如果这里抛异常，上面的decreaseBalance也会回滚
        accountMapper.increaseBalance(toId, amount);
    }
}
```

`@Transactional` 就是告诉Spring："下面这个方法里的所有SQL，要么全成功，要么全回滚。"

## 本章小结

| 概念 | 要点 |
|------|------|
| 事务 | 一组SQL操作的原子集合，COMMIT或ROLLBACK |
| ACID | 原子性、一致性、隔离性、持久性 |
| 脏读 | 读到别的事务未提交的修改 |
| 不可重复读 | 同一事务内两次读同一行结果不同 |
| 幻读 | 同一事务内两次查询范围行数不同 |
| 隔离级别 | READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ → SERIALIZABLE |
| MVCC | 多版本并发控制，通过DB_TRX_ID+DB_ROLL_PTR+ReadView实现快照读 |
| ReadView | 决定当前事务能看到哪些版本的数据 |
| MySQL默认 | REPEATABLE READ，解决了脏读和不可重复读 |

3. **MySQL默认隔离级别是什么？它解决了哪些问题？还有哪个问题没完全解决？**

<details>
<summary>参考答案（做完再看）</summary>

1. ```sql
START TRANSACTION;
UPDATE accounts SET balance = balance - 500 WHERE id = 1;
UPDATE accounts SET balance = balance + 500 WHERE id = 2;
-- 实际生产还要先检查余额是否够
SELECT balance FROM accounts WHERE id = 1;
-- 如果 balance >= 0: COMMIT; 否则: ROLLBACK;
COMMIT;
```

2. 不可重复读：事务A内两次读同一行，第一次age=25，第二次age=30（因为事务B在中间UPDATE并提交了）。区别是"同一行的值变了"。幻读：事务A内两次 `SELECT * FROM users WHERE age > 20`，第一次返回5行，第二次返回6行（因为事务B在中间INSERT了新数据并提交了）。区别是"行数变了"——不可重复读针对的是UPDATE（已有行的值），幻读针对的是INSERT/DELETE（多了或少了行）。

3. MySQL默认隔离级别是REPEATABLE READ。它解决了脏读和不可重复读。幻读在InnoDB中通过MVCC+间隙锁（Gap Lock）部分解决——对于快照读（普通SELECT）不会出现幻读，但对于当前读（SELECT ... FOR UPDATE）理论上可能出现幻读（但InnoDB的间隙锁实际也解决了这个问题）。
</details>
