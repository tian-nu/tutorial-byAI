# 第106章 · 单元测试

> "你怎么证明自己写的代码是对的？跑一遍看看？如果只有一组输入，那确实可以手动跑。但如果你的函数有几十种边界情况呢？如果同事改了你的代码，怎么确保他没改坏？答案就是单元测试——让机器替你自动验证每一段逻辑。它不是负担，而是你的安全网。"

---

## 106.1 什么是单元测试

### 类比：工厂的质检流水线

一条流水线生产手机。每台手机下线后，机器人会自动检查：屏幕亮不亮？按键灵不灵？摄像头能不能拍照？

单元测试（对单个函数输入输出自动验证的测试方法，详见附录I）就是代码的质检机器人：**自动运行你的函数，给定输入，检查输出是否和预期一致。**

### 为什么必须写测试

**没有测试的痛苦经历：**
1. 你写了 `CalculateDiscount` 函数，手动跑了几个例子，看起来都对
2. 三个月后，同事修改了 `CalculateDiscount` 的逻辑
3. 测试环境跑了一遍主要流程——正常
4. 上线后第二天，财务部炸了："为什么会员的折扣算错了！"
5. 排查发现：同事改了普通用户的逻辑，但破坏了会员的特例计算——而那个特例只有特定的时间段才会触发

**有测试的世界：**
1. 你写 `CalculateDiscount` 时，同时写了15个测试用例（覆盖普通用户、会员、黑五促销、0元订单等）
2. 同事修改逻辑后运行 `go test`——3个测试用例红了
3. 同事立刻知道自己的修改破坏了哪些边界情况
4. 修复后所有测试通过 → 安全上线

### 测试金字塔

```
        /\
       /E2E\         端到端测试：模拟真实用户操作（少而慢）
      /------\
     / 集成测试 \     集成测试：测多个模块协作（适量）
    /----------\
   /   单元测试   \   单元测试：测单个函数/方法（多而快）
  /--------------\
```

按照**测试金字塔**（将测试分为单元、集成、端到端三层，越底层越应多写的策略模型，详见附录I）原则，一个健康项目的测试分布应该是：**大量单元测试 + 适量集成测试 + 少量端到端测试**。

单元测试跑得最快（秒级）、写得最快、维护成本最低——所以你应该写最多的单元测试。

## 106.2 Go testing包

### 基本规则

1. 测试文件必须以 `_test.go` 结尾
2. 测试函数必须以 `Test` 开头，接受 `*testing.T` 参数
3. 测试文件和被测代码在同一个包中

**示例：** `calculator.go` 对应 `calculator_test.go`

### 第一个测试

被测代码 `calculator.go`：

```go
package calc

func Add(a, b int) int {
	return a + b
}
```

测试代码 `calc_test.go`：

```go
package calc

import "testing"

func TestAdd(t *testing.T) {
	got := Add(2, 3)
	want := 5

	if got != want {
		t.Errorf("Add(2, 3) = %d; want %d", got, want)
	}
}
```

运行：
```bash
go test ./...
go test -v ./...          # 显示详细信息（每个测试函数名和结果）
```

### t.Error vs t.Fatal vs t.Errorf

| 方法 | 行为 |
|------|------|
| `t.Error(args...)` | 标记失败，但继续执行当前测试的后续代码 |
| `t.Errorf(format, args...)` | 同Error，但支持格式化字符串 |
| `t.Fatal(args...)` | 标记失败，立即停止当前测试函数 |
| `t.Fatalf(format, args...)` | 同Fatal，但支持格式化 |
| `t.Log(args...)` | 仅在`-v`模式下打印日志（即使测试通过） |
| `t.Logf(format, args...)` | 同上，带格式化 |

**什么时候用Fatal？** 当前步骤失败后，后续步骤没有意义时。比如测试数据库操作时，如果第一步"连接数据库"失败，后面增删改查都跑不了——用Fatal终止。

```go
func TestDatabaseInsert(t *testing.T) {
    db, err := connectDB()
    if err != nil {
        t.Fatalf("数据库连接失败: %v", err)  // 连不上数据库，后面没法测
    }

    id, err := InsertUser(db, "Alice")
    if err != nil {
        t.Logf("插入失败: %v", err)
        t.Errorf("期望成功插入，但发生了错误")
    }
}
```

### t.Run：子测试

把一个测试函数拆成多个独立的子测试——某个子测试失败不影响其他：

```go
func TestAdd(t *testing.T) {
	t.Run("正数相加", func(t *testing.T) {
		if Add(2, 3) != 5 {
			t.Error("2+3 应该等于5")
		}
	})

	t.Run("负数相加", func(t *testing.T) {
		if Add(-1, -1) != -2 {
			t.Error("-1+-1 应该等于-2")
		}
	})

	t.Run("零加正数", func(t *testing.T) {
		if Add(0, 5) != 5 {
			t.Error("0+5 应该等于5")
		}
	})
}
```

`t.Run` 还可以和 `go test -run` 配合：`go test -run "TestAdd/正数相加"` 只跑这一个子测试。

## 106.3 表格驱动测试（Go推荐风格）

Go社区强烈推荐**表格驱动测试**：定义一个包含输入和期望输出的结构体切片，然后循环执行。

### 示例：测试计算器

```go
func TestAdd(t *testing.T) {
	tests := []struct {
		name string
		a    int
		b    int
		want int
	}{
		{"正数相加", 2, 3, 5},
		{"负数相加", -1, -1, -2},
		{"正负数相加", 1, -1, 0},
		{"零相加", 0, 0, 0},
		{"大数相加", 1000000, 2000000, 3000000},
	}

	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			got := Add(tt.a, tt.b)
			if got != tt.want {
				t.Errorf("Add(%d, %d) = %d; want %d", tt.a, tt.b, got, tt.want)
			}
		})
	}
}
```

### 表格驱动测试的优势

1. **易于扩展**：想加新用例？在切片里加一行
2. **一目了然**：所有输入和期望输出集中在一个地方
3. **每个用例独立**：通过 `t.Run`，某个用例失败不影响其他
4. **文档作用**：表格本身就是最好的使用示例

### 更多实战例子

**测试字符串处理函数：**

```go
func TestToUpper(t *testing.T) {
	tests := []struct {
		name  string
		input string
		want  string
	}{
		{"全小写", "hello", "HELLO"},
		{"全大写", "HELLO", "HELLO"},
		{"混合", "Hello World", "HELLO WORLD"},
		{"空字符串", "", ""},
		{"含数字", "hello123", "HELLO123"},
		{"中文", "你好", "你好"},
		{"特殊字符", "a@b#c", "A@B#C"},
	}

	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			got := strings.ToUpper(tt.input)
			if got != tt.want {
				t.Errorf("ToUpper(%q) = %q; want %q", tt.input, got, tt.want)
			}
		})
	}
}
```

**测试错误情况：**

```go
func TestDivide(t *testing.T) {
	tests := []struct {
		name      string
		a, b      float64
		want      float64
		wantError bool
	}{
		{"正常除法", 10, 2, 5, false},
		{"除不尽", 7, 3, 2.3333333333333335, false},
		{"除数为零", 10, 0, 0, true},
		{"零除以某数", 0, 5, 0, false},
	}

	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			got, err := Divide(tt.a, tt.b)
			if (err != nil) != tt.wantError {
				t.Errorf("Divide() error = %v, wantError %v", err, tt.wantError)
			}
			if !tt.wantError && got != tt.want {
				t.Errorf("Divide() = %v; want %v", got, tt.want)
			}
		})
	}
}
```

## 106.4 测试覆盖率

### 什么是测试覆盖率

测试覆盖率 = 被测试执行过的代码行数 / 总代码行数。它回答了一个朴素的问题：你的测试跑到了哪些代码，没跑到哪些？

### 查看覆盖率

```bash
go test -cover                          # 命令行显示覆盖率百分比
go test -coverprofile=coverage.out      # 生成覆盖率数据文件
go tool cover -html=coverage.out        # 在浏览器中可视化（绿色=覆盖，红色=未覆盖）
go tool cover -func=coverage.out        # 按函数查看覆盖率
```

浏览器打开后，你可以看到每行代码的颜色：
- **绿色**：被测试覆盖了
- **红色**：没被测试覆盖
- **灰色**：不是可执行代码（如注释、空行）

### 覆盖率不是全部

⚠️ **100%覆盖率 ≠ 没问题。** 因为覆盖率只能告诉你"代码被执行了"，不能告诉你"代码被执行正确了"。

```go
func Divide(a, b int) int {
    return a / b   // 覆盖率100%，但如果b=0会panic
}
```

测试 `Divide(10, 2)` 就能达到100%覆盖率——但没测 `b=0` 的情况。

好测试的核心不是覆盖率而是**边界条件覆盖**：正常值、边界值、异常值、空值、极大值。

### 合理覆盖率目标

| 项目类型 | 建议覆盖率 |
|----------|-----------|
| 核心业务逻辑 | ≥80% |
| 工具函数/库 | ≥90% |
| CRUD代码 | ≥60% |
| 前端展示逻辑 | 不用强求 |

## 106.5 testify断言库

Go标准库的 `if got != want { t.Error(...) }` 太啰嗦了。`testify` 让断言代码简洁得多。

### 安装

```bash
go get github.com/stretchr/testify
```

### assert vs require

- `assert`：断言失败后继续执行（标记测试失败但不终止）
- `require`：断言失败后立即终止当前测试

```go
import (
    "testing"
    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
)

func TestWithTestify(t *testing.T) {
    assert.Equal(t, 5, Add(2, 3))
    assert.NotEqual(t, 6, Add(2, 3))
    assert.True(t, IsValid("hello"))
    assert.False(t, IsValid(""))
    assert.Nil(t, err)
    assert.NotNil(t, result)
    assert.Contains(t, "hello world", "world")
    assert.Greater(t, 10, 5)
    assert.Less(t, 3, 7)
    assert.Error(t, err)
    assert.NoError(t, err)
    assert.Len(t, slice, 3)
    assert.Empty(t, "")
    assert.NotEmpty(t, "data")
    assert.JSONEq(t, `{"name":"Alice"}`, string(jsonBytes))
}

func TestRequireExample(t *testing.T) {
    db, err := connectDB()
    require.NoError(t, err)    // 如果err!=nil，测试立即终止
    require.NotNil(t, db)

    id, err := InsertUser(db, "Alice")
    assert.NoError(t, err)
    assert.Greater(t, id, 0)
}
```

### 对比：有无testify

**标准库写法：**
```go
if got != want {
    t.Errorf("Add(2,3) = %d; want %d", got, want)
}
```

**testify写法：**
```go
assert.Equal(t, want, got)
```

简洁了不止一点。

## 106.6 测试辅助函数和Setup/Teardown

### Setup和Teardown

有些测试需要准备环境（创建临时数据库、初始化配置），测完后需要清理。

```go
func TestMain(m *testing.M) {
    setup()
    code := m.Run()
    teardown()
    os.Exit(code)
}

func setup() {
    fmt.Println("所有测试开始前的准备工作")
}

func teardown() {
    fmt.Println("所有测试结束后的清理工作")
}
```

### t.Cleanup（Go 1.14+）

给每个测试注册清理函数，测试结束后自动执行：

```go
func TestSomething(t *testing.T) {
    tmpDir := t.TempDir()           // 自动创建临时目录，测试结束自动删除

    db, err := sql.Open("sqlite3", ":memory:")
    require.NoError(t, err)
    t.Cleanup(func() {
        db.Close()
    })

    file, err := os.CreateTemp("", "test-*.txt")
    require.NoError(t, err)
    t.Cleanup(func() {
        os.Remove(file.Name())
    })

    // 正常写测试逻辑...
}
```

`t.Cleanup` 的好处：清理函数紧挨着创建代码，逻辑清晰，不怕遗漏。

## 106.7 实战：为Blog系统的Repository层写完整测试

### 被测代码

假设我们有之前Blog系统的Repository层：

```go
package repository

import (
    "database/sql"
    "errors"
)

type Post struct {
    ID      int64
    Title   string
    Content string
    UserID  int64
}

type PostRepo struct {
    db *sql.DB
}

func NewPostRepo(db *sql.DB) *PostRepo {
    return &PostRepo{db: db}
}

func (r *PostRepo) GetByID(id int64) (*Post, error) {
    post := &Post{}
    err := r.db.QueryRow(
        "SELECT id, title, content, user_id FROM posts WHERE id = ?", id,
    ).Scan(&post.ID, &post.Title, &post.Content, &post.UserID)
    if errors.Is(err, sql.ErrNoRows) {
        return nil, nil
    }
    return post, err
}

func (r *PostRepo) Create(post *Post) (int64, error) {
    result, err := r.db.Exec(
        "INSERT INTO posts (title, content, user_id) VALUES (?, ?, ?)",
        post.Title, post.Content, post.UserID,
    )
    if err != nil {
        return 0, err
    }
    return result.LastInsertId()
}
```

### 测试代码

```go
package repository

import (
    "database/sql"
    "testing"

    _ "github.com/mattn/go-sqlite3"
    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
)

func setupTestDB(t *testing.T) *sql.DB {
    db, err := sql.Open("sqlite3", ":memory:")
    require.NoError(t, err)

    _, err = db.Exec(`
        CREATE TABLE posts (
            id INTEGER PRIMARY KEY AUTOINCREMENT,
            title TEXT NOT NULL,
            content TEXT NOT NULL,
            user_id INTEGER NOT NULL
        )
    `)
    require.NoError(t, err)

    t.Cleanup(func() {
        db.Close()
    })

    return db
}

func TestPostRepo_Create(t *testing.T) {
    db := setupTestDB(t)
    repo := NewPostRepo(db)

    post := &Post{
        Title:   "测试文章",
        Content: "这是测试内容",
        UserID:  1,
    }

    id, err := repo.Create(post)
    require.NoError(t, err)
    assert.Greater(t, id, int64(0))
    assert.Equal(t, int64(1), id)
}

func TestPostRepo_GetByID(t *testing.T) {
    db := setupTestDB(t)
    repo := NewPostRepo(db)

    t.Run("存在的数据", func(t *testing.T) {
        post := &Post{Title: "Test", Content: "Content", UserID: 1}
        id, err := repo.Create(post)
        require.NoError(t, err)

        got, err := repo.GetByID(id)
        require.NoError(t, err)
        require.NotNil(t, got)
        assert.Equal(t, "Test", got.Title)
        assert.Equal(t, "Content", got.Content)
    })

    t.Run("不存在的数据", func(t *testing.T) {
        got, err := repo.GetByID(99999)
        require.NoError(t, err)
        assert.Nil(t, got)
    })

    t.Run("负数ID", func(t *testing.T) {
        got, err := repo.GetByID(-1)
        require.NoError(t, err)
        assert.Nil(t, got)
    })
}
```

**测试策略解析：**

1. 用SQLite内存模式替代MySQL——测试环境不需要真实MySQL，跑得飞快
2. `setupTestDB` 封装了数据库初始化，每个测试都能得到干净的数据库
3. `t.Cleanup` 确保数据库连接被关闭
4. 对 `GetByID` 测试了三种情况：找到、找不到、边界值（负数）
5. 用 `require.NoError` 做前置条件检查（出错就没法继续），用 `assert.Equal` 做断言

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 单元测试 | 自动验证单个函数行为的代码 |
| 测试金字塔 | 大量单元测试 + 适量集成测试 + 少量端到端测试 |
| _test.go | 测试文件命名规范 |
| TestXxx | 测试函数命名规范，参数 `*testing.T` |
| t.Error/Fatal | Error标记失败继续，Fatal终止 |
| t.Run | 子测试，独立运行，互不影响 |
| 表格驱动测试 | 结构体切片定义用例，循环执行，Go推荐风格 |
| 测试覆盖率 | go test -cover，目标80%+，但边界条件比覆盖率高更重要 |
| testify | assert.Equal/NoError/NotNil等，让断言更简洁 |
| require vs assert | require失败立即终止，assert继续 |
| t.Cleanup | 注册清理函数，测试结束自动执行 |
| SQLite内存模式 | 数据库单元测试的最佳方案 |
