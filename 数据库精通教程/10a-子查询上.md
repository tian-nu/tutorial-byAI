## 我在做什么？

你已经在第 09 章学会了用 `GROUP BY` 统计"各城市用户数"、用 `HAVING` 筛选"订单数超过 5 的城市"。这些查询都只需要一次 `SELECT`。

但有些问题，一次 `SELECT` 搞不定。比如：
- "查一下哪些订单的金额超过了平均订单金额"——你得先算出平均金额，再用这个结果去比较。
- "查一下买过东西的用户"——你得先去订单表找出哪些 `user_id` 下过单，再回用户表查这些用户。

这种"查询中套查询"的需求，要靠**子查询**（Subquery）*此术语见附录E*。

学完这一章，你能用子查询写出"超过平均值的订单""买过东西的用户"这类需要两步计算的查询。

## 一、什么是子查询？SELECT 里面套 SELECT

子查询，顾名思义：**一个查询嵌套在另一个查询里面**。里面的叫"子查询"（内层查询），外面的叫"外层查询"（主查询）。

最直观的例子：

```sql
-- 你想查"订单金额超过平均金额的订单"
-- 首先，你得知道平均金额是多少：
SELECT AVG(amount) FROM orders;  -- 假设返回 250.00

-- 然后，用这个结果去查：
SELECT * FROM orders WHERE amount > 250.00;
```

这两个步骤，能不能合并成一条 SQL？能——用子查询：

```sql
SELECT * FROM orders
WHERE amount > (SELECT AVG(amount) FROM orders);
```

上面这条 SQL 的执行顺序是：**先执行括号里的子查询，算出平均值，再执行外层查询**。MySQL 自动帮你做了"两步走"。

> **想多一点**：子查询就像你让助手去帮你查一个数，然后你拿着这个数做决策。你不需要自己跑两步——你只需要说"帮我查一下那些金额大于（你去查一下平均金额）的订单"，助手自动帮你跑两步，最后把结果给你。这就是子查询的优雅之处：**一次请求，完成多次查询**。

## 二、子查询的三种返回类型

在深入之前，先搞清楚子查询的"分类法"。按返回结果的不同，子查询分为三类：

| 类型 | 返回什么 | 能放在哪 | 本章讲吗 |
|------|---------|---------|---------|
| **标量子查询** | 单个值（一行一列） | SELECT / WHERE / HAVING | ✅ 本章 |
| **列子查询** | 一列多行 | WHERE / HAVING（配合 IN） | ✅ 本章 |
| **行子查询** | 一行多列 | WHERE（配合 `=`） | ❌ 10b 章 |
| **表子查询** | 多行多列 | FROM（做派生表） | ❌ 10b 章 |

本章先搞定前两种。

## 三、标量子查询：返回单个值

**标量子查询**（Scalar Subquery）*此术语见附录E* 返回的是**一个值**——一行一列，就像数学里的"标量"（一个数字）。

### 3.1 用在 WHERE 中

最常见的场景：**和某个聚合值比较**。

```sql
-- 场景：查"订单金额超过平均值的所有订单"
SELECT id, user_id, amount
FROM orders
WHERE amount > (SELECT AVG(amount) FROM orders);
```

**执行过程**：
1. 子查询 `SELECT AVG(amount) FROM orders` 算出平均值，假设得到 `250.00`
2. 外层查询变成 `SELECT ... WHERE amount > 250.00`
3. 返回结果

### 3.2 用在 SELECT 中

你可以在 `SELECT` 后面放标量子查询，把计算结果作为一列展示：

```sql
-- 场景：查每个订单，同时显示"它和平均值的差距"
SELECT
    id,
    amount,
    (SELECT AVG(amount) FROM orders) AS avg_amount,
    amount - (SELECT AVG(amount) FROM orders) AS diff_from_avg
FROM orders;
```

输出效果：

```
+----+--------+------------+---------------+
| id | amount | avg_amount | diff_from_avg |
+----+--------+------------+---------------+
|  1 | 300.00 |    250.00  |        50.00  |
|  2 | 150.00 |    250.00  |      -100.00  |
|  3 | 400.00 |    250.00  |       150.00  |
|  4 | 200.00 |    250.00  |       -50.00  |
+----+--------+------------+---------------+
```

### 3.3 用在 HAVING 中

```sql
-- 场景：查"订单总额超过平均值的用户"
SELECT user_id, SUM(amount) AS total
FROM orders
GROUP BY user_id
HAVING SUM(amount) > (SELECT AVG(amount) FROM orders);
```

> **注意**：`HAVING` 后面不能直接放聚合函数嵌套（比如 `HAVING SUM(amount) > AVG(amount)` 是不行的），但可以用标量子查询来引用外部聚合结果。

## 四、列子查询：返回一列多行

**列子查询**（Column Subquery）*此术语见附录E* 返回的是**一列、多行**。它必须配合 `IN`、`NOT IN`、`ANY`、`ALL` 等多值运算符使用。

### 4.1 配合 IN：查"在某个集合中"

```sql
-- 场景：查"下过订单的用户"
-- 思路：先从 orders 表找出所有下过单的 user_id，再用它去过滤 users 表
SELECT id, username
FROM users
WHERE id IN (SELECT DISTINCT user_id FROM orders);
```

**执行过程**：
1. 子查询 `SELECT DISTINCT user_id FROM orders` 返回 `(1, 2, 3, 5, ...)`
2. 外层查询变成 `WHERE id IN (1, 2, 3, 5, ...)`
3. 返回这些 id 对应的用户

### 4.2 配合 NOT IN：查"不在某个集合中"

```sql
-- 场景：查"从未下过单的用户"
SELECT id, username
FROM users
WHERE id NOT IN (SELECT DISTINCT user_id FROM orders);
```

**这种写法看起来很合理，但有一个巨大的陷阱**——见下一节。

---

## 五、NOT IN 的 NULL 陷阱（重点！）

### 5.1 先看一个"翻车"现场

假设 `orders` 表中有这样一条数据：

```sql
-- orders 表里有一条订单，user_id 是 NULL
INSERT INTO orders (user_id, amount) VALUES (NULL, 100.00);
```

现在，你执行"从未下过单的用户"查询：

```sql
SELECT id, username
FROM users
WHERE id NOT IN (SELECT user_id FROM orders);
```

你期待的结果是：返回那些 `id` 不在 `user_id` 列表中的用户。

**但实际结果可能是：空！一行都没有！**

### 5.2 为什么会这样？

关键在 `NOT IN` 的工作原理。当你写：

```sql
WHERE id NOT IN (1, 2, NULL, 3)
```

MySQL 在内部会把它展开为：

```sql
WHERE id != 1 AND id != 2 AND id != NULL AND id != 3
```

问题来了：**`id != NULL` 的结果是什么？**

回顾第 07 章学过的 NULL 陷阱：任何值与 NULL 比较，结果都是 `NULL`（未知），不是 `TRUE` 也不是 `FALSE`。

所以：
- `id != 1` → TRUE（假设 id=5）
- `id != 2` → TRUE
- `id != NULL` → **NULL（未知！）**
- `id != 3` → TRUE

`TRUE AND TRUE AND NULL AND TRUE` = **NULL**

在 WHERE 条件中，只有结果为 `TRUE` 的行才被返回。`NULL` 不是 `TRUE`，所以**这行被排除了**。而且**每一行都这样**，所以最终结果为空！

### 5.3 用实验验证

```sql
-- 实验 1：列中有 NULL
SELECT 1 WHERE 5 NOT IN (1, 2, NULL, 3);
-- 结果：空（什么都没有）

-- 实验 2：列中没有 NULL
SELECT 1 WHERE 5 NOT IN (1, 2, 3);
-- 结果：1（有返回）

-- 实验 3：NULL 本身
SELECT NULL = NULL;
-- 结果：NULL（不是 1）
```

### 5.4 安全写法

**方案一**：子查询中排除 NULL

```sql
SELECT id, username
FROM users
WHERE id NOT IN (
    SELECT user_id FROM orders WHERE user_id IS NOT NULL
);
```

**方案二**：用 `NOT EXISTS`（第 10b 章讲）

```sql
SELECT id, username
FROM users u
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id
);
```

> **想多一点**：这个陷阱是 SQL 面试的高频考点。面试官问"NOT IN 有什么坑"，标准答案就是"子查询结果含 NULL 时，整个条件返回空"。本质原因是 NULL 的三值逻辑——SQL 不是布尔代数，是**三值逻辑**：TRUE、FALSE、UNKNOWN。任何与 NULL 的比较都落入 UNKNOWN 这个黑洞。

## 六、子查询 vs 分步查询：为什么不用两条 SQL？

你可能会想："我写成两条 SQL 不就行了？先查平均值，再用结果去查，清楚明了。"

```sql
-- 分步查询
-- 步骤 1：查平均金额
SELECT AVG(amount) FROM orders;  -- 返回 250.00

-- 步骤 2：用结果查
SELECT * FROM orders WHERE amount > 250.00;
```

分步查询确实更清晰。但子查询的优势在于：

| 维度 | 子查询 | 分步查询 |
|------|--------|---------|
| 网络往返 | 1 次 | 2 次（应用层需要拿到结果再发请求） |
| 数据一致性 | 子查询和外层查询在同一快照中执行 | 两次查询之间数据可能被修改 |
| 代码简洁 | 一条 SQL 搞定 | 需要应用层代码拼接 |
| 可读性 | 复杂时较难读 | 简单时更清晰 |

**我的建议**：简单场景用子查询（方便），复杂场景可以考虑分步（清晰）。但不管怎样，你**必须会读子查询**——当你在别人的代码里看到它时，不能一脸懵。

## 我做得对不对？

### 验证方法

请在你的 `my_shop` 数据库中执行以下验证：

```sql
-- 1. 确认环境：有 users 和 orders 表
USE my_shop;
SELECT COUNT(*) AS user_count FROM users;
SELECT COUNT(*) AS order_count FROM orders;

-- 2. 验证标量子查询：查"超过平均金额的订单"
SELECT AVG(amount) FROM orders;  -- 先记住这个值
SELECT * FROM orders
WHERE amount > (SELECT AVG(amount) FROM orders);
-- 手动验证：orders 表中 amount 大于上面平均值的行，应该全部出现在结果中

-- 3. 验证列子查询：查"下过订单的用户"
SELECT DISTINCT user_id FROM orders;  -- 记住这些 user_id
SELECT id, username FROM users
WHERE id IN (SELECT DISTINCT user_id FROM orders);
-- 手动验证：返回的用户 id 应该都在上一步的 user_id 列表中

-- 4. 验证 NOT IN NULL 陷阱
-- 先插入一条 user_id 为 NULL 的订单
INSERT INTO orders (user_id, amount) VALUES (NULL, 999.99);
-- 再执行 NOT IN 查询
SELECT id, username FROM users
WHERE id NOT IN (SELECT user_id FROM orders);
-- 预期：返回空（因为子查询结果中有 NULL）
-- 清理测试数据
DELETE FROM orders WHERE amount = 999.99 AND user_id IS NULL;
```

## 不对怎么办？

### 常见错误 1：子查询返回多行，却用了 `=` 比较

```sql
-- ❌ 错误：子查询返回多行，= 只能比较单个值
SELECT * FROM orders
WHERE amount = (SELECT amount FROM orders WHERE user_id = 1);
-- 如果 user_id=1 有多条订单，子查询返回多行 → 报错：
-- ERROR 1242 (21000): Subquery returns more than 1 row

-- ✅ 正确：用 IN 代替 =
SELECT * FROM orders
WHERE amount IN (SELECT amount FROM orders WHERE user_id = 1);
```

**记住**：`=` 配标量子查询，`IN` 配列子查询。

### 常见错误 2：NOT IN 遇上 NULL（见第五节）

```sql
-- ❌ 危险：子查询可能含 NULL
SELECT * FROM users
WHERE id NOT IN (SELECT user_id FROM orders);

-- ✅ 安全：子查询中排除 NULL
SELECT * FROM users
WHERE id NOT IN (SELECT user_id FROM orders WHERE user_id IS NOT NULL);
```

### 常见错误 3：子查询没有用括号

```sql
-- ❌ 错误：忘记括号
SELECT * FROM orders
WHERE amount > SELECT AVG(amount) FROM orders;
-- 语法错误！

-- ✅ 正确：子查询必须用括号括起来
SELECT * FROM orders
WHERE amount > (SELECT AVG(amount) FROM orders);
```

### 常见错误 4：标量子查询返回了空结果

```sql
-- 假设 orders 表是空的
SELECT * FROM orders
WHERE amount > (SELECT AVG(amount) FROM orders);
-- 子查询返回 NULL（空表的 AVG 是 NULL）
-- 外层变成 WHERE amount > NULL → 结果永远是空

-- ✅ 防御：用 COALESCE 给默认值
SELECT * FROM orders
WHERE amount > (SELECT COALESCE(AVG(amount), 0) FROM orders);
```

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 子查询是什么 | SELECT 中嵌套 SELECT，内层先执行 | 子查询必须用括号括起来 |
| 标量子查询 | 返回单个值，用于 `=` / `>` / `<` 等比较 | 如果返回多行会报错：Subquery returns more than 1 row |
