## 先讲一个真实的故事（你务必要读完）

2019年，一位程序员正在做一个Side Project。他在代码里用到了AWS（📖 术语：Amazon Web Services，亚马逊的云服务，全球最大的云计算平台）的S3存储服务。为了方便，他把AWS的API密钥直接写在了代码里：

```python
# ❌ 他把这行代码提交到了GitHub上
AWS_ACCESS_KEY = "AKIAIOSFODNN7EXAMPLE"
AWS_SECRET_KEY = "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"
```

然后他做了程序员最日常的一个动作：**`git push`**。

他把代码推送到了一个**公开的GitHub仓库**。这没什么，开源嘛，大家都这样。

**但问题在于**：他忘了把密钥删掉。那两行密钥就静静地躺在GitHub上，任何人都能看到。

---

当天凌晨2点，地球另一端的某个地方，一个自动化脚本正在扫描GitHub的每一次新提交。这个脚本不关心代码质量，它只关心一件事：**有没有人把AWS密钥提交到公开仓库。**

它找到了这位程序员的密钥。用时：**不到30秒。**

接下来的事情发生得很快：

1. 脚本用这个密钥登录了AWS
2. 脚本在AWS上启动了**几百台最高配置的服务器**（EC2实例）
3. 这些服务器没有做任何正经事——它们开始**挖矿**（利用服务器算力产生加密货币，相当于用别人的电费给自己赚钱）
4. 挖矿跑了整整一夜

**第二天早上，这位程序员醒来，打开手机，看到AWS发来的邮件：**

> "尊敬的客户，您的账户已产生 **$6,000** 费用。如需详细了解，请登录控制台查看账单。"

六千美元。一觉醒来，一个月的工资没了。

---

### 这不是个例

这个故事里提到的"自动化脚本扫描GitHub找密钥"是真实存在的，而且每天都在运行。它不是某个黑客的业余爱好——这是一个**产业**。

在GitHub上：
- 每天有数千个密钥被意外提交
- 有专门的爬虫24小时不间断扫描
- 从密钥泄露到被利用，平均时间**不到5分钟**
- 到2026年，这种情况越来越多，因为AI让更多人开始写代码——但没教他们安全

**你可能会想："我只是个新手，谁会盯上我？"**

答案是：**不是"人"盯上你，是"脚本"盯上你。** 脚本不关心你是谁，它只关心你有没有密钥。你写的代码量再小，只要密钥是真的，你就中招了。

## 你真的理解AI工具是怎么工作的吗？

在使用Cursor或Trae时，你可能会觉得"AI就在我的电脑里运行"——这是最大的误解。

**真相：你给AI看的每一行代码，都会通过网络上传到AI厂商的云端服务器处理。**

```
你的操作流程：
  你选中代码 → 发送给AI → AI分析 → AI返回结果

实际上发生了什么：
  你的代码 → 通过互联网 → 上传到Cursor/Anthropic的服务器
  → AI模型在云端处理 → 结果通过互联网 → 返回你的电脑
```

**AI没有在你的电脑里运行。** 它运行在几千公里外的数据中心里。你发给它的每一条消息、每一行代码、每一个文件——都经过了互联网，到达了AI厂商的服务器。

📖 **隐私（Privacy）** = 你的信息不被你不希望的人看到或使用的权利。在AI编程中，隐私意味着：你发给AI的代码，谁能看到？

## 🚨 哪些信息绝对不能发给AI？

### 第一类：密钥和令牌（泄露 = 直接经济损失）

📖 **密钥（API Key）** = 你访问某个在线服务的"门禁卡"。比如你要用OpenAI的API，就需要一个密钥。持有这个密钥的人，就可以以你的身份调用服务，产生的费用记在你账上。

```
❌ 绝对不能发给AI的内容：

# API密钥
OPENAI_API_KEY = "sk-proj-abc123def456..."
GITHUB_TOKEN = "ghp_xxxxxxxxxxxxxxxxxxxx"
AWS_ACCESS_KEY_ID = "AKIAIOSFODNN7EXAMPLE"
AWS_SECRET_ACCESS_KEY = "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"

# 数据库密码
DATABASE_URL = "postgresql://admin:MyPassword123@localhost:5432/mydb"
MYSQL_PASSWORD = "MySecretPassword"

# 服务器密码
SSH_PASSWORD = "root_password_123"
SERVER_IP = "10.0.0.1"
```

### 第二类：公司内部代码（泄露 = 法律风险）

如果你在上班，公司代码是公司的资产。把公司代码发给AI，等于把公司资产上传到了第三方服务器。很多公司有明确的合规政策禁止这种行为。

**⚠️ 如果你不确定公司是否允许使用AI编程工具，先问你的主管。**

### 第三类：用户隐私数据（泄露 = 法律+道德双重问题）

```
❌ 绝对不能发给AI的：
- 用户的真实姓名、手机号、邮箱
- 用户的身份证号、地址
- 任何可识别到具体个人的数据

✅ 可以发给AI的（脱敏后）：
- 数据结构定义（字段名、类型）
- 将真实数据替换为假数据：user_id=12345 → user_id=sample_001
```

## 安全实践：怎么在用AI的同时保护自己？

### 实践一：用占位符（最推荐、最简单）

**永远不要在你的代码里写真实的密钥。** 用占位符代替：

```python
# ❌ 错误：真实密钥写死在代码里
OPENAI_API_KEY = "sk-proj-abc123def456ghi789..."
# 这段代码一旦发给AI，你的密钥就泄露了

# ✅ 正确：用占位符
OPENAI_API_KEY = "your_openai_api_key_here"  # 替换为你的真实密钥
# 或者用环境变量
import os
OPENAI_API_KEY = os.environ.get("OPENAI_API_KEY", "your_api_key_here")
```

**在所有发给AI的代码中，养成替换密钥的习惯：**

```
真实密钥 → 占位符
sk-proj-abc123def456... → your_openai_api_key_here
ghp_xxxxxxxxxxxxx → your_github_token_here
postgresql://admin:pass123@host/db → your_database_url_here
```

### 实践二：使用环境变量（专业做法）

```python
# ✅ 最安全的做法：密钥存在 .env 文件里，不写进代码
import os
from dotenv import load_dotenv

load_dotenv()  # 从 .env 文件加载环境变量
OPENAI_API_KEY = os.environ.get("OPENAI_API_KEY")

# .env 文件内容（这个文件永远不要发给AI，也不要提交到GitHub）：
# OPENAI_API_KEY=sk-proj-abc123def456...
```

```bash
# 在 .gitignore 里加上 .env，确保它不会被提交到GitHub
echo ".env" >> .gitignore
```

### 实践三：使用本地部署模型（最高安全级别）

如果你处理的是极度敏感的数据（比如医疗、金融），可以考虑使用**本地部署的AI模型**。这些模型在你的电脑上运行，数据不会离开你的电脑。

- Ollama：免费开源的本地模型运行工具（[ollama.com](https://ollama.com)）
- LM Studio：图形化界面的本地模型管理器

**但注意**：本地模型的代码能力通常不如云端模型（Cloud、GPT-4、DeepSeek等），因为它们能用的算力有限。这是一个"安全 vs 能力"的权衡。

### 实践四：企业版私有部署

如果你在团队中，可以考虑使用企业版AI编程工具，它们通常提供私有部署选项：
- Cursor Business：企业版，数据不用于训练
- GitHub Copilot Business：企业版，有数据保护协议
- 自建方案：很多公司搭建自己的AI编程助手，代码不出公司内网

## Cursor / Trae 的数据使用政策解读

你可能关心："我用Cursor和Trae，我的代码到底安不安全？"

### Cursor 的隐私政策（关键点）

Cursor 官方承诺：
- **默认情况下，你的代码不会用于训练模型**（如果你开启了"Privacy Mode"）
- 代码会传输到Cursor的服务器进行处理，但处理完成后不会长期存储
- 企业版提供额外的数据保护

**但注意**：即使Cursor不用于训练，你的代码**仍然经过了他们的服务器**。如果你在代码里写了密钥，理论上Cursor的服务器能看到。

### Trae 的隐私政策（关键点）

Trae（字节跳动出品）：
- 代码处理在云端进行
- 有数据安全合规承诺
- 详情见 Trae 官网的隐私政策页

### 一句话总结

**无论用什么工具，请默认一个原则：发给AI的代码，都当作"可能被第三方看到"来对待。** 这不是说AI厂商会偷你的代码，而是"安全习惯"——你永远不知道哪一天策略会变、哪一次传输会出问题。
