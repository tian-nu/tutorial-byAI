# 第95章 · 分布式基础理论

## 95.1 CAP定理

### 什么是CAP

CAP定理（分布式系统中一致性、可用性、分区容错三选二的法则，详见附录I）由UC Berkeley的Eric Brewer在2000年提出，是分布式系统的**第一定律**。它说：**在一个分布式系统中，一致性（Consistency）、可用性（Availability）、分区容错性（Partition Tolerance）三者不可兼得——你最多只能同时满足两个。**

### 用现实比喻理解CAP

你开了一家全国连锁的银行，在北京和上海各有一个分行（两个网络分区的节点）。每个分行都能开户和存取款。

**一致性（C，所有节点数据时刻保持一致，详见附录I）**：无论你在北京还是上海查余额，看到的数字永远一样。上海的账户变动立刻同步到北京。

**可用性（A，系统始终能正常响应请求，详见附录I）**：只要走进任何一家分行，它一定给你服务——即使和另一家分行联系不上了。

**分区容错性（P，网络分区后系统仍能继续运行，详见附录I）**：北京和上海之间的电话线断了（网络分区），两边的业务照常运行。

现在问题来了：当电话线断了时——

```
北京分行                                上海分行
账户A: 1000元                          账户A: 1000元
    │                                      │
    │ 客户取500元                           │ 客户取800元
    │                                      │
    ▼                                      ▼
账户A: 500元                            账户A: 200元
    │                                      │
    │ ❌ 电话线断了，无法同步                │
    │                                      │
    └───────── ????? ──────────────────────┘
```

当电话线恢复后——账户A到底还有多少钱？500还是200？两个分行记录了不同的事实——这就是**不一致**。

如果你想保证**一致性**（电话线恢复后两边数字一样）：在电话线断的时候，必须拒绝其中一方的操作。"先生对不起，我们现在联系不上北京总部，暂时不能给您办理业务"——牺牲了**可用性**。

如果你想保证**可用性**（任何时候都能办业务）：电话线恢复后两边发现数字不一样了——牺牲了**一致性**。

### 为什么P必须选

因为网络分区不是"可能发生"——是**一定会发生**。交换机故障、网线被挖断、机房断电、DDoS攻击……任何分布式系统都无法避免网络分区。

所以问题不是"选不选P"，而是"选了P之后，在C和A之间怎么取舍"。

### CP系统 vs AP系统

**CP系统（牺牲可用性）**：

当发生网络分区时，选择"宁可不可用，也不能不一致"。

- 银行转账系统：转1000元，你宁可等10分钟，也不能"你以为转了其实没转"
- Etcd、ZooKeeper：选主时必须达成一致，不一致的话整个集群拒绝服务

**AP系统（牺牲一致性）**：

当发生网络分区时，选择"宁可暂时不一致，也必须可用"。

- 电商商品浏览：你看商品详情时，库存显示99件，实际可能已经卖完了——但总比页面打不开强
- DNS解析：域名解析记录的生效有延迟（几分钟到几小时），但你任何时候查询都能得到结果
- Cassandra、Eureka：网络可以不完美，但服务必须响应

### CAP不是"三选二"，而是"二选一"

很多人误解CAP为"三个里面选两个"。更准确的说法是：

1. **P必须选**——网络分区不可避免
2. **剩下的C和A，你只能二选一**——但不是非黑即白，而是在光谱上选一个点

实际上很少有系统是"100% CP"或"100% AP"——大多数是"网络正常时是CA（既一致又可用），网络分区时在CA之间做一个有倾向性的选择"。

## 95.2 BASE理论

### 为什么需要BASE

CAP定理让建设分布式系统的人很沮丧——要么牺牲一致性，要么牺牲可用性，没有完美方案。BASE理论提供了一个务实的折中方案：**不完全放弃一致性，而是接受"最终一致性"。**

### BASE的含义

**BA — Basically Available（基本可用）**

系统可以损失部分功能来保证核心功能的可用。

- 双十一时：可以查看订单但不能修改订单（"修改"是非核心功能）
- 高峰时：搜索结果的准确度降低一点（但不返回错误）

**S — Soft State（软状态）**

系统可以存在中间状态，不要求所有节点的数据副本在任意时刻都一致。

- 你在朋友圈发了一条状态→你自己立刻看到→你朋友可能过3秒才刷出来
- 这3秒内，你的朋友圈列表和朋友的朋友圈列表不一致——这就是"软状态"

**E — Eventually Consistent（最终一致性）**

不强求实时一致性，但保证**在没有新的更新之后，所有副本最终会达到一致状态**。

- DNS: 修改了一个域名解析记录→24小时内全球各地陆续生效
- 银行的"到账短信"：转账成功→可能5分钟后才收到短信通知

### BASE vs ACID的对比

前面学过数据库的ACID（原子性、一致性、隔离性、持久性）。那ACID和BASE是什么关系？

| 维度 | ACID | BASE |
|------|------|------|
| 适用场景 | 单机数据库 | 分布式系统 |
| 一致性 | 强一致性（写后立刻一致） | 最终一致性（过一段时间一致） |
| 核心追求 | 数据绝对正确 | 系统基本可用 |
| 典型场景 | 银行转账（钱不能少） | 社交网络的点赞数（差几个无所谓） |
| 实现复杂度 | 低（数据库帮你做） | 高（自己要设计补偿机制） |

**关键洞察**：ACID和BASE不是对立的——一个系统可以某些模块用ACID保证"核心交易"的正确性，另一些模块用BASE保证"非核心功能"的可用性。

## 95.3 一致性协议

### 两阶段提交（2PC）

2PC是分布式事务的"祖宗算法"。它把一次跨节点的提交操作分成两个阶段：准备阶段 + 提交阶段。

**参与角色**：
- **协调者（Coordinator）**：负责统一指挥
- **参与者（Participants）**：各个涉及到的服务/数据库

**流程**：

```
阶段一：准备（Prepare）
协调者 → 参与者1: "你准备好提交了吗？"
协调者 → 参与者2: "你准备好提交了吗？"
协调者 → 参与者3: "你准备好提交了吗？"

参与者1: "准备好了！" (YES)
参与者2: "准备好了！" (YES)
参与者3: "准备好了！" (YES)

阶段二：提交（Commit）
协调者 → 所有参与者: "全部提交！"

参与者1: "提交成功"
参与者2: "提交成功"
参与者3: "提交成功"
```

如果任何一个参与者在阶段一说"不行"（NO），协调者会通知所有人"回滚"（Rollback）。

**2PC的问题**：
1. **同步阻塞**：阶段一中，所有参与者锁定了资源，直到阶段二完成才能释放。这段时间别人没法访问这些数据
2. **单点故障**：协调者挂了→所有参与者不知道该怎么办→一直等着→资源永远锁死
3. **网络中断导致数据不一致**：阶段二的提交指令发出后，网络断了→只有部分参与者收到并提交了→数据不一致

### 三阶段提交（3PC）

3PC在2PC的基础上增加了第三个阶段，核心改进是**给参与者加了超时机制**（不会永远等）：

1. **CanCommit**：协调者问"你们能做吗？"
2. **PreCommit**：全部能→协调者说"准备好，但别提交"
3. **DoCommit**：协调者说"提交！"

增加的PreCommit阶段给了参与者一个"缓冲期"——如果协调者挂了，参与者在超时后不需要永远等待，可以根据PreCommit阶段的状态自行决定提交还是回滚。

**3PC仍然不完美**：如果网络刚好在DoCommit阶段分割，仍可能出现数据不一致。

## 95.4 Raft算法

### 为什么需要共识算法

分布式系统中，多个节点需要对"谁是Leader"、"这条数据该是什么"达成一致。如果只是简单的投票（"同意的人多就通过"），网络分区时可能选出两个Leader——两个领导者同时发号施令，系统就乱了。

Raft算法让一致性变得**可理解**——它的设计目标之一就是"比Paxos好懂"。

### Raft的核心：三个角色 + 两个RPC

**角色**：

| 角色 | 职责 |
|------|------|
| Leader | 处理所有客户端请求，同步日志到Follower |
| Follower | 被动接收Leader的日志，不主动发起请求 |
| Candidate | 选举过程中的临时角色，竞选Leader |

角色的转换：

```
     选举超时             收到Leader心跳
Follower ──────→ Candidate ──────→ Leader
   ↑                                    │
   └──────────── 发现更高级的Term ───────┘
```

**两个RPC**：

1. **RequestVote RPC**：Candidate请求其他节点"投我一票"
2. **AppendEntries RPC**：Leader向Follower同步日志（也充当心跳）

### 选举过程（Leader Election）

想象一个五节点集群（Node A~E）。所有节点启动时都是Follower。

```
时间线：
T0: 所有节点是Follower，每个人自己计时（随机150~300ms）
T1: Node C的计时器先到 → 变成Candidate → Term+1 → 给自己投一票
T2: Node C向所有人发RequestVote:"Term 1，日志最新索引是5，选我做Leader！"
T3: Node A/B/D收到：
      - 检查Term（如果比自己的新或同Term）→ OK
      - 检查日志（如果不比自己旧）→ OK
      - 没给别人投过票 → 投给C
T4: C收到3票（自己+3票 = 共4票，超过半数）→ 变成Leader
T5: C开始向所有人发心跳（AppendEntries，empty）→ "我还活着，我是Leader"
```

**如果出现"平票"**（比如4个节点，2:2）→ 没有节点拿到多数票 → 所有Candidate回到Follower → 重新计时 → 新一轮选举。

### 日志复制（Log Replication）

Leader选出后，所有客户端请求都通过Leader处理：

```
客户端 → Leader: "SET x=10"
Leader:
  1. 把这条命令写入自己的日志
  2. 通过AppendEntries把这条日志发给所有Follower
  3. 等待超过半数的Follower确认"收到了"
  4. 提交（commit）这条日志到自己状态机
  5. 告诉Follower"可以提交了"
  6. 返回客户端"写入成功"
```

**只要超过半数的节点有这条日志，这条日志就是"已提交"的**——即使Leader挂了，新Leader一定包含所有已提交的日志（因为新Leader必须拿到多数票，而多数票中至少有一个节点有所有已提交日志）。

这是Raft最精妙的设计：**"多数派"同时保证了选举安全性和日志一致性。**

### 安全性保证

Raft保证5条安全属性：

1. **选举安全**：一个Term最多一个Leader
2. **Leader只追加**：Leader从不覆盖或删除已有日志
3. **日志匹配**：如果两个日志的相同索引处有相同的Term，则之前的所有日志完全相同
4. **Leader完整性**：已提交的日志在新Leader上一定存在
5. **状态机安全**：所有节点按相同顺序应用相同的日志

## 95.5 分布式事务

回到实际开发。分布式事务是CAP定理的"现实投射"——你需要在多个数据库/服务之间保证操作的一致性。

### Seata

Seata（Simple Extensible Autonomous Transaction Architecture）是阿里开源的分布式事务解决方案，支持三种模式：

**AT模式（自动补偿）**——最常用：

```
没有分布式事务：
  下单 → 扣库存 → 库存扣成功了但下单失败了？→ 库存白白扣了

Seata AT模式：
  1. 开启全局事务
  2. 下单服务执行SQL → Seata记录"执行前快照"（before image）
  3. 库存服务执行SQL → Seata记录"执行前快照"
  4. 如果全部成功 → 全局提交
  5. 如果任何一个失败 → 全局回滚 → 用快照自动还原数据
```

**TCC模式（Try-Confirm-Cancel）**——性能最好，但代码侵入性强：

```
Try阶段（预留资源）：
  订单服务: "冻结"这个订单号
  库存服务: "冻结"N件库存
  账户服务: "冻结"N元

Confirm阶段（确认）：
  全部Try成功 → 全部Confirm → 冻结变为真正的扣减

Cancel阶段（取消）：
  任何一个Try失败 → 全部Cancel → 释放冻结的资源
```

**Saga模式**——长事务的终极方案：

```
Saga = 一系列本地事务 + 每个事务有对应的补偿操作

步骤1: 创建订单
  补偿: 取消订单
步骤2: 扣库存
  补偿: 恢复库存
步骤3: 扣款
  补偿: 退款
步骤4: 发通知
  补偿: (不需要补偿——发错了就发错了)

如果步骤3失败 → 从步骤3开始反向执行补偿：
  退款 → 恢复库存 → 取消订单
```

Saga的优点是**不阻塞资源**——每个步骤提交后资源就释放了。缺点是**不隔离**——步骤1到步骤3之间，数据可能被其他人修改了。

### 分布式事务的选择

| 场景 | 推荐方案 |
|------|---------|
| 电商下单 | AT模式（简单、对代码无侵入） |
| 金融转账 | TCC模式（强一致性） |
| 长流程（旅游预订） | Saga（不阻塞资源） |
| 简单场景 | 不要分布式事务！用补偿+幂等 |

🤔 **想多一点**：分布式事务的"终极真理"

走过CAP、BASE、2PC、Raft、Seata之后，你可能觉得分布式事务非常复杂。但有一个更深刻的真理：**最好的分布式事务是没有分布式事务。**

在可能的情况下，通过**业务设计**避免跨服务事务：
- 把紧密关联的数据放在同一个服务中
- 使用最终一致性（如订单→MQ→库存，允许短暂不一致）
- 设计幂等的补偿机制（"你收到退款通知时，即使多发了一次也没关系"）

分布式事务是最后的手段，不是首选方案。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| CAP定理 | C一致性/A可用性/P分区容错，P必须选，CA二选一 |
| CP系统 | 网络分区时牺牲可用性保一致性（银行转账、Etcd） |
| AP系统 | 网络分区时牺牲一致性保可用性（DNS、商品浏览） |
| BASE理论 | 基本可用 + 软状态 + 最终一致性 |
| ACID vs BASE | 单机强一致 vs 分布式最终一致，不是对立的 |
| 2PC | 准备→提交，同步阻塞+单点故障 |
| 3PC | 增加超时机制和缓冲阶段，仍不完美 |
| Raft | Leader/Follower/Candidate，选主+日志复制，多数派原则 |
| 日志复制 | 超过半数确认即提交，保证一致性和新Leader完整性 |
| Seata AT | before image快照，自动回滚 |
| Seata TCC | Try→Confirm→Cancel，高性能+高侵入 |
| Seata Saga | 正向事务+反向补偿，长流程 |
