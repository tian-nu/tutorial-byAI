## 6.1 Java异常体系

### 6.1.1 异常是什么？

想象你在ATM机上取钱：银行卡插反了、余额不足、机器故障——这些"意外情况"就是异常。程序世界也一样：网络断了、文件找不到了、数组越界了……异常处理让程序不是粗暴地崩溃，而是优雅地应对问题。

Java把所有"不正常情况"都抽象成了对象，根类是`Throwable`。

### 6.1.2 Throwable继承树

以下ASCII图展示了完整的异常体系：

```
Throwable（一切异常和错误的根）
│
├── Error（严重问题，程序不处理）
│   ├── OutOfMemoryError        —— 堆内存耗尽，JVM要挂了
│   ├── StackOverflowError      —— 方法递归太深，栈溢出
│   ├── NoClassDefFoundError    —— 类在编译时存在但运行时找不到
│   └── VirtualMachineError     —— JVM内部错误
│
└── Exception（程序可以处理的异常）
    │
    ├── RuntimeException（非受检异常 — unchecked）
    │   ├── NullPointerException           —— 对null调用方法/属性
    │   ├── IndexOutOfBoundsException      —— 数组/集合索引越界
    │   │   ├── ArrayIndexOutOfBoundsException
    │   │   └── StringIndexOutOfBoundsException
    │   ├── ArithmeticException            —— 除以零等算术异常
    │   ├── IllegalArgumentException       —— 方法收到非法参数
    │   │   └── NumberFormatException      —— 字符串转数字格式错误
    │   ├── ClassCastException             —— 类型强制转换失败
    │   ├── ConcurrentModificationException—— 并发修改集合
    │   └── UnsupportedOperationException  —— 不支持的操作
    │
    └── 受检异常（checked — 编译器强制处理）
        ├── IOException                    —— IO操作异常
        │   ├── FileNotFoundException      —— 文件未找到
        │   ├── EOFException               —— 意外到达文件末尾
        │   └── SocketException            —— 网络连接异常
        ├── SQLException                   —— 数据库操作异常
        ├── ClassNotFoundException         —— 类未找到（反射）
        ├── ParseException                 —— 解析异常
        └── InterruptedException           —— 线程被中断
```

**设计哲学：**

- **Error**：属于JVM层面的严重问题，比如内存耗尽、栈溢出。应用程序**不应该也不必要**去try-catch这些异常——因为catch了也无能为力，不如让程序快速失败（fail-fast）。
- **受检异常（Checked Exception）**：编译器强制要求你必须处理（try-catch或throws声明）。这类异常通常是由外部因素引起的、可预见的——文件不存在、网络断开、数据库挂了——程序员**有能力**在代码中做出应对。
- **非受检异常（Unchecked Exception / RuntimeException）**：编译器不强制处理。这类异常通常是**程序员的bug**——空指针、数组越界、类型转换错误——应该通过完善代码来避免，而不是到处try-catch。

> 🚨 **坑点：过度使用受检异常**
>
> 受检异常的理念是好的（强制开发者思考异常情况），但在实践中容易被滥用。如果一个底层方法声明了`throws Exception`，那么所有调用链上的方法要么继续声明throws，要么try-catch——导致代码中充斥着大量无意义的异常声明。如果你在设计API，考虑清楚：**这个异常是调用者可以合理恢复的（用受检），还是调用者基本无法恢复的（用非受检）**。

## 6.2 异常处理机制

### 6.2.1 try-catch-finally完整语法

Java异常处理有五条语句：`try`、`catch`、`finally`、`throw`、`throws`。

```java
try {
    // 可能抛出异常的代码
    int result = 10 / 0;                // ArithmeticException
    String s = null;
    System.out.println(s.length());     // NullPointerException（这一行不会执行，因为上一行已经抛异常了）
} catch (ArithmeticException e) {
    // 捕获算术异常
    System.out.println("数学错误：" + e.getMessage());
} catch (NullPointerException e) {
    // 捕获空指针异常
    System.out.println("空指针错误：" + e.getMessage());
} catch (Exception e) {
    // 兜底：捕获所有其他异常
    System.out.println("未知错误：" + e.getMessage());
} finally {
    // 无论是否发生异常都会执行
    System.out.println("finally块：资源清理工作在这里做");
}

// 程序继续执行（因为异常被catch捕获了）
System.out.println("程序正常结束");
```

**多个catch的先后顺序非常重要**：

```java
// 错误写法
try {
    // ...
} catch (Exception e) {           // 父类在前！
    // 这个catch会吞掉所有异常
} catch (ArithmeticException e) {  // 编译错误：这行永远不会执行
}
```

Java要求catch从具体到宽泛排列——子类异常在前，父类在后。

> 🚨 **坑点：catch范围太大吞掉不该吞的异常**
>
> ```java
> try {
>     // 这里有多行代码，每个都可能有不同类型的异常
>     readFile();
>     parseData();
>     saveToDatabase();
> } catch (Exception e) {
>     // 不管什么异常，统统丢掉！甚至连Error都（可能）被意外吞了
>     e.printStackTrace(); // 只打印，程序继续——生产环境的数据就这样出错了没人知道
> }
> ```
>
> 过于宽泛的catch使得下游代码"假装一切正常"，实际上数据可能已经坏了。

> 🚨 **坑点：空catch块——生产环境的"黑盒问题"**
>
> ```java
> try {
>     criticalOperation();
> } catch (Exception e) {
>     // 什么都不做！异常被彻底隐藏
> }
> // 后面代码"正常"执行，但criticalOperation的失败没人知道
> // 发现问题可能需要几小时的排查
> ```
>
> **原则**：如果无法处理，至少记录日志并重新抛出或转换为业务异常。

### 6.2.2 finally的执行规则与陷阱

finally的设计意图是**无论是否发生异常，都要执行的清理代码**——关闭文件、释放数据库连接等。

```java
// finally一定会执行的场景
public static int test1() {
    try {
        System.out.println("try块");
        return 1;
    } finally {
        System.out.println("finally块");  // 即使try中有return，finally也会执行
    }
}
// 输出：
// try块
// finally块
// 返回 1
```

**finally不执行的唯一例外**：
- `System.exit(0)`被调用（JVM进程直接终止）
- JVM崩溃（如OOM导致进程被杀）
- 守护线程中所有非守护线程结束时
- 无限循环等死锁情况

> 🚨 **大坑：finally中有return，会覆盖try中的返回值和异常！**
>
> ```java
> // 这个方法的返回值永远是3！除零异常也被吞了！
> public static int dangerousMethod() {
>     try {
>         return 1 / 0;  // 抛出 ArithmeticException
>     } finally {
>         return 3;      // finally中的return覆盖了一切！
>     }
> }
>
> System.out.println(dangerousMethod()); // 3（不是异常！异常被吞了！）
> ```
>
> 这个陷阱非常隐蔽：程序员以为异常会被抛出去，但实际上被finally的return"吃"掉了。**永远不要在finally块中使用return、throw、continue或break**。

```java
// 正确做法：finally只做清理，不改变控制流
public static int safeMethod() throws IOException {
    FileInputStream fis = null;
    try {
        fis = new FileInputStream("data.txt");
        return fis.read();  // 可能抛 IOException
    } finally {
        if (fis != null) {
            fis.close();    // 关闭资源，不改变返回值或异常
        }
    }
}
```

### 6.2.3 try-with-resources — 优雅的资源管理（Java 7+）

上面的代码即使努力写了正确的finally，关闭资源的代码也很啰嗦。try-with-resources就是为简化这个场景而生的：

```java
// 传统写法（啰嗦，容易出错）
FileInputStream fis = null;
try {
    fis = new FileInputStream("data.txt");
    // 使用 fis...
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (fis != null) {
        try {
            fis.close();  // close()本身也可能抛异常！
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

// try-with-resources 写法（简洁优雅）
try (FileInputStream fis = new FileInputStream("data.txt")) {
    // 使用 fis...
} catch (IOException e) {
    e.printStackTrace();
}
// fis 会被自动关闭，不需要手动写finally
```

**工作原理**：任何实现了`java.lang.AutoCloseable`接口的类都可以放在try的括号中。当try块结束时（无论正常还是异常），资源的`close()`方法会被自动调用。

**多个资源**：

```java
try (FileInputStream fis = new FileInputStream("input.txt");
     FileOutputStream fos = new FileOutputStream("output.txt")) {
    // 使用 fis 和 fos...
}
// 关闭顺序是声明的逆序：先关 fos，再关 fis
```

> 🚨 **坑点：资源的关闭顺序是声明的逆序**
>
> 如果你在try-with-resources中声明了多个资源，JDK保证它们的关闭顺序是**后声明先关闭**（类似栈的LIFO）。这个细节很重要——如果你依赖特定的关闭顺序（比如先关输出流再关输入流），需要确保声明顺序正确。

**异常抑制机制**：如果try块抛出一个异常，而close()也抛出一个异常，第一个异常会被优先保留，close()的异常作为"被抑制的异常"附加在第一个异常上，可以通过`ex.getSuppressed()`获取。

### 6.2.4 throws与throw

这两个词长得像，但含义完全不同：

```java
// throws：方法声明中"声明"可能抛出的异常
public void readFile(String path) throws IOException {
    FileInputStream fis = new FileInputStream(path); // FileNotFoundException是IOException的子类
    // 如果这里不try-catch，就必须在方法签名上throws
}

// throw：方法体内"手动抛出"一个异常
public void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("年龄必须在0-150之间，收到：" + age);
    }
    this.age = age;
}
```

**调用带有throws声明的方法时**，你有两种选择：
1. 用try-catch捕获处理
2. 继续throws向上传播（甩锅给调用者）

```java
// 选择1：自己处理
public void processFile() {
    try {
        readFile("data.txt");
    } catch (IOException e) {
        System.out.println("文件处理失败：" + e.getMessage());
    }
}

// 选择2：甩锅
public void processFile() throws IOException {
    readFile("data.txt");
}
```

### 6.2.5 自定义异常

Java内置的异常不够语义化时，自定义异常让错误信息更有表达力：

```java
// 业务异常基类（继承RuntimeException——非受检）
public class BusinessException extends RuntimeException {
    private final int errorCode;

    public BusinessException(int errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    public int getErrorCode() {
        return errorCode;
    }
}

// 具体业务异常
public class EmployeeNotFoundException extends BusinessException {
    public EmployeeNotFoundException(Long id) {
        super(40401, "员工不存在，ID：" + id);
    }
}

public class InvalidSalaryException extends BusinessException {
    public InvalidSalaryException(double salary) {
        super(40002, "薪资数值不合法：" + salary);
    }
}

// 使用
public Employee findEmployee(Long id) {
    return employeeRepo.findById(id)
            .orElseThrow(() -> new EmployeeNotFoundException(id));
}
```

**设计建议**：
- 继承`RuntimeException`而非`Exception`（避免强制调用者四处try-catch）
- 添加错误码字段（`errorCode`），便于前端根据错误码做不同处理
- **预告**：第16章Spring Boot全局异常处理会用`@RestControllerAdvice`统一拦截这些自定义异常，返回规范的JSON错误响应

## 6.3 IO流

### 6.3.1 流的概念

**流（Stream）** 是Java中数据传输的抽象管道。站在程序的角度：
- **输入流**：从外部（文件、网络、键盘）读取数据 → 程序
- **输出流**：从程序写出数据 → 外部（文件、网络、屏幕）

```
输入流：
┌──────┐  读取  ┌──────────┐
│ 文件  │ ────→ │  程序     │
│ 网络  │ ←─流─→ │(内存变量) │
│ 键盘  │       └──────────┘

输出流：
┌──────────┐  写出  ┌──────┐
│  程序     │ ────→ │ 文件  │
│(内存变量) │ ←─流─→ │ 网络  │
└──────────┘       │ 屏幕  │
                   └──────┘
```

Java IO分为两大阵营：

| 维度 | 字节流 | 字符流 |
|------|--------|--------|
| 抽象基类 | InputStream / OutputStream | Reader / Writer |
| 处理单位 | 1字节（byte） | 2字节（char） |
| 用途 | 二进制文件（图片、视频、音频） | 文本文件（txt、csv、json） |
| 缓冲流 | BufferedInputStream / BufferedOutputStream | BufferedReader / BufferedWriter |

### 6.3.2 文件字节流

最基本的文件读写：

```java
import java.io.*;

public class ByteStreamDemo {
    public static void main(String[] args) {
        // ===== 写入文件 =====
        try (FileOutputStream fos = new FileOutputStream("hello.txt")) {
            fos.write("Hello, Java!".getBytes());  // 字符串转字节数组写入
            fos.write(65);  // 写入单个字节：'A' 的 ASCII 码是 65
        } catch (IOException e) {
            e.printStackTrace();
        }

        // ===== 读取文件 =====
        try (FileInputStream fis = new FileInputStream("hello.txt")) {
            int data;
            while ((data = fis.read()) != -1) {  // read() 返回 -1 表示文件结束
                System.out.print((char) data);   // 逐字节读取并转字符
            }
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

> 🚨 **坑点：输出流默认覆盖原文件**
>
> ```java
> new FileOutputStream("data.txt");      // 覆盖已有内容从头写
> new FileOutputStream("data.txt", true); // true = append模式，追加到文件末尾
> ```
>
> 如果你只是想追加几行日志，却又忘记了加`true`，文件内容就会被清空。这在生产环境是灾难性的。

### 6.3.3 缓冲流 — 提升IO性能

逐字节读写效率极低（每次read/write都要访问磁盘）。缓冲流内部维护了一个缓存区（默认8192字节=8KB），批量读写：

```java
// 无缓冲：每次读写一个字节都要一次磁盘IO —— 极慢
// 有缓冲：一次读8KB到内存，程序从内存取 —— 磁盘IO次数减少上千倍

public class BufferedCopyDemo {
    public static void main(String[] args) {
        long start = System.currentTimeMillis();

        try (BufferedInputStream bis = new BufferedInputStream(
                     new FileInputStream("source.mp4"));
             BufferedOutputStream bos = new BufferedOutputStream(
                     new FileOutputStream("target.mp4"))) {

            byte[] buffer = new byte[8192];  // 8KB缓冲区
            int bytesRead;
            while ((bytesRead = bis.read(buffer)) != -1) {
                bos.write(buffer, 0, bytesRead);
            }
        } catch (IOException e) {
            e.printStackTrace();
        }

        long end = System.currentTimeMillis();
        System.out.println("复制完成，耗时：" + (end - start) + "ms");
    }
}
```

注意这里的**装饰器模式**：`BufferedInputStream`包装了`FileInputStream`，在不改变接口的情况下增强了功能。IO流大量使用这种设计模式。

### 6.3.4 字符流 — 处理文本的正确方式

字节流读文本会出现编码问题（一个中文用3字节UTF-8编码，字节流按字节读会乱码）。字符流以**字符**为单位读写，自动处理编解码：

```java
// FileWriter / FileReader（简单但无法指定编码）
try (FileWriter writer = new FileWriter("poem.txt");
     FileReader reader = new FileReader("poem.txt")) {

    writer.write("床前明月光，疑是地上霜。");
    writer.flush();  // 强制刷新缓冲区到磁盘

    // 读取
    char[] buffer = new char[128];
    int charsRead = reader.read(buffer);
    System.out.println(new String(buffer, 0, charsRead));
} catch (IOException e) {
    e.printStackTrace();
}
```

> 🚨 **坑点：FileReader/FileWriter无法指定编码，使用的是系统默认编码**
>
> Windows默认编码可能是GBK，Linux是UTF-8，同一个程序在不同系统上读写文件的行为不一致。**始终使用转换流显式指定编码**：

```java
// 正确方式：用转换流明确指定UTF-8
try (BufferedReader reader = new BufferedReader(
             new InputStreamReader(
                 new FileInputStream("data.txt"), StandardCharsets.UTF_8));
     BufferedWriter writer = new BufferedWriter(
             new OutputStreamWriter(
                 new FileOutputStream("output.txt"), StandardCharsets.UTF_8))) {

    // 逐行读取
    String line;
    while ((line = reader.readLine()) != null) {
        writer.write(line);
        writer.newLine();  // 写入换行符（平台无关）
    }
} catch (IOException e) {
    e.printStackTrace();
}
```

**转换流**（`InputStreamReader`/`OutputStreamWriter`）是字节流到字符流的桥梁：字节流 → 按指定编码解码 → 字符流。

**BufferedReader的readLine()**是处理大文本文件的首选方式——一次读一行，内存压力小：

```java
// 统计文件中某个词出现的次数
try (BufferedReader reader = new BufferedReader(
             new InputStreamReader(new FileInputStream("article.txt"), StandardCharsets.UTF_8))) {
    String line;
    int count = 0;
    while ((line = reader.readLine()) != null) {
        if (line.contains("Java")) {
            count++;
        }
    }
    System.out.println("包含'Java'的行数：" + count);
}
```

> 🚨 **坑点：流使用后必须关闭，否则资源泄露**
>
> 每个进程能打开的文件数量是有限制的（Linux默认1024个文件描述符，Windows也是有限制的）。如果打开文件后不关闭，文件句柄会被耗尽，导致后续所有文件操作失败，表现为`Too many open files`错误。**始终使用try-with-resources确保资源被关闭**。

### 6.3.5 NIO.2 Files工具类 — 告别繁琐的IO代码

Java 7引入的NIO.2（`java.nio.file`包）提供了一套现代化API，用一两行代码完成之前几十行才能做的事：

```java
import java.nio.file.*;
import java.nio.charset.StandardCharsets;
import java.util.List;

public class FilesDemo {
    public static void main(String[] args) throws IOException {
        // ===== 写入 =====
        String content = "第一行\n第二行\n第三行";
        Files.writeString(Path.of("data.txt"), content, StandardCharsets.UTF_8);
        // Files.write(Path.of("data.txt"), content.getBytes(StandardCharsets.UTF_8)); // 字节版本

        // ===== 读取全部行 =====
        List<String> lines = Files.readAllLines(Path.of("data.txt"), StandardCharsets.UTF_8);
        lines.forEach(System.out::println);

        // ===== 读取为字符串（Java 11+） =====
        String allContent = Files.readString(Path.of("data.txt"), StandardCharsets.UTF_8);
        System.out.println(allContent);

        // ===== 复制 =====
        Files.copy(Path.of("data.txt"), Path.of("data_backup.txt"),
                   StandardCopyOption.REPLACE_EXISTING);  // 如果目标存在就覆盖

        // ===== 移动/重命名 =====
        Files.move(Path.of("data_backup.txt"), Path.of("archive.txt"),
                   StandardCopyOption.REPLACE_EXISTING);

        // ===== 删除 =====
        Files.deleteIfExists(Path.of("archive.txt"));  // 安全删除

        // ===== 创建目录 =====
        Files.createDirectories(Path.of("output/2026/05")); // 自动创建所有不存在的父目录

        // ===== 遍历目录 =====
        try (var stream = Files.walk(Path.of("."))) {       // walk递归遍历
            stream.filter(Files::isRegularFile)
                  .filter(p -> p.toString().endsWith(".java"))
                  .forEach(System.out::println);
        }

        // ===== 判断文件属性 =====
        Path path = Path.of("data.txt");
        System.out.println("是否存在：" + Files.exists(path));
        System.out.println("是否可读：" + Files.isReadable(path));
        System.out.println("文件大小：" + Files.size(path) + " 字节");
        System.out.println("最后修改：" + Files.getLastModifiedTime(path));
    }
}
```

>

> Files.delete(Path.of("not_exist.txt")); // 抛异常！
> Files.deleteIfExists(Path.of("not_exist.txt")); // 安全，返回false而已

### 6.3.6 Path接口 — 替代传统File类

```java
// 旧式 File（Java 1.0）
File file = new File("dir/subdir/file.txt");

// 新式 Path（Java 7+ NIO.2）
Path path = Path.of("dir", "subdir", "file.txt");  // Java 11+ 工厂方法
// 等价于 Paths.get("dir/subdir/file.txt")         // Java 7+

// Path的常用操作
Path path = Path.of("/home/user/docs/report.pdf");

System.out.println(path.getFileName());     // report.pdf
System.out.println(path.getParent());       // /home/user/docs
System.out.println(path.getRoot());         // / （Linux）或 C:\（Windows）
System.out.println(path.getNameCount());    // 4个路径组件

// 路径解析
Path base = Path.of("/home/user");
Path resolved = base.resolve("projects");   // /home/user/projects

// 相对路径计算
Path absPath = Path.of("/home/user/docs/report.pdf");
Path basePath = Path.of("/home/user");
Path relative = basePath.relativize(absPath); // docs/report.pdf
```

Path替代File不是简单地换API，而是设计理念的提升：Path是接口（可以有不同实现的文件系统），File是具体类——Path更灵活、更适合现代Java的面向接口编程。

## 本章小结

1. **Throwable树**分Error（不处理）、受检异常（强制处理）、非受检异常（bug应修复）。异常的核心不是怎么catch，而是**何时应该捕获、何时应该传播**。
2. **try-catch-finally**中catch要由具体到宽泛排列；finally一定会执行（除exit/JVM崩溃外），但**finally中的return会覆盖try的返回值甚至吞掉异常**——这是最隐蔽的陷阱。
3. **try-with-resources**让资源管理变得优雅，任何实现AutoCloseable的类都可以自动关闭，资源关闭顺序是声明逆序。
4. **throw**手动抛出异常（方法体内），**throws**声明异常（方法签名上）。自定义异常继承RuntimeException并带errorCode。
5. **字节流**处理二进制文件，**字符流**处理文本文件。始终通过InputStreamReader/OutputStreamWriter显式指定UTF-8编码，避免跨平台乱码。
6. **缓冲流**通过内存缓冲区大幅减少磁盘IO次数，是文件操作的标配包装。
7. **NIO.2 Files工具类**让文件操作变得简洁——readAllLines、write、copy、walk等一行代码搞定。

1. 下面代码最终打印什么？

   public static int test() {
       try {
           System.out.println("A");
           return 1 / 0;
       } catch (ArithmeticException e) {
           System.out.println("B");
           return 2;
       } finally {
           System.out.println("C");
       }
   }
   public static void main(String[] args) {
       System.out.println(test());
   }
   ```

2. 如果一个try块中打开了三个文件A.txt、B.txt、C.txt，try-with-resources中它们的声明顺序是A→B→C，那么关闭顺序是什么？为什么这样设计？

3. `BufferedReader`和`FileReader`有什么区别？为什么直接使用`FileReader`不是最佳实践？

4. 写出用NIO.2将一个大文件按行读取并过滤出包含"ERROR"的行的最简代码。

5. try-with-resources中，如果try块抛出一个异常，close()也抛出一个异常，最终哪个异常会被抛给调用者？被"吞掉"的那个异常如何获取？

<details>
<summary>点击查看参考答案</summary>

1. 控制台输出：
   ```
   A
   B
   C
   2
   ```
   try中的`return 1/0`抛出ArithmeticException，跳到catch块，catch块执行`System.out.println("B")`，然后遇到`return 2`——在返回之前，finally块执行`System.out.println("C")`，然后返回2。

2. 关闭顺序是C→B→A（声明的逆序）。这样设计类似栈的后进先出（LIFO），原因：后打开的资源可能依赖先打开的资源。例如输出流B可能引用了输入流A的内容，先关A可能导致B的close失败。

3. BufferedReader是装饰器，包装任意Reader增加缓冲功能；FileReader是直接面向文件的字符流。直接使用FileReader的问题：(1)每次read都触发磁盘IO，效率低；(2)使用系统默认编码，跨平台时会乱码。最佳实践是`new BufferedReader(new InputStreamReader(new FileInputStream(...), StandardCharsets.UTF_8))`。

4. ```java
    Files.lines(Path.of("app.log"), StandardCharsets.UTF_8)
         .filter(line -> line.contains("ERROR"))
         .forEach(System.out::println);
    ```

5. try块中的异常会被优先抛出给调用者；close()中的异常会被作为"被抑制的异常"通过`try异常.getSuppressed()`获取。这个设计是合理的：真正导致问题的是try中的异常，而关闭失败是次要的。
