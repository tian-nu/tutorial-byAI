### 第零篇答案与解析

| 题号 | 答案 | 解析 |
|------|------|------|
| B1 | **B** | DNS解析是第一步，须先将域名翻译为IP地址才能进行后续连接。A（HTTP请求）、C（TCP握手）、D（TLS握手）均在DNS解析之后。 |
| B2 | **B** | 内存（RAM）是临时存储，速度快、价格贵、断电丢失；硬盘是永久存储，速度慢、价格低。A说反了；C中文件系统是硬盘的概念；D速度差异巨大（约1000倍）。 |
| B3 | **D** | "尽力而为地发送"是UDP的特点，不是TCP的。TCP通过序号确认、超时重传、流量控制等机制保证可靠传输。 |
| B4 | **C** | MySQL默认端口3306；80是HTTP；443是HTTPS；6379是Redis。 |
| B5 | **B** | 404 = Not Found，资源未找到。A是500；C是401；D是200。 |
| B6 | **C** | 同一进程内的线程共享内存空间，这是线程间通信效率高的原因。A错，一个进程可以有多个线程；B错，进程才是资源分配的基本单位；D错，进程间内存隔离。 |
| A7 | **C** | 三次握手第二步：服务端收到SYN后，回复 SYN + ACK。第一步客户端发SYN，第三步客户端发ACK。 |
| A8 | **C** | WebSocket适合需要服务端主动推送的实时双向通信场景。A、B、D都是典型的请求-响应模式，用HTTP即可。 |
| A9 | **C** | HTTPS的TLS握手阶段用非对称加密安全交换"会话密钥"，之后用对称加密传输大量数据（因为对称加密速度快得多）。A错，HTTPS混合使用二者；B说反了；D无此定论——二者用途不同。 |
| T10 | **C** | 陷阱在于"永远等价"——`localhost` 在IPv6环境下可能解析为 `::1` 而非 `127.0.0.1`，取决于系统的hosts配置。A、B、D均为正确描述。 |

### 第一篇答案与解析

| 题号 | 答案 | 解析 |
|------|------|------|
| B1 | **C** | Go中声明未初始化的变量自动获得零值（int为0，string为""，bool为false等）。A错，还可以用 `:=`；B错，`:=` 只能在函数内使用；D错，Go支持类型推断。 |
| B2 | **B** | `go` 关键字启动Goroutine：`go funcName()`。Go语言没有 `async`、`thread`、`spawn` 关键字。 |
| B3 | **C** | 数组长度是类型的一部分，`[3]int` 和 `[5]int` 不能互相赋值。A错，数组长度固定；B错，数组才是值类型，切片是引用类型（底层指针）；D错，切片可以用 `make` 创建。 |
| B4 | **B** | Go接口采用隐式实现（Duck Typing），无需显式声明，只要类型实现了接口的所有方法即可。这是Go接口设计的核心特色。 |
| B5 | **C** | `defer` 在函数返回前执行，多个defer按LIFO（后进先出/栈）顺序执行。常用于资源释放、文件关闭等。 |
| B6 | **C** | Go Map的双值返回：`v, ok := m[key]`，`ok` 为 `true` 表示键存在。直接访问不存在键会返回零值（不会panic），容易误判。 |
| A7 | **C** | 从已关闭的Channel接收数据不会panic，会立即返回零值（第二个返回值为false表示已关闭）。A、B、D均正确。 |
| A8 | **B** | 这是经典"闭包陷阱"——goroutine引用的是循环变量 `i` 的地址。当goroutine真正执行时，`i` 可能已经变为5，所以可能输出 `55555`。修复方式是 `i := i` 拷贝变量或通过参数传递。 |
| A9 | **C** | 值接收者操作的是副本，指针接收者操作的是原始值。A错，Go支持指针；B错，`new` 分配零值内存返回指针，`make` 只用于slice/map/channel的初始化；D错，`&` 可以取地址。 |
| T10 | **B** | 陷阱在于：`s[0] = 100` 修改了底层数组，所以第一个元素变了。但 `s = append(s, 200)` 是在函数内对局部变量 `s` 重新赋值，不影响 `main` 中的 `s` 变量。又因为原slice容量为2，append 200后长度为2未超过容量，没有触发扩容换底层数组，但外层的 `s` 变量（len=1）看不到新元素。最终输出 `[100]`。 |

### 第二篇答案与解析

| 题号 | 答案 | 解析 |
|------|------|------|
| B1 | **C** | O(1)表示常数时间，不随数据量增长。O(n)是线性增长；O(log n)是对数增长；O(n²)是平方增长。 |
| B2 | **B** | 栈是LIFO（Last In First Out），像一叠盘子。A描述的是队列；C是数组；D是哈希表/Map。 |
| B3 | **C** | Go 1.8+ 的 `sort` 包使用 pdqsort（Pattern-Defeating Quicksort），是快排、堆排和插入排序的混合体。 |
| B4 | **B** | 二分查找核心前提是数据有序。每次将搜索范围折半，无序数据无法使用。 |
| B5 | **C** | 前进/后退是典型的栈场景——后访问的页面先回退。前进是另一个栈。 |
| B6 | **B** | 哈希冲突是指不同键映射到同一桶。A错，最坏情况（大量冲突）可退化为O(n)；C错，Go的map键不能重复（新值覆盖旧值），但并非所有哈希表实现都如此；D错，键可以是任何可比较类型。 |
| A7 | **B** | 二叉搜索树的性质：左子树 < 根 < 右子树。中序遍历（左→根→右）恰好产生升序序列。 |
| A8 | **C** | Go 1.18+的扩容策略：容量<256时新容量≈2倍；>=256时按 (old+3*256)/4 的公式缓慢增长。B不一定，若append后长度未超容量则不会扩容。 |
| A9 | **B** | 当pivot每次都取到极值（最小或最大），分区极度不平衡，递归深度变为n，时间复杂度退化为O(n²)。 |
| T10 | **D** | 陷阱：`heap.Pop` 移除并返回的是堆中的**最小元素**（最小堆）或**最大元素**（最大堆），而非任意元素。这是堆的定义决定的。A、B、C均正确。 |

### 第三篇答案与解析

| 题号 | 答案 | 解析 |
|------|------|------|
| B1 | **B** | `DELETE FROM users`（无WHERE条件）删除所有行但表结构保留。A会删除整张表；C和D不是标准SQL语法。 |
| B2 | **B** | 索引相当于书的目录，记录"某个值在哪个数据页"。A、C、D都是牵强比喻。 |
| B3 | **C** | Redis没有使用B+树，它使用SDS、链表、哈希表、跳表、压缩列表等内存数据结构。B+树是MySQL InnoDB的索引结构。A、B、D都是Redis快的原因。 |
| B4 | **B** | LEFT JOIN返回左表所有行，右表无匹配则填NULL。A只返回匹配行；C返回右表所有行；D返回两表所有行。 |
| B5 | **C** | I = Isolation（隔离性）：并发事务互不干扰。A是原子性（Atomicity）；B是一致性（Consistency）；D是持久性（Durability）。 |
| B6 | **B** | 缓存击穿：一个热点Key过期，大量并发请求直接打到数据库。A描述的是缓存雪崩；C描述的是缓存穿透。 |
| A7 | **D** | 联合索引遵循最左前缀原则，`WHERE b=2 AND c=3` 跳过了最左列 `a`，无法使用索引。A、B、C均能匹配。 |
| A8 | **B** | InnoDB的行锁基于索引实现。如果查询没有走索引，会升级为表锁。这正是"索引失效导致锁升级"这个常见坑点的原理。 |
| A9 | **B** | Cache Aside模式（旁路缓存）：读时先读缓存→缓存未命中则读DB→写入缓存；写时先更新DB→再删除缓存。这是最常用的策略。A和C可能导致数据不一致；D浪费缓存价值。 |
| T10 | **C** | 此题考查InnoDB的特殊性。标准SQL中REPEATABLE READ不能防幻读，但**InnoDB通过MVCC + Next-Key Lock（间隙锁）在REPEATABLE READ级别下也可以防止幻读**。A错，READ UNCOMMITTED无法防脏读；B错，READ COMMITTED无法防不可重复读；D错，SERIALIZABLE允许并发读，只是禁止并发写。 |

### 第十篇答案与解析

| 题号 | 答案 | 解析 |
|------|------|------|
| B1 | **C** | RESTful风格：用HTTP方法表达操作（DELETE删除），URL用名词表示资源（`/todos/1`）。A和D在URL中使用了动词；B既有动词又不符合资源命名规范。 |
| B2 | **C** | bcrypt加盐哈希是密码存储的行业标准。A明文是安全禁忌；B的MD5已不安全且无盐；D加密可逆（有密钥），不如哈希（不可逆）。 |
| B3 | **C** | "文章↔标签"是典型的多对多关系，通常通过一张中间表（`article_tags`）来实现。 |
| B4 | **B** | Redis预减库存（内存级原子操作）+ 消息队列异步下单是秒杀系统的经典架构。A无法抗住高并发；C不切实际；D单纯加CPU不能解决数据库瓶颈。 |
| B5 | **B** | WebSocket提供全双工、低延迟的实时通信，是IM系统的最佳选择。A轮询浪费资源且延迟高；C和D不适合即时场景。 |
| B6 | **D** | 微服务间绝对不应直接访问对方的数据库！这会破坏服务自治，造成紧耦合。应通过API/RPC/消息队列等接口进行通信。 |
| A7 | **B** | 最佳组合：Redis原子操作做第一层拦截（快），数据库乐观锁做最终保障（准），MQ异步下单（削峰）。A只解决表面问题；C治标不治本；D只能增加注册复杂度，不防超卖。 |
| A8 | **C** | 微服务应"数据去中心化"——每个服务有自己的数据库。共享数据库会破坏服务的独立性和可伸缩性，是微服务反模式。 |
| A9 | **B** | 离线消息的标准处理方式：消息存入数据库并标记为未读，B上线后拉取。A丢弃丢失消息不可接受；C邮件通知只是补充手段；D直接拒绝不友好。 |
| T10 | **B** | 陷阱：六个项目有意安排了递进难度——从单体Todo（项目一）到带认证的系统（项目二）→带文章标签关联的博客（项目三）→高并发秒杀（项目四）→实时通信IM（项目五）→分布式微服务（项目六）。每个项目引入新的技术挑战，不建议跳过。 |
