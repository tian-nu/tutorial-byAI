## 69.1 中间件是什么

中间件（Middleware）是一个在请求到达handler**之前**和handler执行**之后**都能插入逻辑的函数。它像洋葱的每一层皮，一层包一层。

### 一个请求经过中间件的流程

```
HTTP请求到达
   ↓
┌──────────────────────────┐
│  中间件1：日志记录         │
│  ├─ c.Next() 之前：记录开始时间、请求路径
│  └─ c.Next() 之后：计算耗时、打印日志
│  ┌────────────────────┐  │
│  │ 中间件2：身份验证    │  │
│  │ ├─ c.Next() 之前：检查Token是否有效
│  │ │ 如果无效→c.Abort()，直接返回401
│  │ └─ c.Next() 之后：什么都不做
│  │ ┌──────────────┐   │  │
│  │ │ Handler      │   │  │
│  │ │ 真正的业务逻辑│   │  │
│  │ └──────────────┘   │  │
│  └────────────────────┘  │
└──────────────────────────┘
   ↓
HTTP响应返回
```

**每个中间件都可以选择：**
- 在handler之前做一些事（鉴权、参数校验）
- 调用 `c.Next()` 把请求传给下一个中间件
- 在handler之后做一些事（记录日志、添加响应头）
- 调用 `c.Abort()` 终止请求链（权限不够，直接返回）

## 69.2 自定义中间件

### 中间件的函数签名

所有中间件都长这样：

```go
func MyMiddleware(c *gin.Context) {
    fmt.Println("请求到达中间件")
    c.Next()
    fmt.Println("请求处理完成，中间件收尾")
}
```

在 `gin.Context` 上操作，调用 `c.Next()` 把控制权交给下一个处理者。

### c.Next() 前后的代码执行区别

```go
r := gin.Default()

r.Use(func(c *gin.Context) {
    fmt.Println("A: 前")
    c.Next()
    fmt.Println("A: 后")
})

r.Use(func(c *gin.Context) {
    fmt.Println("B: 前")
    c.Next()
    fmt.Println("B: 后")
})

r.GET("/test", func(c *gin.Context) {
    fmt.Println("Handler执行")
    c.String(200, "OK")
})
```

访问 `/test`，终端输出：

```
A: 前
B: 前
Handler执行
B: 后
A: 后
```

**规律**：
- `c.Next()` 之前的代码，按中间件注册顺序**正序执行**
- `c.Next()` 之后的代码，按中间件注册顺序**逆序执行**
- 这就像一个栈——先进后出（FILO）

### c.Abort()：中断请求

```go
r.Use(func(c *gin.Context) {
    token := c.GetHeader("Authorization")
    if token == "" {
        c.JSON(401, gin.H{"msg": "未登录"})
        c.Abort()
        return
    }
    c.Next()
})
```

`c.Abort()` 的作用是**阻止后续的中间件和handler执行**。但注意：`Abort` 不会阻止**当前中间件内 `Abort` 之后的代码执行**——你需要自己 `return`。

```go
r.Use(func(c *gin.Context) {
    fmt.Println("中间件A")
    c.Abort()
    fmt.Println("Abort之后这行仍会打印！")  
})

r.GET("/test", func(c *gin.Context) {
    fmt.Println("Handler不会执行，因为被Abort了")
})
```

正确的写法：

```go
r.Use(func(c *gin.Context) {
    if shouldAbort {
        c.JSON(401, gin.H{"msg": "未登录"})
        c.Abort()
        return
    }
    c.Next()
})
```

### c.IsAborted()：检查是否已被中断

后续中间件可以通过 `c.IsAborted()` 判断前面的中间件是否已经中断：

```go
func MiddlewareA(c *gin.Context) {
    if someCondition {
        c.Abort()
        return
    }
    c.Next()
}

func MiddlewareB(c *gin.Context) {
    if c.IsAborted() {
        return
    }
    c.Next()
}
```

## 69.3 内置中间件

Gin自带几个开箱即用的中间件，分别解决常见问题。

### Logger：请求日志

```go
r := gin.New()
r.Use(gin.Logger())
```

每次请求会在终端打印类似这样的日志：

```
[GIN] 2024/01/15 - 14:32:05 | 200 |    2.3412ms | 127.0.0.1 | GET "/users/1"
```

- 日期时间
- HTTP状态码（绿色=成功，黄色=重定向，红色=错误）
- 请求耗时
- 客户端IP
- HTTP方法和路径

### Logger自定义格式

```go
r.Use(gin.LoggerWithFormatter(func(param gin.LogFormatterParams) string {
    return fmt.Sprintf("[%s] %s %s %d %s\n",
        param.TimeStamp.Format("2006-01-02 15:04:05"),
        param.Method,
        param.Path,
        param.StatusCode,
        param.Latency,
    )
}))
```

### Recovery：Panic恢复

```go
r := gin.New()
r.Use(gin.Recovery())
```

当你的代码意外panic时（比如数组越界、空指针），没有Recovery的话**整个服务进程会挂掉**。有了Recovery，Gin会：
1. 捕获panic
2. 打印堆栈信息到stderr
3. 给客户端返回500 Internal Server Error
4. 服务继续运行，不受影响

### 自定义Recovery

```go
r.Use(gin.CustomRecovery(func(c *gin.Context, err any) {
    c.JSON(500, gin.H{
        "code": 500,
        "msg":  "服务器内部错误，请稍后重试",
    })
}))
```

给客户端返回友好的错误提示，真实错误信息记录到日志系统（第73章）。

### CORS：跨域请求

```go
import "github.com/gin-contrib/cors"

r.Use(cors.Default())
```

第76章会详细讲解CORS的原理和配置。

## 69.4 中间件执行顺序

中间件可以在三个层级注册，执行顺序很重要：

```
全局中间件（r.Use） → 路由组中间件（group.Use） → 路由级中间件（r.GET第三个参数）
```

```go
r := gin.Default()

r.Use(GlobalMiddleware)

auth := r.Group("/api")
auth.Use(AuthMiddleware)
{
    auth.GET("/profile", ProfileMiddleware, profileHandler)
}
```

执行顺序：

```
GlobalMiddleware → AuthMiddleware → ProfileMiddleware → profileHandler
```

每个中间件内部的 `c.Next()` 前后代码按"洋葱模型"执行。

### 全局vs路由组vs路由级

| 注册位置 | 语法 | 影响范围 |
|---------|------|---------|
| 全局 | `r.Use(mw)` | 所有路由 |
| 路由组 | `group.Use(mw)` | 该组内所有路由 |
| 路由级 | `r.GET("/", mw1, mw2, handler)` | 仅这一个路由 |

优先级：全局 → 路由组 → 路由级。越靠近路由的中间件越晚执行（越接近洋葱芯）。

## 69.5 实战中间件

### 鉴权中间件

以下为简化的鉴权示例，完整JWT实现见第79章。

```go
func AuthMiddleware(c *gin.Context) {
    token := c.GetHeader("Authorization")

    if token == "" {
        c.JSON(http.StatusUnauthorized, gin.H{
            "code": 401,
            "msg":  "缺少认证令牌",
        })
        c.Abort()
        return
    }

    if !strings.HasPrefix(token, "Bearer ") {
        c.JSON(http.StatusUnauthorized, gin.H{
            "code": 401,
            "msg":  "令牌格式错误",
        })
        c.Abort()
        return
    }

    token = strings.TrimPrefix(token, "Bearer ")

    userID, err := parseToken(token)
    if err != nil {
        c.JSON(http.StatusUnauthorized, gin.H{
            "code": 401,
            "msg":  "令牌无效或已过期",
        })
        c.Abort()
        return
    }

    c.Set("userID", userID)
	c.Next()
}

// parseToken 简化版JWT解析，完整实现见第79章
// 生产环境请使用 github.com/golang-jwt/jwt/v5
func parseToken(tokenString string) (int, error) {
	parts := strings.Split(tokenString, ".")
	if len(parts) != 3 {
		return 0, fmt.Errorf("无效的token格式")
	}
	// 此处省略签名验证，仅作示例演示解码
	return 1, nil
}
```

### 耗时统计中间件

```go
func CostMiddleware(c *gin.Context) {
    start := time.Now()

    c.Next()

    cost := time.Since(start)
    c.Header("X-Request-Cost", cost.String())

    if cost > 500*time.Millisecond {
        fmt.Printf("[慢查询警告] %s %s 耗时 %v\n",
            c.Request.Method,
            c.Request.URL.Path,
            cost,
        )
    }
}
```

### 请求ID中间件

为每个请求生成唯一ID，便于追踪：

```go
import "github.com/google/uuid"

func RequestIDMiddleware(c *gin.Context) {
    requestID := uuid.New().String()
    c.Set("requestID", requestID)
    c.Header("X-Request-ID", requestID)
    c.Next()
}
```

搭配日志系统使用，当用户反馈问题时，用 `X-Request-ID` 就能在日志里搜索到这次请求的完整记录。

## 🤔 想多一点：中间件的滥用陷阱

### 1. 不要在中间件里做太重的事

中间件每个请求都要执行。如果你在中间件里查数据库、调远程服务，那每个请求（包括静态文件、健康检查）都会拖慢。耗时的操作应该放handler里，只对需要的路由执行。

### 2. 多个中间件的循环问题

中间件A调 `c.Next()` 进入中间件B，B调 `c.Next()` 进入handler，handler返回后回到B，B返回后回到A。这个链条如果在某个环节忘了 `c.Next()` 或用了 `c.Abort()`，可能导致部分代码未执行或资源未释放。用 `defer` 确保资源释放：

```go
func DBMiddleware(c *gin.Context) {
    db := connectDB()
    defer db.Close()

    c.Set("db", db)
    c.Next()
}
```

### 3. 中间件的顺序决定行为

以下两种顺序产生完全不同的结果：

```go
r.Use(LoggerMiddleware, AuthMiddleware)
```

Logger记录所有请求（包括401未登录的）。

```go
r.Use(AuthMiddleware, LoggerMiddleware)
```

Logger只记录通过鉴权的请求（被Abort的请求Logger的 `c.Next()` 后的代码不会执行到）。

**先放全局观察类（Logger、RequestID），再放业务拦截类（Auth、限流），最后放数据准备类（DB注入）。**

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 中间件签名 | `func(c *gin.Context)`，调 `c.Next()` 传递请求 |
| c.Next()前 | 正序执行，做预处理（鉴权、准备数据） |
| c.Next()后 | 逆序执行，做后处理（记录日志、添加响应头） |
| c.Abort() | 中断请求链，阻止后续中间件和handler执行 |
| c.IsAborted() | 判断前置中间件是否中断 |
| Logger | `gin.Logger()` 自动打印请求日志 |
| Recovery | `gin.Recovery()` 捕获panic防止进程崩溃 |
| 洋葱模型 | 中间件层层包裹handler，先进后出 |
| 注册层级 | 全局 `r.Use` → 路由组 `group.Use` → 路由级 |
| 鉴权中间件 | 检查Token→解析用户ID→`c.Set("userID", id)`→`c.Next()` |
| 耗时统计 | `time.Now()` 在 `c.Next()` 前后记录 |
| 请求ID | `uuid.New()` + `c.Header("X-Request-ID")` |
| 顺序原则 | 先观察（Logger/RequestID）→ 再拦截（Auth）→ 最后准备数据（DB） |
