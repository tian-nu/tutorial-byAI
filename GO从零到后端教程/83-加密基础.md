## 83.1 哈希算法

### 哈希是什么

哈希就是把**任意长度的数据**变成**固定长度的摘要**。就像把一整本书压缩成一个64位的指纹——指纹能代表这本书，但你无法从指纹还原出整本书。

三个核心特性：
1. **不可逆**：从哈希值无法推出原始数据（单向性）
2. **定长输出**：不管输入是1字节还是1GB，输出长度固定（如SHA256永远是256位）
3. **雪崩效应**：原始数据改一个bit，哈希值变化一半以上——完全不同的输出

### MD5（已死，但你要知道它的墓碑）

MD5输出128位（32个十六进制字符）。1996年发现碰撞漏洞——可以构造两个不同内容但相同MD5值的文件。**2008年彻底宣告死亡。**

```go
import "crypto/md5"

data := []byte("hello world")
hash := md5.Sum(data)
fmt.Printf("%x\n", hash)
```

如果有人跟你说"我们系统用MD5存密码"——跑。这话的意思是"我们的安全策略停留在1995年"。

MD5唯一还能用的场景：**非安全性校验**（比如文件传输完整性检查，"这个文件传完整了吗？"而不是"这个文件被人恶意替换了吗？"）。对于后者，你该用SHA256。

### SHA256（当前的安全标准）

SHA256输出256位（64个十六进制字符）。至今没有有效的碰撞攻击。

```go
import "crypto/sha256"

data := []byte("hello world")
hash := sha256.Sum256(data)
fmt.Printf("%x\n", hash)
```

输出：
```
b94d27b9934d3e08a52e52d7da7dabfac484efe37a5380ee9088f7ace2efcde9
```

### SHA512和SHA-3

SHA512输出512位，比SHA256更长，理论上更安全但更慢。对大多数场景SHA256已经足够（256位意味着有2^256种可能——这个数字比宇宙中所有原子的数量还大）。

SHA-3是最新的SHA标准（2015年发布），和SHA256用了完全不同的内部结构，不是SHA256的直接后继。如果SHA256有一天被攻破，SHA-3是备选方案。

### 哈希的雪崩效应演示

```go
func main() {
    a := sha256.Sum256([]byte("hello"))
    b := sha256.Sum256([]byte("hallo"))
    
    fmt.Printf("hello: %x\n", a)
    fmt.Printf("hallo: %x\n", b)
}
```

输出（注意两个哈希完全不同——只改了一个字母）：

```
hello: 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
hallo: d3751d33f9cd5049c4af2b462735457e4d3baf130bcbb87f389e349fbaeb20b9
```

这就是雪崩效应——输入微小的改变导致输出天翻地覆。

## 83.2 哈希的三大应用场景

### 场景一：密码存储（+盐）

第77章已经详细讲过——哈希+盐+慢哈希（bcrypt）。这里补充一个纯粹用SHA256的代码示例（仅供学习，生产用bcrypt）：

```go
func hashPassword(password string) (string, string) {
    saltBytes := make([]byte, 16)
    rand.Read(saltBytes)
    salt := hex.EncodeToString(saltBytes)

    hash := sha256.Sum256([]byte(salt + password))

    return salt, hex.EncodeToString(hash[:])
}
```

### 场景二：文件完整性校验

下载文件时，官网通常会提供SHA256校验码：

```
ubuntu-22.04.iso
SHA256: a435f6f393dda581172490eda72f965f2b...
```

下载完成后自己算一次：

```go
func fileSHA256(path string) (string, error) {
    f, err := os.Open(path)
    if err != nil {
        return "", err
    }
    defer f.Close()

    h := sha256.New()
    io.Copy(h, f)

    return hex.EncodeToString(h.Sum(nil)), nil
}
```

如果自己算的和官方提供的不一致：要么文件损坏了（网络传输错误），要么文件被篡改了（有人替换了下载链接中的文件）。

### 场景三：数字签名的基础

数字签名并不是对整份文件做非对称加密（文件太大，太慢），而是先对文件做哈希，然后对**哈希值**做签名。详见83.5。

## 83.3 对称加密：AES

### 比喻：一把钥匙开一把锁

对称加密就像一个密码箱：你用同一把钥匙锁上，也是用同一把钥匙打开。加密和解密用的是**同一个密钥**。

- 速度快（比非对称加密快100-1000倍）
- 适合加密大量数据
- 但密钥分发困难——你怎么把钥匙安全地交给对方？

### AES简介

AES（Advanced Encryption Standard，当今最广泛使用的对称加密标准，详见附录I）是当今最广泛使用的对称加密标准。支持三种密钥长度：

| AES类型 | 密钥长度 | 安全级别 |
|---------|---------|---------|
| AES-128 | 128位（16字节） | 安全 |
| AES-256 | 256位（32字节） | 更安全 |
| AES-192 | 192位（24字节） | 较少使用 |

AES本身是"块加密"——每次加密固定大小的数据块（16字节）。要加密任意长度的数据，需要配合"工作模式"。

### 工作模式：CTR和GCM

**CTR模式（Counter）**：把AES变成"流加密"——把块密码变成一个随机字节流，然后用这个字节流和原文做XOR。CTR速度快，但**不提供完整性校验**——密文被修改了你知道不了。

**GCM模式（Galois/Counter Mode）**：CTR + 完整性校验。GCM是目前最推荐的工作模式——同时提供加密和防篡改。

用AES-GCM的好处：如果有人改了密文中的任何一个字节，解密时会发现并报错。这防止了"密文篡改攻击"——攻击者不需要解密，只需要在你不知道的情况下修改密文（比如修改加密的金额数据）。

### Go中使用AES-GCM

```go
package main

import (
	"crypto/aes"
	"crypto/cipher"
	"crypto/rand"
	"encoding/hex"
	"fmt"
	"io"
)

func EncryptAESGCM(plaintext []byte, key []byte) (string, error) {
	block, err := aes.NewCipher(key)
	if err != nil {
		return "", err
	}

	gcm, err := cipher.NewGCM(block)
	if err != nil {
		return "", err
	}

	nonce := make([]byte, gcm.NonceSize())
	if _, err := io.ReadFull(rand.Reader, nonce); err != nil {
		return "", err
	}

	ciphertext := gcm.Seal(nonce, nonce, plaintext, nil)
	return hex.EncodeToString(ciphertext), nil
}

func DecryptAESGCM(ciphertextHex string, key []byte) ([]byte, error) {
	ciphertext, _ := hex.DecodeString(ciphertextHex)

	block, err := aes.NewCipher(key)
	if err != nil {
		return nil, err
	}

	gcm, err := cipher.NewGCM(block)
	if err != nil {
		return nil, err
	}

	nonceSize := gcm.NonceSize()
	if len(ciphertext) < nonceSize {
		return nil, fmt.Errorf("密文太短")
	}

	nonce, ciphertext := ciphertext[:nonceSize], ciphertext[nonceSize:]
	return gcm.Open(nil, nonce, ciphertext, nil)
}
```

使用示例：

```go
func main() {
	key := make([]byte, 32)
	rand.Read(key)

	plaintext := []byte("这是一段需要加密的敏感数据")

	encrypted, _ := EncryptAESGCM(plaintext, key)
	fmt.Println("加密后:", encrypted)

	decrypted, _ := DecryptAESGCM(encrypted, key)
	fmt.Println("解密后:", string(decrypted))
}
```

### 关键概念：Nonce（或IV）

Nonce（Number used Once）是一个**不需要保密的随机数**，但必须满足：
1. **每次加密都不同**——即使加密同一段数据、用同一个密钥
2. **解密时需要同一个Nonce**——所以通常把Nonce附在密文前面一起传输

为什么需要Nonce？如果没有它，同一段明文用同一个密钥加密，每次输出的密文都一样。攻击者可以从密文的规律里推断明文内容。Nonce打破了这种确定性。

GCM的 `Seal` 方法自动把nonce拼接在密文前面，`Open` 时自动分离。你不需要手动管理。

### AES密钥管理

密钥从哪里来？**绝不能硬编码在代码里。**

```go
key := []byte("hardcoded-key-1234567890!!!")  // ❌ 绝对不能这样
```

正确姿势：

```go
key, _ := hex.DecodeString(os.Getenv("AES_KEY"))
```

或从专门的密钥管理服务（KMS）获取——比如AWS KMS、阿里云KMS。

## 83.4 非对称加密：RSA

### 比喻：信箱和钥匙

非对称加密（使用公钥和私钥配对的加密方式，详见附录I）就像街边的信箱：任何人都可以把信塞进去（用**公钥**加密），但只有有钥匙的人能取出来（用**私钥**解密）。

- 公钥是公开的——你可以把它贴在网上，发给任何人
- 私钥是你独享的——除了你，没有人能解开用公钥加密的内容

### 为什么需要非对称加密？

对称加密的问题：Alice和Bob怎么安全地交换密钥？如果他们在不安全的信道上传输密钥，密钥可能被Eve截获。

非对称加密的答案：Bob生成一对密钥（公钥+私钥），**把公钥公开发给Alice**。Alice用公钥加密消息，只有Bob的私钥能解密。Eve截获了公钥也没用——公钥只能加密，不能解密。

### RSA（最经典的非对称加密算法，详见附录I）的Go实现

```go
package main

import (
	"crypto/rand"
	"crypto/rsa"
	"crypto/sha256"
	"encoding/base64"
	"fmt"
)

func GenerateRSAKeyPair() (*rsa.PrivateKey, error) {
	return rsa.GenerateKey(rand.Reader, 2048)
}

func RSAEncrypt(pubKey *rsa.PublicKey, plaintext []byte) (string, error) {
	ciphertext, err := rsa.EncryptOAEP(
		sha256.New(),
		rand.Reader,
		pubKey,
		plaintext,
		nil,
	)
	if err != nil {
		return "", err
	}
	return base64.StdEncoding.EncodeToString(ciphertext), nil
}

func RSADecrypt(privKey *rsa.PrivateKey, cipherB64 string) ([]byte, error) {
	ciphertext, _ := base64.StdEncoding.DecodeString(cipherB64)

	plaintext, err := rsa.DecryptOAEP(
		sha256.New(),
		rand.Reader,
		privKey,
		ciphertext,
		nil,
	)
	if err != nil {
		return nil, err
	}
	return plaintext, nil
}
```

使用：

```go
func main() {
	privKey, _ := GenerateRSAKeyPair()
	pubKey := &privKey.PublicKey

	message := []byte("用公钥加密的秘密消息")
	encrypted, _ := RSAEncrypt(pubKey, message)
	fmt.Println("加密:", encrypted)

	decrypted, _ := RSADecrypt(privKey, encrypted)
	fmt.Println("解密:", string(decrypted))
}
```

### OAEP填充

RSA加密必须配合填充（Padding）方案使用，裸RSA是不安全的。OAEP（Optimal Asymmetric Encryption Padding）是当前标准——它能防止多种数学攻击。

### RSA的性能问题

RSA加密非常慢（比AES慢1000倍以上），而且有长度限制——RSA-2048最多加密245字节的数据。所以你永远不会用RSA加密一整份文件。

实际做法是**混合加密**：
1. 生成一个随机的AES密钥
2. 用AES加密大量数据
3. 用RSA公钥加密那个AES密钥
4. 把 `[RSA加密的AES密钥] + [AES加密的数据]` 一起发给对方

HTTPS就是用的这种混合方案！

### RSA密钥序列化

把密钥保存到文件：

```go
func SavePrivateKey(path string, key *rsa.PrivateKey) error {
	data := x509.MarshalPKCS1PrivateKey(key)
	block := &pem.Block{
		Type:  "RSA PRIVATE KEY",
		Bytes: data,
	}
	f, _ := os.Create(path)
	defer f.Close()
	return pem.Encode(f, block)
}

func LoadPrivateKey(path string) (*rsa.PrivateKey, error) {
	data, err := os.ReadFile(path)
	if err != nil {
		return nil, err
	}
	block, _ := pem.Decode(data)
	return x509.ParsePKCS1PrivateKey(block.Bytes)
}
```

## 83.5 数字签名

### 签名 vs 加密

很多人搞混RSA的两种用法：

| | 公钥加密，私钥解密 | 私钥签名，公钥验签 |
|---|---|---|
| **方向** | 公钥→私钥 | 私钥→公钥 |
| **目的** | 保密——只有你能看 | 防伪——证明是我发的 |
| **速度** | 慢 | 慢 |
| **场景** | 发送加密消息 | 签署文件、JWT RS256签名 |

加密问的是"谁能看"，签名问的是"谁发的"。

### 数字签名的完整逻辑

1. 发送者（Alice）对消息做SHA256哈希
2. Alice用自己的**私钥**对哈希值做签名（加密处理）
3. Alice把 `[原始消息] + [签名]` 发给Bob
4. Bob拿到后：
   - 对原始消息做SHA256哈希 → 得到 `hash1`
   - 用Alice的**公钥**解密签名 → 得到 `hash2`
   - 对比 `hash1 == hash2`？

如果相等，证明两点：
- **消息没被篡改**（完整性）：如果消息被改过，hash1就会变
- **确实是Alice发的**（不可抵赖性）：只有Alice的私钥能产生匹配的签名

### Go中的RSA签名

```go
func RSASign(privKey *rsa.PrivateKey, message []byte) (string, error) {
	hashed := sha256.Sum256(message)

	signature, err := rsa.SignPKCS1v15(rand.Reader, privKey, crypto.SHA256, hashed[:])
	if err != nil {
		return "", err
	}
	return base64.StdEncoding.EncodeToString(signature), nil
}

func RSAVerify(pubKey *rsa.PublicKey, message []byte, sigB64 string) error {
	hashed := sha256.Sum256(message)

	signature, _ := base64.StdEncoding.DecodeString(sigB64)

	return rsa.VerifyPKCS1v15(pubKey, crypto.SHA256, hashed[:], signature)
}
```

### ECDSA：更现代的选择

ECDSA（椭圆曲线数字签名算法）比RSA签名更快、密钥更短、同等安全级别更高。JWT中的ES256就是ECDSA。

Go的 `crypto/ecdsa` 包用法类似：

```go
privKey, _ := ecdsa.GenerateKey(elliptic.P256(), rand.Reader)

hashed := sha256.Sum256(message)
sig, _ := ecdsa.SignASN1(rand.Reader, privKey, hashed[:])
valid := ecdsa.VerifyASN1(&privKey.PublicKey, hashed[:], sig)
```

ECDSA-256的密钥只有32字节，而同等安全级别的RSA需要3072位（384字节）。这也是为什么现代系统越来越多地转向椭圆曲线算法。

## 83.6 四种加密工具的对比与选择

| | 哈希 (SHA256) | 对称加密 (AES) | 非对称加密 (RSA) | 数字签名 |
|---|---|---|---|---|
| **密钥** | 无 | 一个（双方共用） | 一对（公钥+私钥） | 一对（私钥签+公钥验） |
| **能不能解密** | 不能 | 能 | 能 | 不是用来加密的 |
| **速度** | 快 | 快 | 慢（1000× AES） | 慢 |
| **用途** | 验证完整性 | 加密大量数据 | 交换密钥/加密小数据 | 证明身份/防篡改 |
| **Go包** | `crypto/sha256` | `crypto/aes` | `crypto/rsa` | `crypto/rsa`或`crypto/ecdsa` |

### 一个后端工程师该记住的原则

1. **哈希用SHA256**，不要碰MD5和SHA1
2. **对称加密用AES-GCM**，不要用ECB和CBC（旧模式，不安全）
3. **非对称加密在HTTPS之外极少手动使用**——理解原理即可，你写的代码99%会依赖TLS而非手动RSA
4. **密码存储用bcrypt**，不是SHA256，不是MD5，不是任何"快速"哈希
5. **密钥从环境变量或KMS获取**，永远不硬编码

### 🤔 想多一点

你可能会想：既然Go的标准库有 `crypto/rsa` 和 `crypto/aes`，为什么不写一个自己的加密通信协议？

答案是：**不要这样做。** 密码学的首要规则是：不要自己发明密码学协议。HTTPS/TLS是由全世界最聪明的密码学家设计、经过二十多年实战检验的——你的三行代码根本比不上。标准库提供的加密原语是用来**对接现有安全协议**的（比如给已有的系统做RSA签名验证），不是让你用来从头发明新协议的。

TLS（下一章深入讨论）已经帮你解决了传输加密的所有问题——对称加密、非对称加密、密钥交换、数字签名。你的工作不是重写TLS，而是正确配置它。

| 知识点 | 要点 |
|--------|------|
| 哈希特性 | 不可逆、定长输出、雪崩效应 |
