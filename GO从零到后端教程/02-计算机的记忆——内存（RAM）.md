# 第2章 · 计算机的记忆——内存（RAM）

> "内存是计算机的短期记忆。你打开一个程序，它在内存里；你写了一行代码定义了一个变量，它在内存里。但内存到底是什么？为什么又有栈又有堆？"

---

## 2.1 内存是什么

### 一个比喻：你的临时工作台

想象你在做一道复杂的菜——需要同时处理很多食材。你不可能每次需要一颗蒜就跑一次冰箱。你会把要用到的食材放在**台面上**，方便随时取用。

内存就是这个台面。

| 存储层次 | 比喻 | 特点 |
|----------|------|------|
| CPU寄存器 | 你手心里攥着的东西 | 极快、极小（几十个） |
| CPU缓存 | 你面前的案板 | 很快、很小（几MB到几十MB） |
| **内存（RAM）** | **你的工作台面** | **快、中等大小（几GB到几十GB）** |
| 硬盘（SSD/HDD） | 厨房的冰箱和储物柜 | 慢、很大（几百GB到几TB） |

### RAM的全称

RAM = **Random Access Memory** = 随机存取存储器

"随机存取"的意思是：不管数据在内存的哪个位置，访问它所花的时间是一样的。不像磁带——你要听第5首歌得先把前面的4首快进过去。内存里每个位置都能"瞬间跳到"。

### 内存的核心特点

1. **速度快**：比硬盘快约1000倍（SSD）到100000倍（HDD）
2. **易失性**：断电就没了。关电脑 = 内存清空。这就是为什么没保存的文档会丢失。
3. **贵**：同样容量，内存比硬盘贵很多
4. **容量有限**：一般电脑16GB~32GB，服务器128GB~512GB或更多

### 为什么需要内存

假设没有内存，所有数据都从硬盘读写：

```
CPU: "我要执行下一条指令"  →  等硬盘 → 10毫秒后拿到指令
CPU: "我要读取变量x的值"    →  等硬盘 → 10毫秒后拿到数据
CPU: "我要存储结果y"        →  等硬盘 → 10毫秒后存完
```

CPU每次干活都要等硬盘10毫秒——就像你每次要写一个字都得跑去图书馆翻书。整个系统会慢到不能用。

而有了内存，等内存只需要约100纳秒——快了10万倍。

### 内存不是什么

- **内存不是硬盘**。很多人说"我电脑内存不够了"其实是想说"硬盘满了"。内存是运行程序的临时空间，硬盘是存放文件的永久空间。
- **内存不是显存**。显卡有自己的专用内存（VRAM），专门存图像数据。

## 2.2 内存地址

### 内存像一个巨大的格子仓库

把内存想象成一座超级大的仓库，里面有一排一排整齐的**格子**。每个格子能存1个字节（8个bit，也就是8个0或1）。

```
地址:  0x0000  0x0001  0x0002  0x0003  0x0004  ...
      ┌───────┬───────┬───────┬───────┬───────┬─────
      │  42   │   7   │  255  │   0   │  100  │  ...
      └───────┴───────┴───────┴───────┴───────┴─────
```

每个格子有一个**唯一的编号**，这就是**内存地址**。CPU通过这个编号来找数据——"请给我0x0003号格子里存的东西"。

### 为什么地址是十六进制

你经常会看到类似 `0x7fff5fbff8c0` 这样的地址。为什么不用十进制？

因为计算机用二进制，而十六进制和二进制的转换非常自然：
- 十六进制的1位 = 二进制的4位
- `0xF` = `1111`，`0xA` = `1010`
- `0xFA` = `1111 1010`（一目了然）

如果用十进制就做不到这么直观的转换。

### 不同类型占几个格子

不同类型的变量占用的内存格子数量不同（在64位系统上）：

| Go类型 | 占用字节数 | 占几个格子 |
|--------|-----------|-----------|
| `bool` | 1 | 1个 |
| `byte` / `uint8` | 1 | 1个 |
| `int16` | 2 | 2个 |
| `int32` / `rune` / `float32` | 4 | 4个 |
| `int64` / `float64` | 8 | 8个 |
| `int`（64位系统） | 8 | 8个 |
| `string`（描述符） | 16 | 16个（指针8字节 + 长度8字节） |
| 指针 `*T` | 8 | 8个 |
| 切片 `[]T`（描述符） | 24 | 24个（指针 + 长度 + 容量） |

所以当你写 `var x int64 = 42` 时，Go做的事情是：
1. 在内存里找8个连续的空格子
2. 把数字42的二进制存进这8个格子里
3. 记住这8个格子起始地址是多少

### Go里的指针本质上就是存地址

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
x := 42          // x存在某个地址，比如 0xc000012090
p := &x          // p存的就是 0xc000012090 这个数字
fmt.Println(p)   // 输出: 0xc000012090
```

指针（pointer）不是什么神秘东西——就是一个存着"内存地址"的变量。后面第28章会深入讲指针。

## 2.3 栈（Stack）和堆（Heap）

这是后端工程师面试必考的知识点，也是写出高效Go代码的基础。

### 栈：便签纸（自动管理、快、小）

栈就像一叠**便签纸**。每次调用一个函数，就在便签纸上写这个函数需要的变量（局部变量），然后把便签纸贴在桌子一角。函数返回了，直接把这张便签纸撕掉——干净利落。

**特点：**
- LIFO（Last In First Out，后进先出）：最后贴的便签纸最先撕
- 自动管理：函数返回，对应的栈空间自动释放
- 速度极快：只需要移动一个"栈顶指针"
- 空间有限：每个Goroutine默认只有2KB起步（Goroutine是Go语言的轻量级线程，第34章会详细讲）

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
func foo() {
    a := 10     // 'a'存在栈上
    b := 20     // 'b'存在栈上
    bar()       // 调用bar，bar的变量压在a和b上面
    // bar返回了，bar的栈空间自动释放
} // foo返回了，a和b自动释放
```

用图来看：

```
foo被调用时：
┌──────────────┐ ← 栈顶
│ bar的局部变量  │ （bar还在运行）
├──────────────┤
│ b = 20       │
│ a = 10       │
├──────────────┤
│ 返回地址等    │
└──────────────┘ ← 栈底

bar返回后：
┌──────────────┐ ← 栈顶
│ b = 20       │
│ a = 10       │
├──────────────┤
│ 返回地址等    │
└──────────────┘ ← 栈底
```

### 堆：储物间（手动/GC管理、慢、大）

堆（动态分配的公共内存区，详见附录I）就像一个大**储物间**。你可以在储物间里申请一块空间放东西，这块空间不会因为函数返回就自动消失。你需要**手动**说"这东西我不需要了"（或者等GC帮你清理）。

**特点：**
- 没有固定顺序：哪里有空位放哪里
- 需要GC（垃圾回收）：Go会自动回收不再使用的堆内存
- 速度比栈慢：分配和回收都有开销
- 空间大：受限于物理内存大小

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
func createUser() *User {
    u := User{Name: "张三"}  // u可能逃逸到堆上
    return &u                 // 因为返回了指针，u不能在函数返回后销毁
}
```

这个例子中，`u` 不能放在栈上——因为函数返回后栈就被销毁了，但外面还拿着 `u` 的指针。所以Go编译器会把 `u` 分配到堆上。

### Go的逃逸分析

Go编译器会自动判断一个变量应该放栈上还是堆上。这个过程叫**逃逸分析**。

```bash
go build -gcflags="-m" main.go
```

你会看到类似这样的输出：
```
./main.go:5:2: moved to heap: u
```
这说明编译器判断 `u` 逃逸到了堆上。

**什么情况下变量会逃逸到堆？**
- 函数返回了变量的指针
- 变量太大，栈放不下
- 变量的大小编译时不确定
- 闭包里引用了外部变量（闭包是函数引用外部变量的机制，第27章会详细讲）

**栈 vs 堆对比：**

| 特性 | 栈（Stack） | 堆（Heap） |
|------|------------|-----------|
| 分配速度 | 极快（移动指针） | 较慢（找空闲块） |
| 回收方式 | 自动（函数返回） | GC回收 |
| 大小限制 | 小（KB~MB级别） | 大（GB级别） |
| 数据生命周期 | 跟随函数调用 | 不受函数调用限制 |
| 访问速度 | 快（缓存友好）  | 较慢（可能分散） |
| 并发安全 | 天然安全（每个Goroutine有自己的栈）（Goroutine是Go语言的轻量级线程，第34章会详细讲） | 需要同步 |

## 2.4 内存泄漏是什么

### 一个比喻

你去储物间申请了一个架子放东西。后来你不用这个架子了，但你没跟管理员说"这个架子可以给别人用了"。架子就一直被你占着，但没人能用——这就是**内存泄漏**。

### 技术定义

**内存泄漏** = 程序申请了内存，但不再使用了，却没有释放。

### Go不是有GC吗？为什么还会泄露？

Go的GC（垃圾回收）会自动回收**没有任何引用指向**的内存。但在以下情况下，内存仍然会泄露：

#### 情况一：Map越来越大

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
var cache = make(map[string]*BigObject)

func handler(w http.ResponseWriter, r *http.Request) {
    key := r.URL.Query().Get("key")
    cache[key] = loadBigObject()  // 只加不删！
    // cache永远增长，内存永远增加
}
```

解决方法：用LRU缓存（有淘汰策略的缓存）而不是无限增长的Map。

#### 情况二：Goroutine阻塞收不到退出信号

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
func leakyFunc() {
    ch := make(chan int)
    go func() {
        val := <-ch  // 永远没人往ch发数据！
        fmt.Println(val)
    }()
    // ch和goroutine都无法被回收
}
```

解决方法：用context控制Goroutine的生命周期，或在合适的时机关闭channel。

#### 情况三：没有关闭HTTP Response Body

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
resp, _ := http.Get("https://example.com")
// 忘了 resp.Body.Close()！
```

响应体的底层连接和缓冲区不会自动回收。

#### 情况四：time.Ticker没有停止

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
ticker := time.NewTicker(time.Second)
// 忘了 ticker.Stop()！
// ticker会一直触发，关联的Goroutine和内存不会回收
```

### 如何检测内存泄漏

Go提供了 `pprof` 工具来分析内存使用：

（下面的Go代码你现在看看就行，第15章开始才正式学Go）
```go
import _ "net/http/pprof"

func main() {
    go func() {
        http.ListenAndServe("localhost:6060", nil)
    }()
    // 你的代码...
}
```

然后运行：
```bash
go tool pprof http://localhost:6060/debug/pprof/heap
```

可以生成内存使用的详细报告，找出哪些地方分配了最多内存。

### 内存泄漏的危害

- 程序占的内存越来越多
- GC越来越频繁（要回收的空间变大）→ CPU占用升高
- 最终可能OOM（Out Of Memory）：系统没有可用内存了，进程被杀
- 在服务器上，这可能导致服务崩溃，大量用户受影响

## 本章小结

| 概念 | 一句话总结 |
|------|-----------|
| 内存（RAM） | 计算机的临时工作台，速度快但断电消失 |
| 内存地址 | 每个字节格子的唯一编号 |
| 栈（Stack） | 便签纸，函数调用的局部变量，自动管理，快但小 |
| 堆（Heap） | 储物间，动态分配的数据，GC管理，慢但大 |
| 逃逸分析 | Go编译器自动判断变量放栈还是堆 |
| 内存泄漏 | 申请了内存但忘了还，最终程序会崩 |
