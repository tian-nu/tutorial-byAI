# 第45章 · 哈希表深入

> "你每天写 `m['key'] = value`，几行代码就搞定了数据存取。但在这看似简单的操作背后，隐藏着计算机科学中最精巧的发明之一——哈希表。用一个近似的比喻：哈希表就像你有1000本不同类别的书放在一个图书馆里，你想要想精确找到某一本，必须预先给每本书分配好"楼层-书架-格子号"，然后拿着这个编号直奔那个位置——这就是O(1)查找的秘密。但如果两本书不幸被分配到了同一个位置呢？这就是本章要揭开的谜底。"

---

## 45.1 哈希函数：把任意东西变成数字

**哈希函数的核心任务：把一个任意大小的输入，映射到一个固定大小的输出。**

```
输入 "hello"    → SHA256 → 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
输入 "Go语言"   → SHA256 → 3f078b59db3ca1443797f3ad1ad55ce4dcbac00eff42a5b90f3f610401dd4856
输入 12345      → 某个哈希 → a7ffc6f8bf1ed76651c14756a061d662f580ff4de43b49fa82d80a4b80f8434a
```

不管输入是1个字符还是1GB的电影文件，SHA256永远输出256位（32字节）的哈希值。在Go中：

```go
import "crypto/sha256"

h := sha256.Sum256([]byte("hello"))
fmt.Printf("%x\n", h)
// 输出: 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
```

### 哈希函数的三个关键性质

**1. 确定性：** 同样的输入，永远产生同样的输出。

**2. 均匀分布：** 不同的输入应该均匀地分布在所有可能的输出值上。理想情况下，任何一个输出值的出现概率都一样——这就是判断哈希函数好坏的黄金标准。

**3. 冲突不可避免：** 输入空间无限大，输出空间有限（比如256位），由鸽巢原理可知——必然有不同的输入产生相同的哈希值。这就像用10个箱子来装1000个球，必有箱子装多个球。关键不是"有没有冲突"，而是"冲突之后怎么办"。

### Go Map的哈希函数

Go的map在编译时根据key类型选择不同的哈希算法：

```go
m := make(map[string]int)
m["hello"] = 42
```

对于字符串key，Go内部会计算字符串的哈希值，然后对桶数量取模，决定这个key存在哪个桶里。

```go
// 简化版
hash := hashFunction("hello")
bucketIndex := hash % numberOfBuckets
```

**比喻**：哈希函数就像一个超强的图书管理员。你给它书名"三体"，它就能告诉你这本书在"3楼-B区-第5排-第12格"。下次你再问"三体在哪？"，它直接跟你说同上位置——O(1)。但如果"三体"和"流浪地球"不幸都被分到了同一个位置——这就叫哈希冲突。

## 45.2 哈希冲突解决方案

哈希冲突不可避免。两大流派解决这个问题：

### 方案一：链地址法（Go采用）

每个桶不是一个单独的位置，而是一个链表的头。冲突的元素挂在同一条链上。

```
桶0: [k1,v1] → [k5,v5] → nil     （k1和k5冲突了）
桶1: [k2,v2] → nil
桶2: [k3,v3] → [k6,v6] → [k7,v7] → nil
桶3: [k4,v4] → nil
```

查找流程：算哈希→找到桶→遍历桶里的链表找key。

**优点**：实现简单，删除方便。

**缺点**：如果冲突严重（链表太长），查找会退化成O(n)。Go通过扩容来解决这个问题——当链表长度超过阈值时，把桶数量翻倍，重新分配。

### 方案二：开放寻址法

每个桶只存一个键值对。如果目标桶已经被占，就按某种规则找下一个空桶。

```
想放k1到桶2 → 桶2已被占 → 看桶3 → 桶3空 → 放入桶3
想找k1 → 算哈希得桶2 → 桶2不是k1 → 看桶3 → 桶3是k1 → 找到了
```

查找规则常见的有三种：
- **线性探测**：冲突后依次找下一个（2→3→4→5...）
- **二次探测**：冲突后跳跃着找（2→3→6→11...二次方的间隔）
- **双重哈希**：冲突后用第二个哈希函数算步长

**优点**：数据都在连续内存中，缓存友好。

**缺点**：删除麻烦（不能直接删，要设"墓碑"标记），装填因子（已用桶/总桶）不能太高。

### Go为什么选链地址法？

Go 1.x的map底层采用了链地址法的变体——每个桶不是存一个链表指针，而是可以存8个键值对。当8个位置都满了，才挂溢出桶（overflow bucket）。这种设计兼顾了内存效率（大部分桶不需要溢出）和查找效率（一个桶内8个元素可以一次性加载到CPU缓存）。

## 45.3 Go Map源码分析

打开Go源码中的 `runtime/map.go`，我们来看map的核心结构。

### hmap：map的"头"

```go
type hmap struct {
    count     int    // map中元素个数（len(map)）
    flags     uint8  // 状态标记
    B         uint8  // 桶数量的对数（桶数量 = 2^B）
    noverflow uint16 // 溢出桶的大概数量
    hash0     uint32 // 哈希种子（随机数，防止哈希碰撞攻击）
    buckets    unsafe.Pointer // 指向桶数组的指针
    oldbuckets unsafe.Pointer // 扩容时指向旧桶数组
    nevacuate  uintptr        // 扩容进度（已迁移的桶数量）
    extra *mapextra           // 溢出桶相关信息
}
```

**关键字段解读**：

- `B`：桶数量的以2为底的对数。B=4时桶数量为16，B=10时桶数量为1024。
- `hash0`：随机种子——每个map在创建时生成一个随机种子。这样同一个key在不同map实例中的哈希位置不同，防止攻击者构造大量哈希冲突让map性能退化（哈希碰撞攻击）。
- `oldbuckets`：渐进式扩容的关键——扩容不是一次性完成的，而是边用边迁移。

### bmap：一个桶

```go
type bmap struct {
    tophash [8]uint8      // 每个key的哈希值高8位，用于快速比较
    keys    [8]keytype    // 8个key
    values  [8]valuetype  // 8个value
    overflow *bmap        // 溢出桶指针
}
```

注意：`bmap`的实际结构比这个复杂（keys和values不是分开的数组，而是key/value交错存储以优化内存对齐），但概念上就是这样。

**一个桶能存8个键值对。** 当第9个键值对被分配到同一个桶时，会创建一个溢出桶（overflow bucket），挂到当前桶后面。

### tophash的妙用

`tophash` 是每个key的哈希值的**高8位**。为什么要存这个？

```
查找key="hello"的流程：

1. 计算 hash("hello") → 64位哈希值
2. 低B位 决定去哪个桶
3. 高8位 就是tophash
4. 在桶的8个tophash中快速扫描：
   - tophash[i] == 0? 空位，key不存在
   - tophash[i] == hashTop8? 可能匹配，再比较完整key
   - 都不匹配? 去溢出桶继续找
```

如果没有tophash，每次查找都要比较完整的key（对于长字符串这是昂贵的操作）。有了tophash，大部分时候只需要比较1个字节就能排除不匹配的key——这是性能优化的关键。

### 两种扩容

Go的map有两种扩容触发条件：

**等量扩容（sameSizeGrow）**：不是因为桶不够，而是因为溢出桶太多了——说明很多桶的链表太长，数据分布不均匀。即使桶数量不变，也需要重新哈希打散数据。

触发条件：溢出桶数量 ≥ 桶数量（即平均每个桶至少有1个溢出桶）。

**翻倍扩容**：桶数量翻倍（B增加1）。新桶数量 = 旧桶数量 × 2。每个旧桶中的数据分裂到两个新桶中。

触发条件：装载因子（元素数/桶数量）> 6.5。

```
举例：桶数量=8, B=3
元素数量从52变成53时，装载因子=53/8=6.625 > 6.5 → 触发翻倍扩容
B变成4，桶数量变成16
旧桶3中的元素会被重新分配到新桶3和新桶11（因为多了一位决定因子）
```

### 渐进式扩容

扩容不是一次性完成的——那样如果map很大（比如100万元素），一次性迁移会让程序卡住。Go采用渐进式扩容：

- 每次对map的**写操作**（插入/删除）时，顺便迁移1~2个旧桶到新桶。
- `nevacuate` 字段追踪已经迁移到第几个桶。
- `oldbuckets` 不为nil时表示正在扩容中。
- 读操作时，先查新桶再查旧桶，确保不丢数据。

这种"蚂蚁搬家"式的设计，把一次巨大的操作拆成了无数次微小的操作——用户感知不到卡顿。

```
扩容过程中：
┌──────────────┐     ┌──────────────────────────┐
│ 旧桶(oldbuckets) │     │ 新桶(buckets)          │
│ [0][1][2][3]  │ →   │ [0][1][2][3][4][5][6][7] │
│    ↑neva=1    │     │                          │
│ 已迁移:桶0    │     │ 已就位:桶0的数据         │
└──────────────┘     └──────────────────────────┘
```

## 🤔 想多一点：Map的遍历顺序为什么是"随机"的？

你注意过吗——Go的map遍历顺序是不固定的。连续两次 `for k, v := range m` 可能产生不同的顺序。

这**不是bug，是feature**。Go团队刻意在遍历时引入随机性——每次遍历从随机的一个桶开始，桶内也从随机的一个位置开始。目的是**防止程序员写出依赖遍历顺序的代码**。

```go
m := map[int]string{1: "a", 2: "b", 3: "c", 4: "d"}
for k, v := range m {
    fmt.Println(k, v)
}
// 你可能看到：2 b, 4 d, 1 a, 3 c
// 再跑一次：  3 c, 1 a, 4 d, 2 b
// 再跑一次：  4 d, 1 a, 2 b, 3 c
```

如果你需要有序遍历，**自己排序key**：

```go
keys := make([]int, 0, len(m))
for k := range m {
    keys = append(keys, k)
}
sort.Ints(keys)
for _, k := range keys {
    fmt.Println(k, m[k])
}
```

这个设计哲学贯穿了Go的很多设计决策：**明确胜过隐晦，让潜在的bug在第一时间暴露，而不是潜伏到生产环境。**

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 哈希函数 | 任意输入→固定输出，确定性、均匀分布、冲突不可避免 |

| 等量扩容 | 溢出桶过多 → 重新哈希打散 |
| 渐进式扩容 | 蚂蚁搬家，每次写操作迁移1~2桶，不卡顿 |
| 遍历随机 | Go刻意随机化遍历起点，防止依赖顺序 |
