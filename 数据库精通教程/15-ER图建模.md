## 我在做什么？

第 14 章你学会了判断表设计是否合理。但那个问题是"事后诸葛亮"——你拿到一张已经建好的表，检查它是否符合范式。真正让你头疼的是另一个问题：**拿到一份需求文档，怎么从零开始设计表结构？**

比如产品经理说："我们要做一个电商系统，有用户、商品、订单、购物车……"然后你面对空白的 SQL 编辑器，无从下手。

这一章教你**从需求到表结构的完整建模流程**。核心工具是 **ER 图（Entity-Relationship Diagram，实体关系图）** *此术语见附录E*——它是数据库设计的"建筑蓝图"，在写任何 SQL 之前，先把表结构画出来。

学完这一章，你能：
- 从需求描述中提取实体、属性、关系
- 画出标准的 Crow's Foot（乌鸦脚）ER 图
- 将 ER 图转化为完整的建表 SQL
- 独立完成一个"博客系统"的数据库建模

---

## 一、建模流程全景

先看这张"路线图"，了解整个建模流程的六步：

```
需求描述 → 实体识别 → 属性识别 → 关系识别 → ER 图绘制 → 转表结构 → 范式检查
  (文字)    (名词)     (形容词)    (动词)     (可视化)    (SQL)     (第14章)
```

每一步对应一个具体操作：
1. **读需求**：把产品文档或口头描述写下来
2. **找实体**：圈出所有名词，判断哪些是"需要独立存数据的东西"
3. **定属性**：每个实体有什么特征？（实体 = 名词，属性 = 形容词）
4. **定关系**：实体之间怎么关联？（关系 = 动词）
5. **画 ER 图**：用 Crow's Foot 表示法画出可视化模型
6. **转 SQL**：把 ER 图翻译成 `CREATE TABLE` 语句
7. **范式检查**：用第 14 章的 1NF/2NF/3NF 标准逐张表审查

> **想多一点**：很多人跳过 ER 图直接写 SQL，觉得"画图太慢了"。但这不是慢，是偷懒。ER 图的价值在于：在纸上改一个箭头，比在生产环境 `ALTER TABLE` 修改外键关系要便宜一万倍。**花 30 分钟画 ER 图，可能省掉后续 30 小时的数据库重构。**

## 二、Crow's Foot 表示法速成

在开始画图之前，先统一"语言"——我们用什么符号画 ER 图？

本教程使用 **Crow's Foot（乌鸦脚）** *此术语见附录E* 表示法，因为它直观、流行，draw.io 和 dbdiagram.io 都原生支持。

### 基本符号

```
实体：用矩形框表示，框内写上实体名（通常用名词复数）

    ┌──────────────┐
    │    users     │
    └──────────────┘

关系线：用线段连接两个实体，端点表示数量关系

    |  表示"一"（one）
    }  表示"多"（many）—— 像乌鸦的脚，所以叫 Crow's Foot
    |o 表示"零或一"（zero or one）
    }o 表示"零或多"（zero or many）
```

### 三种基础关系

```
1:1 关系（一对一）
    ┌──────────┐         ┌──────────┐
    │  users   │|───────||  id_card │
    └──────────┘         └──────────┘
    一个用户 ←→ 一张身份证

1:N 关系（一对多）
    ┌──────────┐         ┌──────────┐
    │  users   │|───────}│  orders  │
    └──────────┘         └──────────┘
    一个用户 ←→ 多张订单

M:N 关系（多对多）
    ┌──────────┐         ┌──────────┐
    │  orders  │}───────}│ products │
    └──────────┘         └──────────┘
    一张订单 ←→ 多个商品，一个商品 ←→ 多张订单
```

### 如何判断关系？

从需求描述中找**动词**：
- "用户**拥有**订单" → 动词是"拥有" → 一个用户拥有多张订单？是 → 1:N
- "订单**包含**商品" → 动词是"包含" → 一张订单包含多个商品，一个商品出现在多张订单？是 → M:N
- "用户**绑定**身份证" → 动词是"绑定" → 一个用户绑定几张身份证？通常一张 → 1:1

## 三、实战：电商系统 ER 建模

### 3.1 需求描述

```
我们要做一个电商系统，功能如下：

1. 用户可以注册（用户名、邮箱、密码、手机号、注册时间）
2. 每个用户可以有多个收货地址（省、市、区、详细地址、收件人、电话）
3. 商品有分类，每个分类下有多个商品（商品名、描述、价格、库存、上架时间）
4. 用户可以把商品加入购物车（每个用户一个购物车，购物车里有多个商品和数量）
5. 用户可以下单，一张订单包含多个商品（每个商品的数量和单价）
6. 订单有状态（待支付、已支付、已发货、已完成、已取消）
7. 订单关联一个收货地址，记录下单时间和总金额
```

### 3.2 第一步：实体识别（找名词）

把需求中的关键名词圈出来：

- **用户**（User）—— 独立存在，需要存数据
- **收货地址**（Address）—— 属于用户，但自己也有独立属性
- **商品**（Product）—— 独立存在
- **分类**（Category）—— 独立存在，有名称
- **购物车**（Cart）—— 需要存"用户放了什么商品"
- **购物车项**（CartItem）—— 购物车里的每一条记录
- **订单**（Order）—— 独立存在
- **订单项**（OrderItem）—— 订单里的每一条记录
- 密码 —— 不是实体，是用户的属性
- 状态 —— 不是实体，是订单的属性
- 价格 —— 不是实体，是商品/订单项的属性
- 时间 —— 不是实体，是各实体的属性

**识别技巧**：实体 = 可以独立存在、有自己生命周期、需要单独存一张表的东西。属性 = 依附于实体存在的描述信息。

### 3.3 第二步：属性识别（找形容词/描述词）

给每个实体列出属性：

```
用户（User）
  - id            (主键)
  - username      (用户名)
  - email         (邮箱)
  - password_hash (密码哈希)
  - phone         (手机号)
  - created_at    (注册时间)

收货地址（Address）
  - id            (主键)
  - user_id       (外键 → 用户)
  - province      (省)
  - city          (市)
  - district      (区)
  - detail        (详细地址)
  - receiver_name (收件人)
  - receiver_phone(收件人电话)
  - is_default    (是否默认地址)

分类（Category）
  - id            (主键)
  - name          (分类名)
  - parent_id     (父分类ID，支持多级分类)

商品（Product）
  - id            (主键)
  - category_id   (外键 → 分类)
  - name          (商品名)
  - description   (描述)
  - price         (价格)
  - stock         (库存)
  - created_at    (上架时间)

购物车项（CartItem）
  - id            (主键)
  - user_id       (外键 → 用户)
  - product_id    (外键 → 商品)
  - quantity      (数量)

订单（Order）
  - id            (主键)
  - user_id       (外键 → 用户)
  - address_id    (外键 → 收货地址)
  - status        (状态：pending/paid/shipped/completed/cancelled)
  - total_amount  (总金额)
  - created_at    (下单时间)

订单项（OrderItem）
  - id            (主键)
  - order_id      (外键 → 订单)
  - product_id    (外键 → 商品)
  - quantity      (数量)
  - unit_price    (下单时的单价，快照！)
```

> **想多一点**：为什么 `OrderItem` 里要存 `unit_price`？商品表里不已经有 `price` 了吗？因为你下单后，商品价格可能调价——如果 `OrderItem` 不存快照，半年后查看历史订单，价格就变成当前价格了，财务数据全乱套。**订单明细必须存快照，这是电商系统的铁律。** 回顾第 14 章：这不算违反 3NF，因为 `unit_price` 直接依赖于 `OrderItem` 的主键，不经过 `product_id` 传递。`product_id → price` 是"当前价格"，`OrderItem.unit_price` 是"历史价格"，它们是两个不同的概念。

### 3.4 第三步：关系识别（找动词）

| 需求描述 | 动词 | 实体A → 实体B | 关系类型 |
|---------|------|-------------|---------|
| 用户有多个收货地址 | 有 | User → Address | 1:N |
| 分类下有多个商品 | 有 | Category → Product | 1:N |
| 用户把商品加入购物车 | 加入 | User → CartItem | 1:N |
| 购物车项关联商品 | 关联 | CartItem → Product | N:1 |
| 用户下单 | 下 | User → Order | 1:N |
| 订单包含多个商品 | 包含 | Order → OrderItem | 1:N |
| 订单项关联商品 | 关联 | OrderItem → Product | N:1 |
| 订单关联收货地址 | 关联 | Order → Address | N:1 |

### 3.5 第四步：画 ER 图

用 ASCII 画出 Crow's Foot 风格的 ER 图：

```
┌──────────────────────┐       ┌──────────────────────┐
│       Category       │       │       Product        │
├──────────────────────┤       ├──────────────────────┤
│ PK  id               │       │ PK  id               │
│     name             │|─────}│ FK  category_id      │
│     parent_id        │       │     name             │
└──────────────────────┘       │     description      │
                               │     price            │
                               │     stock            │
                               │     created_at       │
                               └──────────┬───────────┘
                                          │
                                          │}
                                          │
┌──────────────────────┐       ┌──────────┴───────────┐
│        User          │       │      CartItem        │
├──────────────────────┤       ├──────────────────────┤
│ PK  id               │|─────}│ PK  id               │
│     username         │       │ FK  user_id          │
│     email            │       │ FK  product_id       │
│     password_hash    │       │     quantity         │
│     phone            │       └──────────────────────┘
│     created_at       │
└──┬──────────┬────────┘
   │          │
   │}         │}
   │          │
   │  ┌───────┴───────────┐
   │  │     Address       │
   │  ├───────────────────┤
   │  │ PK  id            │
   │  │ FK  user_id       │
   │  │     province      │
   │  │     city          │
   │  │     district      │
   │  │     detail        │
   │  │     receiver_name │
   │  │     receiver_phone │
   │  │     is_default    │
   │  └───────────────────┘
   │
   │}
   │
   │  ┌──────────────────────┐       ┌──────────────────────┐
   │  │       Order          │       │      OrderItem       │
   │  ├──────────────────────┤       ├──────────────────────┤
   └──│ FK  user_id          │|─────}│ FK  order_id         │
      │ PK  id               │       │ PK  id               │
      │ FK  address_id ──────│──┐    │ FK  product_id       │
      │     status           │  │    │     quantity          │
      │     total_amount     │  │    │     unit_price        │
      │     created_at       │  │    └──────────────────────┘
      └──────────────────────┘  │
                                │    (N:1 from Order to Address)
                                │
                                └──── Address 表
```

**符号解读**：
- `|` 在起点 = "最多一个"（一）
- `}` 在终点 = "多个"（多）
- `|─────}` = 一对多（起点一，终点多）
- `}|─────|` = 多对一（起点多，终点一）

> **画图工具推荐**：
> - **draw.io**（免费离线）：新建 → 选择 "Entity Relationship Diagram" → 左侧选 "Crow's Foot" 模板
> - **dbdiagram.io**（在线）：用 DSL 语法描述，自动生成 ER 图，还能直接导出 SQL
> - 初期用纸笔就够了——不追求工具，追求理解

### 3.6 第五步：ER 图转 SQL

把上面的 ER 图翻译成完整的建表语句：

```sql
-- 创建电商数据库
CREATE DATABASE e_shop
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;

USE e_shop;

-- 1. 分类表
CREATE TABLE categories (
    id        INT PRIMARY KEY AUTO_INCREMENT,
    name      VARCHAR(50) NOT NULL COMMENT '分类名',
    parent_id INT DEFAULT NULL COMMENT '父分类ID，NULL表示顶级分类',
    FOREIGN KEY (parent_id) REFERENCES categories(id)
        ON DELETE SET NULL
) COMMENT='商品分类表';

-- 2. 商品表
CREATE TABLE products (
    id          INT PRIMARY KEY AUTO_INCREMENT,
    category_id INT NOT NULL COMMENT '所属分类',
    name        VARCHAR(200) NOT NULL COMMENT '商品名',
    description TEXT COMMENT '商品描述',
    price       DECIMAL(10,2) NOT NULL COMMENT '价格',
    stock       INT NOT NULL DEFAULT 0 COMMENT '库存',
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '上架时间',
    FOREIGN KEY (category_id) REFERENCES categories(id)
        ON DELETE RESTRICT
) COMMENT='商品表';

-- 3. 用户表
CREATE TABLE users (
    id            INT PRIMARY KEY AUTO_INCREMENT,
    username      VARCHAR(50) NOT NULL COMMENT '用户名',
    email         VARCHAR(100) NOT NULL COMMENT '邮箱',
    password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希值',
    phone         VARCHAR(20) COMMENT '手机号',
    created_at    DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '注册时间',
    UNIQUE KEY uk_username (username),
    UNIQUE KEY uk_email (email)
) COMMENT='用户表';

-- 4. 收货地址表
CREATE TABLE addresses (
    id             INT PRIMARY KEY AUTO_INCREMENT,
    user_id        INT NOT NULL COMMENT '所属用户',
    province       VARCHAR(20) NOT NULL COMMENT '省',
    city           VARCHAR(20) NOT NULL COMMENT '市',
    district       VARCHAR(30) NOT NULL COMMENT '区',
    detail         VARCHAR(200) NOT NULL COMMENT '详细地址',
    receiver_name  VARCHAR(50) NOT NULL COMMENT '收件人',
    receiver_phone VARCHAR(20) NOT NULL COMMENT '收件人电话',
    is_default     TINYINT(1) DEFAULT 0 COMMENT '是否默认地址：0否，1是',
    FOREIGN KEY (user_id) REFERENCES users(id)
        ON DELETE CASCADE
) COMMENT='收货地址表';

-- 5. 购物车项表
CREATE TABLE cart_items (
    id         INT PRIMARY KEY AUTO_INCREMENT,
    user_id    INT NOT NULL COMMENT '用户',
    product_id INT NOT NULL COMMENT '商品',
    quantity   INT NOT NULL DEFAULT 1 COMMENT '数量',
    FOREIGN KEY (user_id) REFERENCES users(id)
        ON DELETE CASCADE,
    FOREIGN KEY (product_id) REFERENCES products(id)
        ON DELETE CASCADE,
    UNIQUE KEY uk_user_product (user_id, product_id)
) COMMENT='购物车表';

-- 6. 订单表
CREATE TABLE orders (
    id           INT PRIMARY KEY AUTO_INCREMENT,
    user_id      INT NOT NULL COMMENT '下单用户',
    address_id   INT NOT NULL COMMENT '收货地址',
    status       ENUM('pending', 'paid', 'shipped', 'completed', 'cancelled')
                 DEFAULT 'pending' COMMENT '订单状态',
    total_amount DECIMAL(10,2) NOT NULL COMMENT '总金额',
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '下单时间',
    FOREIGN KEY (user_id) REFERENCES users(id)
        ON DELETE RESTRICT,
    FOREIGN KEY (address_id) REFERENCES addresses(id)
        ON DELETE RESTRICT
) COMMENT='订单表';

-- 7. 订单项表
CREATE TABLE order_items (
    id         INT PRIMARY KEY AUTO_INCREMENT,
    order_id   INT NOT NULL COMMENT '所属订单',
    product_id INT NOT NULL COMMENT '商品',
    quantity   INT NOT NULL COMMENT '数量',
    unit_price DECIMAL(10,2) NOT NULL COMMENT '下单时单价（快照）',
    FOREIGN KEY (order_id) REFERENCES orders(id)
        ON DELETE CASCADE,
    FOREIGN KEY (product_id) REFERENCES products(id)
        ON DELETE RESTRICT
) COMMENT='订单明细表';
```

### 3.7 第六步：范式检查

用第 14 章的标准逐张表审查：

| 表 | 1NF | 2NF | 3NF | 说明 |
|---|:---:|:---:|:---:|------|
| categories | ✅ | ✅ | ✅ | 单主键，无传递依赖 |
| products | ✅ | ✅ | ✅ | `category_id` 是外键，不是传递依赖（`price` 直接依赖 `id`） |
| users | ✅ | ✅ | ✅ | 单主键 |
| addresses | ✅ | ✅ | ✅ | `is_default` 依赖 `(user_id, id)`，但外键不构成传递依赖 |
| cart_items | ✅ | ✅ | ✅ | `quantity` 完全依赖联合主键 `(user_id, product_id)` |
| orders | ✅ | ✅ | ✅ | `status` 和 `total_amount` 都直接依赖 `id` |
| order_items | ✅ | ✅ | ✅ | `unit_price` 是快照而非传递依赖，见 3.3 节"想多一点" |

**全部通过！** 🎉

## 四、M:N 关系转换：中间表

M:N 关系在 ER 图中很常见，但转表时需要特殊处理：**必须建中间表**。

### 示例：订单 ↔ 商品（M:N）

```
┌──────────┐         ┌──────────┐
│  orders  │}───────}│ products │
└──────────┘         └──────────┘
```

直接转 SQL 时，你不能在 `orders` 表里加 `product_id`（因为一个订单有多个商品），也不能在 `products` 表里加 `order_id`（因为一个商品出现在多个订单中）。**M:N 必须拆成两个 1:N + 一张中间表**：

```
┌──────────┐     ┌──────────────┐     ┌──────────┐
│  orders  │|───}│ order_items  │}───|| products │
└──────────┘     └──────────────┘     └──────────┘
   1:N              N:1 (两次)
```

中间表 `order_items` 的联合主键 = `(order_id, product_id)`，恰好是两边主键的组合。

### 常见错误：M:N 忘记建中间表

```sql
-- ❌ 错误：在 orders 表里加 product_id
CREATE TABLE orders_wrong (
    id         INT PRIMARY KEY,
    user_id    INT,
    product_id INT,   -- 只能存一个商品！
    quantity   INT
);
-- 如果订单 1 包含 3 个商品，你只能插入 3 条订单记录，
-- 但 user_id 和 total_amount 会重复 3 次——违反 2NF！

-- ✅ 正确：建中间表
CREATE TABLE orders (
    id           INT PRIMARY KEY,
    user_id      INT,
    total_amount DECIMAL(10,2),
    -- 不存 product_id！
    FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE TABLE order_items (
    id         INT PRIMARY KEY AUTO_INCREMENT,
    order_id   INT,
    product_id INT,
    quantity   INT,
    unit_price DECIMAL(10,2),
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);
```

## 五、1:1 关系转换：外键放哪边？

1:1 关系比较少见，但也会遇到（如用户 ↔ 身份证、用户 ↔ 用户配置）。外键放哪边都可以，但通常遵循以下原则：

```sql
-- 场景：用户和身份证是 1:1
-- 方案 A：外键放身份证表（推荐——身份证"属于"用户）
CREATE TABLE id_cards (
    id      INT PRIMARY KEY,
    user_id INT UNIQUE NOT NULL,  -- UNIQUE 保证 1:1
    id_number VARCHAR(18) NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 方案 B：外键放用户表（不推荐——用户表的"主表"地位被弱化）
CREATE TABLE users (
    id         INT PRIMARY KEY,
    username   VARCHAR(50),
    id_card_id INT UNIQUE,  -- 用户表"依赖"身份证表
    FOREIGN KEY (id_card_id) REFERENCES id_cards(id)
);
```

**选择标准**：把外键放在"从属"方。身份证从属于用户 → 外键放 `id_cards`。如果两个实体是平等的（如"夫妻"），选查询更频繁的那张表放外键。

### 练习：独立完成"博客系统"ER 建模

**需求**：

3. 写出完整建表 SQL
4. 用 1NF/2NF/3NF 标准逐表检查

> **提示**：标签（Tag）和文章（Article）之间是 M:N 关系，别忘了中间表。

## 不对怎么办？

### 常见错误 1：M:N 忘记建中间表

```sql
-- ❌ 在 articles 表里加 tag_id
CREATE TABLE articles (
    id      INT PRIMARY KEY,
    title   VARCHAR(200),
    tag_id  INT  -- 只能存一个标签！
);
-- 一篇文章有多个标签怎么办？存多个 tag_id 用逗号分隔？→ 违反 1NF！

-- ✅ 正确：建中间表 article_tags
CREATE TABLE article_tags (
    article_id INT,
    tag_id     INT,
    PRIMARY KEY (article_id, tag_id),
    FOREIGN KEY (article_id) REFERENCES articles(id),
    FOREIGN KEY (tag_id) REFERENCES tags(id)
);
```

### 常见错误 2：把属性当实体

```sql
-- ❌ 订单状态单独建表
CREATE TABLE order_statuses (
    id   INT PRIMARY KEY,
    name VARCHAR(20)
);
INSERT INTO order_statuses VALUES (1, 'pending'), (2, 'paid'), ...;

-- 然后在 orders 表里加 status_id 外键
-- 这不算错，但过度设计了。状态值很少变化，用 ENUM 或 VARCHAR 即可
```

**判断标准**：如果"东西"有独立生命周期、会增删改、有多个属性 → 是实体。如果只是一个描述值，基本不变 → 是属性。

### 常见错误 3：关系方向搞反

```sql
-- ❌ 在 products 表里加 order_id 外键
CREATE TABLE products (
    id       INT PRIMARY KEY,
    order_id INT,  -- 一个商品只属于一个订单？不对！
    FOREIGN KEY (order_id) REFERENCES orders(id)
);
-- 一个商品可以出现在多个订单中，所以外键不能放在 products 表
```

**判断标准**：外键放在"多"的一方。如果关系是 1:N，外键放 N 方。如果 M:N，建中间表。

### 常见错误 4：忘记订单快照

```sql
-- ❌ order_items 不存 unit_price，只存 product_id
CREATE TABLE order_items (
    order_id   INT,
    product_id INT,
    quantity   INT,
    -- 没有 unit_price！
    PRIMARY KEY (order_id, product_id)
);
-- 查询历史订单时，JOIN products 拿到的价格是当前价格，不是下单时的价格！
```

已在 3.3 节"想多一点"中详细解释了。**永远对订单明细存快照。**

## 本章小结

| 本章学了什么 | 对应命令/概念 | 注意事项 |
|-------------|-------------|---------|
| 建模流程 | 需求 → 实体 → 属性 → 关系 → ER 图 → SQL → 范式检查 | 先画图再写 SQL，省掉后续重构成本 |
| 实体识别 | 从需求中找名词 | 实体 = 有独立生命周期、需要存数据的东西 |
| 属性识别 | 从需求中找描述词 | 属性依附于实体，不是独立存在的 |
| 关系识别 | 从需求中找动词 | 1:1（一用户一身份证）、1:N（一用户多订单）、M:N（多订单多商品） |
| Crow's Foot 表示法 | `\|` = 一，`}` = 多，`\|o` = 零或一 | 最流行的 ER 图表示法，draw.io 原生支持 |
| 1:N 转表 | 外键放 N 方 | 多的一方存外键 |
| M:N 转表 | 必须建中间表 | 中间表联合主键 = 两边主键组合 |
| 1:1 转表 | 外键放"从属"方 | 加 UNIQUE 约束确保 1:1 |
| 订单快照 | `order_items.unit_price` | 历史数据必须存快照，不能用当前价格 |
| 范式检查 | 用 1NF/2NF/3NF 逐表审查 | 在 ER 图阶段就发现问题，而不是上线后 |
