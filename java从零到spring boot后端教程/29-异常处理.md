# 第29章 · 异常处理

> "程序不可能永远正常运行。文件可能被用户删了、网络可能突然断开、数据库可能宕机。如果你不处理这些'意外'，你的程序就会崩掉——用户看到的是一堆红色的错误信息，然后卸载你的App。Java的异常处理机制让你优雅地说：'我知道可能会出错，如果出错了，我这样处理。'"

---

## 29.1 异常是什么？

**异常**（Exception。此术语需进附录）是程序运行时发生的**意外事件**，它会打断正常的执行流程。

```java
int[] arr = new int[3];
System.out.println(arr[5]);  // ArrayIndexOutOfBoundsException
```

如果没有捕获这个异常，程序会崩溃并打印**堆栈跟踪**（Stack Trace）：

```
Exception in thread "main" java.lang.ArrayIndexOutOfBoundsException:
Index 5 out of bounds for length 3
    at com.example.Main.main(Main.java:10)
```

### 异常的分级体系

```
java.lang.Throwable
    ├── java.lang.Error           ← 严重错误，不用处理（如OutOfMemoryError）
    └── java.lang.Exception       ← 可以处理的异常
        ├── RuntimeException      ← 非受检异常（unchecked）
        │   ├── NullPointerException
        │   ├── ArrayIndexOutOfBoundsException
        │   ├── ArithmeticException（如 5/0）
        │   └── IllegalArgumentException
        └── 其他Exception          ← 受检异常（checked）
            ├── IOException
            ├── SQLException
            ├── FileNotFoundException
            └── ClassNotFoundException
```

---

## 29.2 受检异常 vs 非受检异常

| 对比维度 | 受检异常（Checked） | 非受检异常（Unchecked） |
|----------|---------------------|-------------------------|
| 父类 | `Exception`（不包括RuntimeException） | `RuntimeException` |
| 编译器强制处理 | **是**——必须try-catch或throws声明 | **否**——编译器不管 |
| 何时发生 | 编译时可预见的（文件不存在、网络断开） | 编程Bug（空指针、越界） |
| 示例 | `IOException`、`SQLException`、`FileNotFoundException` | `NullPointerException`、`ArrayIndexOutOfBoundsException` |
| 哲学 | "你能预见到，你应该处理" | "这些是代码Bug，你应该修好而不是捕获" |

```java
// 受检异常——必须处理
public void readFile(String path) {
    FileReader reader = new FileReader(path);  // ❌ 编译错误！
    // 必须try-catch或者throws
}

// 非受检异常——编译通过，运行时报错
public void doSomething() {
    String s = null;
    s.length();  // 编译通过！运行时NullPointerException
}
```

## 29.3 try-catch：捕获异常

最基本的异常处理结构：

```java
try {
    // 可能出错的代码
    int result = 10 / 0;
} catch (ArithmeticException e) {
    // 捕获到异常后执行的代码
    System.out.println("除数不能为0！");
    System.out.println("异常信息：" + e.getMessage());
}
System.out.println("程序继续执行");  // ← 这一行会执行！
```

**关键点**：catch住异常后，程序**不会崩溃**，catch块后面的代码会继续执行。

### 多个catch块

```java
try {
    int[] arr = new int[3];
    int result = 10 / 0;        // 先触发这个异常
    System.out.println(arr[5]); // 永远不会执行到这里
} catch (ArithmeticException e) {
    System.out.println("算术异常：" + e.getMessage());
} catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("数组越界：" + e.getMessage());
} catch (Exception e) {
    System.out.println("其他异常：" + e.getMessage());  // 兜底
}
```

**catch顺序很重要**：子类异常必须放在父类异常前面。如果把 `catch (Exception e)` 放在最前面，后面的catch块永远不会被执行（编译报错）。

### Java 7+ 多异常合并

```java
try {
    // ...
} catch (IOException | SQLException e) {
    // 捕获这两种异常，做同样的处理
    System.out.println("IO或数据库异常：" + e.getMessage());
}
```

## 29.4 finally：不管有没有异常，我都要执行

`finally` 块里的代码**无论如何都会执行**——无论try里有异常还是正常结束，无论catch了还是没catch：

```java
try {
    System.out.println("打开数据库连接");
    int result = 10 / 0;
    System.out.println("查询数据");  // 不会执行（前面抛异常了）
} catch (ArithmeticException e) {
    System.out.println("捕获异常");
} finally {
    System.out.println("关闭数据库连接");  // 一定会执行！
}
// 输出：
// 打开数据库连接
// 捕获异常
// 关闭数据库连接
```

### finally唯一不执行的情况

1. 在try或catch里调用了 `System.exit(0)` （直接终止JVM）
2. 操作系统杀死了进程
3. 机器断电

除此之外，`finally` 一定会执行。

## 29.5 try-with-resources：自动关闭资源（Java 7+）

以前为了确保资源关闭，要写一堆finally代码：

```java
// 老式写法（Java 7之前）
BufferedReader reader = null;
try {
    reader = new BufferedReader(new FileReader("test.txt"));
    String line = reader.readLine();
    System.out.println(line);
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (reader != null) {
        try {
            reader.close();  // close也可能抛异常！
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

Java 7引入**try-with-resources**，代码量减少70%：

```java
// try-with-resources（Java 7+）
try (BufferedReader reader = new BufferedReader(new FileReader("test.txt"))) {
    String line = reader.readLine();
    System.out.println(line);
} catch (IOException e) {
    e.printStackTrace();
}
// reader自动关闭，不需要finally！
```

**要求**：资源类必须实现了 `AutoCloseable` 接口（有 `close()` 方法）。

```java
// 多个资源：用分号分隔
try (FileInputStream input = new FileInputStream("in.txt");
     FileOutputStream output = new FileOutputStream("out.txt")) {
    // 两者都会自动关闭
} catch (IOException e) {
    e.printStackTrace();
}
```

## 29.6 throw 与 throws：制造异常和声明异常

### throw：手动抛出异常

```java
public void checkAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("年龄必须在0-150之间，实际：" + age);
    }
    System.out.println("年龄有效：" + age);
}
```

`throw` 后面跟的是一个**异常对象**（用 `new` 创建）。一旦执行 `throw`，方法立即结束。

### throws：声明"我这个方法可能抛出异常"

```java
public void readFile(String path) throws IOException {
    FileReader reader = new FileReader(path);
    // ...
}
```

`throws` 写在方法签名后面，告诉调用者："调用我可能会遇到IOException，你得处理"。这是受检异常的传播机制。

```java
// 调用方必须处理
try {
    readFile("data.txt");
} catch (IOException e) {
    System.out.println("文件读取失败");
}
```

## 29.7 自定义异常：你的业务规则需要自己的异常

内置异常不够用时，自己定义一个：

```java
// 自定义受检异常
public class InsufficientBalanceException extends Exception {
    public InsufficientBalanceException(String message) {
        super(message);
    }
}

// 自定义非受检异常
public class InvalidUsernameException extends RuntimeException {
    public InvalidUsernameException(String message) {
        super(message);
    }
}

// 使用
public void withdraw(double amount) throws InsufficientBalanceException {
    if (amount > balance) {
        throw new InsufficientBalanceException(
            "余额不足！当前余额：" + balance + "，尝试取出：" + amount
        );
    }
    balance -= amount;
}
```

**命名规范**：异常类名以 `Exception` 结尾。

## 29.8 异常处理最佳实践

| 原则 | 说明 | 反例 |
|------|------|------|
| **不要吞异常** | 不要让catch块为空 | `catch(Exception e) { }` ← 最差写法 |
| **精确捕获** | 捕获具体的异常类型，不要一律 `catch(Exception)` | `catch(Exception e)` 笼统 |
| **尽早抛出，延迟捕获** | 底层代码抛出具体异常，高层统一处理和记录日志 | 底层吞异常然后返回null |
| **异常信息要具体** | 包含关键参数值 | `throw new RuntimeException("错误")` |
| **不要用异常控制流程** | 异常是意外情况，不是if-else的替代 | 用try-catch代替if判断null |

### ❌ 错误示范

```java
// 错误一：吞异常
try {
    doSomething();
} catch (Exception e) {
    // 什么都不做——错误被悄然吞没！
}

// 错误二：空泛捕获
try {
    doSomething();
} catch (Exception e) {
    e.printStackTrace();  // 生产环境不应该这样
}

// 错误三：用异常控制流程
try {
    Integer.parseInt(str);
} catch (NumberFormatException e) {
    num = 0;  // 如果经常发生，应该先判断str是否为数字
}
```

### ✅ 正确示范

```java
// 正确：记录日志+返回有意义的信息
try {
    doSomething();
} catch (IOException e) {
    log.error("读取文件失败，路径：{}", path, e);  // 记录完整堆栈
    throw new ServiceException("文件服务暂不可用，请稍后重试", e);
}

// 正确：判断代替异常
if (str != null && str.matches("\\d+")) {
    num = Integer.parseInt(str);
} else {
    num = 0;
}
```

## 本章小结

| 知识点 | 核心要点 |
|--------|----------|
| 异常体系 | Throwable → Error（严重，不处理）+ Exception（可处理） |
| 受检异常 | 继承Exception（非RuntimeException）。编译器强制处理 |
| 非受检异常 | 继承RuntimeException。编程Bug，编译器不强制 |
| try-catch | 捕获异常。多个catch时子类在前 |
| finally | 无论如何都执行（除非System.exit）。用于释放资源 |
| try-with-resources | Java 7+。实现AutoCloseable的资源自动关闭 |
| throw | 手动抛出异常对象 |
| throws | 方法签名声明可能抛出的受检异常 |
| 自定义异常 | 继承Exception或RuntimeException。命名以Exception结尾 |
| 最佳实践 | 不要吞异常、精确捕获、尽早抛延迟捕、不要用异常控制流程 |
