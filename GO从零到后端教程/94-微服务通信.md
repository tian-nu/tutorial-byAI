# 第94章 · 微服务通信

> "服务拆完了，但服务不能各自为战——用户服务需要调订单服务查用户的订单列表，订单服务需要调支付服务发起支付。服务之间怎么说话？本章介绍三种主流方式：REST（最简单）、gRPC（Google开源的高性能远程过程调用框架，详见附录I）、消息队列（最解耦），并给出选择决策树。"

## 94.1 REST

### 是什么

REST（Representational State Transfer）是用HTTP协议在服务之间传递JSON/XML数据。你已经在第12章学过了——和浏览器调后端API一模一样。

```
order-service                        user-service
     │                                     │
     │  GET /api/users/123                 │
     │──────────────────────────────────→  │
     │                                     │
     │  200 OK                             │
     │  {"id":"123", "name":"张三"}        │
     │←──────────────────────────────────  │
     │                                     │
```

### 优点

- **最通用**：任何语言都有HTTP客户端，浏览器也能直接调
- **最直观**：URL + HTTP方法 + JSON——人类可读，调试方便
- **生态完善**：Swagger/OpenAPI自动生成文档和客户端SDK
- **无状态**：天然适合水平扩展

### 缺点

- **文本协议**：JSON比二进制大很多（字段名每次都传一遍），解析慢
- **HTTP/1.1的队头阻塞**：同一连接上的请求必须排队（HTTP/2解决了，但不是所有环境都支持）
- **没有类型安全**：字段拼错了编译器不知道，运行时才发现
- **接口定义靠文档**：没有强制契约，服务提供方改了字段名，调用了才知道

### Go中的REST调用

```go
type User struct {
	ID    string `json:"id"`
	Name  string `json:"name"`
	Email string `json:"email"`
}

func GetUser(userID string) (*User, error) {
	client := &http.Client{Timeout: 3 * time.Second}
	resp, err := client.Get("http://user-service:8080/api/users/" + userID)
	if err != nil {
		return nil, fmt.Errorf("调用用户服务失败: %w", err)
	}
	defer resp.Body.Close()

	if resp.StatusCode != http.StatusOK {
		return nil, fmt.Errorf("用户服务返回: %d", resp.StatusCode)
	}

	var user User
	if err := json.NewDecoder(resp.Body).Decode(&user); err != nil {
		return nil, fmt.Errorf("解析响应失败: %w", err)
	}
	return &user, nil
}
```

**最佳实践**：封装重试、超时、熔断（回顾97章高并发设计）：

```go
func GetUserWithRetry(userID string) (*User, error) {
	for i := 0; i < 3; i++ {
		user, err := GetUser(userID)
		if err == nil {
			return user, nil
		}
		time.Sleep(time.Duration(100*(i+1)) * time.Millisecond)
	}
	return nil, fmt.Errorf("重试3次后仍失败")
}
```

## 94.2 gRPC入门

### 什么是gRPC

gRPC是Google开源的高性能RPC框架。和REST的区别：REST是"我在网页上写了一段话发给你，你看了之后理解我的意思"，gRPC是"我直接把一个压缩包扔给你，你用解压软件打开——里面的数据精确到了字段类型"。

核心优势：
- **Protocol Buffers（protobuf，高效的二进制序列化格式，详见附录I）**：二进制序列化，体积小、速度快
- **HTTP/2**：多路复用、服务端推送、头部压缩
- **强类型契约**：`.proto` 文件定义接口，编译器自动生成客户端和服务端代码
- **多语言**：官方支持10+语言，包括Go、Java、Python、C++等

### 什么时候用gRPC

- 服务间的高频调用（每秒成千上万次）
- 需要流式传输（上传大文件、实时数据推送）
- 微服务内部通信（不直接暴露给浏览器）
- 严格的接口契约（支付、金融场景）

### Protocol Buffers定义服务

**user.proto**：

```protobuf
syntax = "proto3";

package user;
option go_package = "myapp/proto/user";

message GetUserRequest {
  string user_id = 1;
}

message GetUserResponse {
  string user_id = 1;
  string name = 2;
  string email = 3;
}

service UserService {
  rpc GetUser(GetUserRequest) returns (GetUserResponse);
}
```

关键语法：
- `message` = 结构体
- `= 1, 2, 3` = 字段编号（不是值，是标识，**绝对不能改**——改了就破坏向后兼容）
- `service` = 接口
- `rpc` = 方法

### 生成Go代码

```bash
protoc --go_out=. --go-grpc_out=. user.proto
```

生成两个文件：`user.pb.go`（数据结构）和 `user_grpc.pb.go`（服务端/客户端代码）。

### Go服务端实现

```go
package main

import (
	"context"
	"log"
	"net"

	"google.golang.org/grpc"
	pb "myapp/proto/user"
)

type userServer struct {
	pb.UnimplementedUserServiceServer
}

func (s *userServer) GetUser(ctx context.Context, req *pb.GetUserRequest) (*pb.GetUserResponse, error) {
	log.Printf("收到请求: user_id=%s", req.UserId)

	return &pb.GetUserResponse{
		UserId: req.UserId,
		Name:   "张三",
		Email:  "zhangsan@example.com",
	}, nil
}

func main() {
	lis, err := net.Listen("tcp", ":50051")
	if err != nil {
		log.Fatalf("监听失败: %v", err)
	}

	s := grpc.NewServer()
	pb.RegisterUserServiceServer(s, &userServer{})

	log.Println("gRPC服务启动在 :50051")
	if err := s.Serve(lis); err != nil {
		log.Fatalf("服务失败: %v", err)
	}
}
```

### Go客户端调用

```go
package main

import (
	"context"
	"log"
	"time"

	"google.golang.org/grpc"
	"google.golang.org/grpc/credentials/insecure"
	pb "myapp/proto/user"
)

func main() {
	conn, err := grpc.Dial("localhost:50051",
		grpc.WithTransportCredentials(insecure.NewCredentials()),
	)
	if err != nil {
		log.Fatalf("连接失败: %v", err)
	}
	defer conn.Close()

	client := pb.NewUserServiceClient(conn)

	ctx, cancel := context.WithTimeout(context.Background(), time.Second)
	defer cancel()

	resp, err := client.GetUser(ctx, &pb.GetUserRequest{UserId: "123"})
	if err != nil {
		log.Fatalf("调用失败: %v", err)
	}

	log.Printf("用户信息: ID=%s, 姓名=%s, 邮箱=%s", resp.UserId, resp.Name, resp.Email)
}
```

### gRPC四种通信模式

| 模式 | 说明 | 场景 |
|------|------|------|
| 一元RPC | 客户端发一个请求，服务端回一个响应 | 查询用户、创建订单 |
| 服务端流式 | 客户端一个请求，服务端持续推送 | 实时行情、日志流 |
| 客户端流式 | 客户端持续上传，服务端回一个响应 | 上传大文件、批量导入 |
| 双向流式 | 两端同时收发 | 聊天、实时协作 |

**服务端流式示例**：

```protobuf
service LogService {
  rpc StreamLogs(LogRequest) returns (stream LogEntry);
}
```

```go
func (s *logServer) StreamLogs(req *pb.LogRequest, stream pb.LogService_StreamLogsServer) error {
	for i := 0; i < 10; i++ {
		stream.Send(&pb.LogEntry{
			Message:   fmt.Sprintf("日志 #%d", i),
			Timestamp: time.Now().Unix(),
		})
		time.Sleep(500 * time.Millisecond)
	}
	return nil
}
```

## 94.3 消息队列通信

消息队列我们已经用第85-90章整整6章讲过了。在微服务通信的上下文中，消息队列是**第三种选择**——用于异步、解耦的场景。

```
order-service ──→ [消息队列] ──→ notification-service
                                   email-service
                                   sms-service
```

不需要等待响应，不需要服务在线——我们第85章已经讲得很透了。

## 94.4 同步 vs 异步的选择决策树

面对一个跨服务调用的需求，按以下顺序思考：

```
问题1: 调用方需要立刻拿到返回值吗？
  │
  ├── 是 → 同步调用（REST或gRPC）
  │       │
  │       ├── 调用方是浏览器吗？
  │       │     ├── 是 → REST（浏览器不直接支持gRPC）
  │       │     └── 否 → 继续下一问
  │       │
  │       ├── 每秒调用次数 > 1000？
  │       │     ├── 是 → gRPC（高性能）
  │       │     └── 否 → 都可以
  │       │
  │       └── 需要严格的接口契约？
  │             ├── 是 → gRPC（.proto强制契约）
  │             └── 否 → REST（更灵活）
  │
  └── 否 → 异步调用（消息队列）
          │
          ├── 需要多个消费者？（短信+邮件+推送都要发）
          │     ├── 是 → RabbitMQ（fanout/topic）
          │     └── 否 → 都可以
          │
          └── 消息量极大（百万/秒）或需要回溯？
                ├── 是 → Kafka
                └── 否 → RabbitMQ
```

### 混合使用

实际系统往往是混合使用的：

```
浏览器 ──→ REST ──→ API网关
                       │
                       ├──→ gRPC ──→ user-service（高频内部调用）
                       ├──→ gRPC ──→ order-service
                       │              │
                       │              └──→ MQ ──→ notification-service
                       │
                       └──→ REST ──→ product-service（低频调用，JSON方便）
```

🤔 **想多一点**：通信方式的选择不只是技术问题

很多时候，技术选型被"组织因素"影响：

- 团队A只会REST → 那就REST，不要说"gRPC更好所以必须换"
- 第三方服务只提供HTTP API → 那就REST，你没得选
- 跨语言团队（Go + Python + Java） → REST是最小公约数
- 所有服务都在K8s里 → gRPC是自然选择（配合服务网格如Istio）

技术最优 ≠ 团队最优。团队最优的方案是：你觉得该用gRPC，但团队只有一个人熟悉——先用REST，等团队成长了再迁移。

## 本章小结

| 通信方式 | 特点 | 适用场景 | Go实现 |
|---------|------|---------|--------|
| REST | HTTP+JSON，最通用 | 浏览器→服务、低频调用、跨语言 | `net/http` + `json` |
| gRPC | HTTP/2+Protobuf，高性能 | 高并发内部调用、流式传输 | `google.golang.org/grpc` |
| 消息队列 | 异步解耦 | 不需要立刻返回、多消费者 | `amqp091-go` / `sarama` |
| 决策树 | 先问"同步还是异步"→再问"性能/契约/消费者" | | |
