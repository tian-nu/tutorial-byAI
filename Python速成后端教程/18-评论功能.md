## 一、目标与完成效果

**一句话目标**：给博客系统加上评论功能——用户可以给文章写评论，查看文章时评论自动嵌套在文章数据里返回，并且只能删除自己的评论。

**完成后的可观测效果**：
- alice 登录后给一篇文章写了一条评论，bob 登录后也给同一篇文章写了一条评论。
- 你用 `GET /posts/{id}` 查看文章，返回的 JSON 里多了一个 `comments` 字段，里面装着两条评论——评论嵌套在文章里，不需要另外请求。
- 你用 `GET /posts/{post_id}/comments` 可以单独获取某篇文章的所有评论。
- bob 尝试删除 alice 的评论 → 返回 403 Forbidden，删不掉。
- alice 删除自己的评论 → 成功。
- 你理解了"嵌套返回"是什么意思——评论数据跟着文章一起返回，就像快递包裹里附带了清单。

## 三、分步操作

### 步骤 1：评论模型——给 `models.py` 加一张 `comments` 表

**我在做什么？** 在数据库里新增一张"评论表"。每条评论记录三件事：谁写的（`author_id`）、写在哪个文章下面（`post_id`）、写了什么（`content`）。

**比喻**：文章是"主贴"，评论是"跟帖"。一篇文章可以有 N 条评论，这是典型的"一对多"关系——一篇文章对多条评论。

打开 `blog_backend/models.py`，在文件末尾追加 `Comment` 模型：

```python
# blog_backend/models.py —— 在文件末尾追加

class Comment(Base):
    __tablename__ = "comments"

    id = Column(Integer, primary_key=True, index=True)
    content = Column(String(500), nullable=False)
    created_at = Column(DateTime, default=datetime.utcnow)

    # 外键：每条评论属于一篇文章
    post_id = Column(Integer, ForeignKey("posts.id"), nullable=False)

    # 外键：每条评论属于一个用户（作者）
    author_id = Column(Integer, ForeignKey("users.id"), nullable=False)

    # 关系：评论 ↔ 文章（双向导航）
    post = relationship("Post", back_populates="comments")

    # 关系：评论 ↔ 用户（双向导航）
    author = relationship("User", back_populates="comments")
```

**逐行解释**：

- `content = Column(String(500), nullable=False)`：评论内容，最长 500 字，不能为空。评论不像文章那么长，500 字足够了。
- `post_id = Column(Integer, ForeignKey("posts.id"), nullable=False)`：指向 `posts` 表的外键——"这条评论挂在哪篇文章下面"。`nullable=False` 意味着每条评论必须属于一篇文章，不能是"流浪评论"。
- `author_id = Column(Integer, ForeignKey("users.id"), nullable=False)`：指向 `users` 表的外键——"这条评论是谁写的"。
- `post = relationship("Post", back_populates="comments")`：建立 Python 对象之间的双向导航。`Comment.post` 可以让你从评论对象直接跳到它所属的文章对象。`back_populates="comments"` 告诉 SQLAlchemy："文章那边也有一个叫 `comments` 的属性指向我"。
- `author = relationship("User", back_populates="comments")`：同理，`Comment.author` 让你从评论对象跳到作者对象。

**现在还需要在 `Post` 和 `User` 模型里补上反向关系**。找到 `Post` 类，在 `author` 的 `relationship` 下面加一行：

```python
# blog_backend/models.py —— 在 Post 类中追加

    # 关系：文章有多条评论
    comments = relationship("Comment", back_populates="post")
```

找到 `User` 类，在 `posts` 的 `relationship` 下面加一行：

```python
# blog_backend/models.py —— 在 User 类中追加

    # 关系：一个用户有多条评论
    comments = relationship("Comment", back_populates="author")
```

**完整关系图**（现在你的三张表之间的 relationship 双向绑定）：

```
User ──(posts)──→ Post ──(comments)──→ Comment
  ↑                  ↑                    │
  └──(comments)──────┴────────────────────┘
```

- `User.posts`：一个用户写的所有文章
- `User.comments`：一个用户写的所有评论
- `Post.author`：文章的作者
- `Post.comments`：文章下面的所有评论
- `Comment.author`：评论的作者
- `Comment.post`：评论所属的文章

**验证方法**：保存文件，服务器自动重启。如果终端没有报错（没有 `ImportError` 或 `AttributeError`），说明模型定义正确。

但此时数据库里还没有 `comments` 表！因为我们用的是 SQLite，需要手动创建表。在终端中执行：

```powershell
python -c "from blog_backend.database import engine, Base; from blog_backend import models; Base.metadata.create_all(bind=engine); print('comments 表已创建')"
```

如果输出 `comments 表已创建`，说明表已建好。**注意**：`create_all` 只会创建还不存在的表，不会删除已有数据，所以是安全的。

> 🤔 想多一点：为什么不在 `database.py` 启动时自动建表？

因为实际项目中，建表通常由数据库迁移工具（如 Alembic）管理，而不是每次启动都执行 `create_all`。我们目前为了简单，手动执行。教程 23 会介绍数据库迁移。

### 步骤 2：评论 Schema——给 `schemas.py` 加评论的数据模型

**我在做什么？** 定义评论的请求和响应格式。就像文章有 `BlogCreate`（创建时用）和 `BlogOut`（返回时用），评论也需要自己的 Schema。

打开 `blog_backend/schemas.py`，在文件末尾追加：

```python
# blog_backend/schemas.py —— 在文件末尾追加

from datetime import datetime
from pydantic import BaseModel


class CommentCreate(BaseModel):
    """创建评论时客户端传的数据"""
    content: str


class CommentOut(BaseModel):
    """返回评论时的数据格式"""
    id: int
    content: str
    post_id: int
    author_id: int
    created_at: datetime

    class Config:
        from_attributes = True  # 允许从 ORM 对象自动转换
```

**逐行解释**：

- `CommentCreate`：只有一个字段 `content`。客户端只需要告诉我们"评论内容是什么"。`post_id` 从 URL 路径参数获取，`author_id` 从 token 解析——都不需要客户端传。
- `CommentOut`：返回给客户端的数据格式。包含评论的所有字段，让客户端知道谁在什么时候写了什么。
- `class Config: from_attributes = True`：告诉 Pydantic "这个 Schema 可以从 SQLAlchemy 的 ORM 对象自动转换"。没有这行，`return comment` 时会报错。

**还需要修改 `BlogOut`**，让它在返回文章时自动包含评论列表。找到 `schemas.py` 中的 `BlogOut` 类，改成：

```python
# blog_backend/schemas.py —— 修改 BlogOut 类

class BlogOut(BaseModel):
    id: int
    title: str
    content: str
    published: bool
    summary: str | None = None
    author_id: int
    created_at: datetime
    updated_at: datetime | None = None
    comments: list[CommentOut] = []  # ← 新增：嵌套评论列表

    class Config:
        from_attributes = True
```

**关键改动**：`comments: list[CommentOut] = []`。这行告诉 Pydantic："当把 ORM 的 `Post` 对象转成 `BlogOut` 时，`Post.comments`（一个 Comment 对象列表）要转成 `CommentOut` 对象列表"。

**注意**：`CommentOut` 在 `BlogOut` 之前定义很重要——Python 代码从上到下执行，`BlogOut` 引用了 `CommentOut`，所以 `CommentOut` 必须先定义。

**验证方法**：

```powershell
python -c "from blog_backend.schemas import CommentCreate, CommentOut, BlogOut; print('Schemas OK')"
```

如果输出 `Schemas OK`，说明 Schema 定义正确。

### 步骤 3：创建评论接口——`POST /posts/{post_id}/comments`

**我在做什么？** 给 `main.py` 加一个接口：用户登录后，可以对某篇文章发表评论。

打开 `blog_backend/main.py`，在文件顶部导入新增的类：

```python
# blog_backend/main.py —— 修改顶部导入

from blog_backend.schemas import BlogCreate, CommentCreate, CommentOut  # 新增 CommentCreate, CommentOut
from blog_backend.models import Post, User, Comment  # 新增 Comment
```

然后在文件末尾追加接口：

```python
# blog_backend/main.py —— 在末尾追加

# === 创建评论 ===
@app.post("/posts/{post_id}/comments", response_model=CommentOut)
def create_comment(
    post_id: int,
    comment: CommentCreate,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)
):
    # 1. 检查文章是否存在
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法评论"}
        )

    # 2. 创建评论
    new_comment = Comment(
        content=comment.content,
        post_id=post_id,
        author_id=current_user.id,
    )
    db.add(new_comment)
    db.commit()
    db.refresh(new_comment)
    return new_comment
```

**逐行解释**：

- `@app.post("/posts/{post_id}/comments", response_model=CommentOut)`：URL 路径包含 `{post_id}`，表示"给第 X 号文章写评论"。`response_model=CommentOut` 告诉 FastAPI 返回数据的格式。
- `post_id: int`：从 URL 路径中提取的文章 ID。
- `comment: CommentCreate`：从请求体中提取评论内容（只有 `content` 字段）。
- `current_user: User = Depends(get_current_user)`：必须登录才能评论。没带 token → 401。
- 第一步检查文章是否存在：对不存在的文章评论没意义，返回 404。
- `new_comment = Comment(...)`：创建评论对象。`post_id` 来自 URL 路径，`author_id` 来自 token，`content` 来自请求体——三个来源，服务端拼起来，不信任客户端传任何身份信息。
- `db.add` → `db.commit` → `db.refresh`：标准的三步"写入数据库"流程（教程 13 已学过）。

**❌ 常见错误 → ✅ 正确示例**：

❌ 错误：忘记设置 `post_id`

```python
new_comment = Comment(
    content=comment.content,
    author_id=current_user.id,
    # 忘记写 post_id=post_id ← 这里漏了！
)
```

**后果**：评论写入了数据库，但 `post_id` 是 `NULL`（或默认值），评论成了"孤儿记录"——数据库里有这条评论，但不知道它属于哪篇文章。此术语需进附录。

✅ 正确：

```python
new_comment = Comment(
    content=comment.content,
    post_id=post_id,          # ← 从 URL 路径获取
    author_id=current_user.id, # ← 从 token 获取
)
```

**验证方法**：

```bash
# 1. alice 登录
$AliceToken = curl -s -X POST http://127.0.0.1:8000/login `
  -H "Content-Type: application/json" `
  -d '{"username":"alice","password":"alice123"}' `
  | python -c "import sys,json; print(json.load(sys.stdin)['access_token'])"

# 2. 先确认有一篇文章（如果没有，先创建一篇）
curl -s -X POST http://127.0.0.1:8000/posts `
  -H "Content-Type: application/json" `

# 3. 给文章 #1 写评论
curl -s -X POST http://127.0.0.1:8000/posts/1/comments `
  -H "Content-Type: application/json" `
  -H "Authorization: Bearer $AliceToken" `

```

### 步骤 4：获取文章评论接口——`GET /posts/{post_id}/comments`

**我在做什么？** 加一个接口：获取某篇文章下的所有评论。

在 `main.py` 末尾追加：

```python
# blog_backend/main.py —— 在末尾追加

# === 获取某篇文章的所有评论 ===
@app.get("/posts/{post_id}/comments", response_model=list[CommentOut])
def get_comments(post_id: int, db: Session = Depends(get_db)):
    # 1. 检查文章是否存在
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在"}
        )

    # 2. 查询该文章的所有评论
    comments = db.query(Comment).filter(Comment.post_id == post_id).all()
    return comments
```

**逐行解释**：

- `response_model=list[CommentOut]`：返回的是一个评论列表，每个元素是 `CommentOut` 格式。
- 这个接口**不需要登录**——任何人都可以看评论，和看文章一样。
- `db.query(Comment).filter(Comment.post_id == post_id).all()`：查 `comments` 表，找出所有 `post_id` 等于目标文章 ID 的行。这就是 SQL 中 `SELECT * FROM comments WHERE post_id = ?` 的 ORM 写法。

**验证方法**：

```bash
curl -s http://127.0.0.1:8000/posts/1/comments
```

**预期输出**：

```json

    "content": "写得太好了！",
    "post_id": 1,
    "author_id": 1,
```

### 步骤 5：嵌套返回——让文章接口自动带上评论

**我在做什么？** 修改 `GET /posts/{id}` 接口，让它在返回文章时自动把评论也带上——不需要客户端再发一次请求。

**比喻**：以前你拿快递，快递柜只给你一个包裹。现在快递柜给你包裹的同时，还附了一张清单，上面写着"此包裹包含以下物品"。你不需要再去查清单——清单跟着包裹一起来了。

**原理**：SQLAlchemy 的 `relationship` 会自动加载关联数据。当你访问 `post.comments` 时，SQLAlchemy 会自动去数据库查询所有 `post_id` 等于当前文章 ID 的评论，然后返回一个 Python 列表。

打开 `main.py`，找到 `get_post` 接口（`GET /posts/{post_id}`），确认它返回的是 `Post` ORM 对象。由于 `BlogOut` 的 `Config` 设置了 `from_attributes=True`，并且 `BlogOut` 新增了 `comments` 字段，Pydantic 会自动把 `post.comments` 转成 `CommentOut` 列表。

**不需要改 `get_post` 的代码！** 因为：
1. `get_post` 返回 `db.query(Post).filter(...).first()`，这是一个 `Post` ORM 对象。
2. `Post` 类有 `comments = relationship(...)`，所以 `post.comments` 自动可用。
3. `BlogOut` 有 `comments: list[CommentOut] = []`，Pydantic 会自动转换。

**验证方法**：

```bash
curl -s http://127.0.0.1:8000/posts/1
```

**预期输出**（注意最后多了一个 `comments` 字段）：

```json
{
  "id": 1,
  "title": "需要评论的文章",
  "content": "来评论吧！",
  "published": true,
  "summary": null,
  "author_id": 1,
  "created_at": "2026-06-05T...",
  "updated_at": null,
  "comments": [
    {
      "id": 1,
      "content": "写得太好了！",
      "post_id": 1,
      "author_id": 1,
      "created_at": "2026-06-05T..."
    }
  ]
}
```

**评论自动嵌套在文章数据里了！** 这就是"嵌套返回"——客户端一次请求拿到文章和所有评论，不需要再发 `GET /posts/1/comments`。

> 🤔 想多一点：`relationship` 什么时候去查数据库？

SQLAlchemy 默认是**懒加载（lazy loading）**——你不访问 `post.comments`，它就不查。当你第一次访问 `post.comments` 时，SQLAlchemy 才去数据库执行 `SELECT * FROM comments WHERE post_id = ?`。这就像你有一本"目录"，翻开目录才知道内容在哪——但真正的内容要等你翻到那一页才加载。

**懒加载的好处**：如果某次请求不需要评论数据（比如只是检查文章是否存在），就不会浪费一次查询。**坏处**：如果需要评论，就会多一次查询。教程 19 会深入讲这个问题（N+1 问题）以及优化方案。

### 步骤 6：删除评论——只能删自己的评论

**我在做什么？** 加一个删除评论的接口，但只有评论的作者本人才能删。

在 `main.py` 末尾追加：

```python
# blog_backend/main.py —— 在末尾追加

# === 删除评论 ===
@app.delete("/comments/{comment_id}")
def delete_comment(
    comment_id: int,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)
):
    # 1. 查找评论
    comment = db.query(Comment).filter(Comment.id == comment_id).first()
    if comment is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"评论 #{comment_id} 不存在"}
        )

    # 2. 检查是不是评论作者本人
    check_owner(comment, current_user)  # ← 复用教程 17 的 check_owner！

    # 3. 是本人，删除
    db.delete(comment)
    db.commit()
    return {"message": f"评论 #{comment_id} 已删除"}
```

**逐行解释**：

- `@app.delete("/comments/{comment_id}")`：URL 路径是 `/comments/{comment_id}`，不是 `/posts/{post_id}/comments/{comment_id}`。因为评论 ID 是全局唯一的，直接通过评论 ID 定位即可。
- `check_owner(comment, current_user)`：**直接复用教程 17 写的 `check_owner` 函数！** 因为 `check_owner` 只检查 `obj.author_id == current_user.id`，不关心 `obj` 是文章还是评论——只要它有 `author_id` 属性就行。
- 这就是为什么教程 17 要把权限检查抽成公共函数——现在加评论功能，一行代码就搞定了权限检查，不用再写一遍 `if comment.author_id != current_user.id: raise HTTPException(...)`。

**比喻**：`check_owner` 就像一个万能门禁卡验证器。不管你是要进房间（文章）还是要进储物柜（评论），只要你有 `author_id` 这个"锁"，验证器就能用。

```bash
  -H "Content-Type: application/json" `
  -d '{"username":"bob","password":"bob123"}' `
  | python -c "import sys,json; print(json.load(sys.stdin)['access_token'])"

# 2. bob 也发一条评论
```

### `blog_backend/main.py`（本教程新增的接口）

```python
from blog_backend.schemas import BlogCreate, CommentCreate, CommentOut
from blog_backend.models import Post, User, Comment
from blog_backend.database import get_db
from blog_backend.auth import get_current_user, check_owner

# ... 之前的接口（GET /posts、POST /posts、PUT /posts/{id}、DELETE /posts/{id} 等）保持不变 ...


# === 创建评论 ===
@app.post("/posts/{post_id}/comments", response_model=CommentOut)
def create_comment(
    post_id: int,
    comment: CommentCreate,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)
):
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在，无法评论"}
        )

    new_comment = Comment(
        content=comment.content,
        post_id=post_id,
        author_id=current_user.id,
    )
    db.add(new_comment)
    db.commit()
    db.refresh(new_comment)
    return new_comment


# === 获取某篇文章的所有评论 ===
@app.get("/posts/{post_id}/comments", response_model=list[CommentOut])
def get_comments(post_id: int, db: Session = Depends(get_db)):
    post = db.query(Post).filter(Post.id == post_id).first()
    if post is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"文章 #{post_id} 不存在"}
        )

    comments = db.query(Comment).filter(Comment.post_id == post_id).all()
    return comments


# === 删除评论 ===
@app.delete("/comments/{comment_id}")
def delete_comment(
    comment_id: int,
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user)
):
    comment = db.query(Comment).filter(Comment.id == comment_id).first()
    if comment is None:
        return JSONResponse(
            status_code=404,
            content={"error": f"评论 #{comment_id} 不存在"}
        )

    check_owner(comment, current_user)

    db.delete(comment)
    db.commit()
    return {"message": f"评论 #{comment_id} 已删除"}
```

## 六、术语附录

| 术语 | 英文 | 通俗解释 | 本章出现位置 |
|------|------|----------|-------------|
| 嵌套返回 | Nested Response | 把关联数据（如评论）嵌在主数据（如文章）的 JSON 里一起返回，而不是让客户端发两次请求。就像快递包裹里附带清单。 | 步骤 5 |
| relationship | — | SQLAlchemy 中定义对象间关联的方法。`relationship("Post", back_populates="comments")` 的意思是"Comment 对象可以通过 `.post` 属性访问关联的 Post 对象"。此术语需进附录。 | 步骤 1 |
| back_populates | — | `relationship` 的参数，用于建立双向导航。两个关联类的值必须互相匹配。比如 `Comment.post` 的 `back_populates="comments"` 对应 `Post.comments` 的 `back_populates="post"`。 | 步骤 1 |
| 孤儿记录 | Orphan Record | 数据库中存在但无法通过正常关联路径访问到的记录。比如 `post_id` 为 NULL 的评论——它属于哪篇文章？不知道。通常由忘记设置外键字段导致。 | 步骤 3 |
| 懒加载 | Lazy Loading | SQLAlchemy 默认行为：你不访问关联属性（如 `post.comments`），就不去数据库查。第一次访问时才查。好处是省查询，坏处是可能造成 N+1 问题（教程 19 详解）。 | 步骤 5 |

## 八、已知坑点与禁止事项

| 坑点 | 现象 | 原因 | 解决 |
|------|------|------|------|
| 忘记设置 `post_id` | 评论创建成功但查看文章时评论不出现 | 评论的 `post_id` 为 NULL，`relationship` 无法匹配 | 创建评论时确保 `post_id=post_id`（从 URL 路径获取） |
| 忘记在 `BlogOut` 加 `comments` 字段 | 文章返回了但评论不出现 | Pydantic 序列化时只序列化 `BlogOut` 中声明的字段，`comments` 没声明就不会输出 | 在 `BlogOut` 中加 `comments: list[CommentOut] = []` |
| 忘记给 `CommentOut` 加 `from_attributes=True` | 接口返回 500 错误 | Pydantic 默认不能从 ORM 对象自动转换，需要显式开启 | `class Config: from_attributes = True` |
| `CommentOut` 定义在 `BlogOut` 之后 | 启动报 `NameError: name 'CommentOut' is not defined` | Python 代码从上到下执行，`BlogOut` 引用 `CommentOut` 时后者还没定义 | 把 `CommentOut` 放在 `BlogOut` 之前定义 |
| 忘记建表 | 创建评论时报 `no such table: comments` | 只定义了 Python 模型，但没有在数据库里建表 | 执行 `Base.metadata.create_all(bind=engine)` |
| 让客户端传 `post_id` 或 `author_id` | 用户可以伪造评论归属 | 客户端传来的数据不可信 | `post_id` 从 URL 路径取，`author_id` 从 token 取 |
