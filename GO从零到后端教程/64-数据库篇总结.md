## 64.1 MySQL vs Redis：全景对比

两张表放在一起对比，你才能看清它们各自的位置：

| 维度 | MySQL | Redis |
|------|-------|-------|
| **定位** | 持久化关系型数据库 | 内存键值缓存/数据结构服务器 |
| **存储介质** | 硬盘（HDD/SSD） | 内存（RAM） |
| **数据模型** | 表（行和列） | 键值对（String/Hash/List/Set/ZSet） |
| **查询方式** | SQL（声明式，功能极强） | 命令（按key精确操作，功能简单） |
| **关联查询** | JOIN（强大） | 不支持（需应用层关联） |
| **事务** | ACID（完善） | 有限的原子性（无回滚） |
| **持久化** | 天然持久化 | RDB/AOF可选，非天然 |
| **单机QPS** | 1000~5000 | 50000~100000+ |
| **扩展方式** | 主从复制/分库分表 | 集群/哨兵/分片 |
| **典型用途** | 业务数据存储 | 缓存/排行榜/计数器/分布式锁/消息队列 |

### 它们不是竞争关系——是搭档

```
用户请求 → Redis（查缓存）
              ↓ miss
           MySQL（查数据库）
              ↓
           回写Redis（下次命中）
```

MySQL是"地面部队"——稳扎稳打，持久可靠。
Redis是"特种部队"——快速突击，处理热点。

**没有Redis系统能跑**（只是慢），**没有MySQL系统会丢数据**（除非你只用Redis）。

### 什么时候只用MySQL就够了

- 数据量小（百万级以下）
- 并发不高（QPS < 500）
- 数据结构复杂（多表JOIN）

### 什么时候必须上Redis

- 高并发读（首页/商品详情页/排行榜）
- 需要原子计数（点赞数/浏览量）
- 需要分布式锁
- 需要过期自动删除（验证码/登录token）
- 需要实时排行榜

### 什么时候两者都不够

- 全文搜索 → Elasticsearch
- 海量数据分析 → ClickHouse / Hadoop
- 真正的消息队列 → RabbitMQ / Kafka
- 时序数据 → InfluxDB / TimescaleDB

## 64.2 面试高频题速查

### MySQL篇

**Q1：一条SQL从输入到返回结果经历了什么？**

答案：连接认证 → SQL解析 → 优化器生成执行计划 → 执行器调用存储引擎API → 引擎读取数据（内存/磁盘）→ 返回结果。

**Q2：B+树为什么适合做索引？**

答案：多叉平衡 → 树高度低（3~4层）→ 磁盘IO次数少；叶子节点存完整数据/主键值；叶子节点链表相连 → 范围查询快；自平衡 → 不会退化为链表。

**Q3：聚簇索引和非聚簇索引的区别？**

答案：聚簇索引（主键索引）的叶子节点存**完整行数据**，一张表只有一个。非聚簇索引（二级索引）的叶子节点存**主键值**，查数据需要**回表**。覆盖索引可以避免回表。

**Q4：联合索引的最左前缀原则是什么？**

答案：联合索引 `(a, b, c)` 等效于三个索引 `(a)`、`(a, b)`、`(a, b, c)`。跳过最左列a则全部失效。范围查询（`>`、`<`）会截断后续字段的索引使用。

**Q5：索引在什么情况下会失效？**

答案：LIKE '%x'、对索引列使用函数、隐式类型转换、OR（不是所有列都有索引）、不等于（大多数情况）、NOT IN。

**Q6：事务的ACID分别是什么？怎么实现的？**

答案：原子性（undo log）、一致性（约束+其他三个共同保证）、隔离性（锁+MVCC）、持久性（redo log）。

**Q7：脏读、不可重复读、幻读的区别？**

答案：脏读=读到未提交数据；不可重复读=同事务内同一行两次读的值不同（UPDATE导致）；幻读=同事务内范围查询行数变化（INSERT/DELETE导致）。

**Q8：MySQL的隔离级别及各自解决的问题？**

| 级别 | 脏读 | 不可重复读 | 幻读 |
|------|:---:|:---:|:---:|
| READ UNCOMMITTED | ✗ | ✗ | ✗ |
| READ COMMITTED | ✓ | ✗ | ✗ |
| REPEATABLE READ（默认） | ✓ | ✓ | 部分✓ |
| SERIALIZABLE | ✓ | ✓ | ✓ |

**Q9：EXPLAIN中type字段的含义？**

答案（从好到差）：system > const > eq_ref > ref > range > index > ALL。目标是至少达到range级别。看到ALL（全表扫描）要警觉。

**Q10：深分页为什么慢？怎么优化？**

答案：`LIMIT 1000000, 10` 需要扫描前100万行再丢弃。优化：游标分页 `WHERE id > last_id LIMIT 10`（前提是id递增且有序）。

### Redis篇

**Q1：Redis为什么这么快？**

答案：①数据在内存；②单线程无锁竞争；③高效数据结构；④I/O多路复用（epoll）。

**Q2：Redis的过期删除策略？**

答案：惰性删除（访问时检查）+ 定期删除（每100ms随机抽查）。两套组合在CPU和内存间取得平衡。如果内存满了还有过期key没删 → 内存淘汰策略介入。

**Q3：Redis的内存淘汰策略有哪些？**

答案：8种，重点是 `noeviction`（默认拒绝→危险）、`allkeys-lru`（淘汰最近最少使用→最常用）、`allkeys-lfu`（淘汰最不频繁使用→热点场景）。

**Q4：RDB和AOF的区别？**

答案：RDB=定时快照（恢复快、有数据丢失窗口）；AOF=实时追加日志（安全、文件大、everysec推荐）；混合持久化=两者结合（4.0+最佳）。

**Q5：缓存穿透、击穿、雪崩分别是什么？怎么解决？**

| 问题 | 定义 | 方案 |
|------|------|------|
| 穿透 | 查不存在的数据 | 布隆过滤器 / 缓存空值 |
| 击穿 | 热点key过期 | 互斥锁 / 永不过期 |
| 雪崩 | 大量key同时过期 | 随机TTL / 多级缓存 / 限流 |

**Q6：如何保证缓存与数据库双写一致性？**

答案：没有完美方案。实践：先写DB再删缓存（通用）→ 延迟双删（更稳健）→ binlog同步（最可靠但复杂）。容忍短暂不一致，用过期时间兜底。

**Q7：Redis分布式锁怎么实现？**

答案：SET NX PX（加锁）+ Lua脚本（原子解锁，验证UUID防止误删别人的锁）+ 看门狗续期。Redlock算法（多节点）争议较大，一般业务单节点够用。

**Q8：Sorted Set的底层数据结构是什么？**

答案：跳表（skiplist）+ 哈希表。跳表支持范围查询和排序，哈希表支持O(1)查找元素分数。
