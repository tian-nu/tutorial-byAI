## 105.1 为什么需要监控

### 没有监控的世界

- 用户投诉"网站打不开了"——你才知道出问题了
- 数据库连接池满了，导致所有请求超时——你只能靠猜来排查
- 不知道流量什么时候达到峰值——扩容全靠蒙
- 内存泄漏悄悄发生——等发现时服务器已经OOM了几百次

### 有了监控之后

- CPU突然飙到99% → 告警通知你 → 你打开Grafana看是哪段时间哪个接口 → 精准定位
- 内存使用持续上升（从不下降） → 你知道有内存泄漏 → 趁早修复
- QPS图显示每天下午3点有个流量峰值 → 你知道该扩容了
- API响应时间从50ms涨到500ms → 你知道要查慢查询或依赖服务

### 监控的黄金四信号（Google SRE）

| 信号 | 含义 | 例子 |
|------|------|------|
| **延迟（Latency）** | 请求花了多长时间 | P99接口响应时间 |
| **流量（Traffic）** | 系统承受多大压力 | QPS（每秒请求数） |
| **错误（Errors）** | 请求失败率 | 5xx错误比例 |
| **饱和度（Saturation）** | 资源使用到极限了吗 | CPU/内存/磁盘 使用率 |

## 105.2 Prometheus基础

### Prometheus是什么

Prometheus是一个**开源的监控和告警系统**，最初由SoundCloud开发，现在是CNCF（云原生计算基金会）的毕业项目——和Kubernetes同等待遇。

**核心特点：**
- **时间序列数据库**：按时间存储指标数据（CPU使用率在10:00是50%，10:01是51%...）
- **Pull模式**：Prometheus主动去拉取指标（而不是应用推送），这样Prometheus自己控制采集节奏
- **PromQL**：强大的查询语言，能做聚合、计算、预测
- **多维数据模型**：每个指标可以带标签（比如 `http_requests_total{method="GET", status="200"}`）

### 架构

```
┌────────────────────────────────────────────┐
│                  Grafana                    │  可视化（图表、面板、Dashboard）
└──────────────────┬─────────────────────────┘
                   │ 查询PromQL
┌──────────────────▼─────────────────────────┐
│                Prometheus                   │  数据存储 + 查询
└──┬──────────┬──────────┬───────────────────┘
   │ Pull     │ Pull     │ Pull
┌──▼────┐ ┌──▼────┐ ┌──▼──────┐
│Node   │ │Go App │ │ MySQL   │             Exporter（指标暴露器）
│Exporter│ │/metrics│ │Exporter│
└───────┘ └───────┘ └─────────┘
      ▲        ▲          ▲
      │        │          │
   Linux   你的Go程序   MySQL数据库
```

### 安装Prometheus

用Docker最方便：

```bash
mkdir -p /opt/prometheus
```

创建 `/opt/prometheus/prometheus.yml`：

```yaml
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: "myapp"
    static_configs:
      - targets: ["host.docker.internal:8080"]

  - job_name: "node"
    static_configs:
      - targets: ["node-exporter:9100"]
```

```bash
docker run -d \
  --name prometheus \
  -p 9090:9090 \
  -v /opt/prometheus/prometheus.yml:/etc/prometheus/prometheus.yml \
  prom/prometheus
```

访问 `http://服务器IP:9090` 看到Prometheus的Web界面。

### PromQL快速入门

```promql
http_requests_total                                      # 所有HTTP请求总数
http_requests_total{status="200"}                        # 状态码为200的请求
rate(http_requests_total[5m])                            # 最近5分钟每秒请求增长率(QPS)
sum(rate(http_requests_total[5m])) by (status)           # 按状态码分组的QPS
histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))  # P99延迟
```

## 105.3 Grafana

### 安装

```bash
docker run -d \
  --name grafana \
  -p 3000:3000 \
  grafana/grafana
```

访问 `http://服务器IP:3000`，默认账号密码 `admin/admin`。

### 连接Prometheus

1. 登录Grafana
2. 左侧菜单 → Connections → Data sources → Add data source → Prometheus
3. URL填 `http://prometheus:9090`（如果Grafana和Prometheus在同一个Docker网络里）
4. 点 Save & test

### 导入Dashboard

不用从零画面板——社区有大量现成的模板：

1. 左侧菜单 → Dashboards → Import
2. 填入Dashboard ID：
   - **Go进程监控**：ID `10826`（Go Processes）
   - **Node Exporter（服务器基础监控）**：ID `1860`
   - **MySQL监控**：ID `7362`
3. 选择Prometheus数据源 → Import

30秒内你就有一个漂亮的监控面板（而不是花半天手动调）。

## 105.4 Go应用接入Prometheus

### 引入依赖

```bash
go get github.com/prometheus/client_golang
```

### 配置/metrics端点

```go
package main

import (
	"log"
	"net/http"

	"github.com/gin-gonic/gin"
	"github.com/prometheus/client_golang/prometheus"
	"github.com/prometheus/client_golang/prometheus/promhttp"
)

var (
	httpRequestsTotal = prometheus.NewCounterVec(
		prometheus.CounterOpts{
			Name: "http_requests_total",
			Help: "Total number of HTTP requests",
		},
		[]string{"method", "path", "status"},
	)

	httpRequestDuration = prometheus.NewHistogramVec(
		prometheus.HistogramOpts{
			Name:    "http_request_duration_seconds",
			Help:    "HTTP request duration in seconds",
			Buckets: prometheus.DefBuckets,
		},
		[]string{"method", "path"},
	)

	activeConnections = prometheus.NewGauge(
		prometheus.GaugeOpts{
			Name: "active_connections",
			Help: "Current number of active connections",
		},
	)
)

func init() {
	prometheus.MustRegister(httpRequestsTotal)
	prometheus.MustRegister(httpRequestDuration)
	prometheus.MustRegister(activeConnections)
}

func main() {
	r := gin.Default()

	r.GET("/metrics", gin.WrapH(promhttp.Handler()))

	r.Use(prometheusMiddleware())

	r.GET("/api/health", func(c *gin.Context) {
		c.JSON(200, gin.H{"status": "ok"})
	})

	log.Fatal(r.Run(":8080"))
}

func prometheusMiddleware() gin.HandlerFunc {
	return func(c *gin.Context) {
		path := c.FullPath()
		if path == "" {
			path = "unknown"
		}

		activeConnections.Inc()
		defer activeConnections.Dec()

		timer := prometheus.NewTimer(httpRequestDuration.WithLabelValues(c.Request.Method, path))
		defer timer.ObserveDuration()

		c.Next()

		status := http.StatusText(c.Writer.Status())
		httpRequestsTotal.WithLabelValues(c.Request.Method, path, status).Inc()
	}
}
```

**四种Metric类型详解：**

**Counter（计数器）**：只增不减的值。
```go
httpRequestsTotal.Inc()   // +1
```
用例：请求总数、错误总数、处理的字节数。

**Gauge（仪表盘）**：可增可减的值。
```go
activeConnections.Inc()   // +1
activeConnections.Dec()   // -1
```
用例：当前连接数、内存使用量、CPU温度。

**Histogram（直方图）**：数据分布（分桶统计）。
```go
httpRequestDuration.Observe(0.05)   // 记录一次耗时0.05秒的请求
```
用例：请求响应时间的分布（有多少0-10ms的、多少10-50ms的……）。

**Summary（摘要）**：类似Histogram，但在客户端计算分位数（不用预定义桶）。
```go
prometheus.NewSummary(prometheus.SummaryOpts{
    Name: "request_duration_summary",
    Objectives: map[float64]float64{0.5: 0.05, 0.9: 0.01, 0.99: 0.001},
})
```

### 验证

启动应用后访问 `http://localhost:8080/metrics`，你会看到：

```
http_requests_total{method="GET",path="/api/health",status="200"} 42
http_request_duration_seconds_bucket{method="GET",path="/api/health",le="0.005"} 35
http_request_duration_seconds_bucket{method="GET",path="/api/health",le="0.01"} 40
active_connections 3
```

这些就是Prometheus拉取的数据。

## 105.5 告警：Alertmanager

### 告警链

```
Prometheus 检测到异常
    │
    ▼
Alertmanager 接收告警
    │
    ├── 去重（相同告警只发一次）
    ├── 分组（相关告警合并发送）
    ├── 静默（已知维护期的告警不发送）
    │
    ▼
通知渠道：飞书/钉钉/企业微信/邮件/Slack/PagerDuty
```

### Alertmanager配置

`/opt/prometheus/alertmanager.yml`：

```yaml
global:
  resolve_timeout: 5m

route:
  group_by: ["alertname"]
  group_wait: 10s
  group_interval: 10s
  repeat_interval: 1h
  receiver: "webhook"

receivers:
  - name: "webhook"
    webhook_configs:
      - url: "https://open.feishu.cn/open-apis/bot/v2/hook/YOUR_BOT_TOKEN"
        send_resolved: true
```

### Prometheus告警规则

`/opt/prometheus/rules.yml`：

```yaml
groups:
  - name: myapp_alerts
    rules:
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.05
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "高错误率警报"
          description: "过去5分钟5xx错误率超过5%，当前值: {{ $value }}"

      - alert: HighLatency
        expr: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) > 1
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "P99延迟超过1秒"
          description: "P99请求延迟为 {{ $value }} 秒"

      - alert: ServiceDown
        expr: up{job="myapp"} == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "服务挂了！"
          description: "myapp服务已停止响应超过1分钟"
```

### 飞书机器人告警

1. 飞书群里 → 群设置 → 群机器人 → 添加自定义机器人
2. 复制Webhook地址
3. 把地址填到Alertmanager配置中
4. 当Prometheus触发规则条件时，群机器人自动发消息

消息示例：
```
🔴 高错误率警报
过去5分钟5xx错误率超过5%，当前值: 0.08
```

## 105.6 日志收集：EFK/ELK栈

### 为什么需要集中日志

当你只有一个服务器时，`docker logs` 或 `journalctl` 就够了。但当你有10台、50台服务器时，不可能一台一台去找日志。你需要把所有服务器的日志汇聚到一个地方，统一搜索。

### EFK栈（轻量版）

```
┌─────────┐    ┌──────────┐    ┌───────────────┐    ┌────────┐
│ 各个服务器 │───>│ Filebeat │───>│ Elasticsearch │<───│ Kibana │
│ 日志文件   │    │ (采集/转发)│    │ (存储/搜索)    │    │ (界面)  │
└─────────┘    └──────────┘    └───────────────┘    └────────┘
```

- **Filebeat**：轻量级日志采集器，安装在每台服务器上，读取日志文件并发送
- **Elasticsearch**：搜索引擎，存储和索引日志（可以搜任意关键词）
- **Kibana**：Web界面，查询日志、画图、建Dashboard

ELK中的"L"是**Logstash**——一个更重的日志处理器（可以做复杂的过滤和转换）。对大多数场景来说Filebeat就够了（简化为EFK）。

### 为什么你可能暂时不需要EFK

EFK栈完整部署需要较多资源（至少4GB内存）。对于个人项目和中小团队：
1. 用 `docker logs` + `grep` 做简单排查
2. 用云服务商的日志服务（阿里云SLS、腾讯云CLS，按量付费，小流量几乎免费）
3. 等日志量上来了再考虑自建EFK

### Go应用推荐日志格式

为方便后续接入日志收集系统，推荐输出JSON格式的结构化日志：

```go
log := map[string]interface{}{
    "level":        "info",
    "timestamp":    time.Now().Format(time.RFC3339),
    "message":      "用户登录成功",
    "user_id":      12345,
    "ip":           "192.168.1.1",
    "request_id":   "abc-123-def",
}
```

结构化日志的好处：在Kibana中可以按 `user_id` 过滤、按 `level` 聚合——比纯文本日志强一万倍。

## 105.7 分布式追踪

### 什么是分布式追踪

一个用户请求可能经过：Nginx → Go服务A → Go服务B → MySQL → Redis。如果这个请求卡了，慢在哪里？

分布式追踪给每个请求一个**全局唯一的Trace ID**，在请求经过的每个服务处记录时间，最终拼出完整的时间线：

```
Trace ID: abc123 (总耗时 500ms)
├── Nginx: 2ms
├── Go服务A: 498ms
│   ├── 业务逻辑: 50ms
│   ├── 调用Go服务B: 400ms  ← 这个慢了！
│   └── 写Redis: 10ms
└── 响应返回: 0ms
```

### Jaeger简介

Jaeger是Uber开源的分布式追踪系统，CNCF毕业项目。

架构：
```
应用代码（集成OpenTelemetry SDK）
    │  发送span数据
    ▼
Jaeger Collector
    │
    ▼
Jaeger (存储 + UI)
    │
    ▼
可视化：调用链、耗时分析、服务依赖图
```

### Go接入OpenTelemetry

```go
import (
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/exporters/jaeger"
    "go.opentelemetry.io/otel/sdk/trace"
)

func initTracer() func() {
    exporter, _ := jaeger.New(jaeger.WithCollectorEndpoint(
        jaeger.WithEndpoint("http://localhost:14268/api/traces"),
    ))

    tp := trace.NewTracerProvider(trace.WithBatcher(exporter))
    otel.SetTracerProvider(tp)

    return func() { tp.Shutdown(context.Background()) }
}

func someHandler(c *gin.Context) {
    tracer := otel.Tracer("myapp")
    ctx, span := tracer.Start(c.Request.Context(), "someHandler")
    defer span.End()

    span.SetAttributes(attribute.String("user_id", "123"))
}
```

### 你什么时候需要分布式追踪

| 架构 | 需要分布式追踪吗 |
|------|-----------------|
| 单体应用 | 不需要，看日志就够了 |
| 2-3个微服务 | 看日志基本够用，追踪是锦上添花 |
| 5个以上微服务 | 强烈建议，否则排查问题像盲人摸象 |
| 10个以上微服务 | 必须，没有追踪系统你会疯掉的 |

## 🤔 想多一点：监控和日志的区别

很多初学者把"监控"和"日志"混为一谈。它们是完全不同的东西：

| | 监控（Metrics） | 日志（Logging） |
|---|---|---|
| **是什么** | 数字（CPU=80%） | 文本（"User 123 logged in"） |
| **用途** | 发现问题（"不对劲！"） | 排查根因（"为什么？"） |
| **存储** | 时间序列数据库 | 搜索引擎或文件系统 |
| **查询方式** | 聚合、图表、趋势 | 关键词搜索、过滤 |
| **数据量** | 较小（每15秒一个点） | 很大（每秒可能上万行） |
| **先行者** | 先看监控发现异常 → | → 再看日志找具体原因 |

监控是**告警**（"CPU高于90%了！"），日志是**调查**（"查一下这个时间点这个接口的日志"）。两者互补，缺一个都会让你在半夜被叫醒时无从下手。
