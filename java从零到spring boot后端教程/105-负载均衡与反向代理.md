# 105-负载均衡与反向代理

> 💡 一家银行开了 5 个柜台。没有大堂经理时，客户全涌向第一个柜台，后面 4 个柜员玩手机。有了大堂经理——她站在门口，看哪个柜台空闲就把下一个客户分过去。负载均衡就是这位"大堂经理"——把请求均匀分到多台服务器，避免一台累死、其余闲死。

## 105.1 正向代理 vs 反向代理

### 正向代理（帮客户端）

```
你（客户端）─→ 代理服务器 ─→ google.com
    │              │
    │    代理知道你是谁，google 不知道你是谁
    │    用途：翻墙、访问内网、隐藏客户端 IP
```

### 反向代理（帮服务端）

```
客户端 ─→ Nginx（反向代理）─┬→ 服务器 A (192.168.1.10)
           │                ├→ 服务器 B (192.168.1.11)
           │                └→ 服务器 C (192.168.1.12)
           客户端不知道后面有几台服务器
           用途：负载均衡、SSL 卸载、缓存、安全防护
```

## 105.2 Nginx 反向代理配置

### 安装

```bash
# Windows: 下载 nginx.org 的 zip，解压后运行 nginx.exe
# Linux: sudo apt install nginx
# macOS: brew install nginx
```

### 基础反向代理

```nginx
# nginx.conf
server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://localhost:8080;   # 转发到 Spring Boot
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

### 负载均衡（Upstream）

```nginx
upstream backend {
    # 默认轮询
    server 192.168.1.10:8080 weight=3;   # weight=3 权重
    server 192.168.1.11:8080 weight=1;
    server 192.168.1.12:8080 backup;    # backup=备用，正常情况下不参与
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://backend;
    }
}
```

## 105.3 四种负载均衡算法

### 1. 轮询（Round Robin）

```
请求1 → 服务器 A
请求2 → 服务器 B
请求3 → 服务器 C
请求4 → 服务器 A  ← 轮流
```

Nginx 默认算法。假设所有服务器性能一致。

### 2. 加权轮询（Weighted Round Robin）

```
请求1-3 → 服务器 A (weight=3)
请求4   → 服务器 B (weight=1)
```

适合服务器配置不同的场景。

### 3. IP Hash

```nginx
upstream backend {
    ip_hash;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}
```

同一 IP 的请求永远转发到同一台服务器。适合需要 Session 保持的场景。

### 4. 最少连接（Least Connections）

```nginx
upstream backend {
    least_conn;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}
```

请求发给当前连接数最少的服务器。适合请求处理时间差异大的场景。

## 105.4 Nginx 常用配置

```nginx
# 静态文件缓存
location /static/ {
    root /var/www;
    expires 30d;            # 缓存 30 天
    add_header Cache-Control "public, immutable";
}

# WebSocket 支持
location /ws/ {
    proxy_pass http://backend;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}

# 限流
limit_req_zone $binary_remote_addr zone=mylimit:10m rate=10r/s;

location /api/ {
    limit_req zone=mylimit burst=20 nodelay;
    proxy_pass http://backend;
}

# HTTPS
server {
    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    location / {
        proxy_pass http://backend;
    }
}

# HTTP → HTTPS 重定向
server {
    listen 80;
    server_name api.example.com;
    return 301 https://$host$request_uri;
}
```

## 105.5 Spring Cloud LoadBalancer（服务端负载均衡）

在微服务中，负载均衡不仅发生在 Nginx 层，也发生在服务调用层：

```
订单服务
  │ 调 user-service
  │
  ▼
LoadBalancer
  │ 轮询选择：
  ├── user-service 实例 1 (192.168.1.10:8081)
  ├── user-service 实例 2 (192.168.1.11:8081)
  └── user-service 实例 3 (192.168.1.12:8081)
```

```java
// 自定义负载均衡策略
@Configuration
public class LoadBalancerConfig {

    @Bean
    public ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
            Environment environment, LoadBalancerClientFactory factory) {
        String name = environment.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
        return new RandomLoadBalancer(
                factory.getLazyProvider(name, ServiceInstanceListSupplier.class),
                name);
    }
}
```

## 105.6 小结

| 知识点 | 核心内容 |
|--------|----------|
| 正向代理 | 帮客户端隐藏身份 |
| 反向代理 | 帮服务端负载均衡 + 隐藏结构 |
| Nginx upstream | 服务器池 + 算法 |
| 四种算法 | 轮询 / 加权轮询 / IP Hash / 最少连接 |
| Spring Cloud LB | 微服务间调用的客户端负载均衡 |

**2. 你的 3 台服务器配置分别是 8 核 32G、4 核 16G、2 核 8G。用什么负载均衡算法最合适？Nginx 怎么配置？**

**3. IP Hash 适用于什么场景？什么时候不该用 IP Hash？**

---

**答案提示**：1→正向代理代表客户端发请求（VPN、Shadowsocks），反向代理代表服务端收请求（Nginx、HAProxy）。2→用加权轮询：`server big weight=4; server medium weight=2; server small weight=1;`。3→适用于需要 Session 保持但不方便做共享 Session 的场景。不该用于 NAT 网络（大量用户共享同一出口 IP → 全部打到同一台服务器）、或移动端 IP 频繁变化的场景。下一章——高并发系统设计。
