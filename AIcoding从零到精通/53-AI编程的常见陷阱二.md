## 陷阱五：上下文爆炸——"我把整个项目都塞进去了"

### 症状

你打开Cursor，选中了项目里10个文件，全部拖到Composer里，然后对AI说："帮我加一个用户头像上传功能，要适配所有已有的代码。"

AI开始工作了。你等啊等——AI输出了500行代码，但是：

- 第2行就改了一个你根本没想让它改的文件
- 第5行引入了一个你项目中不存在的库
- 前面自己写的逻辑，后面自己又推翻了
- 最后输出的代码，和你的项目风格完全不一样

**你给的上下文太多了，AI"撑住了"。** 就像你给一个人同时塞了10本书让他读，然后问他："第三本书第47页写的什么？"——他反而什么都记不住了。

### 为什么会这样？

AI的"记忆"是有限的（📖 术语：上下文窗口，Context Window）。想象AI有一个"黑板"，它只能在这块黑板上读写。你给它的文件越多，黑板上写的东西越多——**当黑板满了，它就会擦掉前面的内容。**

这就是为什么：
- 你给了10个文件，AI可能只"认真看了"前3个
- 后面它开始"瞎猜"，因为前面的信息已经被擦掉了
- 它输出的代码和你的项目风格不一致——因为它"忘了"你的项目风格

更糟糕的是：**当上下文太满时，AI的输出质量会断崖式下降。** 不是慢慢变差，是突然从80分掉到20分。

### 怎么办？

**铁律：只给AI"相关的"代码，不要给"所有的"代码。**

```
❌ 坏的上下文管理：
  整个项目10个文件全拖进去 → AI崩溃

✅ 好的上下文管理：
  只给AI看：
  1. 你要修改的目标文件（比如 user.py）
  2. 目标文件引用的关键文件（比如 models.py 里 User 模型的定义）
  3. 项目配置文件（比如 requirements.txt 或 package.json，让AI知道用了哪些库）
  → 通常3-4个文件就够了
```

**具体操作指南：**

```
场景一：你要在某个函数里加新功能
  → 只给AI看那个函数所在的文件，以及它调用的关键依赖

场景二：你要新增一个API接口
  → 给AI看：路由文件 + 数据模型定义 + 一个已有的类似接口作为参考

场景三：你要修一个bug
  → 给AI看：报错信息 + 出错的代码文件 + 相关的配置
  → 不需要给整个项目！
```

**一个实用技巧**：如果你不确定该给AI看哪些文件，先问自己："一个人要理解这段代码，最少需要看哪些文件？"——答案就是你该给AI的文件。

> **记住：AI不是"越多越好"，是"越精准越好"。** 3个精准的文件远远好过10个无关的文件。

## 陷阱六：工具频繁切换——"我成了工具收藏家"

### 症状

你用了Cursor一周，感觉不错。然后朋友说："你试试Windsurf，比Cursor还好用！"

你安装Windsurf，用了两天。又看到一篇推文："GitHub Copilot X 革命性更新！"

你切换Copilot，配置了半天。然后YouTube推送："Claude Code 是2026年最好的AI编程工具！"

你又去试Claude Code……

**三个月后**，你用过Cursor、Windsurf、Copilot、Claude Code、Trae、Cody、Codeium……但你一个都不精通。每换一个工具，你就要重新学快捷键、重新适应交互方式、重新配置项目。

你变成了**工具收藏家**——收藏了工具箱里所有的锤子，但一把都没用好。

### 为什么这很危险？

1. **学习成本累积**：每个工具都有独特的交互方式。Cursor的Tab补全和Trae的Builder完全不同。你每换一个工具，就浪费了之前积累的"肌肉记忆"。

2. **深度不够**：高级功能需要时间沉淀。Cursor的Agent模式、Composer的 `.cursorrules` 配置——这些是你用了1-2个月才能掌握的。你换工具太快，永远停留在"初级用户"水平。

3. **决策疲劳**：每次写代码前你都在想："这次用哪个工具？"——这个决策本身消耗了你的精力，而这些精力本该用来思考代码本身。

4. **FOMO陷阱**：FOMO（Fear Of Missing Out，害怕错过）是工具频繁切换的根本原因。你总觉得"下一个工具更好"，但你忘了——**最好的工具是你用熟的那个。**

### 怎么办？

**铁律：选1-2个主力工具，至少用满3个月，再考虑换。**

```
推荐策略：
1. 选定主力工具：Cursor 或 Trae（本书重点讲的就是这两个）
2. 选定备用工具：另一个（比如主力Cursor，备用Trae）
3. 约定"试用期"：新工具出来，先给30分钟试用，不好用就删，别纠结
4. 关注"能力"而非"工具"：你真正需要的是"写好代码"，不是"用遍所有工具"
```

**一个判断标准**：如果一个新工具让你觉得"好厉害！"，但你没有具体的使用场景——那它就是**痒点**，不是**痛点**。痒点不值得换工具，痛点才值得。

> **记住：工具是为你服务的，不是让你为它服务的。** 你不需要成为工具专家，你需要成为"用工具解决问题"的专家。

## 陷阱七：忽略版本控制——"AI改乱了，我回不去了"

### 症状

你让AI帮你改代码。AI改了A文件，又改了B文件，还改了C文件。

你跑了一下——报错了。你说："撤销刚才的改动。"

但是……你发现：

- Cursor的 `Ctrl+Z` 只能撤销文本编辑，跨越多个文件的改动它管不了
- 你根本记不住AI改了哪些文件
- 有个文件你手动改过，AI又改了一遍，你的手动修改被覆盖了
- 现在项目一团乱，你只想回到"让AI改之前"的状态——但回不去了

**你为什么不早点用Git？**

### Git是什么？

📖 **Git（版本控制）** = 给你代码的每一次修改拍一张"快照"。你可以随时回到任何一张快照。就像打游戏时的"存档"——打BOSS之前存个档，打不过就读档重来。

Git是最最基础的程序员工具，在AI时代它**更加重要**——因为AI会帮你做大量修改，你需要一个"后悔药"。

### 为什么AI时代Git更加重要？

```
传统编程：
  你手动写代码 → 改动很小（一次改几行）→ Ctrl+Z 够用

AI编程：
  AI帮你写代码 → 改动很大（一次改几百行、多个文件）→ Ctrl+Z 不够用！
  → 你必须用Git！
```

**AI编程的铁律：每次让AI改代码之前，先commit（存档）。**

### 怎么办？

**最简Git工作流（5分钟学会）：**

```bash
# 第一步：初始化（只需要做一次）
git init                                    # 在项目根目录创建Git仓库
git add .                                   # 把所有文件加入跟踪
git commit -m "初始版本"                     # 拍第一张快照

# 第二步：每次让AI改代码之前
git add .                                   # 把当前改动加入暂存区
git commit -m "让AI改代码之前"              # 拍快照！保存当前状态

# 第三步：AI改完，如果满意
git add .                                   # 把AI改的内容加入暂存区
git commit -m "AI帮我加了用户登录功能"      # 存档

# 第四步：AI改完，如果不满意 → 回退！
git checkout -- .                           # 放弃所有改动，回到上次commit的状态
# 或者更精确地：
git diff                                    # 先看看AI改了什么
git checkout -- <文件名>                    # 只回退某个文件
```

**一句话总结：**

```
AI改代码之前 → git commit（存档）
AI改坏了 → git checkout（读档）
AI改好了 → git commit（存档）
```

> **如果你只用Git做一件事，就做这件事：每次让AI改代码之前，先commit。** 这会救你无数次。

## 小结

| 陷阱 | 一句话 | 应对策略 |
|------|--------|----------|
| 上下文爆炸 | 塞太多无关代码，AI输出质量断崖式下降 | 只给3-4个相关文件，精准而不是多 |
| 工具频繁切换 | 变成工具收藏家，一个都不精通 | 选1-2个主力工具，用满3个月再考虑换 |
| 忽略版本控制 | AI改乱了回不去，欲哭无泪 | 每次让AI改之前先 `git commit` |

## 自测

1. 你要让AI帮你在一个已有项目里加一个新功能。你应该给AI看哪些文件？给出一个"最少但够用"的文件清单。

2. 你的朋友推荐了一个新出的AI编程工具，说"比Cursor好用10倍"。你应该怎么处理？给出你的判断流程。

3. Git在AI编程时代最重要的一个用法是什么？用一句话回答。

> **答案**：
> 1. 最少给3个文件：①你要修改的目标文件 ②目标文件依赖的关键模型/配置 ③一个已有的类似功能文件作为参考。如果项目有 `requirements.txt` 或 `package.json`，也加上。不需要给整个项目。
> 2. ①先判断：这个新工具解决了我现在遇到的什么具体问题？如果只是"听起来很厉害"但没有实际痛点，不换。②给30分钟试用，如果30分钟内没有明显效率提升，放弃。③真正有价值的工具，等它出1-2个月稳定版再考虑是否纳入工作流。④主力工具用满3个月再评估是否换。
> 3. 每次让AI改代码之前，先 `git commit` 存档。这样AI改坏了可以随时回退。
