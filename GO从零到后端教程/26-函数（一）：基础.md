# 第26章 · 函数（一）：基础

> "你每天刷牙用的牙膏，本质上是氟化物+摩擦剂+发泡剂的组合。每次用的时候，你不会自己重新配——你只是挤一下。函数就是编程世界的'牙膏'：把一段逻辑封装好，要用的时候'挤一下'就行。"

---

## 26.1 函数是什么

在现实生活中，很多事情都有固定的流程：

- **泡方便面**：烧水 → 撕开包装 → 放面饼 → 倒开水 → 等3分钟 → 倒调料包 → 搅拌 → 吃
- **寄快递**：打包 → 称重 → 填单 → 付钱 → 拿快递单号
- **登录网站**：输入用户名 → 输入密码 → 点击登录 → 等待验证

这些流程的共同特点是：**输入什么，输出什么，过程是固定的**。泡面的输入是面饼和水，输出是一碗泡好的面。寄快递的输入是物品和地址，输出是一个快递单号。

编程里的函数跟这个完全一样：**一段有名字的代码，接收输入（参数），执行操作，返回输出（返回值）**。

用Go写一个泡面函数：

```go
func makeInstantNoodles(waterTemperature int) string {
    if waterTemperature < 80 {
        return "水不够烫，再烧一会"
    }
    return "泡面完成，请享用"
}
```

用一次就是"挤一次牙膏"：

```go
result := makeInstantNoodles(90)
fmt.Println(result)
```

不用函数的话，每次泡面你都要把整个过程重新写一遍。那不是编程，那是罚抄作业。

## 26.2 函数的声明：解剖 func

Go里函数的语法结构就像搭积木，每一块都不能少：

```
func  函数名  (参数列表)  返回值类型  {
    函数体
    return 结果
}
```

拆开来看每一块：

### func 关键字

Go用 `func` 开头，就像写作文用"今天"开头一样——这是规矩。没有为什么，语言设计者这么定的。

### 函数名

函数名就是它的名字。Go的命名规则你已经知道了：**驼峰命名**。

```go
func calculateTotalPrice(price, quantity int) int {
    return price * quantity
}
```

`calculateTotalPrice`——一看就知道是"计算总价"。好的函数名应该让你一眼看懂它干什么，而不是让你猜。比如：

| 烂名字 | 好名字 | 为什么 |
|--------|--------|--------|
| `do()` | `sendEmail()` | `do`太模糊了，do什么？ |
| `calc()` | `calculateTax()` | `calc`可以缩写但`calculate`更清晰 |
| `get()` | `getUserByID()` | 获取什么？从哪获取？ |
| `p()` | `print()` | 单字母只在极短函数里偶尔用 |

### 参数列表

参数是函数的"输入口"。你想让函数帮你算两个数的和，你得告诉它"哪两个数"。

```go
func add(a int, b int) int {
    return a + b
}
```

`a int` 的意思是：参数名叫 `a`，类型是 `int`。`b int` 也一样。

如果多个参数类型相同，可以简写：

```go
func add(a, b int) int {
    return a + b
}
```

`a, b int` 等价于 `a int, b int`。Go会自动把 `int` 分配给前面逗号分隔的每一个参数名。

**没有参数怎么办？** 括号里什么都不写：

```go
func sayHello() string {
    return "你好，世界"
}
```

### 返回值类型

函数干完活，总要给个交代。交代的内容就是返回值。

```go
func getPi() float64 {
    return 3.14159
}
```

`float64` 写在参数括号和函数体花括号之间，表示"这个函数会返回一个float64类型的值"。

**没有返回值怎么办？** 不写返回值类型：

```go
func printGreeting(name string) {
    fmt.Println("你好，", name)
}
```

这种函数叫做"只干活不交差型"。比如往屏幕上打印东西、往数据库写数据——做了事，但不需要返回什么东西给调用者。

### 函数体

花括号 `{}` 里面的就是函数体——你要执行的代码。

```go
func isAdult(age int) bool {
    if age >= 18 {
        return true
    }
    return false
}
```

**return 必须要写**，只要函数声明了返回值类型。Go不会帮你"默认返回什么"。你承诺了返回 `int`，就必须每一个分支都 `return` 一个 `int`。

如果写了返回类型但不return，编译器会直接报错——Go在这方面非常严格。

## 26.3 参数传递：Go只有值传递

这是一个让很多初学者困惑的地方，我们用一个比喻来讲清楚。

### 比喻：复印机

你去复印店复印身份证：

- **值传递**：复印机把你的身份证复印了一份，给了别人一份复印件。别人在复印件上乱画，你的原件毫发无损。
- **引用传递**：你把身份证原件直接给别人。别人在上面乱画，你的原件就毁了。

Go语言里，**所有参数传递都是值传递**——永远是"复印件"。不存在引用传递。

但问题来了：为什么有时候传了切片进去，函数里改了，外面也变了？这跟"复印件"的矛盾怎么解释？

### 基本类型的值传递：确实互不影响

```go
func double1(n int) {
    n = n * 2
}

func main() {
    x := 10
    double1(x)
    fmt.Println(x)
}
```

输出是 `10`。为什么？因为 `double1(x)` 传的是x的"复印件"——一个x的副本。函数里把副本改成了20，但原件x还是10。

这就好比你复印了一份自己的成绩单，你在复印件上把60分涂改成90分——原件还是60分，不会因为你涂改复印件就变。

### 切片的值传递：复印件和原件"共享同一张纸"

```go
func changeFirst(s []int) {
    s[0] = 999
}

func main() {
    nums := []int{1, 2, 3}
    changeFirst(nums)
    fmt.Println(nums)
}
```

输出是 `[999 2 3]`。明明说好了值传递，怎么外面也变了？这不是自相矛盾吗？

**不矛盾。** 关键是要理解切片的底层结构。

切片不是一个简单的数据，它是一个**描述符**——包含三个信息：
- **指针**：指向底层数组的某个位置
- **长度**：切片里有几个元素
- **容量**：底层数组从指针位置开始还能装几个

当你传切片给函数时，Go确实复制了一份"描述符"。但是，**描述符里的指针，还是指向同一个底层数组**。

```
原件nums的描述符：  {指针→[1,2,3], 长度=3, 容量=3}
                    ↓ 复制
副本s的描述符：     {指针→[1,2,3], 长度=3, 容量=3}
```

副本的指针和原件的指针指向同一个地方。你通过副本修改 `s[0]`，改的是底层数组 `[1,2,3]` 的第一个元素。而原件nums也指向同一个底层数组，所以原件也看到了变化。

**比喻**：切片不是一张纸，而是一张"取货单"。你把取货单复印了一份给别人——这是值传递。但复印的取货单和原件指向同一个仓库。别人拿着复印的取货单去仓库把货改了，你拿着原件去取货的时候发现货也变了。

所以，**切片的值传递不矛盾**——传的是"描述符的副本"，但指向的是同一个底层数据。

### Map的值传递：又是副本指向同一份数据

```go
func addToMap(m map[string]int) {
    m["新来的"] = 100
}

func main() {
    scores := map[string]int{"张三": 90}
    addToMap(scores)
    fmt.Println(scores)
}
```

输出是 `map[张三:90 新来的:100]`。跟切片一样的道理——Map内部也是通过指针指向底层哈希表的。传Map副本的时候，副本里的指针仍然指向同一个哈希表。

### 指针的值传递：把地址作为值传过去

```go
func changeValue(p *int) {
    *p = 999
}

func main() {
    x := 10
    changeValue(&x)
    fmt.Println(x)
}
```

输出是 `999`。这里也是值传递，传的是 `&x`（x的地址）的副本。但副本的地址值和原件的地址值是一样的——都指向x。通过地址找到x，然后修改它。

**比喻**：变量x是一栋房子。`&x` 是写在纸条上的房屋地址。你把纸条复印了一份给朋友——这是值传递。朋友拿着复印的纸条，根据地址找到了你的房子，把门漆成了红色。房子确实被改了，但纸条本身是副本来着。

### 总结表格

| 类型 | 传参行为 | 函数内修改会影响外部吗 |
|------|---------|---------------------|
| int、float、bool、string | 复制整个值 | ❌ 不会 |
| 数组 | 复制整个数组 | ❌ 不会 |
| 切片 | 复制描述符（指针不变） | ✅ 修改元素会 |
| Map | 复制描述符（指针不变） | ✅ 修改键值会 |
| 指针 | 复制地址值 | ✅ 解引用修改会 |
| 结构体 | 复制整个结构体 | ❌ 不会（除非里面有切片/map/指针） |

## 26.4 多返回值：Go的特色菜

很多语言（比如Java）的函数只能返回一个值。你要返回多个东西怎么办？要么包成一个对象，要么用输出参数。

Go直接支持多返回值——一个函数可以返回多个值，用逗号隔开：

```go
func divide(a, b int) (int, int) {
    quotient := a / b
    remainder := a % b
    return quotient, remainder
}

q, r := divide(10, 3)
fmt.Println(q, r)
```

输出是 `3 1`。被除数是10，除数是3，商是3，余数是1。

### 典型模式：返回值 + error

Go里最常见的多返回值模式是：**第一个返回值是真正的结果，第二个返回值是error**。

```go
func findUser(id int) (string, error) {
    if id <= 0 {
        return "", fmt.Errorf("用户ID不能为负数: %d", id)
    }
    return fmt.Sprintf("用户_%d", id), nil
}

name, err := findUser(-1)
if err != nil {
    fmt.Println("出错了:", err)
    return
}
fmt.Println("找到了:", name)
```

这几乎成了Go的"标准姿势"。任何一个可能出错的函数，都返回 `(结果, error)`。

为什么不像其他语言那样抛异常？Go的设计哲学是：**错误是正常的程序流程的一部分，不是什么"异常"**。文件找不到、网络超时、用户输入不合法——这些都是预期内可能发生的事，不应该用异常来处理。所以我们直接把错误作为返回值，调用者显式地检查它。

### 多返回值的接收

你可以接收全部返回值，也可以只接收部分（用 `_` 忽略不需要的）：

```go
q, _ := divide(10, 3)

_, r := divide(10, 3)

q, r := divide(10, 3)
```

`_` 叫"空白标识符"，相当于"这个值我不要了，扔掉"。

## 26.5 命名返回值

Go允许你给返回值取名字：

```go
func calculateStats(numbers []int) (sum int, avg float64) {
    for _, n := range numbers {
        sum += n
    }
    if len(numbers) > 0 {
        avg = float64(sum) / float64(len(numbers))
    }
    return
}
```

注意最后一行只有一个 `return`，没有跟任何值。因为 `sum` 和 `avg` 已经在函数签名里声明过了，Go会自动把这些命名了的值返回出去。这叫"裸返回"（naked return）。

### 好处

**自文档化**。函数签名本身就是文档：

```go
func splitPath(path string) (dir, file string) {
    ...
}
```

你一看就知道：给一个路径，返回目录名和文件名。不需要再去看注释或函数体。

裸返回在短函数里很方便：

```go
func nameAndAge() (name string, age int) {
    name = "张三"
    age = 25
    return
}
```

### 坏处

在长函数里，裸返回是灾难。

```go
// 注意：下面代码中 item.Price 涉及结构体，结构体详见第29章
func processOrder(orderID int) (totalPrice float64, err error) {

    sum = 0
    err = validateOrder(orderID)
    if err != nil {
        return
    }


    items, err := loadItems(orderID)



    if err != nil {
        return
    }
    for _, item := range items {
        sum += item.Price
    }
    return
}
```

50行之后你再看这个函数，`return` 到底返回了什么？你要一行一行往上追踪 `sum` 和 `err` 在哪里被修改过。这是阅读的噩梦。

**命名返回值还会在 `:=` 上产生隐蔽的问题**：

```go
func demo() (result int, err error) {
    result, err := someFunction()
    return
}
```

看起来没问题，但是 `:=` 会在函数作用域里**新声明** `result` 和 `err`，而不是给命名返回值赋值。结果裸返回的还是初始零值。这种bug非常难排查。

### 选择建议

| 场景 | 建议 |
|------|------|
| 函数很短（10行以内） | 可以用命名返回值，裸返回OK |
| 函数很长 | 不要用命名返回值，显式return |
| 返回值含义不清 | 用命名返回值作自文档化，但显式return |
| 有error返回值 | 谨慎使用，尤其注意变量遮蔽问题 |

## 🤔 想多一点：为什么Go只有值传递？

这不是Go懒惰，而是**故意的设计选择**。

引用传递有个问题：你看函数签名 `func f(x *int)`，你知道x是通过"引用"传进去的，函数里改了x外面也会变。但如果所有语言特性都支持引用传递，代码的可预测性就会下降。

Go的哲学是：**让行为显式化**。如果你想修改外部变量，就传指针——函数签名里 `*int` 明确告诉你"这个参数可能会被修改"。如果你看到 `int`，你就可以放心——这个变量不会被改。

这减少了"猜测"：一个变量到底会不会被函数悄悄改掉？看参数类型就知道了。
