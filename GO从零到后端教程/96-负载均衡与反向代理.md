# 第96章 · 负载均衡与反向代理

> "你的Go服务写得再快，一台机器也只能扛几千QPS。流量十倍百倍地涨，怎么办？加机器。但加机器之后，用户请求怎么'均匀'地分到各台机器上？这就是负载均衡（将请求分摊到多台服务器，详见附录I）要解决的问题。而反向代理（站在服务器前接客并转发请求的代理，详见附录I）是解决这个问题的第一站——它就像你服务大楼的前台，帮你接待、分流、挡刀。"

---

## 96.1 正向代理 vs 反向代理

### 正向代理：帮客户端上网

你上不了Google→开一个代理软件→你所有请求先经过代理→代理帮你访问Google→返回给你。

```
你（客户端） ──→ 代理服务器 ──→ Google（目标服务器）
                  ↑
              它帮你隐藏了你的IP
```

正向代理的"正"是指：**代理站在客户端这一边。**

VPN、公司内网的出口网关都是正向代理。

### 反向代理：帮服务器接客

你访问 `api.example.com`→你以为是直接连到后端的Go服务→实际上先连到Nginx（高性能Web服务器和反向代理工具，详见附录I）→Nginx把请求转发给后面的Go服务→Go服务处理完→通过Nginx返回给你。

```
你（客户端） ──→ Nginx（反向代理） ──→ Go服务1（10.0.1.5:8080）
                                     ──→ Go服务2（10.0.1.6:8080）
                                     ──→ Go服务3（10.0.1.7:8080）
```

反向代理的"反"是指：**代理站在服务器这一边。**

你完全不知道后面有几台Go服务、它们在哪个IP上、用的什么语言——你只知道"我访问了 api.example.com"。

### 对比表

| 维度 | 正向代理 | 反向代理 |
|------|---------|---------|
| 站在哪边 | 客户端 | 服务端 |
| 隐藏谁 | 客户端的IP | 服务端的IP和结构 |
| 谁配 | 客户端或网络管理员 | 服务端运维 |
| 典型工具 | Shadowsocks、Squid | Nginx、HAProxy、Traefik |
| 主要用途 | 翻墙、内网访问外网 | 负载均衡、SSL终止、缓存 |

## 96.2 Nginx入门

### 安装Nginx

**Docker方式（推荐）**：

```bash
docker run -d \
  --name nginx-tutorial \
  -p 80:80 \
  nginx:1.25
```

### 基本配置结构

Nginx的配置文件是 `/etc/nginx/nginx.conf`：

```nginx
worker_processes auto;

events {
    worker_connections 1024;
}

http {
    include       mime.types;
    default_type  application/octet-stream;

    server {
        listen       80;
        server_name  localhost;

        location / {
            root   /usr/share/nginx/html;
            index  index.html;
        }
    }
}
```

关键块：
- `worker_processes`：工作进程数（通常设为CPU核数）
- `events.worker_connections`：每个进程的最大连接数
- `http`：HTTP相关的全局配置
- `server`：一个虚拟主机（一个域名对应一个server块）
- `location`：URL路径匹配规则

### 反向代理配置

最常用的场景：把80端口的请求转发到Go服务（8080端口）。

```nginx
server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

`proxy_set_header` 的作用：告诉Go服务"真正的客户端是谁"——否则Go服务只能看到Nginx的IP。

### location匹配规则

`location` 是Nginx的核心路由机制，有多种匹配语法：

```nginx
location = /api/health {
    return 200 "OK";
}

location ^~ /api/admin/ {
    return 403 "Forbidden";
}

location ~* \.(jpg|png|css|js)$ {
    root /var/www/static;
}

location /api/ {
    proxy_pass http://backend:8080;
}

location / {
    root /var/www/html;
}
```

| 语法 | 含义 | 优先级 |
|------|------|--------|
| `= /path` | 精确匹配 | 最高 |
| `^~ /path/` | 前缀匹配（不检查正则） | 高 |
| `~ pattern` | 区分大小写的正则匹配 | 中 |
| `~* pattern` | 不区分大小写的正则匹配 | 中 |
| `/path/` | 普通前缀匹配 | 低 |

**匹配原则**：精确 > 前缀（^~） > 正则（按顺序） > 普通前缀（最长匹配）。

### 负载均衡 upstream

`upstream` 定义一组后端服务器，Nginx把请求分发到这些服务器上：

```nginx
upstream backend_servers {
    server 10.0.1.5:8080 weight=3;
    server 10.0.1.6:8080 weight=2;
    server 10.0.1.7:8080 weight=1;
    server 10.0.1.8:8080 backup;
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://backend_servers;
        proxy_connect_timeout 3s;
        proxy_read_timeout 30s;
    }
}
```

`backup` 表示备用服务器——其他服务器都挂了才用它。

## 96.3 负载均衡算法

### 循环赛 vs 生活中的比喻

负载均衡算法就是回答"下一个请求给谁"这个问题。就像发扑克牌——你可以一张一张轮流发（轮询），也可以看你朋友的牌运来决定给谁多一张。

### 轮询（Round Robin）

一人一张，轮流来。最公平，也最常用（Nginx默认）。

```
请求1 → 服务器A
请求2 → 服务器B
请求3 → 服务器C
请求4 → 服务器A
...
```

但问题是：如果各服务器性能不同（A是32核，C是4核），轮询会把C累死。

### 加权轮询（Weighted Round Robin）

给每台服务器一个"权重"，高权重的服务器分配更多请求。

```nginx
upstream backend {
    server 10.0.1.5:8080 weight=5;
    server 10.0.1.6:8080 weight=3;
    server 10.0.1.7:8080 weight=2;
}
```

每10个请求，大致分配：A得5个、B得3个、C得2个。

### IP Hash

同一客户端的请求永远发给同一台服务器。

```nginx
upstream backend {
    ip_hash;
    server 10.0.1.5:8080;
    server 10.0.1.6:8080;
}
```

原理：`hash(客户端IP) % 服务器数` → 固定的服务器。

**用途**：保持会话（Session）。用户登录后，Nginx把它一直导向同一台服务器。但有个大坑：如果加了一台服务器，hash取模的结果全变了→用户的Session全丢了。

### 最少连接（Least Connections）

把请求发给当前"最闲"的服务器。

```nginx
upstream backend {
    least_conn;
    server 10.0.1.5:8080;
    server 10.0.1.6:8080;
}
```

适合**长连接**场景（如WebSocket、文件下载）。

### 一致性Hash

IP Hash的升级版——加机器后，大部分hash映射不变，只有少量用户受影响。

主要在更专业的负载均衡器（如HAProxy、LVS）中使用，Nginx需要第三方模块。

### 算法选择

| 场景 | 推荐算法 |
|------|---------|
| 无状态服务（REST API） | 加权轮询 |
| 有状态服务（Session）| IP Hash / 一致性Hash |
| WebSocket / 长连接 | 最少连接 |
| 混合长连接+短连接 | 最少连接 |

## 96.4 健康检查

### 主动健康检查

Nginx定期探测后端服务器，挂了就自动踢出。

```nginx
upstream backend {
    server 10.0.1.5:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.6:8080 max_fails=3 fail_timeout=30s;

    keepalive 32;
}

server {
    location /api/ {
        proxy_pass http://backend;

        # 主动健康检查（仅Nginx Plus商业版支持）
        # health_check interval=10s
        #              fails=3
        #              passes=2
        #              uri=/api/health;
        # 开源版默认使用upstream中的max_fails/fail_timeout被动健康检查
    }
}
```

- `max_fails=3`：失败3次就标记为"不可用"
- `fail_timeout=30s`：标记为不可用后，30秒后重新尝试
- `health_check`：主动健康检查（需要Nginx Plus或第三方模块）

被踢出去的服务器在 fail_timeout 后会重新加入，Nginx尝试探测——如果还是失败，继续踢。

### 被动健康检查

不主动探测，而是在实际请求失败时判断：

```nginx
server 10.0.1.5:8080 max_fails=3 fail_timeout=60s;
```

普通用户请求失败也算一次"fail"。3次失败后该服务器被标记为"不可用"60秒。这种方式不需要额外带宽，但发现问题比主动检查慢。

## 96.5 会话保持问题

### 问题：登录了但"掉线"了

用户在你的网站登录成功→Session存在服务器A的内存里→下一个请求被Nginx转发到服务器B→B的Session里没有这个用户→用户被要求重新登录。

这就是**Session不一致**的问题——Nginx的轮询导致的。

### 解决方案一：IP Hash

```nginx
upstream backend {
    ip_hash;
    server 10.0.1.5:8080;
    server 10.0.1.6:8080;
}
```

同一IP永远走同一台服务器→Session不会丢。

**问题**：
- 公司/学校内网的所有用户看起来是一个IP→全部打到同一台服务器→负载不均
- 用户切换网络（WiFi→4G）→IP变了→Session丢了
- 加服务器→IP和服务器映射全变→所有用户掉线

### 解决方案二：Cookie Sticky

Nginx在用户第一次请求时，在Cookie里写入"你该去哪台服务器"，后续请求带着这个Cookie→Nginx按Cookie的内容路由。

```nginx
upstream backend {
    server 10.0.1.5:8080;
    server 10.0.1.6:8080;

    sticky cookie srv_id expires=1h domain=.example.com path=/;
}
```

需要Nginx Plus或开源版的 `nginx-sticky-module`。

### 解决方案三：Redis共享Session

**最好的方案**——Session不存服务器本地，存在Redis中：

```go
store, _ := redis.NewStore(10, "tcp", "localhost:6379", "", []byte("secret-key"))

session, _ := store.Get(r, "session-name")
session.Values["user_id"] = "123"
session.Save(r, w)
```

无论Nginx把请求发到哪台Go服务器，Go服务都从Redis读Session——每台机器看到相同的Session数据。

这就是**无状态化**的精髓：服务器不存任何用户相关的状态→所有服务器完全等价→Nginx怎么轮询都没关系。

🤔 **想多一点**：负载均衡的真正价值

很多人以为负载均衡只是"让流量均匀分散"。但它的真正价值在于：

1. **灰度发布**：先让1%的流量走新版本→没问题→慢慢加大比例→全部切换。Nginx upstream权重就是最简单的灰度发布工具
2. **蓝绿部署**：保留一半机器跑旧版本，一半跑新版本→新版本出问题→立刻切回旧版本
3. **故障隔离**：一台机器OOM了→Nginx把它踢出→用户无感知
4. **扩容偷偷进行**：半夜悄悄加两台机器→改Nginx配置→reload→没有用户知道

负载均衡不只是"分流量"，而是**运维能力的放大器**。有了它，你可以随时调整架构而不惊动用户。

## 本章小结

| 知识点 | 要点 |
|--------|------|
| 正向代理 | 站在客户端这边，隐藏客户端IP |
| 反向代理 | 站在服务端这边，隐藏服务端结构，提供负载均衡 |
| Nginx基本配置 | `server`块管域名，`location`管路径，`upstream`管后端 |
| 轮询 | 默认算法，一人一张轮流来 |
| 加权轮询 | 高性能机器权重高，分更多流量 |
| IP Hash | 同一IP固定走同一服务器，加机器会全乱 |
| 最少连接 | 发给最忙的服务器，适合长连接 |
| 一致性Hash | IP Hash的升级版，加机器只影响少量用户 |
| 健康检查 | 主动（定期探活）+ 被动（请求失败计数），自动踢出/加入 |
| Session共享 | IP Hash/Cookie Sticky/Redis集中存储——Redis方案最优雅 |
