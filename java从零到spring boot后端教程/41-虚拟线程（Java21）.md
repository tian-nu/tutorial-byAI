# 第41章 虚拟线程（Java 21）

> 🚀 **这是 Java 21 最具革命性的特性。** 如果你问我"学 Java 这些年来最激动人心的变化是什么"，我会毫不犹豫地说：虚拟线程。它让 Java 终于能像 Go 的 goroutine 一样轻松处理百万级并发。Spring Boot 3.2 已经内置支持——你现在写的后端代码，很快就能用上它。

## 41.1 传统线程的痛点

### 41.1.1 平台线程的昂贵成本

前几章学的线程，本质上都是**平台线程**（Platform Thread）——与操作系统线程一一对应。

```
每个平台线程的代价：
├── 线程栈内存：约 1MB（默认）
├── 操作系统调度：内核态切换，成本较高
├── 创建/销毁：系统调用开销
└── 上下文切换：约 1-10 微秒（看起来小，但百万次就是秒级）
```

**简单算一笔账：**

```
1 个线程 = 1MB 栈内存
1000 个线程 = 1GB 栈内存
1000000 个线程 = 1TB 栈内存 ← 根本不可能！
```

这就是为什么我们在第 40 章费尽心思学线程池——不是为了方便，而是**被迫**。不控制线程数，系统就崩了。

### 41.1.2 "一个请求一个线程"模型的困境

```java
// 典型的 Servlet/Spring Boot 模式
@GetMapping("/users/{id}")
public User getUser(@PathVariable Long id) {
    User user = userRepo.findById(id);         // 可能阻塞（数据库查询）
    List<Order> orders = orderRepo.findByUser(id); // 又阻塞
    user.setOrders(orders);
    return user;  // 整个过程中线程被占用，什么都做不了
}
```

每个 HTTP 请求占用一个平台线程，线程大部分时间在**等待**（数据库 I/O、网络 I/O）。一个 200 线程的 Tomcat 只能同时处理 200 个请求——第 201 个只能排队。

## 41.2 虚拟线程：轻量级革命

### 41.2.1 什么是虚拟线程

> 🏷️ 此术语需进附录：**虚拟线程**（Virtual Thread）——Java 21 正式发布的轻量级线程实现（Project Loom）。虚拟线程由 JVM 管理而非操作系统，创建成本极低（每个约几百字节），可以轻松创建数十万甚至百万个。当虚拟线程阻塞时，底层平台线程被释放去执行其他虚拟线程。

```
平台线程（Platform Thread）：        虚拟线程（Virtual Thread）：
┌─────────────────────┐            ┌─────────────────────┐
│     OS Thread       │            │   JVM 管理的协程     │
│  ~1MB 栈内存        │            │  ~几百字节对象       │
│  操作系统调度       │            │  JVM 调度（用户态）  │
│  创建 ~1ms          │            │  创建 ~1μs           │
│  最多数千到数万     │            │  可达百万级          │
│  切换 ~10μs         │            │  切换 ~纳秒级        │
└─────────────────────┘            └─────────────────────┘
```

**核心机制**：多个虚拟线程复用在少量平台线程上。当虚拟线程执行阻塞操作（I/O、sleep）时，它自动从平台线程"卸载"，平台线程立刻去执行另一个虚拟线程。对开发者来说，虚拟线程**看起来、用起来和普通线程一模一样**。

```
                       ┌──────────────┐
                       │ OS Thread 1  │──→ VT1 → VT4 → VT7 → ...
                       ├──────────────┤
                       │ OS Thread 2  │──→ VT2 → VT5 → VT8 → ...
                       ├──────────────┤
                       │ OS Thread 3  │──→ VT3 → VT6 → VT9 → ...
                       └──────────────┘
                            ↑ 平台线程（少数几个）
                            ↑ 虚拟线程（成千上万个）在上面复用
```

> 🤔 **想多一点**：虚拟线程和 Go 的 goroutine 思路类似，但实现不同。Go 的协程栈可以动态增长（从几 KB 开始），Java 虚拟线程的栈存储在堆上的 Continuation 对象中，可以在阻塞时被挂起和恢复。两者都实现了"百万并发"的目标。

### 41.2.2 创建虚拟线程

Java 21 提供了多种创建方式：

```java
// 方式1：Thread.startVirtualThread() —— 最简便
Thread vThread = Thread.startVirtualThread(() -> {
    System.out.println("我在虚拟线程中运行！");
});
vThread.join();

// 方式2：Thread.ofVirtual().start()
Thread vThread2 = Thread.ofVirtual()
    .name("my-virtual-thread")
    .start(() -> {
        System.out.println("命名虚拟线程");
    });
vThread2.join();

// 方式3：Thread.ofVirtual().unstarted() —— 创建但暂不启动
Thread vThread3 = Thread.ofVirtual()
    .unstarted(() -> System.out.println("稍后启动"));
vThread3.start();
vThread3.join();

// 方式4：虚拟线程池 —— Executors.newVirtualThreadPerTaskExecutor()
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10000; i++) {
        int taskId = i;
        executor.submit(() -> {
            System.out.println("任务 " + taskId + " 在 " + Thread.currentThread());
        });
    }
} // try-with-resources 自动关闭
```

> ⚠️ **注意**：虚拟线程永远不能池化！`Executors.newVirtualThreadPerTaskExecutor()` 创建的是"每任务一个虚拟线程"的执行器，不是线程池。虚拟线程太便宜了，用完就扔。

## 41.3 百万并发的震撼

### 41.3.1 传统线程 vs 虚拟线程压力测试

```java
import java.time.Duration;
import java.util.concurrent.*;
import java.util.concurrent.atomic.AtomicInteger;

public class VirtualThreadBenchmark {
    public static void main(String[] args) throws Exception {
        int TASK_COUNT = 100_000;

        // --- 测试1：虚拟线程 ---
        AtomicInteger counter = new AtomicInteger(0);
        long start = System.currentTimeMillis();

        try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
            for (int i = 0; i < TASK_COUNT; i++) {
                executor.submit(() -> {
                    try {
                        Thread.sleep(Duration.ofSeconds(1)); // 模拟 I/O 等待
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                    }
                    counter.incrementAndGet();
                });
            }
        }

        System.out.println("虚拟线程: " + TASK_COUNT + " 个任务耗时 "
            + (System.currentTimeMillis() - start) + "ms, 完成: " + counter.get());

        // --- 测试2：传统平台线程池（200 个线程）---
        counter.set(0);
        start = System.currentTimeMillis();

        try (ExecutorService pool = Executors.newFixedThreadPool(200)) {
            for (int i = 0; i < TASK_COUNT; i++) {
                pool.submit(() -> {
                    try {
                        Thread.sleep(Duration.ofSeconds(1));
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                    }
                    counter.incrementAndGet();
                });
            }
        }

        System.out.println("平台线程(200): " + TASK_COUNT + " 个任务耗时 "
            + (System.currentTimeMillis() - start) + "ms, 完成: " + counter.get());
    }
}
```

**典型结果**（你的机器可能不同）：

```
虚拟线程: 100000 个任务耗时 ~1200ms, 完成: 100000    ← 几乎同时完成！
平台线程(200): 100000 个任务耗时 ~500000ms, 完成: 100000 ← 排队等了很久
```

10 万个并发任务，虚拟线程约 1.2 秒，200 线程池约 500 秒。**差距是 400+ 倍。**

### 41.3.2 原理分析

为什么虚拟线程快这么多？

```
传统平台线程（200个线程池）：
任务1-200 → 立即执行 → sleep 1s
任务201-400 → 排队等待 → sleep 1s
任务401-600 → 排队等待 → sleep 1s
...
10万任务需要 500 轮 → 500 秒

虚拟线程：
任务1-100000 → 全部立即启动 → 同时 sleep 1s
1 秒后全部恢复执行
1.2 秒完成（加上调度开销）
```

## 41.4 适用场景与不适用场景

### ✅ 适用场景（IO 密集型）

虚拟线程的超级能力体现在 **I/O 等待** 上：

```java
// 后端服务：同时调用多个微服务
CompletableFuture<String> user = asyncCall(() -> userService.getUser(id));
CompletableFuture<String> order = asyncCall(() -> orderService.getOrders(id));
CompletableFuture<String> payment = asyncCall(() -> paymentService.getHistory(id));

// 用虚拟线程，每个远程调用一个虚拟线程，完全不用线程池
```

- **Spring Boot Web 应用**：每个请求一个虚拟线程
- **数据库查询聚合**：同时查多个表
- **微服务调用**：并行调用多个下游服务
- **消息消费**：Kafka/RabbitMQ 消费者
- **文件处理**：并发读写大量文件

### ❌ 不适用场景（CPU 密集型）

虚拟线程不能加速纯 CPU 计算：

```java
// ❌ CPU 密集型：虚拟线程帮不上忙
ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();
for (int i = 0; i < 10000; i++) {
    executor.submit(() -> {
        // 纯计算：斐波那契、加密、压缩等
        fibonacci(50);  // 虚拟线程不会有任何优势
    });
}
```

原因：虚拟线程不增加 CPU 算力。它只是让 I/O 等待期间不浪费线程资源。纯 CPU 计算还是依赖物理核心数。

## 41.5 虚拟线程 vs 线程池

| 对比维度 | 传统线程池 | 虚拟线程 |
|---------|-----------|---------|
| 创建成本 | ~1ms | ~1μs |
| 内存占用 | ~1MB/线程 | ~几百字节/线程 |
| 最大数量 | 数千（受内存限制） | 百万级 |
| 阻塞代价 | 线程被占用 | 自动释放平台线程 |
| 需要池化 | 必须 | **不需要！**用完即弃 |
| 编程模型 | 复杂（池大小、队列） | 简单（每任务一线程） |
| 调试 | 线程名有意义 | 需要命名 |
| 适用 | CPU 密集 | IO 密集 |

> 🏷️ 此术语需进附录：**调度方式对比**——平台线程由操作系统内核抢占式调度（时间片轮转），虚拟线程由 JVM 在用户态协作式调度（在阻塞点自动切换）。

## 41.6 Spring Boot 3.2+ 集成虚拟线程

Spring Boot 3.2（基于 Java 21）只需一行配置即可启用虚拟线程：

```properties
# application.properties
spring.threads.virtual.enabled=true
```

这会让 Tomcat/Jetty 的请求处理线程、`@Async` 方法、`@Scheduled` 任务都使用虚拟线程。

```java
// 使用虚拟线程 Executor
@Bean
public Executor taskExecutor() {
    return Executors.newVirtualThreadPerTaskExecutor();
}

@Bean
public TomcatProtocolHandlerCustomizer<?> protocolHandlerVirtualThreadExecutorCustomizer() {
    return protocolHandler -> {
        protocolHandler.setExecutor(Executors.newVirtualThreadPerTaskExecutor());
    };
}
```

## 41.7 ❌ 常见错误与注意事项

### 错误1：把虚拟线程放进线程池

```java
// ❌ 错误！虚拟线程不需要池化
ExecutorService pool = Executors.newFixedThreadPool(10);  // 平台线程池
for (int i = 0; i < 10000; i++) {
    Thread vThread = Thread.ofVirtual().unstarted(() -> work());
    pool.submit(vThread);  // 在平台线程中启动虚拟线程——多此一举！
}

// ✅ 正确：直接用虚拟线程执行器
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10000; i++) {
        executor.submit(() -> work());
    }
}
```

### 错误2：在 synchronized 块中使用虚拟线程

```java
// ⚠️ 虚拟线程在 synchronized 块中阻塞时，会"钉住"平台线程
// （Java 21 的已知限制，未来版本可能修复）
synchronized (lock) {
    Thread.sleep(1000);  // 虚拟线程阻塞了，但平台线程也被占用！
    // 这失去了虚拟线程的优势
}

// ✅ 替代方案：使用 ReentrantLock
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    Thread.sleep(1000);  // 虚拟线程卸载，平台线程去执行其他任务
} finally {
    lock.unlock();
}
```

### 错误3：ThreadLocal 导致内存问题

```java
// ⚠️ 传统线程池中 ThreadLocal 是常用模式
// 虚拟线程是"用完即弃"的，大量虚拟线程 + ThreadLocal = 大量对象

// ✅ 如果必须用 ThreadLocal，考虑作用域限制的 ScopedValue（Java 21 预览）或及时清理
```

### 错误4：假设虚拟线程加速一切

```java
// ❌ 误区：虚拟线程让所有代码变快
// 虚拟线程只在 "I/O 密集型 + 高并发" 场景下有优势
// 单次查询、CPU 计算、低并发场景下用虚拟线程和不用的性能差不多
```

## 41.8 小结

| 知识点 | 一句话总结 |
|--------|-----------|
| 传统线程痛点 | 1MB 栈内存，OS 线程开销，必须用线程池限制数量 |
| 虚拟线程 | JVM 管理，几百字节，百万级并发，阻塞时自动释放平台线程 |
| 创建方式 | `Thread.startVirtualThread()` / `Thread.ofVirtual()` / `Executors.newVirtualThreadPerTaskExecutor()` |
| 适用场景 | IO 密集型高并发：Web 请求、数据库查询、微服务调用 |
| 不适用场景 | CPU 密集型计算、synchronized 块内阻塞 |
| Spring Boot 3.2 | `spring.threads.virtual.enabled=true` 一键启用 |

## 41.9 自测题

**1.** 虚拟线程和传统平台线程的本质区别是什么？为什么虚拟线程能支持百万并发？

**2.** 以下场景中，哪些适合用虚拟线程？

- A) 计算 1 到 1 亿的质数
- B) 同时向 1000 个下游 API 发送 HTTP 请求
- C) 处理 10 万用户的数据库查询（每个查询耗时 50ms）
- D) 对一张大图做傅里叶变换

**3.** 在 Java 21 中，为什么在 `synchronized` 块中调用阻塞操作（如 `Thread.sleep()`）会影响虚拟线程的性能？

> 答案提示：1. 虚拟线程由 JVM 而非 OS 管理，栈存在堆上（对象级别），阻塞时自动卸载释放平台线程；2. B 和 C 适合（IO 密集），A 和 D 不适合（CPU 密集）；3. Java 21 中虚拟线程在 synchronized 块内阻塞时会"钉住"底层平台线程，这是当前实现的一个限制，建议用 ReentrantLock 替代。
